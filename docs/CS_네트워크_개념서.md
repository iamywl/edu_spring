# CS 네트워크 개념서: HTTP 아래에서는 무슨 일이 일어나는가?

> "Spring이 `@GetMapping`으로 요청을 받을 때, 그 요청은 어디서 어떻게 여기까지 왔는가? 이 질문에 답할 수 없다면, 당신은 API를 다룰 뿐 네트워크를 다루는 것이 아니다."

---

## 들어가며

### 이 책의 목적

`SPRING_개념서.md`의 **Chapter 3(웹 애플리케이션의 구조)**는 HTTP를 **API 레벨**에서 설명한다. 요청/응답이 무엇인지, 메서드와 상태 코드가 무엇인지, REST를 어떻게 설계하는지 — 즉 "**HTTP라는 언어로 무엇을 말하는가**"를 다룬다.

이 책은 그 **아래**를 판다. 브라우저 주소창에 `https://api.example.com/users/1`을 입력하고 Enter를 누른 순간부터, Spring의 컨트롤러 메서드가 호출되기까지 — 그 사이에 **패킷 단위에서 실제로 무슨 일이 일어나는가**를 설명한다.

- SPRING §3.1은 "HTTP 요청은 편지와 같다"고 말한다. 이 책은 **그 편지가 어떤 봉투에 담겨, 어떤 우편 시스템을 거쳐, 어떻게 목적지 서버까지 배달되는지**를 설명한다.
- SPRING §5.7은 "TLS도 대칭/비대칭 원리다"라고 짚고 넘어간다. 이 책의 §6에서 **그 TLS 핸드셰이크를 본격적으로** 파헤친다.

**왜 비전공자가 이걸 배워야 하는가?** REST API만 알면 당장 CRUD는 짤 수 있다. 하지만 이런 순간에 벽에 부딪힌다:

- "왜 첫 요청만 느리고 그다음은 빠르지?" (DNS 캐시, TCP keep-alive)
- "왜 `localhost`는 되는데 IP로는 안 되지?" (바인딩 주소, 포트)
- "타임아웃이 났는데 서버 로그엔 아무것도 없다?" (연결이 서버에 도달조차 못 함 — 네트워크 계층)
- "HTTPS인데 왜 인증서 오류가 나지?" (CA 신뢰 체인)

이 질문들은 **API 문서에 답이 없다.** 네트워크 스택을 이해해야만 답할 수 있다. 이 책을 다 읽으면 "**API를 하나 호출하면 실제로 무슨 일이 벌어지는가**"를 계층별로 설명할 수 있게 된다. 그것이 전공자와 비전공자를 가르는 선이다.

### 이 책과 실습(lab)의 관계

이 책은 개념이고, `chapter-cs-network/labs/observe_http.sh`는 **그 개념을 눈으로 보는 실습**이다. 각 장 끝에 "**실습으로 확인**" 표시가 있다. 개념을 읽고 → 스크립트를 돌려 실제 패킷/헤더/인증서를 보면, 추상적이던 계층이 손에 잡힌다.


## Chapter 1: 왜 네트워크를 "층(Layer)"으로 나누는가

> "복잡한 문제를 다루는 인류의 유일한 방법은, 그것을 작은 문제로 쪼개는 것이다."

### 1.1 계층화(Layering)라는 아이디어

두 컴퓨터가 대화하려면 해결할 문제가 산더미다:

- 전선/전파로 0과 1을 어떻게 실어 보내지?
- 같은 케이블에 여러 기기가 붙어 있는데 누구에게 보낼지 어떻게 정하지?
- 지구 반대편 컴퓨터까지 어떻게 길을 찾아가지?
- 중간에 데이터가 유실되면 어떻게 다시 보내지?
- 도착한 바이트를 "HTTP 요청"으로 어떻게 해석하지?

이 모든 걸 **하나의 거대한 프로그램**으로 짜면 어떻게 될까? 아무도 이해하지 못하고, 하나를 고치면 전부 무너진다. 그래서 네트워크는 **계층(layer)**으로 나눈다.

**핵심 원리 — 각 층은 "바로 아래 층의 서비스만 쓰고, 바로 위 층에게만 서비스를 준다."**

```
  위 층은 아래 층이 "어떻게" 하는지 몰라도 된다.
  아래 층은 위 층이 "무엇을" 보내는지 몰라도 된다.

  ┌─────────────┐
  │  HTTP        │  "GET /users/1 보내줘"  ← 어떻게 배달되는지 모름/관심없음
  ├─────────────┤
  │  TCP         │  "이 바이트들을 순서대로, 유실 없이 배달"  ← 내용이 뭔지 모름
  ├─────────────┤
  │  IP          │  "이 덩어리를 저 IP 주소로"  ← 신뢰성은 내 일 아님
  ├─────────────┤
  │  링크/물리   │  "옆 장비로 전기신호 전송"  ← 목적지가 어디인지 모름
  └─────────────┘
```

**왜 이렇게 나누면 좋은가?**

- **교체 가능성**: Wi-Fi든 이더넷이든 4G든, 아래 층만 바뀌고 위의 TCP/HTTP는 그대로다. 카페에서 노트북으로 짜던 코드가 집 이더넷에서도 똑같이 돈다.
- **책임 분리**: HTTP를 짜는 사람은 "패킷 유실 재전송"을 신경 쓰지 않아도 된다. 그건 TCP의 일이다. (이것은 SPRING §3.4 "레이어드 아키텍처"에서 Controller/Service/Repository로 책임을 나눈 것과 **똑같은 철학**이다.)
- **표준화**: 층의 경계(인터페이스)만 표준으로 정하면, 그 안은 누가 어떻게 구현하든 상관없다.

### 1.2 OSI 7계층과 TCP/IP 4계층

역사적으로 두 개의 참조 모델이 있다. **OSI 7계층**은 교과서적 개념 모델이고, **TCP/IP 4계층**은 실제 인터넷이 돌아가는 실무 모델이다. 둘을 나란히 놓고 봐야 한다.

```
   OSI 7계층 (개념)              TCP/IP 4계층 (실제)         이 층에 사는 것들
 ┌──────────────────┐        ┌──────────────────┐
 │ 7. 응용 (App)     │        │                   │
 │ 6. 표현(Present.) │  ───→  │  응용 (App)        │   HTTP, DNS, TLS, FTP, SMTP
 │ 5. 세션(Session)  │        │                   │
 ├──────────────────┤        ├──────────────────┤
 │ 4. 전송(Transport)│  ───→  │  전송 (Transport)  │   TCP, UDP     (포트, 신뢰성)
 ├──────────────────┤        ├──────────────────┤
 │ 3. 네트워크(Net.) │  ───→  │  인터넷 (Internet) │   IP, ICMP     (IP주소, 라우팅)
 ├──────────────────┤        ├──────────────────┤
 │ 2. 데이터링크    │  ───→  │                   │
 │ 1. 물리(Physical) │        │  링크 (Link)       │   이더넷, Wi-Fi, MAC 주소
 └──────────────────┘        └──────────────────┘
```

**왜 두 개가 있는가?** OSI는 국제표준화기구가 "이상적으로 이렇게 나누면 좋겠다"고 설계한 모델이다. 반면 인터넷은 OSI가 완성되기 전에 이미 TCP/IP로 돌아가기 시작했고, 실전에서 이겼다. 그래서 **개념은 OSI 7계층 용어로 말하고(예: "그건 4계층 문제야"), 실제 구현은 TCP/IP로** 이해하는 게 현실이다.

**실무에서 자주 쓰는 표현:**
- "**L4** 로드밸런서" = 전송 계층(TCP/포트)에서 분배
- "**L7** 로드밸런서" = 응용 계층(HTTP 경로/헤더)에서 분배
- "**3계층에서 안 닿아**" = IP 라우팅 문제 (방화벽, 라우팅 테이블)

이 책은 아래에서 위로 올라간다: 물리/링크/네트워크(Ch2) → 전송(Ch3) → DNS(Ch4) → HTTP(Ch5) → TLS(Ch6). 마지막에 전부 합쳐 하나의 요청을 추적한다(Ch7).

### 1.3 캡슐화(Encapsulation): 봉투 속의 봉투

계층이 실제로 어떻게 협력하는가? 답은 **캡슐화**다. 각 층은 위 층에서 받은 데이터에 **자기 헤더(header)를 붙여서** 아래로 내린다. 마치 편지를 봉투에 넣고, 그 봉투를 더 큰 소포 상자에 넣고, 그 상자에 운송장을 붙이는 것과 같다.

```
  [보내는 쪽 — 위에서 아래로 내려가며 헤더를 붙인다]

  응용:   ┌────────────────────────────────┐
          │  GET /users/1 HTTP/1.1 ...      │   ← 실제 데이터 (HTTP 메시지)
          └────────────────────────────────┘
                        │ TCP가 감쌈
  전송:   ┌──────┬────────────────────────────────┐
          │TCP헤더│  GET /users/1 HTTP/1.1 ...      │   ← 세그먼트(Segment)
          └──────┴────────────────────────────────┘   TCP헤더: 출발/도착 포트, 순서번호
                        │ IP가 감쌈
  네트워크:┌──────┬──────┬────────────────────────────────┐
          │IP헤더 │TCP헤더│  GET /users/1 HTTP/1.1 ...      │   ← 패킷(Packet)
          └──────┴──────┴────────────────────────────────┘   IP헤더: 출발/도착 IP 주소
                        │ 이더넷이 감쌈
  링크:   ┌────┬──────┬──────┬────────────────────────┬────┐
          │이더넷│IP헤더 │TCP헤더│  GET /users/1 ...      │FCS │   ← 프레임(Frame)
          └────┴──────┴──────┴────────────────────────┴────┘   이더넷헤더: 출발/도착 MAC
                        │
                        ▼  전선/전파로 0과 1이 흐른다
```

받는 쪽에서는 **정확히 반대(역캡슐화, decapsulation)**로 진행한다. 링크 층이 이더넷 헤더를 떼고 IP에게 주고, IP가 IP 헤더를 떼고 TCP에게 주고, TCP가 TCP 헤더를 떼고 HTTP에게 원래 메시지를 준다. **각 층은 자기 헤더만 읽고 떼어낸다.** 아래 층 헤더는 어떻게 붙었는지 신경 쓰지 않는다.

**용어 정리 — 층마다 데이터 단위의 이름이 다르다(PDU):**

| 계층 | 데이터 단위 이름 | 붙는 주요 정보 |
|------|----------------|--------------|
| 응용 | 메시지(Message) | HTTP 헤더/바디 |
| 전송(TCP) | 세그먼트(Segment) | 포트, 순서번호(seq), 확인응답(ack) |
| 전송(UDP) | 데이터그램(Datagram) | 포트 |
| 네트워크(IP) | 패킷(Packet) | 출발/도착 IP |
| 링크 | 프레임(Frame) | 출발/도착 MAC, 오류검출(FCS) |

**왜 이 그림이 중요한가?** 나중에 `curl -v`나 `openssl`, `Wireshark`를 볼 때, 실제로 이 봉투들이 열리는 것을 보게 된다. "IP 헤더 안에 TCP 헤더가 있고, 그 안에 HTTP가 있다"는 이 중첩 구조가 네트워크의 전부라 해도 과언이 아니다.

> **실습으로 확인**: `RawHttpClient.java`는 이 스택의 **가장 위 두 칸(HTTP over TCP)**을 직접 만든다. `Socket`을 열면 TCP가 준비되고, 거기에 `"GET / HTTP/1.1..."` 텍스트를 쓰면 그게 HTTP다. 나머지 아래 층(IP, 링크)은 운영체제가 알아서 감싼다.

## Chapter 2: 아래 세 층 — 물리, 링크, 네트워크

> "인터넷은 '네트워크들의 네트워크(inter-network)'다. 서로 다른 수많은 네트워크를 하나로 묶는 접착제가 바로 IP다."

### 2.1 물리 계층과 링크 계층: 옆 장비에게 건네기

**물리 계층(L1)**은 가장 밑바닥이다. 0과 1을 실제 **전기 신호, 광 신호, 전파**로 바꿔 케이블/공기 중으로 보낸다. 이더넷 케이블의 전압, Wi-Fi의 주파수가 여기 산다. 개발자가 직접 다룰 일은 거의 없다.

**링크 계층(L2)**은 "**같은 네트워크 안에서 바로 옆 장비에게**" 데이터를 건넨다. 여기서 핵심이 **MAC 주소**다.

- **MAC 주소**(예: `a4:83:e7:1c:2b:9f`)는 네트워크 카드(NIC)에 **공장에서 새겨진 물리적 주소**다. 원칙적으로 전 세계에서 유일하고, 바뀌지 않는다.
- 스위치는 MAC 주소를 보고 "이 프레임은 3번 포트에 연결된 기기 것"이라고 판단해 전달한다.

### 2.2 MAC vs IP: 왜 주소가 두 개나 필요한가?

여기서 비전공자가 가장 헷갈리는 지점이다. **MAC 주소가 이미 있는데 왜 IP 주소가 또 필요한가?**

비유가 정확하다:

```
  MAC 주소 = 주민등록번호 (사람에게 고정, 평생 안 바뀜, 하지만 '위치' 정보 없음)
  IP 주소  = 현재 사는 집 주소 (위치 정보 있음, 이사하면 바뀜)

  택배를 보낼 때:
    - "홍길동(주민번호 900101-...)에게 보내"  →  전국 어디 있는지 알 수 없다! 배달 불가.
    - "서울시 강남구 테헤란로 1(주소)로 보내"  →  택배 시스템이 경로를 찾을 수 있다.
```

**MAC 주소로는 라우팅(경로 탐색)을 할 수 없다.** MAC 주소에는 "위치" 정보가 없기 때문이다. 40억 개의 MAC 주소가 지구상에 흩어져 있는데, 그중 하나를 찾으려면 전 세계 모든 장비에게 물어봐야 한다 — 불가능하다.

반면 **IP 주소는 '계층적'이라 위치를 담는다.** `211.234.x.x`는 "한국의 어느 통신사 → 어느 지역 → 어느 서브넷"처럼 주소 자체가 지도 역할을 한다. 그래서 라우터가 "이 IP는 저쪽 방향"이라고 경로를 계산할 수 있다.

**실제 동작 (핵심만):**
- **먼 곳(다른 네트워크)으로 갈 때** → IP 주소로 라우팅. 여러 라우터를 건너뛰며(hop) 목적지 네트워크까지 간다.
- **목적지 네트워크에 도착한 그 마지막 순간** → "이 IP를 가진 기기의 MAC 주소가 뭐지?"를 **ARP**라는 프로토콜로 물어, 최종적으로 MAC 주소로 프레임을 건넨다.

```
  내 PC ──프레임(MAC:라우터)──→ [라우터1] ──→ [라우터2] ──→ ... ──→ [목적지 라우터] ──프레임(MAC:서버)──→ 서버
         └── 매 구간(hop)마다 MAC 주소는 "다음 장비"로 바뀐다
         └── 하지만 IP 주소(출발지/목적지)는 여정 내내 그대로다 ★

  ★ 이것이 핵심: IP는 "최종 목적지"(변하지 않음), MAC은 "바로 다음 발걸음"(매번 바뀜)
```

### 2.3 IP 주소: IPv4와 IPv6

**IPv4**는 32비트 주소다. `192.168.0.1`처럼 0~255 숫자 4개(각 8비트)로 쓴다.

```
  192   .   168   .   0   .   1
  ↓        ↓        ↓      ↓
  11000000 10101000 00000000 00000001   ← 실제로는 32개의 비트
```

32비트면 총 2³² ≈ **43억 개**. 인터넷 초기엔 넘쳐 보였지만, 스마트폰·IoT 시대에 **주소가 고갈**됐다. 임시 해결책이 **NAT**(공유기가 사설 IP `192.168.x.x`를 하나의 공인 IP로 변환)와, 근본 해결책이 **IPv6**다.

**IPv6**는 128비트다. `2001:0db8:85a3::8a2e:0370:7334`처럼 16진수로 쓴다. 2¹²⁸는 사실상 무한(지구의 모래알보다 많다)이라 고갈 걱정이 없다.

```
  IPv4:  192.168.0.1                              (32비트,  ~43억 개)
  IPv6:  2001:db8:85a3::8a2e:370:7334             (128비트, 사실상 무한)
         └── '::' 는 연속된 0 묶음을 생략한 표기
```

### 2.4 서브넷과 CIDR: 네트워크를 나누는 선

IP 주소는 **"네트워크 부분"과 "호스트 부분"**으로 나뉜다. `192.168.0.1/24`에서 `/24`가 **CIDR 표기**로, "**앞 24비트가 네트워크 주소**"라는 뜻이다.

```
  192.168.0.1 / 24
  11000000.10101000.00000000 . 00000001
  └───────── 24비트 ────────┘   └─8비트─┘
       네트워크 부분(고정)        호스트 부분(가변)

  → 이 네트워크(192.168.0.0/24)에는 192.168.0.1 ~ 192.168.0.254 까지 호스트가 들어간다
    (192.168.0.0 은 네트워크 주소, 192.168.0.255 는 브로드캐스트 주소로 예약)
```

**왜 나누는가?** "**같은 서브넷이면 옆집(직접 전달), 다른 서브넷이면 라우터를 거쳐야 함(먼 곳)**"을 구분하기 위해서다. 내 PC가 어떤 IP로 데이터를 보낼 때 가장 먼저 하는 계산이 바로 이것이다: "목적지가 내 서브넷 안인가, 밖인가?" 밖이면 **게이트웨이(라우터)**에게 넘긴다. §2.2에서 본 "먼 곳 vs 옆 장비"의 판단이 여기서 일어난다.

### 2.5 라우팅의 직관: 다음 한 걸음만 안다

인터넷에는 중앙 지휘부가 없다. 그런데 어떻게 서울의 패킷이 뉴욕 서버에 도착할까? 비밀은 **"각 라우터는 전체 경로를 모른다. 오직 다음 한 걸음(next hop)만 안다."**

```
  네비게이션 비유:
    각 교차로(라우터)에 표지판이 있다.
    "뉴욕 방향? → 저쪽 3번 도로로."
    다음 교차로에 또 표지판. "뉴욕 방향? → 저쪽으로."
    → 아무도 전체 지도를 외우지 않지만, 각자 '다음 방향'만 알면 결국 도착한다.
```

각 라우터는 **라우팅 테이블**(목적지 IP 대역 → 다음에 넘길 방향)을 갖고 있고, 패킷의 목적지 IP를 보고 "이건 저쪽으로" 하나만 결정해 넘긴다. 이 과정을 목적지에 닿을 때까지 반복하는 것이 라우팅이다. (`traceroute` 명령으로 이 hop들을 실제로 볼 수 있다.)

## Chapter 3: 전송 계층 — 신뢰성은 여기서 만들어진다

> "IP는 '최선을 다하지만 보장은 안 하는(best-effort)' 배달부다. 패킷을 잃어버릴 수도, 순서를 뒤바꿀 수도, 중복 배달할 수도 있다. 이 엉망진창을 '순서대로, 빠짐없이'로 바꿔주는 것이 TCP다."

### 3.1 포트와 소켓: 한 컴퓨터 안의 여러 문

IP 주소는 **컴퓨터**를 특정한다. 하지만 한 컴퓨터에는 웹서버, 데이터베이스, SSH가 동시에 돌아간다. 도착한 데이터를 **누구에게** 줘야 하나? 그것을 정하는 것이 **포트(port)**다.

```
  IP 주소 = 아파트 동 (건물 하나)
  포트    = 호수 (건물 안의 각 집)

  211.234.1.5 : 443    →  443호에 사는 웹서버(HTTPS)
  211.234.1.5 : 3306   →  3306호에 사는 MySQL
  211.234.1.5 : 22     →  22호에 사는 SSH
```

**잘 알려진 포트(well-known):** 80(HTTP), 443(HTTPS), 22(SSH), 3306(MySQL), 5432(PostgreSQL), 8080(개발용 HTTP — Spring Boot 기본).

**소켓(Socket)**은 "**IP + 포트**"의 조합이자, 프로그램이 네트워크와 대화하는 **창구(추상화)**다. 하나의 연결은 **4개 값**으로 유일하게 식별된다:

```
  연결 = (출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)

  예) 내 브라우저가 example.com에 접속:
      (192.168.0.10, 51234) ────────→ (93.184.216.34, 443)
       └ 내 PC, 랜덤 포트              └ 서버, 443(HTTPS)

  탭을 하나 더 열면 출발지 포트만 51235로 바뀌어 '다른 연결'이 된다.
```

> **실습으로 확인**: `RawHttpClient.java`의 `new Socket(host, 80)`이 바로 이 소켓을 여는 코드다. 자바가 목적지 IP:포트로 향하는 창구를 만들고, 운영체제가 랜덤한 출발지 포트를 배정한다.

### 3.2 TCP vs UDP: 신뢰성이냐 속도냐

전송 계층에는 두 프로토콜이 있고, **선택은 트레이드오프**다.

```
  ┌──────────────┬────────────────────────┬────────────────────────┐
  │              │  TCP                     │  UDP                    │
  ├──────────────┼────────────────────────┼────────────────────────┤
  │ 연결          │ 연결형(handshake 필요)   │ 비연결형(그냥 던짐)      │
  │ 신뢰성        │ 보장(유실 시 재전송)     │ 보장 안 함(잃으면 끝)    │
  │ 순서          │ 보장(순서대로 재조립)    │ 보장 안 함              │
  │ 속도/오버헤드 │ 느림/무거움             │ 빠름/가벼움             │
  │ 비유          │ 등기우편(수령 확인)      │ 광고 전단지(그냥 뿌림)   │
  │ 쓰는 곳       │ HTTP, 파일전송, DB       │ DNS질의, 영상통화, 게임  │
  └──────────────┴────────────────────────┴────────────────────────┘
```

**왜 둘 다 필요한가?**
- 웹페이지, 계좌이체는 **한 글자도 틀리면 안 된다** → TCP. 느려도 정확해야 한다.
- 실시간 영상통화는 **0.1초 전 프레임을 재전송받아 봐야 이미 늦었다** → UDP. 조금 깨져도 지금 프레임이 중요하다.

HTTP/1.1과 HTTP/2는 TCP 위에서 돈다. 그래서 **Spring이 받는 모든 요청의 밑바닥엔 TCP가 있다.** (참고: 최신 HTTP/3는 UDP 기반의 QUIC를 쓰지만, 그 위에서 신뢰성을 스스로 다시 구현한다 — 이 책 범위 밖.)

### 3.3 TCP 3-way handshake: 대화를 시작하는 악수

TCP는 **연결형**이다. 데이터를 보내기 전에 양쪽이 "**대화 시작합시다**"를 합의하는데, 이것이 **3-way handshake**다. 왜 3번인가? 양쪽 모두 "**나도 보낼 수 있고, 상대도 받을 수 있다**"를 확인해야 하기 때문이다.

```
  클라이언트                                            서버
      │                                                 │
      │  ①  SYN  (seq=100)                              │
      │  "대화 시작하자. 내 시작 순서번호는 100이야"      │
      │ ──────────────────────────────────────────────→ │
      │                                                 │
      │  ②  SYN + ACK  (seq=300, ack=101)               │
      │  "좋아. 네 100 잘 받았어(다음엔 101 줘).          │
      │   내 시작 순서번호는 300이야"                     │
      │ ←────────────────────────────────────────────── │
      │                                                 │
      │  ③  ACK  (ack=301)                              │
      │  "네 300도 잘 받았어(다음엔 301 줘). 연결 완료!"  │
      │ ──────────────────────────────────────────────→ │
      │                                                 │
      │ ══════════ 연결 수립(ESTABLISHED) ══════════════ │
      │        이제부터 실제 데이터(HTTP 등)를 주고받음    │
```

- **SYN**(Synchronize): "연결하자 + 내 순서번호 알려줌"
- **ACK**(Acknowledge): "네 걸 잘 받았음 + 다음에 기대하는 번호"
- **왜 순서번호(seq)를 랜덤하게 시작하나?** 예전 연결의 잔여 패킷이나 위조 패킷과 섞이지 않게 하려고. 보안·정확성 둘 다를 위한 것이다.

**왜 이걸 알아야 하는가?** "첫 요청은 왜 느린가?"의 답이 여기 있다. 연결 하나 맺는 데 **최소 1번의 왕복(RTT)**이 든다. 지구 반대편 서버라면 이 악수만으로도 수백 ms가 걸린다. 그래서 §5.2의 **keep-alive**로 이 악수를 아끼는 것이다. `curl -v`를 돌리면 `Connected to ...`라는 줄이 바로 이 handshake 완료 순간이다.

> **실습으로 확인**: `observe_http.sh`의 `curl -v` 출력에서 `* Connected to example.com (...) port 443`이 3-way handshake가 끝난 지점이다. 그 직후부터 TLS 핸드셰이크(§6)가 이어진다.

### 3.4 신뢰성: 순서번호, 확인응답, 재전송

TCP가 "빠짐없이, 순서대로"를 어떻게 보장하는가? **모든 바이트에 순서번호(sequence number)를 매기고, 받은 쪽이 확인응답(ACK)을 보낸다.**

```
  보낸 쪽                                받는 쪽
    │  데이터[seq=1~100] ───────────────→  │  받음. "1~100 확인, 다음 101 줘" (ACK=101)
    │  ←──────────────────────────────────│
    │  데이터[seq=101~200] ──────X (유실)   │  ...안 옴...
    │                                      │
    │  (일정 시간 ACK 안 오면 = 타임아웃)    │
    │  데이터[seq=101~200] 재전송 ────────→ │  받음. "101~200 확인, 다음 201 줘"
    │  ←──────────────────────────────────│
```

- **순서번호**: 각 바이트에 번호 → 받는 쪽이 뒤죽박죽 도착해도 **번호순으로 재조립**한다.
- **확인응답(ACK)**: "여기까지 받았다"를 알림 → 보낸 쪽은 ACK가 온 데이터만 안심하고 버린다.
- **재전송**: ACK가 안 오면(타임아웃) 또는 중복 ACK가 오면 → **다시 보낸다.** 이것이 "유실돼도 결국 도착"을 만든다.

이 덕분에 HTTP를 짜는 사람은 "패킷이 유실되면 어쩌지?"를 **전혀 신경 쓰지 않는다.** TCP가 아래에서 다 처리해준다. 이것이 §1.1에서 말한 **책임 분리**의 실전 사례다.

### 3.5 흐름 제어와 혼잡 제어: 넘치지 않게, 막히지 않게

두 가지 "제어"가 더 있다. 이름이 비슷해 헷갈리니 **누굴 위한 배려인지**로 구분하라.

```
  흐름 제어(Flow Control)   →  '받는 상대'를 배려. 상대 버퍼가 넘치지 않게.
  혼잡 제어(Congestion C.)  →  '중간 네트워크'를 배려. 도로(회선)가 막히지 않게.
```

- **흐름 제어**: 받는 쪽이 "내 버퍼에 아직 이만큼 공간 있어(window size)"를 알려준다. 보내는 쪽은 그 창(window) 크기만큼만 보낸다. → **느린 수신자가 빠른 송신자에 압도되지 않게.**
- **혼잡 제어**: 네트워크 중간이 막히면(패킷 유실 감지) 전송 속도를 **확 줄였다가 천천히 다시 늘린다**(slow start). → 모두가 동시에 마구 보내 인터넷 전체가 마비되는 것을 막는다. 도로에 차가 몰리면 서로 양보해 정체를 푸는 것과 같다.

비전공자 수준에서는 "**TCP는 받는 쪽과 도로 상황을 봐가며 알아서 속도를 조절한다**"는 직관이면 충분하다.

### 3.6 연결 종료: 4-way close

시작이 악수 3번이면, 끝은 인사 4번이다. **양쪽이 각자 독립적으로 "나는 이제 보낼 거 없어"를 선언**해야 하기 때문이다.

```
  클라이언트                                         서버
      │  ① FIN   "나 이제 보낼 거 없어"  ──────────→   │
      │  ② ACK   "알았어"                ←──────────   │
      │            (서버는 아직 보낼 게 남았을 수 있음)  │
      │  ③ FIN   "나도 이제 다 보냈어"    ←──────────   │
      │  ④ ACK   "알았어. 잘 가"          ──────────→   │
      │                                               │
      │  (클라이언트는 잠깐 TIME_WAIT 상태로 대기 후 종료)│
```

- **왜 3번(handshake)이 아니라 4번인가?** 시작할 땐 서버의 "받았다(ACK)"와 "나도 시작(SYN)"을 **한 번에** 보낼 수 있었다(②SYN+ACK). 하지만 끝낼 땐, 클라이언트가 "끝"이라 해도 서버는 **아직 보낼 데이터가 남아 있을 수 있어서** ACK와 FIN을 나눠 보낸다.
- **TIME_WAIT**: 마지막 ACK가 유실될 경우를 대비해 잠시 기다린다. 서버를 재시작할 때 "Address already in use" 오류를 만나는 이유가 바로 이 TIME_WAIT 때문일 때가 많다.

> **실습으로 확인**: `RawHttpClient.java`에서 `Connection: close` 헤더를 보내면, 응답 후 서버가 이 종료 절차를 시작한다. `try-with-resources`로 소켓을 닫으면 클라이언트 쪽 FIN이 나간다.

## Chapter 4: DNS — 이름을 주소로

> "사람은 `google.com`을 기억하지만, 네트워크는 `142.250.x.x`로 길을 찾는다. 이 둘을 이어주는 전화번호부가 DNS다."

### 4.1 DNS가 해결하는 문제

§2에서 봤듯 실제 통신은 **IP 주소**로 이뤄진다. 하지만 사람이 `93.184.216.34`를 외울 순 없다. 그래서 `example.com` 같은 **도메인 이름**을 IP로 **번역**하는 시스템이 필요하다 — 그것이 **DNS(Domain Name System)**다.

```
  사람이 보는 것:   www.example.com
                       │  DNS가 번역
                       ▼
  네트워크가 쓰는 것: 93.184.216.34
```

**왜 이름을 쓰는가?** 기억하기 쉬운 것만이 아니다. 서버 IP가 바뀌어도(이사, 서버 증설) **이름은 그대로 두고 DNS 기록만 바꾸면** 사용자는 아무것도 안 해도 된다. 이름과 위치를 분리한 것 — §2.2에서 본 "MAC(고정 이름) vs IP(위치)"와 같은 발상의 한 층 위 버전이다.

### 4.2 재귀적 질의와 반복적 질의

DNS는 하나의 거대한 서버가 아니라 **계층적으로 분산된 시스템**이다. 도메인은 오른쪽부터 계층을 이룬다: `www.example.com.` = `루트(.) → com → example → www`.

내 PC가 이름을 물으면, 보통 통신사/회사의 **DNS 리졸버(recursive resolver)**가 나 대신 여러 서버를 돌며 답을 찾아온다.

```
  ① 내 PC ──"www.example.com 주소 뭐야?"──→ [리졸버(재귀)]
                                                │  (리졸버가 나 대신 아래를 다 돌아줌 = 재귀적)
     ┌──────────────────────────────────────────┤
     │  ② 리졸버 → [루트 서버]:  "com은 누가 알아?"
     │           ← "com 네임서버 주소는 여기"          ┐
     │  ③ 리졸버 → [com 서버]:   "example.com은 누가?"  │ 리졸버가 단계별로
     │           ← "example.com 네임서버는 여기"        │ 직접 물어봄 = 반복적
     │  ④ 리졸버 → [example.com 서버]: "www 주소는?"    │
     │           ← "93.184.216.34 야"                  ┘
     └──────────────────────────────────────────┐
  ⑤ 리졸버 ──"93.184.216.34 야"──→ 내 PC ◀───────┘
```

- **재귀적 질의(recursive)**: 내 PC → 리졸버. "**답을 통째로 가져다줘**"라고 위임한다.
- **반복적 질의(iterative)**: 리졸버 → 루트/TLD/권한 서버. "**다음에 물어볼 곳만 알려줘**"를 반복한다. §2.5의 라우팅("다음 한 걸음만 안다")과 닮았다.

### 4.3 레코드 타입과 캐싱/TTL

DNS가 저장하는 정보는 **레코드(record)** 단위다. 주요한 것만:

| 레코드 | 의미 | 예시 |
|-------|------|------|
| **A** | 도메인 → IPv4 주소 | `example.com → 93.184.216.34` |
| **AAAA** | 도메인 → IPv6 주소 | `example.com → 2606:2800:...` |
| **CNAME** | 도메인 → 다른 도메인(별칭) | `www.example.com → example.com` |
| **MX** | 메일 서버 | (이메일 라우팅용) |
| **NS** | 이 도메인의 권한 네임서버 | |

**CNAME이 왜 유용한가?** `www.myshop.com`, `shop.myshop.com`이 모두 같은 서버를 가리켜야 할 때, 각각 A 레코드로 IP를 박아두면 서버 IP가 바뀔 때 전부 고쳐야 한다. 대신 CNAME으로 모두 `myshop.com`을 가리키게 하면, **A 레코드 하나만 바꾸면** 전부 따라간다.

**캐싱과 TTL — 왜 첫 접속만 느린가:**

매번 이 재귀 질의를 다 돌면 너무 느리다. 그래서 결과를 **캐시**한다. 각 레코드에는 **TTL(Time To Live)**이 붙어 "이 답을 몇 초간 재사용해도 된다"를 정한다.

```
  첫 접속:  DNS 리졸버 왕복(수 ms~수십 ms) 발생  ──→  느림
  이후:     내 PC/브라우저/리졸버 캐시에서 즉시 응답  ──→  빠름 (TTL 만료 전까지)

  TTL=300  →  5분간 캐시 재사용. 5분 뒤엔 다시 물어봄.
```

**왜 TTL을 짧게/길게 하나?** 길면(예: 하루) 캐시 효율이 좋지만, 서버 IP를 바꿔도 전 세계가 하루 동안 옛 IP를 본다. 짧으면(예: 60초) IP 변경이 빨리 반영되지만 DNS 트래픽이 는다. **서버 이전을 앞두면 미리 TTL을 짧게 낮춰두는 것**이 실무 요령이다.

> **실습으로 확인**: `observe_http.sh`의 `dig example.com` 출력에서 A 레코드와 그 옆의 **TTL 숫자**를 직접 본다. 두 번 연속 돌리면 두 번째 TTL이 줄어든 것(캐시가 카운트다운 중)을 볼 수도 있다.

## Chapter 5: 응용 계층 HTTP — 텍스트로 나누는 대화

> "SPRING §3.1은 HTTP '메서드와 상태 코드'를 설명했다. 여기서는 그 요청/응답이 **바이트 수준에서 실제로 어떻게 생겼는지**, 그리고 연결·상태·캐시가 어떻게 얽히는지를 본다."

### 5.1 요청과 응답의 해부 (실제 바이트)

SPRING §3.1이 "HTTP는 편지 같다"고 했다. 그 편지의 **실제 글자**는 이렇게 생겼다. 놀랍게도 **사람이 읽을 수 있는 평범한 텍스트**다.

```
  ── HTTP 요청(Request) ──────────────────────────────
  GET /users/1 HTTP/1.1              ← 요청 라인: [메서드] [경로] [버전]
  Host: api.example.com              ┐
  Accept: application/json           │ 헤더(header): "이름: 값" 여러 줄
  Authorization: Bearer eyJ...       │
  Connection: keep-alive             ┘
                                     ← 빈 줄(CRLF) = 헤더 끝 신호 ★
  (GET은 보통 바디 없음)               ← 바디(body): POST/PUT이면 여기에 데이터

  ── HTTP 응답(Response) ─────────────────────────────
  HTTP/1.1 200 OK                    ← 상태 라인: [버전] [상태코드] [설명]
  Content-Type: application/json     ┐
  Content-Length: 34                 │ 헤더
  Cache-Control: max-age=60          ┘
                                     ← 빈 줄(CRLF)
  {"id":1,"name":"김철수","age":28}    ← 바디(실제 데이터)
```

**★ 핵심: 빈 줄(`\r\n\r\n`)이 헤더와 바디를 가른다.** 서버는 이 빈 줄을 만나기 전까지를 헤더로, 이후를 바디로 해석한다. 그리고 각 줄의 끝은 `\r\n`(CRLF)이다. 이것을 알아야 §RawHttpClient에서 왜 `"\r\n\r\n"`을 손으로 써주는지 이해된다.

- **요청 라인**의 세 조각(메서드/경로/버전)이 SPRING §3.1의 "무엇을/어디로"에 정확히 대응한다.
- **`Host` 헤더가 왜 필수인가?** 한 IP의 한 서버(한 443 포트)가 `a.com`, `b.com` 수백 개 사이트를 호스팅한다. 서버는 `Host` 헤더를 보고 "이 요청은 어느 사이트 것"인지 안다. 그래서 HTTP/1.1에서 `Host`는 생략 불가다.

> **실습으로 확인**: `RawHttpClient.java`가 바로 이 요청 텍스트를 손으로 조립해 소켓에 쓴다. 그리고 서버가 돌려주는 응답 텍스트(상태 라인 + 헤더 + 바디)를 그대로 출력한다. "HTTP = TCP 위의 텍스트"임을 두 눈으로 확인하는 것이 이 데모의 목적이다.

### 5.2 연결 재사용: keep-alive(HTTP/1.1)와 멀티플렉싱(HTTP/2)

한 웹페이지엔 HTML 1개, CSS 몇 개, 이미지 수십 개가 있다. 이걸 어떻게 실어 나르는가 — 여기서 HTTP 버전이 갈린다.

**HTTP/1.0**: 요청 하나마다 TCP 연결을 새로 맺고 끊었다. §3.3의 3-way handshake를 **파일마다** 반복 — 끔찍하게 느리다.

**HTTP/1.1 — keep-alive(지속 연결):** 한 번 맺은 TCP 연결을 **끊지 않고 재사용**한다. 악수 비용을 크게 아낀다. 하지만 한계가 있다: 한 연결에서 **요청을 하나씩 순서대로** 처리한다. 앞의 느린 응답이 뒤를 막는 **HOL(Head-of-Line) 블로킹**이 생긴다.

```
  HTTP/1.1 (한 연결, 순차):
    연결 ─[요청A]→ ←[응답A]─ [요청B]→ ←[응답B]─ ...   앞이 막히면 뒤가 대기
    (그래서 브라우저는 연결을 6개쯤 동시에 열어 회피했다)

  HTTP/2 (한 연결, 멀티플렉싱):
    연결 ─[A][B][C] 동시 전송→   ←[C][A][B] 뒤섞여 도착─
    하나의 TCP 연결 안에서 여러 '스트림'이 병렬로 오간다. 앞이 뒤를 안 막음.
```

**HTTP/2 — 멀티플렉싱(multiplexing):** 하나의 TCP 연결 안에서 요청/응답을 **작은 조각(프레임)으로 잘게 나눠 뒤섞어** 병렬로 보낸다. 받는 쪽이 조각을 스트림 번호로 재조립한다. 덕분에 연결 하나로 수십 개 요청을 동시에 처리하고, 헤더도 압축한다. **주소창에 https를 치면 요즘 대부분 HTTP/2로 협상**된다(§6 TLS 핸드셰이크 중에 정해진다).

### 5.3 상태 없음(Stateless)과 쿠키/세션

**HTTP는 상태가 없다(stateless).** 서버는 각 요청을 **독립적으로** 처리하고, 방금 누가 왔었는지 **기억하지 않는다.**

**왜 상태가 없게 설계했나?** 그래야 서버를 여러 대로 쉽게 늘릴 수 있다(확장성). 요청마다 자기완결적이면 어느 서버가 처리하든 상관없다. 하지만 문제가 생긴다 — **로그인하면 그다음 요청부터 "나 아까 로그인한 그 사람이야"를 어떻게 아나?**

해결책이 **쿠키(cookie)**다:

```
  ① 로그인 요청 ──────────────────────────────→ 서버
  ② ←── 응답 헤더:  Set-Cookie: session=abc123   (서버: "이 팔찌 차고 다녀")
  ③ 이후 모든 요청 헤더:  Cookie: session=abc123   (브라우저가 자동 첨부)
  ④ 서버는 abc123을 보고 "아, 그 사용자!" 하고 알아본다
```

- **세션(Session) 방식**: 서버가 `abc123 → 사용자 정보`를 **서버 메모리/DB에 저장**하고, 쿠키엔 열쇠(session id)만 담는다.
- **토큰(JWT) 방식**: 상태를 서버에 안 두고 **토큰 자체에 정보를 담아** 서명한다.

이 세션 vs 토큰의 **깊은 비교, 그리고 JWT 서명(HMAC/대칭·비대칭)**은 SPRING §5.3과 §5.7에서 이미 다뤘다. 여기서는 "**stateless한 HTTP 위에 상태를 얹는 장치가 쿠키/세션/토큰**"이라는 **네트워크 층위의 위치**만 확인하면 된다. → 자세히는 SPRING §5.3, §5.7 참조.

### 5.4 캐싱 헤더: 두 번은 안 물어본다

같은 자원(로고 이미지, CSS)을 매번 서버에서 다시 받는 건 낭비다. HTTP는 **캐싱 헤더**로 "이건 얼마간 재사용해도 돼"를 표현한다.

```
  Cache-Control: max-age=3600    → "3600초(1시간)간 다시 묻지 말고 재사용해"
  ETag: "v3-abc"                 → 자원의 지문(버전표). 바뀌면 지문이 바뀜.
  Last-Modified: (날짜)           → 마지막 수정 시각

  재검증(revalidation):
    브라우저 → If-None-Match: "v3-abc"  ("나 v3 갖고 있는데 아직 유효해?")
    서버 →  304 Not Modified            ("안 바뀌었어. 갖고 있는 거 써")  ← 바디 없음 = 절약!
```

**왜 중요한가?** DNS의 TTL(§4.3)과 발상이 같다 — **"바뀌지 않는 것을 매번 다시 가져오지 말자."** `max-age`로 아예 안 묻거나, `ETag`로 "바뀌었는지만" 가볍게 확인(변화 없으면 `304`로 바디를 통째로 아낌)한다. 이것이 웹이 빠른 큰 이유 중 하나다.

## Chapter 6: HTTPS/TLS — 도청과 위조를 막는 계층

> "SPRING §5.7이 'TLS도 대칭/비대칭 원리'라고 짚고 넘어갔다. 그 미뤄둔 이야기를 지금 편다. §5.7을 먼저(또는 나란히) 읽으면 이 장이 훨씬 선명해진다."

### 6.1 HTTPS가 지키는 세 가지

`http://`는 **평문**이다. §5.1에서 봤듯 HTTP는 사람이 읽는 텍스트고, 중간의 누구든(공용 Wi-Fi, 통신사, 해커) **그대로 엿보거나 바꿀 수 있다.** 비밀번호를 http로 보내면 카페 옆자리에서 훔쳐볼 수 있다는 뜻이다.

`https://`는 HTTP를 **TLS**(Transport Layer Security)라는 보안 계층으로 감싼다. TLS는 세 가지를 보장한다:

```
  ① 기밀성(Confidentiality) → 암호화. 중간에서 봐도 알 수 없음.       (도청 방지)
  ② 무결성(Integrity)       → 변조 감지. 한 글자라도 바뀌면 들킴.     (위조 방지)
  ③ 인증(Authentication)    → 상대가 진짜 그 서버 맞는지 확인.        (사칭 방지)
```

**③ 인증이 왜 별도로 필요한가?** 암호화만 하면 "몰래 엿보는 것"은 막지만, 애초에 **가짜 서버(피싱)와 암호화 통신**을 하면 소용없다. 그래서 "**너 진짜 example.com 서버 맞아?**"를 증명하는 절차가 필요하다 — 그게 인증서(§6.3)다.

**HTTP 스택에서 TLS의 위치:** TLS는 TCP(신뢰성) 위, HTTP(응용) 아래에 낀다.

```
  ┌──────────┐
  │  HTTP     │  ← 응용 데이터
  ├──────────┤
  │  TLS      │  ← ★ 여기서 암호화. HTTP 입장에선 "그냥 안전한 TCP"처럼 보임
  ├──────────┤
  │  TCP      │  ← 신뢰성(§3)
  ├──────────┤
  │  IP       │
  └──────────┘
```

### 6.2 TLS 핸드셰이크: 비대칭으로 대칭키를 건네고, 대칭으로 달린다

이것이 §5.7이 "미뤄둔" 바로 그 이야기다. TLS의 천재적인 아이디어는 **두 종류의 암호를 조합**하는 것이다.

- **비대칭키(공개키/개인키)**: 안전하지만 **느리다.** (SPRING §5.7의 RS256과 같은 원리)
- **대칭키(하나의 비밀키)**: 빠르지만 **키를 어떻게 안전하게 공유하지?**가 숙제.

**해법: 느리고 안전한 비대칭으로 "대칭키(세션 키)"를 딱 한 번 안전하게 건네고, 그다음부터는 빠른 대칭키로 실제 데이터를 암호화한다.** §5.7 마지막 문단이 정확히 이 말이었다.

```
  클라이언트(브라우저)                                     서버(example.com)
      │                                                     │
      │  ① ClientHello                                       │
      │  "TLS 하자. 내가 지원하는 암호 목록/버전은 이것들이야" │
      │ ───────────────────────────────────────────────────→│
      │                                                     │
      │  ② ServerHello + 인증서(Certificate)                 │
      │  "이 암호로 하자. 그리고 이게 내 인증서(공개키 포함)야"│
      │ ←───────────────────────────────────────────────────│
      │                                                     │
      │  ③ [인증서 검증]  ★ 이 서버가 진짜 example.com인가?    │
      │     - CA 서명이 유효한가? 도메인이 맞나? 만료 안 됐나?  │
      │                                                     │
      │  ④ 키 교환                                            │
      │  "이 공개키(또는 키 교환값)로 '세션 키'를 안전하게 합의"│
      │ ←──────────────────────────────────────────────────→│
      │                                                     │
      │  ⑤ 이제 양쪽 다 같은 '세션 키(대칭키)'를 갖게 됨        │
      │ ═══════════ 이후 모든 HTTP는 세션 키로 암호화 ════════ │
      │        빠른 대칭 암호로 실제 데이터를 주고받음          │
```

핵심은 ④~⑤다. 비대칭 방식으로 **오직 이 세션에만 쓸 대칭키(세션 키)**를 양쪽이 안전하게 갖는다. 도청자가 오가는 걸 다 봐도, 개인키가 없으면 세션 키를 알아낼 수 없다. 세션 키가 정해진 뒤엔 비대칭은 버리고 **빠른 대칭키로만** 통신한다 — 안전함과 빠름을 모두 취한 것이다.

**왜 첫 HTTPS 요청은 더 느린가?** 이제 명확하다. TCP 3-way handshake(§3.3) **위에** TLS 핸드셰이크의 왕복이 더해진다. 그래서 §5.2의 keep-alive와 §5.4의 캐시가 더더욱 값지다 — 비싼 핸드셰이크를 한 번만 치르고 재사용하니까.

### 6.3 인증서와 CA: "진짜 그 서버인가"를 어떻게 믿나

②에서 서버가 준 "공개키"를 어떻게 믿는가? 공격자도 "나 example.com이야, 이게 내 공개키야" 하고 자기 공개키를 내밀 수 있다(중간자 공격). 이 신뢰 문제를 푸는 게 **인증서(Certificate)**와 **CA(Certificate Authority, 인증 기관)**다.

```
  인증서 = 공개키 + 도메인 이름(example.com) + 유효기간을,
          믿을 수 있는 제3자(CA)가 '자기 개인키로 서명'해 준 문서

  신뢰 체인(chain of trust):
    ┌─────────────────┐
    │ Root CA          │  ← 브라우저/OS에 '이미 신뢰됨'으로 내장된 최상위 기관
    │   서명 ↓          │
    │ 중간 CA           │
    │   서명 ↓          │
    │ example.com 인증서 │  ← 서버가 제시하는 잎(leaf) 인증서
    └─────────────────┘

  브라우저는 이 체인을 위로 따라가, 뿌리(Root CA)가 '내가 이미 신뢰하는 기관'이면 통과.
```

**논리 구조가 §5.7의 비대칭 서명과 완전히 똑같다.** CA는 **개인키로 인증서에 서명**하고, 브라우저는 이미 갖고 있는 **CA의 공개키로 서명을 검증**한다. 서명이 유효하면 "믿을 수 있는 CA가 이 도메인의 공개키임을 보증했다"는 뜻이다. §5.7에서 "공개키로 검증만 가능하고 위조는 불가능"이라 한 그 성질이, 여기서 **가짜 서버를 걸러내는** 데 쓰인다.

**그래서 실무에서:**
- **인증서 만료** → 브라우저가 빨간 경고. (그래서 자동 갱신을 건다)
- **도메인 불일치**(`a.com` 인증서를 `b.com`이 제시) → 경고.
- **사설/자체 서명 인증서** → Root CA 체인이 없어 경고. 개발용으로 `-k`(curl)나 예외 등록으로 우회하지만, 운영에선 절대 금물.

> **실습으로 확인**: `observe_http.sh`의 `openssl s_client -connect example.com:443`가 **실제 인증서 체인, TLS 버전, 암호 스위트**를 그대로 보여준다. `curl -v`의 `* TLSv1.3 ... handshake` 줄들이 위 ①~⑤ 과정의 실제 로그다.

## Chapter 7: 전부 합치기 — API 하나를 호출하면 무슨 일이 벌어지는가

> "이것이 이 책의 지불(payoff)이다. 앞의 모든 장이 이 한 장을 위한 준비였다. 주소창에 `https://api.example.com/users/1`을 치고 Enter를 누른 순간부터, Spring 컨트롤러가 호출되기까지 — 계층별로 추적한다."

```
  당신:  브라우저(또는 앱)에서  https://api.example.com/users/1  요청

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 1. URL 파싱                                            [응용 준비]     │
  │   https  → 프로토콜(TLS+HTTP, 기본 포트 443)                                │
  │   api.example.com → 호스트(이름, 아직 IP 모름)                             │
  │   /users/1 → 경로                                                          │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 2. DNS 조회                                       [응용/DNS · Ch4]     │
  │   "api.example.com의 IP?"  → (캐시에 없으면) 리졸버가 재귀 질의            │
  │   루트 → com → example.com 네임서버 → A 레코드: 93.184.216.34             │
  │   ※ 캐시에 있으면 이 단계 건너뜀 → "두 번째 접속이 빠른" 이유              │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │  이제 목적지 IP를 안다
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 3. TCP 3-way handshake                          [전송 · Ch3]         │
  │   SYN → SYN-ACK → ACK   (93.184.216.34 의 443 포트로)                     │
  │   → 신뢰성 있는 연결(소켓) 수립. 왕복 1회(RTT) 비용.                       │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 4. TLS handshake (https라서)                    [보안 · Ch6]         │
  │   ClientHello → ServerHello+인증서 → 인증서 검증(CA 체인) → 세션키 합의    │
  │   → 이후 통신은 대칭 세션키로 암호화. 여기서 HTTP/2 여부도 협상.           │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │  이제 '안전한 관' 완성
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 5. HTTP 요청 전송                                [응용 · Ch5]         │
  │   GET /users/1 HTTP/1.1                                                    │
  │   Host: api.example.com                                                    │
  │   Accept: application/json                                                 │
  │   (빈 줄)                                                                  │
  │   → 이 텍스트가 TLS로 암호화되어 TCP 세그먼트에 담겨 IP 패킷으로 전송       │
  │   ※ 캡슐화(§1.3): HTTP→(TLS)→TCP→IP→이더넷 순으로 봉투에 감싸짐           │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │  IP 라우팅(§2.5)으로 여러 hop을 건너 서버 도착
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 6. 서버 도착 → 역캡슐화 → Spring                 [서버 측]           │
  │   이더넷→IP→TCP 헤더를 차례로 벗김 → TLS 복호화 → HTTP 메시지 복원         │
  │   → 톰캣이 요청을 스레드에 배정(SPRING §2.7) → DispatcherServlet          │
  │   → @GetMapping("/users/{id}") 컨트롤러 호출  ★ 여기가 SPRING §3.3의 입구  │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │  Service→Repository→DB (SPRING §3.4)
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 7. HTTP 응답 → 당신에게                          [역방향 전 계층]     │
  │   200 OK + {"id":1,...}  →  같은 TLS 세션키로 암호화                       │
  │   →  TCP가 순서/신뢰 보장하며 전송  →  브라우저가 복호화·파싱·렌더링        │
  │   →  Connection: keep-alive면 연결 유지(§5.2), 다음 요청은 3·4 단계 생략!  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**각 단계를 계층에 매핑하면:**

| STEP | 하는 일 | 계층 | 참조 장 |
|------|--------|------|--------|
| 2 | 이름 → IP | 응용(DNS) | Ch4 |
| 3 | 연결 수립 | 전송(TCP) | Ch3 |
| 4 | 암호화 협상 | 보안(TLS) | Ch6 |
| 5 | 요청 전송 | 응용(HTTP) + 전 계층 캡슐화 | Ch5, §1.3 |
| 5~7 | 라우팅/전달 | 네트워크(IP)·링크 | Ch2 |
| 6 | 컨트롤러 진입 | (서버 응용) | **SPRING §3.3** |

**이제 앞머리의 질문들에 답할 수 있다:**
- "첫 요청만 느린 이유?" → STEP 2(DNS)·3(TCP)·4(TLS)가 첫 요청에만 붙고, 이후엔 캐시·keep-alive로 대부분 생략된다.
- "타임아웃인데 서버 로그가 없는 이유?" → STEP 3에서 연결조차 안 됐다면(방화벽·라우팅) 요청이 STEP 6의 Spring에 **도달조차 못 한** 것이다.
- "HTTPS 인증서 오류?" → STEP 4의 인증서 검증(§6.3) 실패다.

**이 여정을 한 문장으로:**
> 이름을 주소로 바꾸고(DNS), 신뢰할 수 있는 관을 뚫고(TCP), 그 관을 암호로 감싸고(TLS), 그 위로 텍스트 한 통(HTTP)을 보내면, 여러 층의 봉투에 감싸여(캡슐화) 여러 길목을 건너(라우팅) 서버에 닿고, 서버가 봉투를 차례로 벗겨(역캡슐화) 마침내 `@GetMapping` 메서드를 부른다.

이 문장을 diagram 없이도 설명할 수 있다면, 당신은 이제 **HTTP 위가 아니라 HTTP 아래를 아는 사람**이다. 전공자의 출발선에 선 것이다.

이 책은 "**API를 호출하면 무슨 일이 벌어지는가**"를 계층으로 풀었다. 여기서 더 깊이 가려면:

- **직접 관찰하라**: `chapter-cs-network/labs/observe_http.sh`를 돌려 DNS·TLS·헤더를 눈으로 보라. `RawHttpClient.java`로 "HTTP는 텍스트"임을 증명하라. **읽은 것을 본 순간 개념이 몸에 붙는다.**
- **더 깊은 도구**: `Wireshark`(패킷을 눈으로), `traceroute`(hop 추적), `tcpdump`(패킷 캡처)로 실제 봉투(§1.3)를 열어보라.
- **되짚어 읽기**: SPRING §3(웹 구조)을 다시 읽으면, 이제 그 "요청"이 어디서 왔는지 보인다. SPRING §5.3·§5.7(세션/JWT/비대칭)은 이 책 §5.3·§6과 짝을 이룬다.

> "왜?를 계속 물어라. `@GetMapping`이 호출되기까지 7단계를 설명할 수 있게 되면, 다음 질문은 '그 7단계 각각은 또 왜 그렇게 생겼나'다. 그 질문이 당신을 전공자 너머로 데려간다."
