# 1장. 왜 네트워크를 "층(Layer)"으로 나누는가

> "복잡한 문제를 다루는 인류의 유일한 방법은, 그것을 작은 문제로 쪼개는 것이다."

## 도입 — 왜 이 장이 먼저인가

이 책의 나머지 여섯 장은 모두 "어느 한 층"에 관한 이야기다. 3장은 전송 계층, 6장은 보안 계층 하는 식이다. 그런데 "층"이라는 개념 자체를 먼저 손에 쥐지 않으면, 각 장이 서로 어떻게 맞물리는지 보이지 않는다. 그래서 이 장은 **네트워크 전체를 조망하는 지도**다.

두 컴퓨터가 대화하려면 해결할 문제가 산더미다.

- 전선/전파로 0과 1을 어떻게 실어 보내지?
- 같은 케이블에 여러 기기가 붙어 있는데 누구에게 보낼지 어떻게 정하지?
- 지구 반대편 컴퓨터까지 어떻게 길을 찾아가지?
- 중간에 데이터가 유실되면 어떻게 다시 보내지?
- 도착한 바이트를 "HTTP 요청"으로 어떻게 해석하지?

이 모든 걸 **하나의 거대한 프로그램**으로 짜면 어떻게 될까? 아무도 이해하지 못하고, 하나를 고치면 전부 무너진다. 그래서 네트워크는 **계층(layer)**으로 나눈다. 이 장을 다 읽으면 "그건 4계층 문제야", "L7 로드밸런서" 같은 실무 표현이 무슨 뜻인지, 그리고 왜 그렇게 나누는 것이 천재적인 설계인지 알게 된다.

## 1.1 계층화(Layering)라는 아이디어

### 개념 — 각 층은 이웃하고만 대화한다

계층화의 **핵심 원리**는 한 문장이다.

> 각 층은 "바로 아래 층의 서비스만 쓰고, 바로 위 층에게만 서비스를 준다."

```
  위 층은 아래 층이 "어떻게" 하는지 몰라도 된다.
  아래 층은 위 층이 "무엇을" 보내는지 몰라도 된다.

  ┌─────────────┐
  │  HTTP        │  "GET /users/1 보내줘"  ← 어떻게 배달되는지 모름/관심없음
  ├─────────────┤
  │  TCP         │  "이 바이트들을 순서대로, 유실 없이 배달"  ← 내용이 뭔지 모름
  ├─────────────┤
  │  IP          │  "이 덩어리를 저 IP 주소로"  ← 신뢰성은 내 일 아님
  ├─────────────┤
  │  링크/물리   │  "옆 장비로 전기신호 전송"  ← 목적지가 어디인지 모름
  └─────────────┘
```

### 원리 — 왜 이렇게 나누면 좋은가

세 가지 이득이 있고, 이 셋은 소프트웨어 공학 전반의 원리이기도 하다.

- **교체 가능성(modularity)**: Wi-Fi든 이더넷이든 4G든, 아래 층만 바뀌고 위의 TCP/HTTP는 그대로다. 카페에서 노트북으로 짜던 코드가 집 이더넷에서도 똑같이 돈다. 반대로 HTTP/1.1을 HTTP/2로 바꿔도 그 아래 TCP는 손댈 필요가 없다.
- **책임 분리(separation of concerns)**: HTTP를 짜는 사람은 "패킷 유실 재전송"을 신경 쓰지 않아도 된다. 그건 TCP의 일이다. 이것은 SPRING §3.4 "레이어드 아키텍처"에서 Controller/Service/Repository로 책임을 나눈 것과 **똑같은 철학**이다. Controller가 SQL을 몰라도 되듯, HTTP는 IP 라우팅을 모른다.
- **표준화(interoperability)**: 층의 경계(인터페이스)만 표준으로 정하면, 그 안은 누가 어떻게 구현하든 상관없다. 그래서 마이크로소프트가 만든 브라우저가 리눅스로 짜인 서버와 아무 문제 없이 대화한다. 양쪽이 각 층의 규약(프로토콜)만 지키면 된다.

### 예시 — "층을 몰라서 생기는 오해"

초보 개발자가 흔히 하는 말: "서버가 안 되는데요?" 이 문장은 사실 층을 뭉뚱그린 것이다. 실제로는 이렇게 쪼개진다.

- 케이블/Wi-Fi가 끊겼나? → **물리/링크(L1/L2)**
- IP가 서버까지 못 닿나(라우팅·방화벽)? → **네트워크(L3)**
- 포트가 안 열렸나(연결 거부)? → **전송(L4)**
- 서버는 떴는데 500을 뱉나? → **응용(L7)**

층을 아는 사람은 "안 돼요"를 "3계층에서 안 닿는다"로 번역할 수 있다. 그 번역이 곧 문제 해결의 절반이다.

## 1.2 OSI 7계층과 TCP/IP 4계층

### 개념 — 두 개의 참조 모델

역사적으로 두 개의 참조 모델이 있다. **OSI 7계층**은 교과서적 개념 모델이고, **TCP/IP 4계층**은 실제 인터넷이 돌아가는 실무 모델이다. 둘을 나란히 놓고 봐야 한다.

```
   OSI 7계층 (개념)              TCP/IP 4계층 (실제)         이 층에 사는 것들
 ┌──────────────────┐        ┌──────────────────┐
 │ 7. 응용 (App)     │        │                   │
 │ 6. 표현(Present.) │  ───→  │  응용 (App)        │   HTTP, DNS, TLS, FTP, SMTP
 │ 5. 세션(Session)  │        │                   │
 ├──────────────────┤        ├──────────────────┤
 │ 4. 전송(Transport)│  ───→  │  전송 (Transport)  │   TCP, UDP     (포트, 신뢰성)
 ├──────────────────┤        ├──────────────────┤
 │ 3. 네트워크(Net.) │  ───→  │  인터넷 (Internet) │   IP, ICMP     (IP주소, 라우팅)
 ├──────────────────┤        ├──────────────────┤
 │ 2. 데이터링크    │  ───→  │                   │
 │ 1. 물리(Physical) │        │  링크 (Link)       │   이더넷, Wi-Fi, MAC 주소
 └──────────────────┘        └──────────────────┘
```

### 각 OSI 계층이 하는 일 (7 → 1)

전공 교재는 7계층 각각에 이름을 붙인다. 실무에서 자주 안 쓰는 5·6계층도 개념만 잡아두면 좋다.

| 계층 | 이름 | 한 문장 역할 | 대표 예 |
|------|------|-------------|---------|
| 7 | 응용(Application) | 사용자가 쓰는 프로토콜 | HTTP, DNS, SMTP |
| 6 | 표현(Presentation) | 인코딩·암호화·압축 | TLS, 문자 인코딩, gzip |
| 5 | 세션(Session) | 대화의 시작·유지·종료 관리 | 세션 수립/재개 |
| 4 | 전송(Transport) | 종단 간(end-to-end) 신뢰성·포트 | TCP, UDP |
| 3 | 네트워크(Network) | 여러 네트워크를 건너는 라우팅 | IP, ICMP |
| 2 | 데이터링크(Data Link) | 같은 네트워크 안 이웃 전달·오류검출 | 이더넷, MAC |
| 1 | 물리(Physical) | 비트를 전기/광/전파 신호로 | 케이블, 전압, 주파수 |

TCP/IP 4계층은 이 중 5·6·7을 "응용"으로, 1·2를 "링크"로 뭉친다. TLS는 OSI로는 대략 6계층(표현)에 해당하지만 실무 TCP/IP 관점에선 "응용과 전송 사이"라고 부른다(6장에서 자세히).

### 원리 — 왜 두 개가 있는가

OSI는 국제표준화기구가 "이상적으로 이렇게 나누면 좋겠다"고 설계한 모델이다. 반면 인터넷은 OSI가 완성되기 전에 이미 TCP/IP로 돌아가기 시작했고, 실전에서 이겼다. 그래서 **개념은 OSI 7계층 용어로 말하고(예: "그건 4계층 문제야"), 실제 구현은 TCP/IP로** 이해하는 게 현실이다.

### 예시 — 실무에서 자주 쓰는 표현

- "**L4** 로드밸런서" = 전송 계층(TCP/포트)에서 분배. IP·포트만 보고 뒤 서버로 넘긴다. 빠르지만 HTTP 경로는 모른다.
- "**L7** 로드밸런서" = 응용 계층(HTTP 경로/헤더)에서 분배. `/api`는 A서버, `/img`는 B서버로 보낼 수 있다. 똑똑하지만 무겁다.
- "**3계층에서 안 닿아**" = IP 라우팅 문제(방화벽, 라우팅 테이블).
- "**포트가 안 열렸어**" = 4계층 문제(서비스 미기동, 방화벽의 포트 차단).

이 책은 아래에서 위로 올라간다: 물리/링크/네트워크(2장) → 전송(3장) → DNS(4장) → HTTP(5장) → TLS(6장). 마지막에 전부 합쳐 하나의 요청을 추적한다(7장).

## 1.3 캡슐화(Encapsulation) — 봉투 속의 봉투

### 개념 — 층은 헤더를 붙이며 협력한다

계층이 실제로 어떻게 협력하는가? 답은 **캡슐화**다. 각 층은 위 층에서 받은 데이터에 **자기 헤더(header)를 붙여서** 아래로 내린다. 마치 편지를 봉투에 넣고, 그 봉투를 더 큰 소포 상자에 넣고, 그 상자에 운송장을 붙이는 것과 같다.

### 원리 — 내려갈 때 감싸고, 올라갈 때 벗긴다

```
  [보내는 쪽 — 위에서 아래로 내려가며 헤더를 붙인다]

  응용:   ┌────────────────────────────────┐
          │  GET /users/1 HTTP/1.1 ...      │   ← 실제 데이터 (HTTP 메시지)
          └────────────────────────────────┘
                        │ TCP가 감쌈
  전송:   ┌──────┬────────────────────────────────┐
          │TCP헤더│  GET /users/1 HTTP/1.1 ...      │   ← 세그먼트(Segment)
          └──────┴────────────────────────────────┘   TCP헤더: 출발/도착 포트, 순서번호
                        │ IP가 감쌈
  네트워크:┌──────┬──────┬────────────────────────────────┐
          │IP헤더 │TCP헤더│  GET /users/1 HTTP/1.1 ...      │   ← 패킷(Packet)
          └──────┴──────┴────────────────────────────────┘   IP헤더: 출발/도착 IP 주소
                        │ 이더넷이 감쌈
  링크:   ┌────┬──────┬──────┬────────────────────────┬────┐
          │이더넷│IP헤더 │TCP헤더│  GET /users/1 ...      │FCS │   ← 프레임(Frame)
          └────┴──────┴──────┴────────────────────────┴────┘   이더넷헤더: 출발/도착 MAC
                        │
                        ▼  전선/전파로 0과 1이 흐른다
```

받는 쪽에서는 **정확히 반대(역캡슐화, decapsulation)**로 진행한다. 링크 층이 이더넷 헤더를 떼고 IP에게 주고, IP가 IP 헤더를 떼고 TCP에게 주고, TCP가 TCP 헤더를 떼고 HTTP에게 원래 메시지를 준다. **각 층은 자기 헤더만 읽고 떼어낸다.** 아래 층 헤더는 어떻게 붙었는지 신경 쓰지 않는다.

### 헤더에 담기는 대표 필드 (미리보기)

각 헤더가 왜 필요한지는 뒤 장에서 다루지만, 캡슐화 그림을 이해하려면 "이 헤더엔 뭐가 들었나"를 대략 알아야 한다.

```
  이더넷 헤더:  출발 MAC | 도착 MAC | 타입          (→ 2장 §2.1)
  IP 헤더:      출발 IP  | 도착 IP  | TTL | 프로토콜  (→ 2장 §2.3)
  TCP 헤더:     출발 포트| 도착 포트| seq | ack | 플래그 | 윈도우  (→ 3장)
  HTTP:         (텍스트 그 자체 — 헤더가 아니라 페이로드)  (→ 5장)
```

여기서 IP 헤더의 "프로토콜" 필드가 "내 안에 든 게 TCP냐 UDP냐"를 알려주고, TCP 헤더의 "도착 포트"가 "이 데이터를 어느 프로그램에 줄까"를 알려준다. 즉 각 헤더는 **바로 위 층으로 올려보낼 때 필요한 정보**를 담는다. 이것이 역캡슐화가 가능한 이유다.

### 용어 정리 — 층마다 데이터 단위의 이름이 다르다(PDU)

| 계층 | 데이터 단위 이름 | 붙는 주요 정보 |
|------|----------------|--------------|
| 응용 | 메시지(Message) | HTTP 헤더/바디 |
| 전송(TCP) | 세그먼트(Segment) | 포트, 순서번호(seq), 확인응답(ack) |
| 전송(UDP) | 데이터그램(Datagram) | 포트 |
| 네트워크(IP) | 패킷(Packet) | 출발/도착 IP |
| 링크 | 프레임(Frame) | 출발/도착 MAC, 오류검출(FCS) |

PDU(Protocol Data Unit)는 "그 층에서 다루는 데이터 한 덩어리"의 정식 이름이다. 시험이나 면접에서 "TCP의 PDU는?" 하고 물으면 답은 "세그먼트"다.

### 예시 — 오버헤드는 공짜가 아니다

캡슐화는 편하지만 대가가 있다. 실제 HTTP 데이터 1바이트를 보내려 해도 그 앞에 이더넷(14B) + IP(20B) + TCP(20B) 헤더가 붙는다. 그래서 아주 짧은 데이터를 자주 보내면 헤더 오버헤드가 커진다. 이것이 TCP가 작은 조각을 모아 보내는 최적화(Nagle 알고리즘)를 갖는 이유이기도 하다. "층으로 나눈 대가"를 실감하는 지점이다.

### 실습 연결

> **실습으로 확인**: `RawHttpClient.java`는 이 스택의 **가장 위 두 칸(HTTP over TCP)**을 직접 만든다. `new Socket(host, 80)`을 열면 TCP가 준비되고, 거기에 `"GET / HTTP/1.1..."` 텍스트를 쓰면 그게 HTTP다. 나머지 아래 층(IP, 링크)은 운영체제가 알아서 감싼다. `./run.sh RawHttpClient`로 실행하면 위 캡슐화 그림의 맨 위 두 층을 손으로 만든 셈이다.
>
> 실제로 봉투가 열리는 것을 보려면 `Wireshark`나 `tcpdump`로 패킷을 캡처하라. 하나의 패킷을 클릭하면 "Ethernet → IP → TCP → HTTP" 트리가 그대로 보인다 — 위 그림이 화면에 재현된다.

## ⚠️ 흔한 오해와 함정

- **"OSI 7계층은 실제 인터넷의 구조다."** 아니다. 실제 인터넷은 TCP/IP 4계층으로 돈다. OSI는 개념·용어를 위한 참조 모델이다. 둘을 1:1로 억지 매핑하면 TLS 같은 것이 어디 속하는지 오히려 헷갈린다.
- **"헤더는 데이터 뒤에 붙는다."** 대부분의 헤더는 **앞(prepend)**에 붙는다. 다만 이더넷의 FCS(오류검출)처럼 **꼬리(trailer)**에 붙는 것도 있다.
- **"한 층이 다른 층 내부를 읽는다."** 원칙적으로 각 층은 자기 헤더만 본다. (단 L4 로드밸런서가 IP·포트만 보고, L7이 HTTP까지 열어보는 것처럼 실무 장비는 여러 층을 걸쳐 보기도 한다 — 이건 최적화이지 계층 모델의 위반이 아니다.)
- **"캡슐화는 그냥 이론이다."** 아니다. `curl -v`, `openssl`, `Wireshark`에서 실제로 이 봉투들이 열린다. 이론이 아니라 화면에 보이는 실체다.

## 연습문제

1. **계층화의 세 가지 이득(교체 가능성·책임 분리·표준화)을 각각 한 문장 예로 설명하라.**
   - *힌트*: "Wi-Fi를 이더넷으로 바꿔도…", "HTTP 개발자는 재전송을 몰라도…", "MS 브라우저가 리눅스 서버와…"
   - *해설*: 교체 가능성 — 아래 층(Wi-Fi→이더넷)을 바꿔도 위 TCP/HTTP는 그대로다. 책임 분리 — 재전송은 TCP의 일이라 HTTP는 신경 안 쓴다. 표준화 — 층 경계 규약만 지키면 구현 주체가 달라도 통신된다.

2. **OSI 7계층 중 TCP/IP 4계층에서 "응용"으로 뭉쳐지는 계층 번호를 모두 대라.**
   - *힌트*: 위쪽 세 개.
   - *해설*: 5(세션)·6(표현)·7(응용). TLS는 대략 6계층에 해당한다.

3. **HTTP 메시지가 링크 계층까지 내려갈 때 데이터 단위 이름이 어떻게 바뀌는가? 순서대로 나열하라.**
   - *힌트*: 메시지 → ? → ? → ?
   - *해설*: 메시지(HTTP) → 세그먼트(TCP) → 패킷(IP) → 프레임(링크).

4. **"3계층에서 안 닿아요"와 "포트가 안 열렸어요"는 각각 어느 계층 문제이며, 무엇을 의심해야 하는가?**
   - *힌트*: L3 vs L4.
   - *해설*: 전자는 네트워크 계층(IP 라우팅·방화벽·라우팅 테이블), 후자는 전송 계층(서비스 미기동·포트 차단).

5. **받는 쪽 IP 계층이 "이 패킷 안에 든 게 TCP인지 UDP인지"를 어떻게 아는가?**
   - *힌트*: IP 헤더의 어떤 필드.
   - *해설*: IP 헤더의 "프로토콜(Protocol)" 필드를 보고 위 층이 TCP인지 UDP인지 판단해 올려보낸다. 각 헤더가 "바로 위 층으로 올릴 정보"를 담기에 역캡슐화가 가능하다.

---

## 요약

- 네트워크는 복잡한 문제를 **층(layer)**으로 쪼갠다. 각 층은 **바로 아래 층의 서비스만 쓰고, 바로 위 층에게만 서비스를 준다.**
- 계층화의 이득은 **교체 가능성·책임 분리·표준화**다. 이는 SPRING §3.4 레이어드 아키텍처와 같은 철학이다.
- **OSI 7계층**은 개념 모델, **TCP/IP 4계층**은 실무 모델이다. "L4/L7", "3계층에서 안 닿아" 같은 표현은 이 모델에서 나온다.
- **캡슐화**는 각 층이 자기 헤더를 붙여 아래로 내리는 것이다. 받는 쪽은 역캡슐화로 헤더를 하나씩 벗긴다. 각 헤더는 "바로 위 층으로 올릴 정보"를 담는다.
- 층마다 데이터 단위(PDU) 이름이 다르다: 메시지 → 세그먼트 → 패킷 → 프레임.
