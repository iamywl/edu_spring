# 3장. 전송 계층 — 신뢰성은 여기서 만들어진다

> "IP는 '최선을 다하지만 보장은 안 하는(best-effort)' 배달부다. 패킷을 잃어버릴 수도, 순서를 뒤바꿀 수도, 중복 배달할 수도 있다. 이 엉망진창을 '순서대로, 빠짐없이'로 바꿔주는 것이 TCP다."

## 도입 — 왜 신뢰성이 별도의 층인가

2장에서 IP는 "저 컴퓨터까지 데이터 덩어리 하나를 배달"한다고 했다. 하지만 IP는 **best-effort(최선을 다하되 보장은 안 함)**다. 패킷이 중간에 유실되거나, 순서가 뒤바뀌거나, 중복 도착해도 IP는 책임지지 않는다. 그런데 웹페이지·계좌이체는 한 바이트도 틀리면 안 된다. 이 간극을 메우는 것이 **전송 계층(L4)**이고, 그 주역이 **TCP**다.

이 장은 이 책에서 가장 밀도가 높다. **Spring이 받는 모든 요청의 밑바닥엔 TCP가 있기** 때문이다. "첫 요청은 왜 느린가", "타임아웃인데 서버 로그가 없다", "서버 재시작 시 Address already in use" — 이 실무 질문들의 답이 모두 이 장에 있다.

## 3.1 포트와 소켓 — 한 컴퓨터 안의 여러 문

### 개념

IP 주소는 **컴퓨터**를 특정한다. 하지만 한 컴퓨터에는 웹서버, 데이터베이스, SSH가 동시에 돌아간다. 도착한 데이터를 **누구에게** 줘야 하나? 그것을 정하는 것이 **포트(port)**다.

```
  IP 주소 = 아파트 동 (건물 하나)
  포트    = 호수 (건물 안의 각 집)

  211.234.1.5 : 443    →  443호에 사는 웹서버(HTTPS)
  211.234.1.5 : 3306   →  3306호에 사는 MySQL
  211.234.1.5 : 22     →  22호에 사는 SSH
```

포트는 16비트라 0~65535까지 있다. 크게 세 구간이다: **잘 알려진 포트(0~1023)**, **등록 포트(1024~49151)**, **동적/임시 포트(49152~65535, 클라이언트가 랜덤 배정받는 출발지 포트)**.

**잘 알려진 포트(well-known):** 80(HTTP), 443(HTTPS), 22(SSH), 25(SMTP), 53(DNS), 3306(MySQL), 5432(PostgreSQL), 8080(개발용 HTTP — Spring Boot 기본).

### 원리 — 소켓은 4-튜플로 유일하게 식별된다

**소켓(Socket)**은 "**IP + 포트**"의 조합이자, 프로그램이 네트워크와 대화하는 **창구(추상화)**다. 하나의 연결은 **4개 값(4-튜플)**으로 유일하게 식별된다.

```
  연결 = (출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)

  예) 내 브라우저가 example.com에 접속:
      (192.168.0.10, 51234) ────────→ (93.184.216.34, 443)
       └ 내 PC, 랜덤 포트              └ 서버, 443(HTTPS)

  탭을 하나 더 열면 출발지 포트만 51235로 바뀌어 '다른 연결'이 된다.
```

이 4-튜플이 하나라도 다르면 서로 다른 연결이다. 그래서 서버의 443 포트 하나로 수만 명이 동시에 접속해도, 각 접속은 "출발지 IP+포트"가 달라 구분된다. 서버가 "동시 접속 처리"를 하는 원리가 여기 있다.

### 예시 — listen 소켓 vs connection 소켓

서버는 처음에 특정 포트(예: 8080)에서 **연결을 기다리는(listen) 소켓** 하나를 연다. 클라이언트가 접속하면, 서버는 그 요청마다 **새 연결 소켓**을 만들어 실제 대화를 한다. listen 소켓은 계속 "다음 손님"을 받고, 연결 소켓은 각 손님과 1:1로 대화한다. 톰캣이 요청마다 스레드를 붙이는 것(SPRING §2.7)이 이 연결 소켓 위에서 일어난다.

### 실습 연결

> **실습으로 확인**: `RawHttpClient.java`의 `new Socket(host, 80)`이 바로 이 소켓을 여는 코드다. 자바가 목적지 IP:포트로 향하는 창구를 만들고, 운영체제가 랜덤한 출발지 포트를 배정한다. `./run.sh RawHttpClient`로 실행한 뒤, 실행 중에 별도 터미널에서 `netstat -an | grep 80`을 보면 위 4-튜플이 실제로 잡히는 것을 볼 수 있다.

## 3.2 TCP vs UDP — 신뢰성이냐 속도냐

### 개념 — 선택은 트레이드오프

전송 계층에는 두 프로토콜이 있다.

```
  ┌──────────────┬────────────────────────┬────────────────────────┐
  │              │  TCP                     │  UDP                    │
  ├──────────────┼────────────────────────┼────────────────────────┤
  │ 연결          │ 연결형(handshake 필요)   │ 비연결형(그냥 던짐)      │
  │ 신뢰성        │ 보장(유실 시 재전송)     │ 보장 안 함(잃으면 끝)    │
  │ 순서          │ 보장(순서대로 재조립)    │ 보장 안 함              │
  │ 흐름/혼잡제어 │ 있음                     │ 없음                    │
  │ 속도/오버헤드 │ 느림/무거움             │ 빠름/가벼움             │
  │ 헤더 크기     │ 20바이트~               │ 8바이트                 │
  │ 비유          │ 등기우편(수령 확인)      │ 광고 전단지(그냥 뿌림)   │
  │ 쓰는 곳       │ HTTP, 파일전송, DB       │ DNS질의, 영상통화, 게임  │
  └──────────────┴────────────────────────┴────────────────────────┘
```

### 원리 — 왜 둘 다 필요한가

- 웹페이지, 계좌이체는 **한 글자도 틀리면 안 된다** → TCP. 느려도 정확해야 한다.
- 실시간 영상통화는 **0.1초 전 프레임을 재전송받아 봐야 이미 늦었다** → UDP. 조금 깨져도 지금 프레임이 중요하다.

"정확성"과 "실시간성"은 자주 상충한다. TCP는 정확성을, UDP는 실시간성을 택한 것이다. UDP는 신뢰성이 필요하면 **응용 계층이 직접** 구현해야 한다(이것이 HTTP/3의 QUIC가 하는 일 — §5.2에서 다룸).

### 예시

HTTP/1.1과 HTTP/2는 TCP 위에서 돈다. 그래서 **Spring이 받는 모든 요청의 밑바닥엔 TCP가 있다.** DNS 질의(4장)는 짧고 빠르면 되니 주로 UDP를 쓴다. 온라인 게임은 "지금 위치"가 중요하니 UDP에 자체 보정을 얹는다.

## 3.3 TCP 3-way handshake — 대화를 시작하는 악수

### 개념 — 왜 3번인가

TCP는 **연결형**이다. 데이터를 보내기 전에 양쪽이 "**대화 시작합시다**"를 합의하는데, 이것이 **3-way handshake**다. 왜 3번인가? 양쪽 모두 "**나도 보낼 수 있고, 상대도 받을 수 있다**"를 확인해야 하기 때문이다. 즉 양방향 통신 능력을 서로 검증한다.

### 원리 — SYN / SYN-ACK / ACK

```
  클라이언트                                            서버
      │                                                 │
      │  ①  SYN  (seq=100)                              │
      │  "대화 시작하자. 내 시작 순서번호는 100이야"      │
      │ ──────────────────────────────────────────────→ │
      │                                                 │
      │  ②  SYN + ACK  (seq=300, ack=101)               │
      │  "좋아. 네 100 잘 받았어(다음엔 101 줘).          │
      │   내 시작 순서번호는 300이야"                     │
      │ ←────────────────────────────────────────────── │
      │                                                 │
      │  ③  ACK  (ack=301)                              │
      │  "네 300도 잘 받았어(다음엔 301 줘). 연결 완료!"  │
      │ ──────────────────────────────────────────────→ │
      │                                                 │
      │ ══════════ 연결 수립(ESTABLISHED) ══════════════ │
      │        이제부터 실제 데이터(HTTP 등)를 주고받음    │
```

- **SYN**(Synchronize): "연결하자 + 내 시작 순서번호(ISN) 알려줌"
- **ACK**(Acknowledge): "네 걸 잘 받았음 + 다음에 기대하는 번호"
- 서버의 응답(②)은 클라이언트에 대한 ACK와 서버 자신의 SYN을 **한 패킷에** 합친 것이다. 그래서 왕복이 3번으로 끝난다.
- **왜 순서번호(seq)를 랜덤하게 시작하나?** 예전 연결의 잔여 패킷이나 위조 패킷과 섞이지 않게 하려고. 보안·정확성 둘 다를 위한 것이다.

### 상태 전이 (전공 교재 관점)

핸드셰이크 동안 각 소켓은 상태 기계(state machine)를 거친다. `netstat`에서 실제로 보이는 상태들이다.

```
  클라이언트:  CLOSED → (SYN 전송) SYN_SENT → (SYN-ACK 수신, ACK 전송) ESTABLISHED
  서버:        LISTEN → (SYN 수신, SYN-ACK 전송) SYN_RCVD → (ACK 수신) ESTABLISHED
```

### 예시 — "첫 요청은 왜 느린가"의 절반

연결 하나 맺는 데 **최소 1번의 왕복(RTT, Round-Trip Time)**이 든다. 지구 반대편 서버라면 이 악수만으로도 수백 ms가 걸린다. 그래서 §5.2의 **keep-alive**로 이 악수를 아끼는 것이다.

### 실습 연결

> **실습으로 확인**: `observe_http.sh`의 `curl -v` 출력에서 `* Connected to example.com (...) port 443`이 3-way handshake가 끝난 지점이다(스크립트의 "[가) TCP 연결 성립]" 블록). 그 직후부터 TLS 핸드셰이크(6장)가 이어진다.

## 3.4 신뢰성 — 순서번호, 확인응답, 재전송

### 개념 — 모든 바이트에 번호를 매긴다

TCP가 "빠짐없이, 순서대로"를 어떻게 보장하는가? **모든 바이트에 순서번호(sequence number)를 매기고, 받은 쪽이 확인응답(ACK)을 보낸다.**

```
  보낸 쪽                                받는 쪽
    │  데이터[seq=1~100] ───────────────→  │  받음. "1~100 확인, 다음 101 줘" (ACK=101)
    │  ←──────────────────────────────────│
    │  데이터[seq=101~200] ──────X (유실)   │  ...안 옴...
    │                                      │
    │  (일정 시간 ACK 안 오면 = 타임아웃)    │
    │  데이터[seq=101~200] 재전송 ────────→ │  받음. "101~200 확인, 다음 201 줘"
    │  ←──────────────────────────────────│
```

### 원리 — 세 가지 장치

- **순서번호(seq)**: 각 바이트에 번호 → 받는 쪽이 뒤죽박죽 도착해도 **번호순으로 재조립**한다.
- **확인응답(ACK)**: "여기까지 받았다"를 알림 → 보낸 쪽은 ACK가 온 데이터만 안심하고 버린다.
- **재전송(retransmission)**: ACK가 안 오면(타임아웃) 또는 **중복 ACK가 3번** 오면 → **다시 보낸다.**

### 재전송의 두 방아쇠 — 타임아웃 vs 빠른 재전송

```
  ① 타임아웃 재전송:  ACK가 RTO(재전송 타임아웃) 시간 안에 안 오면 재전송.
                      RTO는 관측된 RTT를 바탕으로 동적으로 계산된다(RTT가 길면 RTO도 길게).

  ② 빠른 재전송(fast retransmit):  같은 ACK가 3번 연속 오면(중복 ACK)
                      "그 다음 것이 유실됐구나" 판단하고 타임아웃을 기다리지 않고 즉시 재전송.

  받는 쪽:  seq 1~100 받음 → "다음 101 줘"(ACK 101)
            seq 201~300 먼저 도착(101~200 유실) → 계속 "101 줘"(중복 ACK 101) 반복
            → 보낸 쪽이 중복 ACK 3개 보면 101~200을 즉시 재전송
```

이 덕분에 HTTP를 짜는 사람은 "패킷이 유실되면 어쩌지?"를 **전혀 신경 쓰지 않는다.** TCP가 아래에서 다 처리해준다. 이것이 §1.1에서 말한 **책임 분리**의 실전 사례다.

## 3.5 흐름 제어와 혼잡 제어 — 넘치지 않게, 막히지 않게

### 개념 — 누굴 위한 배려인가로 구분하라

두 가지 "제어"가 있다. 이름이 비슷해 헷갈리니 **누굴 위한 배려인지**로 구분하라.

```
  흐름 제어(Flow Control)   →  '받는 상대'를 배려. 상대 버퍼가 넘치지 않게.
  혼잡 제어(Congestion C.)  →  '중간 네트워크'를 배려. 도로(회선)가 막히지 않게.
```

### 원리 1 — 흐름 제어와 슬라이딩 윈도우

받는 쪽이 "내 버퍼에 아직 이만큼 공간 있어(**수신 윈도우, rwnd**)"를 매 ACK마다 알려준다. 보내는 쪽은 그 창(window) 크기만큼만 ACK를 기다리지 않고 **미리 보낼 수 있다**. 이것이 **슬라이딩 윈도우(sliding window)**다.

```
  슬라이딩 윈도우 — "ACK를 기다리지 않고 윈도우 크기만큼 연속 전송"

  이미 ACK 받음  │  보냈고 ACK 대기중  │  아직 안 보냄
  ───────────────┼─────────────────────┼───────────────
   1..100        │  101 .. 400          │  401 ..
                 └──── 윈도우(300B) ────┘
                 ACK 101이 오면 → 윈도우가 오른쪽으로 '미끄러진다(slide)'
                 → 401.. 를 새로 보낼 수 있게 됨
```

윈도우가 없다면 한 조각 보내고 ACK 올 때까지 놀아야 해서 지독히 느리다(stop-and-wait). 윈도우 덕분에 "여러 조각을 파이프에 채워" RTT를 낭비하지 않는다. → **느린 수신자가 빠른 송신자에 압도되지 않게** 하면서도 처리량을 높인다.

### 원리 2 — 혼잡 제어 (slow start와 그 이후)

네트워크 중간이 막히면(패킷 유실 감지) 전송 속도를 **확 줄였다가 천천히 다시 늘린다.** 이를 위해 보내는 쪽은 **혼잡 윈도우(cwnd)**를 따로 두고, 실제 전송량은 `min(rwnd, cwnd)`로 정한다.

```
  전송 속도(cwnd)
    │            ╱|          ╱          ← 혼잡 회피(선형 증가)
    │          ╱  |        ╱
    │        ╱    | (유실!)╱
    │      ╱      |      ╲╱   ← 유실 감지 → cwnd 확 줄임 → 다시 증가
    │   ╱ slow start(지수 증가)
    │ ╱
    └────────────────────────────→ 시간

  slow start:  1 → 2 → 4 → 8 ... 지수적으로 빠르게 늘림
  혼잡 회피:   임계값(ssthresh) 넘으면 조심스럽게 선형 증가
  유실 감지:   "도로가 막혔다" → cwnd를 줄이고 다시 조심스럽게
```

→ 모두가 동시에 마구 보내 인터넷 전체가 마비되는 것을 막는다. 도로에 차가 몰리면 서로 양보해 정체를 푸는 것과 같다.

### 예시 — 비전공자 수준의 직관

전공 시험이 아니라면 "**TCP는 받는 쪽(흐름 제어)과 도로 상황(혼잡 제어)을 봐가며, 슬라이딩 윈도우로 알아서 속도를 조절한다**"는 직관이면 충분하다. 핵심은 흐름 제어는 **상대 버퍼**, 혼잡 제어는 **네트워크**를 본다는 구분이다.

## 3.6 연결 종료 — 4-way close

### 개념 — 시작은 3번, 끝은 4번

시작이 악수 3번이면, 끝은 인사 4번이다. **양쪽이 각자 독립적으로 "나는 이제 보낼 거 없어"를 선언**해야 하기 때문이다.

```
  클라이언트                                         서버
      │  ① FIN   "나 이제 보낼 거 없어"  ──────────→   │
      │  ② ACK   "알았어"                ←──────────   │
      │            (서버는 아직 보낼 게 남았을 수 있음)  │
      │  ③ FIN   "나도 이제 다 보냈어"    ←──────────   │
      │  ④ ACK   "알았어. 잘 가"          ──────────→   │
      │                                               │
      │  (클라이언트는 잠깐 TIME_WAIT 상태로 대기 후 종료)│
```

### 원리 — 왜 3번(handshake)이 아니라 4번인가

시작할 땐 서버의 "받았다(ACK)"와 "나도 시작(SYN)"을 **한 번에** 보낼 수 있었다(②SYN+ACK). 하지만 끝낼 땐, 클라이언트가 "끝"이라 해도 서버는 **아직 보낼 데이터가 남아 있을 수 있어서** ACK(②)와 FIN(③)을 나눠 보낸다. TCP 연결은 **양방향(전이중)**이라, 두 방향을 각각 독립적으로 닫아야 한다.

### TIME_WAIT — "Address already in use"의 정체

능동적으로 먼저 닫은 쪽(보통 클라이언트)은 마지막 ACK(④)를 보낸 뒤 바로 소켓을 없애지 않고 **TIME_WAIT** 상태로 잠시(대략 2×MSL) 기다린다.

```
  TIME_WAIT가 필요한 이유:
    - 마지막 ACK(④)가 유실될 수 있다. 그러면 상대가 FIN(③)을 재전송한다.
      → 내가 이미 소켓을 없앴다면 재전송된 FIN에 응답할 수 없다.
      → 그래서 잠시 남아 있다가 재전송에 대응한다.
    - 예전 연결의 잔여 패킷이 새 연결에 섞이는 것도 방지한다.

  실무 함정: 서버를 껐다 바로 켜면 "Address already in use" 오류가 자주 난다.
            직전 연결이 TIME_WAIT로 포트를 잠시 붙잡고 있어서다.
            (SO_REUSEADDR 옵션으로 완화할 수 있다.)
```

### 실습 연결

> **실습으로 확인**: `RawHttpClient.java`에서 `Connection: close` 헤더를 보내면, 응답 후 서버가 이 종료 절차를 시작한다. `try-with-resources`로 소켓을 닫으면 클라이언트 쪽 FIN이 나간다. `./run.sh RawHttpClient` 실행 직후 `netstat -an | grep 80`을 여러 번 보면 `TIME_WAIT` 상태가 잠깐 남았다 사라지는 것을 볼 수 있다.

## ⚠️ 흔한 오해와 함정

- **"3-way handshake에서 데이터가 오간다."** 아니다. 핸드셰이크는 순서번호 합의와 연결 수립만 한다(순수 SYN/ACK). 실제 HTTP 데이터는 그 이후에 흐른다. (TLS 1.3/TCP Fast Open 같은 최적화는 예외적으로 데이터를 앞당기지만, 기본 원리는 "먼저 연결, 그다음 데이터".)
- **"UDP는 TCP보다 무조건 나쁘다."** 아니다. 실시간성이 중요하면 UDP가 옳다. 신뢰성/실시간성 트레이드오프의 문제다.
- **"흐름 제어와 혼잡 제어는 같은 것이다."** 다르다. 흐름 제어는 **상대 수신 버퍼**를, 혼잡 제어는 **중간 네트워크**를 배려한다. 실제 전송량은 둘 중 작은 쪽으로 제한된다(`min(rwnd, cwnd)`).
- **"타임아웃이 났으니 서버가 에러를 냈다."** 아닐 수 있다. 3-way handshake조차 성립 못 하면(방화벽·라우팅) 요청은 서버 애플리케이션에 **도달조차 못 한다.** 그래서 서버 로그가 비어 있다 — 이건 L3/L4 문제이지 L7(Spring) 문제가 아니다.
- **"서버 재시작 시 포트 오류는 버그다."** 대개는 TIME_WAIT 때문이다. 잠시 기다리거나 SO_REUSEADDR로 해결된다.

## 연습문제

1. **하나의 TCP 연결을 유일하게 식별하는 값 4개는 무엇인가? 브라우저 탭을 하나 더 열면 그중 무엇이 바뀌는가?**
   - *힌트*: 4-튜플.
   - *해설*: (출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트). 새 탭은 보통 출발지 포트만 바뀌어 별개 연결이 된다.

2. **연결 시작은 왜 3번, 종료는 왜 4번인가?**
   - *힌트*: 시작 시 합칠 수 있었던 것과, 종료 시 나눠야 하는 이유.
   - *해설*: 시작은 서버의 ACK와 SYN을 한 패킷(SYN-ACK)으로 합쳐 3번. 종료는 양방향을 독립적으로 닫아야 하고, FIN을 받은 쪽이 아직 보낼 데이터가 남았을 수 있어 ACK와 FIN을 나눠 4번.

3. **TCP가 유실을 감지하는 두 가지 방아쇠는 무엇인가?**
   - *힌트*: 시간과 신호.
   - *해설*: (1) 타임아웃(RTO 안에 ACK 미도착), (2) 빠른 재전송(중복 ACK 3회 수신).

4. **흐름 제어와 혼잡 제어의 차이를 "누구를 배려하는가"로 설명하라. 실제 전송량은 어떻게 결정되나?**
   - *힌트*: 상대 버퍼 vs 도로.
   - *해설*: 흐름 제어는 상대 수신 버퍼(rwnd), 혼잡 제어는 중간 네트워크(cwnd)를 배려. 실제 전송량은 둘 중 작은 값 `min(rwnd, cwnd)`.

5. **서버를 껐다 즉시 켰더니 "Address already in use"가 났다. 어떤 TCP 상태 때문이며 왜 존재하는가?**
   - *힌트*: 마지막 ACK가 유실될 가능성.
   - *해설*: TIME_WAIT. 마지막 ACK가 유실돼 상대가 FIN을 재전송할 경우 대응하고, 이전 연결의 잔여 패킷 혼입을 막기 위해 잠시 소켓/포트를 붙잡는다.

## 요약

- IP는 best-effort. **전송 계층(TCP)이 "순서대로, 빠짐없이"라는 신뢰성을 만든다.**
- **포트**로 한 컴퓨터 안 여러 프로그램을 구분하고, **소켓**은 4-튜플(출발/도착 IP·포트)로 연결을 유일하게 식별한다.
- **TCP는 신뢰·순서 보장(무겁다), UDP는 비연결·비보장(가볍고 빠르다).** 트레이드오프다.
- **3-way handshake(SYN/SYN-ACK/ACK)**로 연결을 수립하며 순서번호(seq)를 합의한다. RTT 1회 비용 → "첫 요청이 느린" 원인의 절반.
- 신뢰성은 **seq(재조립)·ACK(확인)·재전송(타임아웃/중복 ACK)**으로 만든다. **슬라이딩 윈도우**로 처리량을 높이고, **흐름 제어(rwnd)·혼잡 제어(cwnd)**로 속도를 조절한다.
- **4-way close**로 양방향을 독립적으로 닫는다. **TIME_WAIT**가 "Address already in use"의 흔한 원인이다.
