# 6장. HTTPS/TLS — 도청과 위조를 막는 계층

> "SPRING §5.7이 'TLS도 대칭/비대칭 원리'라고 짚고 넘어갔다. 그 미뤄둔 이야기를 지금 편다. §5.7을 먼저(또는 나란히) 읽으면 이 장이 훨씬 선명해진다."

## 도입 — 평문의 위험과 SPRING §5.7의 약속

5장에서 봤듯 HTTP는 사람이 읽는 텍스트다. 그런데 `http://`는 **평문**이다. 중간의 누구든(공용 Wi-Fi, 통신사, 해커) **그대로 엿보거나 바꿀 수 있다.** 비밀번호를 http로 보내면 카페 옆자리에서 훔쳐볼 수 있다는 뜻이다.

`https://`는 HTTP를 **TLS(Transport Layer Security)**라는 보안 계층으로 감싼다. SPRING §5.7은 JWT 서명의 대칭/비대칭 원리를 다루며 "TLS도 같은 원리"라고 짚고 **미뤄뒀다**. 이 장이 바로 그 약속을 갚는다. §5.7의 "개인키로 서명·공개키로 검증, 공개키론 위조 불가"라는 성질이, 여기서는 **가짜 서버를 걸러내는** 데 쓰인다. 이 책의 6장과 SPRING §5.7은 같은 암호 원리의 앞뒤 짝이다.

## 6.1 HTTPS가 지키는 세 가지

### 개념

TLS는 세 가지를 보장한다.

```
  ① 기밀성(Confidentiality) → 암호화. 중간에서 봐도 알 수 없음.       (도청 방지)
  ② 무결성(Integrity)       → 변조 감지. 한 글자라도 바뀌면 들킴.     (위조 방지)
  ③ 인증(Authentication)    → 상대가 진짜 그 서버 맞는지 확인.        (사칭 방지)
```

### 원리 — 왜 인증이 별도로 필요한가

암호화(①)만 하면 "몰래 엿보는 것"은 막지만, 애초에 **가짜 서버(피싱)와 암호화 통신**을 하면 소용없다. 도둑과 암호로 대화해 봐야 비밀이 도둑에게 간다. 그래서 "**너 진짜 example.com 서버 맞아?**"를 증명하는 절차(③ 인증)가 필요하다 — 그게 인증서(§6.3)다. 무결성(②)은 전송 중 한 비트라도 바뀌면 검증에 실패하게 만들어 변조를 잡아낸다.

### 예시 — HTTP 스택에서 TLS의 위치

TLS는 TCP(신뢰성) 위, HTTP(응용) 아래에 낀다. HTTP 입장에선 그냥 "안전해진 TCP"처럼 보인다.

```
  ┌──────────┐
  │  HTTP     │  ← 응용 데이터
  ├──────────┤
  │  TLS      │  ← ★ 여기서 암호화. HTTP 입장에선 "그냥 안전한 TCP"처럼 보임
  ├──────────┤
  │  TCP      │  ← 신뢰성(3장)
  ├──────────┤
  │  IP       │
  └──────────┘
```

이 위치 덕분에 §1.1의 계층 원리가 다시 빛난다. HTTP는 코드 한 줄 바꾸지 않아도 TLS가 아래에서 암호화를 대신해 준다.

## 6.2 TLS 핸드셰이크 — 비대칭으로 대칭키를 건네고, 대칭으로 달린다

### 개념 — 두 종류의 암호를 조합한다

이것이 §5.7이 "미뤄둔" 바로 그 이야기다. TLS의 천재적인 아이디어는 **두 종류의 암호를 조합**하는 것이다.

- **비대칭키(공개키/개인키)**: 안전하지만 **느리다.** (SPRING §5.7의 RS256과 같은 원리)
- **대칭키(하나의 비밀키)**: 빠르지만 **키를 어떻게 안전하게 공유하지?**가 숙제.

**해법: 느리고 안전한 비대칭으로 "대칭키(세션 키)"를 딱 한 번 안전하게 합의하고, 그다음부터는 빠른 대칭키로 실제 데이터를 암호화한다.** §5.7 마지막 문단이 정확히 이 말이었다.

### 원리 — TLS 1.3 핸드셰이크 (1-RTT)

과거 TLS 1.2는 2왕복(2-RTT)이 필요했지만, **현재 표준인 TLS 1.3은 1왕복(1-RTT)**으로 줄였다. 클라이언트가 처음부터 키 교환값을 함께 보내기 때문이다.

```
  클라이언트(브라우저)                                     서버(example.com)
      │                                                     │
      │  ① ClientHello  (+ key_share, 지원 암호/버전, ALPN)  │
      │  "TLS 1.3 하자. 내 키 교환 공개값은 이거야.           │
      │   HTTP/2 쓸래(ALPN). 지원 암호 목록은 이것들"         │
      │ ───────────────────────────────────────────────────→│
      │                                                     │
      │  ② ServerHello (+ key_share) + 인증서 + Finished     │
      │  "이 암호로 하자. 내 키 교환 공개값은 이거,           │
      │   내 인증서는 이거(도메인·공개키·CA서명). 검증 데이터."│
      │ ←───────────────────────────────────────────────────│
      │                                                     │
      │  ③ [인증서 검증]  ★ 이 서버가 진짜 example.com인가?    │
      │     - CA 서명이 유효한가? 도메인이 맞나? 만료 안 됐나?  │
      │     - 양쪽 key_share로 '세션 키'를 각자 계산(합의 완료) │
      │                                                     │
      │  ④ Finished                                          │
      │ ───────────────────────────────────────────────────→│
      │                                                     │
      │  ⑤ 이제 양쪽 다 같은 '세션 키(대칭키)'를 갖게 됨        │
      │ ═══════════ 이후 모든 HTTP는 세션 키로 암호화 ════════ │
      │        빠른 대칭 암호로 실제 데이터를 주고받음          │
```

핵심은 ①~③이다. 양쪽이 `key_share`(키 교환 공개값)를 주고받아, **오직 이 세션에만 쓸 대칭키(세션 키)**를 각자 독립적으로 계산해 낸다(디피-헬만 방식). 도청자가 오가는 값을 다 봐도, 그 값만으로는 세션 키를 계산할 수 없다. 세션 키가 정해진 뒤엔 비대칭은 버리고 **빠른 대칭키로만** 통신한다 — 안전함과 빠름을 모두 취한 것이다.

### 원리 — 왜 "임시 키 교환"인가 (전방향 비밀성)

TLS 1.3은 매 세션마다 **일회용 키 교환값**을 쓴다(ephemeral). 그래서 훗날 서버의 개인키가 유출돼도, 과거에 녹음해 둔 트래픽은 여전히 복호화할 수 없다(각 세션의 대칭키가 개인키가 아니라 그때그때의 일회용 값에서 나왔기 때문). 이 성질을 **전방향 비밀성(Forward Secrecy)**이라 부른다. 전공 교재가 TLS 1.3의 개선점으로 꼭 짚는 항목이다.

### 예시 — 왜 첫 HTTPS 요청은 더 느린가

이제 명확하다. TCP 3-way handshake(§3.3) **위에** TLS 핸드셰이크의 왕복(1-RTT)이 더해진다. 그래서 §5.2의 keep-alive와 §5.4의 캐시가 더더욱 값지다 — 비싼 핸드셰이크를 한 번만 치르고 재사용하니까. (TLS 1.3은 재접속 시 **0-RTT**로 첫 데이터를 즉시 보내는 옵션도 있다.) 또한 ①의 ALPN에서 HTTP/2 사용 여부가 이때 결정된다(§5.2).

## 6.3 인증서와 CA — "진짜 그 서버인가"를 어떻게 믿나

### 개념 — 공개키를 어떻게 신뢰하나

②에서 서버가 준 "공개키"를 어떻게 믿는가? 공격자도 "나 example.com이야, 이게 내 공개키야" 하고 자기 공개키를 내밀 수 있다(중간자 공격, MITM). 이 신뢰 문제를 푸는 게 **인증서(Certificate)**와 **CA(Certificate Authority, 인증 기관)**다.

### 원리 — 신뢰 체인(chain of trust)

```
  인증서 = 공개키 + 도메인 이름(example.com) + 유효기간을,
          믿을 수 있는 제3자(CA)가 '자기 개인키로 서명'해 준 문서

  신뢰 체인(chain of trust):
    ┌─────────────────┐
    │ Root CA          │  ← 브라우저/OS에 '이미 신뢰됨'으로 내장된 최상위 기관
    │   서명 ↓          │
    │ 중간 CA           │
    │   서명 ↓          │
    │ example.com 인증서 │  ← 서버가 제시하는 잎(leaf) 인증서
    └─────────────────┘

  브라우저는 이 체인을 위로 따라가, 뿌리(Root CA)가 '내가 이미 신뢰하는 기관'이면 통과.
```

**논리 구조가 §5.7의 비대칭 서명과 완전히 똑같다.** CA는 **개인키로 인증서에 서명**하고, 브라우저는 이미 갖고 있는 **CA의 공개키로 서명을 검증**한다. 서명이 유효하면 "믿을 수 있는 CA가 이 도메인의 공개키임을 보증했다"는 뜻이다. §5.7에서 "공개키로 검증만 가능하고 위조는 불가능"이라 한 그 성질이, 여기서 **가짜 서버를 걸러내는** 데 쓰인다.

### 원리 — 왜 중간 CA가 끼는가

Root CA의 개인키는 극도로 소중해서 평소엔 오프라인 금고에 보관하고 직접 서명에 쓰지 않는다. 대신 **중간 CA**에게 서명 권한을 위임하고, 실제 서버 인증서는 중간 CA가 발급한다. 그래서 서버는 보통 [자기 인증서 + 중간 CA 인증서]를 함께 제시하고, 브라우저는 그것을 자신이 내장한 Root CA까지 이어 검증한다. Root가 뚫려도 그것 하나로 전체가 무너지지 않게 하는 위험 분산이다.

### 예시 — 실무에서 만나는 인증서 오류

- **인증서 만료** → 브라우저가 빨간 경고. (그래서 자동 갱신을 건다 — 예: Let's Encrypt는 90일마다 자동 갱신)
- **도메인 불일치**(`a.com` 인증서를 `b.com`이 제시) → 경고. 인증서의 도메인 필드와 접속한 호스트가 다르기 때문.
- **사설/자체 서명 인증서** → Root CA 체인이 없어 경고. 개발용으로 `-k`(curl)나 예외 등록으로 우회하지만, 운영에선 절대 금물.
- **체인 불완전**(중간 CA 인증서 누락) → 일부 클라이언트에서 검증 실패. 서버가 중간 CA까지 함께 보내야 한다.

### 실습 연결

> **실습으로 확인**: `observe_http.sh`의 STEP 3이 `openssl s_client -connect example.com:443 -servername example.com`으로 **실제 인증서 체인, TLS 버전, 암호 스위트**를 그대로 보여준다. 출력에서 `s:`(subject, 인증서 주인)와 `i:`(issuer, 서명한 상위 CA)가 leaf → 중간 CA로 이어지는 신뢰 사슬이다. `Verify return code: 0 (ok)`이면 검증 통과. STEP 2(`curl -v`)의 `[나) TLS 핸드셰이크]` 블록(TLS 버전·인증서·ALPN 줄들)이 §6.2 ①~⑤의 실제 로그다. 참고로 `-servername`(SNI)은 §5.1의 `Host` 헤더와 같은 이유 — 한 IP가 여러 도메인을 호스팅하므로 암호화 전에 "어느 도메인인지" 알려주는 것이다.

## ⚠️ 흔한 오해와 함정

- **"HTTPS는 데이터를 비대칭키로 암호화한다."** 아니다. 비대칭은 **세션 키 합의**에만 잠깐 쓰고, 실제 데이터는 빠른 **대칭키**로 암호화한다. 비대칭만 쓰면 너무 느리다.
- **"HTTPS면 상대가 100% 안전한 사이트다."** 아니다. TLS는 "이 연결이 인증서에 적힌 도메인과 안전하게 연결됐다"만 보장한다. 그 도메인 자체가 악성(피싱)일 수 있다. 자물쇠 아이콘 ≠ 착한 사이트.
- **"인증서는 데이터를 암호화한다."** 인증서는 암호화 도구가 아니라 **신원 보증서**(공개키 + 도메인 + CA 서명)다. 암호화는 세션 키가 한다.
- **"자체 서명 인증서도 운영에 써도 된다."** 안 된다. Root CA 체인이 없어 클라이언트가 신뢰하지 않는다. 개발용 우회(`-k`)를 운영에 쓰면 MITM에 무방비다.
- **"TLS 1.3도 2왕복이 필요하다."** 아니다. TLS 1.3은 1-RTT(재접속 시 0-RTT)로 줄였다. 2-RTT는 옛 TLS 1.2 이야기다.
- **"서버 개인키가 유출되면 과거 트래픽도 다 털린다."** TLS 1.3의 전방향 비밀성(일회용 키 교환) 덕분에 과거 녹음 트래픽은 안전하다.

## 연습문제

1. **TLS가 보장하는 세 가지는 무엇이며, 암호화만으로 부족해 "인증"이 왜 별도로 필요한가?**
   - *힌트*: 피싱 서버와 암호 통신.
   - *해설*: 기밀성·무결성·인증. 암호화만 하면 가짜 서버와도 안전하게(?) 대화하게 되므로, "진짜 그 서버인가"를 확인하는 인증(인증서)이 필요하다.

2. **TLS는 왜 비대칭키와 대칭키를 섞어 쓰는가? 각각 어디에 쓰이나?**
   - *힌트*: 느림/빠름, 키 공유 문제.
   - *해설*: 비대칭은 안전하나 느리고, 대칭은 빠르나 키 공유가 문제다. 그래서 비대칭으로 세션 키를 안전하게 합의하고, 이후 실제 데이터는 빠른 대칭키로 암호화한다.

3. **브라우저는 서버 인증서의 공개키를 어떻게 신뢰하는가? SPRING §5.7의 어떤 성질을 쓰는가?**
   - *힌트*: CA 서명, 공개키 검증.
   - *해설*: CA가 개인키로 인증서에 서명하고, 브라우저는 내장된 CA 공개키로 서명을 검증한다. "공개키로 검증만 가능·위조 불가"(§5.7)라는 성질을 이용해 가짜 서버를 걸러낸다.

4. **Root CA가 서버 인증서를 직접 서명하지 않고 중간 CA를 두는 이유는?**
   - *힌트*: 개인키 보호, 위험 분산.
   - *해설*: Root 개인키는 극히 소중해 오프라인 보관하고, 서명 권한을 중간 CA에 위임한다. Root가 노출·오남용될 위험을 줄이고 사고 시 영향을 국소화한다.

5. **"첫 HTTPS 요청이 http보다 느린" 이유와, 그 비용을 줄이는 이 책의 두 장치를 대라.**
   - *힌트*: handshake, keep-alive/캐시.
   - *해설*: TCP 3-way handshake 위에 TLS 핸드셰이크 왕복이 더해져서다. §5.2 keep-alive/멀티플렉싱과 §5.4 캐시(및 TLS 세션 재사용)로 이 비용을 재사용해 아낀다.

## 요약

- HTTP는 평문이라 도청·변조에 노출된다. **HTTPS = HTTP over TLS**이며 TLS는 **기밀성·무결성·인증** 세 가지를 지킨다. 인증이 별도로 필요한 이유는 가짜 서버(피싱) 방어다.
- TLS는 TCP 위·HTTP 아래에 낀다. HTTP는 코드 변경 없이 "안전해진 TCP"를 쓴다(§1.1 계층 원리).
- **TLS 1.3 핸드셰이크(1-RTT)**: `key_share`를 주고받아 세션 대칭키를 각자 계산 → 이후 빠른 대칭키로 통신. **비대칭으로 키 합의, 대칭으로 데이터.** 일회용 키 교환으로 **전방향 비밀성**을 얻는다. ALPN으로 HTTP/2 여부도 이때 정한다.
- **인증서 = 공개키 + 도메인 + CA 서명.** 브라우저는 leaf → 중간 CA → Root CA 신뢰 체인을 검증한다. 논리 구조는 SPRING §5.7의 비대칭 서명과 동일하다.
- 첫 HTTPS가 느린 것은 handshake 때문 → keep-alive·캐시·세션 재사용으로 상쇄한다.
