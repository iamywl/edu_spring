# Chapter 1: 관계 모델 (The Relational Model)

> "우리가 매일 쓰는 '테이블'은 사실 1970년 에드거 커드(Edgar F. Codd)가 만든 **수학 이론**의 산물이다. 테이블이 왜 하필 '행과 열'의 모양인지, 거기엔 이유가 있다."

---

## 1.1 왜 관계형인가? — 다른 방식과 비교

데이터를 저장하는 방법은 여러 가지다. 관계형이 왜 40년 넘게 왕좌를 지키는지 알려면, 관계형이 **없던 시절**과 **대안들**을 알아야 한다.

**대안 1: 그냥 파일에 저장 (플랫 파일)**
```
users.txt:
1,김철수,kim@x.com,컴퓨터공학
2,이영희,lee@x.com,컴퓨터공학
```
문제: "컴퓨터공학과 학생 몇 명?"을 알려면 파일 전체를 코드로 읽어 세야 한다. 두 파일을 합치기(JOIN)도 손으로 코딩. 동시에 두 명이 수정하면 깨진다.

**대안 2: 계층형(Hierarchical) / 네트워크형 DB (1960~70년대)**
데이터를 부모-자식 트리로 저장했다. "학과 → 학생 → 수강" 같은 구조.
문제: 트리 구조가 **미리 정해진 경로**로만 탐색 가능. "이 강좌를 듣는 학생 목록"처럼 트리를 거꾸로 타는 질의가 지옥이었다.

**관계형의 답:** 데이터를 **테이블(관계)**로만 표현하고, 테이블끼리는 **값(키)으로 연결**한다. 그리고 **"무엇을 원하는지"만 선언하면(SQL) DB가 "어떻게 찾을지"는 알아서** 한다.

```
계층형/네트워크형:  "학과에서 출발해 → 학생으로 → 수강으로 내려가라" (경로를 내가 지정)
관계형:             "성적이 A인 학생 이름을 다오"                    (경로는 DB가 결정)
```

이 **"어떻게가 아니라 무엇을"** 이라는 선언적 성질이 관계형의 혁명이었다. 개발자는 데이터의 **논리적 구조**만 알면 되고, 물리적으로 어떻게 저장/탐색되는지(인덱스, 조인 알고리즘)는 DB가 최적화한다.

> **NoSQL은?** 2000년대에 등장한 MongoDB(문서형), Redis(키-값), Cassandra(컬럼형)는 "대규모 확장"과 "유연한 스키마"를 위해 관계형의 일부(JOIN, 강한 일관성)를 포기한 것들이다. 관계형을 **대체**한 게 아니라, "정합성이 덜 중요하고 규모가 극단적인" 특정 상황을 위한 **다른 트레이드오프**다. 대부분의 업무 시스템(은행, 쇼핑몰, 이 프로젝트의 게시판)은 여전히 관계형이 정답이다. — 이 프로젝트가 PostgreSQL을 쓰는 이유다.

## 1.2 용어: 관계 / 튜플 / 애트리뷰트

관계형의 이론 용어와 실무 용어를 나란히 두자. 시험/면접에서는 이론 용어로, 실무에서는 오른쪽으로 부른다.

```
이론 용어              실무 용어         예시
──────────────────────────────────────────────────────
Relation (관계)     =  Table (테이블)    users 테이블
Tuple (튜플)        =  Row (행/레코드)   {1, "김철수", "kim@x.com"}
Attribute (애트리뷰트) = Column (열/필드)  name, email
Domain (도메인)     =  컬럼의 타입/제약   name의 도메인 = "50자 이하 문자열"
Degree (차수)       =  컬럼의 개수        users의 차수 = 3
Cardinality (카디널리티) = 행의 개수      users의 카디널리티 = 2
```

```
                    users 테이블 (Relation)
        ┌─────────┬──────────┬────────────────┐
        │ id      │ name     │ email          │  ← Attribute (컬럼) 3개 → Degree 3
        ├─────────┼──────────┼────────────────┤
Tuple → │ 1       │ 김철수    │ kim@x.com      │
Tuple → │ 2       │ 이영희    │ lee@x.com      │  ← Tuple (행) 2개 → Cardinality 2
        └─────────┴──────────┴────────────────┘
```

**관계 모델의 두 가지 수학적 약속 (중요):**

1. **튜플에는 순서가 없다.** 테이블은 "집합(set)"이다. 1번 행이 위에 있든 아래 있든 논리적으로 동일하다. → 그래서 "몇 번째 행"이라는 개념이 없고, 원하는 순서로 보려면 반드시 `ORDER BY`를 써야 한다.
2. **컬럼 값은 원자적(atomic)이다.** 한 칸에는 값이 딱 하나. `phone` 칸에 "010-1111, 010-2222"처럼 여러 값을 넣으면 안 된다. (이게 뒤에 나올 **1NF**의 기초다.)

## 1.3 키(Key): 행을 어떻게 구별하는가

관계 모델에서 튜플에는 순서가 없다고 했다. 그러면 특정 행을 어떻게 **콕 집어** 가리킬까? 답은 **키(key)**, 즉 "그 행을 유일하게 식별하는 컬럼(들)"이다.

### 슈퍼키(Super Key)
행을 유일하게 구별할 수 있는 컬럼들의 **모든** 조합. 예: `{id}`, `{id, name}`, `{email}`, `{email, name}`... 낭비되는 컬럼이 붙어 있어도 유일하기만 하면 슈퍼키다.

### 후보키(Candidate Key)
슈퍼키 중에서 **군더더기 없는 최소 조합**. `{id, name}`은 `name`을 빼도 여전히 유일하므로 후보키가 아니다. `{id}`와 `{email}`은 각각 하나만으로 유일하고 더 뺄 게 없으므로 **후보키**다. → 후보키는 여러 개일 수 있다.

### 기본키(Primary Key, PK)
여러 후보키 중 **대표로 선택한 하나**. 관례상 변하지 않고 짧은 것을 고른다.
```sql
CREATE TABLE users (
    id    BIGSERIAL PRIMARY KEY,   -- 기본키로 id 선택 (email도 후보키지만)
    email TEXT UNIQUE NOT NULL,     -- 나머지 후보키는 UNIQUE로 유일성 유지
    name  TEXT NOT NULL
);
```
**왜 email 대신 id를?** 이메일은 사람이 **바꿀 수 있다**. 기본키가 바뀌면 그것을 참조하는 모든 곳(외래키)이 연쇄로 바뀌어야 해서 위험하다. 그래서 실무는 **의미 없는 대리키(surrogate key, 자동증가 id)**를 PK로 쓴다.

**PK의 두 규칙:** ① NULL 불가(NULL은 "값 없음"이라 식별에 못 씀), ② 중복 불가.

### 외래키(Foreign Key, FK)
**다른 테이블의 PK를 가리키는 컬럼.** 테이블 사이의 "관계"를 값으로 표현하는 장치다.
```sql
CREATE TABLE posts (
    id        BIGSERIAL PRIMARY KEY,
    title     TEXT NOT NULL,
    author_id BIGINT NOT NULL REFERENCES users(id)  -- users.id를 가리키는 FK
);
```
```
      users                          posts
   ┌──────────┐                 ┌──────────────────┐
   │ id  (PK) │◄────────────────│ author_id  (FK)  │
   │ email    │   "이 게시글의   │ id  (PK)         │
   │ name     │    작성자는      │ title            │
   └──────────┘    users의 이 행"└──────────────────┘
```
**FK가 지키는 것 = 참조 무결성(referential integrity).** `posts.author_id = 5`인데 `users`에 id=5가 없다면? 그건 "존재하지 않는 사람이 쓴 글"이다. FK 제약은 이런 **미아 데이터(orphan)**를 DB가 원천 거부하게 한다.
```sql
INSERT INTO posts (title, author_id) VALUES ('제목', 999);
-- ERROR: insert or update on table "posts" violates foreign key constraint
--        (users에 id=999가 없으므로 거부)
```

> **JPA 연결:** `@Id`가 PK, `@ManyToOne @JoinColumn(name="author_id")`가 FK, `@Column(unique=true)`가 후보키의 UNIQUE 제약을 만든다. Spring 개념서 4.2/4.4절의 그 애노테이션들이 **여기서 배운 키 이론을 테이블로 번역**한 것이다. 자세한 매핑은 이 책 7장에서 정리한다.

# Chapter 2: 정규화 (Normalization)

> "정규화는 'DB 설계에서 가장 중요하지만 가장 자주 건너뛰는' 주제다. 정규화를 모르면, 처음엔 잘 돌던 테이블이 데이터가 쌓일수록 조용히 썩어간다."

정규화(normalization)란 **데이터 중복과 이상 현상을 없애기 위해, 하나의 큰 테이블을 여러 개의 작은 테이블로 쪼개는 체계적 절차**다. "왜 쪼개야 하는가"부터 시작하자.

> 이 장의 모든 내용은 `chapter-cs-database/labs/01_normalization.sql`에서 **실제로 실행**해 볼 수 있다.

---

## 2.1 문제: 하나에 다 때려박은 테이블

수강신청을 관리한다고 하자. "그냥 필요한 걸 다 한 테이블에" 넣으면 이렇게 된다.

```
enrollment (정규화 안 됨 — 이대로 두면 안 되는 예)
┌──────────┬────────────┬──────────┬────────────┬──────────┬──────────────┬──────────┬───────┐
│student_id│student_name│dept_name │dept_office │course_id │course_title  │professor │grade  │
├──────────┼────────────┼──────────┼────────────┼──────────┼──────────────┼──────────┼───────┤
│ 1001     │ 김철수      │컴퓨터공학 │ 공학관 301  │ CS101    │ 자료구조      │ 이교수    │ A     │
│ 1001     │ 김철수      │컴퓨터공학 │ 공학관 301  │ CS102    │ 데이터베이스   │ 박교수    │ B     │
│ 1002     │ 이영희      │컴퓨터공학 │ 공학관 301  │ CS101    │ 자료구조      │ 이교수    │ A     │
│ 1003     │ 박민수      │전자공학   │ 공학관 502  │ EE201    │ 회로이론      │ 최교수    │ C     │
└──────────┴────────────┴──────────┴────────────┴──────────┴──────────────┴──────────┴───────┘
```

한눈에 보이는 문제: `컴퓨터공학 / 공학관 301`이 **세 번** 반복되고, `CS101 / 자료구조 / 이교수`도 **두 번** 반복된다. 이 중복이 세 가지 재앙을 부른다.

### 갱신 이상 (Update Anomaly)
컴퓨터공학과 사무실이 이사를 갔다. `dept_office`를 고쳐야 하는데, 이 값이 **여러 행에 흩어져** 있다. 세 행을 다 고쳐야 하는데 하나라도 빠뜨리면?
```
1001 김철수 → 공학관 301   ← 안 고침
1002 이영희 → 신공학관 101  ← 고침
```
같은 학과인데 사무실이 **두 개**로 갈라진다. 데이터가 **거짓말**을 하기 시작한다.

### 삽입 이상 (Insert Anomaly)
새 강좌 `CS201 운영체제(정교수)`를 개설했는데 아직 수강생이 없다. 이 테이블은 "수강신청 한 건"이 있어야 행이 생기는 구조다. 학생 없이 강좌만 넣으려면 학생 컬럼을 **전부 NULL**로 둔 억지 행을 만들어야 한다. → "강좌 등록"이라는 별개의 사실을 저장할 방법이 없다.

### 삭제 이상 (Delete Anomaly)
박민수(1003)가 자퇴해 그의 행을 지운다. 그런데 박민수는 `EE201 회로이론`의 유일한 수강생이었다. 그 행을 지우는 순간 **"EE201 회로이론은 최교수 담당"이라는 강좌 정보까지 함께 증발**한다. → 학생을 지웠을 뿐인데 무관한 사실이 사라진다.

**세 이상 현상의 근본 원인은 하나:** 서로 다른 종류의 사실(**학생**, **강좌**, **수강**)을 **한 테이블에 섞어** 놓았기 때문이다. 정규화는 이 사실들을 종류별로 분리한다.

## 2.2 함수적 종속 (Functional Dependency)

정규화를 하려면 먼저 데이터 안의 "규칙"을 표현하는 언어가 필요하다. 그게 **함수적 종속(FD)**이다.

> **X → Y** ("X가 Y를 함수적으로 결정한다") = "X 값이 정해지면 Y 값이 **유일하게** 정해진다"

수학 함수 `y = f(x)`와 같다. x 하나에 y 하나. 위 테이블에서 성립하는 FD들:

```
student_id             → student_name, dept_name    (학번을 알면 이름·학과가 정해짐)
dept_name              → dept_office                 (학과를 알면 사무실이 정해짐)
course_id              → course_title, professor     (강좌코드를 알면 제목·교수가 정해짐)
(student_id, course_id) → grade                      (학생+강좌 조합이 정해지면 성적이 정해짐)
```

`→`의 **왼쪽을 결정자(determinant)**라 부른다. 이 테이블의 후보키는 `(student_id, course_id)` 복합키다(이 둘이 있어야 한 행이 특정됨).

**핵심 통찰:** 이상 현상은 "결정자가 후보키가 아닌 FD"에서 나온다. `dept_name → dept_office`에서 `dept_name`은 이 테이블의 키가 아니다. 그래서 `dept_name`이 반복될 때마다 `dept_office`도 딸려 반복되고, 그게 중복과 이상 현상을 만든다. **정규화 = 이런 "나쁜 FD"들을 각자의 테이블로 독립시키는 것.**

## 2.3 단계별 정규화: 1NF → 2NF → 3NF → BCNF

정규형(Normal Form)은 "이 조건을 만족하면 이 등급"이라는 계단이다. 위로 갈수록 엄격하고, 각 단계는 앞 단계를 **포함**한다.

### 제1정규형 (1NF): 모든 값이 원자적

> **규칙:** 모든 컬럼의 값은 더 쪼갤 수 없는 **원자값** 하나여야 한다. (반복 그룹·다중값 금지)

위반 예:
```
┌──────────┬───────────────────┐
│student_id│ courses           │
│ 1001     │ CS101, CS102      │  ← 한 칸에 값 두 개! 1NF 위반
└──────────┴───────────────────┘
```
"CS101을 듣는 학생"을 찾으려면 문자열을 쪼개 검색해야 한다 → 관계 모델의 힘을 잃는다. 1NF는 이걸 각 값이 **한 행씩** 차지하도록 편다.
```
┌──────────┬──────────┐
│student_id│ course   │
│ 1001     │ CS101    │
│ 1001     │ CS102    │  ← 각 칸에 값 하나. 1NF 만족
└──────────┴──────────┘
```
우리의 `enrollment` 예제는 각 칸에 값이 하나뿐이라 **이미 1NF**다.

### 제2정규형 (2NF): 부분 종속 제거

> **규칙:** 1NF + 후보키의 **일부에만** 종속되는 비(非)키 컬럼이 없어야 한다. (**부분 종속** 제거)

이건 **복합 키(키가 2개 이상의 컬럼)**일 때만 문제가 된다. 우리 키는 `(student_id, course_id)`다. 여기서:
```
student_name, dept_name  은 student_id 에만 종속 (course_id 와 무관!)  ← 부분 종속
course_title, professor  은 course_id 에만 종속 (student_id 와 무관!) ← 부분 종속
grade                    은 (student_id, course_id) 전체에 종속        ← 완전 종속 (OK)
```
`student_name`은 학번만 알면 정해진다. 강좌와 상관없다. 그런데도 이 테이블에선 김철수가 두 강좌를 들으면 이름이 두 번 저장된다. → 부분 종속이 중복의 원인. **분리한다:**

```
student 테이블 (student_id 가 결정하는 것들)
┌──────────┬────────────┬──────────┐
│student_id│student_name│dept_name │
│ 1001     │ 김철수      │컴퓨터공학 │
│ 1002     │ 이영희      │컴퓨터공학 │
│ 1003     │ 박민수      │전자공학   │
└──────────┴────────────┴──────────┘   (dept_office 는 아직 여기 있음 — 3NF에서 처리)

course 테이블 (course_id 가 결정하는 것들)
┌──────────┬──────────────┬──────────┐
│course_id │course_title  │professor │
│ CS101    │ 자료구조      │ 이교수    │
│ CS102    │ 데이터베이스   │ 박교수    │
│ EE201    │ 회로이론      │ 최교수    │
└──────────┴──────────────┴──────────┘

enrollment 테이블 (완전 종속인 grade만 남김 + 두 FK)
┌──────────┬──────────┬───────┐
│student_id│course_id │grade  │   PK=(student_id, course_id)
│ 1001     │ CS101    │ A     │   student_id → student(student_id)
│ 1001     │ CS102    │ B     │   course_id  → course(course_id)
│ 1002     │ CS101    │ A     │
│ 1003     │ EE201    │ C     │
└──────────┴──────────┴───────┘
```
**이미 삽입/삭제 이상이 해결된다:** 수강생 없는 강좌는 `course`에 그냥 넣으면 되고(삽입 이상 해결), 수강 기록을 지워도 강좌 정보는 `course`에 남는다(삭제 이상 해결).

### 제3정규형 (3NF): 이행적 종속 제거

> **규칙:** 2NF + 비(非)키 컬럼이 **다른 비키 컬럼을 거쳐** 키에 종속되면 안 된다. (**이행적 종속** 제거)

`student` 테이블을 보자:
```
student_id → dept_name → dept_office
(학번)       (학과)       (사무실)
```
`dept_office`는 키(`student_id`)에 **직접** 종속되는 게 아니라, `dept_name`을 **거쳐서** 종속된다. 이걸 **이행적 종속(transitive dependency)**이라 한다. 결과: 컴퓨터공학 학생이 100명이면 "공학관 301"이 100번 반복 → 갱신 이상의 원인. **분리한다:**

```
department 테이블 (dept_name 이 결정하는 것)          student 테이블 (사무실을 뺌)
┌──────────┬────────────┐                          ┌──────────┬────────────┬──────────┐
│dept_name │dept_office │                          │student_id│student_name│dept_name │← FK
│컴퓨터공학 │ 공학관 301  │◄─────────────────────────│ 1001     │ 김철수      │컴퓨터공학 │
│전자공학   │ 공학관 502  │                          │ 1002     │ 이영희      │컴퓨터공학 │
└──────────┴────────────┘                          │ 1003     │ 박민수      │전자공학   │
                                                    └──────────┴────────────┴──────────┘
```
**갱신 이상 해결:** 학과 사무실 이전은 이제 `department`의 **딱 한 행**만 고치면 끝. 모순이 원천적으로 불가능하다.

### 보이스-코드 정규형 (BCNF): 3NF의 엄격한 버전

> **규칙:** 모든 FD의 **결정자가 후보키**여야 한다. (3NF의 예외 케이스까지 막는 강화판)

3NF는 "비키 컬럼의 이행 종속"을 막지만, 결정자가 **후보키의 일부인 컬럼**일 때 드물게 빈틈이 남는다. BCNF는 "`X → Y`가 성립하면 X는 반드시 후보키"라고 못 박는다.

우리 예제에서 `department`를 분리한 결과, `dept_name → dept_office`의 결정자 `dept_name`이 `department` 테이블의 **후보키(PK)**가 되었다. 즉 department를 분리한 순간 우리는 이미 **BCNF까지** 달성했다. (대부분의 실무 스키마는 3NF만 잘 지키면 자연히 BCNF가 된다. BCNF가 별도로 문제되는 건 "복합 후보키가 겹치는" 특수한 경우다.)

**정규화 여정 요약:**
```
1개 거대 테이블 (이상 현상 3종)
   │ 1NF: 원자값 확인 (이미 만족)
   ▼
   │ 2NF: 부분 종속 제거 → student / course / enrollment 로 분리
   ▼
   │ 3NF: 이행적 종속 제거 → department 분리
   ▼
   │ BCNF: 모든 결정자가 후보키 (department 분리로 자동 달성)
   ▼
4개 테이블: department, student, course, enrollment  (중복·이상 현상 0)
```

## 2.4 정규화 vs 비정규화: 언제 일부러 안 하나?

정규화의 대가는 **JOIN**이다. 데이터를 여러 테이블로 쪼갰으니, 원래 형태로 보려면 다시 합쳐야 한다.
```sql
-- 원래의 "한 테이블처럼" 보려면 JOIN으로 합친다
SELECT s.student_name, d.dept_office, c.course_title, e.grade
  FROM enrollment e
  JOIN student    s ON s.student_id = e.student_id
  JOIN department d ON d.dept_name  = s.dept_name
  JOIN course     c ON c.course_id  = e.course_id;
```
정규화의 철학은 **"저장은 나눠서(중복 제거), 조회는 JOIN으로 합쳐서"**다.

**비정규화(denormalization)**는 성능을 위해 **일부러 중복을 허용**하는 것이다. 예: 조회가 극단적으로 많고 JOIN 비용이 부담되는 통계 화면에서, 미리 계산한 값을 중복 저장. 하지만 이건 "정규화를 몰라서"가 아니라 "정규화를 **알고 나서** 의도적으로 어기는" 고급 최적화다. **원칙: 일단 3NF까지 정규화하고, 측정된 성능 문제가 있을 때만 신중히 비정규화한다.**

> **JPA 연결:** `@Entity` 하나가 보통 정규화된 테이블 하나에 대응한다. 여러 Entity로 나뉜 데이터를 `@ManyToOne` 등으로 참조하다가 JOIN이 폭발하는 게 바로 Spring 개념서 4.4절의 **N+1 문제**다. 즉 "정규화가 만든 JOIN 필요성"과 "ORM이 그걸 자동 번역하며 생기는 N+1"은 동전의 양면이다. → 7장에서 이어짐.

# Chapter 3: 트랜잭션과 ACID

> Spring 개념서 4.5절은 "`@Transactional`이 ACID를 보장한다"고 요약했다. 이 장은 그 **ACID 네 글자 각각이 정확히 무엇을 약속하며, 지켜지지 않으면 어떤 재앙이 나는지**를 실패 시나리오로 판다.

실습: `chapter-cs-database/labs/02_transactions_acid.sql`

---

## 3.1 트랜잭션이란

**트랜잭션(transaction)**은 **"전부 성공하거나 전부 실패하는, 나눌 수 없는 작업 묶음"**이다. 데이터베이스는 트랜잭션에게 네 가지를 약속한다: **ACID**.

```
A - Atomicity  (원자성) : 전부 or 전무. 중간 상태는 없다.
C - Consistency(일관성) : 트랜잭션 전후로 DB 규칙(제약)이 항상 참.
I - Isolation  (격리성) : 동시 트랜잭션끼리 서로의 중간 상태를 못 본다.
D - Durability (지속성) : 커밋된 데이터는 정전에도 살아남는다.
```

---

## 3.2 Atomicity (원자성) — "돈이 증발하는" 시나리오

계좌 이체: A에서 5만원 출금, B에 5만원 입금. 두 UPDATE다.
```sql
UPDATE account SET balance = balance - 50000 WHERE id = 'A';  -- (1) 출금
-- ★ 여기서 서버가 죽으면? ★
UPDATE account SET balance = balance + 50000 WHERE id = 'B';  -- (2) 입금
```
(1)만 반영되고 (2) 전에 서버가 죽으면 → **5만원이 이 세상에서 사라진다.** 원자성은 (1)과 (2)를 하나로 묶어 **"둘 다 되거나 둘 다 안 되게"** 한다.
```sql
BEGIN;
  UPDATE account SET balance = balance - 50000 WHERE id = 'A';
  UPDATE account SET balance = balance + 50000 WHERE id = 'B';
COMMIT;   -- 여기까지 와야 둘 다 확정. 중간에 죽으면 자동 ROLLBACK.
```
**DB는 이걸 어떻게 구현하나?** 커밋 전 변경은 임시 상태이고, 실패 시 **되돌리기 위한 정보(undo)**를 로그에 갖고 있다. `ROLLBACK`(명시적)이나 크래시(암묵적) 시 이 정보로 원상복구한다.

---

## 3.3 Consistency (일관성) — 제약을 어기는 커밋 거부

**일관성**은 "트랜잭션이 끝난 뒤에도 DB의 **모든 규칙(제약조건)**이 참"을 보장한다. 규칙 = `NOT NULL`, `UNIQUE`, `FOREIGN KEY`, `CHECK` 등.
```sql
CREATE TABLE account (
    id TEXT PRIMARY KEY,
    balance INT NOT NULL CHECK (balance >= 0)   -- 규칙: 잔액은 0 이상
);

BEGIN;
  UPDATE account SET balance = balance - 200000 WHERE id = 'A';  -- 잔액 부족!
  -- balance가 음수가 되어 CHECK 위반 → ERROR
COMMIT;   -- 트랜잭션 전체가 롤백됨
```
잔액이 음수가 되는 순간 DB가 **거부**하고, 그 트랜잭션 전체가 무효가 된다. 이 덕분에 "출금은 됐는데 CHECK 때문에 입금 트랜잭션만 실패한" 어중간한 상태가 안 생긴다(원자성과 협력). **일관성은 개발자가 정의한 비즈니스 규칙을 DB가 마지막 방어선으로 지켜주는 것**이다.

---

## 3.4 Isolation (격리성) — 동시에 돌 때의 간섭

여러 트랜잭션이 **동시에** 돌면 서로 간섭할 수 있다. 격리성은 "각 트랜잭션이 마치 **혼자 도는 것처럼**" 보이게 한다. 하지만 완벽한 격리는 느리므로, 실무는 **격리 수준**으로 강도를 조절한다. → 이게 워낙 중요해 **4장 전체**로 다룬다.

---

## 3.5 Durability (지속성) — 커밋 후 정전

`COMMIT`이 성공을 응답한 순간, 그 데이터는 **디스크에 안전**해야 한다. 그 직후 정전이 나도 되살아나야 한다. 어떻게?

**WAL (Write-Ahead Logging, 선행 기록 로그):** DB는 실제 데이터 파일을 고치기 **전에**, "무엇을 바꿀 것인지"를 먼저 **로그 파일에 기록하고 디스크에 강제로 flush**한다. 커밋은 "이 로그가 디스크에 남았다"가 확인되면 성공으로 친다.
```
COMMIT 요청
   │
   ▼
[1] 변경 내용을 WAL 로그에 기록 → 디스크에 fsync (여기까지 되면 안전 확정)
   │
   ▼
[2] COMMIT 성공 응답
   │
   ▼  (실제 데이터 파일 반영은 나중에 천천히 — 이미 로그에 있으니 안전)
[3] 정전이 나도, 재시작 시 WAL 로그를 다시 읽어(replay) 복구
```
**왜 로그를 먼저?** 커다란 데이터 파일 여기저기를 랜덤하게 고쳐 디스크에 쓰는 건 느리다. 반면 로그는 **순차적으로 뒤에 이어 쓰기**라 빠르다. "일단 빠른 로그에 안전하게 적어두고, 실제 반영은 여유 있을 때"가 WAL의 아이디어다.

> **JPA 연결:** `@Transactional` 메서드는 시작 시 `BEGIN`, 정상 종료 시 `COMMIT`, 예외 발생 시 `ROLLBACK`을 프록시가 자동 호출한다(Spring 개념서 2.6/4.5절). 즉 이 장의 `BEGIN/COMMIT/ROLLBACK`이 그 애노테이션 뒤에서 실제로 일어나는 일이다.

# Chapter 4: 동시성 제어 (Concurrency Control)

> Spring 개념서 4.6절은 격리 수준과 이상 현상을 매트릭스로 요약하고 MVCC를 짧게 소개했다. 이 장은 그 매트릭스가 **왜** 그렇게 생겼는지, 락(lock)과 MVCC라는 **두 가지 구현 메커니즘**으로 파고든다.

실습: `chapter-cs-database/labs/03_isolation_levels.sql` (두 세션 필요)

---

## 4.1 세 가지 이상 현상 (Anomalies)

격리가 약하면 동시 트랜잭션 사이에 세 가지 이상 현상이 생긴다. **시간 순서대로** 그려 보자.

### Dirty Read (더티 리드): 커밋 안 된 값을 읽음
```
시간   세션 A                          세션 B
──────────────────────────────────────────────────────────
 t1    BEGIN
 t2    UPDATE balance=999 (커밋 안 함)
 t3                                    SELECT balance  → 999 를 읽음 (!)
 t4    ROLLBACK  (999는 없던 일이 됨)
                                       B는 존재한 적 없는 값을 읽었다 = 재앙
```

### Non-repeatable Read (반복 불가능한 읽기): 두 번 읽으니 값이 다름
```
시간   세션 A                          세션 B
──────────────────────────────────────────────────────────
 t1    BEGIN
 t2    SELECT balance  → 100  (1번째)
 t3                                    UPDATE balance=55; COMMIT
 t4    SELECT balance  → 55   (2번째, 값이 바뀜!)
       같은 트랜잭션에서 같은 행이 100→55. 계산이 어긋난다.
```

### Phantom Read (팬텀 리드): 같은 조건인데 행 개수가 다름
```
시간   세션 A                          세션 B
──────────────────────────────────────────────────────────
 t1    BEGIN
 t2    SELECT count(*) WHERE age>20 → 5 (1번째)
 t3                                    INSERT 새 행 (age=30); COMMIT
 t4    SELECT count(*) WHERE age>20 → 6 (2번째, 유령 행 등장!)
```
(Non-repeatable는 **기존 행의 값**이 바뀌는 것, Phantom은 **행의 집합(개수)**이 바뀌는 것. 미묘하지만 다르다.)

---

## 4.2 격리 수준 매트릭스

격리 수준을 올릴수록 이상 현상을 더 막지만, 동시성(속도)은 떨어진다.
```
격리 수준             │ Dirty Read │ Non-repeatable │ Phantom  │ 특징
─────────────────────┼────────────┼────────────────┼──────────┼──────────────
READ UNCOMMITTED     │   O 발생    │   O 발생        │  O 발생   │ 가장 약함/빠름
READ COMMITTED       │   X 차단    │   O 발생        │  O 발생   │ ★PostgreSQL 기본
REPEATABLE READ      │   X 차단    │   X 차단        │  O* 발생  │ MySQL(InnoDB) 기본
SERIALIZABLE         │   X 차단    │   X 차단        │  X 차단   │ 가장 강함/느림
```
`*` PostgreSQL의 REPEATABLE READ는 MVCC 스냅샷 덕분에 **팬텀까지 대부분 차단**한다 (표준보다 강함).

**PostgreSQL 기본 = READ COMMITTED.** 즉 "커밋된 값만 읽는다(더티 리드 차단)"까지는 공짜지만, 그 이상은 개발자가 필요할 때 올려야 한다. 이 프로젝트도 별도 설정이 없으니 READ COMMITTED로 돈다.
```sql
-- 트랜잭션 단위로 격리 수준 지정
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- ...
COMMIT;
```
> 실습 `03_isolation_levels.sql`에서 두 psql 세션으로 "READ COMMITTED에서 더티 리드가 **안** 생기는 것"과 "반복 불가능한 읽기가 생기다가 REPEATABLE READ로 올리면 사라지는 것"을 직접 관찰한다.

---

## 4.3 락 기반 동시성 제어 (Locking)

격리를 구현하는 **첫 번째 방법**: 데이터에 **자물쇠(lock)**를 걸어 다른 트랜잭션을 막는다.

**두 종류의 락:**
```
공유 락 (Shared, S-lock)   : "읽기 락". 여러 트랜잭션이 동시에 걸 수 있다.
                             내가 읽는 동안 남이 못 바꾸게 한다. (읽기끼리는 공존)
배타 락 (Exclusive, X-lock) : "쓰기 락". 나 혼자만. 남의 어떤 락도 못 걸게 한다.
```
```
락 호환성 표 (걸려 있는 것 vs 걸려는 것)
              걸려는 S락   걸려는 X락
   기존 S락 │   허용 O      대기 X
   기존 X락 │   대기 X      대기 X
```
읽기끼리는 서로 안 막지만, 쓰기가 끼면 무조건 줄을 서야 한다.

### 2단계 잠금 (2-Phase Locking, 2PL)
"언제 락을 걸고 풀 것인가"의 규칙. **직렬 가능성(serializability)**을 보장하는 고전 프로토콜이다.
```
        락 개수
          │      ┌──── 확장 단계 ────┐┌──── 수축 단계 ────┐
          │     /                    \
          │    / (락을 걸기만 함)       \ (락을 풀기만 함)
          │   /                        \
          └──┴──────────────────────────┴────────────► 시간
             한번 락을 풀기 시작하면, 다시는 새 락을 걸지 않는다.
```
핵심 규칙: **모든 락을 다 건 뒤에야(확장 단계 끝) 풀기 시작(수축 단계)한다.** 이 규칙을 지키면 여러 트랜잭션이 동시에 돌아도 결과가 "하나씩 순서대로 돈 것"과 같아진다.

### 교착 상태 (Deadlock)
락의 부작용. 두 트랜잭션이 서로가 가진 락을 기다리며 **영원히 멈춘다.**
```
세션 A: X락(계좌1) 보유  →  계좌2의 락을 기다림
세션 B: X락(계좌2) 보유  →  계좌1의 락을 기다림
       A는 B를, B는 A를 기다림 → 둘 다 영원히 멈춤 (deadlock)
```
PostgreSQL은 교착을 **자동 감지**해서 한쪽 트랜잭션을 강제 실패시켜(`deadlock detected` 에러) 푼다. 예방책: **모든 트랜잭션이 자원을 같은 순서로 잠그기**(예: 항상 작은 id부터). → 위 예에서 둘 다 "계좌1 먼저"로 잠그면 교착이 안 생긴다.

---

## 4.4 MVCC — PostgreSQL이 읽기 락을 없앤 방법

락 방식의 문제: **읽기가 쓰기를 막고, 쓰기가 읽기를 막는다.** 조회가 많은 시스템에서 이건 치명적이다.

**MVCC(Multi-Version Concurrency Control, 다중 버전 동시성 제어)**의 아이디어: **데이터를 덮어쓰지 말고, 새 버전을 추가하라.** 각 트랜잭션은 자기가 시작한 시점의 **스냅샷(버전)**을 본다.
```
account id='A' 의 시간에 따른 버전들:

  버전1: balance=100 (트랜잭션 10이 만듦, 트랜잭션 20이 지움)
  버전2: balance=55  (트랜잭션 20이 만듦, 아직 유효)

  트랜잭션 15가 읽으면? → 자기(15) 시작 시점 기준 유효한 "버전1(100)"을 본다.
  트랜잭션 25가 읽으면? → "버전2(55)"를 본다.
  ★ 둘 다 락을 기다리지 않는다. 각자 자기 버전을 읽을 뿐. ★
```
**결과: "읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다."** SELECT는 (거의) 락 없이 돌아 동시성이 극대화된다. 쓰기끼리(같은 행 UPDATE)만 여전히 락으로 조정한다.

**대가:** 옛 버전이 쌓인다. PostgreSQL은 **VACUUM**이라는 청소 작업으로 아무도 안 보는 죽은 버전(dead tuple)을 주기적으로 회수한다. → MVCC의 편안함은 "버전 쓰레기를 나중에 치우는" 비용과 맞바꾼 것.

이제 4.2절 매트릭스의 `*`가 이해된다: PostgreSQL의 REPEATABLE READ는 **트랜잭션 시작 시점의 스냅샷을 끝까지 고정**해서 보므로, 남이 새 행을 INSERT/커밋해도 그 스냅샷엔 안 보인다 → 팬텀까지 막힌다.

---

## 4.5 낙관적 락 vs 비관적 락

동시 수정 충돌을 다루는 두 철학. (Spring/JPA에서 직접 고르는 부분이라 중요하다.)

**비관적 락 (Pessimistic Locking):** "충돌이 **날 것**이라 가정하고, 미리 락을 건다."
```sql
SELECT * FROM account WHERE id = 'A' FOR UPDATE;  -- 이 행에 X락. 남은 못 건드림.
-- 계산...
UPDATE account SET balance = ... WHERE id = 'A';
```
읽는 순간부터 락으로 막으니 충돌이 원천 봉쇄된다. 대신 락 대기로 동시성이 떨어진다. 충돌이 잦은 곳(재고 차감, 좌석 예매)에 적합.

**낙관적 락 (Optimistic Locking):** "충돌은 **드물 것**이라 가정하고, 락 없이 진행하다 커밋 시 검사한다." **버전 컬럼**으로 구현한다.
```
1. 읽을 때 version=3 을 함께 읽는다.
2. 락 없이 자유롭게 작업.
3. 저장할 때: UPDATE ... SET version=4 WHERE id='A' AND version=3
   → 그 사이 남이 먼저 저장했다면 version이 이미 4라 WHERE가 0행 매칭 → 실패 감지!
   → 실패하면 재시도하거나 에러.
```
락을 안 걸어 빠르지만, 충돌 시 재시도 비용이 든다. 충돌이 드문 곳(게시글 수정 등)에 적합.
```
             비관적 락                     낙관적 락
가정         "충돌 자주 남"                 "충돌 드묾"
방법         미리 X락(FOR UPDATE)           버전 컬럼으로 커밋 시 검사
동시성       낮음 (대기 발생)               높음 (락 없음)
충돌 시      기다렸다 진행                  실패 → 재시도
적합         재고/예매 등 경쟁 심한 곳       일반 수정 등 대부분
```

> **JPA 연결:** JPA는 이 둘을 애노테이션으로 제공한다.
> - 낙관적 락: Entity에 `@Version private Long version;` 필드 하나만 추가하면, JPA가 위 "version WHERE 검사"를 자동으로 넣고, 충돌 시 `OptimisticLockException`을 던진다.
> - 비관적 락: `@Lock(LockModeType.PESSIMISTIC_WRITE)`를 Repository 메서드에 붙이면 `SELECT ... FOR UPDATE`가 나간다.
>
> 즉 이 절의 CS 이론이 JPA에서 애노테이션 한 줄로 노출되는 것이다.

# Chapter 5: 인덱싱 (Indexing)

> Spring 개념서 4.7절은 "`WHERE email=?`가 왜 빠른가 = 인덱스(B-tree), O(log n)"를 소개했다. 이 장은 B+tree 구조, 복합/커버링 인덱스, 그리고 `EXPLAIN ANALYZE`로 그걸 **눈으로 확인**하는 데까지 깊이 들어간다.

실습: `chapter-cs-database/labs/04_indexing_explain.sql`

---

## 5.1 왜 전체 스캔은 O(n)이고 인덱스는 O(log n)인가

인덱스가 없으면 `WHERE email='kim@x.com'`은 테이블을 **처음부터 끝까지** 한 행씩 비교한다. 이게 **전체 테이블 스캔(Sequential Scan, Seq Scan)**이고 **O(n)**이다. 100만 행이면 최악 100만 번 비교.
```
Seq Scan:  [행1][행2][행3]...[행999999][행1000000]  ← 다 훑어야 함 = O(n)
```
**인덱스**는 책 뒤의 "찾아보기"다. "트랜잭션 → 412p"처럼, 값을 **정렬된 색인**으로 미리 만들어 둔다. 정렬돼 있으니 **이진 탐색**류로 후보를 절반씩 줄여 O(log n)에 도달한다.
```
100만 행:  Seq Scan = 최대 1,000,000 번 비교
           Index    = log₂(1,000,000) ≈ 20 번 비교      → 약 5만 배 차이
```

---

## 5.2 B-tree / B+tree 구조

DB 인덱스의 표준 자료구조는 **B-tree**, 정확히는 **B+tree**다. "정렬된 값을 계층적으로 저장한 **균형 트리**"다.

```
                        ┌──────────────┐
   루트(root)  ─────────│   [ 30 | 70 ] │      값을 범위로 분기
                        └───┬────┬────┬─┘
                 ┌──────────┘    │    └──────────┐
          ┌──────▼─────┐  ┌──────▼─────┐  ┌──────▼─────┐
  내부노드 │ [10 | 20]  │  │ [40 | 55]  │  │ [80 | 90]  │   더 좁은 범위
          └──┬──┬───┬──┘  └──┬──┬───┬──┘  └──┬──┬───┬──┘
             ▼  ▼   ▼        ▼  ▼   ▼        ▼  ▼   ▼
  리프노드 [실제 값 + 행 위치 포인터, 정렬된 상태로 저장]
          [5][10] → [15][20] → [25][30] → [40][55] → ...  (리프끼리 연결됨)
```

**B+tree의 두 가지 핵심 성질:**
1. **모든 실제 데이터(포인터)는 리프 노드에만 있다.** 내부 노드는 "길 안내판(범위)"일 뿐. → 내부 노드에 더 많은 키를 담을 수 있어 트리가 **낮아지고**(디스크 접근 횟수 감소), 검색이 항상 리프까지 같은 깊이로 내려간다.
2. **리프 노드끼리 링크드 리스트로 연결**돼 있다. → `WHERE age BETWEEN 20 AND 30` 같은 **범위 검색**이 빠르다. 시작점을 O(log n)에 찾고, 그다음은 리프를 옆으로 쭉 따라가면 된다.

**왜 이진 트리가 아니라 B+tree인가?** 이진 트리는 노드마다 자식이 2개뿐이라 100만 건이면 높이가 약 20. 그런데 DB는 디스크에서 읽으므로 **높이 = 디스크 접근 횟수**가 곧 비용이다. B+tree는 노드 하나(=디스크 블록 하나)에 키를 **수백 개** 담아 자식이 수백 갈래다. → 100만 건도 높이 **3~4**면 끝. 디스크를 3~4번만 읽어 원하는 행에 도달한다. **B+tree는 "디스크 친화적"으로 설계된 트리**다.

---

## 5.3 복합 인덱스와 왼쪽 접두사(Leftmost-Prefix) 규칙

여러 컬럼을 묶어 만드는 인덱스가 **복합 인덱스**다.
```sql
CREATE INDEX idx_country_age ON users (country, age);  -- (country, age) 순서로 정렬
```
이 인덱스는 **`(country, age)` 사전 순으로 정렬**된다. 마치 전화번호부가 `(성, 이름)` 순인 것과 같다.
```
정렬 상태:  (DE, 20) (DE, 41) (KR, 25) (KR, 30) (KR, 55) (US, 33) ...
            └── country로 먼저 정렬, 같은 country 안에서 age로 정렬 ──┘
```
**왼쪽 접두사 규칙:** 이 인덱스는 **왼쪽 컬럼부터 순서대로** 조건이 주어져야 쓸 수 있다.
```
WHERE country='KR' AND age=30   → O 사용 (country, age 둘 다 왼쪽부터)
WHERE country='KR'              → O 사용 (왼쪽 접두사 country만으로도 OK)
WHERE age=30                    → X 못 씀! (앞의 country를 건너뜀)
```
**왜 `age=30`만으로는 못 쓰나?** 전화번호부에서 "성은 모르고 이름이 '철수'인 사람"을 찾는 것과 같다. 이름은 성 안에서만 정렬돼 있으므로 '철수'가 책 전체에 흩어져 있다 → 인덱스가 무용지물. **교훈: 복합 인덱스는 "자주 함께 쓰는 조건의 앞 컬럼 순서"를 신중히 정해야 한다.**

---

## 5.4 커버링 인덱스 (Covering Index)

보통 인덱스로 위치를 찾은 뒤, 실제 행 데이터를 얻으려면 **테이블 본체(heap)를 한 번 더** 읽어야 한다.
```
1. 인덱스에서 email='kim' 위치 찾기
2. 그 포인터로 테이블 본체 가서 나머지 컬럼(name, country...) 읽기  ← 추가 접근
```
그런데 **필요한 컬럼이 전부 인덱스 안에 들어 있으면**, 2단계를 건너뛴다. 이걸 **커버링 인덱스**(그리고 **Index Only Scan**)라 한다.
```sql
CREATE INDEX idx ON users (email) INCLUDE (country);
-- SELECT email, country FROM users WHERE email='kim@x.com'
-- → email로 찾고, country도 인덱스에 있으니 테이블 본체를 안 읽어도 됨 = 더 빠름
```

---

## 5.5 인덱스의 대가: 언제 인덱스를 걸지 말아야 하나

인덱스는 **정렬된 색인**이므로, 행을 INSERT/UPDATE/DELETE할 때마다 **색인도 함께 갱신**해야 한다. 즉:
```
인덱스의 트레이드오프:
  얻는 것:  읽기(SELECT) 속도 O(n) → O(log n)
  잃는 것:  쓰기(INSERT/UPDATE/DELETE) 속도 저하 + 디스크 공간 추가
```
그래서 **"모든 컬럼에 인덱스"는 나쁘다.** 인덱스를 걸면 **안 좋은** 경우:
- **값의 종류가 거의 없는 컬럼** (예: 성별 M/F, 참/거짓): 인덱스로 좁혀도 절반이 남아 이득이 적다.
- **쓰기가 매우 빈번한 컬럼**: 갱신 비용이 조회 이득을 초과.
- **작은 테이블**: 어차피 Seq Scan이 빠르다(옵티마이저가 인덱스를 무시하기도).
- **거의 조회 안 하는 컬럼**: 공간·쓰기 비용만 낭비.

**원칙: `WHERE` / `JOIN ON` / `ORDER BY`에 자주 쓰는 컬럼에만, `EXPLAIN`으로 효과를 확인하며 건다.**

---

## 5.6 EXPLAIN (ANALYZE) 읽는 법

`EXPLAIN`은 옵티마이저가 세운 **실행 계획**을 보여준다. `ANALYZE`를 붙이면 **실제로 실행하고 걸린 시간**까지 측정한다.
```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM big_user WHERE email = 'user777777@example.com';
```
**인덱스 없을 때:**
```
Seq Scan on big_user  (cost=0.00..18334.00 rows=1 width=...) (actual time=45.2..201.7 rows=1 loops=1)
  Filter: (email = 'user777777@example.com')
  Rows Removed by Filter: 999999          ← 99만9999개를 헛되이 읽고 버림!
Execution Time: 201.9 ms
```
**인덱스 만든 후:**
```
Index Scan using idx_big_user_email on big_user  (cost=0.42..8.44 rows=1 ...) (actual time=0.03..0.04 rows=1 loops=1)
  Index Cond: (email = 'user777777@example.com')
Execution Time: 0.08 ms                    ← 약 2500배 빨라짐
```
**읽는 법 정리:**
```
Seq Scan            = 테이블 전체 훑기 (인덱스 못/안 씀). 큰 테이블에서 보이면 경고 신호.
Index Scan          = 인덱스로 바로 찾아감. 우리가 원하는 것.
Index Only Scan     = 커버링 인덱스로 테이블 본체도 안 봄. 가장 빠름.
cost=시작..총합       = 옵티마이저의 추정 비용 (상대값, 작을수록 좋음)
actual time=...      = ANALYZE가 잰 실제 시간(ms)
rows=...             = 처리한 행 수 (추정 rows vs actual rows가 크게 다르면 통계가 낡은 것 → ANALYZE)
Rows Removed by Filter = 읽었다가 버린 행 수. 크면 인덱스가 필요하다는 신호.
Execution Time       = 실제 총 실행 시간
```

# Chapter 6: 쿼리 처리 기초 (Query Processing)

> "SQL은 '무엇을 원하는지'만 쓴다(선언형). 그러면 DB는 '어떻게 가져올지'를 스스로 정한다. 그 '어떻게'를 만드는 것이 쿼리 처리다."

---

## 6.1 논리적 계획 → 물리적 계획

SQL 한 줄이 실행되기까지의 파이프라인:
```
SQL 텍스트
   │  파서(Parser): 문법 검사 → 파스 트리
   ▼
논리적 계획 (Logical Plan): "무엇을" — 관계 대수로 표현
   │  예: users ⋈ posts 를 필터링하고 특정 컬럼을 뽑아라
   │  옵티마이저(Optimizer)가 개입: 같은 결과를 내는 여러 방법 중 비용 최소 선택
   ▼
물리적 계획 (Physical Plan): "어떻게" — 실제 실행 방법
   │  예: posts를 Index Scan → users와 Hash Join → 정렬
   ▼
실행 (Executor): 물리 계획대로 데이터를 실제로 읽고 가공
```
**옵티마이저(optimizer)**가 핵심이다. "이 조인을 어떤 순서로, 어떤 알고리즘으로, 인덱스는 쓸지"를 **통계정보(테이블 크기, 값 분포)**를 바탕으로 **비용을 추정**해 가장 싼 계획을 고른다. `EXPLAIN`이 보여주는 것이 바로 이 물리적 계획이다.

> `EXPLAIN`에서 추정 `rows`가 실제와 크게 다르면, 통계가 낡은 것이다. `ANALYZE 테이블명;`으로 통계를 갱신하면 계획이 개선된다(실습 04에서 대량 INSERT 후 `ANALYZE`를 부른 이유).

---

## 6.2 조인 알고리즘 3종

두 테이블을 합치는 `JOIN`을 실제로 수행하는 방법은 셋이다. 옵티마이저가 상황에 맞게 고른다.

### Nested Loop Join (중첩 루프 조인)
```
for 각 행 r in 테이블A:            ← 바깥 루프
    for 각 행 s in 테이블B:        ← 안쪽 루프
        if r.key == s.key: 결과에 추가
```
가장 단순. A의 각 행마다 B를 뒤진다. B에 **인덱스가 있으면** 안쪽 루프가 O(log n)이 되어 매우 빠르다. → **한쪽이 작고 다른 쪽에 인덱스가 있을 때** 최적. (JPA의 N+1 문제가 사실상 애플리케이션 레벨의 나쁜 nested loop다.)

### Hash Join (해시 조인)
```
1. 작은 테이블B로 해시 테이블을 만든다: key → 행  (메모리에)
2. 큰 테이블A를 한 번 훑으며, 각 행의 key로 해시 테이블을 O(1) 조회해 매칭
```
큰 테이블 두 개를 **인덱스 없이** 등가 조인(`=`)할 때 강력하다. 해시를 만드는 초기 비용은 있지만 그 뒤가 빠르다. → **대용량 + 등가 조인 + 인덱스 없음**에 최적.

### Merge Join (병합 조인)
```
두 테이블을 각각 조인 키로 '정렬'한 뒤, 두 정렬 목록을 지퍼처럼 나란히 훑으며 병합
A: [10][20][30]      →  포인터를 양쪽에서 앞으로 밀며 같은 값 매칭
B: [20][30][40]
```
양쪽이 **이미 정렬돼 있거나**(예: 인덱스 순서대로 읽음) 정렬 비용이 쌀 때 최적. 범위 조인이나 정렬이 어차피 필요한 쿼리에 유리.
```
알고리즘        최적 상황                                    비용 감각
─────────────────────────────────────────────────────────────────────
Nested Loop    한쪽이 작음 + 다른 쪽에 인덱스               O(A) × 인덱스조회
Hash Join      큰 두 테이블 + 등가(=) 조인 + 인덱스 없음     O(A + B) (+해시 메모리)
Merge Join     양쪽이 정렬돼 있음 / 정렬이 값쌈             O(A + B) (+정렬 비용)
```
`EXPLAIN`에서 이 이름들("Nested Loop", "Hash Join", "Merge Join")을 직접 보게 된다. 어떤 게 나오는지로 옵티마이저의 판단을 읽을 수 있다.

# Chapter 7: JPA로 돌아오기 — 이론과 ORM의 연결

> 이제 이 책에서 배운 DB 이론과 Spring 개념서의 JPA를 **한 장의 지도**로 잇는다. "JPA 애노테이션 한 줄이 이 책의 어떤 이론을 부르는가"를 표로 정리한다.

---

## 7.1 애노테이션 ↔ DB 이론 매핑

```
JPA / Spring                        이 책의 이론                     생성되는 것
──────────────────────────────────────────────────────────────────────────────────
@Entity, @Table                     관계(Relation) = 테이블 (1장)     CREATE TABLE
@Id                                 기본키 PK (1.3절)                 PRIMARY KEY
@GeneratedValue                     대리키(surrogate key) (1.3절)     BIGSERIAL / SEQUENCE
@Column(unique=true)                후보키의 UNIQUE 제약 (1.3절)      UNIQUE + 자동 인덱스 (5장)
@Column(nullable=false)             일관성 제약 NOT NULL (3.3절)      NOT NULL
@ManyToOne @JoinColumn              외래키 FK, 참조 무결성 (1.3절)    FOREIGN KEY REFERENCES
@Table(indexes=@Index(...))         명시적 인덱스 (5장)               CREATE INDEX
@Transactional                      트랜잭션 + ACID (3장)             BEGIN/COMMIT/ROLLBACK
@Transactional(isolation=...)       격리 수준 (4.2절)                SET TRANSACTION ISOLATION
@Version                            낙관적 락 (4.5절)                UPDATE ... WHERE version=?
@Lock(PESSIMISTIC_WRITE)            비관적 락 (4.5절)                SELECT ... FOR UPDATE
```

**즉 JPA는 이 책의 이론을 "애노테이션이라는 표면"으로 노출한 것이다.** `@Column(unique=true)`를 붙이는 순간, 1장의 후보키 이론과 5장의 인덱스 이론이 동시에 작동한다.

---

## 7.2 왜 JPA를 쓰면서도 DB 이론을 알아야 하는가

JPA는 DB를 **가려주지만, 완전히 숨기지는 못한다.** 이 책의 이론을 모르면 다음을 영영 이해할 수 없다:

1. **N+1 문제 (Spring 4.4절)** — 정규화(2장)로 나눈 테이블을 ORM이 자동 JOIN하려다, 나쁜 nested loop(6.2절)를 애플리케이션 레벨에서 반복하는 것. 이론을 알아야 "왜 fetch join으로 hash/merge 한 방에 끝내야 하는지"가 보인다.

2. **느린 쿼리** — `findByEmail`이 느리다면? 5장의 인덱스가 없거나 못 타는 것이다. `EXPLAIN`(5.6절)을 읽을 줄 알아야 원인을 찾는다.

3. **데이터 꼬임/동시성 버그** — 두 요청이 같은 재고를 동시에 깎아 음수가 됐다면? 4장의 격리 수준과 락을 알아야 `@Version`이나 `@Lock`으로 고칠 수 있다.

4. **이상한 스키마** — 처음엔 잘 돌던 Entity가 데이터가 쌓이며 중복·모순으로 썩는다면? 2장의 정규화를 안 한 것이다.

> **결론:** JPA는 "SQL을 안 써도 되게" 해주지만, "DB를 몰라도 되게" 해주지는 **않는다.** 이 책의 관계 모델·정규화·트랜잭션·동시성·인덱스·쿼리처리 여섯 기둥이, JPA라는 편리한 표면 아래에서 언제나 돌고 있다. 전공자 수준이란, 그 표면을 걷어내고 아래를 볼 수 있다는 뜻이다.
