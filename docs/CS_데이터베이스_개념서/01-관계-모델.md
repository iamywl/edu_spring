# 1장: 관계 모델 (The Relational Model)

> "우리가 매일 쓰는 '테이블'은 사실 1970년 에드거 커드(Edgar F. Codd)가 만든 **수학 이론**의 산물이다. 테이블이 왜 하필 '행과 열'의 모양인지, 거기엔 이유가 있다."

---

## 도입: 왜 이 장이 필요한가

JPA를 쓰다 보면 `@Entity` 클래스를 만들고, 필드를 선언하고, `@ManyToOne`으로 연결하고, 스프링이 알아서 테이블을 만들어준다. 편리하다. 그런데 다음 질문들에 답할 수 있는가?

- `@Id`는 왜 반드시 있어야 하나?
- 왜 연관관계는 FK(외래 키)로 표현되나?
- 왜 `@Column(unique=true)`와 `@Id`는 다른 의미인가?
- JPA가 생성한 SQL에서 왜 `NULL = NULL`이 `FALSE`가 나오나?
- 어떤 조인을 써도 결과가 같을 것이라는 믿음은 왜 맞는가?

이 질문들의 답은 **관계 모델(relational model)**에 있다. 관계 모델은 테이블이라는 개념의 수학적 토대다. 이 장은 그 토대를 쌓는다: 왜 관계형인지, 테이블의 구성 요소들이 정확히 무엇인지, 키와 무결성 제약이 어떻게 돌아가는지, 그리고 SQL의 바닥에 흐르는 **관계 대수**가 무엇인지.

이 장을 읽고 나면 `CREATE TABLE`의 모든 줄이 이전과 달리 보일 것이다.

## 1.1 왜 관계형인가? — 대안들과의 비교

관계형 DB가 40년 넘게 왕좌를 지키는 이유를 알려면 **관계형이 없던 시절**로 돌아가야 한다.

### 대안 1: 플랫 파일 (Flat File)

```
users.txt:
1,김철수,kim@x.com,컴퓨터공학
2,이영희,lee@x.com,컴퓨터공학

posts.txt:
101,김철수가 쓴 글,1
102,이영희가 쓴 글,2
```

"컴퓨터공학과 학생이 쓴 글 목록"을 얻으려면 프로그램이 두 파일을 전부 읽어 직접 매칭해야 한다. 두 프로세스가 동시에 쓰면 파일이 깨진다. 수천만 행이 되면 전체 스캔만으로 수십 초가 걸린다. 인덱스도 없고, 트랜잭션도 없고, 권한 관리도 없다.

### 대안 2: 계층형 / 네트워크형 DB (1960~70년대)

IBM IMS(1966), CODASYL(1969) 계열이 대표적이다. 데이터를 **부모-자식 트리** 혹은 **미리 정의된 포인터 그래프**로 저장했다.

```
계층형 구조 예:
학과
 └── 학생
      └── 수강
```

"컴퓨터공학과 → CS101 수강생"을 찾으려면 학과 노드에서 출발해 학생으로 내려가고, 다시 수강으로 내려가는 **미리 정해진 경로**를 따라가야 한다. 그런데 "CS101 강좌를 듣는 모든 학생의 학과"처럼 **경로를 거꾸로 타거나, 여러 경로를 동시에 탐색해야 하는 질의는 지옥**이었다. 새 접근 패턴이 생길 때마다 스키마(포인터 구조)를 재설계해야 했다.

### 관계형의 답: 선언적 질의

에드거 커드(Edgar F. Codd)는 1970년 논문 "A Relational Model of Data for Large Shared Data Banks"에서 혁명적인 아이디어를 내놓았다.

1. 데이터를 **테이블(관계, Relation)**로만 표현한다.
2. 테이블끼리는 **포인터가 아닌 값(키)**으로 연결한다.
3. 사용자는 **"무엇을 원하는지"만 선언**하고, **"어떻게 찾을지"는 DB가 결정**한다.

```
계층형/네트워크형:
  "학과 노드에서 출발 → 학생 노드로 이동 → 수강 노드로 내려가라"
  (경로를 프로그래머가 명시)

관계형:
  SELECT s.student_name FROM student s
  JOIN enrollment e ON e.student_id = s.student_id
  WHERE e.course_id = 'CS101';
  (원하는 것만 선언, 어떻게 찾을지는 DB 옵티마이저가 결정)
```

이 **선언적 성질(declarative nature)**이 관계형의 혁명이었다. 프로그래머는 데이터의 **논리적 구조**만 알면 된다. 물리적으로 어떻게 저장되고, 어떤 인덱스를 어떤 순서로 사용할지(5장, 6장)는 DB가 최적화한다.

> **NoSQL은 관계형의 대체가 아니다.** 2000년대 MongoDB(문서형), Redis(키-값), Cassandra(와이드 컬럼), Neo4j(그래프)는 "대규모 수평 확장"과 "유연한 스키마"를 위해 관계형의 일부(강한 일관성, JOIN, 스키마 강제)를 포기한 것들이다. 이것은 관계형의 **대체**가 아니라, 특정 상황에서의 **다른 트레이드오프 선택**이다. 정합성이 핵심인 금융, 커머스, 이 프로젝트의 게시판 같은 업무 시스템은 여전히 관계형이 정답이다.

## 1.2 용어: 관계 / 튜플 / 애트리뷰트

관계 모델의 이론 용어와 실무 용어를 나란히 정리하자. 이 매핑을 모르면 DB 논문이나 면접에서 말이 안 통한다.

```
이론 용어                    실무 용어              예
────────────────────────────────────────────────────────────────
Relation (관계)          =   Table (테이블)         users 테이블
Tuple (튜플)             =   Row / Record (행)      {1, "김철수", "kim@x.com"}
Attribute (애트리뷰트)   =   Column / Field (컬럼)  name, email
Domain (도메인)          =   타입 + 제약            name: 50자 이하 문자열
Degree (차수)            =   컬럼의 개수            users의 차수 = 3
Cardinality (카디널리티) =   행의 개수              users의 카디널리티 = 2
Schema (스키마)          =   테이블 정의            users(id, name, email)
Instance (인스턴스)      =   현재 저장된 데이터     현재 테이블에 들어있는 행들
```

```
                  users 테이블 (Relation, Schema: users(id, name, email))
                  ┌─────────┬──────────────┬────────────────────┐
  Attribute  →   │  id     │  name        │  email             │  ← Degree = 3
                  ├─────────┼──────────────┼────────────────────┤
  Tuple →        │  1      │  김철수       │  kim@x.com         │
  Tuple →        │  2      │  이영희       │  lee@x.com         │  ← Cardinality = 2
                  └─────────┴──────────────┴────────────────────┘
                    ↑                             ↑
                   INT 타입                   TEXT 타입 + 유일(UNIQUE)
                   (도메인)                       (도메인)
```

### 관계 모델의 두 가지 수학적 약속

이 두 약속은 관계 모델의 핵심 전제이며, 이하 모든 내용의 기초다.

**약속 1: 튜플에는 순서가 없다.**

테이블은 수학적으로 **집합(set)**이다. 원소(튜플)의 순서는 집합의 의미와 무관하다. `{A, B}` = `{B, A}`. 따라서:
- "3번째 행"이라는 개념이 존재하지 않는다.
- 특정 순서로 결과를 보고 싶다면 **반드시 `ORDER BY`**를 써야 한다.
- `ORDER BY` 없는 SELECT의 반환 순서는 보장되지 않는다. 어제 된 순서가 오늘 안 될 수 있다.

```sql
-- 잘못된 기대: "항상 id 순으로 나오겠지"
SELECT * FROM users;
-- 올바른 방법: 순서를 원한다면 명시적으로
SELECT * FROM users ORDER BY id;
```

**약속 2: 컬럼 값은 원자적(atomic)이다.**

한 칸에는 더 이상 쪼갤 수 없는 값 하나만 들어간다. 한 칸에 "010-1111, 010-2222"처럼 여러 전화번호를 넣거나, 배열을 넣거나, JSON 덩어리를 넣는 것은 관계 모델의 **1NF(제1정규형)**를 위반한다(2장에서 상세히). 이 원자성이 있어야 WHERE 조건이 의미 있고, 인덱스가 작동한다.

> PostgreSQL은 `ARRAY`, `JSON`, `JSONB` 타입을 지원한다. 이것들은 기술적으로 비원자적이다. 이는 실용적 확장이지, 원자성 원칙이 틀렸다는 뜻이 아니다. 비원자 컬럼을 쓸 때는 그 트레이드오프(검색 불편, 정규화 어려움)를 의식해야 한다.

## 1.3 키(Key): 행을 어떻게 구별하는가

관계 모델에서 튜플에는 순서가 없다. 그렇다면 특정 행을 어떻게 **콕 집어** 가리킬 수 있을까? 답이 **키(key)**다.

### 슈퍼키 (Super Key)

행을 유일하게 구별할 수 있는 컬럼 집합이면 **모두** 슈퍼키다. 불필요한 컬럼이 섞여 있어도 유일성만 있으면 슈퍼키다.

```
users(id, email, name)에서 가능한 슈퍼키:
  {id}              ← 유일 (id는 각 행마다 다름)
  {email}           ← 유일 (email도 각 행마다 다름)
  {id, email}       ← 유일 (하지만 email은 군더더기)
  {id, name}        ← 유일 (하지만 name은 군더더기)
  {id, email, name} ← 유일 (하지만 두 컬럼이 군더더기)
  {name}            ← 유일하지 않을 수 있음 (동명이인) → 슈퍼키 아님
```

슈퍼키의 수는 기하급수적으로 늘어날 수 있다. 실제로 중요한 건 다음 개념이다.

### 후보키 (Candidate Key)

슈퍼키 중에서 **더 이상 컬럼을 뺄 수 없는 최소 슈퍼키**. 즉, 어느 컬럼 하나라도 빼면 유일성이 깨지는 슈퍼키.

```
users에서 후보키:
  {id}     ← 최소. 더 뺄 수 없음. 후보키.
  {email}  ← 최소. 더 뺄 수 없음. 후보키.
  (id, email은 email을 빼도 {id}가 유일하므로 후보키가 아니라 슈퍼키)
```

**후보키는 여러 개일 수 있다.** `id`도 되고 `email`도 된다. 둘 다 최소이면서 유일하다.

### 기본키 (Primary Key, PK)

여러 후보키 중 **대표로 지정한 하나**. 다음 두 규칙을 반드시 지켜야 한다:
1. **NULL 불가**: NULL은 "값 없음"이라 식별 불가.
2. **중복 불가**: 두 행이 같은 PK를 가지면 안 됨.

```sql
CREATE TABLE users (
    id    BIGSERIAL    PRIMARY KEY,   -- PK: NOT NULL + UNIQUE 자동 보장
    email TEXT         UNIQUE NOT NULL, -- 나머지 후보키는 UNIQUE로 유일성 유지
    name  TEXT         NOT NULL
);
```

**왜 `email` 대신 자동증가 `id`를 PK로 쓰나?**

이메일 주소는 사람이 **바꿀 수 있다**. 기본키가 바뀌면 그것을 가리키는 FK(외래 키)가 있는 모든 테이블의 모든 행을 연쇄로 업데이트해야 한다. 수백만 행이 있다면 재앙이다. 그래서 실무에서는 **비즈니스 의미가 없는 대리키(surrogate key)**, 즉 자동증가 정수(`BIGSERIAL`, `GENERATED ALWAYS AS IDENTITY`)를 PK로 쓴다. 이메일이나 주민번호처럼 의미 있는 자연키(natural key)는 PK가 아닌 `UNIQUE NOT NULL`로 두어 유일성만 보장한다.

### 외래키 (Foreign Key, FK)

**다른 테이블의 PK(또는 UNIQUE 컬럼)를 가리키는 컬럼.** 테이블 사이의 "관계"를 값으로 표현하는 장치다.

```sql
CREATE TABLE posts (
    id        BIGSERIAL PRIMARY KEY,
    title     TEXT      NOT NULL,
    author_id BIGINT    NOT NULL
        REFERENCES users(id)         -- users.id를 가리키는 FK
        ON DELETE CASCADE            -- 사용자 삭제 시 해당 글도 함께 삭제
);
```

```
      users                              posts
   ┌─────────────────┐               ┌─────────────────────┐
   │ id  (PK) ◄──────┼───────────────┤ author_id  (FK)     │
   │ email           │  "이 게시글의  │ id         (PK)     │
   │ name            │   작성자는 →  │ title               │
   └─────────────────┘   users의     └─────────────────────┘
                         이 행"
```

FK가 지키는 것이 **참조 무결성(referential integrity)**이다. `posts.author_id = 999`인데 `users`에 `id = 999`인 행이 없다면, 그것은 "존재하지 않는 사람이 쓴 글"이다. DB가 이를 원천 거부한다:

```sql
INSERT INTO posts (title, author_id) VALUES ('제목', 999);
-- ERROR:  insert or update on table "posts" violates foreign key constraint
--         "posts_author_id_fkey" on table "posts"
-- DETAIL: Key (author_id)=(999) is not present in table "users".
```

### FK의 ON DELETE / ON UPDATE 옵션

참조하는 쪽(users)에서 행이 삭제되거나 PK가 변경될 때 참조당하는 쪽(posts)에서 어떻게 반응할지를 지정한다.

```
옵션              설명                               언제 쓰나
─────────────────────────────────────────────────────────────────────────────
CASCADE          부모 삭제 시 자식도 삭제            게시글과 댓글: 글 삭제 시 댓글도 삭제
RESTRICT         부모 삭제 불가 (자식 있으면 오류)    사용자 삭제 전 게시글 먼저 지워야
SET NULL         자식의 FK를 NULL로 설정             담당자 삭제 시 태스크의 담당자를 NULL로
SET DEFAULT      자식의 FK를 기본값으로 설정          드묾
NO ACTION        기본값; RESTRICT와 유사             (지연 제약에서 동작 다름)
```

```sql
-- 다양한 ON DELETE 예시 (PostgreSQL 16)
CREATE TABLE comments (
    id        BIGSERIAL PRIMARY KEY,
    post_id   BIGINT REFERENCES posts(id) ON DELETE CASCADE,   -- 글 삭제 → 댓글도 삭제
    author_id BIGINT REFERENCES users(id) ON DELETE SET NULL,  -- 사용자 삭제 → 댓글 작성자 NULL
    content   TEXT NOT NULL
);
```

## 1.4 무결성 제약의 종류

관계 모델은 데이터의 "올바름"을 보장하기 위해 여러 종류의 제약을 정의한다.

### 도메인 무결성 (Domain Integrity)

각 컬럼의 값이 정의된 도메인(타입 + 범위)을 만족해야 한다.

```sql
CREATE TABLE product (
    id       BIGSERIAL   PRIMARY KEY,
    name     TEXT        NOT NULL,
    price    NUMERIC(10,2) NOT NULL CHECK (price >= 0),  -- 음수 가격 불가
    category TEXT        CHECK (category IN ('전자', '의류', '식품')),
    stock    INT         DEFAULT 0 CHECK (stock >= 0)
);
```

`CHECK` 제약이 도메인 무결성의 SQL 표현이다. PostgreSQL은 `CHECK` 위반 시 즉시 오류를 반환한다.

### 개체 무결성 (Entity Integrity)

모든 기본키 컬럼은 NULL이 아니어야 한다. 기본키는 행을 유일하게 식별하는 식별자인데, NULL이 섞이면 "알 수 없는 값"인 행이 생겨 식별이 불가능해진다.

```sql
-- PostgreSQL에서 PRIMARY KEY는 자동으로 NOT NULL + UNIQUE를 적용한다.
-- 아래 두 선언은 동일하다:
id BIGSERIAL PRIMARY KEY
id BIGSERIAL NOT NULL UNIQUE  -- (내부적으로 이렇게 동작)
```

### 참조 무결성 (Referential Integrity)

외래 키 값은 반드시 참조하는 테이블의 PK에 존재하거나 NULL이어야 한다. 앞서 FK에서 다뤘다. "미아 데이터(orphan row)"를 방지한다.

### 무결성 제약 요약

```
무결성 종류         SQL 표현                     지키는 것
─────────────────────────────────────────────────────────────────────────
도메인 무결성       CHECK, NOT NULL, 타입          컬럼 값이 허용 범위 안에 있어야
개체 무결성         PRIMARY KEY (= NOT NULL+UNIQUE) PK는 NULL·중복 불가
참조 무결성         FOREIGN KEY REFERENCES         FK 값은 참조 대상에 존재해야
사용자 정의 무결성  CHECK, 트리거                  비즈니스 규칙 (재고≥0, 종료일≥시작일)
```

## 1.5 관계 대수 (Relational Algebra)

SQL은 어디서 왔을까? 관계 모델은 커드의 논문에서 **관계 대수(relational algebra)**도 함께 정의했다. SQL은 이 관계 대수를 인간 친화적으로 표현한 언어다. 관계 대수를 알면 SQL이 왜 그렇게 생겼는지, 어떤 질의가 어떤 대수 연산으로 구성되는지 이해할 수 있다.

관계 대수의 연산은 모두 **"테이블을 입력으로 받아 테이블을 출력"**한다. 연산들을 합성(compose)하면 복잡한 질의를 만들 수 있다.

### 기본 단항 연산

**선택 σ (Sigma, Selection)**

조건을 만족하는 행만 추출. 수평 필터링.

```
σ_{조건}(R)

예: σ_{dept_name='컴퓨터공학'}(student)
─── 결과 ───────────────────────────
student_id  student_name  dept_name
1001        김철수         컴퓨터공학
1002        이영희         컴퓨터공학
```

대응 SQL:

```sql
SELECT * FROM student WHERE dept_name = '컴퓨터공학';
```

**투영 π (Pi, Projection)**

지정한 컬럼만 추출. 수직 필터링. 중복 행은 제거된다(집합 의미).

```
π_{컬럼1, 컬럼2, ...}(R)

예: π_{student_name, dept_name}(student)
─── 결과 ─────────────────────
student_name  dept_name
김철수         컴퓨터공학
이영희         컴퓨터공학
박민수         전자공학
```

대응 SQL:

```sql
SELECT DISTINCT student_name, dept_name FROM student;
```

> SQL의 `SELECT`는 이름만 같고 기능은 σ(선택)이 아닌 π(투영)에 더 가깝다. SQL에서 행 필터는 `WHERE`, 컬럼 선택은 `SELECT` 절이다.

**리네임 ρ (Rho, Rename)**

관계(테이블) 또는 애트리뷰트의 이름을 바꾼다. SQL의 `AS`에 해당.

```
ρ_{새이름}(R)
ρ_{새테이블(새컬럼1, 새컬럼2)}(R)

예: ρ_{s(sid, sname, dept)}(student)
→ student 테이블을 s로, 컬럼명을 sid/sname/dept로 바꿔 참조
```

대응 SQL:

```sql
SELECT student_id AS sid, student_name AS sname, dept_name AS dept
FROM student AS s;
```

### 집합 연산

두 관계가 **같은 스키마(컬럼 수·타입이 동일)**여야 적용 가능하다.

**합집합 R ∪ S (Union)**

두 테이블의 행을 합친다. 중복 행은 제거.

```sql
-- A 학과이거나 B 학과인 학생
SELECT student_id FROM student WHERE dept_name = '컴퓨터공학'
UNION
SELECT student_id FROM student WHERE dept_name = '전자공학';
-- (UNION ALL 은 중복 유지)
```

**차집합 R − S (Difference)**

R에만 있고 S에는 없는 행.

```sql
-- 수강신청하지 않은 학생
SELECT student_id FROM student
EXCEPT
SELECT student_id FROM enrollment;
```

**교집합 R ∩ S (Intersection)**

두 테이블 모두에 있는 행.

```sql
-- CS101과 CS102를 모두 듣는 학생
SELECT student_id FROM enrollment WHERE course_id = 'CS101'
INTERSECT
SELECT student_id FROM enrollment WHERE course_id = 'CS102';
```

### 이항 연산

**카티션 곱 R × S (Cartesian Product)**

두 테이블의 모든 행 조합. R이 m행, S가 n행이면 결과는 m × n행. 실제로는 거의 직접 쓰지 않고 조인의 중간 단계로 존재한다.

```
student (3행)  ×  course (3행)  =  9행
student_id  student_name  course_id  course_title
1001        김철수         CS101      자료구조
1001        김철수         CS102      데이터베이스
1001        김철수         EE201      회로이론
1002        이영희         CS101      자료구조
...                                   (총 9행 — 의미 없는 조합 다수)
```

대응 SQL:

```sql
SELECT * FROM student, course;   -- 또는 CROSS JOIN
```

**조인 ⋈ (Natural Join / Theta Join)**

카티션 곱 후 조건을 만족하는 행만 선택. 이것이 SQL의 JOIN이다.

```
enrollment ⋈ student  (공통 컬럼 student_id 로 결합)

─── 결과 ─────────────────────────────────────────────────
student_id  student_name  dept_name   course_id  grade
1001        김철수         컴퓨터공학  CS101      A
1001        김철수         컴퓨터공학  CS102      B
1002        이영희         컴퓨터공학  CS101      A
1003        박민수         전자공학    EE201      C
```

대응 SQL (세타 조인, 더 명시적):

```sql
SELECT e.student_id, s.student_name, s.dept_name, e.course_id, e.grade
FROM enrollment e
JOIN student s ON e.student_id = s.student_id;
```

### 관계 대수의 완결성

아래 다섯 가지가 **관계 대수의 기본 연산(최소 완전 집합)**이다:

```
σ (선택), π (투영), ∪ (합집합), − (차집합), × (카티션 곱)
```

이 다섯 가지만으로 나머지 모든 연산(교집합, 조인, 나누기 등)을 표현할 수 있다. SQL의 모든 질의는 이 다섯 연산의 조합으로 표현된다. "SQL은 관계 대수의 실용적 표현"이다.

```
SQL 절        대응 관계 대수 연산
──────────────────────────────────────
SELECT cols   π (투영)
WHERE cond    σ (선택)
CROSS JOIN    × (카티션 곱)
JOIN ON       ⋈ (조인)
UNION         ∪ (합집합)
EXCEPT        − (차집합)
INTERSECT     ∩ (교집합)
AS            ρ (리네임)
```

## 1.6 NULL과 3값 논리 (Three-Valued Logic)

관계 모델에서 가장 많이 오해받는 개념이 **NULL**이다. NULL은 "값이 없음" 또는 "알 수 없음"을 나타낸다. 여기서 "알 수 없음"이 핵심이다.

### NULL은 값이 아니다

NULL은 `0`도 아니고 `''`(빈 문자열)도 아니다. 말 그대로 "값이 무엇인지 알 수 없음"이다.

```sql
-- 이것들은 모두 NULL이 아니다
SELECT 0 IS NULL;          -- false
SELECT '' IS NULL;         -- false
SELECT NULL IS NULL;       -- true
SELECT NULL = NULL;        -- NULL (!)
```

`NULL = NULL`이 `TRUE`가 아닌 `NULL`인 이유: "알 수 없는 값 A"와 "알 수 없는 값 B"가 같은지는 알 수 없다. 따라서 결과도 "알 수 없음(NULL/UNKNOWN)"이다.

### 3값 논리 진리표

관계형 DB는 TRUE / FALSE 의 2값 논리가 아니라 **TRUE / FALSE / UNKNOWN(NULL)** 의 3값 논리를 사용한다.

```
AND 진리표
─────────────────────────────────────
AND      │ TRUE    │ FALSE   │ UNKNOWN
─────────┼─────────┼─────────┼────────
TRUE     │ TRUE    │ FALSE   │ UNKNOWN
FALSE    │ FALSE   │ FALSE   │ FALSE
UNKNOWN  │ UNKNOWN │ FALSE   │ UNKNOWN

OR 진리표
─────────────────────────────────────
OR       │ TRUE    │ FALSE   │ UNKNOWN
─────────┼─────────┼─────────┼────────
TRUE     │ TRUE    │ TRUE    │ TRUE
FALSE    │ TRUE    │ FALSE   │ UNKNOWN
UNKNOWN  │ TRUE    │ UNKNOWN │ UNKNOWN

NOT 진리표
──────────────────────
NOT TRUE    = FALSE
NOT FALSE   = TRUE
NOT UNKNOWN = UNKNOWN
```

WHERE 절은 **TRUE인 행만** 반환한다. UNKNOWN(NULL이 포함된 비교)은 FALSE처럼 처리되어 결과에서 제외된다.

### NULL 비교의 핵심 규칙

```sql
-- NULL 비교는 항상 = 대신 IS NULL / IS NOT NULL 을 써야 한다
SELECT * FROM users WHERE email = NULL;      -- 항상 0행 반환 (UNKNOWN)
SELECT * FROM users WHERE email IS NULL;     -- 올바른 방법
SELECT * FROM users WHERE email IS NOT NULL; -- 올바른 방법

-- COALESCE: NULL을 기본값으로 치환
SELECT COALESCE(phone, '번호 없음') FROM users;

-- NULLIF: 특정 값을 NULL로 치환
SELECT NULLIF(division_count, 0);  -- 0이면 NULL 반환 (0으로 나누기 방지)
```

### NOT IN과 NULL의 치명적 함정

```sql
-- 학생 테이블에 NULL이 포함된 경우의 NOT IN 함정
CREATE TEMP TABLE some_ids (id BIGINT);
INSERT INTO some_ids VALUES (1), (2), (NULL);

-- 직관: "3번 학생은 {1,2,NULL}에 없으니 나와야 하는 거 아냐?"
SELECT * FROM student WHERE student_id NOT IN (SELECT id FROM some_ids);
-- 결과: 0행!! (3번도 안 나옴)
```

왜 그럴까? `NOT IN`은 내부적으로 `≠ 1 AND ≠ 2 AND ≠ NULL`로 전개된다. `3 ≠ NULL`은 UNKNOWN이고, `AND UNKNOWN`이 붙으면 전체가 UNKNOWN이 되어 WHERE가 FALSE처럼 처리한다.

```sql
-- 안전한 대안: NOT EXISTS
SELECT * FROM student s
WHERE NOT EXISTS (
    SELECT 1 FROM some_ids si WHERE si.id = s.student_id
);

-- 또는 IS NOT DISTINCT FROM 을 사용한 명시적 비교
-- 또는 서브쿼리에 WHERE id IS NOT NULL 추가
SELECT * FROM student
WHERE student_id NOT IN (SELECT id FROM some_ids WHERE id IS NOT NULL);
```

이 함정은 현업에서 아주 흔하게 발생하는 버그다. **서브쿼리에 NULL이 있을 가능성이 있으면 `NOT IN` 대신 `NOT EXISTS`를 쓰는 것이 안전하다.**

### PostgreSQL의 NULL 관련 유용한 함수

```sql
-- IS DISTINCT FROM: NULL을 "다른 값"으로 취급하는 비교 (3값 논리를 2값으로)
SELECT NULL IS DISTINCT FROM NULL;     -- false  (NULL = NULL 취급)
SELECT 1 IS DISTINCT FROM NULL;        -- true
SELECT 1 IS NOT DISTINCT FROM NULL;    -- false

-- NULL 정렬: ORDER BY 시 NULL 위치 지정
SELECT name FROM users ORDER BY name NULLS LAST;   -- NULL을 마지막에
SELECT name FROM users ORDER BY name NULLS FIRST;  -- NULL을 처음에
```

## 1.7 커드의 12가지 규칙 (Codd's 12 Rules)

커드는 1985년 "관계형 DBMS가 진짜 관계형이라면 반드시 지켜야 하는 규칙 12가지(실질 13개, 0번부터 12번)"를 제시했다. 모두 외울 필요는 없지만 핵심을 알아두면 좋다.

```
규칙 0 (기반): 관계형 DBMS는 관계형 수단만으로 DB를 관리할 수 있어야 한다.
규칙 1 (정보): 모든 데이터는 테이블의 셀 값으로만 표현된다.
규칙 2 (보장된 접근): 모든 데이터는 테이블명 + PK값 + 컬럼명으로 접근 가능해야 한다.
규칙 3 (NULL 지원): NULL은 "정보 없음"을 나타내는 1등 시민이어야 한다.
규칙 4 (동적 카탈로그): 스키마 정보 자체도 테이블로 조회할 수 있어야 한다.
규칙 5 (완전한 DML): INSERT/UPDATE/DELETE/SELECT가 모두 지원되어야 한다.
규칙 6 (뷰 갱신): 이론적으로 갱신 가능한 뷰는 실제로도 갱신 가능해야 한다.
규칙 7 (고수준 DML): INSERT/UPDATE/DELETE는 집합 연산이어야 한다 (행 단위가 아니라).
규칙 8 (물리적 독립): 물리적 저장 방식이 바뀌어도 응용이 영향받지 않아야 한다.
규칙 9 (논리적 독립): 뷰·조인 등을 써도 기저 테이블 구조 변경의 영향을 최소화해야 한다.
규칙 10 (무결성 독립): 무결성 규칙은 응용 코드가 아닌 DB에 저장되어야 한다.
규칙 11 (분산 독립): 분산 환경에서도 동일하게 작동해야 한다.
규칙 12 (비전복): 저수준 인터페이스로 무결성 제약을 우회할 수 없어야 한다.
```

현실의 DBMS(Oracle, PostgreSQL, MySQL)는 이 규칙 전부를 완벽히 만족하지는 않는다. 그러나 이 규칙들은 "관계형 DB의 이상적 모습"을 정의하는 기준점으로서 의미가 있다.

## ⚠️ 흔한 오해와 함정

**함정 1: `ORDER BY` 없이 특정 순서를 기대한다.**

"PK 순으로 입력했으니 PK 순으로 나오겠지"는 틀린 가정이다. 관계는 집합이고 집합에는 순서가 없다. 같은 쿼리라도 인덱스 사용 여부, 병렬 처리, MVCC 버전에 따라 반환 순서가 달라진다. **항상 `ORDER BY`로 순서를 명시하라.**

**함정 2: `NULL = NULL`이 `TRUE`라고 생각한다.**

`WHERE col = NULL`은 항상 0행이다. `WHERE col IS NULL`을 써야 한다. 이 실수는 데이터가 "사라지는" 버그로 이어진다.

**함정 3: `NOT IN` 서브쿼리에 NULL이 섞이면 전체 결과가 0행이 된다.**

서브쿼리가 NULL을 반환할 가능성이 있으면 `NOT IN` 대신 `NOT EXISTS`를 쓴다. 1.6절의 예제를 다시 확인하라.

**함정 4: 자연키(natural key)를 PK로 쓴다.**

학번, 이메일, 주민번호, 사업자번호는 "지금 유일"하지만 나중에 바뀔 수 있다. PK가 바뀌면 FK가 있는 모든 테이블이 연쇄로 영향을 받는다. 대리키(`BIGSERIAL`, `UUID`)를 PK로 쓰고 자연키는 `UNIQUE NOT NULL`로 두어라.

**함정 5: FK를 빼면 "더 빠르다"고 생각한다.**

FK는 INSERT/UPDATE 시 추가 검사를 하므로 약간의 오버헤드가 있다. 그러나 FK 없이 참조 무결성을 애플리케이션 레이어에서 관리하는 것은 훨씬 위험하다. 동시 접근, 버그, 배치 작업에서 미아 데이터가 조용히 쌓인다. **FK는 성능이 아닌 정확성의 문제다.** 진짜 성능 문제가 측정된 뒤에야 고민하라.

**함정 6: 관계 대수를 "학문적 장식"이라고 무시한다.**

SQL 쿼리가 왜 느린지, 옵티마이저가 어떤 계획을 세우는지 이해하려면 관계 대수가 필수 배경이다. 6장(쿼리 처리)에서 이 대수 연산이 물리 계획으로 변환되는 과정을 직접 보게 된다.

## 연습문제

**문제 1 (기본)**

다음 테이블에서 가능한 모든 슈퍼키, 후보키를 찾고, 기본키를 선택하는 이유를 설명하라.

```
employee(emp_id, ssn, email, name, dept_id)
- emp_id: 사원번호 (자동 부여, 변경 없음)
- ssn: 주민등록번호 (유일, 하지만 변경될 수 있는 민감 정보)
- email: 회사 이메일 (유일, 퇴사 후 재부여 가능)
- name: 이름 (중복 가능)
- dept_id: 부서 ID (FK)
```

> 힌트: 슈퍼키는 `{emp_id}`, `{ssn}`, `{email}`, `{emp_id, ssn}`, `{emp_id, email}`, ... (총 몇 개인지 세어보라). 후보키는 어떤 집합인가? `ssn`을 PK로 안 쓰는 이유는 개인정보 보호법과 변경 가능성 두 가지다.

> 해설: 후보키는 `{emp_id}`, `{ssn}`, `{email}` — 세 개다. `name`은 동명이인이 가능하므로 후보키가 아니다. 기본키는 `emp_id`가 최적: 의미 없는 대리키라 변경 없고, 민감 정보가 아니다.

**문제 2 (NULL 논리)**

다음 질의의 결과가 몇 행인지 예측하고, 이유를 설명하라.

```sql
CREATE TEMP TABLE t (val INT);
INSERT INTO t VALUES (1), (2), (NULL), (3);

-- 질의 A
SELECT * FROM t WHERE val NOT IN (1, 2);

-- 질의 B
SELECT * FROM t WHERE val NOT IN (1, 2, NULL);

-- 질의 C
SELECT * FROM t WHERE val != ALL(ARRAY[1, 2]);
```

> 힌트: `NOT IN` 목록에 NULL이 있으면 모든 비교가 UNKNOWN이 된다. `!= ALL`은 어떻게 다른가?

> 해설: 질의 A → `{3}` 1행 (val=NULL은 `NULL NOT IN (1,2)` = UNKNOWN → 제외). 질의 B → 0행 (NULL이 목록에 있어 3 ≠ NULL = UNKNOWN → 3도 제외). 질의 C → `{3}` 1행 (`!= ALL`은 NULL 원소와 비교 시 UNKNOWN 이므로 NULL 행 제외, 3은 1과도 2와도 다르므로 포함).

**문제 3 (관계 대수)**

아래 스키마와 데이터가 있다.

```
student(student_id, student_name, dept_name)
enrollment(student_id, course_id, grade)
course(course_id, course_title, professor)
```

다음 요구사항을 관계 대수 식으로 표현하고, 대응하는 SQL을 작성하라.

(가) 이교수가 담당하는 강좌를 수강하는 학생의 이름 목록  
(나) 어떤 강좌도 수강하지 않는 학생의 이름 목록

> 힌트: (가)는 σ → ⋈ → π 조합. (나)는 student의 student_id에서 enrollment의 student_id를 뺀 차집합(−)이 필요하다.

> 해설:  
> (가): π_{student_name}( σ_{professor='이교수'}(course) ⋈ enrollment ⋈ student )  
> SQL: `SELECT DISTINCT s.student_name FROM student s JOIN enrollment e ON s.student_id = e.student_id JOIN course c ON e.course_id = c.course_id WHERE c.professor = '이교수';`  
> (나): π_{student_name}( student ) − π_{student_name}( student ⋈ enrollment )  
> SQL: `SELECT student_name FROM student WHERE student_id NOT IN (SELECT student_id FROM enrollment);`  
> (단, enrollment.student_id에 NULL이 없다면 안전. 있으면 NOT EXISTS를 써야 한다.)

**문제 4 (FK와 참조 무결성)**

다음 DDL에서 `ON DELETE` 옵션을 어떻게 설정해야 하는지 요구사항을 읽고 채워라.

```sql
CREATE TABLE department (dept_id BIGSERIAL PRIMARY KEY, name TEXT NOT NULL);

CREATE TABLE employee (
    emp_id  BIGSERIAL PRIMARY KEY,
    name    TEXT NOT NULL,
    dept_id BIGINT REFERENCES department(dept_id) ON DELETE ___A___
);

CREATE TABLE project (
    proj_id BIGSERIAL PRIMARY KEY,
    name    TEXT NOT NULL,
    lead_id BIGINT REFERENCES employee(emp_id) ON DELETE ___B___
);

CREATE TABLE project_member (
    proj_id BIGINT REFERENCES project(proj_id) ON DELETE ___C___,
    emp_id  BIGINT REFERENCES employee(emp_id) ON DELETE ___D___,
    PRIMARY KEY (proj_id, emp_id)
);
```

요구사항:  
A. 부서가 삭제될 때 직원은 삭제하지 말고, 소속 부서를 "소속 없음"으로 표시해야 한다.  
B. 프로젝트 리더가 퇴사해도 프로젝트는 존재해야 하며, 리더를 미정으로 표시한다.  
C. 프로젝트가 삭제되면 참여 기록도 함께 삭제해야 한다.  
D. 직원이 삭제되면 그 직원의 프로젝트 참여 기록도 함께 삭제해야 한다.

> 해설: A → `SET NULL`, B → `SET NULL`, C → `CASCADE`, D → `CASCADE`

**문제 5 (종합)**

"대학교 도서관 대출 시스템"을 설계한다고 하자. 관리해야 할 개체(entity)와 관계(relationship)를 파악하고, 최소 4개의 테이블을 정의하는 DDL을 PostgreSQL 16 문법으로 작성하라. 각 테이블에 적절한 PK, FK, CHECK, NOT NULL을 포함하라. 대리키(surrogate key)를 사용해야 하며 그 이유를 주석으로 달아라.

> 힌트: 개체 후보: 도서(book), 회원(member), 대출 기록(loan). 관계: 회원이 도서를 대출한다. 대출에는 대출일, 반납예정일, 실반납일이 있다. 한 회원이 같은 책을 여러 번 대출할 수 있다.

> 해설 예시:
> ```sql
> CREATE TABLE member (
>     member_id   BIGSERIAL PRIMARY KEY,  -- 대리키: 이름/이메일은 변경 가능
>     email       TEXT UNIQUE NOT NULL,
>     name        TEXT NOT NULL
> );
> CREATE TABLE book (
>     book_id     BIGSERIAL PRIMARY KEY,  -- 대리키: ISBN도 개정판에서 바뀜
>     isbn        TEXT UNIQUE,
>     title       TEXT NOT NULL,
>     author      TEXT NOT NULL
> );
> CREATE TABLE loan (
>     loan_id         BIGSERIAL PRIMARY KEY,
>     member_id       BIGINT NOT NULL REFERENCES member(member_id) ON DELETE RESTRICT,
>     book_id         BIGINT NOT NULL REFERENCES book(book_id) ON DELETE RESTRICT,
>     loan_date       DATE NOT NULL DEFAULT CURRENT_DATE,
>     due_date        DATE NOT NULL CHECK (due_date > loan_date),
>     return_date     DATE CHECK (return_date >= loan_date)
> );
> ```

**JPA와의 연결:**

- `@Id` = 기본키(개체 무결성, NULL 불가)
- `@Column(unique=true)` = 후보키의 UNIQUE 제약
- `@ManyToOne @JoinColumn` = 외래키(참조 무결성)
- `@Column(nullable=false)` = NOT NULL (도메인 무결성)
- `@Check` / `@Column(columnDefinition="...")` = CHECK 제약 (도메인 무결성)
