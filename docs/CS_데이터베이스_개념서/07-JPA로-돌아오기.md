# Chapter 7: JPA로 돌아오기 — 이론과 ORM의 연결

> "이제 이 책에서 배운 DB 이론과 Spring 개념서의 JPA를 **한 장의 지도**로 잇는다. 우리는 관계 모델에서 출발해 정규화·트랜잭션·동시성·인덱스·쿼리 처리라는 여섯 기둥을 세웠다. 이 장은 그 기둥들이 `@Entity` 한 줄, `@Transactional` 한 줄 뒤에서 실제로 어떻게 돌고 있는지를 보여준다. 다시 말해, 우리가 걷어냈던 'JPA라는 표면'을 다시 씌우되, 이번엔 **그 아래가 훤히 보이는** 상태로 씌운다."

## 이 장을 읽기 전에

이 장은 이 책의 **종착역**이자 Spring 개념서 4장으로 돌아가는 **환승역**이다. 앞의 여섯 장이 "데이터베이스란 무엇인가"를 CS 과목으로 팠다면, 이 장은 "그 이론이 JPA/Spring이라는 실무 도구에서 어떻게 노출되는가"를 정리한다.

- 이 책 1장(관계 모델) → JPA의 `@Entity`/`@Id`/`@ManyToOne`
- 이 책 2장(정규화) → 여러 `@Entity`로의 분리와 그로 인한 JOIN·N+1
- 이 책 3장(트랜잭션) → `@Transactional`
- 이 책 4장(동시성) → `@Version`/`@Lock`, 격리 수준

## 7.1 애노테이션 ↔ DB 이론 매핑

JPA 애노테이션은 대부분 이 책의 어떤 이론을 **테이블 DDL/DML로 번역**하는 장치다. 아래 표가 이 책 전체의 요약이자 이 장의 뼈대다.

```
JPA / Spring                        이 책의 이론                        생성/실행되는 것
──────────────────────────────────────────────────────────────────────────────────────
@Entity, @Table                     관계(Relation) = 테이블 (1장)       CREATE TABLE
@Id                                 기본키 PK (1.3절)                   PRIMARY KEY
@GeneratedValue(IDENTITY/SEQUENCE)  대리키(surrogate key) (1.3절)       BIGSERIAL / SEQUENCE
@Column(unique = true)              후보키의 UNIQUE 제약 (1.3절)         UNIQUE + 자동 인덱스 (5장)
@Column(nullable = false)           개체/도메인 무결성 NOT NULL (1장/3.3절) NOT NULL
@Column(length), 타입 매핑           도메인(domain) (1.2절)              VARCHAR(n), 타입 제약
@ManyToOne @JoinColumn              외래키 FK, 참조 무결성 (1.3절)       FOREIGN KEY REFERENCES
@OneToMany / @ManyToMany            관계 대수의 조인 대상 (1장/2장)      JOIN / 조인 테이블
@Table(indexes = @Index(...))       명시적 인덱스 (5장)                 CREATE INDEX
@Transactional                      트랜잭션 + ACID (3장)               BEGIN / COMMIT / ROLLBACK
@Transactional(isolation = ...)     격리 수준 (4.2절)                   SET TRANSACTION ISOLATION LEVEL
@Transactional(propagation=NESTED)  SAVEPOINT / 부분 롤백 (3장)         SAVEPOINT / ROLLBACK TO SAVEPOINT
@Version                            낙관적 락 (4.5절)                   UPDATE ... WHERE version = ?
@Lock(PESSIMISTIC_WRITE)            비관적 락 (4.5절)                   SELECT ... FOR UPDATE
@Check (Hibernate) / CHECK          일관성 제약 (3.3절)                 CHECK (...)
```

**핵심 통찰: 애노테이션 한 줄이 종종 여러 이론을 동시에 부른다.**

```
   @Column(unique = true) private String email;
        │
        ├──▶ 1장(키 이론)  : email을 후보키로 선언 → UNIQUE 제약
        │
        └──▶ 5장(인덱스)   : UNIQUE는 내부적으로 B+tree 인덱스를 만듦
                             → WHERE email = ? 가 O(log n)으로 빨라짐 (부수 효과)
```

이 하나의 애노테이션을 붙이는 순간, 1장의 후보키 이론과 5장의 인덱스 이론이 **동시에** 작동한다. 이것이 "JPA는 이 책의 이론을 애노테이션이라는 표면으로 노출한 것"이라는 말의 의미다.

### 예제로 보는 번역: Entity → 이론 → DDL

```java
@Entity
@Table(name = "posts", indexes = @Index(name = "idx_posts_author", columnList = "author_id"))
public class Post {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;                       // 1.3절 PK + 대리키

    @Column(nullable = false, length = 200)
    private String title;                  // 1장 도메인 + 개체 무결성

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "author_id", nullable = false)
    private User author;                   // 1.3절 FK + 참조 무결성

    @Version
    private Long version;                  // 4.5절 낙관적 락
}
```

이 Entity가 Hibernate에 의해 번역되면, 이 책의 이론이 그대로 PostgreSQL 16 DDL로 나타난다.

```sql
CREATE TABLE posts (
    id        BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,  -- @Id + @GeneratedValue
    title     VARCHAR(200) NOT NULL,                                -- @Column
    author_id BIGINT NOT NULL REFERENCES users(id),                 -- @ManyToOne @JoinColumn
    version   BIGINT                                                -- @Version
);
CREATE INDEX idx_posts_author ON posts (author_id);                 -- @Table(indexes=...)
```

> **직접 확인하기:** Spring 앱을 `spring.jpa.hibernate.ddl-auto`가 켜진 상태로 띄우고, `spring.jpa.show-sql=true`로 로그를 보면 Hibernate가 만들어내는 `CREATE TABLE` DDL을 눈으로 볼 수 있다. 그 DDL이 곧 이 책 1~5장의 이론이다. 인프라 PostgreSQL(`docker/docker-compose-infra.yml`, DB `edu_spring`)에 접속해 `\d posts`로 실제 테이블 구조와 제약·인덱스를 확인해 보라.

## 7.2 정규화가 만든 JOIN, 그리고 N+1이라는 청구서

이 책 2장은 "저장은 나눠서(정규화), 조회는 JOIN으로 합쳐서"를 가르쳤다. JPA는 이 JOIN을 `@ManyToOne`/`@OneToMany` 연관관계로 자동화한다. 편리하지만, 이 자동화가 이 책 6장의 조인 알고리즘을 **잘못** 부를 때 생기는 것이 바로 **N+1 문제**다.

### N+1은 애플리케이션 레벨의 나쁜 Nested Loop다

```
게시글 10개를 조회하고, 각 게시글의 작성자 이름을 출력하려 한다.

  [1]  SELECT * FROM posts LIMIT 10;                    ← 게시글 10개 (쿼리 1번)
  [2]  for each post:
          SELECT * FROM users WHERE id = post.author_id;  ← 작성자 1명씩 (쿼리 N번)

  총 쿼리 = 1 + N = 1 + 10 = 11번   ← "N+1 문제"
```

이 패턴을 6장의 눈으로 보면 정체가 드러난다. **DB 안에서 한 번의 JOIN(Hash/Merge Join, O(A+B))으로 끝날 일을, 애플리케이션이 바깥 루프를 돌며 안쪽 조회를 N번 날리는 Nested Loop로 펼친 것**이다. 게다가 이 Nested Loop는 네트워크 왕복(round-trip)까지 N번 발생시켜, DB 내부 Nested Loop보다 훨씬 비싸다.

```
DB가 JOIN으로 처리하면:            JPA가 N+1로 펼치면:
┌──────────────────────┐          ┌──────────────────────┐
│ posts ⋈ users        │          │ posts 조회 (왕복 1회) │
│ = Hash Join 한 방     │          │  + users 조회 × N회   │
│ 네트워크 왕복 1회      │          │ 네트워크 왕복 N+1회    │
│ O(A + B)             │          │ O(A) × (왕복 비용)    │
└──────────────────────┘          └──────────────────────┘
```

### 해결: fetch join으로 "DB에게 JOIN을 맡겨라"

```java
// N+1을 유발하는 코드 (지연 로딩된 author를 루프에서 하나씩 접근)
List<Post> posts = postRepository.findAll();
for (Post p : posts) {
    System.out.println(p.getAuthor().getName());  // 여기서 매번 SELECT
}

// fetch join: 한 번의 JOIN으로 author까지 함께 가져온다
@Query("SELECT p FROM Post p JOIN FETCH p.author")
List<Post> findAllWithAuthor();
```

`JOIN FETCH`는 Hibernate에게 "6장의 Hash/Merge Join으로 한 방에 합쳐서 가져오라"고 지시하는 것이다. 실행되는 SQL은 이 책 2.4절의 그 JOIN과 동일하다.

```sql
SELECT p.*, u.*
  FROM posts p
  JOIN users u ON u.id = p.author_id;   -- 왕복 1회, DB가 조인 알고리즘 선택
```

> **상호 참조:** N+1 문제의 JPA 관점 상세(지연/즉시 로딩, `@BatchSize`, `@EntityGraph`)는 [`../SPRING_개념서/README.md`](../SPRING_개념서/README.md) 4.4절에 있다. 이 책은 그 문제의 **정체**(정규화가 만든 JOIN 필요성 + ORM이 그걸 나쁜 Nested Loop로 번역)를 6장의 이론으로 설명했다. 두 문서를 함께 읽어야 "왜 fetch join으로 고쳐야 하는지"가 완전히 이해된다.

## 7.3 트랜잭션과 동시성: 애노테이션 뒤의 BEGIN/COMMIT

### @Transactional = 3장의 BEGIN/COMMIT/ROLLBACK

Spring의 `@Transactional`은 이 책 3장의 트랜잭션 경계를 **프록시(AOP)**로 자동 삽입한다.

```
@Transactional 메서드 호출
   │
   ▼
프록시가 가로챔 → BEGIN          (3장: 트랜잭션 시작)
   │
   ▼
실제 메서드 본문 실행
   │
   ├─ 정상 종료 → COMMIT          (3장: 변경 확정, WAL로 지속성 보장)
   │
   └─ 런타임 예외 → ROLLBACK      (3장: 원자성 — 전부 되돌림)
```

즉 이 책 3장에서 손으로 쓴 `BEGIN; ... COMMIT;`이, `@Transactional` 뒤에서 프록시에 의해 자동으로 일어난다. 예외 시 전체 롤백은 3.2절의 **원자성**이고, 커밋 후 정전 생존은 3.5절의 **WAL 지속성**이다.

```java
@Transactional
public void transfer(String from, String to, int amount) {
    accountRepository.withdraw(from, amount);   // UPDATE ... balance - amount
    accountRepository.deposit(to, amount);       // UPDATE ... balance + amount
    // 메서드 정상 종료 시 프록시가 COMMIT.
    // 중간에 예외(예: CHECK balance>=0 위반)가 나면 프록시가 ROLLBACK → 3.2절 원자성.
}
```

`@Transactional(propagation = Propagation.NESTED)`는 이 책 3장에서 본 **SAVEPOINT**를 내부적으로 사용해 부분 롤백을 구현한다(랩 `02_transactions_acid.sql`의 실험 4).

### 격리 수준과 락: 4장의 선택을 애노테이션으로

```java
// 4.2절 격리 수준 지정
@Transactional(isolation = Isolation.REPEATABLE_READ)
public Report buildReport() { ... }

// 4.5절 낙관적 락: @Version 필드 하나로 UPDATE ... WHERE version=? 자동 생성
@Version private Long version;

// 4.5절 비관적 락: SELECT ... FOR UPDATE 발생
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT s FROM Stock s WHERE s.id = :id")
Stock findByIdForUpdate(@Param("id") Long id);
```

이 프로젝트의 PostgreSQL은 기본 격리 수준이 **READ COMMITTED**(4.2절)다. 재고 차감처럼 경쟁이 심한 곳에서 음수 재고 같은 버그가 나면, 4.5절의 두 철학 중 하나를 골라야 한다.

```
동시성 버그 상황                  이 책 4장의 처방              JPA 애노테이션
────────────────────────────────────────────────────────────────────────
읽고-계산하고-쓰는 사이 경합       낙관적 락 (충돌 드묾)         @Version
재고/좌석 등 강한 경쟁            비관적 락 (충돌 잦음)         @Lock(PESSIMISTIC_WRITE)
집계/리포트 중 값이 바뀜          격리 수준 상향               @Transactional(isolation=...)
```

> **상호 참조:** `@Transactional`의 프록시 동작 원리는 [`../SPRING_개념서/README.md`](../SPRING_개념서/README.md) 2.6절(AOP/프록시)과 4.5절, 격리 수준과 락의 JPA 사용법은 4.6절에 있다. 이 책은 그 애노테이션들이 부르는 **DB 내부 메커니즘**(ACID 구현, 2PL, MVCC, SSI)을 3~4장에서 팠다.

## 7.4 인덱스와 느린 쿼리: `findByEmail`이 느릴 때

`UserRepository.findByEmail(String email)` 한 줄은 아래 SQL로 번역된다.

```sql
SELECT * FROM users WHERE email = ?;
```

이게 느리다면 원인은 이 책 5장에 있다. `email` 컬럼에 인덱스가 없어 **Seq Scan(O(n))**을 하고 있는 것이다. 진단과 처방:

```
1. 5.6절 EXPLAIN으로 확인:
     EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'kim@x.com';
     → "Seq Scan on users ... Rows Removed by Filter: 999999" 가 보이면 인덱스 부재.

2. 처방 — JPA에서 인덱스 선언 (@Table):
     @Table(name="users", indexes = @Index(name="idx_users_email", columnList="email"))
     또는 @Column(unique=true) — UNIQUE는 자동으로 인덱스를 만든다 (5장).

3. 다시 EXPLAIN → "Index Scan using idx_users_email" 로 바뀌면 O(log n) 달성.
```

**복합 조건 쿼리**는 5.3절의 왼쪽 접두사 규칙을 기억해야 한다. `findByCountryAndAge(...)`가 자주 쓰인다면 `@Index(columnList = "country, age")`처럼 **자주 함께 쓰는 조건의 앞 컬럼 순서**를 맞춰 복합 인덱스를 건다. 반대로 `findByAge(...)`만 필요한데 `(country, age)` 인덱스를 걸면 못 탄다.

> **상호 참조:** 인덱스의 JPA 관점 소개는 [`../SPRING_개념서/README.md`](../SPRING_개념서/README.md) 4.7절, `EXPLAIN` 읽는 법과 B+tree 내부는 이 책 [5장](05-인덱싱.md)에 있다. 느린 쿼리를 만나면 항상 5.6절의 `EXPLAIN (ANALYZE)`부터 돌려라 — 추측하지 말고 측정하라.

## 7.5 왜 JPA를 쓰면서도 DB 이론을 알아야 하는가

JPA는 DB를 **가려주지만, 완전히 숨기지는 못한다.** 이 책의 이론을 모르면 다음 네 가지를 영영 이해할 수 없다.

**1. N+1 문제 (Spring 4.4절 · 이 책 2·6장)**
정규화(2장)로 나눈 테이블을 ORM이 자동 JOIN하려다, 나쁜 Nested Loop(6.2절)를 애플리케이션 레벨에서 반복하는 것. 이론을 알아야 "왜 fetch join으로 Hash/Merge Join 한 방에 끝내야 하는지"가 보인다.

**2. 느린 쿼리 (이 책 5장)**
`findByEmail`이 느리다면? 5장의 인덱스가 없거나 못 타는 것이다. `EXPLAIN`(5.6절)을 읽을 줄 알아야 원인을 찾는다.

**3. 데이터 꼬임/동시성 버그 (이 책 4장)**
두 요청이 같은 재고를 동시에 깎아 음수가 됐다면? 4장의 격리 수준과 락을 알아야 `@Version`이나 `@Lock`으로 고칠 수 있다. write skew(4.4절) 같은 미묘한 버그는 SERIALIZABLE이나 명시적 락이 필요하다.

**4. 이상한 스키마 (이 책 2장)**
처음엔 잘 돌던 Entity가 데이터가 쌓이며 중복·모순으로 썩는다면? 2장의 정규화를 안 한 것이다. 함수 종속을 찾아 3NF/BCNF로 분해해야 한다.

```
              JPA (편리한 표면)
   ┌───────────────────────────────────────┐
   │  @Entity  @Transactional  @Version ... │
   └───────────────────────────────────────┘
                     ▲  이 표면을 걷어낼 수 있는가?
   ═══════════════════════════════════════════  ← 전공자 vs 사용자의 경계
                     │
   ┌───────────────────────────────────────┐
   │ 1장 관계모델   2장 정규화   3장 트랜잭션 │
   │ 4장 동시성     5장 인덱스   6장 쿼리처리 │  ← 언제나 아래에서 돌고 있는 여섯 기둥
   └───────────────────────────────────────┘
```

> **결론:** JPA는 "SQL을 안 써도 되게" 해주지만, "DB를 몰라도 되게" 해주지는 **않는다.** 이 책의 관계 모델·정규화·트랜잭션·동시성·인덱스·쿼리 처리 여섯 기둥이, JPA라는 편리한 표면 아래에서 언제나 돌고 있다. **전공자 수준이란, 그 표면을 걷어내고 아래를 볼 수 있다는 뜻이다.**

## ⚠️ 흔한 오해와 함정

**"JPA를 쓰면 SQL과 DB를 몰라도 된다."**
정반대다. JPA는 쉬운 80%를 자동화하는 대신, 어려운 20%(N+1, 락, 인덱스, 스키마 설계)를 만났을 때 **DB 이론을 아는 사람만** 고칠 수 있게 만든다. 추상화는 새는 법이다(leaky abstraction).

**"`@Column(unique=true)`는 그냥 중복 검사일 뿐이다."**
아니다. UNIQUE 제약은 후보키(1장)를 선언하는 동시에, 그 제약을 강제하기 위해 **B+tree 인덱스(5장)를 자동 생성**한다. 그래서 UNIQUE 컬럼 조회는 부수적으로 빨라진다.

**"`@Transactional`만 붙이면 동시성 문제가 다 해결된다."**
아니다. `@Transactional`은 원자성·격리성의 **경계**만 그어줄 뿐, 격리 수준(기본 READ COMMITTED)이 낮으면 반복 불가능한 읽기·팬텀·write skew가 여전히 발생한다. 동시성 버그는 4장의 지식으로 격리 수준을 올리거나 `@Version`/`@Lock`을 명시해야 막힌다.

**"연관관계를 EAGER로 걸면 N+1이 사라진다."**
EAGER는 N+1을 **다른 시점으로 옮길 뿐**(때로는 더 많은 쿼리를 유발) 근본 해결이 아니다. 진짜 해결은 필요할 때 `JOIN FETCH`/`@EntityGraph`로 명시적 JOIN을 지시하는 것이다.

**"인덱스는 많을수록 좋다."**
5.5절에서 봤듯 인덱스는 쓰기(INSERT/UPDATE/DELETE)마다 갱신 비용이 든다. `@Index`를 남발하면 조회는 빨라져도 쓰기가 느려지고 디스크를 낭비한다.

## 연습문제

**문제 1.** 아래 Entity가 Hibernate에 의해 어떤 PostgreSQL DDL로 번역될지 쓰고, 각 줄이 이 책의 몇 장 이론에 대응하는지 표시하라.
```java
@Entity @Table(name = "members")
public class Member {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY) private Long id;
    @Column(unique = true, nullable = false, length = 100) private String email;
    @ManyToOne @JoinColumn(name = "team_id") private Team team;
    @Version private Long version;
}
```
<details><summary>힌트</summary>7.1절의 매핑 표를 참고. `@Id`→PK, `@GeneratedValue`→IDENTITY, `@Column(unique)`→UNIQUE(+인덱스), `@ManyToOne`→FK, `@Version`→낙관적 락 컬럼.</details>
<details><summary>해설</summary>

```sql
CREATE TABLE members (
    id      BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,  -- 1.3절 PK+대리키
    email   VARCHAR(100) NOT NULL UNIQUE,                         -- 1.3절 후보키 + 5장 인덱스 자동 + 1장 NOT NULL
    team_id BIGINT REFERENCES teams(id),                          -- 1.3절 FK/참조 무결성
    version BIGINT                                                -- 4.5절 낙관적 락
);
-- UNIQUE 제약 때문에 email에 B+tree 인덱스가 자동 생성됨(5장).
```
</details>

**문제 2.** 게시판에서 "각 게시글과 그 작성자 이름"을 목록으로 보여주는데, 게시글 20개를 조회하니 SQL 로그에 SELECT가 21번 찍혔다. (a) 이 현상의 이름은? (b) 이 책 6장의 어떤 조인 개념으로 설명되는가? (c) JPA에서 어떻게 고치며, 그때 실행되는 SQL은?
<details><summary>힌트</summary>7.2절. 1 + N번의 쿼리, 애플리케이션 레벨 Nested Loop.</details>
<details><summary>해설</summary>(a) N+1 문제(1+20=21). (b) DB의 한 번 JOIN(Hash/Merge, O(A+B))으로 끝날 일을 애플리케이션이 바깥 루프 + 안쪽 조회 N번의 **나쁜 Nested Loop**로 펼친 것. (c) `@Query("SELECT p FROM Post p JOIN FETCH p.author")` 또는 `@EntityGraph`로 해결. 실행 SQL은 `SELECT p.*, u.* FROM posts p JOIN users u ON u.id = p.author_id`로 왕복 1회.</details>

**문제 3.** 온라인 강의 좌석 예약에서, 정원이 1석 남았는데 두 사용자가 동시에 예약해 정원이 -1이 되는 버그가 보고됐다. 기본 격리 수준(READ COMMITTED)에서 왜 이런 일이 생기는지 4장 개념으로 설명하고, JPA로 고치는 두 가지 방법을 제시하라.
<details><summary>힌트</summary>두 트랜잭션이 각자 "남은 좌석 1"을 읽고 각자 차감. 4.4절 write skew / 4.5절 락.</details>
<details><summary>해설</summary>두 트랜잭션이 커밋 전 서로의 변경을 못 보므로 각자 "남은 1석"을 읽고 각자 예약을 확정 → 이중 예약(일종의 lost update / write skew, 4.4절). 해결: (1) **비관적 락** — `@Lock(PESSIMISTIC_WRITE)`로 `SELECT ... FOR UPDATE`를 걸어 좌석 행을 먼저 잠근다(경쟁 심하니 적합). (2) **낙관적 락** — 좌석 Entity에 `@Version`을 두어, 뒤늦은 커밋이 `WHERE version=?` 불일치로 `OptimisticLockException`을 맞고 재시도. 또는 (3) `@Transactional(isolation = SERIALIZABLE)`로 SSI에게 맡긴다.</details>

**문제 4.** `memberRepository.findByLastNameAndFirstName(...)`이 자주 호출되어 느리다. `@Index`로 복합 인덱스를 걸려 한다. `@Index(columnList = "first_name, last_name")`과 `@Index(columnList = "last_name, first_name")` 중 무엇이 이 쿼리에 유효한가? 5장의 어떤 규칙 때문인가? 만약 `findByLastName(...)`도 자주 쓴다면 어느 쪽이 더 이득인가?
<details><summary>힌트</summary>5.3절 왼쪽 접두사 규칙. 쿼리는 last_name과 first_name을 함께 쓴다.</details>
<details><summary>해설</summary>둘 다 두 컬럼을 모두 WHERE에 쓰므로 이 쿼리 자체에는 둘 다 유효하다. 하지만 `findByLastName(...)`(last_name 단독)도 자주 쓴다면 `(last_name, first_name)` 인덱스가 낫다 — 왼쪽 접두사 규칙(5.3절)에 의해 이 인덱스는 `last_name` 단독 조건에도 쓸 수 있지만, `(first_name, last_name)`은 `last_name` 단독 조회에 못 쓰기 때문. **앞 컬럼을 "단독으로도 자주 쓰는 컬럼"으로 두는 것**이 원칙.</details>

**문제 5.** "우리 서비스는 JPA를 쓰니 SQL과 DB 이론은 몰라도 된다"는 주장에 대해, 이 책의 각 장을 근거로 반박하라(최소 3개 장 인용).
<details><summary>해설</summary>예시 반박: (2장) 정규화를 모르면 Entity 설계가 이상 현상으로 썩는다. (5장) 인덱스와 EXPLAIN을 모르면 느린 쿼리를 진단·처방할 수 없다. (4장) 격리 수준·락을 모르면 동시성 버그(이중 예약, 음수 재고)를 못 고친다. (6장) 조인 이론을 모르면 N+1의 정체와 fetch join 해법을 이해할 수 없다. → JPA는 쉬운 80%를 자동화하지만, 문제가 터지는 어려운 20%는 DB 이론을 아는 사람만 해결한다(leaky abstraction).</details>

## 요약

- JPA 애노테이션은 대부분 이 책의 이론을 **테이블 DDL/DML로 번역**하는 장치다. 애노테이션 한 줄이 종종 여러 이론(예: `@Column(unique)` = 후보키 + 인덱스)을 동시에 부른다.
- **N+1 문제**는 정규화(2장)가 만든 JOIN 필요성을 ORM이 애플리케이션 레벨의 **나쁜 Nested Loop(6장)**로 펼친 것이다. `JOIN FETCH`로 DB의 Hash/Merge Join에 맡겨 해결한다.
- `@Transactional`은 프록시가 3장의 `BEGIN/COMMIT/ROLLBACK`을 자동 삽입하는 것이고, 예외 시 롤백은 **원자성**, 커밋 후 생존은 **WAL 지속성**이다.
- 동시성 버그는 4장의 격리 수준·락 지식으로 `@Version`(낙관적)·`@Lock`(비관적)·`isolation` 상향으로 고친다.
- 느린 쿼리는 5장의 `EXPLAIN (ANALYZE)`로 진단하고, `@Index`(왼쪽 접두사 규칙 준수)로 처방한다.
- **JPA는 SQL을 안 써도 되게 해주지만, DB를 몰라도 되게 해주지는 않는다.** 전공자 수준이란 그 표면을 걷어내고 아래의 여섯 기둥을 볼 수 있다는 뜻이다.
