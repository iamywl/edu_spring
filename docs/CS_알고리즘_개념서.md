## Chapter 1: Big-O — 복잡도를 "추론"하는 기술

### 1.1 왜 시간을 초로 재지 않는가?

"이 코드 빠른가요?"라는 질문에 "0.3초요"라고 답하면 안 된다. 왜?

- 어떤 컴퓨터에서 쟀는가? (내 노트북 vs 서버)
- 데이터가 100개일 때인가, 1억 개일 때인가?
- JIT 컴파일러가 워밍업됐는가?

초(second)는 **환경에 따라 달라진다**. 우리가 알고 싶은 것은 절대 속도가 아니라
**"입력이 커질 때 연산량이 어떻게 증가하는가"** 라는 *경향성*이다.

이 경향성을 표현하는 언어가 **Big-O 표기법**이다.

```
  입력 크기 n에 대해 연산 횟수가...

  일정하다             → O(1)      "상수 시간"
  아주 천천히 는다     → O(log n)  "로그 시간"
  비례해서 는다        → O(n)      "선형 시간"
  n log n으로 는다     → O(n log n) "선형로그 시간"
  제곱으로 폭발한다    → O(n²)     "제곱 시간"
  지수로 폭발한다      → O(2ⁿ)     "지수 시간" (사실상 못 씀)
```

### 1.2 복잡도를 "유도"하는 3단계

Big-O는 외우는 게 아니라 **유도하는 것**이다. 규칙은 셋뿐이다.

**규칙 1: 지배적인 연산을 센다.**
"입력 n이 커질 때 몇 번 실행되는가"를 세는 것이 핵심이다.

```java
for (int i = 0; i < n; i++) {   // n번 반복
    sum += arr[i];              // 이 줄이 n번 실행 → O(n)
}
```

**규칙 2: 상수와 낮은 차수 항을 버린다.**

```
  3n + 5     → O(n)       (계수 3, 상수 5 버림)
  n² + 100n  → O(n²)      (n이 커지면 n²이 지배)
  2ⁿ + n¹⁰⁰  → O(2ⁿ)      (지수가 다항식을 이긴다)
```

> **왜?** 버려도 되는가?
>
> n이 충분히 커지면, 큰 항이 작은 항을 압도한다.
> n=1,000,000일 때 n²=1조인데 100n=1억이다. 100n은 있으나 마나다.
> Big-O는 "충분히 큰 n에서의 지배적 성장률"만 본다.

**규칙 3: 중첩된 반복은 곱한다.**

```java
for (int i = 0; i < n; i++) {       // n번
    for (int j = 0; j < n; j++) {   // × n번
        doSomething();              // = n² 번 → O(n²)
    }
}
```

반복이 **매번 절반씩 줄면** O(log n)이다.

```java
while (n > 1) {
    n = n / 2;   // 매번 절반 → log₂(n)번이면 1이 됨 → O(log n)
}
```

> **왜 log인가?** n을 몇 번 2로 나눠야 1이 되는가? 그것이 log₂(n)이다.
> 1,000,000을 반으로 계속 나누면 약 20번이면 1이 된다(2²⁰ ≈ 100만).
> 그래서 "절반씩 줄이기"는 항상 O(log n)이다. **이진 탐색의 핵심이다.**

### 1.3 Big-O, Θ, Ω — 상한, 딱 맞음, 하한

세 기호는 헷갈리지만 뜻은 간단하다.

```
  Ω(하한) ────── Θ(정확) ────── O(상한)
  "최소 이만큼"   "딱 이만큼"     "많아야 이만큼"
```

- **O(f)** = 상한. "아무리 나빠도 f보다는 빠르다(같거나)."
- **Ω(f)** = 하한. "아무리 좋아도 f보다 느리다(같거나)."
- **Θ(f)** = 상한과 하한이 같음. "정확히 f로 자란다."

실무에서 "O(n)"이라 말할 때는 보통 Θ(n)을 뜻한다.
하지만 이론적으로 중요한 순간이 있다:
**"비교 기반 정렬은 Ω(n log n)이다"** — 이건 "그 어떤 비교 정렬도 n log n보다 빠를 수 없다"는
**하한**의 선언이다(§3.5에서 증명한다).

### 1.4 최선 / 평균 / 최악

같은 알고리즘도 입력에 따라 성능이 다르다.

**예: 선형 탐색으로 값 찾기**
- **최선(best)**: 찾는 값이 맨 앞 → O(1)
- **평균(average)**: 대략 중간쯤 → O(n)
- **최악(worst)**: 맨 뒤에 있거나 없음 → O(n)

**예: 퀵 정렬**
- **최선/평균**: 피벗이 잘 나뉨 → O(n log n)
- **최악**: 피벗이 매번 한쪽으로 치우침(이미 정렬된 입력) → O(n²)

> **실무 원칙:** 보통 **최악의 경우**를 기준으로 판단한다.
> "평균은 빠른데 가끔 멈춘다"는 서비스에서는 재앙이기 때문이다.

### 1.5 분할 상환 분석 (Amortized Analysis)

`ArrayList.add()`는 O(1)이라고 배웠다. 그런데 내부 배열이 꽉 차면
2배 큰 배열로 **전부 복사(O(n))**해야 한다. 그럼 O(n) 아닌가?

```
  용량 4 배열이 꽉 참
        ↓
  용량 8 새 배열 생성 + 기존 4개 복사 (이 순간만 비용 발생)
        ↓
  이후 4번의 add는 복사 없이 O(1)
```

핵심은 **확장이 매번 일어나지 않는다**는 것이다. 확장 비용을
그 전후의 저렴한 add들에 "나눠서 분담"시키면, add 한 번당 평균 비용은 O(1)이다.
이것이 **분할 상환(amortized) O(1)** 이다.

> **비유:** 1년에 한 번 100만 원짜리 보험료를 낸다면,
> 매달 부담은 약 8만 원이다. "가끔 크게, 나머진 0"을
> "매번 조금씩"으로 평탄화하는 것이 분할 상환이다.

`DataStructuresFromScratch.java`를 실행하면 이 "복사 이사"가 언제 일어나는지
직접 출력으로 확인할 수 있다.

## Chapter 2: 탐색 — 선형 vs 이진

> 실행 예제: `SearchAlgorithms.java`

### 2.1 선형 탐색 O(n): 우직하게 하나씩

```
  [2, 5, 8, 12, 16, 23, 38, ...]   target=23 찾기

  2? 아님 → 5? 아님 → 8? 아님 → 12? 아님 → 16? 아님 → 23? 찾음!
  (6번 비교)
```

앞에서부터 하나씩 비교한다. 최악의 경우 n번 비교 → **O(n)**.
장점: **정렬 안 된 배열에서도 된다.** 아무 전제가 없다.

```java
static int linearSearch(int[] arr, int target) {
    for (int i = 0; i < arr.length; i++) {
        if (arr[i] == target) return i;  // 최악 n번 비교
    }
    return -1;
}
```

### 2.2 이진 탐색 O(log n): 절반씩 버리기

**정렬된** 배열에서는 훨씬 영리하게 할 수 있다.

```
  [2, 5, 8, 12, 16, 23, 38, 56, 72, 91]   target=23

  ① 가운데(16) 확인: 23 > 16 → 왼쪽 절반 버림, 오른쪽만 봄
     [.., .., .., .., .., 23, 38, 56, 72, 91]
  ② 가운데(56) 확인: 23 < 56 → 오른쪽 절반 버림
     [.., .., .., .., .., 23, 38, .., .., ..]
  ③ 가운데(23) 확인: 일치! (3번 비교)
```

매 단계마다 후보가 **절반**으로 준다. n을 몇 번 절반내면 1이 되는가?
→ log₂(n)번. 그래서 **O(log n)**.

```java
static int binarySearch(int[] arr, int target) {
    int lo = 0, hi = arr.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;   // (lo+hi)/2 는 오버플로 위험 → 이 형태 권장
        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) lo = mid + 1;  // 오른쪽 절반
        else hi = mid - 1;                          // 왼쪽 절반
    }
    return -1;
}
```

### 2.3 왜 "정렬"이 대전제인가?

이진 탐색은 "가운데가 target보다 작으니 왼쪽은 볼 필요 없다"고 **판단**한다.
이 판단은 **정렬되어 있을 때만** 옳다.

정렬 안 된 배열 `[38, 2, 91, 16, ...]`에서 가운데가 target보다 작다고 해서
왼쪽에 답이 없다는 보장이 전혀 없다. → **결과가 틀린다.**

`SearchAlgorithms.java`는 정렬 안 된 배열에 이진 탐색을 시도해 실제로
**존재하는 값을 못 찾는(FAIL)** 것을 보여준다.

### 2.4 언제 무엇을 쓰는가?

```
  한 번만 찾을 거면?        → 선형 탐색 O(n)
                             (정렬 비용 O(n log n)이 아까움)

  같은 배열에서 여러 번?    → 한 번 정렬 O(n log n) 후
                             매 탐색 이진 탐색 O(log n)
```

**측정 결과** (`SearchAlgorithms.java` 출력):

| N | 선형(비교) | 이진(비교) |
|---|---|---|
| 100 | 100 | 6 |
| 1,000 | 1,000 | 9 |
| 100,000 | 100,000 | 16 |
| 1,000,000 | 1,000,000 | 19 |

N이 10,000배 커져도 이진은 비교가 6 → 19로 **겨우 3배**밖에 안 는다. 이것이 로그의 힘이다.

**Java 연결고리:** `Arrays.binarySearch`, `Collections.binarySearch`, 그리고
`TreeMap`/`TreeSet`(정렬 트리, §6) 이 모두 이 "절반 버리기" 원리 위에 서 있다.

## Chapter 3: 정렬 — O(n²)과 O(n log n)

> 실행 예제: `SortingAlgorithms.java`

정렬은 CS의 "Hello World"다. 같은 문제를 푸는 여러 방법의 복잡도 차이를
가장 극명하게 보여주기 때문이다. 그리고 여기서 **안정성**과 **제자리** 개념을 처음 만난다.

### 3.0 두 가지 핵심 성질

**안정성(stability):** 값이 같은 원소들의 **원래 순서**가 보존되는가?

```
  입력:  3a  1b  3c  2d  1e  3f    (알파벳은 입력 순번)
  안정 정렬:  1b 1e 2d 3a 3c 3f    ← 3들이 a,c,f 순서 그대로
  불안정:     1b 1e 2d 3c 3a 3f    ← 3들의 순서가 뒤바뀜
```

> **왜 중요한가?** "이름순 정렬된 목록"을 다시 "부서순"으로 정렬할 때,
> 안정 정렬이면 같은 부서 안에서 이름순이 유지된다. 불안정이면 깨진다.

**제자리(in-place):** 입력 배열 외에 추가 메모리를 거의 안 쓰는가?

### 3.1 O(n²) 삼형제

#### 버블 정렬 (Bubble) — 안정, 제자리

인접한 두 원소를 비교해 크면 뒤로 보낸다. 한 바퀴 돌면 가장 큰 값이 맨 뒤로 "떠오른다".

```
  [5, 2, 9, 1]
   ↑↑ 5>2 교환 → [2, 5, 9, 1]
      ↑↑ 5<9 유지
         ↑↑ 9>1 교환 → [2, 5, 1, 9]   (9가 맨 뒤로 떠오름)
  ... 반복
```

이중 반복 → **O(n²)**. 가장 느리지만 가장 이해하기 쉽다.

#### 선택 정렬 (Selection) — 불안정, 제자리

남은 부분에서 **최솟값을 골라** 맨 앞과 교환한다.
비교는 O(n²)이지만 **교환은 최대 n-1번**(교환 비용이 클 때 유리).

#### 삽입 정렬 (Insertion) — 안정, 제자리

손패 정렬처럼, 앞쪽 정렬된 부분에 현재 원소를 **제자리에 끼워 넣는다.**

```
  손에 [2, 5, 8] 이 정렬돼 있고, 새 카드 4가 들어오면
  → 5,8을 오른쪽으로 밀고 4를 끼움 → [2, 4, 5, 8]
```

> **왜 실무에서 쓰이나?** **거의 정렬된 배열**에서는 각 원소가 조금만 이동하므로
> 거의 O(n)에 가깝다. 그래서 Java의 `Arrays.sort`(TimSort)도 작은 조각은 삽입 정렬로 처리한다.

### 3.2 O(n log n) 형제

#### 병합 정렬 (Merge) — 안정, 추가 메모리 필요

**분할 정복(divide and conquer):** 반으로 쪼개 각각 정렬한 뒤, 두 정렬된 배열을 병합한다.

```
        [5, 2, 9, 1, 6, 3]
        /               \
    [5, 2, 9]         [1, 6, 3]        ← 반으로 쪼갬 (log n 깊이)
     /    \            /    \
  [5]  [2, 9]       [1]  [6, 3]
        ...                ...
        ↓ 병합하며 올라옴
    [2, 5, 9]         [1, 3, 6]
        \               /
        [1, 2, 3, 5, 6, 9]              ← 두 정렬 배열 병합 (각 층 O(n))
```

쪼개는 깊이 = log n, 각 깊이의 병합 비용 = O(n) → **O(n log n)**.
안정 정렬이지만 병합할 새 배열이 필요해 **제자리가 아니다.**

#### 퀵 정렬 (Quick) — 불안정, 제자리, 평균 최고속

**피벗(pivot)** 하나를 골라 "작은 것 왼쪽 / 큰 것 오른쪽"으로 분할한다.
피벗은 제자리를 찾고, 좌우를 재귀로 정렬한다.

```
  [5, 2, 9, 1, 6, 3]   피벗=3
   → 분할: [2, 1] 3 [5, 9, 6]     (3은 제자리 확정)
             ↓            ↓
          재귀 정렬     재귀 정렬
```

평균 O(n log n)이고 제자리라 실무에서 매우 빠르다.
단, 피벗이 매번 한쪽으로 치우치면(이미 정렬된 입력 등) **최악 O(n²)**.

### 3.3 측정: O(n²) vs O(n log n)

`SortingAlgorithms.java` 벤치마크 (단위 ms, 값은 환경마다 다름):

| N | 버블 | 선택 | 삽입 | 병합 | 퀵 |
|---|---|---|---|---|---|
| 1,000 | 0.7 | 1.2 | 0.5 | 0.1 | 0.1 |
| 5,000 | 13.8 | 5.2 | 1.4 | 0.5 | 0.2 |
| 20,000 | 212.6 | 80.1 | 21.9 | 2.0 | 1.1 |

N을 4배(5,000→20,000) 키우면 O(n²)은 약 16배로 폭증, O(n log n)은 약 4~5배로만 는다.
**N이 커질수록 격차는 벌어진다.**

### 3.4 선택 가이드

```
  안정성이 필요한가? ──── YES ──> 병합 정렬 (또는 TimSort)
       │
       NO
       │
  메모리가 빠듯한가? ──── YES ──> 퀵 정렬 (제자리)
       │
  작거나 거의 정렬됨? ─── YES ──> 삽입 정렬
       │
       └──> 실무에선 그냥 Arrays.sort / Collections.sort (검증된 하이브리드)
```

### 3.5 왜 비교 정렬은 Ω(n log n)보다 빠를 수 없는가?

이것은 **하한(lower bound)**의 증명이다. 매우 아름답다.

- n개를 나열하는 순서(순열)의 경우의 수는 **n!** 가지다.
- 비교 한 번은 결과가 두 갈래(예/아니오)뿐이다.
- k번 비교로 구별할 수 있는 경우의 수는 최대 **2ᵏ**.
- 모든 순서를 구별하려면 **2ᵏ ≥ n!** 이어야 한다.
- 양변에 log를 취하면 **k ≥ log₂(n!) ≈ n log n** (스털링 근사).

즉, **비교만으로** 정렬하는 그 어떤 알고리즘도 최악의 경우 n log n번의 비교가 필요하다.
그래서 "비교 정렬의 하한은 Ω(n log n)"이다.

> **반전:** 그럼 O(n) 정렬은 불가능한가? 가능하다 — 단, "비교"를 안 할 때.
> 계수 정렬(counting sort), 기수 정렬(radix sort)은 값을 인덱스로 쓰거나 자릿수로 나눠
> 비교 없이 O(n)에 정렬한다. 대신 값의 범위 등 특수 조건이 필요하다.

**Java 연결고리:** `Arrays.sort`는 기본형(int 등)엔 Dual-Pivot 퀵 정렬,
객체엔 **TimSort**(병합+삽입 하이브리드, 안정 정렬)를 쓴다.

## Chapter 4: 재귀와 동적 계획법(DP)

> 실행 예제: `RecursionAndDP.java`

### 4.1 재귀: 함수가 자기 자신을 부른다

**재귀(recursion)**란 함수가 자기 자신을 호출하는 것이다.
큰 문제를 "같은 모양의 더 작은 문제"로 쪼갤 수 있으면 코드가 놀랍도록 간결해진다.

```java
static long factorial(int n) {
    if (n <= 1) return 1;          // ① 기저 조건 (base case): 재귀의 바닥
    return n * factorial(n - 1);   // ② 재귀 호출: 더 작은 문제로
}
```

모든 재귀에는 **반드시 두 가지**가 있어야 한다:
1. **기저 조건(base case)**: 더 이상 자기를 안 부르고 답을 반환하는 경우.
2. **재귀 단계**: 자기보다 **작은** 입력으로 자기를 부르는 경우 (바닥을 향해 가야 함).

### 4.2 콜 스택과 StackOverflowError

`docs/JAVA_개념서.md` §1.4에서 배운 **스택(Stack) 메모리**를 기억하는가?
함수를 호출할 때마다 스택에 **스택 프레임**이 쌓인다.

```
  factorial(3) 호출 중...

  ┌─────────────────┐
  │ factorial(1)    │  ← 맨 위 (기저 조건, 곧 반환)
  ├─────────────────┤
  │ factorial(2)    │
  ├─────────────────┤
  │ factorial(3)    │
  ├─────────────────┤
  │ main            │
  └─────────────────┘
```

기저 조건에 닿으면 위에서부터 하나씩 **반환하며 스택이 풀린다**.

> **함정:** 기저 조건이 없거나 재귀가 너무 깊으면 스택이 넘쳐
> **StackOverflowError**가 난다. `RecursionAndDP.java`는 실제로 재귀를 계속 파고들어
> 이 JVM에서 **약 45,000 깊이** 근처에서 스택이 터지는 것을 측정해 보여준다.

### 4.3 분할 정복 (Divide and Conquer)

재귀의 강력한 응용이 **분할 정복**이다: 문제를 여러 조각으로 나눠 각각 재귀로 풀고, 합친다.

- **병합 정렬**(§3.2): 반으로 나눠 각각 정렬 후 병합.
- **퀵 정렬**(§3.2): 피벗으로 나눠 각각 정렬.
- **이진 탐색**(§2.2): 절반을 버리는 것도 일종의 분할.

### 4.4 재귀의 함정 → DP: 피보나치

피보나치를 재귀로 쓰면 우아하다:

```java
static long fibNaive(int n) {
    if (n < 2) return n;
    return fibNaive(n - 1) + fibNaive(n - 2);
}
```

하지만 이것은 **재앙**이다. 같은 값을 지수적으로 다시 계산하기 때문이다.

```
              fib(5)
            /        \
        fib(4)      fib(3)       ← fib(3)이 벌써 두 번!
        /    \       /    \
   fib(3)  fib(2) fib(2) fib(1)  ← fib(3), fib(2) 중복 폭발
   ...
```

이 "**겹치는 부분 문제(overlapping subproblems)**"가 지수 폭발의 원인이다.

**측정** (`RecursionAndDP.java`):

| n | 순진한 재귀(호출 수) | 메모(호출 수) |
|---|---|---|
| 10 | 177 | 19 |
| 20 | 21,891 | 39 |
| 30 | 2,692,537 | 59 |
| 40 | **331,160,281** | 79 |

n=40에서 순진한 재귀는 **3억 3천만 번** 호출한다. 순진한 재귀는 **O(2ⁿ)** 이다.

### 4.5 동적 계획법(DP): 저장하고 재사용하기

해결책은 간단하다: **한 번 계산한 답을 저장(memo)해 두고 재사용하라.**
이것이 **동적 계획법(Dynamic Programming, DP)** 이다. 두 가지 방식이 있다.

**① 메모이제이션 (Memoization) — 위에서 아래로 (top-down)**
재귀는 그대로 두되, 계산한 값을 캐시에 저장한다.

```java
static long fibMemo(int n, long[] memo) {
    if (n < 2) return n;
    if (memo[n] != -1) return memo[n];   // 캐시 적중 → 재계산 회피!
    return memo[n] = fibMemo(n-1, memo) + fibMemo(n-2, memo);
}
```

**② 타뷸레이션 (Tabulation) — 아래에서 위로 (bottom-up)**
작은 문제부터 표를 채워 올라간다. 재귀가 없어 **스택 안전**하다.

```java
static long fibTab(int n) {
    long[] dp = new long[n + 1];
    dp[0] = 0; dp[1] = 1;
    for (int i = 2; i <= n; i++) dp[i] = dp[i-1] + dp[i-2];
    return dp[n];
}
```

둘 다 **O(n)**. 같은 문제인데 O(2ⁿ) → O(n)! **이것이 DP의 위력이다.**

### 4.6 0/1 배낭 문제: DP의 대표 최적화 문제

**문제:** 무게 한도 W인 배낭에, 각 물건을 넣거나(1) 안 넣거나(0) 해서 가치 합을 최대로.

**점화식:** `dp[i][w]` = 앞 i개 물건, 용량 w일 때의 최대 가치.

```
  물건 i를 안 넣는다 → dp[i-1][w]              (이전 상태 그대로)
  물건 i를 넣는다    → dp[i-1][w - 무게ᵢ] + 가치ᵢ  (넣을 수 있을 때만)

  dp[i][w] = 둘 중 더 큰 값
```

```java
for (int i = 1; i <= n; i++)
    for (int cap = 0; cap <= W; cap++) {
        dp[i][cap] = dp[i-1][cap];                        // 안 넣음
        if (w[i-1] <= cap)                                // 넣을 수 있으면
            dp[i][cap] = Math.max(dp[i][cap],
                         dp[i-1][cap - w[i-1]] + v[i-1]); // 넣은 경우와 비교
    }
```

시간/공간 **O(n·W)**.

### 4.7 그리디 vs DP

**그리디(greedy)**는 매 단계 "지금 당장 최선"을 고른다. 빠르지만 **항상 최적은 아니다.**

`RecursionAndDP.java`가 보여주는 0/1 배낭 실패 사례 (W=50):

| 물건 | 무게 | 가치 | 비율 |
|---|---|---|---|
| A | 10 | 60 | 6.0 |
| B | 20 | 100 | 5.0 |
| C | 30 | 120 | 4.0 |

- **그리디**(비율 큰 순): A(60)+B(100) = **160**, 남은 용량 20으론 C 못 담음.
- **DP**: B(100)+C(120) = **220**. 정답!

"지금 당장 최선(A)"이 전체 최선을 망쳤다. 이것이 그리디가 실패하는 전형이다.

```
  그리디: 빠름 O(...), 단 '탐욕 선택 속성'이 성립할 때만 최적
          (동전 거스름돈, 최소 신장 트리 등)
  DP    : 최적 보장, 단 비용 O(n·W) 등. '겹치는 부분문제 + 최적 부분구조'가 조건
```

## Chapter 5: 자료구조 직접 만들기 (1)

> 실행 예제: `DataStructuresFromScratch.java`

당신은 `ArrayList`, `LinkedList`, 스택, 큐를 **써봤다**. 이제 라이브러리 없이
**바닥부터** 직접 만들어, Big-O가 왜 그렇게 나오는지 코드로 확인한다.

### 5.1 동적 배열 (Dynamic Array) = ArrayList의 뼈대

내부는 **고정 크기 배열**이다. 꽉 차면 2배 큰 배열로 "복사 이사"한다(§1.5 분할 상환).

```
  data = [10, 20, _, _]   size=2, capacity=4
  add(30) → [10, 20, 30, _]   size=3
  add(40) → [10, 20, 30, 40]  size=4  (꽉 참!)
  add(50) → 용량 2배로 확장! 새 배열 [_,_,_,_,_,_,_,_] 에 복사 후 삽입
```

| 연산 | 복잡도 | 이유 |
|---|---|---|
| `get/set(i)` | **O(1)** | 배열은 인덱스로 즉시 접근 |
| `add(끝)` | **평균 O(1)** | 가끔 확장 O(n), 분할 상환하면 O(1) |
| `insert(중간)` / `removeAt(중간)` | **O(n)** | 뒤 원소들을 밀거나 당김 |

`DataStructuresFromScratch.java`를 실행하면 용량이 **2→4→8**로 확장되는 순간을 직접 출력한다.

### 5.2 단일 연결 리스트 = LinkedList의 뼈대

노드가 "**값 + 다음 노드 참조**"로 이어진 사슬이다. 배열과 달리 **연속 메모리가 아니다.**

```
  head → [5|•]→ [10|•]→ [20|•]→ [30|null]
         값 next  값 next  ...
```

| 연산 | 복잡도 | 이유 |
|---|---|---|
| `addFirst` / `removeFirst` | **O(1)** | head 참조만 바꿈 |
| `addLast` | **O(1)** | tail 포인터 유지 시 |
| `get(i)` | **O(n)** | head부터 화살표를 i번 따라가야 함 |

> **핵심 대비:** 배열은 **인덱스 랜덤 접근**이 강하고(O(1)),
> 연결 리스트는 **앞/끝 삽입·삭제**가 강하다(O(1)). 무엇을 자주 하느냐로 고른다.
> (`docs/JAVA_개념서.md` §5.3의 "ArrayList vs LinkedList"가 바로 이 이야기다.)

### 5.3 스택 (Stack) — LIFO

"접시 쌓기". 위에서만 넣고(push) 위에서만 뺀다(pop). **Last In, First Out.**
연결 리스트의 head를 top으로 쓰면 push/pop 모두 **O(1)**.

```
  push(1) push(2) push(3)      pop() → 3
  ┌───┐                        (마지막에 넣은 게 먼저 나옴)
  │ 3 │ ← top
  ├───┤
  │ 2 │
  ├───┤
  │ 1 │
  └───┘
```

**쓰임:** 함수 콜 스택(§4.2), 실행 취소(undo), 괄호 짝 검사, DFS(§7.2).

### 5.4 큐 (Queue) — FIFO

"매표소 줄서기". 뒤로 들어와(enqueue) 앞에서 나간다(dequeue). **First In, First Out.**
head/tail 포인터를 둔 연결 리스트로 둘 다 **O(1)**.

```
  enqueue: 뒤 →  [1][2][3]  → dequeue: 앞
  (1이 먼저 들어왔으니 1이 먼저 나감)
```

**쓰임:** 작업 대기열, **BFS**(§7.2), 프린터 스풀.

> 스택(LIFO)과 큐(FIFO)는 "어느 쪽에서 빼느냐"만 다르다. 이 작은 차이가
> 그래프 탐색에서 DFS냐 BFS냐를 가른다.

## Chapter 6: 트리와 힙

> 실행 예제: `TreeAndHeap.java`

드디어 `TreeMap`과 `PriorityQueue`의 정체를 밝힌다.

### 6.1 이진 탐색 트리 (Binary Search Tree, BST)

"**왼쪽 < 부모 < 오른쪽**" 규칙으로 값을 정리한 트리다.

```
            50
          /    \
        30      70          규칙: 모든 노드에서
       /  \    /  \         왼쪽 서브트리 < 자기 < 오른쪽 서브트리
      20  40  60  80
```

**탐색:** 값과 비교해 왼/오 **한쪽으로만** 내려간다. 이진 탐색(§2.2)과 똑같은 원리!

```
  contains(60)? 50에서 시작 → 60>50 오른쪽 → 70에서 60<70 왼쪽 → 60 발견 (3번)
```

**중위 순회(inorder): 왼쪽 → 자기 → 오른쪽** 순서로 방문하면 **정렬된 순서**가 나온다!

```java
inorder(node.left);   // 먼저 왼쪽(더 작은 값들)
visit(node.value);    // 그 다음 자기
inorder(node.right);  // 마지막 오른쪽(더 큰 값들)
// → 20, 30, 40, 50, 60, 70, 80  (자동 정렬!)
```

| 연산 | 복잡도 |
|---|---|
| 탐색 / 삽입 | **O(트리 높이)** |

균형 잡히면 높이 ≈ log n → **O(log n)**.

### 6.2 BST의 함정: 퇴화 (Degeneration)

정렬된 값을 **순서대로** 넣으면 한쪽으로만 자란다. → 사실상 연결 리스트!

```
  1, 2, 3, 4를 순서대로 삽입:
      1
       \
        2
         \
          3        높이 = N → 탐색이 O(N)으로 퇴화!
           \
            4
```

**측정** (`TreeAndHeap.java`):

| N | 균형 BST 높이 | 퇴화 BST 높이 |
|---|---|---|
| 7 | 3 | 7 |
| 31 | 5 | 31 |
| 63 | 6 | 63 |

균형이면 높이 ≈ log₂(N), 퇴화하면 높이 = N.

> **그래서 `java.util.TreeMap` / `TreeSet`은** 스스로 균형을 맞추는
> **레드-블랙 트리(Red-Black Tree)**를 써서 **항상 O(log n)을 보장**한다.
> (`docs/JAVA_개념서.md`에서 TreeMap이 "정렬을 유지한다"고 한 것의 정체가 이것이다.)
> `HashMap`도 버킷 충돌이 심하면 내부적으로 이 레드-블랙 트리로 전환(treeify)한다.

### 6.3 최소 힙 (MinHeap) = PriorityQueue

"**부모 ≤ 자식**" 규칙만 지키는 **완전 이진 트리**다.
따라서 **루트가 항상 최솟값**이다. 신기하게도 **배열 하나**로 표현한다.

```
  트리 모양            배열 표현 (0-index)
       1               [1, 3, 2, 8, 9, 5, 7]
      / \
     3   2             부모(i)의 자식 = 2i+1, 2i+2
    / \ / \            자식(i)의 부모 = (i-1)/2
   8  9 5  7
```

| 연산 | 복잡도 | 방법 |
|---|---|---|
| 최솟값 조회 `peekMin` | **O(1)** | 루트(arr[0]) |
| 삽입 `add` | **O(log n)** | 끝에 넣고 위로 올림(sift-up) |
| 최솟값 삭제 `pollMin` | **O(log n)** | 루트 제거 후 아래로 내림(sift-down) |

**삽입(sift-up):** 맨 끝에 넣고, 부모보다 작으면 부모와 교환하며 올라간다.
한 층 올라갈 때마다 절반씩 → O(log n).

**삭제(sift-down):** 루트를 꺼내고, 맨 끝 원소를 루트로 옮긴 뒤,
두 자식 중 더 작은 쪽과 교환하며 내려간다. → O(log n).

### 6.4 힙 정렬 (Heap Sort)

n개를 힙에 넣고(각 O(log n)) 하나씩 최솟값을 빼면(각 O(log n)) 정렬된다.
전체 **O(n log n)**. `TreeAndHeap.java`에 작은 구현이 있다.

**Java 연결고리:** `PriorityQueue`가 바로 이 이진 힙이다. 다익스트라(§7.3)의 핵심 도구다.

## Chapter 7: 그래프

> 실행 예제: `GraphAlgorithms.java`

### 7.1 그래프란? 그리고 표현법

**정점(vertex)**들이 **간선(edge)**으로 연결된 관계망이다.
지하철 노선도, SNS 친구 관계, 도로망, 웹 링크... 세상의 연결은 대부분 그래프다.

**표현 ① 인접 리스트 (Adjacency List)** — 각 정점마다 이웃 목록.

```
  0 → [1, 2]
  1 → [0, 3, 4]
  2 → [0, 4]
  ...
```
공간 **O(V+E)**. 간선이 적은(희소) 그래프에 효율적. **대부분 이걸 쓴다.**

**표현 ② 인접 행렬 (Adjacency Matrix)** — V×V 표에 연결/가중치.

```
     0  1  2
  0  0  4  1     두 정점 연결 확인이 O(1)이지만
  1  4  0  2     공간 O(V²) (희소 그래프에선 낭비)
  2  1  2  0
```

### 7.2 BFS와 DFS

**BFS (너비 우선 탐색) — 큐(FIFO) 사용**
가까운 정점부터 물결처럼 퍼진다. 가중치 없는 그래프의 **최단 경로(간선 수)**를 준다.

```
  0 --- 1 --- 3
  |     |     |          BFS(0) = [0, 1, 2, 3, 4, 5]
  2 --- 4 --- 5          (0의 이웃 → 그 다음 층 → ...)
```

```java
queue.add(start);
while (!queue.isEmpty()) {
    int cur = queue.poll();          // 가장 먼저 들어온 것부터 (FIFO)
    for (이웃 : cur의 이웃들)
        if (!방문) { 방문표시; queue.add(이웃); }
}
```

**DFS (깊이 우선 탐색) — 스택/재귀 사용**
한 방향으로 **끝까지 파고든** 뒤 되돌아온다.

```
  DFS(0) = [0, 1, 3, 5, 4, 2]   (한 길로 깊이 → 막히면 되돌아옴)
```

```java
void dfs(v) {
    방문표시(v);
    for (이웃 : v의 이웃들)
        if (!방문) dfs(이웃);   // 재귀 = 콜 스택을 씀
}
```

> **연결:** BFS는 §5.4의 **큐**, DFS는 §5.3의 **스택**(재귀 콜 스택)을 쓴다.
> 자료구조가 알고리즘을 결정한다.

### 7.3 최단 경로: 다익스트라 (Dijkstra)

**가중치가 있는** 그래프에서 한 정점부터 모든 정점까지의 최단 거리를 구한다.

핵심 아이디어: **우선순위 큐(최소 힙, §6.3)**에 `(거리, 정점)`을 넣고,
**"지금까지 가장 가까운 정점"**을 계속 꺼내 확정해 나간다.

```
        (0)
       4/   \1
      (1)   (2)              간선: 0-1(4), 0-2(1), 2-1(2),
     1/ \5  /8                     1-3(1), 1-4(5), 2-4(8)
    (3)  (4)
```

`GraphAlgorithms.java` 출력 (출발 0):

```
  0 → 0 : 0
  0 → 1 : 3    ← 직선 0-1은 4지만, 0→2(1)→1(2)=3 이 더 짧다!
  0 → 2 : 1
  0 → 3 : 4
  0 → 4 : 8
```

**핵심 관찰:** 0→1 직선은 4지만, 0→2→1로 **우회**하면 3이다.
다익스트라는 가장 가까운 정점부터 확정하며 이런 우회 최단경로를 찾아낸다.

**완화(relaxation):** "u를 거쳐 가면 더 짧아지는가?"를 확인해 거리를 갱신한다.

```java
int nd = d + edge.weight;          // u를 거쳐 이웃으로 가는 거리
if (nd < dist[edge.to]) {          // 더 짧으면 갱신
    dist[edge.to] = nd;
    pq.add(new int[]{nd, edge.to});
}
```

복잡도 **O((V+E) log V)**.

> **주의:** 다익스트라는 **음수 간선**이 있으면 틀린다("가장 가까운 것 확정" 전제가 깨짐).
> 음수 간선이 있으면 벨만-포드(Bellman-Ford) 등 다른 알고리즘이 필요하다.

## 마치며: 전공자의 사고방식

이 책을 통과했다면, 당신은 이제 다음을 할 수 있다.

- 코드를 보고 **Big-O를 유도**한다("이 이중 반복은 O(n²)이군").
- 자료구조를 선택할 때 **"어떤 연산이 자주 일어나는가"**로 판단한다.
- `ArrayList`, `TreeMap`, `PriorityQueue`가 **왜 그런 성능인지** 안다 —
  그 안에 배열, 레드-블랙 트리, 이진 힙이 들어있음을 알기 때문이다.
- 재귀의 위험(스택)과 힘(분할 정복)을 알고, 지수 시간을 **DP로 다항 시간**으로 낮춘다.
- 그래프 문제를 BFS/DFS/다익스트라로 접근한다.

이것이 "라이브러리를 쓰는 사람"에서 "라이브러리를 이해하고 만들 수 있는 사람"으로,
비전공자에서 전공자로 가는 다리다.

**다음 단계 (이 책을 넘어서):**
- 균형 트리의 구체적 구현(AVL, 레드-블랙 트리)
- 해시 함수 설계와 오픈 어드레싱
- 고급 그래프(최소 신장 트리, 위상 정렬, 강연결 요소)
- NP-완전과 근사 알고리즘

각 개념을 배울 때마다, 반드시 **직접 구현하고 측정**하라.
"안다"와 "만들 수 있다" 사이의 거리는, 오직 손으로 짜본 코드만이 메운다.
