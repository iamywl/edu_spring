> 대상: 비전공자로 시작해 전공자(CS-major) 수준의 감각을 갖추려는 사람
> 짝이 되는 실습 코드: `chapter-cs-concurrency/` (아래 각 절에서 "▶ 데모"로 연결)

## 들어가며

### 이 책이 답하려는 질문

당신은 이미 `JAVA_개념서.md`에서 **"프로그램 하나가 어떻게 실행되는가"**를 배웠다.
스택, 힙, 바이트코드, JIT, GC — 전부 **스레드 한 개짜리** 이야기였다.

그런데 실무의 서버는 그렇지 않다. `SPRING_개념서.md §2.7 "서버는 멀티스레드다"`에서 봤듯,
톰캣은 요청마다 워커 스레드를 하나씩 붙여 **수백 개의 스레드가 동시에** 같은 코드를,
심지어 같은 객체를 실행한다.

바로 여기서 세상에서 가장 재현하기 어려운 버그들이 태어난다.

- 평소엔 멀쩡하다가 트래픽이 몰릴 때만 남의 데이터가 튀어나온다.
- 숫자를 100번 더했는데 결과가 73이다.
- 서버가 아무 에러 없이 그냥 "멈춘다".

이 책은 **"왜 이런 일이 벌어지는가, 그리고 무엇으로 막는가"**를 그림과 실행되는 코드로 설명한다.
외우는 게 아니라 **눈으로 보고 이해**하게 하는 것이 목표다.

### 왜 운영체제부터 시작하는가?

동시성 버그의 근본 원인은 자바가 아니라 **운영체제(OS)와 CPU의 동작 방식**에 있다.
"스레드를 언제 멈추고 다른 스레드로 바꿀지"는 OS 스케줄러가 정하고,
"count++가 왜 쪼개지는지"는 CPU 명령어 수준의 이야기다.
그래서 먼저 OS를 얕게라도 알아야 자바의 `synchronized`, `volatile`, `Atomic`이 **무엇을 고치는지** 보인다.

## Chapter 1: 프로세스와 스레드

### 1.1 프로세스: 실행 중인 프로그램

**프로그램**은 디스크에 저장된 죽어있는 파일(`.class`, `.exe`)이다.
이것을 실행하면 OS가 메모리에 올리고 CPU를 배정한다. 이 **"실행 중인 프로그램 인스턴스"**가 프로세스다.

OS는 프로세스마다 **독립된 주소 공간(address space)**을 준다.

```
  ┌─────────── 프로세스 A ───────────┐   ┌─────────── 프로세스 B ───────────┐
  │  코드 | 데이터 | 힙 | 스택        │   │  코드 | 데이터 | 힙 | 스택        │
  │  주소 0x0000 ~ 0xFFFF (A만의 것)  │   │  주소 0x0000 ~ 0xFFFF (B만의 것)  │
  └──────────────────────────────────┘   └──────────────────────────────────┘
        서로의 메모리를 볼 수 없다 (OS가 격리한다)
```

> **왜?** 격리하는가?
>
> 브라우저가 죽어도 음악 앱은 멀쩡해야 한다. 한 프로그램의 버그가 다른 프로그램의
> 메모리를 망가뜨리면 안 되기 때문이다. 프로세스 간 격리는 **안정성과 보안의 기본**이다.
> (재미있게도 A의 `0x1000`과 B의 `0x1000`은 실제로는 물리 메모리의 다른 위치다.
>  이 마법은 §7 가상 메모리에서 설명한다.)

### 1.2 스레드: 프로세스 안의 실행 흐름

프로세스가 "집"이라면, **스레드(thread)**는 그 집 안에서 일하는 "일꾼"이다.
한 프로세스 안에 스레드가 여러 개일 수 있고, 이들은 **같은 집(주소 공간)을 공유**한다.

이것이 핵심이다. **무엇을 공유하고 무엇을 따로 갖는가:**

```
  ┌──────────────────── 하나의 프로세스 (JVM) ────────────────────┐
  │                                                                │
  │   [공유]  코드(Method Area) · 힙(Heap, 모든 객체)              │  ← 모든 스레드가 함께 본다
  │            ┌──────┐ ┌──────┐ ┌──────┐                         │
  │            │객체 A│ │객체 B│ │static│                         │
  │            └──────┘ └──────┘ └──────┘                         │
  │                                                                │
  │   [따로]  스레드마다 자기만의 스택 + 레지스터(PC 등)          │  ← 절대 안 섞인다
  │   ┌─ Thread-1 ─┐   ┌─ Thread-2 ─┐   ┌─ Thread-3 ─┐          │
  │   │ 스택        │   │ 스택        │   │ 스택        │          │
  │   │ (지역변수)  │   │ (지역변수)  │   │ (지역변수)  │          │
  │   │ 프로그램    │   │ 프로그램    │   │ 프로그램    │          │
  │   │ 카운터(PC)  │   │ 카운터(PC)  │   │ 카운터(PC)  │          │
  │   └────────────┘   └────────────┘   └────────────┘          │
  └────────────────────────────────────────────────────────────────┘
```

- **공유(shared)**: 힙(객체들), 코드, static 변수, 열린 파일 등.
- **전용(private)**: 각 스레드의 **스택**(그래서 지역 변수·매개변수는 안 섞인다)과
  **레지스터**(특히 다음 실행할 명령을 가리키는 **프로그램 카운터 PC**).

> **여기서 모든 동시성 문제가 나온다.**
> 지역 변수는 스택에 있어 스레드마다 따로다 → 안전하다.
> 하지만 **힙의 객체나 static 필드는 공유된다** → 여러 스레드가 동시에 건드리면 위험하다.
> `SPRING_개념서 §2.7`에서 "Service는 stateless여야 안전하다"고 한 이유가 정확히 이것이다.
> 필드(공유)에 요청별 데이터를 담으면 섞이고, 지역 변수(전용)에 담으면 안 섞인다.

### 1.3 문맥 교환(Context Switch)과 "스레드는 왜 싸긴 싼가"

CPU 코어 하나는 한 순간에 스레드 하나만 실행한다.
그럼 어떻게 여러 스레드가 "동시에" 도는 것처럼 보일까?
OS 스케줄러가 스레드들을 **아주 빠르게 번갈아** 실행하기 때문이다.

한 스레드를 잠시 멈추고 다른 스레드로 바꾸는 것을 **문맥 교환(context switch)**이라 한다.
멈출 때 그 스레드의 레지스터(PC 등)를 저장하고, 재개할 스레드의 레지스터를 복원한다.

```
  시간 →
  CPU:  [T1][T2][T1][T3][T2][T1] ...   각 조각이 수 밀리초(time slice)
         ↑저장/복원(문맥 교환) 비용이 매 전환마다 발생
```

> **왜?** 스레드가 프로세스보다 "싸다"고 하는가?
>
> - **생성 비용**: 새 프로세스는 주소 공간 전체를 새로 만들어야 한다. 스레드는 기존 주소 공간을
>   공유하므로 스택만 새로 잡으면 된다 → 훨씬 가볍다.
> - **문맥 교환 비용**: 같은 프로세스 내 스레드끼리는 주소 공간(페이지 테이블 등)이 같아서
>   교환 시 갈아끼울 것이 적다. 프로세스 간 전환은 더 무겁다.
> - **통신 비용**: 스레드는 힙을 공유하니 그냥 같은 객체를 읽으면 된다. 프로세스 간 통신(IPC)은
>   별도 채널(파이프, 소켓)이 필요해 번거롭다.
>
> "싸긴 싸다"고 조심스럽게 말하는 이유: 스레드도 공짜는 아니다. 스택 메모리(보통 ~1MB),
> 커널의 스케줄링 등록 비용이 든다. 그래서 스레드 수천 개를 매번 만들면 서버가 죽는다.
> → 그래서 **스레드 풀**을 쓴다(§6). ▶ 데모: `ExecutorAndFutures.java`

## Chapter 2: 동시성 vs 병렬성, 그리고 비결정성

### 2.1 동시성(Concurrency) ≠ 병렬성(Parallelism)

- **동시성**: 여러 작업을 **번갈아** 진행해서 "동시에 진행되는 것처럼" 다루는 것.
  코어가 1개여도 성립한다(시분할).
- **병렬성**: 여러 작업이 **물리적으로 진짜 동시에** 실행되는 것. 코어가 여러 개여야 성립한다.

```
  동시성(코어 1개, 시분할)         병렬성(코어 2개, 진짜 동시)
  코어: [A][B][A][B][A][B]         코어1: [A][A][A][A]
        빠르게 번갈아                코어2: [B][B][B][B]
```

> 유명한 비유: **동시성은 한 명의 요리사가 여러 냄비를 오가며 돌보는 것,
> 병렬성은 요리사가 여러 명 있어 각자 한 냄비씩 맡는 것.**

핵심은 이것이다. **동시성이든 병렬성이든, 공유 데이터를 건드리면 똑같이 위험하다.**
코어가 1개(동시성만)여도, 문맥 교환이 하필 나쁜 순간에 일어나면 데이터가 깨진다.

### 2.2 스케줄러와 비결정성(Nondeterminism)

**언제 문맥 교환이 일어나는가?** 당신은 모른다. OS 스케줄러가 정한다. 그리고 그 타이밍은
CPU 부하, 다른 프로세스, 인터럽트 등에 따라 **매 실행마다 달라진다.**

그래서 멀티스레드 프로그램은 **비결정적**이다. 같은 코드를 같은 입력으로 돌려도
스레드들이 끼어드는 순서가 매번 다르다.

> **왜?** 이게 무서운가?
>
> 버그가 "가끔만" 난다. 테스트에선 통과하고, 배포하면 트래픽 몰릴 때만 터진다.
> 재현이 안 되니 디버깅이 지옥이다. 그래서 동시성은 "운으로 맞기를 바라는" 게 아니라
> **어떤 순서로 끼어들어도 안전하도록** 설계해야 한다.
>
> ▶ 데모: `RaceConditionDemo.java` — 같은 프로그램을 5번 돌리면 결과가 매번 다르게 나온다.

## Chapter 3: 핵심 문제 — 경쟁 상태와 임계 구역

### 3.1 count++ 는 원자적이지 않다

가장 유명한 동시성 버그를 보자. 여러 스레드가 공유 카운터를 증가시킨다.

```java
static long count = 0;
// 스레드 8개가 각각 100,000번씩 실행
count++;   // 기대: 800,000
```

결과는 800,000보다 **작게** 나온다. 왜? `count++`가 한 줄처럼 보이지만
**실제로는 세 단계**이기 때문이다. `JAVA_개념서 §1.5`에서 배운 바이트코드로 열어보면:

```
  count++  ═══컴파일═══▶   ① getstatic count   // 힙에서 현재 값을 읽어 operand stack에 올림 (read)
                          ② ladd (with 1)      // 스택 위에서 +1                        (modify)
                          ③ putstatic count    // 결과를 다시 힙에 저장                 (write)
```

이 **read-modify-write** 세 단계 "사이"에 다른 스레드가 끼어들 수 있다.

```
  count = 100 인 상태에서 Thread A, B가 거의 동시에 count++:

  시간 →
  A:  read(100) ──────────────────── +1=101 ── write(101)
  B:        read(100) ── +1=101 ── write(101)
                                                  ▲
              둘 다 100을 읽음 → 둘 다 101을 씀 → 101 (102가 아니라!)
              두 번 더했는데 한 번만 오름. 갱신 하나가 "증발"했다.
```

이것을 **잃어버린 갱신(lost update)**이라 하고, 이런 버그가 생길 수 있는 상태를 **경쟁 상태(race condition)**라 한다.

### 3.2 임계 구역(Critical Section)

**임계 구역**이란 "한 번에 한 스레드만 실행해야 안전한 코드 조각"이다.
위 예에서 `count++`의 read-modify-write 전체가 임계 구역이다.

```
  임계 구역 = ┌───────────────────────┐
             │ read → modify → write │  ← 이 3단계는 통째로, 방해받지 않고 끝나야 한다
             └───────────────────────┘
```

**동시성 해결의 본질**은 이것이다: **임계 구역을 "쪼개지지 않게(atomic)" 만들어라.**
어떻게? 다음 챕터의 동기화 도구들로.

> ▶ 데모: `RaceConditionDemo.java` — 실제로 결과가 800,000보다 작게, 매번 다르게 나오는 걸 보여준다.
> 대조군으로 단일 스레드는 항상 정답이 나오는 것도 함께 보여준다(경쟁 자체가 없으므로).

## Chapter 4: 동기화(Synchronization) — 무엇이 무엇을 고치는가

동시성 도구들은 "다 비슷한 거 아냐?" 싶지만, 각자 **고치는 문제가 다르다.**
특히 **원자성(atomicity)**과 **가시성(visibility)**을 구분하는 것이 전공자 수준의 핵심이다.

- **원자성**: read-modify-write가 쪼개지지 않는가? (§3의 문제)
- **가시성**: 한 스레드가 쓴 값을 다른 스레드가 즉시 볼 수 있는가? (§5의 문제)

### 4.1 뮤텍스/락과 `synchronized` (모니터)

가장 직관적인 해법: **임계 구역에 자물쇠를 채운다.** 한 스레드가 자물쇠(락, lock)를 쥐면,
다른 스레드는 그것이 풀릴 때까지 문 앞에서 기다린다. 이런 "한 번에 하나만" 자물쇠를 **뮤텍스(mutex)**라 한다.

자바는 모든 객체에 **모니터(monitor)**라는 내장 락을 심어놨고, `synchronized`로 쓴다.

```java
synchronized (lock) {   // lock 객체의 모니터를 획득 (없으면 대기)
    count++;            // 임계 구역: 한 번에 한 스레드만 여기 들어옴
}                       // 블록을 벗어나면 모니터 해제 (예외로 벗어나도 자동 해제)
```

```
  Thread A ── 락 획득 ──▶ [ count++ ] ── 락 해제 ──▶
  Thread B ── 락 대기 ─────────(문 앞에서 기다림)────▶ [ count++ ] ...
```

- **고치는 것**: 원자성 O, 가시성 O (락 해제 시점에 변경 내용이 다른 스레드에 보이도록 보장).
- **대가**: 대기하는 스레드는 논다(경합이 심하면 느려진다).

### 4.2 원자 변수(Atomic)와 CAS

락은 "막고 기다리는" 방식이다. 반대로 **락 없이(lock-free)** 원자성을 얻는 방법이 있다:
**CAS(Compare-And-Swap)**. CPU가 제공하는 원자적 명령이다.

> "메모리의 값이 **아직 V이면** V+1로 **바꿔라**. 그 사이 누가 바꿔서 V가 아니면 **실패**를 알려라."

읽기·비교·쓰기가 CPU 명령 하나로 원자적으로 처리된다. 실패하면? **다시 시도**한다(재시도 루프).

```java
AtomicLong count = new AtomicLong(0);
count.incrementAndGet();   // 내부적으로 CAS 재시도 루프
```

```
  incrementAndGet() 내부:
  ┌─▶ v = 현재값 읽기
  │   새값 = v + 1
  │   CAS(기대=v, 새값): 그 사이 안 바뀌었으면 성공 ──▶ 끝
  └── 바뀌었으면 실패 → 처음부터 재시도
```

- **고치는 것**: 단일 변수의 원자성 O, 가시성 O.
- **장점**: 락처럼 "재우고 깨우는" 비용이 없어 경합이 낮으면 매우 빠르다.
- **한계**: **하나의 변수**에만 잘 맞는다. 여러 변수를 함께 원자적으로 바꿔야 하면 락이 낫다.

### 4.3 `java.util.concurrent`의 명시적 락 (`ReentrantLock`)

`synchronized`는 편하지만 유연성이 없다(블록을 벗어날 때만 풀림, 타임아웃 불가).
`ReentrantLock`은 `lock()`/`unlock()`을 **직접** 호출하는 명시적 락이다.

```java
lock.lock();
try {
    count++;          // 임계 구역
} finally {
    lock.unlock();    // 반드시 finally에서! (예외가 나도 풀리도록)
}
```

- `synchronized`와 같은 효과(원자성·가시성) + 추가 기능: `tryLock()`(못 잡으면 포기),
  타임아웃, 공정성(먼저 온 순서대로), 여러 조건 변수 등.
- 대가: `unlock()`을 직접 해야 해서 실수하면 락이 안 풀린다(그래서 `finally` 필수).

### 4.4 `volatile`: 가시성은 고치지만 원자성은 못 고친다 (가장 흔한 함정)

`volatile`은 필드에 붙이는 키워드다. **"이 변수는 항상 메인 메모리에서 최신 값을 읽고, 즉시 쓴다"**를 보장한다.
즉 **가시성**을 고친다(§5에서 왜 필요한지 설명). **하지만 원자성은 고치지 못한다.**

```java
volatile long count = 0;
count++;   // 여전히 위험! read-modify-write 3단계는 그대로 쪼개진다.
```

`volatile`은 각 단계에서 "최신 값을 보게"는 해주지만, 세 단계가 **통째로** 안 쪼개지게 하지는 못한다.
그래서 `count++`는 여전히 갱신을 잃는다.

```
  volatile이 고치는 것:  값이 "보이는가"      → O (가시성)
  volatile이 못 고치는 것: 3단계가 "안 쪼개지는가" → X (원자성)
```

> **한 문장 요약**: **가시성 ≠ 원자성.** `volatile`은 플래그(boolean 하나 읽고 쓰기)엔 완벽하지만,
> `count++` 같은 복합 연산엔 부족하다. 그럴 땐 `synchronized`/`Atomic`/`Lock`을 써라.

### 4.5 정리 표

| 도구            | 원자성 | 가시성 | 특징 / 언제 쓰나 |
|-----------------|:------:|:------:|------------------|
| `volatile`      |   ✗    |   ✓    | 상태 플래그(예: `stop` boolean). 복합 연산엔 부족 |
| `synchronized`  |   ✓    |   ✓    | 가장 단순. 임계 구역 보호의 기본 |
| `AtomicXxx`(CAS)|   ✓    |   ✓    | 단일 변수 카운터에 최적. lock-free로 빠름 |
| `ReentrantLock` |   ✓    |   ✓    | 타임아웃/tryLock/공정성 필요할 때 |

> ▶ 데모: `SynchronizationDemo.java` — 같은 카운터를 세 방법으로 고쳐 셋 다 정답이 나오는 것,
> 그리고 `volatile`만 쓰면 **여전히 틀리는 것**을 나란히 보여준다.

## Chapter 5: 메모리 가시성과 자바 메모리 모델(JMM)

### 5.1 "쓴 값이 왜 안 보이는가"

지금까지는 "3단계가 쪼개진다"(원자성)만 얘기했다. 그런데 더 미묘한 문제가 있다.
**한 스레드가 변수에 쓴 값을, 다른 스레드가 영영 못 볼 수도 있다.**

왜? 성능 때문에 CPU와 컴파일러가 다음을 한다:

- 각 CPU 코어는 자기 **캐시**를 갖는다. 스레드가 변수를 쓰면 일단 자기 캐시에만 쓰고,
  메인 메모리로 내려가는 것은 나중일 수 있다. → 다른 코어의 스레드는 **낡은 값**을 본다.
- 컴파일러/CPU가 **명령 순서를 재배치(reordering)**한다. 단일 스레드에선 결과가 같지만,
  다른 스레드가 보면 순서가 뒤집혀 보인다.

```
  코어 1 (Thread A)          메인 메모리          코어 2 (Thread B)
  ┌──────────┐              ┌──────────┐        ┌──────────┐
  │ 캐시      │ ──느리게──▶  │ ready=?  │ ◀──?── │ 캐시      │
  │ ready=true│              └──────────┘        │ ready=?  │
  └──────────┘                                    └──────────┘
   A가 true로 썼지만 캐시에만 있으면, B는 아직 false를 본다 → 무한 대기 가능
```

### 5.2 happens-before: "이 규칙 안에서는 보이는 걸 보장한다"

이 혼돈에 질서를 주는 규약이 **자바 메모리 모델(Java Memory Model, JMM)**이고,
그 핵심 개념이 **happens-before(선행 발생)** 관계다.

직관적으로: **"연산 A가 연산 B보다 happens-before이면, A의 결과(쓴 값)는 B에게 반드시 보인다."**

happens-before가 성립하는 대표적인 경우:

- **락**: 어떤 락을 **해제(unlock)** 한 것은, 그 락을 다음에 **획득(lock)** 하는 것보다 happens-before.
  → `synchronized` 블록 안에서 쓴 값은, 다음에 그 락을 잡은 스레드에게 보인다.
- **volatile**: `volatile` 변수에 **쓴** 것은, 그 변수를 다음에 **읽는** 것보다 happens-before.
- **스레드 시작/종료**: `thread.start()` 이전의 모든 것은 새 스레드에게 보인다.
  `thread.join()`이 끝나면 그 스레드가 한 모든 것이 보인다.

```
  Thread A                          Thread B
  data = 42;                        │
  synchronized(lock){ ready=true; } │  ← unlock
        ─────── happens-before ─────▶ synchronized(lock){ if(ready) ... }  ← lock
                                      여기서 data==42, ready==true 가 보장된다
```

> **왜?** JMM이라는 규칙이 필요한가?
>
> "재배치·캐시를 절대 하지 마라"고 하면 성능이 폭락한다. 그래서 JMM은 **"대부분은 자유롭게 최적화하되,
> happens-before 경계에서는 반드시 보이게 하라"**는 절충안이다. `synchronized`/`volatile`/`Atomic`은
> 이 경계를 만들어준다. 그래서 이들을 쓰면 원자성뿐 아니라 **가시성까지** 공짜로 따라온다.

## Chapter 6: 교착 상태(Deadlock)

### 6.1 서로가 서로를 기다리는 정지

락은 문제를 풀지만 새 문제를 만든다. 두 스레드가 **두 개의 락을 반대 순서로** 잡으면?

```
  Thread-1: 락 A 잡음 ──▶ 락 B를 원함
  Thread-2: 락 B 잡음 ──▶ 락 A를 원함

     Thread-1 ──잡음──▶ [락 A] ◀──원함── Thread-2
         │                                  ▲
         └──원함──▶ [락 B] ──잡음──────────┘

  A는 B를 기다리고, B는 A를 기다린다 → 영원히 아무도 못 나아감 = 교착(Deadlock)
```

에러도 없이, 로그도 없이, 관련 스레드가 그냥 **얼어붙는다.** 서버라면 그 요청들이 응답 없이 쌓인다.

### 6.2 코프만(Coffman)의 네 가지 조건

교착은 다음 **네 가지가 모두** 성립할 때만 발생한다.

1. **상호 배제(Mutual Exclusion)**: 자원(락)은 한 번에 한 스레드만 쥔다.
2. **점유 대기(Hold and Wait)**: 자원 하나를 쥔 채로 다른 자원을 기다린다.
3. **비선점(No Preemption)**: 쥔 자원을 강제로 뺏을 수 없다.
4. **순환 대기(Circular Wait)**: 대기 관계가 원(사이클)을 이룬다.

> **핵심**: 이 넷은 **모두** 필요하다. 따라서 **하나만 깨도 교착은 예방된다.**

### 6.3 예방: 락 순서를 통일하라

가장 실용적인 방법은 **④ 순환 대기를 깨는 것**이다.
**모든 스레드가 락을 항상 같은 순서로 잡게** 하면 사이클이 생길 수 없다.

```
  규칙: 누구든 락을 잡을 땐 반드시 "A → B" 순서로.

  Thread-1: A 잡고 → B 원함
  Thread-2: A 원함 (아직 A도 못 잡음, 그래서 B도 안 건드림)
            → 사이클이 생길 수 없다 → 교착 불가능
```

> 실무 팁: 계좌 이체처럼 두 객체의 락을 잡아야 할 땐, 객체의 고유 ID(예: 계좌번호) 순으로 정렬해
> **항상 작은 ID부터** 잡는다. 그러면 모든 스레드가 동일 순서를 지켜 교착이 원천 봉쇄된다.
>
> ▶ 데모: `DeadlockDemo.java` — 반대 순서로 교착을 일부러 일으키고(감시견 타임아웃으로 감지·탈출),
> 그다음 락 순서를 통일해 교착이 사라지는 것을 보여준다. **프로그램은 절대 멈추지 않는다.**

## Chapter 7: 더 높은 수준의 도구들

락과 원자 변수는 "원재료"다. 실무에서는 이것들을 감싼 **더 안전한 고수준 도구**를 쓴다.

### 7.1 스레드 풀과 `ExecutorService`

§1.3에서 봤듯 스레드는 만들 때마다 비용이 든다. **스레드 풀**은 스레드를 미리 몇 개 만들어 두고
**재사용**한다. 우리는 스레드가 아니라 **작업(task)**만 던진다.

```
  작업들 ─▶ [작업 큐] ─▶ ┌─ 워커1 ─┐
                          ├─ 워커2 ─┤  (고정 개수를 계속 재사용)
                          └─ 워커3 ─┘
```

```java
ExecutorService pool = Executors.newFixedThreadPool(3);
Future<Integer> f = pool.submit(() -> 2 + 2);  // 작업 제출 → 영수증(Future) 즉시 반환
int result = f.get();                           // 결과가 준비될 때까지 대기 후 회수
pool.shutdown();                                // 다 쓰면 반드시 종료
```

- **`Future<T>`**: "지금은 없지만 나중에 나올 결과"를 담는 영수증. `get()`으로 회수한다.
- **왜 프레임워크가 이걸 쓰나**: `SPRING_개념서 §2.7`의 **톰캣 스레드 풀**이 바로 이것이다.
  요청이 올 때마다 스레드를 새로 만들면 트래픽에 서버가 죽으니, 풀에서 워커를 빌려 쓴다.
  DB 커넥션도 같은 이유로 **커넥션 풀(HikariCP)**로 재사용한다.

> ▶ 데모: `ExecutorAndFutures.java` — 스레드 3개 풀로 작업 9개를 처리하며 워커 이름이
> 3종류만 반복 등장(=재사용)하는 것, `Future`로 결과를 회수하는 것을 보여준다.

### 7.2 `BlockingQueue`와 생산자-소비자

데이터를 만드는 쪽(**생산자**)과 쓰는 쪽(**소비자**)의 속도가 다를 때, 둘 사이에 **버퍼(큐)**를 둔다.
`BlockingQueue`는 이 패턴을 위해 만들어진 스레드 안전한 큐로, 두 가지를 자동으로 해준다:

- 큐가 **가득 차면** `put()`하는 생산자가 자리가 날 때까지 **잠들어 대기**.
- 큐가 **비면** `take()`하는 소비자가 데이터가 올 때까지 **잠들어 대기**.

바쁜 대기(`while`로 CPU를 태우는 것)가 아니라 진짜로 스레드를 재운다.

```
  생산자 ──put()──▶ [ □ □ □ ] (크기 5, 유한 버퍼) ──take()──▶ 소비자
           가득 차면 대기          ▲ 비면 대기
```

**유한(bounded) 버퍼**가 핵심이다. 생산자가 소비자보다 빠르면 버퍼가 차고, 그러면 생산자가
자동으로 느려진다 — 이것을 **배압(back-pressure)**이라 한다. 메모리가 무한정 부풀지 않게 막아준다.

> ▶ 데모: `ProducerConsumer.java` — 크기 5 버퍼로, 빠른 생산자가 버퍼가 찰 때마다 블록되는 것과
> 만든 개수와 소비한 개수가 정확히 일치(유실·중복 없음)하는 것을 보여준다.

## Chapter 8: 곁다리 — OS 메모리 관리와 JVM 힙

동시성과 직접 관련은 적지만, "프로세스마다 독립된 주소 공간"(§1.1)의 마법을 이해하면
JVM 힙도 더 선명해진다.

### 8.1 가상 메모리(Virtual Memory)와 페이징(Paging)

§1.1에서 "프로세스 A의 `0x1000`과 B의 `0x1000`은 실제로는 다른 물리 위치"라 했다.
어떻게? **가상 메모리** 덕분이다.

- 각 프로세스는 **가상 주소**를 본다(자기가 메모리를 통째로 가진 것처럼).
- OS와 CPU(MMU)가 가상 주소를 **물리 주소**로 번역한다. 이 번역표가 **페이지 테이블**이다.
- 메모리는 **페이지(page)**라는 고정 크기(보통 4KB) 조각으로 관리된다.

```
  프로세스 A의 가상주소        페이지 테이블        물리 메모리(RAM)
     0x1000  ──────────▶  [번역]  ──────────▶   0x7A3000
  프로세스 B의 가상주소
     0x1000  ──────────▶  [번역]  ──────────▶   0x22F000   (A와 다른 곳!)
```

> **왜?** 이렇게 하는가?
>
> - **격리**: 프로세스는 자기 페이지 테이블에 없는 물리 메모리에 접근할 수 없다 → 남의 메모리 못 건드림.
> - **환상**: 물리 RAM보다 큰 메모리를 쓰는 것처럼 보이게 한다. 안 쓰는 페이지는 디스크로 내보낸다(스와핑).
> - **유연성**: 물리적으로 흩어진 페이지들을 가상 공간에선 연속된 것처럼 보이게 한다.

### 8.2 JVM 힙과의 관계

JVM의 **힙(Heap)**은 결국 OS로부터 받은 이 가상 메모리의 한 덩어리다.

- JVM은 시작할 때 OS에 큰 메모리 영역을 요청하고(`-Xmx`가 그 최대치), 그 안에서 객체를 할당한다.
- 그래서 GC(`JAVA_개념서 §1.8`)가 청소하는 "힙"도, 그 밑을 보면 결국 OS가 페이지로 관리하는 가상 메모리다.
- **실무 감각**: JVM 힙(`-Xmx`)을 물리 RAM보다 크게 잡으면, OS가 부족한 만큼 디스크로 스와핑하며
  성능이 급락한다. "JVM 힙 크기 ≤ 사용 가능한 물리 메모리"를 지켜야 하는 이유가 여기 있다.

## 마치며: 하나의 이야기로 연결하기

```
  [OS] 스레드는 스택/레지스터는 따로, 힙/코드는 공유          (§1)
        └─ 스케줄러가 문맥 교환 → 순서는 비결정적            (§2)
              └─ 공유 데이터 + count++(3단계) = 경쟁 상태     (§3)
                    ├─ 원자성 문제 → synchronized/Atomic/Lock (§4)
                    ├─ 가시성 문제 → volatile/JMM(happens-before) (§4,§5)
                    └─ 락을 잘못 쓰면 → 교착(코프만 4조건, 락 순서로 예방) (§6)
                          └─ 실무에선 고수준 도구로 감싼다
                                (스레드 풀·Future·BlockingQueue)        (§7)
                                   └─ 이것이 톰캣/스프링이 요청을 처리하는 방식
                                      (SPRING_개념서 §2.7)
```

전공자 수준의 감각이란 이 그림이 **하나의 이야기**로 보이는 것이다.
"Service를 stateless로 짜라", "싱글톤 Bean에 상태를 두지 마라"는 스프링의 조언이
왜 나왔는지 — 이제 그 밑바닥의 OS·CPU·JMM까지 내려가 설명할 수 있다면,
당신은 더 이상 "동시성은 무섭다"에서 멈추지 않는다.

### 실습 순서 (chapter-cs-concurrency)

1. `RaceConditionDemo` — 문제를 눈으로 본다(잃어버린 갱신, 비결정성).
2. `SynchronizationDemo` — 세 가지로 고치고, volatile의 함정을 본다.
3. `DeadlockDemo` — 락이 만드는 새 문제(교착)와 그 예방을 본다.
4. `ExecutorAndFutures` — 실무의 스레드 풀과 Future.
5. `ProducerConsumer` — BlockingQueue로 스레드 간 안전한 데이터 전달.

각 파일은 독립 실행 가능하며, 출력이 개념을 **증명**하도록 짜여 있다.
