## 왜 이 장이 필요한가 — 백엔드 개발자 관점

앞의 두 장은 배경이었다. 이 장이 **문제의 핵심**이다. 쇼핑몰 재고, 게시글 조회수, 은행 잔액 — 이 모든 것이
"공유 값을 여러 요청이 동시에 갱신"하는 상황이고, 여기서 **잃어버린 갱신(lost update)**이 발생한다.
"재고를 3개 깎았는데 2개만 줄었다", "동시에 결제했더니 잔액이 음수가 됐다" 같은 사고가 바로 이 장의 주제다.

이 장을 끝내면 다음을 설명할 수 있다.

- `count++`가 **왜 원자적이지 않은지** 바이트코드 3단계로.
- **경쟁 상태(race condition)**와 **잃어버린 갱신**이 무엇인지, 타임라인으로.
- **임계 구역(critical section)**이 무엇이고, 왜 이것을 "쪼개지지 않게" 만드는 것이 해결의 본질인지.

## 3.1 count++ 는 원자적이지 않다

### 개념

가장 유명한 동시성 버그를 보자. 여러 스레드가 공유 카운터를 증가시킨다.

```java
static long count = 0;
// 스레드 8개가 각각 100,000번씩 실행
count++;   // 기대: 800,000
```

결과는 800,000보다 **작게** 나온다. 그것도 매번 다른 값으로. 왜?

### 원리 — 바이트코드 3단계

`count++`가 한 줄처럼 보이지만 **실제로는 세 단계**다. [`JAVA_개념서 §1.5 바이트코드`](../JAVA_교재/심화/)에서 배운
방식대로 `javap -c`로 열어 보면, 계산이 **operand stack** 위에서 벌어지는 것을 확인할 수 있다.

```
  count++  ═══컴파일═══▶   ① getstatic count   // 힙에서 현재 값을 읽어 operand stack에 올림 (read)
                          ② ladd (with 1)      // 스택 위에서 +1                        (modify)
                          ③ putstatic count    // 결과를 다시 힙에 저장                 (write)
```

이 **read-modify-write** 세 단계는 서로 다른 CPU 명령이고, 그 "사이"에 [2장](02-동시성과-병렬성-스케줄러.md)에서
배운 문맥 교환이 언제든 끼어들 수 있다. "①과 ③ 사이"에 다른 스레드가 끼어들면 갱신이 사라진다.

```
  count = 100 인 상태에서 Thread A, B가 거의 동시에 count++:

  시간 →
  A:  read(100) ──────────────────── +1=101 ── write(101)
  B:        read(100) ── +1=101 ── write(101)
                                                  ▲
              둘 다 100을 읽음 → 둘 다 101을 씀 → 101 (102가 아니라!)
              두 번 더했는데 한 번만 오름. 갱신 하나가 "증발"했다.
```

이 스택 다이어그램으로 보면 더 선명하다.

```
  A의 operand stack        B의 operand stack       힙의 count
  ┌──────┐                 ┌──────┐               ┌──────┐
  │ 100  │ ← A가 read      │      │               │ 100  │
  ├──────┤                 ├──────┤               ├──────┤
  │ 100  │                 │ 100  │ ← B도 read    │ 100  │  (아직 아무도 안 씀)
  ├──────┤                 ├──────┤               ├──────┤
  │ 101  │ ← A가 +1        │ 101  │ ← B도 +1      │ 100  │
  ├──────┤                 ├──────┤               ├──────┤
  │      │ ← A가 write ────┼──────┼──────────────▶│ 101  │  A가 101 씀
  │      │                 │      │ ← B가 write ──▶│ 101  │  B도 101 씀 (덮어씀!)
  └──────┘                 └──────┘               └──────┘
                                    결과 101 — 하나의 +1이 사라졌다
```

### 자바 코드와 데모

[`RaceConditionDemo`](../../chapter-cs-concurrency/src/main/java/com/edu/concurrency/RaceConditionDemo.java)가
정확히 이 실험을 한다. 핵심은 공유 필드 하나를 8개 스레드가 각각 10만 번 증가시키는 것이다.

```java
static long unsafeCounter = 0;                 // 모든 스레드가 공유하는 필드
static final int THREADS = 8, INCREMENTS = 100_000;

// 각 스레드가 실행하는 부분:
for (int i = 0; i < INCREMENTS; i++) {
    unsafeCounter++;   // ← read-modify-write 3단계라서 위험
}
// 기대: 8 × 100,000 = 800,000, 그러나 실제는 그보다 작고 매번 다름
```

데모는 `CountDownLatch`로 8개 스레드를 **동시에 출발**시켜 경쟁을 극대화하고(그래야 현상이 잘 보인다),
같은 실험을 5번 반복해 매번 다른 값이 나오는 것을 보여준다. 대조군으로 **단일 스레드**로 같은 총량을 증가시키면
경쟁 자체가 없으므로 항상 정답이 나온다.

> ▶ 데모: `./run.sh RaceConditionDemo` — 결과가 800,000보다 작게, **매번 다르게** 나오는 것을 확인하라.
> "얼마나 증발했는가"까지 출력한다. 단일 스레드 대조군이 항상 정답인 것도 나란히 보여준다.

이 "잃어버린 갱신"은 자바만의 문제가 아니다. [`CS_데이터베이스_개념서`](../CS_데이터베이스_개념서/)의
동시 트랜잭션에서도 똑같이 발생하며, DB는 이를 락이나 MVCC로 막는다(4·5장에서 다시 언급한다).

## 3.2 경쟁 상태(Race Condition)

### 개념

`count++`처럼 **여러 스레드의 접근 순서(타이밍)에 따라 결과가 달라지고, 그중 어떤 순서가 잘못된 결과를 내는**
상황을 **경쟁 상태(race condition)**라 한다. 이름 그대로 스레드들이 "경쟁"하는데, 누가 이기느냐(어떤 순서로
끼어드느냐)에 따라 데이터가 깨진다.

경쟁 상태의 대표적 결과 두 가지.

- **잃어버린 갱신(lost update)**: 위에서 본 것. 여러 갱신 중 일부가 다른 갱신에 덮여 사라진다.
- **불일치한 상태 관찰(inconsistent read)**: 두 필드를 함께 바꿔야 하는데(예: 계좌 A에서 빼고 B에 더하기),
  한쪽만 바뀐 중간 상태를 다른 스레드가 읽어 버린다. → "돈이 잠깐 사라지거나 두 배가 된" 것처럼 보인다.

### 원리 — 경쟁 상태의 필요조건

경쟁 상태는 다음이 **모두** 성립할 때 생긴다.

1. **공유 데이터**가 있다(힙의 객체나 static 필드 — [1장](01-프로세스와-스레드.md)의 "공유" 영역).
2. 적어도 하나의 스레드가 그 데이터를 **쓴다**(모두 읽기만 하면 경쟁 없음).
3. 접근이 **동기화되지 않았다**(순서를 강제하는 장치가 없다).

거꾸로 말하면, 이 셋 중 하나만 없애도 경쟁 상태는 사라진다. 그래서 실무의 방어책이 이렇게 나온다.

- 1을 없애기 → **공유하지 마라.** 지역 변수·불변 객체·stateless 설계(스프링 Service).
- 2를 없애기 → **불변(immutable)으로 만들어라.** 한 번 만들고 안 바꾸면 쓰기가 없어 안전하다.
- 3을 없애기 → **동기화하라.** 락·Atomic 등([4장](04-동기화-락-cas-volatile.md)).

## 3.3 임계 구역(Critical Section)

### 개념

**임계 구역(critical section)**이란 "한 번에 한 스레드만 실행해야 안전한 코드 조각"이다. 위 예에서
`count++`의 read-modify-write 전체가 임계 구역이다. 임계 구역은 "쪼개지면 안 되는" 연산 묶음을 가리킨다.

```
  임계 구역 = ┌───────────────────────┐
             │ read → modify → write │  ← 이 3단계는 통째로, 방해받지 않고 끝나야 한다
             └───────────────────────┘
```

### 원리 — 원자성(atomicity)이 목표다

**동시성 해결의 본질**은 이것이다: **임계 구역을 "쪼개지지 않게(atomic)" 만들어라.** 즉 한 스레드가 임계 구역을
들어가면, 그 스레드가 끝낼 때까지 다른 스레드는 못 들어오게 한다(**상호 배제, mutual exclusion**).

좋은 임계 구역 보호 장치가 갖춰야 할 세 성질(전공 교재의 고전적 조건).

1. **상호 배제(mutual exclusion)**: 한 번에 한 스레드만 임계 구역 안에 있다.
2. **진행(progress)**: 아무도 임계 구역에 없으면, 들어가려는 스레드는 결국 들어갈 수 있어야 한다(무한 지연 금지).
3. **유한 대기(bounded waiting)**: 특정 스레드가 영영 못 들어가는 기아가 없어야 한다.

임계 구역은 **작게** 유지하는 것이 좋다. 크게 잡으면 그만큼 다른 스레드가 오래 기다려 성능(동시 처리량)이 떨어진다.
"필요한 만큼만 좁게 잠근다"가 원칙이다.

### 어떻게 만드나?

임계 구역을 원자적으로 만드는 도구가 다음 장의 주제다.

- **락/뮤텍스(`synchronized`, `ReentrantLock`)**: 임계 구역 입구에 자물쇠를 채운다.
- **원자 변수(`AtomicXxx`, CAS)**: 단일 변수의 read-modify-write를 하드웨어 명령 하나로 원자화한다.

> ▶ 데모: `./run.sh RaceConditionDemo` — 문제(잃어버린 갱신, 비결정성)를 눈으로 본 뒤,
> `./run.sh SynchronizationDemo` 로 넘어가 이 임계 구역을 세 가지 방법으로 원자화하는 것을 확인하라.

## ⚠️ 흔한 오해와 함정

- **"`count++`는 한 줄이니 원자적"** → 가장 흔하고 치명적인 오해. 한 줄이어도 바이트코드로는 3단계이고,
  그 사이에 끼어들 수 있다. "소스 코드 한 줄"과 "원자적"은 아무 관계가 없다.
- **"결과가 800,000보다 크게 나올 수도 있다"** → 아니다. 잃어버린 갱신은 갱신을 **잃는** 것이라 결과는 항상
  기대값 이하다(더 커지는 경우는 없다). 더 커지는 버그는 다른 종류의 문제다.
- **"읽기만 하면 안전하다"** → 대체로 맞다(경쟁 상태 조건 2가 깨짐). 단, 다른 스레드가 그 값을 **쓰는** 경우엔
  가시성 문제([5장](05-메모리-가시성과-JMM.md))가 남아 낡은 값을 볼 수 있다.
- **"임계 구역은 크게 잡을수록 안전하다"** → 안전성은 올라갈지 몰라도 성능이 급락한다. **필요한 만큼만 좁게**가 원칙.
  너무 크게 잡으면 사실상 단일 스레드처럼 동작해 멀티스레드의 이점이 사라진다.

## 연습문제

1. **(핵심)** `count++`가 원자적이지 않은 이유를 바이트코드 3단계 이름과 함께 설명하라.
   <details><summary>힌트/해설</summary>
   read(getfield/getstatic) → modify(iadd/ladd) → write(putfield/putstatic). 이 세 명령 사이에 문맥 교환이
   끼어들면, 두 스레드가 같은 옛 값을 읽고 같은 값을 써서 갱신 하나가 증발한다(lost update).</details>

2. **(개념)** 경쟁 상태가 성립하기 위한 세 가지 조건을 적고, 각 조건을 없애는 실무 방어책을 하나씩 연결하라.
   <details><summary>힌트/해설</summary>
   ① 공유 데이터 → 공유 안 하기(지역 변수/stateless). ② 쓰기 존재 → 불변 객체로 쓰기 제거.
   ③ 미동기화 → 락/Atomic으로 동기화. 하나만 깨도 경쟁 상태는 사라진다.</details>

3. **(응용)** 8개 스레드가 각 10만 번 증가시켰는데 결과가 792,341이 나왔다. "얼마의 갱신이 증발했는가"와
   "이 값이 800,000보다 클 수 있는가"를 답하라.
   <details><summary>힌트/해설</summary>
   증발 = 800,000 − 792,341 = 7,659. 800,000보다 클 수는 없다(갱신을 잃기만 하므로 항상 기대값 이하).</details>

4. **(함정)** 다음 코드는 두 필드를 함께 갱신한다. `synchronized` 없이 두 스레드가 `transfer`를 동시에 호출하면
   어떤 경쟁 상태가 생기는가?
   ```java
   void transfer() { balanceA -= 100; balanceB += 100; }
   ```
   <details><summary>힌트/해설</summary>
   각 `-=`, `+=`가 read-modify-write라 잃어버린 갱신이 나고, 더해서 "A는 줄었는데 B는 아직 안 는" 중간 상태를
   다른 스레드가 읽는 불일치 관찰도 생긴다. 두 필드를 한 임계 구역으로 묶어 보호해야 한다.</details>

5. **(심화)** 임계 구역을 "작게 유지하라"는 원칙과 "여러 필드를 원자적으로 묶어야 한다"는 요구가 충돌하는 예를
   들고, 어떻게 균형을 잡을지 논하라.
   <details><summary>힌트/해설</summary>
   위 transfer처럼 두 필드를 함께 바꿔야 하면 둘을 한 임계 구역으로 묶어야 한다(작게만 잡으면 불일치 발생).
   균형: "논리적으로 함께 원자적이어야 하는 최소 단위"까지만 묶고, 그 안에서 I/O나 무거운 연산은 밖으로 뺀다.
   락을 잡은 채 네트워크 호출 같은 느린 작업을 하지 않는 것이 핵심.</details>

## 요약

- `count++`는 한 줄이지만 바이트코드로 **read → modify → write 3단계**이고, 그 사이에 문맥 교환이 끼어들 수 있다.
- 그래서 여러 스레드가 공유 값을 증가시키면 **잃어버린 갱신(lost update)**이 나고, 결과는 항상 기대값 이하이며 매번 다르다.
- 이렇게 **접근 순서에 따라 결과가 달라지고 잘못될 수 있는 상황**이 **경쟁 상태(race condition)**다.
  조건은 공유 데이터 + 쓰기 + 미동기화 셋이며, 하나만 없애도 사라진다.
- **임계 구역**은 한 번에 한 스레드만 실행해야 안전한 코드 조각. 해결의 본질은 이것을 **원자적(쪼개지지 않게)**으로
  만드는 것이며, 상호 배제·진행·유한 대기를 보장하는 도구가 다음 장의 주제다.
- ▶ 데모: `./run.sh RaceConditionDemo`
