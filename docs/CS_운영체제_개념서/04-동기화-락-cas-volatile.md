## 왜 이 장이 필요한가 — 백엔드 개발자 관점

3장에서 문제(경쟁 상태)를 봤으니, 이제 고칠 차례다. 문제는 도구가 여러 개인데 **"다 비슷한 거 아냐?"** 싶지만,
각자 **고치는 문제가 다르다.** 특히 이 장의 핵심은 두 개념을 구분하는 것이다.

- **원자성(atomicity)**: read-modify-write가 쪼개지지 않는가? ([3장](03-경쟁-상태와-임계-구역.md)의 문제)
- **가시성(visibility)**: 한 스레드가 쓴 값을 다른 스레드가 즉시 볼 수 있는가? ([5장](05-메모리-가시성과-JMM.md)의 문제)

`volatile`을 `count++`에 붙여 놓고 "동기화했다"고 착각하는 것이 가장 흔한 실무 실수다. **`volatile`은 가시성만
고치고 원자성은 못 고친다.** 이 장을 끝내면 언제 무엇을 써야 하는지가 헷갈리지 않는다.

## 4.1 락과 뮤텍스, 그리고 `synchronized`(모니터)

### 개념 — 뮤텍스

가장 직관적인 해법: **임계 구역에 자물쇠를 채운다.** 한 스레드가 자물쇠(**락, lock**)를 쥐면, 다른 스레드는 그것이
풀릴 때까지 문 앞에서 기다린다(그리고 [2장](02-동시성과-병렬성-스케줄러.md)의 "대기(Blocked)" 상태로 잠든다).
이런 **"한 번에 하나만"** 자물쇠를 **뮤텍스(mutex, MUTual EXclusion의 줄임말)**라 한다.

### 원리 — 세마포어와 뮤텍스의 관계

전공 교재에서는 뮤텍스보다 더 일반적인 **세마포어(semaphore)**를 먼저 배운다. 세마포어는 다익스트라가 제안한
동기화 원시 도구로, **정수 카운터 + 두 연산**으로 이루어진다.

- **`P`(wait, acquire)**: 카운터를 1 줄인다. 0 미만이 되면 스레드는 대기.
- **`V`(signal, release)**: 카운터를 1 늘린다. 대기 중인 스레드가 있으면 하나 깨운다.

```
  세마포어 S = 자원이 몇 개 남았는가?
    P(S): S--;  만약 S<0 이면 대기 큐에 들어가 잠든다
    V(S): S++;  만약 대기자가 있으면 하나 깨운다

  ┌ 이진 세마포어(S 초기값=1) = 뮤텍스: 자원 1개 = "한 번에 한 스레드"
  └ 계수 세마포어(S 초기값=N) = 자원 N개까지 동시 허용 (예: 커넥션 풀 크기 N)
```

즉 **뮤텍스는 "초기값이 1인 이진 세마포어"의 특수한 경우**로 볼 수 있다. 다만 뮤텍스에는 세마포어에 없는
중요한 규칙이 하나 있다: **락을 잡은 스레드만 그 락을 풀 수 있다(소유권, ownership).** 세마포어는 아무 스레드나
`V`를 호출할 수 있어 "신호 전달"에도 쓰이지만, 뮤텍스는 오직 상호 배제 전용이다.

계수 세마포어는 실무에서 **자원 개수 제한**에 쓰인다. 예를 들어 DB 커넥션이 10개면, 초기값 10짜리 세마포어로
"동시에 10개까지만 커넥션을 쓰게" 제한할 수 있다. 자바에는 `java.util.concurrent.Semaphore`가 있다.

```java
Semaphore permits = new Semaphore(10);   // 동시에 10개까지 허용
permits.acquire();      // P: 자리 하나 확보 (없으면 대기)
try {
    // 최대 10개 스레드만 여기 동시에 들어옴
} finally {
    permits.release();  // V: 자리 반납
}
```

### 원리 — 자바의 모니터와 `synchronized`

자바는 **모든 객체에 내장 락**을 심어놨다. 이 내장 락과 대기/통지 메커니즘을 합친 것을 **모니터(monitor)**라 한다.
모니터는 "상호 배제(뮤텍스) + 조건 대기(wait/notify)"를 언어 차원에서 묶은 고전적 동시성 구조물이다.
`synchronized` 키워드로 이 모니터를 쓴다.

```java
synchronized (lock) {   // lock 객체의 모니터(내장 락)를 획득 (없으면 대기)
    count++;            // 임계 구역: 한 번에 한 스레드만 여기 들어옴
}                       // 블록을 벗어나면 모니터 해제 (예외로 벗어나도 자동 해제)
```

```
  Thread A ── 락 획득 ──▶ [ count++ ] ── 락 해제 ──▶
  Thread B ── 락 대기 ─────────(문 앞에서 잠들어 기다림)──▶ [ count++ ] ...
             (Blocked 상태)                              (락 얻고 Running)
```

- **고치는 것**: 원자성 O, 가시성 O. (가시성까지 되는 이유는 [5장](05-메모리-가시성과-JMM.md)의 happens-before에서.)
- **대가**: 대기하는 스레드는 논다(경합이 심하면 느려진다). 임계 구역이 크면 병목이 된다.
- **재진입(reentrant)**: 자바 모니터는 재진입 가능하다. 이미 락을 쥔 스레드가 같은 락을 다시 요청하면 통과한다.
  (그래서 `synchronized` 메서드가 다른 `synchronized` 메서드를 호출해도 자기 자신을 막지 않는다.)

`synchronized`는 블록뿐 아니라 메서드에도 붙일 수 있다. 인스턴스 메서드면 `this`의 모니터를, static 메서드면
그 클래스 객체의 모니터를 쓴다.

```java
public synchronized void inc() { count++; }   // this 를 락으로 사용
```

## 4.2 원자 변수(Atomic)와 CAS

### 개념

락은 "막고 기다리는" 방식이다. 반대로 **락 없이(lock-free)** 원자성을 얻는 방법이 있다:
**CAS(Compare-And-Swap)**. CPU가 제공하는 원자적 명령이다.

> "메모리의 값이 **아직 V이면** V+1로 **바꿔라**. 그 사이 누가 바꿔서 V가 아니면 **실패**를 알려라."

읽기·비교·쓰기가 CPU 명령 하나로 원자적으로 처리된다(하드웨어가 그 순간 다른 코어의 접근을 막는다).
실패하면? **다시 시도**한다(재시도 루프, retry loop).

### 원리 — 재시도 루프

```java
AtomicLong count = new AtomicLong(0);
count.incrementAndGet();   // 내부적으로 CAS 재시도 루프
```

```
  incrementAndGet() 내부:
  ┌─▶ v = 현재값 읽기
  │   새값 = v + 1
  │   CAS(기대=v, 새값): 그 사이 안 바뀌었으면 성공 ──▶ 끝
  └── 바뀌었으면 실패 → 처음부터 재시도
```

CAS가 왜 원자적인가? 3장의 문제는 "read와 write 사이에 끼어듦"이었다. CAS는 "값이 여전히 내가 읽은 그 값일
때만 쓴다"를 **하드웨어가 한 명령으로** 보장한다. 그 사이 누가 바꿨으면 조건이 깨져 실패하고, 최신 값으로 다시
시도한다. 결국 "끼어듦"이 있어도 잘못된 값을 쓰지 않는다.

- **고치는 것**: 단일 변수의 원자성 O, 가시성 O.
- **장점**: 락처럼 "재우고 깨우는" 비용이 없어 경합이 낮으면 매우 빠르다(**낙관적** 접근 — 대개 안 부딪힌다고 가정).
- **한계**: **하나의 변수**에만 잘 맞는다. 여러 변수를 함께 원자적으로 바꿔야 하면 락이 낫다.
  또 경합이 극심하면 재시도가 잦아져 오히려 느려질 수 있다.

> **낙관적 vs 비관적**: CAS는 "충돌은 드물 것"이라 보고 일단 시도한 뒤 실패하면 재시도하는 **낙관적(optimistic)**
> 방식이다. 락은 "충돌할 테니 미리 막자"는 **비관적(pessimistic)** 방식이다. 이 구분은
> [`CS_데이터베이스_개념서`](../CS_데이터베이스_개념서/)의 낙관적 락(버전 컬럼)/비관적 락(`SELECT FOR UPDATE`)과
> 똑같은 사고방식이다. 개념이 계층을 넘어 반복된다.

## 4.3 명시적 락 `ReentrantLock`

### 개념

`synchronized`는 편하지만 유연성이 없다(블록을 벗어날 때만 풀림, 타임아웃 불가, 조건 대기 하나뿐).
`java.util.concurrent.locks.ReentrantLock`은 `lock()`/`unlock()`을 **직접** 호출하는 명시적 락이다.

```java
lock.lock();
try {
    count++;          // 임계 구역
} finally {
    lock.unlock();    // 반드시 finally에서! (예외가 나도 풀리도록)
}
```

### 원리 — `synchronized` 대비 추가 기능

같은 효과(원자성·가시성) + 다음 기능들.

- **`tryLock()`**: 락을 못 잡으면 기다리지 않고 즉시 실패를 돌려준다. **교착 회피**의 열쇠([6장](06-교착-상태-데드락.md)).
- **`tryLock(시간)`**: 정해진 시간만 기다리고 포기(타임아웃). `DeadlockDemo`가 이 기법으로 hang을 막는다.
- **공정성(fairness)**: 먼저 기다린 스레드가 먼저 락을 얻도록 옵션을 줄 수 있다(기아 방지, 대신 약간 느림).
- **여러 조건 변수(`Condition`)**: `newCondition()`으로 여러 대기 조건을 나눠 관리(모니터의 wait/notify는 하나뿐).

- **대가**: `unlock()`을 직접 해야 해서 실수하면 락이 안 풀려 다른 스레드가 영영 대기한다. → **반드시 `finally`.**

"이름의 Reentrant"는 §4.1의 재진입 가능성을 뜻한다. 같은 스레드가 이미 쥔 락을 다시 `lock()`하면 카운트만 올라가고
통과하며, 같은 횟수만큼 `unlock()`해야 완전히 풀린다.

## 4.4 `volatile`: 가시성은 고치지만 원자성은 못 고친다 (가장 흔한 함정)

### 개념

`volatile`은 필드에 붙이는 키워드다. **"이 변수는 항상 메인 메모리에서 최신 값을 읽고, 즉시 쓴다"**를 보장한다.
즉 **가시성**을 고친다([5장](05-메모리-가시성과-JMM.md)에서 왜 필요한지 상세히 설명). **하지만 원자성은 고치지 못한다.**

### 원리 — 왜 `count++`엔 부족한가

```java
volatile long count = 0;
count++;   // 여전히 위험! read-modify-write 3단계는 그대로 쪼개진다.
```

`volatile`은 각 단계(read, write)에서 "최신 값을 보게"는 해준다. 그러나 세 단계가 **통째로** 안 쪼개지게 하지는
못한다. read와 write는 여전히 별개의 명령이고, 그 사이에 다른 스레드가 끼어들 수 있다. 그래서 `count++`는
여전히 갱신을 잃는다.

```
  volatile이 고치는 것:  값이 "보이는가"      → O (가시성)
  volatile이 못 고치는 것: 3단계가 "안 쪼개지는가" → X (원자성)
```

**그럼 `volatile`은 언제 쓰나?** read-modify-write가 아닌, **단순 읽기/쓰기 하나짜리** 상태에 쓴다. 대표 예가
**중단 플래그(stop flag)**다.

```java
volatile boolean running = true;   // 한 스레드가 false로 쓰고, 다른 스레드가 읽기만 함

// 워커 스레드:
while (running) { doWork(); }       // running이 false로 바뀌면 즉시 보고 종료

// 다른 스레드:
running = false;                    // 쓰기 하나 — read-modify-write 아님 → volatile로 충분
```

만약 여기서 `volatile`을 빼면, 워커 스레드가 자기 CPU 캐시의 낡은 `running=true`만 보고 **영영 안 멈출 수** 있다
(5장의 가시성 문제). 이때는 `volatile` 하나면 완벽하다 — 원자성이 필요 없는 상황이기 때문이다.

> ### 한 문장 요약
>
> **가시성 ≠ 원자성.** `volatile`은 플래그(boolean 하나 읽고 쓰기)엔 완벽하지만, `count++` 같은 복합 연산엔
> 부족하다. 그럴 땐 `synchronized`/`Atomic`/`Lock`을 써라.

## 4.5 정리 — 무엇을 언제 쓰나

| 도구            | 원자성 | 가시성 | 특징 / 언제 쓰나 |
|-----------------|:------:|:------:|------------------|
| `volatile`      |   ✗    |   ✓    | 상태 플래그(예: `stop`/`running` boolean). 복합 연산엔 부족 |
| `synchronized`  |   ✓    |   ✓    | 가장 단순. 임계 구역 보호의 기본. 여러 필드 묶어 보호 가능 |
| `AtomicXxx`(CAS)|   ✓    |   ✓    | 단일 변수 카운터에 최적. lock-free로 경합 낮으면 빠름 |
| `ReentrantLock` |   ✓    |   ✓    | 타임아웃/tryLock/공정성/여러 Condition 필요할 때 |
| `Semaphore`     |  (N개) |   ✓    | 자원 개수 제한(예: 동시 접속 N개, 커넥션 풀) |

선택 가이드.

1. **단일 boolean 플래그(읽기/쓰기만)** → `volatile`.
2. **단일 숫자 카운터** → `AtomicXxx`(CAS).
3. **여러 필드를 함께 원자적으로 바꿔야 함** → `synchronized` 또는 `ReentrantLock`.
4. **타임아웃/tryLock/공정성이 필요** → `ReentrantLock`.
5. **동시 사용 개수를 N개로 제한** → `Semaphore`.

> ▶ 데모: `./run.sh SynchronizationDemo` — 같은 카운터를 `synchronized`/`Atomic`/`ReentrantLock` 세 방법으로
> 고쳐 **셋 다 정답(800,000)**이 나오는 것, 그리고 `volatile`만 쓰면 **여전히 틀리는 것**을 나란히 보여준다.
> "volatile은 보이게 할 뿐 쪼개지지 않게 하지는 못한다"를 출력으로 증명한다.

`SynchronizationDemo`의 핵심 골격.

```java
// ① synchronized: 한 번에 한 스레드만 이 블록 안에서 3단계를 끝냄
synchronized (lock) { syncCounter++; }

// ② Atomic(CAS): 락 없이 원자적으로 증가
atomicCounter.incrementAndGet();

// ③ ReentrantLock: 명시적으로 잠그고, finally에서 반드시 해제
reentrantLock.lock();
try { reentrantCounter++; } finally { reentrantLock.unlock(); }

// ④ 반례 — volatile 만: 여전히 샌다 (가시성 O, 원자성 X)
volatileCounter++;   // 결과가 정답보다 작다 → "여전히 틀림"
```

## ⚠️ 흔한 오해와 함정

- **"`volatile`을 붙이면 스레드 안전해진다"** → 이 장에서 가장 중요한 함정. `volatile`은 **가시성만** 고친다.
  `count++`, `x = x + 1`, `list.add()` 같은 복합/비원자 연산엔 무력하다.
- **"`Atomic`이면 무조건 락보다 빠르다"** → 경합이 낮을 때만. 극심한 경합에서는 CAS 재시도가 폭증해 락보다
  느려질 수 있다.
- **"`synchronized(new Object())` 로 잠그면 된다"** → 매번 새 객체를 잠그면 서로 다른 락이라 상호 배제가 안 된다.
  **모든 스레드가 같은 락 객체**를 공유해야 의미가 있다.
- **"`ReentrantLock`은 `finally` 없어도 대개 괜찮다"** → 아니다. 임계 구역에서 예외가 나면 `unlock()`이 안 불려
  락이 영영 안 풀린다. `finally`는 선택이 아니라 필수다.
- **"세마포어와 뮤텍스는 같은 것"** → 이진 세마포어는 뮤텍스와 비슷하지만, 뮤텍스는 **소유권**(잡은 스레드만 해제)이
  있고 세마포어는 없다. 세마포어는 신호 전달과 자원 개수 제한에도 쓴다.

## 연습문제

1. **(핵심)** 원자성과 가시성의 차이를 한 문장씩으로 정의하고, `volatile`이 이 중 무엇을 고치고 무엇을 못 고치는지 답하라.
   <details><summary>힌트/해설</summary>
   원자성 = 여러 단계가 쪼개지지 않음. 가시성 = 한 스레드의 쓰기가 다른 스레드에 보임. `volatile`은 가시성 O,
   원자성 X. 그래서 플래그엔 충분하고 count++엔 부족.</details>

2. **(개념)** 이진 세마포어와 뮤텍스의 공통점과 차이점을 설명하라. 계수 세마포어의 실무 용례도 하나 들라.
   <details><summary>힌트/해설</summary>
   공통점: 초기값 1이면 "한 번에 하나"라는 상호 배제. 차이점: 뮤텍스는 소유권(잡은 스레드만 해제)이 있고 세마포어는
   없다. 계수 세마포어 용례: 커넥션 풀/동시 다운로드 수 등 "자원 N개 동시 사용 제한".</details>

3. **(응용)** 다음 각 상황에 `volatile`/`Atomic`/`synchronized`/`Semaphore` 중 무엇이 적절한가?
   (a) 서버 종료 플래그 (b) 조회수 카운터 (c) 계좌 A→B 이체(두 필드) (d) 동시 다운로드 3개 제한.
   <details><summary>힌트/해설</summary>
   (a) volatile — 단일 boolean 읽기/쓰기. (b) AtomicLong — 단일 카운터. (c) synchronized/ReentrantLock —
   두 필드를 한 임계 구역으로. (d) Semaphore(3).</details>

4. **(함정)** 다음 코드가 여전히 경쟁 상태인 이유는?
   ```java
   volatile int count = 0;
   void inc() { count++; }
   ```
   <details><summary>힌트/해설</summary>
   `count++`가 read-modify-write 3단계인데 `volatile`은 원자성을 주지 않는다. 각 단계에서 최신 값을 보긴 하지만
   세 단계 사이에 끼어듦이 여전히 가능해 갱신을 잃는다. `AtomicInteger`나 `synchronized`로 바꿔야 한다.</details>

5. **(심화)** CAS 재시도 루프가 "낙관적", 락이 "비관적"이라 불리는 이유를 설명하고, 각각이 유리한 경합 수준을
   논하라. DB의 낙관적/비관적 락과 연결해도 좋다.
   <details><summary>힌트/해설</summary>
   CAS: "충돌은 드물다"고 보고 일단 시도, 실패 시 재시도(낙관적). 경합이 낮을 때 빠르다. 락: "충돌한다"고 보고
   미리 막음(비관적). 경합이 높을 때 재시도 폭증을 피할 수 있어 유리. DB에서 버전 컬럼=낙관적,
   SELECT FOR UPDATE=비관적으로 같은 트레이드오프가 반복된다.</details>

## 요약

- 동기화 도구는 고치는 문제가 다르다. 핵심 구분은 **원자성(쪼개짐 방지)** vs **가시성(쓴 값 보이기)**.
- **뮤텍스**는 상호 배제 자물쇠, **세마포어**는 그 일반형(카운터 + P/V; 이진이면 뮤텍스 유사, 계수면 자원 N개 제한).
- 자바는 모든 객체에 **모니터(내장 락)**를 심어 `synchronized`로 쓴다 — 원자성·가시성 O, 재진입 가능, 자동 해제.
- **CAS/`Atomic`**은 락 없이 단일 변수를 원자화하는 낙관적 방식 — 경합 낮으면 빠름, 여러 변수엔 부적합.
- **`ReentrantLock`**은 `synchronized` + 타임아웃/tryLock/공정성/여러 Condition. 대신 `finally`에서 반드시 `unlock`.
- **`volatile`**은 **가시성만** 고친다 — 플래그엔 완벽, `count++`엔 무력. 이 구분이 이 장의 핵심.
