## 왜 이 장이 필요한가 — 백엔드 개발자 관점

이 장은 동시성과 직접 관련은 적은 **곁다리(OS 메모리 관리)**다. 하지만 [1장 §1.1](01-프로세스와-스레드.md)에서
미뤄 둔 마법 — **"프로세스 A의 `0x1000`과 B의 `0x1000`이 실제로는 다른 물리 위치"** — 을 여기서 푼다. 그리고
이 이해는 실무에서 대단히 실용적이다. **"JVM 힙(`-Xmx`)을 물리 RAM보다 크게 잡으면 왜 서버가 급격히 느려지는가"**,
**"컨테이너 메모리 제한과 힙 크기의 관계"** 같은 운영 문제가 전부 이 장의 페이징·스와핑에서 나온다.

이 장을 끝내면 다음을 설명할 수 있다.

- **가상 메모리·페이징**이 무엇이고, 프로세스 격리의 마법을 어떻게 만드는지.
- 가상 주소가 물리 주소로 번역되는 원리(페이지 테이블, MMU).
- JVM 힙이 결국 이 가상 메모리의 한 덩어리라는 것, 그리고 `-Xmx`와 스와핑의 관계.

---

## 8.1 가상 메모리(Virtual Memory)와 페이징(Paging)

### 개념

[1장](01-프로세스와-스레드.md)에서 "프로세스 A의 `0x1000`과 B의 `0x1000`은 실제로는 다른 물리 위치"라 했다.
어떻게? **가상 메모리(virtual memory)** 덕분이다.

- 각 프로세스는 **가상 주소(virtual address)**를 본다 — 자기가 메모리를 통째로 가진 것처럼.
- OS와 CPU의 **MMU(Memory Management Unit)**가 가상 주소를 **물리 주소(physical address)**로 번역한다.
  이 번역표가 **페이지 테이블(page table)**이다.
- 메모리는 **페이지(page)**라는 고정 크기(보통 4KB) 조각으로 관리된다. 물리 메모리 쪽 조각은 **프레임(frame)**이라 부른다.

### 원리 — 주소 번역

```
  프로세스 A의 가상주소        페이지 테이블(A)      물리 메모리(RAM)
     0x1000  ──────────▶  [번역]  ──────────▶   0x7A3000
  프로세스 B의 가상주소        페이지 테이블(B)
     0x1000  ──────────▶  [번역]  ──────────▶   0x22F000   (A와 다른 곳!)
```

가상 주소는 "페이지 번호 + 페이지 안 오프셋"으로 나뉜다. MMU가 페이지 번호로 페이지 테이블을 찾아 대응하는
물리 프레임 번호를 얻고, 오프셋을 붙여 최종 물리 주소를 만든다. 페이지 테이블은 프로세스마다 따로 있어서,
A의 페이지 번호와 B의 같은 페이지 번호가 서로 다른 프레임으로 번역된다 — 그래서 같은 가상 주소가 다른 물리
위치를 가리키는 것이다.

이 번역은 메모리 접근마다 일어나므로 느리면 큰 문제다. 그래서 CPU는 **TLB(Translation Lookaside Buffer)**라는
작은 캐시에 최근 번역 결과를 저장해 대부분의 접근을 빠르게 처리한다. ([1장](01-프로세스와-스레드.md)에서 프로세스
간 문맥 교환이 스레드 간보다 무겁다고 한 이유 중 하나가, 프로세스가 바뀌면 이 TLB를 비워야 하기 때문이다.)

> ### 왜 이렇게 하는가?
>
> - **격리**: 프로세스는 자기 페이지 테이블에 없는 물리 메모리에 접근할 수 없다 → 남의 메모리를 못 건드린다.
>   [1장](01-프로세스와-스레드.md)에서 말한 "프로세스 격리(안정성·보안)"가 바로 이 메커니즘으로 실현된다.
> - **환상(추상화)**: 물리 RAM보다 큰 메모리를 쓰는 것처럼 보이게 한다. 지금 안 쓰는 페이지는 디스크로 내보내고
>   (스와핑), 필요할 때 다시 불러온다. 그래서 프로그램은 물리 RAM 크기를 신경 쓰지 않고 짤 수 있다.
> - **유연성**: 물리적으로 흩어진 프레임들을 가상 공간에선 **연속된 것처럼** 보이게 한다. 그래서 힙이 "이어진 큰
>   덩어리"처럼 다뤄진다.

### 곁가지 — 페이지 폴트와 스와핑

접근하려는 페이지가 물리 메모리에 없으면(디스크로 나가 있으면) **페이지 폴트(page fault)**가 발생한다. OS가
디스크에서 그 페이지를 물리 프레임으로 불러오고, 자리가 없으면 오래 안 쓴 페이지를 골라 디스크로 내보낸다
(페이지 교체 정책, 예: LRU). 이 "디스크로 내보내고 불러오는" 왕복을 **스와핑(swapping)**이라 한다.

디스크는 RAM보다 수만 배 느리므로, 페이지 폴트가 잦아지면 성능이 급락한다. 프로그램이 실제 일보다 페이지
교체에 시간을 더 쓰는 병적 상태를 **스래싱(thrashing)**이라 한다. 이것이 다음 절의 실무 경고로 이어진다.

## 8.2 JVM 힙과의 관계

### 개념

JVM의 **힙(Heap)**은 결국 OS로부터 받은 이 가상 메모리의 한 덩어리다. [`JAVA_개념서 §1.8 가비지 컬렉션`](../JAVA_교재/심화/)에서
GC가 청소하는 "힙"도, 그 밑을 보면 OS가 페이지로 관리하는 가상 메모리다.

### 원리 — `-Xmx`와 스와핑

- JVM은 시작할 때 OS에 큰 메모리 영역을 요청하고(`-Xmx`가 그 힙 최대치), 그 안에서 `new`로 객체를 할당한다.
  이 영역은 가상 주소 공간의 한 구획이고, 실제로 쓰는 페이지만 물리 프레임에 매핑된다.
- GC는 이 힙 안에서 도달 불가능한 객체를 청소한다. 하지만 GC가 훑는 힙 자체가 페이지들이므로, 힙이 물리 RAM보다
  크면 GC가 스왑된 페이지들을 건드리며 대량의 페이지 폴트를 유발한다.

> ### 실무 감각 — "힙 크기 ≤ 사용 가능한 물리 메모리"
>
> JVM 힙(`-Xmx`)을 물리 RAM보다 크게 잡으면, OS가 부족한 만큼 디스크로 **스와핑**하며 성능이 급락한다.
> 특히 GC가 힙 전체를 훑을 때 스왑된 페이지를 마구 불러오면 스래싱이 나 서버가 사실상 멈춘 것처럼 느려진다.
> 그래서 **"JVM 힙 크기 ≤ 사용 가능한 물리 메모리"**를 지켜야 한다.
>
> 컨테이너(도커/쿠버네티스) 환경에선 한 걸음 더 나아가, **컨테이너 메모리 제한(limit)보다 힙을 작게** 잡아야
> 한다. 힙 외에도 스택·메타스페이스·네이티브 메모리가 추가로 들기 때문이다. 힙이 제한에 육박하면 컨테이너가
> **OOMKilled**로 강제 종료된다. 최신 JVM은 컨테이너 메모리 제한을 인식해 힙 기본값을 자동 조정한다.

이렇게 [1장](01-프로세스와-스레드.md)의 "프로세스 격리"부터 [`JAVA_개념서`](../JAVA_교재/심화/)의 "GC가 청소하는
힙"까지가 하나의 메모리 이야기로 이어진다. 겉으로는 자바 튜닝 옵션처럼 보이는 `-Xmx`가 사실은 OS의 페이징 위에서
동작한다는 것을 알면, "메모리를 넉넉히 주면 항상 빨라진다"는 순진한 가정을 하지 않게 된다.

---

## ⚠️ 흔한 오해와 함정

- **"프로세스가 보는 주소가 곧 물리 메모리 주소"** → 아니다. 프로세스는 **가상 주소**를 보고, MMU가 페이지 테이블로
  물리 주소로 번역한다. 그래서 서로 다른 프로세스의 같은 가상 주소가 다른 물리 위치일 수 있다.
- **"`-Xmx`를 크게 주면 무조건 빨라진다"** → 아니다. 물리 RAM을 넘으면 스와핑으로 오히려 급격히 느려진다.
  힙은 사용 가능한 물리 메모리 안에 들어와야 한다.
- **"컨테이너 메모리 제한 = 힙 크기로 잡으면 된다"** → 위험하다. 힙 외에 스택·메타스페이스·네이티브 메모리가
  더 든다. 제한보다 힙을 작게 잡지 않으면 OOMKilled를 당한다.
- **"GC가 메모리를 관리하니 OS 페이징과는 무관"** → 아니다. 힙은 결국 가상 메모리 페이지들이라, 힙이 물리 RAM을
  넘으면 GC가 스왑된 페이지를 대량으로 건드려 스래싱을 유발한다.

## 연습문제

1. **(핵심)** 서로 다른 두 프로세스가 같은 가상 주소 `0x1000`을 봐도 데이터가 안 섞이는 이유를, 페이지 테이블로
   설명하라.
   <details><summary>힌트/해설</summary>
   페이지 테이블이 프로세스마다 따로 있어, 같은 가상 페이지 번호가 서로 다른 물리 프레임으로 번역되기 때문.
   이것이 프로세스 격리(안정성·보안)의 실제 메커니즘.</details>

2. **(개념)** 페이지 폴트와 스와핑, 스래싱을 각각 설명하고 셋의 관계를 밝혀라.
   <details><summary>힌트/해설</summary>
   페이지 폴트 = 접근하려는 페이지가 물리 메모리에 없음. 그러면 디스크에서 불러오고 자리 없으면 내보냄(스와핑).
   페이지 폴트가 너무 잦아 일보다 교체에 시간을 다 쓰는 병적 상태 = 스래싱.</details>

3. **(응용)** 물리 RAM 8GB 서버에서 `-Xmx12g`로 JVM을 띄웠다. 어떤 일이 벌어질 수 있는가?
   <details><summary>힌트/해설</summary>
   힙이 물리 RAM을 초과해 OS가 스와핑을 시작한다. 특히 GC가 힙 전체를 훑을 때 스왑된 페이지를 대량으로 불러와
   스래싱이 나 서버가 극도로 느려진다. 힙은 사용 가능한 물리 메모리 안에 잡아야 한다.</details>

4. **(함정)** 컨테이너 메모리 제한 2GB에 `-Xmx2g`로 잡았더니 OOMKilled가 났다. 왜인가?
   <details><summary>힌트/해설</summary>
   힙 외에 스택·메타스페이스·네이티브·GC 자료구조 등이 추가로 메모리를 쓴다. 힙만 2GB로 잡으면 전체 사용량이
   제한 2GB를 넘어 컨테이너가 프로세스를 죽인다. 힙은 제한보다 넉넉히 작게 잡아야 한다.</details>

5. **(심화)** TLB가 무엇이며, 프로세스 간 문맥 교환이 스레드 간보다 무거운 이유와 어떻게 연결되는지 설명하라.
   ([1장](01-프로세스와-스레드.md)과 연계)
   <details><summary>힌트/해설</summary>
   TLB는 최근 가상→물리 번역 결과를 캐싱하는 CPU 안 작은 캐시. 프로세스가 바뀌면 주소 공간(페이지 테이블)이
   달라져 TLB를 비워야 하고, 이후 번역이 다시 느려진다. 같은 프로세스 내 스레드 전환은 주소 공간이 같아 이
   비용이 없다 → 그래서 스레드 전환이 더 싸다.</details>

## 요약

- **가상 메모리**로 각 프로세스는 자기만의 가상 주소를 보고, **MMU**가 **페이지 테이블**로 물리 주소로 번역한다.
  메모리는 **페이지(≈4KB)** 단위로 관리된다.
- 페이지 테이블이 프로세스마다 따로라, 같은 가상 주소가 다른 물리 위치를 가리킨다 → **프로세스 격리**의 실제 구현.
  **TLB**가 번역을 캐싱해 속도를 챙긴다.
- 물리 메모리에 없는 페이지 접근은 **페이지 폴트** → **스와핑**(디스크 왕복). 너무 잦으면 **스래싱**으로 급락한다.
- **JVM 힙**은 이 가상 메모리의 한 덩어리이고, GC가 훑는 대상도 결국 페이지들이다.
- 실무: **힙(`-Xmx`) ≤ 사용 가능한 물리 메모리**, 컨테이너에선 **메모리 제한보다 힙을 작게**(안 그러면 스래싱/OOMKilled).
