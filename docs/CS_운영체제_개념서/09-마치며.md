## 전체를 하나의 그림으로

전공자 수준의 감각이란, 이 책의 여덟 장이 **따로따로가 아니라 하나의 이야기**로 보이는 것이다.

```
  [OS] 스레드는 스택/레지스터는 따로, 힙/코드는 공유          (1장)
        └─ 스케줄러가 문맥 교환 → 순서는 비결정적            (2장)
              └─ 공유 데이터 + count++(3단계) = 경쟁 상태     (3장)
                    ├─ 원자성 문제 → synchronized/Atomic/Lock (4장)
                    ├─ 가시성 문제 → volatile/JMM(happens-before) (4·5장)
                    └─ 락을 잘못 쓰면 → 교착(코프만 4조건, 락 순서로 예방) (6장)
                          └─ 실무에선 고수준 도구로 감싼다
                                (스레드 풀·Future·BlockingQueue)        (7장)
                                   └─ 이것이 톰캣/스프링이 요청을 처리하는 방식
                                      (SPRING_개념서 §2.7)
  [OS] 프로세스 격리의 마법 = 가상 메모리·페이징 → JVM 힙도 그 위에서 (8장)
```

## 각 장이 답한 "왜"

- **1장**: 왜 어떤 데이터는 안전하고 어떤 데이터는 위험한가? → 스택(전용)은 안전, 힙·static(공유)은 위험.
- **2장**: 왜 동시성 버그는 재현이 안 되는가? → 스케줄러가 순서를 정하고, 그 순서는 매번 다르다(비결정성).
- **3장**: 왜 `count++`가 틀리는가? → 바이트코드 3단계(read-modify-write) 사이에 끼어들어 갱신을 잃는다.
- **4장**: 무엇으로 고치는가? → 원자성은 락/CAS/Lock, 가시성은 volatile/JMM. **원자성 ≠ 가시성**이 핵심.
- **5장**: 왜 쓴 값이 안 보이는가? → 캐시·재배치 때문. **happens-before** 경계가 있어야 보인다.
- **6장**: 락이 만드는 새 문제는? → 교착. 코프만 4조건 중 하나(락 순서 통일로 순환 대기)만 깨도 예방된다.
- **7장**: 실무는 어떻게 쓰는가? → 스레드 풀·Future·BlockingQueue로 감싼다. 톰캣/HikariCP가 바로 이것.
- **8장**: 프로세스 격리의 정체는? → 가상 메모리·페이징. JVM 힙도 그 위에 있고, `-Xmx`는 물리 RAM을 넘으면 안 된다.

## 이제 이 조언들이 왜 나왔는지 설명할 수 있다

- **"Service를 stateless로 짜라"** → 싱글톤 Bean은 모든 요청 스레드가 공유하므로(1장), 필드에 요청별 상태를 담으면
  섞인다(3장). 지역 변수에만 담으면 스택은 스레드 전용이라 안전하다.
- **"싱글톤 Bean에 상태를 두지 마라"** → 같은 이유. 공유되는 필드가 경쟁 상태의 씨앗이다.
- **"재고 차감은 원자적 UPDATE나 락으로"** → 잃어버린 갱신(3장)을 막기 위해. DB에서는 락이나 MVCC(5장)로 푼다.
- **"커넥션 풀 크기를 적절히"** → 계수 세마포어로 자원을 제한하는 것(4장)이자 스레드 풀과 같은 재사용 원리(7장).
- **"힙을 물리 메모리보다 크게 잡지 마라"** → 스와핑·스래싱으로 성능이 급락하기 때문(8장).

당신이 이 인과의 사슬을 밑바닥의 OS·CPU·JMM까지 내려가 설명할 수 있다면, 더 이상 "동시성은 무섭다"에서
멈추지 않는다. 동시성은 무서운 게 아니라 **규칙이 분명한** 세계다.

## 실습 순서 (chapter-cs-concurrency)

각 데모는 독립 실행 가능하며, 출력이 개념을 **증명**하도록 짜여 있다. 프로젝트 루트에서 실행한다.

1. `./run.sh RaceConditionDemo` — 문제를 눈으로 본다(잃어버린 갱신, 비결정성). → [3장](03-경쟁-상태와-임계-구역.md)
2. `./run.sh SynchronizationDemo` — 세 가지로 고치고, volatile의 함정을 본다. → [4장](04-동기화-락-cas-volatile.md)
3. `./run.sh DeadlockDemo` — 락이 만드는 새 문제(교착)와 그 예방을 본다. → [6장](06-교착-상태-데드락.md)
4. `./run.sh ExecutorAndFutures` — 실무의 스레드 풀과 Future. → [7장](07-고수준-동시성-도구.md)
5. `./run.sh ProducerConsumer` — BlockingQueue로 스레드 간 안전한 데이터 전달. → [7장](07-고수준-동시성-도구.md)
