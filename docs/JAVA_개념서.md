# Java 개념서: 왜 그렇게 동작하는가?

> "코드를 외우는 것이 아니라, 왜 그런지를 이해하는 것이 진짜 실력이다."

---

## 들어가며

### 이 책의 목적

프로그래밍 책을 펼치면 대부분 이렇게 시작한다.

```
int a = 10;
```

그리고 "변수 a에 10을 대입한다"라고 설명한다.

하지만 정작 중요한 질문에는 답하지 않는다.

- **왜** 변수라는 게 필요한가?
- **왜** `int`라고 써야 하는가?
- 컴퓨터 내부에서는 **실제로** 무슨 일이 벌어지는가?

이 책은 다르다. 이 책의 목표는 단 하나다.

> **"왜?"에 답하는 것.**

코드를 100줄 외우는 것보다, "왜 그렇게 동작하는가?"를 이해하는 것이 훨씬 가치 있다.
문법은 검색하면 나온다. 하지만 개념은 머릿속에 있어야 한다.

### Java를 배우기 전에 알아야 할 것

프로그래밍을 배운다는 것은 **새로운 언어를 배우는 것**과 같다.

영어를 배울 때 문법만 외우면 말을 못 한다.
문법의 **이유**를 이해하고, 그 언어로 **생각하는 법**을 익혀야 한다.

Java도 마찬가지다. 문법을 외우는 것이 아니라,
**컴퓨터가 생각하는 방식**을 이해하고,
**그 방식으로 문제를 해결하는 법**을 익히는 것이다.

자, 이제 시작하자.

## Chapter 1: 컴퓨터는 어떻게 프로그램을 실행하는가?

### 1.1 프로그래밍 언어란?

#### 인간의 언어와 기계의 언어 사이에서

컴퓨터는 놀랍도록 멍청하다.

컴퓨터가 이해하는 것은 오직 하나, **0과 1**뿐이다.
전기가 흐르면 1, 안 흐르면 0. 그게 전부다.

```
01001000 01100101 01101100 01101100 01101111
```

위의 숫자들은 "Hello"를 의미한다. 이것이 **기계어**다.
사람이 이걸로 프로그램을 만드는 건 사실상 불가능하다.

그래서 **프로그래밍 언어**가 탄생했다.

프로그래밍 언어는 **인간과 컴퓨터 사이의 통역**이다.

```
  사람이 이해하는 말        프로그래밍 언어          컴퓨터가 이해하는 말
 ┌───────────────┐    ┌───────────────┐    ┌───────────────┐
 │ "화면에 Hello │ -> │ System.out    │ -> │ 01001000...   │
 │  라고 출력해" │    │ .println(     │    │ 01100101...   │
 │               │    │  "Hello");    │    │               │
 └───────────────┘    └───────────────┘    └───────────────┘
```

#### 컴파일 언어 vs 인터프리터 언어

이 "번역" 과정에는 크게 두 가지 방식이 있다.

**컴파일 언어 (C, C++ 등)**
- 비유: **책을 통째로 번역한 후 출판하는 것**
- 소스코드 전체를 한 번에 기계어로 변환
- 번역이 끝나면 실행이 매우 빠름
- 하지만 번역 과정(컴파일)이 필요하고, OS마다 다시 번역해야 함

**인터프리터 언어 (Python, JavaScript 등)**
- 비유: **동시통역사가 한 문장씩 통역하는 것**
- 한 줄씩 읽으면서 바로 실행
- 바로 실행할 수 있어 편리함
- 하지만 매번 통역하므로 실행 속도가 느림

#### 왜? Java는 둘 다라고?

Java는 독특하다. **두 가지 방식을 모두 사용**한다.

```
  [Java 소스코드]        [바이트코드]           [실행]
   Hello.java    ──>    Hello.class    ──>    결과 출력
                 컴파일               JVM이 해석
                (javac)            (인터프리팅 + JIT)
```

1단계에서는 **컴파일**한다. 하지만 기계어가 아니라 **바이트코드**라는 중간 언어로 변환한다.
2단계에서는 JVM이 바이트코드를 **인터프리팅**한다.

> 왜 이렇게 복잡하게 만들었는가? 다음 절에서 알아보자.

### 1.2 Java의 특별함: "Write Once, Run Anywhere"

#### C/C++의 문제점

1990년대, 소프트웨어 개발자들에게는 큰 고민이 있었다.

Windows용으로 만든 프로그램은 Mac에서 안 돌아간다.
Mac용으로 만든 프로그램은 Linux에서 안 돌아간다.

왜냐하면 각 운영체제(OS)가 이해하는 기계어가 **다르기** 때문이다.

```
  같은 프로그램인데...

  Windows: 01001000 01100101 ...  (x86 기계어)
  Mac:     10100011 00110100 ...  (다른 기계어)
  Linux:   01101001 10010011 ...  (또 다른 기계어)
```

결국 같은 프로그램을 OS마다 따로 컴파일해야 했다.
3개의 OS를 지원하려면, 3번 컴파일해야 했다.

#### Java의 해결책: JVM이라는 가상 머신

1995년, Sun Microsystems의 제임스 고슬링(James Gosling)이 **기가 막힌 아이디어**를 낸다.

> "프로그램이 OS와 직접 대화하지 말고, 중간에 통역사를 두자!"

그 통역사가 바로 **JVM (Java Virtual Machine)**이다.

```
  ┌─────────────────────────────────────────┐
  │           Java 프로그램 (.class)         │
  │         "한 번만 작성하면 된다"           │
  └──────────────────┬──────────────────────┘
                     │
          ┌──────────┼──────────┐
          v          v          v
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ Windows  │ │   Mac    │ │  Linux   │
  │   JVM    │ │   JVM    │ │   JVM    │
  │ (통역사) │ │ (통역사) │ │ (통역사) │
  └────┬─────┘ └────┬─────┘ └────┬─────┘
       v             v            v
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ Windows  │ │   macOS  │ │  Linux   │
  │   OS     │ │   OS     │ │   OS     │
  └──────────┘ └──────────┘ └──────────┘
```

Java 프로그램은 JVM하고만 대화한다.
JVM이 각 OS에 맞게 통역해준다.

> **비유: Java는 에스페란토어, JVM은 각 나라의 통역사**
>
> 에스페란토어(세계 공용어)로 연설문을 한 번 쓰면,
> 한국인 통역사는 한국어로, 일본인 통역사는 일본어로,
> 미국인 통역사는 영어로 번역해준다.
>
> Java 바이트코드를 한 번 만들면,
> Windows JVM은 Windows 기계어로,
> Mac JVM은 Mac 기계어로 변환해준다.

이것이 Java의 유명한 슬로건이다:

> **"Write Once, Run Anywhere" (한 번 작성하면, 어디서든 실행된다)**

#### 왜? 이것이 왜 혁명적이었는가?

인터넷이 발전하면서, 다양한 기기에서 같은 프로그램을 실행해야 했다.
서버는 Linux, 개발자 컴퓨터는 Windows나 Mac...
매번 다시 컴파일하는 건 악몽이었다.

Java는 이 문제를 단번에 해결했고, 그래서 폭발적으로 인기를 얻었다.

### 1.3 JDK, JRE, JVM의 관계

이 세 가지는 초보자를 가장 혼란스럽게 만드는 용어다.
하지만 실제로는 매우 간단하다.

```
  ┌─────────────────────────────────────────────────────┐
  │                    JDK                               │
  │  (Java Development Kit - 개발자용 전체 패키지)       │
  │                                                      │
  │  ┌─────────────────────────────────────────────┐     │
  │  │                  JRE                         │     │
  │  │  (Java Runtime Environment - 실행 환경)      │     │
  │  │                                              │     │
  │  │  ┌───────────────────────────────────┐       │     │
  │  │  │              JVM                  │       │     │
  │  │  │  (Java Virtual Machine)           │       │     │
  │  │  │  바이트코드를 실행하는 엔진       │       │     │
  │  │  └───────────────────────────────────┘       │     │
  │  │                                              │     │
  │  │  + 기본 라이브러리 (java.lang, java.util 등) │     │
  │  └─────────────────────────────────────────────┘     │
  │                                                      │
  │  + 개발도구 (javac, jar, javadoc, jdb 등)            │
  └─────────────────────────────────────────────────────┘
```

**비유로 이해하기:**

| 구성요소 | 비유 | 역할 |
|----------|------|------|
| **JVM** | 통역사 | 바이트코드를 실행 |
| **JRE** | 통역사 + 사전 | 실행에 필요한 모든 것 |
| **JDK** | 통역사 + 사전 + 펜과 종이 | 개발에 필요한 모든 것 |

- Java 프로그램을 **만들려면**: JDK 필요
- Java 프로그램을 **실행만 하려면**: JRE 필요 (단, Java 11부터는 JRE가 별도 배포되지 않으므로 JDK를 설치하면 된다)
- 둘 다 내부에 JVM 포함

> **면접 팁**: "JDK, JRE, JVM의 차이를 설명하라"는 단골 면접 질문이다.
> 포함 관계(JDK > JRE > JVM)를 그려가며 설명하면 좋은 인상을 준다.

### 1.4 Java 메모리 구조

Java 프로그램이 실행되면, JVM은 메모리를 여러 영역으로 나누어 사용한다.
이것을 이해하지 않으면, 나중에 버그를 만들거나 성능 문제를 겪는다.

```
  ┌──────────────────────────────────────────────────┐
  │                 JVM 메모리 구조                    │
  │                                                    │
  │  ┌────────────────┐  ┌────────────────────────┐   │
  │  │   Method Area   │  │         Heap           │   │
  │  │ (클래스 정보)   │  │   (객체들이 사는 곳)   │   │
  │  │                 │  │                        │   │
  │  │ - 클래스 구조   │  │  ┌──────┐ ┌──────┐    │   │
  │  │ - 메서드 코드   │  │  │객체 A│ │객체 B│    │   │
  │  │ - static 변수   │  │  └──────┘ └──────┘    │   │
  │  │                 │  │       ┌──────┐         │   │
  │  └────────────────┘  │       │객체 C│         │   │
  │                       │       └──────┘         │   │
  │  ┌────────────────┐  │                        │   │
  │  │     Stack       │  └────────────────────────┘   │
  │  │ (메서드 호출)   │                               │
  │  │                 │                               │
  │  │ ┌──────────┐   │                               │
  │  │ │ method3  │   │                               │
  │  │ ├──────────┤   │                               │
  │  │ │ method2  │   │                               │
  │  │ ├──────────┤   │                               │
  │  │ │ method1  │   │                               │
  │  │ ├──────────┤   │                               │
  │  │ │  main    │   │                               │
  │  │ └──────────┘   │                               │
  │  └────────────────┘                               │
  └──────────────────────────────────────────────────┘
```

#### Stack: 접시 쌓기

스택은 **접시를 쌓는 것**과 같다.

새 접시를 올릴 때는 맨 위에 올린다.
접시를 뺄 때도 맨 위에서 뺀다.
**마지막에 들어간 것이 먼저 나옵니다 (LIFO: Last In, First Out).**

메서드를 호출하면 스택에 "쌓이고",
메서드가 끝나면 스택에서 "빠진다".

```
  main() 호출      method1() 호출    method2() 호출    method2() 끝

                                      ┌──────────┐
                    ┌──────────┐      ├──────────┤     ┌──────────┐
  ┌──────────┐     ├──────────┤      │ method1  │     ├──────────┤
  │   main   │     │   main   │      ├──────────┤     │   main   │
  └──────────┘     └──────────┘      │   main   │     └──────────┘
                                      └──────────┘
```

각 메서드의 **지역 변수**도 스택에 함께 저장된다.
메서드가 끝나면 지역 변수도 함께 사라진다.

> **왜?** 이런 구조를 사용하는가?
>
> 메서드 호출은 자연스럽게 "들어갔다 나오는" 구조다.
> A가 B를 호출하고, B가 C를 호출하면, C가 먼저 끝나고, B가 끝나고, A가 끝난다.
> 이것이 정확히 스택의 동작 방식이다.

#### Heap: 거대한 창고

힙은 **객체(Object)가 살아가는 곳**이다.

비유하면 **거대한 창고**다.
`new`라는 키워드로 객체를 만들면, 이 창고 어딘가에 공간이 배정된다.
그리고 그 위치(주소)를 받아온다.

스택의 변수는 이 주소를 가지고 있어서, 힙에 있는 객체를 찾아갈 수 있다.

```
  Stack                          Heap
  ┌──────────────┐              ┌─────────────────────┐
  │ myDog = 0x42 │ ──────────> │ Dog 객체             │
  │              │              │  name = "멍멍이"     │
  │              │              │  age = 3             │
  └──────────────┘              └─────────────────────┘
```

> **왜?** 스택과 힙을 나누는가?
>
> 스택은 빠르지만 작고, 메서드가 끝나면 사라진다.
> 객체는 여러 메서드에서 공유해야 하고, 크기도 다양하다.
> 그래서 넓고 유연한 공간인 힙에 저장하는 것이다.

#### Method Area: 클래스의 도서관

클래스 자체의 정보(설계도)가 저장되는 곳이다.
프로그램 시작 시 클래스가 로딩되면, 이곳에 클래스 구조, 메서드 코드, static 변수 등이 저장된다.

> **흔한 오해**: "static 변수는 스택에 있다" -- 아니다! Method Area에 있다.

## Chapter 2: 데이터를 담는 그릇 - 변수

### 2.1 변수란 무엇인가?

#### 비유: 변수는 이름표가 붙은 상자

여러분의 방에 상자가 여러 개 있다고 상상해보라.

- "나이"라고 적힌 상자에는 숫자 25가 들어있다.
- "이름"이라고 적힌 상자에는 "홍길동"이 들어있다.
- "키"라고 적힌 상자에는 175.5가 들어있다.

**변수**는 바로 이 **이름표가 붙은 상자**다.

```
  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │  나이   │  │  이름   │  │   키    │
  │┌───────┐│  │┌───────┐│  │┌───────┐│
  ││  25   ││  ││홍길동 ││  ││ 175.5 ││
  │└───────┘│  │└───────┘│  │└───────┘│
  └─────────┘  └─────────┘  └─────────┘
    int age    String name   double height
```

#### 왜? 타입이 필요한가?

"상자에 그냥 아무거나 넣으면 안 되는가?"

좋은 질문이다. 하지만 컴퓨터 입장에서 생각해보자.

컴퓨터의 메모리는 한정되어 있다.
정수 하나를 저장하려면 4바이트(32비트)가 필요하고,
실수 하나를 저장하려면 8바이트(64비트)가 필요하다.

상자를 만들 때 **크기를 미리 정하지 않으면**,
컴퓨터는 얼마나 큰 공간을 준비해야 하는지 알 수 없다.

> 이사할 때 짐의 종류를 알아야 박스 크기를 정하는 것과 같다.
> 책을 넣을 건지, 이불을 넣을 건지에 따라 필요한 상자 크기가 다르기 때문이다.

타입(Type)은 **상자의 크기와 종류를 결정**하는 것이다.

---

### 2.2 기본형 vs 참조형: 근본적인 차이

이것은 Java에서 **가장 중요한 개념 중 하나**다.
이것을 이해하지 못하면 수많은 버그에 시달리게 된다.

#### 기본형(Primitive Type): 상자 안에 값이 직접 들어있다

```java
int age = 25;
```

이 경우, 메모리의 상자에 **25라는 값 자체**가 들어있다.

```
  Stack
  ┌───────────┐
  │ age = 25  │    <-- 값 자체가 여기 있음
  └───────────┘
```

기본형은 8가지: `byte`, `short`, `int`, `long`, `float`, `double`, `char`, `boolean`

#### 참조형(Reference Type): 상자 안에 주소가 들어있다

```java
String name = new String("홍길동");
```

이 경우, 상자에는 **"홍길동"이라는 값이 아니라, 그 값이 있는 장소의 주소**가 들어있다.

```
  Stack                              Heap
  ┌─────────────────┐              ┌──────────────┐
  │ name = 0x1234   │ ──────────> │  "홍길동"     │
  └─────────────────┘              └──────────────┘
         주소                          실제 값
```

> **비유: 리모컨**
>
> 기본형 변수는 **TV 자체**를 들고 있는 것이다. (작은 TV라 가능)
> 참조형 변수는 **TV 리모컨**을 들고 있는 것이다. (리모컨으로 TV를 조작)
>
> 리모컨(참조)을 복사하면? 같은 TV를 가리키는 리모컨이 2개가 된다.
> 한 리모컨으로 채널을 바꾸면, 다른 리모컨으로 봐도 채널이 바뀌어 있다!

#### == 비교의 함정

이 차이 때문에 **==** 연산자가 완전히 다르게 동작한다.

```
  기본형: == 는 "값"을 비교
  int a = 10;
  int b = 10;
  a == b  =>  true  (값이 같으니까)

  참조형: == 는 "주소"를 비교
  String a = new String("hello");
  String b = new String("hello");
  a == b  =>  false  (값은 같지만, 다른 객체니까!)
```

```
  Stack                    Heap
  ┌──────────────┐        ┌──────────────┐
  │ a = 0x100    │ ────> │   "hello"    │  0x100
  ├──────────────┤        └──────────────┘
  │ b = 0x200    │ ────> ┌──────────────┐
  └──────────────┘        │   "hello"    │  0x200
                          └──────────────┘

  a == b  =>  0x100 == 0x200  =>  false!
  a.equals(b)  =>  "hello" == "hello"  =>  true!
```

> **흔한 실수**: `==`로 문자열을 비교하는 것은 Java 초보자의 대표적인 실수다.
> 참조형은 반드시 `.equals()`를 사용해야 한다.

#### String Pool: 문자열의 특수한 대우

그런데 이상한 점이 있다.

```java
String a = "hello";       // 리터럴로 생성
String b = "hello";       // 리터럴로 생성
System.out.println(a == b);  // true! 왜???
```

`new`를 사용하지 않고 리터럴("")로 문자열을 만들면,
Java는 **String Pool**이라는 특별한 공간에서 같은 문자열을 **재사용**한다.

```
  Stack                    String Pool (Heap 내부)
  ┌──────────────┐        ┌──────────────┐
  │ a = 0x100    │ ────> │              │
  ├──────────────┤  ┌──> │   "hello"    │  0x100
  │ b = 0x100    │ ─┘    │              │
  └──────────────┘        └──────────────┘

  같은 주소를 가리키므로 a == b 는 true!
```

> **왜?** String Pool이 존재하는가?
>
> 문자열은 프로그램에서 엄청나게 많이 사용된다.
> 같은 문자열을 매번 새로 만들면 메모리 낭비가 심하다.
> 그래서 Java는 리터럴 문자열을 Pool에서 재사용하여 메모리를 절약한다.

> **면접 팁**: "`==`와 `equals()`의 차이를 설명하라"는 거의 반드시 나오는 질문이다.
> String Pool까지 설명하면 높은 점수를 받을 수 있다.

### 2.3 왜 int가 가장 많이 쓰이는가?

Java의 정수형은 4가지다:

| 타입 | 크기 | 범위 |
|------|------|------|
| byte | 1바이트 | -128 ~ 127 |
| short | 2바이트 | -32,768 ~ 32,767 |
| **int** | **4바이트** | **약 -21억 ~ 21억** |
| long | 8바이트 | 매우 큰 수 |

그런데 왜 거의 항상 `int`를 쓰는가?

#### CPU 아키텍처와의 관계

현대 컴퓨터의 CPU는 대부분 **32비트 또는 64비트** 단위로 데이터를 처리한다.
`int`는 **32비트(4바이트)**로, CPU가 한 번에 처리하기 **가장 효율적인 크기**다.

`byte`나 `short`를 사용해도, CPU 내부에서는 어차피 32비트로 변환해서 처리한다.
즉, **메모리를 아끼려고 byte를 쓰더라도, 처리 속도는 int와 같거나 오히려 느릴 수 있다.**

> **비유**: 택배 상자를 보낸다고 할 때, 작은 물건이라도 최소 박스 규격이 정해져 있는 것과 같다.
> 작은 물건을 보내도 최소 박스 크기는 동일하다.

#### 언제 다른 타입을 쓰는가?

- **byte**: 파일 입출력, 네트워크 통신 (바이트 단위로 데이터를 다룰 때)
- **long**: 21억을 넘는 숫자 (시간 밀리초, 파일 크기 등)
- **double**: 소수점이 필요할 때 (금액 계산에는 BigDecimal 사용 권장!)

### 2.4 형변환이 필요한 이유

#### 작은 그릇 -> 큰 그릇: 자동 형변환 (묵시적)

작은 컵의 물을 큰 컵에 부으면 아무 문제 없다.

```
  ┌───┐          ┌───────┐
  │ 5 │  ────>   │   5   │
  └───┘          └───────┘
  int (4바이트)   long (8바이트)

  데이터 손실 없음! 자동으로 변환됨.
```

#### 큰 그릇 -> 작은 그릇: 강제 형변환 (명시적)

하지만 큰 컵의 물을 작은 컵에 부으면? **넘칩니다!**

```
  ┌───────────┐          ┌───┐
  │   300     │  ────>   │44 │  <-- 데이터 손실!
  └───────────┘          └───┘
  int (값: 300)          byte (최대 127, 넘침!)
```

Java는 이런 위험한 변환을 **자동으로 해주지 않는다.**
프로그래머가 **명시적으로 캐스팅**해야 한다.

> **왜?** 자동으로 안 해주는가?
>
> 데이터가 손실될 수 있기 때문이다.
> "나는 데이터 손실의 위험을 알고 있고, 그래도 변환하겠다"는 의사를 코드로 표현하는 것이다.

#### 실수하기 쉬운 함정: 정수 나눗셈

```java
int result = 5 / 2;    // 결과: 2 (2.5가 아님!)
```

왜? `int / int`의 결과는 `int`이기 때문이다.
소수점 이하는 **버려집니다**.

이것은 초보자가 가장 자주 빠지는 함정이다.
해결법: 한쪽을 `double`로 만들어야 한다.

```java
double result = 5.0 / 2;    // 결과: 2.5
double result = (double) 5 / 2;  // 결과: 2.5
```

## Chapter 3: 프로그램의 흐름을 제어하다

### 3.1 조건문: 갈림길에서의 선택

프로그램은 항상 위에서 아래로 순서대로 실행된다.
하지만 현실 세계에서는 **상황에 따라 다르게 행동**해야 한다.

"비가 오면 우산을 가져가고, 안 오면 그냥 나간다."

이것이 **조건문**이다.

#### if-else: 인생의 분기점

```
             [비가 오는가?]
              /         \
           예             아니오
            |               |
     [우산을 가져간다]  [그냥 나간다]
            \               /
             [집을 나선다]
```

`if-else`는 **예/아니오**로 갈라지는 모든 상황에 사용한다.
인생에서 매 순간 선택을 하듯, 프로그램도 매 순간 선택한다.

#### switch: 엘리베이터 버튼

`if-else`가 "예/아니오"의 분기점이라면,
`switch`는 **여러 개 중 하나를 고르는 것**이다.

비유: 엘리베이터에서 1층, 2층, 3층... 버튼 중 하나를 누르는 것.

```
         [몇 층?]
        / | | | \
       1  2  3  4  기타
       |  |  |  |   |
       v  v  v  v   v
```

`if-else`로도 가능하지만, 선택지가 많을 때 `switch`가 훨씬 깔끔하다.

#### Java 14+ switch expression: 왜 개선되었는가?

전통적인 `switch`에는 문제가 있었다:
- `break`를 빼먹으면 다음 case로 **떨어지는(fall-through)** 버그
- 코드가 장황함

새로운 switch expression은 이 문제를 해결한다.
화살표(`->`)를 사용하면 break가 필요 없고, 값을 직접 반환할 수 있다.

> **왜?** 기존 방식에 문제가 있다면 새로운 방식을 만드는 것.
> 이것이 Java가 발전하는 방식이다.

---

### 3.2 반복문: 같은 일을 반복하는 기계

컴퓨터의 가장 큰 장점은 **같은 일을 지치지 않고 반복**할 수 있다는 것이다.

#### for: 횟수가 정해진 반복

> "100번 절하기" -- 몇 번 해야 하는지 알고 있을 때

```
  시작 ──> [횟수 확인] ──> 조건 만족? ──> 실행 ──> 증가 ──> [횟수 확인] ──> ...
                              |
                          조건 불만족
                              |
                             끝
```

#### while: 조건이 충족될 때까지

> "배가 부를 때까지 먹기" -- 언제 끝날지 모를 때

횟수가 아니라 **조건**이 중요할 때 사용한다.
"언제까지? 조건이 거짓이 될 때까지."

#### do-while: 최소 한 번은 실행

> "메뉴판은 한 번은 봐야지" -- 일단 한 번 하고, 그 다음부터 조건 검사

while은 조건을 먼저 확인하지만, do-while은 **먼저 실행하고 조건을 확인**한다.
실무에서는 잘 쓰이지 않지만, "최소 한 번은 실행해야 하는" 상황에 사용한다.

#### 무한루프의 위험

```
  while(true) {
      // 여기서 영원히 돌아간다
      // 탈출 조건이 없으면 프로그램이 멈추지 않는다!
  }
```

무한루프는 **의도한 것인지 실수인지**가 중요하다.

- 의도한 무한루프: 서버 프로그램 (항상 켜져 있어야 하니까)
- 실수로 만든 무한루프: **버그** (컴퓨터가 멈추는 것처럼 보임)

> 반복문을 쓸 때는 항상 자문하라: "이 반복은 **언제 끝나는가?**"

---

### 3.3 왜 enhanced for를 선호하는가?

```java
// 일반 for (인덱스 사용)
for (int i = 0; i < list.size(); i++) {
    System.out.println(list.get(i));
}

// enhanced for (향상된 for)
for (String item : list) {
    System.out.println(item);
}
```

**왜 enhanced for를 선호하는가?**

1. **인덱스 실수 방지**: `i < list.size()` 대신 `i <= list.size()`로 쓰는 실수(off-by-one 에러)를 원천 봉쇄
2. **의도가 명확**: "리스트의 각 항목에 대해" → 코드만 봐도 의도를 알 수 있음
3. **코드가 짧고 깔끔**: 불필요한 인덱스 변수가 없음

**그럼 일반 for는 언제 쓰는가?**

- 인덱스 값 자체가 필요할 때 (몇 번째인지 알아야 할 때)
- 역순으로 순회할 때
- 한 번에 2칸씩 건너뛸 때
- 순회 중 요소를 삭제해야 할 때 (이 경우도 Iterator가 더 좋지만)

## Chapter 4: 객체지향이란 무엇인가? (핵심 장)

> 이 장은 Java의 **핵심**이자, 프로그래밍 패러다임의 **혁명**을 다룬다.
> 천천히, 꼼꼼히 읽어야 한다.

### 4.1 절차지향 vs 객체지향

#### 절차지향: 레시피

요리 레시피를 생각해보자.

```
1. 냄비에 물을 끓인다
2. 면을 넣는다
3. 3분간 기다린다
4. 스프를 넣는다
5. 저어준다
6. 그릇에 담는다
```

**위에서 아래로 순서대로 실행.** 이것이 절차지향 프로그래밍이다.
C언어가 대표적이다.

간단한 프로그램에서는 잘 동작한다. 하지만...

#### 문제: 프로그램이 커지면?

호텔 주방을 상상해보자. 100가지 메뉴를 만들어야 한다.
레시피 100개를 위에서 아래로 쭉 나열하면?

- 어디서부터 어디까지가 하나의 메뉴인지 헷갈림
- 같은 재료 손질 과정이 여러 레시피에 중복됨
- 레시피 하나를 수정하면 다른 레시피에 영향을 줄 수 있음
- 새로운 셰프(개발자)가 오면 전체를 이해하기 어려움

#### 객체지향: 레고 블록

이번에는 **레고**를 생각해보자.

레고의 핵심은 **독립적인 블록**이다.

```
  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
  │  바퀴  │  │  차체  │  │  엔진  │  │ 핸들   │
  │ 블록   │  │ 블록   │  │ 블록   │  │ 블록   │
  └────────┘  └────────┘  └────────┘  └────────┘
       \           \          /           /
        \           \        /           /
         ┌──────────────────────────────┐
         │          자동차              │
         │    (블록들을 조립한 결과)    │
         └──────────────────────────────┘
```

각 블록은:
- **독립적**이다 (바퀴 블록은 혼자서도 의미가 있다)
- **재사용** 가능하다 (같은 바퀴 블록을 트럭에도 쓸 수 있다)
- **교체** 가능하다 (바퀴를 큰 바퀴로 바꿔도 다른 블록은 영향 없다)

이것이 **객체지향 프로그래밍**의 핵심 아이디어다.

> **왜?** 객체지향이 등장했는가?
>
> 1960~70년대, 소프트웨어가 점점 커지면서 **"소프트웨어 위기(Software Crisis)"**가 왔다.
> 프로그램이 너무 복잡해져서, 버그를 잡기도 어렵고, 수정하기도 어렵고, 이해하기도 어려웠다.
>
> 해결책: **복잡한 시스템을 독립적인 부품(객체)으로 나누자.**
> 각 부품은 자신의 데이터와 기능을 스스로 관리하게 하자.
> 이것이 객체지향의 탄생 배경이다.

---

### 4.2 클래스와 객체: 설계도와 실체

#### 클래스 = 설계도

자동차 공장을 생각해보자.

자동차를 만들려면 먼저 **설계도**가 필요하다.
설계도에는 자동차의 속성(색상, 배기량, 좌석 수)과
행동(달리다, 멈추다, 경적 울리다)이 정의되어 있다.

```
  ┌─────────────────────────────────┐
  │         Car 설계도 (클래스)      │
  │                                  │
  │  속성 (필드):                    │
  │  - 색상 (color)                  │
  │  - 속도 (speed)                  │
  │  - 연료량 (fuel)                 │
  │                                  │
  │  행동 (메서드):                  │
  │  - 달리다 (drive)                │
  │  - 멈추다 (stop)                 │
  │  - 주유하다 (refuel)             │
  └─────────────────────────────────┘
```

하지만 **설계도는 자동차가 아니다.**
설계도로는 달릴 수 없다.

#### 객체 = 설계도로 만든 실체

설계도(클래스)를 바탕으로 **실제 자동차(객체)**를 만든다.
같은 설계도에서 여러 대의 자동차를 만들 수 있다.

```
  Car 설계도 (클래스)
       │
       ├──> 빨간 자동차 (객체1): color="빨강", speed=0, fuel=100
       │
       ├──> 파란 자동차 (객체2): color="파랑", speed=60, fuel=80
       │
       └──> 흰색 자동차 (객체3): color="흰색", speed=100, fuel=30
```

> **다른 비유들:**
> - 클래스 = **붕어빵 틀**, 객체 = **붕어빵** (같은 틀에서 여러 붕어빵 생산)
> - 클래스 = **쿠키 커터**, 객체 = **쿠키**
> - 클래스 = **주민등록증 양식**, 객체 = **당신의 주민등록증**

**핵심 공식:**

> **필드(상태) + 메서드(행동) = 객체**

현실 세계의 모든 것을 이 공식으로 표현할 수 있다.

| 객체 | 상태(필드) | 행동(메서드) |
|------|-----------|-------------|
| 학생 | 이름, 학번, 성적 | 공부하다, 시험보다 |
| 은행계좌 | 계좌번호, 잔액, 소유자 | 입금하다, 출금하다, 조회하다 |
| 스마트폰 | 배터리, 화면밝기, 볼륨 | 전화하다, 문자보내다, 충전하다 |

### 4.3 캡슐화: 왜 숨기는가?

#### 비유: 자동차 운전자와 엔진

여러분은 자동차를 운전할 줄 안다.
핸들을 돌리면 방향이 바뀌고, 페달을 밟으면 속도가 올라간다.

하지만 **엔진 내부에서 연료가 어떻게 폭발하는지** 알 필요가 없다.
**변속기가 어떤 원리로 기어를 바꾸는지** 알 필요가 없다.

복잡한 내부 구조는 **숨겨져 있고**, 운전자에게는 **핸들과 페달만 노출**된다.

이것이 **캡슐화(Encapsulation)**이다.

```
  ┌──────────────────────────────────────┐
  │             자동차 (객체)            │
  │                                      │
  │   [공개된 인터페이스]                │
  │   ┌──────┐ ┌──────┐ ┌──────┐       │
  │   │핸들  │ │페달  │ │기어  │       │
  │   └──────┘ └──────┘ └──────┘       │
  │                                      │
  │   - - - - 숨겨진 내부 구조 - - - -  │
  │   ┌────────────────────────────┐    │
  │   │ 엔진, 변속기, 전자 시스템  │    │
  │   │ (private - 직접 접근 불가) │    │
  │   └────────────────────────────┘    │
  └──────────────────────────────────────┘
```

#### 왜? 숨기는가?

**1. 보호: 잘못된 사용 방지**

만약 운전자가 엔진 내부를 직접 만질 수 있다면?
실수로 잘못 건드려서 자동차가 고장날 수 있다.

```
  만약 나이(age)를 직접 접근할 수 있다면?
  person.age = -5;     // 나이가 -5? 말이 안 됨!
  person.age = 99999;  // 이것도 문제!

  setter로 보호하면?
  public void setAge(int age) {
      if (age < 0 || age > 150) {
          throw new IllegalArgumentException("잘못된 나이!");
      }
      this.age = age;
  }
```

**2. 유연성: 내부를 바꿔도 외부에 영향 없음**

자동차 엔진을 가솔린에서 전기모터로 교체해도,
운전 방법(핸들, 페달)은 똑같다.

**3. 복잡성 관리: 알 필요 없는 것은 숨긴다**

> **TV 리모컨 비유**:
> 리모컨의 버튼만 누르면 된다.
> 리모컨 내부의 적외선 회로가 어떻게 동작하는지 알 필요 없다.
> `public` 메서드 = 리모컨의 버튼
> `private` 필드/메서드 = 리모컨 내부의 회로

> **면접 팁**: "캡슐화란 무엇이고 왜 필요한가?"
> 데이터 보호, 유연성, 복잡성 관리 세 가지를 자동차 비유와 함께 설명하면 훌륭하다.

### 4.4 상속: 코드의 유전자

#### 비유: 부모에서 자식으로 전달되는 특성

부모님에게서 외모, 성격, 재능 등을 물려받는 것처럼,
프로그래밍에서도 **부모 클래스의 특성을 자식 클래스가 물려받는다.**

```
            ┌─────────────┐
            │    동물      │
            │ (Animal)     │
            │              │
            │ 이름, 나이   │
            │ 먹다(), 자다()│
            └──────┬──────┘
                   │ 상속
          ┌────────┴────────┐
          v                  v
  ┌─────────────┐   ┌─────────────┐
  │     개       │   │   고양이    │
  │   (Dog)      │   │   (Cat)     │
  │              │   │             │
  │ + 짖다()    │   │ + 할퀴다() │
  └─────────────┘   └─────────────┘
```

- `Dog`은 `Animal`의 모든 것(이름, 나이, 먹다, 자다)을 물려받음
- 거기에 **자기만의 것(짖다)**을 추가함
- `Cat`도 마찬가지로 물려받고, **자기만의 것(할퀴다)**을 추가함

#### 왜? 상속을 사용하는가?

**1. 코드 중복 제거**

Dog과 Cat에 "이름", "나이", "먹다()", "자다()"를 각각 따로 만들면 코드가 중복된다.
공통 부분을 Animal에 한 번만 작성하고 물려받으면 중복이 사라진다.

**2. 다형성의 기반 (다음 절에서 자세히)**

상속이 있어야 "여러 형태"로 동작하는 것이 가능해진다.

#### 상속의 위험: 너무 깊은 상속 체인

```
  생물 → 동물 → 포유류 → 영장류 → 사람 → 학생 → 대학생 → 컴공학생
```

이렇게 상속이 깊어지면:
- 어떤 메서드가 어디서 정의된 건지 추적하기 어려움
- 부모 클래스를 수정하면 모든 자식에게 영향
- 코드를 이해하려면 모든 부모 클래스를 올라가며 봐야 함

> **황금 규칙**: 상속 깊이는 **2~3단계 이내**가 적절하다.

#### 상속보다 합성을 선호하라 (Composition over Inheritance)

이것은 경험 많은 개발자들 사이에서 널리 인정받는 원칙이다.

```
  상속: "나는 ~이다" (IS-A)
  합성: "나는 ~을 가지고 있다" (HAS-A)

  상속: class 전기차 extends 자동차     // 전기차는 자동차이다
  합성: class 자동차 {
           Engine engine;               // 자동차는 엔진을 가지고 있다
        }
```

합성의 장점:
- 런타임에 엔진을 교체할 수 있음 (유연성)
- 엔진 클래스를 수정해도 자동차에 미치는 영향이 적음 (느슨한 결합)

> **왜?** 상속보다 합성을 선호하는가?
>
> 상속은 **가장 강한 결합**이다. 부모가 변하면 자식은 반드시 영향을 받는다.
> 합성은 **느슨한 결합**이다. 부품을 교체하듯 유연하게 변경할 수 있다.
>
> "정말로 IS-A 관계일 때만 상속을, 그 외에는 합성을 사용하라."

### 4.5 다형성: 같은 명령, 다른 행동

#### 비유: "말해봐"

동물원에서 동물들에게 "말해봐!"라고 하면:

- 개는 "멍멍!"
- 고양이는 "야옹!"
- 오리는 "꽥꽥!"

**같은 명령**(말해봐)인데 **다른 행동**(각자의 울음소리)을 한다.
이것이 **다형성(Polymorphism)**이다.

```
        "말해봐!" (speak())
              │
    ┌─────────┼─────────┐
    v         v         v
  ┌─────┐  ┌─────┐  ┌─────┐
  │ 개  │  │고양이│  │오리 │
  │멍멍!│  │야옹! │  │꽥꽥!│
  └─────┘  └─────┘  └─────┘
```

#### 업캐스팅: 자식을 부모 타입으로 다루기

다형성의 핵심은 **업캐스팅**이다.

```java
Animal myPet = new Dog();   // Dog을 Animal 타입으로 다룸
myPet.speak();              // "멍멍!" (Dog의 speak 실행)
```

Dog 객체를 Animal 타입 변수에 넣었다.
하지만 `speak()`를 호출하면, **실제 객체인 Dog의 speak()**가 실행된다.

> 택배 상자에 "과일"이라고 적혀있지만, 열어보면 "사과"가 들어있는 것과 같다.
> 겉으로는 Animal이지만, 안에는 Dog이 들어있다.

#### 왜? 다형성이 강력한가?

**실전 예시: 결제 시스템**

온라인 쇼핑몰에서 결제 수단이 여러 개 있다.

```
                 ┌──────────────┐
                 │  Payment     │
                 │ (결제)       │
                 │              │
                 │ + pay()      │
                 └──────┬───────┘
                        │
          ┌─────────────┼─────────────┐
          v             v             v
  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
  │ KakaoPay     │ │ NaverPay     │ │ CardPayment  │
  │              │ │              │ │              │
  │ pay() {      │ │ pay() {      │ │ pay() {      │
  │  카카오 결제 │ │  네이버 결제 │ │  카드 결제   │
  │ }            │ │ }            │ │ }            │
  └──────────────┘ └──────────────┘ └──────────────┘
```

다형성이 없다면:

```
  if (결제방식 == "카카오") {
      카카오 결제 로직...
  } else if (결제방식 == "네이버") {
      네이버 결제 로직...
  } else if (결제방식 == "카드") {
      카드 결제 로직...
  }
  // 새 결제 방식이 추가될 때마다 if문 추가... 끝이 없다
```

다형성이 있다면:

```
  Payment payment = getPaymentMethod();  // 어떤 결제든 OK
  payment.pay();                          // 알아서 해당 방식으로 결제됨!
  // 새 결제 방식 추가? 새 클래스만 만들면 됨. 여기는 수정 불필요!
```

> 이것이 바로 **OCP (Open-Closed Principle)**: 확장에는 열려있고, 수정에는 닫혀있다.
> 새로운 기능을 추가할 때, **기존 코드를 수정하지 않아도 된다.**

### 4.6 추상 클래스 vs 인터페이스: 언제 무엇을?

이 둘의 차이는 Java 면접에서 **반드시** 나오는 질문이다.
하지만 단순히 문법적 차이가 아니라, **설계 철학의 차이**를 이해해야 한다.

#### 추상 클래스: "나는 ~이다" (IS-A)

추상 클래스는 **"불완전한 설계도"**다.

```
  ┌─────────────────────────────┐
  │     Animal (추상 클래스)     │
  │                              │
  │  이름, 나이      (완성됨)   │
  │  먹다()          (완성됨)   │
  │  speak() = ???   (미완성!)  │
  │                              │
  │  "나는 동물이라는 건 아는데, │
  │   어떻게 우는지는 모르겠어"  │
  └──────────────┬──────────────┘
                 │
        ┌────────┴────────┐
        v                  v
  ┌───────────┐     ┌───────────┐
  │    Dog     │     │    Cat    │
  │ speak() {  │     │ speak() { │
  │  "멍멍"   │     │  "야옹"  │
  │ }          │     │ }         │
  └───────────┘     └───────────┘
```

추상 클래스는 **공통된 상태(필드)와 행동(메서드)**을 공유하면서,
일부 행동은 **자식 클래스가 반드시 구현하도록** 강제한다.

#### 인터페이스: "나는 ~할 수 있다" (CAN-DO)

인터페이스는 **"능력의 계약서"**다.

```
  ┌────────────┐    ┌─────────────┐    ┌──────────────┐
  │  Flyable   │    │  Swimmable  │    │  Runnable    │
  │ (날 수 있는)│    │ (수영 가능) │    │ (달릴 수 있는)│
  │            │    │             │    │              │
  │ + fly()   │    │ + swim()   │    │ + run()     │
  └─────┬──────┘    └──────┬──────┘    └──────┬───────┘
        │                  │                   │
        └──────────┐       │                   │
                   v       v                   v
              ┌─────────────────┐    ┌─────────────┐
              │      오리       │    │     개       │
              │ fly(), swim()  │    │   run()     │
              └─────────────────┘    └─────────────┘
```

오리는 **날 수도 있고**, **수영도 할 수 있다** -- 여러 인터페이스를 동시에 구현 가능.
클래스는 하나만 상속받을 수 있지만, 인터페이스는 여러 개 구현할 수 있다.

#### 선택 가이드

```
  "이 관계가 IS-A인가, CAN-DO인가?"

  Dog IS-A Animal (개는 동물이다)        → 추상 클래스 (상속)
  Duck CAN-DO Fly (오리는 날 수 있다)    → 인터페이스 (구현)

  상태(필드)를 공유해야 하는가?
  YES → 추상 클래스
  NO  → 인터페이스

  다중 구현이 필요한가?
  YES → 인터페이스 (Java는 다중 상속 불가)
  NO  → 둘 다 가능
```

#### Java 8 이후: 인터페이스의 변화

Java 8부터 인터페이스에 **default method**가 추가되었다.
즉, 인터페이스도 구현된 메서드를 가질 수 있게 되었다.

> **왜?** 이런 변화가 있었는가?
>
> 기존 인터페이스에 새 메서드를 추가하면,
> 그 인터페이스를 구현한 **모든 클래스**가 새 메서드를 구현해야 했다.
> 수천 개의 클래스가 영향을 받는 대참사!
>
> default method는 "기본 구현을 제공하니, 필요한 클래스만 재정의해라"는 해결책이다.

> **현대 Java의 트렌드**: 추상 클래스보다 **인터페이스를 선호**한다.
> 인터페이스에 default method가 추가되면서, 추상 클래스의 필요성이 줄었다.

### 4.7 SOLID 원칙 (간단히)

좋은 객체지향 설계를 위한 5가지 원칙이다.
지금 당장 모두 이해할 필요는 없지만, **이름과 핵심만** 기억해두자.

| 원칙 | 이름 | 한 줄 설명 | 비유 |
|------|------|-----------|------|
| **S** | 단일 책임 원칙 (SRP) | 하나의 클래스는 하나의 책임만 | 요리사는 요리만, 서빙은 웨이터가 |
| **O** | 개방-폐쇄 원칙 (OCP) | 확장에 열려있고, 수정에 닫혀있어야 | 플러그를 꽂으면 새 기능 추가 (본체 개조 X) |
| **L** | 리스코프 치환 원칙 (LSP) | 자식은 부모를 대체할 수 있어야 | 알바생도 정직원이 하던 일을 할 수 있어야 |
| **I** | 인터페이스 분리 원칙 (ISP) | 큰 인터페이스를 작은 것으로 분리 | 만능 리모컨보다 TV 리모컨, 에어컨 리모컨이 낫다 |
| **D** | 의존성 역전 원칙 (DIP) | 구체적인 것이 아니라 추상적인 것에 의존 | "삼성 TV"가 아니라 "TV"에 의존 |

> 이 원칙들은 처음에는 추상적으로 느껴진다.
> 실제 코드를 작성하면서 "아, 이래서 그런 거구나!" 하고 깨닫는 순간이 올 것이다.

## Chapter 5: 데이터를 효율적으로 다루다 - 컬렉션

### 5.1 배열의 한계와 컬렉션의 등장

#### 배열: 고정된 크기의 방

배열은 **아파트**와 같다. 한 번 지으면 방의 개수를 바꿀 수 없다.

```
  int[] scores = new int[5];

  ┌─────┬─────┬─────┬─────┬─────┐
  │  0  │  1  │  2  │  3  │  4  │  <- 인덱스
  ├─────┼─────┼─────┼─────┼─────┤
  │ 90  │ 85  │ 77  │ 92  │ 88  │  <- 값
  └─────┴─────┴─────┴─────┴─────┘

  6번째 점수를 추가하고 싶다면? 불가능! 크기가 5로 고정!
```

배열의 한계:
- **크기 고정**: 데이터가 몇 개인지 미리 알아야 함
- **삽입/삭제 어려움**: 중간에 넣거나 빼려면 나머지를 전부 밀거나 당겨야 함
- **기능 부족**: 정렬, 검색 등을 직접 구현해야 함

#### 왜? 컬렉션 프레임워크가 필요한가?

현실에서 데이터의 개수는 미리 알 수 없는 경우가 대부분이다.
회원 수, 주문 수, 게시글 수... 모두 동적으로 변한다.

**컬렉션 프레임워크**는 이런 문제를 해결하기 위해 Java가 제공하는 **자료구조 모음**이다.

```
                    ┌─────────────┐
                    │  Collection │                │    Map     │
                    └──────┬──────┘                └─────┬──────┘
                           │                             │
              ┌────────────┤                        ┌────┴────┐
              v            v                        v         v
        ┌──────────┐ ┌──────────┐              ┌─────────┐ ┌─────────┐
        │   List   │ │   Set    │              │ HashMap │ │ TreeMap │
        │ (순서O,  │ │ (순서X,  │              └─────────┘ └─────────┘
        │  중복O)  │ │  중복X)  │
        └────┬─────┘ └────┬─────┘
             │             │
        ┌────┴────┐   ┌───┴───┐
        v         v   v       v
    ArrayList  LinkedList HashSet TreeSet

  ※ Map은 Collection 인터페이스를 상속하지 않는 별도의 인터페이스이다.
```

---

### 5.2 List, Set, Map: 어떤 상황에서 무엇을?

#### List: 순서가 중요하고 중복 허용 (출석부)

학교 출석부를 생각해보자.
- 1번 김철수, 2번 이영희, 3번 박민수...
- **순서**가 있고, 같은 이름이 있을 수도 있음(동명이인)

List는 **데이터의 순서를 보장**하고, **중복을 허용**한다.

사용 예: 장바구니 목록, 채팅 메시지, 순위 리스트

#### Set: 중복을 허용하지 않음 (회원 ID 집합)

회원 ID를 모아놓은 집합을 생각해보자.
- 같은 ID가 두 개 있으면 안 됨
- 순서는 중요하지 않음 (누가 먼저 가입했는지는 다른 문제)

Set은 **중복을 자동으로 제거**한다.

사용 예: 태그 목록, 방문자 IP, 고유 값 추출

#### Map: 키-값 쌍 (사전)

영어 사전을 생각해보자.
- "apple" → "사과"
- "banana" → "바나나"
- **단어(키)**로 **뜻(값)**을 찾음

Map은 **키(key)**와 **값(value)**의 쌍으로 데이터를 저장한다.

사용 예: 사용자 ID → 사용자 정보, 상품코드 → 상품 정보, 설정 이름 → 설정 값

#### 선택 가이드 플로우차트

```
  데이터를 저장하려는데...

  키-값 쌍인가? ──── YES ──> Map (HashMap)
       │
       NO
       │
  중복을 허용하는가? ── NO ──> Set (HashSet)
       │
      YES
       │
  순서가 중요한가? ── YES ──> List (ArrayList)
       │
       NO
       │
  ──> 그래도 List (ArrayList)  // 대부분 List로 충분
```

---

### 5.3 ArrayList vs LinkedList: 진짜 차이

#### ArrayList: 아파트

ArrayList는 **아파트**와 같다.
각 호실에 번호(인덱스)가 있어서, "503호"라고 하면 바로 찾아갈 수 있다.

```
  ArrayList (연속된 메모리 공간)
  ┌─────┬─────┬─────┬─────┬─────┐
  │  0  │  1  │  2  │  3  │  4  │
  ├─────┼─────┼─────┼─────┼─────┤
  │ 사과│바나나│ 포도│ 딸기│수박  │
  └─────┴─────┴─────┴─────┴─────┘

  장점: "2번 뭐야?" → 바로 접근 가능! O(1)
  단점: 중간에 삽입/삭제 → 나머지를 전부 밀거나 당겨야 함 O(n)
```

중간에 삽입하면?

```
  "귤"을 1번 위치에 삽입하기

  [사과][바나나][포도][딸기][수박]
        ↓ 밀기 →→→→→
  [사과][  귤 ][바나나][포도][딸기][수박]

  뒤의 모든 요소를 한 칸씩 밀어야 함!
```

#### LinkedList: 기차

LinkedList는 **기차**와 같다.
각 칸이 다음 칸의 위치를 알고 있어서, 연결되어 있다.

```
  LinkedList (떨어진 메모리 공간, 포인터로 연결)

  [사과] → [바나나] → [포도] → [딸기] → [수박] → null

  장점: 삽입/삭제 자체는 연결만 바꾸면 됨 O(1) (Iterator 사용 시)
  단점: 특정 위치를 찾으려면 처음부터 하나씩 세어가야 함 O(n)
```

중간에 삽입하면?

```
  "귤"을 1번 위치에 삽입하기

  [사과] → [바나나] → [포도] → ...
       ↘            ↗
        [  귤  ]

  연결(화살표)만 바꾸면 끝! 나머지는 그대로!
```

#### 실무에서 99%는 ArrayList

이론적으로 LinkedList가 삽입/삭제에서 유리하지만,
**실무에서는 거의 항상 ArrayList가 더 빠르다.**

이유:
1. **CPU 캐시 효율성**: ArrayList는 메모리가 연속되어 있어 CPU 캐시에 잘 올라감
2. **대부분의 작업은 조회**: 실제 프로그램에서 삽입/삭제보다 조회가 훨씬 많음
3. **끝에 추가하는 경우가 대부분**: 중간 삽입은 드물고, 끝에 추가하는 건 ArrayList도 빠름

> **결론**: 특별한 이유가 없으면 **ArrayList를 쓰는 것이 좋다.**

### 5.4 HashMap의 동작 원리

HashMap은 Java에서 **가장 많이 사용되는 자료구조 중 하나**다.
면접에서도 자주 나오니, 동작 원리를 이해해둬야 한다.

#### 해시 함수란?

해시 함수는 **임의의 데이터를 고정된 크기의 숫자로 변환**하는 함수다.

```
  해시 함수
  "apple"   ──>  12
  "banana"  ──>  7
  "cherry"  ──>  3
  "date"    ──>  12  <-- 다른 입력인데 같은 결과! (해시 충돌)
```

비유: 도서관의 **분류 번호 시스템**
- 책 제목이 아무리 길어도, "520.3"처럼 짧은 번호로 변환
- 번호를 보면 어느 서가에 있는지 바로 알 수 있음

#### 버킷 구조

HashMap은 내부적으로 **배열(버킷)**을 가지고 있다.
키의 해시값으로 어느 버킷에 저장할지 결정한다.

```
  HashMap 내부 구조 (버킷 배열)

  버킷 인덱스
  ┌────┐
  │ 0  │ → null
  ├────┤
  │ 1  │ → null
  ├────┤
  │ 2  │ → null
  ├────┤
  │ 3  │ → [cherry: 체리]
  ├────┤
  │ 4  │ → null
  ├────┤
  │ 5  │ → null
  ├────┤
  │ 6  │ → null
  ├────┤
  │ 7  │ → [banana: 바나나]
  ├────┤
  │ ...│
  ├────┤
  │ 12 │ → [apple: 사과] → [date: 대추]  <-- 충돌! 체이닝
  └────┘
```

#### 해시 충돌과 해결

서로 다른 키가 같은 해시값을 가지면 **해시 충돌**이 발생한다.
Java의 HashMap은 **체이닝(Chaining)** 방식으로 해결한다.
같은 버킷에 연결 리스트(또는 트리)로 여러 항목을 저장한다.

#### equals()와 hashCode()를 같이 오버라이드해야 하는 이유

이것은 Java에서 **가장 중요한 규칙 중 하나**다.

HashMap에서 키를 찾는 과정:
1. `hashCode()`로 **버킷 위치**를 찾음
2. `equals()`로 **정확한 키**를 찾음

```
  "apple"을 찾자!

  1단계: hashCode("apple") = 12  → 12번 버킷으로!
  2단계: 12번 버킷에 [apple][date]가 있음
         equals("apple", "apple") = true → 찾았다!
```

만약 `hashCode()`만 오버라이드하고 `equals()`를 안 하면?
→ 같은 버킷까지는 찾아가지만, 정확한 키를 못 찾음

만약 `equals()`만 오버라이드하고 `hashCode()`를 안 하면?
→ 다른 버킷으로 가버려서, 아예 찾지도 못함

> **규칙**: `equals()`를 오버라이드하면 반드시 `hashCode()`도 오버라이드하라!
>
> 비유: 도서관에서 책을 찾을 때
> - hashCode = 서가 번호 (몇 번 서가에 있는지)
> - equals = 정확한 책 제목 확인 (서가에서 어떤 책인지)
> 둘 다 있어야 책을 찾을 수 있다!

> **면접 팁**: "equals()와 hashCode()의 관계를 설명하라"는 중급 면접에서 자주 나온다.
> HashMap의 동작 원리와 연결해서 설명하면 좋은 인상을 준다.

### 5.5 제네릭: 타입 안전성의 보장

#### 제네릭이 없던 시절의 문제

Java 5 이전에는 제네릭이 없었다.
컬렉션에 어떤 타입이든 넣을 수 있었고, 꺼낼 때 캐스팅해야 했다.

```
  제네릭 없는 세계:

  List list = new ArrayList();
  list.add("문자열");
  list.add(123);          // 숫자도 들어감!
  list.add(new Dog());    // 개 객체도 들어감!

  // 꺼낼 때...
  String s = (String) list.get(1);  // 실행 시 에러! 123은 String이 아님!
  // ClassCastException 폭탄!
```

비유: **아무거나 넣을 수 있는 상자**

```
  ┌───────────────────┐
  │     ??? 상자      │
  │                    │
  │  "문자열"         │
  │  123              │
  │  Dog 객체         │
  │                    │
  │  꺼낼 때 뭐가     │
  │  나올지 모름!     │
  └───────────────────┘
```

#### 제네릭으로 해결

```
  제네릭이 있는 세계:

  List<String> list = new ArrayList<>();
  list.add("문자열");     // OK
  list.add(123);          // 컴파일 에러! String만 가능!
```

비유: **"사과 전용 상자"**

```
  ┌───────────────────┐
  │   사과 전용 상자   │
  │   <Apple>          │
  │                    │
  │  🍎 사과1         │
  │  🍎 사과2         │
  │  🍌 바나나? ──── X │  <-- 넣을 수 없음!
  │                    │
  └───────────────────┘
```

> **왜?** 제네릭이 강력한가?
>
> **실행 시 터지는 에러를 컴파일 시에 잡아준다.**
>
> 실행 시 에러 = 사용자가 보는 에러 (치명적!)
> 컴파일 시 에러 = 개발자만 보는 에러 (개발 중에 수정 가능)
>
> 에러는 **빨리 발견할수록** 좋다.
> 제네릭은 에러를 가장 빨리 (컴파일 시점에) 발견하게 해준다.

## Chapter 6: 함수형 프로그래밍으로의 진화

### 6.1 왜 함수형 프로그래밍인가?

Java는 원래 **순수 객체지향 언어**였다.
그런데 Java 8(2014년)부터 **함수형 프로그래밍** 요소를 도입했다.

왜?

#### 명령형 vs 선언형

**명령형 (Imperative)**: "이렇게 해라" (How)

```
  요리사에게: "냄비를 꺼내고, 물을 넣고, 불을 켜고,
              면을 넣고, 3분 기다리고, 스프를 넣고, 저어라."
```

**선언형 (Declarative)**: "이것을 원한다" (What)

```
  요리사에게: "라면 하나 주세요."
```

함수형 프로그래밍은 **선언형**에 가깝다.
**"어떻게"보다 "무엇을"**에 집중한다.

비유: 택시를 탈 때

- 명령형: "직진하라, 다음 사거리에서 좌회전, 200m 직진, 우회전..."
- 선언형: "강남역으로 가라." (어떻게 갈지는 기사가 알아서)

> **왜?** 선언형이 좋은가?
>
> 1. **가독성**: 코드가 "무엇을 하는지" 바로 보임
> 2. **실수 감소**: 구현 세부사항에서 실수할 가능성이 줄어듦
> 3. **유지보수**: 의도가 명확하므로 나중에 수정하기 쉬움

---

### 6.2 Lambda: 이름 없는 함수

#### 왜 Lambda가 등장했는가?

Java에서 "동작"을 전달하려면, 원래 **익명 클래스**를 사용해야 했다.

비유로 설명하면:

> "커피 한 잔만 주문하는데 10장짜리 주문서를 작성해야 하는 상황"

전통적인 방식은 너무 **장황(verbose)**했다.

```
  // 익명 클래스 (전통 방식) - 커피 한 잔 주문에 10줄
  button.setOnClickListener(new OnClickListener() {
      @Override
      public void onClick(View v) {
          System.out.println("클릭됨!");
      }
  });

  // Lambda (현대 방식) - 핵심만 딱 1줄
  button.setOnClickListener(v -> System.out.println("클릭됨!"));
```

#### 비유: 이름 없는 일회용 레시피

평소에는 요리법에 이름을 붙인다: "김치찌개 레시피", "된장찌개 레시피"

하지만 한 번만 쓰고 버릴 간단한 레시피라면?
이름 없이 그냥 적으면 된다.

```
  일반 메서드: "김치찌개_만들기"라는 이름이 있음 (재사용)
  Lambda:      이름 없음. 그 자리에서 한 번 쓰고 끝 (일회용)
```

Lambda는 **짧고, 이름이 없고, 한 번 쓸 동작을 간결하게 표현**하는 도구다.

> **흔한 오해**: "Lambda를 쓰면 무조건 좋다"
>
> Lambda가 너무 길어지면(5줄 이상) 오히려 가독성이 떨어진다.
> 그럴 때는 일반 메서드로 추출하는 것이 낫다.

---

### 6.3 Stream: 데이터의 파이프라인

#### 비유: 공장의 컨베이어 벨트

공장에서 제품을 만드는 과정을 생각해보자.

```
  [원재료]                                              [완제품]
     │                                                     ^
     v                                                     │
  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
  │ 원재료│ -> │ 세척 │ -> │ 불량 │ -> │ 포장 │ -> │ 수집 │
  │ 투입 │    │      │    │ 제거 │    │      │    │      │
  └──────┘    └──────┘    └──────┘    └──────┘    └──────┘
  (stream())   (map)      (filter)    (map)      (collect)
```

Stream은 **데이터가 컨베이어 벨트를 따라 흘러가면서 가공되는 것**이다.

**명령형 (전통 방식)**: 하나하나 직접 처리

```
  직원이 직접:
  1. 원재료 상자를 열어 하나씩 꺼낸다
  2. 각각을 세척한다
  3. 불량품인지 확인한다
  4. 불량이 아니면 포장한다
  5. 새 상자에 넣는다
```

**선언형 (Stream 방식)**: 파이프라인을 설정

```
  컨베이어 벨트에:
  세척기 → 불량검출기 → 포장기를 설치하고, 재료를 흘려보낸다.
  자동으로 처리되어 끝에서 완제품이 나온다.
```

#### 중간 연산과 최종 연산

| 종류 | 역할 | 비유 |
|------|------|------|
| **중간 연산** | 데이터를 변환/필터링 | 컨베이어 벨트의 각 기계 |
| **최종 연산** | 결과를 만들어냄 | 컨베이어 벨트 끝의 수집 상자 |

중간 연산: `filter()`, `map()`, `sorted()`, `distinct()` ...
최종 연산: `collect()`, `forEach()`, `count()`, `reduce()` ...

> **핵심 규칙**: 최종 연산이 호출되기 전까지, 중간 연산은 **실행되지 않습니다!**

#### Lazy Evaluation: 왜 바로 실행하지 않는가?

```
  100만 개의 데이터에서 조건에 맞는 3개만 찾고 싶다.

  즉시 실행(Eager): 100만 개를 모두 필터링하고, 그 중에서 3개를 고른다.
  지연 실행(Lazy):  하나씩 확인하면서 3개가 모이면 바로 멈춘다.
```

Lazy Evaluation은 **필요한 만큼만 처리**하므로 효율적이다.
Stream이 lazy한 이유가 바로 이것이다.

> **비유**: 서점에서 좋은 책 3권을 찾고 싶다.
> - Eager: 서점의 모든 책을 다 읽어보고, 좋은 것 3권을 고른다. (비효율!)
> - Lazy: 한 권씩 읽어보다가 좋은 책 3권이 모이면 그만둔다. (효율!)

---

### 6.4 Optional: null과의 전쟁

#### NullPointerException: Java 개발자의 영원한 적

null 참조의 발명자 토니 호어(Tony Hoare)는 null을 만든 것을
**"10억 달러짜리 실수"**라고 고백했다.

```
  String name = null;
  int length = name.length();  // NullPointerException! 프로그램 사망!
```

null은 **"아무것도 없음"**을 의미한다.
문제는, 어떤 변수가 null인지 아닌지 **겉으로 보이지 않는다**는 것이다.

비유: **지뢰밭**
- 코드 곳곳에 null이 숨어있고
- 언제 밟을지(NullPointerException) 모른다

#### Optional: null을 명시적으로 처리

Optional은 **"값이 있을 수도 있고 없을 수도 있다"**를 명시적으로 표현한다.

비유: **선물 상자**

```
  일반 변수:   값이 있는지 없는지 열어봐야 안다 (위험!)

  Optional:    상자에 "내용물 있음" 또는 "비어있음"이라고 적혀있다 (안전!)

  ┌──────────────┐        ┌──────────────┐
  │  Optional    │        │  Optional    │
  │ ┌──────────┐ │        │              │
  │ │  "홍길동" │ │        │   (비어있음) │
  │ └──────────┘ │        │              │
  │   isPresent  │        │   isEmpty    │
  │   = true     │        │   = true     │
  └──────────────┘        └──────────────┘
```

#### Optional을 잘못 사용하는 방법 (Anti-Pattern)

> **주의**: Optional을 사용한다고 해서 무조건 좋은 코드가 되는 것은 아니다.

흔한 잘못된 사용:

1. **Optional을 필드로 사용** -- Optional은 반환값에만 사용하는 것이 권장된다
2. **Optional을 메서드 파라미터로 사용** -- 호출자를 불편하게 만든다
3. **`isPresent()` + `get()` 조합** -- 이러면 null 체크와 다를 바 없다
4. **컬렉션을 Optional로 감싸기** -- 빈 컬렉션을 반환하는 것이 더 좋다

> **좋은 사용법**: `orElse()`, `orElseGet()`, `orElseThrow()`, `map()`, `flatMap()`
> 이 메서드들을 체이닝해서 사용하는 것이 Optional의 올바른 사용법이다.

## Chapter 7: 예외처리 - 실패에 대비하다

### 7.1 왜 예외처리가 필요한가?

프로그램은 **항상 실패할 수 있다.**

- 파일을 열려는데 파일이 없으면?
- 네트워크로 데이터를 보내는데 인터넷이 끊기면?
- 숫자를 0으로 나누면?
- 사용자가 예상과 다른 입력을 하면?

예외처리가 없으면, 프로그램은 **그냥 죽는다.**

```
  예외처리 없는 세계:

  사용자: "파일 열어줘"
  프로그램: (파일 없음) → 에러 발생 → 프로그램 종료
  사용자: "????? 내 작업하던 거 다 날아갔어!"
```

```
  예외처리 있는 세계:

  사용자: "파일 열어줘"
  프로그램: (파일 없음) → "파일을 찾을 수 없습니다. 경로를 확인해주세요."
  사용자: "아, 경로를 잘못 입력했구나."
```

비유: **안전벨트와 에어백**

자동차 사고가 나면 안전벨트와 에어백이 탑승자를 보호한다.
사고(예외)가 나지 않으면 좋지만, **나더라도 대비가 되어 있어야** 한다.

#### 예외의 계층 구조

```
                    ┌─────────────┐
                    │  Throwable  │
                    └──────┬──────┘
                           │
                ┌──────────┴──────────┐
                v                      v
        ┌─────────────┐       ┌─────────────┐
        │    Error     │       │  Exception  │
        │ (심각한 오류)│       │ (복구 가능) │
        │              │       │             │
        │ OutOfMemory  │       │     ┌───────┴────────┐
        │ StackOverflow│       │     v                 v
        └─────────────┘       │  ┌────────────┐ ┌──────────────┐
                               │  │  Checked   │ │  Unchecked   │
                               │  │ Exception  │ │ (Runtime     │
                               │  │            │ │  Exception)  │
                               │  │ IOException│ │ NPE, IOOBE  │
                               │  │ SQLException│ │ ClassCast   │
                               │  └────────────┘ └──────────────┘
                               └─────────────────────────────────┘
```

- **Error**: 프로그래머가 처리할 수 없는 심각한 오류 (메모리 부족 등). 건드리지 말아야 한다.
- **Checked Exception**: 예측 가능한 외부 실패. **반드시 처리해야 함.**
- **Unchecked Exception**: 프로그래머의 실수. **처리를 강제하지 않음.**

---

### 7.2 Checked vs Unchecked: 근본적 차이

이 구분은 Java의 독특한 설계이며, **많은 논쟁**이 있는 부분이다.

#### Checked Exception: 예측 가능한 실패

파일을 열 때, 파일이 없을 수 있다는 것은 **예측 가능**하다.
네트워크 요청 시, 연결이 끊길 수 있다는 것도 **예측 가능**하다.

이런 **예측 가능한 실패**에 대해 Java는 **반드시 대비하라고 강제**한다.

비유: **비행기 탑승 전 안전 교육**
"비상구는 여기 있고, 산소마스크는 이렇게 쓰고, 구명조끼는 이렇게..."
비상 상황이 일어날 가능성은 낮지만, **반드시** 교육을 받아야 한다.

#### Unchecked Exception: 프로그래머의 실수

```
  int[] arr = {1, 2, 3};
  int value = arr[10];  // ArrayIndexOutOfBoundsException!
```

이것은 **프로그래머의 실수**다.
Java는 이런 실수에 대해 처리를 강제하지 않는다.

> **왜?** 강제하지 않는가?
>
> 만약 모든 예외를 처리하도록 강제하면,
> 코드의 모든 줄에 try-catch를 써야 한다.
> 그러면 코드가 너무 장황해져서 오히려 읽기 어려워진다.
>
> 프로그래머의 실수는 **예외를 처리하는 것이 아니라, 실수를 하지 않는 것**이 정답이다.

#### 최근 트렌드: Unchecked 예외 선호

흥미롭게도, 현대 Java 개발에서는 **Unchecked 예외를 더 선호**하는 추세다.

이유:
1. Checked 예외는 코드를 장황하게 만듦
2. catch 블록에서 제대로 처리하지 않고 무시하는 경우가 많음
3. Spring Framework 등 현대 프레임워크가 Unchecked 예외를 사용

> **중요**: 이것은 "예외처리를 하지 말라"는 뜻이 아니다.
> "강제하지 않되, 필요한 곳에서 자발적으로 처리하라"는 철학이다.

---

### 7.3 예외처리 Best Practice

#### 1. 구체적인 예외를 잡아라

```
  나쁜 예: catch (Exception e)      // 모든 예외를 뭉뚱그려서 잡음
  좋은 예: catch (FileNotFoundException e)  // 정확히 어떤 예외인지 명시
```

비유: 병원에서 "어디가 아픈가?" → "아프다" (X) → "오른쪽 무릎이 아프다" (O)
구체적이어야 정확한 치료(대응)가 가능하다.

#### 2. 예외를 삼키지 마라

```
  // 최악의 코드 - 절대 하지 마세요!
  try {
      riskyOperation();
  } catch (Exception e) {
      // 아무것도 안 함... 예외를 삼켜버림!
  }
```

비유: 화재 경보기가 울리는데, **소리를 끄고 아무것도 안 하는 것**과 같다.
문제는 여전히 존재하지만, 아무도 모른다. 나중에 더 큰 재앙이 온다.

최소한 로그라도 남겨야 한다:

```java
catch (Exception e) {
    log.error("파일 처리 중 오류 발생", e);
    throw new BusinessException("파일 처리 실패", e);
}
```

#### 3. 비즈니스 예외는 커스텀 예외로

```
  일반 예외: RuntimeException("잔액 부족")
  → 어디서 발생한 건지, 어떤 종류의 오류인지 불명확

  커스텀 예외: InsufficientBalanceException(계좌번호, 요청금액, 잔액)
  → 명확하고, 필요한 정보를 모두 담고 있음
```

비유: 병원 진단서에 "아픔"이라고만 쓰면 안 된다.
"우측 슬개골 인대 염증"이라고 구체적으로 써야 치료가 가능하다.

#### 4. finally 대신 try-with-resources

파일, 데이터베이스 연결 등 **자원(resource)**을 사용한 후에는 반드시 닫아야 한다.

전통적인 방식(finally)은 코드가 지저분해진다.
Java 7부터 도입된 **try-with-resources**를 사용하면 자동으로 닫힌다.

> **왜?** 자원을 닫지 않으면?
>
> 메모리 누수, 파일 잠금, 데이터베이스 연결 고갈 등
> 심각한 문제가 발생한다.
> try-with-resources는 이런 실수를 원천 봉쇄한다.

## Chapter 8: Java의 현대적 기능들

Java는 정체된 언어가 아니다.
6개월마다 새 버전이 출시되며, **개발자의 생산성을 높이는 기능**들이 계속 추가되고 있다.

### 8.1 record: DTO를 위한 완벽한 도구

#### 왜 record가 만들어졌는가?

Java에서 데이터를 담는 클래스를 만들려면, 엄청나게 많은 코드가 필요했다.

```
  이름, 나이만 담는 간단한 클래스인데...

  - private 필드 2개
  - 생성자 1개
  - getter 2개
  - equals() 메서드
  - hashCode() 메서드
  - toString() 메서드

  → 총 50줄 이상의 코드!
```

이것을 **보일러플레이트 코드(boilerplate code)**라고 한다.
"틀에 박힌, 반복적인, 지루한" 코드다.

```
  record 사용 전:                     record 사용 후:

  public class Person {              public record Person(
      private String name;               String name,
      private int age;                   int age
                                     ) {}
      public Person(String name,
          int age) {                 // 끝! 이것으로 모든 것이 자동 생성됨!
          this.name = name;          // - 생성자
          this.age = age;            // - getter (name(), age())
      }                              // - equals()
                                     // - hashCode()
      public String getName() {      // - toString()
          return name;
      }

      public int getAge() {
          return age;
      }

      // equals, hashCode, toString도 구현해야...
      // ... (수십 줄 더)
  }
```

#### 불변성(Immutability)이 왜 중요한가?

record로 만든 객체는 **불변(immutable)**이다.
한 번 만들면 값을 바꿀 수 없다.

> **왜?** 불변이 좋은가?
>
> **비유: 공증된 계약서**
>
> 계약서를 작성한 후 공증을 받으면, 누구도 내용을 수정할 수 없다.
> 그래서 **믿을 수 있다.**
>
> 불변 객체도 마찬가지다.
> 한 번 만들어지면 변하지 않으므로:
> - 멀티스레드에서 안전하다 (여러 곳에서 동시에 읽어도 OK)
> - 예측 가능하다 (내가 준 값이 나중에 바뀌지 않음을 보장)
> - 부작용(side effect)이 없다

---

### 8.2 sealed class: 상속을 제한하는 이유

Java 17에서 도입된 sealed class는 **상속할 수 있는 클래스를 제한**한다.

#### 왜? 아무나 상속하면 안 되는가?

비유: **프랜차이즈 레스토랑**

```
  본사(sealed class)가 허가한 가맹점만 운영할 수 있다.

  sealed class Shape permits Circle, Rectangle, Triangle {}

  "Shape를 상속할 수 있는 것은 Circle, Rectangle, Triangle뿐이다."
  "그 외에는 아무도 상속할 수 없다."
```

이것이 왜 필요한가?

1. **안정성**: 내가 만든 클래스를 예상치 못한 방식으로 확장하는 것을 방지
2. **완전한 분기 처리**: "Shape는 이 3개뿐이야"라는 것을 컴파일러가 알 수 있음
3. **패턴 매칭과의 시너지**: switch에서 모든 경우를 빠짐없이 처리할 수 있음

```
  sealed class를 switch로 처리할 때:

  switch(shape) {
      case Circle c    -> 원 처리
      case Rectangle r -> 사각형 처리
      case Triangle t  -> 삼각형 처리
      // 컴파일러가 "이 3개가 전부"라는 것을 알기 때문에
      // default가 필요 없다! 빠뜨린 case가 있으면 컴파일 에러!
  }
```

---

### 8.3 Text Blocks, Pattern Matching, Switch Expressions

#### Text Blocks (Java 15+)

여러 줄 문자열을 작성할 때의 고통을 해결한다.

```
  이전:
  String json = "{\n" +
                "  \"name\": \"홍길동\",\n" +
                "  \"age\": 25\n" +
                "}";

  이후:
  String json = """
                {
                  "name": "홍길동",
                  "age": 25
                }
                """;
```

> **왜?** 가독성이 압도적으로 좋아진다.
> SQL, JSON, HTML 등 여러 줄 텍스트를 다룰 때 특히 유용하다.

#### Pattern Matching (instanceof)

타입 확인과 캐스팅을 한 번에 한다.

```
  이전:
  if (obj instanceof String) {
      String s = (String) obj;   // 캐스팅 필요
      System.out.println(s.length());
  }

  이후:
  if (obj instanceof String s) {  // 확인 + 캐스팅 동시에!
      System.out.println(s.length());
  }
```

> **왜?** instanceof로 이미 타입을 확인했는데, 다시 캐스팅하는 것은 **중복**이다.
> Pattern Matching은 이 중복을 제거한다.

#### Switch Expressions (Java 14+)

switch를 "문(statement)"이 아니라 "식(expression)"으로 사용한다.
즉, switch가 **값을 반환**할 수 있다.

```
  이전:
  String result;
  switch (day) {
      case MONDAY: case TUESDAY:
          result = "업무일";
          break;                    // break 빼먹으면 버그!
      case SATURDAY: case SUNDAY:
          result = "주말";
          break;
      default:
          result = "기타";
  }

  이후:
  String result = switch (day) {
      case MONDAY, TUESDAY -> "업무일";   // 값을 바로 반환!
      case SATURDAY, SUNDAY -> "주말";
      default -> "기타";
  };
```

> **왜?** break를 빼먹는 실수를 원천 봉쇄하고, 코드가 훨씬 간결해진다.

### 지금까지 배운 것의 의미

이 책을 통해 여러분은 Java의 **"왜?"**를 이해했다.

- 왜 Java는 JVM 위에서 돌아가는가
- 왜 변수에 타입이 필요한가
- 왜 기본형과 참조형이 다른가
- 왜 객체지향이 등장했는가
- 왜 캡슐화, 상속, 다형성이 중요한가
- 왜 컬렉션 프레임워크가 필요한가
- 왜 함수형 프로그래밍 요소가 추가되었는가
- 왜 예외처리가 필요한가

이것은 **문법을 외운 것이 아니라, 철학을 이해한 것**이다.

### 추천 학습 순서

1. **이 책의 내용을 완전히 이해하라.** 개념 없이 코드를 작성하면 모래 위에 집을 짓는 것과 같다.

2. **직접 코드를 작성하라.** 읽기만 해서는 절대 프로그래밍을 배울 수 없다. 자전거 타기를 책으로만 배울 수 없는 것과 같다.

3. **에러 메시지를 두려워하지 마라.** 에러 메시지는 적이 아니라 선생님이다. "여기가 잘못되었어"라고 알려주는 것이다.

4. **왜?를 계속 물어보라.** "이렇게 하니까 동작한다"에서 멈추지 말고, "왜 이렇게 동작하는가?"까지 파고들어야 한다. 그것이 진짜 실력이다.

5. **작은 프로젝트를 만들어보라.** 계산기, 주소록, 간단한 게임... 직접 만들면서 배우는 것이 가장 빠르다.

---

> **마지막으로 기억할 것:**
>
> 프로그래밍은 **암기가 아니라 이해**다.
> 문법은 잊어도 다시 찾으면 된다.
> 하지만 "왜?"를 이해하면, 어떤 언어를 배우든 빠르게 적응할 수 있다.
>
> 지금 Java를 배우면서 익힌 **객체지향적 사고**, **메모리에 대한 이해**, **추상화의 힘**은
> 여러분이 어떤 프로그래밍 언어를 만나든, 어떤 기술 스택을 사용하든
> 평생 여러분의 **기반**이 될 것이다.
