# 2장. 데이터를 담는 그릇 - 변수

1장에서 우리는 컴퓨터가 어떻게 프로그램을 실행하는지, 그리고 우리가 쓴 자바 코드가 어떤 과정을 거쳐 CPU 위에서 살아 움직이는지를 살펴보았다. 이제 그 프로그램이 실제로 **다루는 대상**, 즉 데이터로 눈을 돌릴 차례다.

프로그램이란 결국 무엇을 하는 물건일까? 사용자의 나이를 입력받아 성인 여부를 판단하고, 장바구니에 담긴 상품의 가격을 더하고, 게임 캐릭터의 체력을 깎는다. 이 모든 일의 공통점은 **어떤 값을 잠시 기억해 두었다가, 나중에 꺼내 쓰거나 바꾼다**는 것이다. 사람은 기억을 머릿속에 담지만, 컴퓨터는 그 값을 **메모리(RAM)** 라는 물리적인 공간에 담는다.

그런데 여기서 근본적인 문제가 하나 생긴다. 메모리는 그저 수십억 개의 전기 스위치가 늘어선 거대한 창고일 뿐, "여기가 나이를 담는 곳이고 저기가 이름을 담는 곳"이라는 이름표가 붙어 있지 않다. 메모리의 각 칸은 `0x7fff5a3c` 같은 삭막한 **주소(address)** 로만 구분된다. 사람이 이런 주소를 일일이 외우면서 프로그램을 짤 수는 없는 노릇이다.

바로 이 지점에서 **변수(variable)** 라는 개념이 등장한다. 변수는 "복잡한 메모리 주소에 사람이 이해할 수 있는 이름표를 붙이는 장치"다. 이 장에서 우리는 단순히 "변수를 선언하는 법"을 배우는 데서 그치지 않는다. **왜** 변수에는 타입이 필요한지, **왜** 자바에서 `int`가 그토록 자주 쓰이는지, **왜** `0.1 + 0.2`가 `0.3`이 아닌지, 그리고 자바 개발자를 평생 헷갈리게 만드는 "자바는 값을 전달하는가 참조를 전달하는가?"라는 질문의 진짜 답까지 파고들 것이다. 표면적인 문법 뒤에 숨은 컴퓨터의 동작 원리를 이해하고 나면, 지금까지 "그냥 그런 거려니" 하고 넘어갔던 수많은 현상이 하나의 논리로 꿰어질 것이다.

## 2.1 변수란 무엇인가?

### 정의

**변수(variable)** 는 값을 저장하기 위해 이름을 붙인 메모리 공간이다. 조금 더 정확히 말하면, 변수는 세 가지 요소로 이루어진다.

1. **이름(name)**: 사람이 값을 가리키기 위해 쓰는 식별자 (예: `age`, `name`)
2. **타입(type)**: 그 공간에 담을 수 있는 값의 종류와 크기 (예: `int`, `String`)
3. **값(value)**: 실제로 담긴 데이터 (예: `25`, `"홍길동"`)

### 원리: 왜 이름표가 필요한가

변수를 이해하는 가장 좋은 비유는 **이름표 붙은 상자**다.

```
   변수 = 이름표 붙은 상자

   ┌──────────┐   ┌──────────────┐   ┌──────────┐
   │  age     │   │    name      │   │  height  │  ← 이름표
   ├──────────┤   ├──────────────┤   ├──────────┤
   │    25    │   │   "홍길동"    │   │  175.5   │  ← 담긴 값
   └──────────┘   └──────────────┘   └──────────┘
    int (정수)     String (문자열)     double (실수)
```

프로그래머는 "메모리 주소 `0x7fff5a3c`에 있는 4바이트를 읽어라"라고 말하는 대신, 그냥 `age`라고 부르면 된다. 컴파일러와 JVM이 이 이름을 실제 메모리 위치로 번역해 준다. 즉 변수는 **인간의 언어와 기계의 주소 사이를 이어 주는 통역사**인 셈이다.

자바에서 변수를 만드는 코드는 다음과 같다.

```java
int age = 25;          // "int 타입의 age라는 상자에 25를 담아라"
String name = "홍길동";  // "String 타입의 name이라는 상자에 홍길동을 담아라"
double height = 175.5; // "double 타입의 height라는 상자에 175.5를 담아라"
```

여기서 `int`, `String`, `double`이 바로 타입이다. 그런데 궁금증이 생긴다. **왜 굳이 타입을 미리 정해줘야 할까?** 파이썬이나 자바스크립트처럼 그냥 `age = 25`라고 쓰면 안 될까?

### 원리: 왜 타입이 필요한가

핵심은 **메모리는 무한하지 않고, 컴퓨터는 미리 준비하지 않으면 공간을 확보하지 못한다**는 데 있다.

메모리에 값을 담으려면, 컴퓨터는 먼저 "이 값이 얼마나 큰 공간을 차지할지"를 알아야 한다. 정수 하나는 보통 4바이트, 실수(소수점) 하나는 8바이트를 차지한다. 만약 크기를 미리 알려주지 않으면, 컴퓨터는 얼마만큼의 창고 공간을 비워둬야 할지 결정할 수 없다.

**이사 박스 비유**를 떠올려 보자. 이삿짐센터에 전화해서 "박스 하나만 보내주세요"라고 하면 직원이 물을 것이다. "책을 담을 작은 박스요? 아니면 이불을 담을 큰 박스요?" 크기를 알아야 트럭에 자리를 어떻게 배치할지 계획할 수 있기 때문이다. 타입은 바로 이 **박스의 크기와 종류를 미리 정해주는 것**이다.

```
   타입이 결정하는 것: 상자의 크기와 담을 수 있는 내용물의 종류

   byte     ┌─┐              1바이트, 아주 작은 정수
            └─┘
   int      ┌─┬─┬─┬─┐        4바이트, 보통 크기의 정수
            └─┴─┴─┴─┘
   double   ┌─┬─┬─┬─┬─┬─┬─┬─┐ 8바이트, 소수점 있는 수
            └─┴─┴─┴─┴─┴─┴─┴─┘
```

타입을 미리 정하면 두 가지 이득이 있다.

- **성능**: 컴파일 시점에 "이 변수는 4바이트다"라는 것이 확정되므로, 실행 중에 크기를 매번 따질 필요가 없어 빠르다.
- **안전성**: `int age`라고 선언해두면, 실수로 `age = "스물다섯"`처럼 문자열을 넣으려 할 때 컴파일러가 미리 오류를 잡아준다. 실행 전에 잘못을 발견하는 것이다.

이처럼 자바가 "정적 타입 언어(statically typed language)"인 이유는, **오류를 최대한 빨리, 즉 프로그램이 실행되기 전에 잡아내기 위해서**다. 이것이 자바가 대규모 시스템에서 신뢰받는 첫 번째 이유다.

## 2.2 기본형 vs 참조형: 근본적 차이 (가장 중요)

이 장에서 가장 중요한 절이다. 자바의 데이터는 크게 두 종류로 나뉘는데, 이 둘의 차이를 정확히 이해하지 못하면 이후에 나올 `==` 비교, `String` 동작, 메서드 호출 등 거의 모든 것에서 혼란을 겪게 된다.

### 정의

- **기본형(Primitive type)**: 값 자체를 직접 저장하는 타입. 8가지가 있다.
  → `byte`, `short`, `int`, `long`, `float`, `double`, `char`, `boolean`
- **참조형(Reference type)**: 값이 있는 곳의 **주소**를 저장하는 타입. 기본형 8가지를 제외한 모든 것 (`String`, 배열, 우리가 만든 클래스 등).

### 원리: 상자 안에 무엇이 들었는가

기본형과 참조형의 가장 근본적인 차이는 **상자 안에 무엇이 들어있는가**이다.

기본형은 상자 안에 값이 **직접** 들어있다.

```java
int age = 25;
```

```
   Stack (스택 메모리)
   ┌──────────┐
   │  age     │
   ├──────────┤
   │    25    │  ← 값이 직접 들어있다
   └──────────┘
```

참조형은 상자 안에 값이 아니라 **주소(참조값)** 가 들어있고, 진짜 값은 다른 곳(Heap)에 있다.

```java
String name = new String("홍길동");
```

```
   Stack (스택)              Heap (힙)
   ┌──────────┐             ┌──────────────┐
   │  name    │             │              │
   ├──────────┤   가리킴     │   "홍길동"    │
   │  0x1234  │ ──────────▶ │  (0x1234)    │
   └──────────┘             └──────────────┘
   상자 안에는 주소만!         실제 값은 여기에
```

> **잠깐, Stack과 Heap이 뭔가요?**
> JVM은 메모리를 여러 영역으로 나눠 쓴다. 그중 **스택(Stack)** 은 메서드가 실행되는 동안 지역 변수를 담는 빠르고 작은 공간이고, **힙(Heap)** 은 `new`로 만든 객체들이 사는 크고 유연한 공간이다. 기본형 지역 변수와 참조형의 "주소"는 스택에, 실제 객체는 힙에 저장된다고 이해하면 된다. (자세한 메모리 구조는 이후 객체를 다루는 장에서 더 깊이 파고든다.)

### 비유: 리모컨과 TV

이 차이를 직관적으로 이해하려면 **리모컨과 TV 비유**가 최고다.

- **기본형**은 TV 그 자체다. 상자 안에 진짜 TV가 들어있다.
- **참조형**은 TV를 가리키는 **리모컨**이다. 상자 안에는 리모컨이 들어있고, 그 리모컨은 저 멀리 있는 TV(힙의 객체)를 가리킨다.

```
   기본형: 상자 = TV 자체
   ┌──────────┐
   │   📺     │  값이 곧 물건
   └──────────┘

   참조형: 상자 = 리모컨,  실제 TV는 멀리
   ┌──────────┐         ┌──────────┐
   │   🎛️     │ ──────▶ │    📺    │
   └──────────┘         └──────────┘
   (리모컨)               (진짜 TV)
```

이 비유의 위력은 **복사할 때** 드러난다. 참조형을 다른 변수에 대입하면, 복사되는 것은 리모컨(주소)뿐이다. 그 결과 **리모컨 2개가 똑같은 TV 하나를 가리키게** 된다. TV가 복제되는 것이 아니다.

```java
Dog dog1 = new Dog("멍멍이");
Dog dog2 = dog1;        // 리모컨(주소)만 복사됨
dog2.name = "바둑이";     // dog2로 TV를 조작하면...

System.out.println(dog1.name);  // "바둑이" — dog1이 가리키는 TV도 같은 것이므로!
```

`dog1`과 `dog2`는 서로 다른 리모컨이지만, 둘 다 같은 강아지 객체 하나를 가리킨다. 그래서 하나를 통해 값을 바꾸면 다른 하나에도 반영된다. 이 원리는 2.8절의 "pass-by-value"에서 결정적인 역할을 하니 꼭 기억해두자.

### 원리: `==`의 함정

이제 자바 초보자를 가장 많이 괴롭히는 주제로 들어가 보자. 바로 `==` 연산자다.

`==`는 항상 **상자 안에 든 것**을 비교한다. 그런데 상자 안에 든 것이 기본형과 참조형에서 다르다는 것이 함정의 출발점이다.

- **기본형**: 상자 안에 값이 있으므로, `==`는 **값을 비교**한다.
- **참조형**: 상자 안에 주소가 있으므로, `==`는 **주소를 비교**한다.

```java
// 기본형: 값을 비교 → 직관적
int a = 10;
int b = 10;
System.out.println(a == b);  // true (10 == 10)

// 참조형: 주소를 비교 → 함정!
String s1 = new String("hello");
String s2 = new String("hello");
System.out.println(s1 == s2);        // false (!!)
System.out.println(s1.equals(s2));   // true
```

`s1`과 `s2`는 내용은 똑같이 "hello"지만, `new`를 두 번 했으므로 힙에 별개의 객체가 두 개 만들어졌다. 그래서 두 변수가 가진 **주소**는 서로 다르다.

```
   Stack                       Heap
   ┌──────────┐               ┌──────────────┐
   │  s1      │               │  "hello"     │
   ├──────────┤ ────────────▶ │  (0x100)     │
   │  0x100   │               └──────────────┘
   └──────────┘
   ┌──────────┐               ┌──────────────┐
   │  s2      │               │  "hello"     │  ← 내용은 같지만
   ├──────────┤ ────────────▶ │  (0x200)     │     다른 주소!
   │  0x200   │               └──────────────┘
   └──────────┘

   s1 == s2  →  0x100 == 0x200  →  false
   s1.equals(s2)  →  내용 비교  →  true
```

**결론: 참조형(객체)의 내용이 같은지 알고 싶으면 반드시 `.equals()`를 써야 한다.** `==`는 "같은 물건인가(동일성)"를 묻고, `.equals()`는 "같은 내용인가(동등성)"를 묻는다. 이 둘을 혼동해 `==`로 문자열을 비교하는 것은 자바에서 가장 흔한 버그 중 하나다.

### 원리: String Pool — 왜 어떤 경우엔 `==`가 true일까?

그런데 여기서 이상한 현상이 하나 있다. 위에서는 `new String`으로 만든 두 문자열이 `==`에서 `false`였는데, 다음은 어떨까?

```java
String a = "hello";   // 리터럴
String b = "hello";   // 리터럴
System.out.println(a == b);  // true (!)
```

`new`를 쓰지 않고 큰따옴표로 직접 쓴 문자열 리터럴은 `==` 비교가 `true`가 된다. 왜일까?

이는 자바의 **String Pool(문자열 풀, 상수 풀)** 이라는 최적화 때문이다. 자바는 문자열 리터럴을 특별한 저장소인 String Pool에 모아두고, 같은 내용의 리터럴이 다시 나오면 **새로 만들지 않고 기존 것을 재사용**한다.

```
   String Pool (힙 안의 특별한 영역)
   ┌──────────────┐
   │  "hello"     │
   │  (0x300)     │
   └──────▲───────┘
          │
     ┌────┴────┐
     │         │
   a=0x300   b=0x300   ← 둘 다 같은 주소를 가리킴!

   a == b  →  0x300 == 0x300  →  true
```

**왜 이런 최적화를 할까?** 문자열은 프로그램에서 정말 자주 쓰이는데, 같은 내용의 문자열("확인", "취소", "OK" 등)이 프로그램 곳곳에서 수백, 수천 번 등장한다. 그때마다 힙에 새 객체를 만들면 메모리가 엄청나게 낭비되고, 문자열은 어차피 한 번 만들어지면 바꿀 수 없는(immutable, 불변) 존재이므로 공유해도 안전하다. 그래서 자바는 "이미 있는 건 재사용하자"는 판단을 한 것이다.

반면 `new String("hello")`은 프로그래머가 "무조건 새 객체를 만들어라"고 명시적으로 지시한 것이므로, Pool을 무시하고 힙에 새로 만든다. 그래서 `new`로 만든 두 문자열은 `==`가 `false`인 것이다.

> **면접 팁**
> "`==`와 `equals()`의 차이는?"은 자바 면접의 단골 질문이다. 모범 답안은 이렇다: "`==`는 참조형에서 주소(동일성)를 비교하고, `equals()`는 내용(동등성)을 비교합니다. 그래서 객체의 내용을 비교할 때는 반드시 `equals()`를 써야 합니다." 여기에 String Pool을 곁들이면 만점이다: "다만 문자열 리터럴은 String Pool에서 재사용되기 때문에 `==`가 `true`가 될 수 있는데, 이것은 우연히 같은 주소를 가리키게 된 것일 뿐 내용 비교로 믿어서는 안 됩니다."

## 2.3 왜 int가 가장 많이 쓰이는가?

### 정의: 정수형 4형제

자바에는 정수를 담는 타입이 네 가지 있다. 크기(차지하는 바이트)에 따라 담을 수 있는 값의 범위가 다르다.

| 타입 | 크기 | 범위 | 대략적 느낌 |
|------|------|------|-------------|
| `byte` | 1바이트 (8비트) | -128 ~ 127 | 아주 작은 수 |
| `short` | 2바이트 (16비트) | -32,768 ~ 32,767 | 작은 수 |
| `int` | 4바이트 (32비트) | 약 -21억 ~ +21억 | 보통 크기 (기본값) |
| `long` | 8바이트 (64비트) | 약 -922경 ~ +922경 | 아주 큰 수 |

정확한 범위는 `int`가 -2,147,483,648 ~ 2,147,483,647이다. 흔히 "약 ±21억"으로 기억한다.

### 원리: CPU 아키텍처와 int의 관계

"메모리를 아끼려면 작은 타입인 `byte`를 쓰는 게 좋지 않나?"라고 생각하기 쉽다. 하지만 실무에서 압도적으로 많이 쓰이는 것은 `int`다. **왜일까?**

답은 **CPU가 데이터를 처리하는 단위**에 있다. 현대의 CPU는 32비트 또는 64비트 단위로 데이터를 처리하도록 설계되어 있다. `int`는 정확히 32비트이므로, CPU가 **한 번에, 가장 효율적으로** 다룰 수 있는 크기다.

역설적이게도 `byte`(8비트)나 `short`(16비트)를 쓴다고 해서 처리 속도가 빨라지지는 않는다. CPU는 이런 작은 값도 계산할 때 **내부적으로 32비트로 늘려서(확장해서) 처리**하기 때문이다. 즉 메모리 공간은 아낄 수 있어도, 계산 속도는 `int`와 같거나 오히려 변환 비용 때문에 약간 느려질 수도 있다.

```
   택배 컨베이어 벨트 비유 — 벨트 폭은 32비트로 고정

   byte(8비트) 물건을 실으면:
   ┌────────────────────────────────┐
   │ [8비트 물건]  □ □ □  ← 나머지 낭비  │   벨트는 어차피 32비트 폭
   └────────────────────────────────┘   작은 물건이라도 벨트 한 칸 차지

   int(32비트) 물건을 실으면:
   ┌────────────────────────────────┐
   │ [======= 32비트 물건 =======]     │   벨트를 딱 맞게 채움 → 가장 효율적
   └────────────────────────────────┘
```

CPU라는 컨베이어 벨트의 폭이 32비트라면, 그 위에 8비트짜리 작은 물건을 올려도 벨트 한 칸을 통째로 차지한다. 오히려 32비트 물건을 올릴 때 벨트를 가장 알뜰하게 쓰게 된다. 이것이 자바가 정수의 **기본 타입**을 `int`로 삼고, 대부분의 예제와 실무 코드가 `int`를 쓰는 이유다.

### 언제 다른 타입을 쓰는가

그렇다고 `int`만 쓰면 되는 것은 아니다. 각 타입에는 어울리는 용도가 있다.

- **`byte`**: 파일 입출력이나 네트워크 통신처럼 **데이터를 바이트 단위로 다뤄야 할 때**. 이럴 땐 메모리와 대역폭 절약이 중요하다.
- **`long`**: `int`의 한계인 21억을 넘을 수 있는 값. 대표적으로 **시간(밀리초 단위 타임스탬프)**, **파일 크기(바이트 단위)**, **조회수·매출 합계** 등이 있다. `System.currentTimeMillis()`가 `long`을 반환하는 것이 좋은 예다.
- **`double`**: 소수점이 있는 실수. 단, **돈 계산에는 쓰지 말 것.** 왜 그런지는 2.6절에서 자세히 다룬다. 돈은 `BigDecimal`을 쓰는 것이 정석이다.

`long` 리터럴을 쓸 때는 숫자 뒤에 `L`을 붙여야 한다는 점도 기억하자.

```java
long timestamp = 1735689600000L;  // L을 안 붙이면 int 범위 초과로 컴파일 에러
```

## 2.4 형변환이 필요한 이유

### 정의

**형변환(type casting)** 은 한 타입의 값을 다른 타입으로 바꾸는 것이다. 두 종류가 있다.

- **자동(묵시적) 형변환**: 작은 타입 → 큰 타입. 컴파일러가 알아서 해준다.
- **강제(명시적) 형변환**: 큰 타입 → 작은 타입. 프로그래머가 직접 `(타입)`을 붙여 지시해야 한다.

### 원리: 물컵 비유

형변환의 방향성을 이해하는 데는 **물컵 비유**가 완벽하다.

```
   작은 컵 → 큰 컵 (자동 형변환)         큰 컵 → 작은 컵 (강제 형변환)
   ┌───┐        ┌─────────┐            ┌─────────┐        ┌───┐
   │ 💧│  붓기   │         │            │ 💧💧💧💧 │  붓기   │💧💧│ 넘침!
   │   │ ─────▶ │   💧    │            │         │ ─────▶ │   │ 💧💧💧
   └───┘        └─────────┘            └─────────┘        └───┘
   안전 (손실 없음)                      위험 (데이터 손실)
```

작은 컵의 물을 큰 컵에 부으면 넘칠 걱정이 없다. 마찬가지로 `int` 값을 `long`에 넣으면 손실이 전혀 없으므로 자바가 알아서 변환해 준다.

```java
int small = 5;
long big = small;   // 자동 형변환: int 5 → long 5, 손실 없음
```

반대로 큰 컵의 물을 작은 컵에 부으면 넘쳐서 물이 손실된다. `long`이나 `int` 값을 `byte`처럼 작은 타입에 넣으면 값이 잘려나갈 수 있다. 그래서 자바는 이 변환을 **자동으로 해주지 않고**, 프로그래머가 `(byte)`처럼 명시적으로 캐스팅하도록 요구한다.

```java
int large = 300;
byte b = (byte) large;   // 강제 형변환 필요, 손실 발생
System.out.println(b);   // 44 (!) — 왜 하필 44인지는 2.5절에서
```

**왜 자바는 큰 → 작은 변환을 자동으로 안 해줄까?** 데이터 손실은 심각한 버그를 일으킬 수 있는데, 만약 자바가 몰래 값을 잘라버린다면 프로그래머는 그 사실을 눈치채지 못한 채 잘못된 값으로 계산을 이어갈 것이다. 그래서 자바는 "손실이 발생할 수 있는 변환은 **당신이 그 위험을 알고 있다는 것을 코드로 명시하라**"고 요구한다. `(byte)`라고 직접 쓰는 행위 자체가 "손실이 나도 괜찮으니 잘라라"는 프로그래머의 의사 표현인 것이다.

### 원리: 정수 나눗셈의 함정

형변환과 관련해 초보자가 반드시 한 번은 걸려 넘어지는 함정이 있다.

```java
double result = 5 / 2;
System.out.println(result);  // 2.0 (!) — 왜 2.5가 아닐까?
```

`5 / 2`는 상식적으로 2.5인데, 결과는 2.0이 나온다. 이유는 **연산이 일어나는 순서**에 있다.

자바에서 `int / int`의 결과는 항상 `int`다. `5`와 `2`는 둘 다 정수(`int`)이므로, `5 / 2`는 정수 나눗셈으로 계산되어 소수점 이하가 **버려진 2**가 된다. 이 정수 2가 계산이 끝난 **후에** `double`인 `result`에 담기면서 2.0이 되는 것이다. 즉 소수점은 대입되기 전에 이미 사라진 상태다.

```
   [잘못된 흐름]
   5 / 2  →  (int 나눗셈)  →  2  →  double에 담김  →  2.0
                              ↑ 여기서 이미 소수점 버려짐

   [올바른 흐름]
   5.0 / 2  →  (double 나눗셈)  →  2.5  →  double에 담김  →  2.5
```

해결책은 **나눗셈이 일어나기 전에** 피연산자 중 하나를 실수로 만드는 것이다.

```java
double a = 5.0 / 2;         // 리터럴을 실수로: 2.5
double b = (double) 5 / 2;  // 캐스팅으로 실수로: 2.5
double c = 5 / 2.0;         // 어느 쪽이든 하나만 실수면 됨: 2.5
```

피연산자 중 하나라도 `double`이면, 자바는 다른 하나도 `double`로 자동 승격시킨 뒤 실수 나눗셈을 수행한다. 그래서 2.5라는 정확한 결과가 나온다.

## 2.5 정수의 진짜 얼굴: 2의 보수

앞 절에서 `(byte) 300`이 `44`가 된다고 했다. **왜 하필 44일까?** 이 질문의 답을 찾으려면 컴퓨터가 음수를 어떻게 저장하는지, 즉 **2의 보수(two's complement)** 를 이해해야 한다.

### 문제: 0과 1밖에 없는데 어떻게 마이너스를 표현하나?

컴퓨터의 메모리에는 0과 1만 저장할 수 있다. 그런데 -5 같은 음수의 마이너스 부호는 어디에 저장할까? 마이너스 기호를 담을 자리는 없다. 결국 0과 1만으로 음수를 표현할 방법을 찾아야 한다.

**소박한 첫 번째 아이디어**는 이렇다. "맨 앞 비트를 부호 전용으로 쓰자. 0이면 양수, 1이면 음수." 이것을 **부호-크기 방식(sign-magnitude)** 이라 한다. 얼핏 그럴듯하지만 두 가지 심각한 문제가 있다.

1. **+0과 -0이 따로 존재한다.** `0000`(+0)과 `1000`(-0)이 둘 다 0인데 비트 패턴이 다르다. 0이 두 개라니 이상하다.
2. **덧셈 회로가 복잡해진다.** 부호가 다른 두 수를 더할 때, CPU가 부호를 보고 "이건 사실 뺄셈이네" 하고 특별히 처리해야 한다. 회로가 복잡하고 느려진다.

그래서 컴퓨터 과학자들은 훨씬 영리한 방식인 **2의 보수**를 채택했다.

### 원리: 2의 보수 — "뒤집고 1 더하기"

어떤 수의 음수를 만드는 규칙은 간단하다. **모든 비트를 뒤집고(0↔1), 1을 더한다.**

`5`를 8비트로 나타내면 `0000 0101`이다. 이것의 음수 `-5`를 만들어 보자.

```
   5      =  0000 0101
   비트 뒤집기 →  1111 1010   (모든 비트 반전)
   1 더하기   →  1111 1011   = -5
```

`-5`의 맨 앞 비트가 1이므로 음수임을 알 수 있다. 이 방식의 진짜 아름다움은 **덧셈을 특별 취급 없이 그냥 하면 뺄셈이 저절로 된다**는 점이다. `5 + (-5)`를 계산해 보자.

```
     0000 0101   ( 5)
   + 1111 1011   (-5)
   -----------
   1 0000 0000   ← 9번째 비트(자리올림)가 생김
```

8비트 공간에는 8자리만 담을 수 있으므로, 맨 앞의 9번째 비트는 **그냥 버려진다(overflow away)**. 남는 것은 `0000 0000`, 즉 0이다. 부호를 신경 쓰지 않고 그냥 더했을 뿐인데 `5 + (-5) = 0`이라는 정답이 나왔다.

```
   1 0000 0000
   ↑ 이 자리올림 비트를 버리면
     0000 0000  =  0  ✓ 정답!
```

이것이 2의 보수의 핵심 가치다. **뺄셈을 음수의 덧셈으로 바꿔서, CPU가 덧셈 회로 하나만으로 뺄셈까지 처리할 수 있게 한다.** 부호를 따로 검사하는 복잡한 회로가 필요 없다. 게다가 0은 오직 `0000 0000` 하나뿐이라 +0/-0 문제도 사라진다.

### 원리: 그래서 (byte)300은 왜 44인가?

이제 2.4절의 미스터리를 풀 수 있다. `byte`는 8비트이므로 총 $2^8 = 256$가지 값을 표현할 수 있고, 그 범위는 -128 ~ 127이다.

300을 2진수로 쓰면 9비트가 필요하다.

```
   300  =  1 0010 1100  (9비트)
```

그런데 `byte`는 8비트짜리 상자다. 맨 앞의 9번째 비트가 들어갈 자리가 없어 잘려나간다. 남는 하위 8비트는 `0010 1100`이고, 이것을 10진수로 읽으면 **44**다.

```
   300  =  1 0010 1100
              └──┬───┘
           하위 8비트만 byte에 저장
                 ↓
              0010 1100  =  44
```

수학적으로는 $300 - 256 = 44$로도 설명된다. `byte`가 표현할 수 있는 256가지를 초과한 만큼이 **한 바퀴 돌아(wraparound)** 처음부터 다시 세어지는 것이다. 이것을 **모듈러 연산(modular arithmetic)** 이라 부른다.

### 원리: 오버플로 — 시계처럼 순환하는 정수

이 순환 성질은 실무에서 매우 중요한 함정을 만든다. `byte`의 최댓값 127에 1을 더해 보자.

```
     0111 1111   (127, byte의 최댓값)
   + 0000 0001   (1)
   -----------
     1000 0000   = -128 (!)
```

`127 + 1`이 128이 아니라 **-128**이 된다. 맨 앞 비트가 1이 되면서 갑자기 음수가 되어버린 것이다. 마치 12시에서 1시간이 지나면 13시가 아니라 1시로 돌아가는 시계처럼, 정수도 최댓값을 넘으면 최솟값으로 순환한다. 이것을 **오버플로(overflow)** 라 한다.

```
   byte의 세계는 원형 시계처럼 순환한다

              -128
           ↗        ↘
       127            -127
        ↑              ↓
        ...    0    ...
        ↑              ↓
        1              -1
           ↖        ↙
              (127 다음은 -128!)
```

**왜 이것이 실무에서 중요한가?** `byte`는 잘 안 쓰니 남 얘기 같지만, **`int`도 똑같이 오버플로한다.** `int`의 최댓값은 약 21억인데, 조회수·시간 누적·금액 합계처럼 값이 계속 커지는 경우 21억을 넘는 순간 **아무 에러도 없이 조용히 음수로 뒤집힌다.** 프로그램은 멈추지 않고, 로그에도 아무것도 남지 않으며, 그저 결과가 틀릴 뿐이다. 이런 버그는 발견하기가 극도로 어렵다.

실무 규칙:
- **두 양수를 더했는데 결과가 음수라면 오버플로를 의심하라.**
- **시간(밀리초), 조회수, 파일 크기, 금액 합계처럼 커질 수 있는 값은 처음부터 `long`으로 선언하라.**

## 2.6 실수의 함정: 왜 0.1 + 0.2 != 0.3?

### 문제: 컴퓨터가 계산을 틀린다?

다음 코드는 자바를 처음 배우는 사람에게 충격을 준다.

```java
System.out.println(0.1 + 0.2);        // 0.30000000000000004
System.out.println(0.1 + 0.2 == 0.3); // false
```

`0.1 + 0.2`가 `0.3`이 아니라 `0.30000000000000004`라니. 컴퓨터가 이렇게 간단한 계산을 틀리다니 버그가 아닐까? **아니다. 이것은 버그가 아니라, 실수를 저장하는 방식이 만든 필연적인 결과다.** 이 현상을 이해하려면 컴퓨터가 소수를 어떻게 저장하는지, 즉 **IEEE-754** 표준을 알아야 한다.

### 원리: IEEE-754 — 2진 과학적 표기법

`double`은 64비트인데, 이 64비트를 세 부분으로 나눠서 실수를 저장한다.

```
   IEEE-754 double (64비트) 구조

   ┌─┬───────────────┬────────────────────────────────────────┐
   │S│   Exponent    │              Mantissa                    │
   │ │   (지수, 11)   │              (가수, 52)                   │
   └─┴───────────────┴────────────────────────────────────────┘
    1        11                        52 비트

   값 = (-1)^S × 1.Mantissa × 2^Exponent
        └─부호─┘ └──유효숫자──┘ └──자릿수──┘
```

이것은 우리가 학교에서 배운 **과학적 표기법**의 2진수 버전이다. 예를 들어 $6.02 \times 10^{23}$처럼 "유효숫자 × 10의 거듭제곱"으로 쓰듯, IEEE-754는 "유효숫자 × **2**의 거듭제곱"으로 실수를 표현한다.

여기서 핵심은 **지수가 2의 거듭제곱**이라는 점이다. 즉 컴퓨터는 모든 소수를 $\frac{1}{2}, \frac{1}{4}, \frac{1}{8}, \frac{1}{16}, \dots$ 같은 **2진 분수들의 합**으로 표현한다.

### 원리: 0.1은 2진수로 딱 떨어지지 않는다

문제는 여기서 발생한다. 우리가 10진수로 아주 깔끔하게 쓰는 `0.1`이, **2진수로는 무한히 반복되는 소수**가 된다.

```
   0.1 (10진수) = 0.0001100110011001100110011... (2진수)
                        └────┬────┘
                    "0011"이 무한히 반복
```

이것은 마치 10진수에서 $\frac{1}{3} = 0.3333\dots$가 무한히 반복되는 것과 정확히 같은 상황이다. 3진법이 있다면 1/3이 딱 떨어지겠지만 10진법에서는 안 떨어지듯이, 0.1은 10진법에서는 딱 떨어져도 2진법에서는 안 떨어진다.

그런데 mantissa(가수)에 배정된 공간은 52비트로 유한하다. 무한히 반복되는 소수를 유한한 52비트에 우겨넣으려면 **어딘가에서 반올림**해야 한다. 그 결과 컴퓨터에 실제로 저장되는 값은 정확한 0.1이 아니라 `0.1000000000000000055511...`처럼 **아주 미세하게 큰** 근사값이 된다.

```
   우리가 쓴 값:     0.1
   실제 저장된 값:   0.1000000000000000055511151231257827...  (미세하게 큼)

   0.1도, 0.2도 각각 이런 미세한 오차를 품고 저장된다.
   → 둘을 더하면 오차도 함께 더해져,
     결과가 0.3과 미세하게 어긋나 0.30000000000000004가 된다.
```

`0.1`과 `0.2`가 각각 아주 작은 오차를 품은 채 저장되고, 이 둘을 더하면 그 오차들도 함께 더해진다. 그래서 결과가 정확한 `0.3`과 미세하게 어긋나는 것이다. 이것은 자바만의 문제가 아니라 IEEE-754를 따르는 **모든 언어(C, 파이썬, 자바스크립트 등)에서 똑같이 나타나는** 보편적 현상이다.

### 원리: 그래서 돈은 BigDecimal로

이 미세한 오차가 별것 아닌 것 같아도, **돈 계산에서는 치명적**이다.

```java
// 0.1원 + 0.2원 = 0.30000000000000004원
// 이런 계산을 수백만 건 처리하면 오차가 누적되어
// 회계가 1원, 10원씩 안 맞기 시작한다.
```

은행이나 결제 시스템에서 단 1원이라도 틀리면 큰 문제가 된다. 그래서 돈은 절대 `double`로 계산하지 않고, `BigDecimal`을 쓴다. `BigDecimal`은 값을 2진 근사값이 아니라 **10진수 그대로 정확하게** 저장하고 계산하므로 오차가 없다.

```java
import java.math.BigDecimal;

BigDecimal a = new BigDecimal("0.1");   // 반드시 문자열로 생성!
BigDecimal b = new BigDecimal("0.2");
System.out.println(a.add(b));           // 0.3 (정확!)
```

> **주의**: `BigDecimal`을 만들 때 `new BigDecimal(0.1)`처럼 `double`을 넘기면, 이미 오차가 낀 `double` 값이 들어가므로 소용없다. 반드시 `new BigDecimal("0.1")`처럼 **문자열**로 생성해야 한다.

### 왜 그냥 오차를 두는가? 그리고 실수 비교법

"오차가 이렇게 문제라면 왜 IEEE-754를 계속 쓰는가?"라는 의문이 들 수 있다. 이유는 세 가지다.

1. **표준**: IEEE-754는 전 세계 CPU 하드웨어가 공통으로 지원하는 국제 표준이다. 하드웨어가 직접 지원하니 어디서든 똑같이 동작한다.
2. **속도**: 하드웨어 가속을 받으므로 엄청나게 빠르다. `BigDecimal`은 정확하지만 훨씬 느리다.
3. **대부분 무의미**: 공학, 그래픽, 물리 시뮬레이션에서는 소수점 15자리 이하의 미세한 오차가 아무 영향도 주지 않는다. 오히려 넓은 범위의 수를 빠르게 근사하는 `double`의 능력이 훨씬 유용하다.

정리하면, `float`/`double`은 **넓은 범위를 빠르게 근사**하는 데 최적화된 도구이고(공학·그래픽·물리에 완벽), 1원도 틀리면 안 되는 **돈에는 부적합**하다.

마지막으로 중요한 실무 규칙 하나. **실수는 절대 `==`로 비교하지 마라.** 미세한 오차 때문에 논리적으로 같아야 할 값이 `==`에서 `false`가 나올 수 있다. 대신 두 값의 차이가 아주 작은지(허용 오차 이내인지)로 비교한다.

```java
double a = 0.1 + 0.2;
double b = 0.3;

// System.out.println(a == b);  // false — 쓰지 말 것!

double EPSILON = 1e-9;  // 아주 작은 허용 오차
System.out.println(Math.abs(a - b) < EPSILON);  // true — 올바른 비교
```

## 2.7 char와 유니코드: 문자는 왜 2바이트?

### 문제: 자바의 char는 왜 2바이트일까?

자바의 `char` 타입은 2바이트(16비트)를 차지한다. 정수 타입인 `short`와 크기가 같다. 그런데 다른 많은 언어에서는 문자를 1바이트로 다룬다. **왜 자바는 문자를 2바이트로 만들었을까?** 이 질문의 답에는 컴퓨터가 문자를 다루는 역사가 담겨 있다.

### 원리: 문자도 사실은 숫자다

컴퓨터는 문자를 저장할 수 없다. 오직 숫자만 저장할 수 있을 뿐이다. 그래서 모든 문자에는 **고유한 번호**가 부여되어 있는데, 이것을 **코드 포인트(code point)** 라 한다.

```
   'A'  →  65
   'a'  →  97
   '0'  →  48
   '가'  →  44032
```

`char c = 'A'`라고 쓰면, 컴퓨터는 실제로는 숫자 65를 저장한다. `char`가 숫자라는 증거는, `char`를 `int`에 넣으면 그 코드 포인트 숫자가 그대로 나온다는 것이다.

```java
char c = '가';
int n = c;              // char → int 자동 변환
System.out.println(n);  // 44032 (가의 코드 포인트)
```

옛날에는 영어권 컴퓨팅이 중심이어서 **ASCII**라는 체계로 128개(7비트)면 충분했다. 알파벳 대소문자, 숫자, 기호 정도만 담으면 됐기 때문이다. 하지만 컴퓨터가 전 세계로 퍼지면서 한글, 한자, 아랍 문자, 그리고 이모지까지 담아야 했고, 128개로는 턱없이 부족해졌다. 그래서 전 세계 모든 문자에 번호를 매기는 거대한 체계인 **유니코드(Unicode)** 가 만들어졌다.

### 원리: 자바의 선택 — UTF-16과 2바이트

자바가 설계된 1990년대 초, 당시 유니코드 진영은 "세상의 모든 문자를 담아도 **65,536자면 충분하다**"고 판단했다. $2^{16} = 65,536$이고, 이는 정확히 2바이트로 표현할 수 있는 개수다. 그래서 자바는 **UTF-16**을 채택했고, 문자 하나를 2바이트인 `char`에 담기로 한 것이다.

```
   char (2바이트 = 16비트) → 65,536가지 문자 표현 가능

   ┌─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┐
   │ 16비트: 0 ~ 65,535 코드 포인트    │
   └─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┘

   'A' = 65,  '가' = 44032  →  모두 char 하나에 들어감
```

### 반전: 2바이트로도 부족했다 — 서로게이트 페어

그런데 시간이 지나며 유니코드의 예상은 빗나갔다. 이모지, 희귀 한자, 고대 문자 등이 추가되면서 문자의 수가 65,536개를 훌쩍 넘어버린 것이다. 이제 어떤 문자는 `char` 하나(2바이트)에 담을 수 없게 되었다.

자바는 이 문제를 **char 두 개를 짝지어** 해결한다. 큰 코드 포인트를 가진 문자는 상위 서로게이트와 하위 서로게이트라는 두 개의 `char`로 나눠 표현하는데, 이 짝을 **서로게이트 페어(surrogate pair)** 라 한다.

```
   😀 (웃는 이모지)는 char 하나에 안 들어간다

   ┌──────────┐  ┌──────────┐
   │ 상위 서로게이트 │  │ 하위 서로게이트 │   → 이 두 개가 합쳐져 😀 한 글자
   │  (char 1)  │  │  (char 2)  │
   └──────────┘  └──────────┘
```

**왜 이것이 함정인가?** 자바의 `String.length()`는 글자 수가 아니라 **`char`의 개수**를 센다. 그래서 서로게이트 페어로 표현되는 문자가 섞이면, 우리가 보는 글자 수와 `length()`가 반환하는 값이 달라진다.

```java
System.out.println("abc".length());   // 3 (예상대로)
System.out.println("😀".length());     // 2 (!) — 이모지 하나인데 char는 2개
System.out.println("a😀b".length());   // 4 (a=1, 😀=2, b=1)
```

이모지가 포함된 문자열의 실제 글자 수를 세려면 `length()`가 아니라 `codePointCount()` 같은 메서드를 써야 한다. 사용자 이름이나 게시글에 이모지가 흔한 요즘, 이 함정은 실제 서비스에서 종종 버그를 일으킨다.

## 2.8 Java는 항상 pass-by-value

이 절은 자바에서 **가장 오해받는 주제**다. 인터넷에는 "자바는 기본형은 값으로 전달하고, 객체는 참조로 전달한다(pass-by-reference)"는 설명이 널려 있다. **이것은 틀렸다.** 정확한 사실은 이것이다.

> **자바는 예외 없이, 언제나 pass-by-value(값에 의한 전달)이다. 참조형(객체)도 마찬가지다.**

이 문장이 왜 참인지를 단계별로 증명해 보자.

### 원리 1: 기본형 전달 — 값이 복사된다

```java
void change(int x) {
    x = 100;   // 파라미터 x를 100으로 바꿈
}

int a = 5;
change(a);
System.out.println(a);  // 5 (100이 아님!)
```

`a`의 값 5가 `x`로 **복사**되어 전달된다. 메서드 안에서 `x`를 100으로 바꿔도, 그것은 복사본 `x`를 바꾼 것일 뿐 원본 `a`와는 아무 상관이 없다. 여기까지는 대부분 직관적으로 이해한다.

```
   a ──(값 5를 복사)──▶ x
   ┌───┐              ┌───┐
   │ 5 │              │ 5 │ → 100 으로 바뀜
   └───┘              └───┘
   원본 a는 그대로 5     복사본 x만 변함
```

### 원리 2: 참조형 전달 — 여기서 오해가 시작된다

```java
class Dog {
    String name;
    Dog(String name) { this.name = name; }
}

void rename(Dog d) {
    d.name = "바둑이";   // 객체 내부의 필드를 바꿈
}

Dog myDog = new Dog("멍멍이");
rename(myDog);
System.out.println(myDog.name);  // "바둑이" — 바뀌었다!
```

이번엔 원본 `myDog`의 이름이 "바둑이"로 바뀌었다. "봐라, 객체는 참조로 전달되니까 원본이 바뀌지 않는가?"라고 생각하기 쉽다. **하지만 이 결론은 틀렸다.** 무슨 일이 일어난 건지 정확히 그려 보자.

`myDog`가 가진 것은 힙에 있는 강아지 객체의 **주소**다. 메서드를 호출하면 이 **주소값이 복사**되어 `d`로 전달된다. 그 결과 `myDog`와 `d`는 서로 다른 변수지만, **똑같은 힙 객체를 가리키는 두 개의 리모컨**이 된다.

```
   Stack                          Heap
   ┌──────────┐                  ┌──────────────┐
   │  myDog   │                  │  Dog 객체     │
   ├──────────┤ ──────────────▶  │  name:멍멍이  │
   │  0x555   │                  │  (0x555)     │
   └──────────┘        ┌───────▶ └──────────────┘
   ┌──────────┐        │
   │  d       │        │   d도 주소값(0x555) 복사본을 가짐
   ├──────────┤ ───────┘   → 둘 다 같은 객체를 가리킴
   │  0x555   │
   └──────────┘
```

`d`를 통해 `d.name`을 바꾸면, `d`가 가리키는 객체(=`myDog`도 가리키는 바로 그 객체)의 필드가 바뀐다. 그래서 `myDog.name`도 "바둑이"가 된 것이다. 이것은 2.2절의 "리모컨 2개, TV 1개" 상황과 정확히 같다. 여기까지만 보면 정말 참조 전달처럼 보인다. 하지만 **결정적인 반례**가 남아 있다.

### 원리 3: 결정적 반례 — 재할당

이번엔 메서드 안에서 필드를 바꾸는 게 아니라, `d`에 **완전히 새로운 객체를 대입**해 보자.

```java
void replace(Dog d) {
    d = new Dog("새강아지");   // d에 새 객체 주소를 대입
}

Dog myDog = new Dog("멍멍이");
replace(myDog);
System.out.println(myDog.name);  // "멍멍이" — 안 바뀌었다! (!)
```

만약 자바가 진짜 pass-by-reference였다면, `d = new Dog(...)`로 인해 `myDog`도 새 강아지를 가리켜야 한다. 하지만 결과는 여전히 "멍멍이"다. **원본은 전혀 바뀌지 않았다.** 왜 그럴까?

`d`는 `myDog`가 가진 주소값의 **복사본**이다. `d`에 새 객체의 주소를 대입하면, 복사본인 `d`만 새 객체를 가리키게 될 뿐, 원본 `myDog`는 여전히 원래 객체를 가리킨다.

```
   replace 호출 직후:                d에 재할당한 후:

   myDog ─▶ [멍멍이](0x555)          myDog ─▶ [멍멍이](0x555)  ← 그대로!
   d     ─▶ [멍멍이](0x555)          d     ─▶ [새강아지](0x999) ← d만 이동
   (같은 객체를 가리킴)                (d만 새 객체로, 원본은 무관)
```

리모컨 비유로 말하면, `d`라는 리모컨의 채널을 다른 TV로 바꿔도, `myDog`라는 리모컨은 여전히 원래 TV를 가리키고 있는 것이다. 만약 진짜 참조 전달이었다면 이 재할당이 원본에 반영되어야 했다. 반영되지 않았다는 사실이 곧 **자바가 참조 전달이 아니라는 증거**다.

### 정리: 하나의 원리로 통합

세 가지 예제를 하나의 문장으로 통합할 수 있다.

> **자바는 언제나 값(복사본)을 전달한다. 기본형은 "값 자체"의 복사본이 전달되고, 참조형은 "객체의 주소값"의 복사본이 전달된다.**

여기서 나오는 두 가지 결론:

1. 참조형에서 전달되는 것은 "객체 그 자체"가 아니라 "객체를 가리키는 주소값"이다. 그래서 그 주소를 따라가 **객체 내부를 고치는 것은 원본에 반영**된다(원리 2).
2. 하지만 파라미터 변수 자체에 **다른 주소를 재할당하는 것은 원본에 아무 영향이 없다**(원리 3). 복사본을 바꾼 것이기 때문이다.

영어로는 이 한 문장이 정답을 완벽히 요약한다.

> **"Java is pass-by-value, always. The value passed for objects is the reference."**
> (자바는 언제나 값에 의한 전달이다. 객체에 대해 전달되는 값은 그 참조(주소)다.)

> **면접 팁**
> "자바는 pass-by-value인가 pass-by-reference인가?"는 중급 면접의 대표 질문이다. "자바는 항상 pass-by-value입니다"라고 단호하게 답한 뒤, "다만 참조형의 경우 전달되는 값이 객체의 주소이기 때문에, 객체 내부는 수정할 수 있지만 파라미터를 다른 객체로 재할당해도 원본에는 영향이 없습니다"라고 재할당 반례를 들어 설명하면 깊이를 인정받는다.

> 실행 데모: ./run.sh PassByValue

## ⚠️ 흔한 오해와 함정

1. **"문자열은 `==`로 비교하면 된다."** → 틀림. `==`는 주소를 비교하므로 `new String`으로 만든 문자열은 내용이 같아도 `false`다. 내용 비교는 반드시 `.equals()`를 써라. 리터럴이 우연히 `==`에서 `true`가 나오는 것(String Pool)에 속지 마라.

2. **"`byte`를 쓰면 메모리도 아끼고 속도도 빠르다."** → 틀림. 메모리는 아낄 수 있지만, CPU는 32비트 단위로 처리하므로 계산 속도는 `int`와 같거나 오히려 느릴 수 있다.

3. **"`double result = 5 / 2;`는 2.5다."** → 틀림. `int / int`는 정수 나눗셈이라 2가 되고, 그 후 2.0으로 담긴다. `5.0 / 2` 또는 `(double) 5 / 2`로 써야 2.5가 된다.

4. **"두 양수를 더했는데 결과가 음수일 리 없다."** → 오버플로를 잊었다. `int`도 21억을 넘으면 조용히 음수로 뒤집힌다. 커질 수 있는 값은 `long`을 써라.

5. **"컴퓨터가 `0.1 + 0.2 = 0.3`을 못 맞추는 건 버그다."** → 버그가 아니라 IEEE-754의 필연적 결과다. 돈은 `BigDecimal`을, 실수 비교는 `Math.abs(a - b) < EPSILON`을 써라.

6. **"`"😀".length()`는 1이다."** → 이모지는 서로게이트 페어로 char 2개이므로 `length()`가 2다. 실제 글자 수는 `codePointCount()`로 세라.

7. **"자바는 객체를 참조로 전달한다(pass-by-reference)."** → 틀림. 자바는 항상 pass-by-value다. 객체는 "주소값의 복사본"이 전달될 뿐이다. 재할당 반례로 확인하라.

## 연습문제

**문제 1.** 다음 코드의 출력을 예상하고, 그 이유를 설명하라.
```java
String a = "java";
String b = "java";
String c = new String("java");
System.out.println(a == b);
System.out.println(a == c);
System.out.println(a.equals(c));
```
> **힌트**: 리터럴은 String Pool을, `new`는 힙에 새 객체를 만든다는 점을 떠올려라.
> **해설**: `a == b`는 `true`(둘 다 Pool의 같은 "java"를 가리킴), `a == c`는 `false`(`c`는 `new`로 만든 별개 객체라 주소가 다름), `a.equals(c)`는 `true`(내용 비교이므로). `==`는 주소, `equals`는 내용을 비교한다는 것을 확인하는 문제다.

**문제 2.** `System.out.println((byte)(127 + 1));`의 결과는 무엇이며, 왜 그런가?
> **힌트**: `byte`의 범위는 -128 ~ 127이고, 2의 보수로 저장된다.
> **해설**: 결과는 **-128**. `127`은 `0111 1111`인데 1을 더하면 `1000 0000`이 되고, 맨 앞 비트가 1이므로 음수 -128이 된다. 시계처럼 최댓값을 넘으면 최솟값으로 순환하는 오버플로 현상이다.

**문제 3.** 다음 중 결과가 `2.5`가 나오는 것을 모두 고르고, 나머지는 왜 아닌지 설명하라.
```java
double a = 5 / 2;
double b = 5.0 / 2;
double c = (double)(5 / 2);
double d = (double) 5 / 2;
```
> **힌트**: 나눗셈이 일어나는 시점에 피연산자가 정수인지 실수인지가 관건이다.
> **해설**: `2.5`가 나오는 것은 **b와 d**. `a`는 정수 나눗셈 후 2.0. `c`는 `5 / 2`를 먼저 정수로 계산(2)한 뒤 캐스팅하므로 2.0. `b`와 `d`는 나눗셈 시점에 이미 실수라 2.5. 캐스팅의 위치가 결과를 가른다.

**문제 4.** 아래 메서드를 호출한 뒤 `arr[0]`의 값은? 그리고 왜 2.8절의 "재할당" 반례와 다른가?
```java
void modify(int[] arr) {
    arr[0] = 999;
}
int[] arr = {1, 2, 3};
modify(arr);
System.out.println(arr[0]);
```
> **힌트**: 배열은 참조형이다. 필드/원소를 고치는 것과 파라미터를 재할당하는 것을 구분하라.
> **해설**: `999`. 배열은 참조형이라 주소값이 복사되어 전달되고, `arr[0] = 999`는 그 주소가 가리키는 **같은 배열 객체 내부**를 고치는 것이므로 원본에 반영된다(원리 2). 만약 메서드 안에서 `arr = new int[]{7,8,9};`처럼 재할당했다면 원본은 안 바뀐다(원리 3).

**문제 5.** `0.1 + 0.2 == 0.3`이 `false`인 이유를 IEEE-754 관점에서 세 문장 이내로 설명하고, 두 실수를 안전하게 비교하는 코드를 작성하라.
> **힌트**: 0.1과 0.2가 2진수로 딱 떨어지는지 생각하라.
> **해설**: 0.1과 0.2는 2진수로 무한히 반복되는 소수라 유한한 mantissa(52비트)에 반올림되어 미세한 오차를 품고 저장되며, 이 오차가 더해져 정확한 0.3과 어긋난다. 안전한 비교: `Math.abs((0.1 + 0.2) - 0.3) < 1e-9`.

## 요약

- 변수는 메모리 주소에 이름표를 붙인 것이며, 타입은 "상자의 크기와 종류"를 미리 정해 메모리 확보와 타입 안전성을 가능하게 한다.
- 기본형은 상자에 값을 직접 담고, 참조형은 주소(리모컨)를 담는다. 그래서 `==`는 기본형에서 값을, 참조형에서 주소를 비교하며, 객체 내용 비교는 `.equals()`로 해야 한다.
- 정수는 2의 보수로 저장되어 뺄셈을 덧셈으로 처리하며, 최댓값을 넘으면 조용히 음수로 순환(오버플로)하므로 커질 값은 `long`을 쓴다.
- 실수는 IEEE-754 근사값이라 `0.1 + 0.2 != 0.3`이 되며, 돈은 `BigDecimal`을, 실수 비교는 허용 오차 방식을 써야 한다.
- 자바는 언제나 pass-by-value다. 참조형은 "주소값의 복사본"이 전달되므로 객체 내부는 고칠 수 있지만 재할당은 원본에 영향을 주지 않는다.
