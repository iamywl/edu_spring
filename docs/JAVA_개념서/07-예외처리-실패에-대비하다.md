# 7장. 예외처리 - 실패에 대비하다

우리는 코드를 짤 때 "모든 것이 잘 될 것"이라는 낙관 위에서 생각한다. 파일은 항상 그 자리에 있고, 네트워크는 항상 연결되어 있고, 사용자는 항상 올바른 값을 입력한다고 가정한다. 하지만 현실의 프로그램은 이 낙관을 배신당하며 살아간다. 디스크가 꽉 차고, Wi-Fi가 끊기고, 사용자는 나이 칸에 "스물여섯"이라고 한글로 적어 넣는다.

문제는 실패가 일어난다는 사실 그 자체가 아니다. 실패는 피할 수 없다. 진짜 문제는 **실패가 일어났을 때 프로그램이 어떻게 반응하는가**이다. 아무 대비가 없는 프로그램은 실패를 만나는 순간 그냥 죽는다. 사용자가 30분 동안 작성한 문서가 저장 직전에 통째로 날아간다. 반면 실패에 대비한 프로그램은 "파일을 찾을 수 없습니다. 경로를 확인해 주세요"라고 말하고, 사용자에게 다시 시도할 기회를 준다.

예외처리(Exception Handling)는 바로 이 "실패에 대비하는 방법"이다. 자동차에 안전벨트와 에어백을 다는 것과 같다. 사고가 안 나게 만드는 것이 아니라, 사고가 나도 살아남게 만드는 것이다. 이 장에서는 자바가 왜 예외를 두 종류(Checked/Unchecked)로 나눴는지, `try-catch-finally`가 실제로 어떻게 동작하는지, 자원을 안전하게 닫는 `try-with-resources`는 무엇을 자동화해 주는지, 그리고 실무에서 예외를 다루는 원칙까지 "왜 그렇게 설계되었는가"의 관점으로 파고든다.

---

## 7.1 왜 예외처리가 필요한가?

### 정의

**예외(Exception)** 란 프로그램 실행 중 정상적인 흐름을 방해하는 사건을 말한다. 그리고 **예외처리** 란 이런 사건이 발생했을 때 프로그램이 비정상 종료되지 않고 적절히 대응하도록 만드는 메커니즘이다.

### 원리: 왜 필요한가?

프로그램은 언제나 실패할 수 있다. 몇 가지만 나열해 보자.

- 읽으려는 파일이 존재하지 않는다.
- 다운로드 도중 인터넷 연결이 끊긴다.
- 어떤 수를 0으로 나눈다.
- 사용자가 예상치 못한 값을 입력한다.

예외처리가 없는 세계에서는 이 중 하나라도 일어나면 프로그램은 그냥 죽는다. 문제가 발생한 지점 이후의 코드는 실행되지 않고, 사용자가 하던 작업은 전부 날아간다.

```
[예외처리 없는 세계]
파일 없음  ──▶  에러 발생  ──▶  프로그램 종료  ──▶  "작업 다 날아감"

[예외처리 있는 세계]
파일 없음  ──▶  예외 감지  ──▶  "파일을 찾을 수 없습니다. 경로를 확인하세요"
                                    │
                                    ▼
                              사용자가 대응 (경로 수정 후 재시도)
```

핵심은 이것이다. 예외처리는 실패를 **없애는** 기술이 아니라, 실패를 **관리 가능한 상태로 만드는** 기술이다. 안전벨트가 사고를 막지 못하지만 사고에서 생존하게 하듯이, 예외처리는 실패를 막지 못하지만 실패에서 프로그램을 살려낸다.

### 예외 계층: Throwable

자바에서 모든 예외와 오류는 `Throwable`이라는 최상위 클래스에서 뻗어 나온다. 이 계층 구조를 이해하는 것이 예외처리의 출발점이다.

```
                    ┌─────────────┐
                    │  Throwable  │   ← 모든 던질 수 있는 것의 최상위
                    └──────┬──────┘
             ┌─────────────┴─────────────┐
             ▼                           ▼
      ┌────────────┐              ┌────────────┐
      │   Error    │              │ Exception  │
      │ (심각/복구불가)│              │ (복구 가능)  │
      └────────────┘              └──────┬─────┘
      · OutOfMemoryError      ┌──────────┴───────────┐
      · StackOverflowError    ▼                      ▼
      → 건드리지 말 것    ┌──────────────┐    ┌──────────────────┐
                        │   Checked    │    │    Unchecked     │
                        │   Exception  │    │(RuntimeException) │
                        └──────────────┘    └──────────────────┘
                        반드시 처리 강제        처리 강제 안 함
                        · IOException         · NullPointerException
                        · SQLException        · ArrayIndexOutOfBounds
                        · FileNotFound        · ClassCastException
                        (예측 가능한 외부 실패)   (프로그래머의 실수)
```

이 계층에서 반드시 기억할 세 가지 갈래가 있다.

**1) Error — 심각한 시스템 오류.** `OutOfMemoryError`(메모리 부족), `StackOverflowError`(재귀 무한 반복 등으로 스택 초과) 같은 것들이다. 이것은 프로그래머가 처리할 수 있는 성질의 것이 아니다. 메모리가 이미 다 떨어진 상황에서 코드로 "메모리를 아껴 쓰세요"라고 대응할 방법은 없다. **Error는 잡으려 하지 말 것.** 이건 코드의 잘못이 아니라 실행 환경 자체가 무너진 상황이다.

**2) Checked Exception — 복구 가능한, 예측 가능한 외부 실패.** `IOException`, `SQLException`, `FileNotFoundException` 등. "파일이 없을 수도 있다", "DB 연결이 끊길 수도 있다"처럼 개발자가 미리 예상할 수 있는 실패다. 자바는 이런 예외에 대해 **반드시 처리하라고 강제** 한다.

**3) Unchecked Exception (RuntimeException) — 프로그래머의 실수.** `NullPointerException`(널 참조), `ArrayIndexOutOfBoundsException`(배열 범위 초과), `ClassCastException`(잘못된 형변환) 등. 이것들은 근본적으로 코드의 버그다. 자바는 이 예외들에 대해서는 처리를 강제하지 않는다.

이 세 갈래의 차이가 왜 존재하는지가 다음 절의 주제다.

## 7.2 Checked vs Unchecked: 근본적 차이

### 정의

- **Checked Exception**: 컴파일러가 처리 여부를 검사(check)하는 예외. `try-catch`로 잡거나 `throws`로 떠넘기지 않으면 **컴파일 자체가 안 된다.**
- **Unchecked Exception**: 컴파일러가 검사하지 않는 예외. `RuntimeException`과 그 하위 클래스들이다. 처리하지 않아도 컴파일된다.

이 구분은 자바의 독특한 설계이며, 지금도 논쟁이 많은 주제다. 왜 이렇게 나눴는지 이해하려면 두 종류의 "실패"가 본질적으로 다르다는 점을 봐야 한다.

### 원리: 왜 둘로 나눴는가?

**Checked는 "예측 가능한 외부 실패"다.** 파일을 여는 순간, 그 파일이 없을 수도 있다는 것은 코드를 잘 짜든 못 짜든 언제나 가능한 일이다. 네트워크가 끊기는 것도 마찬가지다. 이런 실패는 개발자의 잘못이 아니라 세상이 불확실하기 때문에 일어난다. 그래서 자바는 "이건 언제든 일어날 수 있으니 **반드시 대비책을 마련하라**"고 강제한다.

비행기 탑승 전 안전 교육을 떠올려 보자. 사고가 날 것이라고 예상해서가 아니라, "일어날 수 있는 일"이기 때문에 승무원이 산소마스크 사용법을 반드시 설명한다. Checked Exception은 이 안전 교육과 같다.

**Unchecked는 "프로그래머의 실수"다.** 다음 코드를 보자.

```java
int[] arr = {1, 2, 3};
System.out.println(arr[10]);   // ArrayIndexOutOfBoundsException
```

배열의 크기는 3인데 인덱스 10에 접근했다. 이것은 외부 세계의 불확실성이 아니라 **명백한 코드 버그**다. 자바는 이런 예외에 대해서는 처리를 강제하지 않는다.

### 왜 실수에는 처리를 강제하지 않는가?

여기서 자바 설계자의 판단이 드러난다. 만약 모든 예외에 처리를 강제한다면 어떻게 될까?

```java
// 만약 배열 접근마다 try-catch를 강제한다면...
try {
    int x = arr[i];
} catch (ArrayIndexOutOfBoundsException e) {
    // ...
}
try {
    String s = name.trim();   // name이 null일 수 있으니 또 try-catch?
} catch (NullPointerException e) {
    // ...
}
```

모든 줄이 `try-catch`로 뒤덮인다. 코드는 읽을 수 없을 만큼 장황해진다. 게다가 더 중요한 논점이 있다. **프로그래머의 실수에 대한 올바른 대응은 "예외를 잡는 것"이 아니라 "실수를 하지 않는 것"이다.** 배열 인덱스가 범위를 벗어나면 `try-catch`로 감쌀 게 아니라 인덱스 계산 로직을 고쳐야 한다. `null`이 문제라면 애초에 `null`이 들어오지 않도록 설계해야 한다.

그래서 자바는 이렇게 나눴다. 외부 실패(Checked)는 "대비하라"고 강제하고, 내부 버그(Unchecked)는 "코드를 고쳐라"라며 강제하지 않는다.

### 최근 트렌드: Unchecked 선호

흥미롭게도 최근의 자바 생태계는 Checked보다 Unchecked를 선호하는 방향으로 흐르고 있다. 이유는 세 가지다.

1. **Checked는 장황하다.** `throws`가 메서드 시그니처를 타고 위로 계속 전파되면서 코드가 지저분해진다.
2. **catch에서 제대로 처리하지 않는 경우가 많다.** 강제로 잡게 하니 `catch (Exception e) {}`처럼 그냥 무시해 버리는 나쁜 습관이 퍼졌다. 강제가 오히려 부실한 처리를 낳은 것이다.
3. **Spring 같은 주요 프레임워크가 Unchecked를 사용한다.** Spring은 `SQLException`(Checked) 같은 것을 `DataAccessException`(Unchecked)으로 감싸서 던진다. 개발자가 원하는 곳에서만 처리하도록 자유를 주는 것이다.

> 오해하면 안 되는 것: 이 트렌드는 **"예외처리를 하지 말라"** 는 뜻이 절대 아니다. "컴파일러가 강제하지 않되, 필요한 곳에서는 자발적으로 처리하라"는 뜻이다. 강제의 유무일 뿐, 처리의 필요성이 사라진 것은 아니다.

## 7.3 try-catch-finally: 예외를 잡는 문법

### 정의

`try-catch-finally`는 예외를 감지하고 처리하는 자바의 기본 문법이다.

- `try`: 예외가 발생할 수 있는 코드를 감싼다.
- `catch`: 예외가 발생하면 실행되는 처리 블록이다.
- `finally`: 예외 발생 여부와 관계없이 **항상** 실행되는 블록이다.

### 원리: 실행 흐름

```
┌──────────────────────────────────────────────┐
│  try {                                         │
│     ①위험한 코드 실행                            │
│  }                                             │
│                                                │
│   예외 발생?                                     │
│   ├─ NO ──────────────▶ catch 건너뜀            │
│   └─ YES ─▶ ②matching catch로 점프              │
│                                                │
│  catch (ExceptionType e) {                     │
│     ③예외 처리                                   │
│  }                                             │
│                                                │
│  finally {                                     │
│     ④항상 실행 (예외 발생하든 안 하든, 심지어      │
│        try/catch 안에서 return 해도 실행됨)      │
│  }                                             │
└──────────────────────────────────────────────┘
```

`finally`가 "항상" 실행된다는 점이 핵심이다. `try` 안에서 `return`을 하더라도, `catch`에서 다시 예외를 던지더라도 `finally`는 실행된다. 그래서 자원 정리(파일 닫기, 연결 해제) 같은 "무슨 일이 있어도 해야 하는 일"을 여기 넣었다.

### 코드 예제

```java
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;

public class TryCatchFinallyExample {
    public static void main(String[] args) {
        BufferedReader reader = null;
        try {
            reader = new BufferedReader(new FileReader("data.txt"));
            String firstLine = reader.readLine();
            System.out.println("첫 줄: " + firstLine);

            int number = Integer.parseInt(firstLine); // 숫자가 아니면 예외
            System.out.println("숫자 변환 성공: " + number);

        } catch (IOException e) {
            // 파일 관련 실패 (Checked)
            System.out.println("파일을 읽는 중 오류: " + e.getMessage());

        } catch (NumberFormatException e) {
            // 숫자 변환 실패 (Unchecked이지만 잡을 수 있음)
            System.out.println("첫 줄이 숫자가 아닙니다: " + e.getMessage());

        } finally {
            // 예외가 나든 안 나든 파일은 반드시 닫아야 한다
            if (reader != null) {
                try {
                    reader.close();
                } catch (IOException e) {
                    System.out.println("파일 닫기 실패: " + e.getMessage());
                }
            }
            System.out.println("정리 완료");
        }
    }
}
```

`finally` 안에서 다시 `try-catch`를 해야 하는 지저분함이 보이는가? `reader.close()` 자체가 `IOException`을 던질 수 있기 때문이다. 이 지저분함을 해결하는 것이 뒤에 나올 `try-with-resources`다.

### multi-catch: 여러 예외를 한 번에

여러 예외를 같은 방식으로 처리하고 싶다면 `|`(파이프)로 묶을 수 있다. 자바 7부터 도입됐다.

```java
try {
    process();
} catch (IOException | SQLException e) {
    // IOException이든 SQLException이든 동일하게 처리
    log.error("작업 실패", e);
}
```

한 가지 주의: multi-catch로 묶은 예외들은 서로 상속 관계에 있으면 안 된다. `catch (IOException | FileNotFoundException e)`는 `FileNotFoundException`이 이미 `IOException`의 하위 타입이라 중복이므로 컴파일 오류다.

### catch 순서 규칙

`catch` 블록은 위에서 아래로 검사되므로, **구체적인 예외를 먼저, 넓은 예외를 나중에** 써야 한다.

```java
try {
    ...
} catch (FileNotFoundException e) {   // 구체적인 것 먼저
    ...
} catch (IOException e) {              // 더 넓은 것 나중
    ...
}
```

순서를 바꿔 `IOException`을 먼저 쓰면, `FileNotFoundException`은 영원히 잡히지 않는 죽은 코드가 되어 컴파일 오류가 난다.

## 7.4 try-with-resources: 자원을 안전하게 닫다

### 정의

`try-with-resources`는 자바 7에서 도입된 문법으로, `try(...)` 괄호 안에 선언한 자원을 블록이 끝날 때 **자동으로 닫아주는** 기능이다.

### 원리: 왜 필요한가?

파일, DB 연결, 소켓 같은 자원은 사용 후 반드시 닫아야 한다. 닫지 않으면 무슨 일이 벌어질까?

- **파일**: 파일 핸들이 계속 열려 파일 잠금이 걸리거나 핸들이 고갈된다.
- **DB 연결**: 커넥션 풀이 고갈되어 결국 새 연결을 못 맺고 서비스가 멈춘다.
- **메모리**: 자원이 회수되지 않아 메모리 누수가 쌓인다.

`finally`로 닫을 수 있지만, 앞에서 봤듯 `finally` 방식은 널 체크와 중첩 `try-catch`로 지저분하다. `try-with-resources`는 이 정리 코드를 컴파일러가 대신 생성해 준다.

```
[finally 방식]                    [try-with-resources 방식]
try {                             try (Resource r = open()) {
    r = open();                       use(r);
    use(r);                       }
} finally {                       // 끝. close()는 자동 호출
    if (r != null) r.close();     // 예외가 나도 자동 호출
}
```

### 동작 원리: AutoCloseable

`try-with-resources`의 괄호 안에 들어갈 수 있는 것은 `AutoCloseable` 인터페이스를 구현한 객체뿐이다. 이 인터페이스는 `close()` 메서드 하나만 요구한다. 블록을 벗어날 때(정상 종료든 예외 발생이든) 컴파일러가 자동으로 `close()`를 호출하는 코드를 삽입한다.

```java
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;

public class TryWithResourcesExample {
    public static void main(String[] args) {
        // reader는 try 블록이 끝나면 자동으로 close()된다
        try (BufferedReader reader = new BufferedReader(new FileReader("data.txt"))) {
            String line;
            while ((line = reader.readLine()) != null) {
                System.out.println(line);
            }
        } catch (IOException e) {
            System.out.println("파일 처리 오류: " + e.getMessage());
        }
        // finally에서 close() 할 필요 없음! 이미 자동으로 닫혔다.
    }
}
```

여러 자원을 세미콜론으로 나열할 수도 있으며, 닫히는 순서는 선언의 **역순** 이다(나중에 연 것을 먼저 닫는다).

```java
try (Connection conn = getConnection();
     PreparedStatement stmt = conn.prepareStatement(sql)) {
    stmt.execute();
}
// 닫히는 순서: stmt.close() → conn.close()
```

### 직접 만들어 보는 AutoCloseable

`AutoCloseable`은 우리가 만든 클래스에도 붙일 수 있다.

```java
public class DatabaseSession implements AutoCloseable {
    public DatabaseSession() {
        System.out.println("DB 세션 시작");
    }

    public void query(String sql) {
        System.out.println("실행: " + sql);
    }

    @Override
    public void close() {
        // try 블록이 끝나면 자동으로 호출된다
        System.out.println("DB 세션 종료 (자원 반납)");
    }
}
```

```java
public class Main {
    public static void main(String[] args) {
        try (DatabaseSession session = new DatabaseSession()) {
            session.query("SELECT * FROM users");
        }
        // 출력:
        // DB 세션 시작
        // 실행: SELECT * FROM users
        // DB 세션 종료 (자원 반납)   ← close()가 자동 호출됨
    }
}
```

## 7.5 커스텀 예외와 throw / throws

### throw vs throws: 헷갈리는 두 단어

이름이 비슷해 자주 혼동되지만 역할이 완전히 다르다.

```
throw   : 예외를 "지금 이 자리에서" 실제로 던진다 (동작).
          throw new IllegalArgumentException("잘못된 값");

throws  : 이 메서드가 예외를 던질 "수 있음"을 시그니처에 선언한다 (약속).
          public void read() throws IOException { ... }
```

```
┌────────────────────────────────────────────────┐
│  public void withdraw(int amount)                │
│         throws InsufficientBalanceException {  ◀──┐ 선언(약속)
│                                                    │
│      if (amount > balance) {                       │
│          throw new InsufficientBalanceException(); ◀── 실제 던짐(동작)
│      }                                             │
│      balance -= amount;                            │
│  }                                                 │
└────────────────────────────────────────────────┘
```

`throws`는 "나는 이런 예외를 던질 수 있으니 나를 호출하는 쪽에서 대비하라"는 계약이다. Checked Exception은 `throws`로 선언하거나 `try-catch`로 처리해야 컴파일된다.

### 커스텀 예외: 왜 만드는가?

비즈니스 로직에서 발생하는 실패를 표준 예외로만 표현하면 정보가 부족하다.

```java
// 나쁜 예: 무슨 일이 일어났는지 불명확
throw new RuntimeException("잔액 부족");
```

이것은 병원에서 "아파요"라고만 말하는 것과 같다. 어디가, 얼마나, 왜 아픈지 알 수 없다. 커스텀 예외는 "우측 슬개골 인대에 염증이 있습니다"처럼 구체적인 진단서를 만들어 준다.

```java
// 좋은 예: 명확한 타입 + 진단에 필요한 정보
public class InsufficientBalanceException extends RuntimeException {
    private final String accountNumber;
    private final long requestedAmount;
    private final long currentBalance;

    public InsufficientBalanceException(String accountNumber,
                                        long requestedAmount,
                                        long currentBalance) {
        super(String.format(
            "잔액 부족 [계좌:%s] 요청금액:%d, 현재잔액:%d",
            accountNumber, requestedAmount, currentBalance));
        this.accountNumber = accountNumber;
        this.requestedAmount = requestedAmount;
        this.currentBalance = currentBalance;
    }

    public String getAccountNumber()  { return accountNumber; }
    public long getRequestedAmount()  { return requestedAmount; }
    public long getCurrentBalance()   { return currentBalance; }
}
```

이제 이 예외를 잡는 쪽은 계좌번호, 요청 금액, 현재 잔액을 모두 꺼내 쓸 수 있다. "얼마가 부족한지 화면에 표시"하는 것 같은 세밀한 대응이 가능해진다.

> 팁: `RuntimeException`을 상속하면 Unchecked 예외가 되고(처리 강제 없음), `Exception`을 상속하면 Checked 예외가 된다(처리 강제). 비즈니스 예외는 대개 Unchecked로 만드는 것이 현대적 관례다.

### 예외 연쇄(Exception Chaining): 원인을 잃지 마라

낮은 계층에서 발생한 예외를 높은 계층의 예외로 감싸 던질 때, 원래 원인(cause)을 반드시 함께 넘겨야 한다.

```java
try {
    jdbcTemplate.update(sql);
} catch (SQLException e) {
    // e를 두 번째 인자로 넘겨 "원인"을 보존한다
    throw new UserRegistrationException("회원 가입 실패", e);
    //                                                  ▲ cause
}
```

왜 중요한가? 원인을 넘기지 않으면 스택 트레이스에서 "회원 가입 실패"까지만 보이고, 정작 진짜 원인인 `SQLException`(예: DB 연결 끊김)이 사라진다. 범인을 잡았는데 진술서를 태워버리는 셈이다. `cause`를 넘기면 스택 트레이스에 `Caused by:` 섹션이 붙어 근본 원인까지 추적할 수 있다.

```
UserRegistrationException: 회원 가입 실패
    at UserService.register(UserService.java:42)
    ...
Caused by: java.sql.SQLException: Connection timed out   ◀── 진짜 원인 보존됨
    at ...
```

### 스택 트레이스 읽는 법

예외가 처리되지 않으면 스택 트레이스가 출력된다. 읽는 순서는 **위에서 아래** 다.

```
Exception in thread "main" java.lang.NullPointerException     ◀ ① 예외 종류
    at com.edu.OrderService.calcTotal(OrderService.java:57)   ◀ ② 예외가 터진 바로 그 지점
    at com.edu.OrderService.checkout(OrderService.java:31)    ◀ ③ 그걸 호출한 곳
    at com.edu.Main.main(Main.java:12)                        ◀ ④ 최초 시작점
```

- **맨 윗줄**: 어떤 예외인지 + 메시지.
- **첫 번째 `at`**: 예외가 실제로 발생한 정확한 파일과 줄 번호. 여기부터 보라.
- **아래로 갈수록**: 호출한 쪽으로 거슬러 올라가는 순서. 즉 호출 스택을 되짚는 것이다.

디버깅의 첫걸음은 스택 트레이스의 첫 번째 `at`이 가리키는 줄을 여는 것이다.

## ⚠️ 흔한 오해와 함정

### 함정 1. 예외를 삼키지 마라

가장 위험한 안티패턴이다.

```java
try {
    riskyOperation();
} catch (Exception e) {
    // 아무것도 안 함  ← 최악!
}
```

이것은 화재경보기가 울리자 소리만 끄고 불은 그대로 두는 것과 같다. 문제가 사라진 게 아니라 **보이지 않게 숨겨진** 것이다. 나중에 원인 모를 버그로 돌아온다. 최소한 로그를 남기거나 다시 던져야 한다.

```java
catch (Exception e) {
    log.error("작업 실패", e);              // 최소한 기록
    throw new BusinessException("처리 실패", e); // 또는 감싸서 다시 던짐
}
```

### 함정 2. `catch (Exception e)`로 뭉뚱그리기

병원에서 "아파요"라고만 하는 것과 같다. 가능한 한 구체적인 예외를 잡아야 각기 다른 대응을 할 수 있다.

```java
// 나쁨: 무엇이 왜 실패했는지 구분 불가
catch (Exception e) { ... }

// 좋음: 실패 종류별로 다르게 대응
catch (FileNotFoundException e) { /* 파일 다시 요청 */ }
catch (IOException e)           { /* 재시도 */ }
```

### 함정 3. `finally`에서 `return` 하지 마라

`finally`의 `return`은 `try`나 `catch`에서 던진 예외조차 **먹어버린다.**

```java
try {
    throw new RuntimeException("중요한 오류");
} finally {
    return 0;   // 위 예외가 사라진다! 절대 하지 말 것
}
```

### 함정 4. Error를 잡으려 하지 마라

`OutOfMemoryError`나 `StackOverflowError`를 `catch (Throwable t)`로 잡는 코드를 종종 보지만, 대부분 무의미하거나 위험하다. 메모리가 이미 고갈된 상태에서 정상 복구는 거의 불가능하다. Error는 프로그램을 고쳐서 예방할 문제이지 잡아서 처리할 문제가 아니다.

### 함정 5. 예외를 흐름 제어에 쓰지 마라

예외를 `if-else` 대용으로 쓰지 말자. 예외 객체 생성과 스택 트레이스 수집은 비용이 크고, 코드 의도도 흐려진다.

```java
// 나쁨: 정상 흐름을 예외로 제어
try {
    return map.get(key).toString();
} catch (NullPointerException e) {
    return "없음";
}
// 좋음: 조건으로 제어
Object v = map.get(key);
return v != null ? v.toString() : "없음";
```

### 면접 팁

- **"Checked와 Unchecked의 차이는?"** → 컴파일러 강제 여부. Checked는 예측 가능한 외부 실패라 처리를 강제하고, Unchecked는 프로그래머의 실수라 강제하지 않는다.
- **"finally는 항상 실행되나?"** → 거의 항상. 예외는 `System.exit()` 호출이나 JVM 강제 종료 정도다.
- **"throw와 throws의 차이는?"** → `throw`는 예외를 실제로 던지는 동작, `throws`는 던질 수 있음을 선언하는 약속.
- **"try-with-resources의 조건은?"** → 자원이 `AutoCloseable`(또는 `Closeable`)을 구현해야 하며, 닫히는 순서는 선언의 역순.

## 연습문제

**문제 1.** 다음 코드의 출력 순서를 예측하라.
```java
public static int test() {
    try {
        return 1;
    } finally {
        System.out.println("finally 실행");
    }
}
// main에서: System.out.println(test());
```
<details>
<summary>힌트</summary>
finally는 return 직전에 실행된다. try의 return 값이 준비된 뒤 finally가 실행되고, 그 다음 실제로 반환된다.
</details>
<details>
<summary>해설</summary>
출력은 <code>finally 실행</code>이 먼저, 그 다음 <code>1</code>이다. try에서 반환값 1을 준비하지만, 실제 반환 직전에 finally 블록이 반드시 실행되기 때문이다.
</details>

**문제 2.** `IOException`을 `catch`하기 전에 `Exception`을 먼저 `catch`하면 어떻게 되는가?
<details>
<summary>힌트</summary>
catch는 위에서 아래로 검사되고, Exception은 IOException의 상위 타입이다.
</details>
<details>
<summary>해설</summary>
컴파일 오류가 난다. 상위 타입인 <code>Exception</code>이 먼저 모든 예외를 잡아버려, 뒤의 <code>IOException catch</code>는 도달할 수 없는(unreachable) 죽은 코드가 되기 때문이다. 구체적인 예외를 먼저 써야 한다.
</details>

**문제 3.** 은행 계좌 클래스에서 출금 시 잔액이 부족하면 던질 커스텀 Unchecked 예외 `InsufficientBalanceException`을 정의하고, 요청 금액과 현재 잔액을 메시지에 포함하도록 만들어라.
<details>
<summary>힌트</summary>
RuntimeException을 상속하고, 생성자에서 super(...)로 메시지를 넘긴다.
</details>
<details>
<summary>해설</summary>

```java
public class InsufficientBalanceException extends RuntimeException {
    public InsufficientBalanceException(long requested, long balance) {
        super(String.format("잔액 부족: 요청 %d, 잔액 %d", requested, balance));
    }
}
```
RuntimeException 상속으로 Unchecked가 되어 throws 선언 없이 던질 수 있다.
</details>

**문제 4.** 아래 `finally` 방식 코드를 `try-with-resources`로 다시 작성하라.
```java
BufferedReader r = null;
try {
    r = new BufferedReader(new FileReader("a.txt"));
    System.out.println(r.readLine());
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (r != null) { try { r.close(); } catch (IOException e) {} }
}
```
<details>
<summary>힌트</summary>
자원 선언을 try 괄호 안으로 옮기면 finally가 통째로 사라진다.
</details>
<details>
<summary>해설</summary>

```java
try (BufferedReader r = new BufferedReader(new FileReader("a.txt"))) {
    System.out.println(r.readLine());
} catch (IOException e) {
    e.printStackTrace();
}
```
BufferedReader는 AutoCloseable을 구현하므로 블록 종료 시 자동으로 close()된다.
</details>

**문제 5.** `SQLException`을 잡아서 커스텀 예외 `DataAccessException`으로 감싸 던질 때, 원래 예외를 반드시 넘겨야 하는 이유는?
<details>
<summary>힌트</summary>
스택 트레이스의 "Caused by:" 섹션을 떠올려 보라.
</details>
<details>
<summary>해설</summary>
원인(cause)을 넘기지 않으면 근본 원인인 SQLException 정보가 사라져 디버깅이 불가능해진다. <code>throw new DataAccessException("메시지", e)</code>처럼 원래 예외를 넘기면 스택 트레이스에 <code>Caused by: SQLException...</code>이 붙어 진짜 원인까지 추적할 수 있다.
</details>

## 요약

- 예외처리는 실패를 없애는 기술이 아니라, 실패에서 프로그램을 살려내는 안전장치다. 실패는 반드시 일어나므로 대비가 필요하다.
- 자바의 예외는 `Throwable` 아래 `Error`(복구 불가, 건드리지 말 것)와 `Exception`으로 나뉘고, `Exception`은 다시 Checked(예측 가능한 외부 실패, 처리 강제)와 Unchecked(프로그래머 실수, 강제 없음)로 갈린다.
- `try-catch-finally`가 기본 문법이며, `finally`는 항상 실행되어 자원 정리에 쓰인다. 자바 7의 `try-with-resources`는 `AutoCloseable` 자원을 자동으로 닫아 이 정리 코드를 없애준다.
- 비즈니스 실패는 정보를 담은 커스텀 예외로 표현하고, 낮은 계층 예외를 감쌀 때는 원인(cause)을 넘겨 연쇄를 유지한다.
- 예외를 삼키지 말고, 구체적으로 잡고, 흐름 제어에 쓰지 마라. `throw`는 동작, `throws`는 선언이다.
