# 8장. 자바의 현대적 기능들

혹시 자바를 "오래되고 장황한 언어"라고 생각하고 있는가? 세미콜론과 중괄호가 넘쳐나고, 데이터 하나 담으려고 50줄짜리 클래스를 써야 하는 언어. 만약 그런 인상을 가지고 있다면, 그것은 **10년 전의 자바**다.

자바는 정체된 언어가 아니다. 2017년 Java 9 이후, 자바는 **6개월마다 새 버전**을 내놓는다. 3월과 9월, 마치 계절이 바뀌듯 정기적으로 새 버전이 나온다. 그리고 이 새 버전들은 단순한 버그 수정이 아니다. 그동안 개발자들이 "왜 자바는 이걸 이렇게 불편하게 해야 하지?"라고 불평하던 지점들을, 하나씩 다른 언어(Kotlin, Scala, Python)의 좋은 점을 흡수하며 개선해 왔다.

그렇다면 왜 이 장이 중요할까? 두 가지 이유가 있다.

첫째, **면접과 실무의 기준선이 올라갔다.** 요즘 실무 코드에서 `record`, `switch expression`, `var`는 흔하게 쓰인다. 이걸 모르면 남의 코드를 읽지 못한다. 남의 코드를 못 읽으면 협업이 안 된다.

둘째, 그리고 더 중요한 것은, 이 기능들이 **하나의 방향성을 가지고 있다**는 점이다. 그 방향은 "**보일러플레이트(boilerplate, 반복되는 뻔한 코드)를 줄이고, 컴파일러가 더 많은 것을 검증하게 만든다**"이다. 즉, 개발자는 의도만 표현하고 나머지 검증은 컴파일러에게 맡기자는 철학이다.

이 장에서 다룰 `record`, `sealed class`, `pattern matching`, `switch expression`은 따로 노는 기능이 아니다. 이들은 서로 맞물려 **"데이터를 안전하게 다루는 새로운 방식"**을 함께 만들어낸다. 각 기능이 왜 등장했는지, 그리고 어떻게 서로 시너지를 내는지를 이해하면, 자바를 바라보는 시각 자체가 바뀔 것이다.

## 8.1 record: DTO를 위한 완벽한 도구

### 정의

`record`는 Java 16에서 정식 도입된, **불변 데이터를 담기 위한 특별한 종류의 클래스**다. 이름 그대로 "기록(record)"을 위한 것이다. 한 줄이면 충분하다.

```java
public record Person(String name, int age) {}
```

이 한 줄이 하는 일을 이해하려면, 먼저 record가 없던 시절 우리가 무엇을 해야 했는지를 봐야 한다.

### 원리(왜 record가 등장했나): 보일러플레이트의 지옥

데이터를 담는 클래스, 즉 DTO(Data Transfer Object)를 만든다고 해보자. 이름과 나이, 딱 두 개의 값만 담고 싶다. 그런데 이 두 값을 "제대로" 담으려면 자바는 다음을 요구한다.

```java
public class Person {
    private final String name;   // 1. 필드
    private final int age;

    public Person(String name, int age) {   // 2. 생성자
        this.name = name;
        this.age = age;
    }

    public String getName() { return name; }   // 3. getter
    public int getAge() { return age; }

    @Override
    public boolean equals(Object o) {          // 4. equals
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Person person = (Person) o;
        return age == person.age && Objects.equals(name, person.name);
    }

    @Override
    public int hashCode() {                     // 5. hashCode
        return Objects.hash(name, age);
    }

    @Override
    public String toString() {                  // 6. toString
        return "Person{name='" + name + "', age=" + age + "}";
    }
}
```

값 두 개를 담으려고 **50줄**을 썼다. 필드 2개, 생성자 1개, getter 2개, `equals`, `hashCode`, `toString`. 이 코드에서 진짜 "정보"는 무엇인가? `name(String)`과 `age(int)`, 그것뿐이다. 나머지는 전부 **기계적으로 반복되는 뻔한 코드**, 즉 보일러플레이트다.

문제는 단순히 길다는 것이 아니다. **위험하다**는 것이다. 필드를 하나 추가했는데 `equals`나 `hashCode`에 반영하는 것을 깜빡하면? 겉으로는 멀쩡한데 `HashMap`에서 값을 못 찾는 미묘한 버그가 생긴다. 사람은 반복 작업에서 실수한다.

record는 이 모든 것을 대신해준다.

```java
public record Person(String name, int age) {}
```

이 한 줄로 자바 컴파일러가 자동 생성하는 것:

| 자동 생성 항목 | 설명 |
|---|---|
| `private final` 필드 | `name`, `age` — 자동으로 불변 필드 |
| 생성자 | `Person(String name, int age)` |
| 접근자(accessor) | `name()`, `age()` — 주의: `getName()`이 아니라 `name()`이다 |
| `equals()` | 모든 필드를 비교 |
| `hashCode()` | 모든 필드 기반 |
| `toString()` | `Person[name=..., age=...]` 형태 |

```
     record Person(String name, int age) {}
                │
                ▼  컴파일러가 자동 생성
     ┌──────────────────────────────────┐
     │ private final String name;       │
     │ private final int age;           │
     │ Person(String, int)  [생성자]     │
     │ name(), age()        [접근자]     │
     │ equals(), hashCode() [비교]       │
     │ toString()           [출력]       │
     └──────────────────────────────────┘
```

주목할 점은 접근자 이름이 `getName()`이 아니라 **`name()`**이라는 것이다. record는 "이건 그냥 데이터다"라는 관점이므로, JavaBeans의 `get` 접두사 관례를 따르지 않는다. 필드 이름 그대로 메서드가 된다.

### 왜 불변(Immutability)이 좋은가

record의 필드는 모두 `final`이다. 즉, 한번 만든 record 객체는 **값을 바꿀 수 없다.** 이것이 우연이 아니라 record의 핵심 설계다.

> **비유: 공증된 계약서**
> 볼펜으로 아무렇게나 쓴 메모는 나중에 슬쩍 고칠 수 있다. 그래서 믿기 어렵다. 반면 공증된 계약서는 **수정이 불가능**하다. 바로 그렇기 때문에 모두가 믿을 수 있다. record는 공증된 계약서다.

불변이 주는 구체적 이점:

1. **멀티스레드 안전(Thread-safe).** 값이 안 바뀌므로 여러 스레드가 동시에 읽어도 아무 문제가 없다. 락(lock)이 필요 없다. 동시성 버그의 상당수는 "누군가 값을 바꿔서" 생기는데, 애초에 못 바꾸면 그 버그가 원천 봉쇄된다.
2. **예측 가능성.** `person`을 함수에 넘겼는데, 그 함수가 내 `person`의 나이를 몰래 100살로 바꿔놓을 걱정을 할 필요가 없다. 값은 만든 그대로 유지된다.
3. **부작용(side effect) 없음.** 공유해도 안전하다. `Map`의 키로 써도 안전하다(키가 바뀌면 해시 구조가 깨지는데, 안 바뀌니까).

### record의 compact constructor(컴팩트 생성자)와 유효성 검증

"불변이면 좋은데, 값을 검증하고 싶으면 어쩌지? 나이가 음수인 사람을 막고 싶은데?" record는 이를 위해 **compact constructor(컴팩트 생성자)**라는 특별한 문법을 제공한다.

```java
public record Person(String name, int age) {
    // 컴팩트 생성자: 매개변수 목록도, this.name = name 도 쓰지 않는다.
    public Person {
        if (age < 0) {
            throw new IllegalArgumentException("나이는 음수일 수 없습니다: " + age);
        }
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException("이름은 비어 있을 수 없습니다");
        }
        // this.name = name; 같은 필드 대입은 자동으로 처리됨! 쓸 필요 없다.
    }
}
```

무엇이 특별한가? 일반 생성자와 달리 **괄호 안에 매개변수를 다시 쓰지 않는다**(`public Person {`). 그리고 마지막의 필드 대입(`this.name = name`)도 컴파일러가 자동으로 붙여준다. 개발자는 오직 **검증 로직만** 작성하면 된다. 이 검증 블록이 끝나면, 자바가 알아서 `this.name = name; this.age = age;`를 실행한다.

값을 정규화(normalize)할 수도 있다. 이때 매개변수 값을 재할당하면 그 값이 필드에 들어간다.

```java
public record Point(int x, int y) {
    public Point {
        x = Math.max(0, x);  // x를 0 이상으로 보정. 이 값이 필드에 저장된다.
        y = Math.max(0, y);
    }
}
```

이 덕분에 record는 **"항상 유효한 상태로만 존재하는 객체"**를 보장할 수 있다. 잘못된 값으로는 애초에 생성 자체가 안 된다.

### record의 한계(무엇을 못 하나)

record는 만능이 아니다. "데이터를 담는" 목적에 최적화된 만큼 제약이 있다. 이것을 아는 것이 record를 올바로 쓰는 길이다.

1. **상속 불가.** record는 다른 클래스를 `extends` 할 수 없다. (내부적으로 이미 `java.lang.Record`를 상속하고 있기 때문이다.) 단, `interface`는 여러 개 구현할 수 있다.
2. **필드 추가 불가.** 인스턴스 필드는 오직 헤더의 컴포넌트(`(String name, int age)`)뿐이다. `private int extra;` 같은 인스턴스 필드를 몸통에 추가할 수 없다. (단, `static` 필드는 가능하다.)
3. **가변(mutable) 상태에 부적합.** 값이 계속 바뀌는 객체(예: 게임 캐릭터의 체력)에는 맞지 않는다. 그런 것은 일반 class를 써야 한다.
4. **암묵적으로 final.** record 자체를 상속할 수 없다. `sealed`와의 조합에서 이 성질이 중요해진다(8.2 참고).

정리하면, record는 **"불변 데이터 캐리어"**다. DTO, 좌표, 응답 객체, 이벤트 같은 "값 그 자체"를 표현할 때 완벽하다. 반면 동작과 상태 변화가 핵심인 객체에는 일반 class가 여전히 필요하다.

## 8.2 sealed class: 상속을 제한하는 이유

### 정의

`sealed class`(봉인된 클래스)는 Java 17에서 정식 도입된 기능으로, **어떤 클래스가 이 클래스를 상속할 수 있는지를 명시적으로 제한**한다.

```java
public sealed class Shape permits Circle, Rectangle, Triangle {}
```

`permits`(허가한다) 키워드 뒤에 나열된 세 클래스만 `Shape`를 상속할 수 있다. 그 외에는 누구도 상속할 수 없다. 컴파일러가 막는다.

### 원리(왜 상속을 제한하나): 개방의 위험

객체지향의 기본 정신은 "상속을 통한 확장"이다. 그런데 왜 이걸 굳이 제한할까? 오히려 좋은 것 아닌가?

> **비유: 프랜차이즈 레스토랑**
> 유명 프랜차이즈 본사는 아무나 그 브랜드로 가게를 열게 두지 않는다. **본사가 심사하고 허가한 가맹점만** 그 간판을 달 수 있다. 왜? 아무나 그 이름으로 장사하면 품질을 통제할 수 없고, 한 곳에서 사고가 나면 브랜드 전체가 무너지기 때문이다.
>
> `sealed`가 정확히 이것이다. "`Shape`라는 간판을 달 수 있는 건 `Circle`, `Rectangle`, `Triangle` 이 세 곳뿐이다"라고 본사(원 클래스)가 못 박는 것이다.

상속이 무제한으로 열려 있으면 어떤 문제가 생길까? 내가 `Shape`를 만들었는데, 어디선가 모르는 누군가가 `WeirdShape extends Shape`를 만들 수 있다. 그러면 나는 "`Shape`의 종류가 몇 가지인지" 절대 알 수 없다. 미래에 무엇이 추가될지 모르니, 모든 코드를 "혹시 모를 미지의 하위 타입"에 대비해 방어적으로 짜야 한다.

### 왜 필요한가: 세 가지 이유

**① 안정성 — 예상 밖의 확장 방지**

`Shape`의 하위 타입을 내가 통제한다. 아무도 몰래 이상한 도형을 끼워넣을 수 없다. 이는 API를 설계할 때 특히 중요하다. "우리가 지원하는 도형은 이게 전부입니다"를 언어 차원에서 보장한다.

**② 완전한 분기 처리 — 컴파일러가 "전부"를 안다**

`Shape`가 `Circle`, `Rectangle`, `Triangle` 셋뿐임을 컴파일러가 안다. 이것이 왜 강력할까? 다음 절에서 이어진다.

**③ 패턴 매칭과의 시너지**

`sealed` + `switch`의 조합은 자바 현대화의 핵심 장면이다.

```java
double area = switch (shape) {
    case Circle c    -> Math.PI * c.radius() * c.radius();
    case Rectangle r -> r.width() * r.height();
    case Triangle t  -> 0.5 * t.base() * t.height();
    // default가 필요 없다!
};
```

`Shape`가 `sealed`이고 셋뿐임을 컴파일러가 알기 때문에, 이 세 case를 모두 처리하면 **"모든 경우를 다뤘다"**고 컴파일러가 인정한다. 그래서 `default:`가 필요 없다.

반대로, 만약 `Triangle` 케이스를 빠뜨리면? **컴파일 에러**가 난다. "Triangle을 처리 안 했잖아!"라고 컴파일러가 잡아준다.

```
        sealed Shape permits Circle, Rectangle, Triangle
                │
                ▼
     switch (shape) {
        case Circle    -> ...   ┐
        case Rectangle -> ...   ├─ 컴파일러: "3개 다 처리했군. OK"
        case Triangle  -> ...   ┘
     }

     ── 만약 Triangle을 빼면? ──
     switch (shape) {
        case Circle    -> ...
        case Rectangle -> ...
     }   ✗ 컴파일 에러: Triangle 처리 안 됨!
```

이것이 왜 혁명적인가? 예전에는 새 도형을 추가할 때, `switch`가 있는 모든 코드를 사람이 일일이 찾아 수정해야 했다. 하나라도 빠뜨리면 런타임에 조용히 `default`로 빠지며 버그가 됐다. 이제는 **새 하위 타입을 추가하는 순간, 처리 안 한 모든 switch가 컴파일 에러로 드러난다.** 컴파일러가 "여기, 여기, 여기 고쳐"라고 알려주는 것이다.

### sealed와 record, interface의 조합

`sealed`의 진짜 위력은 record, interface와 결합할 때 나온다. `sealed`는 class뿐 아니라 **interface**에도 붙일 수 있고, 하위 타입으로 **record**를 쓰는 것이 자바의 현대적 정석이다.

```java
// sealed interface: 이 인터페이스를 구현할 수 있는 건 아래 셋뿐
public sealed interface Shape permits Circle, Rectangle, Triangle {}

// 각 도형은 불변 데이터이므로 record가 안성맞춤
public record Circle(double radius) implements Shape {}
public record Rectangle(double width, double height) implements Shape {}
public record Triangle(double base, double height) implements Shape {}
```

이렇게 하면 무엇이 완벽해지는가?

- `record` → 각 도형은 불변이고 보일러플레이트가 없다.
- `sealed interface` → 도형의 종류가 정확히 셋으로 고정된다.
- `switch` → 컴파일러가 완전성을 보장한다.

이 세 개가 만나면, 함수형 언어에서 말하는 **대수적 데이터 타입(ADT, Algebraic Data Type)**을 자바에서 표현하게 된다. "이 타입은 A이거나 B이거나 C, 셋 중 하나다"를 언어가 보장하고, 그 각각을 안전하게 분기 처리한다.

> **permits 생략 규칙**: 하위 타입들을 **같은 파일**(또는 같은 패키지/모듈 안)에 선언하면 `permits`를 생략할 수 있다. 컴파일러가 알아서 찾는다. 파일이 나뉘어 있을 때만 명시적으로 `permits`를 쓴다.

또한 `sealed`의 하위 타입은 상속 정책을 반드시 하나 선택해야 한다.

| 키워드 | 의미 |
|---|---|
| `final` | 더 이상 상속 불가 (record는 자동으로 이것) |
| `sealed` | 이 하위 타입도 다시 permits로 제한 |
| `non-sealed` | 여기서부터는 다시 자유롭게 상속 허용 (봉인 해제) |

---

## 8.3 Text Blocks, Pattern Matching, Switch Expressions

이 절에서는 코드를 짧고 명확하게 만드는 세 가지 문법을 묶어서 본다. 세 가지 모두 "예전에는 왜 이렇게 불편했지?"라는 물음에 대한 답이다.

### 8.3.1 Text Blocks — 여러 줄 문자열 (Java 15+)

**정의:** `"""`(따옴표 세 개)로 감싸는, 여러 줄 문자열 리터럴.

JSON이나 SQL, HTML을 문자열로 쓸 때 예전에는 이런 지옥을 견뎌야 했다.

```java
// 이전: 이스케이프와 + 연결의 지옥
String json = "{\n" +
              "  \"name\": \"홍길동\",\n" +
              "  \"age\": 25\n" +
              "}";
```

큰따옴표는 `\"`로 이스케이프, 줄바꿈은 `\n`, 각 줄은 `+`로 연결. 실제 JSON이 어떻게 생겼는지 이 코드만 봐서는 알기 어렵다. 오타 내기도 쉽다.

Text Block은 있는 그대로 쓴다.

```java
// 이후: 보이는 그대로
String json = """
        {
          "name": "홍길동",
          "age": 25
        }
        """;
```

**왜 좋은가?** 가독성이다. 코드에 적힌 문자열의 모양이 실제 결과와 똑같다. 이스케이프도, `+`도 없다. SQL, JSON, HTML처럼 여러 줄에 걸친 구조화된 텍스트를 다룰 때 특히 빛난다.

> **들여쓰기 규칙**: Text Block은 가장 왼쪽에 있는 줄(닫는 `"""` 포함)을 기준으로 공통 들여쓰기를 자동으로 제거한다. 그래서 코드 안에서는 예쁘게 들여써도, 실제 문자열에는 그 들여쓰기가 들어가지 않는다.

### 8.3.2 var — 지역 변수 타입 추론 (Java 10+)

**정의:** `var`는 **지역 변수**의 타입을 컴파일러가 우변으로부터 알아서 추론하게 하는 키워드다.

```java
// 이전: 타입을 좌우에 두 번 쓴다
Map<String, List<Person>> groups = new HashMap<String, List<Person>>();

// 이후: 우변만 봐도 뻔하니, 좌변은 var
var groups = new HashMap<String, List<Person>>();
```

**왜 좋은가?** 위 예처럼 타입 이름이 길고 명백할 때, 좌변에 또 쓰는 것은 중복이다. `var`는 이 중복을 없앤다.

여기서 흔한 오해 하나를 미리 짚자. **`var`는 자바스크립트의 `var`가 아니다.** 자바의 `var`는 **여전히 정적 타입**이다. `var x = 10;`이라고 쓴 순간 `x`의 타입은 영원히 `int`로 고정된다. 나중에 `x = "hello";`는 컴파일 에러다. 타입이 사라진 게 아니라, **타입을 컴파일러가 추론했을 뿐**이다.

```
   var x = 10;
        │
        ▼  컴파일러가 우변(10)을 보고 추론
   int x = 10;   ← 실제로는 이것과 완전히 동일. 런타임 비용 0.
```

**주의할 점:**
- `var`는 **지역 변수에만** 쓸 수 있다. 필드, 메서드 매개변수, 반환 타입에는 못 쓴다.
- 초기값이 반드시 있어야 한다(`var x;`는 불가). 추론할 근거가 없기 때문이다.
- 타입이 우변에서 명백하지 않으면 오히려 가독성이 나빠진다. `var result = process();`는 result가 뭔지 알 수 없다. 이럴 땐 명시적 타입이 낫다.

### 8.3.3 Pattern Matching for instanceof — 타입 확인과 캐스팅을 한 번에

**정의:** `instanceof`로 타입을 확인하는 **동시에**, 그 타입의 변수로 바로 바인딩하는 문법(Java 16+).

예전에는 항상 두 단계였다.

```java
// 이전: 확인 → 캐스팅. 같은 타입을 두 번 쓴다.
if (obj instanceof String) {
    String s = (String) obj;   // 방금 String인 걸 확인했는데 또 캐스팅
    System.out.println(s.length());
}
```

`instanceof String`으로 이미 String임을 확인했다. 그런데 값을 쓰려면 또 `(String) obj`로 캐스팅해야 했다. 명백한 중복이다.

```java
// 이후: 확인하면서 바로 s에 담는다
if (obj instanceof String s) {
    System.out.println(s.length());   // s는 이미 String
}
```

`obj instanceof String s`가 참이면, `s`라는 String 변수가 자동으로 생겨서 바로 쓸 수 있다. **왜 좋은가?** "확인했으니 그 타입인 게 당연한데 왜 또 캐스팅?"이라는 중복을 제거했다. 실수(잘못된 타입으로 캐스팅)의 여지도 줄어든다.

### 8.3.4 record deconstruction patterns — 레코드 분해 패턴

패턴 매칭은 record와 만나면 한 단계 더 나아간다. record는 구조가 명확하므로, 그 구조를 **분해(deconstruct)**하며 매칭할 수 있다(Java 21+).

```java
record Point(int x, int y) {}

Object obj = new Point(3, 4);

// 타입 확인 + 내부 값 분해를 한 번에
if (obj instanceof Point(int x, int y)) {
    System.out.println("좌표 합: " + (x + y));  // x=3, y=4가 바로 꺼내진다
}
```

`Point(int x, int y)`는 "obj가 Point이면, 그 안의 x와 y를 각각 꺼내 이 이름으로 쓰겠다"는 뜻이다. `p.x()`, `p.y()`를 따로 호출할 필요가 없다. 중첩된 record도 분해된다.

```java
record Line(Point start, Point end) {}

if (obj instanceof Line(Point(var x1, var y1), Point(var x2, var y2))) {
    // 중첩 구조를 한 번에 풀어낸다. var로 타입 추론도 가능.
    System.out.println("길이 계산 가능");
}
```

이것이 함수형 언어에서 온 강력한 도구다. "이 데이터가 이런 모양이면, 그 안의 값들을 이렇게 꺼내 쓰겠다"를 한 줄로 표현한다.

### 8.3.5 Switch Expressions — 문(statement)이 아닌 식(expression) (Java 14+)

**정의:** `switch`를 값을 **반환하는 식(expression)**으로 쓰는 문법.

예전 switch는 "문장(statement)"이었다. 값을 만들려면 변수를 미리 선언하고, 각 case에서 대입하고, `break`를 붙여야 했다.

```java
// 이전: 장황하고 위험하다
String result;
switch (day) {
    case MONDAY:
    case TUESDAY:
    case WEDNESDAY:
    case THURSDAY:
    case FRIDAY:
        result = "업무일";
        break;   // 이걸 빼먹으면? 아래 case로 줄줄이 흘러내린다(fall-through)!
    case SATURDAY:
    case SUNDAY:
        result = "주말";
        break;
    default:
        result = "기타";
}
```

이 코드의 악명 높은 위험은 **fall-through**다. `break`를 하나라도 빠뜨리면, 실행이 다음 case로 그냥 흘러 내려간다. 이건 문법 에러가 아니라 **조용한 논리 버그**다.

Switch Expression은 이 모든 문제를 없앤다.

```java
// 이후: 간결하고 안전하다
String result = switch (day) {
    case MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY -> "업무일";
    case SATURDAY, SUNDAY -> "주말";
    default -> "기타";
};
```

무엇이 달라졌나?

1. **`->` 화살표 문법.** `break`가 아예 없다. 화살표 오른쪽만 실행되고 자동으로 끝난다. **fall-through 버그가 원천 봉쇄**된다.
2. **여러 값을 콤마로.** `case MONDAY, TUESDAY` 처럼 묶을 수 있다.
3. **값을 반환.** `switch` 전체가 하나의 값이 되어 `result`에 바로 대입된다.

여러 줄이 필요하면 중괄호와 `yield`를 쓴다.

```java
int score = switch (grade) {
    case "A" -> 100;
    case "B" -> {
        System.out.println("B등급 처리");
        yield 80;   // 블록에서 값을 낼 땐 yield
    }
    default -> 0;
};
```

### 8.3.6 when — guarded pattern (가드 패턴)

패턴 매칭 switch에서 "타입은 맞는데, 추가 조건까지 봐야 할 때" `when` 절을 붙인다(Java 21+).

```java
String describe(Object obj) {
    return switch (obj) {
        case Integer i when i < 0  -> "음수 정수";
        case Integer i when i == 0 -> "영";
        case Integer i            -> "양수 정수";   // 위 조건에 안 걸린 나머지
        case String s when s.isBlank() -> "빈 문자열";
        case String s              -> "문자열: " + s;
        default                    -> "알 수 없음";
    };
}
```

`case Integer i when i < 0`은 "Integer이면서 **그리고** 0보다 작을 때"를 뜻한다. 타입 매칭과 조건 검사를 하나의 case로 묶는 것이다.

> **주의: 순서가 중요하다.** 위에서 아래로 검사하므로, 더 구체적인 조건(`when` 붙은 것)을 먼저, 일반 조건을 나중에 써야 한다. `case Integer i`(모든 정수)를 맨 위에 두면 그 아래 `when` 케이스는 절대 도달할 수 없어 컴파일 에러가 난다.

### 면접 팁

- **"record와 class의 차이는?"** — record는 불변 데이터 캐리어이며 `equals`/`hashCode`/`toString`/접근자를 자동 생성하고 상속이 불가능하다. class는 상태 변화와 동작 중심이며 상속이 가능하다.
- **"sealed class는 왜 쓰나?"** — 하위 타입을 컴파일 타임에 고정해 switch의 완전성 검사를 가능하게 하고, 예상 밖의 확장을 막기 위해서다. record + sealed + switch로 안전한 대수적 데이터 타입을 표현한다.
- **"var를 쓰면 성능이 느려지나?"** — 아니다. var는 컴파일 타임에 타입이 결정되며 런타임 비용이 전혀 없다. 자바스크립트의 동적 var와 다르다.
- **"switch expression이 기존 switch보다 나은 점은?"** — 값을 반환하고, fall-through(break 누락) 버그를 없애며, 여러 라벨을 콤마로 묶어 간결하다.

## ⚠️ 흔한 오해와 함정

**① "record는 값을 절대 못 바꾸니 완전히 안전하다"** — 반은 맞고 반은 틀리다. record의 **참조 필드**가 가변 객체(예: `List`)라면, record 자체는 그 참조를 못 바꾸지만 참조가 가리키는 리스트 내용은 바뀔 수 있다. 진짜 불변을 원하면 컴팩트 생성자에서 `List.copyOf(...)`로 방어적 복사를 하라.

```java
public record Team(String name, List<String> members) {
    public Team {
        members = List.copyOf(members);  // 불변 복사본으로 교체
    }
}
```

**② "record의 getter는 getName()이다"** — 아니다. record 접근자는 `name()`이다. `get` 접두사가 없다. 이걸 몰라서 컴파일 에러를 내는 경우가 흔하다.

**③ "var는 아무 데나 쓸 수 있다"** — 지역 변수 전용이다. 필드, 메서드 매개변수, 반환 타입, `var x;`(초기값 없음)에는 못 쓴다.

**④ "sealed만 붙이면 끝"** — 하위 타입은 반드시 `final`, `sealed`, `non-sealed` 중 하나를 선택해야 한다. record 하위 타입은 자동으로 `final`이라 이 걱정이 없다(그래서 record와 궁합이 좋다).

**⑤ "switch expression에서도 break를 쓴다"** — 화살표(`->`) 문법에서는 `break`를 쓰지 않는다. 블록에서 값을 낼 때는 `break`가 아니라 `yield`다.

**⑥ "패턴 매칭 switch에 default는 항상 필요하다"** — sealed 타입을 모든 하위 타입 case로 완전히 커버하면 `default`가 필요 없고, 오히려 새 하위 타입 추가 시 컴파일 에러로 잡아주는 이점을 잃으므로 안 쓰는 게 낫다.

## 연습문제

**문제 1.** 다음 일반 클래스를 record로 바꿔라. 그리고 record의 접근자 메서드로 이름을 출력하는 코드도 써라.
```java
public class Book {
    private final String title;
    private final int price;
    // 생성자, getter, equals, hashCode, toString ...
}
```
<details><summary>힌트</summary>한 줄로 끝난다. 접근자는 <code>get</code> 없이 필드명 그대로다.</details>
<details><summary>해설</summary>

```java
public record Book(String title, int price) {}

Book b = new Book("자바 개념서", 30000);
System.out.println(b.title());  // getTitle()이 아니라 title()
```
</details>

**문제 2.** `Book`의 가격은 0 이상이어야 한다. 음수면 예외를 던지도록 컴팩트 생성자를 추가하라.
<details><summary>힌트</summary>매개변수 목록도, <code>this.price = price</code>도 쓰지 않는다.</details>
<details><summary>해설</summary>

```java
public record Book(String title, int price) {
    public Book {
        if (price < 0) throw new IllegalArgumentException("가격은 음수일 수 없습니다");
    }
}
```
필드 대입은 자동으로 처리되므로 검증만 쓰면 된다.
</details>

**문제 3.** 다음 `sealed interface`와 record들이 주어졌을 때, 각 결제 수단의 수수료를 계산하는 switch expression을 작성하라. `default`는 쓰지 말 것.
```java
sealed interface Pay permits Card, Cash, Point {}
record Card(int amount) implements Pay {}
record Cash(int amount) implements Pay {}
record Point(int amount) implements Pay {}
// 수수료: Card 2%, Cash 0%, Point 1%
```
<details><summary>힌트</summary>sealed라서 세 case를 모두 처리하면 default가 필요 없다. record 분해 패턴을 써 보라.</details>
<details><summary>해설</summary>

```java
double fee = switch (pay) {
    case Card(int amount)  -> amount * 0.02;
    case Cash(int amount)  -> 0;
    case Point(int amount) -> amount * 0.01;
};
```
세 하위 타입을 모두 다뤘으므로 컴파일러가 완전성을 인정한다. 하나라도 빼면 컴파일 에러다.
</details>

**문제 4.** guarded pattern(`when`)을 사용해, `Object`를 받아 "0", "양수", "음수", "정수 아님"을 구분하는 메서드를 작성하라.
<details><summary>힌트</summary>더 구체적인 조건(when)을 먼저, 일반 조건을 나중에 배치하라.</details>
<details><summary>해설</summary>

```java
String classify(Object obj) {
    return switch (obj) {
        case Integer i when i == 0 -> "0";
        case Integer i when i > 0  -> "양수";
        case Integer i             -> "음수";
        default                    -> "정수 아님";
    };
}
```
`case Integer i`를 맨 위에 두면 아래 when 케이스에 도달 불가라서 컴파일 에러가 난다.
</details>

**문제 5.** 다음 코드에는 fall-through 버그가 잠재해 있다. 이를 switch expression으로 안전하게 바꿔라.
```java
String type;
switch (code) {
    case 1: type = "일반"; break;
    case 2: type = "우수";   // break 누락!
    case 3: type = "VIP"; break;
    default: type = "미정";
}
```
<details><summary>힌트</summary>화살표 문법에는 break가 없다. code 2일 때 원래 의도는 "우수"였을 것이다.</details>
<details><summary>해설</summary>

```java
String type = switch (code) {
    case 1 -> "일반";
    case 2 -> "우수";
    case 3 -> "VIP";
    default -> "미정";
};
```
화살표 문법은 fall-through가 없으므로, code가 2면 정확히 "우수"만 나온다. 원래 코드는 break 누락으로 2일 때 "VIP"가 나오는 버그였다.
</details>

- `record`는 불변 데이터 캐리어로 접근자/equals/hashCode/toString을 자동 생성하며, 컴팩트 생성자로 유효성 검증을 할 수 있다(단, 상속 불가·인스턴스 필드 추가 불가).
- `sealed`는 하위 타입을 제한해 안정성을 확보하고, record·switch와 결합해 완전성이 보장되는 안전한 분기(대수적 데이터 타입)를 가능하게 한다.
- Text Block, `var`, pattern matching(record 분해·`when` 가드), switch expression은 코드를 짧고 안전하게 만들며, 특히 switch expression은 fall-through 버그를 원천 봉쇄한다.
- 이 기능들은 따로 노는 게 아니라 "데이터를 안전하게 다루는 하나의 현대적 방식"으로 서로 맞물린다.
