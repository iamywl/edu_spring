# Chapter 01. 자바 시작하기

> 모든 프로그래밍 학습은 "내가 쓴 코드가 실제로 돌아가는 것"을 눈으로 확인하는 데서 시작한다.
> 이 장에서는 자바라는 언어가 무엇이고 어떻게 실행되는지 이해한 뒤,
> **JDK나 이클립스를 직접 설치하지 않고 Docker 컨테이너 하나로** 첫 자바 프로그램을 실행해 본다.

**이 장에서 배우는 것**

- 프로그래밍 언어로서 자바의 특징과 "한 번 작성하면 어디서나 실행"(WORA) 철학 (1.1)
- Docker 컨테이너로 JDK 21 실행 환경을 띄우는 방법 — 로컬 설치 불필요 (1.2)
- 환경 변수(PATH / JAVA_HOME)의 개념과, 컨테이너 안에서는 왜 신경 쓸 필요가 없는지 (1.3)
- 소스 코드가 바이트코드를 거쳐 JVM 위에서 실행되는 과정, JDK/JRE/JVM의 구분 (1.4)
- `compile.sh` → `run.sh`로 이어지는 실제 실행 흐름 (1.5)
- 이클립스 대신 VS Code Dev Containers로 컨테이너에 붙어 개발하는 방법 (1.6 ~ 1.8)
- 클래스·메서드·문장·블록 등 코드를 읽기 위한 기본 용어 (1.9)
- 주석의 세 가지 형태와 Javadoc (1.10), 실행문과 세미콜론 (1.11)

> 이 교재는 원서의 목차를 따르되, "OS별 JDK 설치 → 이클립스 설정"으로 이어지는 원서의 환경 구축 부분을
> **Docker 판**으로 각색했다. 설치에 시간을 쏟는 대신, 어느 컴퓨터에서든 동일하게 동작하는
> 컨테이너 환경에서 바로 코드를 실행하는 데 집중한다.

## 1.1 프로그래밍 언어와 자바

컴퓨터는 0과 1로 된 기계어만 이해한다. 사람이 기계어를 직접 쓰는 것은 사실상 불가능하므로, 사람이 읽고 쓰기 좋은 **프로그래밍 언어**로 코드를 작성한 뒤 이를 기계가 이해할 수 있는 형태로 바꾼다. 자바(Java)는 그중에서도 가장 널리 쓰이는 언어 중 하나로, 1995년 썬 마이크로시스템즈(현재 오라클)가 발표했다.

자바의 대표적인 특징은 다음과 같다.

- **객체지향 언어(OOP)**: 데이터와 기능을 "객체"라는 단위로 묶어 설계한다. (Chapter 02에서 본격적으로 다룬다.)
- **자동 메모리 관리(가비지 컬렉션)**: 사용이 끝난 메모리를 JVM이 알아서 회수해 준다. C/C++처럼 개발자가 직접 메모리를 해제할 필요가 없다.
- **풍부한 표준 라이브러리와 생태계**: 문자열·컬렉션·입출력·네트워크 등 방대한 기능이 기본 제공되며, Spring 같은 거대한 프레임워크 생태계를 갖췄다.
- **플랫폼 독립성(WORA)**: 아래에서 설명한다.

### WORA — Write Once, Run Anywhere

자바의 가장 강력한 슬로건은 **"한 번 작성하면 어디서나 실행된다"(Write Once, Run Anywhere)** 이다. 보통의 언어는 윈도우용, 맥용, 리눅스용으로 각각 컴파일해야 하지만, 자바는 소스 코드를 **바이트코드(bytecode)** 라는 중간 형태로 컴파일한다. 이 바이트코드는 특정 OS에 종속되지 않고, 각 OS에 맞게 설치된 **JVM(Java Virtual Machine)** 이 그 위에서 바이트코드를 실행한다.

```
   소스코드(.java)  ──컴파일──▶  바이트코드(.class)  ──▶  JVM(Windows)  ──▶ 실행
                                        │              JVM(macOS)     ──▶ 실행
                                        └──────────────▶ JVM(Linux)   ──▶ 실행
```

즉, 바이트코드는 하나만 만들면 되고, "OS별 차이"는 JVM이 흡수한다. 이것이 자바가 서버·안드로이드·데스크톱을 아우르며 오래 살아남은 핵심 이유다.

> 흥미롭게도 우리가 이 교재에서 쓰는 **Docker** 역시 "어디서나 동일하게 실행"을 지향한다.
> 자바는 JVM 위에서, Docker는 컨테이너 위에서 이식성을 달성한다. 이 둘을 합치면
> "어떤 컴퓨터에서든, JDK를 따로 깔지 않고도, 똑같은 자바 실행 환경"을 얻는다.

## 1.2 개발 환경 — Docker로 JDK 21 컨테이너 실행

원서에서는 이 자리에서 OS별로 JDK를 내려받아 설치하고 환경 변수를 잡는 과정을 설명한다. 하지만 **이 교재는 그 과정을 통째로 생략한다.** JDK 21이 이미 포함된 Docker 컨테이너를 띄우기만 하면 되기 때문이다.

> **로컬에 자바를 설치할 필요가 없다.**
> 필요한 것은 **Docker Desktop 하나뿐**이다. JDK 21은 컨테이너 이미지 안에 들어 있다.

### 준비물

- **Docker & Docker Compose** (필수 — 이것만 있으면 된다)
- 코드를 읽고 수정할 IDE (VS Code 권장)

Docker Desktop 설치가 아직이라면, 화면 캡처와 함께 처음부터 끝까지 안내하는
[맥북 Docker 실습 가이드](../맥북_Docker_실습가이드.md)를 먼저 따라 하자.

### 컨테이너 띄우고 첫 실행하기

프로젝트 최상위 디렉터리에서 다음 두 명령을 실행한다.

```bash
# 1. JDK 21이 포함된 컨테이너를 백그라운드로 띄운다 (최초 1회)
docker compose up -d

# 2. 컨테이너 안으로 들어가 예제 실행기를 띄운다
docker exec -it java-edu ./run.sh
```

`docker compose up -d`는 프로젝트 루트의 `docker-compose.yml`을 읽어 `java-edu`라는 이름의 컨테이너를 만든다. 이 컨테이너 안에는 JDK 21이 설치되어 있고, Chapter 01~03과 CS 트랙, 그리고 이 교재의 실습 코드가 볼륨으로 연결되어 있다.

`docker exec -it java-edu ./run.sh`는 이미 떠 있는 컨테이너 안에서 `run.sh`(예제 실행기)를 실행한다. 인자 없이 실행하면 **대화형 계층 메뉴**(카테고리 선택 → 개념 선택)가 뜨고, 클래스 이름을 주면 해당 예제가 바로 실행된다.

```bash
# 대화형 메뉴 대신, 특정 예제를 바로 실행
docker exec -it java-edu ./run.sh VariablesAndTypes
```

실습을 마쳤으면 컨테이너를 정리한다.

```bash
docker compose down
```

> 소스 코드가 볼륨으로 마운트되어 있으므로, 호스트(맥북)에서 파일을 수정하면 컨테이너에 즉시 반영된다.
> 이미지를 다시 빌드할 필요 없이 `./compile.sh`만 실행하면 된다. (자세한 흐름은 1.5에서.)

## 1.3 환경 변수 — 컨테이너 안엔 이미 설정되어 있다

원서는 이 자리에서 `PATH`와 `JAVA_HOME` 환경 변수를 손수 설정하는 방법을 설명한다. **우리는 이 작업을 할 필요가 없다.** 컨테이너 이미지 안에 이미 설정되어 있기 때문이다. 다만 "왜 이런 설정이 필요한가"라는 개념은 알아 둘 가치가 있다.

### PATH — 명령어를 어디서 찾을지 알려주는 지도

터미널에서 `java`나 `javac`를 입력하면, 셸은 그 실행 파일이 실제로 어디에 있는지 **`PATH` 환경 변수에 나열된 디렉터리들**을 뒤져서 찾는다. JDK를 설치한 뒤 그 `bin` 폴더를 `PATH`에 추가하지 않으면, "command not found: javac" 같은 오류가 난다. 원서에서 설치 후 `PATH`를 잡으라고 하는 이유가 이것이다.

### JAVA_HOME — "자바가 설치된 곳"을 가리키는 이정표

`JAVA_HOME`은 JDK가 설치된 최상위 경로를 담는 환경 변수다. Gradle, Maven 같은 빌드 도구나 Spring, 톰캣 같은 프로그램은 "어떤 자바를 쓸지"를 이 변수로 판단한다. 컴퓨터에 여러 버전의 자바가 깔려 있을 때 특히 중요하다.

### 컨테이너에서는?

컨테이너 이미지(`eclipse-temurin:21-jdk` 기반)에는 JDK 21이 표준 위치에 설치되어 있고, `java`·`javac`가 이미 `PATH`에 잡혀 있다. 직접 확인해 볼 수 있다.

```bash
docker exec -it java-edu java -version
# openjdk version "21..." 처럼 버전이 출력되면 정상
```

즉, **"자바가 설치된 위치를 시스템에 알려 주는" 번거로운 작업을 컨테이너가 대신 끝내 놓은 상태**다. 이것이 로컬 설치 대비 Docker 방식의 큰 장점이다 — "내 컴퓨터에선 PATH가 꼬여서 안 돼요" 같은 문제가 원천적으로 사라진다.

## 1.4 바이트코드와 JVM

자바 프로그램이 실행되는 과정을 조금 더 자세히 들여다보자. 핵심은 **소스 → 컴파일 → 실행**의 두 단계다.

```
  Hello.java          javac          Hello.class          java
 (사람이 읽는  ) ──컴파일──▶ (JVM이 읽는  ) ──실행──▶  JVM ──▶ 결과 출력
 (  소스코드   )              ( 바이트코드 )
```

1. 개발자가 `.java` 소스 파일을 작성한다.
2. **`javac`(자바 컴파일러)** 가 소스를 **`.class` 바이트코드**로 컴파일한다. 이 바이트코드는 특정 CPU/OS의 기계어가 아니라, JVM이라는 가상 기계를 위한 명령어다.
3. **`java`** 명령으로 JVM을 띄우면, JVM이 `.class`를 읽어 그 자리(각 OS)에 맞는 기계어로 바꿔 가며 실행한다.

이 "한 단계 더 거치는" 구조 덕분에 1.1에서 말한 WORA가 가능해진다.

### JDK / JRE / JVM 구분

세 약어가 자주 혼동되므로 정리한다. 포함 관계는 **JDK ⊃ JRE ⊃ JVM** 이다.

```
┌──────────────────────────── JDK (개발 키트) ────────────────────────────┐
│   javac(컴파일러), jshell, javadoc 등 개발 도구                          │
│  ┌──────────────────────── JRE (실행 환경) ────────────────────────┐    │
│  │   표준 라이브러리 (String, List, ...)                            │    │
│  │  ┌──────────────────── JVM (가상 머신) ────────────────────┐    │    │
│  │  │   바이트코드를 실제로 실행하는 엔진                       │    │    │
│  │  └────────────────────────────────────────────────────────┘    │    │
│  └────────────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────────────┘
```

| 약어 | 이름 | 역할 | 담고 있는 것 |
|------|------|------|--------------|
| **JVM** | Java Virtual Machine | 바이트코드를 실행 | 실행 엔진 |
| **JRE** | Java Runtime Environment | 자바 프로그램을 **실행**하는 데 필요한 최소 세트 | JVM + 표준 라이브러리 |
| **JDK** | Java Development Kit | 자바 프로그램을 **개발**(컴파일 포함)하는 전체 세트 | JRE + `javac` 등 개발 도구 |

코드를 작성하고 컴파일하려면 `javac`가 필요하므로 **개발에는 JDK**가 있어야 한다. 우리 컨테이너에 `21-jdk` 이미지를 쓴 이유가 이것이다. (참고로 Spring Boot 챕터의 최종 실행 이미지는 컴파일이 끝난 JAR만 돌리면 되므로 더 가벼운 `21-jre`를 쓴다.)

## 1.5 소스 작성부터 실행까지

이제 실제 실행 흐름을 정리하자. 이 프로젝트에서는 두 개의 스크립트가 핵심이다.

- **`compile.sh`** — `/app/src` 아래의 모든 `.java`를 찾아 `javac`로 컴파일하고 결과를 `/app/out`에 넣는다.
- **`run.sh`** — 컴파일된 클래스를 `java` 명령으로 실행한다. (필요하면 자동으로 재컴파일도 한다.)

### 컨테이너 안에서의 실행 흐름

```
 [호스트에서 코드 수정]        [컨테이너 안]
   Hello.java 저장   ──▶  ./compile.sh  ──▶  ./run.sh
   (볼륨 마운트로              (javac)          (java)
    즉시 반영)              .java → .class    .class 실행 → 출력
```

컨테이너 안에서 직접 실행하면 다음과 같다.

```bash
# 컨테이너 안 터미널에서
./compile.sh                    # 소스 전체 컴파일
./run.sh                        # 대화형 메뉴 (카테고리 → 개념 선택)
./run.sh VariablesAndTypes      # 특정 개념을 이름으로 바로 실행
./run.sh all                    # 전체 예제 실행
```

호스트(맥북)에서 한 줄로 실행하고 싶다면 `docker exec`를 앞에 붙인다.

```bash
docker exec -it java-edu ./compile.sh
docker exec -it java-edu ./run.sh VariablesAndTypes
```

`compile.sh`가 하는 일은 결국 1.4에서 본 `javac`, `run.sh`가 하는 일은 `java` 명령을 우리 대신 편하게 감싸 준 것이다. 즉 이 두 스크립트를 쓰는 것은 곧 "소스 → 바이트코드 → JVM 실행"을 손으로 하는 것과 같다.

> 코드를 수정한 뒤에는 **저장 → `./compile.sh` → `./run.sh`** 순서만 기억하면 된다.
> 볼륨 마운트 덕분에 이미지를 다시 빌드(`docker build`)할 필요가 없다.

## 1.6 개발 도구 — 이클립스 대신 VS Code

원서의 이 대목(1.6~1.8)은 통합 개발 환경(IDE)으로 **이클립스**를 설치·설정하고, 프로젝트를 만들고, 코드를 실행하는 방법을 다룬다. 이 교재에서는 이클립스 대신 **VS Code**를 컨테이너에 붙여 사용한다.

IDE는 "코드 편집기 + 컴파일/실행 + 디버깅 + 자동완성"을 한 곳에 모은 도구다. 우리는 실행 환경(JDK)을 컨테이너가 이미 제공하므로, IDE는 **코드를 읽고·쓰고, 컨테이너 안에서 실행 버튼을 누르는** 역할만 하면 된다.

### 왜 이클립스가 아니라 VS Code인가

- 컨테이너에 "붙어서(attach)" 작업하는 워크플로가 매끄럽다 (Dev Containers 확장).
- 이 프로젝트는 `.devcontainer/` 설정을 포함하고 있어, VS Code가 자바 확장 팩을 자동으로 설치하고 접속 즉시 컴파일까지 수행하도록 구성돼 있다.

## 1.7 VS Code로 실행 중인 컨테이너에 접속하기 (Dev Containers Attach)

이 프로젝트의 권장 개발 방식이다. 컨테이너를 로컬 폴더처럼 열어, 그 안에서 파일을 수정하고 터미널로 바로 실행한다.

1. 먼저 컨테이너를 띄운다 (1.2와 동일).

   ```bash
   docker compose up -d
   ```

2. VS Code에서 **Dev Containers 확장**을 설치한다. (확장 탭에서 "Dev Containers" 검색)

3. `F1`(또는 `Cmd+Shift+P`)을 눌러 명령 팔레트를 열고
   **"Dev Containers: Attach to Running Container"** 를 선택한 뒤, 목록에서 **`java-edu`** 를 고른다.

4. 새 VS Code 창이 컨테이너 내부(`/app`)를 연다. 터미널을 열고( `` Ctrl+` `` ) 바로 실행한다.

   ```bash
   ./compile.sh
   ./run.sh                        # 대화형 메뉴
   ./run.sh VariablesAndTypes      # Ch01 - 변수와 타입
   ```

> 이 프로젝트는 `.devcontainer/devcontainer.json`에 자바 확장 팩(`vscode-java-pack` 등)과
> `postStartCommand`(접속 시 자동 컴파일)를 정의해 두었다. 그래서 접속만 하면 개발 준비가 끝난다.

## 1.8 코드 수정하고 결과 확인하기

VS Code로 컨테이너에 붙었다면, 이제 원서에서 이클립스로 하던 "편집 → 실행" 사이클을 그대로 할 수 있다.

1. 왼쪽 탐색기에서 소스 파일(예: `src/com/edu/basics/VariablesAndTypes.java`)을 연다.
2. 코드를 수정하고 저장한다(`Cmd+S`). 볼륨 마운트로 컨테이너에 즉시 반영된다.
3. 터미널에서 `./compile.sh`로 다시 컴파일한 뒤 `./run.sh`로 실행해 변화를 확인한다.

```
  편집(VS Code)  ──▶  저장  ──▶  ./compile.sh  ──▶  ./run.sh  ──▶  결과 확인
        ▲                                                             │
        └─────────────────────  다시 수정  ◀──────────────────────────┘
```

이 짧은 순환을 반복하며 "코드를 바꾸면 결과가 어떻게 달라지는가"를 몸으로 익히는 것이, 이 교재 전체를 관통하는 학습법이다.

## 1.9 코드 용어 이해

앞으로 예제를 읽으려면 몇 가지 기본 용어를 알아야 한다. 가장 단순한 자바 프로그램을 예로 들자.

```java
public class Hello {                       // ← 클래스(class)
    public static void main(String[] args) {   // ← 메서드(method)
        System.out.println("Hello, Java!");     // ← 문장(statement)
    }
}
```

- **클래스(class)**: 자바 코드의 가장 큰 담는 단위. 모든 코드는 클래스 안에 들어간다. 소스 파일 이름(`Hello.java`)은 보통 그 안의 공개 클래스 이름(`Hello`)과 일치해야 한다.
- **메서드(method)**: 특정 일을 수행하는 코드 묶음. `main`은 프로그램이 시작될 때 JVM이 가장 먼저 호출하는 **진입점(entry point)** 메서드다. 모든 실행 예제에는 `main`이 있다.
- **문장(statement)**: 하나의 실행 단위. 위에서 `System.out.println(...)`이 한 문장이며, 끝에 세미콜론(`;`)이 붙는다. (1.11 참고)
- **블록(block)**: 중괄호 `{ }`로 감싼 코드 영역. 클래스의 몸통, 메서드의 몸통이 모두 블록이다. 블록은 코드의 "범위"를 정한다.

```
 클래스 블록 { ... 메서드 블록 { ... 문장; 문장; ... } ... }
```

이 관계 — "클래스 안에 메서드, 메서드 안에 문장" — 만 잡아 두면 이후 모든 예제 구조가 눈에 들어온다.

## 1.10 주석

**주석(comment)** 은 컴파일러가 무시하는, 사람을 위한 설명이다. 코드의 의도를 남기거나, 잠시 코드를 실행에서 제외할 때 쓴다. 자바에는 세 가지 형태가 있다.

```java
// 한 줄 주석: 이 기호부터 줄 끝까지 무시된다.

/*
   여러 줄 주석: 이 기호 사이의 모든 내용이 무시된다.
   여러 줄에 걸쳐 설명을 쓸 수 있다.
*/

/**
 * 문서화 주석(Javadoc): /** 로 시작한다.
 * javadoc 도구가 이 주석을 읽어 HTML API 문서를 자동 생성한다.
 * @param name 인사할 대상의 이름
 * @return 인사말 문자열
 */
public String greet(String name) {
    return "Hello, " + name;
}
```

| 형태 | 기호 | 용도 |
|------|------|------|
| 한 줄 주석 | `//` | 짧은 설명, 한 줄 코드 비활성화 |
| 여러 줄 주석 | `/* */` | 여러 줄에 걸친 설명, 여러 줄 비활성화 |
| 문서화 주석 | `/** */` | 클래스·메서드의 공식 설명(Javadoc 문서 생성) |

> 주석은 "무엇을 하는지"보다 **"왜 이렇게 했는지"** 를 남길 때 가장 가치가 있다.
> 코드만 봐도 아는 내용을 굳이 반복하는 주석은 오히려 방해가 된다.

## 1.11 실행문과 세미콜론

자바에서 **실행문(statement)** 은 하나의 명령을 나타내며, 반드시 **세미콜론(`;`)** 으로 끝난다. 세미콜론은 "이 문장이 여기서 끝났다"는 마침표 역할을 한다.

```java
int x = 10;                         // 대입문
System.out.println(x);              // 메서드 호출문
x = x + 1;                          // 다시 대입
```

줄 바꿈이 문장의 끝을 뜻하는 파이썬 같은 언어와 달리, 자바는 **세미콜론**으로 문장을 구분한다. 그래서 한 문장을 여러 줄에 걸쳐 써도 되고, 세미콜론만 정확하면 한 줄에 여러 문장을 쓸 수도 있다.

```java
int a = 1; int b = 2;               // 한 줄에 두 문장 (권장하지는 않음)

int total =                          // 한 문장을 여러 줄로
    a + b;
```

초보자가 가장 자주 만나는 컴파일 오류가 바로 **세미콜론 누락**이다. `';' expected` 같은 오류 메시지가 나오면 바로 앞 줄 끝에 세미콜론이 빠지지 않았는지 확인하자.

한편, 블록을 여는 중괄호 `{ }` 뒤에는 세미콜론을 붙이지 않는다. (`if`문, 메서드, 클래스 선언 등)

```java
if (x > 0) {        // 여기엔 세미콜론 없음
    x = 0;          // 블록 안 문장엔 세미콜론
}                   // 닫는 중괄호 뒤에도 세미콜론 없음
```

## 확인문제

1. 자바의 슬로건 WORA(Write Once, Run Anywhere)를 가능하게 하는 핵심 요소는 무엇인가? 소스 코드가 실행되기까지의 과정을 바이트코드와 JVM을 포함해 설명하라.

2. 이 교재에서는 로컬에 JDK를 설치하지 않는다. 대신 어떤 두 명령으로 JDK 21 실행 환경을 띄우고 예제 실행기를 시작하는가? 순서대로 쓰라.

3. `PATH`와 `JAVA_HOME` 환경 변수는 각각 무슨 역할을 하는가? 그리고 이 프로젝트의 컨테이너에서는 왜 이 값들을 직접 설정하지 않아도 되는가?

4. JDK, JRE, JVM의 포함 관계를 부등호로 나타내고, 자바 코드를 "컴파일"하려면 이 중 무엇이 반드시 필요한지, 그 이유와 함께 답하라.

5. 다음 코드에서 클래스·메서드·문장·블록에 해당하는 부분을 각각 하나씩 지목하고, 컴파일 오류가 나는 부분을 찾아 고쳐라.

   ```java
   public class Test {
       public static void main(String[] args) {
           int n = 5
           System.out.println(n)
       }
   }
   ```

## 요약

- **자바**는 객체지향·자동 메모리 관리·풍부한 생태계를 갖춘 언어이며, 핵심 철학은 **WORA(한 번 작성하면 어디서나 실행)** 이다. 소스를 **바이트코드(.class)** 로 컴파일하고, 각 OS의 **JVM**이 그 위에서 실행하는 구조 덕분에 플랫폼 독립성이 확보된다.
- 이 교재는 원서의 "OS별 JDK 설치 + 이클립스 설정"을 **Docker 판**으로 각색한다. `docker compose up -d`로 JDK 21 컨테이너(`java-edu`)를 띄우고, `docker exec -it java-edu ./run.sh`로 예제를 실행한다. **로컬에 자바를 설치할 필요가 없다.**
- `PATH`(명령어를 찾는 경로)와 `JAVA_HOME`(자바 설치 위치)은 로컬 설치 시 신경 써야 할 환경 변수지만, **컨테이너에는 이미 설정되어 있어** 개발자가 손댈 필요가 없다.
- 실행 과정은 **소스(.java) → `javac` → 바이트코드(.class) → `java`(JVM) → 결과** 이며, **JDK ⊃ JRE ⊃ JVM** 관계가 성립한다. 컴파일러(`javac`)가 필요하므로 개발에는 **JDK**를 쓴다.
- 실제 실행은 **`./compile.sh`(컴파일) → `./run.sh`(실행)** 두 스크립트로 이뤄진다. 코드를 수정하면 볼륨 마운트로 즉시 반영되므로 재빌드가 필요 없다.
- IDE는 이클립스 대신 **VS Code**를 쓰며, **Dev Containers: Attach to Running Container**로 `java-edu` 컨테이너에 붙어 편집·실행·디버깅한다.
- 코드는 **클래스 안에 메서드, 메서드 안에 문장**으로 구성되고, 문장은 **세미콜론(`;`)** 으로 끝난다. 주석은 `//`, `/* */`, 그리고 문서를 생성하는 **Javadoc(`/** */`)** 세 가지가 있다.
