# Chapter 03 · 연산자

프로그램은 결국 **데이터를 꺼내 계산하고, 그 결과를 다시 저장하는 일**의 반복이다. 변수(Chapter 02)가 값을 담는 그릇이라면, 연산자(operator)는 그 값을 **더하고, 비교하고, 뒤집고, 옮기는 도구**다.

연산자는 언뜻 초등학교 산수처럼 보인다. `+`, `-`, `*`, `/`는 우리가 아는 그것이다. 하지만 컴퓨터의 연산에는 **학교에서 배우지 않은 규칙**이 숨어 있다.

- `17 / 5`는 왜 `3.4`가 아니라 `3`인가?
- `Integer.MAX_VALUE + 1`은 왜 예외가 아니라 **가장 작은 음수**가 되는가?
- `0.1 + 0.2`는 왜 `0.3`이 아닌가?
- `"hello" == "hello"`는 `true`인데, 왜 어떤 문자열 비교는 `false`가 되는가?

이 장은 자바가 제공하는 연산자들을 **부호/증감 → 산술 → 오버플로우 → 정확한 계산 → 특수값 → 비교 → 논리 → 비트 → 이동 → 대입 → 삼항 → 우선순위** 순서로 하나씩 다룬다. 각 소절은 실행 가능한 데모 클래스와 짝을 이루므로, 개념을 읽은 뒤 반드시 `./run.sh`로 직접 눈으로 확인하자.

> 이 장은 "무엇을(문법)"에 집중한다. "왜 그렇게 동작하는가(원리)"의 깊은 이야기(2의 보수, IEEE-754의 비트 구조, String Pool 등)는 [JAVA 개념서 2장 『데이터를 담는 그릇 - 변수』](../JAVA_개념서/02-데이터를-담는-그릇-변수.md)에서 상세히 다룬다. 필요할 때마다 상호참조로 안내한다.

## 3.1 부호/증감 연산자

**단항 연산자(unary operator)** 는 피연산자 하나에만 작용한다. 부호 연산자 `+`, `-`와 증감 연산자 `++`, `--`가 여기에 속한다.

- 단항 `+`는 **아무 일도 하지 않는다.** 값을 그대로 둔다.
- 단항 `-`는 **부호를 뒤집는다.** `-a`는 `0 - a`와 같은 의미다.
- `++`는 값을 1 증가, `--`는 값을 1 감소시킨다.

증감 연산자의 진짜 핵심은 **위치**다. 변수 앞에 붙는 **전위(prefix)** 와 뒤에 붙는 **후위(postfix)** 는 "언제 증가하는가"가 다르다.

```
전위 ++x  :  ① 먼저 1 증가  →  ② 증가된 값을 식에 사용
후위 x++  :  ① 먼저 현재 값을 식에 사용  →  ② 그 다음 1 증가

  int x = 5;  int r = ++x;   // r = 6, x = 6   (증가 후 사용)
  int y = 5;  int r = y++;   // r = 5, y = 6   (사용 후 증가)
```

변수를 **단독으로** 쓸 때(`i++;`처럼 한 줄에 홀로)는 전위/후위 결과가 같다. 하지만 **다른 식과 섞이면** 결과가 완전히 달라진다.

```java
int i = 3;
int result = i++ + i++;   // (3) + (4) = 7,  이후 i = 5
// 첫 i++는 3을 내놓고 i=4, 둘째 i++는 4를 내놓고 i=5
```

> 💻 실습: `./run.sh SignAndIncrement`

## 3.2 산술 연산자

이항 산술 연산자 `+`, `-`, `*`, `/`, `%`(나머지)는 두 값을 계산한다. 여기에는 초보자가 반드시 넘어야 할 **세 가지 함정**이 있다.

**① 정수 나눗셈은 소수부를 버린다.** 두 피연산자가 모두 `int`면 결과 타입도 `int`이므로 소수부가 존재할 수 없다. 반올림이 아니라 **0 방향으로 절삭**한다.

```
17 / 5   = 3      (3.4가 아님)
-17 / 5  = -3     (0 방향 절삭이므로 -4가 아님)
```

**② 나머지 `%`의 부호는 왼쪽(피제수)을 따른다.** 오른쪽 값의 부호는 무관하다. 항상 `a == (a/b)*b + (a%b)` 항등식이 성립한다.

```
-17 % 5  = -2     (부호는 왼쪽 -17을 따름)
17 % -5  = 2      (오른쪽 부호는 무관)
```

**③ `+`는 문자열이 끼면 '연결(concatenation)'로 의미가 바뀐다.** 그리고 왼쪽부터 계산되므로 위치에 따라 결과가 달라진다.

```
1 + 2 + "cm"   →  "3cm"     (1+2=3 먼저, 그 뒤 문자열 연결)
"cm" + 1 + 2   →  "cm12"    (문자열이 먼저라 1,2가 각각 연결)
```

한쪽이라도 `double`이면 실수 나눗셈이 되어 소수부가 보존된다.

```java
System.out.println(17 / 5);      // 3
System.out.println(17 / 5.0);    // 3.4
System.out.println((17 / 5) * 1.0); // 3.0  ← 정수나눗셈이 먼저라 3이 된 뒤 3.0
```

> 💻 실습: `./run.sh Arithmetic`

> 정수 나눗셈이 왜 절삭인지, 형변환이 언제 일어나는지는 [개념서 §2.4 형변환](../JAVA_개념서/02-데이터를-담는-그릇-변수.md#24-형변환이-필요한-이유)을 참고하라.

## 3.3 오버플로우·언더플로우

`int`는 **32비트 고정 크기**다. 그래서 표현할 수 있는 범위가 정해져 있다(약 -21억 ~ +21억). 이 범위를 넘어가면 어떻게 될까?

자바는 **예외를 던지지 않는다.** 대신 값이 시계처럼 **조용히 순환(wraparound)** 한다. 이것이 오버플로우가 가장 위험한 이유다 — 눈치채기 어렵다.

```
Integer.MAX_VALUE = 2147483647
        + 1
     ─────────────
Integer.MIN_VALUE = -2147483648   ← 가장 큰 수 + 1 = 가장 작은 수!
```

왜 이런 일이 벌어질까? **2의 보수** 표현 때문이다. `MAX_VALUE`의 비트는 `0111...1`인데, 여기에 1을 더하면 `1000...0`이 되고, 2의 보수에서 최상위 비트 1은 "가장 작은 음수"를 뜻한다.

```
  0111 1111 ... 1111   (MAX_VALUE)
+ 0000 0000 ... 0001
─────────────────────
  1000 0000 ... 0000   (MIN_VALUE)  ← 최상위 비트가 부호로 뒤집힘
```

곱셈에서도 조용히 넘칠 수 있다. 해결책은 두 가지다.

```java
int big = 100_000;
System.out.println(big * big);         // 1410065408  ← 오버플로우! 엉뚱한 값
System.out.println((long) big * big);  // 10000000000 ← 한쪽을 long으로 승격

Math.multiplyExact(big, big);          // 오버플로우 시 ArithmeticException을 던진다
```

(참고) 부동소수에서는 반대로 **언더플로우**가 있다. `Double.MIN_VALUE`보다 작아지면 값이 `0.0`으로 가라앉는다.

> 💻 실습: `./run.sh OverflowUnderflow`

> 2의 보수가 왜 이렇게 설계되었고, `(byte)300`이 왜 `44`가 되는지는 [개념서 §2.5 정수의 진짜 얼굴: 2의 보수](../JAVA_개념서/02-데이터를-담는-그릇-변수.md#25-정수의-진짜-얼굴-2의-보수)에서 깊이 다룬다.

## 3.4 정확한 계산은 정수 연산으로

`0.1 + 0.2`의 결과는 `0.3`이 **아니다.** `0.30000000000000004`다.

```java
double sum = 0.1 + 0.2;
System.out.println(sum);          // 0.30000000000000004
System.out.println(sum == 0.3);   // false !
```

이것은 버그가 아니다. `double`은 10진 소수를 **2진 분수로 근사할 뿐**이라, `0.1`이나 `0.2`처럼 2진수로 딱 떨어지지 않는 수는 미세한 오차를 품는다. 그 오차들이 더해진 것이다.

**결론: 돈이나 정확한 수량 계산에는 절대 `double`을 쓰지 마라.** 두 가지 대안이 있다.

**① 최소 단위 정수로 다루기 (권장).** 금액을 '원'이 아니라 '전(1/100원)' 같은 최소 단위 정수로 바꿔 계산하면 오차가 원천 차단된다.

```java
long priceCents = 120050;   // 1200.50원을 '전' 단위 정수로
long taxCents   = 35025;    // 350.25원
long total = priceCents + taxCents;   // 155075전 = 1550.75원, 오차 없음
```

**② `BigDecimal` 사용.** 단, **반드시 문자열 생성자**를 써야 한다. `double`을 넘기면 그 오차가 그대로 딸려 들어온다.

```java
new BigDecimal("0.1").add(new BigDecimal("0.2"));  // 0.3  정확!
new BigDecimal(0.1);    // 0.1000000000000000055... ← double 오차가 그대로!
new BigDecimal("0.1");  // 0.1                       ← 문자열이면 정확
```

> 💻 실습: `./run.sh PreciseCalculation`

> `0.1`이 왜 2진수로 무한소수가 되는지, IEEE-754 비트 구조는 [개념서 §2.6 실수의 함정](../JAVA_개념서/02-데이터를-담는-그릇-변수.md#26-실수의-함정-왜-01--02--03)에서 그림과 함께 설명한다.

## 3.5 NaN·Infinity

나눗셈에서 0으로 나누면 어떻게 될까? **정수와 실수가 완전히 다르게** 동작한다.

```
정수  10 / 0     →  ArithmeticException (예외 발생!)
실수  1.0 / 0.0  →  Infinity (예외 없음)
실수  0.0 / 0.0  →  NaN (Not a Number)
```

정수의 세계에는 '무한대'가 없어서 예외로 처리한다. 하지만 실수(IEEE-754)에는 무한대(`Infinity`)와 "숫자가 아님"(`NaN`)이라는 특수값이 정의되어 있어, 예외 없이 이런 값을 만들어낸다.

`NaN`에는 아주 이상한 성질이 있다. **자기 자신과도 `==`가 `false`다.**

```java
double nan = 0.0 / 0.0;
System.out.println(nan == nan);   // false !  (역설적으로 nan != nan 은 true)
```

그래서 `NaN`은 `==`로 검사하면 안 된다. 반드시 전용 메서드를 쓴다.

```java
Double.isNaN(0.0 / 0.0);       // true
Double.isInfinite(1.0 / 0.0);  // true
```

> 💻 실습: `./run.sh NanAndInfinity`

## 3.6 비교 연산자

비교 연산자 `==`, `!=`, `<`, `>`, `<=`, `>=`의 결과는 **항상 `boolean`**(true/false)이다. 기본 사용은 직관적이지만, 두 가지 함정이 있다.

**① 실수는 `==`로 비교하면 안 된다.** §3.4에서 봤듯 오차 때문에 논리적으로 같아야 할 값이 다르게 나온다. **차이의 절댓값이 허용오차(epsilon)보다 작은가**로 판단한다.

```java
double x = 0.1 + 0.2;
System.out.println(x == 0.3);                    // false  (오차 때문)
System.out.println(Math.abs(x - 0.3) < 1e-9);    // true   (올바른 방법)
```

**② 문자열의 `==`는 '내용'이 아니라 '같은 객체인지'를 본다.** 내용 비교는 반드시 `.equals()`를 써야 한다.

```java
String a = "hello";
String b = "hello";              // 리터럴은 String Pool을 공유
String c = new String("hello");  // new는 새 객체를 강제로 만든다

a == b        // true   (리터럴은 같은 객체를 공유하기도 함)
a == c        // false  (new로 만든 c는 다른 객체)
a.equals(c)   // true   (내용 비교는 항상 true)
```

`==`는 "같은 물건인가(동일성)"를, `.equals()`는 "같은 내용인가(동등성)"를 묻는다. 이 둘의 혼동은 자바에서 **가장 흔한 버그** 중 하나다.

> 💻 실습: `./run.sh Comparison`

> `==` vs `equals()`, String Pool의 원리는 [개념서 §2.2 기본형 vs 참조형](../JAVA_개념서/02-데이터를-담는-그릇-변수.md#22-기본형-vs-참조형-근본적-차이-가장-중요)에서 면접 답안까지 정리한다.

## 3.7 논리 연산자

논리 연산자는 `boolean` 값들을 조합한다. 논리곱 `&&`(AND), 논리합 `||`(OR), 부정 `!`(NOT)이 있다.

```
  A       B     A && B   A || B
─────────────────────────────────
true    true     true     true
true    false    false    true
false   true     false    true
false   false    false    false
```

`&&`와 `||`의 핵심은 **단축평가(short-circuit)** 다. 결과가 확정되면 오른쪽을 **아예 계산하지 않는다.**

```
&&  :  왼쪽이 false면  →  오른쪽 계산 안 하고 false 확정
||  :  왼쪽이 true면   →  오른쪽 계산 안 하고 true 확정
```

이 성질 덕분에 **null 방어 패턴**이 가능하다. 왼쪽에서 멈추므로 `NullPointerException`이 나지 않는다.

```java
String s = null;
boolean ok = (s != null) && (s.length() > 0);  // NPE 없이 안전
//            └ false면 여기서 멈춤. s.length()는 호출조차 안 됨
```

(참고) `&`와 `|`도 `boolean`에 쓸 수 있지만 **단축평가를 하지 않는다.** 항상 양쪽을 다 계산하므로, 오른쪽에 부작용(side effect)이 있으면 결과가 달라진다.

> 💻 실습: `./run.sh Logical`

## 3.8 비트 논리 연산자

비트 연산자는 정수를 **2진수로 펼쳐서 같은 자리끼리** 논리 연산을 한다. `&`(AND), `|`(OR), `^`(XOR), `~`(NOT)이 있다.

```
  a = 1100  (12)
  b = 1010  (10)
─────────────────
a & b = 1000  (8)    둘 다 1일 때만 1     → 마스킹, 비트 확인
a | b = 1110  (14)   하나라도 1이면 1     → 비트 켜기
a ^ b = 0110  (6)    서로 다르면 1        → 토글
~a           반전    모든 비트 뒤집기     → ~x == -x - 1
```

`^`(XOR)에는 유용한 성질이 있다. **같은 값으로 두 번 XOR하면 원래 값으로 돌아온다.**

```java
(a ^ b) ^ b   →  a   // b가 사라지고 a로 원복
```

`~`(NOT)는 2의 보수 때문에 `~x == -x - 1`이 성립한다. `~5`는 `-6`이다.

**실전 예: 권한 플래그.** 비트 하나하나를 on/off 스위치로 쓰는 기법이다.

```java
int READ = 0b100, WRITE = 0b010, EXEC = 0b001;
int perm = READ | WRITE;              // 읽기+쓰기 권한 부여 → 110
boolean canWrite = (perm & WRITE) != 0;   // true  (마스킹으로 확인)
boolean canExec  = (perm & EXEC)  != 0;   // false
```

> 💻 실습: `./run.sh BitwiseLogic`

## 3.9 비트 이동 연산자

시프트 연산자는 비트를 **좌우로 밀어낸다.** 세 종류가 있다.

```
x << n   왼쪽 시프트       = x * 2^n     오른쪽을 0으로 채움
x >> n   산술 오른쪽 시프트  = x / 2^n     '부호 비트'로 채움 (음수는 음수 유지)
x >>> n  논리 오른쪽 시프트                항상 '0'으로 채움
```

왼쪽 시프트는 곱하기, 오른쪽 시프트는 나누기와 같다.

```java
3 << 4    // 48   (3 * 16)
64 >> 3   // 8    (64 / 8)
```

`>>`와 `>>>`의 차이는 **음수에서 극명하게** 드러난다. `>>`는 부호 비트로 채워 음수를 유지하지만, `>>>`는 무조건 0으로 채워 **거대한 양수**가 된다.

```java
int neg = -8;             // 11111111 ... 11111000
neg >> 1     // -4        (부호 비트 1로 채움 → 여전히 음수)
neg >>> 1    // 2147483644  (0으로 채움 → 최상위가 0이 되어 거대한 양수!)
```

**결론: 음수를 오른쪽으로 밀 때는 `>>`와 `>>>`를 반드시 구분하라.**

> 💻 실습: `./run.sh BitShift`

## 3.10 대입 연산자

기본 대입 `=`는 오른쪽 값을 왼쪽 변수에 **저장**한다. 대입식 자체가 "저장된 값"을 결과로 내기 때문에 **연쇄 대입**이 가능하다.

```java
int a, b, c;
a = b = c = 5;   // 오른쪽부터: c=5, 그 값(5)이 b로, 다시 a로 → 모두 5
```

**복합 대입 연산자**(`+=`, `-=`, `*=`, `/=`, `%=`, `<<=`, `&=` ...)는 `a += b`처럼 연산과 대입을 합친 축약이다.

```java
int x = 20;
x += 5;   // 25    (x = x + 5)
x *= 2;   // 50
x <<= 2;  // 200   (비트 시프트도 복합 대입 가능)
```

그런데 `a += b`는 `a = a + b`의 **완전한 동의어가 아니다.** 복합 대입에는 **암묵적 형변환(자동 캐스트)이 숨어 있다.**

```java
byte b = 10;
// b = b + 1;   // 컴파일 에러! (b + 1은 int라서 byte에 담으려면 명시적 캐스트 필요)
b += 1;         // OK! += 가 결과를 byte로 자동 캐스트 해준다
```

> 💻 실습: `./run.sh Assignment`

## 3.11 삼항(조건) 연산자

`조건 ? A : B`는 자바의 **유일한 삼항(피연산자 3개) 연산자**다. "조건이 true면 A, 아니면 B"를 **값으로** 돌려준다.

```
       조건    ? 참일 때 : 거짓일 때
        ↓          ↓        ↓
(score >= 60) ? "합격"  : "불합격"
```

`if`문과 무엇이 다를까? `if`는 **문장(statement)** 이라 값을 내지 못하지만, 삼항은 **식(expression)** 이라 대입이나 반환에 바로 쓸 수 있다.

```java
int max = (a > b) ? a : b;              // 한 줄로 최댓값
int abs = (n < 0) ? -n : n;             // 절댓값
String grade = (s >= 90) ? "A"          // 중첩도 가능
             : (s >= 80) ? "B"
             : (s >= 70) ? "C"
             : "F";
```

삼항은 **한 값을 두 갈래(또는 몇 갈래) 중에 고를 때** 가장 빛난다. 다만 중첩이 3단계를 넘으면 가독성이 급격히 나빠지므로 그때는 `if-else`나 `switch`가 낫다.

> 💻 실습: `./run.sh Ternary`

## 3.12 우선순위·방향

한 식에 여러 연산자가 섞이면 자바는 **두 가지 규칙**으로 계산 순서를 정한다.

**① 우선순위(precedence): 높은 것부터 먼저 묶는다.** 곱셈이 덧셈보다 먼저인 것처럼.

```
2 + 3 * 4   →  2 + 12  →  14     (3*4가 먼저)
```

대략적인 우선순위 순서는 다음과 같다(높음 → 낮음):

```
단항(++ -- ! ~)  >  산술(* / %)  >  산술(+ -)  >  시프트(<< >>)
   >  비교(< > <= >=)  >  등가(== !=)  >  비트(& ^ |)  >  논리(&& ||)  >  삼항(?:)  >  대입(=)
```

**② 결합 방향(associativity): 우선순위가 같으면 이 방향대로 묶는다.** 대부분 왼쪽→오른쪽이지만, **대입(`=`)과 삼항(`?:`)만 오른쪽→왼쪽**이다.

```
100 - 20 - 5   →  (100-20)-5 = 75    (왼쪽→오른쪽. 만약 오른쪽부터라면 85)
a = b = c = 10                        (오른쪽→왼쪽. c=10부터 대입)
```

여러 종류가 섞이면 우선순위표를 떠올려야 한다.

```java
boolean r = 2 + 3 > 4 && 1 < 2;
// 산술 먼저: (2+3)=5  →  비교: (5>4)=true, (1<2)=true  →  논리: true && true = true
```

**조언: 우선순위표를 외우기보다 괄호로 의도를 드러내라.** 괄호는 런타임 비용이 전혀 없다. 애매하면 무조건 괄호를 쓰는 것이 읽는 사람과 미래의 자신을 위한 최선이다.

> 💻 실습: `./run.sh PrecedenceAndAssociativity`

## ⚠️ 흔한 오해와 함정

- **`i++`와 `++i`가 반복문에서 다르다?** — `for (int i=0; i<n; i++)`에서 `i++`를 `++i`로 바꿔도 **결과는 같다.** 단독으로 쓰였기 때문이다. 차이는 오직 "값으로 쓰일 때(식에 섞일 때)"만 나타난다.
- **`7 / 2`가 `3.5`일 것이다?** — 아니다. 둘 다 정수라 결과도 정수 `3`이다. 실수 결과가 필요하면 한쪽을 `double`로: `7 / 2.0` 또는 `(double) 7 / 2`.
- **오버플로우가 나면 예외가 뜰 것이다?** — 정수 오버플로우는 **조용히 순환할 뿐 예외를 던지지 않는다.** 그래서 더 위험하다. 예외로 잡고 싶으면 `Math.addExact`, `Math.multiplyExact`를 쓴다.
- **`double`로 돈 계산해도 소수점 몇 자리는 괜찮겠지?** — 위험하다. 오차는 누적되고, `==` 비교는 어긋난다. **최소 단위 정수나 `BigDecimal`(문자열 생성자)** 을 써라.
- **`new BigDecimal(0.1)`이면 정확하겠지?** — 아니다. `double` `0.1`은 이미 오차를 품었고, 그게 그대로 들어간다. 반드시 `new BigDecimal("0.1")`처럼 **문자열**로 생성하라.
- **`nan == nan`은 당연히 `true`?** — `false`다. `NaN`은 자기 자신과도 같지 않다. 검사는 `Double.isNaN()`으로.
- **문자열 비교는 `==`로 하면 되지 않나?** — `==`는 객체 주소를 비교한다. 리터럴이 우연히 같은 객체를 가리켜 `true`가 나올 수 있지만, **믿으면 안 된다.** 내용 비교는 항상 `.equals()`.
- **음수를 `>>>`로 반으로 나누면 되겠지?** — 안 된다. `>>>`는 부호를 무시하고 0을 채우므로 음수가 **거대한 양수**로 변한다. 부호 있는 나눗셈은 `>>`.
- **`&&`나 `||` 대신 `&`, `|`를 써도 똑같지 않나?** — 논리값에는 결과가 같아 보이지만 `&`, `|`는 **단축평가를 안 한다.** 오른쪽에 부작용이나 NPE 위험이 있으면 결과가 달라진다.

## 확인문제

**문제 1.** 다음 코드의 출력은?

```java
int i = 3;
int result = i++ + ++i;
System.out.println("result=" + result + ", i=" + i);
```

<details>
<summary>힌트</summary>
`i++`는 현재 값을 먼저 내놓고 증가, `++i`는 증가한 뒤 값을 내놓는다. 왼쪽부터 순서대로 따라가 보자.
</details>

<details>
<summary>해설</summary>

`result=8, i=5`

- `i++`: 현재 값 **3**을 내놓고 `i`는 4가 된다.
- `++i`: `i`를 5로 만든 뒤 **5**를 내놓는다.
- `3 + 5 = 8`, 최종 `i=5`.

관련: §3.1

</details>

---

**문제 2.** 다음 세 식의 결과를 각각 구하라.

```java
System.out.println(-17 / 5);
System.out.println(-17 % 5);
System.out.println(17 % -5);
```

<details>
<summary>힌트</summary>
정수 나눗셈은 0 방향 절삭이다. 나머지의 부호는 왼쪽 값을 따른다.
</details>

<details>
<summary>해설</summary>

`-3`, `-2`, `2`

- `-17 / 5 = -3` : 0 방향으로 절삭하므로 -4가 아니라 -3.
- `-17 % 5 = -2` : 부호는 왼쪽(-17)을 따라 음수. 항등식 확인: `(-3)*5 + (-2) = -17` ✓
- `17 % -5 = 2` : 오른쪽 부호는 무관하므로 왼쪽(17)을 따라 양수.

관련: §3.2

</details>

---

**문제 3.** 아래 코드가 화면에 무엇을 출력하는가? 그리고 왜 예외가 나지 않는가?

```java
double a = 1.0 / 0.0;
double b = 0.0 / 0.0;
System.out.println(a);
System.out.println(b == b);
```

<details>
<summary>힌트</summary>
실수 나눗셈은 0으로 나눠도 예외 대신 특수값을 만든다. 그 특수값 중 하나는 자기 자신과도 같지 않다.
</details>

<details>
<summary>해설</summary>

출력: `Infinity` / `false`

- `1.0 / 0.0`은 `Infinity`(양의 무한대). 실수(IEEE-754)에는 무한대가 정의되어 있어 예외 없이 값이 나온다. (정수 `1 / 0`이었다면 `ArithmeticException`)
- `0.0 / 0.0`은 `NaN`. `NaN`은 자기 자신과의 `==`조차 `false`다. 그래서 `b == b`는 `false`.
- `NaN` 검사는 `Double.isNaN(b)`으로 해야 한다.

관련: §3.5

</details>

---

**문제 4.** `int perm`이 읽기(`0b100`)와 실행(`0b001`) 권한을 가지고 있다. 여기에 **쓰기(`0b010`) 권한을 추가**하고, 그 후 **읽기 권한이 있는지 확인**하는 코드를 비트 연산자로 작성하라.

<details>
<summary>힌트</summary>
비트를 켤 때는 `|`(OR), 특정 비트가 켜져 있는지 확인할 때는 `&`(AND) 마스킹을 쓴다.
</details>

<details>
<summary>해설</summary>

```java
int READ = 0b100, WRITE = 0b010, EXEC = 0b001;
int perm = READ | EXEC;           // 101

perm = perm | WRITE;              // 쓰기 추가 → 111  (perm |= WRITE; 로도 가능)
boolean canRead = (perm & READ) != 0;   // true
```

- `|`로 비트를 켜서 권한을 추가한다.
- `& READ` 마스킹의 결과가 0이 아니면 그 비트가 켜져 있다는 뜻이다.

관련: §3.8

</details>

**문제 5.** 다음 코드는 컴파일 에러가 난다. 왜인지 설명하고, `+=`를 이용해 한 줄로 고쳐라.

```java
byte b = 10;
b = b + 1;   // 컴파일 에러
```

<details>
<summary>힌트</summary>
`b + 1`의 결과 타입이 무엇인지 생각해보자. 복합 대입 연산자에는 무엇이 숨어 있는가?
</details>

<details>
<summary>해설</summary>

**에러 원인:** `b + 1`에서 `byte`는 산술 연산 시 `int`로 승격되므로 결과가 `int`다. `int`를 `byte`에 담으려면 명시적 캐스트가 필요한데, 자동으로 좁혀지지 않아 컴파일 에러가 난다.

**해결:**

```java
b += 1;   // OK
```

복합 대입 `+=`에는 **암묵적 형변환**이 포함되어 있어, 결과를 자동으로 `byte`로 캐스트해준다. (또는 `b = (byte)(b + 1);` 로 명시 캐스트해도 된다.)

관련: §3.10, [개념서 §2.4 형변환](../JAVA_개념서/02-데이터를-담는-그릇-변수.md#24-형변환이-필요한-이유)

</details>

## 요약

- **증감 연산자**는 위치가 핵심이다. 전위 `++x`는 증가 후 사용, 후위 `x++`는 사용 후 증가. 단독으로 쓰면 결과가 같다. (§3.1)
- **정수 나눗셈**은 소수부를 0 방향으로 버리고, **나머지 `%`의 부호는 왼쪽 값**을 따른다. `+`는 문자열이 끼면 연결이 된다. (§3.2)
- **정수 오버플로우는 예외 없이 조용히 순환**한다. 2의 보수 때문에 `MAX + 1 == MIN`. 큰 계산은 `long`이나 `Math.*Exact`로. (§3.3)
- **`double`로 돈 계산 금지.** 오차 때문에 `0.1 + 0.2 != 0.3`. 최소 단위 정수나 `BigDecimal`(문자열 생성자)을 써라. (§3.4)
- 실수 0 나눗셈은 예외 대신 **`Infinity`/`NaN`** 을 만든다. `NaN`은 `==`로 못 잡으니 `Double.isNaN()`으로. (§3.5)
- **실수는 `==` 대신 허용오차**로 비교하고, **문자열 내용은 `.equals()`** 로 비교한다. `==`는 객체 동일성이다. (§3.6)
- **`&&`, `||`는 단축평가**를 한다. 이 성질이 null 방어 패턴을 가능하게 한다. `&`, `|`는 단축평가하지 않는다. (§3.7)
- **비트 연산**(`& | ^ ~`)은 자리별 논리 연산이며, 마스킹/플래그에 쓰인다. XOR은 두 번 하면 원복된다. (§3.8)
- **시프트**에서 음수를 밀 때는 `>>`(부호 유지)와 `>>>`(0 채움, 거대 양수)를 반드시 구분한다. (§3.9)
- **복합 대입**(`+=` 등)에는 암묵적 형변환이 숨어 있어 `b += 1`은 되지만 `b = b + 1`은 안 될 수 있다. (§3.10)
- **삼항 `?:`** 는 값을 돌려주는 식이라 대입/반환에 바로 쓴다. 중첩이 깊으면 `if`가 낫다. (§3.11)
- 계산 순서는 **우선순위**(높은 것 먼저)와 **결합 방향**(대입·삼항만 오른쪽→왼쪽)으로 정해진다. **괄호로 의도를 드러내라.** (§3.12)
