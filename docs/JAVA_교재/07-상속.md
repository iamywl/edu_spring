# Chapter 07. 상속 (Inheritance)

앞 장에서 우리는 클래스로 하나의 부품(객체)을 만드는 법을 배웠습니다. 그런데 실무 코드를 보면 비슷한 클래스가 계속 반복됩니다. 자동차, 스포츠카, 트럭 — 이들은 모두 "시동을 켠다", "정지한다" 같은 공통 기능을 가집니다. 매번 같은 코드를 복사하면 어떻게 될까요? 시동 로직을 하나 고치는 순간 열 군데를 똑같이 고쳐야 하고, 하나라도 빠뜨리면 버그가 됩니다.

**상속(inheritance)**은 이 문제를 해결합니다. 공통 기능을 부모 클래스에 한 번만 작성하고, 자식 클래스가 그것을 "물려받아" 재사용하는 것입니다. 그리고 상속은 단순한 코드 재사용을 넘어, 이 책의 뒤에서 계속 만나게 될 **다형성**의 토대가 됩니다. 하나의 부모 타입으로 여러 자식을 통일해서 다루는 이 능력이야말로 객체지향의 심장입니다.

이 장은 "부모의 코드를 물려받는다"는 단순한 감각에서 시작해, 생성자 호출 순서, 메소드 재정의, 타입 변환, 동적 바인딩, 추상 클래스, 그리고 자바 17의 봉인된 클래스까지 차근차근 올라갑니다.

> 이 장은 [JAVA 개념서 4장 「객체지향이란 무엇인가?」](../JAVA_교재/심화/04-객체지향이란-무엇인가.md)의 §4.4(상속)·§4.5(다형성)·§4.6(추상 클래스)를 실습 중심으로 다시 풀어낸 것입니다. "왜 그렇게 동작하는가"가 더 궁금하면 개념서를 함께 펼쳐 두세요.

## 7.1 상속 개념

### 개념

상속은 이미 만들어진 클래스(부모, super class)의 필드와 메소드를, 새로 만드는 클래스(자식, sub class)가 물려받아 재사용하는 문법입니다. 얻는 것은 두 가지입니다.

- **코드 재사용**: 공통 기능을 부모에 한 번만 작성하고 여러 자식이 공유합니다. 중복이 사라져 유지보수가 쉬워집니다.
- **is-a 관계 표현**: 상속은 "자식 **is-a** 부모"가 성립할 때만 씁니다. 억지로 물려받으면 설계가 무너집니다.

**is-a 관계**란 "A는 B의 한 종류다"가 자연스러운지를 따지는 것입니다.

```
  ┌─────────────────────────────────────────────┐
  │  SportsCar is-a Car   (스포츠카는 자동차다)  → O 상속 OK   │
  │  Engine    is-a Car   (엔진은 자동차다)      → X 상속 아님  │
  │                        └ 이건 has-a (자동차가 엔진을 "가진다") │
  └─────────────────────────────────────────────┘
```

is-a면 상속(extends), has-a면 포함(필드로 가지기)입니다. 이 구분을 헷갈리면 상속 계층이 뒤엉킵니다.

### 코드

```java
static class Car {                 // 부모: 모든 자동차의 공통 기능
    String name;
    void start() { System.out.println(name + " 시동을 켠다."); }
    void stop()  { System.out.println(name + " 정지한다."); }
}

static class SportsCar extends Car {   // 자식: start/stop 을 한 줄도 안 썼지만 사용 가능
    void boost() { System.out.println(name + " 부스트!"); }
}
```

`SportsCar`는 `start()`, `stop()`을 직접 작성하지 않았지만 부모에게서 물려받아 그대로 호출할 수 있습니다.

## 7.2 클래스 상속

### 개념

자바에서 상속은 **`extends`** 키워드로 표현합니다.

```
class 자식 extends 부모 { ... }
```

핵심 규칙 두 가지를 기억하세요.

- **단일 상속(single inheritance)**: 자바 클래스는 부모를 **오직 하나**만 가질 수 있습니다. 여러 부모를 동시에 `extends`할 수 없습니다. (여러 부모의 같은 메소드가 충돌하는 "다이아몬드 문제"를 원천 차단하기 위함입니다. 여러 타입이 필요하면 인터페이스 `implements`를 씁니다.)
- **다단계 상속(계층 상속)은 가능**: `A → B → C`처럼 여러 단계로 이어질 수 있습니다. 가장 아래 자식은 위쪽 조상들의 기능을 연쇄적으로 모두 물려받습니다.

```
        Animal          ← breathe()
          │  extends
         Dog            ← bark()  (+ breathe 물려받음)
          │  extends
        Puppy           ← whine() (+ bark + breathe 물려받음)
```

`Puppy` 객체 하나는 `whine()`, `bark()`, `breathe()`를 모두 쓸 수 있습니다.

### 코드

```java
static class Animal { void breathe() { ... } }
static class Dog   extends Animal { void bark()  { ... } }
static class Puppy extends Dog    { void whine() { ... } }

Puppy p = new Puppy("초코");
p.breathe();  // Animal 에서
p.bark();     // Dog 에서
p.whine();    // Puppy 자신
```

## 7.3 부모 생성자 호출

### 개념

자식 객체를 만들 때는 **부모 부분이 먼저 초기화**되어야 합니다. 집을 지을 때 1층(부모)이 없으면 2층(자식)을 올릴 수 없는 것과 같습니다. 그래서 자식 생성자의 첫 줄에서는 **반드시 부모 생성자가 먼저 호출**됩니다.

- **`super(...)`**: 부모 생성자를 명시적으로 호출합니다. 반드시 자식 생성자의 **첫 문장**이어야 합니다.
- `super(...)`를 생략하면 컴파일러가 자동으로 매개변수 없는 `super()`를 넣어 줍니다. 따라서 부모에 **기본 생성자가 없고 매개변수 생성자만 있으면**, 자식은 `super(...)`를 직접 써야 합니다(안 쓰면 컴파일 오류).

실행 순서는 **부모 생성자 → 자식 생성자 본문**입니다(위에서 아래로).

```
  new Student("홍길동", 20240101)
        │
        ├─▶ ① super(name) 호출 → Person(부모) 생성자 실행
        │
        └─▶ ② Person 끝난 뒤 Student(자식) 본문 실행
```

### 코드

```java
static class Person {
    Person(String name) { System.out.println("① 부모 생성자"); }
}
static class Student extends Person {
    Student(String name, int id) {
        super(name);                       // (필수) 첫 문장, 부모 먼저
        System.out.println("② 자식 생성자");
    }
}
```

## 7.4 메소드 재정의 (Overriding)

### 개념

부모에게 물려받은 메소드가 자식에게 맞지 않으면, 자식이 **같은 메소드를 다시 정의**할 수 있습니다. 이것이 **메소드 재정의(overriding)**입니다. 모든 동물이 소리를 내지만, 고양이는 "야옹", 개는 "멍멍"인 것처럼요.

재정의 규칙:

1. 메소드 **이름 / 매개변수 목록 / 리턴 타입**이 부모와 같아야 합니다.
2. 접근 제한자는 부모와 **같거나 더 넓어야** 합니다 (예: `protected` → `public` 가능, `protected` → `private` 불가).
3. **`@Override`** 애노테이션을 붙이면, 규칙을 어겼을 때 컴파일러가 오류로 잡아 줍니다(강력 권장).

`@Override`는 특히 오타 사고를 막아 줍니다. `sound()`를 재정의하려다 `soundd()`라고 치면, 재정의가 아니라 **전혀 새로운 메소드**가 되어 버립니다. `@Override`가 있으면 이 실수를 컴파일 단계에서 잡아 줍니다.

`super.메소드()`를 쓰면 재정의한 자식 안에서 **부모의 원래 메소드**도 함께 호출할 수 있습니다.

### 코드

```java
static class Animal { void sound() { System.out.println("동물이 소리를 낸다."); } }

static class Cat extends Animal {
    @Override void sound() { System.out.println("야옹"); }
}
static class Dog extends Animal {
    @Override void sound() {
        super.sound();                 // 부모 원래 동작도 실행
        System.out.println("(그리고) 멍멍");
    }
}
```

## 7.5 final 클래스와 final 메소드

### 개념

`final` 키워드는 "더 이상 바꿀 수 없음"을 뜻합니다. 상속과 관련해서는 두 가지로 쓰입니다.

- **final 클래스**: **상속을 금지**합니다. 이 클래스를 `extends`할 수 없습니다.
  예) `java.lang.String`은 final 클래스라 상속이 불가능합니다. 동작이 바뀌면 안 되는(보안·불변) 클래스에 씁니다.
- **final 메소드**: **재정의를 금지**합니다. 자식이 이 메소드를 다시 정의할 수 없습니다. 부모가 정한 동작을 그대로 써야 하는 핵심 로직(예: 잔액 계산)에 씁니다.

```
  final class Password        → extends 불가 (상속 자체를 막음)
  class Account {
      final int getBalance()  → 재정의 불가 (호출은 가능)
      String describe()       → 재정의 가능 (final 아님)
  }
```

주의: final 클래스도 **사용은 가능**합니다(객체 생성·메소드 호출 OK). 막히는 것은 **상속뿐**입니다. final 메소드도 **호출은 가능**하고 **재정의만** 막힙니다.

> 참고: `final 변수`는 값을 한 번만 대입하는 상수를 뜻합니다(2장 참고). 이 절의 주제는 상속·재정의 금지입니다.

### 코드

```java
static final class Password { ... }        // extends 하면 컴파일 오류

static class Account {
    final int getBalance() { return balance; }   // 자식이 재정의 못 함
    String describe() { return "일반 계좌"; }      // 재정의 가능
}
static class SavingsAccount extends Account {
    @Override String describe() { return "적금 계좌"; }  // OK
    // @Override int getBalance() { ... }  // 컴파일 오류!
}
```

## 7.6 protected 접근 제한자

### 개념

접근 제한자는 필드·메소드를 "어디까지 접근 가능하게 할지" 정합니다. 상속과 가장 관련 깊은 것이 **`protected`**입니다.

접근 범위(좁음 → 넓음):

| 제한자 | 접근 가능 범위 |
|--------|----------------|
| `private` | 같은 클래스 안에서만 |
| (default) | 같은 패키지 안에서만 |
| `protected` | 같은 패키지 **+ 다른 패키지라도 자식(상속) 클래스** |
| `public` | 어디서나 |

즉 `protected`는 **"외부에는 감추되, 자식에게는 물려주고 싶은"** 멤버에 씁니다. `private` 필드는 자식조차 직접 접근할 수 없지만, `protected` 필드는 자식이 직접 접근할 수 있습니다.

```
  Employee
    protected name        ──▶ 자식 Manager 에서 직접 접근 O
    protected baseSalary  ──▶ 자식 Manager 에서 직접 접근 O
    private   ssn         ──▶ 자식이라도 직접 접근 X (완전 차단)
```

### 코드

```java
static class Employee {
    protected int baseSalary;   // 자식이 직접 접근 가능
    private   String ssn;       // 자식도 접근 불가
}
static class Manager extends Employee {
    @Override protected int calculatePay() {
        return baseSalary + bonus;   // 부모 protected 필드에 직접 접근
        // return ssn;               // private → 컴파일 오류
    }
}
```

## 7.7 타입 변환 (업캐스팅 / 다운캐스팅)

### 개념

상속 관계에서는 부모 타입과 자식 타입 사이에 변환이 일어납니다.

- **업캐스팅(upcasting)**: 자식 타입 → 부모 타입. "자식 is-a 부모"이므로 **항상 안전**하며 **자동(암시적)**으로 됩니다. 단, 부모 타입 변수로는 **부모가 가진 멤버만** 호출할 수 있습니다.
- **다운캐스팅(downcasting)**: 부모 타입 → 자식 타입. **강제(명시적) 캐스트 `(자식)`**가 필요합니다. 실제 객체가 그 자식이 아니면 실행 중 **`ClassCastException`**이 발생합니다.

```
              업캐스팅 (자동, 안전)
        Bird  ───────────────────▶  Animal
     (자식 객체)      캐스트 불필요    (부모 타입 변수)
                                        │  animal.fly()  → 안 보임(컴파일 오류)
                                        │  animal.breathe() → OK
        Bird  ◀───────────────────  Animal
     (다시 자식)    (Bird) 강제 캐스트
              다운캐스팅 (수동, 위험 — 실제 객체 확인 필요)
```

다운캐스팅은 위험하므로, **하기 전에 `instanceof`로 확인**하는 것이 안전합니다(7.9).

### 코드

```java
Bird bird = new Bird();
Animal animal = bird;            // 업캐스팅: 자동
animal.breathe();                // 부모 멤버만 호출 가능

Bird back = (Bird) animal;       // 다운캐스팅: 강제 캐스트
back.fly();                      // 이제 자식 멤버 호출 가능

Animal plain = new Animal();
Bird wrong = (Bird) plain;       // 실행 중 ClassCastException!
```

## 7.8 다형성 (Polymorphism)

### 개념

**다형성**이란 "하나의 부모 타입 참조로 여러 자식 객체를 다룰 수 있고, 실제 실행되는 메소드는 **객체의 실제 타입**에 따라 결정되는" 성질입니다. 상속의 진짜 목적이자 이 장의 절정입니다.

핵심 두 가지:

- **부모 타입 참조**: 부모 타입 변수 하나로 어떤 자식 객체든 가리킬 수 있습니다(업캐스팅). 덕분에 배열·매개변수를 부모 타입 하나로 통일해 다룰 수 있습니다.
- **동적 바인딩(dynamic binding)**: 부모 타입 변수로 재정의된 메소드를 호출하면, **변수의 타입이 아니라 실제 객체의 타입**에 정의된 메소드가 실행됩니다. 어떤 메소드가 불릴지는 **컴파일 시점이 아니라 실행 시점**에 결정됩니다.

```
  Shape[] shapes = { Circle, Rectangle, Circle };   ← 변수 타입은 전부 Shape
       │
       │  for (Shape s : shapes) s.area();
       ▼
  ┌──────────── 동적 바인딩 ────────────┐
  │  s 가 실제로 Circle    → Circle.area()    실행 │
  │  s 가 실제로 Rectangle → Rectangle.area() 실행 │
  └──────────────────────────────────────┘
   (같은 s.area() 한 줄이지만, 실제 객체마다 다른 버전이 실행됨)
```

이 덕분에 새로운 도형(`Triangle`)이 추가돼도, 순회하는 코드는 한 줄도 바꿀 필요가 없습니다. 이것이 다형성이 주는 **확장에 열려 있는** 설계입니다.

### 코드

```java
static class Shape { double area() { return 0; } }
static class Circle    extends Shape { @Override double area() { return Math.PI*r*r; } }
static class Rectangle extends Shape { @Override double area() { return w*h; } }

Shape[] shapes = { new Circle(2), new Rectangle(3, 4), new Circle(1) };
double total = 0;
for (Shape s : shapes) total += s.area();   // 실제 객체 타입의 area() 실행
```

## 7.9 객체 타입 확인 (instanceof)

### 개념

부모 타입 변수가 **실제로 어떤 자식 객체를 가리키는지** 알아야 할 때 **`instanceof`**를 씁니다.

```
객체 instanceof 타입   →  객체가 그 타입(또는 그 자식)이면 true, 아니면 false
```

- 다운캐스팅(7.7) 전에 `instanceof`로 확인하면 `ClassCastException`을 예방할 수 있습니다.
- **패턴 매칭 instanceof (Java 16+)**: 확인과 동시에 캐스팅된 변수를 바로 만들어 줍니다. 별도 캐스트 코드가 필요 없습니다.

```
  [전통 방식]                          [패턴 매칭 (Java 16+)]
  if (a instanceof Dog) {              if (a instanceof Dog d) {
      Dog d = (Dog) a;   ← 캐스트 한 줄       d.bark();   ← d 를 바로 사용
      d.bark();          더 필요       }
  }
```

### 코드

```java
// 전통 방식: 확인 후 직접 캐스트
if (a instanceof Dog) {
    Dog d = (Dog) a;
    d.bark();
}

// 패턴 매칭: 확인 + 캐스팅을 한 번에 (Java 16+)
if (a instanceof Dog d) {
    d.bark();
}
```

## 7.10 추상 클래스 (Abstract Class)

### 개념

**추상 클래스**는 "아직 완성되지 않은, 상속 전용" 클래스입니다. `abstract` 키워드로 선언합니다.

- **추상 메소드(abstract method)**: 선언만 있고 본문 `{ }`이 없는 메소드. 자식이 **반드시** 재정의(구현)해야 합니다.
- **추상 클래스(abstract class)**: 추상 메소드를 하나라도 가지면 추상 클래스가 됩니다. **직접 `new`로 객체를 만들 수 없습니다**(미완성이므로). 일반 메소드·필드도 함께 가질 수 있어 공통 코드를 재사용할 수 있습니다.

쓰는 이유: **"공통 뼈대는 부모가 제공하고, 자식마다 달라지는 부분만 추상 메소드로 강제 구현"**하게 만들기 위해서입니다.

```
        Payment (abstract)          ← new Payment(...) 불가!
          ├─ int amount             ← 공통 필드
          ├─ abstract void pay();   ← 자식마다 다름 → 강제 구현
          └─ void printReceipt()    ← 공통 코드 → 자식이 그대로 재사용
                │
       ┌────────┴────────┐
  CardPayment        CashPayment
  pay() 구현          pay() 구현
```

`pay()`처럼 결제 수단마다 달라지는 부분은 강제 구현하게 하고, `printReceipt()`처럼 공통인 부분은 부모가 한 번만 작성합니다.

### 코드

```java
static abstract class Payment {
    int amount;
    abstract void pay();                        // 자식이 반드시 구현
    void printReceipt() {                       // 공통 → 재사용
        System.out.println("영수증 : " + amount + "원 결제 완료");
    }
}
static class CardPayment extends Payment {
    @Override void pay() { System.out.println("카드로 결제"); }
}

Payment[] payments = { new CardPayment(10000), new CashPayment(5000) };
for (Payment p : payments) {   // 부모 타입으로 통일 (다형성!)
    p.pay();           // 자식마다 다른 구현
    p.printReceipt();  // 부모 공통 코드
}
```

> 💻 실습: `./run.sh AbstractClass`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch07/AbstractClass.java`
>
> 추상 클래스와 인터페이스를 **언제 무엇으로** 골라야 하는지는 [개념서 §4.6](../JAVA_교재/심화/04-객체지향이란-무엇인가.md#46-추상-클래스-vs-인터페이스-언제-무엇을)에서 다룹니다.

## 7.11 봉인된 클래스 (Sealed Class) — Java 17+

### 개념

**`sealed`(봉인된) 클래스**는 "누가 나를 상속할 수 있는지"를 부모가 직접 지정하는 클래스입니다. `final`(상속 완전 금지)과 일반 클래스(누구나 상속 가능)의 **중간 단계**입니다.

```java
sealed class Shape permits Circle, Rectangle, Triangle { ... }
```

- **`permits`**: 상속을 허용할 자식 목록을 명시합니다. 목록에 없는 클래스는 상속할 수 없습니다.
- `sealed`의 자식은 반드시 아래 셋 중 하나여야 합니다:

| 자식의 형태 | 의미 |
|-------------|------|
| `final` | 더 이상 상속 불가 — 여기서 집합을 닫음 |
| `sealed` | 이 자식도 다시 자신의 `permits`로 상속 대상을 제한 |
| `non-sealed` | 봉인을 풀어, 이 아래로는 누구나 상속 가능 |

```
             sealed Shape permits Circle, Rectangle
                       │  (이 둘 외에는 상속 불가)
              ┌────────┴─────────┐
        Circle (final)      Rectangle (non-sealed)
        상속 닫음                  │  봉인 해제
                             Square (자유 상속 가능)
```

쓰는 이유: 자식의 종류를 **"닫힌 집합"**으로 고정할 수 있어, `switch`에서 모든 경우를 안전하게 다루기 좋습니다. 컴파일러가 "모든 경우를 다뤘다"고 인정하면 `default`가 필요 없고, 새 자식을 추가하면 처리 안 한 `switch`가 컴파일 오류로 드러납니다(도메인 모델링에 유용).

### 코드

```java
sealed static abstract class Shape permits Circle, Rectangle {
    abstract double area();
}
static final      class Circle    extends Shape { ... }   // 집합을 닫음
static non-sealed class Rectangle extends Shape { ... }   // 봉인 해제
static            class Square    extends Rectangle { }   // non-sealed 라 자유 상속

// 닫힌 집합이라 switch 로 안전하게 분기 (default 불필요)
String kind = switch (s) {
    case Circle c    -> "원";
    case Rectangle r -> "사각형 계열";
};
```

> 💻 실습: `./run.sh SealedClass`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch07/SealedClass.java`
>
> `sealed` + `record` + `switch`가 어떻게 시너지를 내는지는 [개념서 §8.2](../JAVA_교재/심화/08-자바의-현대적-기능들.md#82-sealed-class-상속을-제한하는-이유)에서 이어집니다.

## ⚠️ 흔한 오해와 함정

1. **상속(is-a)과 포함(has-a)을 헷갈린다.**
   "코드를 재사용하고 싶다"는 이유만으로 상속하면 안 됩니다. `Engine extends Car`는 틀렸습니다. 엔진은 자동차의 한 종류가 아니라, 자동차가 **가지는** 부품입니다. is-a가 어색하면 상속 대신 필드로 포함하세요.

2. **`super(...)`는 생성자 첫 줄이어야 한다.**
   자식 생성자에서 다른 코드를 먼저 쓰고 나서 `super(...)`를 호출하면 컴파일 오류입니다. 또한 부모에 매개변수 생성자만 있으면 `super(...)` 생략도 불가합니다(컴파일러가 넣어 주는 `super()`가 없으니까요).

3. **재정의(overriding)와 오버로딩(overloading)은 다르다.**
   재정의는 부모의 메소드를 자식이 **같은 시그니처로 다시** 정의하는 것이고, 오버로딩은 **같은 이름·다른 매개변수**로 여러 메소드를 만드는 것입니다. 이름만 비슷하지 완전히 다른 개념입니다. 재정의할 때 매개변수를 살짝 바꾸면 재정의가 아니라 오버로딩이 되어 버립니다 — 그래서 `@Override`가 중요합니다.

4. **다운캐스팅은 "확인 없이" 하면 터진다.**
   `(Bird) animal`은 컴파일은 통과해도, 실제 객체가 `Bird`가 아니면 실행 중 `ClassCastException`이 납니다. `instanceof`로 먼저 확인하거나 패턴 매칭을 쓰세요.

5. **필드는 다형성(동적 바인딩)이 적용되지 않는다.**
   동적 바인딩은 **메소드**에만 적용됩니다. 부모와 자식에 같은 이름의 **필드**가 있으면, 어느 필드가 보이는지는 "변수의 타입"으로 결정됩니다(필드는 재정의가 아니라 "숨김"). 상태는 필드로, 행동은 메소드로 — 다형성은 행동에서 빛납니다.

6. **추상 클래스는 `new` 할 수 없다.**
   `new Payment(...)`는 컴파일 오류입니다. 추상 클래스는 미완성이라 반드시 완성된 자식을 통해서만 객체가 됩니다. 다만 추상 클래스 **타입의 변수**는 만들 수 있습니다(`Payment p = new CardPayment(...)`).

**과제 4. 도형 다형성 — 부모 타입 배열 순회 (응용)**

**과제 5. instanceof 확인과 안전한 다운캐스팅 (응용)**

부모 클래스 `Animal`(필드 `name`, 생성자)과, 이를 상속하는 `Dog`(메소드 `bark()`)·`Cat`(메소드 `meow()`)을 작성하라. `main`에서 `Dog`, `Cat`, 그리고 순수 `Animal` 객체를 섞은 **`Animal[]` 배열**을 만든 뒤, `for`로 순회하면서 `instanceof`로 실제 타입을 확인하고 **안전하게 다운캐스팅**해 `Dog`면 `bark()`, `Cat`이면 `meow()`, 그 외에는 `"특정 소리가 없다"`를 출력하라. 전통 방식(`if (a instanceof Dog) { Dog d = (Dog) a; ... }`)으로 한 번, **패턴 매칭(`if (a instanceof Dog d) { ... }`, Java 16+)**으로 한 번, 두 방식 모두 작성해 결과가 같음을 확인하라.

- 힌트: 확인 없이 `(Dog) a`로 캐스팅하면 실제 객체가 `Dog`가 아닐 때 `ClassCastException`이 난다. `instanceof`로 먼저 걸러라. 패턴 매칭은 확인과 캐스팅을 한 줄로 합쳐 준다.

**과제 6. 추상 클래스로 강제 구현하기 (응용)**

`abstract` 클래스 `Payment`(필드 `amount`, 생성자)를 작성하되, **추상 메소드 `pay()`**(본문 `{ }` 없이 선언만)와 **공통 일반 메소드 `printReceipt()`**(`"영수증 : OOO원 결제 완료"` 출력)를 함께 두어라. 자식 `CardPayment`와 `CashPayment`는 각각 `pay()`를 **반드시 구현**해 결제 방식을 다르게 출력하라. `main`에서 `new Payment(...)`가 왜 안 되는지 주석으로 남기고, **`Payment[]` 배열**에 두 자식을 담아 순회하며 `pay()`(자식마다 다른 구현)와 `printReceipt()`(부모 공통 코드)를 함께 호출하라.

- 힌트: 추상 메소드는 `abstract void pay();`처럼 세미콜론으로 끝난다. 추상 클래스는 `new`로 직접 만들 수 없지만(`new Payment(...)` 컴파일 오류), 추상 클래스 **타입의 변수**(`Payment p = new CardPayment(...)`)는 만들 수 있다.
> 답안 비교: `./run.sh AbstractClass`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch07/AbstractClass.java`

**과제 7. 업캐스팅·다운캐스팅과 ClassCastException (응용)**

부모 클래스 `Animal`에 메소드 `breathe()`(`"숨을 쉰다."` 출력)를 두고, 이를 상속하는 자식 `Bird`에 고유 메소드 `fly()`(`"난다."` 출력)를 추가하라. `main`에서 다음 세 가지를 순서대로 실행하라. (1) `Bird` 객체를 만들어 **`Animal` 타입 변수에 업캐스팅**(자동)하고, 그 변수로 `breathe()`는 호출되지만 `fly()`는 **컴파일 오류**라는 점을 주석으로 남겨라. (2) 그 `Animal` 변수를 **`(Bird)`로 다운캐스팅**해 `fly()`를 호출하라(실제 객체가 `Bird`이므로 안전). (3) 순수 `Animal` 객체(`new Animal()`)를 `(Bird)`로 강제 캐스팅해 보고, 실행 중 발생하는 **`ClassCastException`을 `try-catch`로 잡아** 예외 이름을 출력하라.

- 힌트: 업캐스팅은 캐스트 연산자 없이 `Animal a = bird;`로 자동이다. 부모 타입 변수로는 **부모가 가진 멤버만** 보인다. 다운캐스팅은 `(Bird) a`처럼 강제 캐스트가 필요하며, 실제 객체가 그 타입이 아니면 `ClassCastException`이 난다. `catch (ClassCastException e)`에서 `e.getClass().getSimpleName()`으로 예외 이름을 얻는다.
> 답안 비교: `./run.sh TypeCasting`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch07/TypeCasting.java`

**과제 8. sealed 클래스와 permits (심화, Java 17+)**

봉인된 부모 `abstract` 클래스 `Shape`를 **`sealed ... permits Circle, Rectangle`**로 선언하고 추상 메소드 `area()`를 두어라. 자식 `Circle`(필드 `radius`)은 **`final`**로 선언해 상속을 닫고 `area()`를 `Math.PI * radius * radius`로 구현하라. 자식 `Rectangle`(필드 `w`, `h`)은 **`non-sealed`**로 선언해 봉인을 풀고 `area()`를 `w * h`로 구현한 뒤, `non-sealed`이므로 자유롭게 상속 가능한 손자 클래스 `Square`(정사각형)를 추가하라. `main`에서 `Shape[]` 배열에 세 객체를 담아 넓이를 출력하고, **닫힌 집합이라 `default` 없이** `switch`로 `Circle`/`Rectangle`을 분기하라.

- 힌트: `sealed`의 자식은 반드시 `final`·`sealed`·`non-sealed` 중 하나여야 한다. `permits` 목록에 없는 클래스는 상속하면 컴파일 오류다. 닫힌 집합이면 컴파일러가 "모든 경우를 다뤘다"고 인정해 `switch`에 `default`가 필요 없다. `Square`는 `non-sealed`인 `Rectangle`의 자식이므로 `permits`에 없어도 괜찮다.
> 답안 비교: `./run.sh SealedClass`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch07/SealedClass.java`

   class Base {
       void print(int x)    { System.out.println("Base.int:" + x); }
   }
   class Derived extends Base {
       void print(String x) { System.out.println("Derived.String:" + x); }
   }
   // main
   Derived d = new Derived();
   d.print(10);
   d.print("hi");
   ```

   }

   }

   ```
   `p.tag`와 `p.who()`는 각각 무엇으로 결정되는가? 그 차이의 이유를 쓰세요.

<summary>정답</summary>

출력: `P / Child`.
- `p.who()`는 **메소드**라 동적 바인딩이 적용되어 실제 객체 `Child`의 `who()`가 실행 → `Child`.
- `p.tag`는 **필드**라 동적 바인딩이 적용되지 않고 "변수의 타입(`Parent`)"으로 결정 → `P`(필드는 재정의가 아니라 "숨김"). 상태는 변수 타입, 행동은 실제 객체 타입 — 다형성은 **메소드(행동)**에서만 빛납니다(⚠️ 함정 5).

   class Animal {}

`(1)`은 실제 객체가 `Dog`이므로 안전하게 다운캐스팅되어 `d`에 정상 대입됩니다. `(2)`는 실제 객체가 `Dog`인데 `Cat`으로 강제 캐스팅하므로 **실행 중 `ClassCastException`**이 발생합니다(컴파일은 통과). 예외로 프로그램이 중단되므로 `"끝"`은 **출력되지 않습니다**. 다운캐스팅 전 `instanceof` 확인이 필요합니다(7.7, 7.9).

**[참·거짓 + 이유]**

15. 다음 문장이 참인지 거짓인지 판단하고 그 이유를 쓰세요.
    - (a) `final` 클래스는 객체를 생성할 수 없다.
    - (b) 추상 클래스는 일반(구현된) 메소드를 가질 수 없고 추상 메소드만 가질 수 있다.
    - (c) 다운캐스팅은 컴파일은 통과해도 실행 중 `ClassCastException`이 날 수 있다.

<details>
<summary>정답</summary>

- (a) **거짓**. `final` 클래스는 **상속(extends)만** 금지될 뿐, 객체 생성·메소드 호출은 정상적으로 됩니다(예: `String`). 막히는 것은 상속뿐입니다(7.5, ⚠️ 함정 참고).
- (b) **거짓**. 추상 클래스는 추상 메소드뿐 아니라 **일반(구현된) 메소드와 필드도 함께** 가질 수 있습니다. 공통 코드를 부모가 제공하고 자식이 재사용하는 것이 추상 클래스의 핵심 활용입니다(7.10).
- (c) **참**. 다운캐스팅은 컴파일은 통과하지만, 실제 객체가 대상 자식 타입이 아니면 실행 중 `ClassCastException`이 납니다. 그래서 `instanceof`(또는 패턴 매칭)로 먼저 확인해야 합니다(7.7, 7.9).
</details>

---

## 요약

- **상속(`extends`)**은 부모의 필드·메소드를 자식이 물려받아 재사용하는 문법이며, **is-a 관계**가 성립할 때만 씁니다. 자바는 **단일 상속**(부모 하나)만 허용하되 다단계 상속은 가능합니다.
- 자식 객체 생성 시 **부모 생성자가 먼저** 호출됩니다. `super(...)`는 자식 생성자의 **첫 문장**이어야 합니다.
- **재정의(overriding)**로 물려받은 메소드를 자식에 맞게 바꾸며, `@Override`가 실수를 막아 줍니다. `final`은 상속·재정의를 금지하고, `protected`는 외부에는 숨기되 자식에게는 공개합니다.
- 상속의 진짜 목적은 **다형성**입니다. 부모 타입 참조 + **동적 바인딩**으로, 하나의 코드가 여러 자식 객체를 실제 타입에 맞게 처리합니다. 다운캐스팅 전에는 `instanceof`(패턴 매칭 포함)로 확인하세요.
- **추상 클래스**는 공통 뼈대는 제공하되 달라지는 부분은 강제 구현하게 하고, **`sealed` 클래스**(Java 17+)는 상속 대상을 닫힌 집합으로 고정해 안전한 `switch` 분기를 가능하게 합니다.
