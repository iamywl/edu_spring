# Chapter 08. 인터페이스

앞 장에서 상속을 통해 "나는 ~이다(IS-A)" 관계를 배웠다면, 이번 장의 인터페이스는 "나는 ~할 수 있다(CAN-DO)" 관계를 다룬다. 인터페이스는 **무엇을 할 수 있는가(규격)** 만 약속하고, **어떻게 하는가(구현)** 는 각 클래스에 맡기는 자바의 핵심 장치다.

이 "규격과 구현의 분리" 덕분에 우리는 구현이 바뀌어도 흔들리지 않는 코드를 쓸 수 있고, 하나의 타입으로 서로 다른 객체를 똑같이 다루는 **다형성**을 얻는다. 스프링의 `@Service`/`@Repository`가 인터페이스 중심으로 설계된 이유, JDBC의 `Connection`이 인터페이스인 이유가 모두 여기에 있다.

이 장은 인터페이스의 역할부터 선언·구현, 상수 필드, 추상/디폴트/정적/private 메소드, 다중 구현과 상속, 타입 변환과 다형성, 봉인된 인터페이스까지 순서대로 다룬다. 각 소절 끝에는 직접 실행해 볼 수 있는 실습 명령이 있으니 코드와 함께 읽기를 권한다.

> 📖 이 장은 개념서 [4장. 객체지향이란 무엇인가?](../JAVA_교재/심화/04-객체지향이란-무엇인가.md)의 4.5(다형성)·4.6(추상 클래스 vs 인터페이스), [8장. 자바의 현대적 기능들](../JAVA_교재/심화/08-자바의-현대적-기능들.md)의 8.2(sealed)·8.3(패턴 매칭)과 짝을 이룬다. "왜?"가 궁금할 때 함께 읽자.

## 8.1 인터페이스의 역할

인터페이스는 **규격(specification)** 이다. "리모컨은 켜고 끄는 기능이 있다"는 약속만 하고, TV를 어떻게 켜는지·오디오를 어떻게 켜는지는 정하지 않는다. 그 세부 구현은 각 구현 클래스의 몫이다.

이 분리에서 두 가지 힘이 나온다.

- **규격과 구현의 분리**: 인터페이스를 사용하는 쪽은 규격만 알면 된다. 구현이 통째로 바뀌어도 사용하는 코드는 고칠 필요가 없다.
- **다형성의 기반**: 같은 인터페이스 타입 변수 하나로 서로 다른 구현 객체를 똑같은 방식으로 다룰 수 있다.

```java
interface RemoteControl {
    void turnOn();    // 어떻게 켜는지는 정의하지 않는다 — 약속만 한다
    void turnOff();
}

static void operate(RemoteControl rc) {  // 규격만 알면 어떤 구현이든 처리
    rc.turnOn();
    rc.turnOff();
}
```

`operate`는 TV가 오든 오디오가 오든 신경 쓰지 않는다. `RemoteControl`이라는 규격만 만족하면 그만이다.

> 💻 실습: `./run.sh InterfaceRole`

## 8.2 인터페이스와 구현 클래스 선언

인터페이스는 `interface` 키워드로 선언한다. 구현 클래스는 `class` 선언 뒤에 `implements 인터페이스이름`을 붙이고, **인터페이스의 추상 메소드를 모두** 재정의해야 한다. 하나라도 빠지면 컴파일되지 않는다.

```java
interface Animal {
    void sound();            // 구현 클래스가 반드시 채워야 하는 추상 메소드
}

class Dog implements Animal {
    @Override public void sound() { System.out.println("멍멍"); }
}

Animal animal = new Cat();   // 인터페이스 타입 변수에 구현 객체 대입 (업캐스팅)
animal.sound();
```

구현 객체는 그 클래스 타입(`Dog dog = new Dog()`)으로도, 인터페이스 타입(`Animal animal = new Cat()`)으로도 다룰 수 있다. 실무에서는 후자, 즉 인터페이스 타입으로 다루는 방식을 선호한다. 구현에 덜 묶이기 때문이다.

> 💻 실습: `./run.sh InterfaceDeclaration`

## 8.3 상수 필드

인터페이스에 선언한 필드는 제한자를 생략해도 자동으로 **`public static final`** 이 된다. 즉 모든 필드는 상수다.

- `public`: 어디서든 접근 가능
- `static`: 객체 없이 `인터페이스이름.상수`로 접근
- `final`: 값을 바꿀 수 없으므로 **선언과 동시에 반드시 초기화**

```java
interface Calculator {
    double PI = 3.14159;          // 사실상 public static final
    int    MAX_INPUT = 1_000_000;
    double area(double radius);
}

System.out.println(Calculator.PI);   // 객체 없이 접근 (static)
// Calculator.PI = 3.14;             // 컴파일 오류: final 변수는 재할당 불가
```

인터페이스 상수는 구현 클래스 안에서 이름만으로 참조할 수 있다(`return PI * radius * radius;`). 다만 인터페이스를 "상수 모음집"으로만 쓰는 것은 안티패턴으로 여겨진다. 진짜 상수 모음이 필요하면 `enum`이나 `final class`를 쓰는 편이 낫다.

> 💻 실습: `./run.sh ConstantField`

## 8.4 추상 메소드

인터페이스의 일반 메소드는 몸통 없이 선언만 하며, 제한자를 생략해도 자동으로 **`public abstract`** 가 된다. 몸통이 없으므로 선언 뒤에 세미콜론으로 끝난다.

핵심은 **구현 강제**다. 인터페이스의 추상 메소드를 하나라도 구현하지 않으면 컴파일 자체가 안 된다. 재정의할 때는 접근 제한을 반드시 `public`으로 열어야 한다(원본이 `public`이므로 더 좁힐 수 없다).

```java
interface Payment {
    void pay(int amount);    // public abstract void pay(int amount);
    boolean isPaid();
}

class CardPayment implements Payment {
    private boolean paid = false;
    @Override public void pay(int amount) { /* ... */ paid = true; }
    @Override public boolean isPaid()     { return paid; }
    // 둘 중 하나라도 빠지면 컴파일 오류
}
```

추상 메소드는 "이 규격을 만족하려면 이 기능을 반드시 제공하라"는 강제 조항인 셈이다.

> 💻 실습: `./run.sh AbstractMethod`

> 📖 추상 메소드와 추상 클래스의 차이가 헷갈린다면 개념서 [4.6 추상 클래스 vs 인터페이스](../JAVA_교재/심화/04-객체지향이란-무엇인가.md)를 참고하자.

## 8.5 디폴트 메소드

자바 8부터 `default` 키워드를 붙이면 인터페이스 안에서 메소드의 **몸통(기본 구현)** 을 가질 수 있다. 구현 클래스는 이 메소드를 재정의하지 않아도 그대로 물려받아 쓸 수 있고, 필요하면 재정의할 수도 있다.

디폴트 메소드가 등장한 이유는 **하위 호환** 때문이다. 이미 세상에 배포된 인터페이스에 새 기능을 추가하려면, 예전 같으면 모든 구현 클래스를 고쳐야 했다. 디폴트 메소드는 기본 구현을 인터페이스에 넣어 기존 코드를 건드리지 않고 기능을 추가하게 해준다(자바 8이 `Collection`에 `stream()`, `forEach()`를 추가할 수 있었던 비결).

```java
interface Logger {
    void log(String message);                 // 추상 메소드 (반드시 구현)

    default void logError(String message) {   // 기본 구현 제공
        log("[ERROR] " + message);            // 내부에서 추상 메소드를 활용
    }
}

class ConsoleLogger implements Logger {
    @Override public void log(String message) { System.out.println(message); }
    // logError 는 재정의하지 않아도 기본 구현이 그대로 쓰인다
}
```

디폴트 메소드는 내부에서 추상 메소드를 호출해 뼈대를 만들고, 세부 동작만 구현 클래스에 맡기는 식으로도 활용된다.

> 💻 실습: `./run.sh DefaultMethod`

> 📖 디폴트 메소드는 개념서 [4.6 — Java 8 이후 default 메서드](../JAVA_교재/심화/04-객체지향이란-무엇인가.md) 절에서 "인터페이스가 왜 코드를 갖게 되었나"라는 맥락으로도 다룬다.

## 8.6 정적 메소드

인터페이스에도 `static` 메소드를 정의할 수 있으며, 이 역시 몸통을 가진다. 호출은 **`인터페이스이름.메소드()`** 로만 한다. 객체나 구현 클래스를 통해서는 호출할 수 없다.

정적 메소드는 주로 인터페이스와 관련된 **도우미(유틸리티)** 기능을 담는다. 대표적으로 객체를 만들어 주는 **정적 팩토리 메소드**, 값을 검증하는 메소드가 있다.

```java
interface Shape {
    double area();

    static Shape createSquare(double side) {   // 정적 팩토리 메소드
        return () -> side * side;              // 함수형 인터페이스이므로 람다로 구현
    }
    static boolean isValidArea(double area) {  // 검증 도우미
        return area > 0;
    }
}

Shape square = Shape.createSquare(4);          // 인터페이스이름.정적메소드()
System.out.println(Shape.isValidArea(square.area()));
```

`java.util.List.of(...)`, `Comparator.comparing(...)`처럼 표준 라이브러리도 인터페이스 정적 메소드를 팩토리로 널리 활용한다.

> 💻 실습: `./run.sh StaticMethodInterface`

## 8.7 private 메소드

자바 9부터 인터페이스에 `private` 메소드를 둘 수 있다. 인터페이스 **내부에서만** 쓰이는 공통 코드를 담는 용도이며, 구현 클래스나 외부에서는 호출할 수 없다.

여러 디폴트 메소드가 같은 로직을 반복한다면, 그 중복을 private 메소드 한 곳에 모아 재사용한다. 인스턴스용(`private`)과 정적용(`private static`) 두 종류가 있다.

```java
interface Notifier {
    String name();

    default void sendInfo(String msg) { System.out.println(format("INFO", msg)); }
    default void sendWarn(String msg) { System.out.println(format("WARN", msg)); }

    private String format(String level, String msg) {   // 두 디폴트 메소드의 공통 로직
        return "[" + level + "] (" + name() + ") " + msg;
    }
}
```

private 메소드는 캡슐화의 연장이다. 인터페이스 사용자에게는 보일 필요 없는 내부 구현을 숨겨, 공개 규격을 깔끔하게 유지한다.

> 💻 실습: `./run.sh PrivateMethodInterface`

## 8.8 다중 인터페이스 구현

클래스는 하나만 `extends`할 수 있지만, 인터페이스는 여러 개를 동시에 `implements`할 수 있다. 각 인터페이스의 추상 메소드를 모두 구현하면, 그 객체는 여러 규격을 동시에 만족한다.

```java
interface Flyable   { void fly();  }
interface Swimmable { void swim(); }

class Duck implements Flyable, Swimmable {   // 두 규격을 동시에 구현
    @Override public void fly()  { System.out.println("납니다"); }
    @Override public void swim() { System.out.println("헤엄칩니다"); }
}

Duck duck = new Duck();
Flyable   f = duck;   // 날 수 있는 것으로 취급
Swimmable s = duck;   // 헤엄칠 수 있는 것으로 취급
```

자바가 클래스 다중 상속을 금지한 이유는 부모가 여럿일 때 생기는 모호함(다이아몬드 문제) 때문이다. 반면 인터페이스는 "무엇을 할 수 있는가"만 약속하므로 여러 개를 동시에 구현해도 충돌 위험이 적다. 이것이 자바가 다중 상속의 이점을 안전하게 얻는 방식이다.

> 💻 실습: `./run.sh MultipleImplementation`

## 8.9 인터페이스 상속

인터페이스도 다른 인터페이스를 `extends`로 상속할 수 있다. 클래스와 달리 인터페이스는 **여러 인터페이스를 동시에** 상속할 수 있다(인터페이스 다중 상속).

자식 인터페이스를 구현하는 클래스는 부모 인터페이스들의 추상 메소드까지 **모두** 구현해야 한다.

```java
interface Reader { void read();  }
interface Writer { void write(); }

interface ReadWriter extends Reader, Writer {  // 두 인터페이스를 동시에 상속
    void flush();                              // 자신만의 추상 메소드 추가
}

class FileStream implements ReadWriter {
    @Override public void read()  { /* ... */ }   // 부모 메소드까지 전부 구현
    @Override public void write() { /* ... */ }
    @Override public void flush() { /* ... */ }
}

Reader r = new FileStream();   // ReadWriter는 Reader를 상속하므로 대입 가능
```

인터페이스 상속은 작은 규격들을 조합해 더 큰 규격을 만드는 방법이다. 자바의 `Closeable`이 `AutoCloseable`을 상속하는 구조가 그 예다.

> 💻 실습: `./run.sh InterfaceInheritance`

## 8.10 타입 변환

구현 객체와 인터페이스 타입 사이에는 두 방향의 변환이 있다.

- **자동 타입 변환(업캐스팅)**: 구현 객체 → 인터페이스 타입. 구현 객체는 언제나 그 인터페이스의 규격을 만족하므로 항상 안전하다. 그래서 자동으로 변환된다.
- **강제 타입 변환(다운캐스팅)**: 인터페이스 타입 → 구현 클래스 타입. 괄호로 `(구현클래스)`를 명시해야 하며, 실제 객체가 그 타입일 때만 안전하다. 주로 구현 클래스에만 있는 **고유 기능**을 쓰려고 할 때 필요하다.

```java
Vehicle vehicle = new Car();   // [업캐스팅] 자동 변환
vehicle.move();
// vehicle.openTrunk();        // 불가: 인터페이스에는 openTrunk가 없음

Car car = (Car) vehicle;       // [다운캐스팅] 명시적 캐스팅 필요
car.openTrunk();               // 이제 Car 고유 기능 사용 가능
```

다운캐스팅은 실제 객체 타입이 맞지 않으면 실행 중 `ClassCastException`이 난다. 그래서 다운캐스팅 전에 8.12의 `instanceof`로 확인하는 것이 안전하다.

> 💻 실습: `./run.sh InterfaceTypeCasting`

> 📖 업캐스팅·다운캐스팅과 동적 바인딩의 원리는 개념서 [4.5 다형성 — 업캐스팅과 동적 바인딩](../JAVA_교재/심화/04-객체지향이란-무엇인가.md)에서 더 깊이 다룬다.

## 8.11 다형성

다형성(polymorphism)은 **"같은 타입, 같은 호출, 다른 실행 결과"** 를 뜻한다. 인터페이스 타입 하나로 여러 구현 객체를 담고 동일한 메소드를 호출하면, 실제 담긴 객체에 따라 서로 다른 동작이 실행된다.

- **배열 다형성**: 인터페이스 타입 배열에 여러 구현 객체를 함께 담는다.
- **매개변수 다형성**: 메소드가 인터페이스 타입을 받으면 어떤 구현이든 처리한다.

```java
Speaker[] speakers = { new Korean(), new English(), new Japanese() };
for (Speaker speaker : speakers) {
    speaker.speak();   // 같은 호출이지만 실제 객체마다 다른 결과
}

static void greet(Speaker speaker) {   // 어떤 Speaker 구현이든 받는다
    speaker.speak();
}
```

다형성은 인터페이스가 주는 가장 큰 실용적 이득이다. 새로운 구현(예: `Chinese`)이 생겨도 `greet`이나 반복문은 한 줄도 고치지 않는다. 확장에는 열려 있고 변경에는 닫혀 있는(OCP) 설계가 여기서 나온다.

> 💻 실습: `./run.sh InterfacePolymorphism`

> 📖 다형성이 결제 시스템 같은 실무에서 왜 강력한지는 개념서 [4.5 다형성](../JAVA_교재/심화/04-객체지향이란-무엇인가.md)에서 예시와 함께 설명한다.

## 8.12 객체 타입 확인

`instanceof` 연산자는 어떤 객체가 특정 인터페이스/클래스 타입인지 확인한다. 다운캐스팅 전에 확인하면 `ClassCastException`을 예방할 수 있다.

자바 16부터는 **패턴 매칭**이 추가되어, 타입 확인과 캐스팅을 한 번에 처리한다. `if (p instanceof CardPayment card)`가 참이면 `card` 변수에 캐스팅된 객체가 자동으로 담긴다.

```java
for (Payment p : payments) {
    p.pay();
    if (p instanceof CardPayment card) {   // 확인 + 자동 캐스팅
        card.checkLimit();                 // CardPayment 고유 기능 안전하게 호출
    }
}
```

패턴 매칭이 없던 시절에는 `if (p instanceof CardPayment)`로 확인한 뒤 `((CardPayment) p).checkLimit()`처럼 직접 캐스팅해야 했다. 패턴 매칭은 이 중복을 없애고 실수 여지를 줄인다.

> 💻 실습: `./run.sh InterfaceInstanceof`

> 📖 `instanceof` 패턴 매칭은 개념서 [8.3.3 Pattern Matching for instanceof](../JAVA_교재/심화/08-자바의-현대적-기능들.md)에서 자세히 다룬다.

## 8.13 봉인된 인터페이스

`sealed` 인터페이스는 이 인터페이스를 구현·상속할 수 있는 타입을 `permits` 절로 **명시적으로 제한**한다(자바 17+). 나열된 타입만 구현할 수 있어 무분별한 확장을 막는다.

허용된 구현 타입은 반드시 다음 셋 중 하나여야 한다.

- `final`: 더 이상 상속 불가
- `sealed`: 다시 자신의 허용 목록을 가짐
- `non-sealed`: 봉인을 풀어 자유로운 확장 허용

```java
sealed interface Shape permits Circle, Rectangle {   // Circle, Rectangle만 구현 가능
    double area();
}
final class Circle    implements Shape { /* ... */ }
final class Rectangle implements Shape { /* ... */ }

String desc = switch (shape) {          // 봉인되어 모든 경우가 보장됨
    case Circle c    -> "원";
    case Rectangle r -> "직사각형";
    // default 불필요: 컴파일러가 모든 경우가 다뤄졌음을 보장
};
```

봉인의 진짜 힘은 `switch` 패턴 매칭과 만날 때 드러난다. 구현 타입이 `permits`로 고정돼 있으니, 컴파일러는 모든 경우가 다뤄졌는지 검사할 수 있고 `default`가 없어도 안전하다. 나중에 구현 타입을 하나 더 추가하면, 처리하지 않은 `switch`가 컴파일 오류로 드러나 누락을 알려준다.

> 💻 실습: `./run.sh SealedInterface`

> 📖 sealed의 등장 배경과 record·switch와의 조합은 개념서 [8.2 sealed class](../JAVA_교재/심화/08-자바의-현대적-기능들.md)에서 깊이 다룬다.

## ⚠️ 흔한 오해와 함정

- **"인터페이스는 상수를 모아 두는 곳이다."** 아니다. 인터페이스 필드는 자동으로 `public static final`이 되지만, 인터페이스를 상수 모음집으로만 쓰는 것(상수 인터페이스 패턴)은 안티패턴이다. 상수만 필요하면 `enum`이나 `final class`를 써라.

- **"디폴트 메소드가 있으면 인터페이스는 상태(필드)도 가질 수 있다."** 아니다. 디폴트/정적 메소드로 몸통은 가질 수 있어도, 인터페이스는 여전히 인스턴스 필드를 가질 수 없다. 필드는 전부 상수(`static final`)뿐이다. 상태가 필요하면 추상 클래스를 고려하라.

- **"인터페이스 메소드를 구현할 때 접근 제한자를 마음대로 정할 수 있다."** 아니다. 인터페이스 추상 메소드는 암묵적으로 `public`이다. 구현 클래스에서 `void sound()`처럼 default 접근으로 좁히면 "접근 권한을 약화할 수 없다"는 컴파일 오류가 난다. 반드시 `public`으로 열어야 한다.

- **"다운캐스팅은 컴파일만 되면 안전하다."** 아니다. `(Car) vehicle`이 컴파일에 성공해도, 실제 객체가 `Car`가 아니면 실행 중 `ClassCastException`이 터진다. 다운캐스팅 전에는 `instanceof`로 확인하라.

- **"클래스도 다중 상속이 되니 인터페이스도 하나만 구현해야 한다."** 반대다. 클래스는 하나만 `extends`하지만, 인터페이스는 여러 개를 동시에 `implements`할 수 있다(8.8). 인터페이스 다중 구현은 자바가 권장하는 조합 방식이다.

- **"디폴트 메소드가 두 인터페이스에 같은 시그니처로 있으면 알아서 하나가 선택된다."** 아니다. 두 부모 인터페이스가 같은 시그니처의 디폴트 메소드를 가지면 충돌이 발생하고, 구현 클래스가 반드시 재정의해서 어느 쪽을 쓸지(`Parent.super.method()`) 명시해야 한다.

## 요약

- **인터페이스는 규격이다.** "무엇을 할 수 있는가"만 약속하고, "어떻게 하는가"는 구현 클래스에 맡긴다. 이 분리가 규격-구현 분리와 다형성의 토대가 된다. (8.1)
- **선언·구현**: `interface`로 규격을, `implements`로 구현을 선언하며, 추상 메소드는 하나도 빠짐없이 구현해야 한다. 실무에서는 구현 객체를 인터페이스 타입으로 다룬다. (8.2)
- **필드는 모두 `public static final` 상수**이고, 일반 메소드는 모두 `public abstract`로 구현이 강제된다. (8.3, 8.4)
- **인터페이스도 몸통 있는 메소드를 가질 수 있다**: `default`(하위 호환용 기본 구현), `static`(팩토리·유틸리티), `private`(내부 공통 로직 재사용). (8.5, 8.6, 8.7)
- **조합의 자유**: 클래스는 여러 인터페이스를 동시에 구현하고(8.8), 인터페이스는 여러 인터페이스를 동시에 상속한다(8.9). 작은 규격을 조합해 큰 규격을 만든다.
- **타입 변환**: 업캐스팅(구현→인터페이스)은 항상 안전해 자동, 다운캐스팅(인터페이스→구현)은 명시적이고 `instanceof` 확인이 안전하다. (8.10, 8.12)
- **다형성**은 인터페이스가 주는 가장 큰 실용적 이득이다. 새 구현이 늘어도 사용하는 코드는 그대로다(OCP). (8.11)
- **봉인된 인터페이스**(`sealed ... permits`)는 구현 타입을 제한해, `switch` 패턴 매칭에서 컴파일러가 모든 경우를 검사하도록 만든다. (8.13)

> 📖 이어서 읽기: 인터페이스가 왜 "이 책의 심장"인 다형성·추상화와 이어지는지는 개념서 [4장. 객체지향이란 무엇인가?](../JAVA_교재/심화/04-객체지향이란-무엇인가.md)를, 현대적 기능(record·sealed·패턴 매칭)의 큰 그림은 [8장. 자바의 현대적 기능들](../JAVA_교재/심화/08-자바의-현대적-기능들.md)을 참고하자.
