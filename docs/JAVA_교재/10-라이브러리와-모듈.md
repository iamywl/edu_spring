# Chapter 10 라이브러리와 모듈

프로그램이 커지면 모든 코드를 직접 짤 수는 없다. 이미 잘 만들어진 남의 코드를 **가져다 쓰고(라이브러리)**, 내가 만든 코드도 남이 함부로 헤집지 못하게 **큰 단위로 묶어 경계를 긋는(모듈)** 일이 필요해진다. 이 장은 자바가 이 두 가지를 어떻게 다루는지를 배운다.

먼저 짚고 넘어갈 것이 있다. **이 장의 모듈 부분은 "개념 + 예시" 위주로만 다루고, 별도 실행은 생략한다.** 이 프로젝트의 챕터들은 하나의 소스 폴더에 클래스를 나란히 두고 `./run.sh <클래스명>` 으로 실행하는 **평면 컴파일 모델**을 쓴다. 반면 자바의 모듈 시스템(JPMS)은 프로젝트를 여러 개의 독립 모듈로 쪼개고 각 모듈 루트에 `module-info.java`를 두어야 비로소 작동한다. 이 둘은 빌드·실행 구조 자체가 다르다. 그래서 10.2 이후의 모듈 이야기는 실습 대신 `module-info.java` 예시 코드로 문법과 개념을 익히는 데 집중한다. (모듈을 실제로 실행해 보고 싶다면 별도의 모듈형 프로젝트를 만들어 `javac --module-source-path`/`java --module`로 컴파일·실행해야 하며, 이는 이 교재의 범위를 벗어난다.)

이 장에서 배우는 것

- 외부 라이브러리와 JAR가 무엇이고 어떻게 가져다 쓰는가 (10.1)
- 자바 모듈 시스템(JPMS)과 `module-info.java`의 정체 (10.2)
- 응용프로그램을 모듈 단위로 쪼개는 방법 (10.3)
- 모듈을 배포 단위인 모듈 JAR로 묶기 (10.4)
- `exports`로 패키지를 감추고 여는 은닉 (10.5)
- `requires transitive`로 의존을 전달하는 전이 의존 (10.6)
- 여러 모듈을 하나로 묶는 집합 모듈 (10.7)
- `opens`로 리플렉션을 허용하기 (10.8)
- `java.base`를 비롯한 자바 표준 모듈 (10.9)

## 10.1 라이브러리

**라이브러리(library)** 란 "이미 만들어져 있어 가져다 쓸 수 있는 코드 묶음"이다. 날짜 계산, JSON 파싱, 데이터베이스 접속처럼 자주 필요하지만 매번 직접 짜기엔 번거로운 기능을, 남이 미리 만들어 배포해 둔 것이다. 우리는 그 안에 든 클래스와 메서드를 **호출만** 하면 된다.

자바에서 라이브러리의 물리적 배포 형태가 **JAR(Java ARchive)** 다. JAR는 여러 개의 `.class` 파일과 리소스를 하나로 압축한 파일(사실상 zip)로, 확장자가 `.jar`다. 라이브러리를 쓴다는 것은 곧 "그 라이브러리의 JAR를 내 **클래스패스(classpath)** 에 올려 두고, 그 안의 클래스를 `import` 해서 호출한다"는 뜻이다.

실무에서는 JAR를 손으로 내려받지 않고 **빌드 도구(Gradle/Maven)** 로 관리한다. 이 프로젝트의 Spring 챕터들이 바로 그렇다. `build.gradle`에 의존성 한 줄을 적으면 빌드 도구가 원격 저장소(Maven Central 등)에서 해당 JAR와 그것이 또 의존하는 JAR들까지 자동으로 내려받아 클래스패스를 구성한다. 예를 들어 이 프로젝트가 PostgreSQL에 접속할 때 쓰는 JDBC 드라이버도 이런 외부 라이브러리 JAR다(→ 『20장. 데이터베이스 입출력』 20.1).

```gradle
// build.gradle — 외부 라이브러리(JAR) 하나를 의존성으로 선언
dependencies {
    // PostgreSQL JDBC 드라이버: org.postgresql:postgresql 라는 좌표(coordinate)의 JAR
    runtimeOnly 'org.postgresql:postgresql:42.7.3'
    // JSON 처리 라이브러리 예시
    implementation 'com.fasterxml.jackson.core:jackson-databind:2.17.0'
}
```

정리하면, **라이브러리 = 재사용 가능한 코드**, **JAR = 그 코드를 담아 배포하는 압축 파일**, **클래스패스 = JAR들을 찾아 올려 두는 목록**이다. 다음 절부터의 모듈은 이 "여러 JAR를 클래스패스에 던져 두던" 방식의 한계를 넘어서기 위해 등장했다.

## 10.2 모듈

라이브러리를 클래스패스에 늘어놓는 방식에는 오래된 문제가 있었다. 클래스패스는 그저 "클래스를 찾을 위치 목록"일 뿐이라, 어떤 JAR가 어떤 JAR를 필요로 하는지, 어떤 패키지는 감춰야 하는지를 **표현할 방법이 없다**. `public` 클래스는 클래스패스에 올라온 누구에게나 다 보였고, 필요한 JAR가 빠져도 실행 도중에야 `NoClassDefFoundError`로 터졌다.

이를 해결하려고 자바 9부터 **JPMS(Java Platform Module System, 자바 플랫폼 모듈 시스템)** 가 도입되었다. **모듈(module)** 은 "관련된 패키지들을 묶고, 이름을 붙이고, **무엇을 필요로 하며(requires) 무엇을 공개하는지(exports)** 를 명시적으로 선언한 상위 단위"다. 패키지가 클래스의 묶음이라면, 모듈은 패키지의 묶음이면서 그 경계를 규정하는 계약서다.

모듈의 정체는 각 모듈 루트에 놓인 **`module-info.java`** 라는 특별한 파일에 있다. 이 파일이 있으면 그 소스 묶음은 "이름 있는 모듈(named module)"이 된다. 문법의 뼈대는 아래와 같다.

```java
// src/com.edu.app/module-info.java  (모듈 소스 루트에 위치)
module com.edu.app {          // 모듈 이름 (보통 최상위 패키지명을 따른다)
    requires java.sql;        // 이 모듈이 필요로 하는 다른 모듈
    exports com.edu.app.api;  // 바깥에 공개(=사용 허용)할 패키지
}
```

여기서 `module`, `requires`, `exports` 같은 단어는 이 파일 안에서만 의미를 갖는 **제한된 키워드**다. 핵심 감각은 딱 두 개다. **`requires`는 "나는 이것이 필요하다"**, **`exports`는 "나는 이것을 남에게 보여준다"** 이다. 나머지 절(10.5~10.8)은 모두 이 두 축의 변주다.

## 10.3 응용프로그램 모듈화

**모듈화(modularization)** 란 하나의 큰 응용프로그램을, 책임에 따라 여러 개의 모듈로 쪼개고 그 사이의 의존을 명시적으로 잇는 설계다. 예를 들어 도메인 로직을 담은 모듈, 그것을 쓰는 앱 모듈, 공용 유틸 모듈로 나누고, "앱은 도메인을 필요로 한다"를 코드로 못박는 식이다.

이렇게 하면 얻는 것이 분명하다. 첫째, **의존 관계가 코드에 드러난다.** 누가 누구에게 기대는지가 `module-info.java`에 적혀 있으므로, 허용되지 않은 접근은 아예 컴파일이 막힌다. 둘째, **강한 캡슐화**가 생긴다. `exports`로 공개하지 않은 패키지는 `public` 클래스라도 다른 모듈에서 볼 수 없다(10.5). 셋째, **신뢰할 수 있는 구성**이 된다. 필요한 모듈이 없으면 실행 전(컴파일/기동 시점)에 오류가 나므로, 예전처럼 한참 돌다가 `NoClassDefFoundError`로 죽는 일이 줄어든다.

아래는 앱 모듈이 도메인 모듈과 표준 로깅 모듈에 의존하는 모습이다.

```java
// app 모듈: 도메인 모듈과 표준 java.logging 모듈을 필요로 한다
module com.edu.app {
    requires com.edu.domain;   // 내가 만든 다른 모듈에 의존
    requires java.logging;     // 자바 표준 모듈에 의존 (10.9)
}

// domain 모듈: 서비스 패키지만 바깥에 공개하고, 내부 구현 패키지는 감춘다
module com.edu.domain {
    exports com.edu.domain.service;   // 공개
    // com.edu.domain.internal 은 exports 하지 않음 → 외부에서 접근 불가 (10.5)
}
```

핵심은 "모듈을 나눈다 = 경계를 긋는다"이다. 경계가 생기면 각 모듈은 자기 공개 API 뒤에서 내부를 자유롭게 바꿀 수 있다.

## 10.4 모듈 JAR

10.1의 JAR가 "그냥 클래스를 담은 압축 파일"이었다면, **모듈 JAR(modular JAR)** 는 그 JAR 안에 **`module-info.class`(컴파일된 module-info.java)** 가 함께 들어 있는 JAR다. 즉 모듈 JAR는 배포 형태(JAR)는 같지만, 자기가 어떤 모듈이며 무엇을 requires/exports 하는지에 대한 정보를 스스로 지니고 다닌다.

그래서 같은 `.jar` 파일이라도 **어디에 올리느냐**에 따라 취급이 달라진다.

- **모듈 경로(module path, `--module-path`/`-p`)** 에 올리면 → 이름 있는 모듈로 인식되어 `module-info`의 규칙(requires/exports)이 적용된다.
- **클래스패스(`-cp`)** 에 올리면 → 예전처럼 그냥 클래스 묶음으로 다뤄진다(module-info는 무시).

빌드 도구가 모듈 소스를 컴파일하면 `module-info.class`가 결과물에 포함되어 자연스레 모듈 JAR가 만들어진다. 명령행으로 직접 만든다면 대략 아래 형태다(실습이 아니라 형태 참고용).

```bash
# module-info.java를 포함해 컴파일 → 모듈 JAR로 패키징
javac -d out/com.edu.domain $(find src/com.edu.domain -name '*.java')
jar --create --file libs/com.edu.domain.jar -C out/com.edu.domain .
# 실행 시 모듈 경로(-p)에 올리고, 시작 모듈/메인클래스를 --module로 지정
java -p libs --module com.edu.app/com.edu.app.Main
```

핵심 한 줄: **모듈 JAR = JAR + module-info.class**, 그리고 그것을 **모듈 경로**에 올려야 모듈로서 작동한다.

## 10.5 패키지 은닉

모듈의 가장 강력한 성질이 **강한 캡슐화(strong encapsulation)** 다. 규칙은 간단하다. **`exports` 하지 않은 패키지는, 그 안의 클래스가 `public`이어도 다른 모듈에서 접근할 수 없다.** 예전 클래스패스 세상에서 `public`은 사실상 "모두에게 공개"였지만, 모듈 세상에서 `public`은 "같은 모듈 안, 또는 exports된 경우에만 공개"로 의미가 좁아진다.

이 덕분에 우리는 패키지를 **공개 API와 내부 구현으로 나눌 수 있다.** 바깥에 쓰라고 내놓을 패키지만 `exports` 하고, 내부 세부 구현 패키지는 exports 하지 않는다. 그러면 라이브러리 사용자는 공개 API에만 기댈 수 있고, 우리는 감춰 둔 내부 패키지를 마음껏 리팩터링해도 사용자의 코드를 깨뜨리지 않는다. 이것이 **패키지 은닉**이다.

특정 대상에게만 공개하고 싶다면 **한정 exports(`exports ... to ...`)** 를 쓴다. 지정한 모듈에게만 그 패키지를 보여주고 나머지에겐 감춘다.

```java
module com.edu.domain {
    exports com.edu.domain.api;                 // 누구에게나 공개
    exports com.edu.domain.spi to com.edu.app;  // 오직 com.edu.app 에게만 공개(한정 exports)
    // com.edu.domain.internal  → exports 없음 → 완전히 은닉 (public 이어도 외부 접근 불가)
}
```

기억할 점: **exports는 "공개할 패키지를 고르는" 화이트리스트**이며, 적지 않은 패키지는 자동으로 감춰진다.

## 10.6 전이 의존

모듈 A가 모듈 B를 `requires` 하고, B의 공개 API가 **모듈 C의 타입을 반환하거나 매개변수로 노출**한다고 하자. 그러면 A는 B를 쓰기 위해 어쩔 수 없이 C의 타입도 다뤄야 한다. 그런데 A가 `requires C`를 적지 않았다면 A에서는 그 타입을 컴파일할 수 없다. 이 불편을 해결하는 것이 **전이 의존(transitive dependency)** 이다.

**`requires transitive C`** 는 "나(B)는 C를 필요로 하며, **나를 requires 하는 모듈에게도 C를 함께 물려준다(전달한다)**"는 선언이다. 즉 B가 `requires transitive C`라고 적으면, A는 `requires B`만 해도 C를 자동으로 읽을 수 있게 된다. 이를 **의미적 의존(implied readability, 함축된 가독성)** 이라고도 부른다.

언제 쓰는가? **B의 공개 API 시그니처에 C의 타입이 새어 나올 때**다. 반대로 C를 B 내부에서만 쓰고 API에는 드러내지 않는다면 그냥 `requires C`(비전이)로 감춘다.

```java
// B 모듈: 공개 API가 C 모듈의 타입을 노출하므로 C를 전이로 물려준다
module com.edu.domain {
    requires transitive com.edu.common.model;  // 이 모듈을 쓰는 쪽도 model을 자동으로 읽음
    requires com.edu.internal.util;            // 내부에서만 씀 → 비전이(전달 안 함)
    exports com.edu.domain.api;
}

// A 모듈: requires domain 하나로 common.model 까지 함께 딸려온다
module com.edu.app {
    requires com.edu.domain;   // com.edu.common.model 을 따로 requires 하지 않아도 사용 가능
}
```

핵심: **transitive = "내 의존을 나를 쓰는 쪽에 물려준다"**. API에 드러나는 타입의 모듈에만 붙인다.

## 10.7 집합 모듈

때로는 "이 기능을 쓰려면 이 모듈 묶음 전체가 필요해"라고 말하고 싶다. 사용자가 하위 모듈들을 하나하나 `requires` 하게 하는 대신, **여러 모듈을 한꺼번에 물려주는 대표 모듈**을 만들 수 있다. 이것을 **집합 모듈(aggregator module)** 이라고 한다.

집합 모듈은 자기 자신은 별다른 코드(패키지)를 갖지 않고, **`requires transitive`만 여러 줄 나열**한다(10.6의 전이 의존을 묶음 용도로 활용). 사용자는 이 집합 모듈 하나만 `requires` 하면, 그 안에서 전이로 물려주는 모든 하위 모듈을 한 번에 읽을 수 있다. 자바 표준에도 이런 집합 모듈이 있는데, 예전 Java EE 기능을 묶어 두었던 `java.se`가 대표적이다.

```java
// 집합 모듈: 자기 패키지는 없고, 하위 모듈들을 transitive 로 모아 물려준다
module com.edu.platform {
    requires transitive com.edu.domain;
    requires transitive com.edu.web;
    requires transitive com.edu.persistence;
    // exports 절 없음 — 이 모듈은 "묶어서 전달"하는 역할만 한다
}

// 사용하는 쪽: 셋을 각각 requires 할 필요 없이 platform 하나면 된다
module com.edu.app {
    requires com.edu.platform;   // domain, web, persistence 를 한 번에 읽음
}
```

핵심: **집합 모듈 = requires transitive 만 모아 둔, 코드 없는 우산 모듈**이다.

## 10.8 리플렉션 허용

강한 캡슐화(10.5)는 컴파일·실행 시의 일반 접근뿐 아니라 **리플렉션(reflection)** 접근까지 막는다. 리플렉션은 실행 중에 클래스의 필드·메서드를 이름으로 들여다보고 조작하는 기법으로, JPA/Hibernate·Jackson·Spring 같은 프레임워크가 우리 객체를 채우고 읽을 때 내부적으로 쓴다. 그런데 모듈이 감춘 패키지에는 이런 프레임워크의 리플렉션도 접근할 수 없어, `InaccessibleObjectException` 같은 문제가 난다.

이때 쓰는 것이 **`opens`** 다. `exports`가 "**컴파일 시점의 일반 접근**을 연다"면, `opens`는 "**실행 시점의 깊은 리플렉션 접근**을 연다"는 뜻이다. 둘은 여는 문의 종류가 다르다. `opens` 된 패키지는 `private` 멤버까지 리플렉션으로 접근할 수 있게 되므로, 프레임워크가 우리 객체의 private 필드에 값을 주입할 수 있다.

`exports`처럼 `opens ... to ...`로 대상을 특정 모듈로 한정할 수 있고, 모듈 전체를 열려면 선언 앞에 `open module`을 붙인다.

```java
module com.edu.app {
    requires com.fasterxml.jackson.databind;

    // Jackson 이 이 패키지의 객체를 리플렉션으로 직렬화/역직렬화하도록 허용
    opens com.edu.app.dto to com.fasterxml.jackson.databind;

    // 특정 대상 없이 누구에게나 리플렉션 허용도 가능
    // opens com.edu.app.entity;
}

// 모듈 전체의 모든 패키지를 리플렉션에 열고 싶다면:
// open module com.edu.app { ... }
```

핵심: **exports = 정적(컴파일) 접근 개방**, **opens = 동적(런타임 리플렉션) 접근 개방**. 프레임워크가 private 필드를 건드려야 하면 opens가 필요하다.

## 10.9 자바 표준 모듈

자바 9부터는 JDK 자신도 하나의 거대한 덩어리가 아니라 **수십 개의 표준 모듈로 쪼개져** 있다. 우리가 늘 쓰던 `java.util`, `java.io`, `java.time` 같은 패키지들도 이제 특정 표준 모듈 안에 담겨 있다. 모듈 이름은 관례적으로 `java.*`(자바 SE 표준)와 `jdk.*`(JDK 고유 도구/구현)로 나뉜다.

가장 중요한 모듈이 **`java.base`** 다. 이 모듈은 `java.lang`, `java.util`, `java.io`, `java.time` 등 언어의 근간이 되는 패키지를 담으며, **모든 모듈이 자동으로 `requires java.base` 한 것으로 취급**된다. 그래서 `String`이나 `List`를 쓰려고 `java.base`를 적을 필요가 없다. 반면 그 밖의 표준 기능은 명시적으로 요구해야 한다.

- `java.base` — 언어 핵심(자동 의존, 적을 필요 없음)
- `java.sql` — JDBC 등 데이터베이스 접속 API (→ 『20장』에서 JDBC 사용)
- `java.logging` — 표준 로깅
- `java.net.http` — HTTP 클라이언트
- `java.xml`, `java.desktop`(Swing/AWT) 등

```java
module com.edu.app {
    // java.base 는 자동 의존이라 생략 — String, List, ... 은 그냥 쓸 수 있다
    requires java.sql;       // JDBC(DriverManager, Connection ...)를 쓰려면 필요 (20장)
    requires java.logging;   // java.util.logging 을 쓰려면 필요
}
```

어떤 표준 패키지가 어느 모듈에 있는지는 `java --list-modules`로 목록을 보거나 API 문서에서 확인한다. 핵심: **`java.base`는 공짜(자동)로 딸려오고, 나머지 표준 기능은 필요한 만큼 requires 로 골라 담는다.**

## ⚠️ 흔한 오해와 함정

- **"public 이면 어디서든 쓸 수 있다"** — 모듈 세상에서는 아니다. `exports` 하지 않은 패키지의 `public` 클래스는 다른 모듈에서 보이지 않는다(10.5). public은 "같은 모듈 안, 또는 exports된 경우"에만 열린다.
- **"exports 하면 리플렉션도 된다"** — 아니다. `exports`는 컴파일 시점 일반 접근만 연다. JPA/Jackson처럼 private 필드를 리플렉션으로 건드려야 하면 별도로 `opens`가 필요하다(10.8).
- **"의존은 requires 하나면 다 전달된다"** — 아니다. 기본 `requires`는 그 모듈에게만 유효하다. 내 API에 노출되는 타입의 모듈을 남에게 물려주려면 `requires transitive`를 써야 한다(10.6).
- **"모듈 JAR를 클래스패스에 올려도 규칙이 적용된다"** — 아니다. 같은 JAR라도 클래스패스(`-cp`)에 올리면 module-info가 무시되고 옛 방식으로 취급된다. 모듈로 작동시키려면 모듈 경로(`-p`)에 올려야 한다(10.4).
- **"java.util 을 쓰려면 requires 를 적어야 한다"** — 아니다. `java.util`은 `java.base`에 있고, java.base는 모든 모듈이 자동으로 의존한다(10.9). 반면 `java.sql` 등은 명시해야 한다.

## 요약

- **라이브러리**는 재사용 가능한 코드 묶음이고, **JAR**는 그것을 담아 배포하는 압축 파일이며, **클래스패스**는 JAR를 찾아 올려 두는 목록이다. 실무에선 Gradle/Maven이 JAR 의존을 자동 관리한다. (10.1)
- **모듈(JPMS)** 은 패키지들을 묶어 이름을 붙이고 `requires`/`exports`로 경계를 명시한 상위 단위이며, 그 정체는 모듈 루트의 **`module-info.java`** 에 있다. (10.2)
- 응용프로그램을 모듈로 쪼개면 **의존 관계가 코드에 드러나고, 강한 캡슐화와 신뢰할 수 있는 구성**을 얻는다. (10.3)
- **모듈 JAR**는 `module-info.class`를 포함한 JAR이며, **모듈 경로(`-p`)** 에 올려야 모듈로 작동한다(클래스패스에 올리면 옛 방식). (10.4)
- **`exports`** 하지 않은 패키지는 `public`이라도 외부에서 접근 불가 — 공개 API만 골라 열고 내부 구현은 은닉한다. `exports ... to`로 대상을 한정한다. (10.5)
- **`requires transitive`** 는 내 의존을 나를 쓰는 모듈에게 물려주며, API 시그니처에 드러나는 타입의 모듈에 붙인다. (10.6)
- **집합 모듈**은 `requires transitive`만 모아 여러 모듈을 우산처럼 한 번에 물려주는, 코드 없는 모듈이다. (10.7)
- **`opens`** 는 런타임 리플렉션(private 포함) 접근을 여는 선언으로, JPA/Jackson/Spring이 우리 객체를 리플렉션으로 다룰 때 필요하다(`exports`와 여는 문이 다르다). (10.8)
- JDK도 표준 모듈로 쪼개져 있으며, **`java.base`** 는 언어 핵심을 담아 모든 모듈에 자동 의존된다. `java.sql`·`java.logging` 등 나머지는 필요할 때 `requires`로 담는다. (10.9)

> 이 장의 모듈 부분은 이 프로젝트의 평면 컴파일 모델과 실행 구조가 다르므로 **개념·예시 위주**로 다루었다. 리플렉션과 `opens`가 실제로 왜 필요한지는 JPA/Jackson을 쓰는 Spring 챕터에서, 라이브러리로서의 JDBC 드라이버(JAR)는 『20장. 데이터베이스 입출력』에서 이어진다.
