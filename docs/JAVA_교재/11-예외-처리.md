# Chapter 11. 예외 처리

프로그램을 짜다 보면 "정상적으로 잘 돌아가는 경우"만 생각하기 쉽다. 하지만 실제 프로그램은 없는 파일을 열려 하고, 0으로 나누고, 사용자가 숫자 칸에 글자를 적어 넣는 상황을 끊임없이 만난다. 이런 비정상 상황을 아무 대비 없이 두면 프로그램은 그 자리에서 멈춰 버린다.

**예외 처리(Exception Handling)** 는 이런 비정상 상황을 미리 예상하고, 프로그램이 죽지 않고 정상 흐름으로 돌아오도록 만드는 장치다. 이 장에서는 예외가 무엇인지, `try-catch-finally`로 어떻게 잡는지, checked/unchecked 예외를 어떻게 구분해 처리하는지, 자원을 안전하게 닫는 `try-with-resources`, 예외를 호출자에게 떠넘기는 `throws`, 그리고 우리만의 예외를 정의하는 방법까지 차례로 익힌다.

## 11.1 예외와 예외 클래스

프로그램 실행 중 발생하는 비정상 상황은 크게 두 가지로 나뉜다.

- **에러(Error)** : JVM 자체의 심각한 문제(메모리 부족, 스택 오버플로 등). 코드로 복구할 수 없다.
- **예외(Exception)** : 없는 파일 접근, 잘못된 입력처럼 "복구 가능한" 비정상 상황. 코드로 처리하여 정상 흐름으로 되돌릴 수 있다.

자바에서 던질 수 있는 모든 것(에러와 예외)의 최상위 조상은 `Throwable`이다. 계층 구조는 다음과 같다.

```
Object
  └─ Throwable                ← 던질 수 있는 것의 최상위 조상
       ├─ Error               ← 시스템 레벨 심각한 오류 (처리 대상 아님)
       │    ├─ OutOfMemoryError
       │    └─ StackOverflowError
       └─ Exception           ← 예외의 최상위
            ├─ IOException            ┐
            ├─ ClassNotFoundException ├─ checked (일반 예외)
            ├─ ...                    ┘
            └─ RuntimeException       ┐
                 ├─ NullPointerException           │
                 ├─ ArrayIndexOutOfBoundsException ├─ unchecked (실행 예외)
                 └─ ArithmeticException            ┘
```

여기서 두 갈래를 반드시 구분해야 한다.

- **checked 예외(일반 예외)** : `Exception`을 상속하되 `RuntimeException`은 상속하지 않는다. 컴파일러가 처리를 **강제**한다. `try-catch` 또는 `throws`가 없으면 컴파일되지 않는다.
- **unchecked 예외(실행 예외)** : `RuntimeException`의 자식. 컴파일러가 처리를 강제하지 **않는다**. 주로 프로그래머의 부주의(널 참조 등)로 발생한다.

`instanceof`로 이 계층 관계를 직접 확인할 수 있다.

```java
Exception e = new ArithmeticException("0으로 나눔");
System.out.println(e instanceof RuntimeException); // true → unchecked
System.out.println(e instanceof Throwable);        // true → 모든 예외의 조상

Throwable err = new StackOverflowError();
System.out.println(err instanceof Exception);      // false → Error는 예외가 아님
```

## 11.2 예외 처리 코드

예외가 발생해도 프로그램이 갑자기 종료되지 않도록 잡는 문법이 `try-catch-finally`다.

```java
try {
    // 예외가 발생할 수 있는 코드
} catch (예외타입 e) {
    // 예외가 발생했을 때 실행 (복구 / 알림)
} finally {
    // 예외 발생 여부와 상관없이 "항상" 실행 (자원 정리 등)
}
```

실행 흐름은 두 갈래다.

- **예외가 없으면** : `try` 전체 실행 → `finally` 실행 (`catch`는 건너뜀)
- **예외가 있으면** : `try` 도중 중단 → 맞는 `catch` 실행 → `finally` 실행

발생 가능한 예외가 여러 종류이면 **다중 catch**로 종류별로 나열한다. `catch`는 위에서부터 순서대로 검사하므로, 반드시 **자식 예외를 먼저, 부모 예외(`Exception`)를 마지막에** 써야 한다. 부모를 먼저 쓰면 자식 `catch`는 영원히 도달할 수 없어 컴파일 오류가 난다.

```java
try {
    int result = arr[index] / divisor;      // 두 가지 예외 가능
} catch (ArithmeticException e) {            // 자식 (0으로 나눔)
    System.out.println("0으로 나눌 수 없음");
} catch (ArrayIndexOutOfBoundsException e) { // 자식 (범위 초과)
    System.out.println("잘못된 인덱스");
} catch (Exception e) {                      // 부모는 반드시 마지막
    System.out.println("기타 예외");
} finally {
    System.out.println("항상 실행되는 정리 코드");
}
```

## 11.3 예외 종류에 따른 처리

11.1에서 본 checked / unchecked 구분은 단순한 분류가 아니라, **처리 방식(강제성)의 차이**를 만든다.

**(1) 일반 예외 = checked 예외**

- `Exception`을 상속하되 `RuntimeException`은 상속하지 않는다. 예) `IOException`, `ClassNotFoundException`.
- 컴파일러가 처리를 **강제**한다. 반드시 `try-catch`로 처리하거나 `throws`로 떠넘겨야 컴파일된다.
- 개발자가 어쩔 수 없는 외부 요인(파일 없음, 네트워크 끊김 등)에 주로 사용한다.

```java
static void readFileLike() throws java.io.IOException {   // throws로 떠넘기거나
    throw new java.io.IOException("파일을 찾을 수 없습니다");
}
// 호출부: 반드시 try-catch로 잡아야 컴파일됨
try {
    readFileLike();
} catch (java.io.IOException e) {
    System.out.println("처리됨: " + e.getMessage());
}
```

**(2) 실행 예외 = unchecked 예외**

- `RuntimeException`을 상속한다. 예) `NullPointerException`, `NumberFormatException`.
- 컴파일러가 처리를 **강제하지 않는다**. `try-catch` 없이도 컴파일되지만, 발생하면 실행 중 종료된다.
- 근본 원인은 코드의 결함이므로, 잡기보다는 코드 수정으로 예방하는 것이 원칙이다(다만 안전을 위해 잡을 수도 있다).

```java
static int parseNumber(String text) {   // throws 없이도 컴파일됨
    return Integer.parseInt(text);       // "abc"면 NumberFormatException
}
```

| 구분 | checked (일반) | unchecked (실행) |
|------|---------------|-----------------|
| 상속 | `Exception` | `RuntimeException` |
| 컴파일 강제 | O (try-catch / throws 필수) | X |
| 대표 예 | `IOException` | `NullPointerException` |
| 대응 원칙 | 처리 또는 위임 | 코드로 예방 |

## 11.4 리소스 자동 닫기 (try-with-resources)

파일, 소켓, DB 연결 같은 **리소스**는 사용 후 반드시 `close()`로 닫아야 한다. `finally`에서 직접 닫으면 코드가 길어지고, 닫기를 빠뜨리는 실수를 하기 쉽다.

`try-with-resources`는 이 닫기를 자동화한다.

```java
try (자원선언1; 자원선언2) {
    // 자원 사용
}   // → try 블록이 끝나면(정상/예외 무관) 선언한 자원의 close()가 "자동" 호출
```

두 가지 규칙을 기억하자.

- **조건** : `try` 괄호 안에 선언하는 객체는 반드시 `AutoCloseable` 인터페이스를 구현해야 한다. 이 인터페이스의 `close()`가 자동으로 불린다.
- **닫는 순서** : 여러 자원을 선언하면 **선언의 역순**으로 닫힌다(나중에 연 것을 먼저 닫음). 자원 간 의존 관계 때문에 역순 닫기가 안전하다.

```java
static class Resource implements AutoCloseable {
    private final String name;
    Resource(String name) { this.name = name; }
    void use() { System.out.println(name + " 사용"); }
    @Override public void close() { System.out.println(name + " 닫힘"); }
}

try (Resource a = new Resource("A");
     Resource b = new Resource("B")) {
    a.use();
    b.use();
}   // 닫힘 순서: B → A (선언 역순)
```

사용 중 예외가 발생해도 `close()`는 반드시 호출된다는 점이 핵심이다.

> 💻 실습: `./run.sh TryWithResources`

## 11.5 예외 떠넘기기 (throws)

메소드 안에서 발생한 예외를 그 자리에서 직접 처리하지 않고, "나를 호출한 쪽에서 처리하라"고 위임할 수 있다. 이것이 **예외 떠넘기기(`throws`)** 다.

```java
리턴타입 메소드명(...) throws 예외타입 {
    // 이 안에서 발생한 예외를 처리하지 않고 호출자에게 넘긴다
}
```

예외를 잡는(`catch`) 곳이 나올 때까지, 예외는 **호출 스택을 거꾸로 거슬러 올라가며 전파**된다. `main → levelA → levelB → levelC` 순으로 호출했고 `levelC`에서 예외가 나면,

```
levelC  (발생, throws)
  → levelB  (안 잡음, throws)
    → levelA  (안 잡음, throws)
      → main  (여기서 try-catch로 최종 처리)
```

끝까지 아무도 잡지 않으면 프로그램은 비정상 종료된다.

```java
static void levelC() throws TaskException {
    throw new TaskException("levelC에서 발생");
}
static void levelB() throws TaskException { levelC(); }  // 안 잡고 떠넘김
static void levelA() throws TaskException { levelB(); }  // 안 잡고 떠넘김

public static void main(String[] args) {
    try {
        levelA();                       // 전파된 예외가 여기까지 옴
    } catch (TaskException e) {
        System.out.println("최종 처리: " + e.getMessage());
    }
}
```

예외 객체의 `getStackTrace()`를 보면 예외가 거쳐 온 호출 경로를 확인할 수 있다. `throw`(예외를 던짐)와 `throws`(떠넘기겠다고 선언)는 이름이 비슷하지만 역할이 다르니 헷갈리지 말자.

> 💻 실습: `./run.sh ExceptionPropagation`

## 11.6 사용자 정의 예외

자바가 제공하는 표준 예외만으로는 "우리 프로그램만의 의미 있는 오류"를 정확히 표현하기 어렵다. 예를 들어 `잔액 부족`을 단순히 `Exception`으로 던지면, 무엇이 잘못되었는지 이름만으로는 알 수 없다. 이럴 때 예외 클래스를 직접 만든다.

**만드는 방법**

- checked 예외로 만들려면 → `Exception`을 상속 (처리 강제)
- unchecked 예외로 만들려면 → `RuntimeException`을 상속 (처리 강제 X)

**관례**

- 클래스 이름은 `...Exception`으로 끝맺는다.
- 메시지를 받는 생성자 `super(message)`를 제공한다.
- 필요하면 원인 코드, 부족한 금액 같은 추가 필드를 넣어 더 풍부한 정보를 담는다.

```java
// checked: 잔액 부족 → 반드시 처리하게 만들고 싶다 → Exception 상속
static class InsufficientBalanceException extends Exception {
    private final int shortage;                 // 추가 정보
    InsufficientBalanceException(String message, int shortage) {
        super(message);
        this.shortage = shortage;
    }
    int getShortage() { return shortage; }
}

// unchecked: 잘못된 입력 → 프로그래머 실수 성격 → RuntimeException 상속
static class InvalidAmountException extends RuntimeException {
    InvalidAmountException(String message) { super(message); }
}
```

`Account.withdraw()`에서 잘못된 금액이면 `InvalidAmountException`(unchecked)을, 잔액 부족이면 `InsufficientBalanceException`(checked)을 던져, 상황별로 의미 있는 예외를 구분해 처리한다.

## ⚠️ 흔한 오해와 함정

**함정 1. 예외를 삼키지 마라 (빈 catch 블록)**
`catch (Exception e) {}`처럼 예외를 잡고 아무것도 하지 않으면, 문제가 조용히 사라져 나중에 원인을 찾을 수 없게 된다. 최소한 로그를 남기거나 메시지를 출력하자.

**함정 2. `catch (Exception e)` 하나로 뭉뚱그리지 마라**
모든 예외를 부모 타입 하나로 잡으면, 정작 상황별로 다르게 대응할 기회를 잃는다. 구체적인 예외를 먼저 잡고, `Exception`은 진짜 최후의 보루로만 쓴다. 또한 다중 catch에서 부모 타입을 자식보다 먼저 쓰면 컴파일 오류가 난다(11.2).

**함정 3. `finally`에서 `return`하지 마라**
`finally`의 `return`은 `try`나 `catch`에서 던진 예외까지 덮어써 사라지게 만든다. `finally`는 정리(close 등) 용도로만 쓴다.

**함정 4. Error를 잡으려 하지 마라**
`OutOfMemoryError`, `StackOverflowError` 같은 `Error`는 JVM 수준의 회복 불가능한 문제다. `catch (Throwable)`로 억지로 잡아도 프로그램을 정상으로 되돌릴 수 없으니, 잡지 않는 것이 원칙이다(11.1).

**함정 5. 예외를 흐름 제어에 쓰지 마라**
반복 종료나 조건 분기를 예외로 구현하면 코드가 느려지고 읽기 어려워진다. 예외는 "예외적인 상황"에만 쓴다. unchecked 예외는 잡기보다 `if` 검증으로 예방하는 편이 낫다(11.3).

## 확인문제

1. 다음 중 `Throwable`의 직계 자식 두 가지는 무엇이며, 그중 코드로 복구할 수 있는 것은 어느 쪽인가?

2. checked 예외와 unchecked 예외를 상속 관계와 "컴파일러의 강제 여부" 두 관점에서 구분해 설명하라. 각각의 대표 예외도 하나씩 들어라.

3. 다음 다중 catch 코드는 컴파일되지 않는다. 이유를 설명하고 올바르게 고쳐라.
   ```java
   try {
       int r = arr[i] / d;
   } catch (Exception e) {
       System.out.println("기타");
   } catch (ArithmeticException e) {
       System.out.println("0으로 나눔");
   }
   ```

4. `try-with-resources`로 자원을 자동으로 닫으려면, 그 자원 클래스는 어떤 인터페이스를 구현해야 하는가? 또한 자원을 `A, B, C` 순으로 선언했다면 `close()`는 어떤 순서로 호출되는가?

5. `main → levelA → levelB → levelC` 순으로 호출되고 `levelC`에서 checked 예외가 발생했다. `levelC`, `levelB`, `levelA`가 모두 예외를 잡지 않고 `throws`로 떠넘긴다면 이 예외는 최종적으로 어디에서 처리되어야 하는가? 아무도 처리하지 않으면 어떻게 되는가?

## 요약

- **예외 계층** : 모든 예외/에러의 최상위는 `Throwable`이며, 그 아래 `Error`(복구 불가)와 `Exception`(복구 가능)으로 나뉜다. `Exception`은 다시 checked와 `RuntimeException`(unchecked)으로 갈린다. (11.1)
- **try-catch-finally** : `try`에서 예외를 감지하고 `catch`에서 처리하며 `finally`는 항상 실행된다. 다중 catch는 자식 예외를 먼저, 부모를 마지막에 쓴다. (11.2)
- **예외 종류** : checked는 컴파일러가 처리를 강제(`try-catch`/`throws`)하고, unchecked는 강제하지 않으며 코드 수정으로 예방한다. (11.3)
- **try-with-resources** : `AutoCloseable`을 구현한 자원을 `try(...)`에 선언하면 블록 종료 시 선언의 역순으로 `close()`가 자동 호출된다. (11.4)
- **throws** : 예외를 처리하지 않고 호출자에게 떠넘긴다. 예외는 호출 스택을 거슬러 전파되며, 끝까지 잡히지 않으면 프로그램이 비정상 종료된다. (11.5)
- **사용자 정의 예외** : `Exception`(checked) 또는 `RuntimeException`(unchecked)을 상속해 우리 도메인에 맞는 의미 있는 예외를 만든다. (11.6)
