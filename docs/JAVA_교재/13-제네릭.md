# Chapter 13 제네릭

프로그램을 짜다 보면 "동작은 똑같은데 다루는 타입만 다른" 코드를 자주 만난다. 정수를 담는 상자, 문자열을 담는 상자, 학생 객체를 담는 상자는 담는 방식이 모두 같지만 타입만 다르다. 이럴 때마다 클래스를 새로 만들 수는 없다.

과거에는 이 문제를 `Object` 로 해결했다. 무엇이든 `Object` 로 받으면 하나의 코드로 모든 타입을 다룰 수 있기 때문이다. 하지만 `Object` 는 꺼낼 때마다 형변환(캐스팅)이 필요하고, 엉뚱한 타입을 넣어도 컴파일러가 막지 못해 실행 중에야 `ClassCastException` 으로 터지곤 했다.

**제네릭(Generics)** 은 이 문제를 정면으로 해결한다. 타입을 미리 못 박지 않고 `<T>` 같은 **타입 파라미터**로 비워 두었다가, 사용하는 시점에 타입을 지정한다. 그 결과 하나의 코드를 여러 타입에 재사용하면서도, 잘못된 타입 사용을 **컴파일 시점에** 차단할 수 있다. JDK 5부터 도입된 이 기능은 오늘날 컬렉션, 스트림, 함수형 인터페이스 등 자바 표준 라이브러리 곳곳에 스며 있다.

이 장에서는 제네릭이 왜 필요한지부터 시작해, 제네릭 타입과 메소드를 직접 선언하는 법, 타입을 특정 범위로 제한하는 법, 그리고 여러 제네릭 타입을 유연하게 받는 와일드카드까지 차례로 익힌다.

> 📖 개념 배경: 제네릭이 등장한 역사적 맥락과 "타입 안전성"의 의미는 『JAVA 개념서』 [5.5 제네릭: 타입 안전성의 보장](../JAVA_교재/심화/05-데이터를-효율적으로-다루다-컬렉션.md)에서 더 깊이 다룬다.

## 13.1 제네릭이란?

제네릭은 클래스나 메소드에서 사용할 타입을 미리 정하지 않고, 사용하는 시점에 지정하는 기능이다. 왜 필요한지는 제네릭 이전 방식과 비교하면 분명해진다.

제네릭이 없던 시절에는 여러 타입을 담기 위해 `Object` 를 썼다. 무엇이든 담을 수 있어 유연하지만, 꺼낼 때 반드시 캐스팅해야 하고 엉뚱한 타입을 넣어도 컴파일러가 막지 못한다.

```java
// 제네릭 이전: Object 상자
Object value = "Hello";
String s = (String) value;   // 꺼낼 때 캐스팅 필요

Object danger = 100;         // Integer 를 넣었는데
String wrong = (String) danger;  // 컴파일은 통과, 실행 중 ClassCastException!
```

제네릭을 쓰면 타입 파라미터 `T` 로 상자를 만든다. 지정한 타입만 들어가므로 캐스팅이 사라지고, 잘못된 타입은 아예 컴파일되지 않는다.

```java
class GenericBox<T> {
    private T value;
    public void set(T value) { this.value = value; }
    public T get() { return value; }
}

GenericBox<String> box = new GenericBox<>();
box.set("World");
String s = box.get();   // 캐스팅 불필요, 반환 타입이 String 으로 확정
// box.set(100);        // 컴파일 오류! (실행 전에 잡아준다)
```

제네릭의 두 가지 핵심 이점은 **타입 안전성**(컴파일 시점에 잘못된 타입 차단)과 **캐스팅 제거**(가독성·안정성 향상)다.

## 13.2 제네릭 타입

**제네릭 타입**이란 타입을 파라미터로 가지는 클래스나 인터페이스를 말한다. 클래스명 뒤에 `<T>` 를 붙여 선언한다.

```java
class 클래스명<T> { ... }
interface 인터페이스명<T> { ... }
```

타입 파라미터 이름은 아무렇게나 지어도 되지만, 관례상 의미를 담은 한 글자를 쓴다.

| 기호 | 의미 |
|------|------|
| `T` | Type (일반 타입) |
| `E` | Element (컬렉션 원소) |
| `K`, `V` | Key, Value (키-값 쌍) |
| `N` | Number (숫자) |

타입 파라미터는 하나뿐 아니라 `<K, V>` 처럼 여러 개를 지정할 수도 있다.

```java
// 타입 파라미터 하나
class Basket<T> implements Container<T> {
    private T item;
    public void put(T item) { this.item = item; }
    public T take() { return item; }
}

// 타입 파라미터 여럿 <K, V>
class Pair<K, V> {
    private final K key;
    private final V value;
    public Pair(K key, V value) { this.key = key; this.value = value; }
    public K getKey()   { return key; }
    public V getValue() { return value; }
}

Basket<String> b = new Basket<>();
b.put("사과");
String fruit = b.take();                    // 캐스팅 불필요

Pair<String, Integer> age = new Pair<>("나이", 20);
```

하나의 제네릭 클래스를 `Basket<String>`, `Basket<Integer>` 처럼 여러 타입으로 재사용할 수 있고, `Pair<String, Pair<String, Integer>>` 처럼 서로 중첩(조합)할 수도 있다.

## 13.3 제네릭 메소드

클래스 전체가 제네릭이 아니어도, **메소드 하나**에 대해서만 타입 파라미터를 선언할 수 있다. 이것이 **제네릭 메소드**다.

```java
[제어자] <타입파라미터...> 반환타입 메소드명(파라미터...) { ... }
//        └ 반환타입 앞에 타입 파라미터를 먼저 선언한다
```

핵심은 반환 타입 **앞에** `<T>` 를 먼저 선언한다는 점이다.

```java
static <T> void printAll(T[] arr) {         // 배열 출력
    for (T item : arr) System.out.print(item + " ");
}

static <T> T firstOf(T a, T b) {            // 반환 타입도 T
    return a;
}

static <K, V> void printEntry(K key, V value) {  // 타입 파라미터 여럿
    System.out.println(key + " -> " + value);
}
```

**타입 추론(Type Inference)** 덕분에 호출할 때 타입을 명시할 필요가 거의 없다. 컴파일러가 인자의 타입을 보고 `T` 를 자동으로 알아낸다.

```java
Integer[] nums = { 10, 20, 30 };
printAll(nums);                   // T = Integer 로 추론
String s = firstOf("첫째", "둘째");  // T = String 으로 추론

// 필요하면 명시적으로 지정할 수도 있다(보통 생략)
String x = GenericMethod.<String>firstOf("명시", "지정");
```

자바 표준 라이브러리의 `List.of(...)` 같은 팩토리 메소드도 내부적으로 제네릭 메소드이며, 인자로부터 타입을 추론한다.

> 💻 실습: `./run.sh GenericMethod`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/GenericMethod.java`

## 13.4 제한된 타입 파라미터

타입 파라미터에 아무 타입이나 오면 곤란한 경우가 있다. "숫자만" 받고 싶거나 "서로 비교 가능한 타입만" 받고 싶을 때다. 이때 `extends` 키워드로 **상한 경계(upper bound)** 를 지정한다.

```java
<T extends 상위타입>
//  → T 는 상위타입(또는 그 자손)만 될 수 있다
//  → 메소드 안에서 T 를 상위타입의 멤버처럼 사용할 수 있다
```

경계를 지정하면 얻는 이점은 두 가지다. 받을 수 있는 타입을 제한하는 것과, **그 상위 타입의 메소드를 T 에 대해 호출할 수 있게** 되는 것이다.

```java
// 숫자 타입만: Number 의 doubleValue() 사용 가능
static <T extends Number> double sum(T[] numbers) {
    double total = 0.0;
    for (T n : numbers) total += n.doubleValue();
    return total;
}

// 비교 가능한 타입만: Comparable 의 compareTo() 사용 가능
static <T extends Comparable<T>> T max(T[] arr) {
    T best = arr[0];
    for (T cur : arr)
        if (cur.compareTo(best) > 0) best = cur;
    return best;
}

sum(new Integer[]{1, 2, 3});          // OK
// sum(new String[]{"a"});            // 컴파일 오류! String 은 Number 가 아님
max(new String[]{"banana", "apple"}); // OK, String 은 Comparable 구현
```

주의할 점은, 인터페이스를 경계로 지정할 때도 `implements` 가 아니라 `extends` 를 쓴다는 것이다. 사용자가 직접 만든 클래스라도 `Comparable` 을 구현하면 `max(...)` 같은 경계 메소드에 넣을 수 있다.

## 13.5 와일드카드

제네릭은 기본적으로 **불공변(invariant)** 이다. 즉 `Integer` 가 `Number` 의 자손이더라도, `List<Integer>` 는 `List<Number>` 의 하위 타입이 **아니다**. 그래서 "여러 타입의 제네릭 객체"를 한 메소드로 받으려면 **와일드카드 `?`** 가 필요하다.

와일드카드에는 세 가지 형태가 있다.

| 형태 | 이름 | 의미 | 용도 |
|------|------|------|------|
| `<?>` | 비한정 | 모든 타입. 원소는 `Object` 로만 취급 | 타입 무관하게 다룰 때 |
| `<? extends T>` | 상한 경계 | T 또는 그 자손 | 읽기(생산) |
| `<? super T>` | 하한 경계 | T 또는 그 조상 | 쓰기(소비) |

```java
// <?> : 어떤 리스트든 받아 크기만 확인 (넣기는 불가, null 만 가능)
static void printSize(List<?> list) {
    System.out.println("크기: " + list.size());
}

// <? extends Number> : 생산자 - 꺼내 "읽기"
static double sumOf(List<? extends Number> list) {
    double total = 0.0;
    for (Number n : list) total += n.doubleValue();   // 최소한 Number 임이 보장
    return total;
}

// <? super Integer> : 소비자 - 값을 "넣기"
static void addNumbers(List<? super Integer> list) {
    for (int i = 1; i <= 3; i++) list.add(i);         // Integer 는 반드시 넣을 수 있다
}

sumOf(List.of(10, 20));        // List<Integer> OK
sumOf(List.of(1.5, 2.5));      // List<Double>  OK, 하나의 메소드로 처리
addNumbers(new ArrayList<Number>());  // List<Number> OK
```

언제 `extends` 를 쓰고 언제 `super` 를 쓸지는 **PECS 원칙**으로 기억한다.

> **PECS: Producer-Extends, Consumer-Super**
> - 데이터를 **생산**(꺼내 읽기)하는 곳 → `? extends T`
> - 데이터를 **소비**(집어넣기)하는 곳 → `? super T`
> - 읽기와 쓰기를 **모두** 해야 하면 와일드카드 대신 정확한 타입을 쓴다.

## ⚠️ 흔한 오해와 함정

**1. 타입 소거(Type Erasure) — 제네릭 타입 정보는 런타임에 사라진다**

자바 제네릭은 컴파일 시점에만 타입을 검사하고, 컴파일이 끝나면 타입 정보를 지운다. 이를 **타입 소거**라 한다. 그래서 `List<String>` 과 `List<Integer>` 는 런타임에는 똑같은 `List` 다.

```java
List<String>  a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
System.out.println(a.getClass() == b.getClass());  // true! 런타임엔 구분 안 됨
```

이 때문에 `if (obj instanceof List<String>)` 같은 검사는 불가능하고, `new T[]` 처럼 타입 파라미터로 배열을 만들 수도 없다. 타입 소거의 자세한 원리는 『JAVA 개념서』 [5.9 제네릭 타입 소거(Type Erasure)](../JAVA_교재/심화/05-데이터를-효율적으로-다루다-컬렉션.md)에서 다룬다.

**2. `List<Integer>` 는 `List<Number>` 의 하위 타입이 아니다 (불공변)**

`Integer` 는 `Number` 의 자손이지만, `List<Integer>` 를 `List<Number>` 변수에 대입할 수는 없다. 이것이 허용되면 `List<Number>` 를 통해 `Double` 을 넣어 타입 안전성이 깨지기 때문이다. 여러 타입을 받아야 한다면 13.5의 와일드카드를 쓴다.

**3. 인터페이스 경계에도 `extends` 를 쓴다**

`<T extends Comparable<T>>` 처럼 경계가 인터페이스여도 `implements` 가 아니라 `extends` 를 쓴다. 제네릭 경계 문법에서는 클래스든 인터페이스든 모두 `extends` 로 통일한다.

**4. 원시 타입(raw type)을 쓰면 제네릭의 이점이 사라진다**

`List list = new ArrayList();` 처럼 타입 파라미터를 생략하면 컴파일러가 경고를 내며, 타입 검사도 사라진다. 항상 `List<String>` 처럼 타입을 명시하자.

**5. `<? extends T>` 로 받은 컬렉션에는 넣을 수 없다**

`List<? extends Number>` 는 "Number 의 어떤 자손 리스트인지 모른다". `Integer` 리스트일 수도 `Double` 리스트일 수도 있으니, 무엇을 넣어도 안전하지 않아 컴파일러가 `add` 를 막는다(읽기 전용). 그래서 PECS에서 extends는 생산(읽기) 전용이다.

**과제 1. 제네릭 클래스 `Box<T>` 작성 (쉬움)**

타입 파라미터 하나를 가진 제네릭 클래스 `Box<T>`를 선언하라. `Box`는 `private T value;` 필드와 `void set(T value)`, `T get()` 메소드를 갖는다. `main`에서 `Box<String>`에 문자열을 담아 **캐스팅 없이** 꺼내 출력하고, 같은 클래스를 `Box<Integer>`로 재사용해 정수를 담아 꺼내 출력하라. 하나의 클래스를 여러 타입으로 재사용하면서도 꺼낼 때 형변환이 필요 없다는 점을 코드로 확인하는 것이 핵심이다.

- 힌트: `class Box<T> { private T value; void set(T v){...} T get(){...} }`. 생성은 `new Box<String>()` 또는 다이아몬드 `new Box<>()`. `box.set(100)`처럼 다른 타입을 넣으려 하면 컴파일 오류가 나는지도 주석으로 실험해 보라.
> 답안 비교: `./run.sh GenericType`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/GenericType.java`

**과제 2. 여러 타입 파라미터 `Pair<K, V>` 작성 (쉬움→응용)**

서로 다른 두 타입 파라미터 `<K, V>`를 가진 제네릭 클래스 `Pair<K, V>`를 선언하라. `final K key`, `final V value` 필드와 이를 받는 생성자, `getKey()`, `getValue()`, 그리고 `(key = value)` 형태로 출력하는 `toString()`을 둔다. `main`에서 `Pair<String, Integer>`(예: 이름-나이)와 `Pair<Integer, String>`(예: 등수-메달)을 각각 만들어 출력하라. 여력이 되면 `Pair<String, Pair<String, Integer>>`처럼 제네릭 타입을 **중첩(조합)**해 안쪽 값을 꺼내 보라.

- 힌트: `class Pair<K, V> { private final K key; private final V value; Pair(K k, V v){...} ... }`. 타입 파라미터는 콤마로 여러 개 선언한다. 중첩은 `new Pair<>("학생", new Pair<>("점수", 95))`.
> 답안 비교: `./run.sh GenericType`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/GenericType.java`

**과제 3. 제네릭 메소드 작성 (응용)**

클래스가 제네릭이 아니어도 메소드 단위로 제네릭을 쓸 수 있음을 확인한다. 다음 세 제네릭 메소드를 직접 선언하라. ① `static <T> void printAll(T[] arr)` — 배열의 모든 원소를 한 줄로 출력. ② `static <T> T firstOf(T a, T b)` — 두 값 중 앞의 것을 그대로 반환(반환 타입도 `T`). ③ `static <K, V> void printEntry(K key, V value)` — 서로 다른 두 타입 파라미터를 받아 `key -> value` 로 출력. `main`에서 `Integer[]`와 `String[]`로 `printAll`을, 문자열/정수로 `firstOf`를 호출해 **타입 추론**(타입을 명시하지 않아도 컴파일러가 `T`를 알아냄)을 확인하고, `printEntry`도 호출하라.

- 힌트: 핵심은 반환 타입 **앞에** `<T>`를 먼저 선언하는 것 — `static <T> T firstOf(T a, T b)`. 호출 시 타입은 보통 생략하지만, `GenericMethod.<String>firstOf("명시", "지정")`처럼 명시할 수도 있다.
> 답안 비교: `./run.sh GenericMethod`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/GenericMethod.java`

**과제 4. 상한 경계 `<T extends Number>` / `<T extends Comparable<T>>` 작성 (응용)**

타입 파라미터를 특정 범위로 제한하는 두 메소드를 작성하라. ① `static <T extends Number> double sum(T[] numbers)` — 숫자 배열을 받아 `doubleValue()`로 합계를 구한다. ② `static <T extends Comparable<T>> T max(T[] arr)` — 비교 가능한 배열을 받아 `compareTo()`로 최댓값을 구한다. `main`에서 `Integer[]`, `Double[]`로 `sum`을, `Integer[]`와 `String[]`로 `max`를 호출하라. 여력이 되면 `Comparable<Student>`를 **직접 구현한** `Student`(이름·점수) 배열을 만들어 `max`에 넣어 최고 점수 학생을 찾아보라.

- 힌트: 경계를 지정하면 메소드 안에서 그 상위 타입의 메소드(`doubleValue`, `compareTo`)를 `T`에 대해 호출할 수 있다. 인터페이스 경계에도 `implements`가 아니라 **`extends`**를 쓴다. `String[]`을 `sum`에 넣으면 왜 컴파일 오류인지 주석으로 남겨 보라.
> 답안 비교: `./run.sh BoundedType`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/BoundedType.java`

**과제 5. 와일드카드와 PECS 작성 (응용)**

와일드카드 세 형태를 직접 써서 불공변성을 우회하라. ① `static void printSize(List<?> list)` — 비한정 와일드카드로 어떤 리스트든 받아 크기를 출력. ② `static double sumOf(List<? extends Number> list)` — **생산자(Producer-Extends)**, 리스트에서 값을 꺼내 읽어 합계를 구한다. ③ `static void addNumbers(List<? super Integer> list)` — **소비자(Consumer-Super)**, 리스트에 정수 1~3을 집어넣는다. `main`에서 `List<Integer>`·`List<Double>`을 `sumOf`로(하나의 메소드로 둘 다 처리됨을), `List<Integer>`·`List<Number>`·`List<Object>`를 `addNumbers`로 처리해 보라.

- 힌트: **PECS** — 꺼내 읽기(생산)만 하면 `? extends T`, 집어넣기(소비)만 하면 `? super T`. `<? extends Number>`로 받은 리스트에는 `add`가 왜 막히는지(읽기 전용), `<? super Integer>` 리스트에는 왜 `Integer`를 넣을 수 있는지 주석으로 정리해 보라.
> 답안 비교: `./run.sh WildcardType`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/WildcardType.java`

**과제 6. 제네릭 인터페이스와 그 구현 클래스 작성 (응용)**

제네릭 **인터페이스** `Container<T>`(메소드 `void put(T item)`, `T take()`)를 선언하고, 이를 구현하는 제네릭 클래스 `Basket<T> implements Container<T>`를 작성하라. `main`에서 `Container<Double> box = new Basket<>();`처럼 **인터페이스 타입 변수**로 참조해 `3.14`를 넣고 캐스팅 없이 꺼내 출력하라. 제네릭 인터페이스를 구현할 때 타입 파라미터 `<T>`가 인터페이스 → 구현 클래스로 그대로 전달되는 구조를 손으로 확인하는 것이 핵심이다.

- 힌트: `interface Container<T> { void put(T item); T take(); }` 와 `class Basket<T> implements Container<T> { ... }`. 구현 클래스도 `<T>`를 선언해야 인터페이스의 `T`와 연결된다. `Container<Double>`로 참조해도 실제 객체는 `Basket<Double>`이다.
> 답안 비교: `./run.sh GenericType`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/GenericType.java`

**과제 7. 두 타입 파라미터 제네릭 메소드 `contains` 작성 (응용)**

배열과 찾을 값을 받아 존재 여부를 반환하는 제네릭 메소드 `static <T> boolean contains(T[] arr, T target)`를 작성하라. 내부에서 `equals`로 각 원소를 비교한다. `main`에서 `Integer[]`에 특정 정수가 있는지, `String[]`에 특정 문자열이 있는지 각각 호출해 `true`/`false`를 출력하라. 파라미터가 둘 다 같은 타입 파라미터 `T`이므로, 배열과 다른 타입의 `target`을 넣으면 컴파일 오류가 나는지도 주석으로 실험해 보라.

- 힌트: `for (T item : arr) if (item.equals(target)) return true;`. 비교에는 `==`가 아니라 `equals`를 쓴다(값 비교). `contains(new Integer[]{1,2,3}, "2")`처럼 타입이 어긋나면 왜 컴파일되지 않는지 주석으로 남겨 보라.
> 답안 비교: `./run.sh GenericMethod`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/GenericMethod.java`

**과제 8. 비한정 와일드카드 `<?>`로 다형적 출력 메소드 작성 (응용)**

어떤 타입의 리스트든 받아 크기와 내용을 출력하는 `static void printSize(List<?> list)`를 작성하라. `main`에서 `List<String>`, `List<Integer>`, `List<Double>`을 각각 넣어 **하나의 메소드로** 모두 처리됨을 확인하라. 그런 다음 메소드 안에서 `list.add(...)`를 시도해 보고(주석 해제), **왜 컴파일 오류인지**(어떤 타입인지 모르므로 `null` 외에는 넣을 수 없음) 주석으로 정리하라. 비한정 와일드카드가 "타입에 무관하게 읽기만" 할 때 쓰는 도구임을 손으로 확인하는 것이 핵심이다.

- 힌트: `static void printSize(List<?> list) { System.out.println(list.size() + " : " + list); }`. `<?>`는 `<? extends Object>`와 사실상 같아, 원소를 꺼내면 `Object`로만 다룰 수 있고 넣기는 `null`만 가능하다. `List<?>`와 제네릭 메소드 `<T> void printSize(List<T> list)`의 차이도 생각해 보라(전자는 타입 이름이 필요 없을 때 더 간결).
> 답안 비교: `./run.sh WildcardType`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch13/WildcardType.java`

## 확인문제

> 학부 중간고사 수준. 개념 서술, 출력 예측, 컴파일 가능 여부 판단, 오류 찾기, 빈칸 채우기, 참·거짓 판별을 고루 담았다. 각 문항의 `정답 보기` 를 열기 전에 먼저 스스로 답을 적어 보라.

**1.** 제네릭이 `Object` 방식에 비해 갖는 두 가지 핵심 이점은 무엇인가? 각 이점을 한 문장으로 서술하라.

<details>
<summary>정답 보기</summary>

① **타입 안전성** — 잘못된 타입 사용을 컴파일 시점에 차단하므로, 실행 중 `ClassCastException` 이 나지 않는다. ② **캐스팅 제거** — 꺼낼 때 형변환이 필요 없어 코드가 간결하고 안전해진다.

</details>

**2.** 다음 코드에서 잘못된 부분을 찾고, 이유를 설명하라.
```java
class Box<T> { T value; }
Box<String> box = new Box<>();
box.value = 100;
```

<details>
<summary>정답 보기</summary>

`box.value = 100;` 이 잘못됐다. `Box<String>` 의 `value` 는 `String` 타입이므로 `Integer` 값 `100` 을 대입하면 **컴파일 오류**다. 제네릭이 실행 전에 타입 오류를 잡아 준 것이며, `Object` 방식이었다면 컴파일은 통과하고 실행 중에야 문제가 드러났을 것이다.

</details>

**3.** 아래 제네릭 메소드 선언에서 빠진 부분을 채워라. (배열의 첫 원소를 반환)
```java
static ______ ______ first(T[] arr) { return arr[0]; }
```

<details>
<summary>정답 보기</summary>

`static <T> T first(T[] arr)` — 첫 빈칸이 `<T>`(타입 파라미터 선언), 둘째 빈칸이 `T`(반환 타입)다. 반환 타입 **앞에** 타입 파라미터 `<T>` 를 먼저 선언해야 한다. 순서를 바꿔 `static T <T> first(...)` 로 쓰면 컴파일 오류다.

</details>

**4.** `<T extends Comparable<T>>` 경계를 지정했을 때, 메소드 안에서 `T` 타입 변수에 대해 어떤 메소드를 호출할 수 있게 되는가? 그리고 그 이유는?

<details>
<summary>정답 보기</summary>

`compareTo(...)` 를 호출할 수 있다. 경계로 `Comparable<T>` 를 지정했으므로 `T` 는 반드시 `Comparable` 을 구현한 타입이고, 따라서 그 인터페이스의 메소드인 `compareTo` 를 안전하게 호출할 수 있다. 경계가 없으면 `T` 는 `Object` 로만 취급되어 `compareTo` 를 부를 수 없다.

</details>

**5.** PECS 원칙에 따라 다음 두 메소드의 빈칸을 `? extends Number` 또는 `? super Integer` 중 알맞은 것으로 채워라.
```java
// 리스트에서 값을 꺼내 합계를 구한다
double sum(List<________> list) { ... }
// 리스트에 정수를 집어넣는다
void fill(List<________> list) { ... }
```

<details>
<summary>정답 보기</summary>

첫 번째는 `? extends Number`(값을 꺼내 읽는 생산자 → Producer-Extends), 두 번째는 `? super Integer`(값을 집어넣는 소비자 → Consumer-Super).

</details>

**6.** (출력 예측) 다음 코드의 출력을 예측하라. 그리고 이 결과가 나오는 원리를 한 단어로 답하라.
```java
List<String>  a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
System.out.println(a.getClass() == b.getClass());
System.out.println(a.getClass().getSimpleName());
```

<details>
<summary>정답 보기</summary>

출력:
```
true
ArrayList
```
`List<String>` 과 `List<Integer>` 는 컴파일 후 **타입 소거(Type Erasure)** 로 타입 파라미터 정보가 지워져, 런타임에는 둘 다 똑같은 `ArrayList` 다. 그래서 `getClass()` 결과가 같고(`true`), 클래스 이름에는 `<String>` 같은 정보가 남지 않는다.

</details>

**7.** (컴파일 가능 여부) 다음 각 문장이 **컴파일되는지** 판단하고, 안 되면 이유를 쓰라. (불공변성 함정)
```java
Integer i = 10;
Number  n = i;                       // (가)
List<Integer> li = new ArrayList<>();
List<Number>  ln = li;               // (나)
```

<details>
<summary>정답 보기</summary>

- **(가) 컴파일 O** — `Integer` 는 `Number` 의 자손이므로, 낱개 값의 대입은 허용된다.
- **(나) 컴파일 X** — 제네릭은 **불공변(invariant)** 이다. `Integer` 가 `Number` 의 자손이더라도 `List<Integer>` 는 `List<Number>` 의 하위 타입이 **아니다**. 만약 이 대입이 허용되면 `ln.add(3.14)`(Double) 로 `List<Integer>` 에 실수를 넣어 타입 안전성이 깨지기 때문에 컴파일러가 막는다.

</details>

**8.** (와일드카드 읽기·쓰기 제약) 다음 메소드 본문에서 컴파일 오류가 나는 줄을 모두 찾고 이유를 설명하라.
```java
static void test(List<? extends Number> list) {
    Number n = list.get(0);   // (가)
    list.add(1);              // (나)
    list.add(null);           // (다)
}
```

<details>
<summary>정답 보기</summary>

- **(가) OK** — `<? extends Number>` 에서 꺼내는 값은 최소한 `Number` 임이 보장되므로 `Number` 로 읽을 수 있다(읽기=생산 가능).
- **(나) 컴파일 오류** — 리스트의 실제 타입이 `List<Integer>` 인지 `List<Double>` 인지 알 수 없으므로, `Integer` 든 무엇이든 넣는 것이 안전하지 않아 컴파일러가 `add` 를 막는다.
- **(다) OK** — `null` 은 어떤 참조 타입에도 안전하게 들어가므로 유일하게 허용된다.

즉 `<? extends T>` 는 사실상 **읽기 전용**이다(PECS의 Producer-Extends).

</details>

**9.** (오류 찾기 — raw type) 다음 코드가 왜 위험한지 설명하고, 실행 시 어떤 예외가 언제 발생하는지 답하라.
```java
List list = new ArrayList();   // 타입 파라미터 생략(raw type)
list.add("hello");
list.add(42);
String s = (String) list.get(1);
```

<details>
<summary>정답 보기</summary>

**원시 타입(raw type)** 을 사용해 타입 검사가 사라진 코드다. `list` 에 `String` 과 `Integer` 가 섞여 들어가도 컴파일러가 막지 못한다. 마지막 줄에서 `list.get(1)`(값 `42`, `Integer`)을 `String` 으로 캐스팅하므로, **실행 시** `ClassCastException` 이 발생한다. 컴파일은 경고만 낼 뿐 통과한다. `List<String>` 처럼 타입을 명시했다면 `list.add(42)` 에서 **컴파일 오류**로 미리 잡혔을 것이다.

</details>

**10.** (오류 찾기 — `new T[]`) 다음 제네릭 메소드는 컴파일되지 않는다. 어느 줄이 문제이며 왜인지 설명하라.
```java
static <T> T[] makeArray(T a, T b) {
    T[] arr = new T[2];   // (?)
    arr[0] = a; arr[1] = b;
    return arr;
}
```

<details>
<summary>정답 보기</summary>

`T[] arr = new T[2];` 가 문제다(`generic array creation` 오류). **타입 소거** 때문에 런타임에는 `T` 가 무엇인지 알 수 없어, `new T[2]` 로 실제 배열을 만들 수 없다. 자바는 배열 생성 시점에 원소의 실제 타입을 알아야 하는데 제네릭 타입 정보는 이미 지워져 있기 때문이다. 회피책으로는 `Object[]` 를 만들어 캐스팅하거나, `T[] arr = (T[]) new Object[2];` 또는 `Array.newInstance(...)` 를 쓴다.

</details>

**11.** (참·거짓) 다음 문장이 참이면 O, 거짓이면 X 로 답하고 **이유**를 쓰라.

> "`<T extends Comparable<T>>` 처럼 경계가 인터페이스일 때는 `extends` 대신 `implements` 를 써야 한다."

<details>
<summary>정답 보기</summary>

**X (거짓)**. 제네릭 경계 문법에서는 경계가 클래스든 인터페이스든 **모두 `extends` 로 통일**한다. `<T implements Comparable<T>>` 는 문법 오류다. 이는 일반 클래스 선언에서 인터페이스를 구현할 때 `implements` 를 쓰는 것과 다른 규칙이다.

</details>

**12.** (참·거짓) 다음 문장이 참이면 O, 거짓이면 X 로 답하고 이유를 쓰라.

> "제네릭 메소드를 호출할 때는 항상 `obj.<Type>method(...)` 처럼 타입을 명시적으로 지정해야 한다."

<details>
<summary>정답 보기</summary>

**X (거짓)**. **타입 추론(Type Inference)** 덕분에 컴파일러가 인자의 타입을 보고 `T` 를 자동으로 알아내므로, 대부분의 경우 타입을 명시하지 않고 그냥 `firstOf("a", "b")` 로 호출한다. 명시적 지정은 추론이 모호하거나 강제하고 싶을 때만 쓰는 선택 사항이다.

</details>

**13.** (개념 서술) **타입 소거(Type Erasure)** 가 무엇인지 설명하고, 그로 인해 **불가능해지는 연산 두 가지**를 예와 함께 들라.

<details>
<summary>정답 보기</summary>

**타입 소거**란, 자바 컴파일러가 컴파일 시점에만 제네릭 타입을 검사하고 컴파일이 끝나면 타입 파라미터 정보를 지워 버리는 것을 말한다(하위 호환성을 위해 도입). 그래서 런타임에는 `List<String>` 과 `List<Integer>` 가 똑같은 `List` 다. 이 때문에 불가능해지는 연산:
- **`new T[]` (제네릭 배열 생성)** — 런타임에 `T` 를 몰라 실제 배열을 만들 수 없다.
- **`obj instanceof List<String>`** — 런타임에 `<String>` 정보가 없어 타입 파라미터까지 검사할 수 없다(`instanceof List<?>` 만 가능).

(그 밖에 `T.class`, `new T()`, 서로 다른 타입 파라미터만 다른 오버로딩 등도 불가능하다.)

</details>

**14.** (개념 서술 — PECS) **PECS 원칙**의 뜻을 풀어 쓰고, 어떤 컬렉션에서 값을 **읽기만** 할 때와 **넣기만** 할 때 각각 어떤 와일드카드를 써야 하는지, 그리고 **읽기와 쓰기를 모두** 해야 할 때는 어떻게 하는지 서술하라.

<details>
<summary>정답 보기</summary>

**PECS = Producer-Extends, Consumer-Super**.
- 컬렉션이 데이터를 **생산**(꺼내 읽기)하는 쪽이면 `<? extends T>` 를 쓴다. 꺼내는 값이 최소한 `T` 임이 보장되어 읽기에 안전하다.
- 컬렉션이 데이터를 **소비**(집어넣기)하는 쪽이면 `<? super T>` 를 쓴다. `T` 를 반드시 받을 수 있어 쓰기에 안전하다.
- **읽기와 쓰기를 모두** 해야 하면 와일드카드를 쓰지 않고 **정확한 타입**(`List<T>`)을 쓴다. 어느 한쪽 와일드카드로는 두 연산을 동시에 안전하게 보장할 수 없기 때문이다.

</details>

**15.** (출력 예측) 다음 코드의 출력을 예측하라. `firstOf` 와 `sum` 은 각각 예제의 `GenericMethod`, `BoundedType` 과 동일한 시그니처를 가진다고 가정한다.
```java
// static <T> T firstOf(T a, T b) { return a; }
// static <T extends Number> double sum(T[] numbers) { ... doubleValue 합계 ... }
System.out.println(firstOf("가", "나"));
System.out.println(firstOf(1, 2) + 10);
System.out.println(sum(new Integer[]{1, 2, 3}));
System.out.println(sum(new Double[]{1.5, 2.5}));
```

<details>
<summary>정답 보기</summary>

출력:
```
가
11
6.0
4.0
```
- `firstOf("가","나")` → 앞의 값 `가`.
- `firstOf(1,2)` → `1`(Integer). `1 + 10 = 11`.
- `sum(Integer[]{1,2,3})` → `doubleValue()` 합계이므로 `double` 인 `6.0`.
- `sum(Double[]{1.5,2.5})` → `4.0`.

`sum` 의 반환 타입이 `double` 이므로 정수 배열을 넣어도 `6` 이 아니라 `6.0` 으로 출력되는 점에 주의한다.

</details>

## 요약

- **제네릭**은 타입을 `<T>` 로 비워 두었다가 사용 시점에 지정하는 기능으로, **타입 안전성**과 **캐스팅 제거**라는 두 이점을 준다. (13.1)
- **제네릭 타입**은 `class 이름<T>` / `interface 이름<T>` 로 선언하며, `<K, V>` 처럼 여러 타입 파라미터도 가질 수 있다. 관례상 `T, E, K, V, N` 을 쓴다. (13.2)
- **제네릭 메소드**는 반환 타입 앞에 `<T>` 를 선언한 메소드로, 클래스가 제네릭이 아니어도 쓸 수 있다. **타입 추론** 덕분에 호출 시 타입을 명시하지 않아도 된다. (13.3)
- **제한된 타입 파라미터** `<T extends 상위타입>` 은 받을 타입을 제한하고, 상위 타입의 메소드를 `T` 에 대해 호출할 수 있게 한다. 인터페이스 경계에도 `extends` 를 쓴다. (13.4)
- **와일드카드** `?` 는 제네릭의 불공변성을 우회한다. `<?>`(비한정), `<? extends T>`(상한·읽기), `<? super T>`(하한·쓰기)의 세 형태가 있고, **PECS**(Producer-Extends, Consumer-Super)로 선택 기준을 기억한다. (13.5)
- 제네릭 타입 정보는 컴파일 후 **타입 소거**로 사라진다. 이 때문에 `new T[]` 나 `instanceof List<String>` 이 불가능하다.
