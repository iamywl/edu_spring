# Chapter 14 멀티 스레드

지금까지 우리가 만든 프로그램은 대부분 **한 줄씩 순서대로** 일하는 하나의 흐름이었다. 하지만 현실의 프로그램은 그렇지 않다. 파일을 내려받으면서 화면을 갱신하고, 여러 사용자의 요청을 동시에 처리하며, 배경에서 자동 저장을 돌린다. 이렇게 **여러 일을 동시에** 처리하려면 실행 흐름이 여러 개 필요하다. 이 흐름 하나하나를 **스레드(thread)**라고 부른다.

하나의 프로그램(프로세스) 안에서 여러 스레드가 함께 도는 것이 **멀티 스레드(multi-thread)**다. 멀티 스레드는 강력하지만, 여러 흐름이 **같은 데이터를 공유**하기 때문에 순서가 꼬이면 값이 깨지는 골치 아픈 문제(경쟁 상태, 가시성 문제)를 일으킨다. 그래서 이 장은 "스레드를 만드는 법"뿐 아니라 "스레드를 **안전하게** 다루는 법"에 무게를 둔다.

이번 장에서는 스레드의 개념부터 시작해 생성·실행·상태·동기화·안전 종료·데몬·스레드풀까지 하나씩 살펴본다. 각 소절에는 실행 가능한 실습 코드가 연결되어 있으니, 개념을 읽은 뒤 직접 실행해 결과를 눈으로 확인하며 익히자.

> 이 장은 자바 API 사용법에 집중한다. "왜 CPU가 스레드를 번갈아 실행하는가", "락과 CAS는 하드웨어 수준에서 어떻게 동작하는가"처럼 더 깊은 원리는 `docs/CS_운영체제_개념서/`를 함께 참고하면 좋다. 각 소절 끝에 관련 문서를 연결해 두었다.

## 14.1 멀티 스레드 개념

**프로세스(process)**는 실행 중인 하나의 프로그램이다. 운영체제로부터 독립된 메모리 공간(코드·힙·스택)을 할당받는다. **스레드(thread)**는 그 프로세스 안에서 실제로 코드를 실행하는 **흐름의 단위**다. 하나의 프로세스는 여러 스레드를 가질 수 있고, 이것이 멀티 스레드다.

여러 스레드가 동시에 일하는 것처럼 보이는 이유는 **동시성(concurrency)** 때문이다. CPU 코어 하나가 여러 스레드를 아주 빠르게 번갈아 실행하면 마치 동시에 도는 것처럼 보인다. 코어가 여러 개라면 실제로 같은 순간에 여러 스레드가 실행되는 **병렬성(parallelism)**이 일어난다. 멀티 스레드가 유용한 대표적인 이유는, 네트워크나 파일 입출력처럼 **대기 시간이 긴 작업**을 기다리는 동안 다른 스레드가 일을 계속할 수 있기 때문이다.

멀티 스레드의 핵심이자 함정은 **메모리 구조**에 있다.

```
[하나의 프로세스]
 ┌─────────────────────────────────────────────┐
 │  힙(heap) / 메서드 영역   ← 모든 스레드가 "공유"  │  ← 동기화 문제의 근원(14.6)
 ├──────────────┬──────────────┬───────────────┤
 │  스레드 A     │  스레드 B     │  스레드 C       │
 │  스택(독립)   │  스택(독립)   │  스택(독립)     │  ← 지역 변수는 서로 간섭 없음
 └──────────────┴──────────────┴───────────────┘
```

- 같은 프로세스의 스레드들은 **힙과 메서드 영역을 공유**한다. 그래서 같은 객체·`static` 변수에 함께 접근할 수 있고, 바로 이 때문에 동기화 문제가 생긴다.
- 반면 각 스레드는 **자기만의 스택**을 가진다. 지역 변수와 메서드 호출 정보는 스레드마다 독립적이라 서로 간섭하지 않는다.

```java
static int sharedHeapValue = 0;   // 힙의 공유 변수: 모든 스레드가 접근

Thread t1 = new Thread(() -> sharedHeapValue += 100);
Thread t2 = new Thread(() -> sharedHeapValue += 200);
t1.start();  t2.start();
t1.join();   t2.join();            // 두 스레드가 끝날 때까지 대기
System.out.println(sharedHeapValue);   // 기대값 300 (공유 힙)
```

## 14.2 메인 스레드

자바 프로그램이 시작되면 JVM은 `main()` 메서드를 실행하기 위해 **메인 스레드(main thread)**를 만든다. 즉 우리가 지금껏 작성해 온 모든 `main()` 코드는 이미 하나의 스레드 위에서 돌고 있었던 셈이다. 메인 스레드가 새 스레드를 만들면 그때부터 프로그램은 여러 스레드를 가지게 된다.

`Thread.currentThread()`로 지금 실행 중인 스레드 객체를 얻어 이름·우선순위·ID 등을 확인할 수 있다.

```java
Thread main = Thread.currentThread();
System.out.println(main.getName());       // 보통 "main"
System.out.println(main.getPriority());   // 기본 5
System.out.println(main.isDaemon());      // false (일반 스레드)
```

여기서 중요한 **종료 조건**을 짚고 넘어가자. 프로그램(프로세스)은 실행 중인 **모든 일반(데몬이 아닌) 스레드가 끝나야** 종료된다. 따라서 메인 스레드가 먼저 `main()`의 끝에 도달해도, 사용자가 만든 일반 스레드가 아직 살아 있으면 프로그램은 끝나지 않는다. (데몬 스레드는 예외인데, 이는 14.8에서 다룬다.)

## 14.3 작업 스레드 생성과 실행

메인 스레드가 만드는 새 스레드를 **작업 스레드(worker thread)**라고 한다. 작업 스레드를 만드는 방법은 크게 세 가지다.

1. **`Thread` 클래스를 상속**받아 `run()`을 재정의한다.
2. **`Runnable` 인터페이스를 구현**해 `Thread`에 전달한다.
3. **람다식**으로 `Runnable`을 간단히 전달한다(자바 8 이상). 실무에서 가장 흔하다.

```java
// (2) Runnable 구현
Thread t = new Thread(new Runnable() {
    @Override public void run() { System.out.println("작업 실행"); }
});

// (3) 람다식 — 위와 같은 일을 훨씬 짧게
Thread t2 = new Thread(() -> System.out.println("작업 실행"));
```

여기서 반드시 구분해야 할 것이 **`start()`와 `run()`의 차이**다. 이 둘을 헷갈리는 것이 초보자의 대표적인 실수다.

- **`run()`을 직접 호출**하면 그냥 평범한 메서드 호출일 뿐이다. **새 스레드가 생기지 않고** 현재 스레드에서 순차적으로 실행된다.
- **`start()`를 호출**해야 JVM에게 새 스레드 생성을 요청한다. 그러면 **새 스레드가** `run()`을 실행한다.

```java
Runnable who = () -> System.out.println(Thread.currentThread().getName());

new Thread(who, "새스레드").run();    // "main" 출력 (새 스레드 안 생김!)
new Thread(who, "새스레드").start();  // "새스레드" 출력 (새 스레드에서 실행)
```

병렬 실행을 원한다면 **반드시 `start()`**를 호출해야 한다.

## 14.4 스레드 이름

모든 스레드에는 **이름**이 있다. 디버깅이나 로그를 볼 때 "어느 스레드가 이 코드를 실행했는가"를 구분하는 데 유용하다.

- `getName()` : 스레드 이름을 얻는다.
- `setName()` : 스레드 이름을 바꾼다.
- 이름을 지정하지 않으면 JVM이 `Thread-0`, `Thread-1`처럼 **기본 이름**을 자동으로 붙인다.
- 생성자 `Thread(Runnable, String)`으로 만들 때 이름을 바로 지정할 수도 있다.
- 메인 스레드의 기본 이름은 `main`이다.

```java
// 생성자로 이름 지정 (권장) — 의미 있는 이름은 로그를 읽기 쉽게 한다
Thread named = new Thread(() -> { /* ... */ }, "결제-처리-스레드");
System.out.println(named.getName());   // 결제-처리-스레드

// 이미 만든 스레드의 이름 변경
Thread t = new Thread(() -> { /* ... */ });
t.setName("주문-처리-스레드");
```

## 14.5 스레드 상태

스레드는 생성부터 소멸까지 여러 **상태(state)**를 거친다. 지금 스레드가 어떤 상태인지는 `Thread.getState()`로 확인할 수 있으며, 값은 `java.lang.Thread.State` 열거형으로 정의되어 있다.

| 상태 | 의미 |
|------|------|
| `NEW` | 스레드 객체를 생성했지만 아직 `start()` 하지 않음 |
| `RUNNABLE` | `start()` 이후 실행 중이거나 실행 대기 중 |
| `BLOCKED` | `synchronized` 락을 얻으려고 기다리는 중 |
| `WAITING` | 시간 제한 없이 대기 (`join()`, `wait()`) |
| `TIMED_WAITING` | 시간 제한이 있는 대기 (`sleep(ms)`, `join(ms)`, `wait(ms)`) |
| `TERMINATED` | `run()`이 끝나 완전히 종료됨 |

상태는 다음과 같이 전이한다. `RUNNABLE`을 중심으로 여러 대기 상태를 오가다가 최종적으로 `TERMINATED`에 도달한다.

```
                    start()                sleep()/wait(ms)/join(ms)
  ┌─────┐  ───────────────▶  ┌──────────┐  ─────────────────────▶  ┌───────────────┐
  │ NEW │                    │ RUNNABLE │                          │ TIMED_WAITING │
  └─────┘                    └──────────┘  ◀─────────────────────  └───────────────┘
                              │  ▲   │        시간 경과 / notify
                    락 획득 실패 │  │ 락 획득  │  wait()/join()
                              ▼  │   │         (제한 없음)
                          ┌─────────┐│         ▼
                          │ BLOCKED ││    ┌─────────┐
                          └─────────┘│    │ WAITING │
                                     │    └─────────┘
                              run() 종료 │
                                     ▼
                              ┌────────────┐
                              │ TERMINATED │
                              └────────────┘
```

```java
Thread t = new Thread(() -> {
    try { Thread.sleep(300); } catch (InterruptedException e) { /* ... */ }
});
System.out.println(t.getState());   // NEW
t.start();
System.out.println(t.getState());   // RUNNABLE
Thread.sleep(100);
System.out.println(t.getState());   // TIMED_WAITING (sleep 중)
t.join();
System.out.println(t.getState());   // TERMINATED
```

## 14.6 스레드 동기화

멀티 스레드의 가장 위험한 함정이 여기에 있다. 여러 스레드가 **하나의 공유 자원을 동시에 수정**하면 결과가 꼬일 수 있는데, 이를 **경쟁 상태(race condition)**라고 한다.

원인은 우리가 무심코 쓰는 `count++`이 **원자적(atomic)이지 않다**는 데 있다. `count++`은 사실 세 단계다.

```
count++  ==  (1) count 값을 읽는다  →  (2) 1을 더한다  →  (3) 다시 저장한다
```

두 스레드가 거의 동시에 (1)에서 같은 값을 읽으면, 각자 1을 더해 같은 값을 저장한다. 결과적으로 **한쪽의 증가가 통째로 사라진다.** 이를 **갱신 손실(lost update)**이라 부른다.

```
초기 count = 10

스레드 A: 읽기(10) ─────────▶ +1 ─────────▶ 쓰기(11)
스레드 B:      읽기(10) ─────────▶ +1 ─────────▶ 쓰기(11)   ← A의 증가가 사라짐!

기대: 12, 실제: 11
```

해결책은 **임계 구역(critical section)**, 즉 공유 자원을 건드리는 코드를 **한 번에 한 스레드만** 실행하도록 막는 것이다. 자바의 가장 기본 도구가 `synchronized`다. `synchronized`가 걸린 메서드/블록은 스레드가 **락(lock)**을 얻어야만 들어갈 수 있고, 한 스레드가 락을 쥐고 있는 동안 다른 스레드는 `BLOCKED` 상태로 기다린다.

```java
class SafeCounter {
    int count = 0;
    synchronized void increment() { count++; }   // 한 번에 한 스레드만 실행
}
```

동기화 없이 4개의 스레드가 각각 5만 번씩 올리면 정답(20만)보다 작은 값이 나오지만, `synchronized`를 적용하면 항상 정확한 값이 나온다.

## 14.7 스레드 안전 종료

돌고 있는 스레드를 밖에서 멈추고 싶을 때가 있다. 예전에는 `Thread.stop()`이 있었지만 **절대 쓰면 안 된다(deprecated).** `stop()`은 스레드를 강제로 즉시 죽이는데, 이때 잡고 있던 락이 풀리고 공유 객체가 **중간 상태(불완전한 상태)**로 남아 데이터가 깨질 수 있기 때문이다.

안전한 방법의 핵심은 **"밖에서 죽이는 것"이 아니라 "스스로 마무리하고 종료하도록 요청하는 것"**이다. 두 가지 방식이 있다.

**(1) `volatile` 플래그 방식** — 스레드가 반복문 조건으로 `boolean` 플래그를 검사하게 하고, 외부에서 그 플래그를 `false`로 바꾸면 스스로 루프를 빠져나가 정상 종료한다. 이때 플래그를 `volatile`로 선언해야, 한 스레드가 바꾼 값이 다른 스레드에 **즉시 보이도록** 보장된다.

```java
static volatile boolean running = true;

Thread worker = new Thread(() -> {
    while (running) { /* 반복 작업 */ }   // 플래그가 false 되면 루프 종료
    System.out.println("정리 후 정상 종료");
});
worker.start();
Thread.sleep(200);
running = false;        // 종료 요청 (강제로 죽이는 게 아님)
worker.join();
```

**(2) `interrupt()` 방식** — 스레드에 "중단 요청"을 보낸다. 스레드가 `sleep`/`wait` 중이면 `InterruptedException`이 발생해 빠져나오고, 실행 중이면 `isInterrupted()`로 확인해 스스로 정리 후 종료할 수 있다.

```java
Thread sleeper = new Thread(() -> {
    try {
        while (!Thread.currentThread().isInterrupted()) {
            Thread.sleep(100);   // 인터럽트되면 여기서 예외 발생
        }
    } catch (InterruptedException e) {
        System.out.println("중단 요청 감지 → 정리 후 정상 종료");
    }
});
sleeper.start();
Thread.sleep(250);
sleeper.interrupt();    // 중단 요청
```

두 방식 모두 스레드가 **스스로 마무리한 뒤 종료**하므로 데이터가 깨지지 않는다.

## 14.8 데몬 스레드

**데몬 스레드(daemon thread)**는 다른 일반 스레드를 돕는 **보조 스레드**다. 가비지 컬렉터, 자동 저장, 로그 flush 같은 배경 작업이 대표적이다.

핵심 특징은 **일반(사용자) 스레드가 모두 끝나면 데몬 스레드도 함께 강제 종료된다**는 것이다. 14.2에서 "프로그램은 모든 일반 스레드가 끝나야 종료된다"고 했는데, 데몬 스레드는 이 계산에서 **빠진다.** 즉 메인 스레드가 끝나면, 무한 반복하던 데몬 스레드도 즉시 함께 멈춘다.

지정 방법은 `setDaemon(true)`이며, 반드시 **`start()` 이전**에 호출해야 한다.

```java
Thread daemon = new Thread(() -> {
    while (true) {                 // 일반 스레드였다면 프로그램이 끝나지 않는다
        // 배경 작업...
        try { Thread.sleep(100); } catch (InterruptedException e) { break; }
    }
});
daemon.setDaemon(true);            // start() 이전에! (지정하면 isDaemon() == true)
daemon.start();

Thread.sleep(500);
// 메인 스레드가 끝나면, 무한 반복하던 데몬 스레드도 자동으로 함께 종료된다.
```

무한 루프를 도는 스레드라도 데몬으로 지정하면 프로그램이 멈추지 않고 정상 종료된다.

> 💻 실습: `./run.sh DaemonThread`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/DaemonThread.java`

## 14.9 스레드풀

지금까지는 필요할 때마다 `new Thread(...)`로 스레드를 직접 만들었다. 하지만 작업이 많아지면 이 방식은 위험하다. 작업마다 스레드를 새로 만들고 없애는 **비용이 크고**, 요청이 몰리면 스레드가 수천 개로 **폭증**해 시스템이 마비될 수 있다.

**스레드풀(thread pool)**은 미리 만들어 둔 소수의 스레드를 **재사용**하며, 스레드가 아니라 **작업(Task)**만 계속 받아 처리한다. 자바에서는 `ExecutorService`가 이 역할을 한다.

사용 흐름은 다음과 같다.

1. `Executors.newFixedThreadPool(n)`으로 스레드 `n`개짜리 풀을 만든다.
2. `submit(작업)`으로 작업을 제출하면, 결과를 담을 `Future`를 돌려받는다.
3. `Future.get()`으로 결과를 받는다. 작업이 끝날 때까지 기다렸다가 반환한다.
4. `shutdown()`으로 새 작업 접수를 막고, `awaitTermination(제한시간)`으로 진행 중인 작업이 끝날 때까지 기다린다.

```java
ExecutorService pool = Executors.newFixedThreadPool(3);

List<Future<Integer>> futures = new ArrayList<>();
for (int i = 1; i <= 5; i++) {
    final int n = i;
    futures.add(pool.submit(() -> n * n));   // 작업 제출 → Future 수령
}

int sum = 0;
for (Future<Integer> f : futures) sum += f.get();   // 결과 회수(완료까지 대기)

pool.shutdown();                                     // 새 작업 접수 중단
pool.awaitTermination(5, TimeUnit.SECONDS);          // 진행 중 작업 완료 대기
```

`shutdown()` + `awaitTermination()`을 **반드시** 호출해야 풀의 스레드가 정리되어 프로그램이 정상 종료된다. 호출하지 않으면 풀 스레드가 남아 프로그램이 끝나지 않을 수 있다.

> 💻 실습: `./run.sh ThreadPool`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadPool.java`
>
> 더 깊이: 스레드풀·`Future`·`BlockingQueue` 같은 고수준 동시성 도구의 설계 원리는 `CS_운영체제_개념서/07-고수준-동시성-도구.md` 참고.

## ⚠️ 흔한 오해와 함정

- **`run()`을 호출하면 스레드가 생긴다고 착각한다.** `run()`은 그냥 메서드 호출이라 **현재 스레드**에서 실행될 뿐, 새 스레드가 생기지 않는다. 병렬 실행을 원하면 반드시 `start()`를 호출해야 한다.
- **`count++`이 원자적이라고 믿는다.** `count++`은 "읽기 → 증가 → 쓰기" 세 단계라, 동기화 없이 여러 스레드가 실행하면 **갱신 손실**이 발생한다. 공유 변수를 여럿이 수정한다면 `synchronized`나 `AtomicInteger` 같은 보호가 필요하다.
- **`volatile`이 동기화를 대신한다고 오해한다.** `volatile`은 값의 **가시성**(바뀐 값이 즉시 보임)만 보장할 뿐, `count++` 같은 복합 연산의 **원자성**은 보장하지 않는다. "여러 단계로 이루어진 갱신"에는 `synchronized`가 필요하다.
- **`Thread.stop()`으로 스레드를 죽인다.** 강제 종료는 락과 데이터를 깨뜨릴 수 있어 사용 금지다. `volatile` 플래그나 `interrupt()`로 **스스로 종료하도록 요청**해야 한다.
- **`setDaemon(true)`을 `start()` 이후에 호출한다.** 데몬 지정은 반드시 `start()` **이전**에 해야 한다. 이후에 호출하면 `IllegalThreadStateException`이 발생한다.
- **`join()`을 빠뜨려 결과가 준비되기 전에 읽는다.** 작업 스레드가 끝나기 전에 공유 결과를 읽으면 미완성 값을 볼 수 있다. 완료를 기다려야 한다면 `join()`(또는 풀의 `Future.get()`, `awaitTermination()`)으로 대기하자.
- **스레드풀을 `shutdown()` 하지 않는다.** 풀의 스레드는 기본적으로 계속 살아 있어, 종료하지 않으면 프로그램이 끝나지 않는다. 작업을 다 제출했으면 `shutdown()` + `awaitTermination()`으로 정리해야 한다.

## ✍️ 직접 작성해보기

> 먼저 답을 보지 말고 직접 타이핑 → 실행 → 답안 비교. 예제를 읽는 게 아니라, 빈 파일에 스스로 코드를 작성해 완성하는 과제다.

각 과제를 자기 손으로 작성해 컴파일·실행한 뒤, 마지막의 `답안 비교` 명령으로 관련 예제 클래스와 결과·구조를 견주어 보라. 이 장의 주제인 스레드 생성·`join`, 경쟁 상태와 `synchronized`, 데몬 스레드, 스레드풀을 실제로 손끝에서 만들어 보는 것이 목적이다. 스레드를 다루는 과제에는 **한 가지 철칙**이 있다 — **프로그램이 반드시 스스로 끝나야 한다.** 모든 과제는 `join()`(또는 스레드풀의 `awaitTermination(제한시간)`)으로 만든 스레드가 끝날 때까지 기다렸다가 종료하도록 작성하라. `join`을 빠뜨리면 결과를 덜 익은 채로 읽거나, 스레드가 남아 프로그램이 멈추지 않을 수 있다.

**과제 1. 람다로 스레드 만들고 join으로 기다리기 (쉬움)**

`main`에서 `Runnable`을 **람다식**으로 만들어 `Thread`에 전달하고, `start()`로 새 스레드를 실행하라. 그 스레드는 자기 이름(`Thread.currentThread().getName()`)과 함께 "작업 완료" 같은 메시지를 출력한다. 스레드를 만들 때 생성자 `new Thread(작업, "일꾼-스레드")`로 **이름을 지정**하고, `start()` 뒤에는 반드시 **`join()`으로 그 스레드가 끝날 때까지 기다린 다음** `main`에서 "메인 스레드 종료"를 출력하라. `join`이 없으면 메인이 먼저 끝나 출력 순서가 뒤섞일 수 있음을 확인하는 것이 핵심이다.

- 힌트: `Thread t = new Thread(() -> System.out.println(...), "일꾼-스레드"); t.start(); t.join();`. `join()`은 `InterruptedException`을 던지므로 `main`에 `throws InterruptedException`을 붙이거나 `try/catch`로 감싼다. **반드시 `join()`으로 종료를 기다려 프로그램이 스스로 끝나게 하라.**
> 답안 비교: `./run.sh ThreadCreation`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadCreation.java`

**과제 2. 여러 스레드를 배열로 만들어 모두 join하기 (쉬움→응용)**

같은 작업을 하는 스레드를 **4개** 만들어 `Thread[]` 배열에 담고, 반복문으로 모두 `start()`한 뒤 다시 반복문으로 모두 `join()`하라. 각 스레드는 0부터 999까지 더한 부분합을 계산해 자기 이름과 함께 출력한다. 모든 스레드를 먼저 다 띄우고(`start`), 그다음 전부 기다리는(`join`) **두 단계 구조**를 지키는 것이 핵심이다. 마지막에 `main`에서 "4개 스레드 모두 종료" 를 출력하라.

- 힌트: `Thread[] ts = new Thread[4];` → 첫 반복문에서 `ts[i] = new Thread(task); ts[i].start();`, 둘째 반복문에서 `for (Thread t : ts) t.join();`. start와 join을 **한 반복문에 섞으면** 사실상 순차 실행이 되니 반드시 반복문을 나눠라. 모든 스레드를 join해 프로그램이 스스로 끝나게 하라.
> 답안 비교: `./run.sh ThreadSync`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadSync.java`

**과제 3. 경쟁 상태 재현 후 synchronized로 고치기 (응용)**

`int count` 필드와 `increment()` 메서드를 가진 카운터 클래스를 두 벌 만들라 — 하나는 `synchronized` **없이**(`UnsafeCounter`), 하나는 `synchronized` **붙여서**(`SafeCounter`). 각 카운터를 스레드 4개가 각각 50,000번씩 `increment()`하게 하고, 모든 스레드를 `join`한 뒤 결과를 정답(`4 × 50000 = 200000`)과 비교 출력하라. 동기화 없는 쪽은 정답보다 **작은 값**(갱신 손실)이 나오고, `synchronized`를 붙인 쪽은 **항상 정확**함을 대비로 드러내는 것이 핵심이다.

- 힌트: `synchronized void increment() { count++; }`. `count++`은 "읽기 → 증가 → 쓰기" 3단계라 원자적이지 않다. 스레드 4개를 배열로 만들어 `start` 후 **모두 `join`**해야 최종 `count`를 정확히 읽을 수 있다(과제 2의 두 단계 구조 재사용). join 없이 값을 읽으면 아직 덜 끝난 값을 볼 수 있다.
> 답안 비교: `./run.sh ThreadSync`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadSync.java`

**과제 4. 데몬 스레드로 배경 작업 돌리기 (응용)**

무한 반복(`while (true)`)하며 0.1초마다 "배경 작업 중" 을 출력하는 스레드를 **람다로** 만들라. 이 스레드를 `start()` **이전에** `setDaemon(true)`로 데몬 지정하고, `isDaemon()`이 `true`인지 출력한 뒤 `start()`하라. 그다음 `main`은 `Thread.sleep(500)`으로 약 0.5초만 일하고 그대로 끝난다. **무한 루프를 도는 스레드인데도** 메인이 끝나면 데몬 스레드가 함께 종료되어 프로그램이 멈추지 않고 정상 종료됨을 확인하는 것이 핵심이다.

- 힌트: `t.setDaemon(true)`는 반드시 `t.start()` **이전**에 호출해야 한다(이후면 `IllegalThreadStateException`). 데몬 스레드는 일반 스레드(여기선 main)가 모두 끝나면 자동 종료되므로, 이 과제는 무한 루프여도 `join` 없이 `main`의 `sleep`이 끝나면서 프로그램이 스스로 종료된다 — 데몬이 아니었다면 영원히 멈추지 않는다.
> 답안 비교: `./run.sh DaemonThread`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/DaemonThread.java`

**과제 5. ExecutorService로 작업 제출하고 결과 회수하기 (응용)**

`Executors.newFixedThreadPool(3)`으로 스레드 3개짜리 풀을 만들고, 1부터 5까지의 **제곱을 계산하는 작업 5개**를 `submit`하라. 각 `submit`이 돌려주는 `Future<Integer>`를 리스트에 모아 두었다가 `Future.get()`으로 결과를 회수해 합(`1+4+9+16+25 = 55`)을 출력하라. 마지막에는 반드시 `shutdown()`으로 새 작업 접수를 막고 **`awaitTermination(5, TimeUnit.SECONDS)`로 제한 시간 안에서 종료를 기다려** 프로그램이 스스로 끝나게 하라.

- 힌트: `pool.submit(() -> n * n)`이 `Future<Integer>`를 돌려준다. 람다 안에서 반복 변수를 쓰려면 `final int n = i;`처럼 지역 변수에 복사한다. `Future.get()`은 예외를 던지므로 `try/catch`로 감싼다. **`shutdown()` 뒤 `awaitTermination(제한시간)`을 빠뜨리면 풀 스레드가 남아 프로그램이 끝나지 않는다** — 이 종료 절차가 이 과제의 핵심이다.
> 답안 비교: `./run.sh ThreadPool`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadPool.java`

**과제 6. 스레드 이름 지정하고 확인하기 (쉬움)**

스레드 이름을 다루는 세 가지 경우를 한 프로그램에서 모두 보여라. (1) 메인 스레드의 이름을 `Thread.currentThread().getName()`으로 출력하고, (2) 이름을 **지정하지 않은** 스레드를 만들어 JVM이 붙인 기본 이름(`Thread-0` 형태)을 `getName()`으로 출력하며, (3) 생성자 `new Thread(작업, "결제-처리-스레드")`로 **이름을 지정한** 스레드와 `setName()`으로 **이름을 바꾼** 스레드를 각각 만들어 실행하라. 각 작업 스레드는 자기 이름을 출력한다. 만든 스레드는 모두 `join()`으로 기다린 뒤 `main`에서 "종료"를 출력하라.

- 힌트: 기본 이름은 `start()` 전에도 `getName()`으로 확인할 수 있다. `setName()`은 `start()` 전에 호출한다. 각 스레드를 `start()` 직후 `join()`하면 이름 출력이 순서대로 찍혀 관찰하기 쉽다. **모든 스레드를 `join()`으로 기다려** 프로그램이 스스로 끝나게 하라.
> 답안 비교: `./run.sh ThreadName`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadName.java`

**과제 7. 스레드 상태 전이 관찰하기 (응용)**

한 스레드의 상태가 `NEW → RUNNABLE → TIMED_WAITING → TERMINATED`로 바뀌는 과정을 `getState()`로 찍어 관찰하라. 작업 스레드는 `Thread.sleep(300)`으로 잠깐 잠들도록 만든다. `main`에서 (1) `start()` **이전**에 상태를 출력(기대: `NEW`), (2) `start()` **직후** 출력(기대: `RUNNABLE`), (3) 대상이 `sleep`에 들어갈 시간을 주기 위해 `Thread.sleep(100)` 한 뒤 출력(기대: `TIMED_WAITING`), (4) `join()`으로 끝까지 기다린 뒤 출력(기대: `TERMINATED`)하라. 각 출력에 기대 상태를 함께 적어 실제 값과 비교하라.

- 힌트: 상태를 잡아내려면 **타이밍**이 중요하다. 대상 스레드가 `sleep(300)`에 진입하기 전에 상태를 읽으면 `RUNNABLE`이 보일 수 있으므로, 관찰 전 `main`이 `Thread.sleep(100)`으로 잠깐 양보한다. 마지막 `join()`이 곧 안전한 종료 절차다 — **`join()`으로 대상이 `TERMINATED`가 될 때까지 기다려** 프로그램이 스스로 끝나게 하라.
> 답안 비교: `./run.sh ThreadState`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadState.java`

**과제 8. volatile 플래그로 스레드 안전하게 종료하기 (응용)**

`static volatile boolean running = true;` 플래그를 두고, `while (running)` 동안 반복 작업(예: 0.1초마다 카운트 출력)을 하는 스레드를 만들어라. `main`은 약 0.5초 뒤 `running = false`로 **종료를 요청**하고, `worker.join()`으로 스레드가 스스로 마무리할 때까지 기다린 다음 "정상 종료"를 출력하라. `stop()` 같은 강제 종료 없이, 스레드가 **스스로 루프를 빠져나와** 정리 메시지를 출력하고 끝나는 것을 확인하는 것이 핵심이다.

- 힌트: 플래그는 반드시 **`volatile`**로 선언해야 `main`이 바꾼 값이 작업 스레드에 **즉시 보인다**(가시성). `volatile`이 없으면 작업 스레드가 캐시된 옛 값을 계속 보아 영영 멈추지 않을 수 있다. `interrupt()` 방식으로 구현해도 좋다. **`running = false` 뒤 반드시 `join()`으로 스레드의 종료를 기다려** 프로그램이 스스로 끝나게 하라.
> 답안 비교: `./run.sh ThreadSafeStop`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadSafeStop.java`

## 확인문제

> 개념 서술·출력 예측·오류 찾기·빈칸 채우기·참거짓 판단을 고루 담았다. 먼저 스스로 답을 적은 뒤 `<details>`를 펼쳐 정답·해설과 견주어 보라. 스레드는 **실행 순서가 매번 달라질 수 있는** 특성(비결정성) 때문에, "정확한 출력"보다 "왜 그렇게 되는가"를 설명하는 것이 훨씬 중요하다.

**1. (개념 서술)** 프로세스와 스레드의 차이를 설명하시오. 같은 프로세스의 스레드들이 **공유하는 메모리 영역**과 **각자 독립적으로 가지는 영역**은 각각 무엇이며, 이 구조가 왜 동기화 문제의 원인이 되는지 답하시오.

<details>
<summary>정답·해설</summary>

**프로세스**는 실행 중인 하나의 프로그램으로, 운영체제로부터 **독립된 메모리 공간**을 할당받는다. **스레드**는 그 프로세스 안에서 실제로 코드를 실행하는 **흐름의 단위**이며, 하나의 프로세스는 여러 스레드를 가질 수 있다.

- **공유 영역**: 같은 프로세스의 스레드들은 **힙(heap)과 메서드 영역**을 공유한다. 그래서 같은 객체와 `static` 변수에 함께 접근할 수 있다.
- **독립 영역**: 각 스레드는 **자기만의 스택(stack)**을 가진다. 지역 변수와 메서드 호출 정보는 스레드마다 독립적이라 서로 간섭하지 않는다.

동기화 문제가 생기는 이유는 바로 **힙 공유** 때문이다. 여러 스레드가 힙의 같은 공유 변수를 동시에 읽고 쓰면 순서가 꼬여 경쟁 상태(race condition)와 갱신 손실이 발생한다. 지역 변수(스택)만 쓴다면 이런 문제는 생기지 않는다.

> 더 깊이: `CS_운영체제_개념서/01-프로세스와-스레드.md`
</details>

**2. (출력 예측 — start vs run)** 다음 두 줄은 서로 무엇이 다른가? 각각 무엇이 출력되며, 어느 스레드에서 실행되는지 설명하시오.
```java
Runnable task = () -> System.out.println(Thread.currentThread().getName());
new Thread(task, "작업").run();
new Thread(task, "작업").start();
```

<details>
<summary>정답·해설</summary>

- 첫째 줄 `run()`은 출력이 **`main`**이다. `run()`은 그냥 평범한 메서드 호출이라 **새 스레드가 생기지 않고** 현재(메인) 스레드에서 순차 실행된다.
- 둘째 줄 `start()`는 출력이 **`작업`**이다. `start()`는 JVM에게 새 스레드 생성을 요청하므로, 이름이 `작업`인 **새 스레드**가 `run()`을 실행한다.

핵심: 병렬 실행을 원한다면 반드시 `start()`를 호출해야 한다. `run()`을 직접 부르면 이름만 붙었을 뿐 실제로는 메인 스레드가 실행한다.
</details>

**3. (개념 서술 — 원자성)** `count++`이 원자적이지 않은 이유를 세 단계로 나누어 설명하고, 두 스레드가 동기화 없이 이 연산을 실행할 때 발생하는 **갱신 손실(lost update)**이 왜 생기는지 서술하시오. 이를 막는 자바 문법은 무엇인가?

<details>
<summary>정답·해설</summary>

`count++`은 사실 **세 단계**로 이루어진다.
1. `count` 값을 **읽는다**.
2. 읽은 값에 **1을 더한다**.
3. 결과를 다시 `count`에 **저장한다**.

두 스레드가 거의 동시에 (1)에서 같은 값(예: 10)을 읽으면, 각자 1을 더해 **둘 다 11을 저장**한다. 두 번 증가했으니 12가 되어야 하지만 결과는 11이 되어 **한쪽의 증가가 통째로 사라진다.** 이것이 갱신 손실이다.

이를 막는 자바 문법은 **`synchronized`**다. `increment()`에 `synchronized`를 붙이면 한 번에 한 스레드만 임계 구역에 들어가므로, 읽기·증가·쓰기가 통째로 보호되어 갱신 손실이 사라진다. (`AtomicInteger`를 쓰는 방법도 있다.)

> 더 깊이: `CS_운영체제_개념서/03-경쟁-상태와-임계-구역.md`, `CS_운영체제_개념서/04-동기화-락-cas-volatile.md`
</details>

**4. (개념 서술 — 안전 종료)** `Thread.stop()`을 사용하면 안 되는 이유를 설명하고, 스레드를 **안전하게 종료**하는 두 가지 방법을 각각 서술하시오. 이때 `volatile` 키워드가 왜 필요한가?

<details>
<summary>정답·해설</summary>

**`stop()`이 금지된 이유**: `stop()`은 스레드를 **강제로 즉시** 죽인다. 이때 그 스레드가 잡고 있던 **락이 갑자기 풀리고**, 공유 객체가 **중간(불완전) 상태**로 남아 데이터가 깨질 수 있다. 그래서 deprecated 되었다.

**안전한 종료 두 가지** — 핵심은 "밖에서 죽이기"가 아니라 "스스로 마무리하고 종료하도록 요청하기"다.
1. **`volatile` 플래그 방식**: 스레드가 `while(running)` 처럼 `boolean` 플래그를 반복문 조건으로 검사하게 하고, 외부에서 `running = false`로 바꾸면 스스로 루프를 빠져나가 정상 종료한다.
2. **`interrupt()` 방식**: 스레드에 "중단 요청"을 보낸다. `sleep`/`wait` 중이면 `InterruptedException`이 발생해 빠져나오고, 실행 중이면 `isInterrupted()`로 확인해 스스로 정리 후 종료한다.

**`volatile`이 필요한 이유**: 한 스레드가 바꾼 플래그 값이 다른 스레드에는 **즉시 보이지 않을 수 있다**(가시성 문제). CPU 캐시에 값이 남아 있으면 종료 요청을 못 볼 수 있다. `volatile`은 값을 항상 메인 메모리에서 읽고 쓰게 하여, 바뀐 값이 **즉시 다른 스레드에 보이도록**(가시성) 보장한다.

> 더 깊이: `CS_운영체제_개념서/05-메모리-가시성과-JMM.md`
</details>

**5. (개념 서술 — 스레드풀)** 필요할 때마다 `new Thread()`로 스레드를 만드는 대신 **스레드풀**을 쓰는 이유를 두 가지 이상 설명하시오. 또한 스레드풀을 다 쓴 뒤 `shutdown()`을 호출하지 않으면 어떤 문제가 생기는가?

<details>
<summary>정답·해설</summary>

**스레드풀을 쓰는 이유**
1. **생성·소멸 비용 절감**: 스레드를 만들고 없애는 데는 비용이 크다. 스레드풀은 미리 만들어 둔 소수의 스레드를 **재사용**하므로 이 비용을 아낀다.
2. **스레드 폭증 방지**: 요청이 몰릴 때마다 `new Thread()`를 하면 스레드가 수천 개로 폭증해 시스템이 마비될 수 있다. 풀은 스레드 개수를 **일정하게 제한**해 자원을 통제한다.
3. (추가) 스레드가 아니라 **작업(Task)**만 제출하면 되므로 코드가 간결하고, `Future`로 결과를 받는 등 고수준 도구를 쓸 수 있다.

**`shutdown()`을 호출하지 않으면**: 풀의 스레드는 기본적으로 작업이 없어도 **계속 살아 있다.** 따라서 `shutdown()`(+`awaitTermination()`)으로 정리하지 않으면 풀 스레드(일반 스레드)가 남아 **프로그램이 끝나지 않는다.**

> 더 깊이: `CS_운영체제_개념서/07-고수준-동시성-도구.md`
</details>

**6. (출력 예측 — 비결정성)** 다음 코드에서 A와 B 중 무엇이 먼저 출력될지 **확정할 수 있는가?** 확정할 수 없다면 그 이유를 설명하고, 무엇을 추가해야 순서를 보장할 수 있는지 답하시오.
```java
new Thread(() -> System.out.println("A"), "t1").start();
System.out.println("B");
```

<details>
<summary>정답·해설</summary>

**확정할 수 없다.** `start()`는 새 스레드에게 실행을 "요청"할 뿐, 그 스레드가 언제 CPU를 배정받아 `A`를 출력할지는 **스케줄러가 결정**한다. 메인 스레드는 `start()` 직후 곧바로 다음 줄로 넘어가 `B`를 출력할 수도 있고, 새 스레드가 먼저 `A`를 찍을 수도 있다. 이렇게 실행 순서가 매번 달라질 수 있는 것이 스레드의 **비결정성(non-determinism)**이다.

순서를 보장하려면 스레드 실행 순서를 명시적으로 조율해야 한다. 예를 들어 `A`를 먼저 보장하려면 새 스레드를 `start()`한 뒤 `join()`으로 끝날 때까지 기다린 다음 `B`를 출력하면 된다.
```java
Thread t = new Thread(() -> System.out.println("A"), "t1");
t.start();
t.join();          // A가 끝날 때까지 대기
System.out.println("B");   // 이제 A → B 순서 보장
```
</details>

**7. (오류 찾기 — 동기화 누락)** 다음 코드는 스레드 4개가 각각 50,000번씩 `increment()`를 호출한다. 기대 결과는 200,000이지만 실행할 때마다 그보다 **작은 값**이 나온다. 무엇이 문제이며, 어떻게 고치는가?
```java
class Counter {
    int count = 0;
    void increment() { count++; }
}
```

<details>
<summary>정답·해설</summary>

**문제**: `increment()`에 **동기화가 없다.** `count++`은 "읽기 → 증가 → 쓰기" 3단계이므로, 여러 스레드가 동시에 실행하면 갱신 손실이 발생해 정답보다 작은 값이 나온다.

**수정**: 메서드에 `synchronized`를 붙여 한 번에 한 스레드만 실행하도록 임계 구역으로 보호한다.
```java
synchronized void increment() { count++; }
```
`AtomicInteger`를 써서 `count.incrementAndGet()`으로 고치는 방법도 있다.

주의: 최종 `count`를 정확히 읽으려면, 4개 스레드를 모두 `join()`한 **뒤에** 값을 읽어야 한다. join 전에 읽으면 아직 덜 끝난 값을 볼 수 있다.
</details>

**8. (오류 찾기 — stop 사용)** 다음 코드의 문제점을 지적하고 안전한 방식으로 고쳐 쓰시오.
```java
Thread worker = new Thread(() -> {
    while (true) { 작업처리(); }
});
worker.start();
Thread.sleep(1000);
worker.stop();   // 1초 뒤 강제 종료
```

<details>
<summary>정답·해설</summary>

**문제점**: `Thread.stop()`은 스레드를 **강제로 즉시 죽여** 잡고 있던 락이 풀리고 공유 데이터가 중간 상태로 깨질 수 있어 **사용 금지**(deprecated)다.

**안전한 수정** — `volatile` 플래그로 스스로 종료하도록 요청한다.
```java
static volatile boolean running = true;

Thread worker = new Thread(() -> {
    while (running) { 작업처리(); }   // 플래그가 false 되면 스스로 종료
    System.out.println("정리 후 정상 종료");
});
worker.start();
Thread.sleep(1000);
running = false;   // 종료 요청 (강제로 죽이는 게 아님)
worker.join();     // 스스로 마무리할 때까지 대기
```
`interrupt()` 방식으로 고쳐도 된다. 플래그는 반드시 **`volatile`**로 선언해야 변경이 다른 스레드에 즉시 보인다.
</details>

**9. (출력 예측 — 데몬 종료 함정)** 다음 코드를 실행하면 "배경 작업 중"이 **몇 번쯤** 출력되고 프로그램은 어떻게 되는가? 만약 `setDaemon(true)` 줄을 **지우면** 어떻게 달라지는가?
```java
Thread t = new Thread(() -> {
    while (true) {
        System.out.println("배경 작업 중");
        try { Thread.sleep(100); } catch (InterruptedException e) { break; }
    }
});
t.setDaemon(true);
t.start();
Thread.sleep(500);
System.out.println("메인 종료");
```

<details>
<summary>정답·해설</summary>

**원래 코드**: 데몬 스레드가 0.1초마다 한 번 출력하고 메인은 약 0.5초 뒤 끝나므로 "배경 작업 중"은 **약 5번**(4~5회, 스케줄링에 따라 편차 있음) 출력된다. 그 뒤 "메인 종료"가 찍히고, **일반 스레드(main)가 끝나면 데몬 스레드도 함께 자동 종료**되어 프로그램은 무한 루프임에도 정상적으로 끝난다.

**`setDaemon(true)`를 지우면**: `t`는 일반 스레드가 된다. 프로그램은 **데몬이 아닌 모든 스레드가 끝나야** 종료되는데, `t`는 `while(true)`라 절대 끝나지 않는다. 따라서 "메인 종료"가 출력된 뒤에도 프로그램은 **영원히 끝나지 않고** "배경 작업 중"을 계속 출력한다.
</details>

**10. (참·거짓 + 이유)** 다음 명제의 참·거짓을 판단하고 **이유**를 서술하시오.
> "`volatile`로 선언한 변수는 여러 스레드가 동시에 `count++` 해도 갱신 손실이 발생하지 않는다."

<details>
<summary>정답·해설</summary>

**거짓.** `volatile`은 **가시성(visibility)**만 보장한다. 즉 한 스레드가 바꾼 값이 다른 스레드에 즉시 보이게 할 뿐이다. 하지만 `count++`은 "읽기 → 증가 → 쓰기"의 **복합 연산**이라, 두 스레드가 같은 값을 읽고 각자 증가시키면 여전히 갱신 손실이 생긴다. `volatile`은 **원자성(atomicity)**을 보장하지 않기 때문이다.

복합 연산의 원자성까지 보장하려면 `synchronized`로 임계 구역을 만들거나 `AtomicInteger`를 써야 한다. 요약하면 **`volatile`은 가시성, `synchronized`는 가시성 + 원자성**을 보장한다.
</details>

**11. (참·거짓 + 이유)** 다음 명제의 참·거짓을 판단하고 **이유**를 서술하시오.
> "`setDaemon(true)`는 `start()` 이후에 호출해도 정상 동작한다."

<details>
<summary>정답·해설</summary>

**거짓.** `setDaemon(true)`는 반드시 **`start()` 이전**에 호출해야 한다. 이미 실행이 시작된(`start()` 이후) 스레드에 대해 호출하면 `IllegalThreadStateException`이 발생한다. 스레드가 데몬인지 여부는 실행 시작 전에 확정되어야 하기 때문이다.
</details>

**12. (빈칸 채우기)** 다음 문장의 빈칸 ㉠~㉤을 채우시오.
> - 스레드 객체를 생성만 하고 아직 `start()`하지 않은 상태는 `( ㉠ )`이다.
> - `Thread.sleep(ms)` 실행으로 진입하는, 시간 제한이 있는 대기 상태는 `( ㉡ )`이다.
> - 다른 스레드가 끝나기를 기다릴 때 호출하는, `InterruptedException`을 던지는 메서드는 `( ㉢ )`이다.
> - `synchronized` 락을 얻지 못해 기다리는 스레드의 상태는 `( ㉣ )`이다.
> - `run()`이 끝나 완전히 종료된 스레드의 상태는 `( ㉤ )`이다.

<details>
<summary>정답·해설</summary>

- ㉠ **`NEW`**
- ㉡ **`TIMED_WAITING`**
- ㉢ **`join()`**
- ㉣ **`BLOCKED`**
- ㉤ **`TERMINATED`**

상태 전이: `NEW → RUNNABLE → (BLOCKED / WAITING / TIMED_WAITING) → TERMINATED`. `getState()`로 확인할 수 있다.

> 관련 실습: `./run.sh ThreadState`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadState.java`
</details>

**13. (빈칸 채우기)** 다음 스레드풀 종료 코드의 빈칸 ㉠~㉢을 채우시오.
```java
ExecutorService pool = Executors.newFixedThreadPool(3);
// ... submit / Future.get() 으로 작업 처리 ...
pool.( ㉠ );                                  // 새 작업 접수 중단
pool.( ㉡ )(5, ( ㉢ ).SECONDS);               // 진행 중 작업 완료를 제한 시간 안에서 대기
```

<details>
<summary>정답·해설</summary>

- ㉠ **`shutdown`**
- ㉡ **`awaitTermination`**
- ㉢ **`TimeUnit`**

즉 `pool.shutdown();` 으로 새 작업 접수를 막고, `pool.awaitTermination(5, TimeUnit.SECONDS);` 로 진행 중인 작업이 끝날 때까지 (최대 5초) 기다린다. 이 두 호출을 빠뜨리면 풀 스레드가 남아 프로그램이 정상 종료되지 않는다.

> 관련 실습: `./run.sh ThreadPool`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadPool.java`
</details>

**14. (개념 서술 — 스레드 이름)** 스레드에 **이름**을 붙이는 방법을 두 가지 이상 제시하고, 이름을 지정하지 않으면 어떤 이름이 붙는지 답하시오. 이름을 붙이는 것이 실무에서 왜 유용한가?

<details>
<summary>정답·해설</summary>

**이름 지정 방법**
1. 생성자 `new Thread(runnable, "이름")` 으로 만들 때 바로 지정한다 (권장).
2. 이미 만든 스레드에 대해 `setName("이름")` 으로 바꾼다.
3. 실행 중인 스레드 이름은 `Thread.currentThread().getName()` 으로 얻는다.

**지정하지 않으면**: JVM이 `Thread-0`, `Thread-1` 처럼 순번이 붙은 **기본 이름**을 자동으로 부여한다. (메인 스레드의 기본 이름은 `main`이다.)

**유용한 이유**: 로그나 스택 트레이스, 디버거에서 "어느 스레드가 이 코드를 실행했는가"를 한눈에 구분할 수 있다. `결제-처리-스레드`처럼 의미 있는 이름을 붙이면 문제 추적이 훨씬 쉬워진다.

> 관련 실습: `./run.sh ThreadName`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch14/ThreadName.java`
</details>

## 요약

- **멀티 스레드**는 한 프로세스 안에서 여러 실행 흐름이 도는 것이다. 스레드들은 **힙을 공유**하고 **스택은 독립**적으로 가지며, 힙 공유가 동기화 문제의 근원이 된다. (`ThreadConcept`)
- **메인 스레드**는 JVM이 `main()` 실행을 위해 만드는 첫 스레드다. 프로그램은 **데몬이 아닌 모든 스레드가 끝나야** 종료된다. (`MainThread`)
- **작업 스레드**는 `Thread` 상속·`Runnable` 구현·람다로 만든다. `run()`은 현재 스레드에서 실행되고, **`start()`만이** 새 스레드를 생성한다. (`ThreadCreation`)
- **스레드 이름**은 `getName()`/`setName()`/생성자로 다룬다. 미지정 시 `Thread-0` 형태의 기본 이름이 붙으며, 의미 있는 이름은 디버깅에 도움이 된다. (`ThreadName`)
- **스레드 상태**는 `NEW → RUNNABLE → (BLOCKED/WAITING/TIMED_WAITING) → TERMINATED`로 전이하며 `getState()`로 확인한다. (`ThreadState`)
- **동기화**는 여러 스레드가 공유 자원을 동시에 고칠 때 생기는 **경쟁 상태·갱신 손실**을 막는다. `synchronized`로 **임계 구역**을 한 번에 한 스레드만 실행하게 한다. (`ThreadSync`)
- **안전 종료**는 `stop()`(금지) 대신 **`volatile` 플래그**나 **`interrupt()`**로 스레드가 스스로 마무리하도록 요청한다. (`ThreadSafeStop`)
- **데몬 스레드**는 배경 보조 스레드로, 일반 스레드가 모두 끝나면 함께 종료된다. `setDaemon(true)`을 **`start()` 이전에** 호출한다. (`DaemonThread`)
- **스레드풀**(`ExecutorService`)은 소수의 스레드를 **재사용**해 작업을 처리한다. `submit`/`Future`로 결과를 받고, **`shutdown()` + `awaitTermination()`**으로 정리해야 정상 종료된다. (`ThreadPool`)
