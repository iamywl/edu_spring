**이 장에서 배우는 것**

- 객체 생성을 축약하는 **생성자 참조** `클래스명::new` (16.6)

## 16.1 람다식이란?

**람다식**은 "메소드(함수) 하나를 식(expression)처럼 간결하게 표현"하는 방법이다. 자바에서 어떤 **동작을 넘겨주려면** 예전에는 **익명 클래스(anonymous class)** 를 써야 했다. 그런데 익명 클래스는 정작 중요한 코드는 한 줄인데도 `new`, 클래스명, 메소드 선언, 중괄호 같은 **껍데기**가 너무 길어 가독성이 떨어진다.

람다식은 이 껍데기를 걷어내고 **`(매개변수) -> 실행할 코드`** 형태만 남긴다. 아래는 같은 동작을 익명 클래스와 람다식으로 각각 구현한 것이다.

```java
@FunctionalInterface
interface Greeting {
    String say(String name);            // 추상 메소드 딱 1개
}

// [1] 익명 클래스 : 껍데기가 길다
Greeting g1 = new Greeting() {
    @Override
    public String say(String name) {
        return "안녕하세요, " + name + "님!";
    }
};

// [2] 람다식 : 핵심 코드만 남긴다
Greeting g2 = (String name) -> { return "반갑습니다, " + name + "님!"; };

// [3] 더 축약 : 타입 생략 + 문장 하나면 중괄호/return 생략
Greeting g3 = name -> "환영합니다, " + name + "님!";
```

람다식이 담기는 그릇은 **함수형 인터페이스(functional interface)** 다. 이는 **추상 메소드가 딱 1개**인 인터페이스로, 람다식은 바로 그 유일한 추상 메소드의 구현을 대신한다. `@FunctionalInterface` 어노테이션을 붙이면 컴파일러가 "추상 메소드 1개" 규칙을 검사해 준다(필수는 아니지만 권장). 자바가 미리 제공하는 표준 함수형 인터페이스도 많은데, 예컨대 `Supplier<T>`는 매개변수 없이 값 하나를 공급한다.

## 16.2 매개변수 없는 람다식

함수형 인터페이스의 추상 메소드가 **매개변수를 받지 않을** 때는, 람다식의 매개변수 자리를 **빈 괄호 `()`** 로 표현한다.

```java
() -> 실행할 코드
```

실행문이 하나면 중괄호와 `return`을 생략할 수 있고, 여러 줄이면 중괄호 `{ }`로 감싸고 `return`을 명시한다.

```java
@FunctionalInterface
interface MessageMaker { String make(); }   // 매개변수 없음, 리턴 있음

MessageMaker m1 = () -> "매개변수 없는 람다식입니다.";   // 한 줄
MessageMaker m2 = () -> {                                // 여러 줄
    String part1 = "여러 줄";
    String part2 = "실행 가능";
    return part1 + " " + part2;
};
```

대표적인 표준 함수형 인터페이스 두 가지를 알아 두면 좋다. **`Runnable`** 은 `void run()` — 입력도 리턴도 없이 **동작만** 수행한다. **`Supplier<T>`** 는 `T get()` — 입력은 없지만 값 하나를 **공급**한다.

```java
Runnable task = () -> System.out.println("동작만 수행");
Supplier<Integer> six = () -> 6;   // 항상 6을 공급
```

## 16.3 매개변수 있는 람다식

추상 메소드가 매개변수를 받을 때는 괄호 안에 매개변수를 쓴다. `(x) -> ...`는 1개, `(x, y) -> ...`는 2개다. 여기에 세 가지 **축약 규칙**이 적용된다.

- 매개변수 **타입은 대부분 생략**할 수 있다(컴파일러가 문맥으로 추론).
- 매개변수가 **정확히 1개**이면 **괄호도 생략**할 수 있다. 예) `x -> ...`
- 매개변수가 **0개이거나 2개 이상**이면 괄호는 **반드시** 필요하다.

```java
@FunctionalInterface interface IntPrinter { void print(int value); }

IntPrinter p1 = (int value) -> System.out.println("받은 값: " + value); // 타입 명시
IntPrinter p2 = value -> System.out.println("제곱: " + value * value);  // 타입/괄호 생략
```

값을 **소비만 하고 리턴하지 않는** 표준 함수형 인터페이스로 **`Consumer<T>`** (`void accept(T)`, 입력 1개)와 **`BiConsumer<T, U>`** (`void accept(T, U)`, 입력 2개)가 있다.

```java
Consumer<String> shout = text -> System.out.println(text.toUpperCase() + "!!!");
BiConsumer<String, Integer> repeat = (word, count) ->
        System.out.println(word + " x " + count);
```

## 16.4 리턴값 있는 람다식

람다식이 값을 리턴하는 방법은 두 가지다.

1. **중괄호 `{ }`** 를 쓰고 그 안에서 **명시적으로 `return`** 한다. → `(a, b) -> { return a + b; }`
2. **중괄호 없이 식(expression) 하나**만 쓴다. 이때는 그 식의 결과가 **자동으로 리턴**되며 `return` 키워드를 **쓰지 않는다(써서는 안 된다).** → `(a, b) -> a + b`

```java
@FunctionalInterface interface IntCalculator { int calc(int a, int b); }

IntCalculator add = (a, b) -> { int r = a + b; return r; }; // 중괄호 + return
IntCalculator mul = (a, b) -> a * b;                        // 식 하나 → 자동 리턴
```

값을 **리턴하는** 표준 함수형 인터페이스가 특히 자주 쓰인다. **`Function<T, R>`** 는 `R apply(T)` — 입력 1개, 리턴 1개. **`BiFunction<T, U, R>`** 는 입력 2개, 리턴 1개. **`Predicate<T>`** 는 `boolean test(T)` — 조건 검사에 쓰이며 boolean을 리턴한다.

```java
Function<String, Integer> length = s -> s.length();
Predicate<Integer> isEven = n -> n % 2 == 0;
```

## 16.5 메소드 참조

람다식이 하는 일이 **이미 존재하는 메소드 하나를 그대로 호출**하는 것뿐이라면, **메소드 참조(method reference)** 로 더 짧게 쓸 수 있다. 메소드 참조는 결국 람다식의 축약 표현이다.

```java
s -> Integer.parseInt(s)   // 람다
Integer::parseInt          // 메소드 참조 (같은 의미)
```

메소드 참조에는 세 종류가 있다(생성자 참조는 16.6에서 다룬다).

- **정적 메소드 참조** : `클래스명::정적메소드` — 예) `Integer::parseInt`
- **인스턴스 메소드 참조** : `클래스명::인스턴스메소드` — **첫 번째 인자가 호출 대상**이 된다. 예) `String::length` (`s.length()`와 같음)
- **특정 객체의 메소드 참조** : `객체참조::인스턴스메소드` — 특정 객체에 고정. 예) `System.out::println`

```java
Function<Integer, Integer> square = MathUtil::square;   // 정적
Function<String, Integer> length = String::length;      // 인스턴스 (첫 인자가 호출 대상)
Consumer<String> printer = System.out::println;         // 특정 객체
```

## 16.6 생성자 참조

람다식이 하는 일이 **새 객체를 생성**하는 것뿐이라면, **생성자 참조(constructor reference)** 로 축약할 수 있다. 형태는 **`클래스명::new`** 다.

```java
() -> new User()          // 람다  →  User::new  (기본 생성자)
name -> new User(name)    // 람다  →  User::new  (인자 1개 생성자)
```

흥미로운 점은, 겉모습이 똑같이 `User::new`인데도 **어떤 생성자가 호출될지는 "대상 함수형 인터페이스의 시그니처"가 결정**한다는 것이다. 인자가 0개인 `Supplier<User>`에 담으면 `new User()`가, 인자가 1개인 `Function<String, User>`에 담으면 `new User(String)`이 호출된다.

```java
Supplier<User> defaultFactory = User::new;          // new User()      호출
Function<String, User> namedFactory = User::new;    // new User(name)  호출

User u1 = defaultFactory.get();
User u2 = namedFactory.apply("홍길동");
```

이렇게 생성자 참조는 객체를 찍어내는 **팩토리(factory)** 처럼 재사용할 수 있어, 컬렉션의 각 요소로부터 객체를 만들 때 특히 편리하다.

> 💻 실습: `./run.sh ConstructorReference`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch16/ConstructorReference.java`

## ⚠️ 흔한 오해와 함정

- **함수형 인터페이스에 추상 메소드가 2개면 람다를 못 쓴다.** 람다식은 "유일한 추상 메소드 1개"를 구현하는 것이므로, 추상 메소드가 2개 이상인 인터페이스에는 람다를 담을 수 없다. `@FunctionalInterface`를 붙여 두면 이 실수를 컴파일 시점에 잡을 수 있다.
- **식(expression) 람다에 `return`을 붙인다.** 중괄호 없는 `(a, b) -> a + b`는 결과가 **자동 리턴**된다. 여기에 `return`을 쓰면 컴파일 오류다. `return`을 쓰려면 반드시 중괄호 `{ }`로 감싸야 한다.
- **매개변수가 2개인데 괄호를 생략한다.** 괄호 생략은 **매개변수가 정확히 1개일 때만** 가능하다. `a, b -> ...`는 오류이며 `(a, b) -> ...`로 써야 한다. 0개일 때도 `()`가 필요하다.
- **인스턴스 메소드 참조에서 "누가 호출 대상인지"를 헷갈린다.** `String::length`는 **첫 번째 인자가 호출 대상**이 되어 `s -> s.length()`와 같다. 반면 `System.out::println`은 **특정 객체(`System.out`)에 고정**된 형태다. 둘의 차이를 구분하자.
- **람다 안에서 바깥 지역 변수를 바꾸려 한다.** 람다가 캡처하는 지역 변수는 **사실상 final(effectively final)** 이어야 한다. 즉 캡처한 변수의 값을 람다 안팎에서 바꾸면 컴파일 오류가 난다. (자세한 이유는 개념서 6장의 "변수 캡처" 참고.)
- **`User::new`가 항상 기본 생성자를 부른다고 생각한다.** 어떤 생성자가 불릴지는 **대상 함수형 인터페이스의 인자 개수/타입**이 결정한다. 같은 `User::new`라도 담기는 그릇에 따라 다른 생성자가 호출된다.

**과제 1. 함수형 인터페이스 정의 후 익명 클래스 → 람다로 축약하기 (쉬움)**

추상 메소드가 딱 하나인 함수형 인터페이스 `interface Greeting { String say(String name); }`을 `@FunctionalInterface`를 붙여 직접 선언하라. `main`에서 **같은 동작**을 세 단계로 작성한다 — ① `new Greeting() { ... }` 익명 클래스로 `g1`, ② `(String name) -> { return ...; }` 기본 람다로 `g2`, ③ `name -> ...` 최대 축약 람다로 `g3`. 각각 `say(...)`를 호출해 셋 다 같은 형태의 인사말을 출력하고, 익명 클래스의 긴 껍데기가 람다에서 어떻게 걷혀 나가는지 눈으로 확인하라.

- 힌트: 함수형 인터페이스는 추상 메소드가 **정확히 1개**여야 한다. 축약 규칙은 (매개변수 타입 생략) → (매개변수 1개이면 괄호 생략) → (실행문 하나이면 중괄호·`return` 생략) 순으로 적용된다. `@FunctionalInterface`를 붙이면 "추상 메소드 1개" 규칙을 컴파일러가 검사해 준다.
> 답안 비교: `./run.sh LambdaIntro`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch16/LambdaIntro.java`

**과제 2. `Consumer`와 `BiConsumer`로 값 소비하기 (쉬움→응용)**

표준 함수형 인터페이스 `Consumer<String>` 변수 `shout`를 람다로 만들어, 받은 문자열을 대문자로 바꾸고 뒤에 `!!!`를 붙여 출력하라(`text -> ...`). 이어서 `BiConsumer<String, Integer>` 변수 `repeat`을 만들어, 문자열을 정수 횟수만큼 반복해 출력하라(`(word, count) -> ...`). 각각 `accept(...)`로 호출해 결과를 확인한다. 매개변수가 1개일 때는 괄호를 생략하고, 2개일 때는 괄호가 반드시 필요하다는 점을 두 람다의 대비로 드러내라.

- 힌트: `Consumer<T>`는 `void accept(T)`(입력 1개·리턴 없음), `BiConsumer<T, U>`는 `void accept(T, U)`(입력 2개·리턴 없음)다. 반복 출력은 `StringBuilder`로 `count`번 `append`하거나 간단히 반복문을 써도 된다. import는 `java.util.function.Consumer`, `java.util.function.BiConsumer`.
> 답안 비교: `./run.sh ParamLambda`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch16/ParamLambda.java`

**과제 3. `Function`·`BiFunction`·`Predicate`로 값 리턴하기 (응용)**

값을 **리턴하는** 표준 함수형 인터페이스 세 가지를 람다로 작성하라 — ① `Function<String, Integer> length`는 문자열의 길이를 리턴(`s -> s.length()`), ② `BiFunction<Integer, Integer, String> compare`는 두 수를 비교해 어느 쪽이 큰지 문자열로 리턴, ③ `Predicate<Integer> isEven`은 짝수인지 여부를 boolean으로 리턴(`n -> n % 2 == 0`). 그리고 직접 정의한 `interface IntCalculator { int calc(int a, int b); }`에 대해 **중괄호+`return`을 쓴 람다**와 **식 하나로 자동 리턴하는 람다**를 각각 하나씩 담아, 두 리턴 방식의 차이를 코드로 보여라.

- 힌트: 중괄호 없는 식 람다는 결과가 **자동 리턴**되므로 `return`을 쓰면 컴파일 오류다. `return`을 쓰려면 반드시 중괄호 `{ }`로 감싸야 한다. import는 `java.util.function.Function`, `BiFunction`, `Predicate`.
> 답안 비교: `./run.sh ReturnLambda`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch16/ReturnLambda.java`

**과제 4. 람다를 메소드 참조로 바꾸기 — 정적/인스턴스/특정 객체 (응용)**

세 종류의 메소드 참조를 각각 작성하고, 대응하는 람다를 주석으로 남겨 대비시켜라 — ① 정적 메소드 참조: 직접 만든 `static int square(int n)`을 `Function<Integer, Integer> square = 클래스::square;`로 참조(람다: `n -> square(n)`), ② 인스턴스 메소드 참조: `Function<String, Integer> length = String::length;`(람다: `s -> s.length()`, **첫 인자가 호출 대상**), ③ 특정 객체 메소드 참조: `Consumer<String> printer = System.out::println;`(람다: `x -> System.out.println(x)`). 세 변수를 모두 호출해 결과를 출력하고, ②와 ③에서 "누가 호출 대상인지"가 어떻게 다른지 주석 한 줄로 설명하라.

- 힌트: `String::length`는 **첫 번째 인자**가 호출 대상이 되어 `s -> s.length()`와 같다. 반면 `System.out::println`은 **특정 객체(`System.out`)에 고정**된 형태다. 메소드 참조는 결국 "메소드 하나를 그대로 호출하는 람다"의 축약이다.
> 답안 비교: `./run.sh MethodReference`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch16/MethodReference.java`

**과제 5. 생성자 참조로 객체 팩토리 만들기 (응용)**

기본 생성자와 `String name`을 받는 생성자를 **모두** 가진 `static class User`를 선언하고 `toString()`을 재정의하라. `main`에서 **똑같은 `User::new`**를 서로 다른 그릇 두 개에 담아라 — `Supplier<User> defaultFactory = User::new;`(인자 0개 → `new User()` 호출), `Function<String, User> namedFactory = User::new;`(인자 1개 → `new User(name)` 호출). 두 팩토리로 객체를 만들어 출력하고, 이어서 `String[]` 이름 배열을 반복하며 `namedFactory.apply(name)`으로 여러 객체를 찍어내 생성자 참조가 **팩토리처럼 재사용**됨을 보여라.

- 힌트: 겉모습이 같은 `User::new`라도 **어떤 생성자가 호출될지는 대상 함수형 인터페이스의 시그니처(인자 개수/타입)**가 결정한다. `Supplier<T>`는 `T get()`(인자 0개), `Function<T, R>`는 `R apply(T)`(인자 1개)다. import는 `java.util.function.Supplier`, `java.util.function.Function`.

**과제 6. 매개변수 없는 람다 — `Runnable`과 `Supplier` (쉬움)**

매개변수를 받지 않는 람다 두 가지를 작성하라. ① 직접 정의한 `interface MessageMaker { String make(); }`에 대해 **한 줄 람다**로 `m1`을, **여러 줄 람다(중괄호 + `return`)**로 `m2`를 담아 각각 `make()`를 호출해 출력하라. ② 표준 인터페이스 `Runnable task`를 `() -> ...`로 만들어 화면에 한 줄을 출력하는 **동작만** 수행하도록 하고 `task.run()`으로 실행하라. ③ 표준 인터페이스 `Supplier<Integer> six`를 `() -> 6`으로 만들어 `six.get()`으로 공급된 값을 출력하라. 세 경우 모두 매개변수 자리가 **빈 괄호 `()`**임을 확인하라.

- 힌트: 매개변수가 **0개**이면 괄호를 생략할 수 없다(`()` 필수). `Runnable`은 `void run()`(입력·리턴 없음), `Supplier<T>`는 `T get()`(입력 없음·리턴 있음)이다. 실행문이 하나면 중괄호와 `return`을 생략하고, 여러 줄이면 중괄호로 감싸고 `return`을 명시한다. import는 `java.util.function.Supplier`.

**과제 7. `BiFunction`과 인스턴스 메소드 참조 `String::concat` (응용)**

두 인자를 받는 메소드 참조를 다뤄 보자. `BiFunction<String, String, String> concat`을 **인스턴스 메소드 참조** `String::concat`으로 만들어라(람다: `(a, b) -> a.concat(b)`). 여기서 **첫 번째 인자 `a`가 호출 대상**, 두 번째 인자 `b`가 `concat`의 인자로 넘어간다는 점을 주석으로 남겨라. `concat.apply("Hello", " World")`를 호출해 결과를 출력한 뒤, 이어서 **특정 객체 메소드 참조**로 대비하라 — `String prefix = "값은 "; Function<String, String> addPrefix = prefix::concat;`을 만들어 `addPrefix.apply("42 입니다")`를 출력하고, 이번엔 `prefix`라는 **특정 객체에 고정**되어 인자가 하나만 필요함을 주석으로 설명하라.

- 힌트: `클래스명::인스턴스메소드`(`String::concat`)는 **첫 인자가 호출 대상**이 되어 인자를 하나 더 소비하므로 `BiFunction`(인자 2개)에 담긴다. `객체::인스턴스메소드`(`prefix::concat`)는 대상이 고정되어 인자가 하나 줄어 `Function`(인자 1개)에 담긴다. import는 `java.util.function.BiFunction`, `java.util.function.Function`.
> 답안 비교: `./run.sh MethodReference`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch16/MethodReference.java`

**과제 8. 익명 클래스 vs 람다 3단 축약 재현 (응용)**

과제 1의 축약 과정을 **표준 함수형 인터페이스**로 다시 밟아 보자. `Function<Integer, Integer>` 변수를 세 방식으로 만들어 모두 `apply(4)`를 호출해 같은 결과(예: 16)를 출력하라 — ① `new Function<Integer, Integer>() { @Override public Integer apply(Integer n) { return n * n; } }` 익명 클래스, ② `(Integer n) -> { return n * n; }` 기본 람다, ③ `n -> n * n` 최대 축약 람다. 세 변수의 코드 길이를 육안으로 비교해, 익명 클래스의 껍데기(`new`·타입·`@Override`·메소드 선언·중괄호)가 람다에서 **어떤 순서로 사라지는지** 주석 한 줄로 정리하라.

- 힌트: 축약은 (매개변수 타입 생략) → (매개변수 1개이면 괄호 생략) → (실행문 하나이면 중괄호·`return` 생략) 순으로 적용된다. `Function<T, R>`의 추상 메소드는 `R apply(T)`다. 익명 클래스와 람다는 **같은 함수형 인터페이스**를 구현하므로 서로 자유롭게 바꿔 담을 수 있다. import는 `java.util.function.Function`.

## 확인문제

> 개념 서술 · 출력 예측 · 오류 찾기 · 빈칸 채우기 · 참/거짓 판정을 고루 담았다. 먼저 스스로 답을 적은 뒤 `<details>`를 열어 정답과 해설을 확인하라.

### 개념 서술

**1.** **람다식**과 **함수형 인터페이스**의 관계를 설명하시오. 함수형 인터페이스가 되기 위한 조건은 무엇이며, `@FunctionalInterface` 어노테이션은 어떤 역할을 하는가?

<details>
<summary>정답 및 해설</summary>

- **관계**: 람다식은 함수형 인터페이스의 **유일한 추상 메소드 구현**을 대신하는 축약 표현이다. 람다식 자체에는 타입이 없으며, 담기는 그릇(함수형 인터페이스)의 추상 메소드 시그니처로부터 매개변수·리턴 타입을 추론한다. 따라서 람다식은 **반드시 함수형 인터페이스 타입의 변수/매개변수에만** 담을 수 있다.
- **조건**: 추상 메소드가 **정확히 1개**여야 한다. 단, `default`/`static` 메소드는 몇 개든 있어도 되고, `Object`의 `public` 메소드(`equals`, `hashCode`, `toString` 등)를 재선언한 것은 추상 메소드 개수에 세지 않는다.
- **`@FunctionalInterface`의 역할**: "추상 메소드 1개" 규칙을 **컴파일 시점에 검사**해 준다. 규칙을 위반하면(추상 메소드가 0개거나 2개 이상) 컴파일 오류를 낸다. 필수는 아니지만 의도를 명확히 하고 실수를 막아 주므로 권장된다.
</details>

**2.** **익명 클래스**와 **람다식**의 차이를 세 가지 이상 설명하시오. (구현 대상, `this`가 가리키는 것, 담을 수 있는 인터페이스 등)

<details>
<summary>정답 및 해설</summary>

- **구현 대상**: 익명 클래스는 추상 메소드가 여러 개인 인터페이스/추상 클래스도 구현할 수 있지만, 람다식은 **추상 메소드가 1개인 함수형 인터페이스만** 구현할 수 있다.
- **`this`가 가리키는 것**: 익명 클래스에서 `this`는 **익명 클래스 인스턴스 자신**을 가리킨다. 람다식에서 `this`는 **람다를 감싼 바깥(enclosing) 인스턴스**를 가리킨다. (람다는 새로운 스코프를 만들지 않는다.)
- **컴파일 결과**: 익명 클래스는 별도의 `.class` 파일(`Outer$1.class` 등)이 생성되지만, 람다식은 보통 별도 클래스 파일 없이 `invokedynamic`으로 처리된다.
- **간결성/껍데기**: 익명 클래스는 `new`·타입·메소드 선언·중괄호 같은 껍데기가 필요하지만, 람다식은 `(매개변수) -> 실행코드`만 남긴다.
</details>

**3.** 메소드 참조의 **세 종류**(정적 / 인스턴스 / 특정 객체)를 각각 예시와 함께 설명하고, 각각을 대응하는 람다식으로도 써 보시오.

<details>
<summary>정답 및 해설</summary>

| 종류 | 형태 | 예시 | 대응 람다 |
|---|---|---|---|
| 정적 메소드 참조 | `클래스명::정적메소드` | `Integer::parseInt` | `s -> Integer.parseInt(s)` |
| 인스턴스 메소드 참조 | `클래스명::인스턴스메소드` | `String::length` | `s -> s.length()` (첫 인자가 호출 대상) |
| 특정 객체 메소드 참조 | `객체참조::인스턴스메소드` | `System.out::println` | `x -> System.out.println(x)` (객체 고정) |

핵심 구분: **`클래스명::인스턴스메소드`** 는 첫 번째 인자가 호출 대상(receiver)이 되어 인자를 하나 소비하고, **`객체참조::인스턴스메소드`** 는 대상이 이미 고정되어 있어 인자가 하나 줄어든다. 그래서 `String::concat`은 `BiFunction`(인자 2개)에, `prefix::concat`은 `Function`(인자 1개)에 담긴다.
</details>

### 출력 예측

**4.** 다음 코드의 출력을 예측하시오. (변수 캡처 · effectively final)
```java
int base = 10;
java.util.function.Function<Integer, Integer> addBase = n -> n + base;
System.out.println(addBase.apply(5));
```

<details>
<summary>정답 및 해설</summary>

**출력: `15`**

람다는 바깥 지역 변수 `base`(값 10)를 **캡처**한다. `base`는 선언 후 값이 바뀌지 않으므로 **effectively final**이며 정상적으로 캡처된다. `addBase.apply(5)`는 `5 + 10 = 15`를 리턴한다.
</details>

**5.** 다음 두 메소드 참조를 사용한 코드의 출력을 예측하시오. (인스턴스 vs 특정 객체 메소드 참조)
```java
java.util.function.Function<String, Integer> f = String::length;
java.util.function.BiFunction<String, String, String> g = String::concat;
System.out.println(f.apply("hello"));
System.out.println(g.apply("ab", "cd"));
```

<details>
<summary>정답 및 해설</summary>

**출력:**
```
5
abcd
```

`String::length`는 `s -> s.length()`이므로 `"hello".length() == 5`. `String::concat`은 `(a, b) -> a.concat(b)`이므로 첫 인자 `"ab"`가 호출 대상, 둘째 인자 `"cd"`가 인자가 되어 `"ab".concat("cd") == "abcd"`.
</details>

**6.** 다음 코드의 출력을 예측하시오. (같은 `User::new`, 다른 생성자 — 대상 시그니처 함정)
```java
static class User {
    String name;
    User()            { this.name = "익명"; }
    User(String name) { this.name = name; }
    public String toString() { return "User(" + name + ")"; }
}
// main
java.util.function.Supplier<User> a = User::new;
java.util.function.Function<String, User> b = User::new;
System.out.println(a.get());
System.out.println(b.apply("홍길동"));
```

<details>
<summary>정답 및 해설</summary>

**출력:**
```
User(익명)
User(홍길동)
```

겉모습이 똑같은 `User::new`라도 **대상 함수형 인터페이스의 시그니처**가 어떤 생성자를 부를지 결정한다. `Supplier<User>`(인자 0개)는 `new User()`를, `Function<String, User>`(인자 1개)는 `new User(String)`을 호출한다.
</details>

**7.** 다음 코드의 출력을 예측하시오. (표준 함수형 인터페이스 시그니처 — `Predicate` boolean 리턴)
```java
java.util.function.Predicate<Integer> isEven = n -> n % 2 == 0;
java.util.function.Consumer<String> shout = t -> System.out.println(t.toUpperCase() + "!");
System.out.println(isEven.test(7));
shout.accept("go");
```

<details>
<summary>정답 및 해설</summary>

**출력:**
```
false
GO!
```

`Predicate<T>`의 추상 메소드는 `boolean test(T)`이므로 `isEven.test(7)`은 `7 % 2 == 0`, 즉 `false`. `Consumer<T>`는 `void accept(T)`로 리턴이 없고 소비만 하며, `"go".toUpperCase() + "!"` → `GO!`를 출력한다.
</details>

### 오류 찾기

**8.** 다음 두 람다식 중 컴파일 오류가 나는 것은 무엇이며 그 이유는? 오류가 나는 쪽을 올바르게 고치시오.
```java
IntCalculator a = (x, y) -> x + y;
IntCalculator b = (x, y) -> return x + y;
```

<details>
<summary>정답 및 해설</summary>

- **`b`가 컴파일 오류.** 중괄호 없는 **식(expression) 람다**는 식의 결과가 자동 리턴되므로 `return`을 쓸 수 없다. `return`은 문(statement)이라 식 자리에 올 수 없다.
- **수정 (택1):**
  - 식 람다로: `IntCalculator b = (x, y) -> x + y;`
  - 블록 람다로: `IntCalculator b = (x, y) -> { return x + y; };`
- `a`는 정상이다.
</details>

**9.** 다음 코드에서 컴파일 오류를 모두 찾고 이유를 설명하시오. (변수 캡처)
```java
int count = 0;
Runnable r = () -> {
    count++;                 // (A)
    System.out.println(count);
};
count = 5;                   // (B)
```

<details>
<summary>정답 및 해설</summary>

**두 곳이 문제이며, 근본 원인은 `count`가 effectively final이 아니게 된다는 것이다.**

- **(A)** 람다 안에서 캡처한 지역 변수 `count`를 `count++`로 **수정**하려 한다 → 컴파일 오류.
- **(B)** 람다 바깥에서도 `count = 5`로 재할당한다 → 이 재할당 때문에 `count`는 effectively final이 아니게 되어, 람다가 `count`를 캡처한 것 자체가 오류가 된다.

**해결**: 값을 누적해야 한다면 지역 변수 대신 배열 `int[] count = {0};`이나 `java.util.concurrent.atomic.AtomicInteger`처럼 **가변 컨테이너**를 캡처하고 그 내용을 바꾼다(참조 자체는 바뀌지 않으므로 OK).
</details>

**10.** 다음 코드의 오류를 찾고 고치시오. (괄호 생략 규칙 · 함수형 인터페이스 조건)
```java
interface Calc { int op(int a, int b); int extra(); }   // (가)
Calc c = a, b -> a + b;                                  // (나)
```

<details>
<summary>정답 및 해설</summary>

- **(가)** `Calc`는 추상 메소드가 `op`, `extra` **2개**라 함수형 인터페이스가 아니다. 람다를 담을 수 없다. → `extra()`를 제거하거나 `default`로 바꿔 추상 메소드를 1개로 만든다.
- **(나)** 매개변수가 **2개**인데 괄호를 생략했다. 괄호 생략은 **매개변수가 정확히 1개일 때만** 가능하다. → `(a, b) -> a + b`로 고친다.
- **수정 예:**
  ```java
  interface Calc { int op(int a, int b); }
  Calc c = (a, b) -> a + b;
  ```
</details>

### 빈칸 채우기

**11.** 익명 클래스를 람다식으로 두 단계(기본형 → 최대 축약형)에 걸쳐 완성하시오.
```java
// 원본 익명 클래스
Greeting g = new Greeting() {
    @Override public String say(String name) { return "Hi, " + name; }
};
// 기본형 람다 (타입 명시, 중괄호+return)
Greeting basic  = ____________ ;
// 최대 축약형 람다
Greeting minial = ____________ ;
```

<details>
<summary>정답 및 해설</summary>

```java
Greeting basic  = (String name) -> { return "Hi, " + name; };
Greeting minial = name -> "Hi, " + name;
```

축약 순서: ① 매개변수 타입 생략 → ② 매개변수 1개이므로 괄호 생략 → ③ 실행문 하나이므로 중괄호와 `return` 생략.
</details>

**12.** 빈칸을 채워 각 람다식을 **메소드 참조**로 완성하시오.
```java
// (a) s -> Integer.parseInt(s)
Function<String, Integer> a = ________ ;
// (b) s -> s.toUpperCase()
Function<String, String> b = ________ ;
// (c) () -> new ArrayList<String>()
Supplier<List<String>> c = ________ ;
// (d) x -> System.out.println(x)
Consumer<Object> d = ________ ;
```

<details>
<summary>정답 및 해설</summary>

</details>

### 참/거짓 + 이유

**13.** 다음 명제의 참·거짓을 판정하고 이유를 쓰시오.

- (a) "함수형 인터페이스는 `default` 메소드를 가질 수 없다."
- (b) "메소드 참조 `String::length`와 람다 `s -> s.length()`는 같은 의미다."
- (c) "람다식 `(a, b) -> { a + b; }`는 `IntCalculator`(추상 메소드 `int calc(int,int)`)에 담을 수 있다."

<details>
<summary>정답 및 해설</summary>

- **(a) 거짓.** 함수형 인터페이스의 조건은 **추상** 메소드가 1개인 것이다. `default`/`static` 메소드는 몇 개든 가질 수 있다.
- **(b) 참.** `String::length`는 클래스명::인스턴스메소드 형태로 첫 인자가 호출 대상이 되어 `s -> s.length()`와 정확히 같다.
- **(c) 거짓.** 중괄호를 쓴 블록 람다인데 `a + b;`는 값을 만들 뿐 `return` 하지 않는다. `int`를 리턴해야 하는 `calc`에 대해 리턴 경로가 없어 **컴파일 오류**다. `-> { return a + b; }` 또는 식 람다 `-> a + b`로 써야 한다.
</details>

**14.** 다음 명제의 참·거짓을 판정하고 이유를 쓰시오.

- (a) "같은 `User::new`라도 담기는 함수형 인터페이스에 따라 호출되는 생성자가 달라질 수 있다."
- (b) "람다식 안의 `this`는 람다 객체 자신을 가리킨다."
- (c) "매개변수가 0개인 람다는 괄호를 생략해 `-> ...`로 쓸 수 있다."

<details>
<summary>정답 및 해설</summary>

- **(a) 참.** 어떤 생성자가 호출될지는 대상 함수형 인터페이스의 시그니처(인자 개수/타입)가 결정한다. `Supplier<User>`면 `new User()`, `Function<String,User>`면 `new User(String)`.
- **(b) 거짓.** 람다는 새 스코프를 만들지 않으므로 `this`는 **람다를 감싼 바깥 인스턴스**를 가리킨다. (익명 클래스라면 익명 인스턴스 자신을 가리키는 것과 대비된다.)
- **(c) 거짓.** 괄호 생략은 매개변수가 **정확히 1개**일 때만 가능하다. 0개일 때는 빈 괄호 `()`가 반드시 필요하다: `() -> ...`.
</details>

## 요약

- **람다식**은 함수 하나를 `(매개변수) -> 실행코드` 형태로 간결히 표현한 것으로, **익명 클래스의 껍데기를 걷어낸** 축약 표현이다. 람다가 담기는 그릇은 **추상 메소드가 1개인 함수형 인터페이스**다. (`LambdaIntro`)
- **매개변수 없는 람다**는 빈 괄호 `() -> ...`로 쓴다. `Runnable`(입력·리턴 없음), `Supplier<T>`(리턴만 있음)가 대표적이다. (`NoParamLambda`)
- **매개변수 있는 람다**는 `(x)`, `(x, y)`로 쓰며, 타입은 대부분 생략 가능하고 **매개변수가 1개일 때만** 괄호도 생략할 수 있다. `Consumer`/`BiConsumer`는 값을 소비만 한다. (`ParamLambda`)
- **리턴값 있는 람다**는 중괄호 + `return`을 쓰거나, 식 하나만 써서 **자동 리턴**한다. `Function`, `BiFunction`, `Predicate`가 값을 리턴하는 표준 인터페이스다. (`ReturnLambda`)
- **메소드 참조**는 이미 있는 메소드를 그대로 호출하는 람다의 축약이다. 정적(`클래스명::정적`), 인스턴스(`클래스명::인스턴스`, 첫 인자가 호출 대상), 특정 객체(`객체::인스턴스`) 세 종류가 있다. (`MethodReference`)
- **생성자 참조**는 `클래스명::new`로 객체 생성을 축약하며, **어떤 생성자가 호출될지는 대상 함수형 인터페이스의 시그니처**가 결정한다. (`ConstructorReference`)
