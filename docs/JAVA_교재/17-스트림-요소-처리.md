# Chapter 17 스트림 요소 처리

컬렉션이나 배열에 담긴 데이터를 다룰 때, 우리는 오랫동안 `for` 문으로 요소를 하나씩 꺼내 가공해 왔다. 이 방식은 **"어떻게 반복할 것인가"**를 일일이 지시하는 명령형(imperative) 처리다. 인덱스를 관리하고, 임시 변수를 두고, 조건을 검사하는 코드가 처리 로직과 뒤섞여 의도가 잘 드러나지 않는다.

**스트림(Stream)**은 이 문제를 다른 관점에서 푼다. 데이터를 파이프에 흘려보내듯(stream) 처리하며, **"무엇을 할 것인가"**만 선언적(declarative)으로 기술한다. "짝수만 골라 → 대문자로 바꿔 → 개수를 센다"처럼 처리 단계를 그대로 코드로 옮길 수 있어, 읽기 쉽고 실수가 줄어든다.

스트림 처리를 이해하는 세 가지 열쇠가 있다.

- **파이프라인(pipeline)** : 소스에서 데이터를 얻어 여러 연산을 연결(체이닝)하여 하나의 흐름을 만든다. `소스 → 중간 연산 → 최종 연산`.
- **지연 평가(lazy evaluation)** : 중간 연산은 곧바로 실행되지 않고, **최종 연산이 호출될 때** 비로소 파이프라인 전체가 한 번에 실행된다.
- **내부 반복(internal iteration)** : 요소를 꺼내 도는 일은 스트림이 대신 수행하고, 개발자는 각 요소에 무엇을 할지(함수)만 넘긴다.

이번 장에서는 스트림의 개념과 파이프라인을 이루는 각 연산을 하나씩 살펴본다. 각 소절에는 실행 가능한 실습 코드가 연결되어 있으니, 개념을 읽은 뒤 직접 실행해 결과를 눈으로 확인하며 익히자. 스트림은 람다와 메서드 참조를 전제로 하므로, 필요하면 [JAVA_개념서 『06 함수형 프로그래밍으로의 진화』](../JAVA_교재/심화/06-함수형-프로그래밍으로의-진화.md)를 함께 참고하자.

## 17.1 스트림이란

**스트림**은 컬렉션·배열 등의 데이터 요소를 하나씩 흘려보내며 처리하는 반복자다. 핵심은 처리 방식의 차이에 있다. 전통적인 `for` 문이 **"어떻게 반복할지"**를 명령형으로 기술한다면, 스트림은 **"무엇을 할지"**를 선언형으로 기술한다.

스트림 처리의 뼈대는 **파이프라인**이다. 여러 연산을 점(`.`)으로 연결하여 하나의 처리 흐름을 만들며, 데이터가 파이프를 통과하듯 한 단계씩 흘러가며 가공된다.

```
소스(source) → 중간 연산(filter, map, sorted ...) → 최종 연산(forEach, count, collect ...)
```

아래는 같은 작업(길이 3 이상인 이름을 대문자로 바꿔 개수를 셈)을 명령형과 선언형으로 각각 작성한 것이다.

```java
List<String> names = List.of("kim", "lee", "park", "choi", "an");

// 선언형(스트림): 무엇을 할지 파이프라인으로 기술
long streamCount = names.stream()               // 소스: 컬렉션 → 스트림
        .filter(name -> name.length() >= 3)     // 중간 연산: 필터링
        .map(String::toUpperCase)               // 중간 연산: 대문자 매핑
        .count();                               // 최종 연산: 개수 집계
```

코드가 "필터링 → 매핑 → 집계"라는 처리 의도를 그대로 드러내어, `for` 문 버전보다 읽기 쉽다.

## 17.2 내부 반복자

반복을 누가 제어하느냐에 따라 반복자는 둘로 나뉜다.

- **외부 반복자(external iterator)** : 개발자가 `for`/`while`이나 `Iterator`로 요소를 **직접 꺼내며** 반복한다. 반복의 제어권이 코드(개발자)에게 있다.
- **내부 반복자(internal iterator)** : 스트림이 요소 반복을 **대신** 수행한다. 개발자는 각 요소에 무엇을 할지(함수)만 넘기고, "어떻게 반복할지"는 스트림에게 **위임(delegation)**한다.

위임의 장점은 두 가지다. 첫째, 개발자는 순수하게 **처리 로직**에만 집중할 수 있다. 둘째, 반복 방식(순서·병렬 여부 등)을 라이브러리가 내부에서 최적화하여 결정할 수 있다. 예컨대 `parallel()` 하나만 붙이면 같은 코드가 병렬로 도는데, 이것이 가능한 이유가 바로 반복 제어를 스트림이 쥐고 있기 때문이다.

```java
List<Integer> numbers = List.of(1, 2, 3, 4, 5, 6);

// 내부 반복자: 반복을 스트림에 위임, 개발자는 처리 로직만 제공
int sum = numbers.stream()      // 반복 제어를 스트림에게 위임
        .mapToInt(Integer::intValue)
        .sum();                 // 요소를 어떻게 꺼내 도는지는 스트림이 담당
```

## 17.3 중간 처리와 최종 처리

파이프라인을 이루는 연산은 두 종류로 나뉜다.

- **중간 연산(intermediate operation)** : 스트림을 입력받아 **또 다른 스트림**을 반환한다. (`filter`, `map`, `sorted`, `distinct`, `peek` ...) 반환값이 스트림이므로 계속 연결(체이닝)할 수 있다.
- **최종 연산(terminal operation)** : 스트림을 소비하여 최종 결과(값·컬렉션·void)를 만든다. (`forEach`, `count`, `collect`, `reduce`, `sum` ...)

여기서 스트림의 가장 중요한 성질인 **지연 평가**가 등장한다. 중간 연산은 정의만 해 두고 **곧바로 실행되지 않는다.** **최종 연산이 호출되는 순간**에야 비로소 파이프라인 전체가 한 번에 실행된다.

```java
List<Integer> numbers = List.of(1, 2, 3, 4, 5);

// 중간 연산만 구성 → 아직 실행되지 않음 (filter 내부 로그가 찍히지 않는다)
Stream<Integer> lazy = numbers.stream()
        .filter(n -> {
            System.out.println("filter 실행: " + n);
            return n % 2 == 0;
        });

// 최종 연산 호출 → 이때 비로소 filter 가 실제로 실행된다
lazy.forEach(n -> System.out.println("짝수: " + n));
```

또 하나 기억할 점은, 최종 연산이 스트림을 **소비**하므로 **한 번 최종 연산을 수행한 스트림은 재사용할 수 없다**는 것이다.

## 17.4 리소스로부터 스트림 얻기

스트림을 다루려면 먼저 스트림을 **얻어야** 한다. 데이터가 담긴 리소스의 종류에 따라 얻는 방법이 다르다.

| 리소스 | 얻는 방법 |
|--------|-----------|
| 컬렉션 | `collection.stream()` |
| 객체 배열 | `Arrays.stream(array)` |
| 기본형 배열 | `Arrays.stream(int[])` → `IntStream` |
| 나열된 값 | `Stream.of(v1, v2, ...)` |
| 정수 범위 | `IntStream.range(a, b)` / `rangeClosed(a, b)` |

```java
// 컬렉션 → stream()
List.of("사과", "바나나", "포도").stream().forEach(System.out::println);

// 기본형 배열 → IntStream (숫자 집계에 유리)
int total = Arrays.stream(new int[]{90, 80, 70}).sum();

// 정수 범위: range 는 끝값 미포함, rangeClosed 는 끝값 포함
IntStream.range(1, 5);        // 1 2 3 4
IntStream.rangeClosed(1, 5);  // 1 2 3 4 5
```

`range(a, b)`는 끝값 `b`를 **포함하지 않고**, `rangeClosed(a, b)`는 끝값 `b`를 **포함한다**. 헷갈리기 쉬우니 이름의 "Closed(닫힌 구간)"로 구분하자.

## 17.5 필터링

**필터링**은 스트림에서 원하는 요소만 걸러내는 중간 연산이다. 두 가지를 자주 쓴다.

- `filter(Predicate)` : 조건을 만족하는(`true`) 요소만 남긴다.
- `distinct()` : 중복 요소를 제거한다. 중복 판정은 `equals`/`hashCode`를 기준으로 한다.

```java
List<Integer> numbers = List.of(1, 2, 2, 3, 4, 4, 5, 6, 6);

// filter + distinct 조합: 중복 없는 짝수
List<Integer> result = numbers.stream()
        .filter(n -> n % 2 == 0)   // 조건이 true 인 요소만 통과
        .distinct()                // 중복 제거
        .toList();                 // [2, 4, 6]
```

두 연산 모두 결과 스트림을 반환하는 중간 연산이라 자유롭게 연결할 수 있다. `toList()`(JDK 16+)는 스트림을 불변 리스트로 수집하는 간편한 최종 연산이다.

## 17.6 매핑

**매핑**은 각 요소를 다른 값이나 타입으로 **변환**하는 중간 연산이다. 필터링이 요소를 "고르는" 일이라면, 매핑은 요소를 "바꾸는" 일이다.

- `map(Function)` : 각 요소를 다른 값/타입으로 **1:1 변환**한다.
- `mapToInt(...)` : 각 요소를 `int`로 변환하여 `IntStream`을 만든다. 숫자 집계(`sum`, `average` 등)에 유리하다.
- `flatMap(Function)` : 각 요소를 스트림으로 변환한 뒤 **하나의 스트림으로 평탄화(1:N)**한다.

```java
List<String> words = List.of("apple", "banana", "cherry");

// map: 문자열 → 대문자 문자열 (1:1)
List<String> upper = words.stream().map(String::toUpperCase).toList();

// mapToInt: 문자열 → 길이(int) → IntStream 으로 합계
int totalLength = words.stream().mapToInt(String::length).sum();
```

`map`과 `flatMap`의 차이가 핵심이다. `map`은 요소 하나를 값 하나로 바꾼다(1:1). 반면 `flatMap`은 요소 하나를 **스트림**으로 바꾼 뒤, 그 스트림들을 모두 이어 붙여 **하나의 평평한 스트림**으로 만든다(1:N). 중첩 리스트를 펼칠 때 쓴다.

```java
List<List<Integer>> nested = List.of(List.of(1, 2), List.of(3, 4), List.of(5, 6));

// flatMap: 중첩 리스트를 하나의 스트림으로 평탄화
List<Integer> flat = nested.stream()
        .flatMap(List::stream)   // 각 내부 리스트를 펼쳐 하나로 합침
        .toList();               // [1, 2, 3, 4, 5, 6]
```

## 17.7 정렬

**정렬**은 요소를 특정 기준으로 순서대로 배열하는 중간 연산이다.

- `sorted()` : 요소의 **자연 순서(Comparable)**로 정렬한다(숫자는 오름차순, 문자열은 사전순).
- `sorted(Comparator)` : 지정한 비교 기준으로 정렬한다.
- `Comparator.comparing(...)` : 정렬 키를 지정한다(`comparingInt` 등 기본형 특화 버전도 있다).
- `reversed()` : 정렬 순서를 뒤집는다(내림차순).
- `thenComparing(...)` : 1차 기준이 같을 때 2차 기준으로 추가 정렬한다.

```java
record Person(String name, int age) {}

List<Person> people = List.of(
        new Person("kim", 30), new Person("lee", 25),
        new Person("park", 30), new Person("choi", 25));

// 나이 내림차순, 동률이면 이름 오름차순
people.stream()
        .sorted(Comparator.comparingInt(Person::age).reversed()
                .thenComparing(Person::name))
        .forEach(System.out::println);
```

`Comparator`는 이렇게 `reversed()`, `thenComparing()`을 이어 붙여 복합 정렬 기준을 선언적으로 조립할 수 있다.

## 17.8 루핑

**루핑**은 스트림의 각 요소를 순회하며 동작을 수행하는 것이다. 두 연산의 성격이 다르니 구분해야 한다.

- `forEach(Consumer)` : **최종 연산**. 각 요소에 대해 지정한 동작을 수행한다(주로 출력·처리).
- `peek(Consumer)` : **중간 연산**. 요소를 소비하지 않고 **"들여다보기"만** 한다. 파이프라인 중간 상태를 확인하는 디버깅·로그에 유용하다.

```java
List<String> items = List.of("연필", "지우개", "공책");

// peek 로 중간 관찰, map 으로 변환, toList 로 최종 수집
List<Integer> lengths = items.stream()
        .peek(item -> System.out.println("[원본] " + item))  // 중간 관찰
        .map(String::length)
        .peek(len -> System.out.println("[길이] " + len))    // 변환 후 관찰
        .toList();
```

주의할 점은 성격 차이에서 온다. `forEach`는 **최종 연산**이라 그 뒤로 스트림을 더 쓸 수 없다. `peek`은 **중간 연산**이라 뒤에 최종 연산이 없으면 (지연 평가 때문에) 아예 실행되지 않는다. `peek`은 흐름을 바꾸지 않고 관찰만 하는 용도이므로, 실제 처리는 `forEach`나 다른 최종 연산에 맡기자.

## 17.9 매칭

**매칭**은 스트림의 요소들이 특정 조건을 만족하는지 검사하여 `boolean`을 돌려주는 최종 연산이다.

- `anyMatch(Predicate)` : **하나라도** 조건을 만족하면 `true`.
- `allMatch(Predicate)` : **모든** 요소가 조건을 만족하면 `true`.
- `noneMatch(Predicate)` : **아무** 요소도 조건을 만족하지 않으면 `true`.

```java
List<Integer> numbers = List.of(2, 4, 6, 8, 10);

boolean anyOver6 = numbers.stream().anyMatch(n -> n > 6);   // true
boolean allEven  = numbers.stream().allMatch(n -> n % 2 == 0); // true
boolean noOdd    = numbers.stream().noneMatch(n -> n % 2 == 1); // true
```

세 연산 모두 **단락 평가(short-circuit)**를 한다. 즉, 결과가 확정되면 나머지 요소를 검사하지 않고 즉시 멈춘다. 예를 들어 `anyMatch`는 조건을 만족하는 요소를 처음 만나는 순간 `true`를 반환하고 종료한다. 덕분에 큰 스트림에서도 불필요한 검사를 줄일 수 있다.

## 17.10 기본 집계

**집계(aggregation)**는 여러 요소를 하나의 결과값으로 압축하는 최종 연산이다. 자주 쓰는 통계 연산이 기본으로 제공된다.

- `count()` : 요소 개수(`long`).
- `sum()` : 합계. 숫자 스트림(`IntStream`/`LongStream`/`DoubleStream`)에서 제공.
- `average()` : 평균. `OptionalDouble` 반환.
- `max()` / `min()` : 최대/최소. `Optional` 반환.
- `summaryStatistics()` : 개수·합·평균·최대·최소를 **한 번에** 계산.

```java
List<Integer> scores = List.of(90, 75, 88, 62, 100, 55);

// sum/average/max 는 숫자 스트림에서 사용 → mapToInt 로 변환
int sum = scores.stream().mapToInt(Integer::intValue).sum();
double avg = scores.stream().mapToInt(Integer::intValue).average().orElse(0);

// 통계를 한 번에
IntSummaryStatistics stats = scores.stream()
        .mapToInt(Integer::intValue)
        .summaryStatistics();
System.out.println(stats.getMax() + " / " + stats.getAverage());
```

`sum`·`average`·`max`·`min`은 `IntStream` 같은 **숫자 스트림**에서 쓸 수 있으므로, 객체 스트림이라면 먼저 `mapToInt` 등으로 변환한다. `average`·`max`·`min`은 스트림이 비어 있을 수 있으므로 `Optional` 계열을 반환한다. `orElse(...)`로 비었을 때의 기본값을 지정하자.

## 17.11 커스텀 집계

`count`·`sum`·`max` 같은 기본 집계로 표현하기 어려운 **직접 정의한 집계**가 필요할 때는 `reduce`를 쓴다. `reduce`는 요소들을 하나의 값으로 **누적(reduce)**하는 최종 연산이다.

`reduce`는 세 가지 형태가 있다.

- `reduce(BinaryOperator)` : 초기값 없이 누적 → `Optional` 반환(스트림이 비면 결과가 없으므로).
- `reduce(identity, BinaryOperator)` : **초기값(identity)**과 함께 누적 → 값을 직접 반환.
- `reduce(identity, accumulator, combiner)` : 병렬 처리 시 부분 결과를 결합하는 방법까지 지정.

```java
List<Integer> numbers = List.of(1, 2, 3, 4, 5);

// 초기값 있음: 합계 (0에서 시작해 계속 더함)
int sum = numbers.stream().reduce(0, (a, b) -> a + b);      // 15

// 초기값 없음: 최댓값 → Optional 반환
Optional<Integer> max = numbers.stream().reduce(Integer::max);

// 커스텀 연산: 모든 요소의 곱 (1에서 시작해 계속 곱함)
int product = numbers.stream().reduce(1, (a, b) -> a * b);  // 120
```

핵심은 **초기값(identity)의 선택**이다. 덧셈 누적에는 `0`, 곱셈 누적에는 `1`, 문자열 이어붙이기에는 `""`처럼, 그 연산에서 "아무 영향도 주지 않는 값"을 초기값으로 골라야 한다. 초기값을 잘못 주면(예: 곱셈에 `0`) 결과가 망가지니 주의하자.

> 💻 실습: `./run.sh CustomAggregation`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch17/CustomAggregation.java`

## 17.12 수집

**수집**은 스트림 요소를 리스트·맵·문자열 등으로 **모으는** 최종 연산이다. `collect(Collector)`에 `Collectors`가 제공하는 다양한 수집기를 넘겨 사용한다.

- `toList()` : `List`로 수집.
- `toMap(키함수, 값함수)` : `Map`으로 수집.
- `groupingBy(키함수)` : 키로 그룹화하여 `Map<K, List<T>>` 생성.
- `joining(구분자, 접두, 접미)` : 문자열을 이어붙임.
- `partitioningBy(Predicate)` : 조건 `true`/`false` **두 그룹**으로 분할.

```java
record Product(String name, String category, int price) {}
// products: (사과,과일,1000) (바나나,과일,1500) (당근,채소,800) (배추,채소,2000)

// groupingBy: 카테고리별로 묶기 → Map<String, List<Product>>
Map<String, List<Product>> byCategory = products.stream()
        .collect(Collectors.groupingBy(Product::category));

// joining: 이름을 하나의 문자열로 → "[사과, 바나나, 당근, 배추]"
String joined = products.stream().map(Product::name)
        .collect(Collectors.joining(", ", "[", "]"));

// partitioningBy: 조건으로 두 그룹 분할 → Map<Boolean, List<Product>>
Map<Boolean, List<Product>> partitioned = products.stream()
        .collect(Collectors.partitioningBy(p -> p.price() >= 1000));
```

`groupingBy`와 `partitioningBy`의 차이를 구분하자. `groupingBy`는 키 함수의 결과에 따라 **여러 그룹**으로 나누고, `partitioningBy`는 조건의 참/거짓에 따라 **정확히 두 그룹**(`true`/`false`)으로 나눈다. `Collectors`는 자바 데이터 처리에서 가장 강력한 도구이니 다양한 수집기를 익혀 두면 좋다.

## 17.13 병렬 처리

**병렬 스트림(parallel stream)**은 여러 스레드가 요소를 나눠 **동시에** 처리하도록 한다. 기존 스트림에 `parallel()`을 붙이거나 컬렉션에서 `parallelStream()`으로 얻는다. 내부적으로는 `ForkJoinPool`이 작업을 분할·병합한다.

```java
long n = 50_000_000L;

// 순차 합계
long seqSum = LongStream.rangeClosed(1, n).sum();

// 병렬 합계: parallel() 하나로 여러 스레드가 나눠 처리
long parSum = LongStream.rangeClosed(1, n).parallel().sum();
// 결과는 동일, 큰 데이터에서는 병렬이 대개 더 빠르다
```

병렬 처리는 만능이 아니다. **언제 이득**이고 **언제 위험**한지 판단해야 한다.

- **이득** : 데이터가 매우 많고, 각 요소 처리 비용이 크며, 처리 순서가 결과에 영향을 주지 않을 때(예: 합계).
- **위험** : 데이터가 적으면 스레드 분할·병합 비용이 더 커서 오히려 느리다. 여러 스레드가 **공유 가변 상태**를 수정하면 경합·오류가 발생한다. 순서가 중요한 작업에서는 결과 순서가 뒤섞일 수 있다.

`parallel()`을 붙이는 것만으로 항상 빨라진다고 착각하기 쉬운데, 반드시 실제 데이터로 **측정**한 뒤 도입하자.

> 💻 실습: `./run.sh ParallelStream`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch17/ParallelStream.java`

## ⚠️ 흔한 오해와 함정

- **중간 연산만 써 놓고 실행되기를 기대한다.** 중간 연산은 **지연 평가**되어, 최종 연산이 없으면 아예 실행되지 않는다. `filter`나 `peek`만 걸어 두고 결과가 안 나온다면 최종 연산(`forEach`·`collect`·`count` 등)이 빠진 것은 아닌지 확인하자.
- **최종 연산을 끝낸 스트림을 재사용한다.** 스트림은 최종 연산 시 소비되어 한 번 쓰면 끝이다. 같은 데이터를 다시 처리하려면 소스에서 **스트림을 새로 얻어야** 한다.
- **`map`과 `flatMap`을 혼동한다.** `map`은 요소를 값 하나로 바꾸고(1:1), `flatMap`은 요소를 스트림으로 바꿔 하나로 펼친다(1:N). 중첩 구조를 평탄화할 때는 `flatMap`이다.
- **`range`와 `rangeClosed`의 끝값을 헷갈린다.** `range(a, b)`는 끝값 `b`를 **포함하지 않고**, `rangeClosed(a, b)`는 **포함한다.** 반복 횟수가 하나 어긋나기 쉬우니 주의하자.
- **`forEach`와 `peek`를 혼동한다.** `forEach`는 최종 연산이라 그 뒤로 스트림을 쓸 수 없고, `peek`는 중간 연산이라 최종 연산이 있어야 실행된다. `peek`는 흐름을 관찰하는 디버깅용이지 처리용이 아니다.
- **`reduce`의 초기값을 잘못 준다.** 초기값은 그 연산에서 결과를 바꾸지 않는 값이어야 한다. 덧셈은 `0`, 곱셈은 `1`, 문자열은 `""`이다. 곱셈에 `0`을 주면 결과가 항상 0이 된다.
- **무조건 `parallel()`을 붙이면 빨라진다고 믿는다.** 데이터가 적거나 공유 가변 상태를 수정하면 병렬이 오히려 느리거나 틀린 결과를 낸다. 측정 없이 병렬을 남용하지 말자.

**과제 5. 조건 검사하기 (anyMatch · allMatch · noneMatch)** — 응용

(3) **아무** 학생도 50점 미만이 아닌가, 즉 50점 미만 학생이 하나도 없는가(`noneMatch`). 마지막으로, 세 연산이 공통으로 수행하는 **단락 평가**의 효과를 직접 보기 위해, `filter`에 출력문을 넣은 파이프라인에 `anyMatch`를 걸어 조건을 처음 만족하는 순간 검사가 멈추는지 로그로 확인하라.

<details>
<summary>힌트</summary>

- 단락 평가 관찰은 `students.stream().peek(s -> System.out.println("검사: " + s.name())).anyMatch(s -> s.score() >= 90)`처럼 `peek`(또는 `filter` 안의 출력문)으로 로그를 남긴 뒤, 조건을 처음 만족하는 요소에서 로그가 멈추는지 본다.
- 세 연산은 결과가 확정되는 즉시 남은 요소를 검사하지 않고 종료한다는 점(short-circuit)을 로그로 확인하는 것이 이 과제의 핵심이다.
</details>

**과제 6. 성적 통계 내기 (count · sum · average · max)** — 응용

**과제 7. 직접 누적하기 (reduce)** — 도전

`reduce`로 기본 집계가 아닌 **직접 정의한 누적**을 구현하라. (1) 초기값을 준 `reduce`로 전체 학생 점수의 **합계**를 구해 출력하라. (2) 초기값 없는 `reduce`로 **최고 점수**를 구하라(결과가 `Optional`임에 주의해 `orElse`로 값을 꺼낸다). (3) 커스텀 연산으로 전체 학생 **이름을 `-`로 이어 붙인 문자열**(예: `"kim-lee-park-..."`)을 `reduce`로 만들어 출력하라.

<details>
<summary>힌트</summary>

- 합계는 `reduce(0, (a, b) -> a + b)`처럼 초기값 `0`에서 시작한다. 덧셈에서 결과를 바꾸지 않는 값이 `0`이기 때문이다. 점수만 필요하므로 먼저 `mapToInt(Student::score)`로 바꾸거나 `map(Student::score)`로 정수 스트림을 만든다.
- 최댓값은 `reduce(Integer::max)`처럼 초기값 없이 누적한다. 스트림이 비어 있을 수 있으므로 결과가 `Optional`이다. `orElse(0)`으로 값을 꺼낸다.
- 이름 이어 붙이기는 `map(Student::name)` 뒤에 `reduce("", (a, b) -> a.isEmpty() ? b : a + "-" + b)`처럼 초기값 `""`에서 시작한다. 첫 요소 앞에는 `-`가 붙지 않도록 초기값이 비었는지 검사한다.
</details>

**과제 8. 학년별로 묶기 (collect · groupingBy)** — 도전

**1. (개념 서술)** 스트림 파이프라인을 이루는 **소스·중간 연산·최종 연산**의 역할을 각각 설명하고, 중간 연산과 최종 연산에 해당하는 메서드를 각각 두 개씩 예로 드시오. 또한 "중간 연산은 스트림을 반환하고 최종 연산은 값·컬렉션·void를 반환한다"는 성질이 **체이닝(연결)**과 어떻게 연결되는지 서술하시오.

<summary>정답 및 해설</summary>

- **소스**: 데이터를 공급하는 출발점(컬렉션·배열·범위 등)이며 `list.stream()`, `Arrays.stream(arr)`, `IntStream.range(...)` 등으로 스트림을 얻는다.

- **최종 연산**: 스트림을 **소비하여 결과(값·컬렉션·void)를 만드는** 연산. 예: `forEach`, `count`, `collect`, `reduce`, `sum`, `anyMatch`. 반환값이 스트림이 아니므로 뒤에 더 이어 붙일 수 없고, 이 시점에 파이프라인 전체가 실행된다.

</details>

**2. (개념 서술)** **내부 반복자(internal iterator)**와 **외부 반복자(external iterator)**의 차이를 반복 제어권의 관점에서 설명하시오. 내부 반복자가 갖는 두 가지 장점을 쓰고, 이 성질이 왜 `parallel()` 한 줄만으로 병렬 처리를 가능하게 하는지 서술하시오.

<details>
<summary>정답 및 해설</summary>

- **외부 반복자**: 개발자가 `for`/`while`/`Iterator`로 요소를 **직접 꺼내며** 반복한다. 반복의 제어권이 개발자 코드에 있다.
- **내부 반복자**: 스트림이 요소 반복을 **대신** 수행하고, 개발자는 각 요소에 무엇을 할지(함수)만 넘긴다. "어떻게 반복할지"를 스트림에 **위임**한다.
- 장점 (1) 개발자는 인덱스 관리·조건 검사 같은 반복 제어에서 벗어나 순수한 **처리 로직**에만 집중한다. (2) 반복 방식(순서·병렬 여부 등)을 라이브러리가 내부에서 최적화해 결정할 수 있다.
- 반복 제어를 스트림이 쥐고 있으므로, `parallel()` 하나만 붙이면 스트림이 요소를 여러 조각으로 나눠 여러 스레드에 분배하고 결과를 병합하는 일을 내부에서 알아서 한다. 외부 반복이라면 개발자가 스레드 분할·병합을 직접 구현해야 하므로 이렇게 간단히 바뀌지 않는다.
</details>

**3. (개념 서술)** 병렬 스트림이 **이득이 되는 조건**과 **위험(손해·오류)이 되는 조건**을 각각 두 가지 이상 서술하시오. 특히 "공유 가변 상태(shared mutable state)"를 병렬 스트림에서 수정하면 왜 문제가 되는지 설명하시오.

<details>
<summary>정답 및 해설</summary>

- **이득 조건**: ① 데이터가 매우 많다. ② 각 요소 처리 비용이 크다. ③ 처리 순서가 결과에 영향을 주지 않는다(예: 합계, `sum`). 이때 스레드 분할·병합 비용보다 병렬 처리 이득이 커진다.
- **위험 조건**: ① 데이터가 적으면 스레드 분할·병합 오버헤드가 처리 이득보다 커서 오히려 느리다. ② 여러 스레드가 **공유 가변 상태**를 수정하면 경합(race condition)이 발생한다. ③ 순서가 중요한 작업에서는 결과 순서가 뒤섞일 수 있다.
- 공유 가변 상태(예: 바깥의 `List`나 카운터 변수)를 여러 스레드가 동기화 없이 동시에 수정하면, 같은 위치에 대한 쓰기가 서로 덮어써져 요소가 누락되거나 예외가 나거나 실행할 때마다 결과가 달라진다. 그래서 병렬 스트림에서는 바깥 변수를 건드리지 말고 `collect`/`reduce`처럼 스트림이 안전하게 결과를 모으는 방식을 써야 한다.
- 결론: `parallel()`은 만능이 아니며, 반드시 실제 데이터로 **측정**한 뒤 도입한다.
</details>

**4. (출력 예측 · 지연 평가)** 다음 코드의 출력을 순서대로 예측하시오. `filter`와 `map` 안의 출력문이 각각 몇 번, 어떤 순서로 찍히는지가 핵심이다.
```java
List.of(1, 2, 3, 4).stream()
        .filter(n -> { System.out.println("filter: " + n); return n % 2 == 0; })
        .map(n -> { System.out.println("map: " + n); return n * 10; })
        .forEach(n -> System.out.println("forEach: " + n));
```

<details>
<summary>정답 및 해설</summary>

```
filter: 1
filter: 2
map: 2
forEach: 20
filter: 3
filter: 4
map: 4
forEach: 40
```

- 스트림은 요소를 하나씩 파이프라인 **끝까지 흘려보낸 뒤** 다음 요소로 넘어간다(요소 단위 처리). 반복문처럼 "전체 filter → 전체 map"이 아니다.
- `1`: filter에서 홀수라 탈락 → map/forEach 미실행. `2`: filter 통과 → map 실행 → forEach 실행. `3`: 탈락. `4`: 통과 → map → forEach.
- 그래서 `filter`는 4번(모든 요소), `map`과 `forEach`는 2번(통과한 요소만) 실행된다. 이것이 최종 연산이 있을 때 파이프라인이 요소별로 한 번에 흐른다는 증거다.
</details>

**5. (출력 예측 · 지연 평가)** 다음 두 코드의 출력을 각각 예측하시오. 최종 연산의 유무가 결과를 어떻게 가르는지 설명하시오.
```java
// (가)
List.of(1, 2, 3).stream()
        .peek(n -> System.out.println("peek: " + n))
        .map(n -> n * 2);

// (나)
List.of(1, 2, 3).stream()
        .peek(n -> System.out.println("peek: " + n))
        .map(n -> n * 2)
        .forEach(n -> System.out.println("결과: " + n));
```

<details>
<summary>정답 및 해설</summary>

- **(가)**: **아무것도 출력되지 않는다.** `peek`·`map`은 모두 중간 연산이고 최종 연산이 없다. 지연 평가 때문에 중간 연산만 정의되고 실행되지 않는다.
- **(나)**:
  ```
  peek: 1
  결과: 2
  peek: 2
  결과: 4
  peek: 3
  결과: 6
  ```
  `forEach`라는 최종 연산이 붙어야 파이프라인이 실행되며, 요소별로 `peek → map → forEach` 순으로 흐른다.
- 교훈: `peek`나 `filter`만 걸어 두고 결과가 안 나온다면 최종 연산이 빠진 것은 아닌지 확인하라. `peek`는 흐름을 관찰하는 디버깅용이지 그 자체로 실행을 트리거하지 않는다.
</details>

**6. (출력 예측 · map vs flatMap)** 다음 코드에서 `a`와 `b`가 각각 어떤 타입·값으로 출력되는지 예측하시오.
```java
List<List<Integer>> nested = List.of(List.of(1, 2), List.of(3, 4), List.of(5));

var a = nested.stream().map(List::stream).toList();
var b = nested.stream().flatMap(List::stream).toList();

System.out.println("a = " + a);
System.out.println("b = " + b);
```

<details>
<summary>정답 및 해설</summary>

- `a`는 각 내부 리스트를 **스트림 객체**로 1:1 매핑만 하므로, 요소가 `Stream` 객체인 리스트가 된다. 출력은 `Stream` 객체의 주소 형태라 사실상 쓸모없다(예: `a = [java.util.stream.ReferencePipeline$Head@..., ...]`). 중첩이 풀리지 않았다.
- `b = [1, 2, 3, 4, 5]`. `flatMap`은 각 요소를 스트림으로 바꾼 뒤 그 스트림들을 **하나의 평평한 스트림으로 이어 붙인다**(1:N). 그래서 중첩이 평탄화된다.
- 핵심: 중첩 구조를 펼치려면 `map`이 아니라 `flatMap`이다. `map(List::stream)`은 "스트림의 스트림"을 만들 뿐 펼치지 않는다.
</details>

**7. (출력 예측 · reduce 초기값)** 다음 코드의 출력을 예측하시오. 세 `reduce`의 초기값 선택이 결과를 어떻게 좌우하는지 설명하시오.
```java
List<Integer> nums = List.of(2, 3, 4);

int p1 = nums.stream().reduce(1, (a, b) -> a * b);
int p2 = nums.stream().reduce(0, (a, b) -> a * b);
int s1 = nums.stream().reduce(10, (a, b) -> a + b);

System.out.println(p1 + " / " + p2 + " / " + s1);
```

<details>
<summary>정답 및 해설</summary>

- 출력: `24 / 0 / 19`
- `p1`: 초기값 `1`에서 곱셈 누적 → `1*2*3*4 = 24`. 곱셈에서 `1`은 결과를 바꾸지 않는 올바른 초기값(항등원)이다.
- `p2`: 초기값 `0`에서 곱셈 누적 → `0*2*3*4 = 0`. **잘못된 초기값**. 곱셈에 `0`을 주면 무엇을 곱해도 0이 되어 결과가 망가진다.
- `s1`: 초기값 `10`에서 덧셈 누적 → `10+2+3+4 = 19`. 덧셈의 항등원은 `0`인데 `10`을 주었으므로 실제 합(`9`)보다 초기값만큼 더해진다. "초기값이 결과에 그대로 반영된다"는 점을 보여준다.
- 교훈: 초기값(identity)은 그 연산에서 결과를 바꾸지 않는 값이어야 한다. 덧셈 `0`, 곱셈 `1`, 문자열 `""`.
</details>

**8. (출력 예측 · groupingBy 함정)** 다음 코드는 학생을 학년별로 묶어 **각 학년의 인원 수**를 세려는 의도다. `byGrade`의 실제 타입과 출력을 예측하고, 흔히 기대하는 결과와 무엇이 다른지 설명하시오.
```java
record Student(String name, int grade, int score) {}
List<Student> students = List.of(
        new Student("kim", 1, 85), new Student("lee", 2, 72),
        new Student("park", 1, 90), new Student("an", 2, 95), new Student("jung", 3, 88));

var byGrade = students.stream()
        .collect(Collectors.groupingBy(Student::grade));
System.out.println(byGrade);
```

<details>
<summary>정답 및 해설</summary>

- `byGrade`의 타입은 `Map<Integer, List<Student>>`이고, 인원 **수**가 아니라 각 학년의 **학생 리스트**가 값으로 들어간다. 출력은 대략 `{1=[Student[name=kim,...], Student[name=park,...]], 2=[...], 3=[...]}` 형태다.
- 함정: `groupingBy(키함수)` 한 인자만 쓰면 다운스트림 수집기가 기본값(`toList()`)이라 그룹마다 요소 리스트가 담긴다. **인원 수**를 원했다면 다운스트림 수집기를 넘겨야 한다:
  ```java
  Collectors.groupingBy(Student::grade, Collectors.counting())  // Map<Integer, Long>
  ```
- 마찬가지로 학년별 평균 점수는 `Collectors.groupingBy(Student::grade, Collectors.averagingInt(Student::score))`처럼 두 번째 인자로 무엇을 집계할지 지정한다. "그룹핑=리스트로 묶기"라는 기본 동작을 기억하고, 개수·평균·합이 필요하면 다운스트림을 반드시 지정하자.
</details>

**9. (오류 찾기 · 스트림 재사용)** 다음 코드는 컴파일은 되지만 실행 시 예외가 발생한다. 무슨 예외가 왜 발생하는지 설명하고, 두 통계를 모두 구하도록 코드를 고치시오.
```java
Stream<Integer> stream = List.of(1, 2, 3, 4, 5).stream();
long count = stream.count();
int sum = stream.mapToInt(Integer::intValue).sum();   // 여기서 문제 발생
```

<details>
<summary>정답 및 해설</summary>

- `java.lang.IllegalStateException: stream has already been operated upon or closed`가 발생한다.
- 이유: 최종 연산(`count()`)이 스트림을 **소비**하면 그 스트림은 닫혀 재사용할 수 없다. 같은 `stream` 변수에 다시 최종 연산(`mapToInt(...).sum()`)을 걸 수 없다.
- 해결책 (1) 필요할 때마다 소스에서 스트림을 **새로 얻는다**:
  ```java
  List<Integer> nums = List.of(1, 2, 3, 4, 5);
  long count = nums.stream().count();
  int sum = nums.stream().mapToInt(Integer::intValue).sum();
  ```
  해결책 (2) 여러 통계가 필요하면 스트림을 한 번만 돌려 `summaryStatistics()`로 개수·합·평균 등을 한꺼번에 뽑는다:
  ```java
  IntSummaryStatistics st = nums.stream().mapToInt(Integer::intValue).summaryStatistics();
  long count = st.getCount();
  long sum = st.getSum();
  ```
</details>

**10. (오류 찾기 · 병렬 부작용)** 다음 코드는 1부터 1000까지 짝수를 리스트에 모으려는 의도다. 실행할 때마다 결과 크기가 달라지거나 예외가 나기도 한다. 원인을 설명하고 올바르게 고치시오.
```java
List<Integer> result = new ArrayList<>();
IntStream.rangeClosed(1, 1000).parallel()
        .filter(n -> n % 2 == 0)
        .forEach(n -> result.add(n));   // 공유 리스트에 병렬로 add
System.out.println(result.size());      // 기대: 500, 실제: 매번 다름
```

<details>
<summary>정답 및 해설</summary>

- 원인: `ArrayList`는 **스레드 안전하지 않은 공유 가변 상태**인데, `parallel()` 스트림의 여러 스레드가 `result.add(n)`을 동기화 없이 동시에 호출한다. 내부 배열의 크기 확장·인덱스 갱신이 겹쳐 요소가 누락되거나(size < 500) `ArrayIndexOutOfBoundsException`/`NullPointerException`이 터지고, 실행마다 결과가 달라진다.
- 해결: 바깥 컬렉션을 직접 수정하지 말고, 스트림이 안전하게 결과를 모으게 한다.
  ```java
  List<Integer> result = IntStream.rangeClosed(1, 1000)
          .parallel()
          .filter(n -> n % 2 == 0)
          .boxed()
          .collect(Collectors.toList());   // 병렬에도 안전하게 수집
  System.out.println(result.size());       // 항상 500
  ```
- 교훈: 병렬 스트림에서는 **외부 가변 상태를 부작용(side effect)으로 수정하지 말고** `collect`/`reduce`로 결과를 모아야 한다.
</details>

**11. (오류 찾기)** 다음 코드는 학생 이름을 쉼표로 이어 붙인 문자열을 만들려는 의도지만 컴파일 오류가 난다. 무엇이 문제이며 어떻게 고치는가?
```java
String names = students.stream()
        .collect(Collectors.joining(", "));
```

<details>
<summary>정답 및 해설</summary>

- 문제: `Collectors.joining(...)`은 **문자열(`CharSequence`) 스트림**에만 쓸 수 있다. 그런데 스트림 요소는 `Student` 객체라 타입이 맞지 않아 컴파일되지 않는다.
- 해결: `joining` 전에 `map(Student::name)`으로 각 요소를 문자열로 바꾼다.
  ```java
  String names = students.stream()
          .map(Student::name)
          .collect(Collectors.joining(", ", "[", "]"));
  ```
- 교훈: `joining`은 이어 붙일 대상이 문자열이어야 하므로, 객체 스트림이라면 먼저 원하는 필드를 문자열로 매핑한다.
</details>

**12. (빈칸 채우기)** 다음 파이프라인은 학생 리스트에서 **점수 80점 이상인 학생을 점수 내림차순으로 정렬해, 이름을 쉼표로 이어 붙인 문자열**을 만든다. 빈칸 ㉠~㉢을 채우시오.
```java
String result = students.stream()
        .( ㉠ )(s -> s.score() >= 80)                       // 80점 이상만
        .sorted(Comparator.comparingInt(Student::score).( ㉡ )())   // 점수 내림차순
        .map(Student::name)
        .collect(Collectors.( ㉢ )(", "));                  // 이름 이어 붙이기
```

<details>
<summary>정답 및 해설</summary>

- ㉠ = `filter` — 조건이 `true`인 요소만 남기는 중간 연산.
- ㉡ = `reversed` — `comparingInt`가 만든 오름차순 비교자를 뒤집어 내림차순으로 정렬.
- ㉢ = `joining` — 문자열을 구분자로 이어 붙이는 수집기.
- 완성된 파이프라인은 `filter → sorted → map → collect(joining)` 순으로, 필터링·정렬·매핑을 거쳐 하나의 문자열로 수집한다. 중간 연산으로 흐름을 구성하고 최종 연산 `collect`로 마무리하는 전형적인 형태다.
</details>

**13. (빈칸 채우기)** 다음 파이프라인은 학생을 **학년별로 묶어 학년마다 평균 점수**를 구한다(`Map<Integer, Double>`). 빈칸 ㉠~㉢을 채우시오.
```java
Map<Integer, Double> avgByGrade = students.stream()
        .collect(Collectors.( ㉠ )(
                Student::( ㉡ ),                          // 그룹화 키: 학년
                Collectors.( ㉢ )(Student::score)));      // 다운스트림: 점수 평균
```

<details>
<summary>정답 및 해설</summary>

- ㉠ = `groupingBy` — 키 함수 결과에 따라 여러 그룹으로 나눈다.
- ㉡ = `grade` — 학년을 그룹화 키로 사용.
- ㉢ = `averagingInt` — 각 그룹 안에서 점수 평균을 구하는 다운스트림 수집기(결과가 `Double`).
- `groupingBy`에 두 번째 인자(다운스트림 수집기)를 넘기면, 그룹마다 요소 리스트 대신 원하는 집계값을 담을 수 있다. `counting()`이면 인원 수(`Long`), `averagingInt(...)`면 평균(`Double`), `mapping(...)`이면 변환 후 수집이 된다.
</details>

**14. (참·거짓 + 이유)** 다음 각 진술의 참·거짓을 판별하고 **이유**를 한두 문장으로 쓰시오.
1. `range(1, 5)`와 `rangeClosed(1, 5)`는 같은 개수의 요소를 만든다.
2. 하나의 스트림에 최종 연산을 두 번 호출할 수 있다.
3. `peek`는 최종 연산이므로 그것만으로 파이프라인이 실행된다.
4. `map`은 요소 개수를 바꿀 수 없지만, `flatMap`은 요소 개수를 바꿀 수 있다.
5. `average()`·`max()`가 `Optional` 계열을 반환하는 이유는 스트림이 비어 있을 수 있기 때문이다.

<details>
<summary>정답 및 해설</summary>

1. **거짓**. `range(1, 5)`는 끝값 미포함이라 `1,2,3,4`(4개), `rangeClosed(1, 5)`는 포함이라 `1,2,3,4,5`(5개). 개수가 하나 다르다.
2. **거짓**. 최종 연산은 스트림을 소비해 닫으므로, 같은 스트림에 두 번째 최종 연산을 걸면 `IllegalStateException`이 발생한다. 소스에서 새 스트림을 얻어야 한다.
3. **거짓**. `peek`는 **중간 연산**이다. 최종 연산이 없으면 지연 평가 때문에 실행되지 않는다. `peek`는 관찰용이지 실행 트리거가 아니다.
4. **참**. `map`은 각 요소를 1:1로 바꿔 개수가 그대로다. `flatMap`은 요소 하나를 0개 이상의 요소를 가진 스트림으로 펼쳐 하나로 합치므로(1:N) 전체 개수가 달라질 수 있다.
5. **참**. 빈 스트림에서는 평균·최대·최소가 존재하지 않는다. 그래서 값이 없을 수 있음을 타입으로 표현하는 `OptionalDouble`/`OptionalInt`/`Optional`을 반환하고, `orElse(...)`로 기본값을 지정한다.
</details>

**15. (종합 출력 예측)** 다음 코드의 최종 출력을 예측하시오. `partitioningBy`가 만드는 맵의 키와, 두 그룹에 각각 어떤 이름이 담기는지가 핵심이다.
```java
record Student(String name, int grade, int score) {}
List<Student> students = List.of(
        new Student("kim", 1, 85), new Student("lee", 2, 72),
        new Student("park", 1, 90), new Student("choi", 3, 60),
        new Student("an", 2, 95), new Student("jung", 3, 88));

Map<Boolean, List<String>> parted = students.stream()
        .collect(Collectors.partitioningBy(
                s -> s.score() >= 80,
                Collectors.mapping(Student::name, Collectors.toList())));

System.out.println("우등(true) : " + parted.get(true));
System.out.println("일반(false): " + parted.get(false));
```

<details>
<summary>정답 및 해설</summary>

```
우등(true) : [kim, park, an, jung]
일반(false): [lee, choi]
```

- `partitioningBy(Predicate)`는 조건의 참/거짓에 따라 **정확히 두 그룹**(`true`/`false`)으로 나누며, 결과 맵의 키는 항상 `true`와 `false` 두 개다(해당 그룹이 비어도 키는 존재한다).
- 두 번째 인자로 `Collectors.mapping(Student::name, Collectors.toList())`를 넘겼으므로, 각 그룹에 `Student` 객체가 아니라 **이름 문자열의 리스트**가 담긴다.
- 80점 이상: kim(85)·park(90)·an(95)·jung(88) → true 그룹. 80점 미만: lee(72)·choi(60) → false 그룹. 리스트 순서는 원본 스트림 순서를 따른다.
- `groupingBy`는 키 개수가 데이터에 따라 달라지지만, `partitioningBy`는 항상 두 그룹이라는 차이를 확인하라.
</details>

## 요약

- **스트림**은 데이터를 파이프라인(`소스 → 중간 연산 → 최종 연산`)으로 흘려보내며, "무엇을 할지"를 선언적으로 기술하는 처리 방식이다. (`StreamIntro`)
- **내부 반복자**는 요소 반복을 스트림에 위임하여, 개발자는 처리 로직에만 집중하고 라이브러리가 반복을 최적화(병렬 등)할 수 있게 한다. (`InternalIterator`)
- **중간 연산**은 스트림을 반환하며 **지연 평가**되고, **최종 연산**이 호출될 때 파이프라인 전체가 실행된다. 소비된 스트림은 재사용할 수 없다. (`IntermediateTerminal`)
- **스트림 얻기**는 리소스별로 `stream()`, `Arrays.stream()`, `Stream.of()`, `IntStream.range/rangeClosed` 등을 쓴다. `range`는 끝값 미포함, `rangeClosed`는 포함이다. (`StreamSources`)
- **필터링**은 `filter`로 조건을 만족하는 요소만 남기고, `distinct`로 중복을 제거하는 중간 연산이다. (`Filtering`)
- **매핑**은 `map`으로 1:1 변환, `mapToInt`로 숫자 스트림 변환, `flatMap`으로 1:N 평탄화를 수행한다. (`Mapping`)
- **정렬**은 `sorted()`(자연 순서)와 `sorted(Comparator)`로 하며, `Comparator.comparing`·`reversed`·`thenComparing`을 조립해 복합 기준을 만든다. (`Sorting`)
- **루핑**은 최종 연산 `forEach`로 각 요소를 처리하고, 중간 연산 `peek`로 흐름을 관찰(디버깅)한다. (`Looping`)
- **매칭**은 `anyMatch`·`allMatch`·`noneMatch`로 조건 만족 여부를 `boolean`으로 검사하며, 단락 평가로 조기 종료한다. (`Matching`)
- **기본 집계**는 `count`·`sum`·`average`·`max`·`min`과 `summaryStatistics`로 통계를 구한다. 빈 스트림 대비로 `Optional` 계열을 반환한다. (`BasicAggregation`)
- **커스텀 집계**는 `reduce`로 요소를 하나의 값으로 누적하며, 연산에 맞는 초기값(identity) 선택이 중요하다. (`CustomAggregation`)
- **수집**은 `collect`와 `Collectors`(`toList`·`toMap`·`groupingBy`·`joining`·`partitioningBy`)로 스트림을 컬렉션·맵·문자열로 모은다. (`Collecting`)
- **병렬 처리**는 `parallel()`로 여러 스레드가 동시에 처리하지만, 데이터가 적거나 공유 가변 상태를 수정하면 오히려 손해이므로 측정 후 도입한다. (`ParallelStream`)
