# Chapter 18 데이터 입출력

프로그램은 계산만 하고 끝나지 않는다. 파일에서 설정을 읽고, 결과를 파일에 저장하고, 네트워크로 데이터를 주고받는다. 이렇게 프로그램이 **외부와 데이터를 주고받는 일**을 입출력(I/O, Input/Output)이라고 한다.

자바는 이 입출력을 **스트림(stream)**이라는 하나의 일관된 모델로 다룬다. 파일이든 네트워크든 콘솔이든, 데이터가 흐르는 통로를 스트림으로 추상화했기 때문에 우리는 대상이 무엇이든 거의 같은 방식으로 코드를 쓸 수 있다.

이번 장에서 다룰 큰 줄기는 다음과 같다.

- **바이트 스트림 vs 문자 스트림** : 모든 데이터를 다루는 바이트 스트림(`InputStream`/`OutputStream`)과, 텍스트에 특화된 문자 스트림(`Reader`/`Writer`).
- **기반 스트림 vs 보조 스트림** : 실제로 데이터를 읽고 쓰는 기반 스트림과, 그 위에 기능(버퍼링·문자 변환·기본 타입 처리)을 덧씌우는 보조 스트림.
- **간편 유틸리티** : 몇 줄로 파일을 읽고 쓰게 해 주는 `File`과 `Files`.

각 소절에는 실행 가능한 실습 코드가 연결되어 있다. 개념을 읽은 뒤 직접 실행해 콘솔 출력을 눈으로 확인하며 익히자. 실습 코드는 모두 임시 파일을 만들어 동작하고 JVM 종료 시 자동으로 지워지므로 안심하고 반복 실행해도 된다.

## 18.1 입출력 스트림

**입출력 스트림(I/O Stream)**은 데이터가 흐르는 통로다. 프로그램과 데이터 소스(파일·네트워크·콘솔 등)를 연결해 데이터를 한 방향으로 흘려보낸다. 핵심 성질은 **단방향**이라는 것이다. 하나의 스트림은 읽기 전용(입력)이거나 쓰기 전용(출력)이며, 하나로 둘 다 하지는 못한다.

방향은 프로그램을 기준으로 나눈다.

- **입력(Input)** : 소스 → 프로그램 (읽기)
- **출력(Output)** : 프로그램 → 대상(destination) (쓰기)

또한 처리 단위에 따라 두 계열로 나뉜다. 텍스트를 다루면 문자 스트림, 그 외 모든 이진 데이터는 바이트 스트림을 쓴다.

| 구분 | 처리 단위 | 최상위 클래스 | 용도 |
|------|-----------|---------------|------|
| 바이트 입력 | 1 byte | `InputStream` | 모든 데이터(이미지·오디오·실행 파일 등) |
| 바이트 출력 | 1 byte | `OutputStream` | 모든 데이터 |
| 문자 입력 | 2 byte | `Reader` | 텍스트 |
| 문자 출력 | 2 byte | `Writer` | 텍스트 |

```
[입력]  데이터 소스  ──▶  스트림  ──▶  프로그램      (읽기)
[출력]  프로그램     ──▶  스트림  ──▶  데이터 대상    (쓰기)
```

바이트 스트림은 1바이트(8bit)씩, 문자 스트림은 1문자(char, 16bit)씩 처리한다. 문자 스트림은 문자 인코딩을 자동으로 처리해 주어 텍스트를 다루기 편하다.

## 18.2 바이트 출력 스트림

`OutputStream`의 대표 하위 클래스인 **`FileOutputStream`**은 파일에 바이트 단위로 데이터를 쓴다. 주요 메서드는 다음과 같다.

- `write(int b)` : 1바이트를 쓴다(`int`의 하위 8비트만 사용).
- `write(byte[] b)` : 바이트 배열 전체를 쓴다.
- `write(byte[] b, int off, int len)` : 배열의 일부만 쓴다.
- `flush()` : 내부 버퍼에 남은 데이터를 실제 대상으로 내보낸다.
- `close()` : 스트림을 닫고 자원을 반환한다.

기본은 **덮어쓰기**다. 생성자 두 번째 인자를 `true`로 주면 기존 내용 뒤에 **이어 쓰기(append)**가 된다.

```java
// try-with-resources: 블록을 벗어나면 자동으로 close() 된다
try (FileOutputStream fos = new FileOutputStream(file)) {
    fos.write(72);   // 'H'
    fos.write(105);  // 'i'
}

// 두 번째 인자 true → 이어쓰기(append)
try (FileOutputStream fos = new FileOutputStream(file, true)) {
    fos.write(new byte[]{33, 10}); // '!' + 개행
}
```

## 18.3 바이트 입력 스트림

`InputStream`의 대표 하위 클래스인 **`FileInputStream`**은 파일에서 바이트 단위로 데이터를 읽는다.

- `read()` : 1바이트를 읽어 **0~255의 `int`**로 반환한다. 더 읽을 데이터가 없으면 **-1(EOF)**을 반환한다.
- `read(byte[] b)` : 배열 크기만큼 읽고, **실제 읽은 바이트 수**를 반환한다.

`read()`의 반환 타입이 `byte`가 아니라 `int`인 이유가 중요하다. 유효한 값(0~255)과 파일의 끝을 뜻하는 **EOF(-1)**를 구분하려면 값의 범위가 더 넓어야 하기 때문이다. 그래서 입력의 표준 패턴은 다음과 같다.

```java
try (FileInputStream fis = new FileInputStream(file)) {
    int b;
    while ((b = fis.read()) != -1) {   // -1이면 파일 끝
        System.out.print((char) b);
    }
}
```

한 바이트씩 읽는 대신 `read(byte[])`로 한 번에 여러 바이트를 읽으면 I/O 호출 횟수가 줄어 더 효율적이다.

> 💻 실습: `./run.sh ByteInputStream`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch18/ByteInputStream.java`

## 18.4 문자 입출력 스트림

텍스트만 다룬다면 바이트 스트림보다 **문자 스트림**이 편하다. `Writer`/`Reader`의 하위 클래스인 **`FileWriter`/`FileReader`**는 텍스트 파일을 문자(char) 단위로 읽고 쓴다.

- 쓰기 : `write(String)`, `write(char[])`, `write(int c)`
- 읽기 : `read()` — 한 문자씩 읽고 EOF에서 -1을 반환

문자 스트림은 문자를 바이트로 바꿀 때 **인코딩(charset)**을 사용한다. 인코딩을 지정하지 않으면 플랫폼 기본값을 따르므로, 환경이 다르면 한글이 깨질 수 있다. JDK 11부터는 `FileWriter`/`FileReader` 생성자에 `Charset`을 직접 지정할 수 있으니 한글은 **UTF-8을 명시**하는 것이 안전하다.

```java
try (FileWriter w = new FileWriter(file, StandardCharsets.UTF_8)) {
    w.write("안녕하세요\n");
}
try (FileReader r = new FileReader(file, StandardCharsets.UTF_8)) {
    int c;
    while ((c = r.read()) != -1) System.out.print((char) c);
}
```

참고로 UTF-8에서 한글 한 글자는 보통 3바이트를 차지한다.

## 18.5 보조 스트림

**보조 스트림(filter stream)**은 스스로 데이터를 읽거나 쓰지 못한다. 대신 다른 스트림에 **연결(wrapping)**되어 버퍼링·문자 변환·기본 타입 처리 같은 부가 기능을 제공한다. 스트림을 겹겹이 감싸 기능을 더하는 이 구조는 **데코레이터(Decorator) 패턴**이다.

연결 구조는 "안쪽에 기반 스트림, 바깥쪽에 보조 스트림"으로 읽으면 된다.

```
쓰기:  프로그램 ─▶ BufferedOutputStream ─▶ FileOutputStream ─▶ 파일
                    (보조: 버퍼링)          (기반: 실제 출력)

읽기:  파일 ─▶ FileInputStream ─▶ BufferedInputStream ─▶ 프로그램
              (기반: 실제 입력)     (보조: 버퍼링)
```

```java
// 바깥이 보조 스트림, 안쪽이 기반 스트림
try (BufferedOutputStream bos =
         new BufferedOutputStream(new FileOutputStream(file))) {
    bos.write(1);
    bos.flush();
} // 바깥만 close() 해도 안쪽 FileOutputStream 까지 함께 닫힌다
```

핵심 규칙 두 가지: 보조 스트림은 여러 개를 겹쳐 연결할 수 있고, **가장 바깥 스트림만 `close()`하면 연결된 안쪽 스트림까지 연쇄적으로 닫힌다.**

> 💻 실습: `./run.sh AuxiliaryStream`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch18/AuxiliaryStream.java`

## 18.6 문자 변환 스트림

바이트 스트림과 문자 스트림 사이에 **다리**를 놓는 보조 스트림이 있다.

- **`InputStreamReader`** : 바이트 입력 스트림 → 문자 입력 스트림
- **`OutputStreamWriter`** : 문자 출력 스트림 → 바이트 출력 스트림

```
쓰기:  문자열 ─▶ OutputStreamWriter ─▶ FileOutputStream ─▶ 파일
                 (문자 → 바이트, charset)

읽기:  파일 ─▶ FileInputStream ─▶ InputStreamReader ─▶ 문자열
                                   (바이트 → 문자, charset)
```

변환 스트림의 진짜 가치는 **인코딩을 명시적으로 지정**할 수 있다는 점이다. 네트워크 소켓처럼 바이트 스트림만 주어지는 상황에서 문자로 다루고 싶을 때, 인코딩을 못 박아 한글 깨짐을 막을 수 있다.

```java
try (OutputStreamWriter osw = new OutputStreamWriter(
         new FileOutputStream(file), StandardCharsets.UTF_8)) {
    osw.write("변환 스트림 테스트");
}
```

쓸 때와 읽을 때 인코딩이 다르면 한글이 깨진다. 양쪽 인코딩을 일치시키자.

> 💻 실습: `./run.sh CharConversionStream`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch18/CharConversionStream.java`

## 18.7 성능 향상 스트림

파일이나 네트워크에 **1바이트/1문자씩** 직접 접근하면 매번 느린 I/O가 발생한다. **버퍼링(buffering)**은 데이터를 메모리 버퍼에 모았다가 한 번에 처리해 실제 I/O 호출 횟수를 크게 줄인다. 그래서 성능이 향상된다.

문자 스트림에는 **`BufferedReader`/`BufferedWriter`**가 있으며, 버퍼링뿐 아니라 편리한 줄 단위 메서드도 제공한다.

- `BufferedReader.readLine()` : 한 줄을 통째로 읽는다. EOF이면 **`null`**을 반환한다.
- `BufferedWriter.newLine()` : 플랫폼에 맞는 줄바꿈 문자를 쓴다.

```java
try (BufferedWriter bw = new BufferedWriter(
         new FileWriter(file, StandardCharsets.UTF_8))) {
    bw.write("첫 줄");
    bw.newLine();
}
try (BufferedReader br = new BufferedReader(
         new FileReader(file, StandardCharsets.UTF_8))) {
    String line;
    while ((line = br.readLine()) != null) {   // null이면 EOF
        System.out.println(line);
    }
}
```

주의할 점: 입력 스트림은 EOF를 **`read()`는 -1**, **`readLine()`은 `null`**로 알린다. 반복 종료 조건이 다르니 헷갈리지 말자.

## 18.8 기본 타입 스트림

**`DataOutputStream`/`DataInputStream`**은 자바의 기본 타입(`int`·`double`·`boolean`·`char` 등)과 문자열을 **이진 형태 그대로** 읽고 쓰는 보조 스트림이다.

- `writeInt`/`readInt`, `writeDouble`/`readDouble`, `writeBoolean`/`readBoolean`
- `writeUTF`/`readUTF` : 문자열(수정된 UTF-8 형식)

가장 중요한 규칙은 **쓴 순서와 읽는 순서가 정확히 일치**해야 한다는 것이다. `int`로 썼으면 `int`로 읽어야 하며, 순서나 타입이 어긋나면 값이 깨진다.

```java
try (DataOutputStream dos = new DataOutputStream(new FileOutputStream(file))) {
    dos.writeUTF("상품A");
    dos.writeInt(30000);
    dos.writeDouble(4.5);
}
try (DataInputStream dis = new DataInputStream(new FileInputStream(file))) {
    String name  = dis.readUTF();    // 쓴 순서와 동일하게
    int    price = dis.readInt();
    double rating = dis.readDouble();
}
```

이진 형태라 사람이 파일을 열어 읽기는 어렵지만, 타입 정보를 유지하며 저장/복원할 수 있어 편리하다.

> 💻 실습: `./run.sh DataStream`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch18/DataStream.java`

## 18.9 프린트 스트림

**`PrintStream`/`PrintWriter`**는 데이터를 **사람이 읽기 좋은 텍스트 형태**로 출력하는 편의 스트림이다. `print()`, `println()`, `printf()`, `format()` 같은 익숙한 메서드를 제공한다.

- **`PrintStream`** : 바이트 기반. 우리가 늘 쓰는 `System.out`, `System.err`이 바로 `PrintStream`이다.
- **`PrintWriter`** : 문자 기반. 텍스트 출력에 더 적합하다.

특징 두 가지를 기억하자. 프린트 스트림은 출력 중 `IOException`을 **던지지 않고** 내부에 담아 둔다(`checkError()`로 확인). 또 `autoFlush` 옵션으로 `println` 등에서 자동 flush 되게 할 수 있다.

```java
System.out.printf("이름=%s, 점수=%d, 비율=%.2f%n", "홍길동", 95, 0.87);

try (PrintWriter pw = new PrintWriter(
         Files.newBufferedWriter(path, StandardCharsets.UTF_8))) {
    pw.println("리포트 제목");
    pw.printf("합계: %,d 원%n", 1234567);   // 천 단위 구분, 정렬 등 서식 지원
}
```

## 18.10 객체 스트림

객체를 바이트 형태로 변환해 저장·전송하는 것을 **직렬화(serialization)**, 그 바이트를 다시 객체로 복원하는 것을 **역직렬화(deserialization)**라고 한다. **`ObjectOutputStream.writeObject()`**로 직렬화하고, **`ObjectInputStream.readObject()`**로 역직렬화한다.

직렬화에는 세 가지 키워드가 얽힌다.

- **`Serializable`** : 직렬화 대상 클래스는 이 마커 인터페이스를 구현해야 한다. 안 하면 `NotSerializableException`이 난다.
- **`transient`** : 이 키워드가 붙은 필드는 직렬화에서 **제외**된다. 역직렬화 시 기본값(참조형은 `null`)으로 복원되므로 비밀번호 같은 민감 정보에 쓴다.
- **`serialVersionUID`** : 클래스 버전 식별자. 직렬화/역직렬화 호환성 검증에 쓰인다.

```java
static class Member implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private transient String password;   // 직렬화 제외 → 복원 시 null
}

try (ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream(file))) {
    oos.writeObject(member);
}
try (ObjectInputStream ois = new ObjectInputStream(new FileInputStream(file))) {
    Member restored = (Member) ois.readObject();   // 형변환 필요
}
```

## 18.11 File과 Files 클래스

스트림이 데이터를 흘려보내는 통로라면, **`File`과 `Files`**는 파일 자체(경로·존재 여부·크기 등)를 다루는 도구다.

- **`java.io.File`** : 파일/디렉터리의 경로를 표현하는 전통적 클래스. `exists()`, `length()`, `isDirectory()`, `list()`, `mkdir()`, `delete()` 등 메타 정보와 관리 기능을 제공한다. 실제 입출력은 스트림이 담당한다.
- **`java.nio.file.Files`** : NIO.2에서 도입된 유틸리티 클래스로 `Path`와 함께 쓴다. 파일 읽기/쓰기를 **한 줄로** 처리하는 정적 메서드를 제공한다.

```java
// Files: 간편 입출력 (현대 자바 권장)
Files.write(path, List.of("첫째 줄", "둘째 줄"), StandardCharsets.UTF_8);
List<String> lines = Files.readAllLines(path, StandardCharsets.UTF_8);
long size = Files.size(path);

// File: 메타 정보 조회
File f = path.toFile();
System.out.println(f.getName() + ", " + f.exists() + ", " + f.length());
```

간단한 파일 작업이라면 스트림을 직접 열지 않고 `Files`의 정적 메서드로 끝내는 편이 훨씬 간결하다. 현대 자바에서는 `File`보다 **`Path`/`Files` 사용이 권장**된다.

## ⚠️ 흔한 오해와 함정

- **스트림을 닫지 않는다.** 스트림은 운영체제 자원(파일 핸들 등)을 점유한다. `close()`를 빠뜨리면 자원이 새고, 버퍼에 남은 데이터가 파일에 반영되지 않을 수 있다. 항상 **try-with-resources**로 열어 자동으로 닫히게 하자.
- **`flush()` 없이 버퍼링 스트림을 방치한다.** 버퍼링 스트림에 쓴 데이터는 일단 메모리 버퍼에 쌓인다. `flush()`나 `close()` 전에는 실제 파일에 반영되지 않는다. try-with-resources를 쓰면 `close()`가 자동 호출되며 flush까지 처리된다.
- **EOF 판정 값을 혼동한다.** 바이트/문자 `read()`는 EOF에서 **-1**을 반환하지만, `BufferedReader.readLine()`은 **`null`**을 반환한다. 반복 종료 조건(`!= -1` vs `!= null`)을 대상에 맞게 써야 한다.
- **인코딩을 지정하지 않아 한글이 깨진다.** 문자 스트림에서 인코딩을 생략하면 플랫폼 기본값을 따른다. 쓸 때와 읽을 때 환경이 다르면 한글이 깨진다. 한글은 **UTF-8을 명시**하자.
- **`DataStream`에서 쓴 순서와 다르게 읽는다.** `writeInt` → `writeDouble` 순서로 썼다면 `readInt` → `readDouble` 순서로 읽어야 한다. 순서나 타입이 어긋나면 값이 깨진다.
- **텍스트에 바이트 스트림을, 이진 데이터에 문자 스트림을 쓴다.** 텍스트는 인코딩을 처리해 주는 문자 스트림이, 이미지·오디오 같은 이진 데이터는 바이트 스트림이 맞다. 이진 데이터를 문자 스트림으로 읽으면 인코딩 변환 과정에서 데이터가 손상될 수 있다.
- **직렬화 대상에 `Serializable`을 빠뜨린다.** `Serializable`을 구현하지 않은 객체를 `writeObject()`에 넘기면 `NotSerializableException`이 난다. 또 `transient` 필드는 복원 시 기본값이 된다는 점을 잊지 말자.

**과제 0. 바이트 하나씩 쓰고 확인하기 (바이트 스트림 · FileOutputStream)** — 쉬움

임시 파일을 하나 만들고, `FileOutputStream`으로 `write(int)`를 이용해 `'H'`(72), `'i'`(105) 두 바이트를 쓴 다음, **두 번째 인자를 `true`로 준 append 모드**의 `FileOutputStream`으로 `write(byte[])`를 이용해 `'!'`(33)과 개행(10)을 **이어 써라.** 마지막에 `Files.size(...)`로 최종 파일 크기를 출력하고(기대값 4), `Files.readAllBytes(...)`로 읽어 각 바이트를 `(b & 0xFF)`로 부호 없는 정수로 출력해 `72 105 33 10`이 나오는지 확인하라. 파일은 `Files.createTempFile(...)`(`.bin`)로 만들고 `deleteOnExit()`로 정리한다.

<details>
<summary>힌트</summary>

- 자바의 `byte`는 부호가 있어 128 이상이 음수로 보일 수 있다. 출력할 때 `(b & 0xFF)`로 **부호 없는 0~255**로 바꿔 확인하라.
</details>

**과제 3. 상품 정보 이진 저장소 (기본 타입 스트림 · DataOutputStream/DataInputStream)** — 응용

상품 하나의 정보를 이진 파일로 저장했다가 복원하는 프로그램을 작성하라. `DataOutputStream`으로 문자열(상품명, `writeUTF`), 정수(가격, `writeInt`), 실수(평점, `writeDouble`), 논리값(재고 여부, `writeBoolean`)을 **순서대로** 임시 파일에 쓴 뒤, `DataInputStream`으로 **똑같은 순서로** 읽어 각 값을 출력하라. 파일은 임시 경로(`.dat`)에 만들고 `deleteOnExit()`로 정리한다.

<details>
<summary>힌트</summary>

- 가장 중요한 규칙: **쓴 순서 = 읽는 순서, 쓴 타입 = 읽는 타입**. `writeUTF → writeInt → writeDouble → writeBoolean` 순으로 썼다면 `readUTF → readInt → readDouble → readBoolean` 순으로 읽어야 한다. 순서가 어긋나면 값이 깨진다.
- 이진 형태라 파일을 텍스트 편집기로 열면 알아보기 어렵다. 그래도 타입 정보를 유지하며 정확히 복원되는 것을 출력으로 확인하라.
</details>

**과제 4. 비밀번호를 숨기는 회원 저장기 (객체 직렬화 · transient)** — 응용

`Serializable`을 구현한 `Member` 클래스(필드: `name`, `age`, 그리고 `transient`가 붙은 `password`)를 만들어라. `serialVersionUID`도 선언한다. 회원 객체를 하나 만들어 `ObjectOutputStream.writeObject()`로 임시 파일에 저장하고, `ObjectInputStream.readObject()`로 다시 읽어 복원하라. 복원된 객체를 출력해, **`transient`가 붙은 `password`가 `null`로 복원**되는 것을 눈으로 확인하라.

<details>
<summary>힌트</summary>

- 직렬화 대상 클래스는 반드시 `implements Serializable` 해야 한다. 빠뜨리면 `writeObject()`에서 `NotSerializableException`이 난다. `private static final long serialVersionUID = 1L;` 도 함께 선언하자.
- 민감 정보인 `password`에 `transient`를 붙이면 직렬화에서 **제외**된다. 그래서 역직렬화 후에는 기본값(참조형이므로 `null`)이 된다.
- `readObject()`의 반환 타입은 `Object`이므로 `(Member) ois.readObject()` 로 **형변환**이 필요하고, 메소드에 `throws ClassNotFoundException`을 붙여야 한다. `toString()`을 오버라이드해두면 복원 결과를 보기 편하다.
</details>

**과제 5. 파일 탐정 (Files 간편 입출력 · File 메타 정보)** — 도전

`Files`와 `File`을 함께 써서 파일을 다루는 프로그램을 작성하라. (1) `Files.write(path, List.of("첫째 줄", "둘째 줄", "셋째 줄"), StandardCharsets.UTF_8)`로 임시 파일에 여러 줄을 **한 번에** 저장하고, `Files.readAllLines(...)`로 다시 읽어 출력한다. (2) 같은 파일에 대해 `File`(`getName`, `exists`, `isDirectory`, `length`)과 `Files`(`Files.exists`, `Files.size`, `Files.isDirectory`)로 **동일한 메타 정보를 두 방식으로** 조회해 비교한다. (3) 임시 파일이 위치한 **상위 디렉터리의 목록**을 `File.list()`로 얻어 앞 몇 개만 출력하라. 파일은 임시 경로에 만들고 정리한다.

<details>
<summary>힌트</summary>

- `Files.write`, `Files.readAllLines`, `Files.size`는 스트림을 직접 열지 않고 **한 줄로** 파일 작업을 끝내는 정적 메서드다. `Path`와 함께 쓴다.
- `File`로 넘어가려면 `File file = temp.toFile();` 한다. `file.length()`(바이트 크기), `Files.size(temp)`가 같은 값을 주는지 비교해보라.
- 상위 디렉터리는 `file.getParentFile()`로 얻고 `parent.list()`로 자식 이름 배열을 받는다. `list()`는 `null`일 수 있으니 확인 후, `Math.min(3, children.length)`처럼 앞 몇 개만 출력하면 깔끔하다.
</details>

**과제 6. 버퍼링 성능 체감기 (보조·성능 스트림 · Buffered vs 비버퍼)** — 도전

같은 데이터를 **버퍼링 없이** 쓸 때와 **버퍼링을 얹어** 쓸 때의 소요 시간을 비교하는 프로그램을 작성하라. 임시 파일 두 개를 만들고, (1) `FileWriter`만으로 `write(int)`를 이용해 문자 하나씩 예: 10만 번 쓰고 걸린 시간을 측정한다. (2) 같은 작업을 `BufferedWriter(new FileWriter(...))`로 감싸 수행하고 시간을 측정한다. 두 시간을 `System.nanoTime()`으로 재서 출력하고, **버퍼링 쪽이 빠른지** 확인하라. 두 파일 모두 `Files.size(...)`로 크기가 같은지도 비교하라. 파일은 임시 경로에 만들고 정리한다.

<details>
<summary>힌트</summary>

- 시간 측정은 `long start = System.nanoTime(); ... long ms = (System.nanoTime() - start) / 1_000_000;` 패턴을 쓴다.
- 버퍼링은 데이터를 메모리에 모았다가 한 번에 내보내 **실제 I/O 호출 횟수를 줄인다.** 그래서 한 문자씩 쓰는 작업일수록 `BufferedWriter`의 효과가 크다.
- 두 스트림 모두 반드시 try-with-resources로 닫아라. 닫히면서 남은 버퍼가 flush되어 파일에 반영되므로, 그래야 두 파일 크기가 같아진다.
- 환경에 따라 절대 시간은 다르지만, 보통 버퍼링 쪽이 뚜렷하게 빠르다.
</details>

### A. 개념 서술

**1.** **바이트 스트림**과 **문자 스트림**의 차이를 처리 단위와 용도 관점에서 설명하고, 각 계열의 **입력·출력 최상위 클래스** 이름을 각각 쓰시오.

<details>
<summary>정답 및 해설</summary>

- **처리 단위** : 바이트 스트림은 1바이트(8bit)씩, 문자 스트림은 1문자(char, 16bit)씩 처리한다.
- **용도** : 바이트 스트림은 이미지·오디오·실행 파일 등 **모든 이진 데이터**를 다루고, 문자 스트림은 **텍스트**에 특화되어 있다. 문자 스트림은 문자↔바이트 변환 시 **인코딩(charset)**을 자동으로 처리해 주므로 텍스트를 다루기 편하다.
- **최상위 클래스**
  - 바이트 입력 : `InputStream` / 바이트 출력 : `OutputStream`
  - 문자 입력 : `Reader` / 문자 출력 : `Writer`
</details>

**2.** **보조 스트림(filter/보조 스트림)**이란 무엇이며 **기반 스트림**과 어떻게 다른가? 보조 스트림을 겹겹이 감싸 기능을 더하는 이 구조가 어떤 디자인 패턴에 해당하는지도 쓰시오.

<details>
<summary>정답 및 해설</summary>

- **기반 스트림**은 파일·네트워크 같은 실제 대상에 **직접 연결되어 데이터를 읽고 쓴다**(`FileInputStream`, `FileOutputStream` 등).
- **보조 스트림**은 스스로 입출력하지 못한다. 다른 스트림을 **감싸(wrapping)** 버퍼링·문자 변환·기본 타입 처리 같은 부가 기능만 더한다(`BufferedInputStream`, `DataOutputStream`, `InputStreamReader` 등).
- 이렇게 스트림을 겹겹이 감싸 기능을 확장하는 구조는 **데코레이터(Decorator) 패턴**이다.
- 핵심 규칙 : 보조 스트림은 여러 개를 겹쳐 연결할 수 있고, **가장 바깥 스트림만 `close()`하면 안쪽 스트림까지 연쇄적으로 닫힌다.**
</details>

**3.** 객체 직렬화에서 `Serializable`, `transient`, `serialVersionUID`는 각각 어떤 역할을 하는가? 어떤 필드에 `transient`를 붙이는 것이 적절한지 예를 들어 설명하시오.

<details>
<summary>정답 및 해설</summary>

- **`Serializable`** : 직렬화 대상 클래스가 구현해야 하는 **마커 인터페이스**다. 구현하지 않은 객체를 `writeObject()`에 넘기면 `NotSerializableException`이 발생한다.
- **`transient`** : 이 키워드가 붙은 필드는 **직렬화에서 제외**된다. 역직렬화 시 기본값(참조형은 `null`, `int`는 `0`, `boolean`은 `false`)으로 복원된다.
- **`serialVersionUID`** : 클래스의 **버전 식별자**로, 직렬화/역직렬화 시 클래스 호환성을 검증하는 데 쓰인다. 명시하지 않으면 컴파일러가 자동 생성하는데, 클래스가 조금만 바뀌어도 값이 달라져 `InvalidClassException`이 날 수 있으므로 직접 선언하는 것이 안전하다.
- **`transient` 적용 예** : 비밀번호·주민번호 같은 **민감 정보**, 또는 다른 필드로부터 다시 계산할 수 있어 저장할 필요가 없는 **파생 값(캐시)** 등에 붙인다.
</details>

**4.** 다음 코드의 출력은 무엇인가? `read()`가 EOF를 만났을 때 어떤 값이 나오는지에 주목하시오.

```java
try (FileOutputStream fos = new FileOutputStream(file)) {
    fos.write(65);   // 'A'
    fos.write(66);   // 'B'
}
try (FileInputStream fis = new FileInputStream(file)) {
    int b;
    while ((b = fis.read()) != -1) {
        System.out.print((char) b);
    }
    System.out.print(fis.read());   // 마지막 한 번 더
}
```

<details>
<summary>정답 및 해설</summary>

출력 : `AB-1`

- `write(65)`, `write(66)`으로 `'A'`(65), `'B'`(66) 두 바이트를 쓴다.
- `while` 루프에서 65, 66을 차례로 읽어 `A`, `B`를 출력한다. 세 번째 `read()`는 EOF이므로 **-1**을 반환해 루프가 끝난다.
- 루프 종료 후 `fis.read()`를 한 번 더 호출하면 이미 EOF에 도달했으므로 여전히 **-1**을 반환하고, `System.out.print(-1)`이 `-1`을 출력한다.
- 핵심 : `read()`의 EOF 신호는 **`null`이 아니라 `-1`**이다.
</details>

**5.** 다음 두 코드의 출력을 각각 예측하시오. 같은 파일을 **바이트 스트림**과 **문자 스트림**으로 읽었을 때 한글이 어떻게 나오는가? (파일은 UTF-8로 저장되어 있다.)

```java
// (가) 문자 스트림으로 읽기
try (FileReader r = new FileReader(file, StandardCharsets.UTF_8)) {
    int c;
    while ((c = r.read()) != -1) System.out.print((char) c);
}

// (나) 바이트 스트림으로 읽어 char로 강제 변환
try (FileInputStream in = new FileInputStream(file)) {
    int b;
    while ((b = in.read()) != -1) System.out.print((char) b);
}
```
(파일 내용은 `"가"` 한 글자만 저장되어 있다.)

<details>
<summary>정답 및 해설</summary>

- **(가)** : `가` — 문자 스트림 `FileReader`는 UTF-8을 해석해 한글 한 글자를 온전히 복원한다.
- **(나)** : 알아볼 수 없는 **깨진 문자 3개** — `"가"`는 UTF-8에서 **3바이트**(`0xEA 0xB0 0x80`)로 저장된다. 바이트 스트림은 이를 한 바이트씩(3번) 읽고 각각을 `(char)`로 캐스팅하므로, 한 글자가 아니라 3개의 엉뚱한 문자가 출력된다.
- 핵심 : 텍스트, 특히 한글은 **바이트 스트림으로 한 바이트씩 읽어 char로 캐스팅하면 안 된다.** 인코딩을 해석하는 문자 스트림(또는 변환 스트림)을 써야 한다.
</details>

**6.** 다음 코드의 출력은? `DataInputStream`으로 읽는 순서가 쓴 순서와 어긋나면 어떤 일이 벌어지는지 판단하시오.

```java
try (DataOutputStream dos = new DataOutputStream(new FileOutputStream(file))) {
    dos.writeInt(100);
    dos.writeDouble(3.14);
}
try (DataInputStream dis = new DataInputStream(new FileInputStream(file))) {
    double d = dis.readDouble();   // 순서를 바꿔 double 부터 읽음
    int    i = dis.readInt();
    System.out.println(d + ", " + i);
}
```

<details>
<summary>정답 및 해설</summary>

- **값이 깨진다.** `writeInt`(4바이트) → `writeDouble`(8바이트) 순으로 총 12바이트가 저장되어 있다.
- 읽을 때 `readDouble()`을 먼저 호출하면 **앞 8바이트**(int 4바이트 + double의 앞 4바이트)를 double로 해석하므로 `3.14`가 아닌 엉뚱한 실수가 나온다.
- 이어서 `readInt()`는 **남은 4바이트**(double의 뒷부분)를 int로 해석하므로 역시 엉뚱한 정수가 된다. 남은 바이트가 부족하면 `EOFException`이 날 수도 있다.
- 핵심 : `DataStream`은 **쓴 순서 = 읽는 순서, 쓴 타입 = 읽는 타입**을 반드시 지켜야 한다. 파일에는 타입 경계 정보가 저장되지 않기 때문이다.
</details>

**8.** 다음 코드의 출력을 예측하시오. `readLine()`이 EOF에서 반환하는 값과, `read()`가 EOF에서 반환하는 값의 차이에 주목하라.

```java
try (BufferedWriter bw = new BufferedWriter(new FileWriter(file, StandardCharsets.UTF_8))) {
    bw.write("one");  bw.newLine();
    bw.write("two");  bw.newLine();
}
try (BufferedReader br = new BufferedReader(new FileReader(file, StandardCharsets.UTF_8))) {
    String line;
    int count = 0;
    while ((line = br.readLine()) != null) count++;
    System.out.println("줄 수 = " + count);
    System.out.println("한 번 더 = " + br.readLine());
}
```

<details>
<summary>정답 및 해설</summary>

```
줄 수 = 2
한 번 더 = null
```

- 루프 종료 후 다시 `readLine()`을 호출해도 EOF이므로 여전히 `null`이다.
- 핵심 : `readLine()`의 EOF 신호는 **`null`**이고, 바이트/문자 `read()`의 EOF 신호는 **`-1`**이다. 종료 조건(`!= null` vs `!= -1`)을 대상에 맞게 써야 한다.

### C. 오류 찾기

**9.** 다음 코드에는 **자원 관리 관점의 문제**가 있다. 무엇이 잘못되었고 어떤 결과가 생길 수 있는지 설명하고, 올바르게 고치시오.

```java
BufferedWriter bw = new BufferedWriter(new FileWriter(file, StandardCharsets.UTF_8));
bw.write("중요한 데이터");
// close() 없이 프로그램 종료
```

<details>
<summary>정답 및 해설</summary>

- **문제 1 (자원 누수)** : `close()`를 호출하지 않아 파일 핸들 등 운영체제 자원이 반환되지 않는다.
- **문제 2 (데이터 유실)** : `BufferedWriter`에 쓴 데이터는 일단 **메모리 버퍼**에 쌓인다. `flush()`나 `close()` 전에는 실제 파일에 반영되지 않으므로, close 없이 종료하면 `"중요한 데이터"`가 파일에 **기록되지 않을 수 있다.**
- **수정** : try-with-resources로 열어 블록을 벗어날 때 자동으로 `close()`(내부적으로 flush 포함)되게 한다.

```java
try (BufferedWriter bw = new BufferedWriter(
        new FileWriter(file, StandardCharsets.UTF_8))) {
    bw.write("중요한 데이터");
}   // 블록을 벗어나면 flush 후 close 가 자동 호출된다
```
</details>

**10.** 다음 코드는 한글을 저장했는데 읽을 때 깨진다. **원인**을 찾고 고치시오.

```java
try (FileWriter w = new FileWriter(file, StandardCharsets.UTF_8)) {
    w.write("한글 테스트");
}
try (FileReader r = new FileReader(file, StandardCharsets.ISO_8859_1)) {

}
```

<details>
<summary>정답 및 해설</summary>

- **원인** : 쓸 때는 **UTF-8**로 인코딩했는데 읽을 때는 **ISO-8859-1**로 디코딩했다. 인코딩이 불일치하면 UTF-8의 멀티바이트(한글 3바이트)가 잘못 해석되어 한글이 깨진다.
- **수정** : 읽을 때도 **쓸 때와 같은 인코딩(UTF-8)**을 명시한다.

```java
try (FileReader r = new FileReader(file, StandardCharsets.UTF_8)) {
    ...
}
```

- 핵심 : 문자 스트림에서 **쓰기 인코딩과 읽기 인코딩은 반드시 일치**해야 한다. 인코딩을 아예 생략하면 플랫폼 기본값을 따르므로 환경이 바뀌면 같은 문제가 생긴다.
</details>

**11.** 다음 코드는 `Member` 객체를 파일에 저장하려다 실행 시 예외가 발생한다. 어떤 예외가 왜 발생하며 어떻게 고쳐야 하는가?

### D. 빈칸 채우기

**12.** 다음은 파일에 UTF-8로 한 줄을 쓰고, 다시 읽어 출력하는 코드다. 빈칸 `(ㄱ)`~`(ㄹ)`을 채워 완성하시오. 스트림은 **try-with-resources**로 열어야 한다.

```java
// 쓰기
try ( ______(ㄱ)______ bw = new BufferedWriter(
          new FileWriter(file, StandardCharsets.UTF_8))) {
    bw.write("hello");
    bw.______(ㄴ)______;          // 플랫폼에 맞는 줄바꿈
}

// 읽기
try (BufferedReader br = new BufferedReader(

    }
}
```

<details>

### E. 참·거짓 (이유 포함)

**13.** 다음 각 문장이 참인지 거짓인지 판별하고 **이유**를 쓰시오.

1. 보조 스트림 `BufferedOutputStream`으로 감싼 경우, 안쪽 `FileOutputStream`을 따로 `close()`해야 자원이 반환된다.
2. `FileInputStream.read()`와 `BufferedReader.readLine()`은 둘 다 EOF에서 `-1`을 반환한다.
3. `transient` 필드는 직렬화에서 제외되므로, 역직렬화 후 그 필드는 항상 원래 값으로 복원된다.

<details>
<summary>정답 및 해설</summary>

1. **거짓.** 가장 바깥 스트림(`BufferedOutputStream`)만 `close()`하면 연결된 안쪽 `FileOutputStream`까지 **연쇄적으로 닫힌다.** 안쪽을 따로 닫을 필요가 없다(오히려 바깥을 닫기 전에 안쪽을 먼저 닫으면 문제가 생긴다).
2. **거짓.** `FileInputStream.read()`는 EOF에서 `-1`을 반환하지만, `BufferedReader.readLine()`은 EOF에서 **`null`**을 반환한다. 종료 조건이 서로 다르다.
3. **거짓.** `transient` 필드는 값이 저장되지 않으므로 역직렬화 후 **타입의 기본값**(참조형 `null`, `int` `0` 등)이 된다. 원래 값으로 복원되지 않는다.
</details>

**14.** 다음 코드에서 파일에 실제로 기록되는 **바이트 수**는 몇 바이트인가? (파일은 append 없이 매번 새로 생성된다.) 그리고 이 코드가 정상 동작하는지 참·거짓으로 판단하고 이유를 쓰시오.

```java
try (FileOutputStream fos = new FileOutputStream(file)) {
    fos.write(72);            // 'H'
    fos.write(105);           // 'i'
    fos.write(new byte[]{33, 10});  // '!' , 개행
}
```

<details>
<summary>정답 및 해설</summary>

- **바이트 수 : 4바이트.** `write(72)` 1바이트, `write(105)` 1바이트, `write(byte[]{33,10})` 2바이트로 총 4바이트가 기록된다(`H`, `i`, `!`, 개행).
- **정상 동작 : 참.** try-with-resources로 열었으므로 블록을 벗어날 때 자동으로 `close()`되며, 이때 flush까지 처리되어 4바이트가 파일에 확실히 반영된다. `flush()`를 명시적으로 호출하지 않아도 close 과정에서 처리된다.
- 참고 : `write(int)`는 int의 **하위 8비트만** 사용한다. 그래서 0~255 범위 값을 한 바이트로 쓴다.
</details>

## 요약

- **입출력 스트림**은 프로그램과 외부(파일·네트워크·콘솔)를 잇는 단방향 통로다. 텍스트는 문자 스트림, 그 외 이진 데이터는 바이트 스트림을 쓴다. (`IoStreamIntro`)
- **바이트 출력 스트림** `FileOutputStream`은 `write(int)`/`write(byte[])`로 바이트를 쓰며, 생성자 두 번째 인자 `true`로 이어쓰기(append)가 된다. (`ByteOutputStream`)
- **바이트 입력 스트림** `FileInputStream`은 `read()`로 한 바이트씩 읽고 **EOF에서 -1**을 반환한다. `while ((b = read()) != -1)`이 표준 패턴이다. (`ByteInputStream`)
- **문자 입출력 스트림** `FileWriter`/`FileReader`는 텍스트를 문자 단위로 처리한다. 한글은 **UTF-8을 명시**해 인코딩 문제를 피한다. (`CharStream`)
- **보조 스트림**은 스스로 입출력하지 못하고 기반 스트림을 감싸(데코레이터) 기능을 더한다. **바깥 스트림만 닫으면** 연결된 안쪽 스트림까지 함께 닫힌다. (`AuxiliaryStream`)
- **문자 변환 스트림** `InputStreamReader`/`OutputStreamWriter`는 바이트↔문자를 이어 주며 **인코딩을 명시**할 수 있다. (`CharConversionStream`)
- **성능 향상 스트림** `BufferedReader`/`BufferedWriter`는 버퍼링으로 I/O 호출을 줄여 빠르다. `readLine()`은 EOF에서 **`null`**을, `newLine()`은 줄바꿈을 쓴다. (`PerformanceStream`)
- **기본 타입 스트림** `DataOutputStream`/`DataInputStream`은 기본 타입·문자열을 이진으로 저장한다. **쓴 순서 그대로 읽어야** 값이 복원된다. (`DataStream`)
- **프린트 스트림** `PrintStream`/`PrintWriter`는 `print`/`println`/`printf`로 사람이 읽기 좋은 텍스트를 출력한다. `System.out`이 대표적인 `PrintStream`이다. (`PrintStreamDemo`)
- **객체 스트림** `ObjectOutputStream`/`ObjectInputStream`은 객체를 직렬화/역직렬화한다. 대상은 `Serializable`을 구현해야 하고, `transient` 필드는 직렬화에서 제외된다. (`ObjectStream`)
- **File과 Files**는 파일의 경로·메타 정보를 다룬다. `File`은 전통적 표현, `Files`는 `Path` 기반의 간편 입출력 유틸리티로 현대 자바에서 권장된다. (`FileAndFiles`)
