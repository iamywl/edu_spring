# Chapter 19 네트워크 입출력

지금까지 다룬 프로그램은 한 대의 컴퓨터 안에서 완결되었다. 하지만 우리가 매일 쓰는 웹, 채팅, 게임, 스트리밍은 모두 **여러 대의 컴퓨터가 서로 데이터를 주고받는** 프로그램이다. 이렇게 컴퓨터들이 연결되어 데이터를 교환하는 환경을 **네트워크(network)**라고 하고, 그 위에서 동작하는 프로그램을 만드는 일을 **네트워크 프로그래밍(network programming)**이라고 한다.

자바는 `java.net` 패키지로 네트워크 프로그래밍을 표준 지원한다. 이번 장에서는 **IP·포트·소켓**이라는 기본 개념부터 시작해, 신뢰성 있는 **TCP** 통신과 빠른 **UDP** 통신, 서버가 여러 요청을 **동시에** 처리하는 법, 데이터를 주고받는 표준 형식인 **JSON**, 그리고 이를 종합한 **간이 TCP 채팅**까지 단계적으로 익힌다.

이 장의 실습 코드는 모두 **자기 자신(loopback, 127.0.0.1)** 안에서 서버와 클라이언트가 함께 동작하도록 작성되어 있다. 인터넷 연결이 없어도, 별도 서버를 띄우지 않아도 그대로 실행되며, 타임아웃과 `join`을 넣어 절대 멈춰 있지(hang) 않고 수초 내에 종료된다. 각 소절을 읽은 뒤 직접 실행해 결과를 눈으로 확인하며 익히자.

> 📖 이 장은 자바 API 사용법에 초점을 둔다. 3-way handshake의 상태 전이, 재전송·혼잡 제어, DNS의 내부 동작 같은 **깊은 네트워크 이론**은 별도 문서 **`docs/CS_네트워크_개념서/`** (특히 `03-전송-계층.md`, `04-DNS.md`)에서 다룬다. 소절마다 해당 참조를 함께 적어 두었다.

## 19.1 네트워크 기초

네트워크로 데이터를 주고받으려면 세 가지 질문에 답할 수 있어야 한다. **어느 컴퓨터에게(IP)**, **그 컴퓨터의 어느 프로그램에게(포트)**, **어떤 약속으로(프로토콜)** 보낼 것인가이다.

- **IP 주소(IP Address)** : 네트워크에 연결된 컴퓨터를 식별하는 주소다. 예) `192.168.0.10`. 자기 자신을 가리키는 특별한 주소 `127.0.0.1`(loopback, `localhost`)이 있다.
- **포트(Port)** : 한 컴퓨터 안에서 실행 중인 여러 프로그램을 구분하는 번호(0~65535)다. `0~1023`은 well-known 포트(HTTP 80, HTTPS 443, SSH 22 등)로 예약되어 있다. **"IP 주소 + 포트"가 통신의 종착지(endpoint)**를 결정한다. 예) `127.0.0.1:8080`.
- **프로토콜(Protocol)** : 데이터를 주고받기 위한 **약속(규칙)**이다. 대표적으로 신뢰성을 보장하는 **TCP**와 빠르지만 보장하지 않는 **UDP**가 있다.

TCP와 UDP는 다음과 같이 대비된다. 이 표는 이 장 전체의 지도 역할을 한다.

| 구분 | TCP | UDP |
|------|-----|-----|
| 연결 여부 | 연결 지향 (연결 후 통신) | 비연결 (그냥 던짐) |
| 신뢰성 | 보장 (순서·재전송·오류검출) | 미보장 (유실·순서뒤바뀜 가능) |
| 속도 | 상대적으로 느림 | 빠름 |
| 비유 | 전화 통화 | 편지·방송 |
| 용도 | 파일전송·웹·채팅 | 스트리밍·게임·DNS |

```java
// IP + 포트 = endpoint. TCP는 "전화", UDP는 "편지"에 비유된다.
System.out.println("endpoint = IP + Port (예: 127.0.0.1:8080)");
System.out.println("well-known : 80(HTTP), 443(HTTPS), 22(SSH)");
```

## 19.2 IP 주소

자바에서 IP 주소는 `java.net.InetAddress` 클래스로 표현한다. 이 클래스는 **생성자가 없고** 정적 팩터리 메서드로 객체를 얻는다.

- `getLocalHost()` : 현재 실행 중인 컴퓨터(로컬 호스트)의 IP를 얻는다.
- `getByName("호스트명")` : 호스트 이름을 IP로 변환한다. 이 과정을 **이름 해석(DNS lookup)**이라고 한다.
- `getAllByName("호스트명")` : 한 이름에 연결된 여러 IP를 배열로 반환한다.

이름 해석에 실패하면 **`UnknownHostException`**(검사 예외)이 발생하므로, 반드시 `try-catch` 또는 `throws`로 처리해야 한다. 아래 예시는 인터넷 없이도 해석되는 `localhost`로 시연한다.

```java
InetAddress local = InetAddress.getByName("localhost");
System.out.println("호스트 이름 : " + local.getHostName());     // localhost
System.out.println("IP 주소     : " + local.getHostAddress());  // 127.0.0.1
System.out.println("loopback?   : " + local.isLoopbackAddress()); // true
```

`getHostAddress()`는 IP 문자열을, `getHostName()`은 호스트 이름을, `isLoopbackAddress()`는 그 주소가 자기 자신(127.0.0.1)인지를 알려준다. 존재하지 않는 이름을 넘기면 예상대로 `UnknownHostException`이 발생한다.

> 💻 실습: `./run.sh GetIpAddress`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch19/GetIpAddress.java`

> 📖 더 깊이: 이름이 IP로 바뀌는 과정(DNS 계층 질의, 캐시 등)은 `docs/CS_네트워크_개념서/04-DNS.md`에서 다룬다.

## 19.3 TCP

TCP는 **연결 지향** 프로토콜이다. 데이터를 주고받기 전에 양쪽이 먼저 연결을 맺고, 연결이 유지되는 동안 순서와 도착을 보장하며 통신한다. 자바에서의 흐름은 다음과 같다.

- **서버** : `ServerSocket`을 특정 포트에 열고 `accept()`로 클라이언트의 연결을 기다린다.
- **클라이언트** : `Socket`으로 서버의 `IP:포트`에 연결(connect)한다.
- 연결이 맺어지면 양쪽 모두 `Socket`의 **입출력 스트림**으로 데이터를 주고받는다.

연결이 시작될 때 TCP는 내부적으로 **3-way handshake**(SYN → SYN-ACK → ACK)를 수행한다. 즉 "대화 시작합시다"를 3번의 메시지로 서로 합의하는데, 이는 **양쪽 모두 보내고 받을 수 있음**을 서로 확인하기 위해서다. 이 과정은 자바 API 아래에서 자동으로 일어나므로, 우리는 `connect()`/`accept()`만 호출하면 된다.

하나의 TCP 연결은 **4-튜플**, 즉 `(출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)`로 유일하게 식별된다. 아래 예시는 한 프로세스 안에서 서버 스레드와 클라이언트가 loopback으로 연결해 한 줄을 주고받는다(서버는 받은 문자열을 그대로 되돌려주는 **echo**).

```java
// 서버: 포트 0 → OS가 빈 포트를 자동 할당(포트 충돌 방지)
ServerSocket serverSocket = new ServerSocket();
serverSocket.bind(new InetSocketAddress("127.0.0.1", 0));
serverSocket.setSoTimeout(5000);          // accept가 5초 넘게 막히면 예외 → hang 방지
int port = serverSocket.getLocalPort();

// 클라이언트: 서버에 연결 후 송·수신
try (Socket socket = new Socket()) {
    socket.connect(new InetSocketAddress("127.0.0.1", port), 5000);
    PrintWriter out = new PrintWriter(socket.getOutputStream(), true);
    BufferedReader in = new BufferedReader(new InputStreamReader(socket.getInputStream()));
    out.println("hello tcp");             // 서버로 송신
    System.out.println(in.readLine());     // 서버 응답 수신 → "echo:hello tcp"
}
```

`setSoTimeout`은 읽기/`accept`가 지정한 밀리초를 넘기면 예외를 던지게 해서, 코드가 영원히 멈추는 것을 막는다. 실습 코드는 이 타임아웃과 `join`으로 반드시 수초 내에 종료된다.

## 19.4 UDP

UDP는 **비연결** 프로토콜이다. TCP처럼 연결을 맺는 과정 없이, **데이터그램(datagram)**이라는 패킷 단위로 데이터를 그냥 던진다. 빠르지만 순서나 도착을 보장하지 않아 패킷이 유실될 수 있다.

- **`DatagramSocket`** : 데이터그램을 보내고 받는 데 쓰는 소켓.
- **`DatagramPacket`** : 실제 데이터와 **목적지(또는 출발지) 주소·포트**를 함께 담는 패킷.

`receive()`는 패킷이 올 때까지 **무한정 대기**하므로, 반드시 `setSoTimeout`으로 타임아웃을 걸어야 한다. 아래 예시는 한 프로세스 안에서 송신 소켓이 수신 소켓으로 `"hello udp"`를 보낸다.

```java
InetAddress loopback = InetAddress.getByName("127.0.0.1");

// 수신 소켓: 포트 0 → OS가 자동 할당, 5초 내 미수신 시 예외
DatagramSocket receiver = new DatagramSocket(0, loopback);
receiver.setSoTimeout(5000);
int recvPort = receiver.getLocalPort();

// 송신: 데이터 + 목적지(IP:포트)를 패킷에 담아 send
byte[] data = "hello udp".getBytes();
DatagramPacket sendPacket = new DatagramPacket(data, data.length, loopback, recvPort);
new DatagramSocket().send(sendPacket);

// 수신: 버퍼를 준비해 receive, 실제 길이만큼 문자열로 변환
byte[] buffer = new byte[1024];
DatagramPacket recvPacket = new DatagramPacket(buffer, buffer.length);
receiver.receive(recvPacket);
System.out.println(new String(recvPacket.getData(), 0, recvPacket.getLength()));
```

주의할 점은 수신 시 반드시 **`getLength()`가 돌려주는 실제 길이만큼만** 문자열로 변환해야 한다는 것이다. 버퍼 전체(`new String(buffer)`)를 쓰면 남은 공간의 널 문자까지 섞여 들어간다.

> 📖 더 깊이: UDP가 신뢰성을 응용 계층에 떠넘기는 이유와, 그 위에 신뢰성을 얹은 QUIC(HTTP/3)은 `docs/CS_네트워크_개념서/03-전송-계층.md` §3.2에서 다룬다.

## 19.5 동시 요청 처리

기본적인 서버는 한 번에 한 명만 상대한다. `accept()` → 요청 처리 → 다음 `accept()` 순서로만 동작하므로, 한 클라이언트를 처리하는 동안 다른 클라이언트는 **줄을 서서 기다려야** 한다. 처리 시간이 길면 서버 전체가 느려진다.

해결책은 **스레드 풀(`ExecutorService`)**이다. `accept()`로 연결을 받을 때마다 그 처리 작업을 스레드 풀에 **제출(submit)**하면, 여러 클라이언트를 **병렬로** 처리할 수 있다. `Executors.newFixedThreadPool(n)`으로 스레드 개수를 제한하면 자원(메모리·CPU)을 관리하면서도 동시성을 얻는다.

```java
ExecutorService workerPool = Executors.newFixedThreadPool(3);
// 서버 루프: 연결을 받을 때마다 처리 작업을 풀에 제출 → 병렬 처리
for (int i = 0; i < 3; i++) {
    Socket client = serverSocket.accept();
    workerPool.submit(() -> handleClient(client));   // 각 연결을 별도 스레드에서
}
workerPool.shutdown();
workerPool.awaitTermination(6, TimeUnit.SECONDS);    // 모든 작업 종료 대기
```

핵심은 `accept()`(연결 수락)와 **실제 요청 처리**를 분리하는 것이다. 수락은 메인 루프가 빠르게 반복하고, 오래 걸리는 처리는 워커 스레드에게 넘긴다. 실습 코드는 클라이언트 3개를 동시에 접속시켜 각기 다른 워커 스레드가 이들을 병렬로 처리함을 보여준다. (Spring 같은 실무 서버 프레임워크도 내부적으로 이런 스레드 풀 모델을 사용한다.)

## 19.6 JSON

네트워크로 데이터를 주고받을 때는 **약속된 형식**이 필요하다. 오늘날 가장 널리 쓰이는 것이 **JSON(JavaScript Object Notation)**이다. 사람이 읽기 쉬운 경량 텍스트 형식이며, 거의 모든 언어가 지원한다.

- **객체**는 중괄호로: `{ "키": 값, ... }`
- **배열**은 대괄호로: `[ 값, 값, ... ]`
- 값의 타입: 문자열(`"..."`), 숫자, 불리언(`true`/`false`), `null`, 객체, 배열.

```json
{ "name": "홍길동", "age": 20, "tags": ["java", "spring"] }
```

중요한 점: **JDK 표준 라이브러리에는 범용 JSON 처리기가 없다.** 아래처럼 문자열을 직접 조립·파싱하는 것은 **원리 학습용**일 뿐이며, 실무에서는 절대 직접 파싱하지 말고 검증된 라이브러리인 **Jackson(`com.fasterxml.jackson`)** 또는 **Gson(`com.google.gson`)**을 써야 한다. 직접 파싱은 중첩 객체·배열·이스케이프·따옴표 안의 쉼표 같은 경우를 모두 놓치기 때문이다.

```java
// 학습용: 문자열 맵을 { "k":"v", ... } JSON으로 "직접 조립"
Map<String, String> user = new LinkedHashMap<>();
user.put("name", "홍길동");
user.put("age", "20");
// → {"name":"홍길동","age":"20"}  (아주 단순한 평면 구조만 처리)
```

## 19.7 TCP 채팅

마지막으로 앞의 개념들을 종합해 **간이 TCP 채팅**을 만든다. 채팅의 핵심은 TCP 연결이 맺어진 뒤, 하나의 소켓을 통해 양쪽이 **여러 줄의 메시지를 번갈아** 주고받는 것이다. 이때 "누가 먼저 보내고 누가 답하는지"를 정하는 **약속(프로토콜)**이 필요하다.

이 예시의 프로토콜은 단순하다: **클라이언트가 먼저 한 줄을 보내면 서버가 한 줄로 답한다. 이를 정해진 횟수(3왕복)만큼 반복한 뒤 종료한다.**

```java
// 서버 측: 받고(readLine) → 답한다(println) 를 ROUNDS 번 반복
for (int i = 1; i <= ROUNDS; i++) {
    String msg = in.readLine();                    // 클라이언트 메시지 수신
    out.println("[" + i + "] 서버가 인사드립니다");   // 서버 응답 송신
}

// 클라이언트 측: 보내고(println) → 받는다(readLine) 를 ROUNDS 번 반복
for (int i = 1; i <= ROUNDS; i++) {
    out.println("[" + i + "] 안녕하세요 서버님");     // 메시지 송신
    String reply = in.readLine();                  // 응답 수신
}
```

읽고 쓰는 **순서가 서로 맞물려야** 한다는 점이 핵심이다. 만약 양쪽이 동시에 `readLine()`으로 상대의 메시지를 기다리기만 하면, 아무도 먼저 보내지 않아 둘 다 영원히 멈추는 **교착(deadlock)**에 빠진다. 그래서 "클라가 먼저 보낸다"는 약속을 코드가 정확히 지켜야 한다.

## ⚠️ 흔한 오해와 함정

- **읽기가 영원히 멈춘다(hang).** `accept()`, `readLine()`, UDP의 `receive()`는 데이터가 올 때까지 **무한정 대기**한다. `setSoTimeout(밀리초)`을 걸지 않으면 코드가 영영 멈출 수 있다. 실습 코드가 타임아웃을 거는 이유가 이것이다.
- **"IP만 있으면 통신한다"고 착각한다.** IP는 컴퓨터만 특정할 뿐, **어느 프로그램**에게 보낼지는 **포트**가 정한다. 통신의 종착지는 항상 **IP + 포트**다.
- **`PrintWriter`로 보냈는데 상대가 못 받는다.** 스트림에는 버퍼가 있어, `flush` 되기 전까지 데이터가 실제로 나가지 않을 수 있다. `new PrintWriter(out, true)`처럼 **autoFlush를 켜거나** `println` 후 `flush()`를 호출해야 한다. 또한 `readLine()`은 **줄바꿈**을 만나야 한 줄을 반환하므로, 보낼 때 `println`(줄바꿈 포함)을 써야 한다.
- **UDP를 신뢰성 있는 통신으로 여긴다.** UDP는 도착·순서·중복 없음을 **보장하지 않는다.** 반드시 전달되어야 하는 데이터라면 TCP를 쓰거나, UDP 위에 재전송 로직을 직접 얹어야 한다.
- **UDP 수신 시 버퍼 전체를 문자열로 만든다.** `new String(buffer)`가 아니라 `new String(buffer, 0, packet.getLength())`처럼 **실제 수신 길이만큼**만 변환해야 한다. 그러지 않으면 남은 버퍼의 널 문자가 섞인다.
- **JSON을 직접 파싱하려 한다.** 학습용으로는 괜찮지만, 실무에서 직접 파싱하면 중첩·이스케이프·특수문자에서 반드시 버그가 난다. **Jackson/Gson**을 쓰자.
- **채팅에서 양쪽이 동시에 받기만 한다.** 서로 상대가 먼저 보내길 기다리면 **교착**에 빠진다. "누가 먼저 보내는가"를 프로토콜로 명확히 정해야 한다.

## ✍️ 직접 작성해보기

읽기만 해서는 소켓 프로그래밍이 손에 붙지 않는다. 아래 과제는 **직접 타이핑해서 처음부터 작성**하는 연습이다. 예제 코드를 복사하지 말고, 스펙과 힌트만 보고 스스로 짜 보자. 막히면 힌트를 참고하고, 그래도 안 되면 각 과제 끝의 참고 클래스 코드를 열어 비교하라.

**모든 과제는 반드시 다음 규칙을 지켜야 한다. 이 규칙을 어기면 프로그램이 멈춰(hang) 종료되지 않는다.**

- 통신은 전부 **`localhost`(127.0.0.1) loopback** 안에서만 한다. 인터넷 연결이나 외부 서버가 필요 없어야 한다.
- 서버/수신 포트는 **`0`으로 바인딩**해 OS가 빈 포트를 자동 할당하게 한다(`ServerSocket`은 `bind(new InetSocketAddress("127.0.0.1", 0))`, `DatagramSocket`은 `new DatagramSocket(0, loopback)`). 그 후 `getLocalPort()`로 실제 포트를 얻어 클라이언트에게 넘긴다.
- **모든 소켓에 `setSoTimeout(5000)`**(5초)을 걸어 `accept()`/`readLine()`/`receive()`가 무한정 막히지 않게 한다.
- 서버를 **별도 스레드**로 띄웠다면 마지막에 **`join(6000)`**(스레드 풀은 `awaitTermination`)으로 반드시 회수하고, 프로그램이 **수초 내에 스스로 종료**되도록 한다.

한 파일에 `main` 하나로 서버(스레드)와 클라이언트를 함께 넣어, 한 프로세스만 실행하면 통신이 완결되도록 만든다.

### 과제 1. 내 컴퓨터의 IP 알아내기 (쉬움)

**스펙**
- `InetAddress`를 사용해 다음 세 가지를 순서대로 출력하는 프로그램을 작성하라.
  1. `getLocalHost()`로 현재 컴퓨터의 **호스트 이름**과 **IP 주소**
  2. `getByName("localhost")`로 얻은 주소의 **IP 주소**와 **loopback 여부**(`isLoopbackAddress()`)
  3. 존재하지 않는 이름(예: `"no.such.host.invalid"`)을 `getByName`에 넘겼을 때 **예외를 잡아** "이름 해석 실패" 메시지 출력
- 인터넷 없이도 실행되어야 한다(`localhost`, `127.0.0.1`만 사용).

**힌트**
- `InetAddress`는 생성자가 없다. `InetAddress.getByName(...)`, `InetAddress.getLocalHost()` 같은 **정적 팩터리 메서드**로 객체를 얻는다.
- IP 문자열은 `getHostAddress()`, 이름은 `getHostName()`으로 얻는다.
- 이름 해석 실패 시 **검사 예외** `UnknownHostException`이 발생하므로 `try-catch`로 감싸야 컴파일된다. 3번은 이 예외가 "정상적으로 발생하는지"를 확인하는 과제다.

### 과제 2. TCP echo 서버와 클라이언트 (기본)

**스펙**
- 한 프로세스 안에서 **서버 스레드**와 **클라이언트**가 loopback으로 TCP 연결해 한 줄을 주고받는 프로그램을 작성하라.
- 서버: `ServerSocket`을 `127.0.0.1:0`에 바인딩하고 `accept()`로 연결을 기다린다. 클라이언트가 보낸 한 줄을 읽어 **`"echo:" + 받은문자열`** 형태로 그대로 되돌려준(echo) 뒤 종료한다.
- 클라이언트: `Socket`으로 서버 포트에 연결해 `"hello tcp"`를 보내고, 서버 응답 한 줄을 받아 출력한다(기대값: `echo:hello tcp`).
- 마지막에 서버 스레드를 `join`으로 회수하고, 스레드가 실제로 종료되었는지 출력한다.

**힌트**
- 서버 포트는 미리 정하지 말고 `serverSocket.getLocalPort()`로 얻어 클라이언트에게 넘긴다.
- 서버 소켓엔 `setSoTimeout(5000)`, 연결된 각 `Socket`에도 `setSoTimeout(5000)`을 건다. 클라이언트 연결은 `socket.connect(new InetSocketAddress("127.0.0.1", port), 5000)`처럼 **연결 타임아웃**도 준다.
- 문자열 한 줄 송수신은 `PrintWriter`(autoFlush를 위해 `new PrintWriter(out, true)`)의 `println`과 `BufferedReader`의 `readLine`을 짝으로 쓴다. `readLine`은 **줄바꿈**을 만나야 반환하므로 보낼 때 `println`을 써야 한다.
- 서버는 별도 `Thread`로 띄우고, 클라이언트 통신이 끝난 뒤 `serverThread.join(6000)`으로 회수한다.

### 과제 3. UDP 데이터그램 송수신 (응용)

**스펙**
- 한 프로세스 안에서 **수신 소켓**과 **송신 소켓**이 loopback으로 UDP 데이터그램 한 개를 주고받는 프로그램을 작성하라.
- 수신 소켓: `127.0.0.1`에 포트 `0`으로 열어 자동 할당된 포트를 얻는다.
- 송신: `"hello udp"`를 바이트로 만들어 목적지(수신 소켓의 `127.0.0.1:포트`)를 담은 `DatagramPacket`으로 보낸다.
- 수신: 버퍼로 `receive`한 뒤 **받은 데이터**와 **보낸 곳의 IP:포트**를 출력한다. 타임아웃이 나면 "타임아웃(패킷 미도착)"을 출력한다.

**힌트**
- UDP는 연결 과정이 없다. `DatagramSocket`으로 보내고 받으며, 데이터와 목적지 주소·포트는 `DatagramPacket`에 함께 담는다.
- `receive()`는 무한정 대기하므로 수신 소켓에 **`setSoTimeout(5000)`**을 반드시 걸고, `SocketTimeoutException`을 `catch`한다.
- 수신 데이터를 문자열로 바꿀 때는 반드시 **실제 길이만큼만**: `new String(packet.getData(), 0, packet.getLength())`. 버퍼 전체(`new String(buffer)`)를 쓰면 남은 널 문자가 섞인다.
- 보낸 곳 정보는 수신한 패킷의 `getAddress()`, `getPort()`로 얻는다.

### 과제 4. 스레드 풀로 다중 클라이언트 동시 처리 (응용·심화)

**스펙**
- TCP 서버가 **클라이언트 3개를 동시에** 상대하도록, 요청 처리를 **스레드 풀(`ExecutorService`)**에 넘기는 프로그램을 작성하라.
- 서버: `127.0.0.1:0`에 바인딩하고, `accept()`로 연결을 받을 때마다 그 처리 작업을 `Executors.newFixedThreadPool(3)`에 `submit`한다. 각 워커는 클라이언트의 요청 한 줄을 읽어 **`"응답:" + 요청`**을 돌려준다.
- 클라이언트: 3개를 각각 별도 스레드로 **동시에** 접속시켜, 서로 다른 요청(예: `"클라이언트#1 요청"` …)을 보내고 응답을 출력한다.
- 마지막에 모든 클라이언트 스레드와 서버 스레드를 `join`하고, 스레드 풀은 `shutdown()` 후 `awaitTermination(6, TimeUnit.SECONDS)`으로 종료를 기다린다.

**힌트**
- 핵심은 **`accept()`(연결 수락)와 실제 처리(읽고 응답)를 분리**하는 것이다. 메인 루프는 `accept`만 빠르게 반복하고, 오래 걸릴 수 있는 처리는 워커 스레드에 맡긴다: `workerPool.submit(() -> handleClient(client))`.
- 워커에서 각 소켓에도 `setSoTimeout(5000)`을 건다. 각 클라이언트 스레드의 로그에 서로 다른 워커 스레드 이름(`Thread.currentThread().getName()`)이 찍히는지 보면 병렬 처리가 확인된다.
- 서버가 3개를 다 받으면 루프를 끝내고 `serverSocket.close()` 하도록 해, 프로그램이 스스로 종료되게 한다.
- 종료 순서: 클라이언트 스레드 `join(6000)` → 서버 스레드 `join(6000)` → `workerPool.shutdown()` → `awaitTermination`.

> 답안 비교: `./run.sh ConcurrentServer`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch19/ConcurrentServer.java`

### 과제 5. loopback 정보 종합 출력 (쉬움·응용)

**스펙**
- `InetAddress`로 다음을 순서대로 출력하는 프로그램을 작성하라.
  1. `getByName("127.0.0.1")`로 얻은 주소의 **IP 문자열**과 **loopback 여부**(`isLoopbackAddress()`)
  2. `getByName("localhost")`로 얻은 주소의 **호스트 이름**과 **IP 문자열** — 위 1번과 같은 loopback을 가리키는지 눈으로 비교
  3. `getLocalHost()`로 현재 컴퓨터의 **호스트 이름**과 **IP 문자열**
- 인터넷 연결 없이 `localhost`/`127.0.0.1`만으로 자기완결적으로 실행되어야 한다. 소켓을 열지 않으므로 스레드·타임아웃은 필요 없지만, 프로그램은 즉시 종료되어야 한다.

**힌트**
- `InetAddress`는 생성자가 없다. `getByName(...)`, `getLocalHost()` 같은 **정적 팩터리 메서드**로 객체를 얻는다.
- IP 문자열은 `getHostAddress()`, 호스트 이름은 `getHostName()`으로 얻는다.
- 세 메서드 모두 `UnknownHostException`(검사 예외)을 던질 수 있으므로 `try-catch` 또는 `throws`로 처리해야 컴파일된다.
- `127.0.0.1`과 `localhost`가 **같은 loopback**을 가리킨다는 것을 출력으로 확인하는 것이 이 과제의 목적이다.

### 과제 6. TCP 다왕복 요청·응답 (기본·심화)

**스펙**
- 한 프로세스 안에서 **서버 스레드**와 **클라이언트**가 loopback으로 TCP 연결해, 한 번의 연결 위에서 **여러 줄(예: 3왕복)**을 번갈아 주고받는 프로그램을 작성하라.
- 프로토콜: **클라이언트가 먼저 한 줄을 보내면 서버가 한 줄로 답한다.** 이를 정해진 횟수(`ROUNDS = 3`)만큼 반복한 뒤 양쪽 모두 종료한다.
  - 클라이언트가 `i`번째로 `"[i] 요청"`을 보내면, 서버는 `"[i] 응답"`으로 답한다.
- 마지막에 서버 스레드를 `join`으로 회수하고, 스레드가 실제로 종료되었는지 출력한다.

**힌트**
- 서버 포트는 `ServerSocket`을 `127.0.0.1:0`에 바인딩한 뒤 `getLocalPort()`로 얻어 클라이언트에게 넘긴다. 서버 소켓과 연결된 각 `Socket`에 모두 `setSoTimeout(5000)`을 건다.
- **읽고 쓰는 순서가 서로 맞물려야** 한다. 서버는 `for`문 안에서 `readLine()`(수신) → `println`(응답) 순으로, 클라이언트는 `println`(송신) → `readLine()`(수신) 순으로 `ROUNDS`번 반복한다. 양쪽이 동시에 `readLine()`만 하면 아무도 먼저 보내지 않아 **교착(deadlock)**에 빠진다.
- `PrintWriter`는 `new PrintWriter(out, true)`로 autoFlush를 켜고, 한 줄 단위 통신이므로 `println`/`readLine` 짝을 쓴다.
- 서버는 별도 `Thread`로 띄우고, 클라이언트 통신이 끝난 뒤 `serverThread.join(6000)`으로 회수해 프로그램이 수초 내에 스스로 종료되게 한다.

### 개념 서술

**1.** 통신의 종착지(endpoint)를 이루는 두 요소는 무엇인가? IP 주소와 포트가 각각 "무엇을 식별"하는지 구분하여 설명하시오.

<details><summary>정답/해설</summary>

종착지는 **IP 주소 + 포트**로 결정된다.
- **IP 주소**는 네트워크에 연결된 **어느 컴퓨터(호스트)**인지를 식별한다.
- **포트**는 그 컴퓨터 안에서 실행 중인 **어느 프로그램(프로세스)**에게 보낼지를 식별한다(0~65535).

IP만으로는 컴퓨터까지만 도달할 뿐, 그 안의 어느 프로그램이 받을지 정할 수 없다. 그래서 `127.0.0.1:8080`처럼 "IP:포트" 쌍이 하나의 endpoint가 된다. 더 깊게는 하나의 TCP 연결은 (출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)의 **4-튜플**로 유일하게 식별된다. → `docs/CS_네트워크_개념서/03-전송-계층.md` §3.1
</details>

**2.** TCP와 UDP의 차이를 **연결 여부**와 **신뢰성** 두 관점에서 설명하고, 각각에 더 어울리는 용도를 하나씩 예로 드시오.

<details><summary>정답/해설</summary>

| 관점 | TCP | UDP |
|------|-----|-----|
| 연결 여부 | **연결 지향** — 3-way handshake로 먼저 연결을 맺고 통신 | **비연결** — 연결 없이 데이터그램을 그냥 던짐 |
| 신뢰성 | **보장** — 순서·도착·오류검출·재전송 | **미보장** — 유실·순서 뒤바뀜·중복 가능 |

- TCP에 어울리는 용도: 파일 전송, 웹(HTTP), 채팅 등 **정확성이 중요한** 통신.
- UDP에 어울리는 용도: 실시간 스트리밍, 온라인 게임, DNS 질의 등 **속도·실시간성이 중요한** 통신.

핵심은 "정확성 vs 실시간성"의 트레이드오프다. → `docs/CS_네트워크_개념서/03-전송-계층.md` §3.2
</details>

**3.** TCP 연결 시작 시 일어나는 **3-way handshake**를 세 메시지의 이름과 함께 순서대로 설명하고, "왜 2번이 아니라 3번인가"를 한 문장으로 답하시오.

<details><summary>정답/해설</summary>

순서: **SYN → SYN-ACK → ACK**
1. 클라이언트가 "연결하자"는 **SYN**을 보낸다.
2. 서버가 "알았다(ACK) + 나도 연결하자(SYN)"를 합친 **SYN-ACK**로 답한다.
3. 클라이언트가 서버의 SYN에 대한 **ACK**를 보내 연결이 확립(ESTABLISHED)된다.

**왜 3번인가:** 양쪽 모두가 "보낼 수 있고 받을 수 있음"을 서로 확인해야 하므로, 클라이언트→서버 방향 확인(1,2)과 서버→클라이언트 방향 확인(2,3)이 모두 필요해 최소 3번의 메시지가 든다.

자바에서는 `connect()`/`accept()` 호출 시 이 과정이 API 아래에서 자동으로 일어난다. 상태 전이(SYN_SENT, ESTABLISHED 등)와 순서번호의 역할은 → `docs/CS_네트워크_개념서/03-전송-계층.md` §3.3
</details>

**4.** 단일 스레드 서버의 한계는 무엇이며, **스레드 풀(`ExecutorService`)**을 사용하면 이 한계가 어떻게 해결되는지, `accept()`와 요청 처리의 **분리** 관점에서 설명하시오.

<details><summary>정답/해설</summary>

단일 스레드 서버는 `accept()` → 요청 처리 → 다음 `accept()` 순서로만 동작한다. 따라서 한 클라이언트를 처리하는 **동안** 다른 클라이언트는 큐에서 대기해야 하고, 처리 시간이 길면 뒤의 요청 전체가 느려진다.

스레드 풀은 **연결 수락(`accept`)과 실제 요청 처리를 분리**한다. 메인 루프는 `accept`만 빠르게 반복하고, 오래 걸릴 수 있는 처리는 `workerPool.submit(() -> handleClient(client))`로 워커 스레드에 넘긴다. 그러면 여러 클라이언트를 **병렬**로 처리할 수 있다. `Executors.newFixedThreadPool(n)`으로 스레드 수를 제한하면 동시성을 얻으면서도 메모리·CPU 자원을 관리할 수 있다. (Spring 같은 실무 서버도 내부적으로 이 스레드 풀 모델을 쓴다.)
</details>

**5.** 포트 번호 `0`을 `ServerSocket`이나 `DatagramSocket`에 바인딩하면 어떤 일이 일어나는가? 실습 코드가 굳이 포트 `0`을 쓰는 이유를 **포트 충돌** 관점에서 설명하시오.

<details><summary>정답/해설</summary>

포트 `0`으로 바인딩하면 OS가 현재 **사용 가능한 빈 포트를 자동으로 하나 골라** 할당한다. 실제 할당된 포트는 `getLocalPort()`로 얻어 클라이언트에게 넘긴다.

실습 코드가 포트 `0`을 쓰는 이유: 특정 포트(예: 8080)를 고정하면 그 포트를 **다른 프로그램이 이미 쓰고 있을 때** `BindException: Address already in use`가 나 실습이 실패한다. 포트 `0`은 항상 빈 포트를 받으므로 **포트 충돌 없이** 어느 환경에서나 자기완결적으로 실행된다.
</details>

### 출력·동작 예측

**6.** 다음 UDP 수신 코드에서 수신 소켓에 **아무 패킷도 오지 않는다면**, 프로그램은 어떻게 동작하는가? `setSoTimeout(5000)`이 **있을 때**와 **없을 때**를 각각 답하시오.

```java
DatagramSocket receiver = new DatagramSocket(0, loopback);
receiver.setSoTimeout(5000);              // (A) 이 줄이 있을 때 / 없을 때
byte[] buffer = new byte[1024];
DatagramPacket packet = new DatagramPacket(buffer, buffer.length);
receiver.receive(packet);                 // 패킷이 오지 않는 상황
```

<details><summary>정답/해설</summary>

- **`setSoTimeout(5000)`이 있을 때:** `receive()`는 최대 5초 대기하다가 패킷이 오지 않으면 `java.net.SocketTimeoutException`을 던진다. 이를 `catch`하면 "타임아웃(패킷 미도착)"을 출력하고 정상 종료할 수 있다.
- **없을 때:** `receive()`는 패킷이 올 때까지 **무한정 대기(block)**한다. 프로그램은 이 줄에서 영영 멈춰(hang) 스스로 종료되지 않는다.

`accept()`, `readLine()`, `receive()`는 모두 데이터가 올 때까지 블로킹하는 메서드이므로 타임아웃이 필수다.
</details>

**7.** 다음 TCP 클라이언트 코드는 서버에 `"hello tcp"`를 보내고 응답을 기다린다. 그런데 서버는 응답을 하는데도 클라이언트의 `in.readLine()`이 응답을 못 받고 멈춘다. **서버 측** 아래 코드에서 원인을 찾고 고치시오.

```java
// 서버 측
PrintWriter out = new PrintWriter(client.getOutputStream());  // autoFlush 미설정
String line = in.readLine();
out.print("echo:" + line);   // print (줄바꿈 없음)
```

<details><summary>정답/해설</summary>

원인은 **두 가지**다.
1. **`flush`가 안 됨:** `new PrintWriter(out)`은 autoFlush가 꺼져 있어, 버퍼에 쌓인 데이터가 실제로 전송되지 않는다. 상대는 아무것도 못 받는다.
2. **줄바꿈이 없음:** 클라이언트가 `readLine()`으로 읽는데, `print`는 줄바꿈을 붙이지 않는다. `readLine()`은 `\n`을 만나야 한 줄을 반환하므로 영원히 대기한다.

수정:
```java
PrintWriter out = new PrintWriter(client.getOutputStream(), true); // autoFlush 켜기
out.println("echo:" + line);   // println → 줄바꿈 포함 + autoFlush로 전송
```
(autoFlush를 안 켰다면 `out.println(...); out.flush();`처럼 명시적으로 flush해도 된다.)
</details>

**8.** 다음 UDP 수신 출력 코드에서, 실제 보낸 문자열은 `"hi"`(2바이트)인데 출력에 이상한 문자가 딸려 나온다. 원인과 수정을 설명하시오.

```java
byte[] buffer = new byte[1024];
DatagramPacket packet = new DatagramPacket(buffer, buffer.length);
receiver.receive(packet);
System.out.println(new String(buffer));   // 버퍼 전체를 문자열로
```

<details><summary>정답/해설</summary>

원인: `new String(buffer)`는 1024바이트 버퍼 **전체**를 문자열로 만든다. 실제 수신한 데이터는 앞 2바이트뿐이고, 나머지 1022바이트는 초기값(0, 즉 널 문자)이라 이 널 문자들이 함께 섞여 나온다.

수정: 실제 수신 길이(`getLength()`)만큼만 변환한다.
```java
System.out.println(new String(packet.getData(), 0, packet.getLength()));
```
UDP는 패킷마다 길이가 다르므로, 반드시 `getLength()`가 돌려주는 **실제 길이**만큼만 잘라 써야 한다.
</details>

### 오류 찾기

**9.** 다음 서버 코드는 여러 요청을 처리하는 루프에서 **연결을 반복할수록 자원이 새어(leak)** 나간다. 어떤 자원이 어떻게 새는지 지적하고 고치시오.

```java
while (running) {
    Socket client = serverSocket.accept();
    BufferedReader in = new BufferedReader(new InputStreamReader(client.getInputStream()));
    PrintWriter out = new PrintWriter(client.getOutputStream(), true);
    out.println("응답:" + in.readLine());
    // client.close() 없음
}
```

<details><summary>정답/해설</summary>

새는 자원은 **연결된 소켓(`client`)** 이다. 루프를 한 번 돌 때마다 `accept()`가 새 `Socket`을 반환하는데 `close()`하지 않으므로, 열린 소켓(파일 디스크립터)이 계속 쌓인다. 오래 돌면 "Too many open files" 오류로 서버가 죽는다. `in`/`out` 스트림도 소켓에 묶여 함께 남는다.

수정: **try-with-resources**로 각 연결을 감싸면 처리가 끝나거나 예외가 나도 소켓이 자동으로 닫힌다.
```java
while (running) {
    try (Socket client = serverSocket.accept()) {   // 블록을 벗어나면 자동 close
        client.setSoTimeout(5000);
        BufferedReader in = new BufferedReader(new InputStreamReader(client.getInputStream()));
        PrintWriter out = new PrintWriter(client.getOutputStream(), true);
        out.println("응답:" + in.readLine());
    }
}
```
</details>

**10.** 다음 코드는 같은 포트를 두 번 열려다 실패한다. 무슨 예외가 나며, 왜 나는지 설명하시오. 또 이런 **포트 충돌**을 실습에서 피하는 방법도 쓰시오.

```java
ServerSocket a = new ServerSocket(8080);
ServerSocket b = new ServerSocket(8080);   // 여기서 실패
```

<details><summary>정답/해설</summary>

두 번째 줄에서 `java.net.BindException: Address already in use`가 발생한다. 하나의 (IP, 포트) 조합은 동시에 하나의 소켓만 **바인딩(listen)**할 수 있는데, 포트 8080이 이미 `a`에 점유되어 있어 `b`가 같은 포트를 열 수 없기 때문이다.

실습에서 피하는 방법: 포트를 고정하지 말고 **포트 `0`으로 바인딩**해 OS가 빈 포트를 자동 할당하게 한다(`new ServerSocket(0)` 또는 `bind(new InetSocketAddress("127.0.0.1", 0))`). 그런 다음 `getLocalPort()`로 실제 포트를 얻어 클라이언트에게 넘긴다. (참고: 방금 닫은 서버 포트를 곧바로 재사용할 때 나는 `Address already in use`는 `TIME_WAIT` 상태 때문일 수 있다. → `docs/CS_네트워크_개념서/03-전송-계층.md` §3.6)
</details>

### 빈칸 채우기

**11.** 다음은 loopback에서 TCP echo 서버를 여는 코드다. 자기완결·hang 방지 규칙에 맞도록 빈칸 (가)~(마)를 채우시오.

```java
ServerSocket serverSocket = new ServerSocket();
serverSocket.bind(new InetSocketAddress("127.0.0.1", (가) ));  // OS 자동 포트
serverSocket.( 나 )(5000);                                     // accept 타임아웃
int port = serverSocket.( 다 )();                               // 실제 할당 포트

try (Socket client = serverSocket.( 라 )()) {                   // 연결 수락
    BufferedReader in = new BufferedReader(new InputStreamReader(client.getInputStream()));
    PrintWriter out = new PrintWriter(client.getOutputStream(), ( 마 ));  // autoFlush
    out.println("echo:" + in.readLine());
}
```

<details><summary>정답/해설</summary>

- (가) `0` — 포트 0으로 바인딩해 OS가 빈 포트를 자동 할당한다.
- (나) `setSoTimeout` — `accept()`가 5초 넘게 막히면 예외를 던져 hang을 막는다.
- (다) `getLocalPort` — 실제로 할당된 포트 번호를 얻는다.
- (라) `accept` — 클라이언트 연결을 수락하고 통신용 `Socket`을 반환한다.
- (마) `true` — autoFlush를 켜서 `println` 시 자동으로 flush되게 한다.
</details>

### 참·거짓 (이유 포함)

**12.** 다음 각 명제가 참인지 거짓인지 판단하고 **이유**를 쓰시오.

1. "IP 주소만 알면 상대 프로그램과 통신할 수 있다."
2. "UDP는 보낸 데이터그램이 반드시 순서대로 도착함을 보장한다."
3. "`ServerSocket.accept()`는 클라이언트가 연결할 때까지 블로킹된다."
4. "TCP 연결은 목적지 IP와 포트, 두 가지만으로 유일하게 식별된다."

<details><summary>정답/해설</summary>

1. **거짓.** IP는 컴퓨터(호스트)만 특정한다. 그 안의 어느 프로그램이 받을지는 **포트**가 정한다. 통신 종착지는 항상 "IP + 포트"다.
2. **거짓.** UDP는 비연결·비신뢰 프로토콜이라 순서·도착·중복 없음을 **보장하지 않는다.** 순서 보장이 필요하면 TCP를 쓰거나 응용 계층에서 직접 처리해야 한다.
3. **참.** `accept()`는 연결 요청이 들어올 때까지 무한정 대기(block)한다. 그래서 `setSoTimeout`으로 타임아웃을 걸지 않으면 영영 멈출 수 있다.
4. **거짓.** 하나의 TCP 연결은 (출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)의 **4-튜플**로 유일하게 식별된다. 목적지만으로는, 같은 서버에 접속한 여러 클라이언트를 구분할 수 없다. → `docs/CS_네트워크_개념서/03-전송-계층.md` §3.1
</details>

## 요약

- **네트워크 통신의 3요소**는 IP(어느 컴퓨터), 포트(그 안의 어느 프로그램), 프로토콜(어떤 약속)이다. 종착지(endpoint)는 항상 **IP + 포트**로 정해진다. (`NetworkBasics`)
- **IP 주소**는 `InetAddress`로 표현한다. 정적 팩터리(`getLocalHost`, `getByName`)로 얻으며, 이름 해석 실패 시 `UnknownHostException`을 처리해야 한다. (`GetIpAddress`)
- **TCP**는 연결 지향이며 신뢰성을 보장한다. 서버는 `ServerSocket.accept()`로 기다리고 클라이언트는 `Socket`으로 연결하며, 연결은 **4-튜플**로 유일하게 식별된다. (`TcpNetworking`)
- **UDP**는 비연결이며 빠르지만 신뢰성을 보장하지 않는다. `DatagramSocket`/`DatagramPacket`으로 데이터그램을 주고받고, `receive()`에는 타임아웃이 필수다. (`UdpNetworking`)
- **동시 요청 처리**는 `accept()`와 요청 처리를 분리하고 **스레드 풀**에 작업을 제출해 여러 클라이언트를 병렬로 상대한다. (`ConcurrentServer`)
- **JSON**은 데이터 교환의 표준 텍스트 형식이다. JDK 표준에는 범용 처리기가 없으므로 실무에서는 **Jackson/Gson**을 쓰고, 직접 파싱은 학습용으로만 한다. (`JsonFormat`)
- **TCP 채팅**은 하나의 소켓으로 여러 줄을 번갈아 주고받는다. "누가 먼저 보내는가"를 프로토콜로 정해 **교착**을 피해야 한다. (`TcpChat`)
- 깊은 네트워크 이론(handshake 상태 전이, 재전송·혼잡 제어, DNS)은 **`docs/CS_네트워크_개념서/`**를 참고하라.
