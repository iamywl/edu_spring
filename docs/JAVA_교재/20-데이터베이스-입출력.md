# Chapter 20 데이터베이스 입출력

프로그램이 다루는 데이터는 프로그램이 꺼지면 함께 사라진다. 하지만 회원 정보, 주문 내역, 게시글은 프로그램을 껐다 켜도 남아 있어야 한다. 이렇게 데이터를 **영구히 저장하고 다시 꺼내오는** 일을 담당하는 것이 **데이터베이스(DB)** 이고, 자바가 그 DB와 대화하기 위한 표준 창구가 **JDBC**다. 이 장은 자바 코드가 데이터베이스에 값을 넣고, 바꾸고, 지우고, 읽어오는 과정을 JDBC로 배운다.

이 장의 실행 모델을 먼저 밝혀 둔다. JDBC 코드는 **PostgreSQL 드라이버 JAR(→ 『10장. 라이브러리와 모듈』 10.1)와 실제로 돌고 있는 데이터베이스**가 있어야 실행된다. 그래서 이 장은 각 소절마다 **JDBC 코드 스니펫**을 보여주고, 장 끝에 **이 프로젝트의 인프라 PostgreSQL로 직접 실행하는 법**을 함께 안내한다. DB 이론 자체(정규화·트랜잭션·인덱스 등)는 `docs/CS_데이터베이스_개념서`에서, 실무에서 이 JDBC 위에 얹는 JPA는 Spring 챕터(`docs/SPRING_개념서/04-데이터베이스와-jpa.md`)에서 더 깊게 다룬다. 이 장은 그 둘 **사이의 바닥층**인 순수 JDBC를 정확히 이해하는 데 목적이 있다.

이 장에서 배우는 것

- JDBC가 무엇이고 왜 필요한가 (20.1)
- 이 프로젝트가 쓰는 DBMS인 PostgreSQL 띄우기 (20.2)
- 웹 DB 도구 Adminer로 눈으로 확인하기 (20.3)
- 실습용 DB(테이블) 구성 (20.4)
- DriverManager로 DB에 연결하기 (20.5)
- 데이터 저장/수정/삭제/읽기 — PreparedStatement, ResultSet (20.6 ~ 20.9)
- 프로시저·함수 호출 — CallableStatement (20.10)
- 트랜잭션 — commit/rollback (20.11)
- 간단한 게시판 CRUD로 종합하기 (20.12)

## 20.1 JDBC 개요

**JDBC(Java DataBase Connectivity)** 는 자바 프로그램이 관계형 데이터베이스에 접속해 SQL을 실행하기 위한 **표준 API**다. 핵심 아이디어는 "**하나의 표준 인터페이스, 여러 개의 드라이버**"다. 자바 표준 라이브러리(`java.sql` 패키지, → 『10장』 10.9)는 `Connection`, `Statement`, `ResultSet` 같은 **인터페이스만** 정의한다. 그 인터페이스를 특정 DBMS(PostgreSQL, MySQL, Oracle...)에 맞게 실제로 구현한 것이 **JDBC 드라이버**이며, 각 DB 벤더가 라이브러리 JAR로 배포한다.

덕분에 우리 코드는 `java.sql`의 인터페이스에만 의존하고, DB를 바꿔야 하면 **드라이버 JAR와 접속 URL만** 갈아 끼우면 된다. 이 프로젝트는 PostgreSQL을 쓰므로 `org.postgresql:postgresql` 드라이버 JAR가 필요하다(→ 『10장』 10.1의 의존성 선언).

JDBC로 DB를 다루는 전형적인 흐름은 언제나 아래 다섯 단계다. 이 장의 모든 소절이 이 흐름의 어느 한 조각이다.

```text
1) 연결 얻기      Connection conn = DriverManager.getConnection(url, user, pw)   (20.5)
2) SQL 준비       PreparedStatement ps = conn.prepareStatement("... ? ...")     (20.6)
3) 값 바인딩      ps.setString(1, ...), ps.setInt(2, ...)                       (20.6)
4) 실행          ps.executeUpdate()  또는  ResultSet rs = ps.executeQuery()    (20.6~20.9)
5) 자원 닫기      (try-with-resources 로 자동 close)                            (전 절 공통)
```

`java.sql`의 주요 인터페이스가 여는 문은 각각 이렇다: **Connection**(DB와의 연결), **Statement/PreparedStatement**(SQL 실행 도구), **ResultSet**(조회 결과 표), **CallableStatement**(프로시저/함수 호출용, 20.10).

## 20.2 DBMS

**DBMS(DataBase Management System)** 는 데이터베이스를 저장·관리하고 SQL 질의에 응답하는 서버 소프트웨어다. 이 프로젝트는 DBMS로 **PostgreSQL 16**을 쓴다. 그리고 그 PostgreSQL은 로컬에 직접 설치하는 대신 **인프라 도커 컨테이너**로 띄운다. 설정은 `docker/docker-compose-infra.yml`에 있으며, 접속 정보는 다음과 같이 고정되어 있다.

| 항목 | 값 |
|------|-----|
| 데이터베이스 이름 | `edu_spring` |
| 사용자 | `edu` |
| 비밀번호 | `edu1234` |
| 포트 | `5432` |

띄우는 명령은 프로젝트 루트에서 다음과 같다. (도커 데몬이 켜져 있어야 한다.)

```bash
# PostgreSQL 만 띄우기
docker compose -f docker/docker-compose-infra.yml up -d postgres

# 컨테이너 이름 확인 (보통 docker 디렉터리명이 프로젝트명이 되어 docker-postgres-1)
docker ps --format '{{.Names}}\t{{.Image}}\t{{.Ports}}'
# 예) docker-postgres-1   postgres:16-alpine   0.0.0.0:5432->5432/tcp
```

> 컨테이너 이름은 환경에 따라 `docker_postgres_1`, `edu_spring-postgres-1` 등으로 다를 수 있으니, 위에서 **실제로 확인한 이름**을 이후 명령에 넣는다. (같은 인프라를 쓰는 `chapter-cs-database/README.md`의 안내와 동일하다.)

이 컨테이너 하나가 이 장의 모든 JDBC 코드가 접속할 **실제 DBMS**다.

## 20.3 Client Tool

DB에 무엇이 들어 있는지 코드 없이 눈으로 보고 SQL을 손으로 실행해 보려면 **클라이언트 도구(client tool)** 가 편하다. 이 프로젝트의 인프라에는 웹 기반 DB 도구인 **Adminer**가 포함되어 있어, 브라우저만으로 DB를 들여다볼 수 있다.

```bash
# postgres 와 adminer 를 함께 띄운다
docker compose -f docker/docker-compose-infra.yml up -d postgres adminer
```

띄운 뒤 브라우저에서 **http://localhost:8081** 에 접속해 아래 정보로 로그인한다.

```text
시스템(System):     PostgreSQL
서버(Server):       postgres      ← 도커 네트워크 내부 서비스명 (localhost 아님!)
사용자(Username):   edu
비밀번호:           edu1234
데이터베이스:       edu_spring
```

여기서 주의할 점 하나. **서버 칸에는 `localhost`가 아니라 `postgres`를 적는다.** Adminer는 같은 도커 네트워크 안에서 도는 다른 컨테이너이므로, PostgreSQL을 **컨테이너 서비스명(`postgres`)** 으로 찾기 때문이다. 반면 우리 자바 코드는 컨테이너 밖(호스트)에서 도니까 `localhost:5432`로 접속한다(20.5). "누구 입장에서 부르느냐"에 따라 주소가 달라진다는 점을 기억하자.

로그인 후 **SQL 명령(SQL command)** 메뉴에서 SQL을 붙여 넣어 실행하고 결과를 표로 확인할 수 있다. 이 장의 테이블 생성(20.4)이나 조회 결과를 코드 없이 검증할 때 유용하다.

## 20.4 DB 구성

JDBC로 데이터를 다루기 전에, 데이터를 담을 **테이블**이 있어야 한다. 이 장은 게시판 예제(20.12)로 마무리하므로, 게시글을 담는 `board` 테이블 하나를 만들어 쓴다. 아래 DDL을 Adminer의 SQL 명령(20.3)이나 psql(장 끝의 실행법)에서 한 번 실행해 두자.

```sql
-- 게시판 테이블 (기존 것이 있으면 지우고 새로 만든다 → 반복 실행 안전)
DROP TABLE IF EXISTS board;
CREATE TABLE board (
    id         BIGSERIAL PRIMARY KEY,          -- 자동 증가 기본키
    title      VARCHAR(200) NOT NULL,          -- 제목
    writer     VARCHAR(50)  NOT NULL,          -- 작성자
    content    TEXT,                           -- 본문
    created_at TIMESTAMP    NOT NULL DEFAULT now()
);
```

몇 가지만 짚는다. `BIGSERIAL`은 PostgreSQL에서 행이 추가될 때마다 자동으로 1씩 증가하는 값을 넣어 주는 타입이라, 우리가 `id`를 직접 지정하지 않아도 된다. `PRIMARY KEY`(기본키)는 각 행을 유일하게 구분하는 열이고, `NOT NULL`은 빈 값을 허용하지 않는 제약이다. 이런 제약(키·정규화 등)의 **이론적 배경**은 `docs/CS_데이터베이스_개념서/01-관계-모델.md`와 `02-정규화.md`에서 다루므로, 여기서는 "게시글 한 건 = 테이블의 한 행"이라는 감각만 잡으면 된다.

## 20.5 DB 연결

이제 자바 코드에서 이 DB에 연결한다. JDBC에서 연결은 **`DriverManager.getConnection(url, user, password)`** 이 담당한다. 이 메서드가 접속 URL을 보고 알맞은 드라이버를 골라 실제 연결(`Connection`)을 맺어 준다.

가장 중요한 것이 **접속 URL** 의 형식이다. PostgreSQL은 `jdbc:postgresql://호스트:포트/DB이름` 꼴을 쓴다. 우리 인프라(20.2)에 맞추면 아래와 같다.

```text
jdbc:postgresql://localhost:5432/edu_spring
        ↑프로토콜        ↑호스트  ↑포트  ↑DB이름
```

호스트가 `localhost`인 이유는 20.3에서 말했듯 **우리 자바 코드는 컨테이너 밖(호스트)에서 돌기 때문**이다. 컨테이너가 5432 포트를 호스트로 열어 두었으므로(`ports: "5432:5432"`) 호스트의 `localhost:5432`로 접속하면 컨테이너 안 PostgreSQL에 닿는다.

```java
import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

public class DbConnect {
    // 접속 정보 (인프라 docker-compose-infra.yml 과 일치해야 한다)
    static final String URL  = "jdbc:postgresql://localhost:5432/edu_spring";
    static final String USER = "edu";
    static final String PW   = "edu1234";

    public static void main(String[] args) {
        // try-with-resources: 블록을 벗어나면 conn.close() 가 자동 호출된다
        try (Connection conn = DriverManager.getConnection(URL, USER, PW)) {
            System.out.println("연결 성공: " + conn.getCatalog());  // edu_spring
        } catch (SQLException e) {
            // 드라이버 JAR가 없거나 DB가 안 떠 있으면 여기로 온다
            e.printStackTrace();
        }
    }
}
```

`try (Connection conn = ...)` 형태의 **try-with-resources**를 쓰면 블록이 끝날 때 연결이 자동으로 닫힌다. DB 연결은 한정된 자원이므로 반드시 닫아야 하며, 이 문법이 그 뒤처리를 대신해 준다. 이후 절의 모든 예제도 이 방식으로 자원을 닫는다.

## 20.6 데이터 저장 (INSERT)

데이터를 넣을 때는 SQL `INSERT`를 실행한다. JDBC에서는 SQL을 **`PreparedStatement`** 로 준비하는 것을 원칙으로 삼는다. `PreparedStatement`는 SQL 문장에서 값이 들어갈 자리를 물음표 `?`(플레이스홀더)로 비워 두고, 나중에 `setXxx`로 값을 **바인딩**하는 방식이다.

왜 값을 문자열로 이어 붙이지 않고 이렇게 하는가? 두 가지 이유다. 첫째, **SQL 인젝션**을 막는다. 사용자 입력을 SQL 문자열에 직접 붙이면 악의적 입력이 SQL 구조를 바꿀 수 있는데, 바인딩은 입력을 항상 "값"으로만 취급해 이를 원천 차단한다. 둘째, 타입 변환과 이스케이프를 드라이버가 알아서 처리해 준다.

값을 바꾸는 SQL(INSERT/UPDATE/DELETE)은 **`executeUpdate()`** 로 실행하며, 반환값은 "영향받은 행 수"다.

```java
String sql = "INSERT INTO board (title, writer, content) VALUES (?, ?, ?)";

try (Connection conn = DriverManager.getConnection(URL, USER, PW);
     PreparedStatement ps = conn.prepareStatement(sql)) {

    ps.setString(1, "첫 글입니다");     // 1번째 ? ← 제목
    ps.setString(2, "홍길동");          // 2번째 ? ← 작성자
    ps.setString(3, "JDBC 저장 예제");  // 3번째 ? ← 본문

    int affected = ps.executeUpdate();  // 값 변경 SQL → executeUpdate
    System.out.println(affected + "행 저장됨");   // 1행 저장됨
}
```

`setString(1, ...)` 의 첫 인자 `1`은 SQL 안 물음표의 **순번(1부터 시작)** 이다. 정수는 `setInt`, 실수는 `setDouble`처럼 타입에 맞는 `setXxx`를 쓴다.

## 20.7 데이터 수정 (UPDATE)

수정도 저장과 구조가 똑같다. SQL만 `UPDATE`가 되고, 어떤 행을 바꿀지 `WHERE` 절로 지정한다. 실행은 역시 값 변경이므로 **`executeUpdate()`** 를 쓰고, 반환값(영향받은 행 수)으로 실제로 몇 건이 바뀌었는지 확인할 수 있다.

```java
String sql = "UPDATE board SET title = ?, content = ? WHERE id = ?";

try (Connection conn = DriverManager.getConnection(URL, USER, PW);
     PreparedStatement ps = conn.prepareStatement(sql)) {

    ps.setString(1, "수정된 제목");
    ps.setString(2, "내용을 고쳤습니다");
    ps.setLong(3, 1L);                  // id = 1 인 글을 수정

    int updated = ps.executeUpdate();
    if (updated == 0) {
        System.out.println("해당 id의 글이 없습니다");   // WHERE 에 맞는 행이 없으면 0
    } else {
        System.out.println(updated + "행 수정됨");
    }
}
```

여기서 `WHERE id = ?`를 빠뜨리면 **테이블의 모든 행이 수정된다.** UPDATE와 DELETE에서 `WHERE` 절은 "얼마나 넓게 영향을 줄지"를 결정하는 안전장치이므로 항상 신중히 지정해야 한다.

## 20.8 데이터 삭제 (DELETE)

삭제 역시 같은 패턴이다. SQL이 `DELETE`가 되고 `WHERE`로 대상을 특정한 뒤 `executeUpdate()`로 실행한다.

```java
String sql = "DELETE FROM board WHERE id = ?";

try (Connection conn = DriverManager.getConnection(URL, USER, PW);
     PreparedStatement ps = conn.prepareStatement(sql)) {

    ps.setLong(1, 1L);                  // id = 1 인 글을 삭제
    int deleted = ps.executeUpdate();
    System.out.println(deleted + "행 삭제됨");
}
```

UPDATE와 마찬가지로 `WHERE`를 생략하면 **테이블의 모든 행이 삭제된다.** 실무에서는 아예 지우지 않고 `deleted` 같은 플래그 열을 두어 "지운 것으로 표시"만 하는 소프트 삭제(soft delete)를 쓰기도 하지만, 이 장에서는 개념을 위해 실제 DELETE를 다룬다.

## 20.9 데이터 읽기 (SELECT)

읽기는 앞의 세 가지와 결정적으로 다르다. 조회 SQL(`SELECT`)은 값을 바꾸지 않고 **결과 표를 돌려주므로**, `executeUpdate()`가 아니라 **`executeQuery()`** 를 쓰고, 그 반환값으로 **`ResultSet`** 을 받는다. `ResultSet`은 조회 결과 표를 담은 객체로, **커서(cursor)** 가 첫 행 앞에 놓인 채 시작한다.

읽는 방식은 "한 행씩 앞으로 이동하며 꺼내기"다. **`rs.next()`** 를 호출하면 커서가 다음 행으로 이동하고, 그 행이 있으면 `true`(없으면 `false`)를 돌려준다. 그래서 보통 `while (rs.next()) { ... }` 반복으로 모든 행을 훑는다. 각 열의 값은 `rs.getXxx("컬럼명")`(또는 열 번호)으로 타입에 맞게 꺼낸다.

```java
String sql = "SELECT id, title, writer, created_at FROM board ORDER BY id DESC";

try (Connection conn = DriverManager.getConnection(URL, USER, PW);
     PreparedStatement ps = conn.prepareStatement(sql);
     ResultSet rs = ps.executeQuery()) {        // 조회 → executeQuery → ResultSet

    while (rs.next()) {                          // 다음 행이 있는 동안 반복
        long   id      = rs.getLong("id");
        String title   = rs.getString("title");
        String writer  = rs.getString("writer");
        Timestamp when = rs.getTimestamp("created_at");
        System.out.printf("[%d] %s / %s / %s%n", id, title, writer, when);
    }
}
```

정리하면 **값 변경 = `executeUpdate()`(행 수 반환), 조회 = `executeQuery()`(`ResultSet` 반환)** 이 JDBC 읽기/쓰기를 가르는 핵심 분기다.

## 20.10 프로시저·함수 호출

DB 안에는 SQL로 미리 정의해 둔 **저장 프로시저(stored procedure)** 나 **함수(function)** 가 있을 수 있다. 여러 SQL을 묶어 서버 쪽에 저장해 둔 로직이다. 이런 것을 자바에서 호출할 때는 `PreparedStatement`의 사촌인 **`CallableStatement`** 를 쓰고, `{ call 이름(...) }` 형태의 **이스케이프 문법**으로 호출한다.

아래는 PostgreSQL에 게시글 수를 세는 함수를 하나 만들어 두고(먼저 SQL로 정의), 자바에서 호출하는 예다.

```sql
-- (DB 쪽) 게시글 개수를 돌려주는 함수 정의 — Adminer/psql 에서 한 번 실행
CREATE OR REPLACE FUNCTION count_board() RETURNS bigint AS $$
    SELECT count(*) FROM board;
$$ LANGUAGE sql;
```

```java
// (자바 쪽) CallableStatement 로 함수 호출
try (Connection conn = DriverManager.getConnection(URL, USER, PW);
     CallableStatement cs = conn.prepareCall("{ ? = call count_board() }")) {

    cs.registerOutParameter(1, java.sql.Types.BIGINT);  // 1번 = 반환(OUT) 파라미터
    cs.execute();
    long total = cs.getLong(1);                         // 결과 꺼내기
    System.out.println("총 게시글 수: " + total);
}
```

프로시저가 결과를 **OUT 파라미터**로 돌려줄 때는 `registerOutParameter`로 그 자리와 타입을 먼저 등록한 뒤, 실행 후 `getXxx`로 꺼낸다. 다만 실무에서는 조회 정도라면 굳이 프로시저를 쓰지 않고 일반 SELECT나 JPA로 처리하는 경우가 많다.

## 20.11 트랜잭션

**트랜잭션(transaction)** 은 "여러 SQL을 하나의 원자적 단위로 묶어, 전부 성공하거나 전부 취소되게 하는" 작업 단위다. 계좌 이체(A에서 빼고 B에 더하기)처럼 두 작업이 함께 성공해야만 의미가 있는 경우, 중간에 실패하면 앞선 변경도 **되돌려야(rollback)** 한다.

JDBC의 기본 동작은 **자동 커밋(auto-commit)** 이라, `executeUpdate()` 한 번마다 즉시 DB에 확정된다. 여러 SQL을 하나로 묶으려면 이 자동 커밋을 꺼야 한다. 순서는 항상 이렇다: **`setAutoCommit(false)` → SQL 여러 개 실행 → 성공하면 `commit()`, 도중 예외가 나면 `rollback()`.**

```java
Connection conn = null;
try {
    conn = DriverManager.getConnection(URL, USER, PW);
    conn.setAutoCommit(false);            // 수동 트랜잭션 시작

    try (PreparedStatement ps = conn.prepareStatement(
             "INSERT INTO board(title, writer) VALUES (?, ?)")) {
        ps.setString(1, "묶음 글 1"); ps.setString(2, "kim"); ps.executeUpdate();
        ps.setString(1, "묶음 글 2"); ps.setString(2, "lee"); ps.executeUpdate();
    }

    conn.commit();                        // 둘 다 성공 → 함께 확정
    System.out.println("트랜잭션 커밋");
} catch (SQLException e) {
    if (conn != null) conn.rollback();    // 하나라도 실패 → 전부 취소
    System.out.println("롤백됨: " + e.getMessage());
} finally {
    if (conn != null) { conn.setAutoCommit(true); conn.close(); }
}
```

이 원자성·일관성이 왜 필요한지, 그리고 격리 수준·동시성 문제는 `docs/CS_데이터베이스_개념서/03-트랜잭션과-ACID.md`와 `04-동시성-제어.md`에서 이론으로 깊게 다룬다. 실무에서는 이 `setAutoCommit`/`commit`/`rollback`을 손으로 부르지 않고 Spring의 `@Transactional`이 대신 처리하는데(→ `docs/SPRING_개념서/04-데이터베이스와-jpa.md`), 그 애너테이션 뒤에서 벌어지는 일이 바로 이 코드다.

## 20.12 게시판 구현

지금까지의 조각(연결 → 저장/수정/삭제/읽기)을 하나로 모으면 간단한 **게시판 CRUD**가 된다. 아래는 게시글을 담는 `Board` 클래스와, `board` 테이블에 대한 CRUD를 모은 DAO(Data Access Object) 골격이다.

```java
// 게시글 한 건을 담는 데이터 클래스
class Board {
    long id; String title; String writer; String content;
    Board(String title, String writer, String content) {
        this.title = title; this.writer = writer; this.content = content;
    }
}

class BoardDao {
    static final String URL="jdbc:postgresql://localhost:5432/edu_spring",
                        USER="edu", PW="edu1234";

    private Connection conn() throws SQLException {
        return DriverManager.getConnection(URL, USER, PW);
    }

    // Create — 저장
    void insert(Board b) throws SQLException {
        String sql = "INSERT INTO board(title, writer, content) VALUES (?,?,?)";
        try (Connection c = conn(); PreparedStatement ps = c.prepareStatement(sql)) {
            ps.setString(1, b.title); ps.setString(2, b.writer); ps.setString(3, b.content);
            ps.executeUpdate();
        }
    }

    // Read — 전체 목록
    List<Board> findAll() throws SQLException {
        String sql = "SELECT id, title, writer, content FROM board ORDER BY id DESC";
        List<Board> list = new ArrayList<>();
        try (Connection c = conn(); PreparedStatement ps = c.prepareStatement(sql);
             ResultSet rs = ps.executeQuery()) {
            while (rs.next()) {
                Board b = new Board(rs.getString("title"), rs.getString("writer"),
                                    rs.getString("content"));
                b.id = rs.getLong("id");
                list.add(b);
            }
        }
        return list;
    }

    // Update — 제목/본문 수정
    void update(long id, String title, String content) throws SQLException {
        String sql = "UPDATE board SET title=?, content=? WHERE id=?";
        try (Connection c = conn(); PreparedStatement ps = c.prepareStatement(sql)) {
            ps.setString(1, title); ps.setString(2, content); ps.setLong(3, id);
            ps.executeUpdate();
        }
    }

    // Delete — 삭제
    void delete(long id) throws SQLException {
        try (Connection c = conn();
             PreparedStatement ps = c.prepareStatement("DELETE FROM board WHERE id=?")) {
            ps.setLong(1, id); ps.executeUpdate();
        }
    }
}
```

CRUD 네 메서드가 모두 20.5~20.9에서 배운 그대로임을 확인하자. 이 정도의 반복적인 연결·매핑 코드를 **자동으로 없애 주는 것**이 실무의 JPA/Hibernate다. JPA에서는 `Board`에 `@Entity`를 붙이고 `boardRepository.save(b)`/`findAll()`만 부르면 위의 SQL과 ResultSet 매핑을 프레임워크가 대신 짜 준다(→ `docs/SPRING_개념서/04-데이터베이스와-jpa.md`). 이 장의 JDBC를 이해하고 나면, JPA가 "무엇을 대신 해 주는지"가 선명해진다.

## 이 장의 JDBC 코드를 직접 실행하려면

JDBC 코드는 **드라이버 JAR + 실제 DB**가 있어야 돈다. 실행 절차는 다음과 같다.

1. **인프라 PostgreSQL 띄우기**(20.2):
   ```bash
   docker compose -f docker/docker-compose-infra.yml up -d postgres adminer
   docker ps --format '{{.Names}}\t{{.Ports}}'   # 컨테이너 이름 확인
   ```
2. **테이블 만들기**(20.4의 DDL): Adminer(http://localhost:8081, 서버=`postgres`, 20.3)의 SQL 명령에 붙여 넣거나, psql로 실행한다.
   ```bash
   docker exec -it docker-postgres-1 psql -U edu -d edu_spring   # 접속 후 DDL 붙여넣기
   ```
   (`docker-postgres-1`은 1단계에서 확인한 실제 이름으로 바꾼다. `chapter-cs-database/README.md`의 실습 방식과 동일하다.)
3. **자바 코드에 PostgreSQL 드라이버 JAR를 붙여 실행**한다. 순수 JDBC 예제를 실행하려면 클래스패스에 드라이버가 있어야 한다(→ 『10장』 10.1). 실무 프로젝트라면 `build.gradle`에 `runtimeOnly 'org.postgresql:postgresql'`을 넣고 빌드 도구로 실행하면 된다. 접속 URL은 항상 `jdbc:postgresql://localhost:5432/edu_spring`, 사용자 `edu`, 비밀번호 `edu1234`이다.
4. 결과는 **Adminer에서 `SELECT * FROM board;`** 로 눈으로 검증한다.

> DB를 내리려면 `docker compose -f docker/docker-compose-infra.yml down`(볼륨 유지) 또는 `down -v`(데이터까지 삭제)를 쓴다.

## ⚠️ 흔한 오해와 함정

- **"SELECT도 executeUpdate로 실행한다"** — 아니다. 조회는 결과 표를 돌려주므로 `executeQuery()`로 실행하고 `ResultSet`을 받는다. `executeUpdate()`는 INSERT/UPDATE/DELETE처럼 값을 바꾸는 SQL 전용이며 영향받은 행 수(정수)를 돌려준다.
- **"값은 SQL 문자열에 직접 이어 붙이면 된다"** — 위험하다. 사용자 입력을 문자열로 붙이면 SQL 인젝션에 노출된다. 항상 `?` 플레이스홀더 + `PreparedStatement`의 `setXxx` 바인딩을 쓴다.
- **"UPDATE/DELETE에 WHERE는 선택이다"** — `WHERE`를 빠뜨리면 테이블의 **모든 행**이 수정/삭제된다. 대상을 좁히는 안전장치이므로 반드시 지정한다.
- **"Adminer 서버 칸에 localhost를 적으면 된다"** — Adminer는 도커 네트워크 안 컨테이너라 서버명은 서비스명 `postgres`다. `localhost`는 컨테이너 밖 우리 자바 코드가 쓰는 주소다.
- **"자원은 안 닫아도 GC가 알아서 한다"** — Connection/Statement/ResultSet은 한정된 DB 자원이라 닫지 않으면 누수된다. `try-with-resources`로 반드시 닫는다.

## 요약

- **JDBC**는 자바가 관계형 DB에 접속해 SQL을 실행하는 표준 API로, `java.sql` 인터페이스 + DBMS별 드라이버(JAR) 구조다. 흐름은 연결→SQL 준비→바인딩→실행→닫기의 5단계다. (20.1)
- 이 프로젝트의 **DBMS**는 인프라 도커의 PostgreSQL(`edu_spring`/`edu`/`edu1234`/`5432`)이며 `docker compose -f docker/docker-compose-infra.yml up -d postgres`로 띄운다. (20.2)
- **Adminer**(http://localhost:8081)로 브라우저에서 DB를 보고 SQL을 실행하며, 서버 칸에는 서비스명 `postgres`를 넣는다. (20.3)
- 데이터를 담을 **테이블(board)** 을 DDL로 만들어 두고 시작한다. (20.4)
- **`DriverManager.getConnection("jdbc:postgresql://localhost:5432/edu_spring", ...)`** 으로 연결하며, try-with-resources로 자동으로 닫는다. (20.5)
- **저장/수정/삭제**는 `PreparedStatement` + `?` 바인딩 + **`executeUpdate()`**(행 수 반환)로, WHERE로 대상을 좁힌다. (20.6~20.8)
- **읽기**는 **`executeQuery()`** 로 **`ResultSet`** 을 받아 `while(rs.next())`로 한 행씩 `getXxx`로 꺼낸다. (20.9)
- **프로시저/함수 호출**은 `CallableStatement`와 `{ call ... }` 문법, OUT 파라미터는 `registerOutParameter`로 다룬다. (20.10)
- **트랜잭션**은 `setAutoCommit(false)` → 실행 → 성공 시 `commit()`, 실패 시 `rollback()`으로 여러 SQL을 원자 단위로 묶는다. (20.11)
- 이 조각들을 모으면 **게시판 CRUD DAO**가 되며, 이 반복 코드를 자동화한 것이 실무의 JPA다. (20.12)

> DB 이론(관계 모델·정규화·트랜잭션·동시성·인덱싱)은 `docs/CS_데이터베이스_개념서`에서, 이 JDBC 위에 얹는 JPA/`@Transactional`은 `docs/SPRING_개념서/04-데이터베이스와-jpa.md`에서 이어진다. 이 장의 순수 JDBC를 이해해야 그 위층이 무엇을 대신해 주는지가 보인다.
