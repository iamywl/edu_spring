# Chapter 1: Spring이 해결하는 문제

> "모든 기술은 문제를 해결하기 위해 탄생한다. 문제를 모르면 기술을 이해할 수 없다."

우리는 대개 기술을 "어떻게 쓰는가"부터 배운다. 어노테이션을 붙이고, 설정 파일을 채우고, 실행 버튼을 누른다. 하지만 그렇게 배운 지식은 오래가지 못한다. 문법은 잊히고, 버전이 바뀌면 무용지물이 되며, 조금만 상황이 달라져도 응용할 수 없다. 진짜 오래 남는 지식은 "왜 이 기술이 이렇게 생겼는가"를 이해하는 것이다.

Spring은 오늘날 Java 진영에서 가장 널리 쓰이는 프레임워크다. 그러나 Spring의 수많은 개념 — IoC, DI, AOP, 자동 설정 — 은 하늘에서 뚝 떨어진 것이 아니다. 그것들은 하나같이 **"당시 개발자들이 겪던 고통"**에 대한 대답이다. 그래서 이 책의 첫 장은 Spring의 기능을 설명하지 않는다. 대신, Spring이 **없던 시절**로 시간을 되돌려, 개발자들이 어떤 지옥을 통과했는지를 먼저 본다.

Spring을 제대로 이해하려면, Spring이 태어나기 전의 고통을 먼저 느껴야 한다. 그 고통을 알아야만 "아, 그래서 이런 설계가 나왔구나"라는 깨달음이 온다. 이 장이 끝날 때쯤이면, 여러분은 Spring의 코드 한 줄 없이도 Spring이 왜 필요한지를 설명할 수 있게 될 것이다.

## 1.1 Spring 이전의 Java 웹 개발

### 개념: 모든 것을 혼자 하던 시대

1990년대 후반, Java로 웹 애플리케이션을 만든다는 것은 곧 **Servlet을 직접 다룬다**는 뜻이었다. Servlet은 자바 웹의 가장 원초적인 형태로, HTTP 요청이 들어오면 `doGet`, `doPost` 같은 메서드가 호출되는 구조다. 개념 자체는 단순했지만, 그 단순함이 오히려 문제였다. **너무 많은 책임이 한 곳에 몰렸기 때문이다.**

### 원리: 왜 Servlet만으로는 지옥이었는가

전형적인 Servlet 코드 한 조각을 보자. 회원 정보를 조회해 HTML로 뿌려주는, 지극히 평범한 기능이다.

```java
public class UserServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp)
            throws ServletException, IOException {

        // 1) HTTP 파라미터 파싱
        String userId = req.getParameter("id");

        Connection conn = null;
        PreparedStatement stmt = null;
        ResultSet rs = null;
        try {
            // 2) DB 연결을 직접 맺음
            Class.forName("com.mysql.jdbc.Driver");
            conn = DriverManager.getConnection(
                "jdbc:mysql://localhost:3306/mydb", "root", "1234");

            // 3) SQL 작성 및 실행
            stmt = conn.prepareStatement("SELECT name, email FROM users WHERE id = ?");
            stmt.setString(1, userId);
            rs = stmt.executeQuery();

            // 4) HTML을 문자열로 직접 조립
            PrintWriter out = resp.getWriter();
            out.println("<html><body>");
            if (rs.next()) {
                out.println("<h1>" + rs.getString("name") + "</h1>");
                out.println("<p>" + rs.getString("email") + "</p>");
            }
            out.println("</body></html>");

        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            // 5) 리소스 정리 — 순서를 틀리면 커넥션 누수
            try { if (rs != null) rs.close(); } catch (Exception e) {}
            try { if (stmt != null) stmt.close(); } catch (Exception e) {}
            try { if (conn != null) conn.close(); } catch (Exception e) {}
        }
    }
}
```

이 짧은 메서드 하나가 **다섯 가지 완전히 다른 일**을 동시에 하고 있다.

```
        ┌─────────────────────────────────────────────┐
        │              doGet() 메서드 하나              │
        ├─────────────────────────────────────────────┤
        │  ① HTTP 파싱     (웹 계층의 일)               │
        │  ② DB 연결       (인프라의 일)                │
        │  ③ SQL 실행      (데이터 접근의 일)           │
        │  ④ HTML 생성     (화면(뷰)의 일)              │
        │  ⑤ 리소스 정리   (자원 관리의 일)             │
        └─────────────────────────────────────────────┘
                    ↑ 다섯 역할이 한 몸에 뒤엉킴
```

이것은 마치 **식당에서 한 사람이 주문받고, 요리하고, 서빙하고, 설거지까지 다 하는 것**과 같다. 손님이 한 명일 때는 그럭저럭 돌아가지만, 손님이 늘거나(트래픽 증가), 메뉴가 바뀌거나(요구사항 변경), 요리사가 바뀌면(개발자 교체) 순식간에 무너진다.

구체적으로 무엇이 문제였는가?

- **재사용 불가**: DB 연결 로직이 모든 Servlet마다 복사-붙여넣기로 반복된다. 접속 정보 하나 바뀌면 수십 개 파일을 고쳐야 한다.
- **테스트 불가**: 비즈니스 로직을 검증하려면 실제 웹 서버와 DB가 필요하다. 단위 테스트라는 개념 자체가 성립하지 않는다.
- **변경에 취약**: HTML 디자인만 바꾸고 싶어도 SQL과 뒤엉킨 자바 코드를 건드려야 한다.
- **자원 누수**: `finally`에서 close 순서를 조금만 틀려도 커넥션이 새고, 결국 서버가 죽는다.

### 원리: EJB — 문제를 풀려다 더 큰 문제를 만들다

Servlet의 무질서를 해결하려는 시도로 등장한 것이 **EJB(Enterprise JavaBeans)**다. EJB는 트랜잭션, 보안, 분산 처리 같은 "엔터프라이즈급" 기능을 표준으로 제공하겠다는 야심 찬 규격이었다. 그러나 결과는 정반대였다. **너무 무겁고, 너무 복잡했다.**

EJB로 간단한 비즈니스 컴포넌트 하나를 만들려면 다음이 전부 필요했다.

```
   EJB로 컴포넌트 하나 만들기 (EJB 2.x 시절)
   ┌──────────────────────────────────────────────┐
   │ 1. Home 인터페이스        (생성/조회용)         │
   │ 2. Remote 인터페이스      (원격 호출용)         │
   │ 3. Bean 구현 클래스       (실제 로직)           │
   │ 4. ejb-jar.xml            (배포 서술자, 수백 줄) │
   │ 5. 벤더별 설정 XML        (WebLogic/JBoss 전용)  │
   └──────────────────────────────────────────────┘
          ↓
   비즈니스 로직 3줄을 위해 파일 5개 + XML 수백 줄
```

EJB의 치명적인 문제들:

- **배포 서술자 XML이 수백 줄**: 코드보다 설정이 더 길었다.
- **원격 호출 인터페이스 강제**: 같은 서버 안에서 부르는데도 원격 호출 규약을 지켜야 했다.
- **특정 앱서버 종속**: WebLogic이나 JBoss 같은 상용 애플리케이션 서버 없이는 아예 돌아가지 않았다. 코드가 벤더에 묶였다.
- **테스트하려면 서버 필수**: 로직 한 줄 검증하려고 무거운 앱서버를 통째로 띄워야 했다.

이건 마치 **동네 카페를 여는데 호텔급 주방 설비를 강제로 들여놓으라는** 격이었다. 개발자들은 자기 시간의 **90%를 설정과 씨름하는 데** 쓰고, 정작 돈이 되는 비즈니스 로직에는 겨우 **10%**밖에 쓰지 못했다.

```
   개발자의 하루 (EJB 시대)
   ┌────────────────────────────────────────────┐
   │ ██████████████████████████████████  90%     │  ← XML/설정/앱서버 씨름
   │ ████                                 10%     │  ← 실제 비즈니스 로직
   └────────────────────────────────────────────┘
```

### 원리: Spring의 등장 — "EJB 없이 만드는 게 더 낫다"

이 고통 한가운데서 한 사람이 반기를 들었다. **로드 존슨(Rod Johnson)**이다. 그는 2002년 『Expert One-on-One J2EE Design and Development』라는 책을 출간했다. 이 책의 주장은 당시로서는 거의 이단에 가까웠다.

> "EJB 없이도 엔터프라이즈 애플리케이션을 만들 수 있다. 아니, **없이 만드는 게 더 낫다.**"

그가 책에서 예제로 제시한 인프라 코드가 개발자들 사이에서 폭발적인 반응을 얻었고, 2003년 이것이 정식 오픈소스 프로젝트로 공개되었다. 그것이 바로 **Spring Framework**다. "겨울(EJB의 침체기) 다음에 오는 봄(Spring)"이라는 이름은 결코 우연이 아니었다.

Spring의 철학은 명확했다.

- **POJO(Plain Old Java Object) 기반**: 특별한 인터페이스를 상속받거나 규격을 따를 필요 없이, 평범한 자바 객체 그대로 쓴다.
- **경량 & 모듈화**: 무거운 앱서버가 필요 없고, 필요한 기능만 골라 쓴다.
- **테스트 용이성**: 서버 없이도 순수 자바 단위 테스트가 가능하다.
- **IoC/DI**: 객체의 생성과 연결을 프레임워크에 맡겨, 개발자는 로직에만 집중한다.

```
   패러다임의 전환
   ┌─────────────────────┐        ┌─────────────────────┐
   │      EJB 방식        │        │     Spring 방식      │
   ├─────────────────────┤        ├─────────────────────┤
   │ 무거운 규격 상속     │  ───▶  │ 평범한 POJO          │
   │ 앱서버 종속          │        │ 어디서든 실행        │
   │ 서버 없이 테스트 불가│        │ 순수 JUnit 테스트    │
   │ 설정 90% / 로직 10%  │        │ 설정 ↓ / 로직 ↑      │
   └─────────────────────┘        └─────────────────────┘
```

Spring 덕분에 개발자는 다시 **비즈니스 로직에 집중**할 수 있게 되었다. 앞의 Servlet 예제가 Spring에서 어떻게 분리되는지는 뒤 장에서 자세히 다루지만, 핵심은 "역할별로 코드를 쪼개고, 그 조립은 프레임워크가 대신 한다"는 것이다. 이 "조립을 대신 해준다"는 아이디어가 바로 다음 절의 주제인 **제어의 역전**이다.

## 1.2 프레임워크 vs 라이브러리

### 개념: 비슷해 보이지만 정반대인 두 가지

많은 초심자가 "라이브러리"와 "프레임워크"를 같은 것으로 여긴다. 둘 다 "남이 만든 코드를 가져다 쓴다"는 점에서 비슷해 보이기 때문이다. 하지만 이 둘은 **주도권(제어권)이 누구에게 있는가**에서 근본적으로 갈린다. 이 차이를 이해하는 것이 Spring 전체를 이해하는 열쇠다.

### 원리: 누가 누구를 부르는가

**라이브러리는 도구 상자**다. 내가 필요할 때 꺼내서 **내가 호출**한다. 제어의 흐름을 쥔 쪽은 언제나 나(내 코드)다.

```java
// 라이브러리: 내가 흐름을 주도하고, 필요할 때 도구를 꺼내 쓴다
public String toJson(User user) {
    ObjectMapper mapper = new ObjectMapper();   // 내가 만들고
    return mapper.writeValueAsString(user);     // 내가 부른다
}                                               // 주도권 = 나
```

**프레임워크는 공장**이다. 내 코드를 **프레임워크가 호출**한다. 나는 규칙에 맞게 부품(메서드)을 만들어 컨베이어 벨트에 얹어 놓을 뿐, 언제 그 부품이 작동할지는 프레임워크가 결정한다.

```java
// 프레임워크: 내가 메서드를 만들어 놓으면, "언제 부를지"는 Spring이 정한다
@RestController
public class UserController {

    @GetMapping("/users/{id}")           // "이 경로 요청이 오면"
    public User getUser(@PathVariable String id) {
        return userService.findById(id); // 이 메서드를 Spring이 대신 호출
    }                                    // 주도권 = 프레임워크
}
```

이 흐름의 방향이 뒤집힌 것 — 즉 **"내가 부르던 것"에서 "불려지는 것"으로 바뀐 것** — 을 **IoC(Inversion of Control, 제어의 역전)**라고 부른다. Spring의 심장이 바로 이 IoC다.

```
   라이브러리 (제어권: 나)          프레임워크 (제어권: 프레임워크)
   ┌──────────────┐                ┌──────────────┐
   │   내 코드     │                │  프레임워크   │
   │      │        │                │      │        │
   │      ▼ 호출   │                │      ▼ 호출   │
   │  [라이브러리] │                │   [내 코드]   │
   └──────────────┘                └──────────────┘
   "내가 도구를 부른다"             "프레임워크가 나를 부른다"
        (Call)                       (Don't call us,
                                      we'll call you)
```

마지막 문장은 소프트웨어 공학에서 유명한 격언으로, **"할리우드 원칙(Hollywood Principle)"**이라 불린다. 오디션에서 배우가 "나 뽑아줘"라고 조를 수 없고, 감독이 "필요하면 우리가 부를게(Don't call us, we'll call you)"라고 말하는 상황에 빗댄 것이다. 프레임워크와 개발자의 관계가 정확히 이렇다.

### 원리: 왜 제어권을 넘겨주는 것이 이득인가

제어권을 프레임워크에 넘긴다는 건 언뜻 손해처럼 들린다. 내 마음대로 못 하는 것 아닌가? 하지만 여기에 프레임워크의 진짜 가치가 있다.

- **비유로 보면**: 라이브러리는 **망치**다. 망치는 못을 어디에 어떻게 박을지 전혀 신경 쓰지 않는다. 전부 내 책임이다. 반면 프레임워크는 **공장의 컨베이어 벨트**다. "당신은 3번 구간에서 나사만 조이세요. 나머지는 우리가 알아서 합니다"라고 말한다.
- **일관된 구조 강제**: Spring은 Controller(요청 처리) → Service(비즈니스 로직) → Repository(데이터 접근)라는 계층 규칙을 사실상 강제한다. 이 규칙 덕분에 팀원 누구의 코드를 봐도 구조가 예측 가능하다.

```
   프레임워크가 강제하는 표준 계층 구조
   ┌────────────┐   ┌────────────┐   ┌──────────────┐
   │ Controller │──▶│  Service   │──▶│  Repository  │──▶ DB
   │ (요청 처리)│   │ (비즈로직) │   │ (데이터 접근)│
   └────────────┘   └────────────┘   └──────────────┘
        누가 짜도 같은 자리에 같은 역할 → 협업/유지보수 폭발적 향상
```

혼자 개발할 때는 자유가 좋다. 하지만 여러 명이, 수년에 걸쳐, 수십만 줄의 코드를 함께 다룰 때는 **"자유"보다 "예측 가능한 규칙"**이 훨씬 강력하다. 프레임워크는 바로 이 규칙을 제공하는 대신 약간의 자유를 가져가는 거래다. 그리고 대부분의 실무 상황에서 이 거래는 압도적으로 남는 장사다.

## 1.3 Spring Boot는 왜 만들어졌는가?

### 개념: EJB를 죽인 Spring이, 또 다른 복잡함이 되다

Spring은 EJB의 복잡함을 해결하며 세상을 구했다. 그런데 시간이 지나자, **Spring 자신이 새로운 복잡함의 원천**이 되어버렸다. 아이러니하게도, EJB의 XML 지옥을 비판하던 Spring이 어느새 자기만의 XML 지옥을 만들어낸 것이다.

### 원리: XML 지옥 — Hello World에 XML 3개

전통적인 Spring MVC 프로젝트에서 "Hello World"를 화면에 띄우기까지 필요한 설정을 보자.

```
   전통 Spring MVC "Hello World"에 필요한 것들
   ┌──────────────────────────────────────────────────┐
   │ web.xml                → 서블릿/필터 등록          │
   │ applicationContext.xml → 빈(Bean) 정의             │
   │ dispatcher-servlet.xml → MVC 설정                  │
   │ pom.xml                → 라이브러리 버전 수동 관리 │
   │ 톰캣(Tomcat)           → 별도 설치 & 설정          │
   │ 로깅 설정              → log4j.xml 등              │
   └──────────────────────────────────────────────────┘
        "Hello World" 한 줄을 위해 XML 수백 줄
```

실제 `web.xml`의 일부만 봐도 이렇다.

```xml
<!-- web.xml : DispatcherServlet을 등록하는 것만으로도 이 정도 -->
<web-app>
    <servlet>
        <servlet-name>dispatcher</servlet-name>
        <servlet-class>
            org.springframework.web.servlet.DispatcherServlet
        </servlet-class>
        <init-param>
            <param-name>contextConfigLocation</param-name>
            <param-value>/WEB-INF/dispatcher-servlet.xml</param-value>
        </init-param>
        <load-on-startup>1</load-on-startup>
    </servlet>
    <servlet-mapping>
        <servlet-name>dispatcher</servlet-name>
        <url-pattern>/</url-pattern>
    </servlet-mapping>
</web-app>
```

```xml
<!-- applicationContext.xml : 빈 하나 등록에도 장황한 태그 -->
<beans xmlns="http://www.springframework.org/schema/beans" ...>
    <bean id="userService" class="com.example.UserServiceImpl">
        <property name="userRepository" ref="userRepository"/>
    </bean>
    <bean id="userRepository" class="com.example.UserRepositoryImpl"/>
    <!-- 의존관계가 늘어날수록 이 XML은 수백 줄로 불어난다 -->
</beans>
```

여기에 더해 개발자는 다음까지 손수 관리해야 했다.

- **라이브러리 버전 충돌**: Spring, Jackson, Hibernate 등의 버전을 하나하나 맞춰야 했고, 조금만 어긋나면 알 수 없는 에러가 터졌다.
- **톰캣 설치/설정**: 웹 애플리케이션 서버를 별도로 깔고 WAR 파일을 배포해야 했다.
- **로깅 설정**: 로그 하나 찍으려고 별도 XML을 또 작성했다.

결국 Spring이 EJB의 복잡함을 해결하려다, **자기 자신이 복잡함**이 되어버린 것이다. 문제를 푼 도구가 시간이 지나 새로운 문제가 되는 것 — 이것은 기술의 역사에서 끊임없이 반복되는 패턴이다.

### 원리: Convention over Configuration — Spring Boot의 대답

2014년, Spring 팀은 이 문제에 대한 답으로 **Spring Boot**를 내놓았다. 핵심 철학은 단 하나다.

> **"Convention over Configuration"** — 설정보다 관례(convention)를 우선한다.

즉, "대부분의 경우 개발자가 원하는 표준 설정"을 프레임워크가 미리 알아서 적용하고, 개발자는 **표준에서 벗어나는 것만** 직접 설정하게 한다. 이것을 가능하게 하는 네 가지 무기가 있다.

**① 자동 설정(Auto Configuration)**
classpath에 어떤 라이브러리가 있는지를 보고, 필요한 설정을 프레임워크가 알아서 해준다. 예를 들어 H2 데이터베이스 라이브러리가 있으면 별도 XML 없이 `DataSource`가 자동으로 구성된다.

**② 내장 서버(Embedded Server)**
톰캣을 애플리케이션 안에 품고 있어서, 별도 서버 설치 없이 **JAR 파일 하나로 실행**된다.

```bash
# 톰캣 설치도, WAR 배포도 필요 없다. JAR 하나면 끝.
java -jar myapp.jar
```

**③ 스타터 의존성(Starter Dependencies)**
`spring-boot-starter-web` 하나만 추가하면, 웹 개발에 필요한 라이브러리들이 **서로 호환되는 검증된 버전으로** 통째로 딸려온다. 버전 충돌 지옥에서 해방된다.

```xml
<!-- 예전엔 라이브러리 10개를 버전까지 손수 맞췄지만... -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <!-- 버전조차 부모 POM이 관리 → 이 한 줄로 끝 -->
</dependency>
```

**④ application.yml (또는 .properties)**
장황한 XML 대신 간결한 키-값 형식으로 설정한다.

```yaml
# XML 수백 줄이 이렇게 줄어든다
server:
  port: 8080
spring:
  datasource:
    url: jdbc:h2:mem:testdb
    username: sa
```

이 모든 것을 합치면, 예전에는 XML 수백 줄과 별도 서버가 필요했던 웹 애플리케이션이 이제는 이 정도로 끝난다.

```java
@SpringBootApplication
public class MyApp {
    public static void main(String[] args) {
        SpringApplication.run(MyApp.class, args);
    }
}
```

### 비유: 개인 카페 vs 스타벅스

이 변화를 커피 주문에 비유하면 이해가 쉽다.

```
   Spring(이전)  =  까다로운 개인 카페
   ┌────────────────────────────────────────────┐
   │ "예가체프 원두, 분쇄도 중간, 물 온도 93도,   │
   │  추출 시간 25초, 우유는 65도로 스티밍..."    │
   │   → 모든 것을 손님이 다 지정해야 나온다      │
   └────────────────────────────────────────────┘

   Spring Boot  =  스타벅스
   ┌────────────────────────────────────────────┐
   │ "아메리카노 톨 사이즈요."                     │
   │   → 표준 레시피는 이미 최적화되어 있고,       │
   │      바꾸고 싶은 것만 말하면 된다             │
   │      ("샷 추가", "얼음 적게" 등)             │
   └────────────────────────────────────────────┘
```

스타벅스라고 커스터마이징이 불가능한 게 아니다. 샷을 추가할 수도, 시럽을 뺄 수도 있다. **차이는 "기본값이 이미 훌륭하게 세팅되어 있어서, 필요한 것만 바꾸면 된다"**는 점이다. Spring Boot의 자동 설정도 똑같다. 표준 설정은 다 되어 있고, 개발자는 표준을 벗어나야 할 때만 개입한다.

정리하면, 기술의 진화는 이런 흐름이었다.

```
   Servlet 지옥 ──▶ EJB 지옥 ──▶ Spring(XML 지옥) ──▶ Spring Boot
   (전부 혼자)    (설정 90%)    (자기가 복잡해짐)    (관례 > 설정)
        ↓             ↓              ↓                   ↓
   무질서         과도한 무게    무거운 설정          "필요한 것만"
```

## ⚠️ 흔한 오해와 함정

**오해 1: "Spring = Spring Boot다"**
가장 흔한 착각이다. Spring(Framework)은 IoC/DI, AOP, 트랜잭션 등을 제공하는 **핵심 프레임워크**이고, Spring Boot는 그 Spring을 **쉽게 쓰도록 자동 설정과 관례를 얹은 도구**다. Spring Boot 없이도 Spring은 쓸 수 있고, Spring Boot 내부는 여전히 Spring으로 돌아간다. Boot는 Spring을 대체한 게 아니라 **감싼** 것이다.

**오해 2: "프레임워크는 무조건 라이브러리보다 좋다"**
아니다. 제어권을 프레임워크에 넘기는 것은 **거래**다. 예측 가능한 구조와 자동화를 얻는 대신, 자유와 유연함을 일부 내준다. 아주 작은 스크립트나 단순한 유틸리티에는 프레임워크가 오히려 과하다(over-engineering). 도구는 문제의 크기에 맞춰 골라야 한다.

**오해 3: "자동 설정은 마법이라 몰라도 된다"**
Spring Boot의 자동 설정은 마법처럼 보이지만, 실제로는 **명확한 규칙(classpath에 무엇이 있는가, 어떤 빈이 이미 등록되어 있는가)에 기반한 조건부 설정**일 뿐이다. 원리를 모르면 자동 설정이 예상과 다르게 동작할 때(예: 원하는 빈이 만들어지지 않을 때) 손도 못 댄다. "마법"이라고 넘기지 말고 "왜 이렇게 자동으로 됐는가"를 항상 물어야 한다. (이 원리는 뒤 장에서 자세히 다룬다.)

**오해 4: "EJB는 그냥 나쁜 기술이었다"**
EJB가 겪은 문제는 사실이지만, EJB가 던졌던 질문(트랜잭션, 보안, 분산을 어떻게 표준화할 것인가)은 정당했다. 실제로 후대의 EJB 3.x는 Spring의 영향을 받아 크게 개선되었다. 기술을 "좋다/나쁘다"로만 나누지 말고, **어떤 문제를 어떤 대가로 풀려 했는가**로 보는 눈을 길러야 한다.

**함정: "설정이 안 보이니 안심이다"**
Spring Boot는 설정을 숨겨준다. 편리하지만, 숨겨진 설정이 무엇을 하고 있는지 모르면 디버깅이 지옥이 된다. 편리함과 이해는 별개다. 편리함을 누리되, 그 아래에서 무슨 일이 벌어지는지는 반드시 이해하려 노력해야 한다.

## 연습문제

**문제 1.** Servlet만으로 웹 개발을 하던 시절, 하나의 `doGet` 메서드가 떠안았던 다섯 가지 책임을 나열하고, 이것이 왜 문제인지 한 문장으로 설명하라.

<details>
<summary>해설</summary>

다섯 가지 책임: ① HTTP 파싱, ② DB 연결, ③ SQL 실행, ④ HTML 생성, ⑤ 리소스 정리.
문제인 이유: 서로 다른 성격의 역할이 한 메서드에 뒤엉켜 있어 재사용·테스트·변경이 모두 어렵고, 하나를 고치면 나머지에 영향이 번지기 때문이다. (식당에서 한 사람이 주문·요리·서빙·설거지를 다 하는 상황을 떠올리면 된다.)
</details>

**문제 2.** EJB 시대의 개발자가 "시간의 90%를 설정에, 10%를 로직에" 썼다고 한다. 그 90%를 잡아먹은 대표적인 요인 세 가지를 들라.

<details>
<summary>해설</summary>

① 수백 줄에 달하는 배포 서술자 XML 작성, ② 원격 호출 인터페이스(Home/Remote) 등 강제된 규격 준수, ③ 특정 앱서버(WebLogic/JBoss) 종속 설정 및 서버를 띄워야만 가능한 테스트. 이 모든 것이 비즈니스 로직과 무관한 "설정 노동"이었다.
</details>

**문제 3.** 라이브러리와 프레임워크의 가장 근본적인 차이는 무엇인가? "할리우드 원칙"이라는 용어를 사용해 설명하라.

<details>
<summary>해설</summary>

근본 차이는 **제어권(주도권)의 방향**이다. 라이브러리는 내가 필요할 때 호출한다(제어권 = 나). 프레임워크는 내가 만든 코드를 프레임워크가 호출한다(제어권 = 프레임워크). 이 뒤집힘이 **IoC(제어의 역전)**이며, "우리한테 전화하지 마, 필요하면 우리가 부를게(Don't call us, we'll call you)"라는 할리우드 원칙으로 표현된다.
</details>

**문제 4.** Spring이 EJB의 복잡함을 해결했는데, 왜 다시 Spring Boot가 필요해졌는가? "XML 지옥"과 "Convention over Configuration"을 연결해 설명하라.

<details>
<summary>해설</summary>

Spring은 EJB를 대체했지만, 시간이 지나며 web.xml·applicationContext.xml·dispatcher-servlet.xml 등 방대한 XML 설정과 라이브러리 버전 관리, 톰캣 설치 등이 새로운 부담(XML 지옥)이 되었다. Spring Boot는 **"관례를 설정보다 우선한다(Convention over Configuration)"**는 철학으로, 표준 설정을 자동화(자동 설정·내장 서버·스타터 의존성·yaml)하고 개발자는 표준을 벗어나는 부분만 설정하게 함으로써 이 지옥을 해소했다.
</details>

**문제 5.** "Spring Boot는 마법이라 원리를 몰라도 된다"는 주장에 대해, 왜 이것이 위험한 생각인지 반박하라.

<details>
<summary>해설</summary>

자동 설정은 마법이 아니라 "classpath에 무엇이 있는가, 어떤 빈이 이미 등록되어 있는가" 같은 명확한 조건에 기반한 규칙일 뿐이다. 원리를 모르면 자동 설정이 예상과 다르게 동작할 때(원하는 빈이 생성되지 않거나, 엉뚱한 기본값이 적용될 때) 원인을 찾을 수 없고 손도 대지 못한다. 편리함과 이해는 별개이며, 실무 디버깅에서는 "왜 자동으로 이렇게 됐는가"를 아는 사람만이 문제를 해결할 수 있다.
</details>

## 요약

- **모든 기술은 문제 해결을 위해 태어난다.** Spring의 개념들도 "당시 개발자의 고통"에 대한 대답이다.
- **Servlet 시대**: 하나의 메서드가 HTTP 파싱·DB 연결·SQL·HTML 생성·리소스 정리를 모두 떠안아 재사용·테스트·변경이 불가능한 무질서였다.
- **EJB 시대**: 이를 표준화하려다 배포 서술자 XML, 원격 인터페이스, 앱서버 종속으로 오히려 더 무거워져 개발자 시간의 90%를 설정에 소모했다.
- **Spring 등장(2002 로드 존슨의 책 → 2003 오픈소스)**: "EJB 없이가 더 낫다"는 철학으로 POJO 기반·경량·테스트 용이·IoC/DI를 내세워 로직에 집중하게 했다.
- **라이브러리 vs 프레임워크**: 차이는 제어권의 방향이다. 라이브러리는 내가 호출하고(도구 상자), 프레임워크는 나를 호출한다(공장). 이 역전이 **IoC**이며 "할리우드 원칙"으로 표현된다.
- **Spring Boot(2014)**: Spring 자신이 XML 지옥이 되자, "Convention over Configuration"을 내세워 자동 설정·내장 서버·스타터 의존성·yaml로 "필요한 것만 설정"하게 만들었다. (개인 카페 → 스타벅스)
- **핵심 태도**: 편리함에 안주하지 말고, 그 아래에서 "왜 이렇게 설계됐는가"를 항상 물어라. 원리를 아는 자만이 문제를 해결한다.
