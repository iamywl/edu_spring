# Chapter 6: 테스트 - 왜 테스트를 작성하는가?

> "테스트 없는 코드는 레거시(legacy)다."
> — Michael Feathers, 『레거시 코드 활용 전략(Working Effectively with Legacy Code)』

---

## 도입: 우리는 왜 테스트를 두려워하는가

당신이 3개월 전에 짠 코드를 다시 열었다고 하자. 요구사항이 바뀌어 함수 하나를 고쳐야 한다.
그런데 손이 떨린다. **"이걸 고치면 어디가 깨질까?"** 코드를 읽어도 확신이 안 선다.
결국 겁이 나서 최소한의 수정만 하고, 원래 있던 문제를 그대로 둔 채 그 옆에 `if` 하나를 덧붙인다.
이렇게 코드는 점점 더 무서운 물건이 되어 간다.

Michael Feathers는 이 상태를 정확히 짚었다. 그가 말하는 **레거시 코드**는 "오래된 코드"가 아니다.
**테스트가 없어서 마음 놓고 바꿀 수 없는 코드**다. 작성한 지 하루밖에 안 된 코드도, 테스트가 없다면
그 순간부터 레거시다. 반대로 10년 된 코드라도 촘촘한 테스트가 지키고 있으면 언제든 자신 있게 뜯어고칠 수 있다.

그래서 이 장의 질문은 "테스트를 어떻게 짜는가"가 아니라 **"왜 짜는가"**부터 시작한다.
도구(JUnit, Mockito, Testcontainers)의 문법은 검색하면 나온다. 하지만 **왜 단위 테스트가 통합 테스트보다 많아야 하는지**,
**왜 진짜 DB 대신 Mock을 쓰다가도 어느 순간 다시 진짜 DB를 Docker로 띄우는지**,
이 판단의 근거는 원리를 이해해야만 몸에 붙는다. 이 장이 그 원리를 그림과 실행되는 코드로 설명한다.

이 장의 코드는 `chapter08-testing` 모듈에서 직접 돌려 볼 수 있다.
읽기만 하면 "그렇구나" 하고 넘어가지만, 실제로 테스트를 깨뜨렸다가 다시 초록불(green)이 켜지는 것을
눈으로 보면 "안전망"이라는 말이 몸으로 이해된다.

## 6.1 테스트의 가치 — 왜 시간을 들여 테스트를 짜는가

테스트를 처음 배우는 사람은 이렇게 생각한다. "코드도 짜야 하는데 테스트까지 짜면 일이 두 배 아닌가?"
맞다. 당장은 두 배다. 하지만 테스트는 **미래의 나에게 보내는 보험**이다. 그 가치는 세 가지로 나뉜다.

### (1) 리팩토링 안전망 — 마음 놓고 뜯어고치기

코드는 한 번 짜고 끝나지 않는다. 요구사항이 바뀌고, 성능을 개선하고, 구조를 다듬는다.
이 모든 "바꾸기"가 **기존 동작을 망가뜨리지 않았는지**를 매번 손으로 확인하는 것은 불가능하다.

테스트가 있으면 다르다. 코드를 고친 뒤 테스트를 돌려 전부 초록불이면, **"내가 건드린 게 기존 동작을 깨지 않았다"**는
확신을 몇 초 만에 얻는다. 빨간불(red)이 뜨면, 그 테스트 이름이 곧 "무엇이 깨졌는지"를 알려 준다.

```
        테스트 없는 리팩토링              테스트 있는 리팩토링
        ┌──────────────┐                  ┌──────────────┐
        │  코드 수정    │                  │  코드 수정    │
        └──────┬───────┘                  └──────┬───────┘
               │                                 │
               ▼                                 ▼
        ┌──────────────┐                  ┌──────────────┐
        │ "괜찮겠지..." │                  │ 테스트 실행   │──▶ 빨간불이면 즉시 발견
        │  (확신 없음)  │                  │  (수 초)      │──▶ 초록불이면 안심
        └──────┬───────┘                  └──────┬───────┘
               │                                 │
               ▼                                 ▼
        운영에서 터진다 💥                  자신 있게 배포 ✅
```

### (2) 문서로서의 테스트 — 코드의 사용 설명서

좋은 테스트는 그 자체로 **살아 있는 명세서(specification)**다. 주석은 코드와 따로 놀다가 낡아 버리지만,
테스트는 낡으면 빨간불이 뜨므로 **거짓말을 할 수 없다**. 특히 한글 메서드명을 쓰면 테스트 목록만 봐도
그 클래스가 무엇을 보장하는지가 한눈에 읽힌다.

```java
class 주문서비스Test {

    @Test
    void 재고가_충분하면_주문에_성공한다() { ... }

    @Test
    void 재고가_부족하면_주문에_실패하고_예외를_던진다() { ... }

    @Test
    void 주문_금액이_10만원_이상이면_무료배송이_적용된다() { ... }
}
```

이 세 줄만 봐도 "주문 서비스가 무엇을 책임지는가"가 문장으로 읽힌다.
JUnit 5부터는 `@DisplayName`으로 더 자연스러운 설명도 붙일 수 있다.

```java
@Test
@DisplayName("주문 금액이 10만 원 이상이면 무료 배송이 적용된다")
void freeShippingOver100000() { ... }
```

### (3) 빠른 피드백 — 버그를 만든 그 자리에서 잡기

버그를 고치는 비용은 **발견이 늦어질수록 기하급수적으로 커진다**. 코드를 짜는 그 순간에 잡으면 1분,
QA 단계에서 잡으면 1시간, 운영(production)에서 고객이 발견하면 하루 이상 + 신뢰 손상이다.

```
버그 발견 시점별 비용
개발 중        │▓                      1분
코드 리뷰      │▓▓▓                    10분
QA 단계        │▓▓▓▓▓▓▓▓               1시간
운영 배포 후   │▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  하루+ (고객 신뢰까지)
```

테스트는 이 피드백 루프를 **개발 중** 단계로 당긴다. 저장하면 테스트가 돌고, 방금 만든 버그가
그 자리에서 빨간불로 튀어나온다.

### 비유: 건물 안전검사

테스트를 짜지 않는 것은 **안전검사 없이 건물을 짓는 것**과 같다.

```
   테스트 없는 개발                     테스트 있는 개발
   ┌────────────────┐                  ┌────────────────┐
   │ 층을 계속 쌓음  │                  │ 한 층 쌓을 때마다│
   │ (검사 안 함)    │                  │  안전검사 통과  │
   └───────┬────────┘                  └───────┬────────┘
           │                                   │
           ▼                                   ▼
   ┌────────────────┐                  ┌────────────────┐
   │ 20층에서 붕괴   │                  │ 3층 결함을      │
   │ (전체 무너짐)   │                  │  즉시 발견·보강 │
   └────────────────┘                  └────────────────┘
   → 어느 층이 원인인지 모름            → 결함 층을 정확히 지목
```

한 층을 올릴 때마다 검사(테스트)를 통과시키면, 20층에서 문제가 생겨도 "방금 올린 층"이 원인임을 안다.
반면 검사 없이 20층을 올린 뒤 무너지면, 어느 층의 결함이 원인인지 알 수 없어 전체를 다시 봐야 한다.
이것이 바로 "빠른 피드백"과 "안전망"이 실무에서 갖는 힘이다.

### 좋은 테스트의 조건 — FIRST 원칙

모든 테스트가 다 가치 있는 것은 아니다. 나쁜 테스트는 오히려 개발 속도를 갉아먹는다.
좋은 단위 테스트가 갖춰야 할 성질을 **FIRST**라는 두문자로 기억한다.

| 글자 | 원칙 | 의미 |
|------|------|------|
| **F** | Fast(빠름) | 수천 개를 몇 초 안에 돌릴 수 있어야 한다. 느리면 아무도 안 돌린다. |
| **I** | Independent(독립적) | 테스트끼리 순서나 상태를 공유하면 안 된다. 하나가 실패해도 다른 것에 영향 없어야 한다. |
| **R** | Repeatable(반복 가능) | 언제, 어디서, 몇 번을 돌려도 같은 결과. "가끔 실패하는(flaky)" 테스트는 최악이다. |
| **S** | Self-validating(자가 검증) | 통과/실패를 스스로 판단. 사람이 로그를 눈으로 읽어야 하면 실패다. |
| **T** | Timely(적시성) | 코드를 짤 때 같이 짠다. 나중으로 미루면 영원히 안 짜게 된다. |

이 원칙들을 지키지 못하는 테스트는 "테스트가 있는 척"만 하는 부채가 된다.

## 6.2 단위 테스트 vs 통합 테스트 — 무엇을, 어디까지 검증할 것인가

테스트에는 크게 두 종류가 있다. **무엇을 격리하고 무엇을 실제로 돌리는가**가 둘을 가른다.

### 단위 테스트(Unit Test) — 클래스 하나를 세상에서 고립시킨다

단위 테스트는 **하나의 클래스(또는 메서드)만** 검증한다. 그 클래스가 의존하는 다른 객체들(Repository, 외부 API 등)은
**Mock(가짜 객체)**으로 대체해 완전히 고립시킨다. Spring 컨텍스트도 띄우지 않는다.
그래서 아주 빠르고, 실패했을 때 "이 클래스의 이 로직이 틀렸다"고 정확히 짚어 준다.

```java
@ExtendWith(MockitoExtension.class)  // Mockito를 JUnit 5에 연결
class UserServiceTest {

    @Mock                       // 가짜 UserRepository를 만든다
    private UserRepository userRepository;

    @Mock                       // 가짜 PasswordEncoder
    private PasswordEncoder passwordEncoder;

    @InjectMocks                // 위 Mock들을 주입받은 진짜 UserService
    private UserService userService;

    @Test
    void 새로운_이메일이면_회원가입에_성공한다() {
        // given(준비): 이 이메일은 아직 없다고 가짜에게 시킨다
        given(userRepository.existsByEmail("new@test.com")).willReturn(false);
        given(passwordEncoder.encode("pw1234")).willReturn("암호화된_비밀번호");
        given(userRepository.save(any(User.class)))
                .willAnswer(invocation -> invocation.getArgument(0));

        // when(실행): 검증 대상 메서드를 호출한다
        User result = userService.register("new@test.com", "pw1234");

        // then(검증): 결과와 상호작용을 확인한다
        assertThat(result.getEmail()).isEqualTo("new@test.com");
        assertThat(result.getPassword()).isEqualTo("암호화된_비밀번호");
        verify(userRepository).save(any(User.class));  // save가 호출됐는가
    }
}
```

여기서 `@Mock`, `@InjectMocks`가 하는 일을 그림으로 보면 이렇다.

```
  진짜 세상                         테스트 세상 (@ExtendWith(MockitoExtension.class))
  ┌─────────────┐                  ┌──────────────────────────────────┐
  │ UserService │                  │  UserService (진짜, @InjectMocks) │
  │   ├─ Repo   │   격리하면        │      ├─ Repo   → 🎭 Mock          │
  │   ├─ Encoder│  ──────────▶      │      ├─ Encoder→ 🎭 Mock          │
  │   └─ Email  │                  │      └─ Email  → 🎭 Mock          │
  └─────────────┘                  └──────────────────────────────────┘
                                   → DB도, 이메일도 안 건드림. 순수 로직만 검증.
```

### 통합 테스트(Integration Test) — 여러 계층을 진짜로 연결한다

통합 테스트는 **Controller → Service → Repository → DB**로 이어지는 여러 계층이
**실제로 함께 잘 동작하는가**를 검증한다. Spring 컨텍스트를 실제로 띄우고, 진짜 DB에 붙는다.
단위 테스트가 놓치는 것 — Bean 배선(wiring), 트랜잭션, JSON 직렬화, SQL 매핑 — 이 여기서 걸린다.

```java
@SpringBootTest                 // 전체 Spring 컨텍스트를 띄운다
@AutoConfigureMockMvc           // HTTP 요청을 흉내 낼 MockMvc를 준비
@Transactional                  // 각 테스트 후 자동 롤백 → DB를 더럽히지 않음
class UserApiIntegrationTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void 회원가입_API를_호출하면_201과_사용자정보를_반환한다() throws Exception {
        // given: 요청 본문
        String requestBody = """
            { "email": "test@test.com", "password": "pw1234" }
            """;

        // when & then: 실제 HTTP 요청을 흉내 내고 응답을 검증
        mockMvc.perform(post("/api/users")
                        .contentType(MediaType.APPLICATION_JSON)
                        .content(requestBody))
                .andExpect(status().isCreated())               // 201
                .andExpect(jsonPath("$.email").value("test@test.com"))
                .andExpect(jsonPath("$.password").doesNotExist()); // 비밀번호 노출 안 됨
    }
}
```

`@Transactional`이 왜 여기 붙는지가 핵심이다. 통합 테스트는 진짜 DB에 쓰기(write)를 하므로,
그대로 두면 테스트를 돌릴 때마다 데이터가 쌓인다. `@Transactional`은 **각 테스트가 끝나면 자동으로 롤백**해서,
DB를 처음 상태로 되돌린다. 덕분에 FIRST 원칙의 R(반복 가능)과 I(독립적)를 지킬 수 있다.

### 두 테스트의 특징 비교

| 항목 | 단위 테스트 | 통합 테스트 |
|------|-------------|-------------|
| **속도** | 매우 빠름 (밀리초) | 느림 (컨텍스트 로딩 초 단위) |
| **범위** | 클래스 하나 | 여러 계층 + DB |
| **Spring 컨텍스트** | 안 띄움 | 띄움 (`@SpringBootTest`) |
| **의존성** | Mock으로 대체 | 실제 Bean/DB 사용 |
| **잡는 버그** | 로직 오류 | 배선·트랜잭션·SQL·직렬화 오류 |
| **실패 시 원인 파악** | 매우 명확 (한 클래스) | 상대적으로 넓음 |

### 테스트 피라미드 — 왜 단위 테스트가 압도적으로 많아야 하는가

테스트를 어떤 비율로 짤 것인가에 대한 고전적 지침이 **테스트 피라미드(Test Pyramid, Mike Cohn)**다.

```
                    /\
                   /  \      E2E 테스트 (적게)
                  / E2E\     - 브라우저까지 전체 흐름
                 /──────\    - 느리고 깨지기 쉬움(flaky)
                /        \
               / 통합테스트 \   통합 테스트 (중간)
              /  Integration \  - 계층 간 연결·DB 검증
             /────────────────\ - 컨텍스트 로딩으로 느림
            /                  \
           /     단위 테스트      \  단위 테스트 (아주 많이)
          /        Unit          \ - 빠르고 정확하게 원인 지목
         /────────────────────────\ - 로직의 모든 분기를 촘촘히
```

**왜 아래쪽(단위)이 넓어야 하는가?** 세 가지 이유가 있다.

1. **속도**: 단위 테스트는 밀리초 단위다. 수천 개를 저장할 때마다 돌려도 개발 흐름을 안 끊는다.
   통합·E2E는 느려서 자주 못 돌린다 → 피드백이 늦어진다.
2. **안정성**: 위로 갈수록 외부 요인(네트워크, 타이밍, 브라우저 렌더링)이 많아져 **가끔 실패하는 flaky 테스트**가 된다.
   단위 테스트는 고립돼 있어 결정적(deterministic)이다.
3. **원인 지목**: 단위 테스트가 깨지면 "이 클래스의 이 로직"이라고 즉시 안다. E2E가 깨지면 "어딘가 잘못됐다"까지만 안다.

거꾸로 뒤집힌 피라미드(통합·E2E만 잔뜩)를 **아이스크림 콘 안티패턴**이라 부른다.
느리고, 자주 깨지고, 원인을 못 짚어서 결국 아무도 테스트를 안 돌리게 된다.

> **참고 — Slice 테스트**: 단위와 통합의 중간에는 Spring이 제공하는 "얇게 저민" 테스트가 있다.
> `@WebMvcTest`는 Controller 계층만, `@DataJpaTest`는 Repository + 내장 DB만 띄운다.
> 전체 컨텍스트(`@SpringBootTest`)보다 가볍게, 특정 계층만 통합적으로 검증하고 싶을 때 쓴다.
> `chapter08-testing`의 `ProductControllerTest`, `ProductRepositoryTestcontainersTest`가 이런 계층별 테스트의 예다.

## 6.3 Mock — 왜 가짜 객체가 필요한가

단위 테스트의 핵심 도구가 **Mock**이다. 왜 진짜 객체 대신 가짜를 쓰는지, 구체적인 상황으로 보자.

### 문제 상황: 진짜 의존성을 그대로 쓰면?

`UserService.createUser()`가 세 개의 협력 객체에 의존한다고 하자.

```java
public User createUser(String email, String rawPassword) {
    if (userRepository.existsByEmail(email)) {         // ① DB 조회
        throw new DuplicateEmailException(email);
    }
    String encoded = passwordEncoder.encode(rawPassword); // ② 암호화
    User user = userRepository.save(new User(email, encoded)); // ③ DB 저장
    emailService.sendWelcomeEmail(email);              // ④ 진짜 이메일 발송
    return user;
}
```

이 메서드를 **진짜 의존성으로** 테스트하면 이런 일이 벌어진다.

- ①③ → **PostgreSQL이 실행 중**이어야 하고, 테스트마다 실제 데이터가 쌓인다.
- ④ → **실제 이메일이 발송된다.** 테스트를 100번 돌리면 100통의 환영 메일이 나간다. 😱

즉, 진짜 의존성은 **느리고, 외부 시스템이 켜져 있어야 하고, 부작용(side effect)을 남긴다.**
우리가 검증하고 싶은 것은 오직 **"createUser의 로직이 올바른가"**인데, 그 로직과 무관한 요소들이 발목을 잡는다.

### 해결: Mock으로 협력 객체를 통제한다

Mock은 우리가 시키는 대로만 행동하는 가짜다. `when`(또는 `given`)으로 **"이렇게 물어보면 이렇게 답해라"**를 정하고,
`verify`로 **"이 메서드가 호출됐는가"**를 검사한다.

```java
@Test
void 이미_존재하는_이메일이면_예외를_던지고_이메일을_보내지_않는다() {
    // given: 가짜 Repository에게 "이 이메일은 이미 있다"고 시킨다
    given(userRepository.existsByEmail("dup@test.com")).willReturn(true);

    // when & then: 예외가 터지는지 확인
    assertThatThrownBy(() -> userService.createUser("dup@test.com", "pw"))
            .isInstanceOf(DuplicateEmailException.class);

    // verify: 중복이므로 저장도, 이메일 발송도 절대 일어나면 안 된다
    verify(userRepository, never()).save(any());
    verify(emailService, never()).sendWelcomeEmail(anyString());
}
```

진짜 DB와 진짜 메일 서버 없이도, **"중복이면 저장·발송을 하지 않는다"**는 로직을 완벽히 검증했다.

### 비유: 스턴트맨

영화의 위험한 장면에서 주연 배우 대신 **스턴트맨**을 쓴다. 카메라에 잡히는 결과(장면)는 똑같지만,
진짜 배우가 위험을 감수할 필요가 없다. Mock이 바로 스턴트맨이다.
검증 대상(주연 = UserService)은 진짜지만, 위험하거나 비싼 협력 객체(DB, 이메일)는 안전한 대역으로 바꾼다.

```
   진짜 DB로 테스트                        Mock으로 테스트
   ┌────────────────────┐                 ┌────────────────────┐
   │ UserService (진짜) │                 │ UserService (진짜) │
   └─────────┬──────────┘                 └─────────┬──────────┘
             │                                      │
     ┌───────┼────────┐                      ┌──────┼───────┐
     ▼       ▼        ▼                       ▼      ▼       ▼
  [PostgreSQL][SMTP][파일]                 🎭Repo  🎭Enc  🎭Email
   느림      진짜메일  부작용                빠름   부작용없음  통제가능
   💥발송됨   💥데이터쌓임                    ✅원하는 대로 응답
```

### 테스트 더블(Test Double)의 종류

"Mock"은 사실 **테스트 더블(가짜 대역의 총칭)** 중 하나다. Gerard Meszaros의 분류에 따르면
용도에 따라 몇 가지로 나뉜다. 실무에서는 뭉뚱그려 "Mock"이라 부르지만, 차이를 알면 도구 선택이 쉬워진다.

| 종류 | 역할 | 예시 |
|------|------|------|
| **Dummy** | 그냥 자리 채우기용. 실제로 안 쓰임 | 파라미터 채우기용 `null` 대체 객체 |
| **Stub** | 미리 정해진 값만 돌려줌 (상태 검증) | `given(...).willReturn(...)` |
| **Mock** | 호출 여부·횟수를 검증 (행위 검증) | `verify(repo).save(...)` |
| **Spy** | 진짜 객체를 감싸되 일부만 가짜로 (`@Spy`) | 실제 로직 + 특정 메서드만 스텁 |
| **Fake** | 가볍게 동작하는 진짜 구현 | `Map` 기반 인메모리 Repository |

Mockito의 `@Mock`은 위 표의 Stub과 Mock 역할을 겸한다. 값을 돌려주게 하면(Stub) 상태 검증에,
`verify`로 호출을 확인하면(Mock) 행위 검증에 쓴다.

### 언제 Mock을, 언제 진짜를 쓸까

Mock은 만능이 아니다. Mock으로는 **"내가 정한 대로 동작한다"**만 검증할 수 있다.
그런데 진짜 DB의 SQL 문법, JSONB 컬럼, 트랜잭션 격리 수준 같은 것은 Mock이 흉내 낼 수 없다.
**"진짜로 붙었을 때 되는가"**는 진짜를 써야만 알 수 있다. 그래서 다음 절의 Testcontainers가 등장한다.

## 6.4 Testcontainers — 왜 진짜 DB를 Docker로 띄우는가

### 문제 상황: H2 내장 DB의 함정

통합 테스트에서 DB가 필요할 때, 가장 쉬운 선택은 **H2 같은 내장(in-memory) DB**다.
설정 없이 메모리에서 순식간에 뜨고 사라지니 편하다. 하지만 여기에 무서운 함정이 있다.

**운영 DB(PostgreSQL)와 테스트 DB(H2)가 다르면, 테스트는 통과하는데 운영에서만 터진다.**

```
  테스트 환경 (H2)                    운영 환경 (PostgreSQL)
  ┌──────────────────┐               ┌──────────────────┐
  │  ✅ 테스트 통과   │               │  💥 런타임 에러   │
  └──────────────────┘               └──────────────────┘
   "우리 코드 완벽해!"                 "왜 배포하니까 터지지?"
```

차이가 나는 대표적인 지점들:

- **JSONB / 배열 타입**: PostgreSQL 고유 타입. H2는 지원이 다르거나 없다.
- **전문 검색(full-text search)**: `to_tsvector`, `@@` 연산자 등은 PostgreSQL 전용.
- **SQL 문법 차이**: `LIMIT/OFFSET`, 시퀀스, 윈도우 함수, `ON CONFLICT`(upsert) 등 방언(dialect)이 다르다.
- **격리 수준·락 동작**: 동시성 관련 동작이 미묘하게 다르다.

즉, H2로 초록불이 떠도 그것은 "H2에서는 된다"일 뿐, **"PostgreSQL에서도 된다"의 증거가 아니다.**

### 해결: Testcontainers — 진짜 DB를 Docker 컨테이너로

**Testcontainers**는 테스트가 시작될 때 **실제 PostgreSQL을 Docker 컨테이너로 띄우고**,
테스트가 끝나면 자동으로 없앤다. 즉, 운영과 **똑같은 DB**를 상대로 테스트한다.

```java
@SpringBootTest
@Testcontainers                  // Testcontainers 생명주기를 JUnit에 연결
class ProductRepositoryTestcontainersTest {

    @Container                   // 컨테이너 생명주기를 관리
    static PostgreSQLContainer<?> postgres =
            new PostgreSQLContainer<>("postgres:16-alpine")
                    .withDatabaseName("testdb")
                    .withUsername("test")
                    .withPassword("test");

    @DynamicPropertySource       // 컨테이너의 랜덤 포트를 Spring 설정에 주입
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired
    private ProductRepository productRepository;

    @Test
    void 상품을_저장하고_이름으로_조회한다() {
        // given
        productRepository.save(new Product("노트북", 1_500_000));

        // when: PostgreSQL 문법 그대로 실행됨
        List<Product> found = productRepository.findByNameContaining("노트");

        // then
        assertThat(found).hasSize(1);
        assertThat(found.get(0).getPrice()).isEqualTo(1_500_000);
    }
}
```

`@DynamicPropertySource`가 왜 필요한지가 포인트다. Testcontainers는 포트 충돌을 막으려고
컨테이너를 **매번 랜덤 포트**에 띄운다. 그 포트를 미리 알 수 없으니, 컨테이너가 뜬 뒤 실제 JDBC URL을
Spring의 `datasource` 설정에 **동적으로** 꽂아 넣는 것이다.

### 실행 흐름

```
  ① 테스트 시작
       │
       ▼
  ② @Container: Docker에서 postgres:16-alpine 컨테이너 기동
       │   (랜덤 포트, 예: 49xxx)
       ▼
  ③ @DynamicPropertySource: 그 포트로 spring.datasource.url 주입
       │
       ▼
  ④ @SpringBootTest: Spring이 진짜 PostgreSQL에 연결
       │
       ▼
  ⑤ 테스트 실행 (진짜 SQL이 진짜 DB에서 돌아감)
       │
       ▼
  ⑥ 테스트 종료: 컨테이너 자동 폐기 (흔적 없음)
```

### 비유: 모형 자동차 vs 실제 도로 주행

H2로 하는 테스트는 **책상 위 모형 자동차를 굴려 보는 것**과 같다. 바퀴가 돌아가는 건 확인되지만,
실제 도로의 요철, 빗길, 경사에서 어떻게 될지는 알 수 없다. Testcontainers는 **실제 자동차를 실제 도로에서 시험 주행**하는 것이다.
운영에 나가기 전, 운영과 똑같은 노면(PostgreSQL)에서 미리 달려 보는 것이다.

```
   H2 (모형 자동차)                    Testcontainers (실제 도로 주행)
   ┌────────────────┐                  ┌────────────────┐
   │  빠르고 편함    │                  │  약간 느림      │
   │  하지만 가짜    │                  │  하지만 진짜    │
   │  노면          │                  │  운영과 동일    │
   └────────────────┘                  └────────────────┘
   → "모형은 잘 굴러갔는데             → "운영에서 될 것을
      진짜 도로에선 사고"                 미리 검증함"
```

> **주의**: Testcontainers는 Docker가 실행 중이어야 하고, 컨테이너 기동에 수 초가 걸린다.
> 그래서 **모든 테스트를 Testcontainers로 짜지는 않는다.** 순수 로직은 Mock 단위 테스트로,
> DB 방언·매핑이 중요한 부분만 Testcontainers 통합 테스트로 — 이것이 테스트 피라미드와 맞물리는 지점이다.
> `chapter08-testing` 모듈에서 이 조합을 실제로 확인할 수 있다.

## 6.5 Given-When-Then — 왜 테스트에도 구조가 필요한가

테스트를 아무렇게나 쓰면 나중에 읽기 어렵다. **Given-When-Then**은 테스트를 세 단계로 나누는 구조로,
행위 주도 개발(BDD)에서 왔다. (같은 개념을 AAA — Arrange-Act-Assert — 라고도 부른다.)

```
  Given (준비)  →  테스트에 필요한 상황·데이터를 세팅
  When  (실행)  →  검증하려는 그 동작 하나를 실행
  Then  (검증)  →  결과가 기대와 같은지 단언(assert)
```

### 예제: 쿠폰 할인 계산

```java
@Test
void 만원_이상_주문에_10퍼센트_쿠폰을_적용하면_할인가가_계산된다() {
    // given: 주문과 쿠폰을 준비한다
    Order order = new Order(20_000);              // 2만 원 주문
    Coupon coupon = Coupon.percentage(10);        // 10% 할인 쿠폰

    // when: 쿠폰을 적용한다 (검증 대상 동작)
    int finalPrice = order.applyCoupon(coupon);

    // then: 할인된 금액을 확인한다
    assertThat(finalPrice).isEqualTo(18_000);     // 2만 원 - 10% = 1만 8천 원
}

@Test
void 최소주문금액_미달이면_쿠폰이_적용되지_않는다() {
    // given
    Order order = new Order(5_000);               // 5천 원 주문
    Coupon coupon = Coupon.percentageWithMinimum(10, 10_000); // 1만 원 이상만 적용

    // when
    int finalPrice = order.applyCoupon(coupon);

    // then: 조건 미달이므로 원가 그대로
    assertThat(finalPrice).isEqualTo(5_000);
}
```

각 단계가 주석으로 명확히 구분되어, 처음 보는 사람도 **"무엇을 준비하고, 무엇을 실행해, 무엇을 기대하는지"**를
3초 만에 읽는다.

### 왜 이 구조가 좋은가

1. **일관성**: 모든 테스트가 같은 골격이라, 다른 사람의 테스트도 익숙하게 읽힌다.
2. **명확성**: "무엇을 검증하는가(When)"가 한 줄로 딱 보인다. When이 여러 줄이면 "테스트가 너무 많은 걸 한다"는 신호다.
3. **유지보수**: 실패했을 때 Then만 보면 "무엇이 기대와 달랐는지"를 바로 안다.

### 비유: 과학 실험 보고서

Given-When-Then은 **과학 실험 보고서의 형식**과 똑같다.

```
   과학 실험 보고서          →      테스트
   ───────────────────             ───────────────────
   실험 조건 (온도·시약)     →      Given (데이터·상태 준비)
   조작 (시약을 넣는다)      →      When  (동작 실행)
   결과 (색이 변한다)        →      Then  (결과 단언)
```

과학자가 "조건 → 조작 → 결과"를 일관되게 기록하기 때문에 누구나 그 실험을 재현하고 검증할 수 있다.
테스트도 똑같다. 이 구조를 지키면 테스트가 **재현 가능하고 검증 가능한 실험**이 된다.

> **팁**: When 단계는 이상적으로 **한 줄**이어야 한다. 실행 코드가 여러 줄로 늘어난다면,
> 그 테스트가 "한 가지"가 아니라 "여러 가지"를 검증하려 한다는 뜻이다. 테스트를 쪼개라.

## ⚠️ 흔한 오해와 함정

### 오해 1: "커버리지 100%면 좋은 테스트다"

**커버리지(coverage)는 "테스트가 코드의 몇 %를 거쳐 갔는가"일 뿐, "제대로 검증했는가"가 아니다.**
극단적으로, 단언(assert) 없이 메서드를 호출만 해도 커버리지는 올라간다.

```java
@Test
void 잘못된_테스트() {
    userService.createUser("test@test.com", "pw");  // 호출만 함
    // assert가 없다! → 커버리지는 올라가지만 아무것도 검증 안 함
}
```

이 테스트는 예외가 안 터지기만 하면 무조건 통과한다. 커버리지 숫자는 참고 지표일 뿐,
**의미 있는 단언이 있는가**가 진짜 척도다. "커버리지 100%"를 목표로 삼으면 이런 껍데기 테스트가 양산된다.

### 오해 2: "Mock을 많이 쓸수록 좋은 단위 테스트다"

Mock 남용은 **구현에 결합된 취약한 테스트**를 만든다. 모든 협력 객체를 Mock으로 바꾸면,
테스트가 "무엇을 하는가(동작)"가 아니라 "어떻게 하는가(구현)"를 검증하게 된다.
그러면 리팩토링으로 내부 구현만 바꿔도 — 동작은 그대로인데 — 테스트가 우수수 깨진다.
이는 6.1의 "리팩토링 안전망"과 정반대다. 값 객체(value object)나 단순한 협력자까지 Mock으로 만들 필요는 없다.

### 오해 3: "통합 테스트만 있으면 단위 테스트는 필요 없다"

통합 테스트는 느리고, 모든 분기(edge case)를 다 검증하기엔 비용이 크다.
"재고가 0, 1, 음수, 최대값일 때"의 모든 경우를 통합 테스트로 짜면 DB 왕복만으로도 테스트 시간이 폭발한다.
이런 촘촘한 경우의 수는 **빠른 단위 테스트의 몫**이다. 6.2의 테스트 피라미드가 뒤집히면(아이스크림 콘)
결국 아무도 안 돌리는 느린 테스트 스위트가 된다.

### 오해 4: "private 메서드도 직접 테스트해야 한다"

private 메서드는 그 자체가 목적이 아니라 **public 메서드의 내부 구현**이다.
private 메서드를 억지로 테스트하려고 접근 제어자를 열거나 리플렉션을 쓰면, 구현에 결합된 취약한 테스트가 된다.
**private 로직은 그것을 호출하는 public 메서드를 통해 간접적으로 검증**하면 된다.
만약 private 메서드가 너무 복잡해서 따로 테스트하고 싶어진다면, 그것은 **"별도 클래스로 분리하라"는 설계 신호**다.

### 오해 5: "테스트가 가끔 실패하는 건 원래 그런 거다"

**가끔 실패하는 테스트(flaky test)는 없는 것보다 나쁘다.** 팀이 "또 그거네" 하며 빨간불을 무시하게 만들고,
결국 진짜 버그가 낀 빨간불도 넘겨 버린다. 원인은 대개 테스트 간 상태 공유, 시간·순서 의존, 외부 시스템 의존이다.
FIRST 원칙의 I(독립)와 R(반복 가능)를 어긴 것이다. flaky 테스트는 발견 즉시 고치거나 격리해야 한다.

## 연습문제

### 문제 1
다음 테스트에는 문제가 있다. 무엇이 잘못됐는지 지적하고 고쳐라.

```java
@Test
void 회원가입_테스트() {
    userService.createUser("a@test.com", "pw");
}
```

<details>
<summary>힌트</summary>
FIRST 원칙 중 'S'(Self-validating)와, 오해 1을 떠올려 보라.
</details>

<details>
<summary>해설</summary>
<b>단언(assert)이 없다.</b> 이 테스트는 예외만 안 터지면 무조건 통과하므로, 실제로는 아무것도 검증하지 못한다.
커버리지만 올리는 껍데기 테스트다. 최소한 반환된 User의 이메일이 맞는지, 또는 Repository의 save가 호출됐는지를
단언해야 한다.

```java
@Test
void 회원가입하면_저장된_사용자를_반환한다() {
    // given
    given(userRepository.existsByEmail("a@test.com")).willReturn(false);
    given(userRepository.save(any())).willAnswer(inv -> inv.getArgument(0));
    // when
    User result = userService.createUser("a@test.com", "pw");
    // then
    assertThat(result.getEmail()).isEqualTo("a@test.com");
    verify(userRepository).save(any());
}
```
</details>

### 문제 2
`OrderService.placeOrder()`가 `PaymentGateway`(외부 결제 API)와 `OrderRepository`에 의존한다.
이 메서드의 **단위 테스트**를 짤 때, 두 의존성을 각각 어떻게 처리해야 하며 그 이유는 무엇인가?

<details>
<summary>힌트</summary>
6.3의 "진짜 의존성을 쓰면 생기는 문제"를 결제 API에 대입해 보라.
</details>

<details>
<summary>해설</summary>
둘 다 <b>Mock으로 대체</b>한다. 진짜 <code>PaymentGateway</code>를 쓰면 <b>실제 결제가 발생</b>하고(부작용, 비용),
외부 API가 켜져 있어야 하며 느리다. 진짜 <code>OrderRepository</code>는 DB를 필요로 한다.
단위 테스트의 목적은 "placeOrder의 <i>로직</i>이 올바른가"이므로, 협력 객체는 Mock으로 통제하고
<code>verify</code>로 "결제가 정확히 한 번 호출됐는가", "주문이 저장됐는가"를 검증하면 된다.
(실제 결제 연동이 되는지는 별도의 통합/계약 테스트의 몫이다.)
</details>

### 문제 3
어떤 Repository 테스트가 H2에서는 통과하는데 운영(PostgreSQL)에 배포하면 실패한다.
쿼리에 `ON CONFLICT ... DO UPDATE`(upsert) 구문이 들어 있다. 왜 이런 일이 생겼고, 어떻게 해결하는가?

<details>
<summary>힌트</summary>
6.4의 "H2 내장 DB의 함정"과 SQL 방언 차이를 보라.
</details>

<details>
<summary>해설</summary>
<code>ON CONFLICT</code>는 <b>PostgreSQL 전용 문법</b>이다. H2는 이를 지원하지 않거나 다르게 처리하므로,
H2 기반 테스트는 이 구문을 제대로 검증하지 못한다(또는 H2용으로 우회된 쿼리를 태운다).
해결책은 <b>Testcontainers로 실제 PostgreSQL을 띄워</b> 운영과 동일한 방언에서 테스트하는 것이다.
<code>@Testcontainers</code> + <code>@Container PostgreSQLContainer</code> + <code>@DynamicPropertySource</code>를 사용한다.
</details>

### 문제 4
다음 테스트를 Given-When-Then 구조로 재작성하고, When 단계가 한 줄이 아니라면 왜 문제인지 설명하라.

```java
@Test
void test() {
    Order order = new Order(20000);
    Coupon coupon = Coupon.percentage(10);
    int price = order.applyCoupon(coupon);
    assertThat(price).isEqualTo(18000);
}
```

<details>
<summary>힌트</summary>
6.5의 세 단계와 "When은 한 줄" 팁을 적용하라.
</details>

<details>
<summary>해설</summary>

```java
@Test
void 만원이상_주문에_10퍼센트_쿠폰을_적용하면_할인된다() {
    // given
    Order order = new Order(20_000);
    Coupon coupon = Coupon.percentage(10);
    // when
    int price = order.applyCoupon(coupon);
    // then
    assertThat(price).isEqualTo(18_000);
}
```

When은 <code>order.applyCoupon(coupon)</code> 한 줄이므로 좋다. 만약 When에 여러 동작이 들어 있다면,
그 테스트가 "한 가지 행위"가 아니라 여러 가지를 한꺼번에 검증한다는 신호이므로 테스트를 쪼개야 한다.
그래야 실패했을 때 "정확히 어느 동작이 틀렸는지"를 알 수 있다.
</details>

### 문제 5
테스트 피라미드가 "뒤집힌" 프로젝트(통합·E2E만 많고 단위 테스트는 거의 없음)에서
개발자들이 겪게 될 구체적인 고통 세 가지를 예상해 보라.

<details>
<summary>힌트</summary>
6.2의 "단위 테스트가 많아야 하는 세 가지 이유"를 뒤집어 생각하라.
</details>

<details>
<summary>해설</summary>
1. <b>느린 피드백</b>: 통합·E2E는 컨텍스트 로딩·DB·네트워크로 느려서 전체 스위트가 수 분~수십 분 걸린다.
   자주 못 돌리니 버그를 늦게 발견한다.<br>
2. <b>flaky(가끔 실패)</b>: 위 계층일수록 타이밍·네트워크·외부 상태에 의존해 무작위로 실패한다.
   빨간불을 신뢰하지 않게 되고, 진짜 버그도 무시하게 된다.<br>
3. <b>원인 파악 곤란</b>: E2E가 깨지면 "어딘가 잘못됐다"까지만 알 뿐, 어느 클래스·어느 로직이 원인인지
   디버깅에 오래 걸린다. (이 상태를 '아이스크림 콘 안티패턴'이라 부른다.)
</details>

## 요약

- **테스트 없는 코드는 레거시다.** 테스트의 목적은 문법이 아니라 **마음 놓고 코드를 바꿀 수 있는 자유**다.
- 테스트의 세 가지 가치: **리팩토링 안전망**, **살아 있는 문서**, **빠른 피드백**. (건물 안전검사 비유)
- 좋은 테스트는 **FIRST**(Fast·Independent·Repeatable·Self-validating·Timely) 원칙을 지킨다.
- **단위 테스트**는 클래스 하나를 Mock으로 고립시켜 빠르게 로직을 검증한다
  (`@ExtendWith(MockitoExtension.class)`, `@Mock`, `@InjectMocks`).
- **통합 테스트**는 여러 계층 + 실제 DB를 연결해 배선·트랜잭션·SQL을 검증한다
  (`@SpringBootTest`, `@AutoConfigureMockMvc`, `@Transactional`, `MockMvc`).
- **테스트 피라미드**: 단위(많이) > 통합(중간) > E2E(적게). 아래가 넓어야 빠르고 안정적이며 원인을 잘 짚는다.
  뒤집히면 아이스크림 콘 안티패턴. 그 중간에 `@WebMvcTest`, `@DataJpaTest` 같은 slice 테스트가 있다.
- **Mock**은 느리고·비싸고·부작용 있는 협력 객체(DB, 이메일, 결제)를 대체하는 안전한 대역(스턴트맨)이다.
  `when`으로 응답을 정하고 `verify`로 호출을 검증한다. 테스트 더블에는 Dummy·Stub·Mock·Spy·Fake가 있다.
- **Testcontainers**는 실제 DB를 Docker로 띄워, H2의 "테스트는 통과하는데 운영에서 터지는" 방언 함정을 없앤다
  (`@Testcontainers`, `@Container`, `@DynamicPropertySource`). 모형 자동차가 아니라 실제 도로 주행.
- **Given-When-Then**은 테스트를 "준비 → 실행 → 검증" 3단계로 구조화한다. 과학 실험 보고서처럼
  일관되고 재현 가능하게 만든다. When은 한 줄이 이상적이다.
- **함정 주의**: 커버리지 100% ≠ 좋은 테스트, Mock 남용, 통합 테스트만으로 충분하다는 착각,
  private 메서드 직접 테스트, flaky 테스트 방치.

실습은 `chapter08-testing` 모듈에서 진행한다. Mock 단위 테스트, MockMvc 통합 테스트,
Testcontainers 테스트가 한 프로젝트 안에 어떻게 공존하는지를 직접 돌려 보라.
