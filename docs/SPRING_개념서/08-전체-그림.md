# Chapter 8: 전체 그림 - 모든 것이 어떻게 연결되는가?

> "나무를 보다가 숲을 잊지 말자. 하지만 숲을 이야기하려면 나무 하나하나를 먼저 알아야 한다."

여기까지 오느라 수고 많았다. 우리는 지금까지 일곱 개의 장에 걸쳐 Spring의 세계를 하나씩 뜯어보았다. Spring이 어떤 고통에서 태어났는지(1장), IoC와 DI와 AOP라는 심장이 어떻게 뛰는지(2장), 웹 요청이 어떤 계층을 지나는지(3장), 데이터가 JPA를 통해 어떻게 저장되는지(4장), 보안은 어떻게 지켜지는지(5장), 코드를 어떻게 검증하는지(6장), 그리고 이 모든 것을 컨테이너에 담아 어떻게 배포하는지(7장)까지.

그런데 지금까지의 여정에는 한 가지 위험이 있었다. **나무 하나하나에 집중하느라 숲 전체를 놓칠 수 있다는 것**이다. IoC를 배울 때는 IoC만, JPA를 배울 때는 JPA만 봤다. 하지만 실제 애플리케이션에서 이것들은 따로 노는 부품이 아니다. 하나의 요청이 들어오면 이 모든 개념이 **동시에, 유기적으로, 하나의 흐름 속에서** 작동한다.

이 마지막 장의 목표는 그래서 명확하다. **흩어진 나무들을 하나의 숲으로 다시 조립하는 것.** 우리는 하나의 게시판 프로젝트를 예로 들어, 요청 하나가 시스템 전체를 관통하는 여정을 따라가 볼 것이다. 그 여정 속에서 지금까지 배운 개념들이 각각 어디에 놓여 어떤 역할을 하는지 보게 될 것이다. 그리고 마지막으로, 이 책을 덮은 뒤 "좋은 개발자"가 되기 위해 무엇을 어떻게 해야 하는지를 이야기하며 여정을 마무리하겠다.

## 8.1 게시판 프로젝트로 보는 전체 아키텍처

### 개념: 요청 하나가 지나가는 길

전체 그림을 먼저 보자. 게시판 서비스가 Docker Compose 위에서 돌고 있고, 하나의 HTTP 요청이 시스템을 어떻게 관통하는지를 나타낸 것이다.

```
   ┌──────────────────────── Docker Compose ───────────────────────────┐
   │                                                                    │
   │   HTTP 요청                                                        │
   │      │                                                             │
   │      ▼                                                             │
   │  ┌────────┐                                                        │
   │  │ Nginx  │  리버스 프록시 (요청 분배, 정적 파일, TLS 종료)        │
   │  └────┬───┘                                                        │
   │       │                                                            │
   │       ▼                                                            │
   │  ┌──────────────────── Spring Boot ──────────────────────────┐    │
   │  │                                                            │    │
   │  │  ┌────────────────────────────────────────────────┐       │    │
   │  │  │  Security Filter Chain                          │       │    │
   │  │  │  (인증/인가 필터: 토큰 검증, 권한 확인)          │       │    │
   │  │  └───────────────────┬────────────────────────────┘       │    │
   │  │                      ▼                                     │    │
   │  │  ┌────────────┐  ┌────────────┐  ┌──────────────┐          │    │
   │  │  │ Controller │─▶│  Service   │─▶│  Repository  │          │    │
   │  │  │ (요청 처리)│  │@Transactional│ │ JpaRepository│          │    │
   │  │  │ @Valid     │  │(비즈니스)   │  │ (데이터 접근)│          │    │
   │  │  └────────────┘  └────────────┘  └──────┬───────┘          │    │
   │  └───────────────────────────────────────┼──────────────────┘    │
   │                                           │                        │
   │                                           ▼                        │
   │                                    ┌────────────┐                  │
   │                                    │ PostgreSQL │  데이터베이스     │
   │                                    └────────────┘                  │
   │                                                                    │
   └────────────────────────────────────────────────────────────────────┘
```

이 다이어그램은 앞선 모든 장의 결정판이다. Docker Compose(7장)라는 울타리 안에서, Nginx가 요청을 받아 Spring Boot로 넘기고, Spring 안에서는 Security Filter Chain(5장)이 먼저 관문을 지키고, 통과한 요청은 Controller → Service → Repository(3장)의 계층을 거쳐 PostgreSQL(4장)에 도달한다. 그리고 이 모든 객체들은 IoC 컨테이너(2장)가 조립해 준 것이다.

이제 이 정지된 그림에 생명을 불어넣어 보자. **세 개의 실제 요청**을 따라가며 각 단계에서 무슨 일이 벌어지는지 본다.

### 원리: 요청의 여정 ① 회원가입 — `POST /api/auth/signup`

첫 번째 여정은 회원가입이다. 아직 로그인하지 않은 사용자가 가입을 요청한다.

```
   회원가입 여정: POST /api/auth/signup
   ┌─────────────────────────────────────────────────────────────┐
   │ 1. Nginx          요청을 받아 Spring Boot로 전달              │
   │       │                                                       │
   │       ▼                                                       │
   │ 2. Security       /api/auth/signup 은 permitAll              │
   │    Filter Chain   → 인증 없이 통과시킴                        │
   │       │                                                       │
   │       ▼                                                       │
   │ 3. AuthController @Valid 로 요청 본문(이메일/비번) 검증       │
   │       │                                                       │
   │       ▼                                                       │
   │ 4. AuthService    createUser():                              │
   │       │           - 이메일 중복 확인                          │
   │       │           - BCrypt 로 비밀번호 암호화                 │
   │       │           - 저장 요청                                 │
   │       ▼                                                       │
   │ 5. UserRepository save() → PostgreSQL 에 INSERT              │
   │       │                                                       │
   │       ▼                                                       │
   │ 6. 응답            201 Created                                │
   └─────────────────────────────────────────────────────────────┘
```

핵심은 **누구나 가입은 할 수 있어야 한다**는 점이다. 그래서 Security 설정에서 이 경로는 `permitAll`로 열려 있다(5장). Controller는 들어온 데이터가 올바른지 `@Valid`로 검증하고(3장), Service는 비즈니스 규칙(이메일 중복 확인, 비밀번호 암호화)을 처리하며, Repository가 실제 저장을 담당한다.

```java
@PostMapping("/signup")
public ResponseEntity<Void> signup(@Valid @RequestBody SignupRequest request) {
    authService.createUser(request);          // 검증된 요청을 서비스로
    return ResponseEntity.status(201).build();
}
```

```java
@Transactional
public void createUser(SignupRequest request) {
    if (userRepository.existsByEmail(request.email())) {   // 중복 확인
        throw new DuplicateEmailException();
    }
    String encoded = passwordEncoder.encode(request.password());  // BCrypt 암호화
    userRepository.save(new User(request.email(), encoded));       // 저장
}
```

여기서 비밀번호를 **평문이 아니라 BCrypt로 단방향 암호화**해 저장한다는 점이 보안의 핵심이다(5장). DB가 유출되어도 원래 비밀번호는 복원할 수 없다.

### 원리: 요청의 여정 ② 로그인 — `POST /api/auth/login`

두 번째 여정은 로그인이다. 방금 가입한 사용자가 이제 신원을 증명하고, 그 대가로 **토큰**을 받는다.

```
   로그인 여정: POST /api/auth/login
   ┌─────────────────────────────────────────────────────────────┐
   │ 1. Security       /api/auth/login 도 permitAll → 통과        │
   │       ▼                                                       │
   │ 2. AuthController  @Valid 로 이메일/비번 검증                 │
   │       ▼                                                       │
   │ 3. AuthService    login():                                   │
   │       │           - 이메일로 사용자 조회                      │
   │       │           - BCrypt 로 비밀번호 '검증'(matches)         │
   │       │           - 성공 시 JWT 토큰 생성                     │
   │       ▼                                                       │
   │ 4. 응답            200 OK + { "token": "eyJ..." }             │
   └─────────────────────────────────────────────────────────────┘
```

로그인 시점에는 비밀번호를 다시 암호화하는 게 아니라, 입력한 비밀번호를 저장된 해시와 **비교(matches)** 한다. BCrypt는 같은 비밀번호라도 매번 다른 해시를 만들지만, 검증은 가능하도록 설계되어 있다(5장). 검증에 성공하면 **JWT 토큰**을 발급한다. 이 토큰은 "나는 인증된 사용자다"라는 서명된 증표로, 이후 요청마다 함께 보내게 된다.

```java
public LoginResponse login(LoginRequest request) {
    User user = userRepository.findByEmail(request.email())
            .orElseThrow(() -> new InvalidCredentialsException());
    if (!passwordEncoder.matches(request.password(), user.getPassword())) {
        throw new InvalidCredentialsException();   // 비밀번호 불일치
    }
    String token = jwtProvider.createToken(user);  // JWT 발급
    return new LoginResponse(token);
}
```

### 원리: 요청의 여정 ③ 게시글 작성 — `POST /api/posts`

마지막 여정은 게시글 작성이다. 이번엔 **인증이 반드시 필요한** 요청이다. 앞서 받은 토큰을 헤더에 실어 보낸다.

```
   게시글 작성 여정: POST /api/posts   (Authorization: Bearer eyJ...)
   ┌─────────────────────────────────────────────────────────────┐
   │ 1. JwtAuthFilter    - 헤더에서 토큰 파싱                      │
   │       │             - 서명 검증(위조 여부 확인)               │
   │       │             - SecurityContext 에 인증정보 저장        │
   │       ▼                                                       │
   │ 2. Authorization    이 경로는 authenticated 필요              │
   │    Filter           → 인증된 사용자만 통과                    │
   │       ▼                                                       │
   │ 3. PostController   @Valid 로 본문 검증                       │
   │       │             @AuthenticationPrincipal 로 로그인 사용자 │
   │       ▼                                                       │
   │ 4. PostService      @Transactional:                          │
   │       │             - 작성자(User) 조회                       │
   │       │             - Post 엔티티 생성                        │
   │       │             - 저장                                    │
   │       ▼                                                       │
   │ 5. PostRepository   save() → PostgreSQL 에 INSERT            │
   │       ▼                                                       │
   │ 6. 응답             201 Created                               │
   └─────────────────────────────────────────────────────────────┘
```

여기서 Security Filter Chain의 진가가 드러난다(5장). `JwtAuthFilter`는 토큰을 파싱하고 **서명을 검증**해 위조되지 않았음을 확인한 뒤, 인증 정보를 `SecurityContext`에 담는다. 그다음 인가 필터가 "이 경로는 인증된 사용자만"이라는 규칙을 적용한다. Controller는 `@AuthenticationPrincipal`로 현재 로그인한 사용자가 누구인지 꺼내 쓴다. Service는 `@Transactional`(2장, 4장) 안에서 작성자를 조회하고 게시글을 만들어 저장한다. 트랜잭션 덕분에, 저장 도중 문제가 생기면 전체가 깔끔하게 롤백된다.

```java
@PostMapping
public ResponseEntity<Void> create(
        @Valid @RequestBody PostRequest request,
        @AuthenticationPrincipal UserPrincipal principal) {   // 현재 사용자
    postService.create(request, principal.getUserId());
    return ResponseEntity.status(201).build();
}
```

```java
@Transactional
public void create(PostRequest request, Long userId) {
    User author = userRepository.findById(userId)
            .orElseThrow(() -> new UserNotFoundException());
    Post post = new Post(request.title(), request.content(), author);
    postRepository.save(post);
}
```

### 정리: 지금까지 배운 모든 것의 조합

이 세 여정 안에, 이 책의 모든 개념이 녹아 있다. 각각이 어느 장에서 왔는지 짚어보자.

```
   요청 하나에 담긴 여덟 개의 개념
   ┌──────────────────┬────────────────────────────────────┬────────┐
   │ 개념             │ 요청 여정에서의 역할                │ 배운 곳│
   ├──────────────────┼────────────────────────────────────┼────────┤
   │ IoC / DI         │ 모든 객체를 컨테이너가 조립·주입     │ 2장    │
   │ 레이어드 아키텍처│ Controller→Service→Repository 계층   │ 3장    │
   │ DTO              │ 요청/응답 데이터의 전용 그릇         │ 3장    │
   │ Validation       │ @Valid 로 입력 데이터 검증           │ 3장    │
   │ JPA / 트랜잭션   │ Repository·@Transactional 로 영속화  │ 4장    │
   │ Security         │ Filter Chain·JWT·BCrypt 로 보호      │ 5장    │
   │ 테스트           │ 각 계층을 단위/통합 테스트로 검증    │ 6장    │
   │ Docker           │ 전체를 컨테이너로 담아 배포          │ 7장    │
   └──────────────────┴────────────────────────────────────┴────────┘
```

- **IoC / DI** — [2장: IoC/DI/프록시/AOP/멀티스레드](02-ioc-di-프록시-aop-멀티스레드.md). Controller·Service·Repository가 서로를 `new`로 만들지 않고 컨테이너가 주입해 준다. `@Transactional`이 작동하는 것도 프록시(AOP) 덕분이다.
- **레이어드 아키텍처 · DTO · Validation** — [3장: 웹 애플리케이션 구조](03-웹-애플리케이션-구조.md). 요청 처리·비즈니스·데이터 접근이 계층으로 분리되고, DTO가 데이터를 나르며, `@Valid`가 입구를 지킨다.
- **JPA · 트랜잭션** — [4장: 데이터베이스와 JPA](04-데이터베이스와-jpa.md). `JpaRepository`가 SQL 없이 저장을 처리하고, `@Transactional`이 원자성을 보장한다.
- **Security** — [5장: 보안 - Spring Security](05-보안-spring-security.md). Filter Chain·JWT·BCrypt가 인증과 인가를 책임진다.
- **테스트** — [6장: 테스트](06-테스트.md). 이 모든 계층이 의도대로 동작하는지 검증한다.
- **Docker** — [7장: Docker](07-docker.md). 완성된 애플리케이션을 어디서든 동일하게 실행되도록 담는다.

그리고 이 전체 흐름이 "왜 이렇게 생겼는가"의 출발점은 [1장: Spring이 해결하는 문제](01-spring이-해결하는-문제.md)에 있다. Servlet 하나가 모든 걸 떠안던 지옥에서 시작해, 지금 우리가 본 이 깔끔한 계층 구조에 도달한 것이다.

## 8.2 실무에서의 Spring Boot

### 개념: 이 책은 지도의 시작점일 뿐이다

여기까지 배운 것은 Spring Boot 애플리케이션의 **뼈대**다. 실무에서는 이 뼈대 위에 훨씬 많은 것이 붙는다. 그것들을 다 다루려면 책 여러 권이 필요하지만, "이런 세계가 더 있다"는 지도만은 그려 두는 것이 좋다. 지도가 있어야 다음에 어디로 갈지 정할 수 있으니까.

```
   실무에서 추가로 마주칠 세계
   ┌───────────────────────────────────────────────────────────┐
   │ 캐싱          Redis — 자주 읽는 데이터를 메모리에 저장       │
   │ 메시지 큐     Kafka / RabbitMQ — 비동기·이벤트 기반 통신     │
   │ MSA           Spring Cloud — 서비스를 잘게 쪼개 독립 배포     │
   │ CI/CD         GitHub Actions / ArgoCD — 자동 빌드·배포        │
   │ 모니터링      Prometheus / Grafana / ELK — 관측과 로그 분석   │
   │ 클라우드      AWS / Kubernetes — 확장 가능한 인프라 운영      │
   └───────────────────────────────────────────────────────────┘
```

- **캐싱(Redis)**: 매번 DB에 묻는 대신 자주 쓰는 데이터를 메모리에 두어 응답을 빠르게 한다.
- **메시지 큐(Kafka, RabbitMQ)**: 서비스끼리 직접 호출하는 대신 메시지를 주고받아 느슨하게 연결하고, 트래픽 폭주를 완충한다.
- **MSA(Spring Cloud)**: 거대한 하나의 애플리케이션(모놀리식)을 작은 서비스 여럿으로 쪼개 독립적으로 개발·배포한다.
- **CI/CD(GitHub Actions, ArgoCD)**: 코드를 push하면 자동으로 테스트·빌드·배포까지 이어지도록 파이프라인을 만든다.
- **모니터링(Prometheus, Grafana, ELK)**: 시스템이 지금 건강한지 지표로 관측하고, 로그를 모아 분석한다.
- **클라우드(AWS, Kubernetes)**: 트래픽에 따라 서버를 자동으로 늘리고 줄이며, 컨테이너를 대규모로 운영한다.

### 원리: 학습 로드맵 — 어디부터 어디까지

이 모든 것을 한꺼번에 공부하려 하면 압도당한다. 순서가 중요하다. 대략 이런 단계로 나아가면 좋다.

```
   학습 로드맵
   ┌──────────────────────────────────────────────────────────┐
   │ Phase 1  기초                                              │
   │   Java · Spring Boot · JPA · REST API · 테스트             │
   │   (← 이 책이 다룬 범위)                                    │
   │        │                                                   │
   │        ▼                                                   │
   │ Phase 2  심화                                              │
   │   Security 심화 · 트랜잭션 전파 · JPA 성능(N+1) · 캐싱      │
   │        │                                                   │
   │        ▼                                                   │
   │ Phase 3  아키텍처                                          │
   │   MSA · 메시지 큐 · 이벤트 기반 · 도메인 주도 설계(DDD)     │
   │        │                                                   │
   │        ▼                                                   │
   │ Phase 4  전문성                                            │
   │   대규모 트래픽 · 클라우드/K8s · 모니터링 · 성능 튜닝       │
   └──────────────────────────────────────────────────────────┘
```

- **Phase 1(기초)**: 이 책이 다룬 영역이다. 여기가 흔들리면 위 단계가 모래성이 된다. 튼튼히 다져라.
- **Phase 2(심화)**: 기초 개념을 더 깊이 판다. 트랜잭션 전파 옵션, JPA의 N+1 문제 같은 "실무에서 아프게 배우는" 것들이다.
- **Phase 3(아키텍처)**: 개별 애플리케이션을 넘어 여러 서비스를 어떻게 나누고 통신시킬지를 다룬다.
- **Phase 4(전문성)**: 대규모 트래픽과 운영의 세계다. 여기서는 코드보다 시스템 전체를 보는 눈이 중요해진다.

Phase를 건너뛰지 말라. 3장에서 왜 계층을 나누는지 이해하지 못한 사람은, Phase 3에서 왜 서비스를 나누는지도 이해하지 못한다. **원리는 규모만 바뀔 뿐 반복된다.**

## 마무리: 좋은 개발자가 되려면

이 책을 시작하며 우리는 "어떻게"가 아니라 "왜"를 묻자고 했다. 마지막으로, 이 책을 덮은 뒤 좋은 개발자로 성장하기 위한 몇 가지 태도를 나누고 싶다.

### 코드를 많이 읽어라

글을 잘 쓰려면 좋은 글을 많이 읽어야 하듯, 코드를 잘 짜려면 좋은 코드를 많이 읽어야 한다.

- **GitHub의 좋은 프로젝트를 읽어라.** 스타가 많은 오픈소스 프로젝트는 수많은 사람의 검증을 거친 코드다. 구조가 어떻게 잡혀 있는지, 이름을 어떻게 짓는지 눈으로 익혀라.
- **Spring 소스코드를 직접 들춰봐라.** 예를 들어 `@Transactional`이 어떻게 작동하는지 궁금하다면 `TransactionInterceptor`를 열어 보라. 이 클래스가 바로 2장에서 배운 **프록시의 심장**이다. "마법"이라 느꼈던 것이 실은 명확한 코드였음을 확인하는 순간, 그 개념은 영원히 내 것이 된다.
- **코드 리뷰에 적극 참여하라.** 남의 코드를 리뷰하며 "왜 이렇게 짰지?"를 묻고, 내 코드가 리뷰받으며 "왜 이렇게 짰냐"는 질문에 답하는 과정에서 가장 빠르게 성장한다.

### "왜?"를 계속 물어라

세상에는 두 종류의 개발자가 있다.

```
   "어떻게" 아는 개발자        vs        "왜" 아는 개발자
   ┌──────────────────┐                ┌──────────────────┐
   │ @Transactional 을 │                │ 왜 프록시로 동작  │
   │ '붙이면' 된다     │                │ 하는지 '이해한다' │
   │                   │                │                   │
   │ 문제가 생기면      │                │ 문제가 생기면      │
   │ 막힌다            │                │ 원인을 추적한다   │
   └──────────────────┘                └──────────────────┘
```

"어떻게"만 아는 사람은 정해진 상황에서만 작동한다. 조금만 상황이 달라지거나 버그가 나면 손을 놓는다. 반면 "왜"를 아는 사람은 처음 보는 상황에서도 원리로부터 답을 유추해 낸다. 항상 이렇게 물어라. **"왜 이 애노테이션이 필요하지? 이걸 빼면 무슨 일이 벌어지지? 이 설계는 어떤 문제를 풀려고 나온 거지?"**

### 완벽하지 않아도 괜찮다

성장에는 시간이 걸린다. 지금 모든 것을 이해하지 못해도 괜찮다.

```
   개발자의 성장 곡선
   1년차   문법과 도구 사용에 급급하다. 검색과 복붙으로 버틴다.
   3년차   구조가 보이기 시작한다. "왜"를 묻기 시작한다.
   5년차   설계를 고민한다. 트레이드오프를 저울질한다.
   7년차   시스템 전체와 팀·비즈니스를 함께 본다.
```

누구나 1년차를 거친다. 지금 막막한 것은 당신이 부족해서가 아니라, 성장의 자연스러운 한 지점에 있기 때문이다. 중요한 건 속도가 아니라 방향이다. **매일 조금씩, "왜"를 물으며 나아가면 된다.**

### 추천 학습 리소스

- **서적**
  - 『토비의 스프링』 — Spring의 원리를 가장 깊이 파고드는 국내 명저.
  - 『자바 ORM 표준 JPA 프로그래밍』(김영한) — JPA를 제대로 이해하려면 반드시.
  - 『Clean Code』 — 읽기 좋은 코드란 무엇인가.
  - 『객체지향의 사실과 오해』 — 객체지향의 본질을 다시 생각하게 하는 책.
- **온라인**
  - Spring 공식 문서 — 가장 정확하고 최신인 1차 자료.
  - Baeldung — 구체적인 예제와 설명이 풍부한 영문 사이트.
  - 인프런 — 체계적인 한국어 강의.
- **실습**
  - 작은 프로젝트를 처음부터 끝까지 만들어 보라(이 책의 게시판처럼).
  - 이미 만든 코드를 리팩토링해 보라. 더 나은 구조를 고민하는 것 자체가 공부다.
  - 오픈소스에 작은 기여라도 시도해 보라.

**코드는 변하지만, 원리는 오래간다.**

## ⚠️ 흔한 오해와 함정

**오해 1: "이 책을 다 읽었으니 실무 준비가 끝났다"**
이 책은 지도의 시작점일 뿐이다. Phase 1(기초)을 다뤘을 뿐, Phase 2~4의 세계가 남아 있다. 무엇보다 실무는 "아는 것"이 아니라 "직접 부딪히며 문제를 푸는 것"으로 배운다. 이 책이 준 것은 지식이 아니라 **더 깊이 배울 수 있는 기반과 방향 감각**이다.

**오해 2: "새로운 도구와 유행을 계속 쫓아야 한다"**
기술의 유행은 빠르게 바뀐다. 매년 새로운 프레임워크, 라이브러리, 아키텍처가 등장한다. 유행을 다 쫓다 보면 어느 것도 깊이 있게 알지 못한다. 도구는 결국 문제를 푸는 수단이다. **"이 도구가 어떤 문제를 어떤 대가로 푸는가"라는 원리를 이해하면**, 새 도구가 나와도 금방 그 본질을 파악할 수 있다. 유행이 아니라 원리를 쫓아라.

**함정: "왜인지 모른 채 남의 코드를 베낀다"**
Stack Overflow나 블로그의 코드를 이해 없이 복사-붙여넣기 하는 것은 가장 위험한 습관이다. 당장은 돌아가는 것처럼 보여도, 조금만 상황이 달라지면 왜 안 되는지 알 수 없고, 심지어 보안 구멍이나 성능 문제를 그대로 들여오기도 한다. **베끼기 전에 반드시 "이 코드가 왜 이렇게 생겼는지"를 이해하라.** 이해한 코드만이 진짜 내 것이다.

## 연습문제

이 책 전체를 관통하는 종합 문제들이다. 여러 장의 개념을 엮어 답해 보자.

**문제 1.** 게시글 작성 요청(`POST /api/posts`) 하나가 시스템을 관통하는 여정을, Security Filter Chain부터 PostgreSQL까지 순서대로 설명하라. 각 단계에서 어느 장의 개념이 작동하는지도 밝혀라.

<details>
<summary>해설</summary>

① `JwtAuthFilter`가 헤더의 토큰을 파싱하고 서명을 검증해 `SecurityContext`에 인증정보 저장(5장 Security). ② 인가 필터가 "authenticated 필요" 규칙 적용, 인증된 사용자만 통과(5장). ③ `PostController`가 `@Valid`로 본문 검증하고 `@AuthenticationPrincipal`로 현재 사용자를 꺼냄(3장 웹 구조·Validation). ④ `PostService`가 `@Transactional` 안에서 작성자 조회 → Post 생성 → 저장(2장 프록시/트랜잭션, 4장 JPA). ⑤ `PostRepository.save()`가 PostgreSQL에 INSERT(4장). ⑥ 201 Created 응답. 이 모든 객체들의 조립은 IoC 컨테이너가 담당한다(2장).
</details>

**문제 2.** 회원가입 시 비밀번호를 BCrypt로 "암호화"하고, 로그인 시에는 "검증(matches)"한다. 이 둘의 차이는 무엇이며, 왜 저장할 때 평문으로 두면 안 되는가? (5장 + 전체)

<details>
<summary>해설</summary>

회원가입 때는 `encode`로 비밀번호를 단방향 해시로 변환해 저장한다(원본으로 되돌릴 수 없음). 로그인 때는 입력한 평문을 저장된 해시와 `matches`로 비교한다. BCrypt는 같은 비밀번호라도 매번 다른 해시를 만들지만(salt 때문), 비교는 가능하도록 설계돼 있다. 평문으로 저장하면 DB가 유출되는 순간 모든 사용자의 비밀번호가 그대로 노출되므로, 단방향 암호화로 저장해 유출되어도 원본을 복원할 수 없게 해야 한다.
</details>

**문제 3.** `@Transactional`이 붙은 서비스 메서드가 중간에 예외로 실패하면 이미 실행된 저장 작업이 롤백된다. 이 동작이 가능한 이유를 2장(프록시/AOP)과 4장(트랜잭션)의 개념을 엮어 설명하라.

<details>
<summary>해설</summary>

Spring은 `@Transactional`이 붙은 빈을 그대로 쓰지 않고, 프록시로 감싸 등록한다(2장 AOP/프록시). 실제 메서드 호출 전에 프록시가 트랜잭션을 시작하고, 정상 종료되면 커밋, 예외가 발생하면 롤백한다. 이 트랜잭션 경계 덕분에 여러 DB 작업이 하나의 단위(원자성)로 묶여, 중간에 실패하면 전체가 없던 일처럼 되돌아간다(4장). 즉, 프록시(2장)가 트랜잭션(4장)의 시작·커밋·롤백을 대신 관리하는 것이다. `TransactionInterceptor`가 그 심장이다.
</details>

**문제 4.** "내 컴퓨터에서는 되는데 서버에서는 안 된다"는 문제와, 이 책의 게시판 프로젝트가 Docker Compose로 배포되는 구조를 연결하라. Compose가 이 문제를 어떻게 막는가? (7장 + 전체)

<details>
<summary>해설</summary>

환경 차이(OS·Java·PostgreSQL 버전 등)가 원인이다. Docker는 앱과 실행 환경을 이미지에 통째로 담아 어디서든 동일하게 실행되게 하고, Docker Compose는 Spring Boot·PostgreSQL·Redis·Nginx 등 여러 컨테이너를 하나의 `docker-compose.yml`로 정의해 버전을 고정(예: `postgres:16`)하고 서비스 이름 기반 통신·`depends_on`+`healthcheck`로 실행 순서와 준비 상태까지 통제한다. 따라서 개발·테스트·운영 어디서든 동일한 환경이 재현되어 "내 컴퓨터에서는 되는데" 문제가 사라진다.
</details>

**문제 5.** "어떻게 아는 개발자"와 "왜 아는 개발자"의 차이를 `@Transactional`을 예로 들어 설명하고, "왜"를 아는 것이 실무에서 왜 결정적인지 논하라.

<details>
<summary>해설</summary>

"어떻게" 아는 개발자는 `@Transactional`을 붙이면 롤백된다는 사실만 안다. 이 애노테이션이 프록시로 동작한다는 사실을 모르면, 같은 클래스 내부 메서드 호출에서는 트랜잭션이 적용되지 않는(self-invocation) 함정에 빠졌을 때 원인을 찾지 못한다. "왜" 아는 개발자는 프록시 원리를 이해하므로 이런 예외 상황에서도 원인을 추적하고 해결한다. 실무의 대부분은 "정상 상황"이 아니라 "예외 상황을 디버깅하는 일"이므로, 원리를 아는 사람만이 처음 보는 문제 앞에서 답을 유추해 낼 수 있다.
</details>

## 요약

이 한 페이지가 책 전체의 요약이다.

- **1장 — 왜 Spring인가**: 모든 기술은 문제를 풀려고 태어난다. Servlet 지옥 → EJB 지옥 → Spring(XML 지옥) → Spring Boot(관례 > 설정)의 흐름이 그 증거다.
- **2장 — Spring의 심장**: IoC/DI로 객체 조립을 컨테이너에 맡기고, 프록시/AOP로 트랜잭션 같은 공통 관심사를 코드 밖에서 처리한다. 멀티스레드 환경에서 빈은 상태를 갖지 않아야 한다.
- **3장 — 웹 구조**: 요청은 Controller(요청 처리) → Service(비즈니스) → Repository(데이터 접근)의 계층을 지나며, DTO가 데이터를 나르고 `@Valid`가 입구를 지킨다.
- **4장 — 데이터와 JPA**: 객체와 관계형 DB의 간극을 JPA가 메운다. `@Transactional`이 여러 작업을 원자적 단위로 묶는다.
- **5장 — 보안**: Security Filter Chain이 관문에서 인증(JWT 서명 검증)과 인가(권한 확인)를 수행하고, 비밀번호는 BCrypt로 단방향 암호화한다.
- **6장 — 테스트**: 각 계층을 단위·통합 테스트로 검증해, "왜 되는지"를 코드로 증명한다.
- **7장 — Docker**: "내 컴퓨터에서는 되는데" 문제를, 앱과 환경을 컨테이너에 통째로 담아 근본적으로 해결한다. Compose로 여러 컨테이너를 지휘하고, 멀티스테이지로 이미지를 가볍게 만든다.
- **8장 — 전체 그림**: 요청 하나(회원가입/로그인/게시글 작성)의 여정 속에서 위 모든 개념이 동시에 유기적으로 작동한다. 실무에는 캐싱·메시지 큐·MSA·CI/CD·모니터링·클라우드가 더해지며, Phase 1~4로 나아간다.
- **관통하는 태도**: 코드를 많이 읽고, 항상 "왜?"를 묻고, 완벽하지 않아도 계속 나아가라. **코드는 변하지만, 원리는 오래간다.**
