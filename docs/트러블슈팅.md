# 트러블슈팅 & 자주 묻는 질문 (FAQ)

이 문서는 처음 실습하는 학생들이 가장 자주 만나는 오류를 **증상 → 원인 → 해결** 순서로 정리한 가이드입니다.
대부분의 문제는 "Docker가 안 켜져 있음", "포트 충돌", "DB를 먼저 안 띄움" 셋 중 하나입니다.

## 이 프로젝트의 기본 정보 (먼저 외워두면 편함)

| 항목 | 값 |
|------|-----|
| 앱(Spring Boot) 포트 | `8080` |
| Adminer(DB 관리 웹) | `http://localhost:8081` |
| PostgreSQL 포트 | `5432` |
| Redis 포트 | `6379` |
| DB 이름 / 사용자 / 비밀번호 | `edu_spring` / `edu` / `edu1234` |
| Java 챕터 컨테이너 이름 | `java-edu` |

> 핵심 규칙: **Spring 챕터(04~09)를 실행하기 전에는 항상 Docker가 켜져 있어야 하고, DB가 떠 있어야 합니다.**

## 1. Docker가 안 켜져 있음

### 증상
```
Cannot connect to the Docker daemon at unix:///var/run/docker.sock.
Is the docker daemon running?
```
또는 `docker compose up` 실행 시 멈추거나 즉시 실패.

### 원인
Docker Desktop이 실행되지 않았습니다. `docker` 명령은 백그라운드의 Docker 데몬이 켜져 있어야 동작합니다.

### 해결
1. **Docker Desktop 앱을 실행**합니다. (맥: 상단 메뉴바에 고래 아이콘이 안정적으로 떠 있어야 함)
2. 고래 아이콘이 "Docker Desktop is running" 상태가 될 때까지 기다립니다. (1~2분 걸릴 수 있음)
3. 터미널에서 확인:
   ```bash
   docker ps
   ```
   오류 없이 표(빈 표라도)가 나오면 정상입니다.

## 2. 포트가 이미 사용 중 (Port already in use)

### 증상
```
Bind for 0.0.0.0:8080 failed: port is already allocated
```
또는 앱이 뜨자마자 죽으면서 `Port 8080 was already in use` 로그가 보임.

### 원인
- 이전에 실행한 챕터의 컨테이너/앱이 **아직 종료되지 않고** 같은 포트(8080, 8081, 5432, 6379)를 잡고 있음.
- 또는 다른 프로그램(다른 DB, 다른 서버)이 그 포트를 쓰고 있음.

### 해결 — (1) 누가 쓰는지 찾기
```bash
# 8080 포트를 점유한 프로세스 확인 (맥/리눅스)
lsof -i :8080

# 또는 한 번에 종료
kill -9 $(lsof -t -i :8080)
```
포트를 5432, 6379, 8081 로 바꿔서 똑같이 확인하면 됩니다.

### 해결 — (2) 떠 있는 Docker 컨테이너 정리 (가장 흔한 원인)
```bash
# 현재 떠 있는 컨테이너 확인
docker ps

# 이전 챕터 컨테이너가 떠 있다면, 그 챕터 폴더에서 내리기
docker compose down

# 공통 인프라(DB 등)를 정리하려면
cd docker
docker compose -f docker-compose-infra.yml down
```
> **두 챕터를 동시에 띄우면 8080이 충돌합니다.** 한 번에 하나의 Spring 챕터만 실행하세요.

### 해결 — (3) 포트 자체를 바꾸기 (위 방법으로 안 될 때)
해당 챕터의 `docker-compose.yml`에서 포트 매핑 `"8080:8080"`을 `"8090:8080"`처럼 왼쪽(호스트) 숫자만 바꾸면 됩니다.
그러면 접속 주소도 `http://localhost:8090`으로 바뀝니다. (`.http` 파일의 `@baseUrl`도 같이 바꿔주세요.)

## 3. `docker compose up --build`가 너무 느림 / 멈춘 것 같음

### 증상
처음 빌드할 때 `Downloading...`, `Resolving dependencies` 같은 로그에서 오래 멈춰 있음.

### 원인
**처음 한 번은** Gradle이 의존성(라이브러리)과 Docker 베이스 이미지(JDK 등)를 인터넷에서 내려받습니다. 네트워크 상태에 따라 수 분이 걸릴 수 있습니다. 멈춘 게 아니라 **다운로드 중**입니다.

### 해결
- 그냥 기다립니다. (첫 빌드 3~10분은 정상)
- 두 번째 실행부터는 캐시 덕분에 훨씬 빠릅니다.
- 정말 멈춘 것 같으면 `Ctrl+C`로 중단 후 다시 `docker compose up --build`. 네트워크가 불안정하면 일시적 실패가 날 수 있습니다.

## 4. Chapter 01~03 (Java 기초): 컴파일/실행 문제

### 증상 A — `class not found` / `Error: Could not find or load main class`
```
Error: Could not find or load main class com.edu.basics.VariablesAndTypes
```

#### 원인
**컴파일을 안 했습니다.** Java는 `.java`(소스)를 `.class`(실행 파일)로 컴파일한 뒤에야 실행할 수 있습니다.
이 프로젝트는 `/app/out`에 컴파일 결과를 두고, 거기서 실행합니다.

#### 해결 — 항상 `compile.sh` → `run.sh` 순서
```bash
# 컨테이너 먼저 띄우기 (최초 1회)
docker compose up -d

# 컨테이너 안에서 실행
docker exec -it java-edu ./compile.sh   # 1) 컴파일
docker exec -it java-edu ./run.sh        # 2) 대화형 메뉴 실행
docker exec -it java-edu ./run.sh VariablesAndTypes   # 또는 클래스 이름으로 바로 실행
docker exec -it java-edu ./run.sh all    # 전체 실행
```
> 코드를 수정했다면 **반드시 `./compile.sh`를 다시 실행**해야 변경이 반영됩니다.
> 볼륨 마운트 덕분에 `docker build`를 다시 할 필요는 없고, `compile.sh`만 다시 돌리면 됩니다.

### 증상 B — 컴파일 실패 (`컴파일 실패! 위의 오류를 확인하세요.`)
#### 원인
수정한 소스 코드에 문법 오류가 있습니다.
#### 해결
출력된 `error:` 줄의 **파일명과 줄 번호**를 보고 해당 위치를 고친 뒤 다시 `./compile.sh`.

### 증상 C — `java-edu` 컨테이너가 없다는 오류
```
Error response from daemon: No such container: java-edu
```
#### 해결
프로젝트 루트(`edu_spring/`)에서 `docker compose up -d`를 먼저 실행해 컨테이너를 띄우세요.

## 5. DB 연결 실패 (Connection refused) — Spring 챕터에서 가장 흔함

### 증상
```
org.postgresql.util.PSQLException: Connection to localhost:5432 refused.
Check that the hostname and port are correct and that the postmaster is accepting TCP/IP connections.
```
또는 앱이 부팅 중 `HikariPool ... Exception during pool initialization` 으로 죽음.

### 원인
**PostgreSQL이 안 떠 있습니다.** Chapter 06처럼 공통 인프라를 따로 띄워야 하는 경우, DB를 먼저 실행하지 않고 앱만 실행하면 연결할 DB가 없어서 실패합니다.

### 해결
```bash
# 공통 인프라(PostgreSQL, Redis, Adminer) 먼저 실행
cd docker
docker compose -f docker-compose-infra.yml up -d

# DB가 떴는지 확인 (healthy 가 보이면 OK)
docker ps
```
- Adminer(`http://localhost:8081`)에 접속해 `edu / edu1234 / edu_spring`로 로그인되면 DB 정상입니다.
- Chapter 09처럼 챕터의 `docker compose up --build` 한 번으로 DB까지 함께 뜨는 경우라면, **postgres 컨테이너가 healthy 될 때까지 앱이 기다렸다가** 부팅됩니다. 첫 실행 시 잠깐 느린 건 정상입니다.

> 비밀번호 인증 실패(`password authentication failed for user "edu"`)가 나면, 예전에 다른 비밀번호로 만든 DB 볼륨이 남아 있을 수 있습니다. 9번 항목의 `down -v`로 초기화하세요.

## 6. JWT 인증 문제 (401 / 403) — Chapter 07, 09

### 증상 A — 401 Unauthorized
보호된 API(글 작성 등)를 호출했는데 401이 떨어짐.
#### 원인 & 해결
- **`Authorization` 헤더를 안 붙였다.** → 요청에 `Authorization: Bearer <토큰>` 헤더를 추가하세요.
- **로그인을 안 하고 토큰이 없다.** → 먼저 `POST /api/auth/login`으로 토큰을 받으세요.
- **토큰이 만료됐다 (`token expired`).** → 다시 로그인해서 새 토큰을 받으세요. (토큰에는 유효기간이 있습니다.)
- **`Bearer ` 접두어를 빼먹었다.** → `Bearer <토큰>` 형식이어야 합니다. `Bearer`와 토큰 사이에 공백 한 칸.

### 증상 B — 403 Forbidden
로그인은 됐는데(토큰 있음) 특정 API에서 403이 떨어짐.
#### 원인
인증은 됐지만 **권한(Role)이 부족**합니다. 예: 일반 `USER` 토큰으로 `ADMIN` 전용 API(`/api/admin/**`) 접근, 또는 남이 쓴 글을 수정/삭제하려는 경우(작성자 본인만 가능).
#### 해결
- 관리자 API는 `role`이 `ADMIN`인 계정으로 회원가입/로그인해서 받은 토큰을 쓰세요.
- 글/댓글 수정·삭제는 **작성한 본인 계정의 토큰**으로 요청하세요.

> **401 vs 403 한 줄 요약**: 401 = "너 누구야?(인증 안 됨)", 403 = "너인 건 알겠는데 권한 없어(인가 실패)".

### 참고 — Chapter 07 회원가입은 `role`이 필수
Chapter 07의 `/api/auth/signup`은 `username`, `password`(8자 이상), **`role`(USER 또는 ADMIN)** 세 개가 모두 필요합니다.
Chapter 09는 `role` 없이 `username`, `password`(6자 이상)만 보냅니다. 챕터별로 다르니 `.http` 파일의 예시를 참고하세요.

## 7. Validation 400 에러 — 입력값 검증 실패

### 증상
```
HTTP/1.1 400 Bad Request
```
정상적으로 보냈다고 생각했는데 400이 떨어짐.

### 원인
요청 본문이 **검증 규칙**을 어겼습니다. 예시:
- 제목(`title`)을 비워서 보냄 (`@NotBlank`)
- 비밀번호가 너무 짧음 (`@Size(min=...)`)
- 이메일 형식이 아님 (`@Email`)

### 해결
- 응답 본문(JSON)의 `message`/`error` 필드를 읽으면 **어떤 필드가 왜 틀렸는지** 나옵니다. 그 메시지대로 값을 고치세요.
- 챕터별 필수/길이 규칙:
  - Ch05 Todo: `title` 필수, 100자 이하
  - Ch06 Member: `name` 2~50자, `email` 이메일 형식
  - Ch07 SignUp: `username` 2~50자, `password` 8자 이상, `role` 필수
  - Ch09 SignUp: `username` 3~50자, `password` 6자 이상 / Post: `title`·`content` 필수

## 8. JPA 관련 문제 — Chapter 06, 09

### 증상 A — `LazyInitializationException`
```
org.hibernate.LazyInitializationException: could not initialize proxy - no Session
```
#### 원인
`@ManyToOne(fetch = LAZY)` 같은 지연 로딩 연관 엔티티를, **트랜잭션(영속성 컨텍스트)이 끝난 뒤**에 접근했습니다. (예: 컨트롤러에서 엔티티를 그대로 반환하면서 연관 객체를 직렬화하려 할 때)
#### 해결
- 서비스 계층(트랜잭션 안)에서 필요한 데이터를 **DTO로 변환**해 반환하세요. 이 프로젝트가 DTO 패턴을 쓰는 이유가 이것입니다.
- 또는 fetch join(`@Query`의 `join fetch`)으로 연관 데이터를 함께 조회하세요.

### 증상 B — 쿼리가 너무 많이 나감 (N+1 문제)
목록 1번 조회인데 로그에 SELECT가 수십 개 찍힘.
#### 원인
목록을 가져온 뒤, 각 항목마다 연관 엔티티를 1건씩 추가 조회해서 쿼리가 (1 + N)개 발생합니다.
#### 해결
fetch join 또는 `@EntityGraph`로 한 번에 조회. (개념과 예시는 **Chapter 06 README**의 연관관계/JPQL 부분 참고.)

### 증상 C — 테이블이 안 만들어짐 / `relation "member" does not exist`
#### 원인
스키마(테이블)가 생성되지 않았습니다. `application.yml`의 `spring.jpa.hibernate.ddl-auto` 설정이 테이블 자동 생성 여부를 결정합니다. (`create`/`update`면 자동 생성, `none`/`validate`면 자동 생성 안 함.)
#### 해결
- 학습용에서는 보통 `ddl-auto: create` 또는 `update`로 두면 앱 시작 시 테이블이 만들어집니다.
- 그래도 안 되면 **DB 볼륨이 꼬였을 수** 있으니 9번의 `down -v`로 초기화 후 다시 실행.

## 9. Gradle Wrapper / JDK 버전 문제 (로컬에서 `./gradlew` 쓸 때)

> Docker로만 실행하면 이 문제는 거의 없습니다. 컨테이너 안에 JDK 21이 들어 있기 때문입니다.
> 아래는 `./gradlew bootRun`처럼 **로컬에서 직접** 빌드할 때 해당됩니다.

### 증상 A — `permission denied: ./gradlew`
```bash
chmod +x gradlew
./gradlew bootRun
```

### 증상 B — JDK 버전 불일치
```
error: invalid source release: 21
```
또는 `Unsupported class file major version`.
#### 원인
로컬에 설치된 JDK가 21보다 낮습니다. 이 프로젝트는 **Java 21** 기준입니다.
#### 해결
- 가장 쉬운 방법: 로컬 빌드 대신 **Docker로 실행**하세요. (JDK 설치 불필요)
- 굳이 로컬에서 한다면 JDK 21을 설치하고, `java -version`이 21을 가리키는지 확인하세요.

## 10. 일반 디버깅 팁

### 스택 트레이스(에러 로그) 읽는 법
- 보통 **맨 위 몇 줄**에 예외 종류와 메시지가 있습니다. (`...Exception: 메시지`)
- `Caused by:` 가 있으면 **그 부분이 진짜 원인**인 경우가 많습니다. 가장 아래쪽 `Caused by`부터 보세요.
- `com.edu.~`로 시작하는 줄을 찾으면 우리가 작성한 코드 중 어디서 터졌는지 알 수 있습니다.

### 로그는 어디서 보나
```bash
# 실행 중인 앱 로그 따라보기 (해당 챕터 폴더에서)
docker compose logs -f app

# 특정 컨테이너 로그
docker logs -f <컨테이너이름>

# 인프라(DB 등) 로그
cd docker && docker compose -f docker-compose-infra.yml logs -f
```
포그라운드로 `docker compose up`(–d 없이) 실행했다면 로그가 터미널에 바로 흐릅니다.

### 깨끗하게 다시 시작하기 (막히면 이걸로 초기화)
```bash
# 컨테이너 + 네트워크 내리기
docker compose down

# 볼륨(DB 데이터)까지 싹 지우고 초기화 ← DB가 꼬였을 때 특효
docker compose down -v

# 그 뒤 다시 빌드해서 실행
docker compose up --build
```
> `down -v`는 **DB 데이터를 모두 삭제**합니다. 실습 데이터가 날아가도 괜찮을 때만 사용하세요.
> 회원가입했던 계정/게시글도 사라지니, 초기화 후에는 회원가입부터 다시 하면 됩니다.

### 그래도 안 될 때 체크리스트
1. Docker Desktop이 정말 켜져 있는가? (`docker ps`)
2. 이전 챕터 컨테이너를 `docker compose down`으로 내렸는가? (8080 충돌)
3. Spring 챕터인데 DB를 먼저 띄웠는가? (`docker-compose-infra.yml up -d` 또는 챕터 compose)
4. 코드를 고쳤다면(Ch01~03) `compile.sh`를 다시 돌렸는가?
5. 인증 API라면 토큰을 받아 `Authorization: Bearer ...` 헤더를 붙였는가?
