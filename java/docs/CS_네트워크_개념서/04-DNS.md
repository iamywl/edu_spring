# 4장. DNS — 이름을 주소로

> "사람은 `google.com`을 기억하지만, 네트워크는 `142.250.x.x`로 길을 찾는다. 이 둘을 이어주는 전화번호부가 DNS다."

## 도입 — 왜 이름과 주소를 분리하나

2장에서 실제 통신은 **IP 주소**로 이뤄진다고 했다. 3장에서 그 IP·포트로 소켓을 열었다. 그런데 우리가 실제로 타이핑하는 것은 `93.184.216.34`가 아니라 `example.com`이다. 그 사이를 잇는 번역기가 **DNS(Domain Name System)**다.

DNS는 응용 계층(L7) 프로토콜이지만, HTTP보다 **먼저** 일어난다. 소켓을 열려면 먼저 목적지 IP를 알아야 하고, 그 IP를 알아내는 것이 DNS이기 때문이다. 그래서 7장의 "URL 입력 → 응답" 여정에서 DNS는 TCP 연결보다 앞선 STEP 2에 놓인다. "첫 접속만 느린" 현상의 나머지 절반(§3.3의 handshake가 절반)도 여기서 나온다.

---

## 4.1 DNS가 해결하는 문제

### 개념

사람이 `93.184.216.34`를 외울 순 없다. 그래서 `example.com` 같은 **도메인 이름**을 IP로 **번역**하는 시스템이 필요하다 — 그것이 DNS다.

```
  사람이 보는 것:   www.example.com
                       │  DNS가 번역
                       ▼
  네트워크가 쓰는 것: 93.184.216.34
```

### 원리 — 이름과 위치의 분리

**왜 이름을 쓰는가?** 기억하기 쉬운 것만이 아니다. 서버 IP가 바뀌어도(이사, 서버 증설, 클라우드 이전) **이름은 그대로 두고 DNS 기록만 바꾸면** 사용자는 아무것도 안 해도 된다. 이름과 위치를 분리한 것 — §2.2에서 본 "MAC(고정 이름) vs IP(위치)"와 같은 발상의 한 층 위 버전이다.

또한 하나의 이름이 **여러 IP**를 가리키게 하면(A 레코드 여러 개), 접속을 여러 서버에 분산할 수 있다(DNS 라운드로빈). 이름이라는 간접 층 하나가 유연성을 크게 늘린다.

### 예시 — 도메인의 계층 구조

도메인은 오른쪽부터 계층을 이룬다. 맨 오른쪽에 보이지 않는 루트(`.`)가 있다.

```
  www . example . com .
   │      │        │   └── 루트(root, 보통 생략)
   │      │        └────── TLD(Top-Level Domain): com, org, kr, ...
   │      └─────────────── 2차 도메인(등록자가 소유): example
   └────────────────────── 서브도메인(호스트): www, api, mail ...
```

이 계층 구조 덕분에 DNS는 하나의 거대한 서버가 아니라 **계층적으로 분산된 시스템**이 된다. `.com`을 아는 서버, `example.com`을 아는 서버가 따로 있고 책임을 나눈다.

## 4.2 재귀적 질의와 반복적 질의

### 개념 — 리졸버가 나 대신 돌아다닌다

내 PC가 이름을 물으면, 보통 통신사/회사의 **DNS 리졸버(recursive resolver)**가 나 대신 여러 서버를 돌며 답을 찾아온다.

### 원리 — 재귀는 위임, 반복은 단계별 추적

```
  ① 내 PC ──"www.example.com 주소 뭐야?"──→ [리졸버(재귀)]
                                                │  (리졸버가 나 대신 아래를 다 돌아줌 = 재귀적)
     ┌──────────────────────────────────────────┤
     │  ② 리졸버 → [루트 서버]:  "com은 누가 알아?"
     │           ← "com 네임서버 주소는 여기"          ┐
     │  ③ 리졸버 → [com 서버]:   "example.com은 누가?"  │ 리졸버가 단계별로
     │           ← "example.com 네임서버는 여기"        │ 직접 물어봄 = 반복적
     │  ④ 리졸버 → [example.com 서버]: "www 주소는?"    │
     │           ← "93.184.216.34 야"                  ┘
     └──────────────────────────────────────────┐
  ⑤ 리졸버 ──"93.184.216.34 야"──→ 내 PC ◀───────┘
```

- **재귀적 질의(recursive)**: 내 PC → 리졸버. "**답을 통째로 가져다줘**"라고 위임한다. 내 PC는 최종 답만 기다린다.
- **반복적 질의(iterative)**: 리졸버 → 루트/TLD/권한 서버. "**다음에 물어볼 곳만 알려줘**"를 반복한다. 각 서버는 "나는 모르지만 저기 물어봐"라고 다음 단계를 알려줄 뿐이다. §2.5의 라우팅("다음 한 걸음만 안다")과 닮았다.

### 예시 — 세 종류의 네임서버

- **루트 네임서버**: "`.com`, `.kr` 같은 TLD는 누가 관리하는지"만 안다. 전 세계에 논리적으로 13개(실제로는 애니캐스트로 수백 대).
- **TLD 네임서버**: `.com`을 관리. "`example.com`의 권한 네임서버가 누구인지"를 안다.
- **권한(authoritative) 네임서버**: `example.com`의 실제 레코드(A, AAAA 등)를 갖고 있는 최종 답의 원천.

## 4.3 레코드 타입과 캐싱/TTL

### 개념 — DNS는 레코드 단위로 저장한다

DNS가 저장하는 정보는 **레코드(record)** 단위다. 주요한 것만:

| 레코드 | 의미 | 예시 |
|-------|------|------|
| **A** | 도메인 → IPv4 주소 | `example.com → 93.184.216.34` |
| **AAAA** | 도메인 → IPv6 주소 | `example.com → 2606:2800:...` |
| **CNAME** | 도메인 → 다른 도메인(별칭) | `www.example.com → example.com` |
| **MX** | 메일 서버 | (이메일 라우팅용) |
| **NS** | 이 도메인의 권한 네임서버 | |
| **TXT** | 임의 텍스트 | 도메인 소유 검증, SPF 등 |

### 원리 1 — CNAME은 왜 유용한가

`www.myshop.com`, `shop.myshop.com`이 모두 같은 서버를 가리켜야 할 때, 각각 A 레코드로 IP를 박아두면 서버 IP가 바뀔 때 전부 고쳐야 한다. 대신 CNAME으로 모두 `myshop.com`을 가리키게 하면, **A 레코드 하나만 바꾸면** 전부 따라간다. §4.1의 "간접 층으로 유연성 확보"를 DNS 안에서 다시 한 것이다.

### 원리 2 — 캐싱과 TTL: 왜 첫 접속만 느린가

매번 §4.2의 재귀 질의를 다 돌면 너무 느리다. 그래서 결과를 **캐시**한다. 각 레코드에는 **TTL(Time To Live)**이 붙어 "이 답을 몇 초간 재사용해도 된다"를 정한다. 캐시는 여러 층에 존재한다: 브라우저 캐시 → OS 캐시 → 리졸버 캐시.

```
  첫 접속:  DNS 리졸버 왕복(수 ms~수십 ms) 발생  ──→  느림
  이후:     내 PC/브라우저/리졸버 캐시에서 즉시 응답  ──→  빠름 (TTL 만료 전까지)

  TTL=300  →  5분간 캐시 재사용. 5분 뒤엔 다시 물어봄.
```

### 예시 — TTL을 짧게/길게 하는 트레이드오프

- **길면(예: 하루)**: 캐시 효율이 좋아 DNS 트래픽이 적다. 하지만 서버 IP를 바꿔도 전 세계가 하루 동안 옛 IP를 본다.
- **짧으면(예: 60초)**: IP 변경이 빨리 반영되지만 DNS 질의 트래픽이 는다.

**서버 이전(마이그레이션)을 앞두면 미리 TTL을 짧게 낮춰두는 것**이 실무 요령이다. 이전 전날 TTL을 60초로 낮춰 두면, 실제 전환 시 전 세계 캐시가 1분 안에 새 IP를 본다. 이 발상은 §5.4의 HTTP 캐싱, §3.4의 재전송 타이머와 같은 "동적 조정"의 사고방식이다.

### 실습 연결

> **실습으로 확인**: `observe_http.sh`의 STEP 1이 `dig +noall +answer example.com A`를 실행해 A 레코드와 그 옆의 **TTL 숫자**를 그대로 보여준다(읽는 법: `이름  TTL(초)  IN  A  IP주소`). AAAA(IPv6)도 함께 조회한다. 스크립트를 두 번 연속 돌리면 두 번째 TTL이 줄어든 것(캐시가 카운트다운 중)을 볼 수도 있다. 여기서 얻은 IP가 3장의 소켓이 향할 목적지이고, 5장의 HTTP 요청이 실려 갈 곳이다.

## ⚠️ 흔한 오해와 함정

- **"DNS는 항상 UDP를 쓴다."** 대개 UDP(53번 포트)를 쓰지만, 응답이 크거나(예: DNSSEC) 존 전송(zone transfer)에는 TCP를 쓴다. "짧고 빠르면 UDP, 커지면 TCP"로 기억하라.
- **"CNAME은 A 레코드처럼 IP를 가리킨다."** 아니다. CNAME은 **다른 이름(도메인)**을 가리키는 별칭이다. 결국 그 이름의 A 레코드까지 한 번 더 따라가야 IP가 나온다.
- **"DNS를 바꾸면 즉시 전 세계에 반영된다."** 아니다. TTL 동안 옛 값이 캐시에 남는다. "DNS 전파(propagation)"에 시간이 걸리는 이유가 TTL이다.
- **"루트 서버가 모든 도메인의 IP를 안다."** 아니다. 루트는 TLD 네임서버 위치만 안다. 각 계층이 "다음 단계"만 알려주는 분산 구조다.
- **"DNS 조회는 웹서버가 한다."** 아니다. 조회는 클라이언트(브라우저/OS)가 소켓을 열기 **전에** 한다. HTTP보다 앞선 단계다.

## 연습문제

1. **DNS가 이름과 IP를 분리해 얻는 실무 이득 두 가지를 대라.**
   - *힌트*: 서버 이전, 부하 분산.
   - *해설*: (1) 서버 IP가 바뀌어도 이름은 유지 → DNS 기록만 수정. (2) 한 이름에 여러 IP를 두어 부하 분산(DNS 라운드로빈).

2. **재귀적 질의와 반복적 질의의 차이를 "누가 돌아다니는가"로 설명하라.**
   - *힌트*: 내 PC는 위임, 리졸버는 추적.
   - *해설*: 재귀는 내 PC가 리졸버에게 "답 통째로 가져와" 위임. 반복은 리졸버가 루트→TLD→권한 서버를 단계별로 직접 물어 추적.

3. **`www.example.com`의 IP를 알아내는 과정에서 루트/TLD/권한 네임서버는 각각 무엇을 알려주는가?**
   - *힌트*: 세 계층.
   - *해설*: 루트는 `.com` TLD 서버 위치, `.com` TLD는 `example.com` 권한 서버 위치, 권한 서버가 최종 A 레코드(IP)를 준다.

4. **서버 IP를 다음 주 화요일에 바꿀 예정이다. DNS 관점에서 무엇을 미리 해두면 좋은가? 이유는?**
   - *힌트*: TTL.
   - *해설*: 전환 며칠 전 TTL을 짧게(예: 60초) 낮춘다. 그러면 전환 시 캐시가 빠르게 새 IP로 갱신된다.

5. **"두 번째 접속이 첫 접속보다 빠른" 이유를 DNS 관점에서(3장 TCP 관점과 함께) 설명하라.**
   - *힌트*: 캐시 + keep-alive.
   - *해설*: 첫 접속은 DNS 재귀 질의(수 ms~수십 ms) + TCP 3-way handshake가 붙는다. 두 번째는 DNS가 캐시(TTL 내)에서 즉시 응답하고, TCP는 keep-alive로 재사용되어 둘 다 생략된다.

## 요약

- **DNS는 도메인 이름 ↔ IP 주소 번역기**다. 응용 계층 프로토콜이며 HTTP보다 먼저 실행된다.
- 이름/위치 분리로 **서버 이전·부하 분산의 유연성**을 얻는다(§2.2 MAC/IP 분리의 상위 버전).
- 조회는 **재귀적 질의(내 PC→리졸버, 위임)**와 **반복적 질의(리졸버→루트/TLD/권한 서버, 단계별 추적)**로 나뉜다. §2.5 라우팅처럼 "다음 단계만 안다".
- 레코드 타입: **A(IPv4)·AAAA(IPv6)·CNAME(별칭)·MX·NS·TXT**. CNAME은 IP 변경 시 한 곳만 고치면 되게 해준다.
- **TTL과 캐시**가 "첫 접속만 느린" 현상의 절반을 만든다. TTL은 캐시 효율 vs 변경 반영 속도의 트레이드오프이며, 서버 이전 전 미리 낮추는 것이 요령이다.
