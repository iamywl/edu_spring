## 도입 — 왜 마지막에 종합하는가

1장부터 6장까지 우리는 스택을 밑바닥부터 한 층씩 올렸다. 층으로 나누는 이유(1장), IP·라우팅(2장), TCP 신뢰성(3장), DNS(4장), HTTP(5장), TLS(6장). 각 장은 그 자체로 완결이지만, 진짜 실력은 **이 조각들이 하나의 요청 안에서 순서대로 어떻게 맞물리는지** 설명할 수 있을 때 드러난다.

이 장은 새 개념을 거의 도입하지 않는다. 대신 앞 여섯 장을 **하나의 타임라인**으로 꿴다. 이 타임라인을 다이어그램 없이 말로 설명할 수 있다면, 당신은 "HTTP 위"가 아니라 "HTTP 아래"를 아는 사람이다.

## 7.1 URL 입력에서 응답까지 — 7단계 추적

```
  당신:  브라우저(또는 앱)에서  https://api.example.com/users/1  요청

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 1. URL 파싱                                            [응용 준비]     │
  │   https  → 프로토콜(TLS+HTTP, 기본 포트 443)                                │
  │   api.example.com → 호스트(이름, 아직 IP 모름)                             │
  │   /users/1 → 경로                                                          │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 2. DNS 조회                                       [응용/DNS · 4장]     │
  │   "api.example.com의 IP?"  → (캐시에 없으면) 리졸버가 재귀 질의            │
  │   루트 → com → example.com 네임서버 → A 레코드: 93.184.216.34             │
  │   ※ 캐시에 있으면 이 단계 건너뜀 → "두 번째 접속이 빠른" 이유              │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │  이제 목적지 IP를 안다
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 3. TCP 3-way handshake                          [전송 · 3장]         │
  │   SYN → SYN-ACK → ACK   (93.184.216.34 의 443 포트로)                     │
  │   → 신뢰성 있는 연결(소켓) 수립. 왕복 1회(RTT) 비용.                       │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 4. TLS handshake (https라서)                    [보안 · 6장]         │
  │   ClientHello(+key_share,ALPN) → ServerHello+인증서 → 검증(CA체인) → 세션키│
  │   → 이후 통신은 대칭 세션키로 암호화. 여기서 HTTP/2 여부도 협상(ALPN).      │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │  이제 '안전한 관' 완성
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 5. HTTP 요청 전송                                [응용 · 5장]         │
  │   GET /users/1 HTTP/1.1                                                    │
  │   Host: api.example.com                                                    │
  │   Accept: application/json                                                 │
  │   (빈 줄)                                                                  │
  │   → 이 텍스트가 TLS로 암호화되어 TCP 세그먼트에 담겨 IP 패킷으로 전송       │
  │   ※ 캡슐화(1장): HTTP→(TLS)→TCP→IP→이더넷 순으로 봉투에 감싸짐             │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │  IP 라우팅(2장)으로 여러 hop을 건너 서버 도착
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 6. 서버 도착 → 역캡슐화 → Spring                 [서버 측]           │
  │   이더넷→IP→TCP 헤더를 차례로 벗김 → TLS 복호화 → HTTP 메시지 복원         │
  │   → 톰캣이 요청을 스레드에 배정(SPRING §2.7) → DispatcherServlet          │
  │   → @GetMapping("/users/{id}") 컨트롤러 호출  ★ 여기가 SPRING §3.3의 입구  │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │  Service→Repository→DB (SPRING §3.4)
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 7. HTTP 응답 → 당신에게                          [역방향 전 계층]     │
  │   200 OK + {"id":1,...}  →  같은 TLS 세션키로 암호화                       │
  │   →  TCP가 순서/신뢰 보장하며 전송  →  브라우저가 복호화·파싱·렌더링        │
  │   →  Connection: keep-alive면 연결 유지(§5.2), 다음 요청은 3·4 단계 생략!  │
  └─────────────────────────────────────────────────────────────────────────┘
```

## 7.2 각 단계를 계층에 매핑하면

| STEP | 하는 일 | 계층 | 참조 장 |
|------|--------|------|--------|
| 1 | URL 파싱 | (클라이언트 응용) | — |
| 2 | 이름 → IP | 응용(DNS) | [4장](04-DNS.md) |
| 3 | 연결 수립 | 전송(TCP) | [3장](03-전송-계층.md) |
| 4 | 암호화 협상 | 보안(TLS) | [6장](06-HTTPS-TLS.md) |
| 5 | 요청 전송 | 응용(HTTP) + 전 계층 캡슐화 | [5장](05-HTTP.md), [§1.3](01-계층-모델.md) |
| 5~7 | 라우팅/전달 | 네트워크(IP)·링크 | [2장](02-물리-링크-네트워크.md) |
| 6 | 컨트롤러 진입 | (서버 응용) | **SPRING §3.3** |

## 7.3 이제 앞머리의 질문들에 답할 수 있다

책 표지에서 던진 "API 문서엔 답이 없는" 질문들을, 이제 단계로 정확히 지목할 수 있다.

- **"첫 요청만 느린 이유?"** → STEP 2(DNS 재귀 질의)·3(TCP handshake, RTT 1회)·4(TLS handshake, RTT 1회 이상)가 첫 요청에만 붙는다. 이후엔 DNS 캐시(§4.3)·TCP keep-alive(§5.2)·TLS 세션 재사용(§6.2)으로 대부분 생략된다.
- **"타임아웃인데 서버 로그가 없는 이유?"** → STEP 3에서 연결조차 안 됐다면(방화벽·라우팅) 요청이 STEP 6의 Spring에 **도달조차 못 한** 것이다. 이건 L3/L4 문제이지 애플리케이션(L7) 문제가 아니다(§3.6 함정 참고).
- **"HTTPS 인증서 오류?"** → STEP 4의 인증서 검증(§6.3) 실패다. 만료·도메인 불일치·체인 불완전·자체 서명 중 하나다.
- **"`localhost`는 되는데 IP로는 안 되는 이유?"** → STEP 3의 포트 바인딩 문제일 수 있다. 서버가 `127.0.0.1`에만 바인딩되어 외부 IP로는 안 닿는 경우다(§2.3, §3.1).
- **"HTTP/1.1인데 왜 느리게 느껴지지?"** → STEP 5에서 HOL 블로킹(§5.2). HTTP/2 멀티플렉싱으로 개선된다(STEP 4의 ALPN에서 협상).

## 7.4 이 여정을 한 문장으로

> 이름을 주소로 바꾸고(DNS·4장), 신뢰할 수 있는 관을 뚫고(TCP·3장), 그 관을 암호로 감싸고(TLS·6장), 그 위로 텍스트 한 통(HTTP·5장)을 보내면, 여러 층의 봉투에 감싸여(캡슐화·1장) 여러 길목을 건너(라우팅·2장) 서버에 닿고, 서버가 봉투를 차례로 벗겨(역캡슐화) 마침내 `@GetMapping` 메서드를 부른다.

이 문장을 diagram 없이도 설명할 수 있다면, 당신은 이제 **HTTP 위가 아니라 HTTP 아래를 아는 사람**이다. 전공자의 출발선에 선 것이다.

### 실습 연결 — 7단계를 한 번에 관찰한다

> **실습으로 확인**: `chapter-cs-network/labs/observe_http.sh` **한 번의 실행**이 이 7단계 중 STEP 2~5를 순서대로 보여준다.
>
> - STEP 2(DNS): `dig`로 A/AAAA 레코드와 TTL (스크립트 섹션 1)
> - STEP 3(TCP): `curl -v`의 `Connected to ...` 줄 (섹션 2 "[가]")
> - STEP 4(TLS): `curl -v`의 TLS/인증서 줄과 `openssl s_client`의 인증서 체인 (섹션 2 "[나]", 섹션 3)
> - STEP 5(HTTP): `curl -v`의 `> `(요청)·`< `(응답) 헤더, 그리고 `nc`로 보낸 평문 요청 원문 (섹션 2 "[다]/[라]", 섹션 4)
>
> 그다음 `./run.sh RawHttpClient`로 소켓을 직접 열어 STEP 3(연결)과 STEP 5(HTTP 텍스트 전송)를 손으로 재현하라. 더 깊이 보려면 `traceroute example.com`으로 STEP 5의 IP 라우팅 hop들(§2.5)을, `Wireshark`/`tcpdump`로 STEP 5의 캡슐화된 봉투(§1.3)를 눈으로 열어 보라.

## ⚠️ 흔한 오해와 함정

- **"모든 요청이 STEP 1~7을 다 거친다."** 아니다. 캐시(DNS)·keep-alive(TCP)·세션 재사용(TLS)이 살아 있으면 두 번째 요청부터는 STEP 2·3·4가 대부분 생략된다. 그것이 "첫 요청만 느린" 이유다.
- **"타임아웃 = 서버 버그."** 대개 아니다. STEP 3에서 연결이 안 되면 요청은 서버 코드에 도달조차 못 한다. 로그가 없으면 L7이 아니라 L3/L4를 의심하라.
- **"http와 https는 STEP만 다르다."** https는 STEP 4(TLS)가 추가되고, STEP 5·7의 HTTP가 세션 키로 암호화된다. 기능적 차이가 아니라 안전한 관을 하나 더 뚫는 것이다.
- **"Spring이 요청을 '받는다'는 건 네트워크의 시작이다."** 반대다. STEP 6은 **여정의 끝(도착지)**이다. `@GetMapping`이 불릴 때 이미 6단계가 지나 있다.

## 연습문제

1. **`https://api.example.com/users/1`을 처음 호출할 때 거치는 7단계를 순서대로 한 줄씩 요약하라.**
   - *힌트*: URL 파싱 → DNS → TCP → TLS → HTTP 요청 → 서버 역캡슐화/컨트롤러 → 응답.
   - *해설*: (1) URL 파싱 (2) DNS로 이름→IP (3) TCP 3-way handshake (4) TLS 핸드셰이크·인증서 검증·세션키 (5) HTTP 요청 텍스트 캡슐화 전송 (6) 서버 역캡슐화→톰캣→`@GetMapping` (7) 응답을 역방향 전 계층으로.

2. **두 번째 요청이 빠른 이유를 STEP 번호와 이 책의 캐시/재사용 장치로 설명하라.**
   - *힌트*: STEP 2·3·4가 생략.
   - *해설*: DNS 캐시(§4.3)로 STEP 2, TCP keep-alive(§5.2)로 STEP 3, TLS 세션 재사용(§6.2)으로 STEP 4가 대부분 생략된다.

3. **요청 후 타임아웃이 났는데 서버 애플리케이션 로그가 완전히 비어 있다. 어느 STEP에서 실패했을 가능성이 크며, 어느 계층 문제인가?**
   - *힌트*: 연결조차 안 됨.
   - *해설*: STEP 3(TCP 연결) 또는 그 이전에서 실패. 요청이 STEP 6의 Spring에 도달조차 못 했다. L3/L4(라우팅·방화벽·포트) 문제다.

4. **HTTPS 접속 시 "인증서가 신뢰되지 않는다"는 경고가 떴다. 어느 STEP·어느 장의 어떤 검증이 실패한 것인가? 원인 후보 셋을 대라.**
   - *힌트*: STEP 4, §6.3.
   - *해설*: STEP 4의 인증서 검증(§6.3) 실패. 만료 / 도메인 불일치 / 자체 서명(또는 체인 불완전) 등.

5. **STEP 5에서 HTTP 요청 텍스트가 서버까지 가는 동안 어떤 봉투들에 감싸이며, 그 봉투들은 어느 장에서 다뤘는가?**
   - *힌트*: 캡슐화.
   - *해설*: HTTP→TLS(6장)→TCP(3장)→IP(2장)→이더넷(2장) 순으로 헤더가 붙는다(캡슐화, §1.3). 서버는 역순으로 벗긴다.

## 요약

- URL 입력 → 응답은 **7단계**다: URL 파싱 → DNS(4장) → TCP handshake(3장) → TLS handshake(6장) → HTTP 요청 캡슐화 전송(5장·1장) → 서버 역캡슐화·컨트롤러 진입(SPRING §3.3) → 응답 역방향 전달.
- 각 단계는 특정 계층에 매핑된다. **`@GetMapping`이 불리는 STEP 6은 여정의 끝(도착지)**이다.
- 첫 요청이 느린 것은 STEP 2·3·4 때문이며, DNS 캐시·keep-alive·TLS 세션 재사용으로 상쇄된다.
- 실무 질문("타임아웃인데 로그 없음", "인증서 오류", "localhost만 됨")은 이제 **정확한 STEP과 계층으로 지목**할 수 있다.
- `observe_http.sh` 한 번이 STEP 2~5를 순서대로 보여주고, `./run.sh RawHttpClient`가 STEP 3·5를 손으로 재현한다.

## 마무리 — 다음으로

이 책은 "**API를 호출하면 무슨 일이 벌어지는가**"를 계층으로 풀었다. 여기서 더 깊이 가려면:

- **직접 관찰하라**: `chapter-cs-network/labs/observe_http.sh`를 돌려 DNS·TLS·헤더를 눈으로 보라. `./run.sh RawHttpClient`로 "HTTP는 텍스트"임을 증명하라. **읽은 것을 본 순간 개념이 몸에 붙는다.**
- **더 깊은 도구**: `Wireshark`(패킷을 눈으로), `traceroute`(hop 추적), `tcpdump`(패킷 캡처)로 실제 봉투(§1.3)를 열어보라.
- **되짚어 읽기**: SPRING §3(웹 구조)을 다시 읽으면, 이제 그 "요청"이 어디서 왔는지 보인다. SPRING §5.3·§5.7(세션/JWT/비대칭)은 이 책 [5장](05-HTTP.md)·[6장](06-HTTPS-TLS.md)과 짝을 이룬다.

> "왜?를 계속 물어라. `@GetMapping`이 호출되기까지 7단계를 설명할 수 있게 되면, 다음 질문은 '그 7단계 각각은 또 왜 그렇게 생겼나'다. 그 질문이 당신을 전공자 너머로 데려간다."
