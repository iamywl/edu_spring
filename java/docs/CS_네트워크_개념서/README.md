# CS 네트워크 개념서 — HTTP 아래에서는 무슨 일이 일어나는가?

> "Spring이 `@GetMapping`으로 요청을 받을 때, 그 요청은 어디서 어떻게 여기까지 왔는가? 이 질문에 답할 수 없다면, 당신은 API를 다룰 뿐 네트워크를 다루는 것이 아니다."

> **🐳 실습 환경 — 이 책의 Java 데모는 `java-sandbox` 컨테이너에서 실행한다** (인터넷 접속 필요)
> ```bash
> cd java && docker compose up -d              # 컨테이너 켜기 (이미 떠 있으면 생략)
> docker exec -it java-sandbox ./run.sh RawHttpClient
> ```
> 셸 관찰 실습은 호스트에서: `bash java/chapter-cs-network/labs/observe_http.sh`

---

## 이 책은 무엇인가

`SPRING_개념서`의 **Chapter 3(웹 애플리케이션의 구조)**는 HTTP를 **API 레벨**에서 설명한다. 요청/응답이 무엇인지, 메서드와 상태 코드가 무엇인지, REST를 어떻게 설계하는지 — 즉 "**HTTP라는 언어로 무엇을 말하는가**"를 다룬다.

이 책은 그 **아래**를 판다. 브라우저 주소창에 `https://api.example.com/users/1`을 입력하고 Enter를 누른 순간부터, Spring의 컨트롤러 메서드가 호출되기까지 — 그 사이에 **패킷 단위에서 실제로 무슨 일이 일어나는가**를, 대학 전공 네트워크 교재 수준으로 계층별로 설명한다.

- SPRING §3.1은 "HTTP 요청은 편지와 같다"고 말한다. 이 책은 **그 편지가 어떤 봉투에 담겨, 어떤 우편 시스템을 거쳐, 어떻게 목적지 서버까지 배달되는지**를 설명한다.
- SPRING §5.7은 "TLS도 대칭/비대칭 원리다"라고 짚고 넘어간다. 이 책의 6장에서 **그 TLS 핸드셰이크를 본격적으로** 파헤친다.

## 학습 목표

이 책을 다 읽으면 다음을 **다이어그램 없이도 말로 설명**할 수 있게 된다.

- 왜 네트워크를 층(layer)으로 나누는가, OSI 7계층과 TCP/IP 4계층은 어떻게 대응하는가, 캡슐화란 무엇인가
- IP 주소와 MAC 주소가 왜 둘 다 필요한가, CIDR/서브넷은 무엇을 나누는가, 라우팅은 어떻게 "중앙 지휘부 없이" 동작하는가
- TCP가 "순서대로, 빠짐없이"를 어떻게 만드는가 — seq/ack, 재전송, 슬라이딩 윈도우, 혼잡 제어
- 3-way handshake와 4-way close의 각 단계, TIME_WAIT의 정체
- DNS 재귀/반복 질의, 레코드 타입, TTL과 캐시
- HTTP 요청/응답의 바이트 수준 구조, HTTP/1.1 vs HTTP/2 vs HTTP/3, 쿠키/세션, 캐싱 헤더
- TLS가 지키는 세 가지(기밀성·무결성·인증), TLS 1.3 핸드셰이크, 인증서 체인과 CA
- 위 전부를 하나로 꿴 "URL 입력 → 응답"의 7단계

이 능력이 전공자와 비전공자를 가르는 선이다.

**왜 비전공자가 이걸 배워야 하는가?** REST API만 알면 당장 CRUD는 짤 수 있다. 하지만 이런 순간에 벽에 부딪힌다.

- "왜 첫 요청만 느리고 그다음은 빠르지?" (DNS 캐시, TCP keep-alive, TLS 세션 재사용)
- "왜 `localhost`는 되는데 IP로는 안 되지?" (바인딩 주소, 포트)
- "타임아웃이 났는데 서버 로그엔 아무것도 없다?" (연결이 서버에 도달조차 못 함 — 네트워크/전송 계층)
- "HTTPS인데 왜 인증서 오류가 나지?" (CA 신뢰 체인)

이 질문들은 **API 문서에 답이 없다.** 네트워크 스택을 이해해야만 답할 수 있다.

## 추천 학습 순서

이 책은 **아래에서 위로 올라간다.** 물리/링크/네트워크(2장) → 전송(3장) → DNS(4장) → HTTP(5장) → TLS(6장). 그리고 마지막에 전부 합쳐 하나의 요청을 추적한다(7장).

1. **1장 → 7장 순서대로** 읽는 것을 기본으로 한다. 각 장이 앞 장의 개념을 딛고 서기 때문이다.
2. 시간이 없다면 **1장(계층/캡슐화) → 3장(TCP) → 7장(종합)**만 읽어도 큰 그림은 잡힌다.
3. **SPRING_개념서와 짝지어 읽기**: 이 책 5장(HTTP)은 SPRING §3(웹 구조) 위에 쌓이고, 이 책 6장(TLS)은 SPRING §5.7(대칭/비대칭 암호)이 "미뤄둔" 이야기를 편다. 5.3장·6장을 읽을 때 SPRING §5.3·§5.7을 나란히 펼쳐 두면 이해가 두 배로 선명해진다.

각 장은 다음 형식을 따른다: **도입(왜 필요한가) → 개념·원리 → ASCII 다이어그램 → 예시 → 실습 연결 → 흔한 오해와 함정 → 연습문제 → 요약 → 이전/다음 장 링크.**

## 실습 안내 — 읽은 것을 눈으로 본다

이 책은 개념이고, `chapter-cs-network/labs/observe_http.sh`는 **그 개념을 눈으로 보는 실습**이다. 각 장 끝에 "**실습으로 확인**" 표시가 있다. 개념을 읽고 → 스크립트를 돌려 실제 패킷/헤더/인증서를 보면, 추상적이던 계층이 손에 잡힌다.

```
  이 책(개념)  ──────→  labs/observe_http.sh(관찰)  ──────→  RawHttpClient.java(직접 구현)
    "왜 그런가"           "정말 그런지 본다"                "HTTP는 그냥 텍스트임을 증명"
```

**셸 실습** — 계층을 눈으로 본다(읽기 전용, 공개 사이트 `example.com`만 조회):

```bash
# chapter-cs-network 디렉터리에서
bash labs/observe_http.sh
```

| 단계 | 사용 도구 | 관찰 대상 | 이 책의 장 |
|------|-----------|-----------|-----------|
| 1 | `dig` / `nslookup` | DNS A/AAAA 레코드와 TTL | 4장 |
| 2 | `curl -v` | TCP 연결 · TLS 핸드셰이크 · 요청/응답 헤더 | 3·5·6장 |
| 3 | `openssl s_client` | 인증서 체인 · TLS 버전/암호 스위트 | §6.3 |
| 4 | `nc` + `printf` | 평문 TCP로 보낸 날것의 HTTP 요청 바이트 | §5.1 |

**Java 데모** — `RawHttpClient.java`로 소켓에 텍스트를 직접 써서 "HTTP = TCP 위의 텍스트"임을 증명한다. 프로젝트의 통합 실습 환경인 `java-sandbox` 컨테이너에서 다음처럼 실행한다.

```bash
cd java && docker compose up -d              # 컨테이너 켜기 (이미 떠 있으면 생략)
docker exec -it java-sandbox ./run.sh RawHttpClient
```

또는 로컬 JDK로 직접:

```bash
javac -d /tmp/netout src/main/java/com/edu/network/RawHttpClient.java
java  -cp /tmp/netout com.edu.network.RawHttpClient
```

> 모든 실습은 인터넷 접속이 필요하다. 공개 테스트 사이트(`example.com`)만 **읽기 전용**으로 조회하며, 아무것도 설치·변경·삭제하지 않는다. 오프라인이면 각 단계가 실패 메시지를 내고 안전하게 종료된다.

## 더 깊이 가려면

- **직접 관찰하라**: `observe_http.sh`를 돌려 DNS·TLS·헤더를 눈으로 보라. `RawHttpClient.java`로 "HTTP는 텍스트"임을 증명하라.
- **더 깊은 도구**: `Wireshark`(패킷을 눈으로), `traceroute`(hop 추적), `tcpdump`(패킷 캡처)로 실제 봉투(1장 캡슐화)를 열어보라.
- **되짚어 읽기**: SPRING §3(웹 구조)을 다시 읽으면, 이제 그 "요청"이 어디서 왔는지 보인다. SPRING §5.3·§5.7(세션/JWT/비대칭)은 이 책 5장·6장과 짝을 이룬다.

> "왜?를 계속 물어라. `@GetMapping`이 호출되기까지 7단계를 설명할 수 있게 되면, 다음 질문은 '그 7단계 각각은 또 왜 그렇게 생겼나'다. 그 질문이 당신을 전공자 너머로 데려간다."
