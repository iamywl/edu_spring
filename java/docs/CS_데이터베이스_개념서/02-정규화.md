## 도입: 왜 이 장이 필요한가

스프링으로 게시판을 만들 때, 처음엔 "그냥 필요한 정보를 다 한 테이블에 넣으면 되지 않을까?"라는 생각이 드는 것은 자연스럽다. JOIN이 없으니 단순하고 빠를 것 같다. 그런데 데이터가 수천, 수십만 건 쌓이면서 다음 증상이 나타나기 시작한다.

- 학과 사무실 번호를 바꿨는데, 어떤 학생 기록은 옛날 번호를 가리키고 있다.
- 새로 개설한 강좌에 아직 수강생이 없어서 DB에 저장할 방법이 없다.
- 특정 학생 정보를 지웠더니, 그 학생이 수강한 강좌 정보까지 사라졌다.

이 세 가지 증상 — **갱신 이상, 삽입 이상, 삭제 이상** — 은 모두 같은 병에서 나온다: 서로 다른 사실을 한 테이블에 섞어 놓는 것.

정규화(normalization)란 **데이터 중복과 이상 현상을 제거하기 위해, 하나의 큰 테이블을 더 작고 올바른 테이블들로 쪼개는 체계적 절차**다. 직관이 아니라 수학적 도구(함수 종속, 폐포, 분해 조건)를 사용한다는 점에서 "체계적"이다.

이 장을 읽으면 다음을 할 수 있다:
- 임의의 스키마에서 함수 종속을 파악하고, 폐포로 후보키를 찾는다.
- 1NF → 2NF → 3NF → BCNF로 단계적으로 정규화한다.
- 무손실 분해와 종속성 보존 두 조건을 검증한다.
- BCNF와 3NF의 트레이드오프를 이해하고 언제 어떤 정규형을 선택할지 판단한다.

## 2.1 문제: 하나에 다 때려박은 테이블

수강 관리 시스템을 만든다고 하자. "그냥 필요한 걸 다 한 테이블에" 넣으면:

```
enrollment_bad (정규화 안 됨 — 이대로 두면 안 되는 예)
┌──────────┬────────────┬──────────┬────────────┬──────────┬──────────────┬──────────┬───────┐
│student_id│student_name│dept_name │dept_office │course_id │course_title  │professor │grade  │
├──────────┼────────────┼──────────┼────────────┼──────────┼──────────────┼──────────┼───────┤
│ 1001     │ 김철수      │컴퓨터공학 │ 공학관 301  │ CS101    │ 자료구조      │ 이교수    │ A     │
│ 1001     │ 김철수      │컴퓨터공학 │ 공학관 301  │ CS102    │ 데이터베이스   │ 박교수    │ B     │
│ 1002     │ 이영희      │컴퓨터공학 │ 공학관 301  │ CS101    │ 자료구조      │ 이교수    │ A     │
│ 1003     │ 박민수      │전자공학   │ 공학관 502  │ EE201    │ 회로이론      │ 최교수    │ C     │
└──────────┴────────────┴──────────┴────────────┴──────────┴──────────────┴──────────┴───────┘
```

한눈에 보이는 문제: `컴퓨터공학 / 공학관 301`이 세 번 반복되고, `CS101 / 자료구조 / 이교수`도 두 번 반복된다.

### 갱신 이상 (Update Anomaly)

컴퓨터공학과가 신공학관 101호로 이사했다. `dept_office`를 고쳐야 하는데, 이 값이 여러 행에 흩어져 있다. 세 행을 모두 고쳐야 한다. 그런데 하나라도 빠뜨리면?

```
1001 김철수 → 공학관 301    ← 업데이트 누락
1002 이영희 → 신공학관 101  ← 업데이트 완료
```

같은 학과인데 사무실이 두 곳으로 갈라진다. 데이터가 거짓말을 하기 시작한다. 이것이 갱신 이상이다.

### 삽입 이상 (Insert Anomaly)

새 강좌 `CS201 운영체제(정교수)`를 개설했는데 아직 수강생이 없다. 이 테이블에서 한 행은 "학생이 강좌를 수강하는 사건" 하나다. 학생 없이 강좌만 넣으려면 학생 관련 컬럼을 전부 NULL로 둔 억지 행을 만들어야 한다.

```
│ NULL │ NULL │ NULL │ NULL │ CS201 │ 운영체제 │ 정교수 │ NULL │  ← 말이 안 되는 행
```

"강좌 정보를 등록한다"는 사실을 표현할 방법이 없다. 이것이 삽입 이상이다.

### 삭제 이상 (Delete Anomaly)

박민수(1003)가 자퇴해 그의 수강 기록을 지운다. 그런데 박민수는 `EE201 회로이론`의 유일한 수강생이었다. 그 행을 지우는 순간 "EE201은 최교수 담당"이라는 강좌 정보도 함께 사라진다.

```
삭제 전: EE201 강좌 정보가 박민수 수강 기록 행에 묻혀 있음
삭제 후: EE201 강좌 정보 증발
```

학생을 지웠을 뿐인데, 완전히 다른 사실(강좌 정보)이 사라진다. 이것이 삭제 이상이다.

### 세 이상 현상의 공통 원인

세 이상 현상 모두 **서로 다른 종류의 사실을 한 테이블에 섞어 놓은 것**에서 비롯된다.

```
한 테이블에 섞인 사실들:
  ① 학생 정보     (student_id, student_name, dept_name)
  ② 학과 정보     (dept_name, dept_office)
  ③ 강좌 정보     (course_id, course_title, professor)
  ④ 수강 기록     (student_id, course_id, grade)
```

정규화는 이 사실들을 분리해 각자의 테이블에 담는 과정이다.

## 2.2 함수 종속 (Functional Dependency)

정규화를 체계적으로 하려면, 데이터 안의 "규칙"을 표현하는 언어가 필요하다. 그것이 **함수 종속(Functional Dependency, FD)**이다.

> **X → Y** ("X가 Y를 함수적으로 결정한다"): X의 값이 정해지면 Y의 값이 **유일하게** 정해진다.

수학 함수 `y = f(x)`와 같다. x 하나에 y 하나. "x가 다르더라도 y가 같을 수 있다"는 괜찮다. "같은 x에 서로 다른 y"가 없어야 함수적 종속이 성립한다.

```
enrollment_bad 에서 성립하는 FD들:
─────────────────────────────────────────────────────────────────────────
student_id              → student_name, dept_name
  (학번이 정해지면 학생 이름과 학과가 유일하게 정해진다)

dept_name               → dept_office
  (학과명이 정해지면 사무실이 유일하게 정해진다)

course_id               → course_title, professor
  (강좌코드가 정해지면 강좌명과 담당교수가 유일하게 정해진다)

(student_id, course_id) → grade
  (학번+강좌코드 조합이 정해지면 성적이 유일하게 정해진다)
─────────────────────────────────────────────────────────────────────────
```

`→`의 왼쪽을 **결정자(determinant)**, 오른쪽을 **종속자(dependent)**라 부른다.

이 테이블의 **후보키는 `(student_id, course_id)`** 복합키다. 이 두 컬럼의 조합만이 한 행을 유일하게 특정한다.

**핵심 통찰:** 이상 현상은 **"결정자가 후보키가 아닌 FD"**에서 발생한다.

`dept_name → dept_office`에서 결정자 `dept_name`은 이 테이블의 (후보)키가 아니다. 그래서 `dept_name` 같은 값을 가진 행이 여러 개 생기고, 거기에 딸린 `dept_office`도 여러 번 중복 저장된다. 이것이 갱신 이상의 뿌리다.

**정규화 = "결정자가 후보키가 아닌 FD"를 제거하는 과정.**

## 2.3 암스트롱 공리 (Armstrong's Axioms)

FD를 다루기 위해 **암스트롱 공리(Armstrong's Axioms)**라는 추론 규칙이 있다. 주어진 FD 집합에서 새로운 FD를 올바르게 유도하는 데 사용한다.

### 세 가지 기본 공리

**공리 1 — 반사성 (Reflexivity)**

> Y ⊆ X 이면 X → Y 가 성립한다.

컬럼 집합은 자신의 부분 집합을 항상 결정한다. 예: `{student_id, name} → student_id`.

```
{A, B} → A         (A는 {A,B}의 부분집합)
{A, B} → {A, B}   (자기 자신도 포함)
```

**공리 2 — 증대성 (Augmentation)**

> X → Y 이면, 임의의 Z에 대해 XZ → YZ 가 성립한다.

양변에 같은 컬럼을 추가해도 FD가 유지된다.

```
student_id → student_name
이면
{student_id, course_id} → {student_name, course_id}   (course_id를 양변에 추가)
```

**공리 3 — 이행성 (Transitivity)**

> X → Y 이고 Y → Z 이면, X → Z 가 성립한다.

```
student_id → dept_name    (학번으로 학과를 앎)
dept_name  → dept_office  (학과로 사무실을 앎)
∴ student_id → dept_office (학번으로 사무실을 앎)  ← 이행 종속
```

### 세 가지 파생 규칙

기본 공리 세 개로부터 유도되는 편의 규칙이다.

**합집합 (Union Rule)**

> X → Y 이고 X → Z 이면, X → YZ.

```
student_id → student_name
student_id → dept_name
∴ student_id → {student_name, dept_name}
```

**분해 (Decomposition Rule)**

> X → YZ 이면, X → Y 이고 X → Z.

```
student_id → {student_name, dept_name}
∴ student_id → student_name   (분해)
∴ student_id → dept_name      (분해)
```

**유사 이행성 (Pseudotransitivity Rule)**

> X → Y 이고 WY → Z 이면, WX → Z.

```
dept_name → dept_office
∴ {student_id, dept_name} → dept_office
  → student_id → dept_name 이므로
  → {student_id} → dept_office 가 성립 (이행성으로도 도출 가능)
```

암스트롱 공리의 세 기본 규칙은 **완전하고(complete) 건전하다(sound)**. 즉:
- 건전성(sound): 이 규칙으로 유도한 FD는 모두 참이다.
- 완전성(complete): 참인 모든 FD는 이 규칙으로 유도할 수 있다.

## 2.4 애트리뷰트 폐포 (Attribute Closure)

**X+** (X의 폐포, X-plus)는 "X가 함수적으로 결정할 수 있는 모든 애트리뷰트의 집합"이다. 폐포를 구하면 X가 후보키인지를 판단할 수 있다.

### 폐포 계산 알고리즘

```
입력: FD 집합 F, 시작 집합 X
출력: X+ (X의 폐포)

알고리즘:
1. closure ← X (시작: X 자신)
2. 변화가 없을 때까지 반복:
     F 안의 모든 FD "A → B"에 대해:
       if A ⊆ closure:
           closure ← closure ∪ B
3. return closure
```

### 예제: `enrollment_bad`에서 `{student_id}`의 폐포

```
FD 집합 F:
  F1: student_id → student_name, dept_name
  F2: dept_name → dept_office
  F3: course_id → course_title, professor
  F4: (student_id, course_id) → grade

X = {student_id}
closure = {student_id}

반복 1:
  F1: student_id ⊆ {student_id} → closure에 student_name, dept_name 추가
  closure = {student_id, student_name, dept_name}

반복 2:
  F2: dept_name ⊆ {student_id, student_name, dept_name} → dept_office 추가
  closure = {student_id, student_name, dept_name, dept_office}

반복 3:
  더 이상 추가할 애트리뷰트 없음 → 종료

{student_id}+ = {student_id, student_name, dept_name, dept_office}
```

`{student_id}+`는 전체 8개 컬럼을 포함하지 않는다(course_id, course_title, professor, grade 가 없다). 따라서 `{student_id}`는 후보키가 아니다.

### 예제: `{student_id, course_id}`의 폐포

```
X = {student_id, course_id}
closure = {student_id, course_id}

반복 1:
  F1: student_id ⊆ closure → student_name, dept_name 추가
  F3: course_id ⊆ closure → course_title, professor 추가
  F4: {student_id, course_id} ⊆ closure → grade 추가
  closure = {student_id, course_id, student_name, dept_name, course_title, professor, grade}

반복 2:
  F2: dept_name ⊆ closure → dept_office 추가
  closure = {student_id, course_id, student_name, dept_name, dept_office, course_title, professor, grade}
  (전체 8개 컬럼)

반복 3:
  더 이상 추가할 것 없음 → 종료

{student_id, course_id}+ = 전체 애트리뷰트 집합
```

폐포가 전체 컬럼 집합과 같으므로, `{student_id, course_id}`는 **슈퍼키**다. 그리고 이 집합에서 어느 하나라도 빼면 슈퍼키가 되지 않으므로 **후보키**이기도 하다.

### 후보키를 폐포로 찾는 방법

```
1. 모든 컬럼 집합의 부분집합을 열거한다.
2. 각 집합의 폐포를 계산한다.
3. 폐포가 전체 컬럼 집합인 집합 = 슈퍼키
4. 어떤 진부분집합도 슈퍼키가 아닌 슈퍼키 = 후보키
```

실제로는 모든 부분집합을 열거할 필요 없이 "결정자가 되는 컬럼"부터 시작해 상향식으로 찾는다.

## 2.5 단계별 정규화: 1NF → 2NF → 3NF → BCNF

정규형(Normal Form)은 계단이다. 위로 갈수록 엄격하고, 각 단계는 앞 단계를 **포함**한다.

```
완전 비정규화 테이블
       │
       ▼ 1NF: 원자값(반복 그룹·다중값 제거)
       │
       ▼ 2NF: 부분 종속 제거 (복합키의 일부에만 종속된 FD 제거)
       │
       ▼ 3NF: 이행적 종속 제거 (비키 컬럼 → 비키 컬럼 체인 제거)
       │
       ▼ BCNF: 모든 결정자가 후보키 (3NF의 남은 예외 제거)
```

### 제1정규형 (1NF): 모든 값이 원자적

> **정의:** 모든 컬럼의 값은 더 이상 쪼갤 수 없는 **원자값** 하나여야 한다. 반복 그룹(repeating group)과 다중값(multi-valued) 속성이 없어야 한다.

**1NF 위반 예 1: 한 칸에 여러 값**

```
┌──────────┬───────────────────┐
│student_id│ courses           │
│ 1001     │ CS101, CS102      │  ← 한 칸에 값 두 개! 1NF 위반
│ 1002     │ CS101             │
└──────────┴───────────────────┘
```

"CS101을 듣는 학생"을 찾으려면 문자열을 파싱해야 한다. 인덱스도 못 건다. 해결: 행으로 풀어낸다.

```
┌──────────┬──────────┐
│student_id│ course   │
│ 1001     │ CS101    │  ← 각 칸에 값 하나. 1NF 만족
│ 1001     │ CS102    │
│ 1002     │ CS101    │
└──────────┴──────────┘
```

**1NF 위반 예 2: 반복 컬럼**

```
┌──────────┬──────────┬──────────┬──────────┐
│student_id│ course1  │ course2  │ course3  │  ← 반복 컬럼! 1NF 위반
│ 1001     │ CS101    │ CS102    │ NULL     │
└──────────┴──────────┴──────────┴──────────┘
```

수강 강좌가 4개이면 컬럼을 또 추가해야 한다. 같은 방식으로 행으로 풀어낸다.

우리의 `enrollment_bad`는 이미 각 칸에 값이 하나뿐이므로 **1NF는 만족**한다.

### 제2정규형 (2NF): 부분 종속 제거

> **정의:** 1NF를 만족하면서, 후보키의 **진부분집합(proper subset)**에 종속되는 비(非)키 컬럼이 없어야 한다. 즉, 모든 비키 컬럼이 후보키 **전체**에 종속(완전 함수 종속)해야 한다.

2NF는 **복합 키(후보키가 2개 이상 컬럼)**일 때만 의미가 있다. 단일 컬럼 후보키는 진부분집합이 없으므로 1NF이면 자동으로 2NF.

우리 테이블의 후보키는 `(student_id, course_id)`다. 이 중 **일부에만** 종속된 것들을 찾자:

```
FD 검사:
─────────────────────────────────────────────────────────────────
student_id → student_name, dept_name   ← student_id 만으로 결정됨
                                          = 후보키 일부에만 종속
                                          = 부분 종속! (2NF 위반)

course_id  → course_title, professor   ← course_id 만으로 결정됨
                                          = 후보키 일부에만 종속
                                          = 부분 종속! (2NF 위반)

(student_id, course_id) → grade        ← 후보키 전체에 종속
                                          = 완전 함수 종속 (OK)
─────────────────────────────────────────────────────────────────
```

**해결: 부분 종속된 애트리뷰트를 자신의 결정자와 함께 새 테이블로 분리한다.**

```
student 테이블 (student_id가 결정하는 것들)
┌──────────┬────────────┬──────────┐
│student_id│student_name│dept_name │    ← dept_office는 dept_name이 결정하므로 여기선 아직 보류
│ 1001     │ 김철수      │컴퓨터공학 │      (3NF에서 처리)
│ 1002     │ 이영희      │컴퓨터공학 │
│ 1003     │ 박민수      │전자공학   │
└──────────┴────────────┴──────────┘

course 테이블 (course_id가 결정하는 것들)
┌──────────┬──────────────┬──────────┐
│course_id │course_title  │professor │
│ CS101    │ 자료구조      │ 이교수    │
│ CS102    │ 데이터베이스   │ 박교수    │
│ EE201    │ 회로이론      │ 최교수    │
└──────────┴──────────────┴──────────┘

enrollment 테이블 (완전 종속인 grade만 + 두 FK)
┌──────────┬──────────┬───────┐
│student_id│course_id │grade  │    PK = (student_id, course_id)
│ 1001     │ CS101    │ A     │    FK: student_id → student(student_id)
│ 1001     │ CS102    │ B     │    FK: course_id  → course(course_id)
│ 1002     │ CS101    │ A     │
│ 1003     │ EE201    │ C     │
└──────────┴──────────┴───────┘
```

**2NF 달성 후 이상 현상 변화:**
- 삽입 이상 해결: 수강생 없는 강좌를 `course` 테이블에 직접 넣을 수 있다.
- 삭제 이상 해결: 수강 기록을 지워도 강좌 정보는 `course`에 남는다.
- 갱신 이상: 아직 남아 있다! `student` 테이블에서 `dept_office`가 `dept_name`에 이행 종속되어 있으므로.

### 제3정규형 (3NF): 이행 종속 제거

> **정의:** 2NF를 만족하면서, 비키 컬럼이 다른 비키 컬럼을 통해 후보키에 **이행적으로(transitively)** 종속되는 경우가 없어야 한다.

더 정확히: **FD X → A가 성립하면 (1) X가 슈퍼키이거나, (2) A가 일부 후보키의 원소(prime attribute)여야 한다.**

`student` 테이블을 보자:

```
student_id  →  dept_name  →  dept_office
(후보키)       (비키)         (비키)

이행 종속: student_id → dept_name → dept_office
          즉 student_id → dept_office 는 dept_name을 '경유'해서 성립
```

`dept_name → dept_office`에서 결정자 `dept_name`은 `student` 테이블의 후보키가 아니다. 그러므로 3NF 위반. **해결: `dept_name → dept_office`를 새 테이블로 분리한다.**

```
department 테이블 (dept_name이 결정하는 것)
┌──────────────┬────────────┐
│dept_name(PK) │dept_office │    ← dept_name이 이제 이 테이블의 PK (후보키)
│ 컴퓨터공학    │ 공학관 301  │
│ 전자공학      │ 공학관 502  │
└──────────────┴────────────┘

student 테이블 (dept_office 제거, dept_name은 FK로 유지)
┌──────────┬────────────┬──────────────┐
│student_id│student_name│dept_name(FK) │    → department(dept_name) 참조
│ 1001     │ 김철수      │ 컴퓨터공학    │
│ 1002     │ 이영희      │ 컴퓨터공학    │
│ 1003     │ 박민수      │ 전자공학      │
└──────────┴────────────┴──────────────┘
```

**3NF 달성 후 갱신 이상 해결:** 학과 사무실 이전은 이제 `department` 테이블의 **한 행만** 고치면 된다. 모순이 원천적으로 불가능하다.

### 정규화 여정 전체 요약 다이어그램

```
1개 거대 테이블: enrollment_bad
(student_id, student_name, dept_name, dept_office,
 course_id, course_title, professor, grade)
│
│  이상현상: 갱신·삽입·삭제 이상 모두 존재
│
├── 1NF 검사: 원자값 확인 → 이미 만족
│
├── 2NF: 부분 종속 제거 (결정자가 복합키의 일부)
│   ├── student_id → student_name, dept_name  ─┐ student 테이블로 분리
│   ├── course_id  → course_title, professor   ─┤ course 테이블로 분리
│   └── (student_id, course_id) → grade        ─┘ enrollment에 남음
│   이상현상: 삽입·삭제 이상 해결, 갱신 이상 아직 존재
│
├── 3NF: 이행 종속 제거 (비키→비키 체인)
│   └── dept_name → dept_office  ─── department 테이블로 분리
│   이상현상: 세 가지 모두 해결
│
└── BCNF: 모든 결정자가 후보키?
    ├── department: dept_name(PK) → dept_office  ✓ 결정자가 PK
    ├── student: student_id(PK) → student_name, dept_name  ✓
    ├── course: course_id(PK) → course_title, professor  ✓
    └── enrollment: (student_id, course_id)(PK) → grade  ✓
    → 모든 결정자가 후보키  → BCNF 자동 달성!

최종 결과: 4개 테이블 (department, student, course, enrollment)
           중복·이상현상 0
```

## 2.6 보이스-코드 정규형 (BCNF)

> **정의:** 모든 FD X → A에서 X는 반드시 **후보키(슈퍼키)**여야 한다. (X가 슈퍼키가 아닌 FD는 모두 제거)

BCNF는 3NF의 강화판이다. 3NF는 "결정자가 슈퍼키가 아니어도, 종속자가 prime attribute(어떤 후보키의 원소)이면 허용한다"는 예외를 인정한다. BCNF는 이 예외 없이 "결정자는 무조건 슈퍼키"를 요구한다.

우리 예제에서는 3NF까지 정규화하자 자연히 BCNF가 달성됐다. BCNF가 3NF와 갈리는 것은 **복합 후보키가 겹치는(overlapping candidate keys)** 경우에 발생한다.

### BCNF와 3NF가 갈리는 예제: 고전적 '학생-과목-교수' 문제

다음 관계를 생각하자:

```
teaching(student, subject, teacher)

비즈니스 규칙:
  - 한 학생은 각 과목에서 정확히 한 교수에게 배운다
  - 각 교수는 정확히 하나의 과목만 담당한다
```

이 규칙에서 성립하는 FD:

```
FD1: (student, subject) → teacher    (학생+과목 → 교수)
FD2: teacher → subject               (교수 → 과목, 교수는 한 과목만 담당)
```

후보키를 폐포로 찾아보자:

```
{student, subject}+ = ?
  FD1 적용: teacher 추가
  FD2 적용: subject 이미 있음
  → {student, subject, teacher} = 전체 = 슈퍼키이며 최소 → 후보키 1

{student, teacher}+ = ?
  FD2 적용: subject 추가
  FD1 적용: teacher 이미 있음
  → {student, subject, teacher} = 전체 = 슈퍼키이며 최소 → 후보키 2
```

후보키가 두 개: `{student, subject}`와 `{student, teacher}`. 겹치는(overlapping) 후보키다.

**3NF 검사:**

FD2: `teacher → subject`. 결정자 `teacher`는 슈퍼키인가? 아니다. 그러면 종속자 `subject`는 prime attribute(어떤 후보키의 원소)인가? 후보키 `{student, subject}` 에 `subject`가 있으니 yes. 따라서 **3NF는 만족**.

**BCNF 검사:**

FD2: `teacher → subject`. 결정자 `teacher`는 슈퍼키인가? `{teacher}+`를 계산하면 `{teacher, subject}` — 전체 컬럼이 아니므로 슈퍼키가 아니다. 따라서 **BCNF 위반**.

```
teaching 테이블 샘플:
┌─────────┬─────────────┬──────────┐
│ student │ subject     │ teacher  │
│ 김철수   │ 데이터베이스  │ 박교수   │  ← 박교수=데이터베이스 중복
│ 이영희   │ 데이터베이스  │ 박교수   │  ← 박교수=데이터베이스 중복
│ 박민수   │ 자료구조     │ 이교수   │
└─────────┴─────────────┴──────────┘
"박교수는 데이터베이스 담당"이 두 번 반복 → 갱신 이상
```

**BCNF 분해:**

FD2(`teacher → subject`)를 분리한다:

```
teacher_subject 테이블          student_teacher 테이블
┌──────────┬─────────────┐      ┌─────────┬──────────┐
│teacher(PK)│ subject     │      │ student │ teacher  │
│ 박교수    │ 데이터베이스  │      │ 김철수   │ 박교수   │
│ 이교수    │ 자료구조     │      │ 이영희   │ 박교수   │
└──────────┴─────────────┘      │ 박민수   │ 이교수   │
                                 └─────────┴──────────┘
PK: teacher                      PK: (student, teacher)
```

이제 `teacher → subject`의 결정자 `teacher`가 `teacher_subject`의 PK(후보키)이므로 BCNF 만족.

## 2.7 BCNF와 3NF의 트레이드오프: 무손실 분해와 종속성 보존

분해가 "올바른" 분해인지 판단하는 두 가지 조건이 있다.

### 무손실 조인 분해 (Lossless-Join Decomposition)

분해된 테이블들을 다시 JOIN했을 때 **원본과 정확히 동일한 결과**가 나와야 한다. 원본보다 많은 행(스퓨리어스 튜플, spurious tuple)이 생기면 정보가 왜곡된 것이다.

**손실 분해 예 (잘못된 분해):**

```
원본 enrollment_bad 의 일부:
┌──────────┬──────────┬────────────┬───────┐
│student_id│course_id │dept_name   │grade  │
│ 1001     │ CS101    │ 컴퓨터공학  │ A     │
│ 1002     │ CS101    │ 컴퓨터공학  │ A     │
└──────────┴──────────┴────────────┴───────┘

잘못된 분해: R1(student_id, dept_name), R2(dept_name, course_id, grade)

R1:
┌──────────┬────────────┐
│student_id│dept_name   │
│ 1001     │ 컴퓨터공학  │
│ 1002     │ 컴퓨터공학  │
└──────────┴────────────┘

R2:
┌────────────┬──────────┬───────┐
│dept_name   │course_id │grade  │
│ 컴퓨터공학  │ CS101    │ A     │
│ 컴퓨터공학  │ CS101    │ A     │  ← 두 수강 기록이 학과로만 묶임
└────────────┴──────────┴───────┘

R1 ⋈ R2 (dept_name 기준):
┌──────────┬────────────┬──────────┬───────┐
│student_id│dept_name   │course_id │grade  │
│ 1001     │ 컴퓨터공학  │ CS101    │ A     │
│ 1002     │ 컴퓨터공학  │ CS101    │ A     │
│ 1001     │ 컴퓨터공학  │ CS101    │ A     │  ← 스퓨리어스 튜플!
│ 1002     │ 컴퓨터공학  │ CS101    │ A     │  ← 스퓨리어스 튜플!
└──────────┴──────────── ┴──────────┴───────┘
원본보다 많은 행 → 손실 분해 (정보 왜곡)
```

**무손실 분해 조건 (두 테이블로 분해 시):**

R을 R1과 R2로 분해했을 때 무손실이 되려면:

> R1 ∩ R2가 R1의 후보키이거나, R1 ∩ R2가 R2의 후보키여야 한다.

즉, **공통 컬럼이 어느 한쪽 테이블의 (후보)키여야** 한다.

```
올바른 분해 예:
R1 = student(student_id, student_name, dept_name)   공통: {student_id}
R2 = enrollment(student_id, course_id, grade)        공통이 R1의 PK

→ {student_id}가 student의 PK → 무손실 분해 ✓
```

### 종속성 보존 (Dependency Preservation)

분해 후에도 원래의 모든 FD를 **각 테이블 내에서 단독으로 검사할 수 있어야** 한다. 여러 테이블을 JOIN해야만 확인할 수 있는 FD가 남으면 종속성이 보존되지 않은 것이다. 종속성이 보존되지 않으면 제약을 관리하기 위해 매번 JOIN을 해야 하고, 성능 저하와 관리 부담이 커진다.

### 3NF vs BCNF: 어느 것을 선택해야 하나?

| | 3NF | BCNF |
|---|---|---|
| 무손실 분해 | 항상 가능 | 항상 가능 |
| 종속성 보존 | **항상 가능** | **보장 불가** |
| 남은 이상 현상 | 아주 드문 경우에 남을 수 있음 | 완전히 제거 |

앞서 본 `teaching(student, subject, teacher)` 예제에서:

```
BCNF 분해: teacher_subject(teacher, subject) + student_teacher(student, teacher)
  FD1: (student, subject) → teacher  — teacher_subject ⋈ student_teacher 로만 확인 가능
  FD2: teacher → subject             — teacher_subject 내에서 단독 확인 가능
  → FD1의 종속성이 보존되지 않음!
```

BCNF 분해 후 "한 학생이 같은 과목을 두 교수에게 배우지 않는다"는 FD1을 강제하려면 INSERT/UPDATE 시마다 JOIN 쿼리를 실행해야 한다. 이것은 비용이 크다.

**결론:**

> **실무 원칙: 대부분의 경우 3NF가 적절한 목표다.** BCNF까지 요구할 때 종속성 보존이 희생되면, 그 트레이드오프를 의식적으로 선택해야 한다. 3NF는 무손실 분해와 종속성 보존을 동시에 달성하는 알고리즘이 항상 존재한다.

## 2.8 4NF와 5NF: 다치 종속과 조인 종속

완전성을 위해 간략히 언급한다.

**다치 종속 (Multivalued Dependency, MVD)**

X →→ Y ("X가 Y를 다치적으로 결정한다"): X의 값이 정해지면 Y 값들의 **집합**이 정해지며, 이 집합은 나머지 애트리뷰트의 값과 독립적이다.

예: `professor_course_hobby(professor, course, hobby)` 에서 교수의 강의 과목과 취미가 독립적으로 여러 개라면 다치 종속이 발생한다. BCNF를 만족해도 이상 현상이 생길 수 있다.

**4NF:** BCNF + 비자명(non-trivial) 다치 종속에서 결정자가 슈퍼키여야 한다.

**5NF(PJNF, Project-Join Normal Form):** 조인 종속(join dependency)과 관련. 세 개 이상으로 분해했다가 JOIN했을 때 무손실인 조건.

4NF, 5NF는 이론적 완결성을 위해 존재하지만 실무에서는 3NF/BCNF로 충분한 경우가 대부분이다.

## 2.9 정규화 vs 비정규화: 언제 일부러 하지 않나?

정규화의 대가는 **JOIN**이다. 데이터를 4개 테이블로 쪼갰으니, 원래 형태로 보려면 JOIN으로 합쳐야 한다.

```sql
-- 수강 기록을 "사람이 읽을 수 있는" 형태로 조회하려면:
SELECT
    s.student_name,
    d.dept_office,
    c.course_title,
    c.professor,
    e.grade
FROM enrollment e
JOIN student    s ON s.student_id = e.student_id
JOIN department d ON d.dept_name  = s.dept_name
JOIN course     c ON c.course_id  = e.course_id
ORDER BY s.student_name, c.course_id;
```

정규화의 철학은 **"저장은 나눠서(중복 제거), 조회는 JOIN으로 합쳐서"**다.

**비정규화(denormalization):** 성능을 위해 **의도적으로 중복을 허용**하는 것. 예: 매일 수십만 건 조회되는 통계 화면에서, 4번 JOIN을 피하기 위해 미리 집계한 값을 별도 컬럼에 저장.

```
비정규화를 고려할 조건:
  1. 정규화된 스키마가 이미 존재하고
  2. EXPLAIN ANALYZE로 JOIN이 측정된 성능 병목임을 확인하고
  3. 중복 데이터 관리 비용(정합성 유지 로직)을 수용할 수 있을 때
```

**원칙: 일단 3NF까지 정규화하고, 측정된 성능 문제가 있을 때만 신중히 비정규화한다.** "더 빠를 것 같아서" 미리 비정규화하는 것은 갱신 이상을 다시 초대하는 것이다.

> **JPA와의 연결:** `@Entity` 하나가 보통 정규화된 테이블 하나에 대응한다. 정규화된 데이터를 `@ManyToOne`, `@OneToMany` 등의 연관관계로 조회할 때, Hibernate가 내부적으로 JOIN SQL을 발행한다. 그런데 Hibernate의 지연 로딩(lazy loading)이 N개의 연관 객체를 N번의 SELECT로 따로 조회하면 — 이것이 Spring 개념서 4.4절의 **N+1 문제**다. "정규화가 만든 JOIN 필요성"과 "ORM이 그것을 자동 번역하며 생기는 N+1"은 동전의 양면이다. → 자세한 내용은 [SPRING_개념서/README.md 4.4절](../../../spring/docs/SPRING_개념서/README.md)과 이 책 7장.

## ⚠️ 흔한 오해와 함정

**함정 1: "정규화하면 무조건 느려진다"는 미신**

JOIN이 추가된다는 것은 사실이지만, 현대 RDBMS(PostgreSQL 포함)는 조인을 매우 효율적으로 처리한다. 특히 **인덱스가 있는 컬럼(FK, PK)**으로 JOIN하는 경우 오버헤드는 극히 작다. 반면 비정규화로 테이블이 거대해지면 전체 스캔 비용이 훨씬 크다. "JOIN이 느리다"는 느낌은 대부분 **인덱스 부재** 또는 **잘못 작성된 JOIN**에서 나오는 것이다. → 5장(인덱싱), 6장(쿼리 처리)에서 확인할 것.

**함정 2: 2NF는 단일 컬럼 PK이면 자동으로 만족한다고 착각**

2NF의 "부분 종속 제거"는 복합 후보키가 있을 때만 문제가 된다. 단일 컬럼 후보키 테이블은 부분 종속 자체가 불가능해 1NF이면 자동으로 2NF다. 하지만 이것은 "2NF를 신경 쓸 필요 없다"는 뜻이 아니다. 복합 키가 생기는 순간(예: 많은 연결 테이블) 2NF 위반 가능성을 반드시 점검해야 한다.

**함정 3: "대충 정규화했다"고 생각하지만 실은 이행 종속이 남아 있음**

가장 흔한 패턴: "학생 테이블에 학과 이름과 학과 사무실을 같이 넣었다. 이 정도는 괜찮겠지." → 데이터가 쌓이면 학과 사무실 변경 시 수천 행을 업데이트해야 한다. 항상 `비키 → 비키` FD가 있는지 확인하자.

**함정 4: BCNF 분해 후 종속성 보존 여부를 확인하지 않음**

BCNF를 달성하기 위해 분해했더니, 원래의 중요한 비즈니스 규칙(FD)을 테이블 내에서 직접 검사할 수 없게 되는 경우가 있다(2.7절의 `teaching` 예제). 이 경우 트리거나 애플리케이션 레이어에서 제약을 관리해야 하는 부담이 생긴다. BCNF를 목표로 한다면 종속성 보존 여부를 반드시 확인하라.

**함정 5: 함수 종속을 "데이터를 보고" 결정한다**

FD는 데이터의 현재 스냅샷이 아니라 **비즈니스 규칙**에서 나온다. 지금 테이블에 `dept_name`이 중복 없이 들어있다고 해서 `dept_name → dept_office`가 성립하는 게 아니다. "학과명이 정해지면 사무실이 유일하게 정해진다는 규칙이 존재한다"는 것이 FD다. 미래 데이터를 포함해도 성립하는 규칙을 찾아야 한다.

**함정 6: NULL이 많은 컬럼을 그냥 둔다**

한 테이블에 선택적 정보(어떤 행에는 있고 어떤 행에는 없는)가 많다면, 그것은 그 테이블에 서로 다른 종류의 개체가 섞여 있다는 신호다. 정규화가 필요한 경우다.

## 연습문제

**문제 1 (FD와 폐포)**

다음 FD 집합이 주어진다.

```
R(A, B, C, D, E)
F = {
  AB → C,
  C → D,
  BD → E,
  E → A
}
```

(가) `{A, B}+` 를 계산하라.  
(나) `{B, C}+` 를 계산하라.  
(다) 후보키를 모두 찾아라.  
(라) `AB → E`가 성립함을 암스트롱 공리를 이용해 증명하라.

> 해설:
>
> (가) `{A, B}+`:
> ```
> closure = {A, B}
> AB → C 적용: {A, B, C}
> C → D 적용: {A, B, C, D}
> BD → E 적용 (B와 D 둘 다 있음): {A, B, C, D, E}
> E → A 적용: A 이미 있음
> → {A, B}+ = {A, B, C, D, E} = 전체
> → {A, B}는 슈퍼키이며 최소 → 후보키
> ```
>
> (나) `{B, C}+`:
> ```
> closure = {B, C}
> C → D 적용: {B, C, D}
> BD → E 적용: {B, C, D, E}
> E → A 적용: {A, B, C, D, E} = 전체
> → {B, C}는 슈퍼키이며 최소 → 후보키
> ```
>
> (다) 후보키: `{A, B}`, `{B, C}`. (B는 어떤 FD의 오른쪽에도 없으므로 모든 후보키에 반드시 포함. A나 C 중 하나와 조합 필요.)
>
> (라) AB → E 증명:
> ```
> 1. AB → C              (주어진 FD)
> 2. C → D               (주어진 FD)
> 3. AB → D              (이행성: 1, 2)
> 4. AB → BD             (증대성: AB → B는 반사성, AB → D는 3에서; 합집합)
> 5. BD → E              (주어진 FD)
> 6. AB → E              (이행성: 4, 5)
> ```

**문제 2 (정규화 실습 — 주문 스키마)**

다음 스키마가 있다. FD를 파악하고 BCNF까지 정규화하라. 각 단계를 보여라.

```
order_item(주문id, 고객id, 고객명, 고객주소, 상품id, 상품명, 단가, 수량)

비즈니스 규칙:
- 주문id는 고객id를 포함(주문은 반드시 한 고객에 속함)
- 한 고객은 여러 상품을 한 주문에 담을 수 있음
- 같은 상품의 단가는 항상 동일 (상품마다 단가가 고정)
- 고객주소는 고객마다 하나
- 수량은 (주문id, 상품id) 조합이 결정
```

> 힌트: FD를 먼저 나열하라. 후보키를 찾아라. 부분 종속 → 2NF, 이행 종속 → 3NF 순서로.

> 해설:
>
> FD 목록:
> ```
> 주문id → 고객id                    (한 주문은 한 고객)
> 고객id → 고객명, 고객주소          (고객 정보)
> 상품id → 상품명, 단가              (상품 정보)
> (주문id, 상품id) → 수량            (주문별 상품 수량)
> ```
>
> 후보키: `(주문id, 상품id)`
>
> 1NF: 각 칸에 원자값 → 이미 만족 가정
>
> 2NF (부분 종속 제거):
> ```
> 주문id → 고객id   (주문id는 복합키의 일부 → 부분 종속)
> 상품id → 상품명, 단가  (상품id는 복합키의 일부 → 부분 종속)
> (주문id, 상품id) → 수량  (완전 종속, OK)
>
> 분리:
> order(주문id PK, 고객id FK)
> product(상품id PK, 상품명, 단가)
> order_item(주문id FK, 상품id FK, 수량)   PK=(주문id, 상품id)
> ```
>
> 3NF (이행 종속 제거):
> ```
> order 테이블: 주문id → 고객id → 고객명, 고객주소 (이행 종속!)
> 분리:
> customer(고객id PK, 고객명, 고객주소)
> order(주문id PK, 고객id FK)
> ```
>
> 최종 스키마:
> ```sql
> CREATE TABLE customer (
>     customer_id BIGSERIAL PRIMARY KEY,
>     name        TEXT NOT NULL,
>     address     TEXT
> );
> CREATE TABLE product (
>     product_id  BIGSERIAL PRIMARY KEY,
>     name        TEXT NOT NULL,
>     price       NUMERIC(10,2) NOT NULL CHECK (price >= 0)
> );
> CREATE TABLE orders (
>     order_id    BIGSERIAL PRIMARY KEY,
>     customer_id BIGINT NOT NULL REFERENCES customer(customer_id)
> );
> CREATE TABLE order_item (
>     order_id    BIGINT NOT NULL REFERENCES orders(order_id) ON DELETE CASCADE,
>     product_id  BIGINT NOT NULL REFERENCES product(product_id),
>     quantity    INT    NOT NULL CHECK (quantity > 0),
>     PRIMARY KEY (order_id, product_id)
> );
> ```

**문제 3 (BCNF와 종속성 보존)**

다음 관계와 FD가 있다.

```
R(강의실, 시간대, 과목, 교수)

FD:
F1: (강의실, 시간대) → 과목
F2: (시간대, 교수)  → 강의실
F3: (강의실, 시간대) → 교수
```

(가) 후보키를 모두 찾아라.  
(나) R이 BCNF인지 판단하라. BCNF를 위반한다면 어느 FD에서 위반되는가?  
(다) BCNF를 달성하도록 무손실 분해를 수행하라. 종속성이 보존되는가?

> 힌트: 후보키가 여러 개이고 서로 겹치는 경우다.

> 해설:
>
> (가) 폐포 계산:
> ```
> {강의실, 시간대}+ = {강의실, 시간대, 과목, 교수} = 전체 → 슈퍼키
>   최소? {강의실}+ = {강의실} ≠ 전체. {시간대}+ = {시간대} ≠ 전체 → 최소 → 후보키 1
>
> {시간대, 교수}+ = ?
>   F2: {강의실, 시간대, 교수}
>   F3: 과목 추가 → {강의실, 시간대, 과목, 교수} = 전체 → 후보키 2
>
> {시간대, 과목}+ = ?
>   직접 결정하는 FD 없음 → 슈퍼키 아님 → 후보키 아님
>
> 후보키: {강의실, 시간대}, {시간대, 교수}
> ```
>
> (나) F2: `(시간대, 교수) → 강의실`. 결정자 `(시간대, 교수)`는 후보키 2이므로 OK.
> F1: `(강의실, 시간대) → 과목`. 결정자 `(강의실, 시간대)`는 후보키 1이므로 OK.
> F3: `(강의실, 시간대) → 교수`. 결정자 `(강의실, 시간대)`는 후보키 1이므로 OK.
> → 모든 결정자가 후보키 → R은 이미 BCNF!
>
> (다) 이미 BCNF이므로 추가 분해 불필요. 이 예제는 "겹치는 후보키가 있어도 BCNF를 만족할 수 있다"를 보여준다.

**문제 4 (무손실 분해 검증)**

`enrollment_bad`를 다음 두 가지 방식으로 분해했다. 각각 무손실 분해인지 판단하라.

```
분해 A:
  R1(student_id, student_name, dept_name, dept_office)
  R2(student_id, course_id, course_title, professor, grade)

분해 B:
  R1(student_id, student_name, dept_name)
  R2(dept_name, dept_office)
  R3(student_id, course_id, grade)
  R4(course_id, course_title, professor)
```

> 힌트: 공통 컬럼이 어느 한쪽 테이블의 후보키인지 확인하라.

> 해설:
>
> 분해 A:
> R1 ∩ R2 = {student_id}. student_id는 R1의 후보키인가? R1에서 student_id → student_name, dept_name이지만 student_id → dept_office는 이행 종속을 통해 성립. student_id가 R1의 PK(후보키) 역할을 한다. → 무손실 ✓
>
> 분해 B:
> R1 ∩ R2 = {dept_name}. dept_name이 R2의 PK → 무손실 ✓
> R1 ∩ R3 = {student_id}. student_id가 R1의 PK → 무손실 ✓
> R3 ∩ R4 = {course_id}. course_id가 R4의 PK → 무손실 ✓
> → 전체 분해가 무손실 ✓

**문제 5 (종합 설계)**

"온라인 강의 플랫폼"을 설계한다. 다음 요구사항을 읽고 FD를 도출하고, 3NF로 정규화된 PostgreSQL 16 스키마 DDL을 작성하라.

```
요구사항:
- 강사(instructor)는 여러 강좌(course)를 개설할 수 있다.
- 학습자(learner)는 여러 강좌를 수강 신청(enrollment)할 수 있다.
- 각 강좌는 하나의 카테고리(category)에 속한다.
- 카테고리에는 이름과 설명이 있다.
- 수강 신청에는 신청일, 완료 여부(completed)가 있다.
- 강사에게는 이름, 이메일, 전문 분야(specialty)가 있다.
- 강좌에는 제목, 설명, 가격, 난이도(difficulty)가 있다.
- 학습자에게는 이름, 이메일이 있다.
- 같은 학습자가 같은 강좌에 여러 번 등록할 수 없다.
- 이메일은 강사와 학습자 각각의 시스템에서 유일하다.
```

> 힌트: 개체: instructor, learner, course, category. 관계: enrollment. 이행 종속을 주의하라(course → category_id → category 설명).

> 해설 예시:
> ```sql
> CREATE TABLE category (
>     category_id   BIGSERIAL PRIMARY KEY,
>     name          TEXT UNIQUE NOT NULL,
>     description   TEXT
> );
> CREATE TABLE instructor (
>     instructor_id BIGSERIAL PRIMARY KEY,
>     email         TEXT UNIQUE NOT NULL,
>     name          TEXT NOT NULL,
>     specialty     TEXT
> );
> CREATE TABLE course (
>     course_id     BIGSERIAL PRIMARY KEY,
>     instructor_id BIGINT NOT NULL REFERENCES instructor(instructor_id),
>     category_id   BIGINT NOT NULL REFERENCES category(category_id),
>     title         TEXT NOT NULL,
>     description   TEXT,
>     price         NUMERIC(10,2) NOT NULL CHECK (price >= 0),
>     difficulty    TEXT CHECK (difficulty IN ('입문', '초급', '중급', '고급'))
> );
> CREATE TABLE learner (
>     learner_id    BIGSERIAL PRIMARY KEY,
>     email         TEXT UNIQUE NOT NULL,
>     name          TEXT NOT NULL
> );
> CREATE TABLE enrollment (
>     learner_id    BIGINT NOT NULL REFERENCES learner(learner_id) ON DELETE CASCADE,
>     course_id     BIGINT NOT NULL REFERENCES course(course_id) ON DELETE CASCADE,
>     enrolled_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
>     completed     BOOLEAN NOT NULL DEFAULT FALSE,
>     PRIMARY KEY (learner_id, course_id)
> );
> ```

## 요약

이 장에서 다룬 핵심 내용:

| 개념 | 핵심 내용 |
|------|-----------|
| 이상 현상 | 갱신·삽입·삭제 이상 — 서로 다른 사실을 한 테이블에 섞어서 발생 |
| 함수 종속 FD | X → Y: X가 정해지면 Y가 유일하게 정해짐. 결정자=X, 종속자=Y |
| 암스트롱 공리 | 반사·증대·이행 (+ 합집합·분해·유사이행). FD를 올바르게 추론하는 규칙 |
| 폐포 X+ | X가 함수적으로 결정할 수 있는 모든 컬럼 집합. 후보키 찾기에 사용 |
| 1NF | 모든 값이 원자적(원자값, 반복 그룹 없음) |
| 2NF | 1NF + 복합키의 일부에만 종속된 FD(부분 종속) 제거 |
| 3NF | 2NF + 비키→비키 체인(이행 종속) 제거. X→A에서 X가 슈퍼키이거나 A가 prime attribute |
| BCNF | 모든 FD의 결정자가 슈퍼키. 3NF의 겹치는 후보키 예외 없음 |
| 무손실 분해 | 분해 후 JOIN = 원본. 공통 컬럼이 어느 한쪽의 (후보)키여야 성립 |
| 종속성 보존 | 분해 후에도 원래 FD를 각 테이블 내에서 단독 검사 가능해야. 3NF는 항상 가능, BCNF는 보장 불가 |
| 비정규화 | 측정된 성능 문제 있을 때만 의도적으로 중복 허용. 먼저 정규화하라. |

**JPA와의 연결:**

- 정규화된 테이블 하나 = `@Entity` 하나
- 테이블 간 FK 참조 = `@ManyToOne`, `@OneToMany`, `@ManyToMany`
- 정규화로 분리된 테이블들을 JOIN하는 과정이 지연 로딩 시 N+1로 나타남
- Spring 개념서 4.2(Entity), 4.3(JpaRepository), 4.4(연관관계/N+1) → [SPRING_개념서/README.md 4장](../../../spring/docs/SPRING_개념서/README.md)
