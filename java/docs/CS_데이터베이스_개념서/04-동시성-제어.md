## 도입: 왜 동시성 제어가 필요한가

3장에서 트랜잭션은 "혼자 도는 것처럼 보이게"라는 격리성(Isolation)을 ACID의 한 성질로 배웠다. 그런데 현실의 데이터베이스는 수백, 수천 개의 트랜잭션이 동시에 실행된다. 각 트랜잭션을 진짜로 하나씩 순서대로 실행하면(직렬 실행) 격리성은 완벽하지만, 처리량(throughput)이 형편없어진다. 동시에 실행하면 처리량은 올라가지만, 트랜잭션들이 서로 간섭하여 잘못된 결과를 낼 수 있다.

**동시성 제어(Concurrency Control)**는 이 둘 사이에서 균형을 잡는 기법의 총체다. 단순히 "락을 건다"는 수준이 아니라, 어떤 실행 순서가 올바른지 정의하고(직렬가능성), 그것을 보장하는 메커니즘(2PL, MVCC, 타임스탬프 순서화)을 제공하는 것이다.

Spring 개념서 4.6절은 격리 수준 매트릭스와 MVCC를 결과 중심으로 소개했다. 이 장은 **그 매트릭스가 왜 그렇게 생겼는지**, 그 뒤에 있는 두 메커니즘(락과 MVCC)을 이론과 실패 시나리오로 파헤친다.

## 4.1 이상 현상 (Anomalies) — 동시 실행이 만들어내는 문제

동시 트랜잭션이 서로 간섭할 때 발생하는 **세 가지 이상 현상**을 타임라인으로 살펴보자. 세션 A와 세션 B가 동시에 실행된다.

### Dirty Read — 아직 커밋되지 않은 값을 읽다

```
[Dirty Read 타임라인]

시각   세션 A (T1)                         세션 B (T2)
────────────────────────────────────────────────────────────
 t1    BEGIN;
 t2    UPDATE account
       SET balance = 999999
       WHERE id = 'A';
       -- 아직 COMMIT 안 함! (임시 값)
 t3                                         BEGIN;
 t4                                         SELECT balance
                                            FROM account
                                            WHERE id = 'A';
                                            -- => 999999 ← 커밋 안 된 값!
 t5    ROLLBACK;
       -- balance는 원래 100000으로 복원
 t6                                         -- B는 존재한 적 없는 999999를
                                            -- 이미 읽어버렸다. 재앙!
```

**왜 위험한가?** T2가 읽은 999999는 T1이 ROLLBACK하면서 사라진 값이다. T2가 이 값을 근거로 의사결정을 내렸다면(예: "잔액이 충분하다" → 대출 승인), 근거 자체가 없어진 것이다.

---

### Non-repeatable Read — 같은 행을 두 번 읽었는데 값이 다르다

```
[Non-repeatable Read 타임라인]

시각   세션 A (T1)                         세션 B (T2)
────────────────────────────────────────────────────────────
 t1    BEGIN;
 t2    SELECT balance
       FROM account WHERE id = 'A';
       -- => 100000
 t3                                         BEGIN;
 t4                                         UPDATE account
                                            SET balance = 55000
                                            WHERE id = 'A';
 t5                                         COMMIT; ← 커밋 완료
 t6    SELECT balance
       FROM account WHERE id = 'A';
       -- => 55000  (!)
       -- 같은 트랜잭션에서 같은 쿼리인데 결과가 다름
 t7    COMMIT;
```

**왜 위험한가?** T1이 "잔액 확인 → 계산 → 재확인" 패턴을 사용할 때, t2와 t6 사이에 T2가 값을 바꾸면 두 SELECT가 다른 값을 반환한다. 계산의 전제가 무너진다.

**Non-repeatable Read vs Dirty Read의 차이**: Dirty Read는 커밋되지 않은 값을 읽는 것, Non-repeatable Read는 **커밋된** 변경으로 인해 같은 행의 값이 달라지는 것이다.

---

### Phantom Read — 같은 조건인데 행 집합이 달라지다

```
[Phantom Read 타임라인]

시각   세션 A (T1)                         세션 B (T2)
────────────────────────────────────────────────────────────
 t1    BEGIN;
 t2    SELECT COUNT(*)
       FROM employee
       WHERE age > 20;
       -- => 5 (행 5개)
 t3                                         BEGIN;
 t4                                         INSERT INTO employee
                                            (name, age) VALUES
                                            ('신입', 25);
 t5                                         COMMIT; ← 새 행 추가됨
 t6    SELECT COUNT(*)
       FROM employee
       WHERE age > 20;
       -- => 6 (!)
       -- 행이 하나 늘었다. 유령(Phantom)이 나타났다.
 t7    COMMIT;
```

**Non-repeatable Read vs Phantom Read의 차이:**
- Non-repeatable Read: **기존 행의 값**이 바뀌는 것
- Phantom Read: **행 집합(개수, 구성)**이 바뀌는 것 — 새 행이 생기거나 기존 행이 사라짐

이 차이는 해결 방법이 다르기 때문에 중요하다. Non-repeatable Read는 해당 행에 락을 걸어 막을 수 있지만, Phantom Read는 "범위" 전체를 보호해야 하므로 더 강한 격리가 필요하다.

## 4.2 격리 수준 — 이상 현상을 얼마나 허용할 것인가

SQL 표준은 네 가지 격리 수준을 정의한다. 격리 수준을 낮출수록 더 많은 이상 현상을 허용하고, 더 높은 동시성(처리량)을 얻는다.

```
[격리 수준 매트릭스]

격리 수준            더티 리드   반복 불가능한 읽기   팬텀 리드   동시성
─────────────────────────────────────────────────────────────────────────
READ UNCOMMITTED     발생 가능    발생 가능           발생 가능   최고 ↑
READ COMMITTED  ★    방지됨       발생 가능           발생 가능   높음
REPEATABLE READ      방지됨       방지됨              발생 가능*  중간
SERIALIZABLE         방지됨       방지됨              방지됨      최저 ↓

★ PostgreSQL 기본값
* PostgreSQL의 REPEATABLE READ는 MVCC 스냅샷으로 Phantom Read도 대부분 방지함
  (SQL 표준보다 더 강한 보장)
```

### 격리 수준 설정 방법

```sql
-- 트랜잭션 시작과 동시에 설정
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM account WHERE id = 'A';
COMMIT;

-- 또는 트랜잭션 시작 시 한 줄로
BEGIN ISOLATION LEVEL SERIALIZABLE;

-- 세션 전체에 기본값으로 설정
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 현재 세션의 격리 수준 확인
SHOW transaction_isolation;
```

### 각 격리 수준의 의미

**READ UNCOMMITTED**: 다른 트랜잭션이 아직 커밋하지 않은 변경도 읽는다. PostgreSQL에서는 실제로 구현되지 않고 READ COMMITTED와 동일하게 동작한다.

**READ COMMITTED** (PostgreSQL 기본): 매 `SELECT`마다 그 시점에 커밋된 최신 값을 읽는다. Dirty Read는 방지되지만, 같은 트랜잭션 안의 두 `SELECT`가 다른 값을 볼 수 있다(Non-repeatable Read).

**REPEATABLE READ**: 트랜잭션 시작 시점의 스냅샷을 트랜잭션 내내 고정한다. 같은 쿼리는 항상 같은 결과를 반환한다. PostgreSQL의 구현(MVCC)은 Phantom Read까지 대부분 방지한다.

**SERIALIZABLE**: 동시에 실행되는 트랜잭션들이 마치 하나씩 순서대로 실행된 것과 동일한 결과를 보장한다. PostgreSQL은 **SSI(Serializable Snapshot Isolation)**로 구현하며 Write Skew(쓰기 왜곡)까지 잡아낸다. (4.5절 참조)

## 4.3 스케줄과 직렬가능성 — "올바른 동시 실행"의 정의

동시성 제어는 "어떤 실행 순서가 올바른가"를 먼저 정의해야 한다. 이를 위해 **스케줄(schedule)**과 **직렬가능성(serializability)**이라는 개념이 필요하다.

### 스케줄

**스케줄(schedule)**은 여러 트랜잭션의 연산이 실제로 실행된 순서다.

```
T1: R(A), W(A), R(B), W(B)
T2: R(A), W(A)

직렬 스케줄(serial schedule):
  T1의 모든 연산이 끝난 후 T2가 시작
  R1(A), W1(A), R1(B), W1(B), R2(A), W2(A)
  또는
  R2(A), W2(A), R1(A), W1(A), R1(B), W1(B)

동시 스케줄(concurrent schedule):
  두 트랜잭션의 연산이 뒤섞여 실행
  R1(A), R2(A), W1(A), R1(B), W2(A), W1(B)
```

직렬 스케줄은 항상 올바르다(트랜잭션들이 독립적으로 실행되므로). 동시 스케줄은 성능은 좋지만 올바름을 보장할 수 없다.

**직렬가능성(serializability)**은 동시 스케줄의 결과가 어떤 직렬 스케줄의 결과와 동일하면 그 동시 스케줄을 "올바르다"고 정의하는 것이다.

### 충돌 연산 (Conflicting Operations)

두 연산이 **충돌(conflict)**한다는 것은, 순서를 바꾸면 결과가 달라질 수 있다는 뜻이다. 충돌의 조건:
1. 서로 다른 트랜잭션의 연산
2. 같은 데이터 항목에 접근
3. 적어도 하나가 쓰기 연산

```
[충돌 연산 유형]

유형               T_i 연산   T_j 연산   충돌 여부
──────────────────────────────────────────────────
Read-Write(RW)     R(X)       W(X)       충돌 ○
Write-Read(WR)     W(X)       R(X)       충돌 ○  ← Dirty Read의 원인
Write-Write(WW)    W(X)       W(X)       충돌 ○  ← 갱신 손실의 원인
Read-Read(RR)      R(X)       R(X)       충돌 ✕  (순서 바꿔도 동일)
```

### 충돌 직렬가능성과 우선 그래프

**충돌 직렬가능성(conflict serializability)**은 충돌 연산들의 상대적 순서를 바꾸지 않고 비충돌 연산만 교환하여 직렬 스케줄로 변환 가능한지를 따진다. 이를 확인하는 도구가 **우선 그래프(precedence graph / serialization graph)**다.

**우선 그래프 작성법**:
1. 각 트랜잭션을 노드로 그린다.
2. T_i의 연산이 T_j의 충돌 연산보다 먼저 실행되면 T_i → T_j 화살표를 추가한다.
3. 그래프에 사이클이 없으면 충돌 직렬가능 스케줄이다.

**예제 1: 직렬가능한 스케줄**

```
스케줄: R1(A), R2(A), W1(A), R1(B), W2(A), W1(B)

충돌 관계 분석:
  W1(A)가 R2(A) 이후  → T2 → T1 이 아니라, T2의 R2(A)는 W1(A) 이전 → T1이 W1(A)로 먼저 씀
  정확히:
  - R2(A) at t2, W1(A) at t3: R2가 먼저 → T2→T1? No: W1이 R2 뒤에 있음 → T1의 W1이 T2의 R2 이후
    이는 T2가 A를 읽은 후 T1이 A를 씀: T2 먼저(읽기), T1 나중(쓰기) → T2 → T1
  - W1(A) at t3, W2(A) at t5: W1이 먼저 → T1 → T2
  - W1(A)가 W2(A) 이전 → T1 → T2

우선 그래프:
  T2 ──→ T1
  T1 ──→ T2  ← 사이클!
```

이 경우 사이클이 있으므로 충돌 직렬가능하지 않다.

**예제 2: 직렬가능한 스케줄**

```
스케줄: R1(A), W1(A), R2(A), W2(A), R1(B), W1(B)

충돌 관계:
  W1(A) at t2, R2(A) at t3: W1 먼저 → T1 → T2
  W1(A) at t2, W2(A) at t4: W1 먼저 → T1 → T2
  (R1(B), W1(B)는 T2와 충돌 없음)

우선 그래프:
  T1 ──→ T2   (사이클 없음 ✓)

결론: 충돌 직렬가능. T1 → T2 순으로 직렬 실행한 것과 동일한 결과.
```

```
[우선 그래프 예제 2]

   ┌────┐        ┌────┐
   │ T1 │ ──────▶│ T2 │
   └────┘        └────┘
   
   사이클 없음 → 충돌 직렬가능 ✓
   직렬 순서: T1, T2
```

### 뷰 직렬가능성 (View Serializability)

**뷰 직렬가능성(view serializability)**은 충돌 직렬가능성보다 더 넓은 개념이다. 세 조건을 만족하면 뷰 직렬가능하다고 한다:

1. **초기 읽기**: 스케줄에서 항목 X를 처음 읽는 트랜잭션이 직렬 스케줄에서도 같다.
2. **읽기 소스**: 스케줄에서 T_i가 T_j의 쓰기 결과를 읽는다면, 직렬 스케줄에서도 동일하다.
3. **최종 쓰기**: 각 항목에 대한 마지막 쓰기가 두 스케줄에서 같은 트랜잭션이다.

충돌 직렬가능 ⊂ 뷰 직렬가능이다. 뷰 직렬가능하지만 충돌 직렬가능하지 않은 스케줄이 존재한다. 그러나 뷰 직렬가능성 테스트는 NP-hard이므로 실무에서는 충돌 직렬가능성(우선 그래프)을 주로 사용한다.

## 4.4 락 기반 동시성 제어

### 공유 락(S)과 배타 락(X)

**락(lock)**은 트랜잭션이 데이터를 접근하기 전에 획득해야 하는 권한이다.

- **공유 락(Shared lock, S-lock)**: 읽기 락. 여러 트랜잭션이 동시에 보유 가능.
- **배타 락(Exclusive lock, X-lock)**: 쓰기 락. 오직 하나의 트랜잭션만 보유 가능.

```
[락 호환성 표]

               요청하는 락
             S(공유)   X(배타)
기존 락  S   허용 ✓    대기 ✗
보유 시  X   대기 ✗    대기 ✗
(없음)   -   허용 ✓    허용 ✓

읽기(S)끼리만 공존 가능.
쓰기(X)는 어떤 것과도 공존 불가.
```

```sql
-- 공유 락(S): 다른 세션의 읽기는 허용, 쓰기는 차단
SELECT * FROM account WHERE id = 'A' FOR SHARE;

-- 배타 락(X): 다른 세션의 읽기/쓰기 모두 차단
SELECT * FROM account WHERE id = 'A' FOR UPDATE;

-- NOWAIT: 락 획득 불가 시 즉시 오류 반환 (대기 안 함)
SELECT * FROM account WHERE id = 'A' FOR UPDATE NOWAIT;

-- SKIP LOCKED: 락이 걸린 행은 건너뜀 (큐 처리에 유용)
SELECT * FROM account WHERE status = 'PENDING'
FOR UPDATE SKIP LOCKED LIMIT 1;
```

### 2PL (2-Phase Locking) — 직렬가능성의 보장

**2단계 락킹(2-Phase Locking, 2PL)**은 충돌 직렬가능성을 보장하는 락 프로토콜이다.

**규칙**: 트랜잭션은 두 단계로 나뉜다.
1. **확장 단계(Growing phase)**: 락을 획득만 할 수 있다 (해제 금지)
2. **수축 단계(Shrinking phase)**: 락을 해제만 할 수 있다 (새 락 획득 금지)

```
[2PL 락 개수 그래프]

락 개수
   │
 3 │         ★ (잠금 지점)
   │        /│\
 2 │       / │ \
   │      /  │  \
 1 │     /   │   \
   │    /    │    \
 0 │───/─────┼─────\──────▶ 시각
   │ 확장단계 │수축단계
             │
       "한 번 풀기 시작하면 새 락 안 검"
```

**왜 2PL이 직렬가능성을 보장하는가?** 락 해제 후 새 락을 얻지 못하므로, 락을 가장 늦게 놓는 시점(잠금 지점, lock point)의 순서가 직렬 순서와 일치한다. 이 순서가 우선 그래프에서 사이클이 없는 순서를 만든다.

```sql
-- 2PL 흉내: 두 계좌 이체 시 두 락을 모두 잡은 후 작업
BEGIN;
    SELECT * FROM account WHERE id = 'A' FOR UPDATE;  -- X-락 획득
    SELECT * FROM account WHERE id = 'B' FOR UPDATE;  -- X-락 획득
    -- 이 후부터는 새 락 획득 없이 작업
    UPDATE account SET balance = balance - 30000 WHERE id = 'A';
    UPDATE account SET balance = balance + 30000 WHERE id = 'B';
COMMIT;  -- 락 해제
```

### Strict 2PL / Rigorous 2PL

기본 2PL에는 문제가 있다. 수축 단계에서 락을 일찍 해제하면, 다른 트랜잭션이 그 데이터를 읽거나 써서 연쇄 롤백(cascading abort)이 발생할 수 있다.

- **Strict 2PL**: 배타 락(X-lock)을 커밋/롤백 시까지 보유. 읽기 락은 일찍 해제 가능.
- **Rigorous 2PL**: 모든 락(S, X)을 커밋/롤백 시까지 보유. 연쇄 롤백 완전 방지.

```
[Cascading Abort 예시 — 기본 2PL의 위험]

T1: W(A), 락해제(A), ......., ROLLBACK
T2:           R(A), W(A), COMMIT

→ T2가 T1이 쓰고 아직 커밋 안 한 값을 읽음(Dirty Read)
  T1이 롤백되면 T2도 롤백되어야 함 → 연쇄(cascading) 롤백

[Strict 2PL로 방지]
T1: W(A), ............., ROLLBACK, 락해제(A)
T2:                                          R(A)  ← T1 롤백 후에야 읽기 가능
→ T2는 항상 확정된 값만 읽음 → 연쇄 롤백 없음
```

PostgreSQL의 락은 실질적으로 Rigorous 2PL에 가깝게 동작한다. 트랜잭션 종료 시 모든 락이 해제된다.

### 교착 상태 (Deadlock)

2PL의 치명적인 문제가 **교착 상태(Deadlock)**다. 두 트랜잭션이 서로가 보유한 락을 기다리면서 둘 다 진행하지 못하는 상황이다.

```
[교착 상태 시나리오]

시각   세션 A (T1)                          세션 B (T2)
─────────────────────────────────────────────────────────
 t1    BEGIN;
 t2    SELECT * FROM account
       WHERE id = 'A' FOR UPDATE;           BEGIN;
       -- A 행 X-락 획득 ✓
 t3                                         SELECT * FROM account
                                            WHERE id = 'B' FOR UPDATE;
                                            -- B 행 X-락 획득 ✓
 t4    SELECT * FROM account
       WHERE id = 'B' FOR UPDATE;
       -- B 행 대기 중... (T2가 B를 보유)
 t5                                         SELECT * FROM account
                                            WHERE id = 'A' FOR UPDATE;
                                            -- A 행 대기 중... (T1이 A를 보유)
 t6    ← T1과 T2가 서로를 기다리며 영원히 대기 →

       PostgreSQL이 자동 감지:
       ERROR: deadlock detected
       DETAIL: Process 12345 waits for ShareLock on transaction 67890
               Process 67890 waits for ShareLock on transaction 12345
       → 하나를 강제 종료 (롤백)하여 다른 쪽이 진행 가능
```

```sql
-- PostgreSQL 데드락 설정
SHOW deadlock_timeout;  -- 기본 1000ms: 이 시간 이상 대기 후 데드락 감지 실행
```

**교착 상태 예방법**:
1. **항상 같은 순서로 잠그기**: 계좌 이체 시 항상 `id가 작은 계좌를 먼저` 잠그는 등 일정한 순서 유지
2. **한 번에 모든 필요한 락 획득**: 트랜잭션 시작 시 필요한 모든 데이터를 미리 잠금
3. **타임아웃 설정**: 너무 오래 대기하면 포기하고 재시도

```sql
-- 교착 방지: 항상 id 작은 것 먼저 잠그기
BEGIN;
-- A='A', B='B'이면 'A' < 'B'이므로 A부터
SELECT * FROM account WHERE id IN ('A', 'B') ORDER BY id FOR UPDATE;
UPDATE account SET balance = balance - 30000 WHERE id = 'A';
UPDATE account SET balance = balance + 30000 WHERE id = 'B';
COMMIT;
```

## 4.5 타임스탬프 순서화 — 락 없이 순서를 강제하다

락 기반 프로토콜과 전혀 다른 접근법이 **타임스탬프 순서화(Timestamp Ordering, TO)**다.

### 개념

각 트랜잭션이 시작될 때 고유한 **타임스탬프(TS)**를 부여한다. 데이터 항목 X마다 두 값을 유지한다:
- `read_TS(X)`: X를 읽은 트랜잭션 중 가장 큰 타임스탬프
- `write_TS(X)`: X를 쓴 트랜잭션 중 가장 큰 타임스탬프

**프로토콜 규칙**:

```
트랜잭션 Ti (타임스탬프 TS(Ti))가 X에 접근할 때:

[읽기 요청 R(X)]
  if TS(Ti) < write_TS(X):
      → Ti보다 나중 트랜잭션이 이미 X를 썼다.
        Ti의 읽기가 너무 늦었음 → Ti를 abort하고 재시작
  else:
      → 읽기 허용. read_TS(X) = max(read_TS(X), TS(Ti))

[쓰기 요청 W(X)]
  if TS(Ti) < read_TS(X):
      → Ti보다 나중 트랜잭션이 이미 X를 읽었다.
        Ti의 쓰기로 그 읽기가 틀려짐 → Ti를 abort하고 재시작
  if TS(Ti) < write_TS(X):
      → Ti보다 나중 트랜잭션이 이미 X를 썼다.
        Ti의 쓰기는 덮어쓸 필요 없음 (Thomas Write Rule)
  else:
      → 쓰기 허용. write_TS(X) = TS(Ti)
```

### 토마스 쓰기 규칙 (Thomas Write Rule)

위 규칙에서 `TS(Ti) < write_TS(X)`인 경우, 기본 TO는 Ti를 abort한다. 그러나 **토마스 쓰기 규칙(Thomas Write Rule)**은 이 쓰기를 그냥 **무시(skip)**해도 된다고 완화한다. 어차피 더 나중 트랜잭션이 이미 덮어써서, Ti의 쓰기가 최종 결과에 영향이 없기 때문이다. 이로써 불필요한 abort를 줄인다.

### 락 vs 타임스탬프 순서화 비교

```
[락 기반 vs 타임스탬프 순서화]

              락 기반 (2PL)              타임스탬프 순서화 (TO)
─────────────────────────────────────────────────────────────────
대기 방식    충돌 시 락 대기(blocking)   충돌 시 abort+재시작
교착 상태    발생 가능 (감지/예방 필요)  교착 없음 (대기 자체가 없음)
기아(starvation) 발생 가능             abort 루프 가능
직렬 순서    락 잠금 지점 순서           타임스탬프 순서
오버헤드     락 테이블 관리              TS/read_TS/write_TS 관리
실무 사용    범용 (대부분 RDBMS)         특수 목적 / 분산 시스템
```

PostgreSQL은 락 기반과 MVCC를 주로 사용하며 순수 TO는 구현하지 않는다. 그러나 MVCC의 스냅샷 메커니즘은 타임스탬프 순서화의 아이디어를 내포한다.

## 4.6 MVCC — 읽기와 쓰기가 서로를 막지 않는다

### 락의 근본 문제

2PL에서 읽기(S-락)와 쓰기(X-락)는 서로를 차단한다. 읽기가 많은 워크로드에서 쓰기 트랜잭션이 읽기를 모두 막아 성능이 저하된다. 반대로 쓰기가 잦으면 읽기가 자주 대기한다.

**"읽기가 쓰기를 막고, 쓰기가 읽기를 막는다"** — 이것이 락의 근본 비용이다.

### MVCC의 아이디어

**MVCC(Multi-Version Concurrency Control, 다중 버전 동시성 제어)**의 핵심 아이디어는 단순하다: **기존 데이터를 덮어쓰지 말고, 새 버전을 추가한다.**

각 트랜잭션은 자신이 시작할 때의 "스냅샷(snapshot)"을 보게 된다. 누군가 데이터를 변경해도 이전 버전이 남아 있으므로, 오래된 트랜잭션은 이전 버전을 계속 읽을 수 있다.

```
[MVCC 버전 관리 예시]

account 행 A의 버전 이력:

버전   balance   생성 트랜잭션   삭제 트랜잭션   가시성
────────────────────────────────────────────────────────────
v1     100000    txn_id=10      txn_id=20      txn_id < 20인 트랜잭션이 봄
v2     70000     txn_id=20      txn_id=35      20 ≤ txn_id < 35인 트랜잭션이 봄
v3     40000     txn_id=35      (살아있음)     txn_id ≥ 35인 트랜잭션이 봄

읽기 트랜잭션 txn_id=25:
  → v2를 봄 (balance=70000). v1은 너무 오래됨, v3는 아직 없음

새 쓰기 트랜잭션 txn_id=40:
  → UPDATE: v3를 삭제하고 v4(balance=20000) 생성
  → txn_id=25 읽기는 영향받지 않음 (여전히 v2를 봄)
```

### 결과: 읽기와 쓰기가 서로를 막지 않는다

```
[MVCC 동시 실행]

T1 (읽기, txn=25):  ────────[SELECT: balance=70000]────────▶
                                                (락 없이 이전 버전 읽음)
T2 (쓰기, txn=35):  ──[UPDATE: 70000→40000]──[COMMIT]──────▶
                     (새 버전 생성, T1에 영향 없음)

결과: T1은 T2의 쓰기와 무관하게 자신의 스냅샷을 읽음
→ "읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다"
```

**단, 같은 행을 동시에 쓰는 경우는 락이 필요하다.** 두 트랜잭션이 같은 행에 UPDATE를 시도하면 먼저 온 쪽이 X-락을 잡고, 나중 쪽은 대기한다.

### PostgreSQL의 MVCC 구현: xmin/xmax

PostgreSQL에서 각 튜플(행)의 헤더에는 두 필드가 있다:

- **xmin**: 이 버전을 생성한 트랜잭션 ID (INSERT 또는 UPDATE한 트랜잭션)
- **xmax**: 이 버전을 삭제한(또는 무효화한) 트랜잭션 ID (DELETE 또는 UPDATE로 대체된 경우)

```sql
-- 튜플의 xmin, xmax 확인 (PostgreSQL 전용 시스템 컬럼)
SELECT id, balance, xmin, xmax FROM account;
-- id  balance  xmin  xmax
-- A   70000    123   456   ← txn 123이 생성, txn 456이 삭제(또는 업데이트)
-- A   40000    456   0     ← txn 456이 생성, 아직 살아있음
```

트랜잭션이 SELECT 할 때 PostgreSQL은 **스냅샷(snapshot)**을 가져온다. 스냅샷은 "이 시점에 커밋된 트랜잭션 ID 집합"이다. xmin이 이 집합에 포함되고 xmax가 포함되지 않은 버전이 가시적이다.

### VACUUM — 죽은 버전 회수

MVCC의 대가는 오래된 버전(dead tuple)이 쌓이는 것이다. 더 이상 어떤 트랜잭션도 참조하지 않는 버전을 **VACUUM**이 회수한다.

```sql
-- 특정 테이블 VACUUM
VACUUM account;

-- 통계도 갱신하는 VACUUM ANALYZE
VACUUM ANALYZE account;

-- 공간을 OS에 반환하는 VACUUM FULL (테이블 잠금 필요)
VACUUM FULL account;

-- VACUUM 자동 실행 여부 확인
SHOW autovacuum;  -- on (기본값)
```

```
[MVCC와 VACUUM]

시간 ─────────────────────────────────────────────────────▶

  v1(100000): ██████████████████████████████▓▓▓▓▓▓
                                             ↑ 모든 TX가 v1 이후 버전 봄
                                               → dead tuple, VACUUM 대상

  v2(70000):  ████████████████████████████████████████▓▓▓▓
                                                       ↑ dead tuple

  v3(40000):  ████████████████████████████████████████████ → 살아있음

  VACUUM:                        ──────────────▶ (dead tuple 회수)
```

## 4.7 스냅샷 격리와 Write Skew — MVCC의 숨겨진 함정

### 스냅샷 격리 (Snapshot Isolation)

PostgreSQL의 REPEATABLE READ는 **스냅샷 격리(Snapshot Isolation, SI)**로 구현된다. 트랜잭션 시작 시 스냅샷을 고정하고, 트랜잭션 내내 그 스냅샷을 읽는다. 이로써:
- Non-repeatable Read 방지 (스냅샷이 고정되므로 같은 행이 항상 같은 값)
- Phantom Read도 방지 (새로 INSERT된 행이 스냅샷에 없으므로 보이지 않음)

이것이 SQL 표준의 REPEATABLE READ가 Phantom Read를 허용하는데 비해, PostgreSQL의 REPEATABLE READ가 Phantom Read까지 대부분 막는 이유다.

### Write Skew — 스냅샷 격리의 이상 현상

그런데 스냅샷 격리는 SQL 표준의 이상 현상 세 가지를 모두 막으면서도, 표준에는 없는 이상 현상인 **Write Skew(쓰기 왜곡)**에 취약하다.

```
[Write Skew 시나리오: 당직 의사 최소 1명 규칙]

테이블: doctors (name TEXT, on_call BOOLEAN)
  Alice: on_call = TRUE
  Bob:   on_call = TRUE
규칙: 동시에 당직인 의사가 항상 최소 1명이어야 한다.

시각   세션 A (T1, Alice가 당직 해제)   세션 B (T2, Bob이 당직 해제)
──────────────────────────────────────────────────────────────────
 t1    BEGIN ISOLATION LEVEL
       REPEATABLE READ;
 t2    SELECT COUNT(*) FROM doctors
       WHERE on_call = TRUE;
       -- => 2 (Alice, Bob 둘 다)         BEGIN ISOLATION LEVEL
                                           REPEATABLE READ;
 t3                                        SELECT COUNT(*) FROM doctors
                                           WHERE on_call = TRUE;
                                           -- => 2 (스냅샷: Alice, Bob 둘 다)
 t4    -- 2명이니까 1명 빠져도 됨
       UPDATE doctors
       SET on_call = FALSE
       WHERE name = 'Alice';
 t5    COMMIT;
                                        -- 2명이니까 1명 빠져도 됨
                                        -- (스냅샷은 t3 시점 고정, T1 커밋 모름)
 t6                                     UPDATE doctors
                                        SET on_call = FALSE
                                        WHERE name = 'Bob';
 t7                                     COMMIT;

최종: Alice on_call=FALSE, Bob on_call=FALSE
→ 당직 의사 0명! 규칙 위반! ← Write Skew
```

**왜 발생하는가?** 두 트랜잭션이 각자의 스냅샷을 읽고 서로 **다른 행**을 갱신했기 때문이다. 같은 행을 건드리지 않았으므로 락 충돌도 없고 스냅샷 격리도 감지하지 못했다. 그러나 결과적으로 제약이 깨졌다.

**Non-repeatable Read vs Write Skew**:
- Non-repeatable Read: 내가 읽은 행의 값이 다른 트랜잭션에 의해 바뀜 (다른 쪽이 같은 행을 씀)
- Write Skew: 내가 읽은 행과 내가 쓴 행이 다름, 두 트랜잭션이 서로 다른 행을 씀

### SERIALIZABLE(SSI)로 Write Skew 방지

PostgreSQL의 SERIALIZABLE은 **SSI(Serializable Snapshot Isolation)**로 구현된다. SSI는 스냅샷 격리에서 **읽기-쓰기 의존 사이클**을 감지하여 Write Skew를 방지한다.

```sql
-- Write Skew 방지: SERIALIZABLE 격리 수준 사용
BEGIN ISOLATION LEVEL SERIALIZABLE;
    SELECT COUNT(*) FROM doctors WHERE on_call = TRUE;  -- 2
    UPDATE doctors SET on_call = FALSE WHERE name = 'Alice';
COMMIT;
-- T1 또는 T2 중 하나가:
-- ERROR:  could not serialize access due to read/write dependencies among transactions
-- DETAIL:  Processes 12345 and 67890 appear to be in a conflict cycle.
```

쓰기 왜곡이 감지되면 PostgreSQL이 한 트랜잭션을 중단시키고, 애플리케이션이 재시도해야 한다.

## 4.8 낙관적 락 vs 비관적 락

### 비관적 락 (Pessimistic Locking)

**비관적 락**은 "충돌이 자주 발생할 것"이라고 가정하고, **먼저 락을 걸어** 다른 트랜잭션의 접근을 차단하는 방식이다.

```sql
-- 비관적 락: SELECT...FOR UPDATE로 미리 잠금
BEGIN;
    -- 잔액 확인과 동시에 X-락 획득
    SELECT balance FROM account WHERE id = 'A' FOR UPDATE;
    -- 다른 세션의 이 행 읽기/쓰기 모두 차단됨

    -- 비즈니스 로직 처리 (안전하게)
    UPDATE account SET balance = balance - 30000 WHERE id = 'A';
COMMIT;  -- 락 해제
```

**적합한 상황**:
- 재고 관리: 같은 상품을 동시에 많이 구매하는 경우 충돌이 빈번
- 좌석/티켓 예매: 같은 좌석을 동시에 여러 명이 예약 시도
- 충돌 가능성이 높고 재시도 비용이 큰 경우

### 낙관적 락 (Optimistic Locking)

**낙관적 락**은 "충돌이 드물 것"이라고 가정하고, 락을 걸지 않고 작업하다가 **커밋 시 충돌을 검사**하는 방식이다.

```sql
-- 낙관적 락: version 컬럼으로 충돌 감지
ALTER TABLE account ADD COLUMN version INT DEFAULT 0;

-- 1) 읽기: 현재 버전을 기억
SELECT id, balance, version FROM account WHERE id = 'A';
-- 결과: id='A', balance=70000, version=3

-- 2) 업데이트: 버전이 여전히 3인지 확인하면서 갱신
UPDATE account
SET balance = balance - 30000,
    version = version + 1
WHERE id = 'A' AND version = 3;  -- version 조건이 핵심

-- 3) 영향 행 수 확인
-- → 1행: 성공 (버전이 아직 3이었음 → 내가 읽은 후 아무도 변경 안 함)
-- → 0행: 실패 (버전이 바뀜 → 다른 트랜잭션이 먼저 변경) → 재시도
```

```
[낙관적 락 흐름]

트랜잭션 A:  [READ v=3]  ───────────────  [UPDATE WHERE v=3] → 1행 → 성공
트랜잭션 B:  [READ v=3]  [UPDATE v=3→4]  [UPDATE WHERE v=3] → 0행 → 실패 → 재시도
              (B가 먼저 커밋하여 v=4)
```

**적합한 상황**:
- 사용자 프로필 수정: 같은 사람을 동시에 수정하는 경우 드묾
- 일반적인 CRUD: 충돌 확률이 낮고 재시도 비용이 작은 경우

### 비교표

```
[낙관적 락 vs 비관적 락]

                   낙관적 락              비관적 락
────────────────────────────────────────────────────────────
충돌 가정          드물다                 자주 발생한다
락 시점            커밋 시 검사           데이터 접근 시 즉시
충돌 감지 방법     version 컬럼 확인       DB 락 대기/실패
충돌 처리          재시도 (애플리케이션)   대기 후 재실행
교착 가능성        없음                   있음
성능 (충돌 드묾)   높음 ↑                 낮음 (대기 발생)
성능 (충돌 빈번)   낮음 (재시도 폭증) ↓   수용 가능 (예상된 대기)
SQL                version 조건부 UPDATE  SELECT...FOR UPDATE
JPA                @Version               @Lock(PESSIMISTIC_WRITE)
```

### JPA 연결

```java
// 낙관적 락: @Version 애노테이션
@Entity
public class Account {
    @Id private String id;
    private int balance;

    @Version
    private int version;  // JPA가 자동으로 version 조건 추가
    // 충돌 시 OptimisticLockException 발생 → 재시도 로직 필요
}

// 비관적 락: @Lock 애노테이션
public interface AccountRepository extends JpaRepository<Account, String> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)  // SELECT...FOR UPDATE
    @Query("SELECT a FROM Account a WHERE a.id = :id")
    Optional<Account> findByIdWithLock(@Param("id") String id);
}
```

JPA의 낙관적/비관적 락 상세 사용법은 `../SPRING_개념서/README.md` 4.6절을 참조하라.

## 4.9 SELECT FOR UPDATE 및 고급 락 옵션 (PostgreSQL 16)

PostgreSQL 16의 행 수준 락 옵션:

```sql
-- FOR UPDATE: 배타 락 (가장 강함)
-- 다른 세션의 FOR UPDATE, FOR NO KEY UPDATE, FOR SHARE, FOR KEY SHARE 모두 차단
SELECT * FROM account WHERE id = 'A' FOR UPDATE;

-- FOR NO KEY UPDATE: 키를 포함하지 않는 배타 락
-- FOR KEY SHARE는 허용, 나머지 차단
SELECT * FROM account WHERE id = 'A' FOR NO KEY UPDATE;

-- FOR SHARE: 공유 락
-- 다른 FOR SHARE/FOR KEY SHARE는 허용, 쓰기 락은 차단
SELECT * FROM account WHERE id = 'A' FOR SHARE;

-- FOR KEY SHARE: 가장 약한 공유 락
-- 외래키 제약 확인에 사용
SELECT * FROM account WHERE id = 'A' FOR KEY SHARE;

-- NOWAIT: 락을 즉시 획득 불가 시 오류 반환 (대기하지 않음)
SELECT * FROM account WHERE id = 'A' FOR UPDATE NOWAIT;
-- ERROR:  could not obtain lock on row in relation "account"

-- SKIP LOCKED: 이미 잠긴 행을 건너뜀 (큐/작업 처리에 유용)
-- 여러 워커가 동시에 작업을 가져갈 때 충돌 없이 분배
SELECT * FROM jobs WHERE status = 'PENDING'
ORDER BY created_at
FOR UPDATE SKIP LOCKED
LIMIT 1;
-- 각 워커가 서로 다른 행을 가져감
```

```

**관찰 목표**:

| 시나리오 | 격리 수준 | 관찰 현상 |
|---------|----------|----------|
| 1 | READ COMMITTED | Dirty Read가 발생하지 않음 (B가 A의 미커밋 값을 못 봄) |
| 2 | READ COMMITTED | Non-repeatable Read 발생 (A가 두 번 읽으면 다른 값) |
| 3 | REPEATABLE READ | Non-repeatable Read 사라짐 (스냅샷 고정) |

**준비 단계** (아무 세션에서 한 번만):

```sql
DROP TABLE IF EXISTS account CASCADE;
CREATE TABLE account (id TEXT PRIMARY KEY, balance INT NOT NULL);
INSERT INTO account (id, balance) VALUES ('A', 100000);
```

**시나리오 2 (Non-repeatable Read) 관찰**:

```sql
-- [세션 A]
BEGIN;
SELECT balance FROM account WHERE id = 'A';  -- => 100000

-- [세션 B]
BEGIN;
UPDATE account SET balance = 55555 WHERE id = 'A';
COMMIT;

-- [세션 A] (같은 트랜잭션 안에서 재조회)
SELECT balance FROM account WHERE id = 'A';  -- => 55555 (!)
-- READ COMMITTED이므로 B의 커밋이 보임 → Non-repeatable Read 발생
COMMIT;
```

**시나리오 3 (REPEATABLE READ로 방지)**:

```sql
-- 값 초기화 (아무 세션에서)
UPDATE account SET balance = 100000 WHERE id = 'A';

-- [세션 A]
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM account WHERE id = 'A';  -- => 100000

-- [세션 B]
BEGIN;
UPDATE account SET balance = 77777 WHERE id = 'A';
COMMIT;

-- [세션 A] (스냅샷 고정 상태에서 재조회)
SELECT balance FROM account WHERE id = 'A';  -- => 100000 (!)
-- 스냅샷이 고정되어 B의 커밋이 보이지 않음 → Non-repeatable Read 없음
COMMIT;

-- [세션 A] (트랜잭션 종료 후 새 조회)
SELECT balance FROM account WHERE id = 'A';  -- => 77777 (이제 최신값)
```

## ⚠️ 흔한 오해와 함정

**오해 1: "REPEATABLE READ면 Phantom Read도 완전히 방지된다"**

SQL 표준상 REPEATABLE READ는 Phantom Read를 방지하지 않는다. PostgreSQL은 MVCC 스냅샷 덕분에 **대부분의** Phantom Read를 방지하지만, `SELECT FOR UPDATE`로 범위를 잠그는 경우나 특정 패턴에서는 여전히 발생할 수 있다. 확실한 보장이 필요하면 SERIALIZABLE을 써야 한다.

**오해 2: "MVCC는 락이 전혀 없다"**

MVCC는 읽기와 쓰기 간의 락을 없앤다. 그러나 **같은 행에 대한 동시 쓰기(UPDATE)는 여전히 배타 락이 필요하다.** 두 트랜잭션이 같은 행에 UPDATE를 시도하면, 먼저 도착한 쪽이 X-락을 잡고 나머지는 대기한다.

**오해 3: "낙관적 락은 항상 락 없이 동작한다"**

낙관적 락에서 DB 레벨의 락은 없지만, 커밋 시 version 컬럼 확인을 위한 `UPDATE WHERE version=?`이 실행될 때 DB 내부적으로 짧은 락이 발생한다. "락이 없다"는 것은 읽기 단계에서 데이터를 미리 잠그지 않는다는 의미다.

**오해 4: "READ UNCOMMITTED에서는 커밋 안 된 값을 읽는다"**

PostgreSQL에서 READ UNCOMMITTED는 실제로 READ COMMITTED와 동일하게 동작한다. PostgreSQL의 MVCC 구현상 커밋되지 않은 값을 노출하는 경로가 없기 때문이다. 즉 PostgreSQL에서는 Dirty Read가 격리 수준과 무관하게 발생하지 않는다.

**오해 5: "교착 상태는 항상 나쁘다"**

교착 상태 자체는 데이터 손상이 아니다. PostgreSQL이 자동으로 감지하고 하나를 롤백한다. 진짜 문제는 **교착 상태가 자주 발생하는 것**이다. 애플리케이션이 일관된 락 순서를 유지하면 대부분 예방 가능하다.

**오해 6: "SERIALIZABLE은 너무 느려서 쓸 수 없다"**

PostgreSQL의 SSI(Serializable Snapshot Isolation)는 전통적인 2PL 기반 직렬화보다 훨씬 효율적이다. 실제로 대부분의 트랜잭션이 충돌 없이 진행될 경우 성능 저하가 크지 않다. 정확성이 중요한 금융 트랜잭션에서는 적극 고려할 만하다.

T1: R(A), W(A), R(B)

| 선행 연산  | 후행 연산 | 충돌 여부 | 엣지       |

**문제 3 (2PL과 교착):** 두 트랜잭션 T1(계좌 A→B 이체)과 T2(계좌 B→A 이체)가 동시에 실행될 때 교착 상태가 발생하는 타임라인을 그리고, 이를 방지하는 방법을 SQL로 보여라.

<details>
<summary>힌트 및 해설</summary>

**교착 타임라인**:
```
T1: LOCK(A) ───────▶ wait for LOCK(B)...
T2:          LOCK(B) ─────────────────▶ wait for LOCK(A)...
             교착!
```

**방지: 항상 id 오름차순으로 잠그기**
```sql
-- T1 (A→B 이체): 'A' < 'B'이므로 A부터
BEGIN;
SELECT * FROM account WHERE id = 'A' FOR UPDATE;
SELECT * FROM account WHERE id = 'B' FOR UPDATE;
UPDATE account SET balance = balance - 30000 WHERE id = 'A';
UPDATE account SET balance = balance + 30000 WHERE id = 'B';
COMMIT;

-- T2 (B→A 이체): 'A' < 'B'이므로 역시 A부터 잠금 (B가 아니라!)
BEGIN;
SELECT * FROM account WHERE id = 'A' FOR UPDATE;  -- A 먼저
SELECT * FROM account WHERE id = 'B' FOR UPDATE;  -- B 나중
UPDATE account SET balance = balance - 30000 WHERE id = 'B';
UPDATE account SET balance = balance + 30000 WHERE id = 'A';
COMMIT;
```

두 트랜잭션 모두 A를 먼저 잠그므로, T2는 T1이 끝날 때까지 A를 기다린다. 교착 발생 불가.
</details>

**문제 4 (MVCC 이해):** PostgreSQL의 MVCC에서 읽기 트랜잭션이 진행 중인 동안 쓰기 트랜잭션이 같은 행을 UPDATE하면, 읽기 트랜잭션은 어떤 버전을 보는가? 또한 쓰기 트랜잭션이 완료된 후 읽기 트랜잭션이 같은 행을 다시 조회하면, READ COMMITTED와 REPEATABLE READ에서 결과가 어떻게 다른가?

<details>
<summary>힌트 및 해설</summary>

**읽기 진행 중 쓰기 발생**: 읽기 트랜잭션은 자신의 스냅샷 시점의 버전을 본다. 쓰기가 새 버전을 추가해도 읽기 트랜잭션의 스냅샷에는 없으므로 보이지 않는다. 이것이 "쓰기가 읽기를 막지 않는다"의 본질이다.

**쓰기 완료 후 재조회**:
- **READ COMMITTED**: 매 SELECT마다 새 스냅샷을 가져온다. 쓰기 트랜잭션이 COMMIT했으므로 새 스냅샷에 새 버전이 포함된다 → 새 값을 봄 (Non-repeatable Read 가능)
- **REPEATABLE READ**: 트랜잭션 시작 시 고정된 스냅샷을 유지한다. 쓰기가 COMMIT돼도 스냅샷이 바뀌지 않는다 → 이전 값을 계속 봄 (Non-repeatable Read 방지)
</details>

**문제 5 (Write Skew 분석):** 온라인 시험 시스템에서 각 시험은 정확히 2명의 감독관이 배정되어야 한다. 두 감독관 Admin1과 Admin2가 동시에 "남은 감독관 1명을 다른 시험에 재배정"하는 트랜잭션을 실행할 때 어떤 이상 현상이 발생할 수 있으며, 해결 방법은 무엇인가?

<details>
<summary>힌트 및 해설</summary>

**발생 이상 현상: Write Skew**

두 Admin이 각자 스냅샷에서 "현재 감독관 2명"을 확인하고, 각자 서로 다른 감독관을 재배정하면 최종적으로 감독관 0명이 된다. 각자 다른 행(감독관 레코드)을 수정했으므로 REPEATABLE READ의 MVCC가 감지하지 못한다.

**해결 방법**:
1. **SERIALIZABLE** 격리 수준 사용 — PostgreSQL SSI가 읽기-쓰기 의존 사이클을 감지하여 한 트랜잭션을 abort시킴
2. **명시적 잠금** — 시험 레코드에 `FOR UPDATE`를 걸어 두 Admin이 동시에 진행하지 못하게 함:
   ```sql
   BEGIN;
   SELECT * FROM exam WHERE id = :exam_id FOR UPDATE;  -- 시험 잠금
   SELECT COUNT(*) FROM exam_supervisors WHERE exam_id = :exam_id;
   -- 2명이면 1명만 재배정 허용
   UPDATE exam_supervisors SET exam_id = :new_exam_id WHERE supervisor_id = :my_id;
   COMMIT;
   ```
3. **낙관적 락** — 시험 엔티티에 version 컬럼을 추가하여 두 번째 커밋이 version 불일치로 실패하게 함
</details>

## 요약

- **이상 현상 세 가지**: Dirty Read(미커밋 읽기), Non-repeatable Read(같은 행 값 변경), Phantom Read(행 집합 변경) — 격리 수준이 높을수록 더 많이 방지된다.
- **격리 수준 매트릭스**: READ UNCOMMITTED → READ COMMITTED(PG 기본) → REPEATABLE READ → SERIALIZABLE 순으로 강해진다.
- **직렬가능성**: 동시 스케줄의 결과가 어떤 직렬 스케줄과 동일할 때 "올바른" 동시 실행이다. 우선 그래프의 사이클 유무로 충돌 직렬가능성을 판정한다.
- **2PL**: 확장 단계(락 획득만)와 수축 단계(락 해제만)로 나눠 충돌 직렬가능성을 보장한다. 교착 상태는 같은 순서로 잠그기로 예방한다.
- **Strict/Rigorous 2PL**: 커밋까지 락을 보유하여 연쇄 롤백을 방지한다.
- **타임스탬프 순서화**: 락 없이 TS(Ti)로 순서를 강제한다. 교착 불가능, 대신 abort+재시작 발생.
- **MVCC**: 새 버전을 추가하고 각 트랜잭션은 스냅샷을 읽는다. 읽기와 쓰기가 서로를 차단하지 않는다. VACUUM이 죽은 버전을 회수한다.
- **Write Skew**: 스냅샷 격리(REPEATABLE READ)의 이상 현상 — 두 트랜잭션이 서로 다른 행을 갱신하여 제약을 위반. SERIALIZABLE(SSI)로 해결.
- **낙관적 vs 비관적 락**: 비관적은 미리 `FOR UPDATE` 잠금(충돌 빈번 시), 낙관적은 version 컬럼으로 커밋 시 검사(충돌 드묾 시). JPA의 `@Version` / `@Lock`과 직접 대응.
- Spring 연결: `../SPRING_개념서/README.md` 4.6절(격리 수준과 이상 현상), 2.6절(트랜잭션 프록시/AOP).
