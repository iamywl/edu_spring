## 5.1 왜 전체 스캔은 O(n)이고 인덱스는 O(log n)인가

### 인덱스가 없을 때: Seq Scan

데이터베이스는 테이블의 행들을 **힙(heap)**이라는 디스크 블록 집합에 저장한다. 특별한 순서가 없다. `INSERT` 순서대로 그냥 쌓인다.

```sql
SELECT * FROM users WHERE email = 'kim@example.com';
```

인덱스가 없다면 DB는 **처음 블록부터 마지막 블록까지 모든 행을 읽고**, 각 행의 `email` 값이 `'kim@example.com'`인지 하나씩 비교한다. 이것이 **순차 스캔(Sequential Scan, Seq Scan)**이다.

```
힙 블록들 (저장 순서대로, 정렬 없음)
┌─────────────────────────────────────────────────────┐
│ [이순신,조선,50] [kim@..,KR,30] [park@..,US,25] ...  │  ← 블록 1
│ [lee@..,JP,40]  [choi@..,KR,22] [hong@..,DE,35] ... │  ← 블록 2
│  ...                                                 │  ← 블록 N
└─────────────────────────────────────────────────────┘
        ↓  ↓  ↓  ↓  ↓  ↓  ... (전부 읽으며 비교)
              kim@.. 발견!
```

행이 N개면 비교도 N번이다. 데이터가 두 배가 되면 스캔 시간도 두 배 → **O(n)**.

### 인덱스가 있을 때: O(log n)

인덱스는 책 뒤의 **찾아보기(색인)**와 같다. "트랜잭션"이라는 단어의 페이지를 찾을 때, 500페이지를 처음부터 넘기지 않고, 색인에서 `트랜잭션 → 412p`를 찾아 바로 뛰어간다.

DB 인덱스는 이 색인을 **정렬된 트리 구조**로 만들어 둔다. 찾고 싶은 값이 트리의 어느 가지에 있는지 절반씩 좁혀가므로 O(log n)이다.

**100만 행 비교:**

| 방법 | 비교 횟수 | 직관 |
|------|-----------|------|
| Seq Scan (인덱스 없음) | 1,000,000번 | 처음부터 끝까지 |
| B+tree Index Scan | ≈ log₂(1,000,000) ≈ 20번 | 절반씩 좁혀가기 |
| **속도 차이** | **약 50,000배** | |

실제로는 B+tree의 분기가 2개가 아니라 수백 개(팬아웃)이기 때문에 20번보다도 훨씬 적은 단계(보통 3~4단계)로 찾는다. 이 내용은 5.2에서 자세히 다룬다.

```
인덱스 없음 (Seq Scan, O(n))

  비교: ● ● ● ● ● ● ● ● ● ● ... (100만 번)
         └──────── 전부 훑기 ────────┘

인덱스 있음 (Index Scan, O(log n))

  루트
  ├─ 왼쪽 절반 탈락
  │   └─ (건너뜀)
  └─ 오른쪽 절반
      ├─ 또 절반 탈락
      └─ 발견! ← 3~4단계 만에 도달
```

## 5.2 B-tree와 B+tree: DB가 선택한 자료구조

### 왜 이진 탐색 트리(BST)가 아닌가

이진 탐색 트리도 O(log n)이다. 그런데 DB는 이진 트리 대신 B+tree를 쓴다. 왜?

**핵심 제약:** 데이터베이스의 데이터는 **디스크(SSD/HDD)**에 있다. 메모리가 아니다. 디스크 접근은 메모리 접근보다 수백~수만 배 느리다. 그리고 디스크는 **블록(page) 단위**로 읽는다 — 1바이트를 읽어도 보통 8KB 블록 전체를 읽는다.

이진 트리는 노드 하나에 **값 1개**만 있다. 100만 개 값이라면 트리 높이가 log₂(1,000,000) ≈ 20이다. 루트에서 리프까지 20개 노드를 각각 다른 디스크 블록에서 읽으면 **디스크 접근 최대 20회**.

B+tree는 노드 하나(디스크 블록 1개, 8KB)에 **수백 개의 키**를 담는다. 자식 갈래 수(팬아웃)가 수백 개이므로:

| 트리 종류 | 노드당 키 수 | 100만 건 높이 | 디스크 접근 횟수 |
|-----------|-------------|--------------|-----------------|
| 이진 탐색 트리 | 1 | ≈ 20 | ≈ 20회 |
| B+tree (팬아웃 200) | ≈ 200 | ≈ log₂₀₀(1,000,000) ≈ 3 | **3~4회** |

디스크 접근 20회 vs 3회. B+tree가 압도적으로 유리하다.

### B-tree와 B+tree의 구조적 차이

두 자료구조를 혼용해서 쓰는 문서가 많지만, 엄밀히는 다르다.

**B-tree (원래 B-tree):**
- 내부 노드에도 **데이터 포인터**가 있다.
- 탐색이 내부 노드에서 끝날 수 있다.
- 범위 검색 시 트리를 다시 오르내려야 한다.

**B+tree:**
- **실제 데이터 포인터는 리프 노드에만** 있다.
- 내부 노드는 **길안내판(라우팅 키)**만 들고 있다.
- 내부 노드가 가벼워지므로 팬아웃이 더 커지고 트리 높이가 낮아진다.
- **리프 노드들이 링크드 리스트로 연결**되어 범위 검색(`BETWEEN`, `>=`, `<=`)이 빠르다.

PostgreSQL, MySQL InnoDB, Oracle — 모두 B+tree를 사용하는 이유가 여기 있다.

```
B+tree 예시 (이메일 인덱스, 간략화)

                       [ M | T ]          ← 루트(내부 노드): 길안내판만
                      /    |    \
           [ C | G ]    [ P | R ]    [ W | Z ]  ← 내부 노드
           /  |  \      /  |  \      /  |  \
         ↓   ↓   ↓    ↓   ↓   ↓   ↓   ↓   ↓

리프 노드(실제 데이터 포인터):
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ A.. │ptr │──▶│ D.. │ptr │──▶│ H.. │ptr │──▶│ K.. │ptr │──▶ ...
│ B.. │ptr │   │ E.. │ptr │   │ I.. │ptr │   │ L.. │ptr │
│ C.. │ptr │   │ F.. │ptr │   │ J.. │ptr │   │ M.. │ptr │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
      ↑ 리프 노드들은 링크드 리스트로 연결 → 범위 검색 O(k) 가능

ptr = 힙(heap) 블록의 실제 위치 (ctid)
```

**B+tree의 두 가지 핵심 성질:**

1. **내부 노드에는 데이터 포인터 없음**: 내부 노드가 오직 키와 자식 포인터만 담으므로 노드 하나에 더 많은 키가 들어간다 → 팬아웃 증가 → 트리 높이 감소 → 디스크 접근 횟수 최소화.

2. **리프 노드가 연결 리스트**: `BETWEEN 'a@..' AND 'z@..'` 같은 범위 검색은, 시작 리프를 찾은 뒤 오른쪽 포인터를 따라 순서대로 읽기만 하면 된다 → O(k), k는 결과 행 수.

### 팬아웃(fanout)과 트리 높이 계산

**실제 값을 계산해보자.**

PostgreSQL의 기본 페이지 크기는 8KB(8,192 바이트)다. 내부 노드의 각 엔트리는 키 값 + 자식 포인터로 구성된다. 예를 들어 `BIGINT` 키(8바이트) + 포인터(6바이트) = 14바이트라면:

```
팬아웃 ≈ 8,192 / 14 ≈ 585 (헤더 오버헤드 고려 시 ≈ 400~500)
```

실제 PostgreSQL에서는 텍스트 키 길이, fill factor 등을 고려해 보통 팬아웃이 **100~400** 사이다.

팬아웃을 400으로 잡으면:

```
높이 1 (루트만)         : 최대 400 건
높이 2 (루트 + 리프)   : 최대 400 × 400 = 160,000 건
높이 3                  : 최대 400³ = 64,000,000 건 (6,400만)
높이 4                  : 최대 400⁴ = 25,600,000,000 건 (256억)
```

**10억 건 테이블도 B+tree 높이 4~5면 찾는다.** 디스크 접근 4~5회.

**fill factor**: B+tree는 노드를 꽉 채우지 않고 일부 여유를 남겨 두는데, 이 비율을 `fill factor`라고 한다. PostgreSQL 기본값은 **90%**다. 나머지 10%는 향후 INSERT/UPDATE로 노드 분할(split)을 줄이기 위한 여유 공간이다.

```sql
-- fill factor를 조정한 인덱스 생성 (쓰기가 매우 빈번할 때 낮춤)
CREATE INDEX idx_orders_created_at ON orders(created_at) WITH (fillfactor = 70);
```

## 5.3 복합 인덱스와 왼쪽 접두사(Leftmost-Prefix) 규칙

### 복합 인덱스의 정렬 구조

```sql
CREATE INDEX idx_country_age ON users(country, age);
```

이 인덱스는 `(country, age)` 쌍을 **사전 순서(dictionary order)**로 정렬한다.

```
정렬 결과 (개념도):

  (DE, 20)
  (DE, 25)
  (DE, 30)   ← country='DE' 범위
  (DE, 45)
  (FR, 22)
  (FR, 31)   ← country='FR' 범위
  ...
  (KR, 20)
  (KR, 30)   ← country='KR' AND age=30 → 바로 찾음
  (KR, 35)
  ...
  (US, 20)
  (US, 40)   ← country='US' 범위
```

비유: 전화번호부는 (성, 이름) 순으로 정렬되어 있다. "김철수"는 빠르게 찾는다. "철수"(성 없이 이름만)를 찾으려면? 전화번호부 전체를 다 뒤져야 한다.

### 왼쪽 접두사 규칙 정리

`CREATE INDEX idx ON users(country, age)` 기준:

| WHERE 조건 | 인덱스 사용 여부 | 이유 |
|-----------|-----------------|------|
| `country='KR' AND age=30` | **O** — 완전 활용 | (country, age) 둘 다 왼쪽부터 순서대로 |
| `country='KR'` | **O** — 부분 활용 | 왼쪽 접두사(country)만 써도 됨 |
| `country='KR' AND age > 25` | **O** — 부분 범위 활용 | country로 탐색 후 age 범위 스캔 |
| `age=30` | **X** — 사용 불가 | 첫 번째 컬럼(country) 없이는 age 위치를 특정 불가 |
| `age=30 AND country='KR'` | **O** — 사용 가능* | 옵티마이저가 조건 순서를 재배열 |

*쿼리 작성 순서가 아니라, 인덱스 컬럼 순서가 기준이다. 옵티마이저는 `age=30 AND country='KR'`을 `country='KR' AND age=30`으로 재배열한다.

### 컬럼 순서 선택의 원칙

**등가 조건 컬럼을 앞에, 범위 조건 컬럼을 뒤에 배치하라.**

```sql
-- 나쁨: age 범위 조건이 앞에 있어 country는 인덱스를 못 씀
-- WHERE age > 20 AND country = 'KR'
CREATE INDEX bad_idx ON users(age, country);

-- 좋음: 등가 조건(country)을 앞에, 범위 조건(age)을 뒤에
CREATE INDEX good_idx ON users(country, age);
```

왜? `country='KR'`은 인덱스에서 정확한 위치를 특정한다. 그 위치에서 `age > 20`을 범위 스캔하면 된다. 반대로 `age > 20`을 먼저 두면 해당 범위가 country 전체에 흩어져 있어 인덱스 효율이 사라진다.

## 5.4 커버링 인덱스(Covering Index)와 Index Only Scan

### 일반 인덱스의 2단계 조회

보통 인덱스 조회는 **두 단계**다:

```
1단계: 인덱스 트리에서 키 검색 → 힙(heap)의 물리적 위치(ctid) 획득
2단계: ctid로 힙 블록 접근 → 실제 행의 모든 컬럼 읽기
```

```
인덱스 블록                   힙 블록 (테이블 본체)
┌─────────────────┐           ┌──────────────────────────────┐
│ kim@.. → ctid   │──────────▶│ id=5, email=kim@.., name=김철수, │
│           (0,5) │           │ country=KR, age=30, ...       │
└─────────────────┘           └──────────────────────────────┘
   1단계                            2단계 (힙 접근)
```

2단계에서 힙 블록을 **무작위**로 읽어야 하기 때문에 랜덤 I/O가 발생한다. 인덱스가 있어도 대량의 행을 반환하는 쿼리라면 옵티마이저가 "차라리 Seq Scan이 낫다"고 판단할 수 있다.

### 커버링 인덱스: 2단계를 건너뛴다

SELECT 컬럼과 WHERE 조건에 필요한 **모든 컬럼**이 인덱스 안에 있으면, 힙 접근 없이 **인덱스만 읽고 쿼리를 완성**할 수 있다. 이것이 **커버링 인덱스(covering index)**이며, EXPLAIN에서 **Index Only Scan**으로 나타난다.

```sql
-- email로 country를 조회하는 패턴이 빈번하다고 가정
SELECT email, country FROM users WHERE email = 'kim@example.com';

-- INCLUDE로 커버링 인덱스 생성 (PostgreSQL 11+)
CREATE INDEX idx_users_email_cover ON users(email) INCLUDE (country);
```

`INCLUDE` 절에 넣은 컬럼은 **정렬 키로 쓰이지 않고**, 단지 리프 노드에 **값으로 저장**된다. 덕분에 `SELECT email, country`처럼 이 두 컬럼만 필요한 쿼리는 힙을 전혀 건드리지 않는다.

```
커버링 인덱스 리프 노드:
┌──────────────────────────────────────────┐
│ key: kim@example.com                     │
│ include: country = 'KR'                  │  ← 여기서 끝! 힙 안 봐도 됨
│ ptr: ctid (사용 안 함)                   │
└──────────────────────────────────────────┘
```

**EXPLAIN에서의 차이:**

```
-- 일반 인덱스
Index Scan using idx_users_email on users
  Index Cond: (email = 'kim@example.com')
  Heap Fetches: 1    ← 힙 1번 읽음

-- 커버링 인덱스
Index Only Scan using idx_users_email_cover on users
  Index Cond: (email = 'kim@example.com')
  Heap Fetches: 0    ← 힙 0번! 인덱스만으로 완료
```

커버링 인덱스는 디스크 I/O를 절반 이하로 줄일 수 있어 빈번하게 실행되는 조회 쿼리에서 큰 효과를 낸다.

## 5.5 PostgreSQL의 인덱스 종류

PostgreSQL은 B-tree 외에도 다양한 인덱스 타입을 제공한다. 상황에 맞는 인덱스를 선택하는 것이 중요하다.

### 5.5.1 B-tree 인덱스 (기본)

```sql
CREATE INDEX idx_email ON users(email);  -- 타입 지정 안 하면 기본이 B-tree
```

- **적합한 경우**: `=`, `<`, `>`, `<=`, `>=`, `BETWEEN`, `LIKE 'prefix%'`
- **부적합한 경우**: `LIKE '%suffix'`(앞 와일드카드), 전문 검색, 배열/JSONB

### 5.5.2 Hash 인덱스

```sql
CREATE INDEX idx_email_hash ON users USING hash(email);
```

- **적합한 경우**: `=` 등가 조건만 있을 때
- **부적합한 경우**: 범위 검색(`>`, `<`, `BETWEEN`), ORDER BY
- B-tree보다 등가 검색이 약간 빠를 수 있지만, 범위를 못 쓰는 제약이 있어 실제로는 거의 B-tree를 선택한다.
- PostgreSQL 10+ 에서 WAL 지원, 재시작 안전성 개선.

### 5.5.3 GiST (Generalized Search Tree)

```sql
-- 전문 검색, 기하 데이터 (예: PostGIS 공간 인덱스)
CREATE INDEX idx_location ON places USING gist(location);

-- 범위 타입
CREATE INDEX idx_period ON events USING gist(during);  -- tsrange
```

- **적합한 경우**: 기하 데이터(점/선/다각형), 범위 타입, 전문 검색, `&&`(겹침) 연산
- 내부적으로 확장 가능한 프레임워크 — 연산자 클래스만 구현하면 된다.

### 5.5.4 GIN (Generalized Inverted Index)

```sql
-- JSONB 컬럼 인덱싱
CREATE INDEX idx_metadata ON products USING gin(metadata);

-- 배열 컬럼 인덱싱
CREATE INDEX idx_tags ON articles USING gin(tags);

-- 전문 검색 (tsvector)
CREATE INDEX idx_content_fts ON articles USING gin(to_tsvector('korean', content));
```

- **적합한 경우**: JSONB, 배열, 전문 검색(tsvector)
- 키 하나가 여러 행을 가리키는 **역 인덱스** 구조
- 빌드 시간이 길고 업데이트 비용이 크지만, 조회 속도가 빠름

### 5.5.5 BRIN (Block Range Index)

```sql
-- 삽입 순서대로 단조 증가하는 타임스탬프에 적합
CREATE INDEX idx_created_at_brin ON events USING brin(created_at);
```

- **적합한 경우**: 대규모 테이블에서 **물리적으로 순서 있는** 데이터 (로그, 시계열)
- 블록 범위별 최솟값/최댓값만 기록 → **매우 작은 인덱스 크기** (B-tree의 수백 분의 1)
- **부적합한 경우**: 무작위 삽입, 높은 선택도 조회

### 5.5.6 부분 인덱스 (Partial Index)

```sql
-- 활성 사용자(active=true)만 인덱싱
CREATE INDEX idx_active_users_email ON users(email) WHERE active = true;

-- 아직 처리 안 된 주문만 인덱싱
CREATE INDEX idx_pending_orders ON orders(created_at) WHERE status = 'PENDING';
```

- WHERE 절로 인덱싱 대상을 **일부 행**으로 제한
- 인덱스 크기가 줄고, 자주 조회되는 부분집합에 집중할 수 있다.
- 쿼리의 WHERE 조건이 인덱스의 WHERE 조건을 **포함**해야 인덱스가 적용된다.

```sql
-- 인덱스가 적용되는 쿼리 (active=true 조건 포함)
SELECT * FROM users WHERE active = true AND email = 'kim@example.com';

-- 인덱스가 적용되지 않는 쿼리 (active 조건 없음)
SELECT * FROM users WHERE email = 'kim@example.com';
```

### 5.5.7 표현식 인덱스 (Expression Index)

```sql
-- lower()를 적용한 표현식 인덱스 → 대소문자 무관 이메일 검색
CREATE INDEX idx_lower_email ON users(lower(email));
```

```sql
-- 이 쿼리에 인덱스가 적용됨
SELECT * FROM users WHERE lower(email) = lower('Kim@Example.com');
```

- WHERE 절에 컬럼 그대로가 아닌 **함수/표현식을 적용**했을 때 사용
- 5.8절에서 다루는 "인덱스가 안 타는 패턴" 중 가장 흔한 실수를 해결하는 방법

### 5.5.8 유니크 인덱스 (Unique Index)

```sql
-- PRIMARY KEY는 자동으로 유니크 인덱스 생성
-- UNIQUE 제약도 자동으로 유니크 인덱스 생성

-- 명시적 유니크 인덱스
CREATE UNIQUE INDEX idx_unique_email ON users(email);

-- 부분 유니크 인덱스 (활성 사용자 중에서만 이메일 유일)
CREATE UNIQUE INDEX idx_unique_active_email ON users(email) WHERE active = true;
```

JPA에서 `@Column(unique = true)`를 붙이면 PostgreSQL은 자동으로 유니크 인덱스를 생성한다. Spring 개념서 4.7절에서 "existsByEmail이 빠른 이유"가 바로 이것이다.

## 5.6 선택도(Selectivity)와 카디널리티: 옵티마이저의 판단 기준

### 선택도란 무엇인가

**선택도(selectivity)**는 조건이 얼마나 많은 행을 걸러내는지를 나타내는 비율이다.

```
선택도 = 조건을 만족하는 행 수 / 전체 행 수
```

- 선택도 **낮음** (0에 가까움): 조건이 극소수 행만 통과시킴 → 인덱스 효율 **높음**
- 선택도 **높음** (1에 가까움): 조건이 대부분 행을 통과시킴 → 인덱스 효율 **낮음**

**카디널리티(cardinality)**는 컬럼의 고유값 수다.

```
email 컬럼: 100만 사용자 중 고유값 100만 개 → 카디널리티 100만, 선택도 ≈ 1/100만 (낮음) → 인덱스 매우 효율적
gender 컬럼: 100만 행, 고유값 2개(M/F) → 카디널리티 2, 선택도 ≈ 1/2 (높음) → 인덱스 거의 무의미
```

### 옵티마이저가 Seq Scan을 선택하는 경우

**인덱스가 있어도 옵티마이저가 Seq Scan을 선택할 수 있다.** 왜?

Index Scan은 인덱스 블록과 힙 블록을 **번갈아 무작위로** 읽는다. 결과 행이 많다면 이 무작위 I/O가 순차 스캔보다 오히려 느릴 수 있다.

```
결과 행이 적을 때 (선택도 낮음): Index Scan 유리
  인덱스 → 힙[3번] → 힙[77번] → 힙[15231번]
  → 3번의 랜덤 I/O, 각각 다른 블록

결과 행이 많을 때 (선택도 높음): Seq Scan이 나을 수 있음
  인덱스 → 힙[거의 모든 블록]
  → 어차피 전체 블록을 읽어야 하는데, 인덱스 오버헤드만 추가됨
  → 차라리 순차로 쭉 읽는 게 I/O 효율에서 유리
```

일반적으로 결과가 전체 행의 **5~10%** 이상이면 옵티마이저는 Seq Scan을 선호하기 시작한다. (정확한 임계치는 통계와 비용 파라미터에 따라 다름.)

### 통계 정보: pg_stats와 ANALYZE

옵티마이저는 통계 정보를 바탕으로 선택도를 **추정**한다.

```sql
-- 특정 컬럼의 통계 정보 조회
SELECT
    tablename,
    attname,
    n_distinct,         -- 고유값 수 (음수면 비율, 예: -0.5 = 전체의 50%)
    correlation,        -- 물리적 순서와 논리적 순서의 상관관계 (-1~1)
    most_common_vals,   -- 가장 빈번한 값 목록
    most_common_freqs,  -- 그 빈도
    histogram_bounds    -- 값 분포 히스토그램
FROM pg_stats
WHERE tablename = 'big_user' AND attname = 'email';
```

```sql
-- 통계 수동 갱신 (INSERT/UPDATE/DELETE 대량 후 필수)
ANALYZE big_user;

-- 전체 DB 통계 갱신 (자동으로도 autovacuum이 수행)
ANALYZE;
```

**EXPLAIN의 추정 rows와 실제 rows가 크게 다르다면** → 통계가 낡았거나 값 분포가 치우쳐 있다는 신호다. `ANALYZE`를 수행하거나, `default_statistics_target`을 높여 통계 샘플 수를 늘린다.

```sql
-- email 컬럼의 통계 샘플 수를 기본(100)에서 500으로 늘림
ALTER TABLE users ALTER COLUMN email SET STATISTICS 500;
ANALYZE users;
```

## 5.7 클러스터형 vs 비클러스터형 인덱스

### PostgreSQL의 힙 기반 구조 (비클러스터)

PostgreSQL은 **힙 기반(heap-based)** 스토리지를 사용한다. 테이블 행은 힙에 **삽입 순서대로** 저장되며, 어떤 인덱스도 **보조(secondary) 인덱스**다. 인덱스의 리프 노드는 행의 물리적 위치인 `ctid (블록번호, 오프셋)`를 가리킨다.

```
PostgreSQL 구조
┌────────────────────────────────┐
│  인덱스 (B+tree)               │
│  email → ctid(0,5)            │
│  kim@. → ctid(3,12)           │
└──────────────┬─────────────────┘
               │ ctid로 직접 접근
               ▼
┌────────────────────────────────┐
│  힙 (테이블 본체)              │
│  블록0: [행1][행2][행3]...     │
│  블록3: ...[행12: kim@..]...   │  ← 무작위 위치
└────────────────────────────────┘
```

### MySQL InnoDB의 클러스터형 PK

MySQL InnoDB는 **기본키(PK)로 클러스터형 인덱스**를 만든다. 테이블 행 자체가 PK 순서대로 B+tree의 리프에 저장된다. PK 조회는 리프에서 바로 행 전체를 읽는다.

세컨더리 인덱스는 리프에 `ctid` 대신 **PK 값**을 가리킨다 → 세컨더리 인덱스 조회는 항상 PK 인덱스를 한 번 더 탐색한다(이중 조회).

| 항목 | PostgreSQL | MySQL InnoDB |
|------|-----------|-------------|
| 기본 구조 | 힙 + 보조 인덱스들 | 클러스터형 PK |
| PK 조회 | 인덱스 → 힙 | 인덱스 리프 = 행 |
| 세컨더리 인덱스 | 인덱스 → ctid → 힙 | 인덱스 → PK → 클러스터 인덱스 |
| 행 삽입 비용 | 힙에 바로 추가(저렴) | PK 순서 유지 필요(비용 가변) |

### HOT Update와 CLUSTER 명령

**HOT(Heap-Only Tuple) Update**: PostgreSQL의 최적화로, 인덱스 컬럼이 변경되지 않는 UPDATE는 기존 인덱스를 수정하지 않고 힙 내에서만 처리한다. 인덱스 갱신 비용 절감.

**CLUSTER 명령**: 특정 인덱스의 순서대로 힙을 물리적으로 재배치한다. 단, 이후 INSERT/UPDATE는 순서를 유지하지 않아 시간이 지나면 무질서해진다. 대형 보고서용 조회에 일시적으로 사용한다.

```sql
-- emails 인덱스 순서로 힙 재배치 (오래 걸리고, 이후 유지 안 됨)
CLUSTER users USING idx_users_email;
```

## 5.8 인덱스가 안 타는 흔한 패턴

인덱스를 만들었는데도 EXPLAIN에서 Seq Scan이 나온다면, 다음 패턴 중 하나일 가능성이 높다.

### 패턴 1: 컬럼에 함수 적용

```sql
-- 인덱스 안 탐: email 컬럼에 lower()가 적용되어 인덱스 키와 불일치
SELECT * FROM users WHERE lower(email) = 'kim@example.com';

-- 해결: 표현식 인덱스 생성
CREATE INDEX idx_lower_email ON users(lower(email));
-- 또는 애초에 DB에 소문자로만 저장
```

```sql
-- 인덱스 안 탐: 날짜 컬럼에 DATE_TRUNC 적용
SELECT * FROM orders WHERE DATE_TRUNC('month', created_at) = '2024-01-01';

-- 해결: 범위 조건으로 재작성
SELECT * FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01';
```

### 패턴 2: 앞 와일드카드 LIKE

```sql
-- 인덱스 안 탐: 앞에 %가 있으면 시작 지점 특정 불가
SELECT * FROM users WHERE email LIKE '%kim%';

-- 인덱스 사용 가능: 뒤에만 % 있으면 B-tree 활용 가능
SELECT * FROM users WHERE email LIKE 'kim%';

-- 해결 (앞 와일드카드): GIN + pg_trgm 확장
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_email_trgm ON users USING gin(email gin_trgm_ops);
SELECT * FROM users WHERE email LIKE '%kim%';  -- 이제 GIN 인덱스 사용
```

### 패턴 3: 암묵적 타입 변환

```sql
-- 인덱스 안 탐: user_id가 INTEGER인데 문자열로 비교
SELECT * FROM users WHERE user_id = '123';  -- 암묵적 캐스팅 발생

-- 해결: 타입 맞추기
SELECT * FROM users WHERE user_id = 123;
```

### 패턴 4: OR 조건

```sql
-- 인덱스 안 탐 (또는 비효율): OR로 서로 다른 컬럼
SELECT * FROM users WHERE email = 'kim@..' OR country = 'KR';

-- 해결 1: UNION ALL로 분리 (각 브랜치에 인덱스 적용)
SELECT * FROM users WHERE email = 'kim@..'
UNION ALL
SELECT * FROM users WHERE country = 'KR' AND email != 'kim@..';

-- 해결 2: 각 컬럼에 인덱스 후 옵티마이저에 맡기기 (BitmapOr 활용)
-- PostgreSQL은 Bitmap Index Scan을 BitmapOr로 합칠 수 있음
```

### 패턴 5: 왼쪽 접두사 위반 (복합 인덱스)

이미 5.3절에서 다뤘지만 다시 강조한다. `idx_on(country, age)` 가 있을 때:

```sql
-- 인덱스 안 탐
SELECT * FROM users WHERE age = 30;  -- country 없이 age만
```

## 5.9 인덱스의 대가: 쓰기 성능 트레이드오프

인덱스는 **공짜가 아니다.** 얻는 것과 잃는 것이 있다.

**얻는 것:**
- SELECT 성능 O(n) → O(log n)
- ORDER BY, JOIN 조건의 정렬 비용 감소

**잃는 것:**
- INSERT: 각 인덱스의 B+tree에 키 추가 (분할 비용 가능)
- UPDATE: 인덱스 컬럼 변경 시 기존 키 삭제 + 새 키 삽입
- DELETE: 인덱스 키 삭제 (즉시 공간 회수가 아닌 VACUUM 때 회수)
- 디스크 공간: 인덱스마다 별도 공간 차지

**인덱스를 걸면 손해인 경우:**

| 상황 | 이유 |
|------|------|
| 카디널리티가 낮은 컬럼 (성별, 상태 플래그) | 선택도 높아 옵티마이저가 Seq Scan 선택 |
| 쓰기가 매우 빈번한 컬럼 | 인덱스 갱신 오버헤드가 조회 이득보다 큼 |
| 행 수가 적은 테이블 (수천 건 이하) | Seq Scan이 이미 충분히 빠름 |
| 거의 조회 안 되는 컬럼 | 공간과 쓰기 비용만 낭비 |

**원칙**: WHERE, JOIN ON, ORDER BY에 자주 등장하는 고유값이 많은 컬럼에만 인덱스를 걸고, 반드시 `EXPLAIN (ANALYZE)`로 효과를 확인한다.

```sql
-- 현재 인덱스 크기 확인
SELECT
    indexrelname AS 인덱스명,
    pg_size_pretty(pg_relation_size(indexrelid)) AS 크기,
    idx_scan AS 스캔횟수,
    idx_tup_read AS 읽은튜플수
FROM pg_stat_user_indexes
WHERE relname = 'users'
ORDER BY pg_relation_size(indexrelid) DESC;

-- 사용되지 않는 인덱스 찾기 (idx_scan이 0인 것들)
SELECT schemaname, tablename, indexrelname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;
```

## 5.10 EXPLAIN (ANALYZE)로 실행계획 읽기

### EXPLAIN의 두 가지 모드

```sql
-- EXPLAIN: 옵티마이저가 세운 실행계획만 출력 (실제 실행 안 함)
EXPLAIN SELECT * FROM users WHERE email = 'kim@example.com';

-- EXPLAIN ANALYZE: 계획을 실제로 실행하고 측정값도 출력
EXPLAIN (ANALYZE) SELECT * FROM users WHERE email = 'kim@example.com';

-- EXPLAIN ANALYZE BUFFERS: I/O 블록 통계까지 출력 (랩 04에서 사용)
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM users WHERE email = 'kim@example.com';
```

### Seq Scan 출력 해석

```
Seq Scan on big_user  (cost=0.00..20406.00 rows=1 width=48)
                       (actual time=95.231..120.445 rows=1 loops=1)
  Filter: (email = 'user777777@example.com')
  Rows Removed by Filter: 999999
Execution Time: 120.610 ms
```

| 항목 | 의미 |
|------|------|
| `cost=0.00..20406.00` | 옵티마이저 추정 비용 (시작비용..총비용). 단위는 상대적(디스크 1페이지 읽기 ≈ 1.0) |
| `rows=1` | 반환될 것으로 추정한 행 수 |
| `width=48` | 행 하나의 평균 바이트 크기 |
| `actual time=95.231..120.445` | 첫 행까지 걸린 실제 시간 .. 마지막 행까지 (ms) |
| `actual rows=1` | 실제로 반환된 행 수 |
| `loops=1` | 이 노드가 반복 실행된 횟수 (nested loop의 안쪽 노드는 >1) |
| `Filter:` | 행을 읽은 후 조건을 검사 |
| `Rows Removed by Filter: 999999` | 조건을 통과 못해 버린 행 수 → **읽었지만 버린 행** |
| `Execution Time: 120.610 ms` | 전체 실제 실행 시간 |

### Index Scan 출력 해석

```
Index Scan using idx_big_user_email on big_user
                       (cost=0.42..8.44 rows=1 width=48)
                       (actual time=0.041..0.043 rows=1 loops=1)
  Index Cond: (email = 'user777777@example.com')
Execution Time: 0.059 ms
```

| 항목 | 의미 |
|------|------|
| `Index Scan using idx_big_user_email` | 어떤 인덱스를 사용했는지 명시 |
| `Index Cond:` | 인덱스로 처리한 조건 (인덱스 트리를 탐색한 조건) |
| `Rows Removed by Filter:` | 없음 → 인덱스 조건이 정확히 걸러줌 |
| 실행 시간 | **0.059 ms** vs Seq Scan의 **120.610 ms** → 약 2,045배 빠름 |

### Index Only Scan 출력 해석

```
Index Only Scan using idx_big_user_email_country on big_user
                       (cost=0.42..4.44 rows=1 width=32)
                       (actual time=0.019..0.021 rows=1 loops=1)
  Index Cond: (email = 'user777777@example.com')
  Heap Fetches: 0
Execution Time: 0.031 ms
```

| 항목 | 의미 |
|------|------|
| `Index Only Scan` | 힙 접근 없이 인덱스만으로 완료 |
| `Heap Fetches: 0` | 힙 블록을 0번 읽음 → 커버링 인덱스 완전 활용 |

### 노드 트리 읽는 법

EXPLAIN 출력은 **들여쓰기로 트리를 표현**한다. **안쪽(더 들여쓴) 노드가 먼저 실행**되고, 결과를 바깥쪽 노드에 전달한다.

```
Hash Join  (cost=...) (actual time=...)       ← 3. 마지막에 조인
  Hash Cond: (o.user_id = u.id)
  ->  Seq Scan on orders o  (...)             ← 1. 먼저 orders 스캔
  ->  Hash  (...)                             ← 2. users로 해시 테이블 빌드
        ->  Index Scan on users u  (...)
```

실행 순서: `Index Scan on users` → `Hash (빌드)` → `Seq Scan on orders` → `Hash Join`

### 실행계획 항목 요약 표

| 항목 | 설명 |
|------|------|
| `Seq Scan` | 테이블 전체 순차 스캔 |
| `Index Scan` | 인덱스로 위치 찾은 후 힙 접근 |
| `Index Only Scan` | 인덱스만으로 결과 반환 (힙 미접근) |
| `Bitmap Index Scan` | 여러 인덱스를 비트맵으로 합쳐서 스캔 |
| `Nested Loop` | 중첩 루프 조인 |
| `Hash Join` | 해시 조인 |
| `Merge Join` | 정렬 후 병합 조인 |
| `cost=시작..총합` | 옵티마이저 추정 비용 (낮을수록 좋음) |
| `rows=N` (추정) | 반환 행 수 추정치 (실제와 크게 다르면 통계 낡음 → ANALYZE) |
| `actual rows=N` | 실제 반환 행 수 (ANALYZE 시에만 보임) |
| `loops=N` | 해당 노드 반복 실행 횟수 |
| `Rows Removed by Filter` | 읽었지만 조건 통과 못해 버린 행 수 |
| `Execution Time` | 전체 실제 실행 시간 (ms) |

## ⚠️ 흔한 오해와 함정

**오해 1: "인덱스는 무조건 빠르다"**

인덱스가 있어도 옵티마이저가 Seq Scan을 선택할 수 있다. 결과 행이 전체의 10% 이상이거나, 테이블이 작거나, 통계가 낡았거나, 선택도가 높은 컬럼이면 Seq Scan이 오히려 빠르다. "EXPLAIN 찍어보기 전까지 인덱스가 쓰인다고 가정하지 말라."

**오해 2: "인덱스를 많이 걸수록 좋다"**

INSERT/UPDATE/DELETE마다 인덱스도 갱신된다. 쓰기 빈도가 높은 테이블에 인덱스가 많으면 오히려 전체 처리량이 줄 수 있다. "일단 다 걸자"는 안티패턴이다.

**오해 3: "복합 인덱스는 순서가 상관없다"**

`(country, age)`와 `(age, country)`는 완전히 다른 인덱스다. 왼쪽 접두사 규칙을 항상 고려해야 한다.

**오해 4: "WHERE에 컬럼을 쓰면 인덱스가 무조건 탄다"**

`WHERE lower(email) = ?`, `WHERE CAST(id AS TEXT) = ?`, `WHERE email LIKE '%kim'` — 이런 패턴은 인덱스를 무력화한다. 함수, 타입 변환, 앞 와일드카드를 피하거나 표현식 인덱스를 만들어야 한다.

**오해 5: "PK는 자동으로 인덱스가 걸린다고 했으니 다른 컬럼은 모두 수동으로"**

PostgreSQL의 `UNIQUE` 제약도 자동으로 유니크 인덱스를 만든다. JPA의 `@Column(unique=true)` 역시 마찬가지다. 이미 만들어진 인덱스를 중복으로 또 만드는 실수를 하지 말 것.

**오해 6: "통계는 자동으로 항상 최신 상태다"**

autovacuum이 통계를 자동 갱신하지만, 대량의 INSERT/DELETE 직후에는 통계가 일시적으로 낡을 수 있다. 대량 데이터 변경 후에는 `ANALYZE 테이블명`을 명시적으로 실행하라.

**함정: 인덱스 컬럼에 NULL 있는 경우**

PostgreSQL의 B-tree 인덱스는 NULL을 인덱싱한다 (일부 다른 DB와 다름). 단, `WHERE col IS NULL` 쿼리도 인덱스를 활용할 수 있다. 그러나 `WHERE col = NULL` (IS NULL이 아닌 = NULL)은 항상 빈 결과이므로 주의.

### 문제 1: 인덱스 설계

아래 쿼리가 자주 실행된다. 어떤 인덱스를 만들겠는가? 이유도 설명하라.

```sql
-- 쿼리 A: 특정 국가의 최근 주문 목록 (페이지네이션)
SELECT order_id, user_id, amount, created_at
FROM orders
WHERE country = 'KR' AND status = 'COMPLETED'
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;

-- 쿼리 B: 특정 사용자의 총 주문 금액 합산
SELECT user_id, SUM(amount)
FROM orders
WHERE user_id = 12345 AND created_at >= '2024-01-01'
GROUP BY user_id;
```

<details>
<summary>힌트</summary>

- 쿼리 A: `WHERE` 조건에 등가 컬럼이 두 개 있고 `ORDER BY`도 있다. 복합 인덱스의 컬럼 순서를 고려하라. 커버링 인덱스를 적용하면 `Index Only Scan`이 가능한가?
- 쿼리 B: `user_id` 등가 조건이 있고 `created_at` 범위 조건이 있다. 어느 컬럼이 앞에 와야 하는가?

</details>

<details>
<summary>해설</summary>

**쿼리 A:**
```sql
-- 등가 조건(country, status) → 앞에, ORDER BY 컬럼(created_at) → 뒤에
CREATE INDEX idx_orders_country_status_date
    ON orders(country, status, created_at DESC);
```
- `WHERE country='KR' AND status='COMPLETED'`로 범위를 좁힌 뒤, `created_at DESC` 순서로 이미 정렬되어 있어 정렬 비용 없음.
- SELECT 컬럼이 `order_id, user_id, amount, created_at`인데 이를 모두 INCLUDE에 넣으면 `Index Only Scan`도 가능:

```sql
CREATE INDEX idx_orders_cover
    ON orders(country, status, created_at DESC)
    INCLUDE (order_id, user_id, amount);
```

**쿼리 B:**
```sql
-- user_id 등가 조건 앞에, created_at 범위 조건 뒤에
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);
```
- 반환 컬럼이 `user_id, SUM(amount)`인데, `amount`는 집계에 필요하므로 커버링을 원하면 `INCLUDE (amount)` 추가:
```sql
CREATE INDEX idx_orders_user_date_cover
    ON orders(user_id, created_at) INCLUDE (amount);
```

</details>

### 문제 2: EXPLAIN 출력 해석

다음 EXPLAIN 출력을 보고 질문에 답하라.

```
Seq Scan on big_user  (cost=0.00..21406.00 rows=200000 width=48)
                      (actual time=0.012..138.433 rows=200000 loops=1)
  Filter: ((country)::text = 'KR'::text)
  Rows Removed by Filter: 800000
Execution Time: 156.221 ms
```

1. 이 쿼리는 어떤 스캔 방식을 사용했는가?
2. `Rows Removed by Filter: 800000`은 무엇을 의미하는가?
3. 이 쿼리에 인덱스를 걸어도 효과가 없을 수 있다. 왜인가?
4. 만약 쿼리가 `WHERE country = 'KR' AND age = 30`으로 바뀐다면 어떤 인덱스를 제안하겠는가?

<details>
<summary>해설</summary>

1. **Seq Scan**: 테이블 전체를 처음부터 끝까지 순차 스캔했다.

2. **100만 행 중 80만 행을 읽었지만 조건(country='KR')을 통과하지 못해 버렸다.** 5개 국가가 균등 분포라면 country='KR'은 전체의 20% = 20만 행을 반환한다. 나머지 80만 행이 `Rows Removed by Filter`다.

3. **선택도가 높기 때문이다.** country 컬럼의 고유값이 5개뿐이라 선택도 = 1/5 = 0.2 (20%)이다. 전체 행의 20%를 반환해야 하므로, 옵티마이저 입장에서는 인덱스를 통한 20만 번의 랜덤 I/O보다 Seq Scan이 효율적이라 판단할 수 있다. 단, 조건이 더 추가되어 반환 행이 줄면 인덱스가 효과를 낸다.

4. `WHERE country = 'KR' AND age = 30`이라면 선택도가 1/5 × 1/50 = 1/250 = 0.4%로 크게 낮아진다. 이때는 인덱스가 효과적이다:
```sql
CREATE INDEX idx_users_country_age ON users(country, age);
```
(country 앞, age 뒤 — 등가 조건 두 개이므로 순서는 바꿔도 되지만 country가 더 선택도가 낮은 축이 앞에 오면 좋다)

</details>

### 문제 3: 인덱스가 타는가?

`CREATE INDEX idx_on_email ON users(email)`이 있다고 가정한다. 아래 쿼리들 중 이 인덱스를 **타지 않는** 것을 모두 고르고 이유를 설명하라.

```sql
-- A
SELECT * FROM users WHERE email = 'kim@example.com';
-- B
SELECT * FROM users WHERE lower(email) = 'kim@example.com';
-- C
SELECT * FROM users WHERE email LIKE 'kim%';
-- D
SELECT * FROM users WHERE email LIKE '%kim';
-- E
SELECT * FROM users WHERE email IS NULL;
-- F
SELECT * FROM users WHERE email != 'kim@example.com';
```

<details>
<summary>해설</summary>

- **A**: 인덱스 탐. `=` 등가 조건, 컬럼 그대로.
- **B**: **인덱스 안 탐.** `lower(email)`은 인덱스 키(`email`)와 다른 표현식이다. `lower(email)` 인덱스가 별도로 필요하다.
- **C**: 인덱스 탐. 뒤 와일드카드 LIKE는 B-tree에서 prefix 검색으로 처리.
- **D**: **인덱스 안 탐 (B-tree).** 앞 와일드카드 LIKE는 시작 위치를 특정할 수 없어 B-tree를 못 쓴다. pg_trgm + GIN 인덱스가 필요.
- **E**: 인덱스 탐 (PostgreSQL). PG의 B-tree는 NULL도 인덱싱하므로 `IS NULL`에 인덱스 활용 가능.
- **F**: 조건에 따라 다름. `!=`는 대부분의 행을 반환하므로 선택도가 높아 옵티마이저가 Seq Scan을 선택할 가능성이 높다. 기술적으로 인덱스를 타는 경우도 있으나 효율이 낮다.

</details>

### 문제 4: 커버링 인덱스 설계

아래 쿼리를 `Index Only Scan`으로 만들려면 어떤 인덱스를 생성해야 하는가?

```sql
SELECT email, name, age FROM users WHERE email LIKE 'kim%' ORDER BY age;
```

기존에 `CREATE INDEX idx_email ON users(email)` 만 있다고 가정하라.

<details>
<summary>해설</summary>

현재 `idx_email`로는 WHERE 조건은 처리하지만, SELECT의 `name, age`를 힙에서 읽어와야 하므로 `Index Scan`이 된다.

`Index Only Scan`을 만들려면 SELECT에 필요한 모든 컬럼(email, name, age)이 인덱스에 있어야 한다. 그리고 `ORDER BY age`가 있으므로 정렬도 고려한다.

```sql
CREATE INDEX idx_email_cover
    ON users(email, age)  -- email 검색 + age 정렬
    INCLUDE (name);       -- name은 정렬 키 불필요, INCLUDE로 저장
```

또는 ORDER BY와 인덱스 정렬 방향을 맞추면 정렬 비용도 없앨 수 있다:
```sql
CREATE INDEX idx_email_age_cover
    ON users(email, age ASC)
    INCLUDE (name);
```

이제 `SELECT email, name, age FROM users WHERE email LIKE 'kim%' ORDER BY age`는 인덱스만으로 처리 가능 → `Index Only Scan`.

</details>

### 문제 5: JPA 연결 — 어떤 인덱스가 생성되는가

아래 JPA Entity 코드에서 PostgreSQL에 어떤 인덱스가 자동으로 생성되는가? 그리고 `findByEmailAndCountry(String email, String country)` 쿼리에 최적화된 인덱스를 `@Table(indexes = ...)` 애노테이션으로 추가하려면 어떻게 작성하는가?

```java
@Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true, nullable = false)
    private String email;

    @Column(nullable = false)
    private String country;

    private Integer age;
}
```

<details>
<summary>해설</summary>

**자동 생성되는 인덱스:**
1. `id` PK → PostgreSQL이 자동으로 유니크 B-tree 인덱스 생성
2. `email`의 `unique = true` → 유니크 B-tree 인덱스 자동 생성 (`uk_users_email` 또는 유사한 이름)

**복합 인덱스 추가:**
```java
@Entity
@Table(
    name = "users",
    indexes = {
        @Index(name = "idx_users_email_country", columnList = "email, country")
    }
)
public class User { ... }
```

`findByEmailAndCountry`는 `WHERE email = ? AND country = ?` 쿼리를 생성하므로, `(email, country)` 복합 인덱스가 최적이다. email이 이미 고유(선택도 매우 낮음)하므로 email을 앞에 배치한다.

단, email에 이미 유니크 인덱스가 있으므로, `WHERE email = ?`만으로도 이미 1건 이하로 좁혀진다. 이 경우 복합 인덱스의 추가 효과는 크지 않다. 만약 email이 unique가 아니라면 `(email, country)` 복합 인덱스가 의미 있다.

Spring 개념서 §4.3([JpaRepository](../../../spring/docs/SPRING_개념서/README.md))과 §4.7([인덱스와 B-tree](../../../spring/docs/SPRING_개념서/README.md)) 참조.

</details>

## 요약

| 개념 | 핵심 |
|------|------|
| Seq Scan vs Index Scan | O(n) vs O(log n). 100만 행: 100만 번 vs 3~4번 디스크 접근 |
| B+tree 구조 | 내부 노드 = 길안내판, 리프 = 데이터 포인터 + 링크드 리스트. 팬아웃 400이면 높이 3~4 |
| 왼쪽 접두사 규칙 | `(country, age)` 인덱스 → `WHERE country` O, `WHERE age` X |
| 커버링 인덱스 | INCLUDE 컬럼 포함 → Index Only Scan, Heap Fetches: 0 |
| 인덱스 종류 | B-tree(범위), Hash(등가), GIN(JSONB/배열), GiST(기하), BRIN(시계열), 부분/표현식 |
| 선택도 | 낮을수록(고유값 많을수록) 인덱스 효율 ↑. 높으면 옵티마이저가 Seq Scan 선택 |
| 인덱스의 대가 | 읽기 O(n)→O(log n) 얻고, 쓰기 저하 + 디스크 공간 잃음 |
| EXPLAIN 항목 | Seq Scan/Index Scan/Index Only Scan, cost, actual time, rows, Rows Removed by Filter, Execution Time |
| 인덱스 안 타는 패턴 | 함수 적용, 앞 와일드카드 LIKE, 타입 변환, 왼쪽 접두사 위반 |

**Spring 개념서 상호참조:**
- 4.7절 인덱스와 B-tree → 이 장의 5.1~5.2
- 4.4절 N+1 문제 → 6장의 Nested Loop 조인과 연결
- 4.3절 JpaRepository → `@Column(unique=true)`, `@Table(indexes=...)` 실체가 이 장의 인덱스
