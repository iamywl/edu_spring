## 도입: 선언형 언어와 실행의 간극

프로그래머가 작성하는 SQL은 **선언형(declarative)**이다.

```sql
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.country = 'KR'
GROUP BY u.name
ORDER BY order_count DESC
LIMIT 10;
```

이 쿼리는 "한국 사용자 중 주문이 많은 상위 10명을 이름과 주문 수로 알려달라"는 **의도**만 담고 있다. 구체적으로:

- `users`와 `orders`를 어떤 순서로 읽을 것인가?
- 조인을 어떤 알고리즘으로 처리할 것인가?
- GROUP BY 결과를 메모리에서 처리할 것인가, 디스크를 쓸 것인가?
- 인덱스를 사용할 것인가, 순차 스캔을 할 것인가?

이 모든 결정을 **쿼리 처리기(Query Processor)**가 자동으로 내린다.

**왜 이것을 알아야 하는가?**

JPA를 쓰는 개발자는 `repository.findTop10ByCountryOrderByOrderCountDesc("KR")` 한 줄을 작성한다. 그런데 이 한 줄이 만들어내는 SQL의 실행이 **0.1ms인지 30초인지**는 쿼리 처리기의 결정에 달려 있다. 그 결정을 이해해야 올바른 인덱스를 걸고, 올바른 쿼리를 작성하고, EXPLAIN 출력을 해석해서 문제를 진단할 수 있다.

이 장이 끝나면, `labs/04_indexing_explain.sql`의 EXPLAIN 출력에서 조인 노드를 읽고 해석할 수 있다. 그리고 두 테이블을 JOIN할 때 어떤 알고리즘이 선택되는지, 그 비용은 어떻게 추정되는지를 설명할 수 있다.

5장(인덱싱)이 "무엇이 찾기를 빠르게 하는가"라면, 이 장은 "데이터베이스가 쿼리 전체를 어떻게 계획하는가"이다.

## 6.1 SQL이 실행되는 파이프라인: 텍스트에서 결과까지

SQL 문자열이 결과 행으로 변환되기까지 다섯 단계를 거친다.

```
SQL 텍스트
    │
    ▼
┌─────────────────────────────────────────┐
│  1. 파서 (Parser)                        │
│     SQL 문법 검사, 파스 트리(parse tree) 생성 │
│     오류: "syntax error at or near ..."  │
└──────────────────┬──────────────────────┘
                   │ 파스 트리
                   ▼
┌─────────────────────────────────────────┐
│  2. 분석기 / 재작성기 (Analyzer/Rewriter) │
│     테이블·컬럼 이름 검증, 타입 체크       │
│     뷰(View) 정의로 확장                  │
│     Rule 시스템 적용 (PostgreSQL 전용)    │
└──────────────────┬──────────────────────┘
                   │ 쿼리 트리
                   ▼
┌─────────────────────────────────────────┐
│  3. 논리적 계획 (Logical Plan)           │
│     관계 대수(relational algebra)로 표현  │
│     "무엇을" — SELECT, JOIN, FILTER, GROUP │
│     여러 동치 계획이 가능 (같은 결과, 다른 순서) │
└──────────────────┬──────────────────────┘
                   │ 논리 계획
                   ▼
┌─────────────────────────────────────────┐
│  4. 옵티마이저 (Optimizer)               │
│     통계 기반으로 가장 비용이 낮은 물리 계획 탐색 │
│     인덱스 선택, 조인 순서, 조인 알고리즘 결정 │
│     → EXPLAIN이 보여주는 것이 바로 이 출력 │
└──────────────────┬──────────────────────┘
                   │ 물리적 계획 (실행 트리)
                   ▼
┌─────────────────────────────────────────┐
│  5. 실행기 (Executor)                    │
│     물리 계획대로 실제 데이터 읽고 처리   │
│     결과 행을 클라이언트로 스트리밍       │
└─────────────────────────────────────────┘
                   │ 결과 행들
                   ▼
             클라이언트 (JDBC, psql)
```

### 논리적 계획과 물리적 계획의 차이

**논리적 계획**: "무엇을 하는가"를 관계 대수 연산으로 표현.

```
π_{name, order_count}(
  σ_{country='KR'}(users)
  ⋈_{users.id = orders.user_id}
  γ_{name, COUNT(id) → order_count}(orders)
)
```

- π = 프로젝션(SELECT 컬럼 선택)
- σ = 셀렉션(WHERE 조건)
- ⋈ = 조인
- γ = 그루핑(GROUP BY)

같은 논리 계획을 표현하는 방법이 여럿이다. 예를 들어:
- users를 먼저 필터링 후 조인?
- orders를 먼저 읽고 조인 후 필터링?
- 조인 전에 GROUP BY?

이 중 어느 순서가 더 빠른지는 논리 계획만으로는 알 수 없다.

**물리적 계획**: "어떻게 하는가"를 구체적인 알고리즘으로 표현.

```
Limit (rows=10)
  └─ Sort (order_count DESC)
       └─ HashAggregate (GROUP BY name)
            └─ Hash Join (users.id = orders.user_id)
                 ├─ Index Scan on users (WHERE country='KR')
                 └─ Seq Scan on orders
```

물리 계획에서는 각 연산의 구현 알고리즘이 명시된다. EXPLAIN이 보여주는 것이 바로 이 물리 계획이다.

### 옵티마이저의 역할

옵티마이저(Optimizer)는 논리 계획을 받아 **비용(cost)**이 가장 낮은 물리 계획을 찾는다.

```
논리 계획 하나에 가능한 물리 계획들:

  계획 A: Seq Scan(orders) → Hash Join → Index Scan(users) → Sort → Limit
    추정 비용: 45,230

  계획 B: Index Scan(users, country='KR') → Nested Loop → Seq Scan(orders) → Sort → Limit
    추정 비용: 12,105  ← 옵티마이저가 이것을 선택

  계획 C: Merge Join(정렬된 양쪽) → Filter → Group → Sort → Limit
    추정 비용: 18,440
```

옵티마이저가 비용을 추정하는 기반이 **통계 정보**다. `pg_stats`에 저장된 테이블 크기, 컬럼 카디널리티, 값 분포 히스토그램을 이용해 "이 조건으로 행이 몇 개나 남을지"를 추정하고, 각 알고리즘의 I/O 비용을 더해 최종 비용을 계산한다.

**추정 rows가 실제 rows와 크게 다르다면**: 통계가 낡았거나(`ANALYZE` 필요), 값 분포가 히스토그램에 잘 잡히지 않는 경우다. 이것이 "나쁜 실행 계획"의 가장 흔한 원인이다.

## 6.2 옵티마이저 심화: 규칙 기반 vs 비용 기반

### 규칙 기반 옵티마이저 (RBO)

초기 DB 시스템(Oracle 구버전 등)은 **규칙(rule) 목록**을 가지고 있었다. "인덱스가 있으면 항상 사용한다", "인덱스 범위 스캔은 전체 스캔보다 항상 낫다" 같은 규칙. 통계 정보 없이 규칙만 적용한다.

**문제**: 규칙이 항상 옳지 않다. 100만 행 중 90%를 반환해야 할 때도 규칙대로 인덱스를 쓰면 오히려 느리다. 5장에서 배운 선택도 개념이 바로 이 한계를 극복하기 위한 것이다.

### 비용 기반 옵티마이저 (CBO)

현대 RDBMS(PostgreSQL, MySQL 8.0+, Oracle CBO 등)는 **비용(cost) 추정**을 기반으로 계획을 선택한다. 여러 후보 계획의 비용을 계산해 최솟값을 고른다.

**PostgreSQL의 비용 단위**: 디스크 페이지 1개를 순차 읽는 비용을 `seq_page_cost = 1.0`으로 기준 잡는다. 다른 비용은 이에 상대적이다:

```sql
SHOW seq_page_cost;         -- 1.0 (기본)
SHOW random_page_cost;      -- 4.0 (기본) — 랜덤 I/O는 순차보다 4배 비쌈
SHOW cpu_tuple_cost;        -- 0.01 (기본)
SHOW cpu_index_tuple_cost;  -- 0.005 (기본)
SHOW cpu_operator_cost;     -- 0.0025 (기본)
```

SSD 환경이라면 `random_page_cost`를 낮춰주면 인덱스가 더 자주 선택된다:
```sql
SET random_page_cost = 1.1;  -- SSD에서 순차와 랜덤 차이 적음
```

### 조인 순서 탐색: 조합 폭발

N개 테이블을 조인하는 경우, 가능한 조인 순서는 N! (팩토리얼)이다:
- 테이블 2개: 2! = 2가지
- 테이블 5개: 5! = 120가지
- 테이블 10개: 10! = 3,628,800가지
- 테이블 20개: 20! ≈ 2.4 × 10^18 (실질적으로 탐색 불가)

PostgreSQL은 두 가지 전략으로 이 조합 폭발을 처리한다:

1. **동적 계획법 (Dynamic Programming, System R 방식)**: 테이블 수가 `join_collapse_limit`(기본 8) 이하이면, 부분 계획들의 최적해를 기억하며 상향식으로 최적 계획을 찾는다.

2. **유전 알고리즘 (GEQO)**: `geqo_threshold`(기본 12) 이상의 테이블을 조인할 때, 진화 알고리즘으로 좋은 계획을 근사 탐색한다. 최적이 아닐 수 있지만 합리적인 시간 안에 찾는다.

```sql
-- 옵티마이저가 탐색하는 조인 순서 범위 조정
SET join_collapse_limit = 1;   -- 쿼리 작성 순서대로만 (자동 최적화 끔)
SET join_collapse_limit = 8;   -- 기본값 (최대 8개 테이블까지 DP)
```

## 6.3 조인 알고리즘 3종: Nested Loop, Hash, Merge

조인은 두 테이블의 행들 중 조건을 만족하는 쌍을 찾는 연산이다. 이를 구현하는 알고리즘 세 가지가 있으며, 상황에 따라 최적 선택이 달라진다. EXPLAIN에서 `Nested Loop`, `Hash Join`, `Merge Join`으로 명시된다.

### 6.3.1 Nested Loop Join (중첩 루프 조인)

**알고리즘:**

```
for each row r in 외부 테이블 A:
    for each row s in 내부 테이블 B:
        if r.key == s.key:
            output(r, s)
```

```
외부 테이블 A (users, 필터 후 5행)    내부 테이블 B (orders)
┌────────┐                           ┌────────────────────┐
│ user_1 │ → B 전체 스캔 or 인덱스   │ 모든 orders 행 탐색 │
│ user_2 │ → B 전체 스캔 or 인덱스   │                    │
│ user_3 │ → B 전체 스캔 or 인덱스   │                    │
│ user_4 │ → B 전체 스캔 or 인덱스   │                    │
│ user_5 │ → B 전체 스캔 or 인덱스   │                    │
└────────┘                           └────────────────────┘
```

**비용:**
- B에 인덱스 없음: O(|A| × |B|) — |A|와 |B|는 각 테이블의 행 수
- B에 인덱스 있음: O(|A| × log|B|) — 내부 테이블을 인덱스로 조회

**최적 상황:**
- 외부 테이블(A)의 결과 행이 **매우 적을 때**
- 내부 테이블(B)에 조인 조건 컬럼에 **인덱스가 있을 때**

**JPA N+1 문제와의 연결:**

JPA의 N+1 문제(Spring 개념서 [§4.4](../../../spring/docs/SPRING_개념서/README.md))는 사실상 **애플리케이션 레벨의 나쁜 Nested Loop**이다.

```java
// N+1: 모든 Post를 가져온 후(1번), 각 Post마다 Author를 별도 조회(N번)
List<Post> posts = postRepository.findAll();   // 쿼리 1번
for (Post post : posts) {
    System.out.println(post.getAuthor().getName());  // 쿼리 N번
}
```

이것은 DB 안에서 최적화된 Nested Loop가 아니라, 애플리케이션과 DB 사이의 네트워크를 N번 오가는 최악의 패턴이다. DB 내에서는 인덱스를 쓰는 Nested Loop가 효율적이지만, 앱 레벨의 N+1은 네트워크 왕복 비용이 추가되어 훨씬 나쁘다.

### 6.3.2 Hash Join (해시 조인)

**알고리즘:**

```
[Build 단계] 작은 테이블 B로 해시 테이블 구축
  for each row s in B:
      hashtable[s.key] = s    (O(|B|) 시간)

[Probe 단계] 큰 테이블 A를 순회하며 해시 테이블 조회
  for each row r in A:
      if hashtable[r.key] exists:
          output(r, hashtable[r.key])    (O(1) 해시 조회)

전체 비용: O(|A| + |B|)  (해시 빌드 O(B) + 프로브 O(A))
```

```
[Build 단계]
users (작은 테이블, 10만 행)
→ 메모리 해시 테이블 구축
  { user_id_1: 행1, user_id_2: 행2, ... }

[Probe 단계]
orders (큰 테이블, 100만 행)
→ 각 행의 user_id로 해시 테이블 O(1) 조회
→ 매칭되면 결과에 추가
```

**비용:**
- O(|A| + |B|) — 양쪽을 각각 한 번씩만 읽음
- 해시 테이블 빌드 초기 비용이 있지만, 이후 조인은 O(1)

**최적 상황:**
- 두 테이블 모두 크고 인덱스가 없을 때
- 등가 조인(`=`)만 있을 때 (해시는 등가 비교만 가능)
- 해시 테이블이 `work_mem` 내에 들어올 때

**work_mem과 Grace Hash Join:**

해시 테이블이 `work_mem`(기본 4MB)을 초과하면 PostgreSQL은 **Grace Hash Join(파티션 해시 조인)**을 사용한다. 양쪽 테이블을 해시값으로 파티션 분할해 디스크에 쓴 뒤, 파티션 단위로 처리한다. 성능이 크게 저하될 수 있다.

```sql
-- 큰 조인 쿼리에서 work_mem 임시 증가 (세션 범위)
SET work_mem = '256MB';
EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
-- Execution Time이 줄면 work_mem 부족이 원인이었던 것
```

### 6.3.3 Merge Join (정렬 병합 조인)

**알고리즘:**

```
사전 조건: 두 테이블 모두 조인 키 기준으로 정렬되어 있어야 함

[정렬 단계] (이미 정렬되어 있지 않다면)
  A → 조인 키로 정렬 → A_sorted
  B → 조인 키로 정렬 → B_sorted

[병합 단계]
  포인터 i = A_sorted의 첫 행
  포인터 j = B_sorted의 첫 행

  while i ≤ |A| and j ≤ |B|:
      if A[i].key == B[j].key:
          output(A[i], B[j])
          j++ (또는 i++)
      elif A[i].key < B[j].key:
          i++
      else:
          j++
```

```
정렬된 A (users, id 기준)     정렬된 B (orders, user_id 기준)
  id=1, 이순신                  user_id=1, 주문A
  id=2, 김철수       ────▶     user_id=1, 주문B    ← 1:N 매칭
  id=3, 박영수                  user_id=2, 주문C
  id=4, 최지수                  user_id=3, 주문D
                                user_id=3, 주문E
                                user_id=5, 주문F

포인터를 나란히 움직이며 키 비교 → 마치 지퍼(zipper)처럼
```

**비용:**
- 정렬 비용 O(|A| log|A| + |B| log|B|) + 병합 O(|A| + |B|)
- 이미 정렬되어 있다면(인덱스 순서와 같다면) 정렬 비용 없음 → O(|A| + |B|)

**최적 상황:**
- 양쪽에 조인 키 인덱스가 있어 **이미 정렬된** 경우
- 범위 조인(`>=`, `<=`, `BETWEEN`)이 있을 때 (Hash Join은 등가만 가능)
- ORDER BY가 있고 조인 키와 정렬 키가 같을 때 — 조인과 정렬을 동시에

**외부 정렬(External Sort):**

정렬할 데이터가 `work_mem`을 초과하면 디스크를 쓰는 **외부 정렬**이 필요하다. EXPLAIN에서 `Sort Method: external merge`로 나타나며, 이때도 `work_mem` 증가가 도움이 된다.

### 세 조인 알고리즘 비교표

| 항목 | Nested Loop | Hash Join | Merge Join |
|------|-------------|-----------|------------|
| 알고리즘 | 이중 반복 | 빌드+프로브 | 정렬+병합 |
| 기본 비용 | O(|A|×|B|) | O(|A|+|B|) | O((A+B)log+A+B) |
| 인덱스 있을 때 | O(|A|×log|B|) | (변화 없음) | O(|A|+|B|) |
| 등가 조인 | O | O | O |
| 범위 조인 | O | X | O |
| 메모리 사용 | 적음 | 해시 테이블 | 정렬 버퍼 |
| **최적 상황** | A 작고 B에 인덱스 | 둘 다 크고 등가 | 이미 정렬 또는 범위 |
| JPA 연결 | N+1의 DB 버전 | 페치 조인 효과 | ORDER BY 조인 |

## 6.4 조인 비용 모델 정량화

이 절은 전공 수준의 비용 모델을 수식으로 다룬다. 실무에서는 EXPLAIN이 계산해주지만, 수식을 이해하면 EXPLAIN 출력을 더 깊이 해석할 수 있다.

**기호 정의:**
- B(R): 테이블 R의 블록(페이지) 수
- |R|: 테이블 R의 튜플(행) 수
- h_i: 인덱스 B+tree의 높이

### Nested Loop Join 비용

```
인덱스 없음 (Block Nested Loop 기준):
  비용 = B(A) + |A| × B(B)
  (A를 블록 단위로 읽고, A 각 행마다 B 전체를 읽음)

인덱스 있음:
  비용 = B(A) + |A| × (h_i + 1)
  (A를 읽고, A 각 행마다 B의 인덱스로 1건 조회 = h_i 단계 + 힙 1회)
```

**예시:**
- users: B(users)=1000 블록, |users|=100,000 행
- orders: B(orders)=5000 블록, |orders|=500,000 행
- orders(user_id)에 인덱스, h_i = 3

```
인덱스 없음:  1000 + 100,000 × 5000 = 500,001,000 블록 읽기 → 매우 비쌈
인덱스 있음:  1000 + 100,000 × (3 + 1) = 1000 + 400,000 = 401,000 블록 읽기
```

약 1,246배 차이. 인덱스의 위력이 수식으로도 명확하다.

### Hash Join 비용

```
[Build 단계]: B(B)  (작은 테이블 B를 읽어 해시 테이블 구축)
[Probe 단계]: B(A)  (큰 테이블 A를 순차 읽으며 해시 테이블 조회)

총 비용: B(A) + B(B)  (메모리에 해시가 맞을 때)

Grace Hash Join (파티션 분할, 메모리 부족 시):
  비용 = 3 × (B(A) + B(B))
  (읽기 + 파티션 쓰기 + 파티션 읽기)
```

**예시 (위와 동일 테이블, 인덱스 없음):**
```
Hash Join:     1000 + 5000 = 6,000 블록 읽기
Nested Loop:   500,001,000 블록 읽기 (인덱스 없을 때)

Hash Join이 83,000배 빠름 (인덱스가 없다면)
```

### Merge Join 비용

```
정렬 비용 (외부 정렬 기준):
  2 × B(R) × ceil(log_{B_B}(B(R)/B_B))
  (B_B = 버퍼 페이지 수, 즉 work_mem / page_size)

이미 정렬되어 있으면:
  비용 = B(A) + B(B)  (Hash Join과 동일)
```

이미 인덱스 순서로 정렬되어 있는 경우, Merge Join은 Hash Join과 동일한 선형 비용으로 동작하면서 범위 조인도 처리할 수 있다.

## 6.5 쿼리 실행 모델: Volcano 모델

PostgreSQL(그리고 대부분의 전통적 RDBMS)은 **Volcano(화산) 모델** 또는 **Iterator(반복자) 모델**로 쿼리를 실행한다.

### Volcano 모델 개념

모든 물리 계획 노드(Seq Scan, Hash Join, Sort 등)는 **동일한 인터페이스**를 구현한다:

```
interface Iterator {
    open()       // 초기화 (자식 노드들도 재귀적으로 open)
    next() → row // 다음 행 1개 반환 (없으면 EOF)
    close()      // 자원 해제
}
```

최상위 노드부터 `next()`를 호출하면, 그 노드가 자식 노드의 `next()`를 호출하고, 이것이 재귀적으로 전파되어 결과를 만든다.

```
Limit.next()
  └─ Sort.next()
       └─ HashAgg.next()
            └─ Hash Join.next()
                 ├─ Index Scan.next() ← 실제 데이터 소스
                 └─ Hash.next()
                      └─ Seq Scan.next() ← 실제 데이터 소스
```

### 파이프라이닝 vs 재료화

**파이프라이닝(Pipelining)**: 행 하나가 생성되는 즉시 상위 노드로 전달. 전체 중간 결과를 메모리에 올리지 않는다. `LIMIT`이 있을 때 특히 유리 — 충분한 행이 모이면 하위 노드를 중단할 수 있다.

```
LIMIT 10 WITH OFFSET 0:
  Sort.next() → 10번 호출 후 Limit이 close() → 하위 노드 중단
  전체 정렬이 필요 없을 수도 있음 (이미 인덱스로 정렬된 경우)
```

**재료화(Materialization)**: 일부 노드는 중간 결과 전체를 메모리(또는 디스크)에 저장해야만 다음 단계로 넘어갈 수 있다.

```
재료화가 필요한 경우:
  - Sort: 모든 행을 받아야 정렬 가능
  - Hash Join의 Build 단계: 해시 테이블을 완전히 구축해야 Probe 가능
  - HashAggregate: 모든 행을 받아야 집계 완료
```

재료화 노드가 `work_mem`을 초과하면 디스크 임시 파일(`Temp File`)을 사용한다. EXPLAIN에서:

```
Sort  (cost=...)
  Sort Key: order_count DESC
  Sort Method: external merge  Disk: 15384kB   ← 디스크 사용! work_mem 늘려야
```

### PostgreSQL 16의 병렬 쿼리

PostgreSQL 10부터 도입된 **병렬 쿼리**는 여러 CPU 코어를 활용해 쿼리 처리를 병렬화한다.

```
Gather  (cost=...) (actual time=...)            ← 병렬 결과 합산
  Workers Planned: 4
  Workers Launched: 4
  ->  Parallel Seq Scan on big_table  (...)     ← 4개 워커가 테이블 나눠 읽기
```

병렬 쿼리가 자동으로 활성화되려면:
- 테이블이 충분히 클 것 (`min_parallel_table_scan_size`, 기본 8MB)
- 비용이 `parallel_setup_cost` + `parallel_tuple_cost`를 정당화할 것
- `max_parallel_workers_per_gather` 가 0이 아닐 것 (기본 2)

병렬 처리가 가능한 연산: `Parallel Seq Scan`, `Parallel Index Scan`, `Parallel Hash Join`, `Parallel Aggregate`.

## 6.6 서브쿼리와 집계 처리

### 상관 서브쿼리의 비효율

**상관 서브쿼리(correlated subquery)**는 외부 쿼리의 값을 참조하는 서브쿼리다.

```sql
-- 각 사용자의 평균보다 많이 주문한 목록
SELECT * FROM orders o
WHERE amount > (
    SELECT AVG(amount)
    FROM orders
    WHERE user_id = o.user_id   -- 외부 쿼리의 o.user_id를 참조
);
```

이 쿼리는 orders를 순회하는 외부 루프마다 서브쿼리를 실행한다. `|orders|`번 서브쿼리 실행 → Nested Loop와 동일한 O(N²) 위험.

**옵티마이저의 비상관화(decorrelation/unnesting)**:

현대 옵티마이저는 많은 경우 상관 서브쿼리를 자동으로 조인으로 변환한다.

```sql
-- 옵티마이저가 내부적으로 변환:
SELECT o.*
FROM orders o
JOIN (
    SELECT user_id, AVG(amount) AS avg_amount
    FROM orders
    GROUP BY user_id
) avg_o ON avg_o.user_id = o.user_id
WHERE o.amount > avg_o.avg_amount;
```

EXPLAIN에서 `SubqueryScan` 대신 `Hash Join`이 나오면 성공적으로 비상관화된 것이다. 옵티마이저가 못 바꾸는 복잡한 상관 서브쿼리는 직접 조인으로 재작성하는 것이 좋다.

### GROUP BY: HashAggregate vs GroupAggregate

GROUP BY도 두 가지 구현 방식이 있다.

**HashAggregate**: 해시 테이블에 각 그룹키 → 집계값을 저장. 인풋이 정렬되지 않아도 됨. `work_mem`을 쓴다.

```
HashAggregate  (Group Key: user_id)
  ->  Seq Scan on orders
```

**GroupAggregate**: 인풋이 그룹키로 **이미 정렬**되어 있을 때 사용. 연속된 같은 키를 순서대로 집계 → 메모리를 거의 안 씀.

```
GroupAggregate  (Group Key: user_id)
  ->  Index Scan on orders using idx_user_id  ← 이미 user_id 순 정렬
```

`work_mem`이 작거나 그룹 수가 매우 많으면 HashAggregate가 디스크를 쓰는 반면, GroupAggregate는 스트리밍 처리가 가능하다. 인덱스가 집계 성능에도 영향을 미치는 이유가 여기 있다.

## 6.7 EXPLAIN 트리 심화 읽기

5장에서 기본 항목을 배웠다. 이 절에서는 실제 EXPLAIN 출력 하나를 **처음부터 끝까지 완전 해부**한다.

### 예제 쿼리와 EXPLAIN 출력

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.country = 'KR'
GROUP BY u.name
ORDER BY order_count DESC
LIMIT 10;
```

```

        (actual time=523.121..523.124 rows=10 loops=1)

  ->  Sort  (cost=18240.30..18490.30 rows=100000 width=40)
              (actual time=523.118..523.120 rows=10 loops=1)
        Sort Key: (count(o.id)) DESC
        Sort Method: top-N heapsort  Memory: 25kB
        ->  HashAggregate  (cost=14815.00..15815.00 rows=100000 width=40)
                            (actual time=510.234..519.876 rows=100000 loops=1)
              Group Key: u.name
              Batches: 1  Memory Usage: 12305kB
              ->  Hash Join  (cost=3530.00..12315.00 rows=500000 width=16)
                              (actual time=31.451..390.321 rows=500000 loops=1)
                    Hash Cond: (o.user_id = u.id)
                    Buffers: shared hit=4498 read=1204
                    ->  Seq Scan on orders o  (cost=0.00..7735.00 rows=500000 width=16)
                                               (actual time=0.008..85.431 rows=500000 loops=1)
                          Buffers: shared hit=3235 read=1204
                    ->  Hash  (cost=2905.00..2905.00 rows=50000 width=12)

                          ->  Index Scan using idx_users_country on users u
                               (cost=0.42..2905.00 rows=50000 width=12)

Execution Time: 523.891 ms
```

### 단계별 해부

**실행 순서 (안쪽 → 바깥쪽):**

```
1. Index Scan on users (country='KR')  → 50,000행
2. Hash (해시 테이블 빌드, 메모리 2.5MB)
3. Seq Scan on orders              → 500,000행
4. Hash Join                       → 500,000행 (o.user_id = u.id 매칭)
5. HashAggregate (GROUP BY u.name) → 100,000행 (그룹별 집계, 12MB)
6. Sort (order_count DESC)         → top-N heapsort, 10개만
7. Limit                           → 10행 반환
```

**각 노드 해석:**

| 노드 | 핵심 관찰 |
|------|-----------|
| `Index Scan on users` | country='KR' → 인덱스로 50,000행 필터링. `Buffers: shared hit=1263` → 1,263 블록 읽음 |
| `Hash (Build)` | 50,000행으로 해시 테이블 빌드. `Batches: 1` → 메모리에 전부 들어감. `Memory: 2521kB` |
| `Seq Scan on orders` | 500,000행 전체 스캔. `read=1204` → 디스크에서 1,204 블록 읽음 (cold) |
| `Hash Join` | `Hash Cond: (o.user_id = u.id)`. 500,000 × O(1) probe → 500,000 매칭 행 |
| `HashAggregate` | `Batches: 1, Memory: 12305kB` → 12MB. 이 값이 `work_mem` 초과 시 디스크 |
| `Sort` | `top-N heapsort Memory: 25kB` → 전체 정렬 대신 상위 10개만 유지하는 힙 정렬. 효율적 |
| `Limit` | `rows=10 loops=1` → 실제 10행 반환 확인 |

**통계 진단:**
- `rows=50000` (추정) vs `actual rows=50000` → 추정이 정확. 통계 정상.
- `rows=500000` (orders 추정) vs `actual rows=500000` → 정확.
- 만약 추정과 실제가 10배 이상 다르면 `ANALYZE` 필요 신호.

**성능 관찰:**
- `Buffers: shared read=1204` → 디스크에서 읽은 블록. 다음 실행에서는 캐시 hit 증가.
- `HashAggregate Memory: 12305kB` → 12MB 사용. `work_mem` 기본값 4MB라면 디스크 배치(Batches > 1) 발생. 이 예에서는 `Batches: 1`이므로 세션에서 `work_mem`이 이미 충분히 설정됨.
- 총 523ms. 주요 병목은 Seq Scan(85ms)과 HashAggregate(~130ms).

**개선 방향:**
1. orders에 `CREATE INDEX idx_orders_user_id ON orders(user_id)`를 만들면 Hash Join 대신 Nested Loop가 선택될 수 있고, orders 전체 Seq Scan을 피할 수 있다.
2. 또는 users(country) 인덱스 이미 있으니, users 행이 50,000개 (5%)로 적다 — 현 Hash Join이 이미 합리적.

## ⚠️ 흔한 오해와 함정

**오해 1: "SQL 작성 순서가 실행 순서다"**

`FROM users JOIN orders ... WHERE ... GROUP BY ... ORDER BY ...` — 이 SQL 키워드 순서는 실행 순서가 아니다. 옵티마이저는 순서를 자유롭게 재배열한다. WHERE 조건을 조인 전에 먼저 적용하거나, 작은 테이블을 해시 테이블로 먼저 빌드하거나. **실행 순서를 알고 싶으면 EXPLAIN을 보라.**

**오해 2: "서브쿼리보다 조인이 항상 빠르다"**

"서브쿼리는 느리다"는 말이 있지만, 현대 옵티마이저는 많은 서브쿼리를 자동으로 조인으로 변환(비상관화)한다. 실제로 느린지는 EXPLAIN으로 확인해야 한다. 단, **상관 서브쿼리**는 비상관화가 안 되는 경우도 있어 주의가 필요하다.

**오해 3: "Hash Join이 항상 제일 빠르다"**

Hash Join은 등가 조인에서 O(|A|+|B|)로 효율적이지만, 해시 테이블이 메모리를 초과하면 Grace Hash Join으로 전환되어 비용이 3배 이상 증가한다. 외부 테이블이 작고 내부 테이블에 인덱스가 있다면 Nested Loop가 더 빠를 수 있다. 상황에 따라 다르므로 EXPLAIN으로 확인.

**오해 4: "통계는 자동이니 신경 안 써도 된다"**

autovacuum이 통계를 자동 갱신하지만 딜레이가 있다. 대량 INSERT/DELETE/UPDATE 직후에는 통계가 낡아 옵티마이저가 잘못된 계획을 세울 수 있다. **대량 데이터 변경 후에는 `ANALYZE`를 명시적으로 실행하라.** EXPLAIN의 추정 rows와 actual rows 차이가 10배 이상이면 통계 갱신을 의심하라.

**오해 5: "EXPLAIN은 실제 실행이 아니니 믿을 수 없다"**

`EXPLAIN` (ANALYZE 없이)은 추정치만 보여주는 것이 맞다. 그러나 `EXPLAIN (ANALYZE)`는 **실제 실행**하고 측정값을 보여준다. 다만 실제로 데이터를 변경하는 DML(INSERT/UPDATE/DELETE)에 `EXPLAIN ANALYZE`를 쓰면 실제로 변경이 일어나므로 **트랜잭션으로 감싸야** 한다:

```sql
BEGIN;
EXPLAIN (ANALYZE) UPDATE orders SET status = 'PROCESSED' WHERE status = 'PENDING';
ROLLBACK;   -- 실제 변경은 롤백
```

**오해 6: "work_mem을 크게 늘리면 무조건 좋다"**

`work_mem`은 **쿼리당**, **노드당** 사용되는 메모리다. 동시 쿼리가 100개이고 쿼리마다 노드가 5개라면 최악의 경우 `100 × 5 × work_mem`이 실제 메모리 사용량이 될 수 있다. 서버 전체 메모리를 초과하면 OOM 위험. 전역 설정보다 특정 쿼리에만 세션 단위로 늘리는 게 안전하다.

**함정: 조인 순서와 드라이빙 테이블**

Nested Loop에서 **외부 테이블(드라이빙 테이블)**이 크고 내부 테이블에 인덱스가 없으면 O(|A|×|B|)로 매우 느릴 수 있다. 옵티마이저가 잘못된 드라이빙 테이블을 선택했다면 `join_collapse_limit = 1`을 설정하고 쿼리에서 원하는 순서를 직접 지정하거나, 조인 힌트 방식(pg_hint_plan 확장)을 검토한다.

### 문제 1: 쿼리 처리 파이프라인

다음 SQL이 PostgreSQL에서 처리되는 5단계(파서, 분석기, 논리계획, 옵티마이저, 실행기)를 간략히 설명하라. 각 단계에서 무슨 일이 일어나는지, 그리고 EXPLAIN이 어느 단계의 출력을 보여주는지 명시하라.

```sql
SELECT u.name, SUM(o.amount)
FROM users u JOIN orders o ON o.user_id = u.id
WHERE u.active = true
GROUP BY u.name
HAVING SUM(o.amount) > 100000;
```

<details>
<summary>해설</summary>

1. **파서**: SQL 텍스트의 문법을 검사하고 파스 트리를 생성. 오타나 문법 오류를 여기서 잡는다.

2. **분석기/재작성기**: `users`, `orders` 테이블이 실제로 존재하는지, `u.name`, `o.amount` 같은 컬럼이 있는지 확인. 타입 호환성 검사. 뷰라면 원본 쿼리로 펼침.

3. **논리적 계획**: 관계 대수로 표현:
   - σ(active=true)(users) — 필터
   - ⋈(users.id=orders.user_id) — 조인
   - γ(name, SUM(amount)) — 집계
   - σ(SUM > 100000) — HAVING 필터
   순서와 알고리즘은 미정.

4. **옵티마이저**: 통계 기반으로 비용 최솟값 계획 탐색:
   - `active=true`로 필터 후 조인? 아니면 조인 후 필터?
   - Hash Join? Nested Loop?
   - HashAggregate? GroupAggregate?
   → **EXPLAIN이 보여주는 것이 바로 옵티마이저의 출력(물리 계획)**

5. **실행기**: 물리 계획대로 실제 데이터 읽고 처리. 결과 행을 JDBC/psql에 스트리밍.

</details>

### 문제 2: 조인 알고리즘 선택

다음 상황에서 각각 어떤 조인 알고리즘이 가장 적합한지 고르고 이유를 설명하라.

**상황 A**: `employees`(100만 행)와 `departments`(50개 행)를 `dept_id`로 등가 조인. `departments.id`에 PK 인덱스 있음.

**상황 B**: `orders`(500만 행)와 `products`(100만 행)를 `product_id`로 등가 조인. 두 테이블 모두 인덱스 없음.

**상황 C**: `event_log`(1,000만 행, `event_time` 기준 정렬)와 `session_log`(500만 행, `start_time` 기준 정렬)를 `event_time BETWEEN start_time AND end_time`으로 범위 조인.

<details>
<summary>해설</summary>

**상황 A: Nested Loop Join**

`departments`가 50행밖에 안 되는 소규모 테이블이고, `departments.id`에 인덱스가 있다. 외부 테이블을 `employees`로 잡고, 각 employees 행마다 `departments.id` 인덱스로 O(log 50) 조회. 총 비용: O(|employees| × log|departments|) = O(100만 × 6) = 600만 → 효율적. Hash Join보다 메모리 적게 쓰고 간단.

**상황 B: Hash Join**

양쪽 모두 크고 인덱스가 없다. Nested Loop는 O(500만 × 100만) = O(5조) → 불가능. Hash Join: 작은 쪽(`products` 100만 행)으로 해시 테이블 빌드 후 `orders` 500만 행 probe. 비용 O(500만 + 100만) = O(600만) → 합리적. 단, products 해시 테이블이 `work_mem`을 초과하면 Grace Hash Join으로 전환되어 비용 증가.

**상황 C: Merge Join**

범위 조인(`BETWEEN`)은 Hash Join으로 처리 불가(Hash는 등가만). Merge Join은 범위 조인도 처리 가능하며, 두 테이블이 이미 시간 컬럼으로 정렬되어 있으므로 정렬 비용 없음 → O(|event_log| + |session_log|) = O(1,500만) 선형 비용. Nested Loop는 O(10억 × 500만) 불가.

</details>

### 문제 3: EXPLAIN 출력 해석

다음 EXPLAIN 출력을 보고 물음에 답하라.

```
Hash Join  (cost=15000.00..45000.00 rows=500000 width=24)
           (actual time=250.11..1820.33 rows=480000 loops=1)
  Hash Cond: (o.user_id = u.id)
  ->  Seq Scan on orders o  (cost=0.00..7500.00 rows=500000 width=16)
                             (actual time=0.01..90.42 rows=500000 loops=1)
  ->  Hash  (cost=10000.00..10000.00 rows=400000 width=12)
             (actual time=240.00..240.01 rows=400000 loops=1)
        Buckets: 524288  Batches: 4  Memory Usage: 4096kB
        ->  Seq Scan on users u  (cost=0.00..10000.00 rows=400000 width=12)
                                  (actual time=0.02..150.33 rows=400000 loops=1)
Planning Time: 3.421 ms
Execution Time: 2045.118 ms
```

1. 어떤 조인 알고리즘이 선택됐는가?
2. 실행 순서를 올바르게 나열하라.
3. `Batches: 4`는 무엇을 의미하는가? 어떻게 개선할 수 있는가?
4. `rows=500000` (Hash Join 추정) vs `actual rows=480000`의 차이가 크게 느껴지는가? 통계 관점에서 평가하라.
5. 이 쿼리의 성능을 개선하기 위해 어떤 인덱스를 제안하겠는가?

<details>
<summary>해설</summary>

1. **Hash Join**: 출력의 첫 줄이 `Hash Join`임. `Hash Cond: (o.user_id = u.id)` — 등가 조인이다.

2. **실행 순서** (안쪽 → 바깥쪽):
   1. `Seq Scan on users u` → 400,000행
   2. `Hash (Build)` → 400,000행으로 해시 테이블 구축
   3. `Seq Scan on orders o` → 500,000행
   4. `Hash Join (Probe)` → 480,000행 반환

3. **`Batches: 4`는 해시 테이블이 메모리를 초과해 4개 파티션으로 나눠 디스크를 사용했다는 의미**. `Memory Usage: 4096kB` = 4MB가 한 배치의 메모리 한계였다. 개선 방법:
   ```sql
   SET work_mem = '64MB';  -- 해시 테이블이 메모리에 들어오도록 늘림
   ```
   `Batches: 1`이 되면 디스크 없이 메모리만으로 처리 → 성능 향상.

4. **추정 500,000 vs 실제 480,000 → 약 4% 오차.** 이 정도는 정상 범위. 통계가 충분히 정확하다. 10배 이상 차이가 날 때 통계 문제를 의심한다.

5. **인덱스 제안:**
   - `users` 테이블에 WHERE 조건이 있다면 해당 컬럼에 인덱스
   - 현재 `Seq Scan on users`와 `Seq Scan on orders`이므로:
     ```sql
     -- users가 필터 조건이 있는 경우
     CREATE INDEX idx_users_country ON users(country);
     
     -- orders의 user_id에 인덱스 → Nested Loop 가능성 검토
     CREATE INDEX idx_orders_user_id ON orders(user_id);
     ```
   - users가 필터 후 매우 작다면 (예: country='KR' → 5만 행) Nested Loop로 전환:
     `Index Scan on users` (5만 행) → Nested Loop → `idx_orders_user_id`로 orders 조회
   - 양쪽 모두 크다면 Hash Join 유지하되 `work_mem` 늘려 `Batches: 1`로 만드는 것이 우선.

</details>

### 문제 4: JPA N+1과 DB 조인

아래 JPA 코드가 실행할 때 DB에서 어떤 일이 일어나는지 설명하고, 이것이 이 장에서 배운 어떤 조인 알고리즘과 가장 유사한지 말하라. 또한 어떻게 수정해야 하는지 JPA 코드와 함께 제시하라. Spring 개념서 §4.4도 참조하라.

```java
// Post 목록을 가져온 후 각 Post의 Author를 조회
List<Post> posts = postRepository.findAll();  // 게시물 N개 조회 쿼리 1번
for (Post post : posts) {
    // post.getAuthor()가 LAZY 로딩이면 여기서 쿼리 발생
    System.out.println(post.getAuthor().getName());
}
```

<details>
<summary>해설</summary>

**DB에서 일어나는 일:**

```sql
-- 1번 (전체 Post 조회)
SELECT * FROM post;  -- N개 결과

-- N번 (각 Post마다 Author 개별 조회)
SELECT * FROM users WHERE id = 1;
SELECT * FROM users WHERE id = 2;
...
SELECT * FROM users WHERE id = N;
```

총 N+1번의 쿼리가 애플리케이션-DB 사이를 왕복.

**유사한 조인 알고리즘:** 외부 테이블(`post`)의 각 행마다 내부 테이블(`users`)을 개별 조회하는 구조는 **Nested Loop Join**과 유사하다. 그러나 DB 내부의 Nested Loop는 최적화된 실행 엔진 안에서 돌고 인덱스도 활용한다. JPA N+1은 **네트워크 왕복 비용이 N번** 추가되는 최악의 형태다. DB 내부에서 50만 행을 처리하는 것보다, 애플리케이션과 DB 사이를 N번 오가는 것이 훨씬 느린 이유가 여기 있다.

**JPA 해결 방법 — FETCH JOIN:**

```java
// Repository에 JPQL fetch join 추가
@Repository
public interface PostRepository extends JpaRepository<Post, Long> {
    @Query("SELECT p FROM Post p JOIN FETCH p.author")
    List<Post> findAllWithAuthor();
}

// 사용
List<Post> posts = postRepository.findAllWithAuthor();
// 단 1번의 JOIN 쿼리:
// SELECT p.*, u.* FROM post p JOIN users u ON p.author_id = u.id
```

이것이 DB에서 Hash Join 또는 Nested Loop Join 하나로 처리되어 1번의 쿼리로 완료된다.

또는 `@EntityGraph`:
```java
@EntityGraph(attributePaths = {"author"})
List<Post> findAll();
```

Spring 개념서 §4.4([N+1 문제](../../../spring/docs/SPRING_개념서/README.md))도 함께 읽으면 JPA 관점의 해결 방법을 더 자세히 볼 수 있다.

</details>

### 문제 5: 옵티마이저 힌트와 통계

다음 쿼리의 EXPLAIN 출력에서 `rows` 추정치가 실제보다 100배 작게 나왔다. 이것이 성능에 어떤 영향을 미치는지 설명하고, 어떻게 진단·수정하는지 단계를 제시하라.

```
Hash Join  (cost=100.00..1200.00 rows=10 width=24)    ← 추정: 10행
           (actual time=30.11..9820.33 rows=1000 loops=1)  ← 실제: 1000행
```

<details>
<summary>해설</summary>

**성능 영향:**

옵티마이저는 Hash Join이 10행만 반환할 것으로 예상했다. 그래서 이 이후 단계(예: Sort, Limit, 상위 조인)의 비용도 10행 기준으로 계획했다. 실제로는 1,000행이 반환되어 다음 단계에서도 100배 더 많은 작업이 발생한다. 예를 들어:

- 이후 Sort 노드가 10행 예상으로 `quicksort`를 선택했는데 1,000행이 오면 더 많은 메모리가 필요
- 이후 Nested Loop의 외부가 1,000행이면 내부를 1,000번 반복 → 10번 예상보다 100배 비쌈

**진단 단계:**

```sql
-- 1. 해당 테이블 통계 확인
SELECT tablename, attname, n_distinct, most_common_vals, most_common_freqs
FROM pg_stats
WHERE tablename = '해당테이블' AND attname = '해당컬럼';

-- 2. 실제 카디널리티와 비교
SELECT COUNT(DISTINCT 해당컬럼) FROM 해당테이블;

-- 3. 통계 갱신
ANALYZE 해당테이블;

-- 4. 통계 샘플 수 늘리기 (값 분포가 불균일할 때)
ALTER TABLE 해당테이블 ALTER COLUMN 해당컬럼 SET STATISTICS 500;
ANALYZE 해당테이블;

-- 5. EXPLAIN 다시 실행해 추정치 개선 확인
EXPLAIN (ANALYZE) ...;
```

**수정 후 기대:**
- `rows=10` → `rows=900~1100` 수준으로 추정치가 개선
- 옵티마이저가 올바른 알고리즘(예: HashAggregate 대신 Sort+GroupAggregate)을 선택

통계 갱신으로도 해결이 안 되면, 조건에 코릴레이션(correlation)이 있는 경우다 — 예를 들어 `country='KR' AND city='Seoul'` 조합의 빈도를 통계가 독립적으로 추정하는 문제. 이때는 `CREATE STATISTICS`로 다중 컬럼 통계를 추가한다:

```sql
CREATE STATISTICS stat_country_city ON country, city FROM users;
ANALYZE users;
```

</details>

| 옵티마이저 | 비용 기반(CBO). 통계(pg_stats)로 선택도 추정. 통계 낡으면 → 나쁜 계획 |
| 조인 순서 탐색 | N! 조합. DP(≤8 테이블) or GEQO(≥12 테이블) |
| Nested Loop | O(|A|×|B|). B에 인덱스 있으면 O(|A|×log|B|). A 작고 B 인덱스 있을 때 최적 |
| Hash Join | O(|A|+|B|). Build(B로 해시 테이블) + Probe(A 순회). 등가 조인, 둘 다 클 때 최적 |
| Merge Join | O(sort + A+B). 이미 정렬 시 O(A+B). 범위 조인 가능. 양쪽 정렬 있을 때 최적 |
| work_mem | Hash Build, Sort, HashAggregate가 이 메모리를 씀. 초과 시 디스크(Batches>1, external merge) |
| Volcano 모델 | 모든 노드가 open/next/close 인터페이스. 파이프라이닝 vs 재료화 |
| EXPLAIN 읽기 | 들여쓰기=자식, 안→밖 순서. loops 곱. 추정 vs actual 비교. Batches, Sort Method 주목 |
| JPA 연결 | N+1 = 앱 레벨 나쁜 Nested Loop. FETCH JOIN = DB 내 최적 조인 1번 |

**Spring 개념서 상호참조:**
- 4.4절 N+1 문제 → 이 장의 6.3.1 (Nested Loop와 N+1 비교)
- 4.7절 인덱스와 B-tree → 5장(인덱싱)과 이 장의 비용 모델 연결
- 4.3절 JpaRepository → FETCH JOIN으로 생성된 SQL이 Hash Join 또는 Nested Loop로 처리
