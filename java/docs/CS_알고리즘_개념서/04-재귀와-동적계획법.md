## 1. 도입

알고리즘을 배우다 보면 "이 문제, 어떻게 분해할 수 있을까?"라는 질문을 자주 만나게 된다. **재귀(Recursion)** 는 문제를 동일한 구조의 더 작은 문제로 반복 분해하는 사고 방식이며, **동적 계획법(Dynamic Programming, DP)** 은 그 과정에서 발생하는 중복 계산을 제거함으로써 지수 시간을 다항 시간으로 끌어내리는 강력한 최적화 기법이다.

이 두 개념은 따로 떼어 볼 수 없다. 재귀를 이해하지 못한 채 DP를 공부하면 점화식이 왜 성립하는지 알 수 없고, DP 없이 재귀만 쓰면 지수 폭발이라는 함정에 빠진다. 이 장에서는 두 개념을 유기적으로 연결하여, 팩토리얼부터 0/1 배낭 문제까지 단계적으로 체화한다.

학습 목표:
- 재귀의 두 필수 요소(기저 조건, 재귀 단계)를 정확히 설명할 수 있다.
- 콜 스택 다이어그램을 그리고 StackOverflowError 발생 조건을 예측할 수 있다.
- 겹치는 부분 문제(Overlapping Subproblems)와 최적 부분 구조(Optimal Substructure)를 공식적으로 정의할 수 있다.
- 메모이제이션(top-down)과 타뷸레이션(bottom-up)의 차이와 트레이드오프를 설명할 수 있다.
- 0/1 배낭 문제를 2차원 DP와 1차원 공간 최적화 DP로 각각 구현할 수 있다.
- 그리디와 DP 적용 조건을 구분하고 그리디 실패 사례를 제시할 수 있다.

## 2. 재귀(Recursion)의 원리

### 2.1 재귀란 무엇인가

**재귀(Recursion)** 란 함수가 자기 자신을 호출하여 문제를 해결하는 기법이다. 수학적으로는 귀납적 정의(Inductive Definition)에 대응한다. 예를 들어 팩토리얼은 다음과 같이 귀납적으로 정의된다.

```
0! = 1                    (기저 조건, Base Case)
n! = n × (n-1)!   (n > 0)  (재귀 단계, Recursive Step)
```

이 두 줄이 재귀 알고리즘의 **두 필수 요소** 다.

| 요소 | 역할 | 없으면? |
|------|------|---------|
| 기저 조건 (Base Case) | 재귀를 멈추는 최소 단위 | 무한 재귀 → StackOverflowError |
| 재귀 단계 (Recursive Step) | 더 작은 입력으로 자신을 호출 | 문제가 전혀 분해되지 않음 |

### 2.2 팩토리얼 Java 구현

```java
public class Factorial {

    /**
     * n! 을 재귀로 계산한다.
     * 기저 조건: n == 0 → 1
     * 재귀 단계: n * factorial(n - 1)
     *
     * 시간 복잡도: O(n)
     * 공간 복잡도: O(n) — 콜 스택 깊이
     */
    public static long factorial(int n) {
        if (n < 0) throw new IllegalArgumentException("n은 음수일 수 없습니다.");
        if (n == 0) return 1;               // 기저 조건
        return n * factorial(n - 1);        // 재귀 단계
    }

    public static void main(String[] args) {
        for (int i = 0; i <= 10; i++) {
            System.out.printf("%2d! = %d%n", i, factorial(i));
        }
    }
}
```

### 2.3 재귀 호출의 확장 및 수축

재귀 함수는 두 단계로 동작한다. **확장(Winding)** 단계에서는 기저 조건에 도달할 때까지 자기 자신을 호출하며 스택 프레임을 쌓는다. **수축(Unwinding)** 단계에서는 기저 조건에서 반환값을 갖고 역방향으로 스택 프레임을 해제하며 결과를 합산한다.

```
factorial(4) 호출 흐름 (확장 → 수축)

  확장 단계 (Call)           수축 단계 (Return)
  ──────────────────         ──────────────────
  factorial(4)               factorial(4) = 4 × 6 = 24
    └─ factorial(3)            └─ factorial(3) = 3 × 2 = 6
         └─ factorial(2)            └─ factorial(2) = 2 × 1 = 2
              └─ factorial(1)            └─ factorial(1) = 1 × 1 = 1
                   └─ factorial(0)            └─ factorial(0) = 1  ← 기저 조건
```

### 2.4 꼬리 재귀(Tail Recursion)와 반복문 변환

일부 언어(Scala, Haskell 등)는 **꼬리 재귀 최적화(TCO, Tail Call Optimization)** 를 지원하여 마지막 호출이 재귀인 경우 스택 프레임을 재사용한다. 그러나 **JVM(Java)은 표준적으로 TCO를 지원하지 않는다.** (자세한 함정은 [11절](#11-️-흔한-오해와-함정) 참고)

꼬리 재귀 형태로 변환하면 반복문으로의 전환이 자명해진다는 교육적 가치는 있다.

```java
// 꼬리 재귀 스타일 (JVM에서 최적화 불가)
public static long factorialTail(int n, long acc) {
    if (n == 0) return acc;
    return factorialTail(n - 1, n * acc);  // 반환값 = 재귀 호출 자체
}

// 반복문으로 변환 (권장)
public static long factorialIter(int n) {
    long acc = 1;
    for (int i = 1; i <= n; i++) acc *= i;
    return acc;
}
```

## 3. 콜 스택과 StackOverflowError

### 3.1 콜 스택의 구조

JVM은 각 메서드 호출마다 **스택 프레임(Stack Frame)** 을 생성하여 스레드의 콜 스택에 쌓는다. 스택 프레임에는 지역 변수, 매개변수, 반환 주소, 이전 프레임 포인터 등이 저장된다.

```
factorial(3) 콜 스택 다이어그램
(스택은 아래에서 위로 성장)

┌─────────────────────────────┐
│  factorial(0)               │  ← 가장 나중에 쌓임 (Top)
│    n=0, 반환값=1            │
├─────────────────────────────┤
│  factorial(1)               │
│    n=1, 대기: 1 × ?         │
├─────────────────────────────┤
│  factorial(2)               │
│    n=2, 대기: 2 × ?         │
├─────────────────────────────┤
│  factorial(3)               │  ← 가장 먼저 쌓임 (Bottom)
│    n=3, 대기: 3 × ?         │
├─────────────────────────────┤
│  main()                     │
└─────────────────────────────┘
      JVM Thread Stack
```

### 3.2 StackOverflowError 발생 조건

JVM의 기본 스택 크기는 플랫폼마다 다르지만, 일반적으로 **약 45,000 수준의 재귀 깊이** 에서 `StackOverflowError`가 발생한다(실측값이며 JVM 옵션 `-Xss`로 조정 가능). 이는 두 가지 상황에서 주로 발생한다.

1. **기저 조건이 없거나 잘못 작성된 경우** — 재귀가 영원히 종료되지 않는다.
2. **입력 크기가 스택 한계를 초과하는 경우** — 기저 조건이 있어도 n이 매우 클 때.

```java
// 실험: StackOverflowError 발생 깊이 측정
public class StackDepthTest {
    static int depth = 0;

    public static void recurse() {
        depth++;
        recurse();  // 기저 조건 없음 → 무한 재귀
    }

    public static void main(String[] args) {
        try {
            recurse();
        } catch (StackOverflowError e) {
            System.out.println("StackOverflowError 발생 깊이: " + depth);
            // 실측: 약 45,000 (JVM 기본 스택 크기, 환경에 따라 다름)
        }
    }
}
```

```
실측 결과 (OpenJDK 17, 기본 Xss):
StackOverflowError 발생 깊이: 약 45,000
```

### 3.3 대응 전략

| 상황 | 대응 방법 |
|------|-----------|
| 기저 조건 누락 | 반드시 기저 조건 작성, 단위 테스트로 검증 |
| 입력 크기 과대 | 반복문(iteration)으로 변환 |
| 재귀 깊이 조정 필요 | JVM 옵션 `-Xss` 증가 (임시방편) |
| 메모이제이션 필요 | 반복적 타뷸레이션(bottom-up) 사용 |

## 4. 분할 정복(Divide and Conquer)

### 4.1 분할 정복의 세 단계

재귀의 가장 고전적인 응용이 **분할 정복** 이다. 다음 세 단계로 구성된다.

```
① 분할 (Divide)   : 문제를 더 작은 부분 문제로 나눈다.
② 정복 (Conquer)  : 부분 문제를 재귀적으로 해결한다.
③ 결합 (Combine)  : 부분 문제의 해를 합쳐 원래 문제의 해를 구한다.
```

### 4.2 대표 알고리즘

| 알고리즘 | 분할 방식 | 결합 비용 | 시간 복잡도 |
|----------|-----------|-----------|-------------|
| 병합 정렬(Merge Sort) | 반으로 분할 | O(n) — 병합 | O(n log n) |
| 퀵 정렬(Quick Sort) | 피벗 기준 분할 | O(1) — 제자리 | 평균 O(n log n) |
| 이진 탐색(Binary Search) | 반으로 분할 | O(1) — 비교 | O(log n) |

```
병합 정렬 분할 트리 예시: [38, 27, 43, 3]

                [38, 27, 43, 3]
               /               \
        [38, 27]             [43, 3]
        /      \             /     \
      [38]    [27]         [43]    [3]
        \      /             \     /
        [27, 38]             [3, 43]
               \               /
                [3, 27, 38, 43]
```

분할 정복과 DP의 차이: 분할 정복은 부분 문제가 서로 **겹치지 않는다**. 병합 정렬의 왼쪽 절반과 오른쪽 절반은 독립적이다. 반면 피보나치 수열처럼 부분 문제가 **겹치면** DP가 필요하다.

## 5. 재귀의 함정: 순진한 피보나치

### 5.1 피보나치 수열의 정의

```
F(0) = 0
F(1) = 1
F(n) = F(n-1) + F(n-2)   (n ≥ 2)
```

### 5.2 순진한 재귀 구현

```java
public static long fibNaive(int n) {
    if (n <= 1) return n;                   // 기저 조건
    return fibNaive(n - 1) + fibNaive(n - 2); // 재귀 단계
}
```

### 5.3 겹치는 부분 문제(Overlapping Subproblems) 시각화

```
fib(5) 재귀 호출 트리

                    fib(5)
                  /        \
             fib(4)        fib(3)
            /      \      /      \
         fib(3)  fib(2) fib(2) fib(1)
         /   \   /   \  /   \
      fib(2) fib(1) fib(1) fib(0) fib(1) fib(0)
      /   \
   fib(1) fib(0)

중복 호출:
  fib(3) : 2회
  fib(2) : 3회
  fib(1) : 5회
  fib(0) : 3회
```

`fib(3)`가 2번, `fib(2)`가 3번 등 동일한 계산이 반복된다. 이것이 **겹치는 부분 문제(Overlapping Subproblems)** 이며, 지수 폭발의 원인이다.

### 5.4 순진 재귀의 호출 횟수 측정

```
n     순진 재귀 호출 횟수    메모이제이션 호출 횟수
────  ─────────────────────  ──────────────────────
 10             177                    19
 20          21,891                    39
 30       2,692,537                    59
 40     331,160,281                    79
```

`n`이 10 증가할 때 순진 재귀는 약 **150배** 증가하는 반면, 메모이제이션은 **20회** 증가할 뿐이다. 순진 재귀의 시간 복잡도는 O(2ⁿ)이다.

**왜 O(2ⁿ)인가?** 각 노드에서 2개의 자식 노드를 생성하고 트리의 높이가 n이므로, 최대 2ⁿ개의 노드가 존재한다. 실제로는 중복 제거로 약 `2^(n/2)` 수준이지만 여전히 지수 복잡도다.

## 6. 동적 계획법(Dynamic Programming)

### 6.1 DP의 두 가지 적용 조건

DP가 적용되려면 문제가 다음 두 조건을 만족해야 한다.

**① 겹치는 부분 문제 (Overlapping Subproblems)**

동일한 부분 문제가 재귀 트리에서 두 번 이상 등장한다. 메모이제이션이나 타뷸레이션으로 중복 계산을 제거할 수 있다.

**② 최적 부분 구조 (Optimal Substructure)**

전체 문제의 최적해가 부분 문제의 최적해로 구성된다. 즉, 부분 문제를 최적으로 풀면 전체 문제도 최적으로 풀린다.

```
[공식 정의]
최적 부분 구조: 문제 P의 최적해 OPT(P)가 존재할 때,
P의 부분 문제 P₁, P₂, ..., Pₖ에 대해
OPT(P) = f(OPT(P₁), OPT(P₂), ..., OPT(Pₖ))
를 만족하는 함수 f가 존재한다.
```

### 6.2 DP 설계 4단계

DP 문제를 풀 때는 다음 4단계를 순서대로 따르는 것이 좋다.

```
① 상태 정의 (State Definition)
   dp[i]가 무엇을 의미하는지 명확히 한다.
   예: dp[i] = i번째 피보나치 수

② 점화식 (Recurrence Relation)
   부분 문제 간의 관계를 수식으로 표현한다.
   예: dp[i] = dp[i-1] + dp[i-2]

③ 기저 조건 (Base Case)
   가장 작은 부분 문제의 해를 직접 정의한다.
   예: dp[0] = 0, dp[1] = 1

④ 계산 순서 (Computation Order)
   부분 문제를 의존 관계에 맞는 순서로 계산한다.
   top-down: 재귀 + 메모이제이션 (자동으로 필요한 것만 계산)
   bottom-up: 반복문으로 작은 것부터 채움
```

### 6.3 방법 1: 메모이제이션(Memoization, Top-Down)

**메모이제이션** 은 재귀 구조를 유지하면서 이미 계산한 결과를 캐시(배열 또는 해시맵)에 저장하는 방식이다. 필요한 부분 문제만 계산하므로 레이지(lazy) 방식이라고도 한다.

```java
public class FibMemo {
    private static long[] cache;

    public static long fibMemo(int n) {
        cache = new long[n + 1];
        Arrays.fill(cache, -1);  // -1: 미계산 표시
        cache[0] = 0;
        cache[1] = 1;
        return solve(n);
    }

    private static long solve(int n) {
        if (cache[n] != -1) return cache[n];     // 캐시 적중(Cache Hit)
        cache[n] = solve(n - 1) + solve(n - 2);  // 계산 후 저장
        return cache[n];
    }
}
```

```
fibMemo(5) 실행 흐름 (캐시 히트 지점 표시)

solve(5)
  └─ solve(4)
       └─ solve(3)
            └─ solve(2)
                 └─ solve(1) = 1  [기저]
                 └─ solve(0) = 0  [기저]
                 cache[2] = 1
            └─ solve(1) = 1  ★캐시 히트
            cache[3] = 2
       └─ solve(2) = 1  ★캐시 히트
       cache[4] = 3
  └─ solve(3) = 2  ★캐시 히트
  cache[5] = 5

총 계산: 6회 (n=5 → 2n-1회)
```

**시간 복잡도:** O(n) — 각 부분 문제를 정확히 1번씩 계산
**공간 복잡도:** O(n) — 캐시 배열 + 콜 스택

### 6.4 방법 2: 타뷸레이션(Tabulation, Bottom-Up)

**타뷸레이션** 은 작은 부분 문제부터 순서대로 계산하여 테이블을 채워 나가는 방식이다. 재귀 호출이 없으므로 콜 스택을 전혀 사용하지 않아 **스택 안전(Stack-Safe)** 하다.

```java
public class FibTab {
    public static long fibTab(int n) {
        if (n <= 1) return n;

        long[] dp = new long[n + 1];
        dp[0] = 0;   // 기저 조건
        dp[1] = 1;   // 기저 조건

        for (int i = 2; i <= n; i++) {
            dp[i] = dp[i - 1] + dp[i - 2];  // 점화식
        }

        return dp[n];
    }
}
```

```
fibTab(6) 테이블 채우기 과정

인덱스: [0] [1] [2] [3] [4] [5] [6]
초기화: [ 0][ 1][ ?][ ?][ ?][ ?][ ?]
i=2:    [ 0][ 1][ 1][ ?][ ?][ ?][ ?]
i=3:    [ 0][ 1][ 1][ 2][ ?][ ?][ ?]
i=4:    [ 0][ 1][ 1][ 2][ 3][ ?][ ?]
i=5:    [ 0][ 1][ 1][ 2][ 3][ 5][ ?]
i=6:    [ 0][ 1][ 1][ 2][ 3][ 5][ 8]
                                   ↑
                              반환값: 8
```

공간을 더 절약하려면 변수 2개만으로도 구현할 수 있다.

```java
public static long fibOptimal(int n) {
    if (n <= 1) return n;
    long prev2 = 0, prev1 = 1;
    for (int i = 2; i <= n; i++) {
        long curr = prev1 + prev2;
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;  // O(1) 공간
}
```

### 6.5 Top-Down vs Bottom-Up 트레이드오프

| 비교 항목 | 메모이제이션 (Top-Down) | 타뷸레이션 (Bottom-Up) |
|-----------|-------------------------|------------------------|
| 구현 스타일 | 재귀 + 캐시 | 반복문 + 테이블 |
| 계산 범위 | 필요한 부분 문제만 계산 | 모든 부분 문제 계산 |
| 콜 스택 사용 | O(n) — 스택 오버플로 위험 | O(1) — 스택 안전 |
| 코드 직관성 | 점화식과 구조가 일치 | 순서를 수동으로 결정해야 |
| 디버깅 용이성 | 재귀 트레이스 필요 | 테이블 직접 확인 가능 |
| 캐시 미스 | 필요한 것만 접근 | 순차 접근으로 캐시 효율적 |
| 추천 상황 | 부분 문제 수가 적을 때 | 입력이 크고 안정성 중요 시 |

## 7. 0/1 배낭 문제(Knapsack Problem)

### 7.1 문제 정의

n개의 물건이 있으며, 각 물건 i는 무게 `weight[i]`와 가치 `value[i]`를 갖는다. 배낭의 최대 용량은 W다. 각 물건은 넣거나(1) 넣지 않거나(0) 둘 중 하나이며, 배낭 용량을 초과하지 않으면서 가치의 합을 최대화하라.

```
[입력 예시]
물건 수 n=4, 배낭 용량 W=7

물건  무게  가치
  1    1     1
  2    3     4
  3    4     5
  4    5     7
```

### 7.2 DP 설계 4단계 적용

**① 상태 정의**

```
dp[i][w] = 첫 i개의 물건만 고려하고 배낭 용량이 w일 때 달성 가능한 최대 가치
```

**② 점화식**

물건 i를 배낭에 **넣지 않는 경우**와 **넣는 경우** 중 더 큰 값을 선택한다.

```
        ┌ dp[i-1][w]                             (물건 i를 넣지 않음)
dp[i][w] = max ┤
        └ dp[i-1][w - weight[i]] + value[i]     (물건 i를 넣음, w ≥ weight[i])
```

**③ 기저 조건**

```
dp[0][w] = 0   (물건이 없으면 가치는 0)
dp[i][0] = 0   (용량이 0이면 가치는 0)
```

**④ 계산 순서**

i를 1부터 n까지, w를 0부터 W까지 순서대로 채운다.

### 7.3 Java 구현 (2차원 DP)

```java
public class Knapsack01 {

    /**
     * 0/1 배낭 문제 — 2차원 DP
     *
     * @param n       물건 수
     * @param W       배낭 최대 용량
     * @param weight  각 물건의 무게 (1-indexed)
     * @param value   각 물건의 가치 (1-indexed)
     * @return 최대 가치
     *
     * 시간 복잡도: O(n × W)
     * 공간 복잡도: O(n × W)
     */
    public static int knapsack(int n, int W, int[] weight, int[] value) {
        int[][] dp = new int[n + 1][W + 1];

        for (int i = 1; i <= n; i++) {
            for (int w = 0; w <= W; w++) {
                // 경우 1: 물건 i를 넣지 않음
                dp[i][w] = dp[i - 1][w];

                // 경우 2: 물건 i를 넣음 (용량이 충분할 때만)
                if (w >= weight[i]) {
                    int withItem = dp[i - 1][w - weight[i]] + value[i];
                    dp[i][w] = Math.max(dp[i][w], withItem);
                }
            }
        }

        return dp[n][W];
    }

    public static void main(String[] args) {
        int n = 4, W = 7;
        int[] weight = {0, 1, 3, 4, 5}; // 1-indexed
        int[] value  = {0, 1, 4, 5, 7}; // 1-indexed

        System.out.println("최대 가치: " + knapsack(n, W, weight, value)); // 9
    }
}
```

### 7.4 DP 테이블 시각화

```
0/1 배낭 dp 테이블 (n=4, W=7)
물건: 1(무게1,가치1), 2(무게3,가치4), 3(무게4,가치5), 4(무게5,가치7)

     w=0  w=1  w=2  w=3  w=4  w=5  w=6  w=7
i=0 [  0][  0][  0][  0][  0][  0][  0][  0]
i=1 [  0][  1][  1][  1][  1][  1][  1][  1]  ← 물건1(무게1,가치1)
i=2 [  0][  1][  1][  4][  5][  5][  5][  5]  ← 물건2(무게3,가치4)
i=3 [  0][  1][  1][  4][  5][  6][  6][  9]  ← 물건3(무게4,가치5)
i=4 [  0][  1][  1][  4][  5][  7][  8][  9]  ← 물건4(무게5,가치7)
                                          ↑
                                    정답: 9
```

### 7.5 공간 최적화: 2D → 1D (역순 갱신)

2차원 dp 테이블 대신 1차원 배열로 공간을 O(W)로 줄일 수 있다. 핵심은 **역순(w = W → weight[i])으로 갱신** 하는 것이다.

**왜 역순인가?** `dp[w]`를 갱신할 때 `dp[w - weight[i]]`는 이번 물건 i가 고려되기 **전** 의 값이어야 한다(즉, `dp[i-1][w-weight[i]]`). 역순으로 순회하면 작은 인덱스가 아직 갱신되지 않았으므로 자동으로 이전 행의 값을 참조하게 된다.

```java
public static int knapsack1D(int n, int W, int[] weight, int[] value) {
    int[] dp = new int[W + 1];  // O(W) 공간

    for (int i = 1; i <= n; i++) {
        // 역순으로 순회해야 같은 물건을 중복 사용하지 않음
        for (int w = W; w >= weight[i]; w--) {
            dp[w] = Math.max(dp[w], dp[w - weight[i]] + value[i]);
        }
    }

    return dp[W];
}
```

```
1D 배열 갱신 과정 (물건2: 무게3, 가치4 추가 시)

이전 상태: [0, 1, 1, 1, 1, 1, 1, 1]
역순 갱신 (w=7→3):
  w=7: max(dp[7], dp[7-3]+4) = max(1, 1+4) = 5
  w=6: max(dp[6], dp[6-3]+4) = max(1, 1+4) = 5
  w=5: max(dp[5], dp[5-3]+4) = max(1, 1+4) = 5
  w=4: max(dp[4], dp[4-3]+4) = max(1, 1+4) = 5
  w=3: max(dp[3], dp[3-3]+4) = max(1, 0+4) = 4

갱신 후:  [0, 1, 1, 4, 5, 5, 5, 5]
```

## 8. 그리디 vs DP: 언제 무엇을 쓰는가

### 8.1 그리디 알고리즘이란

**그리디(Greedy)** 알고리즘은 매 단계에서 현재 가장 좋아 보이는 선택을 하는 방식이다. 미래를 고려하지 않고 지금 당장 최선의 결정을 내린다.

### 8.2 그리디 실패 사례: 0/1 배낭

```
[예시] 배낭 용량 W=50
물건  무게  가치  가치/무게 비율
  A    10    60      6.0
  B    20   100      5.0
  C    30   120      4.0

그리디 전략: 가치/무게 비율이 높은 순서로 선택
  → A 선택 (비율 6.0, 총무게=10, 총가치=60)
  → B 선택 (비율 5.0, 총무게=30, 총가치=160)
  → C 선택 불가 (30 추가 시 총무게=60 > W=50)
  그리디 결과: A + B = 160

DP 최적해:
  B + C = 100 + 120 = 220 > 160

그리디가 최적해를 놓쳤다!
```

### 8.3 탐욕 선택 속성(Greedy Choice Property)

그리디가 항상 최적해를 보장하려면 **탐욕 선택 속성** 이 성립해야 한다. 즉, 전역 최적해가 각 단계에서의 국소 최적 선택으로 구성됨을 수학적으로 증명할 수 있어야 한다.

| 알고리즘 | 탐욕 선택 속성 | 근거 |
|----------|----------------|------|
| 동전 거스름돈 (표준 동전 체계) | 성립 | 큰 단위가 항상 작은 단위의 배수 |
| 분할 배낭 (Fractional Knapsack) | 성립 | 물건을 쪼갤 수 있으므로 비율 정렬 최적 |
| 프림/크루스칼 MST | 성립 | 컷(cut) 속성에 의해 증명 가능 |
| 0/1 배낭 | 불성립 | 반례 위에 제시 |
| 최단 경로 (음수 간선 없음) | 성립 (다익스트라) | 이완(relaxation) 속성 |

### 8.4 DP vs 그리디 선택 기준

```
문제 분석 흐름도

    최적 부분 구조 성립?
         /          \
       No           Yes
        ↓            ↓
    완전 탐색     겹치는 부분 문제?
    (백트래킹)    /              \
               Yes              No
                ↓                ↓
               DP          분할 정복 or
                           그리디 (탐욕 선택 속성 확인)
```

|----------|-----------|-----------|-----------|-------------|------|

| 피보나치 공간 최적 | O(n) | O(n) | O(n) | O(1) | 변수 2개 |

## 11. ⚠️ 흔한 오해와 함정

### 함정 1: 기저 조건 누락으로 인한 무한 재귀

가장 흔한 실수다. 기저 조건을 아예 빠뜨리거나 조건이 절대 만족되지 않는 경우가 있다.

```java
// 잘못된 예: n이 음수로 전달되면 기저조건(n==0)에 절대 도달하지 않음
public static long factorial(int n) {
    if (n == 0) return 1;         // n=-1이면 이 조건은 절대 true가 안 됨
    return n * factorial(n - 1);  // 음수에서 더 음수로 → 무한 재귀
}

// 올바른 예
public static long factorial(int n) {
    if (n < 0) throw new IllegalArgumentException("n >= 0 이어야 합니다.");
    if (n == 0) return 1;
    return n * factorial(n - 1);
}
```

**교훈:** 재귀 함수를 작성할 때 기저 조건을 먼저 작성하고, 입력 유효성 검사를 포함하라.

### 함정 2: JVM에서 꼬리 재귀 최적화(TCO) 기대

Java는 언어 명세상 꼬리 재귀 최적화를 **보장하지 않는다**. JIT 컴파일러가 부분적으로 최적화할 수 있지만 스펙에는 없다. Kotlin JVM도 마찬가지다 (`tailrec` 키워드는 컴파일러가 반복문으로 **직접 변환**하는 것이지, JVM TCO가 아님).

```java
// 이 코드도 깊은 입력에서는 StackOverflowError 발생!
public static long factorialTail(int n, long acc) {
    if (n == 0) return acc;
    return factorialTail(n - 1, n * acc);  // JVM은 여전히 새 프레임 생성
}
```

**교훈:** Java에서 깊은 재귀가 필요하다면 반복문으로 직접 변환하거나, 명시적 스택 자료구조를 사용하라.

### 함정 3: 메모 배열 초기화 값과 실제 반환값의 충돌

캐시 미스를 `-1`로 표시할 때, 실제 계산 결과도 `-1`이 될 수 있는 문제가 있다.

```java
// 위험한 패턴: 피보나치처럼 음수가 나오지 않는 문제에서는 -1 초기화가 안전
// 하지만 문제에 따라 결과값이 -1이 될 수 있음
int[] cache = new int[n + 1];
Arrays.fill(cache, -1);  // -1 = "계산하지 않음"

// 만약 dp 결과가 -1이 되는 문제라면?
// cache[i] = -1인지 확인 로직이 실제값 -1과 충돌!
```

```java
// 안전한 패턴 1: boolean 배열로 계산 여부 별도 관리
boolean[] computed = new boolean[n + 1];
int[] cache = new int[n + 1];

// 안전한 패턴 2: Integer 배열 사용 (null로 미계산 표시)
Integer[] cache = new Integer[n + 1];  // null = 미계산

// 안전한 패턴 3: Long.MIN_VALUE 같은 도달 불가능 값 사용
long[] cache = new long[n + 1];
Arrays.fill(cache, Long.MIN_VALUE);
```

**교훈:** 메모이제이션 초기화 값은 절대 실제 결과값이 될 수 없는 값(또는 별도 플래그)으로 선택하라.

### 함정 4: DP 상태 정의 실수

상태를 잘못 정의하면 점화식이 성립하지 않거나 정답을 구할 수 없다.

```
잘못된 예 (0/1 배낭):
  dp[i] = i번째 물건까지의 최대 가치 (용량 정보 없음)
  → 용량 제약이 상태에 반영되지 않아 의미 없음

올바른 예:
  dp[i][w] = 첫 i개 물건, 용량 w에서의 최대 가치
  → 두 차원 모두 상태에 필수적으로 포함
```

**교훈:** 상태는 해당 부분 문제를 **완전히 특정**할 수 있는 최소한의 정보를 담아야 한다. 점화식에 등장하는 모든 변수가 상태에 포함되어야 한다.

### 함정 5: 그리디를 항상 최적으로 오인

동전 거스름돈 문제에서 "항상 큰 단위 먼저"가 작동하는 경험 때문에 그리디가 항상 최적이라 착각하기 쉽다.

```
반례: 동전 {1, 3, 4}, 목표 금액 6
그리디 (큰 것 먼저): 4 + 1 + 1 = 3개
DP 최적:             3 + 3    = 2개  ← 더 적음
```

**교훈:** 그리디를 적용하기 전 **탐욕 선택 속성을 반드시 증명**하거나, 반례를 탐색하라. 증명 없이 그리디를 적용하는 것은 위험하다.

### 함정 6: 1D 배낭에서 순방향 갱신

1D 공간 최적화 배낭에서 `w`를 작은 값부터 큰 값으로 갱신하면(순방향), 같은 물건을 여러 번 담는 완전 배낭(Unbounded Knapsack)이 되어버린다.

```java
// 잘못된 예 (순방향 → 완전 배낭이 됨)
for (int w = weight[i]; w <= W; w++) {  // w가 증가하는 방향
    dp[w] = Math.max(dp[w], dp[w - weight[i]] + value[i]);
    // dp[w-weight[i]]가 이미 현재 i로 갱신된 값일 수 있음 → 중복 사용
}

// 올바른 예 (역순 → 0/1 배낭)
for (int w = W; w >= weight[i]; w--) {  // w가 감소하는 방향
    dp[w] = Math.max(dp[w], dp[w - weight[i]] + value[i]);
}
```

**교훈:** 0/1 배낭 1D DP에서는 **반드시 역순(큰 w에서 작은 w 방향)으로 갱신**하라.

## 12. 연습문제

### 문제 1 — 동전 거스름돈 DP (난이도: ★★☆)

**문제:** 동전 종류 배열 `coins[]`와 목표 금액 `amount`가 주어진다. 동전의 수를 최소화하여 `amount`를 만들어라. 불가능하면 -1을 반환하라. (각 동전은 무한히 사용 가능)

**예시:** `coins = [1, 5, 6, 9]`, `amount = 11` → 최소 2개 (`5 + 6`)

**힌트:** `dp[i]` = 금액 i를 만드는 최소 동전 수. 점화식: `dp[i] = min(dp[i - coins[j]] + 1)` for all valid j.

<details>
<summary>해설 보기</summary>

```java
public static int coinChange(int[] coins, int amount) {
    int[] dp = new int[amount + 1];
    Arrays.fill(dp, amount + 1);  // 불가능한 큰 값으로 초기화
    dp[0] = 0;                    // 기저 조건: 0원을 만드는 데 동전 0개

    for (int i = 1; i <= amount; i++) {
        for (int coin : coins) {
            if (i >= coin) {
                dp[i] = Math.min(dp[i], dp[i - coin] + 1);
            }
        }
    }

    return dp[amount] > amount ? -1 : dp[amount];
}
```

핵심 설명:
- `dp[i] = amount + 1`로 초기화하면 "불가능"을 나타내면서 min 연산에서 자연스럽게 무시된다.
- 최종적으로 `dp[amount] > amount`이면 amount+1 이상의 동전이 필요하다는 의미이므로 불가능.
- 이 문제는 완전 배낭(동전 무한 사용)이므로 순방향 갱신이 올바르다.
- 시간 복잡도: O(amount × |coins|), 공간 복잡도: O(amount)
</details>

### 문제 2 — 최장 증가 부분수열(LIS) (난이도: ★★★☆)

**문제:** 정수 배열 `nums[]`가 주어진다. 가장 긴 순증가 부분수열(Longest Increasing Subsequence)의 길이를 구하라.

**예시:** `nums = [10, 9, 2, 5, 3, 7, 101, 18]` → 4 (`[2, 3, 7, 101]` 또는 `[2, 5, 7, 101]`)

**힌트:** `dp[i]` = `nums[i]`를 마지막 원소로 하는 LIS의 최대 길이. 점화식: `dp[i] = max(dp[j] + 1)` for all `j < i && nums[j] < nums[i]`.

<details>
<summary>해설 보기</summary>

```java
public static int lengthOfLIS(int[] nums) {
    int n = nums.length;
    if (n == 0) return 0;

    int[] dp = new int[n];
    Arrays.fill(dp, 1);  // 각 원소 자체가 길이 1의 LIS

    int maxLen = 1;
    for (int i = 1; i < n; i++) {
        for (int j = 0; j < i; j++) {
            if (nums[j] < nums[i]) {
                dp[i] = Math.max(dp[i], dp[j] + 1);
            }
        }
        maxLen = Math.max(maxLen, dp[i]);
    }

    return maxLen;
}
```

핵심 설명:
- 상태 정의: `dp[i]`는 인덱스 i의 원소를 마지막으로 포함하는 LIS 길이.
- 기저 조건: 모든 `dp[i] = 1` (원소 자신만 포함).
- 이중 루프로 O(n²) 시간에 해결 가능.
- **도전:** 이진 탐색을 활용하면 O(n log n)으로 개선할 수 있다 (patience sort 기법).
</details>

### 문제 3 — 0/1 배낭을 1차원 배열로 공간 최적화 구현 (난이도: ★★☆)

**문제:** 아래 입력으로 0/1 배낭 문제를 1차원 DP 배열만을 사용하여 풀어라. 역순 갱신이 왜 필요한지 코드 주석으로 설명하라.

```
n=3, W=6
물건1: 무게=2, 가치=3
물건2: 무게=3, 가치=4
물건3: 무게=4, 가치=5
```

**힌트:** 물건마다 `w = W down to weight[i]`로 역순 순회. 정답은 `dp[6] = 7` (물건1 + 물건2).

<details>
<summary>해설 보기</summary>

```java
public static int knapsack1D(int n, int W, int[] weight, int[] value) {
    int[] dp = new int[W + 1];

    for (int i = 1; i <= n; i++) {
        // 역순 순회: dp[w-weight[i]]가 아직 이번 물건 i로
        // 갱신되지 않은 상태(= 이전 물건까지의 결과)를 참조하기 위함
        for (int w = W; w >= weight[i]; w--) {
            dp[w] = Math.max(dp[w], dp[w - weight[i]] + value[i]);
        }
    }

    return dp[W];
}

// 테스트
int n = 3, W = 6;
int[] weight = {0, 2, 3, 4};
int[] value  = {0, 3, 4, 5};
System.out.println(knapsack1D(n, W, weight, value)); // 7
```

갱신 과정 추적:

```
초기:      [0, 0, 0, 0, 0, 0, 0]
물건1(w=2,v=3) 처리 후: [0, 0, 3, 3, 3, 3, 3]
물건2(w=3,v=4) 처리 후: [0, 0, 3, 4, 4, 7, 7]
물건3(w=4,v=5) 처리 후: [0, 0, 3, 4, 5, 7, 8]
                                             ↑
                                        dp[6] = 7  (물건1+물건2)
```

</details>

### 문제 4 — 메모이제이션 vs 타뷸레이션 직접 비교 (난이도: ★★☆)

**문제:** 피보나치 수 `fib(n)`을 메모이제이션(top-down)과 타뷸레이션(bottom-up) 두 가지 방법으로 각각 구현하라. n=30, 40, 45에 대해 호출/루프 횟수를 측정하고, 두 방법의 공간 복잡도 차이를 분석하는 표를 작성하라.

**힌트:** 카운터 변수를 전역 또는 배열로 두어 각 함수 호출마다 증가시킨다.

<details>
<summary>해설 보기</summary>

```java
public class FibComparison {
    static int memoCallCount = 0;
    static int tabLoopCount = 0;
    static long[] cache;

    // Top-Down (메모이제이션)
    static long fibMemo(int n) {
        cache = new long[n + 1];
        Arrays.fill(cache, -1);
        cache[0] = 0; cache[1] = 1;
        return memoSolve(n);
    }

    static long memoSolve(int n) {
        memoCallCount++;
        if (cache[n] != -1) return cache[n];
        return cache[n] = memoSolve(n-1) + memoSolve(n-2);
    }

    // Bottom-Up (타뷸레이션)
    static long fibTab(int n) {
        long[] dp = new long[n + 1];
        dp[0] = 0; dp[1] = 1;
        for (int i = 2; i <= n; i++) {
            tabLoopCount++;
            dp[i] = dp[i-1] + dp[i-2];
        }
        return dp[n];
    }
}
```

결과표:

```
n   | 메모 호출수 | 타뷸 루프수 | 메모 공간   | 타뷸 공간
----+------------+------------+------------+-----------
 30 |    59      |    29      | O(n)+스택  | O(n)
 40 |    79      |    39      | O(n)+스택  | O(n)
 45 |    89      |    44      | O(n)+스택  | O(n)
```

결론: 총 연산 횟수는 비슷하지만, 메모이제이션은 콜 스택이 추가로 쌓인다. 큰 n에서는 타뷸레이션이 더 안전하다.
</details>

### 문제 5 — 등산 계단 오르기 (난이도: ★★)

**문제:** n개의 계단을 오를 때 한 번에 1칸 또는 2칸을 오를 수 있다. n번째 계단에 오르는 방법의 수를 구하라.

**예시:** n=4 → 5가지 (1+1+1+1, 1+1+2, 1+2+1, 2+1+1, 2+2)

**힌트:** 피보나치 수열과 동일한 점화식이 성립한다. 상태 정의부터 시작하라.

<details>
<summary>해설 보기</summary>

DP 설계:
- 상태: `dp[i]` = i번째 계단에 오르는 방법의 수
- 기저: `dp[1] = 1`, `dp[2] = 2`
- 점화식: `dp[i] = dp[i-1] + dp[i-2]` (마지막에 1칸 or 2칸 오름)

```java
public static int climbStairs(int n) {
    if (n <= 2) return n;
    int[] dp = new int[n + 1];
    dp[1] = 1;
    dp[2] = 2;
    for (int i = 3; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }
    return dp[n];
}
```

n=4: dp = [_, 1, 2, 3, 5] → 5가지

이 문제는 피보나치 수열의 변형으로, LeetCode 70번 "Climbing Stairs"와 동일한 문제다. DP의 기본 중의 기본이므로 반드시 손으로 테이블을 채워보라.
</details>

## 13. 요약

### 핵심 개념 정리

| 개념 | 정의 | 핵심 조건 |
|------|------|-----------|
| 재귀 | 함수가 자기 자신을 호출 | 기저 조건 + 재귀 단계 |
| 분할 정복 | 독립적인 부분 문제로 분해 | 부분 문제가 겹치지 않음 |
| DP | 겹치는 부분 문제를 저장·재사용 | 겹치는 부분 문제 + 최적 부분 구조 |
| 메모이제이션 | 재귀 + 캐시 (top-down) | 필요한 부분만 계산 |
| 타뷸레이션 | 반복문 + 테이블 (bottom-up) | 스택 안전, 순서 결정 필요 |
| 그리디 | 매 단계 국소 최선 선택 | 탐욕 선택 속성 증명 필수 |

### 복잡도 비교 요약

```
피보나치:
  순진 재귀 O(2ⁿ) → 메모/타뷸 O(n) → 공간 최적 O(1)

0/1 배낭:
  완전 탐색 O(2ⁿ) → 2D DP O(nW) / O(nW) → 1D DP O(nW) / O(W)
```

### DP 설계 체크리스트

```
□ 1. 상태를 명확히 정의했는가? (dp[i]의 의미)
□ 2. 점화식이 올바른가? (모든 경우를 빠짐없이 포함)
□ 3. 기저 조건이 완전한가? (최소 부분 문제 모두 처리)
□ 4. 계산 순서가 의존 관계와 일치하는가? (부분 문제 먼저)
□ 5. 캐시/테이블 초기화 값이 실제 결과값과 충돌하지 않는가?
□ 6. 공간 최적화 가능성을 검토했는가?
```
