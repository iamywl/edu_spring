## 왜 이 장이 필요한가 — 백엔드 개발자 관점

동시성 버그의 **모든 원인은 단 한 문장으로 요약된다: "여러 실행 흐름이 같은 데이터를 공유하기 때문"**이다.
그렇다면 무엇이 공유되고 무엇이 공유되지 않는지를 정확히 아는 것이 출발점이다.

이 구분을 알면 [`SPRING_개념서 §2.7`](../../../spring/docs/SPRING_개념서/)의 핵심 조언 — **"Service는 stateless여야 안전하다"** —
이 왜 나왔는지가 즉시 보인다. 필드(공유)에 요청별 데이터를 담으면 스레드끼리 섞이고,
지역 변수(전용)에 담으면 절대 안 섞인다. 이 장은 그 "공유 vs 전용"의 경계를 OS 수준에서 그린다.

이 장을 끝내면 다음을 설명할 수 있다.

- 프로그램·프로세스·스레드가 각각 무엇인지.
- 프로세스는 왜 서로 격리되는지(안정성·보안).
- 스레드가 **무엇을 공유하고(힙·코드·static) 무엇을 따로 갖는지(스택·레지스터)**.
- 문맥 교환(context switch)이 무엇이며, 스레드가 프로세스보다 왜 "싼지".

## 1.1 프로세스: 실행 중인 프로그램

### 개념

**프로그램(program)**은 디스크에 저장된 죽어있는 파일이다. `.class`, `.jar`, `.exe` 같은 바이트 덩어리일 뿐,
그 자체로는 아무 일도 하지 않는다. 요리책에 적힌 레시피와 같다 — 읽을 수는 있어도 저절로 요리가 되지는 않는다.

이것을 **실행**하면 OS가 파일을 메모리에 올리고, CPU 시간을 배정하고, 필요한 자원(파일 핸들, 네트워크 소켓 등)을
할당한다. 이렇게 **"실행 중인 프로그램의 인스턴스"**가 곧 **프로세스(process)**다. 레시피(프로그램)로 실제 요리를
하는 행위(프로세스)인 셈이다. 같은 프로그램을 두 번 실행하면 프로세스는 둘이 된다(크롬 창을 두 개 띄우면
크롬 프로세스가 여러 개 생기는 것과 같다).

### 원리 — 독립된 주소 공간

OS는 프로세스마다 **독립된 주소 공간(address space)**을 준다. 각 프로세스는 "메모리 전체가 내 것"인 것처럼
0번지부터 시작하는 자기만의 주소를 본다.

```
  ┌─────────── 프로세스 A ───────────┐   ┌─────────── 프로세스 B ───────────┐
  │  코드 | 데이터 | 힙 | 스택        │   │  코드 | 데이터 | 힙 | 스택        │
  │  주소 0x0000 ~ 0xFFFF (A만의 것)  │   │  주소 0x0000 ~ 0xFFFF (B만의 것)  │
  └──────────────────────────────────┘   └──────────────────────────────────┘
        서로의 메모리를 볼 수 없다 (OS가 격리한다)
```

한 프로세스의 메모리 구성은 대략 이렇게 나뉜다(자바 기준 개념적으로).

- **코드(text) 영역**: 실행할 명령어. 읽기 전용.
- **데이터 영역**: 전역/정적 데이터(자바의 static, 클래스 메타데이터가 사는 곳에 대응).
- **힙(heap)**: 실행 중 동적으로 할당되는 객체. 자바의 `new`가 여기에 만든다.
- **스택(stack)**: 함수 호출 정보, 지역 변수, 매개변수. 호출할 때 쌓이고 반환할 때 걷힌다.

> ### 왜 프로세스를 격리하는가?
>
> 브라우저가 죽어도 음악 앱은 멀쩡해야 한다. 한 프로그램의 버그가 다른 프로그램의 메모리를 망가뜨리면 안 되기
> 때문이다. 프로세스 간 격리는 **안정성과 보안의 기본**이다. 만약 격리가 없다면 악성 프로그램이 은행 앱의
> 메모리를 몰래 읽어 비밀번호를 훔칠 수 있을 것이다.
>
> 재미있게도 A의 `0x1000`과 B의 `0x1000`은 실제로는 물리 메모리의 **서로 다른 위치**다. 같은 주소를 봐도
> 진짜 RAM에서는 딴 곳을 가리킨다. 이 마법은 [8장 가상 메모리](08-가상-메모리와-JVM-힙.md)에서 설명한다.

## 1.2 스레드: 프로세스 안의 실행 흐름

### 개념

프로세스가 "집"이라면, **스레드(thread)**는 그 집 안에서 일하는 "일꾼"이다. 한 프로세스 안에 스레드가 여러 개일 수
있고, 이들은 **같은 집(주소 공간)을 공유**한다. 프로세스는 최소 하나의 스레드(메인 스레드)로 시작하며,
필요하면 스레드를 더 만들어 여러 일을 동시에 진행한다.

자바로 보면, JVM 프로세스 하나 안에서 `main` 스레드가 시작되고, `new Thread()`나 스레드 풀로 스레드를 더 만든다.
톰캣 서버라면 요청마다 워커 스레드가 붙어 수백 개가 동시에 돈다.

### 원리 — 무엇을 공유하고 무엇을 따로 갖는가

이것이 이 책 전체에서 가장 중요한 그림이다. 여기서 모든 동시성 문제가 태어나기 때문이다.

```
  ┌──────────────────── 하나의 프로세스 (JVM) ────────────────────┐
  │                                                                │
  │   [공유]  코드(Method Area) · 힙(Heap, 모든 객체)              │  ← 모든 스레드가 함께 본다
  │            ┌──────┐ ┌──────┐ ┌──────┐                         │
  │            │객체 A│ │객체 B│ │static│                         │
  │            └──────┘ └──────┘ └──────┘                         │
  │                                                                │
  │   [따로]  스레드마다 자기만의 스택 + 레지스터(PC 등)          │  ← 절대 안 섞인다
  │   ┌─ Thread-1 ─┐   ┌─ Thread-2 ─┐   ┌─ Thread-3 ─┐          │
  │   │ 스택        │   │ 스택        │   │ 스택        │          │
  │   │ (지역변수)  │   │ (지역변수)  │   │ (지역변수)  │          │
  │   │ 프로그램    │   │ 프로그램    │   │ 프로그램    │          │
  │   │ 카운터(PC)  │   │ 카운터(PC)  │   │ 카운터(PC)  │          │
  │   └────────────┘   └────────────┘   └────────────┘          │
  └────────────────────────────────────────────────────────────────┘
```

- **공유(shared)**: 힙(모든 객체), 코드, static 변수, 열린 파일 핸들 등. 프로세스 전체에 하나뿐이다.
- **전용(private)**: 각 스레드의 **스택**(그래서 지역 변수·매개변수는 안 섞인다)과
  **레지스터**(특히 다음 실행할 명령을 가리키는 **프로그램 카운터, PC**).

왜 이렇게 나뉠까? 스택은 "지금 어느 함수의 어느 지점에 있는가"라는 **실행 위치**를 담는다. 스레드가 여럿이면
각자 다른 함수의 다른 지점에 있어야 하므로 스택은 반드시 따로여야 한다. 반면 힙의 객체는 "데이터"이고,
스레드가 협력하려면 같은 데이터를 봐야 하므로 공유된다.

### 자바 코드로 확인

```java
public class ShareVsPrivate {
    static int sharedField = 0;          // 힙/데이터 영역 → 모든 스레드가 공유

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            int localVar = 0;            // 스택 → 스레드마다 따로 (절대 안 섞임)
            for (int i = 0; i < 1000; i++) {
                localVar++;              // 안전: 내 스택의 내 변수
                sharedField++;           // 위험: 모두가 공유하는 필드
            }
            System.out.println(Thread.currentThread().getName()
                    + " localVar=" + localVar);   // 항상 1000 (안 섞임)
        };
        Thread t1 = new Thread(task, "T1");
        Thread t2 = new Thread(task, "T2");
        t1.start(); t2.start();
        t1.join();  t2.join();
        System.out.println("sharedField=" + sharedField); // 2000이 아닐 수 있다!
    }
}
```

`localVar`는 각 스레드의 스택에 따로 존재하므로 언제나 정확히 1000이 나온다. 그러나 `sharedField`는 두 스레드가
공유하므로 2000이 아닐 수 있다 — 왜 그런지는 [3장](03-경쟁-상태와-임계-구역.md)에서 파헤친다. 지금 기억할 것은
단 하나: **섞이는 것은 오직 공유되는 필드/힙의 데이터뿐이다.**

> ### 여기서 모든 동시성 문제가 나온다
>
> 지역 변수는 스택에 있어 스레드마다 따로다 → **안전하다.**
> 하지만 **힙의 객체나 static 필드는 공유된다** → 여러 스레드가 동시에 건드리면 **위험하다.**
>
> [`SPRING_개념서 §2.7`](../../../spring/docs/SPRING_개념서/)에서 "Service는 stateless여야 안전하다"고 한 이유가 정확히 이것이다.
> 싱글톤 Service Bean은 프로세스에 하나뿐이라 모든 요청 스레드가 공유한다. 그 Bean의 **필드**(공유)에 요청별
> 데이터를 담으면 요청끼리 섞이고, 메서드 안 **지역 변수**(전용)에만 담으면 수백 스레드가 동시에 들어와도 안 섞인다.
>
> ```
> 안전 (stateless):                     위험 (stateful):
>   String process(Order o) {             private Order current;  // 공유 필드!
>     String r = ...;   // 지역변수         void process(Order o) {
>     return r;         // 안 섞임            this.current = o;   // 요청끼리 덮어씀
>   }                                        ...
>                                          }
> ```

## 1.3 문맥 교환(Context Switch)과 "스레드는 왜 싸긴 싼가"

### 개념

CPU 코어 하나는 한 순간에 스레드 **하나**만 실행한다. 그럼 어떻게 여러 스레드가 "동시에" 도는 것처럼 보일까?
OS 스케줄러가 스레드들을 **아주 빠르게 번갈아** 실행하기 때문이다. 사람 눈에는 동시로 보이지만, 실제로는
초당 수백~수천 번 번갈아 도는 것이다.

한 스레드를 잠시 멈추고 다른 스레드로 바꾸는 것을 **문맥 교환(context switch)**이라 한다.
"문맥(context)"이란 그 스레드의 실행 상태 — 레지스터 값들, 특히 프로그램 카운터(PC) — 를 말한다.

### 원리 — 저장하고 복원한다

문맥 교환의 절차는 이렇다.

1. 실행 중이던 스레드 T1을 멈춘다.
2. T1의 문맥(레지스터, PC 등)을 메모리에 **저장**한다. — "T1이 어디까지 했는지" 기록.
3. 다음에 돌릴 스레드 T2의 문맥을 메모리에서 **복원**한다. — "T2가 아까 멈췄던 지점"으로 되돌림.
4. T2를 실행한다.

```
  시간 →
  CPU:  [T1][T2][T1][T3][T2][T1] ...   각 조각이 수 밀리초(time slice)
         ↑저장/복원(문맥 교환) 비용이 매 전환마다 발생
```

각 조각을 **타임 슬라이스(time slice, quantum)**라 한다. 스케줄러는 슬라이스가 끝나거나, 스레드가 I/O를 기다려
스스로 CPU를 놓거나(블로킹), 더 급한 스레드가 나타나면 문맥 교환을 일으킨다. **바로 이 교환이 "언제, 어느 명령과
어느 명령 사이에서" 일어날지 우리가 통제할 수 없다는 점**이 다음 장(비결정성)과 그다음 장(경쟁 상태)의 씨앗이 된다.

> ### 왜 스레드가 프로세스보다 "싸다"고 하는가?
>
> - **생성 비용**: 새 프로세스는 주소 공간 전체를 새로 만들어야 한다(페이지 테이블 등). 스레드는 기존 주소 공간을
>   공유하므로 스택만 새로 잡으면 된다 → 훨씬 가볍다.
> - **문맥 교환 비용**: 같은 프로세스 내 스레드끼리는 주소 공간(페이지 테이블 등)이 같아서 교환 시 갈아끼울 것이
>   적다. 프로세스 간 전환은 주소 공간까지 바꿔야 하므로 더 무겁다(TLB 무효화 등).
> - **통신 비용**: 스레드는 힙을 공유하니 그냥 같은 객체를 읽으면 된다. 프로세스 간 통신(IPC)은 별도 채널
>   (파이프, 소켓, 공유 메모리)이 필요해 번거롭다.
>
> **"싸긴 싸다"고 조심스럽게 말하는 이유**: 스레드도 공짜는 아니다. 스택 메모리(보통 ~1MB), 커널의 스케줄링
> 등록 비용이 든다. 그래서 스레드 수천 개를 매번 만들면 서버가 죽는다. → 그래서 **스레드 풀**을 쓴다([7장](07-고수준-동시성-도구.md)).
> ▶ 데모: `./run.sh ExecutorAndFutures`  📁 `chapter-cs-concurrency/src/main/java/com/edu/concurrency/ExecutorAndFutures.java`

### 곁가지 — 사용자 스레드 vs 플랫폼 스레드, 그리고 가상 스레드

전통적으로 자바 스레드 하나는 OS 스레드 하나에 1:1로 대응한다(**플랫폼 스레드**). OS 스레드는 무겁기 때문에
수만 개를 만들면 메모리와 스케줄링 비용이 감당이 안 된다.

자바 21의 **가상 스레드(virtual thread)**는 이 한계를 넘으려는 시도다. 수천 개의 가상 스레드를 소수의 OS 스레드
위에서 돌려, 블로킹 I/O가 많은 서버에서도 스레드를 저렴하게 쓸 수 있게 한다. 개념적으로는 "스레드를 아주 싸게
만드는" 방향이며, 이 책이 다루는 공유/동기화 문제는 가상 스레드에서도 **그대로** 적용된다(공유 데이터를 건드리는
한 위험은 똑같다). 이 책의 데모는 전통적 플랫폼 스레드와 스레드 풀 기준이지만, 원리는 동일하다.

## ⚠️ 흔한 오해와 함정

- **"스레드는 메모리를 다 공유한다"** → 아니다. **힙·코드·static은 공유하지만 스택과 레지스터는 스레드마다 따로다.**
  이 절반의 오해가 "지역 변수도 위험한 거 아냐?"라는 불필요한 걱정을 낳는다. 지역 변수는 언제나 안전하다.
- **"멀티스레드는 항상 빠르다"** → 아니다. 문맥 교환에도 비용이 들고, 공유 데이터에 락을 걸면 대기가 생긴다.
  코어 수보다 스레드가 너무 많으면 교환 비용만 늘어 오히려 느려질 수 있다.
- **"프로세스가 여러 개면 데이터가 공유된다"** → 아니다. 프로세스는 주소 공간이 격리되어 **직접 공유되지 않는다.**
  공유하려면 별도의 IPC가 필요하다. 데이터를 그냥 공유하는 건 같은 프로세스 안의 스레드들이다.
- **"static 변수는 스레드마다 따로"** → 아니다. static은 클래스당 하나뿐이라 **모든 스레드가 공유**한다.
  스레드마다 따로 두고 싶으면 `ThreadLocal`을 써야 한다.

## 연습문제

1. **(개념)** 프로그램과 프로세스의 차이를 한 문장으로 설명하고, 같은 프로그램을 두 번 실행하면 프로세스가
   몇 개 생기는지 답하라.
   <details><summary>힌트/해설</summary>
   프로그램은 디스크의 정적 파일, 프로세스는 그것을 실행 중인 인스턴스다. 두 번 실행하면 프로세스는 2개.
   각 프로세스는 독립된 주소 공간을 가지므로 서로의 메모리를 못 본다.</details>

2. **(핵심)** 다음 중 여러 스레드가 동시에 접근할 때 "섞일 위험이 있는" 것을 모두 고르라.
   (a) 메서드의 지역 변수 (b) 인스턴스 필드 (c) static 필드 (d) 매개변수 (e) 힙에 있는 공유 객체.
   <details><summary>힌트/해설</summary>
   위험: (b), (c), (e) — 모두 공유되는 힙/데이터 영역. 안전: (a), (d) — 각 스레드의 스택에 따로 존재.
   이 구분이 stateless Service의 근거다.</details>

3. **(응용)** 톰캣이 요청을 "요청마다 새 프로세스"가 아니라 "요청마다 스레드"로 처리하는 이유를 §1.3의
   세 가지 비용(생성·문맥 교환·통신) 관점에서 설명하라.
   <details><summary>힌트/해설</summary>
   프로세스는 주소 공간을 새로 만들어야 해 생성·교환 비용이 크고, 커넥션 풀·캐시 같은 공유 자원에 접근하기도
   어렵다. 스레드는 같은 힙을 공유해 이 자원들에 바로 접근하고, 생성·교환도 훨씬 싸다. 그래서 thread-per-request.</details>

4. **(함정)** 아래 코드에서 각 스레드가 출력하는 `count`의 마지막 값이 항상 100인 이유는?
   ```java
   Runnable r = () -> { int count = 0; for (int i=0;i<100;i++) count++; System.out.println(count); };
   ```
   <details><summary>힌트/해설</summary>
   `count`가 지역 변수라 각 스레드의 스택에 따로 존재하기 때문. 공유되지 않으므로 경쟁 자체가 없다.
   만약 `count`를 static 필드로 바꾸면 결과가 무너질 수 있다.</details>

5. **(심화)** `ThreadLocal`은 왜 필요한가? "스레드마다 따로 갖는 것은 스택인데, 필드를 스레드별로 갖고 싶을 때"라는
   맥락에서 설명하라.
   <details><summary>힌트/해설</summary>
   필드는 원래 공유된다. 그러나 트랜잭션 컨텍스트, 사용자 정보처럼 "스레드별로 하나씩" 두고 싶은 값이 있다.
   `ThreadLocal`은 값을 스레드별 저장소에 보관해 공유 없이 스레드마다 독립된 값을 갖게 해 준다. 스프링의
   트랜잭션·시큐리티 컨텍스트가 이 방식을 쓴다.</details>

## 요약

- **프로그램**은 디스크의 정적 파일, **프로세스**는 그것의 실행 인스턴스, **스레드**는 프로세스 안의 실행 흐름.
- OS는 프로세스마다 **독립된 주소 공간**을 줘서 격리한다(안정성·보안).
- 한 프로세스 안의 스레드들은 **힙·코드·static을 공유**하고, **스택·레지스터(PC)는 각자 따로** 갖는다.
- **공유되는 데이터를 여러 스레드가 건드리는 것** — 이것이 모든 동시성 문제의 뿌리다. 지역 변수는 안전하다.
- **문맥 교환**은 스레드를 번갈아 실행하려고 문맥을 저장/복원하는 것. 그 타이밍을 우리가 통제할 수 없다는 점이
  다음 장들의 핵심 씨앗이다.
- 스레드는 프로세스보다 생성·교환·통신이 싸지만 공짜는 아니다 → 스레드 풀이 필요한 이유(7장).
