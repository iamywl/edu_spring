## 왜 이 장이 필요한가 — 백엔드 개발자 관점

락은 경쟁 상태를 풀지만 **새로운 문제**를 만든다. 두 스레드가 서로가 쥔 락을 기다리면 **둘 다 영원히 멈춘다.**
서버라면 그 요청들이 응답 없이 쌓이고, 스레드 풀이 다 소진되면 **서버 전체가 응답 불능**이 된다. 에러 로그도
스택 트레이스도 없이 "그냥 느려지다 멈추는" 가장 진단하기 까다로운 장애 중 하나가 교착이다.

이 장을 끝내면 다음을 설명할 수 있다.

- 교착 상태가 무엇이고 어떻게 발생하는지(순환 대기).
- **코프만(Coffman)의 네 가지 조건**과, 하나만 깨도 예방된다는 원리.
- 교착에 대응하는 네 전략 — **예방·회피·탐지·복구**의 구분.

## 6.1 서로가 서로를 기다리는 정지

### 개념

두 스레드가 **두 개의 락을 반대 순서로** 잡으면 교착이 생긴다.

```java
// Thread-1
lockA.lock();          // A를 먼저 잡고
lockB.lock();          // B를 원함

// Thread-2
lockB.lock();          // B를 먼저 잡고
lockA.lock();          // A를 원함
```

### 원리 — 순환 대기

```
  Thread-1: 락 A 잡음 ──▶ 락 B를 원함
  Thread-2: 락 B 잡음 ──▶ 락 A를 원함

     Thread-1 ──잡음──▶ [락 A] ◀──원함── Thread-2
         │                                  ▲
         └──원함──▶ [락 B] ──잡음──────────┘

  A는 B를 기다리고, B는 A를 기다린다 → 영원히 아무도 못 나아감 = 교착(Deadlock)
```

에러도 없이, 로그도 없이, 관련 스레드가 그냥 **얼어붙는다.** 이 대기 관계를 그래프로 그리면 **사이클(원)**이
생기는데, 이 사이클이 교착의 시각적 표식이다.

```
  자원 할당 그래프 (사이클 = 교착):
    T1 ──보유──▶ A     T2 ──보유──▶ B
    T1 ──요청──▶ B     T2 ──요청──▶ A
    → T1→B→T2→A→T1 로 도는 사이클 존재 → 교착
```

교착은 락 두 개뿐 아니라, 세 스레드가 원형으로 물고 물릴 때도 생긴다(T1→T2→T3→T1). 핵심은 "대기 관계가
원을 이루는가"다.

## 6.2 코프만(Coffman)의 네 가지 조건

### 개념

교착은 다음 **네 가지가 모두** 성립할 때만 발생한다(1971년 코프만 등이 정리).

1. **상호 배제(Mutual Exclusion)**: 자원(락)은 한 번에 한 스레드만 쥔다.
2. **점유 대기(Hold and Wait)**: 자원 하나를 쥔 채로 다른 자원을 기다린다.
3. **비선점(No Preemption)**: 쥔 자원을 강제로 뺏을 수 없다.
4. **순환 대기(Circular Wait)**: 대기 관계가 원(사이클)을 이룬다.

### 원리 — 하나만 깨도 예방된다

**핵심**: 이 넷은 **모두 필요**(필요조건들의 곱)하다. 따라서 **하나만 깨도 교착은 예방된다.** 각 조건을 깨는
전략이 곧 예방법이다.

| 깰 조건 | 방법 | 대가/한계 |
|---------|------|-----------|
| ① 상호 배제 | 자원을 공유 가능하게(읽기 전용, 불변) | 근본적으로 배타적 자원엔 불가 |
| ② 점유 대기 | 필요한 락을 **한꺼번에** 모두 잡거나, 하나라도 못 잡으면 전부 놓기 | 자원 활용도 저하, 기아 위험 |
| ③ 비선점 | 락을 못 잡으면 이미 쥔 것을 **놓고 물러남**(선점 허용) | 롤백 비용, 라이브락 위험 |
| ④ 순환 대기 | **락 획득 순서를 전역적으로 통일** | 실무에서 가장 실용적 ✔ |

실무에서 가장 자주 쓰는 것은 **④를 깨는 것**이다. 다음 절에서 본다.

## 6.3 예방(Prevention): 락 순서를 통일하라

### 개념

**모든 스레드가 락을 항상 같은 순서로 잡게** 하면 순환 대기가 생길 수 없다.

```
  규칙: 누구든 락을 잡을 땐 반드시 "A → B" 순서로.

  Thread-1: A 잡고 → B 원함
  Thread-2: A 원함 (아직 A도 못 잡음, 그래서 B도 안 건드림)
            → 사이클이 생길 수 없다 → 교착 불가능
```

### 원리 — 왜 사이클이 못 생기나

모두가 "번호가 낮은 락부터" 잡는다고 하자. 어떤 스레드가 락 3을 쥔 채 락 5를 기다린다면, 그 스레드는 3을 5보다
먼저 잡은 것이다(규칙상). 그런데 사이클이 생기려면 "락 5를 쥔 스레드가 락 3을 기다리는" 반대 방향이 필요한데,
그 스레드는 5를 잡기 전에 이미 3을 잡았어야 한다(규칙 위반). 즉 **전역 순서를 지키면 대기 방향이 한쪽으로만
흘러 원이 닫히지 않는다.**

### 자바 코드 — 계좌 이체의 락 순서 정하기

```java
void transfer(Account from, Account to, long amount) {
    // 두 계좌의 고유 ID로 순서를 정해 "항상 작은 ID부터" 잠근다.
    Account first  = from.id() < to.id() ? from : to;
    Account second = from.id() < to.id() ? to   : from;
    synchronized (first) {
        synchronized (second) {
            from.withdraw(amount);
            to.deposit(amount);
        }
    }
}
```

`transfer(A, B)`와 `transfer(B, A)`가 동시에 일어나도, 둘 다 **작은 ID부터** 잠그므로 순서가 같아 교착이 원천
봉쇄된다. 정렬 기준은 어떤 전역적으로 일관된 값이든 좋다(ID, 해시, 이름 등).

## 6.4 회피·탐지·복구 — 나머지 세 전략

예방(prevention)이 "조건을 아예 못 생기게 막는" 것이라면, 나머지 전략들도 알아 두면 시야가 넓어진다.
운영체제 교재는 교착 대응을 보통 **예방·회피·탐지·무시**로 나눈다.

### 회피(Avoidance)

**실행 중에 "이 락을 지금 주면 교착 위험이 있는가"를 판단해, 위험하면 주지 않는** 방식이다.
고전 알고리즘이 **은행원 알고리즘(Banker's Algorithm)** — 자원을 할당했을 때 모두가 결국 완료 가능한
"안전 상태(safe state)"가 유지되는지 검사하고, 안전하지 않으면 할당을 미룬다. 각 스레드의 최대 자원 요구량을
미리 알아야 해서 실무 코드에서는 드물지만, 개념은 알아 둘 가치가 있다.

실무에서 회피에 가장 가까운 도구는 [4장](04-동기화-락-cas-volatile.md)의 **`tryLock(타임아웃)`**이다.
"정해진 시간 안에 못 잡으면 포기하고 물러나기"는 점유 대기·비선점을 부드럽게 깨는 실용적 회피다.

```java
if (lockB.tryLock(3, TimeUnit.SECONDS)) {     // 3초 안에 못 잡으면
    try { /* ... */ } finally { lockB.unlock(); }
} else {
    // 포기하고 이미 쥔 락도 놓고 물러남 → 교착 회피
}
```

### 탐지(Detection)와 복구(Recovery)

**교착이 생기게 두되, 주기적으로 탐지해 발견하면 복구**하는 방식이다. 자원 할당 그래프에서 **사이클을 찾으면**
교착이다. 발견하면 관련 스레드 중 하나를 강제 종료하거나 자원을 뺏어(선점) 사이클을 끊는다(복구).

자바에서는 `ThreadMXBean.findDeadlockedThreads()`로 교착에 빠진 스레드를 탐지할 수 있고, `jstack`으로 스레드
덤프를 뜨면 `Found one Java-level deadlock:` 메시지로 어떤 스레드가 어떤 락을 물고 있는지 보여준다. DB도
교착을 탐지해 한 트랜잭션을 희생시켜(deadlock victim, 강제 롤백) 사이클을 끊는다 — 탐지·복구 전략의 실사용 예다.

### 무시(Ostrich Algorithm)

교착이 드물고 손해가 작다면 **그냥 무시**하고, 나면 재시작하는 선택도 있다("타조 알고리즘" — 모래에 머리를 묻는).
많은 범용 OS가 애플리케이션 수준 교착에 대해선 사실상 이 전략을 택한다.

### 데모 — 유발과 예방을 안전하게 보여주기

[`DeadlockDemo`](../../chapter-cs-concurrency/src/main/java/com/edu/concurrency/DeadlockDemo.java)는
진짜 `synchronized`로 교착을 만들면 프로그램이 영원히 멈춰 교육용으로 위험하므로, `tryLock(타임아웃)` 기반의
**감시견(watchdog)**으로 교착을 감지·탈출한다. 즉 회피 기법으로 안전하게 "교착이 발생했다"만 보여준 뒤,
락 순서를 통일한 버전이 교착 없이 완료되는 것을 대조한다.

```java
// 1) 유발: 반대 순서로 잡되, tryLock 타임아웃으로 hang 없이 탈출
lockA.lock();
boolean gotB = lockB.tryLock(3, TimeUnit.SECONDS);   // 3초 뒤 포기 → "교착 감지"
// 2) 해결: 두 스레드 모두 A→B 순서로만 잡음 → 순환 대기 불가능
```

> ▶ 데모: `./run.sh DeadlockDemo` — 반대 순서로 교착을 일부러 일으키고(감시견 타임아웃으로 감지·탈출),
> 그다음 락 순서를 통일해 교착이 사라지는 것을 보여준다. **프로그램은 절대 멈추지 않는다.**

## ⚠️ 흔한 오해와 함정

- **"교착은 코드가 잘못돼서 에러가 난다"** → 아니다. 교착은 **에러 없이 조용히 멈춘다.** 그래서 더 진단하기 어렵다.
  증상은 "특정 요청들이 응답이 없다", "스레드 풀이 고갈된다"로 나타난다.
- **"락을 하나만 쓰면 교착은 절대 없다"** → 대체로 맞다(순환 대기가 성립하려면 락이 둘 이상 필요). 다만 재진입
  불가능한 락을 같은 스레드가 두 번 잡으려 하면 자기 자신과 교착할 수 있다(자바 모니터·ReentrantLock은 재진입
  가능해 이 문제는 없다).
- **"코프만 조건 넷을 다 없애야 예방된다"** → 반대다. **하나만 깨도** 예방된다. 넷이 모두 필요조건이기 때문.
- **"tryLock을 쓰면 교착이 근본적으로 사라진다"** → tryLock은 **회피**(빠져나오기)일 뿐 예방이 아니다.
  포기 후 재시도가 계속 실패하면 **라이브락(livelock)** — 서로 양보만 하다 진행 못 함 — 이 생길 수 있다.
  근본 해결은 락 순서 통일(예방)이다.
- **"교착과 기아는 같은 것"** → 다르다. 교착은 서로 물려 아무도 못 가는 것, 기아는 특정 스레드만 계속 밀려
  자원을 못 얻는 것. 원인과 해법이 다르다.

## 연습문제

1. **(핵심)** 코프만의 네 조건을 나열하고, 각 조건을 깨는 방법을 하나씩 적어라. 실무에서 가장 실용적인 것은?
   <details><summary>힌트/해설</summary>
   ① 상호배제→자원 공유(불변), ② 점유대기→한꺼번에 잡기/전부 놓기, ③ 비선점→tryLock으로 물러나기,
   ④ 순환대기→락 순서 통일. 가장 실용적: ④ 락 순서 통일.</details>

2. **(개념)** 예방·회피·탐지·복구를 각각 한 문장으로 정의하고, `tryLock(타임아웃)`은 이 중 무엇에 해당하는지 답하라.
   <details><summary>힌트/해설</summary>
   예방=조건을 아예 못 생기게, 회피=실행 중 위험한 할당을 미룸/물러남, 탐지=생긴 뒤 사이클 찾기, 복구=희생자를
   골라 사이클 끊기. tryLock(타임아웃)은 회피에 해당(못 잡으면 물러남).</details>

3. **(응용)** `transfer(A,B)`와 `transfer(B,A)`가 동시에 호출될 때 교착이 나는 코드를, 락 순서 통일로 고쳐라.
   <details><summary>힌트/해설</summary>
   두 계좌의 고유 ID를 비교해 "항상 작은 ID부터" 잠근다(§6.3 코드). 두 호출 모두 같은 순서로 잡으므로 순환 대기
   불가.</details>

4. **(함정)** 두 스레드가 서로 tryLock 실패 시 락을 놓고 즉시 재시도하기를 반복하며 영영 진행하지 못한다.
   이것을 무엇이라 하며, 어떻게 완화하는가?
   <details><summary>힌트/해설</summary>
   라이브락(livelock) — 멈추진 않지만 서로 양보만 하며 진행 못 함. 재시도 전 무작위 백오프(random backoff)를
   주거나, 근본적으로 락 순서를 통일(예방)해 애초에 충돌을 없앤다.</details>

5. **(심화)** 자원 할당 그래프에서 "사이클이 있으면 교착"이 항상 성립하는가? 자원이 각 종류당 1개일 때와 여러 개일
   때를 나눠 논하라.
   <details><summary>힌트/해설</summary>
   자원 인스턴스가 각 종류당 1개면 "사이클 = 교착"이 성립. 인스턴스가 여러 개면 사이클이 있어도 교착이 아닐 수
   있다(다른 인스턴스로 요구가 충족될 수 있으므로). 여러 개일 땐 사이클은 "교착 가능성"의 필요조건일 뿐.</details>

## 요약

- **교착(deadlock)**은 스레드들이 서로가 쥔 락을 기다리며 **에러 없이 영원히 멈추는** 상태. 대기 관계가 **사이클**을 이룬다.
- **코프만 4조건**(상호 배제·점유 대기·비선점·순환 대기)이 **모두** 성립해야 발생하므로, **하나만 깨도 예방**된다.
- 실무에서 가장 실용적인 예방은 **④ 순환 대기 깨기 = 락 획득 순서를 전역적으로 통일**(예: 항상 작은 ID부터).
- 교착 대응 전략: **예방**(조건 차단), **회피**(tryLock/은행원 알고리즘), **탐지+복구**(사이클 찾아 희생자 롤백, DB가 사용), **무시**.
- ▶ 데모: `./run.sh DeadlockDemo` — 교착 유발(감시견으로 감지·탈출) → 락 순서 통일로 해결. 절대 멈추지 않는다.
