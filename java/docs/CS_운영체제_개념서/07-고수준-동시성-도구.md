## 왜 이 장이 필요한가 — 백엔드 개발자 관점

지금까지 본 락·CAS·`volatile`은 **"원재료"**다. 이것들로 직접 스레드를 만들고 큐를 잠그면 코드가 복잡하고 버그가
잦다. 실무에서는 이것들을 감싼 **더 안전한 고수준 도구**를 쓴다. 그리고 이 도구들은 남의 이야기가 아니다 —
**톰캣 스레드 풀, HikariCP 커넥션 풀, 메시지 큐 소비자**가 전부 이 장의 개념 그대로다.

이 장을 끝내면 다음을 설명할 수 있다.

- **스레드 풀**과 `ExecutorService`가 왜 필요하고 어떻게 도는지.
- **`Future`**로 "나중에 나올 결과"를 받는 법.
- **`BlockingQueue`**와 생산자-소비자, 그리고 **배압(back-pressure)**의 의미.

## 7.1 스레드 풀과 `ExecutorService`

### 개념

[1장 §1.3](01-프로세스와-스레드.md)에서 봤듯 스레드는 만들 때마다 비용이 든다(스택 ~1MB, 커널 등록).
요청이 초당 수천 개인데 매번 `new Thread()`를 하면 서버가 죽는다. **스레드 풀(thread pool)**은 스레드를 미리 몇 개
만들어 두고 **재사용**한다. 우리는 스레드가 아니라 **작업(task)**만 던지고, 놀고 있는 워커가 집어 처리한다.

### 원리 — 작업 큐 + 워커

```
  작업들 ─▶ [작업 큐] ─▶ ┌─ 워커1 ─┐
                          ├─ 워커2 ─┤  (고정 개수를 계속 재사용)
                          └─ 워커3 ─┘
```

작업을 `submit`하면 큐에 들어가고, 각 워커 스레드는 "큐에서 작업을 꺼내 → 실행 → 다시 큐에서 꺼내"를 반복한다.
워커 스레드는 죽지 않고 살아 있으면서 여러 작업을 처리하므로, **생성 비용이 한 번만** 든다.

### 자바 코드

```java
ExecutorService pool = Executors.newFixedThreadPool(3);   // 워커 3개
Future<Integer> f = pool.submit(() -> 2 + 2);  // 작업 제출 → 영수증(Future) 즉시 반환
int result = f.get();                           // 결과가 준비될 때까지 대기 후 회수
pool.shutdown();                                // 다 쓰면 반드시 종료 (안 하면 프로그램이 안 끝남)
```

- **`Future<T>`**: "지금은 없지만 나중에 나올 결과"를 담는 영수증. `get()`으로 회수한다(준비될 때까지 블록).
- **`shutdown()`**: 새 작업은 거부하고 진행 중인 작업은 마저 끝낸다. `awaitTermination`으로 완료를 기다린다.

### 원리 — 왜 프레임워크가 이걸 쓰나

[`SPRING_개념서 §2.7`](../../../spring/docs/SPRING_개념서/)의 **톰캣 스레드 풀**이 바로 이것이다. 요청이 올 때마다 스레드를 새로
만들면 트래픽에 서버가 죽으니, 풀에서 워커를 빌려 요청을 처리하고 반납한다(thread-per-request). DB 커넥션도 같은
이유로 **커넥션 풀(HikariCP)**로 재사용한다 — 커넥션 생성이 비싸기 때문. [4장](04-동기화-락-cas-volatile.md)의
계수 세마포어(자원 N개 제한)와 커넥션 풀 크기가 같은 개념이라는 점도 다시 떠올려 보라.

풀 크기 선택은 실무에서 중요한 튜닝 포인트다. CPU 바운드 작업은 대략 코어 수 정도, I/O 바운드 작업(DB·네트워크
대기가 많음)은 그보다 크게 잡는다. 너무 크면 문맥 교환 오버헤드([1장](01-프로세스와-스레드.md))가 늘고, 너무
작으면 처리량이 떨어진다.

> ▶ 데모: `./run.sh ExecutorAndFutures` — 스레드 3개 풀로 작업 9개를 처리하며 **워커 이름이 3종류만 반복 등장**
> (=재사용)하는 것, `Future`로 결과를 회수하는 것을 보여준다. 대조로 `new Thread()`는 매번 이름이 달라(재사용
> 없음) 대량 트래픽에 부적합함을 나란히 보여준다.

```java
// ExecutorAndFutures 골격
ExecutorService pool = Executors.newFixedThreadPool(3);
List<Future<Integer>> futures = new ArrayList<>();
for (int n = 1; n <= 9; n++) {
    final int v = n;
    futures.add(pool.submit(() -> { /* 워커 이름 출력 */ return v * v; }));
}
for (Future<Integer> f : futures) sum += f.get();   // 순서대로 결과 회수
pool.shutdown();
```

## 7.2 `BlockingQueue`와 생산자-소비자

### 개념

데이터를 만드는 쪽(**생산자, producer**)과 쓰는 쪽(**소비자, consumer**)의 속도가 다를 때, 둘 사이에 **버퍼(큐)**를
둔다. 직접 락으로 큐를 관리하면 복잡하고 버그가 잦지만, **`BlockingQueue`**는 이 패턴 전용의 스레드 안전한 큐로
두 가지를 자동으로 해 준다.

- **스레드 안전(thread-safe)**: 여러 생산자/소비자가 동시에 써도 내부 락으로 안전.
- **블로킹(blocking)**:
  - 큐가 **가득 차면** `put()`하는 생산자가 자리가 날 때까지 **잠들어 대기**.
  - 큐가 **비면** `take()`하는 소비자가 데이터가 올 때까지 **잠들어 대기**.

여기서 "잠들어 대기"가 핵심이다. `while` 루프로 CPU를 태우는 **바쁜 대기(busy-wait)**가 아니라, 진짜로 스레드를
[2장](02-동시성과-병렬성-스케줄러.md)의 "대기(Blocked)" 상태로 재운다. 그래서 CPU를 낭비하지 않는다.

```
  생산자 ──put()──▶ [ □ □ □ ] (크기 5, 유한 버퍼) ──take()──▶ 소비자
           가득 차면 대기          ▲ 비면 대기
```

### 원리 — 유한 버퍼와 배압(back-pressure)

**유한(bounded) 버퍼**가 핵심이다. 생산자가 소비자보다 빠르면 버퍼가 차고, 그러면 생산자의 `put()`이 블록되어
**자동으로 느려진다** — 이것을 **배압(back-pressure)**이라 한다.

배압이 없으면(무한 버퍼면) 빠른 생산자가 데이터를 계속 쌓아 **메모리가 무한정 부풀다 OutOfMemoryError**로
죽는다. 유한 버퍼는 "소비자가 감당할 수 있는 만큼만 생산하게" 자연스럽게 속도를 맞춰 준다. 이 개념은 메시지 큐,
리액티브 스트림, 네트워크 흐름 제어([`CS_네트워크_개념서`](../CS_네트워크_개념서/)의 TCP 흐름 제어)까지 관통하는
중요한 아이디어다.

### 자바 코드

```java
BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(5);   // 크기 5 유한 버퍼

// 생산자
queue.put(item);          // 가득 차면 여기서 잠들어 대기

// 소비자
Integer item = queue.take();   // 비면 여기서 잠들어 대기
```

[`ProducerConsumer`](../../chapter-cs-concurrency/src/main/java/com/edu/concurrency/ProducerConsumer.java)는
크기 5 버퍼에 빠른 생산자와 느린 소비자를 두어, 버퍼가 자주 가득 차 생산자가 블록되는 것을 관찰한다. 소비자를
깔끔히 끝내기 위해 **독약(poison pill)** 신호(특별한 종료 값)를 큐에 넣어, 소비자가 그것을 받으면 스스로 종료한다.
그래서 프로그램이 절대 hang 하지 않는다.

```java
// ProducerConsumer 골격
static final int POISON_PILL = -1;
// 생산자: 다 만든 뒤 종료 신호를 넣는다
queue.put(POISON_PILL);
// 소비자: 종료 신호를 받으면 루프 탈출
Integer item = queue.take();
if (item == POISON_PILL) break;
```

> ▶ 데모: `./run.sh ProducerConsumer` — 크기 5 버퍼로, 빠른 생산자가 버퍼가 찰 때마다 블록되는 것과
> **만든 개수와 소비한 개수가 정확히 일치**(유실·중복 없음)하는 것을 보여준다.

## 7.3 `CompletableFuture` — "기다리지 말고, 다음 할 일을 예약하라"

### 개념

§7.1의 `Future`에는 아쉬움이 있다: 결과를 쓰려면 결국 `get()`으로 **스레드가 서서 기다려야** 한다.
`CompletableFuture`(Java 8+)는 발상을 뒤집는다 — **"결과가 나오면 이걸 해줘"라고 다음 단계를 예약**하고,
현재 스레드는 딴 일을 한다. 여러 비동기 작업을 레고처럼 **조합(composition)**할 수 있는 것이 핵심이다.

```
  Future 방식:              CompletableFuture 방식:
  결과 나올 때까지 get()에서   supplyAsync(작업)
  블록 ─── 기다림 ───▶ 사용     .thenApply(변환 예약)        ← 예약만 하고 바로 반환
                              .thenCombine(다른 작업, 합침 예약)
                              ...결과가 나오는 순간 예약된 단계들이 이어서 실행됨
```

### 원리 — 핵심 메서드 다섯 가지

| 메서드 | 하는 일 | 언제 |
|--------|---------|------|
| `supplyAsync(공급자)` | 백그라운드에서 값을 만드는 작업 시작 | 파이프라인의 출발점 |
| `thenApply(f)` | 결과를 **변환** (스트림의 `map` 같은 것) | 결과 가공 |
| `thenCompose(f)` | 결과로 **또 다른 비동기 호출**을 연결 (`flatMap`) | 의존적인 두 호출 (사용자ID → 주문조회) |
| `thenCombine(cf, f)` | **독립적인 두 결과**를 만나서 합침 | 무관한 두 호출 (날씨 + 환율) |
| `allOf(cf...)` | 여러 개가 **전부** 끝나기를 기다림 | N개 동시 호출 후 취합 |

예외는 파이프라인을 타고 흐르다가 `exceptionally`(실패 시에만, `catch` 같은 것) 또는
`handle`(성공/실패 모두 지나는 합류 지점)에서 복구한다.

### 원리 — 왜 배우나: 순차 900ms vs 병렬 300ms

외부 API 3개(각 300ms)를 **순차**로 부르면 900ms, **동시에 던지고 `allOf`로 취합**하면 가장 느린
하나의 시간(~300ms)만 든다. I/O 대기가 대부분인 백엔드에서 이 차이는 그대로 응답 시간 차이가 된다.
(Spring 의 `@Async`, WebClient 의 리액티브 조합이 전부 이 아이디어 위에 있다.)

```java
// 순차: 300 × 3 = 900ms
String a = callApiA(); String b = callApiB(); String c = callApiC();

// 병렬: max(300, 300, 300) = 300ms
CompletableFuture<String> fa = CompletableFuture.supplyAsync(() -> callApiA(), pool);
CompletableFuture<String> fb = CompletableFuture.supplyAsync(() -> callApiB(), pool);
CompletableFuture<String> fc = CompletableFuture.supplyAsync(() -> callApiC(), pool);
CompletableFuture.allOf(fa, fb, fc).get(5, TimeUnit.SECONDS);   // 셋 다 끝날 때까지만
```

> ▶ 데모: `./run.sh CompletableFutureDemo` — 외부 네트워크 없이 `Thread.sleep`으로 만든 가짜 API로
> ① supplyAsync→thenApply 파이프라인, ② thenCompose(의존) vs thenCombine(독립), ③ **순차 ~900ms vs
> 병렬 ~300ms 시간 증명**, ④ exceptionally/handle 예외 복구를 차례로 보여준다.

**예상 출력 해석.**

- 1부에서 `"(메인 스레드는 블록되지 않고 여기까지 먼저 도달했다)"`가 작업 시작 로그보다 먼저/직후에
  찍힌다 — `supplyAsync`가 예약만 하고 즉시 반환한다는 증거.
- 2부(b)에서 날씨(200ms)+환율(200ms)을 합쳤는데 소요가 **~200ms** — 동시에 진행됐다는 증거.
- 3부에서 순차 ~900ms vs 병렬 ~300ms 가 숫자로 찍힌다.
- 4부에서 예외가 나면 `thenApply` 단계는 **건너뛰고** `exceptionally`로 직행해 대체값으로 복구된다.

### 확인문제

1. `thenApply`와 `thenCompose`의 차이를 설명하고, "사용자 ID를 조회한 뒤 그 ID로 주문 목록 API를
   호출"하는 상황엔 어느 쪽이 맞는지 답하라.
   <details><summary>정답</summary>
   thenApply는 결과를 일반 값으로 변환한다(A → B). thenCompose는 결과를 받아 또 다른
   CompletableFuture를 시작하고 그것을 평평하게 이어준다(A → CF&lt;B&gt;). 주문 조회는 "또 다른 비동기
   호출"이므로 thenCompose가 맞다. thenApply를 쓰면 CF&lt;CF&lt;B&gt;&gt;로 이중 포장된다(스트림의 map vs
   flatMap 관계).</details>

2. 각 300ms 걸리는 독립적인 API 4개를 (a) 순차 호출, (b) `supplyAsync` 4개 + `allOf`로 호출하면
   각각 대략 몇 ms인가? (b)가 성립하기 위한 전제 조건 하나는?
   <details><summary>정답</summary>
   (a) 약 1,200ms(300×4). (b) 약 300ms(가장 느린 하나). 전제: 네 호출이 서로의 결과에 의존하지 않는
   독립 작업이어야 하고, 이를 동시에 실행할 스레드가 풀에 충분해야 한다(풀 크기가 1이면 도로
   순차가 된다).</details>

3. `exceptionally`와 `handle`의 차이를 try-catch-finally 에 비유해 설명하라.
   <details><summary>정답</summary>
   exceptionally는 예외가 났을 때만 호출되어 대체값을 돌려주는 catch 에 해당한다(성공 시 건너뜀).
   handle은 성공이든 실패든 항상 지나는 합류 지점으로 finally 에 가깝되, (결과, 예외) 두 인자 중
   null 이 아닌 쪽을 보고 반환값을 새로 정할 수 있다는 점이 다르다.</details>

## ⚠️ 흔한 오해와 함정

- **"풀을 shutdown 안 해도 된다"** → 아니다. 안 하면 워커 스레드가 살아 있어 JVM이 종료되지 않는다.
  다 쓴 풀은 `shutdown()` → `awaitTermination()` 순으로 정리한다.
- **"`Future.get()`은 즉시 결과를 준다"** → 아니다. 결과가 준비될 때까지 **블록**한다. 여러 작업을 `submit`한 뒤
  각각 `get()`하면 병렬로 진행되지만, `submit` → `get` → `submit` → `get`으로 하나씩 기다리면 사실상 순차 실행이 된다.
- **"무한 버퍼가 더 안전하다(안 막히니까)"** → 반대다. 무한 버퍼는 배압이 없어 빠른 생산자가 메모리를 폭발시킨다.
  **유한 버퍼가 배압으로 안정성을 준다.**
- **"BlockingQueue는 busy-wait로 기다린다"** → 아니다. 진짜로 스레드를 재운다(Blocked 상태). CPU를 태우지 않는다.
- **"스레드 풀 크기는 크면 클수록 좋다"** → 아니다. 코어 수를 크게 넘기면 문맥 교환 비용만 늘어 오히려 느려진다.
  CPU 바운드는 코어 수 근처, I/O 바운드는 그보다 크게가 대략의 기준.

## 연습문제

1. **(핵심)** `new Thread()`를 요청마다 만드는 것과 스레드 풀을 쓰는 것의 차이를, [1장](01-프로세스와-스레드.md)의
   스레드 생성 비용과 연결해 설명하라.
   <details><summary>힌트/해설</summary>
   new Thread는 매번 스택·커널 등록 비용을 낸다. 대량 트래픽에서 감당 불가. 풀은 워커를 재사용해 생성 비용을
   한 번만 내고, 작업만 큐로 넘긴다. 톰캣이 이 방식.</details>

2. **(개념)** `Future`란 무엇이며 `get()`의 동작을 설명하라. "영수증" 비유로.
   <details><summary>힌트/해설</summary>
   Future는 "나중에 나올 결과"의 영수증. submit하면 즉시 받고, get()하면 결과가 준비될 때까지 블록하다 값을 준다.
   여러 작업을 submit한 뒤 모아서 get하면 병렬성이 산다.</details>

3. **(응용)** 배압(back-pressure)이 무엇이며, 유한 버퍼가 어떻게 이를 만드는지 설명하라. 무한 버퍼의 위험은?
   <details><summary>힌트/해설</summary>
   배압 = 소비자가 못 따라오면 생산자가 자동으로 느려지는 것. 유한 버퍼가 가득 차면 put()이 블록되어 생산자가
   멈춰 속도가 맞춰진다. 무한 버퍼는 배압이 없어 메모리 폭발(OOM) 위험.</details>

4. **(함정)** 다음 코드가 사실상 순차 실행이 되는 이유는?
   ```java
   for (int i=0;i<9;i++){ Future<Integer> f = pool.submit(task); int r = f.get(); }
   ```
   <details><summary>힌트/해설</summary>
   submit 직후 바로 get으로 블록해 다음 작업을 제출하기 전에 결과를 기다리기 때문. 병렬성을 살리려면 먼저 9개를
   전부 submit해 futures에 모은 뒤, 그다음에 get으로 회수해야 한다.</details>

5. **(심화)** `BlockingQueue`의 "독약(poison pill)" 패턴이 필요한 이유와, 소비자가 여러 개일 때 어떻게 처리해야
   하는지 논하라.
   <details><summary>힌트/해설</summary>
   소비자는 큐가 비면 take()에서 영원히 잠들 수 있어, "이제 끝"을 알려 줄 신호가 필요하다. 소비자가 N개면
   독약도 N개 넣거나, 독약을 받은 소비자가 다시 독약을 큐에 넣어 전파하게 해야 모두 종료된다.</details>

## 요약

- **스레드 풀(`ExecutorService`)**은 워커를 미리 만들어 재사용한다. 우리는 스레드가 아니라 **작업**만 큐로 던진다.
  톰캣 스레드 풀·HikariCP 커넥션 풀이 이 개념. 다 쓰면 `shutdown()`.
- **`Future`**는 "나중에 나올 결과"의 영수증. `get()`은 준비될 때까지 블록한다. 먼저 다 submit한 뒤 모아서 get해야
  병렬성이 산다.
- **`BlockingQueue`**는 스레드 안전 + 블로킹(가득 차면 put 대기, 비면 take 대기)으로 생산자-소비자를 안전하게
  구현한다. 바쁜 대기가 아니라 진짜로 재운다.
- **유한 버퍼**는 **배압(back-pressure)**을 만들어 빠른 생산자를 자동으로 늦추고 메모리 폭발을 막는다.
  이 개념은 메시지 큐·리액티브·TCP 흐름 제어까지 관통한다.
- **`CompletableFuture`**는 "결과가 나오면 할 일"을 예약하는 비동기 조합 도구다. 변환은 `thenApply`,
  의존 호출 연결은 `thenCompose`, 독립 결과 합침은 `thenCombine`, 전체 대기는 `allOf`, 예외 복구는
  `exceptionally`/`handle`. 독립적인 I/O는 동시에 던져야 한다(순차 900ms → 병렬 300ms).
