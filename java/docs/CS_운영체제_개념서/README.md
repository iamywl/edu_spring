## 이 책에 대하여

당신은 이미 [`JAVA_개념서`](../JAVA_교재/심화/)에서 **"프로그램 하나가 어떻게 실행되는가"**를 배웠다.
스택, 힙, 바이트코드, JIT, GC — 전부 **스레드 한 개짜리** 이야기였다.

그런데 실무의 서버는 그렇지 않다. [`SPRING_개념서 §2.7 "서버는 멀티스레드다"`](../../../spring/docs/SPRING_개념서/)에서 봤듯,
톰캣은 요청마다 워커 스레드를 하나씩 붙여 **수백 개의 스레드가 동시에** 같은 코드를,
심지어 같은 객체를 실행한다. 바로 여기서 세상에서 가장 재현하기 어려운 버그들이 태어난다.

- 평소엔 멀쩡하다가 트래픽이 몰릴 때만 남의 데이터가 튀어나온다.
- 숫자를 100번 더했는데 결과가 73이다.
- 서버가 아무 에러 없이 그냥 "멈춘다".

이 책은 **"왜 이런 일이 벌어지는가, 그리고 무엇으로 막는가"**를 그림과 실행되는 코드로 설명한다.
외우는 게 아니라 **눈으로 보고 이해**하게 하는 것이 목표다.

## 이 책의 학습 목표

이 책을 끝까지 읽고 데모를 직접 돌려 보면 다음을 스스로 설명할 수 있게 된다.

1. 프로세스와 스레드가 **무엇을 공유하고 무엇을 따로 갖는지**, 그리고 그것이 왜 모든 동시성 문제의 뿌리인지.
2. `count++`가 **왜 원자적이지 않은지** 바이트코드 3단계로.
3. **원자성(atomicity)과 가시성(visibility)**의 차이 — 이 둘을 구분하는 것이 전공자 수준의 핵심이다.
4. `synchronized` / `AtomicXxx` / `ReentrantLock` / `volatile`이 **각각 무엇을 고치고 무엇을 못 고치는지**.
5. 뮤텍스·세마포어·모니터의 관계와, 자바 메모리 모델(JMM)의 **happens-before** 규칙.
6. 교착 상태(deadlock)의 **코프만 4조건**과 예방·회피·탐지·복구의 구분.
7. 실무의 고수준 도구(스레드 풀, `Future`, `BlockingQueue`)가 왜 이렇게 생겼는지.
8. 가상 메모리·페이징이 "프로세스 격리"의 마법을 어떻게 만드는지, 그리고 그것이 JVM 힙과 어떻게 이어지는지.

## 추천 읽기 순서

**처음 읽는다면** 1장부터 순서대로 읽어라. 이 책은 앞 장이 뒤 장의 근거가 되도록 **하나의 이야기**로 엮여 있다.
1→2에서 "왜 순서가 예측 불가능한가"를 이해하고, 3에서 그로 인한 버그를 눈으로 보고, 4·5에서 고치는 도구를,
6에서 그 도구가 만드는 새 문제를, 7에서 실무의 포장을 본다. 8은 곁다리(메모리 관리)이니 나중에 읽어도 좋다.

**이미 동시성을 좀 안다면** 4장(원자성 vs 가시성)과 5장(JMM/happens-before), 6장(예방·회피·탐지 구분)만
골라 읽어도 밀도가 높다.

## 실행 데모 안내

각 장은 실제로 **실행되는 코드**와 연결된다. `java/` 디렉토리(컨테이너 안에서는 `/app`)에서 다음처럼 돌린다.

```bash
# java/ 디렉토리에서 실행
./run.sh RaceConditionDemo        # 3장 - 경쟁 상태(잃어버린 갱신, 비결정성)
./run.sh SynchronizationDemo      # 4장 - 동기화 3종 + volatile 반례
./run.sh AtomicCasDemo            # 4장 §4.6 - CAS 재시도 루프 직접 구현, LongAdder 비교
./run.sh ReentrantLockDemo        # 4장 §4.6 - tryLock 교착 회피·공정성·Condition·재진입
./run.sh VolatileVisibilityDemo   # 5장 §5.3 - 가시성 문제 재현과 happens-before
./run.sh DeadlockDemo             # 6장 - 교착 유발/해결
./run.sh ExecutorAndFutures       # 7장 - 스레드 풀과 Future
./run.sh ProducerConsumer         # 7장 - 생산자-소비자(BlockingQueue)
./run.sh CompletableFutureDemo    # 7장 §7.3 - 비동기 조합(순차 vs 병렬 시간 증명)
```

인자 없이 `./run.sh` 만 치면 대화형 메뉴가 뜬다. 각 데모는 **출력이 개념을 증명**하도록 짜여 있으니,
장을 읽은 뒤 반드시 직접 돌려 보고 결과를 눈으로 확인하기를 권한다. 특히 `RaceConditionDemo`는
**여러 번 돌릴 때마다 결과가 다르게** 나오는 것이 관찰 포인트다.
