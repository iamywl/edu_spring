# Chapter 06. 클래스

프로그램이 커질수록 "데이터"와 "그 데이터를 다루는 기능"이 여기저기 흩어지면 관리가 어려워진다. 자바는 이 둘을 하나로 묶는 도구로 **클래스(class)** 를 제공한다. 클래스는 새로운 데이터 타입을 정의하는 설계도이고, 그 설계도로 찍어낸 실체가 **객체(object)** 다. 이 장에서는 클래스를 선언하고, 객체를 생성하고, 필드·생성자·메서드라는 세 가지 구성 멤버를 다루는 법을 배운다. 나아가 인스턴스와 정적 멤버의 차이, final과 상수, 패키지와 접근 제한자를 통한 캡슐화, 그리고 이를 실전에서 활용하는 Getter/Setter와 싱글톤 패턴까지 다룬다.

객체지향의 큰 그림(상속·다형성·SOLID 등)은 『JAVA 개념서 4장. 객체지향이란 무엇인가?』에서 개념적으로 정리한다. 이 장은 그 개념을 **문법과 코드로 실습**하는 데 초점을 둔다.

## 6.1 객체지향 프로그래밍(OOP)

**객체지향 프로그래밍(OOP)** 은 현실 세계의 사물을 소프트웨어의 "객체"로 모델링하는 방식이다. 데이터(필드)와 그 데이터를 다루는 기능(메서드)을 하나의 객체로 묶어서 관리한다.

- **절차지향**: 데이터와 함수가 분리되어 있고, 함수를 순서대로 호출해 처리한다.
- **객체지향**: 데이터와 함수를 객체 안에 함께 넣어, 객체가 스스로 일을 처리한다.

절차지향은 데이터를 함수에 "넘겨서" 처리하지만, 객체지향은 객체가 자기 데이터를 품고 "스스로" 처리한다.

```java
// 절차지향: 데이터와 함수가 분리
int total = calcTotal(prices);       // prices를 함수에 넘김

// 객체지향: 데이터와 기능이 Cart 안에 함께
Cart cart = new Cart(prices);
int total = cart.calcTotal();        // 객체가 스스로 계산
```

OOP의 4대 특징은 **캡슐화, 상속, 다형성, 추상화** 다. 이 중 캡슐화는 이 장 전반(접근 제한자, Getter/Setter)에서 실습하고, 나머지 세 가지는 개념서 4장에서 다룬다.

## 6.2 객체와 클래스

- **클래스(class)**: 객체를 만들기 위한 "설계도". 어떤 필드와 메서드를 가질지 정의한다.
- **객체(object)**: 설계도로 실제로 만들어진 "실체". **인스턴스(instance)** 라고도 한다.

비유하자면 클래스는 "붕어빵 틀", 객체는 그 틀로 찍어낸 "붕어빵" 하나하나다. 하나의 클래스로 여러 객체를 만들 수 있고, 각 객체는 서로 **독립된 상태** 를 가진다.

```java
Car sonata = new Car();   // 붕어빵 1개
Car avante = new Car();   // 또 다른 붕어빵 1개
sonata.accelerate(50);    // sonata의 speed만 바뀜
avante.accelerate(30);    // avante의 speed만 바뀜 (서로 무관)
```

단, 두 참조 변수가 **같은 객체** 를 가리키면 상태를 공유한다. `Car ref = sonata;` 처럼 대입하면 새 객체가 생기는 게 아니라 같은 객체를 함께 가리킨다(→ 6.4에서 자세히).

## 6.3 클래스 선언

클래스는 `class` 키워드로 선언한다.

```java
[접근제한자] class 클래스이름 {
    // 필드; 생성자; 메서드;
}
```

규칙과 관례:

- 클래스 이름은 대문자로 시작하는 **파스칼 표기법(PascalCase)** 을 쓴다. 예: `Student`, `ClassDeclaration`
- 하나의 `.java` 파일에 `public` 클래스는 최대 1개이며, 파일명과 이름이 같아야 한다.
- 멤버가 하나도 없는 빈 클래스도 성립하며, 객체로 만들 수 있다.

```java
class Empty { }                    // 빈 클래스도 유효

class Student {
    String name;                   // 필드
    int grade;
    String intro() {               // 메서드
        return name + "(" + grade + "학년)";
    }
}
```

클래스를 선언한다는 것은 곧 프로그램에 **새로운 데이터 타입** 을 정의하는 일이다. `Student s`에서 `Student`는 우리가 만든 타입이다.

## 6.4 객체 생성과 클래스 변수(참조 변수)

객체는 **`new` 연산자** 로 생성한다.

```java
클래스이름 변수 = new 클래스이름();
```

`new`는 힙(heap) 메모리에 객체를 만들고, 그 객체의 **주소(참조)** 를 돌려준다. 왼쪽의 변수(참조 변수)는 값 자체가 아니라 객체의 "주소"를 담는다. 그래서 참조 변수를 대입하면 **객체가 복사되는 게 아니라 주소가 복사** 된다.

```java
Point p1 = new Point();
p1.x = 10;
Point p2 = p1;      // 주소 복사 → 같은 객체를 가리킴
p2.x = 99;          // p2로 바꿔도
System.out.println(p1.x);      // 99 (같은 객체이므로 함께 바뀜)
System.out.println(p1 == p2);  // true

Point p3 = new Point();        // new는 매번 새 객체
System.out.println(p1 == p3);  // false
```

아직 아무 객체도 가리키지 않는 참조 변수는 `null` 이다. `null` 상태에서 멤버에 접근하면 `NullPointerException`(NPE) 이 발생한다. 참조와 값의 차이는 개념서 2장(변수/메모리)과도 연결된다.

## 6.5 클래스의 구성 멤버

클래스는 크게 세 종류의 멤버로 구성된다.

1. **필드(field)**: 객체의 데이터(상태)를 저장한다. → 6.6
2. **생성자(constructor)**: 객체가 생성될 때 초기화를 담당한다. 클래스 이름과 같다. → 6.7
3. **메서드(method)**: 객체의 기능(행위)을 정의한다. → 6.8

```java
class Account {
    String owner;   int balance;                 // 1) 필드
    Account(String owner, int balance) {         // 2) 생성자
        this.owner = owner;  this.balance = balance;
    }
    void deposit(int amount) { balance += amount; }   // 3) 메서드
}
```

세 멤버가 협력하여 "상태를 가지고 행동하는" 객체가 완성된다. 필드는 데이터, 생성자는 초기화, 메서드는 행위를 담당한다.

> 💻 실습: `./run.sh ClassMembers`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch06/ClassMembers.java`

## 6.6 필드 선언과 사용

**필드** 는 객체의 데이터(상태)를 저장하는 변수로, 클래스 블록 안에 선언한다.

```java
[접근제한자] 타입 필드이름 [= 초기값];
```

지역 변수와 달리, 필드는 초기값을 주지 않아도 **타입별 기본값** 으로 자동 초기화된다.

| 타입 | 기본값 |
|---|---|
| 정수형(int 등) | `0` |
| 실수형(double 등) | `0.0` |
| boolean | `false` |
| char | `''` (숫자 0) |
| 참조형(String 등) | `null` |

필드는 **선언과 동시에** 초기화하거나 **생성자에서** 초기화할 수 있다.

```java
class Product {
    String name = "무명";   // 선언 시 초기화
    int price;              // 생성자에서 초기화
    Product(int price) { this.price = price; }
}
```

## 6.7 생성자 선언과 호출 (오버로딩 · this())

**생성자** 는 `new`로 객체를 만들 때 호출되어 필드를 초기화하는 특별한 멤버다.

- 이름이 **클래스 이름과 같고, 리턴 타입이 없다**(void도 아님).
- 생성자를 하나도 선언하지 않으면 컴파일러가 **기본 생성자** 를 자동으로 넣어준다.
- 매개변수를 달리하여 여러 개 만들 수 있다 → **생성자 오버로딩**.
- **`this()`**: 같은 클래스의 다른 생성자를 호출한다(반드시 생성자 첫 줄에서만).
- **`this`**: 현재 객체 자신의 참조. 필드와 매개변수 이름이 겹칠 때 구분에 쓴다.

```java
class Book {
    String title;  int price;
    Book()              { this("제목없음", 0); }     // this()로 위임
    Book(String title)  { this(title, 10000); }       // 오버로딩 + 위임
    Book(String title, int price) {
        this.title = title;    // this.title(필드) = title(매개변수)
        this.price = price;
    }
}
```

생성자는 "완전히 초기화된 객체" 만 세상에 나오도록 보장하는 장치다. 생성자와 `this`의 개념적 배경은 개념서 4.2에서도 다룬다.

## 6.8 메소드 선언과 호출

**메서드** 는 객체의 기능(행위)을 정의한 코드 블록이다.

```java
[접근제한자] 리턴타입 메서드이름(매개변수목록) {
    실행문;
    return 값;   // 리턴타입이 void면 생략
}
```

- **매개변수(parameter)**: 메서드에 전달되는 입력값.
- **리턴값(return)**: 메서드가 돌려주는 결과. 리턴 타입이 `void`면 반환값이 없다.
- **오버로딩(overloading)**: 이름이 같고 매개변수(개수/타입)가 다른 메서드를 여러 개 두는 것.
- **가변인자(varargs)**: `타입... 이름` 형태로, 개수가 정해지지 않은 인자를 배열처럼 받는다.

```java
class Calculator {
    int    add(int a, int b)           { return a + b; }
    double add(double a, double b)     { return a + b; }   // 오버로딩(타입)
    int    add(int a, int b, int c)    { return a+b+c; }   // 오버로딩(개수)
    int    sum(int... nums) {                              // 가변인자
        int total = 0;
        for (int n : nums) total += n;
        return total;
    }
}
```

메서드로 로직에 이름을 붙여 재사용하고, 오버로딩과 가변인자로 유연성을 얻는다. (오버로딩과 오버라이딩의 구분은 개념서 4.5 참고.)

## 6.9 인스턴스 멤버

**인스턴스 멤버** 는 객체(인스턴스)마다 따로 존재하는 필드·메서드다(`static`이 붙지 않은 멤버).

- **인스턴스 필드**: 객체를 `new` 할 때마다 새로 만들어져, 객체마다 독립된 값을 가진다.
- **인스턴스 메서드**: 반드시 객체를 통해 호출한다(`객체.메서드()`).

`this` 키워드는 "이 메서드를 호출한 바로 그 객체 자신"을 가리킨다. 따라서 인스턴스 메서드 안의 `this.필드` 는 각 객체의 자기 필드를 의미한다.

```java
class Counter {
    int count;                              // 인스턴스 필드: 객체마다 별도
    void increase() { this.count++; }       // this = 호출한 객체
    Counter reset() { this.count = 0; return this; }   // this 리턴 → 체이닝
}

Counter a = new Counter(), b = new Counter();
a.increase(); a.increase();   // a.count = 2
b.increase();                 // b.count = 1 (a와 무관)
b.reset().increase();         // 메서드 체이닝
```

`this` 덕분에 하나의 메서드 코드가 각 객체에 맞게 동작한다.

> 💻 실습: `./run.sh InstanceMember`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch06/InstanceMember.java`

## 6.10 정적 멤버

`static`이 붙은 멤버(정적 필드/메서드)는 객체가 아니라 **클래스** 에 소속된다.

- **정적 필드**: 모든 객체가 **공유** 하는 하나의 값. `클래스이름.필드` 로 접근.
- **정적 메서드**: 객체 없이 `클래스이름.메서드()` 로 호출. **인스턴스 멤버와 `this`는 사용 불가**.
- **정적 블록(`static { }`)**: 클래스가 메모리에 로딩될 때 딱 한 번 실행되어 정적 필드를 초기화.

정적 멤버는 객체를 생성하기 전에도 사용할 수 있다.

```java
class Company {
    static String name;             // 공유 필드
    static int employeeCount;       // 공유 필드
    String empName;                 // 인스턴스 필드 (객체마다 다름)
    static { name = "에듀컴퍼니"; }   // 로딩 시 1회 실행
    Company(String empName) { this.empName = empName; employeeCount++; }
    static int getEmployeeCount() { return employeeCount; }
}
```

공유 상태나 유틸 기능처럼 "객체와 무관한, 클래스 전체의 공통 데이터/기능"에는 정적 멤버가 자연스럽다. static과 instance의 개념적 대비는 개념서 4.8에서 다룬다.

> 💻 실습: `./run.sh StaticMember`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch06/StaticMember.java`

## 6.11 final 필드와 상수

- **final 필드**: 한 번 값이 정해지면 다시 바꿀 수 없는 필드. **반드시 초기화** 해야 한다(선언 시 또는 생성자에서). 객체마다 값이 다를 수 있다.
- **상수(constant)**: `static final` 로 선언한, 값이 고정된 공유 값. 관례상 이름을 **대문자와 밑줄(`_`)** 로 쓴다. 예: `MAX_SPEED`, `PI`

```java
class Circle {
    static final double PI = 3.14159;   // 상수 (모든 원이 공유)
    final int radius;                   // final 인스턴스 필드
    Circle(int radius) { this.radius = radius; }   // 생성자에서 한 번만 초기화
    double area() { return PI * radius * radius; }
}
// c1.radius = 7;  → 컴파일 오류 (final은 재할당 불가)
```

지역 변수에도 `final`을 붙일 수 있다(`final int MAX = 100;`). `final`은 "변하면 안 되는 값"을 코드로 못 박아, 매직 넘버를 의미 있는 이름으로 바꾸고 실수로 인한 변경을 막아 안정성과 가독성을 높인다.

## 6.12 패키지

**패키지(package)** 는 관련 있는 클래스들을 묶는 "폴더" 같은 개념이다.

- 파일 맨 위의 `package com.edu.javabook.ch06;` 가 그 클래스의 소속 패키지다.
- 패키지는 **디렉토리 구조와 일치** 한다(`com/edu/javabook/ch06/`).
- 이름 충돌을 막고(같은 이름 클래스도 패키지가 다르면 공존), 접근 범위를 나눈다.
- **완전한 이름(FQCN)** = 패키지명 + 클래스명. 예: `com.edu.javabook.ch06.PackageDemo`

```java
package com.edu.javabook.ch06;   // 소속 패키지 선언 (파일 최상단)

PackageDemo.class.getPackageName();  // "com.edu.javabook.ch06"
PackageDemo.class.getName();         // FQCN
PackageDemo.class.getSimpleName();   // "PackageDemo" (단순 이름)
```

다른 패키지의 클래스는 `import` 로 불러오거나 완전한 이름으로 사용한다. 단, `java.lang` 패키지(예: `String`, `Object`)는 `import` 없이 자동으로 쓸 수 있다. 패키지는 대규모 코드를 논리적으로 조직화하는 기본 단위다.

## 6.13 접근 제한자

**접근 제한자(access modifier)** 는 멤버(또는 클래스)에 대한 "접근 허용 범위"를 정한다.

| 제한자 | 접근 가능 범위 |
|---|---|
| `public` | 어디서나 (모든 패키지) |
| `protected` | 같은 패키지 + 다른 패키지의 자식 클래스 |
| `default`(키워드 없음) | 같은 패키지 안에서만 (package-private) |
| `private` | 해당 클래스 내부에서만 |

접근 범위(좁음 → 넓음): **`private` < `default` < `protected` < `public`**

```java
class Sample {
    public    int pub = 1;   // 어디서나
    protected int pro = 2;   // 같은 패키지 + 자식
    int def = 3;             // default: 같은 패키지
    private   int pri = 4;   // 자기 클래스 안에서만
    public int getPri() { return pri; }   // private는 메서드로 열어줌
}
```

캡슐화를 위해 필드는 보통 `private`로 숨기고, 필요한 것만 `public` 메서드로 연다. 최소한만 공개하면 결합도가 낮아지고 유지보수가 쉬워진다. 캡슐화의 "왜?"는 개념서 4.3에서 다룬다.

> 💻 실습: `./run.sh AccessModifier`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch06/AccessModifier.java`

## 6.14 Getter와 Setter

**캡슐화의 실천 방법**: 필드를 `private`로 숨기고, 값을 읽고 쓰는 통로로 메서드를 제공한다.

- **Getter**: 필드 값을 읽는 메서드. 관례상 `getXxx()` (boolean은 `isXxx()`).
- **Setter**: 필드 값을 바꾸는 메서드. 관례상 `setXxx(값)`.

Setter에서 **유효성 검증** 을 하면 잘못된 값이 들어오는 것을 막을 수 있다. 필드를 `public`으로 열어두면 아무 값이나 들어갈 수 있어 위험하다.

```java
class Person {
    private String name;
    private int age;
    public String getName() { return name; }      // Getter
    public int    getAge()  { return age; }
    public void setAge(int age) {                  // Setter + 유효성 검증
        if (age < 0 || age > 150) return;          // 잘못된 값 거부
        this.age = age;
    }
}
```

Getter/Setter로 접근을 통제하면 객체가 항상 "올바른 상태"를 유지한다. 필드를 감추고 메서드로만 접근시켜 잘못된 데이터를 원천 차단하는 것이다.

> 💻 실습: `./run.sh GetterSetter`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch06/GetterSetter.java`

## 6.15 싱글톤 패턴

**싱글톤(Singleton)** 은 클래스의 인스턴스를 **단 하나만** 생성해 공유하는 디자인 패턴이다.

만드는 방법:

1. 자기 자신 타입의 정적 필드 하나를 `private`로 둔다.
2. 생성자를 `private`로 막아 외부에서 `new` 를 못 하게 한다.
3. 유일한 인스턴스를 돌려주는 `public static` 메서드(`getInstance`)를 제공한다.

```java
class Settings {
    private static final Settings INSTANCE = new Settings();  // 1) 유일 인스턴스
    private Settings() { }                                    // 2) new 차단
    public static Settings getInstance() { return INSTANCE; } // 3) 반환 통로
}

Settings s1 = Settings.getInstance();
Settings s2 = Settings.getInstance();
System.out.println(s1 == s2);   // true (항상 같은 객체)
```

설정 정보, 로그 관리자처럼 "전체에서 하나만 있으면 되는" 자원에 적합하다. 인스턴스를 하나로 강제해 상태 공유와 자원 낭비 방지를 동시에 얻는다. 이 절은 앞서 배운 `static`, `final`, `private 생성자`, 접근 제한자가 모두 결합된 종합 예제다.

> 💻 실습: `./run.sh SingletonPattern`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch06/SingletonPattern.java`

## ⚠️ 흔한 오해와 함정

- **"참조 변수를 대입하면 객체가 복사된다"** — 아니다. 대입되는 것은 객체의 **주소** 뿐이다. `p2 = p1` 이후 `p2`로 값을 바꾸면 `p1`이 보는 객체도 함께 바뀐다. 진짜 복사가 필요하면 별도의 복사 로직이나 `new`가 필요하다.
- **"필드는 초기화하지 않으면 쓰레기 값이 들어간다"** — 아니다. 필드는 타입별 기본값(0, 0.0, false, null 등)으로 자동 초기화된다. 단, **지역 변수** 는 자동 초기화되지 않아 반드시 직접 초기화해야 한다.
- **"생성자에도 리턴 타입을 쓴다"** — 생성자는 리턴 타입이 없다. `void`조차 붙이지 않는다. `void Book() { }` 처럼 쓰면 그것은 생성자가 아니라 그냥 메서드가 되어버린다.
- **"정적 메서드 안에서 인스턴스 필드나 `this`를 쓸 수 있다"** — 쓸 수 없다. `static` 메서드는 특정 객체에 묶여 있지 않으므로 `this`가 존재하지 않고, 인스턴스 멤버에 직접 접근할 수 없다.
- **"`null` 참조의 메서드/필드에 접근해도 괜찮다"** — `NullPointerException`이 발생한다. 객체를 사용하기 전에 `new`로 생성했는지, 혹은 `null`인지 확인해야 한다.
- **"필드를 `public`으로 열고 Setter도 만들면 편하다"** — 필드가 `public`이면 Setter의 유효성 검증을 우회해 아무 값이나 직접 넣을 수 있다. 캡슐화를 원한다면 필드는 반드시 `private`로 두어야 한다.

### 과제 1. 클래스와 객체 — `Car` 클래스 (쉬움)

자동차를 표현하는 **`Car` 클래스를 설계·작성하라.** 이 과제의 핵심은 **하나의 클래스로 만든 여러 객체가 서로 독립된 상태를 가진다**는 것을 확인하는 것이다.

### 과제 5. 인스턴스 vs static — `Company`(직원) 클래스 (보통→응용)

회사에 소속된 직원을 표현하는 클래스를 통해 **인스턴스 멤버와 정적 멤버의 차이를 드러내도록 설계·작성하라.**

- **정적(static) 필드**: 회사 이름(모든 직원이 공유), 전체 직원 수(`employeeCount`)를 둔다.
- **인스턴스 필드**: 직원 이름(객체마다 다름)을 둔다.
- 생성자에서 직원 이름을 초기화하고, 객체가 만들어질 때마다 정적 필드 `employeeCount` 를 1씩 증가시켜라.
- 전체 직원 수를 돌려주는 **정적 메서드** `getEmployeeCount()` 를 만들어라.
- `main`에서 직원 객체를 여러 개 만든 뒤, 공유되는 값(회사 이름·직원 수)과 객체마다 다른 값(직원 이름)이 어떻게 구분되는지 출력하라.

> 힌트: `static` 필드는 객체가 아니라 **클래스에 하나만** 존재해 모든 객체가 공유한다. `static` 메서드 안에서는 `this` 나 인스턴스 필드를 쓸 수 없다(특정 객체에 묶여 있지 않으므로). 정적 필드는 `클래스이름.필드` 로 접근한다.

### 과제 7. final과 상수 — `Circle` 클래스 (보통)

원을 표현하는 **`Circle` 클래스를 설계·작성하라.** 이 과제의 핵심은 **`final` 필드**와 **`static final` 상수**의 차이를 드러내는 것이다.

- **상수(`static final`)**: 원주율 `PI = 3.14159` 를 상수로 선언하라. 관례상 이름은 **대문자**로 쓴다.
- **`final` 인스턴스 필드**: 반지름(`radius`, int)을 `final` 로 선언하고 **생성자에서 한 번만** 초기화하라.
- 메서드: 넓이를 돌려주는 `area()` (= `PI * radius * radius`).
- `main`에서 반지름이 다른 원 객체를 두 개 만들어 각각의 넓이를 출력하라.
- (실험) 코드 어딘가에 `c1.radius = 7;` 또는 `Circle.PI = 3.14;` 를 **주석으로** 적어두고, 왜 이 줄이 컴파일 오류인지 한 줄 주석으로 설명해보라.

> 힌트: `final` 필드는 **반드시 초기화**해야 하며(선언 시 또는 생성자에서) 한 번 정해지면 **재할당 불가**다. `radius` 는 객체마다 다르지만(인스턴스), `PI` 는 모든 원이 공유하는 고정값(`static final`)이다. 상수는 매직 넘버(3.14159)를 의미 있는 이름으로 바꿔 가독성을 높인다.

### 과제 8. 싱글톤 패턴 — `Settings` 클래스 (응용·종합)

애플리케이션 전체에서 **단 하나만** 존재해야 하는 설정 객체 **`Settings` 클래스를 싱글톤 패턴으로 설계·작성하라.**

- 자기 자신 타입의 **`private static` 필드**에 유일한 인스턴스를 하나 만들어 둔다.
- 생성자를 **`private`** 로 막아 외부에서 `new Settings()` 를 못 하게 하라.
- 유일한 인스턴스를 돌려주는 **`public static getInstance()`** 메서드를 제공하라.
- (선택) 설정값 하나(예: 테마 이름)를 필드로 두고 Getter/Setter로 접근하게 해보라.
- `main`에서 `getInstance()` 를 두 번 호출해 받은 두 참조가 **`==` 비교에서 `true`**(같은 객체)임을 확인하라.

> 힌트: 이 과제는 이 장에서 배운 `static`, `final`, `private 생성자`, 접근 제한자가 모두 결합된 종합 문제다. 인스턴스를 `private static final INSTANCE = new Settings();` 로 두면 클래스 로딩 시 한 번만 생성된다. 생성자가 `private` 라 외부에서는 오직 `getInstance()` 로만 객체를 얻을 수 있다.

> 답안 비교: `./run.sh SingletonPattern`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch06/SingletonPattern.java`

**[개념 서술]**

**1.** 클래스와 객체(인스턴스)의 관계를 "붕어빵" 비유로 설명하고, 하나의 클래스로 만든 두 객체가 서로 독립된 상태를 가진다는 말의 의미를 서술하시오.

<details><summary>정답·해설</summary>

- **클래스 = 붕어빵 틀(설계도)**, **객체 = 그 틀로 찍어낸 붕어빵 하나하나(실체·인스턴스)** 다. 클래스는 어떤 필드·메서드를 가질지 정의만 하고, 실제 데이터를 담는 것은 `new`로 만든 객체다.
- 하나의 클래스로 여러 객체를 만들 수 있고, 각 인스턴스 필드는 객체마다 **별도의 메모리**에 존재한다. 따라서 `a.speed`를 바꿔도 `b.speed`는 영향을 받지 않는다 — 이것이 "서로 독립된 상태"의 의미다.
- 단, 두 참조 변수가 **같은 객체**를 가리키면(`b = a`) 상태를 공유한다(2번 문제 참고).

</details>

---

**2.** `static` 필드(정적 필드)와 인스턴스 필드의 차이를 "공유"라는 키워드로 설명하고, `static` 메서드 안에서 `this`를 사용할 수 없는 이유를 서술하시오.

<details><summary>정답·해설</summary>

- **인스턴스 필드**는 객체를 `new` 할 때마다 새로 만들어져 **객체마다 별도의 값**을 가진다. **정적 필드**는 클래스에 **단 하나만** 존재해 **모든 객체가 공유**한다. 한 객체가 정적 필드를 바꾸면 다른 모든 객체·클래스 전체에 그 변경이 반영된다.
- `static` 메서드는 특정 객체가 아니라 **클래스에 소속**되어 객체 없이도 호출된다. `this`는 "이 메서드를 호출한 그 객체 자신"을 뜻하는데, 정적 메서드에는 그런 "그 객체"가 존재하지 않으므로 `this`가 성립하지 않는다. 같은 이유로 정적 메서드에서 인스턴스 필드/메서드에 직접 접근할 수 없다.

</details>

---

**3.** 필드를 `private`로 숨기고 Getter/Setter로만 접근하게 하는 이유(캡슐화)를 설명하고, Setter에 유효성 검증을 넣었을 때 얻는 이점을 예를 들어 서술하시오.

<details><summary>정답·해설</summary>

- **캡슐화**: 필드를 `private`로 감추면 외부에서 필드에 **직접** 접근·변경할 수 없다. 값을 읽고 쓰는 통로를 `public` 메서드(Getter/Setter)로만 열어 접근을 **통제**할 수 있다. 내부 구현이 바뀌어도 외부에 영향이 적어(낮은 결합도) 유지보수가 쉬워진다.
- **유효성 검증의 이점**: Setter 안에서 값을 검사하면 잘못된 값이 필드에 들어오는 것을 원천 차단해 객체가 항상 "올바른 상태"를 유지한다. 예: `setAge(int age)`에서 `age < 0 || age > 150`이면 거부 → 나이 필드가 절대 음수나 비현실적 값이 되지 않는다. 만약 필드가 `public`이면 이 검증을 우회해 아무 값이나 들어갈 수 있다.

</details>

---

**4.** **캡슐화(encapsulation)** 를 정의하고, 접근 제한자 `private` `default` `protected` `public`을 접근 범위가 **좁은 것부터 넓은 순서로** 나열한 뒤 각각의 접근 가능 범위를 한 줄로 설명하시오.

<details><summary>정답·해설</summary>

- **캡슐화**: 데이터(필드)와 그 데이터를 다루는 기능(메서드)을 하나의 객체로 묶고, 내부 구현을 숨긴 채 필요한 부분만 외부에 공개하는 것.
- 범위 좁음 → 넓음: **`private` < `default` < `protected` < `public`**
  - `private`: 해당 클래스 내부에서만
  - `default`(키워드 없음): 같은 패키지 안에서만(package-private)
  - `protected`: 같은 패키지 + 다른 패키지의 자식 클래스
  - `public`: 어디서나(모든 패키지)

</details>

**12.** 다음 코드에서 잘못된 부분을 모두 찾고 이유를 설명하시오.

class Circle {
    static final double PI = 3.14159;
    final int radius;                 // (a)
    Circle(int radius) { }            // (b)

- **(a)** 자체는 정상(선언은 유효). 다만 (b)에서 초기화하지 않아 문제가 된다.

**14.** 다음은 캡슐화된 `Person` 클래스다. Getter와 검증 있는 Setter를 완성하도록 빈칸 (가)~(다)를 채우시오.
```java
class Person {
    private int age;
    public int ___(가)___() { return age; }        // Getter
    public void setAge(int age) {
        if (___(나)___) return;                     // 0~150 벗어나면 거부
        ___(다)___ = age;                           // 필드에 대입
    }
}
```

<details><summary>정답·해설</summary>

- **(가)** `getAge` — 필드 값을 읽는 Getter 관례 이름(`getXxx`).
- **(나)** `age < 0 || age > 150` — 유효 범위(0~150)를 벗어나면 거부.
- **(다)** `this.age` — 매개변수 `age`와 필드 `age`가 겹치므로 `this.`로 필드를 지목.
- (참고) boolean 필드의 Getter는 관례상 `isXxx()`를 쓴다.

</details>

**[참·거짓 + 이유]**

**15.** 다음 서술의 참·거짓을 판정하고 **이유**를 쓰시오.
1. "생성자는 리턴 타입으로 `void`를 붙여야 한다."
2. "필드는 초기화하지 않으면 쓰레기 값이 들어간다."
3. "정적 메서드는 객체를 만들지 않고 `클래스이름.메서드()`로 호출할 수 있다."
4. "싱글톤 패턴에서는 생성자를 `public`으로 열어 두어야 한다."

<details><summary>정답·해설</summary>

1. **거짓.** 생성자는 리턴 타입이 **아예 없다**(`void`조차 붙이지 않는다). `void Book(){}`처럼 쓰면 생성자가 아니라 그냥 메서드가 되어 버린다.
2. **거짓.** **필드**는 타입별 기본값(int `0`, double `0.0`, boolean `false`, 참조형 `null`)으로 자동 초기화된다. 자동 초기화되지 않는 것은 **지역 변수**다.
3. **참.** 정적 멤버는 클래스에 소속되어 객체 생성 전에도 `클래스이름.메서드()`로 호출할 수 있다.
4. **거짓.** 싱글톤은 외부에서 `new`로 인스턴스를 여러 개 만들지 못하도록 생성자를 **`private`** 로 막고, 오직 `public static getInstance()`로만 유일 인스턴스를 얻게 한다.

</details>

## 요약

- **클래스** 는 객체를 만드는 설계도이자 새로운 **데이터 타입** 이며, **객체(인스턴스)** 는 `new`로 만든 실체다. 하나의 클래스로 만든 객체들은 서로 독립된 상태를 가진다.
- 자바에서 객체는 항상 **참조(주소)** 를 통해 다뤄진다. 참조 변수 대입은 객체 복사가 아니라 주소 복사이며, 아무 객체도 없으면 `null`이다.
- 클래스의 3대 구성 멤버는 **필드**(데이터/상태), **생성자**(초기화), **메서드**(행위)다. 필드는 타입별 기본값으로 자동 초기화되고, 생성자는 리턴 타입이 없으며 오버로딩과 `this()`로 유연하게 초기화한다.
- **인스턴스 멤버** 는 객체마다 따로 존재하고, **정적 멤버(`static`)** 는 클래스에 소속되어 모든 객체가 공유한다. `static` 메서드는 `this`와 인스턴스 멤버를 쓸 수 없다.
- **`final`** 은 재할당을 막고, **`static final` 상수** 는 고정된 공유 값에 의미 있는 이름을 부여한다.
- **패키지** 로 클래스를 조직화하고, **접근 제한자**(`public` > `protected` > `default` > `private`)로 공개 범위를 통제한다. 필드를 `private`로 숨기고 **Getter/Setter** 로 여는 것이 **캡슐화** 의 실천이며, Setter의 유효성 검증으로 객체를 항상 올바른 상태로 유지한다.
- **싱글톤 패턴** 은 `private static` 필드 + `private` 생성자 + `public static getInstance()` 로 인스턴스를 하나로 강제하는, 이 장의 문법이 종합된 실전 패턴이다.
- 객체지향의 나머지 축(상속·다형성·추상화·SOLID)은 『JAVA 개념서 4장』에서 이어진다.
