지금까지 우리는 클래스를 하나의 독립된 파일, 독립된 타입으로만 다뤄 왔다. 그런데 실제 코드를 보면 "클래스 안에 또 클래스가" 선언되어 있거나, 이름도 없이 `new Runnable() { ... }` 같은 형태로 객체가 즉석에서 튀어나오는 경우를 자주 만난다. 이것이 바로 **중첩 선언(nested declaration)** 과 **익명 객체(anonymous object)** 다.

왜 클래스를 밖으로 빼지 않고 굳이 다른 클래스 안에 두는가? 답은 **관계와 응집도**다. 어떤 타입이 특정 클래스 안에서만 의미를 가진다면, 그것을 바깥에 두는 것은 오히려 관계를 흐린다. 중첩 선언은 "이 타입은 저 클래스에 속한 것"이라는 소속을 코드 구조로 드러내고, 캡슐화를 강화하며, 관련 코드를 한자리에 모아 준다.

이 장에서는 선언 위치에 따라 나뉘는 네 가지 중첩 클래스(인스턴스 멤버 클래스, 정적 멤버 클래스, 로컬 클래스, 익명 클래스)와, 클래스 안에 두는 중첩 인터페이스를 차례로 살펴본다. 4장에서 배운 클래스/인터페이스/추상 클래스(→ JAVA_개념서 『4장. 객체지향이란 무엇인가?』 4.2, 4.6)와 6장의 람다/함수형 인터페이스(→ JAVA_개념서 『6장. 함수형 프로그래밍으로의 진화』 6.2)가 여기서 하나로 이어진다.

실습 코드는 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/` 아래에 있으며, 각 소절 끝의 `./run.sh <클래스명>` 으로 직접 실행해 볼 수 있다.

## 9.1 중첩 클래스

**중첩 클래스(nested class)** 란 말 그대로 "클래스 안에 선언된 클래스"다. 자바는 중첩 클래스를 **선언 위치**에 따라 네 가지로 분류한다. 종류를 결정하는 것은 이름이나 용도가 아니라 오직 "어디에 선언했는가"다.

| 종류 | 선언 위치 | static | 핵심 성질 |
|------|-----------|--------|-----------|
| 인스턴스 멤버 클래스 | 바깥 클래스의 멤버 자리 | 없음 | 바깥 **객체**가 있어야 생성 (9.2) |
| 정적 멤버 클래스 | 바깥 클래스의 멤버 자리 | 있음 | 바깥 객체 없이 생성 (9.3) |
| 로컬 클래스 | 메서드/초기화 블록 안 | — | 그 메서드 안에서만 존재 (9.4) |
| 익명 클래스 | 선언과 동시에 생성 | — | 이름 없이 일회성 사용 (9.7) |

가장 크게 갈리는 기준은 **static의 유무**다. 멤버 위치에 선언하되 static이 없으면 바깥 객체에 묶이고(인스턴스 멤버 클래스), static이 붙으면 바깥 객체와 무관한 독립 타입이 된다(정적 멤버 클래스). 메서드 안으로 들어가면 지역적인 로컬 클래스가 되고, 이름조차 붙이지 않고 즉석에서 만들면 익명 클래스가 된다.

```java
public class NestedClassIntro {
    class InstanceMember { }          // (1) 인스턴스 멤버 클래스 : static 없음
    static class StaticMember { }     // (2) 정적 멤버 클래스     : static 있음

    String useLocalClass() {
        class Local { }               // (3) 로컬 클래스 : 메서드 안
        return new Local().toString();
    }
    // (4) 익명 클래스는 new 부모타입() { ... } 로 즉석 생성 (9.7 참고)
}
```

## 9.2 인스턴스 멤버 클래스

**인스턴스 멤버 클래스**(비정적 내부 클래스)는 바깥 클래스의 멤버 자리에 **static 없이** 선언한 클래스다. 핵심 규칙은 단 하나, **바깥 객체가 반드시 있어야** 내부 클래스 객체를 만들 수 있다는 것이다. 즉 내부 객체는 항상 자신을 만든 바깥 객체 하나에 소속된다.

바깥 밖에서 생성할 때의 문법은 일반적인 `new`와 다르다. 먼저 바깥 객체를 만들고, 그 객체를 통해 `바깥객체.new 내부클래스()` 형태로 생성한다.

인스턴스 멤버 클래스는 바깥 객체에 묶여 있으므로 바깥 객체의 **인스턴스 필드/메서드에 직접 접근**할 수 있고, 그 값을 읽거나 바꿀 수도 있다. 대신 바깥 객체에 종속되기 때문에 (상수를 제외한) static 멤버는 가질 수 없다. "바깥 객체와 상태를 공유하는 부속품"이 필요할 때 알맞다.

```java
public class InstanceMemberClass {
    private String carName;
    private int fuel;

    class Engine {                        // static 없음 → 바깥 객체 소속
        void drive() {
            fuel--;                        // 바깥 객체의 필드를 직접 수정
            System.out.println(carName + " 주행 → 남은 연료 " + fuel);
        }
    }
}
// 사용 : 바깥 객체를 먼저 만들고 → 바깥객체.new 내부()
InstanceMemberClass car = new InstanceMemberClass("소나타", 2);
InstanceMemberClass.Engine engine = car.new Engine();
engine.drive();
```

바깥 객체마다 서로 다른 상태를 공유하므로, `car`의 엔진과 `car2`의 엔진은 각기 다른 연료를 바라본다.

## 9.3 정적 멤버 클래스

**정적 멤버 클래스(static nested class)** 는 바깥 클래스의 멤버 자리에 **static을 붙여** 선언한 클래스다. 인스턴스 멤버 클래스와 정반대로, **바깥 객체가 필요 없다**. 바깥 클래스 이름을 통해 `new 바깥클래스.내부클래스()` 로 곧바로 생성한다.

바깥 객체와 무관하기 때문에 몇 가지 접근 규칙이 자연스럽게 따라온다.

- 바깥의 **인스턴스** 필드/메서드에는 접근할 수 없다. (연결된 바깥 객체가 없으므로)
- 바깥의 **static** 필드/메서드에는 접근할 수 있다.
- 자기 자신도 static 멤버를 자유롭게 가질 수 있다.

정적 멤버 클래스는 특정 바깥 객체에 소속되지 않는 **독립적인 도우미 타입**에 적합하다. 좌표 `Point`, 연결 리스트의 `Node`, 빌더 패턴의 `Builder` 등이 대표적이다. 이들은 바깥 클래스와 개념적으로 묶여 있지만 개별 객체 상태와는 얽힐 필요가 없다.

```java
public class StaticMemberClass {
    static int createdCount = 0;

    static class Point {                   // static 붙음 → 바깥 객체 불필요
        int x, y;
        Point(int x, int y) {
            this.x = x; this.y = y;
            createdCount++;                // 바깥의 static 필드는 접근 가능
        }
        static double distance(Point a, Point b) { /* ... */ }
    }
}
// 사용 : 바깥 객체 없이 바로 생성
StaticMemberClass.Point a = new StaticMemberClass.Point(0, 0);
```

## 9.4 로컬 클래스

**로컬 클래스(local class)** 는 클래스 멤버 자리가 아니라 **메서드(또는 초기화 블록) 안**에 선언한 클래스다. 지역 변수처럼 그 메서드 안에서만 존재하며, 메서드가 끝나면 이름은 사라진다. 특정 메서드 안에서만 잠깐 쓰는 보조 타입을 만들 때 사용한다.

로컬 클래스의 가장 중요한 규칙은 **effectively final 캡처**다. 로컬 클래스는 자신을 감싼 메서드의 지역 변수와 매개변수를 사용(캡처)할 수 있는데, 그 변수는 값이 변하지 않아야 한다. `final`로 선언되었거나, `final`을 붙이지 않았더라도 "한 번 대입한 뒤 다시 바뀌지 않는(effectively final)" 변수여야 한다.

왜 이런 제약이 있을까? 로컬 클래스 객체는 메서드가 끝난 뒤에도 살아남을 수 있다. 그런데 메서드가 끝나면 그 지역 변수는 사라진다. 그래서 자바는 변수 자체를 참조하는 대신 **그 값의 복사본을 캡처**하는데, 값이 도중에 바뀌면 캡처한 복사본과 어긋나 버린다. 이 혼란을 막기 위해 "변하지 않는 값만" 캡처하도록 강제하는 것이다. (같은 캡처 규칙이 람다에도 적용된다 → JAVA_개념서 『6장. 함수형 프로그래밍으로의 진화』 6.3 "Lambda의 변수 캡처".)

```java
static void makeGreeter(String prefix) {   // prefix : 재대입 없음 → effectively final
    int mark = prefix.length();            // 이 지역 변수도 이후 변경 없으면 캡처 가능

    class Greeter {                        // 로컬 클래스 : 이 메서드 안에서만 존재
        void greet(String name) {
            System.out.println(prefix + name + " (prefix 길이=" + mark + ")");
        }
    }
    new Greeter().greet("홍길동");
}
```

## 9.5 바깥 멤버 접근

중첩 클래스가 바깥의 멤버에 접근할 수 있다는 것은 앞에서 보았다. 이 소절은 특히 **이름이 겹칠 때 무엇을 가리키는가**, 그리고 **바깥 객체 자신을 어떻게 가리키는가**에 초점을 둔다.

핵심은 두 개의 `this`를 구분하는 것이다. 내부 클래스 안에서 그냥 `this`는 지금 실행 중인 **내부 클래스 객체 자신**을 가리킨다. 반면 그 내부 객체를 소유한 **바깥 객체 자신**을 가리키려면 `바깥클래스.this` 라고 써야 한다. 이를 **Outer.this** 문법이라고 부른다.

이름이 겹칠 때의 우선순위는 안쪽부터다.

> 로컬 변수/매개변수 → 내부 클래스 필드(`this.필드`) → 바깥 필드(`Outer.this.필드`)

가장 안쪽(로컬 변수/매개변수)이 우선하고, 내부 클래스 필드는 `this.필드`로, 이름이 겹친 바깥 필드는 반드시 `바깥클래스.this.필드`로 명시해야 접근된다. 반대로 겹치지 않는 바깥 멤버는 그냥 이름만으로 접근된다.

```java
public class OuterMemberAccess {
    private int value = 100;               // 바깥 필드
    class Inner {
        private int value = 20;            // 내부 필드 (바깥과 이름이 겹침)
        void show(int value) {             // 매개변수까지 세 개가 겹침
            System.out.println(value);                          // 매개변수 (5)
            System.out.println(this.value);                     // 내부 필드 (20)
            System.out.println(OuterMemberAccess.this.value);   // 바깥 필드 (100)
        }
    }
}
```

## 9.6 중첩 인터페이스

**중첩 인터페이스(nested interface)** 는 클래스 안에 선언한 인터페이스다. 특정 클래스와 강하게 연관된 인터페이스를 그 클래스 안에 두면 관계가 분명해지고, 이름 충돌을 피하며, 응집도가 높아진다. 버튼 클래스 안에 클릭 이벤트 리스너 인터페이스를 두는 UI 콜백 패턴이 대표적인 예다.

기억할 규칙은 하나다. **클래스 안의 인터페이스는 자동으로 static**이다(암묵적 static). 그래서 바깥 객체 없이 `바깥클래스.인터페이스` 형태로 참조한다. 구현 방법은 자유롭다. 별도의 이름 있는 클래스로 구현해도 되고, 익명 클래스(9.7)로 즉석 구현해도 되며, 추상 메서드가 하나뿐인 함수형 인터페이스라면 람다로도 구현할 수 있다(→ JAVA_개념서 『4장』 4.6 인터페이스, 『6장』 6.2 함수형 인터페이스).

```java
public class NestedInterface {
    static class Button {
        interface OnClick {                // 암묵적 static, 추상 메서드 1개 → 함수형
            void onClick(String buttonName);
        }
        private OnClick listener;
        void setOnClick(OnClick l) { this.listener = l; }
        void press() { if (listener != null) listener.onClick("확인"); }
    }
}
// 익명 클래스로 구현
button.setOnClick(new NestedInterface.Button.OnClick() {
    public void onClick(String name) { System.out.println(name + " 처리"); }
});
// 함수형 인터페이스이므로 람다로도 구현 가능
button.setOnClick(name -> System.out.println(name + " 처리"));
```

## 9.7 익명 객체

**익명 객체(익명 클래스)** 는 이름이 없는 클래스를 선언과 동시에 곧바로 하나 생성한 객체다. 어떤 인터페이스나 (추상)클래스를 **딱 한 번만** 구현해서 쓸 때, 별도의 이름 있는 클래스를 만들지 않고 그 자리에서 즉석으로 정의한다.

문법은 `new 부모타입() { ...재정의/추가... }` 형태다.

- 부모타입이 **인터페이스**면 그 인터페이스를 구현한다.
- 부모타입이 **(추상)클래스**면 그 클래스를 상속한다.
- 중괄호 `{ }` 안에서 메서드를 재정의하거나 필드/메서드를 추가한다.

익명 클래스도 로컬 클래스처럼 effectively final 지역 변수를 캡처할 수 있다. 추상 메서드가 여러 개거나 필드/추가 메서드가 필요하면 익명 클래스가 제격이다. 하지만 추상 메서드가 **정확히 하나**인 함수형 인터페이스라면, 익명 클래스 대신 **람다식**으로 훨씬 짧게 쓸 수 있다. 실제로 자바에서 람다는 이 익명 클래스 문법의 군더더기를 걷어내며 등장했다(→ JAVA_개념서 『6장. 함수형 프로그래밍으로의 진화』 6.2 "익명 클래스에서 Lambda로").

```java
interface Greeting { String greet(String name); }

// [1] 인터페이스를 익명 클래스로 즉석 구현
Greeting korean = new Greeting() {
    @Override public String greet(String name) { return "안녕하세요, " + name + "님"; }
};

// [2] 지역 변수 캡처 (effectively final)
String suffix = "!!!";
Greeting excited = new Greeting() {
    @Override public String greet(String name) { return "야호 " + name + suffix; }
};

// [3] 함수형 인터페이스는 람다가 더 간결
Greeting english = name -> "Hello, " + name;
```

## ⚠️ 흔한 오해와 함정

- **"내부 클래스는 그냥 `new`로 만들면 된다"** — 인스턴스 멤버 클래스는 아니다. 바깥 객체를 먼저 만든 뒤 `바깥객체.new 내부()` 로 생성해야 한다. 그냥 `new 내부()`는 바깥 인스턴스 문맥이 없을 때 컴파일 오류가 난다. 반면 정적 멤버 클래스는 `new 바깥.내부()`로 바로 만든다.
- **"static이 붙든 안 붙든 별 차이 없다"** — static 하나가 성질을 완전히 바꾼다. static이 없으면 바깥 객체에 묶여 바깥 인스턴스 필드에 접근할 수 있지만, static이 붙으면 바깥 객체와 무관해져 바깥 인스턴스 필드에 접근할 수 없다.
- **"로컬 클래스에서 지역 변수를 마음대로 바꿔 쓸 수 있다"** — 캡처한 지역 변수는 effectively final이어야 한다. 캡처 후(또는 이전이라도) 그 변수에 재대입하면 "effectively final이 아니다"라는 컴파일 오류가 난다.
- **"익명 클래스 안의 `this`는 바깥 객체다"** — 아니다. 익명/내부 클래스 안의 `this`는 그 익명/내부 객체 자신이다. 바깥 객체를 가리키려면 `바깥클래스.this`(Outer.this)를 써야 한다.
- **"익명 클래스는 이제 필요 없다, 다 람다로 하면 된다"** — 람다는 추상 메서드가 하나인 함수형 인터페이스에만 쓸 수 있다. 추상 클래스를 상속하거나, 추상 메서드가 여럿이거나, 상태를 담을 필드가 필요하면 익명 클래스를 써야 한다.

**과제 1. 정적 멤버 클래스로 좌표 만들기 (쉬움)**

바깥 클래스 안에 `static class Point`(정적 멤버 클래스)를 선언하라. `Point`는 `int x, y` 필드와 두 값을 받는 생성자를 갖는다. `main`에서 바깥 객체 없이 `new 바깥클래스.Point(3, 4)` 형태로 두 개의 점을 만들고, 각 점을 `(x, y)` 형태로 출력하라. 정적 멤버 클래스는 **바깥 객체가 없어도** 곧바로 생성된다는 점을 코드로 확인하는 것이 핵심이다.

- 힌트: `static class Point { int x, y; Point(int x, int y){ ... } }`, 생성은 `new 바깥클래스.Point(...)`. 바깥 객체(`new 바깥클래스()`)를 먼저 만들 필요가 없다.
> 답안 비교: `./run.sh StaticMemberClass`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/StaticMemberClass.java`

**과제 2. 인스턴스 멤버 클래스로 바깥 상태 공유하기 (쉬움→응용)**

바깥 클래스에 `private int fuel;`(연료) 필드와 이를 초기화하는 생성자를 두고, 그 안에 `class Engine`(인스턴스 멤버 클래스, static 없음)을 선언하라. `Engine.drive()`는 호출될 때마다 바깥 객체의 `fuel`을 1씩 줄이고 남은 연료를 출력한다. `main`에서 바깥 객체를 먼저 만든 뒤 **`바깥객체.new Engine()`** 문법으로 엔진을 생성하고, `drive()`를 두어 번 호출해 연료가 줄어드는 것을 보여라.

- 힌트: 인스턴스 멤버 클래스는 바깥 객체에 소속되므로 바깥의 인스턴스 필드(`fuel`)에 직접 접근·수정할 수 있다. 생성 문법은 `new 내부()`가 아니라 `바깥객체.new Engine()`이다.
> 답안 비교: `./run.sh InstanceMemberClass`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/InstanceMemberClass.java`

**과제 3. 로컬 클래스와 변수 캡처 (응용)**

`static void makeGreeter(String prefix)` 같은 메서드를 만들고, **그 메서드 안에** `class Greeter`(로컬 클래스)를 선언하라. `Greeter.greet(String name)`은 메서드의 매개변수 `prefix`를 캡처해 `prefix + name` 형태로 인사말을 출력한다. `main`에서 서로 다른 `prefix`로 이 메서드를 두 번 호출해 결과를 확인하라. 마지막으로, 메서드 안에서 `prefix`에 재대입하는 줄을 **주석으로** 남기고 그 주석을 풀면 왜 "effectively final이 아니다" 오류가 나는지 한 줄로 설명하는 주석을 덧붙여라.

- 힌트: 로컬 클래스는 감싼 메서드의 지역 변수·매개변수를 캡처하되, 그 변수는 **effectively final**(한 번 대입 후 변경 없음)이어야 한다. 캡처한 변수에 다시 대입하면 컴파일 오류가 난다.
> 답안 비교: `./run.sh LocalClass`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/LocalClass.java`

**과제 4. 콜백 인터페이스를 익명 객체로 즉석 구현 (응용)**

`greet(String name)` 추상 메서드 하나를 가진 인터페이스 `Greeting`을 정의하라(또는 예제의 것을 참고해 직접 선언). `main`에서 이 인터페이스를 **익명 객체**(`new Greeting() { @Override public String greet(String name){ ... } }`)로 즉석 구현한 변수 `korean`을 만들어 한국어 인사를, effectively final 지역 변수(`String suffix = "!!!";`)를 **캡처**하는 또 다른 익명 객체 `excited`를 만들어 감탄사 붙인 인사를 각각 출력하라. 이름 있는 클래스를 따로 만들지 않고 그 자리에서 구현하는 것이 핵심이다.

- 힌트: 문법은 `new 인터페이스() { ...메서드 재정의... }`. 익명 객체도 로컬 클래스처럼 바깥 지역 변수를 캡처할 수 있다(effectively final).
> 답안 비교: `./run.sh AnonymousObject`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/AnonymousObject.java`

**과제 5. 익명 객체 vs 람다 — 중첩 인터페이스로 버튼 콜백 (응용)**

바깥 클래스 안에 `static class Button`을 두고, 그 안에 클릭 콜백용 **중첩 인터페이스** `interface OnClick { void onClick(String name); }`을 선언하라. `Button`은 `setOnClick(OnClick l)`로 리스너를 등록하고 `press()`로 `onClick("확인")`을 호출한다. `main`에서 **같은 리스너를 두 가지 방식**으로 등록해 보라 — 먼저 **익명 객체**(`new Button.OnClick() { ... }`)로, 그다음 동일 동작을 **람다**(`name -> ...`)로. 추상 메서드가 하나뿐인 함수형 인터페이스이므로 람다로 더 짧게 쓸 수 있음을 두 코드의 대비로 드러내라. (여력이 되면, 왜 추상 메서드가 여럿이거나 상태 필드가 필요하면 람다로 못 바꾸는지 주석 한 줄을 덧붙여라.)

- 힌트: 클래스 안의 인터페이스는 **암묵적 static**이라 `바깥.Button.OnClick`으로 참조한다. 익명 객체는 `new Button.OnClick(){ public void onClick(String name){...} }`, 람다는 `button.setOnClick(name -> ...)`.
> 답안 비교: `./run.sh NestedInterface`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/NestedInterface.java`

**과제 6. 바깥 멤버 접근과 Outer.this (응용)**

바깥 클래스에 인스턴스 필드 `private int value = 100;`을 두고, 그 안에 인스턴스 멤버 클래스 `Inner`를 선언하라. `Inner`도 같은 이름의 필드 `private int value = 20;`을 갖는다. `Inner`의 메서드 `show(int value)`(매개변수까지 세 이름이 겹침) 안에서 (1) 매개변수 `value`, (2) 내부 필드 `this.value`, (3) 바깥 필드 `바깥클래스.this.value` 세 값을 각각 구분해 출력하라. 또 바깥 객체 자신을 반환하는 메서드 `outer()`를 만들어 `return 바깥클래스.this;`로 반환하고, `main`에서 `inner.outer() == outer`가 `true`임을 확인하라. 이름이 겹칠 때의 우선순위(로컬/매개변수 → `this.` → `Outer.this.`)를 코드로 드러내는 것이 핵심이다.

- 힌트: 내부 클래스에서 그냥 `value`는 가장 안쪽(매개변수), `this.value`는 내부 필드, 겹친 바깥 필드는 반드시 `바깥클래스.this.value`로 명시한다. 바깥 객체 자신은 `바깥클래스.this`다.
> 답안 비교: `./run.sh OuterMemberAccess`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/OuterMemberAccess.java`

**과제 7. 네 가지 중첩 클래스 한눈에 비교하기 (종합)**

하나의 바깥 클래스 안에 (1) 인스턴스 멤버 클래스, (2) 정적 멤버 클래스를 각각 선언하고, 어떤 메서드 안에는 (3) 로컬 클래스를 선언하라. `main`에서 각각을 올바른 문법으로 생성해(정적은 `new 바깥.정적내부()`, 인스턴스는 `바깥객체.new 인스턴스내부()`, 로컬은 그 메서드 호출로) 각 객체가 자신의 종류를 한 줄씩 출력하게 하라. 마지막으로 (4) 익명 클래스를 `Runnable`이나 임의의 인터페이스로 즉석 생성해 실행하라. 네 종류가 **생성 문법**부터 다르다는 점을 하나의 파일에서 대조하는 것이 목적이다.

- 힌트: 정적 멤버 클래스는 바깥 객체 불필요, 인스턴스 멤버 클래스는 바깥 객체 필수, 로컬 클래스는 메서드 안에서만 생성, 익명 클래스는 `new 부모타입(){...}`. `NestedClassIntro`의 구조를 참고하되 실제 실행 코드까지 채워라.
> 답안 비교: `./run.sh StaticMemberClass` 및 `./run.sh InstanceMemberClass`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/InstanceMemberClass.java`

**과제 8. 중첩 인터페이스를 별도 클래스로 구현하기 (심화)**

과제 5의 `Button`/`OnClick` 구조를 재사용하되, 이번에는 콜백을 **익명 객체나 람다가 아니라 이름 있는 별도의 클래스**로 구현하라. 즉 `OnClick`을 구현하는 최상위(또는 정적 멤버) 클래스 `LogClickHandler`를 만들고, 이 클래스는 자신의 필드(예: `private int clickCount;`)에 클릭 횟수를 누적한 뒤 매 클릭마다 `"n번째 클릭: 이름"`을 출력하게 하라. `main`에서 `button.setOnClick(new LogClickHandler())`로 등록하고 `press()`를 여러 번 호출해 횟수가 누적되는지 확인하라. 상태(필드)가 필요할 때는 람다보다 이름 있는 구현 클래스가 알맞다는 점을 체감하는 것이 목적이다.

- 힌트: 중첩 인터페이스는 암묵적 static이므로 `class LogClickHandler implements 바깥.Button.OnClick { ... }`로 구현한다. 상태를 누적해야 하므로 필드가 필요하고, 이는 람다로 표현하기 어렵다(과제 5의 마지막 주석과 연결).
> 답안 비교: `./run.sh NestedInterface`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch09/NestedInterface.java`

## 확인문제

> 개념 이해 → 출력 예측 → 오류 찾기 → 서술까지, 학부 중간고사 수준으로 구성했다. 먼저 스스로 풀고 `<details>`의 정답·해설을 펼쳐 확인하라.

### 개념·서술형

**1.** 중첩 클래스를 네 종류로 나누는 **분류 기준**은 무엇인가? 그리고 인스턴스 멤버 클래스와 정적 멤버 클래스를 가르는 결정적 차이는 무엇인가?

<details>
<summary>정답·해설</summary>

분류 기준은 **"어디에 선언했는가(선언 위치)"** 다. 이름이나 용도가 아니다. 멤버 자리에 static 없이 두면 인스턴스 멤버 클래스, static을 붙이면 정적 멤버 클래스, 메서드/초기화 블록 안에 두면 로컬 클래스, 이름 없이 `new 부모타입(){...}`로 즉석 생성하면 익명 클래스다.

인스턴스 멤버 클래스와 정적 멤버 클래스를 가르는 결정적 차이는 **static의 유무 → 바깥 객체에 대한 소속 여부**다. static이 없으면 바깥 **객체**에 묶여 `바깥객체.new 내부()`로만 생성되고 바깥의 인스턴스 멤버에 직접 접근한다. static이 붙으면 바깥 객체와 무관한 독립 타입이 되어 `new 바깥.내부()`로 생성하고 바깥의 인스턴스 멤버에는 접근할 수 없다.

</details>

**2.** 정적 멤버 클래스는 바깥의 인스턴스 필드에는 접근할 수 없지만 바깥의 static 필드에는 접근할 수 있다. 그 이유를 "바깥 객체" 개념을 사용해 설명하라.

<details>
<summary>정답·해설</summary>

정적 멤버 클래스는 바깥 객체와 무관하게 생성되므로(`new 바깥.내부()`), 자신과 연결된 **바깥 객체가 존재하지 않는다**. 인스턴스 필드는 특정 바깥 객체에 속한 값이므로 "어느 객체의 필드인가"를 특정할 수 없어 접근이 불가능하다. 반면 static 필드는 특정 객체가 아니라 **클래스 자체에 하나만 존재**하는 값이므로, 바깥 객체가 없어도 클래스 이름만으로 접근할 수 있다. (`StaticMemberClass`의 `createdCount++`가 이 경우다.)

</details>

**3.** 로컬 클래스(그리고 익명 클래스)가 캡처하는 지역 변수는 왜 effectively final이어야 하는가? 변하는 값을 그대로 참조하도록 허용하면 어떤 문제가 생기는가?

<details>
<summary>정답·해설</summary>

로컬 클래스/익명 클래스 객체는 자신을 감싼 **메서드가 끝난 뒤에도 살아남을 수 있다**. 그런데 메서드가 끝나면 그 지역 변수(스택에 있던 값)는 사라진다. 그래서 자바는 변수 자체를 참조하는 대신 **그 값의 복사본을 캡처**한다. 만약 원본 변수가 도중에 바뀌는 것을 허용하면, 캡처된 복사본과 원본이 서로 어긋나 둘 중 어느 값이 맞는지 혼란이 생긴다. 이 불일치를 원천 차단하기 위해 "한 번 대입한 뒤 바뀌지 않는(effectively final) 값만" 캡처하도록 강제한다. (같은 규칙이 람다에도 적용된다.)

</details>

**4.** 정적 멤버 클래스와 인스턴스 멤버 클래스를 (1) 생성 문법, (2) 바깥 인스턴스 필드 접근 가능 여부, (3) 자기 자신의 static 멤버 보유 가능 여부의 세 관점에서 비교해 설명하라.

<details>
<summary>정답·해설</summary>

| 관점 | 정적 멤버 클래스 | 인스턴스 멤버 클래스 |
|------|------------------|----------------------|
| (1) 생성 문법 | `new 바깥.내부()` — 바깥 객체 불필요 | `바깥객체.new 내부()` — 바깥 객체 필수 |
| (2) 바깥 인스턴스 필드 접근 | 불가 (연결된 바깥 객체 없음) | 가능 (바깥 객체에 소속) |
| (3) 자기 static 멤버 | 자유롭게 보유 가능 | (상수 제외) 보유 불가 |

핵심은 "바깥 객체에 묶이는가"다. 정적 멤버 클래스는 독립적이라 자체 static 멤버도 자유롭고 바깥 static 멤버에만 접근한다. 인스턴스 멤버 클래스는 바깥 객체에 종속되어 바깥 인스턴스 필드를 공유하지만 자체 static 멤버는 가질 수 없다.

</details>

**5.** 익명 클래스와 람다는 둘 다 "즉석에서 구현"을 제공한다. 그럼에도 익명 클래스로만 할 수 있고 람다로는 할 수 없는 경우를 **세 가지** 서술하라.

<details>
<summary>정답·해설</summary>

1. **추상 클래스(또는 일반 클래스)를 상속**해야 할 때. 람다는 오직 함수형 인터페이스에만 쓸 수 있다. (예: `new Animal(){ String sound(){...} }`)
2. **추상 메서드가 둘 이상**인 인터페이스를 구현할 때. 람다는 추상 메서드가 정확히 하나인 함수형 인터페이스만 대상으로 한다.
3. **상태를 담을 필드나 추가 메서드**가 필요할 때. 익명 클래스는 `{ }` 안에 필드·메서드를 추가할 수 있지만 람다는 식 하나로 표현식만 담는다.

(참고: `this`의 의미도 다르다. 익명 클래스의 `this`는 그 익명 객체 자신이지만, 람다의 `this`는 자신을 감싼 바깥 객체를 가리킨다.)

</details>

### 출력 예측

**6.** 다음 코드의 출력은 무엇인가? (`InstanceMemberClass`의 구조를 참고)
```java
InstanceMemberClass car = new InstanceMemberClass("소나타", 2);
InstanceMemberClass.Engine engine = car.new Engine();
engine.drive();
engine.drive();
engine.drive();
```

<details>
<summary>정답·해설</summary>

```
소나타 주행 → 남은 연료 1
소나타 주행 → 남은 연료 0
소나타 연료 없음 → 주행 불가
```

인스턴스 멤버 클래스 `Engine`은 바깥 객체 `car`의 인스턴스 필드 `fuel`을 직접 수정한다. `fuel`은 2에서 시작해 `drive()`가 호출될 때마다 1씩 감소한다. 세 번째 호출에서는 `fuel <= 0`(0)이므로 "주행 불가"가 출력되고 `fuel`은 더 줄지 않는다.

</details>

**7.** `OuterMemberAccess`의 `Inner.show(int value)` 안에서 다음 세 줄이 실행될 때, `inner.show(5)` 호출의 출력을 순서대로 예측하라. 바깥 필드 `value = 100`, 내부 필드 `value = 20`이다.
```java
System.out.println(value);
System.out.println(this.value);
System.out.println(OuterMemberAccess.this.value);
```

<details>
<summary>정답·해설</summary>

```
5
20
100
```

이름이 겹칠 때 우선순위는 안쪽부터다. 그냥 `value`는 가장 안쪽인 **매개변수(5)**, `this.value`는 **내부 클래스 필드(20)**, `OuterMemberAccess.this.value`는 **바깥 필드(100)** 를 가리킨다. 바깥 객체 자신은 `Outer.this`로만 명시할 수 있다.

</details>

**8.** 다음 로컬 클래스 코드의 출력을 예측하라. (`prefix`, `mark` 캡처)
```java
static void makeGreeter(String prefix) {
    int mark = prefix.length();
    class Greeter {
        void greet(String name) {
            System.out.println(prefix + name + " (prefix 길이=" + mark + ")");
        }
    }
    new Greeter().greet("홍길동");
}
// 호출:
makeGreeter("안녕하세요, ");
```

<details>
<summary>정답·해설</summary>

```
안녕하세요, 홍길동 (prefix 길이=6)
```

로컬 클래스 `Greeter`는 감싼 메서드의 매개변수 `prefix`와 지역 변수 `mark`를 캡처한다. `mark`는 `"안녕하세요, "`의 길이인데, 한글 5자 + 쉼표 + 공백 = **6**이다. (자바 `String.length()`는 한글도 1문자로 센다.)

</details>

**9.** 다음 익명 객체 코드의 함정에 주의하며 출력을 예측하라.
```java
int base = 10;
Runnable r = new Runnable() {
    int base = 99;               // 익명 클래스 자신의 필드
    @Override public void run() {
        System.out.println(base);        // (가)
        System.out.println(this.base);   // (나)
    }
};
r.run();
```

<details>
<summary>정답·해설</summary>

```
99
99
```

익명 클래스 안에 같은 이름의 필드 `base = 99`가 선언되어 있으므로, `run()` 안에서 그냥 `base`는 **가장 가까운 스코프인 익명 클래스 자신의 필드(99)** 를 가리킨다. `this.base` 역시 익명 클래스 객체 자신의 필드이므로 99다. 바깥 지역 변수 `base = 10`을 가리키려면 캡처된 이름이 겹치지 않아야 하는데, 여기서는 겹쳐 가려졌다(shadowing). 함정: 익명 클래스의 `this`는 바깥이 아니라 **익명 객체 자신**을 가리킨다.

</details>

**10.** `NestedInterface`에서 다음 코드의 출력을 예측하라.
```java
NestedInterface.Button ok = new NestedInterface.Button("확인");
ok.setOnClick(name -> System.out.println("  → " + name + " 처리: 저장 완료"));
ok.press();
```
(`press()`는 `"이름 + 버튼 눌림"`을 먼저 출력하고, 리스너가 있으면 `onClick(name)`을 호출한다.)

<details>
<summary>정답·해설</summary>

```
확인 버튼 눌림
  → 확인 처리: 저장 완료
```

중첩 인터페이스 `OnClick`은 추상 메서드가 하나인 함수형 인터페이스이므로 람다로 구현할 수 있다. `press()`가 먼저 `"확인 버튼 눌림"`을 출력하고, 등록된 람다 리스너에 버튼 이름 `"확인"`을 넘겨 `onClick`을 실행한다.

</details>

### 오류 찾기

**11.** 다음 코드는 컴파일 오류가 난다. 어느 줄에서, 왜 오류가 나는지 설명하고 바르게 고쳐라.
```java
static void makeCounter() {
    int count = 0;
    class Counter {
        void tick() { System.out.println(count); }
    }
    count = 5;                 // (★)
    new Counter().tick();
}
```

<details>
<summary>정답·해설</summary>

**(★) 줄에서 오류**가 난다. 로컬 클래스 `Counter`가 지역 변수 `count`를 캡처했는데, `count = 5;`로 **재대입**하는 순간 `count`는 effectively final이 아니게 된다. 컴파일러는 "local variables referenced from an inner class must be final or effectively final" 오류를 낸다.

고치는 방법: 재대입을 없앤다. 예를 들어 `int count = 5;`로 처음부터 원하는 값을 대입하고 이후 바꾸지 않거나, 값을 바꿔야 한다면 캡처 대신 별도의 `int[] holder = {0}` 같은 배열/객체로 우회한다.

</details>

**12.** 다음 코드에서 인스턴스 멤버 클래스 생성 문법의 잘못을 찾아 고쳐라.
```java
public class Outer {
    class Inner { void hi() { System.out.println("hi"); } }
    public static void main(String[] args) {
        Inner in = new Inner();     // (★)
        in.hi();
    }
}
```

<details>
<summary>정답·해설</summary>

**(★) `new Inner()`가 오류**다. `Inner`는 static이 없는 인스턴스 멤버 클래스이므로 반드시 **바깥 객체**가 있어야 생성된다. 그런데 `main`은 static 메서드라 연결된 바깥 인스턴스(`this`)가 없다. 컴파일러는 "an enclosing instance ... is required" 오류를 낸다.

고침: 바깥 객체를 먼저 만든 뒤 `바깥객체.new 내부()`로 생성한다.
```java
Outer outer = new Outer();
Outer.Inner in = outer.new Inner();
in.hi();
```
(만약 `Inner`를 `static class Inner`로 바꾼다면 `new Inner()`로도 생성할 수 있다.)

</details>

### 빈칸 채우기

**13.** 빈칸 (가)~(다)를 채워라.
```java
// (가) 바깥 밖에서 인스턴스 멤버 클래스 Engine 객체 생성
InstanceMemberClass car = new InstanceMemberClass("소나타", 2);
InstanceMemberClass.Engine engine = ____(가)____;

// (나) 정적 멤버 클래스 Point 객체 생성 (바깥 객체 없이)
StaticMemberClass.Point p = ____(나)____;

// (다) 내부 클래스 안에서 바깥 객체 자신을 가리키는 표현
//     (바깥 클래스 이름이 OuterMemberAccess 일 때)
OuterMemberAccess self = ____(다)____;
```

<details>
<summary>정답·해설</summary>

- **(가)** `car.new Engine()` — 인스턴스 멤버 클래스는 `바깥객체.new 내부()` 문법.
- **(나)** `new StaticMemberClass.Point(3, 4)` (인자는 예시) — 정적 멤버 클래스는 `new 바깥.내부(...)`로 바깥 객체 없이 생성.
- **(다)** `OuterMemberAccess.this` — 내부 클래스에서 바깥 객체 자신은 `바깥클래스.this`(Outer.this).

</details>

**14.** 다음 문장의 빈칸을 채워라. "클래스 안에 선언한 인터페이스는 자동으로 ____(가)____ 이므로, 바깥 객체 없이 `____(나)____` 형태로 참조한다. 이 인터페이스의 추상 메서드가 정확히 하나면 ____(다)____ 로도 구현할 수 있다."

<details>
<summary>정답·해설</summary>

- **(가)** static (암묵적 static)
- **(나)** 바깥클래스.인터페이스 (예: `NestedInterface.Button.OnClick`)
- **(다)** 람다(람다식)

</details>

### 참·거짓 + 이유

**15.** 다음 각 명제의 참·거짓을 판정하고 이유를 밝혀라.

(a) 익명 클래스 안의 `this`는 자신을 감싼 바깥 객체를 가리킨다.

(b) 정적 멤버 클래스는 자기 자신의 static 필드를 가질 수 있다.

(c) 로컬 클래스는 감싼 메서드가 끝나면 그 객체도 즉시 사라진다.

(d) 추상 메서드가 두 개인 인터페이스도 람다로 구현할 수 있다.

<details>
<summary>정답·해설</summary>

- **(a) 거짓.** 익명 클래스(및 내부 클래스)의 `this`는 그 익명/내부 **객체 자신**을 가리킨다. 바깥 객체를 가리키려면 `바깥클래스.this`(Outer.this)를 써야 한다. (단, 람다의 `this`는 예외적으로 바깥 객체를 가리킨다.)
- **(b) 참.** 정적 멤버 클래스는 바깥 객체와 무관한 독립 타입이라 자기 static 멤버를 자유롭게 가질 수 있다. (인스턴스 멤버 클래스는 상수를 제외하면 불가.)
- **(c) 거짓.** 로컬 클래스 객체는 메서드가 끝난 뒤에도 참조가 남아 있으면 살아남을 수 있다. 바로 이 때문에 지역 변수를 effectively final 값으로 **복사(캡처)** 하는 규칙이 존재한다.
- **(d) 거짓.** 람다는 추상 메서드가 **정확히 하나**인 함수형 인터페이스에만 쓸 수 있다. 추상 메서드가 둘 이상이면 익명 클래스 등으로 구현해야 한다.

</details>

## 요약

- **중첩 클래스**는 클래스 안에 선언한 클래스로, **선언 위치**에 따라 인스턴스 멤버 클래스 / 정적 멤버 클래스 / 로컬 클래스 / 익명 클래스로 나뉜다. (9.1)
- **인스턴스 멤버 클래스**(static 없음)는 바깥 객체에 소속되며 `바깥객체.new 내부()`로 생성하고, 바깥의 인스턴스 멤버에 직접 접근한다. (9.2)
- **정적 멤버 클래스**(static 있음)는 바깥 객체 없이 `new 바깥.내부()`로 생성하는 독립 도우미 타입이며, 바깥의 static 멤버에만 접근한다. (9.3)
- **로컬 클래스**는 메서드 안에서만 존재하며, 감싼 메서드의 **effectively final** 지역 변수만 캡처한다. (9.4)
- 이름이 겹치면 안쪽(로컬 → `this.` → `Outer.this.`)이 우선하고, 겹친 바깥 필드와 바깥 객체 자신은 **`바깥클래스.this`(Outer.this)** 로 명시한다. (9.5)
- **중첩 인터페이스**는 클래스 안의 인터페이스로 **암묵적 static**이며 `바깥.인터페이스`로 참조한다. (9.6)
- **익명 객체**는 이름 없이 즉석에서 인터페이스를 구현하거나 (추상)클래스를 상속하는 일회성 객체다. 추상 메서드가 하나뿐이면 **람다**가 더 간결하다. (9.7)
