## 11.1 예외와 예외 클래스

프로그램 실행 중 발생하는 비정상 상황은 크게 두 가지로 나뉜다.

- **에러(Error)** : JVM 자체의 심각한 문제(메모리 부족, 스택 오버플로 등). 코드로 복구할 수 없다.
- **예외(Exception)** : 없는 파일 접근, 잘못된 입력처럼 "복구 가능한" 비정상 상황. 코드로 처리하여 정상 흐름으로 되돌릴 수 있다.

자바에서 던질 수 있는 모든 것(에러와 예외)의 최상위 조상은 `Throwable`이다. 계층 구조는 다음과 같다.

```
Object
  └─ Throwable                ← 던질 수 있는 것의 최상위 조상
       ├─ Error               ← 시스템 레벨 심각한 오류 (처리 대상 아님)
       │    ├─ OutOfMemoryError
       │    └─ StackOverflowError
       └─ Exception           ← 예외의 최상위
            ├─ IOException            ┐
            ├─ ClassNotFoundException ├─ checked (일반 예외)
            ├─ ...                    ┘
            └─ RuntimeException       ┐
                 ├─ NullPointerException           │
                 ├─ ArrayIndexOutOfBoundsException ├─ unchecked (실행 예외)
                 └─ ArithmeticException            ┘
```

여기서 두 갈래를 반드시 구분해야 한다.

- **checked 예외(일반 예외)** : `Exception`을 상속하되 `RuntimeException`은 상속하지 않는다. 컴파일러가 처리를 **강제**한다. `try-catch` 또는 `throws`가 없으면 컴파일되지 않는다.
- **unchecked 예외(실행 예외)** : `RuntimeException`의 자식. 컴파일러가 처리를 강제하지 **않는다**. 주로 프로그래머의 부주의(널 참조 등)로 발생한다.

`instanceof`로 이 계층 관계를 직접 확인할 수 있다.

```java
Exception e = new ArithmeticException("0으로 나눔");
System.out.println(e instanceof RuntimeException); // true → unchecked
System.out.println(e instanceof Throwable);        // true → 모든 예외의 조상

Throwable err = new StackOverflowError();
System.out.println(err instanceof Exception);      // false → Error는 예외가 아님
```

## 11.2 예외 처리 코드

예외가 발생해도 프로그램이 갑자기 종료되지 않도록 잡는 문법이 `try-catch-finally`다.

```java
try {
    // 예외가 발생할 수 있는 코드
} catch (예외타입 e) {
    // 예외가 발생했을 때 실행 (복구 / 알림)
} finally {
    // 예외 발생 여부와 상관없이 "항상" 실행 (자원 정리 등)
}
```

실행 흐름은 두 갈래다.

- **예외가 없으면** : `try` 전체 실행 → `finally` 실행 (`catch`는 건너뜀)
- **예외가 있으면** : `try` 도중 중단 → 맞는 `catch` 실행 → `finally` 실행

발생 가능한 예외가 여러 종류이면 **다중 catch**로 종류별로 나열한다. `catch`는 위에서부터 순서대로 검사하므로, 반드시 **자식 예외를 먼저, 부모 예외(`Exception`)를 마지막에** 써야 한다. 부모를 먼저 쓰면 자식 `catch`는 영원히 도달할 수 없어 컴파일 오류가 난다.

```java
try {
    int result = arr[index] / divisor;      // 두 가지 예외 가능
} catch (ArithmeticException e) {            // 자식 (0으로 나눔)
    System.out.println("0으로 나눌 수 없음");
} catch (ArrayIndexOutOfBoundsException e) { // 자식 (범위 초과)
    System.out.println("잘못된 인덱스");
} catch (Exception e) {                      // 부모는 반드시 마지막
    System.out.println("기타 예외");
} finally {
    System.out.println("항상 실행되는 정리 코드");
}
```

## 11.3 예외 종류에 따른 처리

11.1에서 본 checked / unchecked 구분은 단순한 분류가 아니라, **처리 방식(강제성)의 차이**를 만든다.

**(1) 일반 예외 = checked 예외**

- `Exception`을 상속하되 `RuntimeException`은 상속하지 않는다. 예) `IOException`, `ClassNotFoundException`.
- 컴파일러가 처리를 **강제**한다. 반드시 `try-catch`로 처리하거나 `throws`로 떠넘겨야 컴파일된다.
- 개발자가 어쩔 수 없는 외부 요인(파일 없음, 네트워크 끊김 등)에 주로 사용한다.

```java
static void readFileLike() throws java.io.IOException {   // throws로 떠넘기거나
    throw new java.io.IOException("파일을 찾을 수 없습니다");
}
// 호출부: 반드시 try-catch로 잡아야 컴파일됨
try {
    readFileLike();
} catch (java.io.IOException e) {
    System.out.println("처리됨: " + e.getMessage());
}
```

**(2) 실행 예외 = unchecked 예외**

- `RuntimeException`을 상속한다. 예) `NullPointerException`, `NumberFormatException`.
- 컴파일러가 처리를 **강제하지 않는다**. `try-catch` 없이도 컴파일되지만, 발생하면 실행 중 종료된다.
- 근본 원인은 코드의 결함이므로, 잡기보다는 코드 수정으로 예방하는 것이 원칙이다(다만 안전을 위해 잡을 수도 있다).

```java
static int parseNumber(String text) {   // throws 없이도 컴파일됨
    return Integer.parseInt(text);       // "abc"면 NumberFormatException
}
```

| 구분 | checked (일반) | unchecked (실행) |
|------|---------------|-----------------|
| 상속 | `Exception` | `RuntimeException` |
| 컴파일 강제 | O (try-catch / throws 필수) | X |
| 대표 예 | `IOException` | `NullPointerException` |
| 대응 원칙 | 처리 또는 위임 | 코드로 예방 |

## 11.4 리소스 자동 닫기 (try-with-resources)

파일, 소켓, DB 연결 같은 **리소스**는 사용 후 반드시 `close()`로 닫아야 한다. `finally`에서 직접 닫으면 코드가 길어지고, 닫기를 빠뜨리는 실수를 하기 쉽다.

`try-with-resources`는 이 닫기를 자동화한다.

```java
try (자원선언1; 자원선언2) {
    // 자원 사용
}   // → try 블록이 끝나면(정상/예외 무관) 선언한 자원의 close()가 "자동" 호출
```

두 가지 규칙을 기억하자.

- **조건** : `try` 괄호 안에 선언하는 객체는 반드시 `AutoCloseable` 인터페이스를 구현해야 한다. 이 인터페이스의 `close()`가 자동으로 불린다.
- **닫는 순서** : 여러 자원을 선언하면 **선언의 역순**으로 닫힌다(나중에 연 것을 먼저 닫음). 자원 간 의존 관계 때문에 역순 닫기가 안전하다.

```java
static class Resource implements AutoCloseable {
    private final String name;
    Resource(String name) { this.name = name; }
    void use() { System.out.println(name + " 사용"); }
    @Override public void close() { System.out.println(name + " 닫힘"); }
}

try (Resource a = new Resource("A");
     Resource b = new Resource("B")) {
    a.use();
    b.use();
}   // 닫힘 순서: B → A (선언 역순)
```

사용 중 예외가 발생해도 `close()`는 반드시 호출된다는 점이 핵심이다.

## 11.5 예외 떠넘기기 (throws)

메소드 안에서 발생한 예외를 그 자리에서 직접 처리하지 않고, "나를 호출한 쪽에서 처리하라"고 위임할 수 있다. 이것이 **예외 떠넘기기(`throws`)** 다.

```java
리턴타입 메소드명(...) throws 예외타입 {
    // 이 안에서 발생한 예외를 처리하지 않고 호출자에게 넘긴다
}
```

예외를 잡는(`catch`) 곳이 나올 때까지, 예외는 **호출 스택을 거꾸로 거슬러 올라가며 전파**된다. `main → levelA → levelB → levelC` 순으로 호출했고 `levelC`에서 예외가 나면,

```
levelC  (발생, throws)
  → levelB  (안 잡음, throws)
    → levelA  (안 잡음, throws)
      → main  (여기서 try-catch로 최종 처리)
```

끝까지 아무도 잡지 않으면 프로그램은 비정상 종료된다.

```java
static void levelC() throws TaskException {
    throw new TaskException("levelC에서 발생");
}
static void levelB() throws TaskException { levelC(); }  // 안 잡고 떠넘김
static void levelA() throws TaskException { levelB(); }  // 안 잡고 떠넘김

public static void main(String[] args) {
    try {
        levelA();                       // 전파된 예외가 여기까지 옴
    } catch (TaskException e) {
        System.out.println("최종 처리: " + e.getMessage());
    }
}
```

예외 객체의 `getStackTrace()`를 보면 예외가 거쳐 온 호출 경로를 확인할 수 있다. `throw`(예외를 던짐)와 `throws`(떠넘기겠다고 선언)는 이름이 비슷하지만 역할이 다르니 헷갈리지 말자.

## 11.6 사용자 정의 예외

자바가 제공하는 표준 예외만으로는 "우리 프로그램만의 의미 있는 오류"를 정확히 표현하기 어렵다. 예를 들어 `잔액 부족`을 단순히 `Exception`으로 던지면, 무엇이 잘못되었는지 이름만으로는 알 수 없다. 이럴 때 예외 클래스를 직접 만든다.

**만드는 방법**

- checked 예외로 만들려면 → `Exception`을 상속 (처리 강제)
- unchecked 예외로 만들려면 → `RuntimeException`을 상속 (처리 강제 X)

**관례**

- 클래스 이름은 `...Exception`으로 끝맺는다.
- 메시지를 받는 생성자 `super(message)`를 제공한다.
- 필요하면 원인 코드, 부족한 금액 같은 추가 필드를 넣어 더 풍부한 정보를 담는다.

```java
// checked: 잔액 부족 → 반드시 처리하게 만들고 싶다 → Exception 상속
static class InsufficientBalanceException extends Exception {
    private final int shortage;                 // 추가 정보
    InsufficientBalanceException(String message, int shortage) {
        super(message);
        this.shortage = shortage;
    }
    int getShortage() { return shortage; }
}

// unchecked: 잘못된 입력 → 프로그래머 실수 성격 → RuntimeException 상속
static class InvalidAmountException extends RuntimeException {
    InvalidAmountException(String message) { super(message); }
}
```

`Account.withdraw()`에서 잘못된 금액이면 `InvalidAmountException`(unchecked)을, 잔액 부족이면 `InsufficientBalanceException`(checked)을 던져, 상황별로 의미 있는 예외를 구분해 처리한다.

> 💻 실습: `./run.sh CustomException`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch11/CustomException.java`

## ⚠️ 흔한 오해와 함정

**함정 1. 예외를 삼키지 마라 (빈 catch 블록)**
`catch (Exception e) {}`처럼 예외를 잡고 아무것도 하지 않으면, 문제가 조용히 사라져 나중에 원인을 찾을 수 없게 된다. 최소한 로그를 남기거나 메시지를 출력하자.

**함정 2. `catch (Exception e)` 하나로 뭉뚱그리지 마라**
모든 예외를 부모 타입 하나로 잡으면, 정작 상황별로 다르게 대응할 기회를 잃는다. 구체적인 예외를 먼저 잡고, `Exception`은 진짜 최후의 보루로만 쓴다. 또한 다중 catch에서 부모 타입을 자식보다 먼저 쓰면 컴파일 오류가 난다(11.2).

**함정 3. `finally`에서 `return`하지 마라**
`finally`의 `return`은 `try`나 `catch`에서 던진 예외까지 덮어써 사라지게 만든다. `finally`는 정리(close 등) 용도로만 쓴다.

**함정 4. Error를 잡으려 하지 마라**
`OutOfMemoryError`, `StackOverflowError` 같은 `Error`는 JVM 수준의 회복 불가능한 문제다. `catch (Throwable)`로 억지로 잡아도 프로그램을 정상으로 되돌릴 수 없으니, 잡지 않는 것이 원칙이다(11.1).

**함정 5. 예외를 흐름 제어에 쓰지 마라**
반복 종료나 조건 분기를 예외로 구현하면 코드가 느려지고 읽기 어려워진다. 예외는 "예외적인 상황"에만 쓴다. unchecked 예외는 잡기보다 `if` 검증으로 예방하는 편이 낫다(11.3).

**과제 1. 안전한 나눗셈기 (try-catch-finally · 다중 catch)** — 쉬움

정수 배열 `int[] arr = {100, 200, 300}`과, 인덱스·나눗수를 받아 `arr[index] / divisor`를 계산해 출력하는 `divide(int[] arr, int index, int divisor)` 메소드를 작성하라. 이때 **0으로 나누기**와 **배열 범위 초과**를 각각 다른 `catch`로 잡고, 어떤 경우든 마지막에 `[finally] 정리 완료`를 출력해야 한다. `divide(arr, 0, 5)`(정상), `divide(arr, 1, 0)`(0으로 나눔), `divide(arr, 9, 5)`(범위 초과) 세 가지를 호출해 흐름을 확인하라.

<details>
<summary>힌트</summary>

- `arr[index]`는 `ArrayIndexOutOfBoundsException`, `/ divisor`는 `ArithmeticException`이 날 수 있다.
- 다중 `catch`는 **자식 예외를 먼저, 부모(`Exception`)를 마지막**에 써야 한다. 부모를 먼저 쓰면 컴파일 오류가 난다.
- `finally`는 예외 발생 여부와 관계없이 **항상** 실행된다. 세 호출 모두에서 finally 줄이 찍히는지 확인하라.
</details>

**과제 2. 문자열 숫자 변환기 (checked vs unchecked)** — 쉬움

두 개의 메소드를 만들어라. (1) `readFileLike()` — 몸통에서 `throw new java.io.IOException("파일 없음")`을 하되, 이 예외를 **직접 잡지 말고** 메소드 선언에 `throws`를 붙여 호출자에게 떠넘긴다. (2) `parseNumber(String text)` — `Integer.parseInt(text)`의 결과를 반환하되, `throws`를 **붙이지 않는다**. `main`에서 `readFileLike()`는 반드시 `try-catch`로 감싸고, `parseNumber`는 `"123"`(정상)과 `"abc"`(예외)를 각각 넣어 결과를 확인하라.

<details>
<summary>힌트</summary>

- `IOException`은 **checked** 예외라 컴파일러가 처리를 강제한다. `throws`를 빼면 컴파일조차 안 된다.
- `Integer.parseInt("abc")`는 `NumberFormatException`(**unchecked**)을 던진다. `throws` 없이도 컴파일되지만, 안 잡으면 실행 중 종료된다.
- 두 메소드를 나란히 보며 "왜 하나는 throws가 필수이고 하나는 선택인가"를 스스로 설명해보라.
</details>

> 답안 비교: `./run.sh ExceptionTypes`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch11/ExceptionTypes.java`

**과제 3. 자동으로 닫히는 자원 (try-with-resources · AutoCloseable)** — 응용

`AutoCloseable`을 구현한 `Resource` 클래스를 만들어라. 생성자에서 `[열림] 이름`, `use()`에서 `[사용] 이름`, `close()`에서 `[닫힘] 이름`을 출력한다. 그런 다음 `try-with-resources`로 자원 `A`, `B`, `C` **세 개를 한 번에 선언**해 각각 `use()`를 호출하고, `try` 블록이 끝날 때 `close()`가 **어떤 순서로** 불리는지 출력으로 확인하라.

<details>
<summary>힌트</summary>

- `try (자원선언) { ... }` 괄호 안에 선언한 객체는 반드시 `AutoCloseable`을 구현해야 하고, 블록 종료 시 `close()`가 **자동** 호출된다(`finally` 불필요).
- 여러 자원은 **선언의 역순**으로 닫힌다. `A; B; C` 순으로 열었다면 닫힘은 `C → B → A`다.
- 여유가 되면 `try` 블록 안에서 일부러 `throw new RuntimeException(...)`을 해보라. 예외가 나도 `close()`가 먼저 호출되는 것을 볼 수 있다.
</details>

> 답안 비교: `./run.sh TryWithResources`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch11/TryWithResources.java`

**과제 4. 예외 전파 추적기 (throws · 호출 스택 전파)** — 응용

`main → levelA → levelB → levelC` 순으로 호출되는 세 메소드를 작성하라. 가장 안쪽 `levelC()`에서 직접 만든 checked 예외(예: `TaskException extends Exception`)를 `throw`하고, `levelC`·`levelB`·`levelA`는 **아무도 잡지 않고 오직 `throws`로만 떠넘긴다**. 예외는 오직 `main`의 `try-catch`에서만 최종 처리되어야 한다. 각 메소드에 진입할 때 자기 이름을 출력해, 예외가 어느 지점에서 발생해 어디까지 거슬러 올라가는지 흐름을 확인하라.

<details>
<summary>힌트</summary>

- checked 예외를 잡지 않고 넘기려면 메소드마다 `... levelC() throws TaskException` 처럼 `throws` 선언이 필요하다. 하나라도 빠지면 컴파일 오류.
- `throw`는 "예외를 던진다"(실행), `throws`는 "떠넘기겠다고 선언한다"(선언)로 역할이 다르다. 헷갈리지 말자.
- `catch`한 예외 객체의 `getStackTrace()`를 돌려보면 예외가 거쳐 온 호출 경로(`levelC → levelB → levelA → main`)를 확인할 수 있다.
</details>

> 답안 비교: `./run.sh ExceptionPropagation`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch11/ExceptionPropagation.java`

**과제 5. 잔액 부족 출금기 (사용자 정의 예외 · 도메인 규칙)** — 도전

간단한 은행 계좌를 표현하는 `Account` 클래스를 만들어라. 잔액을 필드로 갖고, `withdraw(int amount)` 메소드는 두 가지 규칙을 지킨다. (1) 출금액이 **0 이하**이면 unchecked 사용자 정의 예외 `InvalidAmountException extends RuntimeException`을 던진다. (2) 출금액이 **잔액보다 크면** checked 사용자 정의 예외 `InsufficientBalanceException extends Exception`을 던지되, **부족한 금액**을 추가 필드로 담아 `getShortage()`로 꺼낼 수 있게 한다. `main`에서 잔액 10,000원인 계좌에 대해 정상 출금(3,000), 잔액 부족(50,000), 잘못된 금액(-100) 세 경우를 각각 호출해 상황별로 다른 예외가 잡히는지 확인하라.

<details>
<summary>힌트</summary>

- 사용자 정의 예외는 checked면 `Exception`을, unchecked면 `RuntimeException`을 상속한다. 관례상 클래스 이름은 `...Exception`으로 끝맺고, `super(message)` 생성자를 제공한다.
- `InsufficientBalanceException`은 부족액을 담을 `private final int shortage;` 필드와 이를 받는 생성자, 그리고 `getShortage()`를 둔다.
- `withdraw`가 checked 예외를 던지므로 메소드 선언에 `throws InsufficientBalanceException`이 필요하다. `main`에서는 `InvalidAmountException`(unchecked)과 `InsufficientBalanceException`(checked)을 **각각의 `catch`로 구분**해 잡아, "왜 이 예외인지"를 메시지로 구분해 출력하라.
</details>

> 답안 비교: `./run.sh CustomException`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch11/CustomException.java`

**과제 6. finally 실행 순서 관찰기 (try-catch-finally · 실행 흐름)** — 응용

`try-catch-finally`의 실행 순서를 눈으로 확인하는 프로그램을 작성하라. `probe(int mode)` 메소드를 만들어 `try` 진입 시 `[try] 진입`, `catch` 시 `[catch] 잡힘`, `finally` 시 `[finally] 정리`를 출력한다. `mode == 0`이면 예외가 나지 않게, `mode == 1`이면 `try` 안에서 일부러 `ArithmeticException`(예: `10 / 0`)이 나게 하라. 두 경우(`probe(0)`, `probe(1)`)를 각각 호출해, **예외가 없을 때는 catch를 건너뛰고**, **예외가 있을 때만 catch가 실행되며**, **어느 경우든 finally는 항상 실행됨**을 출력으로 확인하라.

<details>
<summary>힌트</summary>

- 예외가 없으면 `[try] → [finally]` 순으로만 찍히고 `[catch]`는 건너뛴다.
- 예외가 있으면 `[try](중단) → [catch] → [finally]` 순으로 찍힌다. 예외 발생 이후의 `try` 코드는 실행되지 않으니, `try` 끝에도 출력문을 하나 두어 그 줄이 `mode==1`에서 안 찍히는지 확인해 보라.
- `finally` 안에는 절대 `return`을 두지 말자(함정 3). 정리 용도로만 쓴다.
</details>

> 답안 비교: `./run.sh TryCatchFinally`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch11/TryCatchFinally.java`

**과제 7. 계층 판별기 (예외 계층 · instanceof)** — 쉬움

여러 예외 객체가 `Throwable` 계층 어디에 속하는지 `instanceof`로 판별해 출력하는 프로그램을 작성하라. `new ArithmeticException("x")`, `new java.io.IOException("y")`, `new NullPointerException()`, `new StackOverflowError()`를 각각 만들어, 각 객체가 `RuntimeException`인지 / `Exception`인지 / `Throwable`인지를 `instanceof`로 검사해 `true/false`를 출력하라. 결과를 보고 "checked vs unchecked", "Error는 Exception이 아니다"를 스스로 설명해 보라.

<details>
<summary>힌트</summary>

- `ArithmeticException`, `NullPointerException`은 `RuntimeException`의 자식(unchecked)이라 세 검사 모두 `true`다.
- `IOException`은 `Exception`이지만 `RuntimeException`은 **아니다**(checked). `instanceof RuntimeException`이 `false`로 나오는지 확인하라.
- `StackOverflowError`는 `Error`라서 `instanceof Exception`이 **`false`** 다. 하지만 `instanceof Throwable`은 `true`다. "모든 예외/에러의 조상은 `Throwable`"임을 눈으로 확인할 수 있다.
</details>

**과제 8. 안전한 자원 관리자 (try-with-resources · 예외 발생 시 close 보장)** — 도전

과제 3의 `Resource`를 재사용하되, 이번엔 **사용 도중 예외가 나도 close가 반드시 호출됨**을 명확히 보여 주는 프로그램을 작성하라. `try-with-resources`로 자원 `A`, `B` 두 개를 선언하고, `use()` 호출 뒤 `try` 블록 안에서 일부러 `throw new RuntimeException("사용 중 오류")`를 던진다. 이 예외를 바깥의 `catch (RuntimeException e)`로 잡아 메시지를 출력하라. 출력에서 **`[닫힘] B` → `[닫힘] A`가 `catch`보다 먼저** 찍히는지 확인하라.

<details>
<summary>힌트</summary>

- 예외가 나기 전에 이미 열린 자원은 블록을 빠져나갈 때 **모두 자동으로 닫힌다.** 닫힘은 선언 역순이므로 `B` 먼저, `A` 나중이다.
- 출력 순서는 대략 `[열림]A → [열림]B → [사용]... → [닫힘]B → [닫힘]A → [catch] 처리`가 된다. close가 catch보다 먼저임을 눈으로 확인하는 것이 이 과제의 핵심이다.
- 이것이 `try-with-resources`의 가장 큰 이점이다. `finally`에 close를 직접 쓰지 않아도, 어떤 경로로 블록을 빠져나가든 자원이 닫힌다.
</details>

## 확인문제

> 아래 문제는 개념 서술, 출력 예측, 오류 찾기, 빈칸 채우기, 참·거짓 판단을 고루 섞었다. 먼저 스스로 답을 적어 본 뒤 `<details>`를 펼쳐 해설과 대조하자. 출력 예측 문제는 **머릿속으로만 풀지 말고** 반드시 종이에 실행 순서를 적어 보라.

**1. (개념 서술)** 다음 중 `Throwable`의 직계 자식 두 가지는 무엇이며, 그중 코드로 복구할 수 있는 것은 어느 쪽인가? 나머지 한쪽을 코드로 잡으려 하면 안 되는 이유도 함께 설명하라.

<details>
<summary>정답 및 해설</summary>

`Throwable`의 직계 자식은 **`Error`** 와 **`Exception`** 이다.

- **코드로 복구할 수 있는 쪽은 `Exception`** 이다. 없는 파일 접근, 잘못된 입력처럼 복구 가능한 비정상 상황을 표현하며, `try-catch`로 잡아 정상 흐름으로 되돌릴 수 있다.
- `Error`는 `OutOfMemoryError`, `StackOverflowError`처럼 **JVM 수준의 회복 불가능한 문제**다. `catch (Throwable)`로 억지로 잡아도 프로그램을 정상 상태로 되돌릴 수 없으므로 잡지 않는 것이 원칙이다. (11.1)
</details>

**2. (개념 서술)** checked 예외와 unchecked 예외를 **상속 관계**와 **컴파일러의 강제 여부** 두 관점에서 구분해 설명하라. 각각의 대표 예외도 하나씩 들고, 각 예외에 대한 "대응 원칙"까지 적어라.

<details>
<summary>정답 및 해설</summary>

| 구분 | checked (일반 예외) | unchecked (실행 예외) |
|------|---------------------|------------------------|
| 상속 관계 | `Exception`을 상속하되 `RuntimeException`은 상속하지 **않음** | `RuntimeException`을 상속 |
| 컴파일러 강제 | O — `try-catch` 또는 `throws` 없으면 컴파일 오류 | X — 없어도 컴파일됨 |
| 대표 예외 | `IOException`, `ClassNotFoundException` | `NullPointerException`, `ArithmeticException` |
| 대응 원칙 | 처리 또는 위임(`throws`) | 코드 수정으로 예방(`if` 검증) |

핵심: unchecked 예외는 대개 **프로그래머의 부주의**(널 참조, 0 나눗셈)로 생기므로, 잡기보다 코드로 예방하는 것이 원칙이다. (11.1, 11.3)
</details>

**3. (오류 찾기)** 다음 다중 catch 코드는 컴파일되지 않는다. 이유를 설명하고 올바르게 고쳐라.
```java
try {
    int r = arr[i] / d;
} catch (Exception e) {
    System.out.println("기타");
} catch (ArithmeticException e) {
    System.out.println("0으로 나눔");
}
```

<details>
<summary>정답 및 해설</summary>

`catch`는 **위에서부터 순서대로** 검사한다. 첫 번째 `catch (Exception e)`가 모든 예외의 부모 타입이므로 `ArithmeticException`을 포함해 **모든 예외를 먼저 잡아 버린다.** 그 결과 아래의 `catch (ArithmeticException e)`는 **영원히 도달할 수 없는 코드**가 되어 컴파일 오류(`exception ArithmeticException has already been caught`)가 난다.

올바른 순서는 **자식 예외를 먼저, 부모(`Exception`)를 마지막**에 두는 것이다.
```java
try {
    int r = arr[i] / d;
} catch (ArithmeticException e) {   // 자식 먼저
    System.out.println("0으로 나눔");
} catch (Exception e) {             // 부모는 마지막
    System.out.println("기타");
}
```
(11.2)
</details>

**4. (개념 서술)** `try-with-resources`로 자원을 자동으로 닫으려면, 그 자원 클래스는 어떤 인터페이스를 구현해야 하는가? 또한 자원을 `A, B, C` 순으로 선언했다면 `close()`는 어떤 순서로 호출되며, 왜 그런 순서인가?

<details>
<summary>정답 및 해설</summary>

- 자원 클래스는 **`AutoCloseable`** 인터페이스를 구현해야 한다. 이 인터페이스의 `close()`가 블록 종료 시 자동 호출된다.
- `A, B, C` 순으로 선언했다면 닫힘 순서는 **`C → B → A`**, 즉 **선언의 역순**이다.
- 이유: 나중에 연 자원이 앞서 연 자원에 **의존**할 수 있기 때문이다(예: A 연결 위에 B 스트림을 얹음). 의존하는 쪽(나중에 연 것)을 먼저 닫아야 안전하다. (11.4)
</details>

**5. (개념 서술)** `main → levelA → levelB → levelC` 순으로 호출되고 `levelC`에서 checked 예외가 발생했다. `levelC`, `levelB`, `levelA`가 모두 예외를 잡지 않고 `throws`로 떠넘긴다면 이 예외는 최종적으로 어디에서 처리되어야 하는가? 아무도 처리하지 않으면 어떻게 되는가?

<details>
<summary>정답 및 해설</summary>

예외는 호출 스택을 **거꾸로 거슬러 올라가며 전파**된다(`levelC → levelB → levelA → main`). 중간 메소드가 모두 `throws`로 떠넘기면, 예외는 최종 호출자인 **`main`까지 올라와 그곳의 `try-catch`에서 처리**되어야 한다.

`main`에서도 잡지 않으면 예외는 JVM까지 전파되어 **프로그램이 비정상 종료**되고, 콘솔에 스택 트레이스가 출력된다. (11.5)
</details>

**6. (출력 예측)** 다음 코드의 출력을 순서대로 예측하라.
```java
static void run() {
    try {
        System.out.println("A");
        int x = 10 / 0;
        System.out.println("B");
    } catch (ArithmeticException e) {
        System.out.println("C");
    } finally {
        System.out.println("D");
    }
    System.out.println("E");
}
```

<details>
<summary>정답 및 해설</summary>

```
A
C
D
E
```
- `"A"` 출력 후 `10 / 0`에서 `ArithmeticException` 발생 → `try` 도중 중단되므로 **`"B"`는 출력되지 않는다.**
- 맞는 `catch`가 실행되어 `"C"` 출력.
- `finally`는 예외 발생 여부와 무관하게 항상 실행되어 `"D"` 출력.
- `try-catch`에서 예외가 정상 처리되었으므로 이후 흐름이 이어져 `"E"` 출력. (11.2)
</details>

**7. (출력 예측 · finally return 함정)** 다음 메소드의 반환값은 무엇인가? 그리고 이 코드가 왜 위험한지 설명하라.
```java
static int test() {
    try {
        return 1;
    } finally {
        return 2;
    }
}
```

<details>
<summary>정답 및 해설</summary>

반환값은 **`2`** 다.

`try`의 `return 1;`이 실행되어 반환값 `1`이 준비되지만, 메소드가 실제로 빠져나가기 **직전에 `finally`가 반드시 실행**된다. 이때 `finally`의 `return 2;`가 앞선 `return 1;`을 **덮어써** 최종 반환값이 `2`가 된다.

위험한 이유: `finally`의 `return`은 반환값뿐 아니라 **`try`나 `catch`에서 던진 예외까지 삼켜** 사라지게 만든다. 그래서 `finally`는 자원 정리(close 등) 용도로만 쓰고, **`return`/`throw`를 두지 않는 것**이 원칙이다. (11.2, 함정 3)
</details>

**8. (출력 예측 · 리소스 close 순서 함정)** 아래 `Res`는 `AutoCloseable` 구현체로 생성자에서 `"open X"`, `close()`에서 `"close X"`를 출력한다. 다음 코드의 출력을 순서대로 예측하라.
```java
try (Res a = new Res("A");
     Res b = new Res("B")) {
    System.out.println("body");
    throw new RuntimeException("boom");
} catch (RuntimeException e) {
    System.out.println("catch " + e.getMessage());
}
```

<details>
<summary>정답 및 해설</summary>

```
open A
open B
body
close B
close A
catch boom
```
- 자원은 선언 순서대로 열린다: `open A` → `open B`.
- `try` 블록 진입 후 `"body"` 출력, 그다음 예외 발생.
- **예외가 나도 `close()`는 반드시 호출**되며, 순서는 **선언의 역순**(`close B` → `close A`). 즉 `catch`보다 **close가 먼저** 일어난다.
- 자원이 모두 닫힌 뒤에야 예외가 `catch`로 전달되어 `catch boom` 출력. (11.4)
</details>

**9. (출력 예측 · 다중 catch 순서)** `arr`의 길이가 3일 때 다음 호출들의 출력을 각각 예측하라.
```java
static void f(int[] arr, int i, int d) {
    try {
        System.out.println("result=" + (arr[i] / d));
    } catch (ArithmeticException e) {
        System.out.println("arith");
    } catch (ArrayIndexOutOfBoundsException e) {
        System.out.println("index");
    } catch (Exception e) {
        System.out.println("other");
    }
}
// 호출:
// (1) f(arr, 0, 2)   // arr = {10, 20, 30}
// (2) f(arr, 1, 0)
// (3) f(arr, 5, 2)
```

<details>
<summary>정답 및 해설</summary>

- (1) `f(arr, 0, 2)` → `arr[0]/2 = 10/2 = 5` → `result=5` (예외 없음)
- (2) `f(arr, 1, 0)` → `arr[1]/0` → `ArithmeticException` → `arith`
- (3) `f(arr, 5, 2)` → `arr[5]` 접근에서 `ArrayIndexOutOfBoundsException` → `index`

포인트: `arr[i] / d`는 **`arr[i]`를 먼저 평가**한 뒤 나눗셈을 한다. (3)에서는 인덱스 접근 단계에서 이미 예외가 나므로 나눗셈까지 가지 않는다. 세 `catch` 모두 자식 → 부모 순으로 배치되어 있어 각 예외가 자기 타입에 정확히 잡힌다. (11.2)
</details>

**10. (오류 찾기)** 다음 코드에서 잘못된 부분을 찾아 이유를 설명하고 고쳐라.
```java
static void save() {
    try {
        writeFile();   // IOException(checked)을 던질 수 있음
    } catch (RuntimeException e) {
        System.out.println("실패");
    }
}
```

<details>
<summary>정답 및 해설</summary>

`writeFile()`이 던지는 `IOException`은 **checked 예외**인데, 위 코드는 `catch (RuntimeException e)`로 **unchecked 예외만** 잡고 있다. `IOException`은 `RuntimeException`의 자식이 아니므로 **잡히지 않으며**, checked 예외가 처리도 위임도 되지 않아 **컴파일 오류**가 난다.

고치는 방법은 둘 중 하나다.
```java
// (a) 직접 잡기
try {
    writeFile();
} catch (IOException e) {          // checked 예외를 정확히 잡음
    System.out.println("실패");
}
// (b) 위임하기
static void save() throws IOException {
    writeFile();
}
```
(11.3)
</details>

**11. (빈칸 채우기)** 아래 문장의 빈칸을 채워라.

(a) `Exception`을 상속하되 `RuntimeException`은 상속하지 않는 예외를 (①)__________ 예외라 하며, 컴파일러가 처리를 (②)__________ 한다.
(b) `try-with-resources`에 선언하는 자원은 반드시 (③)__________ 인터페이스를 구현해야 한다.
(c) 예외를 처리하지 않고 호출자에게 넘기겠다고 메소드 선언에 붙이는 키워드는 (④)__________ 이고, 실제로 예외를 던지는 키워드는 (⑤)__________ 이다.
(d) 다중 catch에서는 (⑥)__________ 예외를 먼저, (⑦)__________ 예외를 마지막에 써야 한다.

<details>
<summary>정답 및 해설</summary>

- ① **checked**(일반) ② **강제**
- ③ **`AutoCloseable`**
- ④ **`throws`** ⑤ **`throw`**
- ⑥ **자식**(구체적인) ⑦ **부모**(`Exception` 같은 상위)

`throws`(떠넘기겠다는 선언)와 `throw`(실제로 던지는 실행문)는 철자가 비슷하지만 역할이 다르다. (11.2, 11.4, 11.5)
</details>

**12. (참·거짓 + 이유)** 다음 각 진술이 참인지 거짓인지 판단하고, 이유를 한 줄로 적어라.

(a) `finally` 블록은 `try`에서 예외가 발생하지 않으면 실행되지 않는다.
(b) `NumberFormatException`은 `try-catch` 없이도 컴파일된다.
(c) `try`에서 `return`을 만나면 `finally`는 건너뛴다.
(d) `catch (Exception e)`는 `Error`도 함께 잡는다.

<details>
<summary>정답 및 해설</summary>

- (a) **거짓.** `finally`는 예외 발생 여부와 **무관하게 항상** 실행된다.
- (b) **참.** `NumberFormatException`은 `RuntimeException`의 자식인 **unchecked 예외**라 컴파일러가 처리를 강제하지 않는다.
- (c) **거짓.** `try`의 `return`이 실행되어도, 메소드를 실제로 빠져나가기 직전에 `finally`가 반드시 실행된다.
- (d) **거짓.** `Exception`과 `Error`는 모두 `Throwable`의 자식이지만 **서로 형제 관계**다. `catch (Exception e)`는 `Error`를 잡지 못한다(`Error`까지 잡으려면 `catch (Throwable)`이 필요하지만 권장되지 않는다). (11.1, 11.2, 11.3)
</details>

**13. (오류 찾기)** 다음 사용자 정의 예외 코드에서 잘못된 부분을 모두 찾아 고쳐라.
```java
class InsufficientBalanceException extends Exception {
    // 생성자 없음
}

static void withdraw(int amount, int balance) {   // throws 없음
    if (amount > balance) {
        throw new InsufficientBalanceException("잔액 부족");
    }
}
```

<details>
<summary>정답 및 해설</summary>

두 가지 문제가 있다.

1. `InsufficientBalanceException`에 **메시지를 받는 생성자가 없어** `new InsufficientBalanceException("잔액 부족")` 호출이 컴파일되지 않는다. `super(message)`를 호출하는 생성자를 추가해야 한다.
2. `InsufficientBalanceException`은 `Exception`을 상속한 **checked 예외**인데 `withdraw`에 `throws` 선언이 없어 컴파일되지 않는다.

```java
class InsufficientBalanceException extends Exception {
    InsufficientBalanceException(String message) {
        super(message);                       // (1) 생성자 추가
    }
}

static void withdraw(int amount, int balance)
        throws InsufficientBalanceException {  // (2) throws 선언 추가
    if (amount > balance) {
        throw new InsufficientBalanceException("잔액 부족");
    }
}
```
(11.6)
</details>

**14. (개념 서술)** `try-with-resources`가 `finally`에서 직접 `close()`를 호출하는 방식에 비해 갖는 이점을 두 가지 이상 서술하라.

<details>
<summary>정답 및 해설</summary>

- **닫기를 빠뜨릴 실수를 원천 차단한다.** 자원을 `try(...)`에 선언만 하면 `close()`가 자동 호출되므로, `finally`에 close 코드를 일일이 쓰다가 누락하는 실수가 없다.
- **코드가 짧고 읽기 쉽다.** 여러 자원을 닫는 `finally` 블록(각각 null 검사 + try-catch로 감싸야 함)이 사라진다.
- **여러 자원을 선언 역순으로 안전하게 닫아 준다.** 순서를 직접 관리할 필요가 없다.
- **예외가 숨는 문제를 줄인다.** 직접 `finally`에서 close할 때 close가 던지는 예외가 원래 예외를 덮어쓰는 문제가, try-with-resources에서는 "suppressed exception"으로 보존된다. (11.4)
</details>

**15. (출력 예측 · 예외 전파)** 다음 코드의 출력을 순서대로 예측하라.
```java
static void c() { System.out.println("c"); throw new RuntimeException("x"); }
static void b() { System.out.println("b"); c(); }
static void a() { System.out.println("a"); b(); }

public static void main(String[] args) {
    try {
        a();
    } catch (RuntimeException e) {
        System.out.println("caught " + e.getMessage());
    }
    System.out.println("end");
}
```

<details>
<summary>정답 및 해설</summary>

```
a
b
c
caught x
end
```
- 호출 순서대로 `a` → `b` → `c` 출력.
- `c()`에서 `RuntimeException`이 발생하고, `c`·`b`·`a` 모두 잡지 않으므로 **호출 스택을 거슬러 `main`까지 전파**된다.
- `main`의 `catch`가 잡아 `caught x` 출력.
- `RuntimeException`은 unchecked라 `a`/`b`/`c`에 `throws` 선언이 없어도 전파된다는 점에 주목하라. 예외가 `main`에서 정상 처리되었으므로 이후 흐름이 이어져 `end`가 출력된다. (11.5)
</details>

## 요약

- **예외 계층** : 모든 예외/에러의 최상위는 `Throwable`이며, 그 아래 `Error`(복구 불가)와 `Exception`(복구 가능)으로 나뉜다. `Exception`은 다시 checked와 `RuntimeException`(unchecked)으로 갈린다. (11.1)
- **try-catch-finally** : `try`에서 예외를 감지하고 `catch`에서 처리하며 `finally`는 항상 실행된다. 다중 catch는 자식 예외를 먼저, 부모를 마지막에 쓴다. (11.2)
- **예외 종류** : checked는 컴파일러가 처리를 강제(`try-catch`/`throws`)하고, unchecked는 강제하지 않으며 코드 수정으로 예방한다. (11.3)
- **try-with-resources** : `AutoCloseable`을 구현한 자원을 `try(...)`에 선언하면 블록 종료 시 선언의 역순으로 `close()`가 자동 호출된다. (11.4)
- **throws** : 예외를 처리하지 않고 호출자에게 떠넘긴다. 예외는 호출 스택을 거슬러 전파되며, 끝까지 잡히지 않으면 프로그램이 비정상 종료된다. (11.5)
- **사용자 정의 예외** : `Exception`(checked) 또는 `RuntimeException`(unchecked)을 상속해 우리 도메인에 맞는 의미 있는 예외를 만든다. (11.6)
