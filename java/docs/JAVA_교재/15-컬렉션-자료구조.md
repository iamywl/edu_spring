프로그램은 결국 여러 개의 데이터를 담고, 찾고, 바꾸고, 지우는 일의 연속이다. 회원 목록을 관리하고, 장바구니에 상품을 넣고, 사용자 ID로 정보를 조회하는 모든 작업이 그렇다. 이때 데이터를 담는 "그릇"으로 배열을 쓸 수도 있지만, 배열은 크기가 고정되어 있고 삽입·삭제·검색 같은 기능을 직접 구현해야 하는 불편함이 있다.

## 15.1 컬렉션 프레임워크

### 개념

**컬렉션 프레임워크**는 여러 데이터를 담고 다루기 위한 표준화된 자료구조 라이브러리다. 배열과 달리 크기가 동적으로 늘어나며, `add`·`remove`·`contains` 같은 편의 메소드를 제공한다.

컬렉션은 크게 두 개의 상위 인터페이스 계층으로 나뉜다.

```
        Iterable
           │
       Collection                        Map  (별도 계층)
      ┌────┼────┐                      ┌───┴───┐
    List  Set  Queue                HashMap  TreeMap ...
     │     │     │
 ArrayList │  ArrayDeque
 LinkedList│  PriorityQueue
        HashSet
        LinkedHashSet
        TreeSet
```

- **Collection** : 원소들의 묶음
  - **List** : 순서 O, 중복 O (ArrayList, LinkedList)
  - **Set** : 순서는 구현체별, 중복 X (HashSet, LinkedHashSet, TreeSet)
  - **Queue** : 대기열 (ArrayDeque, PriorityQueue)
- **Map** : 키(Key) → 값(Value) 매핑, 키 중복 X (HashMap, TreeMap)
  - Map은 Collection을 상속하지 않는 **별도 계층**이라는 점에 주의한다.

배열은 생성 시 크기가 고정되지만, 컬렉션은 동적으로 커진다.

```java
// 배열: 크기 3으로 고정 → 더 넣으려면 새 배열로 복사해야 함
int[] arr = new int[3];

// 컬렉션(List): 자유롭게 추가 가능
List<Integer> list = new ArrayList<>();
list.add(10);
list.add(20);
list.add(30);
list.add(40);   // 배열과 달리 계속 추가 가능
```

## 15.2 List 컬렉션

### 개념

**List**는 순서가 있고 중복을 허용하는 컬렉션이다. 인덱스(0부터)로 접근한다.

| 구현체 | 내부 구조 | get(i) 조회 | 앞/뒤 삽입·삭제 |
|--------|-----------|:-----------:|:----------------:|
| ArrayList | 배열 | O(1) | 중간은 O(n) |
| LinkedList | 이중 연결 리스트 | O(n) | O(1) |

- **임의 인덱스 조회가 잦다** → ArrayList
- **앞/뒤 삽입·삭제가 잦다** → LinkedList

```java
List<String> fruits = new ArrayList<>();
fruits.add("사과");        // 맨 뒤에 추가
fruits.add("바나나");
fruits.add("사과");        // List는 중복 허용
fruits.add(1, "딸기");     // 인덱스 1에 끼워넣기
fruits.set(0, "청사과");   // 0번 교체
fruits.remove("바나나");   // 값으로 삭제
fruits.remove(0);          // 인덱스로 삭제
System.out.println(fruits.get(0));  // 인덱스로 조회
```

순회는 for-index, for-each, Iterator 세 가지 방법이 있다.

> 💻 실습: `./run.sh ListCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/ListCollection.java`

### 내부 구조 더 알아보기

ArrayList의 `add`가 왜 평균 O(1)인지(동적 배열의 amortized 분석), LinkedList의 노드 구조는 [JAVA_개념서 5장](../JAVA_교재/심화/05-데이터를-효율적으로-다루다-컬렉션.md)과 [CS_알고리즘 - 자료구조 직접 구현](../CS_알고리즘_개념서/05-자료구조-직접-구현.md)에서 직접 구현하며 유도한다.

## 15.3 Set 컬렉션

### 개념

**Set**은 중복을 허용하지 않는 컬렉션이다. 같은 값을 여러 번 `add`해도 하나만 저장된다. 인덱스 개념이 없어 `get(i)` 같은 위치 접근은 제공하지 않는다.

세 구현체는 **순서 유지 방식**이 다르다.

| 구현체 | 순서 | 내부 구조 |
|--------|------|-----------|
| HashSet | 보장 안 함 (가장 빠름) | 해시 테이블 |
| LinkedHashSet | 삽입 순서 유지 | 해시 테이블 + 연결 리스트 |
| TreeSet | 정렬(오름차순) 유지 | 균형 이진 트리 |

```java
Set<String> set = new HashSet<>();
set.add("사과");
set.add("바나나");
set.add("사과");   // 중복 → 무시됨
System.out.println(set.size());            // 2
System.out.println(set.contains("사과"));  // true

// 같은 입력 50 10 40 20 30 을 넣었을 때
new HashSet<>();        // 순서 무의미
new LinkedHashSet<>();  // 50 10 40 20 30 (넣은 순서)
new TreeSet<>();        // 10 20 30 40 50 (자동 정렬)
```

대표적인 활용은 리스트의 **중복 제거**다.

> 💻 실습: `./run.sh SetCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/SetCollection.java`

### 내부 구조 더 알아보기

HashSet이 어떻게 중복을 판별하는지(`hashCode`/`equals` 계약)와 해시 충돌 처리는 [JAVA_개념서 5장의 HashMap 내부 동작](../JAVA_교재/심화/05-데이터를-효율적으로-다루다-컬렉션.md)에서 다룬다. Set은 내부적으로 Map을 사용한다.

## 15.4 Map 컬렉션

### 개념

**Map**은 키(Key) → 값(Value) 쌍으로 저장하는 컬렉션이다.

- 키는 중복될 수 없다(같은 키에 `put`하면 값이 덮어써진다).
- 값은 중복될 수 있다.
- 인덱스가 아니라 **키**로 값을 꺼낸다.

구현체별 키 순서는 Set과 같은 규칙을 따른다: HashMap(순서 X), LinkedHashMap(삽입 순서), TreeMap(키 정렬).

```java
Map<String, Integer> scores = new HashMap<>();
scores.put("국어", 90);
scores.put("수학", 85);
scores.put("수학", 100);                       // 같은 키 → 85를 100으로 덮어씀
scores.get("수학");                            // 100
scores.get("과학");                            // null (없는 키)
scores.getOrDefault("과학", 0);                // 0 (기본값 제공)
scores.containsKey("국어");                    // true
scores.remove("국어");

// 순회 3가지
for (String key : scores.keySet()) { ... }     // 키만
for (Integer v : scores.values()) { ... }      // 값만
for (Map.Entry<String, Integer> e : scores.entrySet()) {
    e.getKey(); e.getValue();                  // 키+값
}
```

> 💻 실습: `./run.sh MapCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/MapCollection.java`

### 내부 구조 더 알아보기

HashMap이 100만 개 중에서 어떻게 한 번에 값을 찾아내는지(버킷·해시 함수·충돌 시 트리화)는 [JAVA_개념서 5장](../JAVA_교재/심화/05-데이터를-효율적으로-다루다-컬렉션.md)의 핵심 주제이며, 기술 면접에서 가장 자주 나온다.

## 15.5 검색 강화 컬렉션

### 개념

TreeSet / TreeMap은 정렬 상태를 유지하므로 **범위 검색과 근사 검색**에 강하다. 이들은 `NavigableSet` / `NavigableMap` 인터페이스를 구현하며 다음 메소드를 제공한다.

**값 하나 찾기** (집합이 `{10, 20, 30, 40, 50}`일 때)

| 메소드 | 의미 | 예: 기준 30 |
|--------|------|:-----------:|
| `first()` / `last()` | 최소 / 최대 | 10 / 50 |
| `floor(x)` | x 이하(≤) 중 최대 | `floor(30)` = 30 |
| `ceiling(x)` | x 이상(≥) 중 최소 | `ceiling(25)` = 30 |
| `lower(x)` | x 미만(<) 중 최대 | `lower(30)` = 20 |
| `higher(x)` | x 초과(>) 중 최소 | `higher(30)` = 40 |

**범위(부분) 뷰**

- `headSet/headMap(x)` : x보다 작은 부분
- `tailSet/tailMap(x)` : x 이상인 부분
- `subSet/subMap(a, b)` : a 이상 b 미만 부분

```java
NavigableSet<Integer> set = new TreeSet<>(List.of(10, 20, 30, 40, 50));
set.floor(25);          // 20
set.ceiling(25);        // 30
set.subSet(20, 50);     // [20, 30, 40]

NavigableMap<Integer, String> map = new TreeMap<>();
map.floorKey(25);       // 25 이하 최대 키
map.headMap(30);        // 키 30 미만
```

> 💻 실습: `./run.sh SearchCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/SearchCollection.java`

### 내부 구조 더 알아보기

TreeMap이 정렬을 유지하면서도 검색·삽입이 O(log n)인 원리(레드-블랙 트리 등 균형 이진 탐색 트리)는 [CS_알고리즘 - 트리와 힙](../CS_알고리즘_개념서/06-트리와-힙.md)에서 다룬다.

## 15.6 LIFO와 FIFO

### 개념

데이터를 넣고 빼는 **순서 규칙**에 따라 두 가지 대표 자료구조가 있다.

- **Stack (LIFO, Last-In-First-Out)** : 나중에 넣은 것이 먼저 나온다. (접시 쌓기)
- **Queue (FIFO, First-In-First-Out)** : 먼저 넣은 것이 먼저 나온다. (줄 서기)

자바에서는 오래된 `java.util.Stack` 대신 **`Deque`(ArrayDeque)** 를 스택·큐 양쪽으로 쓰기를 권장한다.

| 용도 | 삽입 | 제거 | 확인 |
|------|------|------|------|
| Stack (LIFO) | `push` (맨 위) | `pop` (맨 위) | `peek` |
| Queue (FIFO) | `offer` (뒤) | `poll` (앞) | `peek` |

**PriorityQueue**는 넣은 순서와 무관하게 우선순위(기본: 작은 값)가 먼저 나온다.

```java
Deque<String> stack = new ArrayDeque<>();
stack.push("A"); stack.push("B"); stack.push("C");
stack.pop();    // C  (나중에 넣은 것부터)

Queue<String> queue = new ArrayDeque<>();
queue.offer("1"); queue.offer("2"); queue.offer("3");
queue.poll();   // 1  (먼저 넣은 것부터)

PriorityQueue<Integer> pq = new PriorityQueue<>();
pq.offer(40); pq.offer(10); pq.offer(30);
pq.poll();      // 10 (작은 값부터)
```

> 💻 실습: `./run.sh StackQueueCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/StackQueueCollection.java`

### 내부 구조 더 알아보기

PriorityQueue의 기반 자료구조인 힙(heap)의 삽입·삭제 O(log n) 원리는 [CS_알고리즘 - 트리와 힙](../CS_알고리즘_개념서/06-트리와-힙.md)에서 직접 구현하며 다룬다.

## 15.7 동기화된 컬렉션

### 개념

`ArrayList`, `HashMap` 같은 기본 컬렉션은 **동기화되지 않았다(not thread-safe)**. 여러 스레드가 동시에 수정하면 데이터가 깨지거나 `ConcurrentModificationException` 같은 예외가 발생할 수 있다. 단일 스레드에서는 문제없지만, 멀티 스레드 환경에서는 대책이 필요하다.

두 가지 해결책이 있다.

1. **동기화 래퍼** : `Collections.synchronizedList/Set/Map(...)` 으로 기존 컬렉션을 감싼다. 모든 메소드가 하나의 락으로 보호된다. 단, **순회(for-each) 시에는 직접 `synchronized` 블록으로 감싸야** 안전하다.
2. **`java.util.concurrent` 패키지** : 애초에 동시성을 위해 설계된 컬렉션. `ConcurrentHashMap`은 부분 잠금으로 동기화 래퍼보다 성능이 우수하다. (`CopyOnWriteArrayList` 등)

```java
// 1) 동기화 래퍼
List<Integer> syncList = Collections.synchronizedList(new ArrayList<>());
synchronized (syncList) {            // 순회는 직접 락으로 감싸야 안전
    for (int v : syncList) { ... }
}

// 2) ConcurrentHashMap (부분 잠금 → 더 빠름)
Map<Integer, Integer> map = new ConcurrentHashMap<>();
map.put(1, 1);
```

> 💻 실습: `./run.sh SynchronizedCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/SynchronizedCollection.java`

### 내부 구조 더 알아보기

동기화가 왜 필요한지(경쟁 상태, 잃어버린 갱신), 락과 CAS의 원리는 [CS_운영체제 개념서 - 동기화·락·CAS·volatile](../CS_운영체제_개념서/04-동기화-락-cas-volatile.md)에서 다룬다.

## 15.8 수정할 수 없는 컬렉션

### 개념

**수정할 수 없는(불변, unmodifiable/immutable) 컬렉션**은 생성 후 내용을 바꿀 수 없다. `add`·`remove`·`set`·`put`을 호출하면 `UnsupportedOperationException`이 발생한다. 실수로(또는 외부에서) 데이터가 변경되는 것을 막아 안전하게 공유할 수 있어, 상수 목록이나 설정값처럼 변하면 안 되는 데이터에 적합하다.

만드는 방법은 두 가지다.

1. `List.of(...)` / `Set.of(...)` / `Map.of(...)` : 처음부터 불변으로 생성 (JDK 9+).
2. `Collections.unmodifiableList/Set/Map(원본)` : 기존 컬렉션의 **읽기 전용 뷰**.

```java
List<String> days = List.of("월", "화", "수");   // 처음부터 불변
days.get(0);        // 읽기는 자유
days.add("목");     // UnsupportedOperationException!

// 읽기 전용 뷰 - 주의: 원본이 바뀌면 뷰도 바뀐다
List<String> origin = new ArrayList<>(List.of("A", "B"));
List<String> readOnly = Collections.unmodifiableList(origin);
readOnly.set(0, "Z");   // UnsupportedOperationException!
origin.add("C");        // 원본 수정 → readOnly에도 "C"가 반영됨
```

완전한 불변이 필요하면 `List.of` 나 `List.copyOf` 를 사용한다.

## ⚠️ 흔한 오해와 함정

- **Map은 Collection이 아니다.** `Map`은 `Collection`을 상속하지 않는 별도 계층이다. `iterator()`가 없으므로 순회하려면 `keySet()`·`values()`·`entrySet()`을 거쳐야 한다.
- **HashSet/HashMap의 순서를 믿지 마라.** "출력 순서"는 보장되지 않으며 JDK 버전이나 원소에 따라 달라질 수 있다. 순서가 필요하면 LinkedHashSet/LinkedHashMap, 정렬이 필요하면 TreeSet/TreeMap을 쓴다.
- **`List.of(...)`는 불변이다.** 여기에 `add`하면 `UnsupportedOperationException`이 난다. 수정 가능한 리스트가 필요하면 `new ArrayList<>(List.of(...))`처럼 복사한다.
- **`unmodifiableList`는 "뷰"일 뿐이다.** 원본을 수정하면 뷰에도 반영된다. 원본과 독립된 완전한 불변이 필요하면 `List.copyOf(원본)`을 쓴다.
- **커스텀 객체를 Set/Map 키로 쓸 때는 `equals`/`hashCode`를 함께 재정의해야 한다.** 그렇지 않으면 중복 판별과 조회가 의도대로 동작하지 않는다.
- **동기화 컬렉션도 순회는 별도로 보호해야 한다.** `Collections.synchronizedList`로 감싸도 for-each 순회 중에는 직접 `synchronized` 블록으로 감싸야 안전하다.
- **스택은 `java.util.Stack` 대신 `ArrayDeque`.** 구형 `Stack`은 `Vector` 기반이라 모든 연산이 동기화되어 느리고, 반복자 순서도 직관과 반대다.

### 과제 2. 중복 방문자 제거 (Set)

### 과제 3. 단어 빈도수 세기 (Map)

한 문장 `"apple banana apple cherry banana apple"`을 공백으로 나눠, **각 단어가 몇 번 나왔는지**를 세어 출력하는 프로그램을 작성하라. 결과는 `apple=3`, `banana=2`, `cherry=1` 처럼 단어별 등장 횟수여야 한다.

- 힌트: `String[] words = sentence.split(" ");` 로 단어 배열을 만든다. `Map<String, Integer> count = new HashMap<>();` 를 준비하고, 각 단어에 대해 `count.put(w, count.getOrDefault(w, 0) + 1);` 로 "기존 값 + 1"을 저장하는 것이 핵심이다.
- 힌트: 없는 키를 `get`하면 `null`이 나와 `+ 1`에서 오류가 난다. 그래서 **`getOrDefault(w, 0)`** 으로 처음 보는 단어에는 0을 기본값으로 준다. 순회 출력은 `entrySet()`을 돌며 `getKey()`/`getValue()`를 쓴다.
- 응용: `HashMap` 대신 `TreeMap`으로 바꾸면 단어가 **사전순으로 정렬**되어 출력된다. 두 결과를 비교해 보자.

### 과제 4. 괄호 열고 닫기 확인 (Deque - 스택)

`"(()())"` 같은 괄호 문자열이 **올바르게 짝지어졌는지** 판별하는 프로그램을, `ArrayDeque`를 **스택(LIFO)** 으로 사용해 작성하라. 여는 괄호 `(`를 만나면 스택에 `push`하고, 닫는 괄호 `)`를 만나면 `pop`한다. 모두 처리한 뒤 스택이 비어 있으면 올바른 괄호다. `"(()())"`는 `올바름`, `"(()"`나 `")("`는 `올바르지 않음`을 출력하라.

- 힌트: `Deque<Character> stack = new ArrayDeque<>();` 로 시작한다. 닫는 괄호인데 **스택이 이미 비어 있으면**(pop할 게 없으면) 그 즉시 올바르지 않다고 판정할 수 있다.
- 힌트: 문자열은 `for (char c : s.toCharArray())` 로 한 글자씩 순회한다. 스택이 "나중에 넣은 여는 괄호부터 꺼내진다"는 LIFO 성질이 괄호 짝 맞추기에 왜 딱 맞는지 생각해 보자.
- 응용: 같은 `ArrayDeque`를 이번엔 **큐(FIFO)** 로 써서, `offer`로 넣고 `poll`로 꺼내면 넣은 순서 그대로 나오는 것도 확인해 보자.

> 답안 비교: `./run.sh StackQueueCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/StackQueueCollection.java`

### 과제 5. 성적 구간 조회 (TreeMap - 범위 검색)

점수를 키로, 학생 이름을 값으로 하는 `TreeMap<Integer, String>`에 `{70:"철수", 82:"영희", 90:"민수", 95:"수지"}`를 넣고, 정렬 기반 **범위·근사 검색**을 실습하는 프로그램을 작성하라. 다음을 각각 구해 출력하라: (1) 85점 **이하** 중 가장 높은 점수의 학생(`floorKey`), (2) 85점 **이상** 중 가장 낮은 점수의 학생(`ceilingKey`), (3) **80점 이상 95점 미만** 구간에 든 학생들(`subMap`).

- 힌트: `TreeMap`은 키를 자동으로 **오름차순 정렬**해 유지하므로 이런 검색이 가능하다. `map.get(map.floorKey(85))` 처럼 먼저 키를 찾고 그 키로 값을 꺼낸다.
- 힌트: `map.subMap(80, 95)`는 80 이상 95 **미만**(95는 제외)의 부분 맵을 돌려준다. 경계값이 포함되는지/제외되는지 헷갈릴 때는 `floor`/`ceiling`(포함)과 `lower`/`higher`(제외)의 차이를 떠올린다.
- 응용: `headMap(80)`(80점 미만)과 `tailMap(90)`(90점 이상)도 출력해, 하위권·상위권을 나눠 보자.

> 답안 비교: `./run.sh SearchCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/SearchCollection.java`

### 과제 6. 우선순위 대기열로 응급실 환자 처리 (PriorityQueue)

응급실에서는 도착 순서가 아니라 **위급도**가 낮은 숫자일수록 먼저 치료한다. 환자를 `(위급도, 이름)` 형태로 `PriorityQueue`에 넣고, 위급도가 **작은 순서대로** 꺼내 치료 순서를 출력하는 프로그램을 작성하라. 예를 들어 `(3,"철수") (1,"영희") (2,"민수") (1,"수지")`를 넣으면, 치료 순서는 위급도 1인 영희·수지가 먼저, 그다음 민수(2), 마지막이 철수(3)여야 한다.

- 힌트: 정수 하나로만 우선순위를 매길 거라면 `PriorityQueue<int[]>`에 `new int[]{위급도, 인덱스}`를 넣고 `PriorityQueue<>(Comparator.comparingInt(a -> a[0]))`로 위급도 기준 비교자를 준다. 또는 위급도를 키로 하는 방식도 가능하지만, **같은 위급도(1이 둘)** 가 있으므로 Map은 부적합함을 확인하자.
- 힌트: `PriorityQueue`는 넣은 순서(FIFO)나 인덱스 순서를 **보장하지 않는다**. 오직 우선순위(비교 결과)만 보장한다. 같은 위급도끼리의 순서는 정해져 있지 않다는 점도 결과로 확인해 보자.
- 응용: 비교자를 `Comparator.reverseOrder()`나 `(a, b) -> b - a`로 바꿔, 위급도가 **큰 값부터** 나오는 최대 우선순위 큐로도 만들어 보자.

> 답안 비교: `./run.sh StackQueueCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/StackQueueCollection.java`

### 과제 7. 스레드 안전 카운터 (동기화 컬렉션)

두 개의 스레드가 각각 같은 리스트에 원소를 1000개씩 `add`한다. 일반 `new ArrayList<>()`로 하면 최종 크기가 2000이 **안 나올 수 있음**(경쟁 상태)을 확인하고, `Collections.synchronizedList(new ArrayList<>())`로 감싸면 항상 2000이 나오는 것을 확인하는 프로그램을 작성하라. 마지막에 리스트의 합계를 구할 때는 **순회를 `synchronized` 블록으로 감싸** 안전하게 구하라.

- 힌트: `Runnable task = () -> { for (int i = 0; i < 1000; i++) list.add(i); };` 를 두 스레드 `t1`, `t2`로 실행하고 `t1.start(); t2.start(); t1.join(); t2.join();` 으로 **끝날 때까지 기다린 뒤** 크기를 출력한다. `join()`을 빠뜨리면 스레드가 끝나기 전에 크기를 읽어 엉뚱한 값이 나온다.
- 힌트: 동기화 리스트라도 `for (int v : list)` 순회 도중 다른 스레드가 수정하면 위험하다. `synchronized (list) { for (int v : list) ... }` 로 감싸야 한다. 성능이 중요한 맵이라면 `synchronizedMap` 대신 `ConcurrentHashMap`을 쓰는 것도 함께 실습해 보자.
- 응용: `ConcurrentHashMap`에 두 스레드가 **같은 키 집합**을 `put`하게 하고, 최종 키 개수가 (두 배가 아니라) 원래 키 개수와 같음을 확인해 보자.

> 답안 비교: `./run.sh SynchronizedCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/SynchronizedCollection.java`

### 과제 8. 설정값을 지키는 불변 컬렉션 (Unmodifiable)

변하면 안 되는 요일 목록을 `List.of("월","화","수","목","금")`으로 만들고, 여기에 `add`를 시도하면 어떤 예외가 나는지 `try-catch`로 잡아 출력하는 프로그램을 작성하라. 이어서 `new ArrayList<>()`에 원소를 채운 `origin`을 `Collections.unmodifiableList(origin)`으로 감싼 `readOnly`를 만들고, (1) `readOnly.set(...)`이 예외를 던지는 것과 (2) **`origin`을 수정하면 `readOnly`에도 반영**되는 것을 각각 확인하라. 마지막으로 `List.copyOf(origin)`은 원본 변경의 영향을 받지 않는 **진짜 불변**임을 보여라.

- 힌트: 두 가지 "불변"을 구분하는 것이 핵심이다. `List.of`/`List.copyOf`는 원본과 **독립된** 불변이고, `Collections.unmodifiableList`는 원본에 연결된 **읽기 전용 뷰**다. 뷰는 자기 자신으로는 수정 못 하지만 원본이 바뀌면 따라 바뀐다.
- 힌트: 예외 타입은 `UnsupportedOperationException`이다. `catch (UnsupportedOperationException e)` 로 잡아 메시지를 출력하면, 프로그램이 죽지 않고 계속 진행되는 것을 볼 수 있다.
- 응용: `readOnly`를 `List.copyOf(origin)`으로 바꾼 뒤 `origin.add(...)`를 해 보고, 이번엔 복사본에 반영되지 **않는** 것을 앞의 뷰 방식과 비교해 보자.

> 답안 비교: `./run.sh UnmodifiableCollection`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch15/UnmodifiableCollection.java`

`Collection`은 "원소 하나하나의 묶음"을 나타내는 인터페이스로 `List`·`Set`·`Queue`가 이를 상속한다. 반면 `Map`은 "키→값 쌍"을 저장하는 별도 계층으로, 원소의 단위가 "하나의 값"이 아니라 "키와 값의 쌍(Entry)"이기 때문에 `Collection`의 `add(E)`·`iterator()` 같은 시그니처가 그대로 들어맞지 않는다. 그래서 Map은 Collection을 상속하지 않는다.

- `values()` : 값만 (Collection)
- `entrySet()` : 키-값 쌍 (`Set<Map.Entry<K,V>>`) — 키와 값을 함께 쓸 때 가장 효율적

| 구분 | 순서 | 중복 | 접근 방식 | 대표 선택 상황 |
|------|------|------|-----------|----------------|
| List | 유지(삽입 순서) | 허용 | 인덱스 | 장바구니, 로그처럼 순서·중복이 의미 있는 목록 |
| Set | 구현체별 | 불허 | 포함 여부(contains) | 방문자 명단, 태그처럼 "있다/없다"만 중요한 집합 |
| Map | 구현체별(키 기준) | 키 불허·값 허용 | 키 | ID→회원, 단어→빈도처럼 키로 값을 찾는 매핑 |

핵심 판단 기준: **순서·중복·인덱스가 필요하면 List**, **중복 제거·존재 확인이 목적이면 Set**, **키로 값을 찾아야 하면 Map**을 고른다.

</details>

**문제 3.** `HashMap`과 `TreeMap`의 차이를 (1) 내부 자료구조, (2) 평균 시간복잡도, (3) 키 순서, (4) 적합한 상황 네 측면에서 비교하시오.

<details>
<summary>정답 및 해설</summary>

| 항목 | HashMap | TreeMap |
|------|---------|---------|
| 내부 자료구조 | 해시 테이블(버킷 배열 + 충돌 시 연결 리스트/트리) | 균형 이진 탐색 트리(레드-블랙 트리) |
| get/put 평균 | O(1) | O(log n) |
| 키 순서 | 보장 안 함 | 키 기준 오름차순 정렬 유지 |
| 적합한 상황 | 순서가 필요 없고 조회가 최대한 빨라야 할 때 | 정렬 상태·범위 검색(`floorKey`·`subMap` 등)이 필요할 때 |

즉 "가장 빠른 단순 조회"는 HashMap, "정렬·범위 검색"은 TreeMap이다. 삽입 순서 유지가 목적이면 `LinkedHashMap`을 쓴다. TreeMap의 O(log n) 원리는 [CS_알고리즘 - 트리와 힙](../CS_알고리즘_개념서/06-트리와-힙.md)에서 다룬다.

</details>

**문제 4.** 다음 상황에서 ArrayList와 LinkedList 중 무엇을 선택해야 하는지 성능 특성을 근거로 고르시오.
- (a) 100만 개의 원소에서 임의의 인덱스로 자주 조회한다.
- (b) 리스트의 맨 앞에 원소를 자주 삽입·삭제한다.

<details>
<summary>정답 및 해설</summary>

- (a) **ArrayList**. 내부가 배열이라 `get(i)`가 O(1)이다. LinkedList는 인덱스 접근 시 앞에서부터 노드를 따라가야 해 O(n)이다.
- (b) **LinkedList**. 맨 앞 삽입·삭제가 O(1)(노드의 링크만 바꿈)이다. ArrayList는 맨 앞에 넣으면 뒤 원소를 전부 한 칸씩 밀어야 해 O(n)이다.

정리: 임의 조회가 잦으면 ArrayList, 양 끝(특히 앞) 삽입·삭제가 잦으면 LinkedList. 다만 실무에서는 캐시 지역성 덕에 ArrayList가 대체로 유리해 기본 선택지로 쓰인다.

</details>

**문제 5.** 같은 입력 `50, 10, 40, 20, 30`을 HashSet, LinkedHashSet, TreeSet에 각각 넣었을 때 출력 순서가 어떻게 다른지 쓰고, 그 이유를 각 구현체의 내부 구조와 연결해 설명하시오.

<details>
<summary>정답 및 해설</summary>

- **HashSet** : 순서 **보장 안 함**. 해시 값에 따라 버킷에 흩어져 저장되므로 삽입·정렬 순서와 무관하다(출력이 JDK/원소에 따라 달라질 수 있다).
- **LinkedHashSet** : `50 10 40 20 30` (넣은 순서). 해시 테이블에 더해 삽입 순서를 잇는 연결 리스트를 유지한다.
- **TreeSet** : `10 20 30 40 50` (오름차순 정렬). 균형 이진 탐색 트리라 항상 정렬 상태를 유지한다.

"순서 무의미→HashSet, 삽입 순서→LinkedHashSet, 정렬→TreeSet"으로 기억한다.

</details>

**문제 6.** 다음 코드의 출력을 예측하시오.

```java
Map<String, Integer> m = new TreeMap<>();
m.put("banana", 2);
m.put("apple", 5);
m.put("cherry", 1);
m.put("apple", 9);
System.out.println(m);
```

<details>
<summary>정답 및 해설</summary>

```
{apple=9, banana=2, cherry=1}
```

두 가지 함정이 있다. (1) **TreeMap이라 키가 사전순 정렬**되어 `apple → banana → cherry` 순으로 나온다. (2) 같은 키 `"apple"`에 두 번 `put`했으므로 **뒤의 값 9가 앞의 5를 덮어쓴다**(키 중복 불가). HashMap이었다면 순서는 보장되지 않지만 값 덮어쓰기는 동일하다.

</details>

**문제 7.** 다음 코드의 출력을 예측하시오.

```java
PriorityQueue<Integer> pq = new PriorityQueue<>();
for (int n : new int[]{40, 10, 30, 20, 50}) pq.offer(n);
System.out.println(pq.peek());
StringBuilder sb = new StringBuilder();
while (!pq.isEmpty()) sb.append(pq.poll()).append(" ");
System.out.println(sb.toString().trim());
```

<details>
<summary>정답 및 해설</summary>

```
10
10 20 30 40 50
```

`PriorityQueue`는 기본적으로 **작은 값이 우선순위가 높다**(min-heap). 따라서 `peek()`은 최솟값 10이고, `poll()`을 반복하면 작은 값부터 차례로 나온다. 함정: **`pq` 자체를 그냥 출력(`System.out.println(pq)`)하면 힙의 내부 배열 순서라 정렬된 모습이 아니다** — 정렬된 순서는 오직 `poll()`로 꺼낼 때만 보장된다. PriorityQueue 기반 힙의 원리는 [CS_알고리즘 - 트리와 힙](../CS_알고리즘_개념서/06-트리와-힙.md) 참고.

</details>

**문제 8.** TreeSet에 `{10, 20, 30, 40, 50}`이 들어 있을 때 다음 결과를 각각 쓰시오.
`floor(35)`, `ceiling(35)`, `floor(30)`, `lower(30)`, `higher(30)`, `subSet(20, 50)`, `headSet(30)`, `tailSet(30)`

<details>
<summary>정답 및 해설</summary>

- `floor(35)` = **30** (35 이하 중 최대)
- `ceiling(35)` = **40** (35 이상 중 최소)
- `floor(30)` = **30** (자기 자신 포함)
- `lower(30)` = **20** (30 미만 중 최대, 자기 제외)
- `higher(30)` = **40** (30 초과 중 최소, 자기 제외)
- `subSet(20, 50)` = **[20, 30, 40]** (20 이상 50 **미만**, 50 제외)
- `headSet(30)` = **[10, 20]** (30 미만)
- `tailSet(30)` = **[30, 40, 50]** (30 이상)

핵심: `floor`/`ceiling`은 경계값 **포함**, `lower`/`higher`는 **제외**. 범위 뷰의 상한 `b`는 **미만**(제외)이다.

</details>

**문제 9.** 다음 코드의 출력을 예측하시오. (unmodifiable 뷰의 함정)

```java
List<String> origin = new ArrayList<>(List.of("A", "B"));
List<String> view = Collections.unmodifiableList(origin);
origin.add("C");
System.out.println(view);
System.out.println(view.size());
```

<details>
<summary>정답 및 해설</summary>

```
[A, B, C]
3
```

`Collections.unmodifiableList`는 원본의 **읽기 전용 뷰**일 뿐, 원본과 독립된 복사본이 아니다. `view`로는 수정할 수 없지만(`view.add(...)`는 예외), **원본 `origin`을 수정하면 뷰에 그대로 비친다**. 원본과 완전히 독립된 불변이 필요하면 `List.copyOf(origin)`을 써야 한다.

</details>

**문제 10.** 다음 코드는 각 줄에서 무엇을 출력하는가? (unmodifiable 예외 함정)

```java
List<Integer> nums = List.of(1, 2, 3);
System.out.println(nums.get(1));   // (a)
try {
    nums.set(0, 99);               // (b)
} catch (Exception e) {
    System.out.println(e.getClass().getSimpleName());
}
```

<details>
<summary>정답 및 해설</summary>

- (a) `2` — `List.of`로 만든 불변 리스트도 **읽기(`get`)는 자유**롭다.
- (b) `UnsupportedOperationException` — 불변 리스트에 `set`/`add`/`remove` 등 수정 연산을 하면 이 예외가 던져진다. (`nums.set(0, 99)` 자체는 실행되지 않고 예외로 빠진다.)

주의: `List.of(...)`에는 `null` 원소를 넣을 수 없어, `null`을 넣으면 생성 시점에 `NullPointerException`이 난다는 점도 함께 기억한다.

</details>

### 오류 찾기

**문제 11.** 다음 코드는 실행 시 예외가 발생한다. 어떤 예외인지, 왜 발생하는지, 어떻게 고쳐야 하는지 설명하시오.

```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4, 5));
for (Integer n : list) {
    if (n % 2 == 0) {
        list.remove(n);      // 짝수를 지우려는 의도
    }
}
```

<details>
<summary>정답 및 해설</summary>

**`ConcurrentModificationException`** 이 발생한다. for-each는 내부적으로 `Iterator`를 쓰는데, 순회 도중에 컬렉션의 구조(크기)를 `list.remove(...)`로 바꾸면 이터레이터가 "예상과 다른 변경(modCount 불일치)"을 감지해 예외를 던진다.

고치는 방법:
1. `Iterator`를 직접 써서 `it.remove()`로 지운다.
   ```java
   Iterator<Integer> it = list.iterator();
   while (it.hasNext()) { if (it.next() % 2 == 0) it.remove(); }
   ```
2. 더 간단히 `list.removeIf(n -> n % 2 == 0);` (JDK 8+)을 쓴다.

또한 `list.remove(n)`은 `Integer`를 넘기면 **값으로**, `int`를 넘기면 **인덱스로** 지운다는 오버로딩 함정도 있으니 주의한다.

</details>

**문제 12.** 다음 코드에서 `set.size()`가 기대와 달리 2가 나온다. 이유와 해결책을 설명하시오.

```java
class Point {
    int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }
}
Set<Point> set = new HashSet<>();
set.add(new Point(1, 1));
set.add(new Point(1, 1));   // 같은 좌표를 또 넣음
System.out.println(set.size());   // 2 ?!
```

<details>
<summary>정답 및 해설</summary>

`Point`가 `equals`와 `hashCode`를 재정의하지 않았기 때문이다. `HashSet`은 `hashCode`로 버킷을 찾고 `equals`로 같은 값인지 판별하는데, 재정의가 없으면 `Object`의 기본 구현(객체 identity 기준)을 쓴다. 두 `new Point(1,1)`은 **서로 다른 객체**이므로 다른 것으로 취급되어 둘 다 저장되어 `size()`가 2가 된다.

해결: `Point`에 `equals`와 `hashCode`를 **함께** 재정의한다(둘 중 하나만 재정의하면 계약 위반). 필드 `x`, `y` 기준으로 재정의하면 같은 좌표는 하나로 취급되어 `size()`가 1이 된다. `record Point(int x, int y) {}`로 선언하면 두 메소드가 자동 생성된다.

</details>

### 참·거짓 판별

**문제 14.** 다음 명제의 참·거짓을 판별하고 **이유**를 쓰시오.

1. "`HashMap`은 키를 넣은 순서대로 순회된다."
2. "`Map`은 `Collection` 인터페이스를 상속한다."
3. "`TreeSet`의 `subSet(a, b)`는 `a` 이상 `b` 이하 원소를 돌려준다."
4. "`Collections.synchronizedList`로 감싸면 for-each 순회도 별도 조치 없이 스레드 안전하다."

<details>
<summary>정답 및 해설</summary>

1. **거짓**. HashMap은 순서를 보장하지 않는다. 삽입 순서를 유지하려면 `LinkedHashMap`을 쓴다.
2. **거짓**. Map은 Collection을 상속하지 않는 **별도 계층**이다. 그래서 `iterator()`가 없고 `keySet`/`values`/`entrySet`으로 순회한다.
3. **거짓**. `subSet(a, b)`는 `a` **이상** `b` **미만**(상한 제외)이다. 상한을 포함하려면 `subSet(a, true, b, true)` 오버로드를 쓴다.
4. **거짓**. 개별 메소드 호출은 락으로 보호되지만, **순회(for-each) 중에는 직접 `synchronized (list) { ... }` 블록으로 감싸야** 안전하다. 감싸지 않으면 순회 도중 다른 스레드의 수정으로 문제가 생길 수 있다.

</details>

**문제 15.** `List.of("a", "b")`로 만든 리스트에 `add`를 호출하면 어떤 예외가 발생하는가? 또한 `Collections.unmodifiableList(원본)`으로 만든 읽기 전용 뷰가 완전한 불변이 아닌 이유를 설명하고, 완전한 불변 리스트를 만드는 방법을 제시하시오.

<details>
<summary>정답 및 해설</summary>

- `List.of(...)`로 만든 리스트에 `add`(또는 `set`/`remove`)를 호출하면 **`UnsupportedOperationException`** 이 발생한다. 이 리스트는 수정 연산을 지원하지 않는다.
- `Collections.unmodifiableList(원본)`은 원본의 **읽기 전용 뷰**일 뿐 독립된 복사본이 아니다. 뷰 자신으로는 수정할 수 없지만 **원본을 수정하면 뷰에 그대로 반영**되므로, 완전한 불변이 아니다.
- 완전한 불변 리스트는 처음부터 `List.of(...)`로 만들거나, 기존 컬렉션을 `List.copyOf(원본)`으로 **복사**해서 만든다. 이렇게 하면 이후 원본이 바뀌어도 영향을 받지 않는다.

</details>

## 요약

- **컬렉션 프레임워크**는 `java.util`이 제공하는 표준 자료구조 라이브러리로, 배열과 달리 크기가 동적이고 편의 메소드를 갖춘다. 크게 **Collection**(List·Set·Queue)과 **Map**(키-값) 두 계층으로 나뉜다.
- **List**는 순서 O·중복 O. 인덱스 조회가 잦으면 `ArrayList`(O(1)), 앞/뒤 삽입·삭제가 잦으면 `LinkedList`(O(1)).
- **Set**은 중복 X. 순서 무의미하면 `HashSet`, 삽입 순서 유지하면 `LinkedHashSet`, 정렬 유지하면 `TreeSet`.
- **Map**은 키→값 저장(키 중복 X). 순서 규칙은 Set과 동일하게 Hash/Linked/Tree로 갈린다. `getOrDefault`·`containsKey`와 `keySet`/`values`/`entrySet` 순회를 기억한다.
- **TreeSet/TreeMap**은 정렬을 유지해 `floor`/`ceiling`/`lower`/`higher`와 `headSet`/`tailSet`/`subSet` 같은 **범위·근사 검색**에 강하다.
- **Stack(LIFO)** 과 **Queue(FIFO)** 는 `ArrayDeque`로 구현하고, **PriorityQueue**는 우선순위 순으로 꺼낸다.
- 멀티 스레드 환경에서는 `Collections.synchronizedXxx` 래퍼나 `ConcurrentHashMap` 같은 **동기화 컬렉션**을 쓴다.
- `List.of`/`Set.of`/`Map.of`와 `Collections.unmodifiableXxx`로 **불변(수정 불가) 컬렉션**을 만들 수 있으며, 완전한 불변에는 `copyOf`를 쓴다.
