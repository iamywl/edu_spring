## 21.1 강화된 내용 개요

**자바 21**은 자바 17에 이은 LTS 버전으로, 그동안 프리뷰 상태로 실험되던 언어·라이브러리 기능들을 정식으로 확정했다. 6개월 주기의 일반 버전과 달리 LTS는 오랫동안 지원되므로, 실무 프로젝트가 기준으로 삼기에 적합하다.

자바 21에서 정식화된 주요 변화는 다음과 같다.

| 구분 | 기능 | 핵심 |
|------|------|------|
| 언어 | `switch`의 `null` 처리 | `case null`, `case null, default` 지원 |
| 언어 | `switch` 패턴 매칭 | 타입 패턴 + `when` 가드 |
| 언어 | 레코드 패턴 | `case Point(int x, int y)`로 분해 |
| 라이브러리 | 가상 스레드 | 경량 스레드로 대규모 동시성 |
| 라이브러리 | 순차 컬렉션 | `getFirst`/`getLast`/`reversed` 통일 API |
| 런타임 | 기본 문자셋 | UTF-8로 통일(자바 18~) |

이후 소절에서 이 기능들을 하나씩 개념과 실습으로 확인한다. 실행 중인 JDK 버전은 코드에서 `System.getProperty("java.version")`으로 직접 확인할 수 있다.

## 21.2 지역 변수 타입 추론 (var)

`var`는 자바 10부터 도입된 **지역 변수 타입 추론(Local Variable Type Inference)** 키워드다. 변수의 타입을 직접 적는 대신 `var`라고 쓰면, 컴파일러가 **우변(초기화 식)을 보고 타입을 자동으로 결정**한다.

가장 중요한 점은 `var`가 **컴파일 타임에 타입이 확정되는 정적 타입**이라는 것이다. 자바스크립트의 `var`처럼 런타임에 타입이 바뀌는 동적 타입이 결코 아니며, 타입 안전성은 그대로 유지된다. 따라서 `var number = 10;`으로 선언한 변수에 나중에 문자열을 넣으려 하면 컴파일 오류가 난다.

```java
var message = "안녕하세요";                 // String으로 추론
var count = 42;                              // int로 추론
var scores = new HashMap<String, List<Integer>>();  // 긴 타입 중복을 제거

for (var fruit : List.of("사과", "바나나")) {  // fruit은 String으로 추론
    System.out.println(fruit);
}
```

`var`에는 몇 가지 규칙이 있다. **지역 변수에서만** 쓸 수 있고(필드·매개변수·반환 타입 불가), **선언과 동시에 초기화**해야 한다(추론할 근거가 필요하기 때문). 또한 `var`는 우변에서 타입이 명확할 때 가독성을 높이지만, `var x = getValue();`처럼 우변이 메서드 호출이라 타입이 감춰질 때는 오히려 읽기 어려워지므로 명시적 타입이 낫다.

## 21.3 switch의 null 처리

전통적인 `switch` 문에 `null`을 넘기면 `NullPointerException`이 발생했다. 그래서 개발자는 `switch` 앞에 항상 별도의 `null` 검사 코드를 두어야 했다. 자바 21은 이 불편을 없애, **`switch`의 `case`로 `null`을 직접 다룰 수 있게** 했다.

두 가지 방식이 있다. `case null`은 `null`일 때 실행할 분기를 **별도로** 명시한다. `case null, default`는 `null`과 나머지 모든 경우를 **하나의 분기로 묶는다**. 이 기능은 화살표(`->`) 스타일 `switch`에서 사용한다.

```java
String describeDay(String day) {
    return switch (day) {
        case "MON", "TUE", "WED", "THU", "FRI" -> "평일";
        case "SAT", "SUN" -> "주말";
        case null -> "요일 정보 없음(null)";   // null을 NPE 없이 처리
        default -> "알 수 없는 요일";
    };
}
```

주의할 점은, `case null`을 **작성했을 때만** 안전하다는 것이다. `case null`이 없는 `switch`에 `null`을 넘기면 자바 21에서도 여전히 `NullPointerException`이 발생한다. 즉 `null` 처리는 자동이 아니라 명시적으로 선언해야 얻어지는 안전장치다.

## 21.4 switch 패턴 매칭

자바 21에서 정식화된 **switch 패턴 매칭**은 `case`에 값 상수뿐 아니라 **타입 패턴**을 쓸 수 있게 한다. `case 타입 변수` 형태로 작성하면, 값이 그 타입일 때 매칭되면서 **동시에 그 타입으로 캐스팅된 변수를 바로 사용**할 수 있다. 예전의 `if - else if + instanceof + 명시적 캐스팅` 사슬을 간결하고 안전한 `switch` 하나로 대체한다.

```java
String describe(Object obj) {
    return switch (obj) {
        case null -> "널 값";
        case Integer i -> "정수: " + i;                       // i는 Integer로 바인딩
        case String s -> "문자열: \"" + s + "\" (길이 " + s.length() + ")";
        default -> "기타 타입: " + obj.getClass().getSimpleName();
    };
}
```

여기에 **`when` 가드**를 붙이면 타입이 같아도 조건으로 세분화할 수 있다. `case 타입 변수 when 조건` 형태이며, 타입이 맞더라도 `when` 조건이 참일 때만 그 분기가 선택된다. 가드는 **위에서부터 순서대로** 검사되므로 좁은 조건을 먼저 배치해야 한다.

```java
String classifyNumber(Object obj) {
    return switch (obj) {
        case Integer i when i < 0   -> "음수";
        case Integer i when i == 0  -> "영(0)";
        case Integer i when i < 100 -> "작은 양수";
        case Integer i             -> "큰 양수(100 이상)";
        default -> "정수가 아님";
    };
}
```

패턴 매칭 `switch`는 컴파일러가 **분기 완전성(exhaustiveness)**을 검사해준다는 장점도 있다.

## 21.5 레코드 패턴

**레코드 패턴(Record Pattern)**은 레코드 인스턴스를 그 **구성요소(component)로 분해(deconstruct)**하면서 매칭하는 문법으로, 자바 21에서 정식화되었다. `case Point(int x, int y)`처럼 쓰면 값이 `Point`일 때 매칭되면서 **동시에 `x`, `y` 변수에 각 구성요소가 바로 추출**된다. `instanceof` 확인 뒤 getter를 반복 호출하던 코드가 한 줄로 줄어든다.

```java
record Point(int x, int y) {}
record Line(Point start, Point end) {}

Object obj = new Point(3, 4);
if (obj instanceof Point(int x, int y)) {   // x, y로 즉시 분해
    System.out.println("x=" + x + ", y=" + y);
}
```

레코드가 레코드를 품고 있으면 **중첩 분해**도 가능하다. 아래처럼 `Line`을 두 `Point`로, 다시 각 좌표로 한 번에 풀어낼 수 있으며, 구성요소 타입을 `var`로 추론하게 두면 더 간결하다.

```java
Object something = new Line(new Point(0, 0), new Point(3, 4));
if (something instanceof Line(Point(var x1, var y1), Point(var x2, var y2))) {
    double len = Math.sqrt(Math.pow(x2 - x1, 2) + Math.pow(y2 - y1, 2));
    System.out.println("길이 = " + len);
}
```

레코드 패턴은 `switch` 패턴 매칭·`when` 가드와 결합해 **데이터 구조 기반 분기**를 아주 간결하게 만든다. 예컨대 `case Point(int x, int y) when x == 0 && y == 0 -> "원점"`처럼 분해와 조건을 동시에 표현할 수 있다.

## 21.6 가상 스레드

**가상 스레드(Virtual Thread)**는 자바 21에서 정식화된 **경량 스레드**다. 기존 **플랫폼 스레드**는 운영체제 스레드에 1:1로 매핑되어 하나 만드는 비용이 크고 개수도 제한적이다. 반면 가상 스레드는 **JVM이 관리하는 아주 가벼운 스레드**여서 수천~수백만 개를 만들어도 부담이 적다. 특히 **I/O 대기가 많은** 작업에서 진가를 발휘하는데, 대기 중에는 실제 운영체제 스레드(캐리어 스레드)를 점유하지 않기 때문이다.

생성 방법은 두 가지다. `Thread.ofVirtual().start(Runnable)`로 하나를 직접 만들거나, `Executors.newVirtualThreadPerTaskExecutor()`로 **작업마다 가상 스레드를 하나씩 할당**하는 `ExecutorService`를 쓴다. 후자는 대량 작업 제출에 편리하다.

```java
// [1] 하나를 직접 생성
Thread vt = Thread.ofVirtual().start(() ->
    System.out.println("isVirtual=" + Thread.currentThread().isVirtual()));
vt.join();   // 끝날 때까지 대기

// [2] 작업마다 가상 스레드 할당 (try-with-resources로 안전하게 종료)
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1000; i++) {
        executor.submit(() -> { /* 짧은 I/O 흉내 */ });
    }
}   // 블록 종료 시 모든 작업 완료를 기다린 뒤 종료
```

**정상 종료**에 유의해야 한다. `try-with-resources`로 `ExecutorService`를 닫거나, `shutdown()` 후 `awaitTermination(...)`으로 모든 작업이 끝날 때까지 기다린 뒤 종료해야 프로그램이 hang 없이 깔끔하게 끝난다.

## 21.7 순차 컬렉션

**순차 컬렉션(Sequenced Collections)**은 자바 21에서 추가된 컬렉션 인터페이스 계층으로, **원소에 정해진 순서(첫 번째 ~ 마지막)가 있는** 컬렉션들에게 **통일된 API**를 제공한다. 예전에는 첫/마지막 원소를 얻는 방법이 컬렉션마다 제각각이었다. `List`는 `list.get(0)`과 `list.get(size-1)`을 썼지만 다른 컬렉션은 방식이 달랐다. 이제 어떤 순차 컬렉션이든 `getFirst()`/`getLast()`로 동일하게 접근한다.

새 인터페이스는 세 가지다.

- `SequencedCollection` : 순서 있는 컬렉션(예: `List`) — `getFirst()`, `getLast()`, `addFirst()`, `addLast()`, `removeFirst()`, `removeLast()`, `reversed()`
- `SequencedSet` : 순서 있는 집합(예: `LinkedHashSet`)
- `SequencedMap` : 순서 있는 맵(예: `LinkedHashMap`) — `firstEntry()`, `lastEntry()`, `putFirst()`, `putLast()`, `reversed()`

```java
SequencedCollection<String> seq = new ArrayList<>(List.of("B", "C", "D"));
seq.addFirst("A");            // 맨 앞에 추가
seq.addLast("E");             // 맨 뒤에 추가
seq.getFirst();               // "A"
seq.getLast();                // "E"
seq.reversed();               // [E, D, C, B, A] (역순 뷰)
```

`reversed()`가 돌려주는 것은 **역순 뷰**로, 원본과 연결되어 있어 별도 복사 없이 역순으로 순회할 수 있다. 순차 컬렉션 덕분에 "첫 원소·마지막 원소·역순"이라는 자주 쓰는 동작이 컬렉션 종류와 무관하게 하나의 API로 통일되었다.

## 21.8 기본 문자셋

자바 18(JEP 400)부터 표준 자바 API의 **기본 문자셋(Default Charset)**이 **UTF-8로 통일**되었고, 이 변경은 자바 21에도 그대로 적용된다. 즉 `Charset.defaultCharset()`이 실행 환경(OS·로케일)과 무관하게 항상 UTF-8을 반환한다.

이 변화가 중요한 이유는 **이식성과 예측 가능성**이다. 예전에는 기본 문자셋이 OS·로케일에 따라 달랐다(윈도우 한글 환경은 MS949/CP949, 리눅스는 UTF-8 등). 그래서 같은 코드가 개발 PC와 서버에서 다르게 동작하고, 파일을 읽고 쓸 때 **한글이 깨지는** 문제가 흔했다. 이제 기본이 UTF-8로 통일되어 "어디서 실행하든 동일하게 동작"한다.

```java
Charset def = Charset.defaultCharset();
System.out.println(def);                                // UTF-8 (자바 18+)
System.out.println(def.equals(StandardCharsets.UTF_8)); // true

String korean = "한글 テスト émoji";
byte[] bytes = korean.getBytes(StandardCharsets.UTF_8); // 문자셋 명시가 가장 안전
String restored = new String(bytes, StandardCharsets.UTF_8);
System.out.println(korean.equals(restored));            // true
```

다만 기본값이 UTF-8로 바뀌었더라도, **실무에서는 문자셋을 명시하는 습관**이 가장 안전하다. 파일 입출력이나 인코딩/디코딩에서 `StandardCharsets.UTF_8`을 직접 지정하면 JDK 버전이나 실행 환경에 관계없이 언제나 같은 결과를 얻는다. (필요하면 `file.encoding` 시스템 속성으로 옛 동작을 흉내 낼 수도 있으나 권장되지 않는다.)

## ⚠️ 흔한 오해와 함정

- **`var`가 동적 타입이라고 오해한다.** `var`는 자바스크립트의 `var`와 전혀 다르다. 컴파일 타임에 타입이 **정적으로 확정**되며, 다른 타입 값을 다시 대입하면 컴파일 오류가 난다. 또 지역 변수에서만 쓸 수 있고 반드시 초기화와 함께 선언해야 한다.
- **자바 21이면 `switch`가 자동으로 `null`을 처리해준다고 생각한다.** 그렇지 않다. `case null`을 **직접 작성했을 때만** 안전하며, `case null`이 없으면 `null` 입력 시 여전히 `NullPointerException`이 발생한다.
- **`when` 가드의 순서를 신경 쓰지 않는다.** 패턴 매칭 `case`는 위에서부터 순서대로 검사된다. 넓은 조건을 위에 두면 아래의 좁은 조건이 영영 선택되지 않는다. 좁은(구체적인) 조건을 먼저 배치해야 한다.
- **레코드 패턴과 일반 타입 패턴을 혼동한다.** `case Point p`는 값을 `Point` 타입 변수 `p` 하나로 받지만, `case Point(int x, int y)`는 **구성요소로 분해**해 `x`, `y`를 각각 꺼낸다. 분해가 필요할 때만 괄호 형태를 쓴다.
- **가상 스레드를 만들고 종료를 기다리지 않는다.** `ExecutorService`를 `try-with-resources`로 닫거나 `shutdown()` + `awaitTermination(...)`으로 완료를 기다리지 않으면, 작업이 끝나기 전에 프로그램이 종료되거나 hang이 발생할 수 있다.
- **가상 스레드가 CPU 연산도 무조건 빠르게 한다고 믿는다.** 가상 스레드의 강점은 **I/O 대기가 많은** 작업의 대규모 동시성이다. CPU를 계속 쓰는 계산 작업은 물리 코어 수의 한계를 그대로 받으므로 개수를 늘린다고 빨라지지 않는다.
- **기본 문자셋이 UTF-8이 되었으니 문자셋을 신경 안 써도 된다고 여긴다.** 예전 JDK나 특정 설정에서는 여전히 다를 수 있다. 파일·네트워크 입출력에서는 `StandardCharsets.UTF_8`을 **명시**하는 것이 가장 안전하다.

`SequencedCollection<String>` 타입으로 `new ArrayList<>(List.of("B", "C", "D"))`를 만든 뒤, `addFirst("A")`와 `addLast("E")`로 양끝에 원소를 추가하라. 그다음 `getFirst()`, `getLast()`, `reversed()`를 호출해 각각 **첫 원소**, **마지막 원소**, **역순 뷰**를 출력하라. 끝으로 `reversed()`가 원본과 연결된 **뷰**임을 확인하기 위해, 역순 뷰를 변수에 담아두고 원본에 `addLast("F")`를 한 뒤 그 역순 뷰를 다시 출력해 변화가 반영되는지 관찰하라.

<details>
<summary>힌트</summary>

**과제 5. 다수 작업을 처리하는 가상 스레드 (Virtual Thread · 정상 종료)** — 도전

`Executors.newVirtualThreadPerTaskExecutor()`로 만든 `ExecutorService`에 짧은 작업 **1000개**를 `submit`하라. 각 작업은 `Thread.sleep(1)`로 짧은 I/O 대기를 흉내 내고 `AtomicInteger`로 완료 개수를 세도록 한다. 첫 작업(id == 0)에서는 `Thread.currentThread().isVirtual()`이 `true`인지 출력하라. 마지막으로 **모든 작업이 끝날 때까지 반드시 기다린 뒤** 종료하고, "제출한 작업 수"와 "완료한 작업 수"가 같은지 출력해 확인하라.

<details>
<summary>힌트</summary>

- 완료를 기다리지 않고 프로그램이 끝나면 작업이 유실되거나 hang이 날 수 있다. **정상 종료**가 이 과제의 핵심이다.
- 정상 종료는 두 방법 중 하나로: (1) `try (ExecutorService ex = ...) { ... }` **try-with-resources** — 블록 종료 시 자동으로 완료를 기다린다. (2) `ex.shutdown()` 후 `ex.awaitTermination(10, TimeUnit.SECONDS)` — 반환값이 `true`면 모두 정상 종료된 것이다. 이 과제에서는 **`awaitTermination`으로 명시적으로 종료**해보자.
- 완료 카운트는 여러 스레드가 동시에 증가시키므로 일반 `int`가 아니라 `AtomicInteger.incrementAndGet()`을 써야 정확하다.
</details>

**과제 6. null도 안전하게 다루는 switch (switch의 null 처리)** — 쉬움

문자열 요일을 받아 설명을 돌려주는 `describeDay(String day)`와 색상 코드를 돌려주는 `colorCode(String color)`를 **화살표(`->`) 스타일 switch**로 작성하라. (1) `describeDay`는 평일/주말을 각각 묶어 처리하고, `null`은 `case null`로 **별도 분기**해 "요일 정보 없음"을 돌려준다. (2) `colorCode`는 `"RED"`, `"GREEN"`, `"BLUE"`를 각각 코드로 매핑하고, `null`과 그 외 모든 값을 `case null, default`로 **한 분기에 묶어** 기본 코드를 돌려준다. 두 메소드에 정상 값과 `null`을 각각 넣어 결과를 출력하라. 마지막으로 `case null`이 **없는** switch에 `null`을 넣으면 `NullPointerException`이 난다는 것을 `try-catch`로 잡아 확인하라.

<details>
<summary>힌트</summary>

- `case null`은 `null`일 때만 실행할 분기를 **별도로** 명시한다. `case null, default`는 `null`과 나머지 전부를 **하나로 묶는다**. 둘 중 어떤 방식인지 과제 지시대로 골라 쓴다.
- `case null`을 두지 않은 switch에 `null`을 넣으면 자바 21에서도 여전히 `NullPointerException`이 난다. 이 점을 `try-catch`로 직접 확인해보는 것이 이 과제의 핵심이다.
- 이 기능은 **화살표(`->`) 스타일** switch에서 쓴다.
</details>

> 답안 비교: `./run.sh SwitchNull`  📁 `chapter-java-textbook/src/main/java/com/edu/javabook/ch21/SwitchNull.java`

</details>

<summary>정답 및 해설</summary>

- **정식화된 언어 기능**(두 가지 이상): ① `switch`의 `null` 처리(`case null`), ② `switch` 패턴 매칭(타입 패턴 + `when` 가드), ③ 레코드 패턴.
- **정식화된 라이브러리/런타임 기능**(두 가지 이상): ① 가상 스레드, ② 순차 컬렉션(`SequencedCollection`/`Set`/`Map`), ③ 기본 문자셋 UTF-8 통일(자바 18~).
</details>

**2. (개념 서술)** `var`는 정적 타입인가 동적 타입인가? 그 근거를 자바스크립트의 `var`와 비교해 설명하고, `var`를 **언제 쓰면 좋고 언제 쓰지 말아야** 하는지 각각 하나씩 예로 드시오.

<details>
<summary>정답 및 해설</summary>

- `var`는 **정적 타입**이다. 컴파일러가 우변(초기화 식)을 보고 타입을 **컴파일 타임에 확정**하며, 이후 다른 타입 값을 대입하면 컴파일 오류가 난다. 자바스크립트의 `var`는 런타임에 타입이 바뀌는 동적 타입이지만, 자바의 `var`는 이름만 같을 뿐 완전히 다르다. 타입 안전성은 그대로 유지된다.
- **쓰면 좋은 예**: 우변에서 타입이 명확할 때. 예) `var scores = new HashMap<String, List<Integer>>();` — 좌우변 타입 중복을 없애 가독성이 좋아진다.
- **쓰지 말아야 하는 예**: 우변이 메서드 호출이라 타입이 감춰질 때. 예) `var x = getValue();` — 반환 타입이 코드에 드러나지 않아 읽기 어려워진다. 이때는 명시적 타입이 낫다.
</details>

**3. (오류 찾기)** 다음 코드에서 컴파일 오류가 나는 줄을 모두 찾고, 각각 왜 오류인지 설명하시오.
```java
var number = 10;
number = "문자열";              // (가)
var empty;                       // (나)
var result = null;               // (다)
for (var i = 0; i < 3; i++) {    // (라)
    System.out.println(i);
}
```

<details>
<summary>정답 및 해설</summary>

오류인 줄은 (가), (나), (다) 세 줄이다.

- **(가)** `number`는 `10`으로 `int`로 확정되었다. 정적 타입이므로 `String` 값을 대입할 수 없다 → 타입 불일치 컴파일 오류.
- **(나)** `var`는 **선언과 동시에 초기화**해야 한다. 초기화 식이 없으면 컴파일러가 타입을 추론할 근거가 없다 → 오류.
- **(다)** `var result = null;`은 `null`만으로는 타입을 추론할 수 없다 → 오류. (명시적으로 `String result = null;`처럼 써야 한다.)
- **(라)** 정상이다. `for`의 초기화식에서 `var i = 0`은 `int`로 추론되며 지역 변수이므로 문제없다.
</details>

**4. (출력 판단)** 다음 코드의 출력은 무엇인가?
```java
var list = List.of("사과", "바나나", "체리");
var map = new HashMap<String, Integer>();
System.out.println(list.getClass().getSimpleName());
System.out.println(map.getClass().getSimpleName());
```

<details>
<summary>정답 및 해설</summary>

```
ListN
HashMap
```

- `List.of(...)`가 3개 이상 원소로 만드는 불변 리스트의 실제 구현 클래스는 `ImmutableCollections.ListN`이고, `getSimpleName()`은 `ListN`을 돌려준다. 우리가 흔히 아는 `ArrayList`가 아니라는 점을 눈으로 확인하는 것이 핵심이다.
- `new HashMap<>()`의 실제 타입은 `HashMap`이므로 그대로 `HashMap`이 출력된다.
- 요점: `var`는 **정적 타입**을 추론할 뿐, `getClass()`가 돌려주는 것은 **런타임의 실제 구현 클래스**다. 둘을 혼동하지 말자. (`LocalVarInference` 참고)
</details>

**5. (빈칸 채우기)** 다음은 `switch`의 `null`을 안전하게 처리하는 코드다. 빈칸 ㉠, ㉡을 채워 완성하시오.
```java
String describeDay(String day) {
    return switch (day) {
        case "MON", "TUE", "WED", "THU", "FRI" -> "평일";
        case "SAT", "SUN" -> "주말";
        case ㉠ -> "요일 정보 없음";        // null을 별도 분기로 처리
        ㉡ -> "알 수 없는 요일";
    };
}
```

<details>
<summary>정답 및 해설</summary>

- ㉠ = `null`
- ㉡ = `default`

`case null`을 별도로 두면 `null` 입력을 `NullPointerException` 없이 안전하게 처리한다. 나머지 모든 값은 `default`가 받는다. 만약 `null`과 나머지를 **하나의 분기로 묶고** 싶으면 `case null, default -> ...`로 쓸 수도 있다. (`SwitchNull` 참고)
</details>

**6. (컴파일/실행 판단)** 자바 21에서 다음 `switch`에 `null`을 넘기면 어떻게 되는가? 컴파일은 되는지, 실행 시 무슨 일이 일어나는지 설명하고, 안전하게 만들려면 어떤 `case`를 추가해야 하는지 쓰시오.
```java
String r = switch (s) {   // s == null
    case "A" -> "에이";
    default -> "기타";
};
```

<details>
<summary>정답 및 해설</summary>

- **컴파일은 정상적으로 된다.** `case null`이 없어도 문법 오류는 아니다.
- 하지만 `s`가 `null`이면 **실행 시 `NullPointerException`이 발생**한다. 자바 21이라도 `switch`가 자동으로 `null`을 처리해주지는 않는다.
- 안전하게 하려면 `case null -> ...`을 추가하거나, `default -> ...`를 `case null, default -> ...`로 바꿔 `null`을 함께 묶으면 된다.
- 요점: `null` 안전 처리는 **명시적으로 `case null`을 작성했을 때만** 얻어지는 안전장치다. (`SwitchNull` 참고)
</details>

**7. (개념 서술)** `switch` 패턴 매칭에서 **타입 패턴**과 **`when` 가드**는 각각 무엇을 하는가? `when` 가드가 있는 `case`들의 **순서**가 왜 중요한지 예를 들어 설명하시오.

<details>
<summary>정답 및 해설</summary>

- **타입 패턴** (`case Integer i ->`): 값이 그 타입일 때 매칭되면서 **동시에 그 타입으로 캐스팅된 변수 `i`를 바로 사용**하게 해준다. `instanceof` + 명시적 캐스팅 사슬을 대체한다.
- **`when` 가드** (`case Integer i when i < 0 ->`): 타입이 같아도 **조건이 참일 때만** 그 분기를 선택하게 한다. 타입 + 값 조건을 함께 표현한다.
- **순서가 중요한 이유**: `case`는 **위에서부터 순서대로** 검사되므로, 넓은(포괄적인) 조건을 위에 두면 아래의 좁은 조건이 영영 선택되지 않는다. 예를 들어 `case Integer i ->`(가드 없는 포괄 케이스)를 `case Integer i when i < 0 ->`보다 **위에** 두면 음수 케이스가 절대 실행되지 않는다. 좁은 조건을 먼저 배치해야 한다. (`SwitchPattern` 참고)
</details>

**8. (출력 판단)** 다음 `classifyNumber`에 `-5`, `0`, `7`, `1000`을 각각 넣으면 무엇이 출력되는가?
```java
static String classifyNumber(Object obj) {
    return switch (obj) {
        case Integer i when i < 0   -> "음수";
        case Integer i when i == 0  -> "영(0)";
        case Integer i when i < 100 -> "작은 양수";
        case Integer i              -> "큰 양수(100 이상)";
        default -> "정수가 아님";
    };
}
```

<details>
<summary>정답 및 해설</summary>

- `-5` → `음수` (첫 가드 `i < 0`이 참)
- `0` → `영(0)` (`i == 0`)
- `7` → `작은 양수` (`i < 100`)
- `1000` → `큰 양수(100 이상)` (앞의 가드가 모두 거짓 → 가드 없는 포괄 케이스)

가드가 **위에서부터** 검사되므로, 좁은 조건(`< 0`, `== 0`, `< 100`)을 먼저 두고 가장 포괄적인 케이스를 마지막에 두었다. (`SwitchPattern` 참고)
</details>

**9. (출력 판단 — 레코드 패턴 분해)** 다음 코드의 출력은 무엇인가?
```java
record Point(int x, int y) {}
record Line(Point start, Point end) {}

Object obj = new Line(new Point(0, 0), new Point(3, 4));
if (obj instanceof Line(Point(var x1, var y1), Point(var x2, var y2))) {
    double len = Math.sqrt(Math.pow(x2 - x1, 2) + Math.pow(y2 - y1, 2));
```

<details>
<summary>정답 및 해설</summary>

```
길이 = 5.0
```

- 거리 = √((3−0)² + (4−0)²) = √(9 + 16) = √25 = 5.0. (3-4-5 직각삼각형)
- 요점: `instanceof`/`switch` 확인 뒤 getter를 반복 호출하지 않고, **한 줄로 중첩 분해**할 수 있다. (`RecordPattern` 참고)

**10. (빈칸 채우기 — 레코드 패턴)** 다음은 좌표를 분해하며 원점을 따로 구분하는 코드다. 빈칸 ㉠, ㉡을 채우시오. (㉠은 원점 여부를 검사하는 부분, ㉡은 일반 점을 분해하는 패턴이다.)
```java
record Point(int x, int y) {}

static String describe(Object obj) {
    return switch (obj) {
        case Point(int x, int y) ㉠ -> "원점(0,0)";
        case ㉡ -> "점 (" + x + ", " + y + ")";
        default -> "알 수 없는 도형";
    };
}
```

<details>
<summary>정답 및 해설</summary>

- ㉠ = `when x == 0 && y == 0`
- ㉡ = `Point(int x, int y)`

원점을 구분하는 가드 케이스를 **먼저** 두고, 그 아래에 일반 점을 분해하는 케이스를 둔다. 순서가 바뀌면(`Point(int x, int y)`가 위로 가면) 원점 케이스가 절대 선택되지 않는다. 레코드 패턴 `Point(int x, int y)`는 값을 `Point p` 하나로 받는 `case Point p`와 달리 **구성요소 `x`, `y`로 분해**한다는 점을 기억하자. (`RecordPattern` 참고)
</details>

**11. (출력 판단 — 순차 컬렉션)** 다음 코드의 출력을 순서대로 쓰시오.
```java
SequencedCollection<String> seq = new ArrayList<>(List.of("B", "C", "D"));
seq.addFirst("A");
seq.addLast("E");
System.out.println(seq.getFirst());
System.out.println(seq.getLast());
System.out.println(seq.reversed());
```

<details>
<summary>정답 및 해설</summary>

```
A
E
[E, D, C, B, A]
```

- 초기 `[B, C, D]`에 `addFirst("A")` → `[A, B, C, D]`, `addLast("E")` → `[A, B, C, D, E]`.

- `reversed()`는 **역순 뷰**로 `[E, D, C, B, A]`.
- 요점: `List`도 자바 21에서 `SequencedCollection`을 구현하므로 `getFirst`/`getLast`/`addFirst`/`addLast`/`reversed`를 통일된 API로 쓸 수 있다. (`SequencedCollection` 참고)
</details>

**12. (참·거짓 + 이유 — 순차 컬렉션 뷰)** 다음 주장이 참인지 거짓인지 판정하고 이유를 쓰시오.
> "`reversed()`는 원본을 역순으로 **복사한 새 컬렉션**을 돌려주므로, 뷰를 받아둔 뒤 원본에 원소를 추가해도 그 뷰에는 반영되지 않는다."

<details>
<summary>정답 및 해설</summary>

**거짓.**

`reversed()`가 돌려주는 것은 **복사본이 아니라 역순 뷰(view)**다. 원본과 연결되어 있어 별도 복사 없이 역순으로 순회하며, **원본을 수정하면 그 변화가 뷰에도 반영된다**. 예를 들어 `var rev = seq.reversed();` 뒤에 `seq.addLast("F")`를 하면, `rev`를 다시 출력했을 때 맨 앞에 `F`가 나타난다(역순이므로). 뷰이기 때문에 메모리를 절약하고 원본과 항상 동기화된다는 점이 핵심이다. (`SequencedCollection` 참고)
</details>

**13. (개념 서술 — 가상 스레드 vs 플랫폼 스레드)** 가상 스레드와 플랫폼 스레드의 차이를 설명하고, 가상 스레드가 특히 유리한 작업 종류와, 반대로 이점이 크지 않은 작업 종류를 각각 쓰시오.

<details>
<summary>정답 및 해설</summary>

- **플랫폼 스레드**는 운영체제 스레드에 **1:1로 매핑**되어 하나 만드는 비용이 크고 개수가 제한적이다.
- **가상 스레드**는 **JVM이 관리하는 경량 스레드**로, 수천~수백만 개를 만들어도 부담이 적다. I/O 대기 중에는 실제 운영체제 스레드(캐리어 스레드)를 점유하지 않고 양보한다.
- **유리한 작업**: **I/O 대기가 많은** 작업(네트워크 호출, DB 조회, 파일 입출력 등)의 대규모 동시성. 대기 시간에 캐리어 스레드를 놓아주므로 적은 OS 스레드로 많은 작업을 소화한다.
- **이점이 크지 않은 작업**: **CPU를 계속 쓰는 계산 작업(CPU-bound)**. 이런 작업은 물리 코어 수의 한계를 그대로 받으므로 가상 스레드를 많이 만들어도 빨라지지 않는다. (`VirtualThread` 참고)
</details>

**14. (오류 찾기/함정 — 가상 스레드 종료)** 다음 코드는 실행하면 "완료한 작업 수"가 1000보다 작게 나오거나 아예 출력이 누락될 수 있다. 무엇이 문제이며 어떻게 고쳐야 하는지 두 가지 방법으로 쓰시오.
```java
ExecutorService ex = Executors.newVirtualThreadPerTaskExecutor();
AtomicInteger done = new AtomicInteger();
for (int i = 0; i < 1000; i++) {
    ex.submit(() -> { Thread.sleep(1); done.incrementAndGet(); });
}
System.out.println("완료한 작업 수 = " + done.get());   // 여기서 바로 출력
```

<details>
<summary>정답 및 해설</summary>

- **문제**: 작업을 `submit`한 뒤 **완료를 기다리지 않고** 곧바로 `done.get()`을 출력한다. 제출한 작업들이 아직 끝나지 않았으므로 카운트가 1000보다 작게 나온다. `ExecutorService`를 종료하지도 않아 hang이 날 수도 있다. (참고로 예시의 `Thread.sleep(1)`은 `InterruptedException` 처리가 필요하지만, 여기서 핵심은 종료 대기 누락이다.)
- **고치는 방법 두 가지**:
  1. **try-with-resources**: `try (ExecutorService ex = Executors.newVirtualThreadPerTaskExecutor()) { ... }` — 블록을 벗어날 때 `close()`가 모든 작업 완료를 기다린 뒤 종료한다. 출력은 블록 **밖**에서 한다.
  2. **shutdown + awaitTermination**: 제출을 마친 뒤 `ex.shutdown();`으로 새 작업 접수를 막고 `boolean ok = ex.awaitTermination(10, TimeUnit.SECONDS);`로 완료를 기다린다. 반환값이 `true`면 모두 정상 종료된 것이고, 그 **뒤에** `done.get()`을 출력한다.
- 요점: 가상 스레드도 **정상 종료(완료 대기)**를 명시해야 결과가 정확하고 프로그램이 hang 없이 끝난다. (`VirtualThread` 참고)
</details>

## 요약

- **자바 21**은 2023년 9월에 나온 LTS 버전으로, 프리뷰였던 패턴 매칭·레코드 패턴·가상 스레드 등을 정식 기능으로 확정했다. (`Java21Overview`)
- **`var`**는 컴파일 타임에 타입이 확정되는 **정적** 지역 변수 타입 추론이다. 지역 변수에서만, 초기화와 함께 쓰며, 우변이 명확할 때만 사용해 가독성을 높인다. (`LocalVarInference`)
- **`switch`의 `null` 처리**는 `case null`(별도 분기) 또는 `case null, default`(묶어서 처리)로 가능하다. `case null`이 없으면 여전히 NPE가 난다. (`SwitchNull`)
- **`switch` 패턴 매칭**은 `case 타입 변수`로 타입에 따라 분기하며 캐스팅된 변수를 바로 쓴다. `when` 가드로 조건을 더하며, 순서대로 검사되므로 좁은 조건을 먼저 둔다. (`SwitchPattern`)
- **레코드 패턴**은 `case Point(int x, int y)`처럼 레코드를 구성요소로 즉시 분해하며, 중첩 분해로 복잡한 구조도 간결히 다룬다. (`RecordPattern`)
- **가상 스레드**는 경량 스레드로 대규모 동시성(특히 I/O 대기 작업)에 적합하다. `try-with-resources`나 `awaitTermination`으로 정상 종료해야 한다. (`VirtualThread`)
- **순차 컬렉션**은 `getFirst`/`getLast`/`reversed`로 첫·마지막 원소 접근과 역순 뷰를 통일된 API로 제공한다(`SequencedCollection`/`Set`/`Map`). (`SequencedCollection`)
- **기본 문자셋**은 자바 18부터 UTF-8로 통일되어(자바 21 포함) 플랫폼 간 문자 처리가 일관돼졌다. 그래도 입출력에는 `StandardCharsets.UTF_8`을 명시하는 것이 안전하다. (`DefaultCharset`)
