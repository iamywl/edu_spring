# 1장. 컴퓨터는 어떻게 프로그램을 실행하는가?

> **🐳 실습 환경 — 이 장은 `java-sandbox` 컨테이너에서 실습한다**
> ```bash
> cd java && docker compose up -d              # 컨테이너 켜기 (이미 떠 있으면 생략)
> docker exec -it java-sandbox ./run.sh StackDemo   # 예제 실행
> ```
> 처음이라면 [java/README.md](../../../README.md)의 "빠른 시작"을 먼저 보라.

## 들어가며 — 왜 이 장부터 시작하는가?

우리가 작성하는 자바 코드는 결국 사람의 언어에 가깝다. `System.out.println("Hello")` 같은 문장은 영어를 조금 아는 사람이라면 "무언가를 출력하는구나" 하고 대충 짐작할 수 있다. 하지만 여러분의 노트북 안에 들어 있는 CPU는 이 문장을 단 한 글자도 이해하지 못한다. CPU가 이해하는 것은 오직 전기 신호의 켜짐(1)과 꺼짐(0)뿐이다.

그렇다면 여기서 근본적인 질문이 하나 생긴다. **인간이 쓴 텍스트 파일 한 장이, 어떻게 실리콘 덩어리 위에서 실제로 "동작"하는 프로그램이 되는가?** 이 질문에 답하지 못한 채로 자바를 배우면, 여러분은 평생 "이렇게 쓰면 된다더라" 수준의 암기에서 벗어나지 못한다. 반대로 이 질문에 답할 수 있게 되면, 나중에 마주칠 대부분의 어려운 문제 — `NullPointerException` 스택 트레이스, 갑자기 서버 응답이 튀는 현상, "왜 처음 요청만 느리지?" 같은 미스터리 — 가 모두 **하나의 그림 위에서 벌어지는 사건**이라는 것을 깨닫게 된다.

전산 전공자에게 이 장의 내용은 훗날 운영체제, 컴파일러, 컴퓨터 구조 과목과 정확히 이어진다. 비전공자에게는 "블랙박스에 대한 두려움"을 없애 주는 장이다. 우리는 이 장에서 소스코드 한 줄이 기계어가 되기까지의 전 과정을 따라가고, 자바만의 독특한 실행 모델(JVM), 프로그램이 사용하는 메모리의 구조(스택/힙/메서드 영역), 바이트코드라는 중간 언어의 실체, "인터프리터인데 왜 빠른가"의 비밀(JIT), 클래스가 메모리에 올라오는 과정(클래스 로딩), 그리고 아무도 신경 쓰지 않아도 알아서 메모리를 청소해 주는 가비지 컬렉션까지 모두 다룬다.

미리 약속하자. 이 장의 목표는 바이트코드 명령어를 외우는 것이 **아니다**. 목표는 **"내가 쓴 코드가 결국 어떤 그림 위에서 굴러가는가"에 대한 지도를 머릿속에 그리는 것**이다. 지도가 있으면 길을 잃지 않는다. 시작해 보자.

## 1.1 프로그래밍 언어란?

### 정의

**프로그래밍 언어(Programming Language)** 란, 인간이 컴퓨터에게 작업을 지시하기 위해 만든 형식 언어(formal language)다. 인간의 자연어와 기계어 사이에 놓인 **통역 계층**이라고 이해하면 정확하다.

### 원리 — 왜 통역이 필요한가?

컴퓨터의 CPU는 0과 1, 즉 이진수(binary)만 이해한다. 예를 들어 다음과 같은 비트 나열이 CPU에게는 "H라는 글자를 처리하라"는 명령일 수 있다.

```
01001000 01100101 01101100 01101100 01101111
   H        e        l        l        o
```

이런 0과 1의 나열을 **기계어(machine code)** 라고 부른다. 사람이 이걸 직접 쓰는 것은 이론적으로 가능하지만 현실적으로 불가능에 가깝다. 그래서 우리는 사람이 읽고 쓸 수 있는 언어(자바, 파이썬 등)로 코드를 작성한 뒤, 그것을 기계어로 "번역"하는 도구를 사용한다.

```
   사람의 생각            프로그래밍 언어             기계어
  "화면에 인사해"   →    System.out             →   01001000
                        .println("Hi")            01100101 ...
  ───────────      ───────────────────       ───────────
   자연어 (모호함)     형식 언어 (엄밀함)        CPU가 실행
```

번역 방식에는 크게 두 가지 철학이 있다. 이 차이를 이해하는 것이 자바를 이해하는 핵심이다.

**① 컴파일 언어 (Compiled Language) — 예: C, C++**

원고 전체를 미리 번역해서 책으로 출판하는 방식이다.

- 프로그램을 실행하기 전에, **소스코드 전체를 한 번에 기계어로 번역**한다.
- 실행 시점에는 이미 번역이 끝나 있으므로 **매우 빠르다.**
- 단점: 번역된 결과물(기계어)은 특정 OS/CPU에 딱 맞춰져 있다. 다른 환경에서 돌리려면 **다시 번역**해야 한다. 번역가가 국가마다 따로 책을 출판해야 하는 셈이다.

**② 인터프리터 언어 (Interpreted Language) — 예: Python, JavaScript**

동시통역사가 옆에서 한 문장씩 실시간으로 통역하는 방식이다.

- 소스코드를 **한 줄씩 읽으면서 즉석에서 실행**한다.
- 장점: 미리 번역할 필요 없이 바로 실행할 수 있고, 유연하다.
- 단점: 실행할 때마다 통역하므로 컴파일 언어보다 **느리다.**

### 그런데 자바는 왜 "둘 다"인가?

자바의 설계자들은 흥미로운 절충안을 택했다. **"1단계는 컴파일하되, 기계어가 아니라 '바이트코드'라는 중간 언어로 번역한다. 2단계는 그 바이트코드를 JVM이 실행한다."**

```
  Hello.java          Hello.class            실제 실행
 (소스코드)   ─javac→  (바이트코드)   ─JVM→   (인터프리팅
                                              + JIT 컴파일)
  ─────────           ────────────           ───────────
  사람이 작성          기계어가 아님!          플랫폼별 실행
                      JVM 전용 언어
```

- **1단계 (컴파일):** `javac`(자바 컴파일러)가 `Hello.java`를 `Hello.class`로 번역한다. 하지만 이 `.class` 파일은 특정 CPU의 기계어가 **아니다.** JVM이라는 가상의 기계를 위한 **바이트코드(bytecode)** 다.
- **2단계 (실행):** JVM이 이 바이트코드를 읽어서 인터프리팅하고, 자주 실행되는 부분은 JIT로 진짜 기계어로 번역해 캐싱한다.

즉 자바는 "미리 어느 정도 번역해 두되(컴파일 언어의 속도 이점), 최종 기계어는 실행 환경에서 결정한다(인터프리터 언어의 이식성 이점)"는 하이브리드 전략을 취한다. 이 전략이 왜 그토록 혁명적이었는지는 다음 절에서 이어진다.

> 💡 **면접 팁:** "자바는 컴파일 언어인가요, 인터프리터 언어인가요?"라고 물으면 "둘 다입니다"가 정답이다. 소스코드는 컴파일되어 바이트코드가 되고(컴파일), 그 바이트코드는 JVM에서 인터프리팅되며(인터프리터), 핫스팟은 JIT로 다시 컴파일된다(다시 컴파일)고 설명하면 확실히 각인시킬 수 있다.

## 1.2 Java의 특별함: "Write Once, Run Anywhere"

### 정의

**WORA(Write Once, Run Anywhere)** 는 "한 번 작성하면 어디서든 실행된다"는 자바의 대표 슬로건이다. 한 번 작성한 자바 프로그램을 재작성/재컴파일 없이 윈도우, 맥, 리눅스 등 어디서든 돌릴 수 있다는 뜻이다.

### 원리 — C/C++의 고통에서 출발하기

이 슬로건이 왜 나왔는지 이해하려면 그 이전 세계의 고통을 알아야 한다. C/C++에서는 소스코드를 각 OS/CPU의 기계어로 직접 컴파일한다. 그런데 문제는, **OS와 CPU마다 기계어가 다르다는 것**이다.

```
                같은 C 소스코드 main.c
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
   Windows 컴파일    Mac 컴파일      Linux 컴파일
       │               │               │
       ▼               ▼               ▼
   main.exe        main(Mac용)     main(Linux용)
   (x86 기계어)     (다른 기계어)    (또 다른 기계어)

   → OS/CPU마다 따로 컴파일해야 한다!
```

인터넷이 폭발적으로 성장하던 1990년대에 이것은 진짜 악몽이었다. 서버는 리눅스인데 개발자 PC는 윈도우나 맥이고, 배포 대상 기기는 제각각이었다. 프로그램 하나를 배포하려고 환경마다 다시 컴파일하고 테스트하는 일이 반복됐다.

### 해결책 — 중간에 통역사를 두자 (JVM)

1995년 James Gosling이 이끄는 팀은 발상을 뒤집었다. **"프로그램이 OS와 직접 대화하게 하지 말고, 중간에 통역사를 하나 두면 되지 않을까?"**

그 통역사가 바로 **JVM(Java Virtual Machine, 자바 가상 머신)** 이다.

```
                    Hello.class (바이트코드)
                    한 번만 만들면 끝!
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
   Windows용 JVM     Mac용 JVM       Linux용 JVM
   (윈도우 통역사)    (맥 통역사)      (리눅스 통역사)
       │               │               │
       ▼               ▼               ▼
   Windows 기계어    Mac 기계어      Linux 기계어

   → 바이트코드는 하나. JVM이 각 OS에 맞게 통역!
```

핵심은 이것이다. **자바 프로그램(`.class`)은 OS와 직접 대화하지 않는다. 오직 JVM하고만 대화한다.** 그리고 각 OS에 맞는 JVM이 미리 만들어져 있어서, 그 JVM이 바이트코드를 해당 OS의 기계어로 통역해 준다.

### 비유 — 에스페란토와 통역사

세계 각국 사람들이 모인 회의를 상상해 보자.

- **방법 A (C/C++):** 발표자가 청중의 언어를 하나하나 미리 배워서, 한국인용 원고, 일본인용 원고, 미국인용 원고를 따로 준비한다. (환경마다 재컴파일)
- **방법 B (Java):** 발표자는 **에스페란토어(만국 공통어 = 바이트코드)** 로 한 번만 원고를 쓴다. 각 나라 부스에 앉은 **통역사(각 OS의 JVM)** 가 에스페란토를 자기 나라 말로 실시간 통역한다.

발표자는 청중이 몇 개 나라 사람인지 신경 쓸 필요가 없다. 통역사가 알아서 처리한다. 이것이 "Write Once, Run Anywhere"다.

### 왜 이것이 혁명적이었나?

앞서 말한 시대적 배경 때문이다. 인터넷과 웹이 커지면서 "한 번 작성해서 아무 데서나 돌린다"는 것은 개발 비용을 극적으로 줄였다. 은행, 통신사, 대기업의 서버 시스템이 대거 자바로 넘어온 이유가 여기에 있다. 오늘날에도 백엔드 서버 시장에서 자바가 강세인 것은 이 결정 하나에서 비롯됐다.

> 💡 **면접 팁:** WORA를 가능하게 하는 주체는 "바이트코드"가 아니라 "각 플랫폼에 맞게 구현된 JVM들"이라는 점을 정확히 말할 수 있어야 한다. 바이트코드는 어디서나 똑같고, 플랫폼 차이를 흡수하는 것은 JVM이다.

## 1.3 JDK, JRE, JVM의 관계

### 정의

자바를 다루면 반드시 마주치는 세 약자다. 이들은 서로 포함 관계(⊃)에 있다.

- **JVM (Java Virtual Machine):** 바이트코드를 실제로 실행하는 엔진.
- **JRE (Java Runtime Environment):** JVM + 실행에 필요한 기본 라이브러리.
- **JDK (Java Development Kit):** JRE + 개발에 필요한 도구들.

### 원리 — 포함 관계 그림

```
┌─────────────────────────────────────────────┐
│  JDK (Java Development Kit) — 개발자용 전체    │
│  ┌───────────────────────────────────────┐   │
│  │  JRE (Runtime Environment) — 실행 환경 │   │
│  │  ┌─────────────────────────────────┐  │   │
│  │  │  JVM — 바이트코드 실행 엔진       │  │   │
│  │  └─────────────────────────────────┘  │   │
│  │  + 기본 라이브러리                     │   │
│  │    (java.lang, java.util, java.io …)  │   │
│  └───────────────────────────────────────┘   │
│  + 개발 도구                                   │
│    (javac, jar, javadoc, jdb …)               │
└─────────────────────────────────────────────┘
```

- **JVM** 은 순수한 실행 엔진이다. 바이트코드를 받아 기계어로 통역해 CPU에 올린다. 하지만 JVM만으로는 아무것도 못 한다. `System.out.println` 같은 것을 쓰려면 `java.lang` 같은 표준 라이브러리가 필요하다.
- **JRE** 는 JVM에 그 표준 라이브러리를 얹은 것이다. 즉 "남이 만든 자바 프로그램을 **실행**만 하는 데" 필요한 최소 세트다.
- **JDK** 는 JRE에 개발 도구를 더한 것이다. 소스코드를 컴파일하는 `javac`, 파일을 묶는 `jar`, 문서를 만드는 `javadoc`, 디버깅하는 `jdb` 등이 포함된다. 즉 "자바 프로그램을 **만드는** 데" 필요한 완전한 세트다.

### 비유 — 통역사, 사전, 펜과 종이

| 구성 요소 | 비유 | 역할 |
|-----------|------|------|
| **JVM** | 통역사 | 바이트코드를 실행(통역)한다 |
| **JRE** | 통역사 + 사전 | 통역에 필요한 어휘집(라이브러리)까지 갖췄다 |
| **JDK** | 통역사 + 사전 + 펜과 종이 | 새 글을 **쓸 수도** 있다(컴파일·개발 도구) |

정리하면 이렇다.

- 자바 프로그램을 **만들려면** → JDK가 필요하다.
- 자바 프로그램을 **실행만 하려면** → JRE면 충분하다.

단, 실무 팁이 하나 있다. **Java 11부터는 JRE가 별도로 배포되지 않는다.** 그래서 요즘은 실행만 할 것이어도 그냥 JDK를 설치하는 것이 일반적이다.

> 💡 **면접 팁:** "JDK, JRE, JVM의 차이는?"은 신입 면접 단골 질문이다. 포함 관계(JDK ⊃ JRE ⊃ JVM)를 그림으로 그려 설명하고, "JVM은 실행 엔진, JRE는 실행 환경(+라이브러리), JDK는 개발 키트(+컴파일러/도구)"라고 한 문장으로 정리한 뒤, "Java 11부터 JRE가 단독 배포되지 않는다"까지 덧붙이면 인상적이다.

## 1.4 Java 메모리 구조

### 정의

프로그램이 실행되는 동안 JVM은 운영체제로부터 메모리를 할당받아 여러 영역으로 나눠 쓴다. 대표적인 세 영역은 **메서드 영역(Method Area)**, **힙(Heap)**, **스택(Stack)** 이다.

### 원리 — 전체 지도

```
┌───────────────────────────────────────────────┐
│                 JVM 메모리                       │
│                                                 │
│  ┌─────────────────────────────────────────┐   │
│  │  Method Area (메서드 영역)                │   │
│  │  · 클래스 구조 정보                        │   │
│  │  · 메서드의 바이트코드                     │   │
│  │  · static 변수                            │   │
│  └─────────────────────────────────────────┘   │
│                                                 │
│  ┌────────────────────┐  ┌──────────────────┐  │
│  │  Heap (힙)          │  │  Stack (스택)     │  │
│  │  · new로 만든 객체   │  │  · 메서드 호출    │  │
│  │  · 모든 인스턴스     │  │  · 지역 변수      │  │
│  │  (공유 창고)        │  │  (스레드마다 1개) │  │
│  └────────────────────┘  └──────────────────┘  │
└───────────────────────────────────────────────┘
```

세 영역은 각각 성격이 뚜렷이 다르다. 하나씩 뜯어보자.

### ① Stack — 접시 쌓기 (LIFO)

스택은 **접시를 쌓았다가 위에서부터 빼는** 구조다. 마지막에 넣은 것이 가장 먼저 나온다. 이를 **LIFO(Last In, First Out)** 라고 한다.

**메서드를 호출할 때마다 스택에 한 칸이 쌓이고, 메서드가 끝나면 그 칸이 빠진다.** 메서드 안에서 선언한 지역 변수도 이 스택 칸에 함께 저장되며, 메서드가 끝나면 함께 사라진다.

```java
public class StackDemo {
    public static void main(String[] args) {
        int a = 10;      // main의 지역 변수
        method1();       // method1 호출 → 스택에 쌓임
    }
    static void method1() {
        int b = 20;
        method2();       // method2 호출 → 또 쌓임
    }
    static void method2() {
        int c = 30;
    }
}
```

이때 스택은 다음과 같이 자란다.

```
  실행 순서 →
  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │          │   │          │   │ method2  │ ← 가장 위(마지막 호출)
  │          │   │ method1  │   │ method1  │
  │  main    │   │  main    │   │  main    │
  └──────────┘   └──────────┘   └──────────┘
   main 실행중    method1 호출   method2 호출

  method2가 먼저 끝나 빠지고, 그다음 method1, 마지막에 main.
```

**왜 하필 LIFO인가?** 메서드 호출 자체가 자연스럽게 "들어갔다 나오는" 구조이기 때문이다. A가 B를 부르고, B가 C를 부르면, 반드시 C가 가장 먼저 끝나고, 그다음 B, 마지막에 A가 끝난다. 이 순서는 항상 스택 구조와 정확히 일치한다. 그래서 메서드 호출을 관리하는 데 스택이 딱 맞는다. 참고로 스택은 **스레드마다 하나씩** 갖는다. 스레드는 각자 자기 일(호출 흐름)을 하므로 스택이 분리되어 있어야 한다.

### ② Heap — 거대한 공유 창고

`new` 키워드로 객체를 만들면, 그 객체는 스택이 아니라 **힙**이라는 거대한 창고 어딘가에 자리를 배정받는다. 그리고 그 자리의 **주소(참조)** 를 돌려준다. 스택의 변수는 이 주소를 붙잡고 있는 것이다.

```java
Dog myDog = new Dog();
```

```
     Stack                       Heap
  ┌──────────────┐          ┌──────────────────┐
  │ myDog = 0x42 │────────▶ │ 0x42: Dog 객체     │
  │  (주소만 보관) │          │   name = "바둑이"  │
  └──────────────┘          │   age  = 3        │
                            └──────────────────┘
   변수는 "주소표"           실제 객체는 창고 안
```

**왜 스택과 힙을 나누는가?** 성격이 다르기 때문이다.

- **스택**은 매우 빠르지만 크기가 작고, 메서드가 끝나면 내용이 사라진다. 잠깐 쓰고 버릴 지역 변수에 적합하다.
- **객체**는 여러 메서드에서 공유될 수 있고, 크기도 제각각이며, 언제까지 살아 있어야 할지 미리 알 수 없다. 그래서 오래 보관할 수 있고 넓은 공간인 **힙**에 둔다.

힙은 모든 스레드가 **공유**한다. 그래서 여러 스레드가 같은 객체를 동시에 건드릴 때 동시성 문제가 생기는데, 그 이야기는 뒤 장에서 다룬다.

### ③ Method Area — 클래스 도서관

메서드 영역에는 **클래스에 관한 정보**가 저장된다. 클래스의 구조(어떤 필드와 메서드가 있는지), 메서드의 바이트코드, 그리고 `static` 변수가 여기에 있다.

```
┌──────────────────────────────────────┐
│  Method Area (클래스 도서관)            │
│                                        │
│  [Dog 클래스]                          │
│    · 필드 목록: name, age              │
│    · 메서드 코드: bark(), eat()        │
│    · static int count = 0  ← 여기!    │
└──────────────────────────────────────┘
```

**흔한 오해:** "`static` 변수는 스택에 있다"고 착각하기 쉬운데, **틀렸다.** `static` 변수는 특정 메서드나 객체에 속하지 않고 **클래스 전체에 딱 하나** 존재하는 값이므로, 클래스 정보가 모여 있는 메서드 영역에 저장된다. (참고: 최신 자바에서 메서드 영역은 힙의 한 부분인 Metaspace로 구현되지만, 개념적으로는 별도 영역으로 이해하면 된다.)

> 💡 **면접 팁:** "지역 변수, 객체, static 변수는 각각 어디에 저장되나?"라는 질문에 즉답할 수 있어야 한다. 지역 변수 → 스택, `new` 객체 → 힙, static 변수와 클래스 메타데이터 → 메서드 영역. 이 세 개를 정확히 짝지으면 메모리 구조를 이해했다는 신호다.

> 실행 데모: ./run.sh StackDemo

## 1.5 바이트코드: 블랙박스를 열어보다

### 정의

**바이트코드(bytecode)** 는 `javac`가 소스코드를 컴파일해 만든, JVM 전용의 중간 언어다. `.class` 파일 안에 들어 있으며, 사람이 아니라 **JVM이 읽으라고** 만든 명령어의 나열이다.

### 원리 — 소스코드가 실행되기까지의 여정

```
   Add.java   ─javac→   Add.class   ─JVM→   실제 실행
  (소스코드)            (바이트코드)
```

`.class`는 이진 파일이라 그냥 열면 알아볼 수 없다. 하지만 JDK에 포함된 `javap`라는 도구로 **역어셈블(disassemble)** 하면, 그 안의 명령어를 사람이 읽을 수 있는 형태로 볼 수 있다. 블랙박스를 여는 도구가 `javap`다.

### 예제 — javap로 들여다보기

다음과 같은 간단한 메서드가 있다고 하자.

```java
class Add {
    int add(int a, int b) {
        return a + b;
    }
}
```

`javac Add.java` 후 `javap -c Add` 하면 대략 이런 바이트코드가 나온다.

```
int add(int, int);
  Code:
     0: iload_1      // 1번 지역변수(a)를 스택에 올려라
     1: iload_2      // 2번 지역변수(b)를 스택에 올려라
     2: iadd         // 스택 위 두 값을 더해라
     3: ireturn      // 결과(int)를 반환해라
```

명령어 이름에는 규칙이 있다.

- 첫 글자는 **자료형**을 뜻한다: `i` = int, `l` = long, `d` = double, `f` = float, `a` = 참조(address).
- `load` = 지역 변수를 (operand) 스택으로 올려라.
- `store` = 스택 값을 지역 변수에 저장해라.
- `add` = 더해라. `ireturn` = int 값을 반환해라.

**여기서 하나 궁금해진다. 왜 `a`가 `iload_1`, `b`가 `iload_2`일까? 0번은 어디 갔지?** 인스턴스 메서드에서는 **0번 지역 변수가 항상 `this`(자기 자신)** 이기 때문이다. 그래서 실제 파라미터는 1번, 2번부터 시작한다. (`static` 메서드였다면 `this`가 없으니 파라미터가 0번부터 시작한다.)

### operand stack — JVM은 스택 계산기다

여기서 자바 실행 모델의 핵심이 드러난다. 많은 CPU는 계산을 **레지스터**에서 하지만, JVM은 **operand stack(피연산자 스택)** 이라는 스택 위에서 계산한다. 앞의 `a + b`(a=3, b=4)가 어떻게 굴러가는지 4단계로 보자.

```
① iload_1 (a=3)      ② iload_2 (b=4)     ③ iadd            ④ ireturn
  ┌─────┐              ┌─────┐             ┌─────┐            ┌─────┐
  │     │              │  4  │             │     │            │     │
  │  3  │              │  3  │             │  7  │  ← 3+4     │     │  → 7 반환
  └─────┘              └─────┘             └─────┘            └─────┘
 operand stack        스택에 두 값          더한 결과만        결과를 꺼내
                                            남음               호출자에게
```

이 방식은 수학의 **RPN(역폴란드 표기법, Reverse Polish Notation)** 과 정확히 같다. `3 4 +`처럼 피연산자를 먼저 쌓고 연산자로 처리한다. 계산기 스택을 떠올리면 딱 맞다.

### stack frame — 스택에 쌓이는 한 칸의 정체

1.4에서 "메서드를 호출하면 스택에 한 칸이 쌓인다"고 했다. 그 한 칸의 정식 이름이 **스택 프레임(stack frame)** 이다. 프레임 안에는 세 가지가 들어 있다.

```
┌─────────────────────────────────────┐
│  Stack Frame (add 메서드)            │
│                                       │
│  ① 지역 변수 배열                     │
│     [0] = this                        │
│     [1] = a                           │
│     [2] = b                           │
│                                       │
│  ② operand stack (계산용 임시 스택)   │
│                                       │
│  ③ 복귀 주소 (끝나면 어디로 돌아갈지)  │
└─────────────────────────────────────┘
   메서드가 끝나면 이 프레임이 통째로 사라진다
```

메서드가 끝나면 이 프레임 전체가 스택에서 pop 되어 사라지고, `this`·`a`·`b` 같은 지역 변수도 함께 소멸한다. 1.4에서 "지역 변수는 메서드가 끝나면 사라진다"고 한 이유가 바로 이것이다.

### 왜 이걸 알아야 하나?

바이트코드 명령어를 외울 필요는 **전혀 없다.** 하지만 "내 코드가 결국 이런 명령어들로 번역된다"는 감각을 갖는 순간, 앞으로 마주칠 여러 개념이 **하나의 그림 위에서** 연결된다.

- `NullPointerException` 스택 트레이스를 보면 → 스택 프레임이 어떻게 쌓였는지 그림이 그려진다.
- 성능 문제를 볼 때 → "어느 지점의 어떤 연산이 뜨거운가"를 스택/힙 위에서 상상할 수 있다.
- 다음 절의 JIT, 뒤의 GC → 모두 이 스택·힙·메서드 영역이라는 무대 위에서 벌어지는 이야기다.

> 💡 **면접 팁:** "자바는 스택 기반 가상 머신인가, 레지스터 기반인가?"라고 물으면 "스택 기반(operand stack 사용)"이 정답이다. 반대로 안드로이드의 Dalvik/ART는 레지스터 기반이라는 점까지 알면 더 좋다.

> 실행 데모: ./run.sh Add

## 1.6 JIT 컴파일: "인터프리터인데 왜 이렇게 빠른가?"

### 정의

**JIT(Just-In-Time) 컴파일러** 는 실행 중에 자주 쓰이는 바이트코드를 진짜 기계어로 번역해 캐싱함으로써, 인터프리터의 느림을 극복하는 JVM의 최적화 엔진이다.

### 원리 — 의문에서 시작하기

1.1에서 "인터프리터는 느리다"고 했다. 그런데 자바도 바이트코드를 인터프리팅한다. **그렇다면 자바는 느려야 하는데, 왜 실제로는 C에 견줄 만큼 빠른가?** 답이 바로 JIT다.

JVM은 처음엔 통역사처럼 굴다가, 나중엔 번역가로 변신한다.

```
  프로그램 시작
      │
      ▼
  인터프리터로 즉시 실행 ────────────┐
      │                            │
      ▼                            │ JVM이 뒤에서 몰래
  호출 횟수 / 반복 횟수를 센다        │ 실행 통계를 수집
      │                            │
      ▼                            │
  자주 실행되는 코드 발견 (hot spot) ◀┘
      │
      ▼
  JIT가 그 부분만 기계어로 번역해 캐시
      │
      ▼
  이후엔 통역 없이 네이티브(기계어)로 바로 실행 → 빠름!
```

**비유:** 동시통역사가 회의에 자꾸 나오는 문장을 아예 통째로 외워 버리는 것과 같다. 처음 몇 번은 한 단어씩 통역하지만, "감사합니다"가 백 번 나오면 그냥 반사적으로 내뱉게 된다. 참고로 오라클의 표준 JVM 이름이 **HotSpot** 인데, 바로 이 "핫스팟을 찾아 최적화한다"는 아이디어에서 온 이름이다.

### 왜 처음부터 다 컴파일하지 않는가?

여기서 반문이 나올 수 있다. "그렇게 빠를 거면 그냥 처음부터 전부 기계어로 컴파일하면 되잖아?" 두 가지 이유가 있다.

**① 시작 속도(startup time)**
전부 컴파일하려면 실행 전에 오래 기다려야 한다. 인터프리터는 번역을 기다릴 필요 없이 **즉시** 실행을 시작할 수 있다. 짧게 실행되고 끝나는 프로그램이라면 컴파일 비용이 오히려 손해다.

**② 실행 정보 수집(profiling)**
어느 코드가 뜨거운지는 **실제로 돌려 봐야** 안다. JIT는 실행 중에 관찰한 정보를 바탕으로, 정적 컴파일러(C 컴파일러 등)가 흉내 낼 수 없는 **공격적인 최적화** 를 한다. 예를 들면 이런 것들이다.

- **인라이닝(inlining):** 자주 호출되는 작은 메서드를 아예 호출 지점에 펼쳐 넣어 호출 비용을 없앤다.
- **분기 예측(branch prediction):** "이 if문은 99% true였다"는 실측 정보로 가장 흔한 경로를 최적화한다.

이 때문에 특정 상황에서는 오래 돌린 자바가 **C보다도 빠를 수** 있다. 실행 중 살아 있는 정보를 활용하기 때문이다.

### 단계적 컴파일 (Tiered Compilation)

현대 JVM은 한 번에 최고 품질로 컴파일하지 않고, 여러 단계(tier)로 나눠 점진적으로 최적화한다.

```
  Tier 0        Tier 1~3         Tier 4
  인터프리터  →  C1 컴파일러   →   C2 컴파일러
  (즉시 실행)   (빠르게 대충      (느리지만
               번역 + 통계 수집)   최고 품질 최적화)

  뜨거워질수록 오른쪽으로 승격
```

- **Tier 0:** 인터프리터. 워밍업 없이 즉시 시작.
- **Tier 1~3 (C1):** 빠르게 대충 컴파일하면서 실행 통계를 더 모은다.
- **Tier 4 (C2):** 컴파일은 느리지만 결과물의 품질이 가장 높다. 진짜 뜨거운 코드에만 적용한다.

**왜 이게 중요한가?** "자바는 워밍업(warm-up)이 필요하다"는 말의 정체가 바로 이것이다. 프로그램을 켜자마자는 인터프리터라 느리고, 조금 돌려 핫스팟이 C2까지 승격되어야 최고 속도가 난다. **벤치마크할 때 첫 실행 결과를 버리는(warm-up 구간을 제외하는) 이유**도 여기에 있다. 처음 몇 번은 아직 JIT가 개입하기 전이라 진짜 성능이 아니다.

> 💡 **면접 팁:** "자바가 느리다는데 사실인가?"라는 질문이 나오면, JIT와 단계적 컴파일을 근거로 "장시간 실행되는 서버에서는 워밍업 후 네이티브에 근접한다"고 답하면 좋다. "인터프리터의 빠른 시작 + JIT의 빠른 정상 속도"라는 두 마리 토끼 전략임을 강조하라.

## 1.7 클래스 로딩

### 정의

**클래스 로딩(class loading)** 은 디스크에 있는 `.class` 파일을 찾아 읽어, JVM의 메서드 영역에 클래스 정보로 올리는 과정이다. 이 일을 하는 주체가 **클래스 로더(ClassLoader)** 다.

### 원리 — .class가 있다고 저절로 실행되진 않는다

`.class` 파일이 디스크에 존재한다고 해서 그 클래스가 저절로 메모리에 올라오는 것은 아니다. 반드시 클래스 로더가 그것을 읽어 메서드 영역에 배치해야 비로소 쓸 수 있다. 이 과정은 세 단계로 이뤄진다.

```
  ① Loading   →   ② Linking   →   ③ Initialization
  (읽어서 올림)     (검증·준비·해석)   (실제 값 채우기)
```

**① Loading(로딩)**
`.class` 파일을 찾아 읽어서, 그 클래스의 구조 정보를 메서드 영역에 올린다.

**② Linking(링킹)** — 세 하위 단계로 나뉜다.

- **Verify(검증):** 이 바이트코드가 안전하고 규칙에 맞는지 확인한다. 손상되었거나 악의적인 코드를 막는 보안 관문이다.
- **Prepare(준비):** `static` 변수를 위한 메모리를 확보하고 **기본값**으로 초기화한다. int면 `0`, 참조형이면 `null`, boolean이면 `false`.
- **Resolve(해석):** 심볼릭 참조(이름으로 된 참조)를 실제 메모리 참조로 바꾼다.

**③ Initialization(초기화)**
드디어 `static` 변수에 **우리가 지정한 실제 값**이 들어가고, `static` 초기화 블록이 실행된다.

여기서 헷갈리기 쉬운 **Prepare vs Initialization** 을 예로 보자.

```java
class Config {
    static int COUNT = 100;
}
```

```
  Prepare 단계:  COUNT = 0     ← 일단 기본값으로 자리만 잡음
  Init 단계:     COUNT = 100   ← 이때 비로소 100이 들어감
```

즉 `static int COUNT = 100`은 두 시점에 걸쳐 처리된다. 준비 단계에서 `0`이 되었다가, 초기화 단계에서 `100`이 되는 것이다.

### 클래스 로더의 계층 구조와 부모 위임

클래스 로더는 하나가 아니라 **계층(hierarchy)** 으로 이루어져 있다.

```
  Bootstrap ClassLoader     (최상위, C로 작성, java.lang.* 등 핵심 JDK)
        ▲
        │ 부모
  Platform ClassLoader      (JDK 확장 / 플랫폼 모듈)
        ▲
        │ 부모
  Application ClassLoader   (우리가 짠 코드 + 외부 라이브러리, classpath)
```

클래스를 로딩할 때는 **부모 위임(parent delegation)** 원칙을 따른다. 즉 "내가 직접 로딩하기 전에, 먼저 부모에게 이 클래스 로딩해 줄 수 있냐고 물어본다." 부모가 처리할 수 있으면 부모가 하고, 못 할 때만 자식이 한다.

**왜 이렇게 하는가? — 보안 때문이다.** 누군가 악의적으로 `java.lang.String`이라는 가짜 클래스를 만들어 넣었다고 하자. 부모 위임이 있으면, `java.lang.String`을 로딩할 때 요청이 최상위 Bootstrap까지 올라가고, **진짜 `String`이 먼저 로딩**된다. 결과적으로 가짜는 절대 끼어들 수 없다. 핵심 클래스가 위조되는 것을 원천 차단하는 것이다.

### Lazy Loading — 필요할 때 로딩한다

JVM은 모든 클래스를 시작할 때 한꺼번에 로딩하지 않는다. **처음 필요해지는 순간**(그 클래스로 `new`를 하거나, `static` 멤버에 처음 접근하는 순간)에 로딩한다. 이를 **지연 로딩(lazy loading)** 이라 한다. 쓰이지 않는 클래스는 메모리에 아예 올라오지 않으므로 메모리를 아낄 수 있다.

> 💡 **면접 팁:** "부모 위임 모델은 왜 있나?"라는 질문의 정답은 "핵심 라이브러리 위조 방지(보안) + 클래스의 유일성 보장"이다. 또 "`static int COUNT = 100`은 언제 100이 되나?"에는 "Prepare에서 0, Initialization에서 100"이라고 답하면 클래스 로딩 단계를 정확히 이해했다는 신호다.

## 1.8 가비지 컬렉션

### 정의

**가비지 컬렉션(Garbage Collection, GC)** 은 더 이상 쓰이지 않는 객체를 JVM이 **자동으로** 찾아 힙에서 제거하고 메모리를 회수하는 기능이다.

### 원리 — C의 고통에서 다시 출발

C에서는 `malloc`으로 메모리를 잡으면 `free`로 직접 반납해야 한다. 반납을 깜빡하면 **메모리 누수(memory leak)** 가 쌓여 결국 프로그램이 뻗고, 반대로 이미 반납한 것을 또 쓰면 **크래시**가 난다. 이 수동 관리가 수많은 버그의 원흉이었다.

자바는 이 부담을 개발자에게서 걷어냈다. GC가 알아서 "이제 아무도 안 쓰는 객체"를 찾아 치운다.

### 그런데 "안 쓰는 객체"를 어떻게 판단하는가? — 도달 가능성

핵심 개념은 **도달 가능성(Reachability)** 이다. GC는 **GC Root** 에서 출발해 참조를 타고 갈 수 있는 객체는 "살아 있다"고 보고, 도달할 수 없는 객체는 "쓰레기"로 판정한다.

GC Root는 대략 이런 것들이다.

- 현재 실행 중인 스택 프레임의 지역 변수
- `static` 변수
- (그 외 JNI 참조 등)

```
  GC Root ──▶ A ──▶ B ──▶ C      ← Root에서 도달 가능 = 살아있음
                              
              D ◀──▶ E            ← 서로만 참조, Root에서 도달 불가 = 쓰레기
```

여기서 아주 중요한 함정이 있다. **D와 E는 서로를 참조하고 있다.** 단순히 "참조가 하나라도 있으면 산다"(참조 카운팅) 방식이라면 D, E는 서로가 서로를 가리키니 영원히 안 죽는다. 하지만 자바 GC는 참조 카운팅이 아니라 **도달 가능성**을 본다. Root에서 D, E로 가는 길이 없으므로 둘 다 쓰레기로 판정되어 회수된다. 이것이 순환 참조(circular reference) 문제를 자연스럽게 해결하는 이유다.

### Mark and Sweep — 표시하고 쓸어담기

가장 기본이 되는 GC 알고리즘이다. 두 단계로 동작한다.

```
  ① Mark (표시)                    ② Sweep (수거)
  Root에서 도달 가능한 객체에         표시 안 된 객체를 회수
  표시(mark)를 남긴다

  Root ─▶ [A✓] ─▶ [B✓]            [A✓] [B✓]  ← 유지
          [D ]    [E ]      →       (D, E는 회수됨)
          표시 없음                  빈 공간 확보
```

### 세대별 GC — 약한 세대 가설

여기서 성능을 위한 중요한 관찰이 등장한다. 실제 프로그램을 분석해 보면 **대부분의 객체는 만들어지자마자 금방 죽는다.** (메서드 안에서 잠깐 쓰고 버리는 임시 객체를 생각하면 된다.) 이를 **약한 세대 가설(Weak Generational Hypothesis)** 이라 한다.

이 관찰을 활용해 힙을 나이대로 나눈다.

```
┌──────────────────────────┐   ┌──────────────────────┐
│  Young Generation         │   │  Old Generation       │
│  (갓 태어난 객체)          │   │  (여러 번 살아남은     │
│  대부분 여기서 금방 죽음    │──▶│   장수 객체 = 승격)    │
└──────────────────────────┘   └──────────────────────┘
   Minor GC: 자주, 빠름          Major(Full) GC: 드물게, 느림
```

- **Young Generation:** 새로 만든 객체가 태어나는 곳. 대부분 여기서 곧 죽는다.
- **Old Generation:** Young에서 GC를 몇 번 견디고 살아남은 객체가 **승격(promotion)** 되어 옮겨 가는 곳. 오래 사는 객체들이다.
- **Minor GC:** Young 영역만 청소한다. 작고 자주 일어나며 빠르다.
- **Major GC(= Full GC):** Old까지 포함해 청소한다. 드물게 일어나지만 느리다.

**왜 나누는가?** "작은 Young 영역만 자주 청소해도, 어차피 쓰레기의 대부분이 Young에 있으니 대부분의 쓰레기를 치울 수 있다." 즉 값싼 청소(Minor GC)를 자주 돌려 대부분을 처리하고, 비싼 청소(Full GC)는 어쩌다 한 번만 하면 된다. 효율의 극대화다.

### Stop-The-World (STW) — GC의 대가

GC가 힙을 정리하는 동안, 객체들이 계속 움직이면 정리를 정확히 할 수 없다. 그래서 GC는 정리 중에 **애플리케이션 스레드를 잠깐 멈춘다.** 이를 **Stop-The-World(STW)** 라고 한다.

```
  앱 실행 ──[STW! 멈춤]── 앱 실행 ──[STW!]── 앱 실행
              │                       │
              GC 작업 중                GC 작업 중
```

STW가 짧으면 문제없지만, **길어지면 서버가 그 시간 동안 응답을 못 한다.** 사용자 입장에서는 "가끔 응답이 툭툭 튀는" 레이턴시 문제로 나타난다. 그래서 현대의 GC들(G1 GC, ZGC 등)은 **어떻게든 STW 시간을 줄이는 것**을 목표로 발전해 왔다. ZGC 같은 최신 GC는 STW를 밀리초 이하로 억제한다.

> 💡 **면접 팁:** GC 관련 최고의 한 방은 **"자바 GC는 참조 카운팅이 아니라 도달 가능성(reachability) 기반"** 이라는 것이다. 그래서 순환 참조도 회수된다는 점, 그리고 STW를 줄이는 것이 현대 GC의 핵심 과제라는 점을 덧붙이면 깊이가 드러난다.

## ⚠️ 흔한 오해와 함정

**① "자바는 인터프리터 언어라서 느리다."**
반은 맞고 반은 틀리다. 처음엔 인터프리팅하지만, JIT가 핫스팟을 네이티브로 컴파일하므로 워밍업 후에는 매우 빠르다. "느리다"는 인식은 주로 짧게 실행되는 프로그램의 시작·워밍업 구간 때문이다.

**② "static 변수는 스택에 있다."**
틀렸다. `static` 변수는 클래스에 하나뿐인 값이므로 **메서드 영역**에 저장된다. 스택에 있는 것은 지역 변수와 매개변수뿐이다.

**③ "new로 만든 객체는 스택에 있다."**
틀렸다. 객체 실체는 항상 **힙**에 있다. 스택에 있는 것은 그 객체를 가리키는 **참조(주소)** 일 뿐이다.

**④ "GC는 참조가 0이 되면(참조 카운팅) 객체를 지운다."**
틀렸다. 자바 GC는 **도달 가능성**을 기준으로 한다. 서로만 참조하는 순환 구조라도 GC Root에서 도달할 수 없으면 회수된다.

**⑤ "System.gc()를 호출하면 즉시 GC가 돈다."**
아니다. `System.gc()`는 JVM에게 "돌려 주면 좋겠다"는 **요청(hint)** 일 뿐, 강제가 아니다. 실무에서는 오히려 성능을 해칠 수 있어 함부로 쓰지 않는다.

**⑥ "바이트코드는 기계어다."**
틀렸다. 바이트코드는 **JVM용 중간 언어**다. 진짜 기계어는 JVM(인터프리터/JIT)이 실행 시점에 만들어 낸다.

**⑦ "실행하려면 무조건 JDK가 있어야 한다."**
개념적으로는 실행만 하려면 JRE면 충분하다. 다만 Java 11부터 JRE가 단독 배포되지 않아 실무에서는 JDK를 설치하는 것이 보통이다.

**문제 1.** 다음 코드에서 변수 `x`, 객체 `list`가 만든 ArrayList 인스턴스, 그리고 `TOTAL`은 각각 JVM 메모리의 어느 영역에 저장되는가?

```java
class Store {
    static int TOTAL = 0;
    void run() {
        int x = 5;
        java.util.List<Integer> list = new java.util.ArrayList<>();
    }
}
```

<details><summary>힌트</summary>지역 변수 / new로 만든 객체 / static 변수의 저장 위치를 각각 떠올려 보라.</details>

<details><summary>해설</summary><code>x</code>는 지역 변수이므로 <b>스택</b>. <code>list</code>라는 참조 변수 자체도 스택에 있지만, 그것이 가리키는 <b>ArrayList 객체 실체는 힙</b>에 있다. <code>TOTAL</code>은 static이므로 <b>메서드 영역</b>.</details>

**문제 2.** 자바는 컴파일 언어인가, 인터프리터 언어인가? 근거와 함께 설명하라.

<details><summary>힌트</summary>`.java` → `.class` → JVM의 두 단계를 떠올려라.</details>

<details><summary>해설</summary>둘 다이다. 소스코드는 <code>javac</code>로 <b>바이트코드로 컴파일</b>되고(컴파일적 성격), 그 바이트코드는 JVM에서 <b>인터프리팅</b>되며(인터프리터적 성격), 핫스팟은 <b>JIT로 다시 기계어로 컴파일</b>된다. 즉 하이브리드다.</details>

**문제 3.** 아래 코드에서 GC가 도는 시점에 객체 A, B, C 중 회수되는 것은? 단 `main`이 실행 중이라고 가정한다.

```java
void main() {
    Node a = new Node();   // A
    Node b = new Node();   // B
    a.next = b;            // A → B
    b.next = a;            // B → A (서로 참조)
    a = null;
    b = null;
}
```

<details><summary>힌트</summary>참조 카운팅이 아니라 도달 가능성으로 판단한다. GC Root에서 A, B로 가는 길이 남아 있는가?</details>

<details><summary>해설</summary>A와 B는 서로를 참조하지만, <code>a = null; b = null;</code> 이후 <b>GC Root(지역 변수)에서 A, B로 가는 길이 사라졌다.</b> 도달 불가능하므로 서로 참조하고 있어도 <b>둘 다 회수 대상</b>이다. 참조 카운팅이었다면 회수되지 못했을 것이다. (C는 코드에 없으므로 무시.)</details>

**문제 4.** `static int COUNT = 100;`이라는 필드가 있을 때, 클래스 로딩의 Prepare 단계와 Initialization 단계에서 `COUNT`의 값은 각각 무엇인가?

<details><summary>힌트</summary>Prepare는 "기본값으로 자리만 잡는" 단계, Initialization은 "실제 값을 채우는" 단계다.</details>

<details><summary>해설</summary>Prepare 단계에서는 <b>0</b>(int의 기본값). Initialization 단계에서 비로소 <b>100</b>이 대입된다.</details>

**문제 5.** "자바는 워밍업이 필요하다"는 말은 무슨 뜻이며, 성능 벤치마크 시 첫 실행 결과를 버리는 이유는 무엇인가?

<details><summary>힌트</summary>단계적 컴파일(Tier 0 → Tier 4)과 JIT의 동작 시점을 떠올려라.</details>

<details><summary>해설</summary>프로그램 시작 직후에는 코드가 아직 <b>인터프리터(Tier 0)</b>로 실행되어 느리다. 자주 실행되는 핫스팟이 발견되어 <b>JIT(C2, Tier 4)</b>로 컴파일되고 나서야 최고 속도가 난다. 이 과정을 워밍업이라 한다. 첫 실행은 아직 JIT 개입 전이라 진짜 정상 성능이 아니므로, 벤치마크에서는 워밍업 구간을 제외한다.</details>

## 요약

- 컴퓨터는 0과 1(기계어)만 이해하며, 프로그래밍 언어는 인간과 기계 사이의 통역이다. 자바는 소스코드를 **바이트코드**로 컴파일한 뒤 **JVM**이 실행하는 하이브리드 방식으로, "Write Once, Run Anywhere(WORA)"를 실현한다. WORA를 가능케 하는 것은 각 플랫폼별 JVM이다.
- 개발·실행 도구는 **JDK ⊃ JRE ⊃ JVM**의 포함 관계를 갖는다. JVM은 실행 엔진, JRE는 실행 환경(+라이브러리), JDK는 개발 키트(+컴파일러/도구)다.
- JVM 메모리는 **메서드 영역**(클래스 정보·static), **힙**(모든 객체), **스택**(메서드 호출·지역 변수)으로 나뉜다. 바이트코드는 **operand stack** 위에서 계산되며, 메서드 호출마다 **스택 프레임**이 쌓였다 사라진다.
- **JIT 컴파일러**는 핫스팟을 네이티브로 번역·캐싱하여 인터프리터의 느림을 극복하고, 단계적 컴파일(Tier)로 워밍업 후 높은 성능을 낸다.
- **클래스 로딩**은 로딩→링킹(검증·준비·해석)→초기화 순으로 진행되며 부모 위임으로 보안을 지킨다. **가비지 컬렉션**은 참조 카운팅이 아니라 **도달 가능성**으로 판정하며, 세대별 GC와 Mark-and-Sweep으로 동작하고 STW를 줄이는 것이 현대 GC의 과제다.
