# 4장. 객체지향이란 무엇인가? (핵심 장)

> **🐳 실습 환경 — 이 장은 `java-sandbox` 컨테이너에서 실습한다**
> ```bash
> cd java && docker compose up -d              # 컨테이너 켜기 (이미 떠 있으면 생략)
> docker exec -it java-sandbox ./run.sh PolymorphismExample   # 예제 실행
> ```
> 처음이라면 [java/README.md](../../../README.md)의 "빠른 시작"을 먼저 보라.

지금까지 우리는 변수에 데이터를 담고(2장), if와 for로 흐름을 제어했습니다(3장). 이 정도만 알아도 계산기나 간단한 로직은 얼마든지 만들 수 있습니다. 그런데 왜 Java를 "객체지향 언어"라고 부를까요? 왜 세상 거의 모든 실무 코드가 `class`라는 단어로 시작할까요?

이 장은 이 책의 심장입니다. 앞의 장들이 "문법"을 다뤘다면, 이 장은 그 문법으로 **어떻게 큰 프로그램을 무너지지 않게 조립하느냐**를 다룹니다. 객체지향(Object-Oriented Programming, OOP)은 단순한 문법 몇 개가 아니라, 프로그래밍 세계를 바꿔놓은 **패러다임 혁명**입니다. 절차적으로 위에서 아래로 명령을 나열하던 사고방식에서, 세상을 "책임을 가진 부품(객체)들의 협력"으로 바라보는 사고방식으로의 전환이죠.

이 장을 제대로 이해하면 이후의 모든 장(컬렉션, 예외 처리, 그리고 Spring까지)이 "왜 그렇게 생겼는지" 자연스럽게 보이기 시작합니다. 반대로 이 장을 대충 넘기면, 앞으로 마주칠 모든 프레임워크가 "외워야 하는 마법"처럼 느껴질 겁니다. 그래서 이 장은 이 책에서 가장 길고, 가장 천천히, 가장 많은 "왜?"를 던집니다. 편하게 따라오세요.

## 4.1 절차지향 vs 객체지향: 왜 사고방식을 바꿔야 했나?

### 정의

- **절차지향(Procedural Programming)**: 프로그램을 "해야 할 일의 순서(절차)"로 바라보는 방식. 데이터와, 그 데이터를 처리하는 함수가 분리되어 있습니다.
- **객체지향(Object-Oriented Programming)**: 프로그램을 "데이터와 그 데이터를 다루는 기능을 함께 묶은 부품(객체)들의 협력"으로 바라보는 방식.

### 원리 (왜?)

절차지향은 **레시피**와 같습니다.

```
[라면 끓이기 - 절차지향]
1. 물을 끓인다
2. 면을 넣는다
3. 스프를 넣는다
4. 3분 기다린다
5. 그릇에 담는다
   ↓ 위에서 아래로 순서대로 실행
```

C 언어가 대표적입니다. 프로그램이 작을 때는 이 방식이 오히려 명확하고 빠릅니다. 문제는 프로그램이 **커질 때** 발생합니다.

호텔 주방을 상상해 보세요. 메뉴가 100개입니다. 절차지향으로 짜면 레시피 100개가 위에서 아래로 주르륵 나열됩니다.

```
[호텔 주방 - 절차지향의 지옥]
1. 물 끓임      ← 라면? 파스타? 국밥? 어디부터가 하나의 메뉴지?
2. 양파 손질
3. 면 넣음
4. 마늘 다짐    ← "양파 손질"이랑 뭐가 다르지? 중복인가?
5. 육수 끓임
...
473. 소금 간
```

여기서 터지는 문제들:

1. **경계가 모호하다**: 어디부터 어디까지가 "하나의 메뉴"인지 코드만 봐서는 알기 어렵습니다.
2. **중복이 넘친다**: "재료 손질"이 여러 메뉴에서 반복됩니다.
3. **수정이 위험하다**: 육수 끓이는 로직 하나를 바꿨더니, 그 로직을 공유하던 다른 5개 메뉴가 동시에 망가집니다.
4. **새 개발자가 이해 못 한다**: 473줄짜리 함수를 처음 보는 사람이 어떻게 파악하겠어요?

객체지향은 이 문제를 **레고 블록**처럼 해결합니다.

```
[자동차 조립 - 객체지향]

  ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
  │  바퀴  │   │  차체  │   │  엔진  │   │  핸들  │
  │ (독립) │   │ (독립) │   │ (독립) │   │ (독립) │
  └────┬───┘   └────┬───┘   └────┬───┘   └────┬───┘
       └────────────┴─────┬──────┴────────────┘
                          ▼
                   ┌─────────────┐
                   │  자동차 완성 │
                   └─────────────┘

각 블록은:  독립적  /  재사용 가능  /  교체 가능
```

엔진 블록이 고장 나면 엔진 블록만 갈아 끼우면 됩니다. 바퀴는 건드리지 않아도 되죠. 각 부품이 **자기 데이터와 자기 기능을 스스로 책임집니다.**

### 왜 등장했나? — 소프트웨어 위기(Software Crisis)

1960~70년대, 하드웨어는 급격히 발전하는데 소프트웨어는 점점 복잡해져서 **버그 잡기, 수정하기, 이해하기가 사람의 한계를 넘어서는** 사태가 벌어졌습니다. 이것을 "소프트웨어 위기"라고 부릅니다. 프로젝트가 예산 초과, 일정 지연, 품질 저하로 줄줄이 실패했죠.

이때 나온 해결책의 핵심 아이디어가 이것입니다:

> **"복잡한 시스템을 독립적인 부품(객체)으로 나누자. 그리고 각 부품이 자기 데이터와 자기 기능을 스스로 관리하게 하자."**

이것이 객체지향의 존재 이유입니다. 객체지향은 "멋있어서" 나온 게 아니라, **복잡함을 다스리기 위해** 나왔습니다. 이 관점을 잊지 마세요. 앞으로 배울 모든 개념(캡슐화, 상속, 다형성)은 전부 "복잡함을 어떻게 다스리는가"에 대한 답입니다.

### 코드로 느껴보기

```java
// 절차지향 스타일 (데이터와 기능이 분리)
int balance = 10000;

int deposit(int currentBalance, int amount) {
    return currentBalance + amount;
}
// 문제: balance를 아무나, 어디서나 마음대로 바꿀 수 있다.
//       balance = -999999; 도 막을 방법이 없다.

// ────────────────────────────────────

// 객체지향 스타일 (데이터 + 기능을 하나로 묶음)
class Account {
    private int balance = 10000;   // 데이터를 객체 안에 가둔다

    void deposit(int amount) {     // 기능도 객체 안에 있다
        this.balance += amount;
    }
}
// balance는 Account가 스스로 책임진다. 밖에서 함부로 못 건드린다.
```

> **면접 팁**: "절차지향과 객체지향의 차이를 한마디로?"라고 물으면 이렇게 답하세요. "절차지향은 데이터와 기능이 분리되어 있고, 객체지향은 데이터와 그것을 다루는 기능을 하나의 객체로 묶습니다. 이렇게 묶으면 복잡한 시스템을 독립적이고 교체 가능한 부품으로 나눌 수 있습니다."

## 4.2 클래스와 객체: 설계도와 실체

### 정의

- **클래스(Class)**: 객체를 만들기 위한 **설계도(틀)**. 상태(필드)와 행동(메서드)이 어떻게 생겼는지 정의합니다.
- **객체(Object) / 인스턴스(Instance)**: 클래스라는 설계도로 **실제로 만들어낸 실체**. 메모리 위에 존재하는 진짜 물건입니다.

### 원리 (왜?)

설계도(클래스)는 자동차가 **아닙니다.** 종이 위의 그림일 뿐이라 달릴 수 없습니다. 설계도로 공장에서 실제로 찍어낸 자동차, 그것이 객체입니다.

```
[Car 클래스 = 설계도]
┌─────────────────────────┐
│         Car             │
├─────────────────────────┤
│ 속성(필드):             │
│   - color               │
│   - speed               │
│   - fuel                │
├─────────────────────────┤
│ 행동(메서드):           │
│   + drive()             │
│   + stop()              │
│   + refuel()            │
└─────────────────────────┘
        │ new Car()  (설계도로 실체를 찍어냄)
        ├──────────────┬──────────────┐
        ▼              ▼              ▼
   ┌────────┐    ┌────────┐    ┌────────┐
   │ 빨간 차 │    │ 파란 차 │    │ 흰 차   │
   │color=RED│   │color=BLU│   │color=WHT│
   └────────┘    └────────┘    └────────┘
```

같은 설계도 하나로 색이 다른 자동차를 여러 대 만들 수 있습니다. 각 자동차는 **독립적인 상태**를 가집니다. 빨간 차의 속도를 올려도 파란 차는 그대로입니다.

익숙한 비유들:

| 설계도 (클래스) | 실체 (객체) |
| :--- | :--- |
| 붕어빵 틀 | 붕어빵 |
| 쿠키 커터 | 쿠키 |
| 주민등록증 양식 | 당신의 주민등록증 |

### 핵심 공식

> **필드(상태) + 메서드(행동) = 객체**

객체는 "어떤 상태를 가지고 있고, 어떤 행동을 할 수 있는가"로 정의됩니다.

| 클래스 | 필드(상태) | 메서드(행동) |
| :--- | :--- | :--- |
| 학생(Student) | 이름, 학번, 성적 | 공부하다, 시험보다 |
| 은행계좌(Account) | 계좌번호, 잔액, 소유자 | 입금, 출금, 조회 |
| 스마트폰(Phone) | 배터리, 모델명, 저장공간 | 전화걸다, 앱실행, 충전 |

### 코드로 확인 — 클래스 정의와 객체 생성

```java
// 1) 클래스 정의 (설계도 그리기)
public class Car {
    // 필드 (상태)
    String color;
    int speed;
    int fuel;

    // 메서드 (행동)
    void drive() {
        speed += 10;
        System.out.println(color + " 자동차가 달립니다. 속도: " + speed);
    }

    void stop() {
        speed = 0;
        System.out.println("정지했습니다.");
    }
}

// 2) 객체 생성 (설계도로 실체 찍어내기)
public class Main {
    public static void main(String[] args) {
        Car redCar = new Car();   // new 키워드로 실체를 만든다
        redCar.color = "빨강";
        redCar.drive();           // "빨강 자동차가 달립니다. 속도: 10"

        Car blueCar = new Car();  // 또 다른 독립적인 실체
        blueCar.color = "파랑";
        blueCar.drive();          // "파랑 자동차가 달립니다. 속도: 10"
        // redCar의 speed는 여전히 10, blueCar의 speed와 무관하다
    }
}
```

`new Car()`가 실행되는 순간, 메모리(정확히는 힙 영역)에 새로운 Car 실체가 하나 태어납니다. `redCar`와 `blueCar`는 서로 다른 메모리 주소를 가리키는, 완전히 별개의 객체입니다.

### 생성자(Constructor): 객체가 태어날 때의 초기 설정

객체를 만들 때마다 `color`, `speed`를 일일이 따로 설정하는 건 번거롭고 실수하기 쉽습니다. **생성자**는 객체가 태어나는 순간 초기값을 세팅해주는 특별한 메서드입니다.

```java
public class Car {
    String color;
    int speed;

    // 생성자: 클래스 이름과 똑같고, 반환 타입이 없다
    public Car(String color) {
        this.color = color;   // this는 "지금 만들어지는 이 객체"를 가리킨다
        this.speed = 0;       // 태어날 때 속도는 항상 0
    }
}

// 사용
Car redCar = new Car("빨강");   // 태어나자마자 color="빨강", speed=0 확정
```

생성자가 있으면 "속도를 0으로 초기화하는 걸 깜빡했다" 같은 실수를 원천 차단할 수 있습니다. 객체는 **항상 유효한 상태로 태어나야 한다**는 원칙을 지켜주는 장치입니다.

### this 키워드

위 코드에서 `this.color = color`가 눈에 걸렸을 겁니다. `this`는 **"지금 이 메서드를 실행하고 있는 바로 그 객체 자신"**을 가리키는 참조입니다.

```java
public Car(String color) {
    this.color = color;
    //   ↑         ↑
    //  객체의     생성자가
    //  필드       받은 매개변수
}
```

매개변수 이름(`color`)과 필드 이름(`color`)이 같을 때, `this.color`는 "이 객체의 필드", 그냥 `color`는 "매개변수"를 뜻합니다. `this`가 없으면 둘을 구분할 수 없어 `color = color`가 되어버려 아무 일도 일어나지 않습니다.

## 4.3 캡슐화: 왜 숨기는가?

### 정의

**캡슐화(Encapsulation)**: 객체의 데이터(필드)를 외부로부터 숨기고, 정해진 통로(공개 메서드)로만 접근하게 하는 것. "복잡한 내부는 감추고, 필요한 조작 버튼만 노출한다."

### 원리 (왜?)

운전자와 엔진을 생각해 봅시다. 운전자는 핸들, 페달, 기어만 알면 운전할 수 있습니다. 엔진 내부에서 연료가 어떻게 폭발하고 피스톤이 어떻게 움직이는지 **몰라도 됩니다.** 아니, 오히려 몰라야 편합니다.

```
[캡슐화된 자동차]

  운전자가 만지는 것 (public - 공개 인터페이스)
  ┌──────────────────────────────┐
  │   🕹 핸들   ⏸ 페달   ⚙ 기어   │
  └──────────────┬───────────────┘
                 │ 이 통로로만 접근 가능
  ═══════════════╪═══════════════════  ← 캡슐(경계)
                 │
  ┌──────────────▼───────────────┐
  │   숨겨진 것 (private)         │
  │   🔧 엔진   🔩 변속기         │
  │   ⛽ 연료분사장치            │
  │   (운전자는 몰라도 됨)        │
  └──────────────────────────────┘
```

### 왜 숨기는가? — 세 가지 이유

**① 보호 (잘못된 사용 방지)**

필드를 그냥 열어두면 이런 말도 안 되는 일이 가능합니다.

```java
person.age = -5;       // 나이가 -5살?
person.age = 99999;    // 99999살?
```

캡슐화하면 setter라는 검문소를 통과해야 하므로 이상한 값을 막을 수 있습니다.

```java
public class Person {
    private int age;    // private: 외부에서 직접 접근 불가

    public int getAge() {          // getter: 읽기 통로
        return age;
    }

    public void setAge(int age) {  // setter: 쓰기 통로 + 검증
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("나이는 0~150 사이여야 합니다: " + age);
        }
        this.age = age;
    }
}

// 사용
Person p = new Person();
p.setAge(30);     // OK
// p.age = -5;    // 컴파일 에러! private이라 접근 불가
p.setAge(-5);     // 예외 발생! 검문소가 막아준다
```

**② 유연성 (내부를 바꿔도 외부는 영향 없음)**

가솔린 엔진을 전기 모터로 통째로 교체해도, 운전자는 여전히 똑같이 페달을 밟고 핸들을 돌립니다. 운전법(공개 인터페이스)이 바뀌지 않았기 때문이죠.

코드도 마찬가지입니다. `getAge()`의 내부 구현을 바꿔도(예: 나이를 생년월일로부터 계산하도록 변경), `getAge()`를 호출하는 수천 곳의 코드는 **단 한 줄도 바꿀 필요가 없습니다.**

```java
// 내부 저장 방식이 바뀌어도...
public int getAge() {
    // return this.age;                          // (이전 방식)
    return LocalDate.now().getYear() - birthYear; // (새 방식)
}
// 이 메서드를 쓰는 바깥 코드 person.getAge()는 그대로 동작한다!
```

**③ 복잡성 관리**

TV 리모컨을 생각하세요. 버튼만 누르면 됩니다. 리모컨 내부의 적외선 회로가 어떻게 신호를 쏘는지 알 필요가 없죠. `public` 메서드가 "버튼"이고, `private` 필드와 메서드가 "내부 회로"입니다. 사용자는 버튼만 알면 되니 머리가 편안합니다.

> **면접 팁**: "캡슐화를 왜 하나요?" → **데이터 보호 / 유연성(변경 용이성) / 복잡성 관리**. 이 세 단어를 순서대로 말하면 완벽합니다. 특히 "getter/setter를 왜 쓰냐"는 질문에 "관례라서요"라고 답하면 안 됩니다. "setter에 검증 로직을 넣어 객체를 항상 유효한 상태로 유지하고, 내부 구현을 바꿔도 외부에 영향을 주지 않기 위해서"라고 답하세요.

### 접근 제어자 정리

| 제어자 | 접근 범위 | 흔한 용도 |
| :--- | :--- | :--- |
| `private` | 같은 클래스 안에서만 | 필드(데이터)는 대부분 이것 |
| `default`(없음) | 같은 패키지 안에서 | 패키지 내부용 |
| `protected` | 같은 패키지 + 자식 클래스 | 상속에서 자식이 쓸 것 |
| `public` | 어디서나 | 공개 메서드(버튼) |

원칙: **필드는 private, 통로(메서드)는 public**. 이것이 캡슐화의 기본 자세입니다.

## 4.4 상속: 코드의 유전자

### 정의

**상속(Inheritance)**: 기존 클래스(부모)의 필드와 메서드를 새 클래스(자식)가 물려받는 것. 자식은 부모의 모든 것을 갖고, 거기에 자기만의 것을 추가하거나 바꿀 수 있습니다.

### 원리 (왜?)

부모가 자식에게 특성을 물려주듯, 클래스도 물려줍니다.

```
[상속 구조]
              ┌──────────────────┐
              │     Animal       │  (부모)
              ├──────────────────┤
              │ 필드: name, age  │
              │ 메서드: eat()    │
              │         sleep()  │
              └───────┬──────────┘
             extends  │  extends
          ┌───────────┴───────────┐
          ▼                       ▼
   ┌──────────────┐        ┌──────────────┐
   │     Dog      │        │     Cat      │  (자식)
   ├──────────────┤        ├──────────────┤
   │ + bark()     │        │ + scratch()  │
   │ (name, age,  │        │ (name, age,  │
   │  eat, sleep  │        │  eat, sleep  │
   │  자동 상속)  │        │  자동 상속)  │
   └──────────────┘        └──────────────┘
```

Dog은 `bark()`만 새로 정의했을 뿐인데, `name`, `age`, `eat()`, `sleep()`을 전부 공짜로 물려받습니다.

### 왜 상속을 쓰나?

1. **코드 중복 제거**: `eat()`, `sleep()`을 Dog에도 쓰고 Cat에도 쓰고... 대신 부모 Animal에 **딱 한 번** 정의하면 됩니다.
2. **다형성의 기반**: (4.5에서 배울) 다형성이라는 강력한 기능은 상속 없이는 성립하지 않습니다.

### 코드로 확인 — extends, super, @Override

```java
// 부모 클래스
public class Animal {
    protected String name;
    protected int age;

    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public void eat() {
        System.out.println(name + "이(가) 먹습니다.");
    }

    public void makeSound() {
        System.out.println("...");
    }
}

// 자식 클래스
public class Dog extends Animal {   // extends: "Animal을 상속받는다"

    public Dog(String name, int age) {
        super(name, age);   // super: 부모의 생성자를 호출한다
    }

    public void bark() {    // 자식만의 새 메서드
        System.out.println(name + ": 멍멍!");
    }

    @Override               // 부모의 메서드를 재정의(오버라이딩)
    public void makeSound() {
        System.out.println(name + ": 멍멍!");
    }
}
```

- **`extends`**: 상속을 선언하는 키워드입니다.
- **`super`**: 부모를 가리킵니다. `super(name, age)`는 부모 생성자 호출, `super.eat()`은 부모의 메서드 호출입니다. 자식 생성자의 첫 줄은 반드시 부모 생성자 호출(`super(...)`)이어야 합니다(생략하면 컴파일러가 `super()`를 자동으로 넣습니다).
- **`@Override`**: "이 메서드는 부모의 메서드를 재정의한 것"이라고 컴파일러에게 알리는 표시(애너테이션)입니다. 오타로 잘못된 재정의를 하면 컴파일러가 잡아줍니다.

### ⚠️ 상속의 위험: 너무 깊은 상속 체인

상속은 강력하지만 남용하면 독이 됩니다.

```
[지나치게 깊은 상속]
생물 → 동물 → 포유류 → 영장류 → 사람 → 학생 → 대학생 → 컴공학생

문제:
- 컴공학생의 어떤 메서드가 어느 조상에서 왔는지 추적 불가
- '포유류'를 수정하면 아래 5단계가 전부 영향받음
- 새 개발자는 8단계를 다 읽어야 이해 가능
```

> **황금 규칙**: 상속 깊이는 **2~3단계 이내**로 유지하세요.

### 상속보다 합성 (Composition over Inheritance)

이것은 실무에서 정말 중요한 원칙입니다.

- **상속(IS-A, "나는 ~이다")**: `class 전기차 extends 자동차` → "전기차는 자동차**이다**"
- **합성(HAS-A, "나는 ~을 가진다")**: `class 자동차 { Engine engine; }` → "자동차는 엔진을 **가진다**"

```
[상속 - IS-A]                 [합성 - HAS-A]
                              ┌────────────────┐
전기차 extends 자동차          │     자동차     │
  ↑ 부모에 강하게 묶임         ├────────────────┤
  ↑ 부모 바뀌면 자식 다 흔들림 │  Engine engine ├──▶ [Engine]
                              │  (부품을 소유)  │    교체 가능!
                              └────────────────┘
```

```java
// 상속 방식 - 부모에 강하게 결합됨
class ElectricCar extends Car { ... }

// 합성 방식 - 부품을 소유함 (교체 가능)
class Car {
    private Engine engine;   // 엔진을 "가진다"

    public Car(Engine engine) {
        this.engine = engine;
    }

    public void setEngine(Engine engine) {  // 런타임에 엔진 교체 가능!
        this.engine = engine;
    }
}
```

**합성의 장점:**

- **런타임 교체 가능**: 실행 중에 엔진을 갈아 끼울 수 있습니다(유연성).
- **느슨한 결합**: Engine을 수정해도 Car가 받는 영향이 적습니다.

**왜 합성을 선호하나?** 상속은 프로그래밍에서 **가장 강한 결합**입니다. 부모가 조금만 변해도 모든 자식이 흔들립니다. 반면 합성은 인터페이스만 유지하면 부품끼리 느슨하게 연결되죠.

> **원칙**: "정말로 IS-A 관계(진짜 그 종류이다)일 때만 상속하고, 그 외에는 합성을 사용하라." 예를 들어 "학생은 사람이다"는 상속이 맞지만, "자동차는 엔진을 가진다"를 "자동차는 엔진이다(상속)"로 만들면 재앙입니다.

## 4.5 다형성: 같은 명령, 다른 행동

### 정의

**다형성(Polymorphism)**: 같은 명령(메서드 호출)에 대해 객체마다 다르게 반응하는 것. "여러(poly) 형태(morph)"라는 뜻입니다.

### 원리 (왜?)

동물들에게 "말해봐!"라고 외치면:

```
        "말해봐!" (같은 명령: makeSound())
            │
   ┌────────┼────────┐
   ▼        ▼        ▼
 [개]     [고양이]   [오리]
 "멍멍!"  "야옹!"   "꽥꽥!"
   └── 같은 명령, 다른 행동 = 다형성 ──┘
```

명령은 똑같이 `makeSound()` 하나지만, 실제 객체가 무엇이냐에 따라 다른 결과가 나옵니다.

### 업캐스팅과 동적 바인딩

다형성의 핵심 마법은 이것입니다.

```java
Animal myPet = new Dog();   // 자식 객체를 부모 타입 변수에 담는다 (업캐스팅)
myPet.makeSound();          // 결과: "멍멍!"  ← Dog의 makeSound가 실행됨!
```

`myPet`의 **타입은 Animal**이지만, 실제로 담긴 **객체는 Dog**입니다. `makeSound()`를 호출하면 컴파일러가 "Animal의 makeSound"를 부를 것 같지만, 실행 시점에는 **실제 객체인 Dog의 makeSound가 실행됩니다.**

택배 상자에 비유하면: 상자 겉면에는 "과일"이라고 적혀 있지만(타입은 Animal), 열어보니 "사과"가 들어있는(실제 객체는 Dog) 격입니다. 열었을 때 나오는 건 실제 내용물입니다.

이 "실행 시점에 실제 객체의 메서드를 찾아 실행하는" 동작을 **동적 바인딩(Dynamic Binding)** 또는 **동적 메서드 디스패치(Dynamic Method Dispatch)**라고 부릅니다.

```
[동적 바인딩의 원리]

컴파일 시점:  myPet.makeSound()  → "makeSound라는 메서드가 있긴 하네" (타입만 체크)
                    │
실행 시점:    myPet이 실제로 가리키는 객체는? → Dog!
                    │
              Dog의 makeSound()를 찾아 실행 → "멍멍!"

즉, "어느 메서드를 부를지"는 컴파일 때가 아니라 실행 때 결정된다.
```

이것이 왜 중요할까요? 컴파일러는 `Animal` 타입만 보고도 코드를 작성하게 해주고, 실제 동작은 실행 시점에 진짜 객체가 결정합니다. 덕분에 "구체적인 것을 몰라도 추상적인 것에 기대어 코드를 짤 수 있게" 됩니다.

### 왜 강력한가? — 결제 시스템 예시

실무의 결제 시스템을 봅시다.

```
              ┌──────────────────┐
              │   Payment (부모)  │
              │   + pay()        │  ← 추상적인 "결제한다"
              └────────┬─────────┘
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
 ┌────────────┐ ┌────────────┐ ┌──────────────┐
 │ KakaoPay   │ │ NaverPay   │ │ CardPayment  │
 │ pay(){카카오}│ │ pay(){네이버}│ │ pay(){카드}   │
 └────────────┘ └────────────┘ └──────────────┘
```

**다형성이 없다면** 이렇게 짜야 합니다.

```java
// 지옥의 if-else
if (method.equals("카카오")) {
    // 카카오페이 결제 로직 30줄...
} else if (method.equals("네이버")) {
    // 네이버페이 결제 로직 30줄...
} else if (method.equals("카드")) {
    // 카드 결제 로직 30줄...
}
// 페이코, 토스, 애플페이... 결제수단이 늘 때마다 else if가 끝없이 추가된다.
```

**다형성이 있다면:**

```java
Payment payment = getPaymentMethod();  // 실제로 뭐가 올지 몰라도 됨
payment.pay();                         // 알아서 맞는 결제가 실행된다

// 새 결제수단(예: 토스페이)이 생기면?
// → class TossPay extends Payment { ... } 새 클래스만 추가하면 끝!
// → 기존 코드는 단 한 줄도 안 바꾼다.
```

이것이 바로 **OCP(Open-Closed Principle, 개방-폐쇄 원칙)**입니다. "확장에는 열려 있고(새 결제수단 추가 자유), 수정에는 닫혀 있다(기존 코드 건드릴 필요 없음)."

### 오버라이딩 vs 오버로딩 — 반드시 구분하라

이 둘은 이름이 비슷해서 초보자가 자주 헷갈립니다. 완전히 다른 개념입니다.

| 구분 | 오버라이딩(Overriding) | 오버로딩(Overloading) |
| :--- | :--- | :--- |
| 뜻 | 부모 메서드를 자식이 **재정의** | 같은 이름 메서드를 **여러 개** 정의 |
| 관계 | 상속 관계에서 발생 | 같은 클래스 안에서 발생 |
| 시그니처 | **같아야** 함(이름·매개변수 동일) | 매개변수가 **달라야** 함 |
| 결정 시점 | 실행 시점(동적) | 컴파일 시점(정적) |
| 키워드 | `@Override` | (없음) |

```java
// 오버로딩: 같은 이름, 다른 매개변수 (한 클래스 안)
class Calculator {
    int add(int a, int b)          { return a + b; }
    double add(double a, double b) { return a + b; }   // 매개변수 타입이 다름
    int add(int a, int b, int c)   { return a + b + c; } // 매개변수 개수가 다름
}

// 오버라이딩: 부모 메서드를 자식이 재정의 (상속 관계)
class Animal { void makeSound() { System.out.println("..."); } }
class Cat extends Animal {
    @Override
    void makeSound() { System.out.println("야옹!"); }   // 시그니처 동일, 내용만 변경
}
```

기억법: **오버로딩(overLOAD)은 메서드를 "잔뜩 쌓는(load)" 것**, **오버라이딩(overRIDE)은 부모 위에 "덮어쓰는(ride over)" 것**.

## 4.6 추상 클래스 vs 인터페이스: 언제 무엇을?

### 면접에 반드시 나온다

이 주제는 자바 면접의 단골입니다. 그런데 대부분 "문법 차이"만 외웁니다(추상 클래스는 하나만, 인터페이스는 여러 개...). 진짜 중요한 건 **설계 철학의 차이**입니다.

### 추상 클래스: "나는 ~이다" (IS-A)

**추상 클래스(Abstract Class)**는 **불완전한 설계도**입니다. 일부는 완성되어 있고, 일부는 "이건 자식이 반드시 채워라"라고 비워둡니다.

```
[추상 클래스 Animal]
┌─────────────────────────────┐
│   Animal (abstract)         │
├─────────────────────────────┤
│ name, age       ← 완성됨    │
│ eat()           ← 완성됨    │
│ makeSound()     ← ??? 미완성│ (abstract, 몸통 없음)
└──────────────┬──────────────┘
       ┌───────┴───────┐
       ▼               ▼
 ┌──────────┐    ┌──────────┐
 │   Dog    │    │   Cat    │
 │makeSound │    │makeSound │  ← 자식이 반드시 구현
 │="멍멍!"  │    │="야옹!"  │
 └──────────┘    └──────────┘
```

```java
public abstract class Animal {
    protected String name;

    public void eat() {                      // 완성된 메서드 (공통 상태/행동)
        System.out.println(name + " 먹는 중");
    }

    public abstract void makeSound();        // 미완성 (몸통 {}이 없음!)
    //     ↑ 자식이 반드시 구현해야 함
}

public class Dog extends Animal {
    @Override
    public void makeSound() {                // 자식이 완성시킴
        System.out.println("멍멍!");
    }
}
```

추상 클래스는 공통된 **상태(필드)와 행동을 공유**하면서, 일부는 자식이 반드시 구현하도록 강제할 때 씁니다. `abstract` 클래스는 그 자체로 `new` 할 수 없습니다(불완전하니까요).

### 인터페이스: "나는 ~할 수 있다" (CAN-DO)

**인터페이스(Interface)**는 **능력의 계약서**입니다. "이 능력을 구현하겠다고 약속한다"는 명세죠.

```
[인터페이스 = 능력]
 ┌──────────┐  ┌───────────┐  ┌──────────┐
 │ Flyable  │  │ Swimmable │  │ Runnable │
 │  fly()   │  │  swim()   │  │  run()   │
 └────┬─────┘  └─────┬─────┘  └────┬─────┘
      │  ┌───────────┘             │
      ▼  ▼                         ▼
   ┌───────────┐              ┌─────────┐
   │   Duck    │              │   Dog   │
   │fly + swim │              │  run    │
   │(둘 다 구현)│              └─────────┘
   └───────────┘
```

```java
public interface Flyable {
    void fly();       // 인터페이스의 메서드는 기본적으로 몸통이 없다
}
public interface Swimmable {
    void swim();
}

// 오리는 날 수도, 헤엄칠 수도 있다 → 여러 인터페이스 동시 구현!
public class Duck implements Flyable, Swimmable {
    @Override public void fly()  { System.out.println("오리가 난다"); }
    @Override public void swim() { System.out.println("오리가 헤엄친다"); }
}
```

- **클래스는 단 하나만 상속**할 수 있습니다(`extends`는 하나).
- **인터페이스는 여러 개 구현**할 수 있습니다(`implements A, B, C`).

이것이 자바가 "다중 상속의 혼란"을 피하면서도 "여러 능력을 조합"할 수 있게 해준 설계입니다.

### 선택 가이드

```
질문 1) IS-A인가, CAN-DO인가?
        "Dog은 Animal이다"     → 추상 클래스
        "Duck은 날 수 있다"    → 인터페이스

질문 2) 공통 상태(필드)를 공유해야 하나?
        YES → 추상 클래스
        NO  → 인터페이스

질문 3) 여러 개를 동시에 구현해야 하나?
        YES → 인터페이스 (자바는 다중 상속 불가!)
```

| 비교 항목 | 추상 클래스 | 인터페이스 |
| :--- | :--- | :--- |
| 관계 | IS-A ("~이다") | CAN-DO ("~할 수 있다") |
| 상태(필드) | 가질 수 있음 | 상수만 가능(원칙적) |
| 다중 적용 | 불가(단일 상속) | 가능(다중 구현) |
| 키워드 | `abstract`, `extends` | `interface`, `implements` |

### Java 8 이후 — default 메서드

원래 인터페이스는 몸통 없는 메서드만 가질 수 있었습니다. 그런데 Java 8부터 **default 메서드**가 추가되어, 인터페이스도 구현된 메서드를 가질 수 있게 됐습니다.

```java
public interface Flyable {
    void fly();

    default void land() {                    // default: 기본 구현 제공
        System.out.println("착륙합니다");
    }
}
```

**왜 이런 게 생겼을까요?** 이미 100개의 클래스가 구현 중인 인터페이스에 새 메서드를 추가한다고 상상해 보세요. 그러면 100개 클래스가 전부 그 메서드를 구현해야 컴파일이 됩니다 — 대참사죠. `default`는 "기본 구현을 제공하되, 필요한 클래스만 재정의하라"는 해결책입니다. 기존 100개 클래스는 손대지 않아도 됩니다.

> **현대 트렌드**: 요즘은 추상 클래스보다 **인터페이스를 선호**합니다. 다중 구현이 가능하고, default 메서드로 유연하게 확장할 수 있으며, Spring 같은 프레임워크가 인터페이스 기반으로 설계되어 있기 때문입니다.

## 4.7 SOLID 원칙: 좋은 객체지향 설계의 나침반

지금까지 배운 캡슐화·상속·다형성을 "어떻게 잘 쓸 것인가"에 대한 다섯 가지 지침이 SOLID입니다. 각 원칙의 앞 글자를 딴 이름입니다. 지금은 "이런 게 있구나" 정도로 눈에 익혀두고, 실무를 하면서 점점 체감하면 됩니다.

| 글자 | 원칙 | 뜻 | 비유 |
| :---: | :--- | :--- | :--- |
| **S** | 단일 책임 (SRP) | 하나의 클래스는 하나의 책임만 가진다 | 요리사는 요리만, 서빙은 서버가 |
| **O** | 개방-폐쇄 (OCP) | 확장에는 열려 있고, 수정에는 닫혀 있다 | 플러그를 꽂으면 기능이 추가됨 |
| **L** | 리스코프 치환 (LSP) | 자식은 언제나 부모를 대체할 수 있어야 한다 | 알바생도 정직원의 일을 할 수 있다 |
| **I** | 인터페이스 분리 (ISP) | 큰 인터페이스 하나보다 작은 여러 개로 | 만능 리모컨보다 TV/에어컨 리모컨 따로 |
| **D** | 의존성 역전 (DIP) | 구체적인 것이 아니라 추상적인 것에 의존하라 | "삼성 TV"가 아니라 그냥 "TV"에 기대라 |

이 중 **OCP**는 이미 4.5 결제 시스템에서 다형성으로 실현했고, **DIP**는 6장 이후 Spring을 배울 때 "인터페이스에 의존하라"는 형태로 계속 만나게 됩니다. SOLID는 외우는 규칙이 아니라, "왜 이 코드가 유지보수하기 좋은가/나쁜가"를 판단하는 안목입니다.

## 4.8 static vs instance: 개별의 것과 공용의 것

객체지향을 배울 때 반드시 짚어야 할 개념이 하나 더 있습니다. `static` 키워드입니다.

### 정의

- **인스턴스 멤버(instance member)**: 객체마다 **따로** 존재. 객체를 만들어야 쓸 수 있음.
- **정적 멤버(static member)**: 클래스에 **하나만** 존재하며 모든 객체가 **공유**. 객체 없이 클래스 이름으로 접근.

### 원리 (왜?)

```
[static vs instance]

     Student 클래스
     ┌────────────────────────────┐
     │ static int count;  ← 공용   │ ← 모든 학생이 공유하는 "총 학생 수"
     └────────────────────────────┘
              │
   ┌──────────┼──────────┐
   ▼          ▼          ▼
[학생A]     [학생B]     [학생C]
name="김"  name="이"  name="박"   ← name은 각자 따로(instance)
   └──────────┴──────────┘
        모두 같은 count를 봄
```

"학생의 이름"은 학생마다 다르니 인스턴스 필드입니다. 하지만 "전체 학생 수"는 개별 학생의 소유가 아니라 클래스 전체의 정보이므로 static이 어울립니다.

```java
public class Student {
    static int count = 0;   // 정적: 모든 객체가 공유
    String name;            // 인스턴스: 객체마다 따로

    public Student(String name) {
        this.name = name;
        count++;            // 학생이 태어날 때마다 공용 카운터 증가
    }
}

Student a = new Student("김");   // count = 1
Student b = new Student("이");   // count = 2
System.out.println(Student.count);  // 2  ← 객체가 아니라 클래스 이름으로 접근
System.out.println(a.name);         // "김" ← 이건 객체별
```

`Math.random()`, `Integer.parseInt()`처럼 객체를 안 만들고 바로 쓰는 메서드들이 전부 static 메서드입니다. **static 메서드 안에서는 인스턴스 필드나 `this`를 쓸 수 없습니다.** static은 특정 객체에 소속된 게 아니기 때문에 "어느 객체의 this?"라고 물으면 답이 없거든요.

## ⚠️ 흔한 오해와 함정

**함정 1: "클래스와 객체는 같은 말 아닌가요?"**
아닙니다. 클래스는 **설계도**, 객체는 그 설계도로 만든 **실체**입니다. 붕어빵 틀(클래스)과 붕어빵(객체)이 다르듯이요. `new`를 하는 순간 객체가 태어납니다.

**함정 2: "getter/setter는 그냥 관례라 다 만들어야 한다"**
그렇지 않습니다. 무의미한 getter/setter를 전부 만들면 캡슐화가 오히려 깨집니다(사실상 필드를 public으로 여는 것과 같아짐). setter에 검증 로직이 없다면, 정말 그 setter가 필요한지 의심해 보세요. 필요한 통로만 여는 게 원칙입니다.

**함정 3: "상속을 많이 쓸수록 객체지향을 잘하는 것이다"**
정반대입니다. 실무에서는 상속을 **아껴 쓰고** 합성을 선호합니다. 깊은 상속은 유지보수의 악몽입니다. "IS-A가 진짜 성립할 때만 상속"이 정답입니다.

**함정 4: "오버라이딩과 오버로딩은 비슷한 거다"**
전혀 다릅니다. 오버라이딩은 상속 관계에서 부모 메서드를 **재정의**(시그니처 동일), 오버로딩은 같은 클래스에서 이름만 같고 매개변수가 **다른** 메서드를 여러 개 만드는 것입니다.

**함정 5: "`Animal a = new Dog();`에서 a.makeSound()는 Animal 걸 부른다"**
아닙니다. **동적 바인딩** 때문에 실제 객체인 Dog의 makeSound()가 실행됩니다. 이게 다형성의 핵심입니다. 타입이 아니라 실제 객체가 메서드를 결정합니다.

**함정 6: "static 메서드에서 this를 쓸 수 있다"**
없습니다. static은 특정 객체에 속하지 않으므로 `this`도, 인스턴스 필드도 접근 불가입니다.

## 연습문제

**문제 1.** `BankAccount` 클래스를 캡슐화하여 설계하세요. `balance`(잔액) 필드는 외부에서 직접 접근할 수 없어야 하고, `deposit(int amount)`와 `withdraw(int amount)` 메서드로만 조작할 수 있어야 합니다. 출금 시 잔액보다 많이 뽑으려 하면 예외를 던지세요.

> **힌트**: `balance`는 `private`. withdraw에서 `if (amount > balance)` 검증.
>
> **해설**:
> ```java
> public class BankAccount {
>     private int balance = 0;
>     public void deposit(int amount) {
>         if (amount <= 0) throw new IllegalArgumentException("입금액은 양수여야 함");
>         balance += amount;
>     }
>     public void withdraw(int amount) {
>         if (amount > balance) throw new IllegalArgumentException("잔액 부족");
>         balance -= amount;
>     }
>     public int getBalance() { return balance; }
> }
> ```
> 핵심은 `balance`를 private으로 숨기고, deposit/withdraw라는 통로에 검증을 넣어 잘못된 상태(음수 잔액 등)를 원천 차단한 것입니다.

**문제 2.** 다음 코드의 출력 결과를 예측하고 이유를 설명하세요.
```java
Animal a = new Cat();   // Cat extends Animal, Cat이 makeSound를 "야옹"으로 오버라이드
a.makeSound();
```

> **힌트**: 타입은 Animal, 실제 객체는 Cat. 동적 바인딩을 떠올리세요.
>
> **해설**: 출력은 **"야옹!"** 입니다. 변수의 타입은 Animal이지만, 실제로 담긴 객체는 Cat입니다. 자바는 실행 시점에 실제 객체(Cat)의 makeSound()를 찾아 실행합니다(동적 바인딩). 이것이 다형성입니다.

**문제 3.** `Flyable`(fly)과 `Swimmable`(swim) 인터페이스를 만들고, 이 둘을 모두 구현하는 `Duck` 클래스를 작성하세요. 그리고 왜 이 상황에서는 추상 클래스가 아니라 인터페이스가 적합한지 한 문장으로 설명하세요.

> **힌트**: "여러 능력의 조합"이 열쇠입니다.
>
> **해설**:
> ```java
> interface Flyable { void fly(); }
> interface Swimmable { void swim(); }
> class Duck implements Flyable, Swimmable {
>     public void fly()  { System.out.println("날다"); }
>     public void swim() { System.out.println("헤엄치다"); }
> }
> ```
> 인터페이스가 적합한 이유: 오리는 "날 수 있고(CAN-DO)" "헤엄칠 수 있는(CAN-DO)" 서로 독립된 두 능력을 동시에 가지는데, 자바 클래스는 단 하나만 상속(extends)할 수 있으므로 여러 능력을 조합하려면 다중 구현이 가능한 인터페이스를 써야 합니다.

**문제 4.** 다음 설계 중 상속(IS-A)이 아니라 합성(HAS-A)으로 바꿔야 하는 것은? 이유도 설명하세요.
```
(A) class 대학생 extends 학생
(B) class 자동차 extends 엔진
(C) class 정사각형 extends 도형
```

> **힌트**: "A는 B이다"를 자연스럽게 말할 수 있는지 소리 내어 읽어보세요.
>
> **해설**: **(B)**. "대학생은 학생이다", "정사각형은 도형이다"는 자연스럽지만, "자동차는 엔진이다"는 말이 안 됩니다. 자동차는 엔진을 **가지는(HAS-A)** 관계이므로 `class 자동차 { private Engine engine; }` 형태의 합성으로 설계해야 합니다.

**문제 5.** `count`라는 필드로 지금까지 생성된 객체의 총 개수를 세고 싶습니다. 이 필드는 `static`이어야 할까요, 인스턴스 필드여야 할까요? 이유를 설명하세요.

> **힌트**: "총 개수"는 특정 객체 하나의 정보인가, 클래스 전체의 정보인가?
>
> **해설**: **static**이어야 합니다. "총 생성 개수"는 특정 객체 하나의 소유물이 아니라 클래스 전체가 공유해야 하는 정보입니다. 인스턴스 필드로 만들면 객체마다 따로 count를 가지므로 각자 자기만 세게 되어 총합을 알 수 없습니다. 생성자에서 `count++`을 하면 모든 객체가 같은 공용 카운터를 증가시킵니다.

## 요약

- **객체지향은 소프트웨어 위기를 극복하기 위한 패러다임 혁명**으로, 복잡한 시스템을 데이터와 기능을 함께 가진 독립적 부품(객체)으로 나누어 다스립니다.
- **클래스는 설계도, 객체는 실체**이며, `필드(상태) + 메서드(행동) = 객체`라는 공식으로 정의됩니다.
- **캡슐화**는 데이터를 숨기고 통로만 여는 것으로, 보호·유연성·복잡성 관리라는 세 가지 이유로 존재합니다.
- **상속**은 코드 중복을 없애고 다형성의 기반이 되지만 남용은 금물이며, "IS-A일 때만 상속, 그 외엔 합성"이 원칙입니다.
- **다형성**은 같은 명령에 다른 반응을 하는 능력으로, 동적 바인딩과 OCP를 통해 확장에 강한 유연한 설계를 가능하게 합니다. 추상 클래스(IS-A)와 인터페이스(CAN-DO)는 설계 철학이 다르며, SOLID 원칙(S/O/L/I/D)이 이 모든 것을 관통하는 좋은 설계의 나침반입니다.
