# 5장. 데이터를 효율적으로 다루다 - 컬렉션

> **🐳 실습 환경 — 이 장은 `java-sandbox` 컨테이너에서 실습한다**
> ```bash
> cd java && docker compose up -d              # 컨테이너 켜기 (이미 떠 있으면 생략)
> docker exec -it java-sandbox ./run.sh HashMapInternals   # 예제 실행
> ```
> 처음이라면 [java/README.md](../../../README.md)의 "빠른 시작"을 먼저 보라.

프로그램은 결국 **데이터를 담고, 찾고, 바꾸고, 지우는 일**의 연속이다. 회원 목록을 관리하고, 장바구니에 상품을 넣고, 사용자 ID로 정보를 조회하는 모든 작업이 그렇다. 그런데 "데이터를 담는다"는 단순해 보이는 이 일이, 어떤 그릇에 담느냐에 따라 프로그램의 성능이 수백 배씩 차이 난다.

왜 그럴까? 100만 명의 회원 중에서 특정 ID를 찾는 데 어떤 그릇은 100만 번을 뒤지고, 어떤 그릇은 단 한 번에 찾아낸다. 어떤 그릇은 데이터를 중간에 끼워 넣을 때 나머지 전부를 밀어야 하고, 어떤 그릇은 연결 고리만 바꾸면 끝난다. **자료구조를 안다는 것은 곧 "이 상황에서 어떤 그릇이 왜 빠른지"를 안다는 뜻**이다.

이 장에서는 자바의 컬렉션 프레임워크를 다룬다. 하지만 단순히 "ArrayList는 이렇게 쓴다"를 외우는 것이 목표가 아니다. **HashMap이 어떻게 100만 개 중에서 한 번에 값을 찾아내는지, ArrayList의 `add`가 왜 평균 O(1)인지, TreeMap이 왜 항상 정렬 상태를 유지하면서도 빠른지** — 그 내부 원리를 직접 유도할 수 있게 되는 것이 목표다. 원리를 이해하면 새로운 자료구조를 만나도 두렵지 않다. 그리고 이 내용은 기술 면접에서 가장 자주 나오는 주제이기도 하다.

## 5.1 배열의 한계와 컬렉션의 등장

### 정의

**배열(Array)** 은 같은 타입의 데이터를 연속된 메모리 공간에 고정된 크기로 저장하는 가장 기본적인 자료구조다. **컬렉션 프레임워크(Collection Framework)** 는 자바가 제공하는, 크기가 동적으로 늘어나고 다양한 기능을 갖춘 자료구조들의 표준 모음이다.

### 원리 (왜 배열만으로는 부족한가?)

배열을 아파트에 비유해 보자. 아파트는 처음 지을 때 세대 수가 정해진다. 5세대짜리 아파트에 6번째 가구가 이사 올 수는 없다.

```java
int[] scores = new int[5];   // 5칸짜리 아파트
scores[0] = 90;
scores[1] = 85;
// ...
scores[5] = 100;             // 런타임 에러! ArrayIndexOutOfBoundsException
                             // 6번째 칸은 애초에 없다
```

배열의 세 가지 한계는 다음과 같다.

1. **크기 고정**: `new int[5]`로 만들면 영원히 5칸이다. 더 넣고 싶으면 새 배열을 만들어 통째로 복사해야 한다.
2. **삽입/삭제가 번거롭다**: 중간에 값을 끼워 넣으려면 뒤의 값들을 한 칸씩 밀어야 하고, 지우면 당겨야 한다.
3. **기능이 빈약하다**: 정렬, 검색, 중복 제거 같은 기능이 없어 매번 직접 구현해야 한다.

그런데 현실의 데이터는 **개수를 미리 알 수 없다**. 오늘 가입할 회원이 몇 명일지, 이 게시글에 댓글이 몇 개 달릴지, 장바구니에 상품을 몇 개 담을지 아무도 모른다. 크기가 고정된 배열로는 이런 동적인 세상을 담기 어렵다. **그래서 "필요한 만큼 알아서 늘어나고, 삽입·삭제·검색·정렬 기능까지 갖춘" 자료구조가 필요해졌고, 그것이 컬렉션 프레임워크다.**

### ASCII 다이어그램: 컬렉션 계층 구조

```
                        Iterable
                           │
                       Collection
              ┌────────────┼────────────┐
              │            │            │
            List          Set         Queue
        (순서 O, 중복 O) (순서 X*, 중복 X) (대기열)
              │            │
   ┌──────────┴──┐    ┌────┴────┐
ArrayList  LinkedList HashSet  TreeSet
                              (정렬됨)

         Map  ◀── ※ Collection을 상속하지 않는다!
          │
   ┌──────┴──────┐
 HashMap      TreeMap
             (키 정렬됨)

  * HashSet은 순서 보장 X, LinkedHashSet은 삽입 순서 유지,
    TreeSet은 정렬 순서 유지
```

여기서 **가장 자주 나오는 함정**: `Map`은 `Collection`을 상속하지 않는다. 컬렉션은 "값들의 모음"이지만 Map은 "키-값 쌍의 모음"이라 구조가 다르기 때문이다. 그래서 `Map`은 `add()`가 아니라 `put()`을 쓴다. 다만 넓은 의미에서 "컬렉션 프레임워크"라고 부를 때는 Map도 포함한다.

### 코드 예제

```java
// 배열: 크기 고정
String[] arr = new String[3];
arr[0] = "사과";
// arr[3] = "포도";  // 불가능

// 컬렉션: 알아서 늘어난다
import java.util.*;

List<String> fruits = new ArrayList<>();
fruits.add("사과");
fruits.add("바나나");
fruits.add("포도");
fruits.add("딸기");   // 몇 개든 계속 추가 가능
System.out.println(fruits.size());  // 4
System.out.println(fruits);         // [사과, 바나나, 포도, 딸기]
```

## 5.2 List, Set, Map: 어떤 상황에서 무엇을?

### 정의

- **List**: 순서가 있고(인덱스 존재), 중복을 허용하는 모음
- **Set**: 순서 개념이 없고(구현에 따라 다름), 중복을 허용하지 않는 모음
- **Map**: 키(key)와 값(value)의 쌍으로 데이터를 저장하는 모음

### 원리 (왜 세 가지로 나뉘는가?)

세상의 데이터에는 저마다 다른 성격이 있다. 그 성격에 맞는 그릇을 고르라고 세 종류로 나눠 놓은 것이다.

**List — 순서가 중요하고 중복이 있을 수 있을 때**
출석부를 생각하자. 1번, 2번, 3번 순서가 있고, "김철수"라는 동명이인이 두 명 있을 수도 있다. 채팅 메시지(도착 순서 중요), 장바구니, 순위표가 여기에 해당한다.

```java
List<String> chat = new ArrayList<>();
chat.add("안녕하세요");
chat.add("반갑습니다");
chat.add("안녕하세요");   // 중복 OK
System.out.println(chat); // [안녕하세요, 반갑습니다, 안녕하세요]
```

**Set — 중복이 있으면 안 될 때**
회원 ID 집합을 생각하자. 같은 ID가 두 번 들어가면 안 된다. 순서는 중요하지 않다. 게시글의 태그 목록, 오늘 방문한 IP 목록, 리스트에서 고유값만 뽑아낼 때 유용하다.

```java
Set<String> tags = new HashSet<>();
tags.add("자바");
tags.add("스프링");
tags.add("자바");        // 이미 있으므로 무시됨
System.out.println(tags.size()); // 2
```

**Map — 키로 값을 찾고 싶을 때**
영어사전을 생각하자. "apple"이라는 단어(키)를 찾으면 "사과"라는 뜻(값)이 나온다. 사용자 ID로 사용자 정보를 찾거나, 상품 코드로 상품을 찾는 상황이다.

```java
Map<String, String> dict = new HashMap<>();
dict.put("apple", "사과");
dict.put("banana", "바나나");
System.out.println(dict.get("apple")); // 사과
```

### ASCII 다이어그램: 선택 플로우차트

```
                 데이터를 어떻게 저장할까?
                          │
              ┌───────────┴───────────┐
        "키로 값을 찾나?"              아니오
              │ 예                      │
              ▼                    "중복을 허용하나?"
           ★ Map ★                     │
                          ┌────────────┴────────────┐
                        아니오                      예
                          │                          │
                   "순서가 중요한가?"              ★ List ★
                          │                     (사실 대부분 List로
                    ┌─────┴─────┐               충분하다)
                   예           아니오
                    │             │
                ★ List ★      ★ Set ★
                             (또는 List)
```

실무 팁: 초보자는 "순서가 안 중요하니 Set을 써야 하나?" 하고 고민하는데, **애매하면 그냥 List를 쓰면 대부분 문제없다.** Set은 "중복 제거가 명확히 필요할 때"만 선택하면 된다.

## 5.3 ArrayList vs LinkedList: 진짜 차이

### 정의

둘 다 `List` 인터페이스를 구현하지만 내부 저장 방식이 완전히 다르다. **ArrayList는 내부적으로 배열**을 쓰고, **LinkedList는 노드들이 포인터로 연결된 연결 리스트**를 쓴다.

### 원리 (내부 구조의 차이가 성능을 가른다)

**ArrayList = 아파트 (연속된 메모리)**
데이터가 메모리에 연속으로 붙어 있다. 그래서 "3번 데이터 줘"라고 하면 시작 주소 + 3칸만큼 이동해 **한 번에 도달**한다. 조회가 O(1)이다.

```
ArrayList (인덱스로 즉시 접근)
┌─────┬──────┬─────┬─────┬─────┐
│  0  │  1   │  2  │  3  │  4  │  ← 인덱스
├─────┼──────┼─────┼─────┼─────┤
│사과 │바나나│포도 │딸기 │수박 │
└─────┴──────┴─────┴─────┴─────┘
 조회: arr[2] → 포도, 한 번에! O(1)

 중간 삽입 (인덱스 1에 "귤" 삽입):
┌─────┬──────┬──────┬─────┬─────┬─────┐
│사과 │ 귤   │바나나│포도 │딸기 │수박 │
└─────┴──────┴──────┴─────┴─────┴─────┘
        ↑ 뒤의 4개를 전부 한 칸씩 밀어야 한다 → O(n)
```

**LinkedList = 기차 (흩어진 메모리, 포인터로 연결)**
데이터(노드)가 메모리 여기저기 흩어져 있고, 각 노드가 "다음은 여기야"라는 주소(포인터)를 들고 있다. 특정 위치를 찾으려면 첫 칸부터 하나씩 세며 가야 한다(O(n)). 대신 삽입·삭제는 앞뒤 연결 고리만 바꾸면 되므로, **그 위치를 이미 알고 있다면(Iterator로 순회 중이라면) O(1)** 이다.

```
LinkedList (포인터로 연결)
┌──────┐   ┌──────┐   ┌──────┐
│ 사과 │──▶│바나나│──▶│ 포도 │──▶ null
└──────┘   └──────┘   └──────┘
 조회: "3번째?" → 사과→바나나→포도, 세며 가야 함 → O(n)

 중간 삽입 ("바나나"와 "포도" 사이에 "귤"):
┌──────┐   ┌──────┐        ┌──────┐
│ 사과 │──▶│바나나│──┐  ┌─▶│ 포도 │──▶ null
└──────┘   └──────┘  │  │  └──────┘
                     ▼  │
                  ┌──────┐
                  │  귤  │
                  └──────┘
   → 연결 고리 2개만 바꾸면 끝, O(1) (위치를 알고 있을 때)
```

### 왜 실무의 99%는 ArrayList인가?

이론만 보면 "삽입/삭제가 많으면 LinkedList가 유리하다"고 생각하기 쉽다. 하지만 **실무에서는 거의 항상 ArrayList가 빠르다.** 세 가지 이유가 있다.

1. **CPU 캐시 효율**: 현대 CPU는 데이터를 읽을 때 주변 메모리를 함께 캐시에 올린다. ArrayList는 연속 메모리라 캐시 적중률이 높다. LinkedList는 노드가 흩어져 있어 매번 캐시 미스가 나고, 이 차이가 이론적 이점을 압도한다.
2. **대부분의 작업은 조회**: 실제 코드는 삽입/삭제보다 "몇 번째 값 읽기"가 훨씬 많다. 조회는 ArrayList가 압도적이다.
3. **삽입은 대부분 "끝에 추가"**: 중간 삽입이 아니라 `add()`로 맨 뒤에 붙이는 경우가 대부분인데, 이건 ArrayList도 평균 O(1)이다(5.6절 상환분석 참고).

**결론: 특별한 이유가 없으면 ArrayList를 쓰라.** LinkedList는 "양 끝에서 빈번하게 넣고 빼는 큐/덱" 같은 특수한 경우에나 고려한다.

### 코드 예제

```java
List<Integer> arrayList = new ArrayList<>();
List<Integer> linkedList = new LinkedList<>();

// 사용법은 동일하다 (List 인터페이스로 추상화됨)
arrayList.add(10);
linkedList.add(10);

// 하지만 내부 동작과 성능은 다르다
int x = arrayList.get(500);   // O(1)
int y = linkedList.get(500);  // O(n) — 앞에서부터 500번 이동
```

## 5.4 HashMap의 동작 원리

### 정의

**HashMap**은 키-값 쌍을 저장하되, **해시 함수**를 이용해 키를 빠르게 찾아내는 자료구조다. 실무에서 가장 많이 쓰이고, 기술 면접에서 가장 자주 나온다.

### 원리 1: 해시 함수

**해시 함수**는 임의의 데이터를 고정된 크기의 숫자로 바꾸는 함수다. 도서관의 분류번호에 비유할 수 있다. 세상의 모든 책 제목을 "813.7" 같은 숫자로 바꿔주는 규칙이다.

```
해시 함수 hash("apple")  → 12
         hash("banana") →  7
         hash("cherry") →  3
         hash("date")   → 12   ← "apple"과 같은 값! (충돌)
```

여기서 핵심 질문: **왜 해시로 바꾸는가?** 숫자로 바꾸면 그 숫자를 배열 인덱스로 써서 "몇 번 칸에 넣을지"를 즉시 계산할 수 있기 때문이다. 이름으로 찾으면 전부 뒤져야 하지만, "813번 서가로 가라"고 하면 한 번에 간다.

### 원리 2: 버킷 구조

HashMap 내부에는 **버킷(bucket)이라 부르는 배열**이 있다. 키의 해시값으로 어느 버킷에 넣을지 결정한다.

```
HashMap 내부 (버킷 배열)
 인덱스
 ┌───┐
 │ 0 │
 ├───┤
 │ 1 │
 ├───┤
 │ 2 │
 ├───┤
 │ 3 │──▶ [cherry → 값]
 ├───┤
 │...│
 ├───┤
 │ 7 │──▶ [banana → 값]
 ├───┤
 │...│
 ├───┤
 │12 │──▶ [apple → 값] ──▶ [date → 값]   ← 충돌! 체이닝
 └───┘
```

### 원리 3: 해시 충돌과 체이닝

`hash("apple")`과 `hash("date")`가 둘 다 12가 나오는 것처럼, **서로 다른 키가 같은 해시값을 갖는 것을 해시 충돌**이라 한다. 무한한 키를 유한한 버킷에 넣으니 충돌은 필연적이다.

충돌이 나면 어떻게 할까? **체이닝(Chaining)** — 같은 버킷에 여러 항목을 연결 리스트(또는 나중에 트리)로 매달아 둔다. 조회할 때는 그 버킷 안의 항목들을 순서대로 확인한다.

### equals()와 hashCode()를 왜 함께 오버라이드해야 하는가?

이것은 면접 단골 질문이다. HashMap이 키를 찾는 과정은 **2단계**다.

```
키 조회 2단계:
 1단계: hashCode()로 어느 버킷인지 결정  → "12번 버킷으로 가"
 2단계: 그 버킷 안에서 equals()로 정확한 키를 찾음
        → "apple이야? date야? equals로 비교!"
```

도서관 비유: `hashCode()`는 **서가 번호**, `equals()`는 **정확한 책 제목**이다. 서가 번호로 대략의 위치를 찾고, 제목으로 정확한 책을 집는다.

이제 왜 둘을 함께 오버라이드해야 하는지 보자.

- **hashCode()만 오버라이드하면**: 올바른 버킷까지는 가지만, equals가 기본(주소 비교)이라 "논리적으로 같은 키"인데도 다르다고 판단해 못 찾는다.
- **equals()만 오버라이드하면**: hashCode가 기본이라 두 논리적으로 같은 객체가 **다른 버킷**으로 가버린다. 아예 엉뚱한 곳을 뒤지니 절대 못 찾는다.

**규칙: equals()를 오버라이드하면 반드시 hashCode()도 오버라이드하라. 논리적으로 같은 두 객체는 반드시 같은 hashCode를 반환해야 한다.**

```java
class Member {
    String id;
    Member(String id) { this.id = id; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Member)) return false;
        return id.equals(((Member) o).id);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id);   // equals에서 쓴 필드로 계산
    }
}

Map<Member, String> map = new HashMap<>();
map.put(new Member("a01"), "홍길동");
System.out.println(map.get(new Member("a01"))); // 홍길동 (둘 다 오버라이드했으므로 찾음)
```

## 5.5 제네릭: 타입 안전성의 보장

### 정의

**제네릭(Generics)** 은 클래스나 메서드가 다룰 데이터의 타입을 `<T>` 같은 형태로 외부에서 지정할 수 있게 하는 기능이다. `List<String>`은 "문자열만 담는 List"라는 뜻이다.

### 원리 (왜 제네릭이 필요했는가?)

**제네릭이 없던 시절 (Java 5 이전)** 을 보자. 컬렉션은 아무 타입이나 받는 `Object` 상자였다.

```java
// Java 5 이전 스타일
List list = new ArrayList();
list.add("문자열");
list.add(123);          // 정수도 들어감
list.add(new Dog());    // 개도 들어감

// 꺼낼 때 지옥이 열린다
String s = (String) list.get(1);  // 실제로는 123(Integer)!
                                   // → 런타임에 ClassCastException 폭탄
```

이건 "아무거나 넣을 수 있는 상자"다. 넣을 때는 편하지만, 꺼낼 때 무엇이 들었는지 몰라 캐스팅해야 하고, 잘못 캐스팅하면 프로그램이 실행 중에 터진다.

**제네릭이 해결한 방식**: "이 상자는 사과 전용"이라고 상자에 라벨을 붙이는 것이다.

```java
List<String> list = new ArrayList<>();
list.add("문자열");    // OK
list.add(123);         // 컴파일 에러! 문자열이 아니잖아
String s = list.get(0); // 캐스팅 불필요, 항상 String
```

### 왜 이게 강력한가? — 에러를 언제 잡느냐의 문제

핵심은 **"실행 시 터질 에러를 컴파일 시점에 잡아준다"** 는 것이다.

```
에러가 발생하는 시점
┌────────────────────┬─────────────────────────────┐
│ 컴파일 시 에러      │ 실행(런타임) 시 에러         │
├────────────────────┼─────────────────────────────┤
│ 개발자만 본다       │ 사용자가 본다 (치명적)       │
│ 빌드가 안 될 뿐     │ 서비스 도중 앱이 죽음        │
│ 그 자리에서 수정    │ 언제 어디서 터질지 모름      │
└────────────────────┴─────────────────────────────┘

  에러는 빨리 발견할수록 좋다.
  제네릭은 에러 발견 시점을 "실행" → "컴파일"로 당긴다.
```

컴파일 에러는 개발자가 빌드하다 발견해 바로 고치면 그만이다. 하지만 런타임 에러는 이미 배포된 후, 사용자가 쓰는 도중에 터진다. **제네릭은 에러를 앞당겨 잡는 안전장치**다.

## 5.6 Big-O를 스스로 유도하는 법

### 정의

**Big-O 표기법**은 입력 크기 n이 커질 때 알고리즘의 작업량이 어떤 비율로 증가하는지를 나타내는 표기다. O(1), O(n), O(log n), O(n²) 등이 있다.

### 원리 (Big-O가 진짜 답하려는 질문)

많은 사람이 "O(n)이 몇 초냐?"고 묻지만, Big-O는 **시간을 재는 게 아니다.** Big-O가 답하려는 질문은 이것이다.

> **"n이 2배, 10배, 100배 커지면 작업량은 몇 배가 되는가?"** — 즉 성장률(growth rate)이다.

결과표를 외우는 게 목적이 아니다. **처음 보는 코드의 복잡도를 스스로 계산할 수 있어야 한다.** 세 가지 규칙만 알면 된다.

### 규칙 1: 지배적인 연산의 실행 횟수를 센다

```java
// 예제 A
int sum = 0;
for (int i = 0; i < n; i++) {
    sum += arr[i];    // n번 실행
}
// → O(n)

// 예제 B (중첩 반복)
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        check();      // n × n = n²번 실행
    }
}
// → O(n²)  — n이 2배면 작업량은 4배로 폭증
```

### 규칙 2: 상수 배수와 낮은 차수 항은 버린다

```java
for (int i = 0; i < n; i++) { ... }   // n번
for (int j = 0; j < n; j++) { ... }   // n번
// 합쳐서 2n → 상수 2를 버리고 O(n)

// 3n² + 100n + 5 → 가장 큰 항만 남기고 O(n²)
```

이유: n이 아주 커지면 n² 앞에서 n이나 상수는 무의미해진다. n = 100만이면 n²는 1조인데, 100n은 1억일 뿐이다. **가장 빠르게 자라는 항이 전체를 지배한다.**

### 규칙 3: 매번 절반으로 줄면 O(log n)

이진 탐색(binary search)을 보자. 정렬된 데이터에서 중간값과 비교해 절반씩 버린다.

```
16개에서 찾기:
16 → 8 → 4 → 2 → 1   (4단계)   log₂16 = 4
1024개에서 찾기:
1024 → 512 → ... → 1  (10단계)  log₂1024 = 10
```

**n이 1024배 커져도 단 10번**이면 된다. 데이터가 폭발적으로 늘어도 작업량은 거의 안 는다 — 이것이 O(log n)의 위력이다.

### 상환분석: ArrayList의 add가 왜 O(1)인가?

여기서 흥미로운 질문. ArrayList는 내부가 고정 크기 배열인데, 꽉 차면 어떻게 계속 추가할까? **더 큰 배열을 새로 만들어 전부 복사한다(O(n)).** 그런데도 왜 `add`를 "평균 O(1)"이라 부를까?

**상환분석(Amortized Analysis)** 때문이다. 복사는 가끔만 일어나고, 그 비용을 여러 번의 add에 나눠 부담시키면 평균적으로 상수가 된다.

```
용량이 2배씩 늘어난다고 하자: 4 → 8 → 16 → 32
16개를 add하는 동안 복사가 일어나는 시점의 복사량:
  4칸 채우고 8로 확장할 때  복사 4
  8칸 채우고 16으로 확장할 때 복사 8
 16칸 채우고 32로 확장할 때  복사 16
 총 복사량 = 4 + 8 + 16 = 28  ≈  상수 × 16

 16번의 add로 나누면 → add 1번당 평균 상수 → O(1)
```

비유: 매일 조금씩 저금하다가 1년에 한 번 큰 지출을 하면, 하루 평균으로 나누면 부담이 작다.

**왜 하필 2배(곱셈)로 늘릴까?** 만약 "1칸씩만 늘린다면" 매번 전체 복사가 일어나 총 비용이 1+2+3+...+n = O(n²)가 된다. **곱셈으로 늘려야만 평균 O(1)이 보장된다.** 이것이 자바 ArrayList가 약 1.5배씩 용량을 늘리는 이유다.

### Big-O 치트 시트

| 자료구조/연산 | 조회(get) | 검색(contains) | 삽입 | 삭제 |
|---|---|---|---|---|
| 배열 | O(1) | O(n) | O(n) | O(n) |
| ArrayList | O(1) | O(n) | O(1)* (끝), O(n) (중간) | O(n) |
| LinkedList | O(n) | O(n) | O(1)** | O(1)** |
| HashMap / HashSet | — | O(1) 평균 | O(1) 평균 | O(1) 평균 |
| TreeMap / TreeSet | — | O(log n) | O(log n) | O(log n) |

\* 상환 O(1) &nbsp;&nbsp; \*\* 위치를 이미 알고 있을 때(Iterator 순회 중)

## 5.7 HashMap 심화: resize, treeify, O(1)의 조건

### 원리 1: 버킷 인덱스를 구하는 방법 — `hash & (n-1)`

버킷이 n개일 때, 해시값을 어느 버킷에 넣을지는 원래 `해시값 % n`(나머지)으로 구할 수 있다. 그런데 HashMap은 나눗셈(`%`) 대신 **비트 AND(`&`)** 를 쓴다.

```
n이 2의 거듭제곱이면, n-1은 하위 비트가 전부 1인 마스크가 된다.

 n = 16       →  16 = 0001 0000
 n - 1 = 15   →  15 = 0000 1111   ← 하위 4비트가 전부 1

 hash & (n-1) 은 hash % n 과 결과가 같다:

 예) hash = 0110 1101 (109)
     &        0000 1111 (15)
     ───────────────────
              0000 1101 (13)  → 13번 버킷

 109 % 16 = 13  ← 동일!
```

**왜 AND를 쓰는가?** 나눗셈은 CPU에서 느린 연산이고, 비트 AND는 매우 빠르다. n을 항상 2의 거듭제곱으로 유지하면 느린 나눗셈을 빠른 AND로 대체할 수 있다. **이것이 HashMap 용량이 항상 2의 거듭제곱(16, 32, 64...)인 이유다.**

### 원리 2: load factor와 resize

버킷이 점점 차면 충돌이 잦아진다. 버킷 대비 원소가 너무 많으면 체인이 길어져 조회가 느려진다. 그래서 일정 비율을 넘으면 **버킷 배열을 2배로 키운다 — 이것이 resize.**

- **load factor(적재율)**: 기본값 0.75
- **threshold(임계치)**: 용량 × load factor

```
용량 16, load factor 0.75
 → threshold = 16 × 0.75 = 12
 → 원소가 12개를 넘으면 용량을 32로 확장(resize)

resize 시 모든 원소의 버킷 위치를 다시 계산(rehash)한다:
 [기존 16개 버킷] ──rehash──▶ [새 32개 버킷]
     각 원소를 새 버킷으로 재배치
```

**왜 0.75인가?** 이것은 트레이드오프의 균형점이다.
- load factor가 **낮으면**: 조금만 차도 resize → 메모리 낭비, 잦은 재배치
- load factor가 **높으면**: 버킷이 빽빽 → 충돌 잦음 → 조회 느려짐
- 0.75는 시간과 공간 사이의 실험적으로 검증된 균형점이다.

### 원리 3: treeify (Java 8+)

체이닝은 연결 리스트라, 한 버킷에 원소가 많이 쌓이면 그 버킷 조회가 O(n)으로 느려진다. 악의적인 입력(모두 같은 해시)이 들어오면 공격에 취약하기도 하다. Java 8부터는 이를 방어한다.

```
한 버킷의 체인 길이가 8을 넘으면:
 연결 리스트 ────treeify────▶ 레드-블랙 트리
   O(n)                        O(log n)

 (단, 버킷 배열 크기가 64 이상일 때만 트리화;
  그보다 작으면 resize를 먼저 함)

 다시 6개 이하로 줄면 → 연결 리스트로 되돌림(untreeify)
```

체인이 길어져도 최악이 O(log n)으로 방어된다.

### 정리: "HashMap은 O(1)"의 진짜 의미

```
 평균 O(1):  해시가 고르게 분산되어 버킷당 원소가 적을 때
 최악 O(log n): 충돌이 심해 treeify가 발동했을 때 (Java 8+)
```

**"HashMap은 O(1)"은 "해시 함수가 값을 고르게 분산시킨다"는 전제 위에서만 성립한다.** 그래서 좋은 `hashCode()` 구현이 중요하다. 모든 키가 같은 값을 반환하는 나쁜 hashCode를 쓰면, 모든 원소가 한 버킷에 몰려 O(n)(혹은 트리화 후 O(log n))으로 전락한다.

## 5.8 균형 이진 탐색 트리: TreeMap은 왜 O(log n)?

### 정의

HashMap은 순서가 없다. 반면 **TreeMap과 TreeSet은 키를 항상 정렬된 상태로 유지**한다. "이름 순으로 순회", "10~20 사이의 키만 조회", "가장 가까운 키 찾기" 같은 작업에 쓴다. 내부는 **균형 이진 탐색 트리(정확히는 레드-블랙 트리)** 다.

### 원리 1: 이진 탐색 트리(BST)

**BST 규칙: 왼쪽 자식 < 부모 < 오른쪽 자식.** 이 규칙 덕분에 값을 찾을 때 매번 절반을 버릴 수 있다.

```
          (8)
         /   \
       (3)    (10)
      /  \       \
    (1)  (6)     (14)

 6을 찾는 과정:
  8과 비교 → 6 < 8, 왼쪽으로     (1단계)
  3과 비교 → 6 > 3, 오른쪽으로   (2단계)
  6과 비교 → 찾음!               (3단계)

 매 비교마다 후보의 절반을 버린다 → O(트리 높이)
```

### 원리 2 (함정): 균형이 무너지면 O(n)

BST의 성능은 **트리 높이**에 달렸다. 그런데 정렬된 데이터를 순서대로 넣으면 재앙이 벌어진다.

```
 1, 2, 3, 4, 5 를 순서대로 삽입하면:

 (1)
    \
    (2)
       \
       (3)
          \
          (4)
             \
             (5)   ← 한쪽으로 쭉! 사실상 연결 리스트

 높이 = n → 탐색 O(n)  ← 트리의 이점이 완전히 사라짐!
```

**O(log n)은 트리가 낮게(균형 있게) 유지될 때만 성립**한다. 그냥 BST로는 이 균형을 보장할 수 없다.

### 원리 3 (해결): 레드-블랙 트리

**레드-블랙 트리**는 각 노드에 빨강/검정 색을 부여하고 몇 가지 규칙을 강제해, **스스로 균형을 유지**하는 트리다.

핵심 규칙 중 하나: **루트에서 각 잎(리프)까지 가는 모든 경로의 검정 노드 수가 같아야 한다.** 이 규칙이 트리가 한쪽으로 치우치는 것을 막는다.

```
 삽입/삭제로 균형이 깨지면:
   회전(rotation) + 색 변경으로 스스로 재정렬

 회전(rotation) 예시 (오른쪽으로 치우친 것을 바로잡음):
        (3)                 (2)
       /            →      /   \
     (2)                 (1)   (3)
     /
   (1)

 회전은 국소적 수술이라 O(1)에 끝난다.
 그 결과 높이가 항상 약 log n으로 유지됨
 → 삽입·삭제·검색이 최악에도 O(log n) 보장
```

### 왜 TreeMap을 쓰는가? — HashMap과 비교

| | HashMap | TreeMap |
|---|---|---|
| 평균 조회 | **O(1)** | O(log n) |
| 순서 | 없음 | **키 정렬됨** |
| 범위 검색 | 불가 | **가능** (`subMap`, `headMap` 등) |
| 가장 가까운 키 | 불가 | **가능** (`floorKey`, `ceilingKey`) |

**정렬·범위 검색이 필요하면 TreeMap, 순수 속도만 필요하면 HashMap.** 참고로 5.7절의 HashMap treeify는 바로 이 레드-블랙 트리를 재활용한 것이다.

```java
TreeMap<Integer, String> scores = new TreeMap<>();
scores.put(90, "A");
scores.put(70, "C");
scores.put(80, "B");

System.out.println(scores);              // {70=C, 80=B, 90=A} — 정렬됨!
System.out.println(scores.firstKey());   // 70
System.out.println(scores.ceilingKey(75)); // 80 (75 이상 가장 작은 키)
System.out.println(scores.subMap(70, 85)); // {70=C, 80=B} (범위 검색)
```

## 5.9 제네릭 타입 소거(Type Erasure)

### 정의

**타입 소거(Type Erasure)** 는 제네릭 타입 정보가 컴파일 시점에만 사용되고, **컴파일이 끝나면 지워지는** 자바 제네릭의 구현 방식이다.

### 원리 (컴파일러가 몰래 하는 일)

제네릭은 컴파일 시점에 타입 오류를 잡아준다. 하지만 컴파일이 끝나면 그 타입 정보는 바이트코드에서 사라진다.

```java
List<String> a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
System.out.println(a.getClass() == b.getClass()); // true !!
// 런타임에는 둘 다 그냥 List일 뿐, <String>/<Integer>는 사라졌다
```

컴파일러가 실제로 하는 일을 풀어 보면 이렇다.

```
컴파일 전 (개발자가 쓴 코드):
   List<String> list = new ArrayList<>();
   list.add("hello");
   String s = list.get(0);

컴파일러가 하는 일:
   1. 타입 확인: list.add(123) 같은 오류를 컴파일 시점에 잡는다
   2. 캐스팅 자동 삽입: get(0)에 (String) 캐스팅을 몰래 넣는다
   3. 타입 정보 삭제: <String>을 지운다

컴파일 후 (실제 바이트코드, 개념적으로):
   List list = new ArrayList();       // 제네릭 지워짐
   list.add("hello");
   String s = (String) list.get(0);   // 캐스팅은 컴파일러가 넣어줌
```

### 왜 타입을 지우는가?

제네릭은 Java 5에서 **뒤늦게** 추가된 기능이다. 그 전에 작성된 수많은 코드(`List list = ...`)와 호환되어야 했다. 만약 제네릭 타입을 런타임까지 유지했다면 기존 바이트코드와 맞지 않았을 것이다. **타입을 지워서 컴파일 후 바이트코드가 제네릭 이전과 동일해지도록** 한 것이다 — 하위 호환성을 위한 선택이다.

### 소거가 낳는 3가지 제약

```java
// 제약 1: new T[] 불가 — 런타임에 T가 무엇인지 모른다
class Box<T> {
    // T[] arr = new T[10];  // 컴파일 에러
}

// 제약 2: instanceof List<String> 불가 — <String>이 지워졌으니 확인 불가
if (obj instanceof List<String>) { }  // 컴파일 에러
if (obj instanceof List<?>) { }        // 이건 OK (와일드카드)
if (obj instanceof List) { }           // 이것도 OK

// 제약 3: raw type을 섞어 쓰면 런타임 ClassCastException 위험
List raw = new ArrayList();   // raw type (제네릭 없이)
raw.add("문자열");
List<Integer> nums = raw;     // 경고만 뜨고 컴파일은 됨
int n = nums.get(0);          // 런타임에 ClassCastException!
                              // (String을 int로 캐스팅 시도)
```

**정리: 제네릭은 컴파일러에게만 보이는 안전장치다. 런타임에는 사라진다.** 그래서 raw type을 섞어 쓰면 제네릭의 보호막이 뚫리고, 제네릭 이전의 ClassCastException 지옥이 되살아난다. raw type은 절대 쓰지 말자.

> 실행 데모: ./run.sh TypeErasureDemo

### 심화: 제네릭 와일드카드와 PECS

제네릭을 제대로 쓰려면 **와일드카드(`?`)** 를 알아야 한다. 다음 코드는 직관에 반한다.

```java
List<Integer> ints = new ArrayList<>();
List<Number> nums = ints;   // 컴파일 에러!
// Integer는 Number의 하위 타입인데 왜 안 될까?
```

만약 저게 허용된다면 `nums.add(3.14)`(Double)를 할 수 있게 되고, 그러면 `List<Integer>`에 Double이 들어가 버린다. 이걸 막으려고 **제네릭은 기본적으로 상속 관계를 물려받지 않는다(불공변, invariant).** 이 경직성을 풀어주는 것이 와일드카드다.

**`? extends T` (상한 경계) — 읽기용(Producer)**

```java
// "T이거나 T의 하위 타입" — 데이터를 꺼내 읽을 때
double sum(List<? extends Number> list) {
    double total = 0;
    for (Number n : list) total += n.doubleValue();  // 읽기 OK
    return total;
    // list.add(...) 는 불가! 정확한 타입을 모르니 넣지 못한다
}
sum(new ArrayList<Integer>());  // OK
sum(new ArrayList<Double>());   // OK
```

**`? super T` (하한 경계) — 쓰기용(Consumer)**

```java
// "T이거나 T의 상위 타입" — 데이터를 집어넣을 때
void addNumbers(List<? super Integer> list) {
    list.add(1);   // 넣기 OK (Integer는 확실히 넣을 수 있다)
    list.add(2);
    // Number n = list.get(0);  // 읽으면 Object로만 나옴
}
addNumbers(new ArrayList<Number>());   // OK
addNumbers(new ArrayList<Object>());   // OK
```

**PECS 원칙**: **Producer Extends, Consumer Super.**
- 컬렉션에서 데이터를 **꺼내 읽기(생산)** 만 하면 `extends`
- 컬렉션에 데이터를 **집어넣기(소비)** 만 하면 `super`

```
 List<? extends T>  ── 여기서 값을 "꺼낸다"(Producer) → extends
 List<? super   T>  ── 여기에 값을 "넣는다"(Consumer) → super
```

`Collections.copy(dest, src)`가 대표적이다. `src`는 읽기만 하니 `extends`, `dest`는 쓰기만 하니 `super`를 쓴다.

## ⚠️ 흔한 오해와 함정

**1. "삽입/삭제가 많으니 LinkedList를 쓰자"**
대부분 틀린 판단이다. CPU 캐시 효율과 실제 사용 패턴 때문에 거의 항상 ArrayList가 빠르다. 벤치마크 없이 LinkedList를 고르지 말자.

**2. `equals()`만 오버라이드하고 `hashCode()`를 빼먹는 것**
HashMap/HashSet에서 객체를 키로 쓸 때 값을 못 찾는 버그가 난다. IDE의 자동 생성 기능을 쓰면 둘을 항상 함께 만들어 준다.

**3. 순회 중 컬렉션 수정 → ConcurrentModificationException**
```java
List<String> list = new ArrayList<>(List.of("a", "b", "c"));
for (String s : list) {
    if (s.equals("b")) list.remove(s);  // 예외 발생!
}
```
for-each(내부적으로 Iterator)로 순회하는 동안 컬렉션 크기가 바뀌면, 자바는 무결성이 깨졌다고 판단해 `ConcurrentModificationException`을 던진다. 올바른 삭제 방법은 세 가지다.
```java
// 방법 1: Iterator의 remove() 사용
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) it.remove();  // 안전
}
// 방법 2: removeIf (가장 간결)
list.removeIf(s -> s.equals("b"));
// 방법 3: 순회용 복사본을 따로 만들기
for (String s : new ArrayList<>(list)) { ... }
```

**4. "HashMap은 무조건 O(1)"**
평균이 O(1)일 뿐, 나쁜 hashCode로 충돌이 몰리면 O(log n)(treeify) 또는 그 이상으로 느려진다. O(1)은 "고르게 분산된다"는 전제가 있다.

**5. raw type 사용**
`List list = ...`처럼 제네릭을 생략하면 컴파일 경고를 무시하는 순간 런타임 ClassCastException 위험이 생긴다. 항상 타입 파라미터를 명시하자.

**6. `HashMap`이 순서를 보장한다고 착각**
HashMap은 순서를 보장하지 않는다. 삽입 순서가 필요하면 `LinkedHashMap`, 정렬 순서가 필요하면 `TreeMap`을 쓴다.

## 연습문제

**문제 1.** 어떤 사이트에서 "오늘 접속한 서로 다른 방문자 IP의 개수"를 세려고 한다. List, Set, Map 중 무엇을 써야 하며 그 이유는?

<details>
<summary>힌트 / 해설</summary>

**힌트**: 같은 IP가 여러 번 접속해도 한 번만 세야 한다. 순서는 중요하지 않다.

**해설**: `Set`(예: `HashSet<String>`)이 정답이다. Set은 중복을 자동으로 제거하므로, 접속할 때마다 `add`하고 마지막에 `size()`를 부르면 고유 IP 개수가 나온다. List는 중복을 허용해 직접 제거해야 하고, Map은 키-값 쌍이 필요 없는 이 상황에 과하다.
</details>

**문제 2.** 다음 코드의 시간 복잡도를 Big-O로 나타내고, 그렇게 판단한 근거를 규칙에 따라 설명하라.
```java
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        System.out.println(i + j);
    }
}
for (int k = 0; k < n; k++) {
    System.out.println(k);
}
```

<details>
<summary>힌트 / 해설</summary>

**힌트**: 중첩 반복과 단일 반복을 각각 계산한 뒤 합치고, 낮은 차수 항을 버린다.

**해설**: 첫 부분은 중첩 반복이라 n × n = n²(규칙 1). 두 번째 부분은 n(규칙 1). 합치면 n² + n인데, 낮은 차수 항 n을 버리면(규칙 2) **O(n²)** 이다.
</details>

**문제 3.** `Member` 클래스를 HashMap의 키로 사용했는데, 분명히 같은 정보로 만든 객체로 `get`을 호출해도 `null`이 반환된다. 무엇을 빠뜨렸을 가능성이 높은가?

<details>
<summary>힌트 / 해설</summary>

**힌트**: HashMap이 키를 찾는 2단계 과정을 떠올려 보자.

**해설**: `equals()`와 `hashCode()`를 오버라이드하지 않았거나 둘 중 하나만 했을 가능성이 높다. 오버라이드하지 않으면 기본 구현(객체 주소 기반)이 쓰여, 내용이 같아도 다른 객체로 취급된다. **두 메서드를 반드시 함께, 같은 필드를 기준으로 오버라이드**해야 한다.
</details>

**문제 4.** ArrayList의 `add`가 "평균 O(1)"인데 내부적으로는 배열을 복사하는 O(n) 작업이 일어난다. 이 모순을 상환분석 관점에서 설명하라. 그리고 만약 용량을 2배가 아니라 매번 1칸씩만 늘린다면 평균 복잡도가 어떻게 되는가?

<details>
<summary>힌트 / 해설</summary>

**힌트**: 복사가 "얼마나 자주" 일어나는지, 그 총비용을 add 횟수로 나눠 보자.

**해설**: 용량을 2배씩 늘리면 복사는 드물게(용량이 꽉 찰 때만) 일어난다. n개를 추가하는 동안 총 복사량은 대략 n에 비례(4+8+16+... ≈ 2n)하므로, add 1회당 평균 비용은 상수 → **상환 O(1)**. 반면 1칸씩만 늘리면 매번 전체 복사가 일어나 총비용이 1+2+...+n = n(n+1)/2 = O(n²)가 되고, add당 평균은 **O(n)** 으로 나빠진다. 곱셈 증가가 필수인 이유다.
</details>

**문제 5.** 다음 두 메서드 시그니처 중 어느 것이 PECS 원칙에 맞는지 판단하라. `List`에서 값을 꺼내 합계를 구하는 메서드다.
```java
double sumA(List<? super Number> list)     // (a)
double sumB(List<? extends Number> list)   // (b)
```

<details>
<summary>힌트 / 해설</summary>

**힌트**: 이 메서드는 리스트에서 값을 **꺼내 읽기만** 한다. Producer인가 Consumer인가?

**해설**: (b)가 정답이다. 값을 꺼내 읽는(생산하는) 용도이므로 **Producer Extends** 원칙에 따라 `? extends Number`를 써야 한다. `? extends Number`면 원소를 `Number`로 안전하게 읽을 수 있다. (a)의 `? super Number`는 넣기(Consumer)용이라, 꺼낼 때 타입이 `Object`로만 보장되어 `doubleValue()`를 부를 수 없다.
</details>

## 요약

- **배열은 크기가 고정**되어 동적인 현실 데이터를 담기 어렵고, 이를 해결하려 등장한 것이 **컬렉션 프레임워크**다. List(순서·중복), Set(중복 제거), Map(키-값)으로 나뉘며 Map은 Collection을 상속하지 않는다.
- **ArrayList는 연속 메모리(조회 O(1))**, LinkedList는 포인터 연결(삽입·삭제 O(1))이지만, CPU 캐시 효율과 실제 사용 패턴 때문에 **실무에서는 거의 항상 ArrayList가 빠르다.**
- **HashMap은 해시 함수로 버킷을 정해 평균 O(1)** 조회를 이루며, 키를 hashCode(버킷)와 equals(정확한 키) 2단계로 찾으므로 **둘을 함께 오버라이드**해야 한다. resize(load factor 0.75), treeify(체인 8 초과 시 레드-블랙 트리화)로 성능을 유지한다.
- **Big-O는 성장률**을 뜻하며, 지배 연산 세기·상수 버리기·절반씩 줄면 O(log n)의 세 규칙으로 스스로 유도할 수 있다. TreeMap은 레드-블랙 트리로 균형을 유지해 정렬·범위 검색을 O(log n)에 제공한다.
- **제네릭은 컴파일 시점 타입 안전장치**이며, 하위 호환을 위해 런타임에 정보가 지워진다(타입 소거). 와일드카드는 PECS(Producer Extends, Consumer Super) 원칙으로 유연하게 활용한다.
