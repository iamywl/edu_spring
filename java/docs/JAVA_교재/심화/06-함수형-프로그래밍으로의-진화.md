## 6.1 왜 함수형 프로그래밍인가?

### 정의

**함수형 프로그래밍(Functional Programming)** 은 "무엇을 계산할 것인가(What)"에 집중하는 프로그래밍 방식입니다. 반대로 지금까지 우리가 써 온 방식은 **명령형 프로그래밍(Imperative Programming)** 으로, "어떻게 계산할 것인가(How)"를 한 단계씩 지시합니다.

### 원리: 명령형 vs 선언형, 무엇이 다른가?

두 방식의 차이를 요리에 비유해 봅시다.

**명령형(Imperative, "이렇게 해라")** 은 요리사에게 조리 과정을 하나하나 지시하는 것입니다.

> "냄비를 꺼내고, 물 550ml를 넣고, 불을 켜고, 물이 끓으면 면을 넣고, 3분 30초 기다린 뒤, 스프를 넣고, 불을 끄세요."

**선언형(Declarative, "이것을 원한다")** 은 결과만 말하는 것입니다.

> "라면 하나 주세요."

같은 결과지만, 후자는 "어떻게 만드는지"를 신경 쓰지 않습니다. 그 과정은 요리사(라이브러리, 런타임)에게 맡깁니다.

또 다른 비유는 택시입니다. 명령형은 기사에게 "직진하세요, 저기서 좌회전, 다음 신호에서 우회전..."이라고 일일이 길을 알려주는 것이고, 선언형은 "강남역으로 가주세요"라고 목적지만 말하는 것입니다. 함수형 프로그래밍은 이 **선언형에 가깝습니다.**

### 왜 선언형이 좋은가?

세 가지 이유가 있습니다.

1. **가독성(Readability)**: 코드가 "무엇을 하는지" 바로 보입니다. 반복문의 세부 동작을 머릿속으로 추적할 필요가 없습니다.
2. **실수 감소(Fewer Bugs)**: 인덱스 계산, 반복 종료 조건 같은 구현 세부사항에서 나오는 실수가 줄어듭니다.
3. **유지보수(Maintainability)**: 의도가 명확해서, 몇 달 뒤에 봐도 코드의 목적을 쉽게 파악할 수 있습니다.

### 다이어그램: 같은 작업, 두 가지 사고방식

```
[문제] 숫자 리스트에서 짝수만 골라 2배로 만들어 합계 구하기

명령형 (How - 과정을 지시):
  ┌──────────────────────────────────────┐
  │ int sum = 0;                          │
  │ for (int i = 0; i < list.size(); i++) │ ← 반복 방법을 직접 관리
  │   int n = list.get(i);                │
  │   if (n % 2 == 0)                      │ ← 조건 직접 검사
  │     sum += n * 2;                      │ ← 누적 직접 관리
  └──────────────────────────────────────┘
       "돌면서, 검사하고, 곱하고, 더해라"

선언형 (What - 결과를 기술):
  ┌──────────────────────────────────────────────────┐
  │ list.stream()                                     │
  │     .filter(n -> n % 2 == 0)   // 짝수만 원한다    │
  │     .map(n -> n * 2)            // 2배를 원한다     │
  │     .reduce(0, Integer::sum);   // 합계를 원한다    │
  └──────────────────────────────────────────────────┘
       "짝수를, 2배로, 합쳐라"
```

### 코드: 두 방식의 실제 비교

```java
import java.util.List;
import java.util.ArrayList;

public class ImperativeVsDeclarative {
    public static void main(String[] args) {
        List<Integer> numbers = List.of(1, 2, 3, 4, 5, 6);

        // 명령형: 어떻게 할지 하나하나 지시
        int sumImperative = 0;
        for (int n : numbers) {
            if (n % 2 == 0) {
                sumImperative += n * 2;
            }
        }
        System.out.println("명령형 결과: " + sumImperative); // 24 (=4+8+12)

        // 선언형: 무엇을 원하는지 기술
        int sumDeclarative = numbers.stream()
                .filter(n -> n % 2 == 0)   // 짝수만
                .map(n -> n * 2)            // 2배
                .reduce(0, Integer::sum);   // 합계
        System.out.println("선언형 결과: " + sumDeclarative); // 24
    }
}
```

두 코드는 같은 결과(24)를 냅니다. 하지만 선언형 코드는 "짝수를 골라서, 2배로 만들어, 더한다"는 의도가 문장처럼 읽힙니다. 이것이 함수형 프로그래밍이 지향하는 세계입니다.

## 6.2 Lambda: 이름 없는 함수

### 정의

**Lambda 표현식(Lambda Expression)** 은 이름이 없는, 한 번 쓰고 버리는 함수입니다. `(매개변수) -> { 실행할 코드 }` 형태로 동작 자체를 값처럼 전달할 수 있게 해 줍니다.

### 원리: 왜 Lambda가 등장했는가?

Java에서 "동작을 전달"하려면 원래 매우 번거로웠습니다. 자바에서는 함수를 그냥 넘길 수 없고, 반드시 객체에 담아야 했기 때문입니다. 그래서 등장한 것이 **익명 클래스(anonymous class)** 였습니다.

문제는 익명 클래스가 지나치게 장황하다는 것입니다. 비유하자면, **커피 한 잔을 주문하는데 10장짜리 주문서를 작성**하는 셈입니다. 정작 하고 싶은 일("클릭되면 메시지 출력")은 한 줄인데, 그 한 줄을 감싸는 껍데기가 대여섯 줄이나 됩니다.

```java
// 익명 클래스: 하고 싶은 일은 한 줄인데 껍데기가 잔뜩
button.setOnClickListener(new OnClickListener() {
    @Override
    public void onClick(View v) {
        System.out.println("클릭됨!");
    }
});

// Lambda: 진짜 하고 싶은 일만 남는다
button.setOnClickListener(v -> System.out.println("클릭됨!"));
```

두 코드는 완전히 동일하게 동작합니다. Lambda는 컴파일러가 "어차피 구현할 메서드가 하나뿐이니까, 껍데기는 내가 알아서 채울게"라고 추론해 주기 때문에 가능합니다.

### 비유: 이름 없는 일회용 레시피

일반 메서드는 "김치찌개_만들기"처럼 **이름이 있어서 여러 번 재사용**할 수 있는 레시피입니다. 반면 Lambda는 **이름이 없고, 그 자리에서 한 번 쓰고 끝나는 일회용 레시피**입니다.

즉, "짧고, 이름 붙일 필요 없고, 한 번만 쓸 동작"을 간결하게 표현할 때 Lambda가 빛을 발합니다.

### 다이어그램: 익명 클래스에서 Lambda로

```
익명 클래스 (5줄의 껍데기)
  ┌────────────────────────────────────┐
  │ new OnClickListener() {  ← 껍데기    │
  │   @Override               ← 껍데기   │
  │   public void onClick() { ← 껍데기   │
  │       핵심 동작 한 줄!!!             │  ← 진짜 하고 싶은 것
  │   }                       ← 껍데기   │
  │ }                         ← 껍데기   │
  └────────────────────────────────────┘

              │ 컴파일러가 껍데기를 추론
              ▼
Lambda (핵심만 남음)
  ┌────────────────────────────────────┐
  │  v -> 핵심 동작 한 줄!!!            │
  └────────────────────────────────────┘
```

### 함수형 인터페이스: Lambda가 담기는 그릇

Lambda는 아무 데나 쓸 수 없습니다. **추상 메서드가 딱 하나뿐인 인터페이스(함수형 인터페이스, Functional Interface)** 에만 담깁니다. 왜냐하면 컴파일러가 "이 Lambda가 어떤 메서드를 구현하는 것인지" 헷갈리지 않아야 하기 때문입니다. 메서드가 하나뿐이면 고민할 필요가 없습니다.

Java는 자주 쓰는 형태를 표준 함수형 인터페이스로 미리 만들어 두었습니다. 이 네 가지만 알면 대부분의 상황을 커버합니다.

```
┌────────────┬─────────────────┬──────────────────────────────┐
│ 인터페이스  │ 형태             │ 의미                          │
├────────────┼─────────────────┼──────────────────────────────┤
│ Function   │ T -> R          │ 입력 받아 변환해서 반환        │
│ Predicate  │ T -> boolean    │ 입력 받아 참/거짓 판별         │
│ Consumer   │ T -> void       │ 입력 받아 소비(출력 등), 반환X │
│ Supplier   │ () -> T         │ 입력 없이 값을 공급            │
└────────────┴─────────────────┴──────────────────────────────┘
```

```java
import java.util.function.*;

public class FunctionalInterfaces {
    public static void main(String[] args) {
        // Function<T, R>: 입력을 변환해 반환 (String -> Integer)
        Function<String, Integer> length = s -> s.length();
        System.out.println(length.apply("hello")); // 5

        // Predicate<T>: 참/거짓 판별 (Integer -> boolean)
        Predicate<Integer> isEven = n -> n % 2 == 0;
        System.out.println(isEven.test(4)); // true
        System.out.println(isEven.test(7)); // false

        // Consumer<T>: 값을 소비, 반환값 없음
        Consumer<String> printer = s -> System.out.println("소비: " + s);
        printer.accept("데이터"); // 소비: 데이터

        // Supplier<T>: 입력 없이 값을 공급
        Supplier<String> greeting = () -> "안녕하세요";
        System.out.println(greeting.get()); // 안녕하세요
    }
}
```

### 메서드 참조(::): Lambda를 더 짧게

Lambda가 "기존에 있는 메서드를 그냥 호출하기만 하는" 경우, 이를 더 짧게 쓸 수 있습니다. 바로 **메서드 참조(Method Reference)** 인 `::` 문법입니다.

```java
// Lambda: 매개변수를 받아서, 그 매개변수의 메서드를 호출만 함
Function<String, Integer> f1 = s -> s.length();

// 메서드 참조: "String의 length 메서드를 참조" — 매개변수가 생략됨
Function<String, Integer> f2 = String::length;

// 둘은 완전히 동일하게 동작
System.out.println(f1.apply("hello")); // 5
System.out.println(f2.apply("hello")); // 5
```

메서드 참조에는 네 가지 형태가 있습니다.

```
┌──────────────────────┬───────────────────────┬──────────────────────┐
│ 종류                  │ 예시                   │ 대응하는 Lambda       │
├──────────────────────┼───────────────────────┼──────────────────────┤
│ 정적 메서드           │ Integer::parseInt      │ s -> Integer.parseInt(s)│
│ 특정 객체의 인스턴스   │ System.out::println    │ x -> System.out.println(x)│
│ 임의 객체의 인스턴스   │ String::toUpperCase    │ s -> s.toUpperCase() │
│ 생성자                │ ArrayList::new         │ () -> new ArrayList()│
└──────────────────────┴───────────────────────┴──────────────────────┘
```

### 흔한 오해: "Lambda는 무조건 좋다"

Lambda가 항상 정답은 아닙니다. Lambda가 너무 길어지면(대략 5줄 이상) 오히려 가독성이 떨어집니다. 이럴 때는 **차라리 이름 있는 일반 메서드로 추출**하고, 그 메서드를 메서드 참조로 넘기는 것이 낫습니다. Lambda는 "짧고 명확할 때" 가장 강력합니다.

## 6.3 Stream: 데이터의 파이프라인

### 정의

**Stream** 은 데이터의 흐름(stream)입니다. 컬렉션의 데이터를 하나씩 흘려보내면서, 파이프라인을 통해 걸러내고(filter), 변환하고(map), 수집하는(collect) 일련의 처리 과정입니다.

### 원리: 공장 컨베이어 벨트

Stream을 이해하는 가장 좋은 비유는 **공장의 컨베이어 벨트**입니다.

```
원재료         세척          불량 제거        포장          수집
(사과들)  →   (씻기)   →   (썩은 것 제거) →  (박스에)  →  (창고에)

list.stream()                                          .collect(...)
              .map(세척)   .filter(불량아님)  .map(포장)
```

명령형 방식은 **직원 한 명이 사과를 하나씩 직접 들고 다니며 씻고, 확인하고, 포장**하는 것입니다. Stream(선언형)은 **컨베이어 벨트를 설치해 두고, 재료를 흘려보내면 각 기계가 자동으로 처리**하는 것입니다. 우리는 "어떤 기계를 어떤 순서로 놓을지"만 설계하면 됩니다.

### 다이어그램: Stream 파이프라인 구조

```
    ┌─────────┐   ┌──────────┐   ┌────────┐   ┌────────┐   ┌──────────┐
    │  소스    │──▶│ 중간연산  │──▶│ 중간연산│──▶│중간연산 │──▶│ 최종연산  │
    │ (컬렉션) │   │ filter   │   │  map   │   │ sorted │   │ collect  │
    └─────────┘   └──────────┘   └────────┘   └────────┘   └──────────┘
     stream()      "거르기"       "변환"       "정렬"       "수집 상자"
                   └──────────── 벨트 위의 기계들 ────────┘      │
                                                                ▼
                                                         [ 결과 나옴 ]
```

### 중간 연산 vs 최종 연산

Stream의 연산은 두 종류입니다.

```
┌────────────┬────────────────────────────────┬─────────────────────┐
│ 종류        │ 역할                            │ 예시                 │
├────────────┼────────────────────────────────┼─────────────────────┤
│ 중간 연산   │ 벨트 위의 각 기계.               │ filter, map,        │
│(Intermediate)│ Stream을 반환 → 이어붙이기 가능 │ sorted, distinct,   │
│            │                                 │ limit               │
├────────────┼────────────────────────────────┼─────────────────────┤
│ 최종 연산   │ 벨트 끝의 수집 상자.             │ collect, forEach,   │
│(Terminal)  │ 결과를 만들고 Stream을 소모       │ count, reduce, sum  │
└────────────┴────────────────────────────────┴─────────────────────┘
```

**핵심 규칙: 최종 연산이 호출되기 전까지 중간 연산은 실행되지 않습니다.** 컨베이어 벨트를 아무리 잘 설치해도, 벨트를 "켜기(최종 연산)" 전까지는 사과 한 알도 움직이지 않는 것과 같습니다.

```java
import java.util.List;

public class LazyDemo {
    public static void main(String[] args) {
        List<Integer> numbers = List.of(1, 2, 3);

        // 중간 연산만 나열 — 아무것도 출력되지 않는다!
        numbers.stream()
               .filter(n -> {
                   System.out.println("필터 검사: " + n);
                   return n % 2 == 1;
               });
        System.out.println("--- 최종 연산이 없어서 위 filter는 실행 안 됨 ---");

        // 최종 연산(forEach)을 붙이면 그제서야 벨트가 돌아간다
        numbers.stream()
               .filter(n -> n % 2 == 1)
               .forEach(n -> System.out.println("홀수: " + n));
        // 홀수: 1
        // 홀수: 3
    }
}
```

### Lazy Evaluation: 필요한 만큼만 처리

Stream의 또 다른 핵심은 **게으른 평가(Lazy Evaluation)** 입니다. Stream은 데이터를 미리 전부 처리하지 않고, "필요한 만큼만" 처리합니다.

100만 개의 데이터에서 조건에 맞는 **3개만** 찾는다고 해 봅시다.

```
Eager(성급한 평가):
  100만 개 전부 필터링 완료 → 그중 3개 꺼냄  (99만 9997개 헛수고!)

Lazy(게으른 평가):
  1개 확인 → 2개 확인 → ... → 3개 모이는 순간 STOP  (나머지는 건드리지도 않음)
```

서점에 비유하면, "좋은 책 3권"을 찾을 때 서점의 모든 책을 다 검토하는 게 아니라, 좋은 책이 3권 모이는 순간 검색을 멈추는 것입니다. 필요한 만큼만 처리하니 훨씬 효율적입니다.

```java
import java.util.stream.Stream;

public class LazyEvaluation {
    public static void main(String[] args) {
        // 무한에 가까운 스트림에서도 3개만 필요하면 3번만 처리
        Stream.iterate(1, n -> n + 1)      // 1, 2, 3, 4, ... 무한
              .filter(n -> {
                  System.out.println("검사: " + n);
                  return n % 2 == 0;
              })
              .limit(3)                     // 짝수 3개면 충분
              .forEach(n -> System.out.println("선택: " + n));
        // 검사: 1
        // 검사: 2
        // 선택: 2
        // 검사: 3
        // 검사: 4
        // 선택: 4
        // 검사: 5
        // 검사: 6
        // 선택: 6   ← 여기서 멈춤! 7 이상은 검사조차 안 함
    }
}
```

### 실전 코드: filter / map / collect / reduce

이제 실제로 자주 쓰이는 Stream 연산들을 코드로 살펴봅시다.

```java
import java.util.List;
import java.util.stream.Collectors;

public class StreamInAction {
    record Person(String name, int age) {}

    public static void main(String[] args) {
        List<Person> people = List.of(
            new Person("김철수", 25),
            new Person("이영희", 30),
            new Person("박민수", 17),
            new Person("최지우", 42)
        );

        // 1) filter + map + collect: 성인의 이름만 리스트로
        List<String> adultNames = people.stream()
                .filter(p -> p.age() >= 20)      // 성인만
                .map(Person::name)                // 이름만 추출 (메서드 참조)
                .collect(Collectors.toList());
        System.out.println(adultNames); // [김철수, 이영희, 최지우]

        // 2) reduce: 모든 나이의 합계
        int totalAge = people.stream()
                .map(Person::age)
                .reduce(0, Integer::sum);         // 초기값 0에서 누적
        System.out.println("나이 합계: " + totalAge); // 114

        // 3) count: 미성년자 수
        long minorCount = people.stream()
                .filter(p -> p.age() < 20)
                .count();
        System.out.println("미성년자 수: " + minorCount); // 1

        // 4) 정렬 + 최대: 가장 나이 많은 사람
        people.stream()
              .max((a, b) -> a.age() - b.age())
              .ifPresent(p -> System.out.println("최고령: " + p.name())); // 최지우
    }
}
```

### Collectors: 수집의 다양한 방법

최종 연산 `collect`는 `Collectors`와 함께 매우 강력해집니다. 데이터를 리스트로 모으는 것뿐 아니라, 그룹으로 묶거나, 통계를 내거나, 문자열로 합칠 수 있습니다.

```java
import java.util.*;
import java.util.stream.Collectors;

public class CollectorsDemo {
    record Person(String name, int age) {}

    public static void main(String[] args) {
        List<Person> people = List.of(
            new Person("김철수", 25),
            new Person("이영희", 30),
            new Person("박민수", 17),
            new Person("최지우", 42)
        );

        // toList: 리스트로 수집
        List<String> names = people.stream()
                .map(Person::name)
                .collect(Collectors.toList());
        System.out.println(names); // [김철수, 이영희, 박민수, 최지우]

        // joining: 문자열로 이어붙이기
        String joined = people.stream()
                .map(Person::name)
                .collect(Collectors.joining(", ", "[", "]"));
        System.out.println(joined); // [김철수, 이영희, 박민수, 최지우]

        // groupingBy: 조건별로 그룹화 (성인/미성년자)
        Map<Boolean, List<Person>> grouped = people.stream()
                .collect(Collectors.groupingBy(p -> p.age() >= 20));
        System.out.println(grouped.get(true).size());  // 3 (성인)
        System.out.println(grouped.get(false).size()); // 1 (미성년자)

        // averagingInt: 평균 나이
        double avgAge = people.stream()
                .collect(Collectors.averagingInt(Person::age));
        System.out.println("평균 나이: " + avgAge); // 28.5
    }
}
```

### Lambda의 변수 캡처: effectively final

Lambda는 바깥의 지역 변수를 사용할 수 있습니다. 이를 **변수 캡처(variable capture)** 라고 합니다. 그런데 한 가지 제약이 있습니다. **캡처된 지역 변수는 final이거나 사실상 final(effectively final)이어야 합니다.**

"사실상 final"이란, `final` 키워드를 명시하지 않았더라도 **한 번 값이 정해진 뒤 다시 바뀌지 않는** 변수를 말합니다.

```java
public class VariableCapture {
    public static void main(String[] args) {
        int multiplier = 3; // final을 안 붙였지만 값을 바꾸지 않으면 "사실상 final"

        // OK: multiplier가 이후 변경되지 않으므로 캡처 가능
        java.util.function.Function<Integer, Integer> multiply = n -> n * multiplier;
        System.out.println(multiply.apply(5)); // 15

        // multiplier = 4; // ← 이 줄의 주석을 풀면 위 Lambda가 컴파일 에러!
        //                    "지역 변수는 final이거나 effectively final이어야 함"
    }
}
```

**왜 이런 제약이 있을까요?** Lambda는 생성될 때 그 변수의 "값을 복사"해서 가지고 갑니다. 만약 원본 변수가 나중에 바뀌면, Lambda가 들고 있는 복사본과 불일치가 생겨 혼란과 버그가 발생합니다. 그래서 Java는 아예 "캡처하는 변수는 바뀌지 못하게" 강제하여 이런 혼란을 원천 차단합니다.

## 6.4 Optional: null과의 전쟁

### 정의

**Optional\<T\>** 은 "값이 있을 수도, 없을 수도 있다"는 것을 **명시적으로** 표현하는 상자입니다. `null`을 직접 다루는 대신, "비어 있을 가능성"을 타입으로 드러냅니다.

### 원리: null이라는 10억 달러짜리 실수

`null`은 프로그래머의 영원한 적입니다. 실행 중에 갑자기 터지는 `NullPointerException(NPE)`은 자바 개발자라면 누구나 겪어 본 고통입니다.

흥미롭게도, null 참조를 발명한 컴퓨터 과학자 **Tony Hoare** 자신이 이를 두고 **"10억 달러짜리 실수(billion-dollar mistake)"** 라고 후회했습니다. 그가 1965년 null을 도입한 이후, 수많은 시스템 오류와 취약점이 null에서 비롯되었기 때문입니다.

```java
String name = null;
name.length(); // 💥 NullPointerException! 실행하는 순간 폭발
```

문제의 본질은 **null이 "아무것도 없음"을 의미하지만, 겉으로는 전혀 보이지 않는다**는 점입니다. `String name`이라는 변수 선언만 봐서는, 이것이 실제 문자열을 가리키는지 null인지 알 수 없습니다. 마치 **지뢰밭**을 걷는 것과 같습니다. 어디에 지뢰(null)가 묻혀 있는지 알 수 없으니, 매번 조심스럽게 `if (name != null)` 검사를 해야 합니다.

### 비유: 선물 상자

Optional은 이 문제를 "선물 상자"로 해결합니다.

- **일반 변수**: 포장 안 된 물건. 열어봐야(사용해봐야) 내용물이 있는지 알 수 있음 → 위험.
- **Optional**: 상자에 "내용물 있음" 또는 "비어 있음"이라고 **적혀 있는** 선물 상자. 열기 전에 상태를 알 수 있음 → 안전.

```
일반 String              Optional<String>
┌──────────┐            ┌──────────────────────┐
│    ???    │            │  isPresent() = true   │
│ (열어봐야  │            │  ┌────────────────┐   │
│  안다)    │            │  │   "실제 값"     │   │
└──────────┘            │  └────────────────┘   │
  위험!                  └──────────────────────┘

                        ┌──────────────────────┐
                        │  isEmpty() = true      │
                        │  ┌────────────────┐   │
                        │  │   (비어 있음)   │   │
                        │  └────────────────┘   │
                        └──────────────────────┘
                            안전! 상태가 명시됨
```

### 잘못된 사용(Anti-Pattern)

Optional은 강력하지만, 잘못 쓰면 오히려 코드를 나쁘게 만듭니다. 다음은 피해야 할 사용법입니다.

1. **필드로 사용하지 말 것**: Optional은 "반환값"에만 쓰도록 설계되었습니다. 필드에 쓰면 직렬화 문제 등 부작용이 생깁니다.
2. **파라미터로 사용하지 말 것**: 메서드 파라미터를 Optional로 받으면, 호출자가 매번 `Optional.of(...)`로 감싸야 해서 불편합니다.
3. **`isPresent() + get()` 조합을 피할 것**: 이는 결국 `if (x != null)`과 다를 바 없습니다. Optional을 쓰는 의미가 없어집니다.
4. **컬렉션을 Optional로 감싸지 말 것**: 값이 없으면 그냥 빈 컬렉션(`List.of()`)을 반환하는 게 낫습니다.

```java
// 나쁜 예: isPresent + get — null 체크와 다를 게 없다
Optional<String> opt = findName();
if (opt.isPresent()) {
    String name = opt.get();
    System.out.println(name);
}

// 좋은 예: ifPresent — 선언적이고 안전
findName().ifPresent(name -> System.out.println(name));
```

### 좋은 사용: 메서드 체이닝

Optional의 진가는 `orElse`, `orElseGet`, `orElseThrow`, `map`, `flatMap` 같은 메서드를 **체이닝**할 때 드러납니다.

```java
import java.util.Optional;

public class OptionalChaining {
    record Address(String city) {}
    record Person(String name, Address address) {}

    public static void main(String[] args) {
        // orElse: 값이 없으면 기본값 반환
        String name1 = Optional.<String>empty().orElse("이름없음");
        System.out.println(name1); // 이름없음

        // orElseGet: 값이 없을 때만 함수를 실행해 기본값 생성 (지연 실행)
        String name2 = Optional.of("홍길동").orElseGet(() -> "생성된 기본값");
        System.out.println(name2); // 홍길동 (Supplier는 실행 안 됨)

        // orElseThrow: 값이 없으면 예외 발생
        try {
            Optional.empty().orElseThrow(() -> new RuntimeException("값이 없습니다"));
        } catch (RuntimeException e) {
            System.out.println("예외: " + e.getMessage()); // 예외: 값이 없습니다
        }

        // map: 값이 있으면 변환, 없으면 그대로 빈 Optional
        Optional<Integer> len = Optional.of("hello").map(String::length);
        System.out.println(len.get()); // 5

        // 실전 체이닝: 사람 -> 주소 -> 도시, 중간에 null이어도 NPE 없음
        Person person = new Person("김철수", new Address("서울"));
        String city = Optional.ofNullable(person)
                .map(Person::address)
                .map(Address::city)
                .orElse("주소없음");
        System.out.println(city); // 서울

        // 만약 주소가 null이었다면?
        Person noAddr = new Person("이영희", null);
        String city2 = Optional.ofNullable(noAddr)
                .map(Person::address)     // null이 되어 빈 Optional
                .map(Address::city)       // 건너뜀 (NPE 안 남!)
                .orElse("주소없음");
        System.out.println(city2); // 주소없음
    }
}
```

**`orElse`와 `orElseGet`의 차이**를 주목하세요. `orElse(x)`는 값이 있든 없든 `x`를 항상 준비하지만, `orElseGet(() -> ...)`은 값이 **없을 때만** 함수를 실행합니다. 기본값을 만드는 비용이 큰 경우(예: DB 조회) `orElseGet`이 훨씬 효율적입니다.

## ⚠️ 흔한 오해와 함정

1. **"Stream은 반복문보다 항상 빠르다"** — 아닙니다. Stream은 파이프라인 구축 비용과 박싱/언박싱 오버헤드가 있어서, 단순 반복에서는 일반 `for`문이 더 빠를 수 있습니다. Stream의 장점은 성능이 아니라 **가독성과 표현력**입니다. (병렬 스트림은 다른 얘기지만, 그것도 만능은 아닙니다.)

2. **"Stream을 재사용할 수 있다"** — 없습니다. Stream은 최종 연산으로 한 번 소모되면 다시 쓸 수 없습니다. 다시 쓰려면 원본 컬렉션에서 `stream()`을 새로 호출해야 합니다. 이미 소모된 Stream을 또 쓰면 `IllegalStateException`이 발생합니다.

3. **"중간 연산만 써도 실행된다"** — 안 됩니다. 최종 연산이 없으면 벨트는 돌지 않습니다. `filter`만 잔뜩 써 놓고 아무 일도 안 일어나서 당황하는 경우가 많습니다.

4. **"Optional.get()을 그냥 쓰면 된다"** — 위험합니다. 값이 없을 때 `get()`을 호출하면 `NoSuchElementException`이 터집니다. 이는 NPE와 다를 게 없습니다. 항상 `orElse`, `orElseThrow`, `ifPresent` 등으로 "값이 없을 때"를 명시적으로 처리하세요.

5. **"Lambda 안에서 지역 변수를 마음대로 바꿀 수 있다"** — 없습니다. 캡처하는 지역 변수는 effectively final이어야 합니다. 카운터를 증가시키고 싶다면 배열이나 `AtomicInteger` 같은 우회책이 필요하지만, 애초에 그런 상황은 Stream의 `reduce`나 `count`로 푸는 게 더 함수형답습니다.

## 연습문제

**문제 1.** 다음 명령형 코드를 Stream을 이용한 선언형 코드로 바꿔 보세요.
```java
List<String> words = List.of("apple", "banana", "cherry", "date");
List<String> result = new ArrayList<>();
for (String w : words) {
    if (w.length() > 5) {
        result.add(w.toUpperCase());
    }
}
```
> **힌트**: `filter`로 조건을 걸고, `map`으로 변환한 뒤, `collect`로 모으세요.
>
> **해설**:
> ```java
> List<String> result = words.stream()
>         .filter(w -> w.length() > 5)   // 5글자 초과
>         .map(String::toUpperCase)       // 대문자로 (메서드 참조)
>         .collect(Collectors.toList());
> // [BANANA, CHERRY] — apple(5), date(4)는 제외됨
> ```

**문제 2.** 다음 코드는 왜 컴파일 에러가 날까요? 그리고 어떻게 고칠 수 있을까요?
```java
int count = 0;
List.of(1, 2, 3).forEach(n -> count += n);
```
> **힌트**: Lambda가 캡처하는 지역 변수의 제약을 떠올려 보세요.
>
> **해설**: `count`는 Lambda에 캡처되는 지역 변수인데, `count += n`으로 값을 변경하려 하기 때문에 "effectively final" 규칙을 위반합니다. 해결책은 애초에 함수형답게 `reduce`나 `sum`을 쓰는 것입니다.
> ```java
> int count = List.of(1, 2, 3).stream()
>         .mapToInt(Integer::intValue)
>         .sum(); // 6
> ```

**문제 3.** 아래 두 표현식의 차이를 설명하세요. 어떤 상황에서 어느 쪽을 써야 할까요?
```java
String a = optional.orElse(expensiveDbCall());
String b = optional.orElseGet(() -> expensiveDbCall());
```
> **힌트**: 각 메서드가 언제 인자(또는 함수)를 평가하는지 생각해 보세요.
>
> **해설**: `orElse(expensiveDbCall())`은 `optional`에 값이 **있든 없든** `expensiveDbCall()`이 항상 먼저 실행됩니다(인자가 미리 평가됨). 반면 `orElseGet(() -> ...)`은 값이 **없을 때만** 함수를 실행합니다. 기본값 생성 비용이 크다면(DB 조회, 네트워크 등) 반드시 `orElseGet`을 쓰세요.

**문제 4.** 다음 Stream 코드는 아무것도 출력하지 않습니다. 왜일까요?
```java
List.of("a", "b", "c").stream()
    .map(s -> s.toUpperCase());
```
> **힌트**: 중간 연산과 최종 연산의 관계를 떠올려 보세요.
>
> **해설**: `map`은 **중간 연산**입니다. 최종 연산이 없으면 게으른 평가 원칙에 따라 벨트가 돌지 않습니다. `.forEach(System.out::println)` 같은 최종 연산을 붙여야 실제로 실행됩니다.

**문제 5.** `Optional.ofNullable`과 `Optional.of`의 차이는 무엇인가요? `null`을 넣으면 각각 어떻게 되나요?
> **힌트**: 두 메서드가 null을 받아들이는지 실험해 보세요.
>
> **해설**: `Optional.of(null)`은 즉시 `NullPointerException`을 던집니다(of는 null을 허용하지 않음). `Optional.ofNullable(null)`은 안전하게 빈 Optional을 반환합니다. **값이 null일 수도 있는 경우엔 반드시 `ofNullable`을 써야** 합니다.

## 요약

- **함수형 프로그래밍은 "어떻게(How)"보다 "무엇을(What)"에 집중하는 선언형 방식**으로, 가독성을 높이고 실수를 줄이며 유지보수를 쉽게 만듭니다.
- **Lambda는 이름 없는 일회용 함수**로, 익명 클래스의 장황함을 없애고 동작을 값처럼 전달합니다. `Function`/`Predicate`/`Consumer`/`Supplier` 같은 함수형 인터페이스에 담기며, `::`(메서드 참조)로 더 짧게 쓸 수 있습니다.
- **Stream은 데이터 처리 파이프라인(컨베이어 벨트)** 으로, 중간 연산(filter/map/sorted)과 최종 연산(collect/reduce/count)으로 구성되며, 최종 연산 전까지 실행되지 않는 게으른 평가(Lazy Evaluation) 덕분에 필요한 만큼만 효율적으로 처리합니다.
- **Optional은 "값이 없을 수 있음"을 타입으로 명시하는 선물 상자**로, null이 일으키는 NPE를 예방합니다. `isPresent()+get()` 같은 안티패턴 대신 `map`/`orElse`/`orElseThrow` 체이닝으로 안전하고 선언적으로 다뤄야 합니다.
- 함수형 요소는 성능이 아니라 **표현력과 안전성**을 위한 도구입니다. 적재적소에 쓸 때 가장 강력합니다.
