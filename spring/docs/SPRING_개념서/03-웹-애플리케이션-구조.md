# Chapter 3: 웹 애플리케이션의 구조

> "HTTP를 모르고 웹 개발을 하는 것은, 도로교통법을 모르고 운전대를 잡는 것과 같다."

> **🐳 실습 환경 — 이 장의 실습은 chapter05(컨테이너 `spring-ch05-web`, DB 불필요)로 진행한다**
> ```bash
> cd spring/chapter05-spring-web && docker compose up --build
> ```
> API 호출은 `requests.http` 파일로 (VS Code REST Client). 컨테이너 상태 확인: `docker ps`

운전면허 학원에서 가장 먼저 배우는 것은 액셀과 브레이크가 아니라 **도로교통법**입니다. 왜 그럴까요? 자동차를 움직이는 기술만 익히면 당장 앞으로는 나아갈 수 있습니다. 하지만 신호등의 의미를 모르고, 차선의 규칙을 모르고, 우선순위를 모른다면 그 운전은 반드시 사고로 이어집니다. 도로는 나 혼자 쓰는 공간이 아니라 **약속을 공유하는 공간**이기 때문입니다.

웹 개발도 정확히 같습니다. 많은 초심자가 Spring Boot의 애노테이션을 외우고 컨트롤러 코드를 복사-붙여넣기 하는 것부터 시작합니다. 당장은 화면에 "Hello World"가 찍히고 데이터도 저장됩니다. 하지만 곧 이런 질문 앞에서 무너집니다.

- "왜 데이터 조회는 GET인데 삭제는 DELETE로 해야 하나요? 그냥 다 POST로 하면 안 되나요?"
- "왜 성공했는데 200을 주고, 없는 걸 찾으면 404를 줘야 하나요? 그냥 다 200 주고 본문에 `success: false`라고 쓰면 안 되나요?"
- "왜 컨트롤러에 다 짜면 되는데 굳이 서비스, 리포지토리로 나누나요?"
- "왜 DB에서 꺼낸 객체를 그대로 응답하면 안 되고 굳이 DTO를 하나 더 만드나요?"

이 모든 질문의 답은 하나의 문장으로 요약됩니다. **웹은 나 혼자 쓰는 공간이 아니라, 수많은 클라이언트와 서버가 약속(프로토콜)을 공유하는 공간이기 때문입니다.** HTTP라는 도로교통법을 이해하고, REST라는 도로 설계 원칙을 이해하고, 레이어드 아키텍처라는 차량 정비 구조를 이해할 때, 비로소 "동작하는 코드"를 넘어 "이해할 수 있고, 유지보수할 수 있고, 예측할 수 있는 코드"를 작성할 수 있습니다.

이 장에서는 웹 애플리케이션이 **왜 지금의 모습으로 만들어졌는가**를 여섯 개의 주제로 파고듭니다. HTTP 프로토콜의 본질에서 시작해, REST API의 설계 철학, Spring MVC가 요청을 처리하는 내부 흐름, 계층을 나누는 이유, DTO가 존재하는 이유, 그리고 검증(Validation)이 서버에 반드시 있어야 하는 이유까지. 문법이 아니라 **원리**를 이해하는 것이 목표입니다.

## 3.1 HTTP 프로토콜의 기초 — 웹의 도로교통법

### 개념: HTTP는 "요청과 응답"이라는 약속이다

HTTP(HyperText Transfer Protocol)는 클라이언트(브라우저, 앱, curl 등)와 서버가 대화하는 규칙입니다. 이 대화는 놀라울 만큼 단순한 구조를 가집니다.

> 클라이언트가 **요청(Request)**을 보내면, 서버가 **응답(Response)**을 돌려준다. 끝.

이 단순함 위에 오늘날의 모든 웹 서비스가 세워져 있습니다. 그런데 이 규칙에는 반드시 알아야 할 중요한 성질이 하나 있습니다. **HTTP는 상태를 기억하지 않는다(Stateless).** 서버는 방금 온 요청과 그 다음 요청이 같은 사람에게서 온 것인지 기본적으로 알지 못합니다. 마치 편지와 같습니다.

### 원리: 편지 비유로 이해하는 요청/응답

HTTP 통신은 우편 시스템과 정확히 닮았습니다. 편지를 쓸 때 우리는 반드시 **봉투(헤더)**에 받는 사람 주소와 방식을 적고, **내용물(본문)**을 넣습니다. 답장도 마찬가지입니다.

```
┌─────────────────────────────────────────────────────────┐
│                     클라이언트 (나)                        │
│                                                           │
│   "이 사람 정보 좀 주세요" 라는 편지를 씁니다.               │
└────────────────────────┬──────────────────────────────────┘
                         │
                         │  요청 (Request) = 보내는 편지
                         │
                         ▼
   ┌───────────────────────────────────────────┐
   │  GET /users/1 HTTP/1.1        ← 무엇을, 어떻게 │  (요청 라인)
   │  Host: api.example.com        ← 받는 사람 주소 │
   │  Accept: application/json     ← 답장은 이 형식으로│  (헤더 = 봉투)
   │  Authorization: Bearer xxx    ← 나 이런 사람이야 │
   │                                              │
   │  (본문 없음 - GET은 보통 내용물이 없다)          │  (바디 = 내용물)
   └───────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│                     서버 (우체국+상대방)                    │
│                                                           │
│   편지를 읽고 → 처리하고 → 답장을 씁니다.                    │
└────────────────────────┬──────────────────────────────────┘
                         │
                         │  응답 (Response) = 답장 편지
                         │
                         ▼
   ┌───────────────────────────────────────────┐
   │  HTTP/1.1 200 OK              ← 처리 결과   │  (상태 라인)
   │  Content-Type: application/json ← 내용 형식  │  (헤더 = 봉투)
   │  Content-Length: 68                          │
   │                                              │
   │  { "id": 1, "name": "홍길동" } ← 실제 데이터  │  (바디 = 내용물)
   └───────────────────────────────────────────┘
```

편지 비유가 강력한 이유는 HTTP의 **무상태(Stateless)** 성질을 자연스럽게 설명하기 때문입니다. 편지를 받는 우체국 직원은 당신이 어제 보낸 편지를 기억하지 않습니다. 매번 편지 봉투에 "나 누구입니다"(Authorization 헤더)라고 적어야만 상대가 나를 알아봅니다. 그래서 웹에서는 로그인 상태를 유지하기 위해 매 요청마다 토큰이나 세션 쿠키를 함께 보내는 것입니다. 상태를 기억하지 않기에, 서버는 수백만 명의 사용자를 각각 기억할 필요 없이 확장(scale-out)할 수 있습니다. 이 무상태성은 불편함이 아니라 웹이 거대해질 수 있었던 **핵심 설계 결정**입니다.

실제 요청/응답의 원본(raw) 형태를 보면 이렇습니다.

```http
GET /users/1 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

```

```http
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 68

{
  "id": 1,
  "name": "홍길동",
  "email": "hong@example.com"
}
```

### 원리: HTTP 메서드 — "무엇을 하려는가"를 동사로 표현하다

편지를 보낼 때 "정보를 알려달라"는 편지와 "이 정보를 등록해달라"는 편지는 목적이 다릅니다. HTTP는 이 **목적(의도)**을 URL이 아니라 **메서드(Method)**로 표현하기로 약속했습니다. 이것이 HTTP 메서드입니다.

| 메서드 | 비유 | 의미 | 예시 |
|--------|------|------|------|
| **GET** | 도서관에서 책을 열람만 함 | 리소스 **조회** (읽기 전용) | `GET /users/1` |
| **POST** | 새 회원 가입 신청서를 제출함 | 리소스 **생성** | `POST /users` |
| **PUT** | 프로필 전체를 새 것으로 교체함 | 리소스 **전체 수정(교체)** | `PUT /users/1` |
| **PATCH** | 프로필에서 이름만 고침 | 리소스 **부분 수정** | `PATCH /users/1` |
| **DELETE** | 회원 탈퇴 처리함 | 리소스 **삭제** | `DELETE /users/1` |

이 표에서 반드시 이해해야 할 두 가지 개념이 있습니다. **안전성(Safe)**과 **멱등성(Idempotent)**입니다. 이 두 단어는 시험에 나오는 용어가 아니라, 실무에서 버그를 막는 실질적인 개념입니다.

**안전성(Safe): "서버 상태를 바꾸지 않는가?"**

GET은 "안전한" 메서드입니다. 몇 번을 호출하든 서버의 데이터가 변하지 않습니다. 도서관에서 책을 100번 꺼내 읽어도 책 내용이 바뀌지 않는 것과 같습니다. 이 성질 덕분에 브라우저는 GET 요청을 마음대로 캐싱하고, 크롤러는 GET 링크를 마음대로 따라갈 수 있습니다. **만약 GET으로 데이터를 삭제하도록 만들었다면?** 검색 엔진 크롤러가 사이트를 돌아다니다가 `GET /users/1/delete` 링크를 무심코 눌러 사용자를 몽땅 삭제해버리는 실제 사고가 과거에 있었습니다. 이것이 "GET은 안전해야 한다"는 약속을 지켜야 하는 이유입니다.

**멱등성(Idempotent): "여러 번 호출해도 결과가 같은가?"**

멱등성은 "같은 요청을 1번 보내든 100번 보내든, 최종 서버 상태가 동일한가"를 묻습니다.

```
멱등성 비교 (같은 요청을 3번 보냈을 때)

GET /users/1        → 조회만. 3번 해도 서버 상태 그대로.        [멱등 O]
PUT /users/1        → "이름=홍길동으로 교체". 3번 해도 결과 동일. [멱등 O]
DELETE /users/1     → "1번 삭제". 3번 해도 결국 '없음' 상태 동일. [멱등 O]

POST /users         → "새 유저 생성". 3번 하면 유저 3명 생성!    [멱등 X]
PATCH /users/1      → 구현에 따라 다름 (조심).                   [상황따라]
```

이 개념이 왜 실무에서 중요할까요? **네트워크는 불안정하기 때문입니다.** 사용자가 결제 버튼을 눌렀는데 응답이 늦어져 다시 눌렀다고 합시다. 만약 결제가 멱등하지 않은 방식(단순 POST 중복 허용)으로 설계되어 있다면 이중 결제가 발생합니다. 그래서 실무에서는 결제처럼 위험한 작업에 **멱등성 키(Idempotency-Key)**를 헤더로 붙여, 같은 키의 요청은 한 번만 처리되도록 만듭니다. PUT과 DELETE가 멱등하도록 설계된 것은 재시도(retry)를 안전하게 만들기 위한 배려입니다.

### 원리: 상태 코드 — "결과를 한 줄로 요약하다"

응답의 첫 줄에는 세 자리 숫자가 붙습니다. 이것이 **상태 코드(Status Code)**입니다. 상태 코드는 응답 본문을 읽지 않고도 "무슨 일이 일어났는가"를 즉시 알려주는 요약본입니다.

```
1xx : 처리 중입니다 (거의 안 씀)
2xx : 성공했습니다              ← "네, 잘 처리됐어요"
3xx : 다른 곳으로 가세요 (리다이렉트)
4xx : 당신(클라이언트)이 잘못했어요  ← "요청이 이상해요"
5xx : 제(서버)가 잘못했어요        ← "저희 쪽 문제예요, 죄송"
```

숫자를 외우기보다 **첫 자리의 의미**를 이해하는 것이 핵심입니다. 4로 시작하면 "요청을 보낸 쪽 잘못", 5로 시작하면 "서버 쪽 잘못"입니다. 이 구분만으로도 장애 발생 시 "우리 서버 문제인가, 클라이언트 문제인가"를 즉시 나눌 수 있습니다.

자주 쓰이는 상태 코드를 사람 말로 옮기면 다음과 같습니다.

| 코드 | 이름 | 사람 말로 하면 | 언제 쓰나 |
|------|------|----------------|-----------|
| **200** OK | 성공 | "네, 여기 있어요" | 조회/수정 성공 |
| **201** Created | 생성됨 | "새로 만들었어요" | POST로 리소스 생성 성공 |
| **204** No Content | 내용 없이 성공 | "처리했는데 돌려드릴 건 없어요" | DELETE 성공, 응답 본문 불필요 시 |
| **301** Moved Permanently | 영구 이동 | "그 주소 이제 여기로 바뀌었어요" | URL 영구 변경 |
| **304** Not Modified | 안 바뀜 | "지난번 거 그대로 쓰세요" | 캐시 유효 시 |
| **400** Bad Request | 잘못된 요청 | "무슨 말인지 모르겠어요" | 요청 형식/값 오류 |
| **401** Unauthorized | 인증 안 됨 | "당신 누구세요? 로그인부터" | 인증 정보 없음/만료 |
| **403** Forbidden | 권한 없음 | "누군진 알겠는데, 그건 안 돼요" | 인증은 됐으나 권한 부족 |
| **404** Not Found | 못 찾음 | "그런 거 여기 없어요" | 리소스 미존재 |
| **405** Method Not Allowed | 허용 안 된 메서드 | "그 방식으론 안 돼요" | GET만 되는데 POST 보냄 |
| **409** Conflict | 충돌 | "이미 있는 거예요 / 상태가 안 맞아요" | 중복 가입, 동시 수정 충돌 |
| **422** Unprocessable Entity | 처리 불가 | "형식은 맞는데 내용이 규칙 위반이에요" | 검증(validation) 실패 |
| **429** Too Many Requests | 요청 과다 | "너무 자주 부르지 마세요" | 요청 제한(rate limit) 초과 |
| **500** Internal Server Error | 서버 오류 | "저희 쪽에서 터졌어요" | 예상 못 한 서버 예외 |
| **502** Bad Gateway | 게이트웨이 오류 | "뒤에 있는 서버가 이상해요" | 프록시가 잘못된 응답 받음 |
| **503** Service Unavailable | 서비스 불가 | "지금 점검/과부하 중이에요" | 서버 일시 중단 |

**왜 상태 코드를 제대로 써야 하는가?** "그냥 다 200 주고 본문에 `{success: false}`라고 쓰면 안 되나요?"라는 질문에 대한 답이 여기 있습니다.

상태 코드는 사람만 읽는 것이 아니라 **수많은 자동화된 시스템이 읽습니다.** 브라우저는 401을 받으면 로그인 페이지로 보낼 수 있고, 3xx를 받으면 자동으로 리다이렉트하며, 2xx만 캐싱합니다. 모니터링 시스템은 5xx의 비율을 세어 장애를 감지합니다. CDN과 프록시는 상태 코드를 보고 응답을 캐싱할지 결정합니다. 로드밸런서는 5xx가 많은 서버를 자동으로 제외합니다. **모든 것을 200으로 주면, 이 자동화된 생태계 전체가 눈이 먼 상태가 됩니다.** 장애가 나도 모니터링은 "정상 200"만 보고 알람을 울리지 않고, 실패한 요청도 캐시에 저장되어 다른 사용자에게 잘못된 결과가 전달됩니다. 상태 코드는 인간을 위한 배려가 아니라 **기계와의 약속**입니다.

## 3.2 REST API — 예측 가능한 웹의 설계 철학

### 개념: REST는 "리소스를 명사로, 행위를 메서드로" 표현하는 규칙이다

3.1에서 배운 HTTP 메서드를 잘 활용하면 아주 우아한 설계 원칙이 나옵니다. 바로 **REST(Representational State Transfer)**입니다. REST의 핵심 규칙은 딱 한 문장으로 요약됩니다.

> **URL에는 "무엇(리소스=명사)"만 담고, "어떻게 할지(행위=동사)"는 HTTP 메서드로 표현하라.**

즉, URL에는 동사가 들어가면 안 됩니다. `getUser`, `createUser`, `deleteUser` 같은 동사는 이미 GET, POST, DELETE라는 메서드로 표현되기 때문입니다.

### 원리: 나쁜 예 vs 좋은 예

말로만 하면 와닿지 않으니 실제 URL 설계를 비교해봅시다.

```
❌ 나쁜 예 (동사가 URL에 섞여 혼란스러움)
──────────────────────────────────────────
GET  /getUser?id=1              ← "get"이 중복 (메서드가 이미 GET)
POST /createUser                ← "create"가 중복
POST /user/delete?id=1          ← 삭제인데 POST? 동사도 URL에
GET  /user/update?id=1&name=김   ← 수정인데 GET? 심지어 안전성 위반!
POST /getUserList               ← 조회인데 POST?
GET  /deleteAllUsers            ← 최악: GET으로 삭제 (크롤러 사고 위험)

  → 규칙이 없다. 개발자마다 제각각. 다음 API가 뭘지 예측 불가능.


✅ 좋은 예 (리소스는 명사, 행위는 메서드)
──────────────────────────────────────────
GET    /users          ← 유저 목록 조회
GET    /users/1        ← 1번 유저 조회
POST   /users          ← 유저 생성
PUT    /users/1        ← 1번 유저 전체 수정
PATCH  /users/1        ← 1번 유저 부분 수정
DELETE /users/1        ← 1번 유저 삭제

  → 규칙이 명확하다. "명사(리소스) + 메서드(행위)"만 알면
    처음 보는 API도 어떻게 동작할지 짐작할 수 있다.
```

계층 관계도 URL로 자연스럽게 표현됩니다.

```
GET    /users/1/orders         ← 1번 유저의 주문 목록
GET    /users/1/orders/99      ← 1번 유저의 99번 주문
POST   /users/1/orders         ← 1번 유저의 새 주문 생성
DELETE /users/1/orders/99      ← 1번 유저의 99번 주문 삭제
```

### 원리: REST가 주는 진짜 가치는 "예측 가능성"이다

REST 규칙을 지키는 것의 핵심 이득은 **예측 가능성(Predictability)**입니다. `/users`라는 API를 본 개발자는 문서를 읽지 않고도 다음을 짐작할 수 있습니다.

- 아마 `/products`, `/orders`도 같은 패턴일 것이다.
- 목록은 `GET /products`, 하나 조회는 `GET /products/{id}`일 것이다.
- 생성은 `POST /products`이고 201을 돌려줄 것이다.
- 삭제는 `DELETE /products/{id}`이고 204를 돌려줄 것이다.

이 예측 가능성은 **인지 부하(cognitive load)를 극적으로 낮춥니다.** 팀에 새 개발자가 들어와도, 다른 회사의 API와 연동해도, 규칙이 일관되기에 학습 곡선이 완만해집니다. 좋은 API 설계는 "외워야 할 것을 줄이는 설계"입니다.

> **참고: HATEOAS — REST의 최종 이상**
>
> REST에는 사실 더 높은 단계가 있습니다. HATEOAS(Hypermedia As The Engine Of Application State)는 응답 안에 "다음에 할 수 있는 행동의 링크"를 함께 담자는 개념입니다. 예를 들어 주문 조회 응답에 `{"status": "PAID", "_links": {"cancel": "/orders/99/cancel", "refund": "/orders/99/refund"}}`처럼 다음 가능한 액션의 URL을 넣어주면, 클라이언트는 URL을 하드코딩하지 않고 서버가 알려준 링크만 따라가면 됩니다. 웹 페이지에서 링크를 클릭하며 돌아다니듯 API도 그렇게 탐색하게 하자는 이상입니다. 실무에서 완전히 구현하는 경우는 드물지만, "응답이 다음 행동을 안내한다"는 아이디어는 알아둘 가치가 있습니다.

### 코드: Spring에서 REST 컨트롤러 작성하기

Spring은 REST 규칙을 코드로 옮기기 매우 쉽게 만들어줍니다.

```java
@RestController                       // JSON 응답을 주는 REST 컨트롤러
@RequestMapping("/users")             // 이 컨트롤러의 공통 경로 (리소스 = users)
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    // GET /users        → 목록 조회
    @GetMapping
    public List<UserSummaryResponse> getUsers() {
        return userService.findAll();
    }

    // GET /users/1      → 단건 조회
    @GetMapping("/{id}")
    public UserDetailResponse getUser(@PathVariable Long id) {
        return userService.findById(id);
    }

    // POST /users       → 생성 (성공 시 201 Created가 어울린다)
    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public UserDetailResponse createUser(@RequestBody CreateUserRequest request) {
        return userService.create(request);
    }

    // PATCH /users/1    → 부분 수정
    @PatchMapping("/{id}")
    public UserDetailResponse updateUser(@PathVariable Long id,
                                         @RequestBody UpdateUserRequest request) {
        return userService.update(id, request);
    }

    // DELETE /users/1   → 삭제 (성공 시 204 No Content가 어울린다)
    @DeleteMapping("/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void deleteUser(@PathVariable Long id) {
        userService.delete(id);
    }
}
```

`@RequestMapping("/users")`로 리소스(명사)를 한 번 지정하고, 각 메서드에 `@GetMapping`, `@PostMapping` 등으로 행위(HTTP 메서드)를 붙였습니다. URL 어디에도 동사가 없다는 점을 눈여겨보세요. 애노테이션 이름 자체가 REST 규칙을 강제하도록 설계되어 있습니다.

## 3.3 Spring MVC의 요청 처리 흐름 — 요청은 어떻게 컨트롤러에 도달하는가

### 개념: 요청 하나가 처리되는 여정

사용자가 `GET /users/1`을 호출했을 때, 그 요청은 서버 안에서 여러 단계를 거쳐 처리되고 응답이 되어 돌아옵니다. Spring MVC는 이 여정을 정교하게 조율합니다. 전체 흐름은 다음과 같습니다.

```
     [클라이언트]
        │  GET /users/1
        ▼
┌──────────────────────┐
│   DispatcherServlet   │  ← 모든 요청의 정문(front controller)
│   (프론트 컨트롤러)     │     "이 요청, 누가 처리하지?"
└──────────┬───────────┘
           │ ① "이 URL 담당자 찾아줘"
           ▼
┌──────────────────────┐
│    HandlerMapping     │  ← URL과 메서드를 보고 담당 컨트롤러 결정
│                       │     "GET /users/1 → UserController.getUser()"
└──────────┬───────────┘
           │ ② 담당자 알려줌
           ▼
┌──────────────────────┐
│      Controller       │  ← 요청을 받고 응답 형태를 결정 (교통 정리)
│   (UserController)    │     "id=1 꺼내서 서비스한테 넘기자"
└──────────┬───────────┘
           │ ③ 비즈니스 처리 위임
           ▼
┌──────────────────────┐
│       Service         │  ← 실제 업무 로직 수행 (핵심 두뇌)
│   (UserService)       │     "규칙 확인하고, 데이터 조합하고..."
└──────────┬───────────┘
           │ ④ 데이터 필요해
           ▼
┌──────────────────────┐
│     Repository        │  ← DB와 대화 (데이터 창고 관리인)
│   (UserRepository)    │     "SELECT * FROM users WHERE id=1"
└──────────┬───────────┘
           │ ⑤ 쿼리 실행
           ▼
┌──────────────────────┐
│      Database         │  ← 실제 데이터 저장소
└──────────────────────┘

  그리고 데이터는 ⑤→④→③ 역순으로 거슬러 올라가고,
  DispatcherServlet이 최종적으로 JSON 응답을 만들어 클라이언트에게 돌려준다.
```

### 원리: 레스토랑 비유로 이해하기

이 흐름은 레스토랑의 운영과 놀랍도록 닮았습니다.

```
   손님(클라이언트)  "스테이크 하나 주세요" (요청)
        │
        ▼
   ┌──────────────────────────────────────────┐
   │  안내 데스크 (DispatcherServlet)             │
   │  "어서오세요. 어떤 걸 원하시죠?              │
   │   → 이 주문은 홀 담당 웨이터에게 배정할게요"   │
   └────────────────┬─────────────────────────┘
                    ▼
   ┌──────────────────────────────────────────┐
   │  웨이터 (Controller)                        │
   │  주문을 받아 적고, 주방에 전달만 한다.        │
   │  (직접 요리하지 않는다!)                     │
   └────────────────┬─────────────────────────┘
                    ▼
   ┌──────────────────────────────────────────┐
   │  주방장 (Service)                           │
   │  실제 요리를 한다. 레시피(비즈니스 규칙)를    │
   │  알고, 재료를 조합해 요리를 완성한다.         │
   └────────────────┬─────────────────────────┘
                    ▼
   ┌──────────────────────────────────────────┐
   │  창고 담당자 (Repository)                    │
   │  냉장고(DB)에서 재료를 꺼내다 준다.           │
   │  요리는 안 한다. 재료 관리만.                 │
   └──────────────────────────────────────────┘

   완성된 요리(응답)는 다시 창고→주방→웨이터를 거쳐 손님에게 전달된다.
```

이 비유의 핵심은 **각자 자기 일만 한다**는 것입니다. 웨이터가 요리를 하지 않고, 주방장이 손님을 안내하지 않으며, 창고 담당자가 레시피를 고민하지 않습니다. 역할이 명확히 나뉘어 있기에 레스토랑은 바쁜 시간에도 무너지지 않고 돌아갑니다. 이 역할 분담이 바로 3.4에서 다룰 **레이어드 아키텍처**입니다.

### 원리: 왜 DispatcherServlet이 필요한가?

"그냥 URL마다 처리 코드를 짜면 되지, 왜 DispatcherServlet이라는 정문을 하나 더 두는가?"라는 의문이 들 수 있습니다.

DispatcherServlet은 **프론트 컨트롤러(Front Controller) 패턴**의 구현입니다. 만약 이런 정문이 없다면, 모든 컨트롤러가 각자 다음과 같은 공통 작업을 **중복으로** 처리해야 합니다.

- 요청 URL을 분석해 어떤 코드를 실행할지 결정
- 요청 본문(JSON)을 자바 객체로 변환(역직렬화)
- 인증/인가 확인
- 예외가 발생하면 적절한 상태 코드로 변환
- 자바 객체를 JSON으로 변환(직렬화)해서 응답
- 로깅, 요청 시간 측정 등 공통 처리

이 모든 것을 컨트롤러마다 반복한다면 코드는 지옥이 됩니다. DispatcherServlet은 **모든 요청이 반드시 거쳐야 하는 단일 입구**를 만들어, 이 공통 작업을 한곳에서 처리합니다. 정문이 하나이기에 그 정문에 보안 검색대(필터), 안내원(인터셉터), 통역사(메시지 컨버터)를 붙이면 모든 요청에 일괄 적용됩니다. **중복을 없애고, 공통 관심사를 한곳에 모으는 것** — 이것이 프론트 컨트롤러를 두는 이유입니다.

정리하면, 개발자는 DispatcherServlet의 존재를 거의 의식하지 않고 컨트롤러만 작성하면 됩니다. 하지만 그 뒤에서 이 "정문"이 요청 분배, 데이터 변환, 예외 처리라는 궂은일을 대신 해주고 있다는 사실을 아는 것과 모르는 것은 큰 차이입니다.

## 3.4 레이어드 아키텍처 — 관심사를 계층으로 나누다

### 개념: 3계층 구조

앞서 레스토랑 비유에서 봤듯, 웹 애플리케이션은 보통 세 개의 계층(Layer)으로 역할을 나눕니다.

```
┌─────────────────────────────────────────────────────────────┐
│                                                               │
│   ① Controller Layer (표현 계층 / Presentation)                │
│   ─────────────────────────────────────────────              │
│   역할     : HTTP 요청을 받고, 응답을 돌려준다 (교통 정리)         │
│   비유     : 웨이터 (주문 받고 전달, 요리는 안 함)               │
│   알아야 할 것 : HTTP, URL, 상태 코드, 요청/응답 형식             │
│   몰라야 할 것 : DB 쿼리, 비즈니스 규칙의 세부 구현               │
│                                                               │
│                        │  DTO 전달                             │
│                        ▼                                       │
│   ② Service Layer (비즈니스 계층 / Business)                    │
│   ─────────────────────────────────────────────              │
│   역할     : 핵심 업무 로직, 규칙 검증, 트랜잭션 관리             │
│   비유     : 주방장 (레시피를 알고 실제로 요리함)                │
│   알아야 할 것 : 도메인 규칙("잔액보다 많이 출금 불가" 등)         │
│   몰라야 할 것 : HTTP의 세부사항, 어떤 DB를 쓰는지의 세부구현      │
│                                                               │
│                        │  엔티티/조건 전달                      │
│                        ▼                                       │
│   ③ Repository Layer (데이터 접근 계층 / Persistence)           │
│   ─────────────────────────────────────────────              │
│   역할     : DB와 대화. 데이터 저장/조회/수정/삭제               │
│   비유     : 창고 담당자 (재료를 꺼내다 줄 뿐)                   │
│   알아야 할 것 : 쿼리, 테이블 구조, DB 접근 방법                 │
│   몰라야 할 것 : 왜 이 데이터가 필요한지(비즈니스 이유), HTTP     │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

### 원리: 왜 계층을 나누는가 — 관심사의 분리(Separation of Concerns)

"계층을 나누면 파일만 많아지는데, 왜 굳이?"라는 의문은 자연스럽습니다. 하지만 계층 분리는 소프트웨어 공학의 가장 오래되고 강력한 원칙인 **관심사의 분리(Separation of Concerns)**의 구현입니다.

계층을 나눌 때의 실질적 이득은 다음과 같습니다.

1. **변경의 영향 범위가 좁아진다.** DB를 MySQL에서 PostgreSQL로 바꿔도, 변경은 Repository 계층에만 머뭅니다. Service와 Controller는 손댈 필요가 없습니다. 반대로 REST API를 GraphQL로 바꿔도 Controller만 바뀌고 핵심 로직(Service)은 그대로입니다. **"바뀌는 이유가 다른 코드는 분리하라"**는 원칙(단일 책임 원칙, SRP)이 여기서 실현됩니다.

2. **테스트가 쉬워진다.** Service를 테스트할 때, 진짜 DB를 띄우지 않고 Repository를 가짜(Mock)로 바꿔치기하면 됩니다. HTTP 서버를 띄우지 않고도 비즈니스 로직만 순수하게 검증할 수 있습니다. 계층이 뭉쳐 있으면 이런 격리 테스트가 불가능합니다.

3. **역할이 명확해 협업이 쉬워진다.** 프론트엔드에 맞춰 응답 형식을 조정하는 사람은 Controller만 보면 되고, 결제 규칙을 다루는 사람은 Service만 보면 됩니다.

핵심 규칙 하나: **의존 방향은 항상 위에서 아래로만 흐릅니다.** Controller → Service → Repository. Repository가 Service를 알아서는 안 되고, Service가 Controller를 알아서는 안 됩니다. 아래 계층은 위 계층의 존재를 몰라야 합니다. 창고 담당자가 "이 재료가 왜 필요한지"(비즈니스)를 몰라도 되는 것처럼요.

### 안티패턴: 웨이터가 요리까지 하는 격 — Controller에 비즈니스 로직

가장 흔한 실수는 **Controller에 비즈니스 로직을 몰아넣는 것**입니다. 이는 웨이터가 주문도 받고, 요리도 하고, 설거지까지 하는 것과 같습니다. 당장은 돌아가지만 곧 재앙이 됩니다.

```java
// ❌ 안티패턴: 컨트롤러가 모든 일을 다 함
@RestController
@RequestMapping("/orders")
public class OrderController {

    private final OrderRepository orderRepository;
    private final ProductRepository productRepository;
    private final UserRepository userRepository;

    @PostMapping
    public Order createOrder(@RequestBody OrderRequest request) {
        // (1) 재고 확인 — 비즈니스 로직!
        Product product = productRepository.findById(request.getProductId()).orElseThrow();
        if (product.getStock() < request.getQuantity()) {
            throw new RuntimeException("재고 부족");
        }

        // (2) 잔액 확인 — 비즈니스 로직!
        User user = userRepository.findById(request.getUserId()).orElseThrow();
        int total = product.getPrice() * request.getQuantity();
        if (user.getBalance() < total) {
            throw new RuntimeException("잔액 부족");
        }

        // (3) 재고 차감, 잔액 차감, 주문 저장 — 전부 비즈니스 로직!
        product.setStock(product.getStock() - request.getQuantity());
        user.setBalance(user.getBalance() - total);
        productRepository.save(product);
        userRepository.save(user);

        Order order = new Order(user, product, request.getQuantity(), total);
        return orderRepository.save(order);
        // 문제: 트랜잭션 관리 없음! (3)에서 재고만 깎이고 서버가 죽으면?
        //       재고는 줄었는데 주문은 없는 데이터 불일치 발생!
    }
}
```

이 코드의 문제는 심각합니다.
- **재사용 불가**: 다른 곳(배치 작업, 관리자 페이지)에서 같은 주문 로직이 필요하면 복사-붙여넣기밖에 없습니다.
- **테스트 불가**: 이 로직을 검증하려면 HTTP 요청을 흉내 내야 합니다.
- **트랜잭션 부재**: 여러 DB 작업이 하나로 묶이지 않아, 중간에 실패하면 데이터가 깨집니다.
- **책임 과다**: 컨트롤러가 HTTP도 알고 비즈니스도 알아, 바뀌는 이유가 여러 개입니다.

### 올바른 구조: 각자 자기 일만 하기

로직을 Service로 옮기고, 트랜잭션 경계를 명확히 합니다.

```java
// ✅ 올바른 구조: 컨트롤러는 교통 정리만
@RestController
@RequestMapping("/orders")
public class OrderController {

    private final OrderService orderService;

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public OrderResponse createOrder(@RequestBody OrderRequest request) {
        // 컨트롤러는 "받아서 넘기고, 결과를 돌려준다"만 한다.
        return orderService.createOrder(request);
    }
}
```

```java
// ✅ 비즈니스 로직은 서비스에 모으고, 트랜잭션으로 묶는다
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final ProductRepository productRepository;
    private final UserRepository userRepository;

    public OrderService(OrderRepository orderRepository,
                        ProductRepository productRepository,
                        UserRepository userRepository) {
        this.orderRepository = orderRepository;
        this.productRepository = productRepository;
        this.userRepository = userRepository;
    }

    @Transactional   // ★ 이 메서드 안의 모든 DB 작업은 '하나'로 묶인다.
                     //   중간에 실패하면 전부 롤백 → 데이터 불일치 방지!
    public OrderResponse createOrder(OrderRequest request) {
        Product product = productRepository.findById(request.getProductId())
                .orElseThrow(() -> new ProductNotFoundException(request.getProductId()));
        User user = userRepository.findById(request.getUserId())
                .orElseThrow(() -> new UserNotFoundException(request.getUserId()));

        // 도메인 규칙(비즈니스 로직)은 서비스가 책임진다
        product.decreaseStock(request.getQuantity());   // 재고 부족 시 내부에서 예외
        int total = product.getPrice() * request.getQuantity();
        user.withdraw(total);                            // 잔액 부족 시 내부에서 예외

        Order order = Order.of(user, product, request.getQuantity(), total);
        Order saved = orderRepository.save(order);

        return OrderResponse.from(saved);   // 엔티티가 아니라 DTO로 변환해 반환
    }
}
```

`@Transactional`이 붙은 메서드는 그 안의 모든 DB 작업을 하나의 트랜잭션으로 묶습니다. 재고 차감 후 주문 저장 직전에 예외가 나면, **재고 차감까지 전부 취소(롤백)**됩니다. "모두 성공하거나, 모두 실패하거나"라는 원자성(Atomicity)이 보장되는 것입니다. 이 트랜잭션 경계는 **Service 계층**에 두는 것이 정석입니다. 하나의 "업무 단위(비즈니스 트랜잭션)"가 바로 Service 메서드에 대응하기 때문입니다.

### 트레이드오프: 계층 분리에도 비용은 있다

계층 분리가 만능은 아닙니다. 정직하게 트레이드오프를 짚자면, 계층을 나누면 **작성해야 할 코드와 파일이 늘어나고**, 데이터가 계층을 오가며 변환되는 **오버헤드**가 생깁니다. 아주 단순한 CRUD 애플리케이션이라면 이 분리가 과하게 느껴질 수 있습니다.

그럼에도 실무에서 계층 분리를 기본으로 삼는 이유는, 소프트웨어가 **거의 항상 예상보다 오래 살고 복잡해지기 때문**입니다. 처음엔 단순했던 주문 로직에 쿠폰, 포인트, 재고 예약, 배송 정책이 붙습니다. 계층이 잘 나뉘어 있으면 이 변화를 각 계층에서 감당할 수 있지만, 뭉쳐 있으면 하나의 거대한 진흙 덩어리(Big Ball of Mud)가 됩니다. "지금의 약간의 번거로움"으로 "미래의 큰 재앙"을 막는 투자라고 이해하면 됩니다.

## 3.5 DTO 패턴 — 왜 엔티티를 직접 노출하면 안 되는가

### 개념: DTO는 계층 사이를 오가는 "전용 운반 객체"다

DTO(Data Transfer Object)는 이름 그대로 **데이터를 전달하기 위한 객체**입니다. Entity(DB 테이블과 매핑되는 객체)를 그대로 클라이언트에게 응답하거나 요청으로 받는 대신, 목적에 맞게 만든 별도의 객체를 사용하는 패턴입니다.

"Entity를 그대로 쓰면 편한데 왜 DTO를 하나 더 만드나요?"라는 질문에는 **세 가지 명확한 이유**가 있습니다.

### 원리: Entity 직접 노출의 세 가지 문제

**문제 1 — 보안(Security): 감춰야 할 정보가 새어 나간다**

User 엔티티에는 비밀번호 해시, 주민번호, 내부 관리용 플래그 등 **절대 외부에 나가면 안 되는 필드**가 있습니다. 엔티티를 그대로 JSON으로 응답하면 이 모든 것이 함께 노출됩니다.

```
User 엔티티를 그대로 응답하면...

{
  "id": 1,
  "name": "홍길동",
  "email": "hong@example.com",
  "password": "$2a$10$abc...",     ← 💀 비밀번호 해시 노출!
  "ssn": "900101-1234567",         ← 💀 주민번호 노출!
  "isAdmin": false,                ← 💀 내부 권한 정보 노출!
  "createdAt": "2024-01-01"
}
```

**문제 2 — 결합(Coupling): DB 구조가 API 스펙에 묶인다**

엔티티를 그대로 응답하면, **DB 테이블 구조가 곧 API 스펙**이 됩니다. 그러면 컬럼명 하나만 바꿔도(예: `name` → `fullName`) 그 API를 쓰는 모든 클라이언트(웹, 앱, 외부 파트너)가 깨집니다. DB는 내부 사정이고 API는 외부 약속인데, 이 둘이 강하게 묶여버리면 어느 한쪽도 자유롭게 바꿀 수 없습니다.

**문제 3 — 유연성(Flexibility): 화면마다 다른 데이터 형태가 필요하다**

목록 화면에서는 이름과 요약만 필요하고, 상세 화면에서는 전체 정보가 필요합니다. 하나의 엔티티로는 이 다양한 요구를 우아하게 표현할 수 없습니다. 목록에도 모든 필드를 실어 보내면 트래픽이 낭비되고, JPA 연관관계까지 딸려 나가면 예상치 못한 추가 쿼리가 발생합니다.

### 원리: 명함 비유

DTO의 존재 이유는 **명함**에 비유하면 명쾌합니다.

> 당신은 집 주소, 주민번호, 통장 잔고, 가족 관계까지 아는 "온전한 나 자신"(Entity)입니다. 하지만 처음 만난 사람에게 이 모든 걸 건네지 않습니다. 상황에 맞게 **명함**(DTO)을 건넵니다. 비즈니스 미팅에서는 회사와 직책이 적힌 명함을, 동호회에서는 별명과 취미가 적힌 명함을 줍니다. 명함에는 상대가 알아야 할 것만, 내가 보여주고 싶은 것만 담깁니다.

Entity가 "온전한 나"라면, DTO는 "상황에 맞게 골라 건네는 명함"입니다. 상대(클라이언트)에게 필요한 정보만, 안전한 정보만, 상황에 맞는 형태로 전달하는 것입니다.

```
┌────────────────────┐         ┌────────────────────┐
│   Entity (User)     │         │  클라이언트 (외부)   │
│  ──────────────     │         │                     │
│  id                 │         │   "이름이랑 이메일만  │
│  name               │  ───▶   │    보여주세요"       │
│  email              │  변환    │                     │
│  password  🔒       │  (DTO)  └────────────────────┘
│  ssn       🔒       │              ▲
│  isAdmin   🔒       │              │
│  createdAt          │      ┌───────┴────────┐
└────────────────────┘      │ UserSummaryResponse │
        │                    │  id, name, email    │  ← 안전한 것만!
        │                    └────────────────────┘
        │
        │  요청도 마찬가지: 클라이언트가 보낸 JSON은
        ▼  CreateUserRequest(DTO)로 받아 검증 후 Entity로 변환
```

### 코드: 목적별 DTO 설계

Java의 `record`는 DTO를 간결하게 표현하는 데 이상적입니다. 불변(immutable)이고, 보일러플레이트가 없기 때문입니다.

```java
// ── 요청 DTO: 클라이언트 → 서버 (생성용) ──
// 클라이언트가 보내야 할 것만 담는다. id나 createdAt은 서버가 정하므로 없다.
public record CreateUserRequest(
        String name,
        String email,
        String password
) {}

// ── 응답 DTO 1: 목록용 (가벼움) ──
// 목록 화면에 필요한 최소 정보만. password, ssn 등은 애초에 없다.
public record UserSummaryResponse(
        Long id,
        String name,
        String email
) {
    // Entity → DTO 변환은 정적 팩토리 메서드로 (변환 책임을 한곳에 모음)
    public static UserSummaryResponse from(User user) {
        return new UserSummaryResponse(user.getId(), user.getName(), user.getEmail());
    }
}

// ── 응답 DTO 2: 상세용 (풍부함) ──
// 상세 화면에 필요한 정보. 그래도 password 같은 민감정보는 절대 담지 않는다.
public record UserDetailResponse(
        Long id,
        String name,
        String email,
        String phoneNumber,
        LocalDateTime createdAt
) {
    public static UserDetailResponse from(User user) {
        return new UserDetailResponse(
                user.getId(),
                user.getName(),
                user.getEmail(),
                user.getPhoneNumber(),
                user.getCreatedAt()
        );
    }
}
```

같은 User라도 목록에서는 `UserSummaryResponse`(가벼움), 상세에서는 `UserDetailResponse`(풍부함)를 씁니다. 어느 쪽에도 `password`나 `ssn`은 **아예 존재하지 않으므로** 실수로라도 노출될 수 없습니다. 이것이 보안을 코드 구조 자체로 보장하는 방식입니다.

서비스는 엔티티를 조회한 뒤 목적에 맞는 DTO로 변환해 반환합니다.

```java
@Service
@Transactional(readOnly = true)   // 조회 전용 → 읽기 최적화
public class UserService {

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    // 목록: 가벼운 DTO로 변환
    public List<UserSummaryResponse> findAll() {
        return userRepository.findAll().stream()
                .map(UserSummaryResponse::from)
                .toList();
    }

    // 상세: 풍부한 DTO로 변환
    public UserDetailResponse findById(Long id) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new UserNotFoundException(id));
        return UserDetailResponse.from(user);
    }
}
```

DTO 변환은 언뜻 "귀찮은 중간 작업"처럼 보이지만, 실제로는 **DB 내부(Entity)와 외부 세계(API)를 분리하는 방화벽**입니다. 이 방화벽 덕분에 DB 구조는 자유롭게 진화하고, API는 안정적으로 유지되며, 민감정보는 원천적으로 차단됩니다.

## 3.6 Validation — 검증은 왜 반드시 서버에 있어야 하는가

### 개념: 검증은 "쓰레기 데이터가 들어오는 것을 막는 문지기"다

Validation(검증)은 들어온 데이터가 규칙에 맞는지 확인하는 일입니다. 이메일 형식이 맞는지, 이름이 비어 있지 않은지, 비밀번호가 최소 길이를 넘는지 등입니다.

여기서 초심자가 가장 많이 하는 오해는 이것입니다. "프론트엔드(브라우저)에서 이미 검증하는데, 서버에서 또 할 필요가 있나요?"

### 원리: 클라이언트 검증만으로는 왜 부족한가 — Postman/curl 우회

답은 명확합니다. **클라이언트 검증은 절대 신뢰할 수 없기 때문입니다.**

브라우저의 자바스크립트 검증은 오직 "정상적인 사용자가 정상적인 브라우저를 쓸 때"만 동작합니다. 하지만 악의적인 사용자나 잘못 만든 프로그램은 브라우저를 거치지 않습니다. Postman, curl, 또는 직접 짠 스크립트로 서버 API를 **직접** 호출할 수 있습니다.

```bash
# 브라우저의 검증을 깡그리 무시하고, curl로 서버에 직접 쓰레기 데이터를 보냄
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "", "email": "이건-이메일이-아님", "password": "1"}'
```

브라우저 화면에서는 "이메일 형식이 올바르지 않습니다"라고 막았더라도, 위 curl 요청은 그 화면을 거치지 않고 서버로 곧장 들어옵니다. **서버에 검증이 없다면, 이 쓰레기 데이터가 그대로 DB에 저장됩니다.**

### 원리: 은행 번호표 vs 창구 직원 비유

이 관계는 은행에 비유하면 완벽하게 이해됩니다.

```
은행에 갔을 때...

  번호표 발급기 (클라이언트 검증)
  ────────────────────────────
  "어떤 업무 보러 오셨나요? 예금? 대출?"
  → 대충 안내해주는 편의 기능. 빠르고 친절함.
  → 하지만 번호표 기계는 당신 신분을 확인하지 않는다.
  → 기계를 안 거치고 곧장 창구로 갈 수도 있다.

  창구 직원 (서버 검증)  ★진짜 문지기★
  ────────────────────────────
  "신분증 주세요. 서명 여기 해주세요. 잔액 확인하겠습니다."
  → 여기서 통과 못 하면 절대 업무 처리 안 됨.
  → 실제 돈이 오가는 최종 관문. 반드시 검증한다.
```

- **클라이언트 검증 = 번호표 기계**: 사용자 편의를 위한 것입니다. 잘못 입력하면 즉시 빨간 글씨로 알려줘서 서버까지 갈 필요 없이 빠르게 고치게 합니다. **UX(사용자 경험)를 위한 배려**입니다.
- **서버 검증 = 창구 직원**: 실제 데이터가 저장되기 전 최종 관문입니다. 여기서 통과 못 하면 절대 처리되지 않습니다. **데이터 무결성과 보안을 위한 필수 방어선**입니다.

결론은 이렇습니다. **클라이언트 검증은 "있으면 좋은 것(편의)"이고, 서버 검증은 "없으면 안 되는 것(필수)"입니다.** 둘 다 하는 것이 정답입니다.

### 코드: Bean Validation과 @Valid

Spring은 표준 스펙인 Bean Validation을 통해 검증을 선언적으로(애노테이션으로) 처리합니다. 요청 DTO의 각 필드에 규칙을 애노테이션으로 붙입니다.

```java
public record CreateUserRequest(

        @NotBlank(message = "이름은 필수입니다")
        @Size(max = 20, message = "이름은 20자 이하여야 합니다")
        String name,

        @NotBlank(message = "이메일은 필수입니다")
        @Email(message = "올바른 이메일 형식이 아닙니다")
        String email,

        @NotBlank(message = "비밀번호는 필수입니다")
        @Size(min = 8, max = 64, message = "비밀번호는 8자 이상 64자 이하여야 합니다")
        String password
) {}
```

자주 쓰는 검증 애노테이션은 다음과 같습니다.

| 애노테이션 | 의미 |
|-----------|------|
| `@NotNull` | null이 아니어야 함 |
| `@NotBlank` | null도, 빈 문자열도, 공백만도 안 됨 (문자열용) |
| `@NotEmpty` | null도 빈 것도 안 됨 (컬렉션/문자열용) |
| `@Size(min, max)` | 길이/크기 범위 |
| `@Email` | 이메일 형식 |
| `@Min` / `@Max` | 숫자 최소/최대 |
| `@Positive` / `@Negative` | 양수/음수 |
| `@Pattern(regexp=...)` | 정규식 일치 |
| `@Past` / `@Future` | 과거/미래 날짜 |

이제 컨트롤러에서 `@Valid`를 붙이면, 컨트롤러 메서드가 실행되기 **전에** 검증이 자동으로 수행됩니다. 규칙 위반 시 예외가 발생하고 메서드 본문은 아예 실행되지 않습니다.

```java
@RestController
@RequestMapping("/users")
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public UserDetailResponse createUser(
            @Valid @RequestBody CreateUserRequest request) {   // ★ @Valid 하나로 검증 자동화
        // 여기 도달했다면 이미 검증을 통과한 것. 안심하고 로직 실행.
        return userService.create(request);
    }
}
```

### 원리: @RestControllerAdvice로 예외를 한곳에서 처리하다

검증에 실패하면 `MethodArgumentNotValidException`이 발생합니다. 이때 별도 처리를 안 하면 Spring 기본 오류 페이지가 튀어나와 클라이언트가 해석하기 어렵습니다. 그래서 **전역 예외 처리기**를 두어, 모든 예외를 한곳에서 일관된 형식으로 변환합니다. 이것이 `@RestControllerAdvice`입니다.

```java
@RestControllerAdvice   // 모든 컨트롤러의 예외를 여기서 가로채 처리
public class GlobalExceptionHandler {

    // 검증 실패 → 400 Bad Request + 어떤 필드가 왜 틀렸는지 안내
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(MethodArgumentNotValidException e) {
        Map<String, String> errors = new HashMap<>();
        for (FieldError error : e.getBindingResult().getFieldErrors()) {
            errors.put(error.getField(), error.getDefaultMessage());
        }
        return new ErrorResponse("VALIDATION_FAILED", "입력값 검증에 실패했습니다", errors);
    }

    // 리소스 없음 → 404 Not Found
    @ExceptionHandler(UserNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(UserNotFoundException e) {
        return new ErrorResponse("USER_NOT_FOUND", e.getMessage(), null);
    }

    // 그 외 예상 못 한 예외 → 500 Internal Server Error
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleUnexpected(Exception e) {
        // 실무에서는 여기서 로그를 반드시 남긴다. 단, 내부 상세는 클라이언트에 노출 X
        return new ErrorResponse("INTERNAL_ERROR", "서버 오류가 발생했습니다", null);
    }
}

// 일관된 오류 응답 형식
public record ErrorResponse(
        String code,           // 기계가 읽을 오류 코드
        String message,        // 사람이 읽을 메시지
        Map<String, String> fieldErrors   // 필드별 상세 (없으면 null)
) {}
```

검증 실패 시 클라이언트는 다음과 같은 깔끔한 응답을 받습니다.

```json
{
  "code": "VALIDATION_FAILED",
  "message": "입력값 검증에 실패했습니다",
  "fieldErrors": {
    "email": "올바른 이메일 형식이 아닙니다",
    "password": "비밀번호는 8자 이상 64자 이하여야 합니다"
  }
}
```

전역 예외 처리기의 이점은 큽니다. **예외 처리 코드가 각 컨트롤러에 흩어지지 않고 한곳에 모입니다.** 상태 코드 매핑 규칙이 일관되고(검증 실패는 항상 400, 리소스 없음은 항상 404), 오류 응답 형식이 통일되어 클라이언트가 예측 가능하게 처리할 수 있습니다. 이것 역시 3.4에서 본 **관심사의 분리**입니다 — "정상 흐름"과 "예외 처리"라는 서로 다른 관심사를 분리한 것입니다.

## 3.7 필터 vs 인터셉터 — 정문 밖의 검색대와 복도의 안내원

### 개념: 컨트롤러 앞에는 두 겹의 관문이 있다

3.3에서 "정문(DispatcherServlet)에 보안 검색대(필터), 안내원(인터셉터)을 붙이면 모든 요청에 일괄 적용된다"고 했습니다. 이 절에서는 그 두 관문을 정면으로 다룹니다. 둘 다 "요청의 앞뒤에 공통 처리를 끼워 넣는" 도구지만, **서 있는 위치가 다릅니다.**

- **필터(Filter)** 는 서블릿 표준(`jakarta.servlet.Filter`) 기술로, **DispatcherServlet 바깥** — 즉 서블릿 컨테이너(톰캣) 레벨에서 동작합니다. 요청이 Spring MVC에 들어가기도 전에 실행됩니다.
- **인터셉터(Interceptor)** 는 Spring MVC(`HandlerInterceptor`) 기술로, **DispatcherServlet 안쪽** — 컨트롤러 호출 앞뒤에서 동작합니다. Spring 내부에 있으므로 "어떤 컨트롤러의 어떤 메서드가 이 요청을 처리하는지"(`HandlerMethod`)까지 알 수 있습니다.

### 원리: 요청 하나가 통과하는 전체 순서

```
클라이언트
   │
   ▼
 Filter ───────────────────────── 서블릿 컨테이너 레벨 (Spring MVC 바깥)
   │                              "일단 건물에 들어오는 모든 사람 검사"
   ▼
 DispatcherServlet ────────────── Spring MVC의 정문
   │
   ▼
 Interceptor.preHandle() ──────── 컨트롤러 실행 전 (false 반환 시 여기서 차단)
   │                              "이 요청은 ○○Controller.○○()가 처리하는군"
   ▼
 Controller ───────────────────── 실제 요청 처리
   │
   ▼
 Interceptor.postHandle() ─────── 컨트롤러 정상 반환 후 (예외 시 건너뜀)
   │
   ▼
 Interceptor.afterCompletion() ── 처리 완전 종료 후 (예외가 나도 항상 호출)
   │
   ▼
 Filter (응답 통과) ────────────── 응답이 나가기 직전 마지막으로 통과
   │
   ▼
클라이언트
```

건물 비유로 보면, **필터는 건물 출입구의 보안 검색대**입니다. 어느 사무실(컨트롤러)에 가는 사람이든 예외 없이 통과해야 하고, 검색대는 방문자가 몇 층 누구를 만나러 가는지까지는 관심이 없습니다. **인터셉터는 사무실 복도의 안내원**입니다. 방문자가 어느 사무실의 누구를 만나러 가는지(HandlerMethod)를 알고, 그 정보에 기반한 세밀한 안내(권한 확인, 기록)를 할 수 있습니다.

### 비교: 언제 Filter를, 언제 Interceptor를 쓰는가

| 구분 | Filter | Interceptor |
|------|--------|-------------|
| 소속 | 서블릿 표준 스펙 | Spring MVC |
| 위치 | DispatcherServlet **앞/뒤** | DispatcherServlet과 컨트롤러 **사이** |
| 핸들러 정보 | 모름 | `HandlerMethod`로 처리 담당 메서드를 앎 |
| `@ControllerAdvice` 예외 처리 | 적용 안 됨 (Spring MVC 바깥) | 적용됨 |
| 콜백 | `doFilter()` 하나 (앞뒤를 감쌈) | `preHandle` / `postHandle` / `afterCompletion` 세 시점 |
| 적합한 용도 | 인증/보안, 인코딩, 요청·응답 본문 변형, 전 요청 로깅 | 인가(권한) 검사, 컨트롤러 단위 측정/로깅, 로그인 사용자 정보 준비 |

선택 기준은 한 줄로 요약됩니다. **Spring MVC와 무관하게 모든 요청에 걸어야 하면 Filter, "누가 처리하는지"라는 Spring 문맥이 필요하면 Interceptor.**

### 원리: Spring Security가 Filter 기반인 이유

다음 장들에서 배울 Spring Security는 인증/인가를 **필터 체인(SecurityFilterChain)** 으로 구현합니다. 위치를 생각하면 필연적인 설계입니다. 인증되지 않은 요청은 애플리케이션 로직 근처에도 가지 못하게 **가장 바깥, 즉 초입에서** 차단해야 안전한데, 그 초입이 바로 서블릿 필터입니다. 필터가 `chain.doFilter()`를 호출하지 않으면 요청은 DispatcherServlet에 도달조차 못 합니다. 또한 필터는 서블릿 표준이므로 Spring MVC를 거치지 않는 요청(정적 리소스 등)까지 빠짐없이 보호할 수 있습니다. 필터를 이해하는 것은 Spring Security의 `SecurityFilterChain`을 이해하는 전제 조건입니다.

#### 확인문제

**1. (개념 서술)** 필터와 인터셉터는 둘 다 "요청 앞뒤에 공통 처리를 끼워 넣는" 도구입니다. 그런데 인터셉터만 "이 요청을 어떤 컨트롤러의 어떤 메서드가 처리하는지"를 알 수 있습니다. 그 이유를 두 도구의 **위치**로 설명해 보세요.

<details>
<summary>정답</summary>

요청을 어떤 컨트롤러 메서드에 배정할지는 DispatcherServlet(정확히는 그 안의 HandlerMapping)이 결정한다. 필터는 DispatcherServlet **바깥**(서블릿 컨테이너 레벨)에서 실행되므로, 필터가 동작하는 시점에는 아직 담당 핸들러가 정해지지 않았다(정해질 곳에 도달하지도 않았다). 반면 인터셉터는 DispatcherServlet **안쪽**에서 핸들러 결정이 끝난 뒤 컨트롤러 호출 직전에 실행되므로, `preHandle`의 handler 파라미터(`HandlerMethod`)로 담당 메서드 정보를 받을 수 있다.
</details>

**2. (예측)** 컨트롤러에서 예외가 발생해 `@RestControllerAdvice`가 처리하지 못하고 그대로 전파되었다고 합시다. 인터셉터의 세 콜백(`preHandle`, `postHandle`, `afterCompletion`) 중 어떤 것이 호출되고 어떤 것이 호출되지 않을까요?

<details>
<summary>정답</summary>

`preHandle`은 컨트롤러 실행 **전**이므로 이미 호출되었고, `postHandle`은 컨트롤러가 **정상 반환한 경우에만** 호출되므로 건너뛴다. `afterCompletion`은 예외 발생 여부와 무관하게 **항상** 호출되며, 발생한 예외가 `ex` 파라미터로 전달된다. 그래서 측정 마무리나 자원 정리는 postHandle이 아니라 afterCompletion에 두어야 한다.
</details>

**3. (설계 판단)** "모든 요청에 대해 인증 토큰을 검사해서, 유효하지 않으면 컨트롤러에 도달하기 전에 401로 차단하고 싶다"는 요구사항이 있습니다. Filter와 Interceptor 중 무엇이 더 적합하며, 왜 그런가요? Spring Security의 설계와 연결해서 답해 보세요.

<details>
<summary>정답</summary>

**Filter**가 적합하다. 인증은 특정 컨트롤러의 문맥이 필요 없는 전면 방어이고, 가능한 한 가장 바깥(초입)에서 차단할수록 안전하기 때문이다. 필터에서 `chain.doFilter()`를 호출하지 않으면 요청은 DispatcherServlet에 도달조차 못 하며, Spring MVC를 거치지 않는 요청(정적 리소스 등)까지 보호할 수 있다. 실제로 Spring Security가 인증/인가를 `SecurityFilterChain`이라는 필터 체인으로 구현한 이유가 바로 이것이다. (반면 "이 컨트롤러 메서드는 관리자만 접근 가능" 같은 핸들러 단위의 세밀한 인가는 인터셉터 또는 Security의 메서드 보안이 어울린다.)
</details>

## ⚠️ 흔한 오해와 함정

**오해 1: "모든 요청을 POST로 하면 편하지 않나?"**
GET/POST 구분은 단순 취향이 아닙니다. GET은 캐싱되고, 북마크되고, 크롤러가 따라가고, 재시도가 안전합니다. 조회를 POST로 하면 이 모든 이점을 잃습니다. 메서드는 "의도의 선언"이며, 브라우저·프록시·모니터링 시스템이 그 의도를 신뢰하고 동작합니다.

**오해 2: "성공/실패 상관없이 다 200을 주고 본문으로 구분하자."**
상태 코드는 인간이 아니라 기계가 먼저 읽습니다. 전부 200을 주면 모니터링은 장애를 못 잡고, 캐시는 실패 응답을 저장하며, 로드밸런서는 죽은 서버를 못 걸러냅니다. 실패는 4xx/5xx로 정직하게 알려야 합니다.

**오해 3: "REST URL에 동사를 넣으면 더 명확하지 않나? `/users/create`처럼."**
행위는 이미 HTTP 메서드가 표현합니다. `POST /users`면 충분합니다. URL에 동사를 넣으면 `POST /users/create`, `POST /users/update`처럼 규칙이 무너지고 예측 가능성이 사라집니다. 단, 순수 CRUD로 표현이 어려운 특수 액션(예: `POST /orders/99/cancel`)은 예외적으로 허용되기도 합니다.

**오해 4: "Entity를 그냥 응답하면 DTO 만드는 수고를 아낄 수 있잖아."**
당장은 그렇습니다. 그러나 비밀번호가 노출되고, DB 컬럼명을 못 바꾸게 되고, JPA 지연 로딩이 응답 직렬화 중에 터지는 문제를 겪게 됩니다. DTO는 수고가 아니라 방화벽입니다.

**오해 5: "프론트에서 검증하니까 서버 검증은 생략해도 돼."**
가장 위험한 오해입니다. 클라이언트 검증은 curl 한 줄로 우회됩니다. 서버 검증이 없으면 쓰레기 데이터와 공격이 그대로 들어옵니다. 서버 검증은 선택이 아니라 필수입니다.

**오해 6: "비즈니스 로직은 어차피 컨트롤러에서 돌아가니 거기 두면 되지."**
당장은 돌아갑니다. 그러나 재사용도, 테스트도, 트랜잭션 관리도 어려워집니다. 웨이터가 요리까지 하면 바쁜 시간에 주방이 마비됩니다. 로직은 Service에, 트랜잭션 경계도 Service에 두세요.

**함정 1: `@NotNull`과 `@NotBlank` 혼동.** 문자열에 `@NotNull`만 걸면 빈 문자열 `""`이나 공백 `"   "`은 통과합니다. 문자열 필수 값에는 `@NotBlank`를 쓰세요.

**함정 2: `@Valid`를 빼먹으면 검증이 아예 안 된다.** DTO에 애노테이션을 아무리 잘 붙여도, 컨트롤러 파라미터에 `@Valid`가 없으면 검증기는 동작하지 않습니다.

**함정 3: `@Transactional`을 컨트롤러에 붙이기.** 트랜잭션 경계는 하나의 업무 단위인 Service 메서드에 두는 것이 정석입니다. 또한 같은 클래스 내부 메서드 호출(self-invocation)에는 `@Transactional`이 프록시 방식 때문에 적용되지 않는 함정도 있습니다(2장 AOP/프록시 참고).

## 연습문제

**문제 1.** 다음 URL 설계 중 REST 원칙에 어긋나는 것을 모두 고르고, 올바르게 고쳐 보세요.
```
(A) GET    /products
(B) POST   /products/create
(C) GET    /getProduct?id=5
(D) DELETE /products/5
(E) GET    /products/5/delete
```
<details><summary>힌트</summary>URL에 동사가 있는지, 그리고 메서드가 안전성/의도에 맞는지 확인하세요. 특히 (E)는 안전성 위반입니다.</details>

<details><summary>해설</summary>
어긋나는 것: (B), (C), (E).
- (B) `POST /products/create` → 동사 중복. `POST /products`로 수정.
- (C) `GET /getProduct?id=5` → 동사 중복 + REST식 경로 미사용. `GET /products/5`로 수정.
- (E) `GET /products/5/delete` → 동사 포함 + 심각한 안전성 위반(GET이 삭제 수행). `DELETE /products/5`로 수정.
정상: (A) 목록 조회, (D) 삭제.
</details>

**문제 2.** 클라이언트가 `POST /users`로 회원가입을 성공했습니다. 이때 서버가 돌려주기에 가장 적절한 상태 코드는 무엇이며, 그 이유는 무엇인가요? 또한 존재하지 않는 유저를 `GET /users/999`로 조회했을 때는 어떤 코드가 적절한가요?
<details><summary>힌트</summary>"새로 만들어졌다"와 "그런 거 없다"를 사람 말로 옮겨 보세요.</details>

<details><summary>해설</summary>
회원가입 성공: **201 Created**. 새 리소스가 생성되었음을 의미하며, 단순 성공(200)보다 정확한 의도를 전달합니다(관례상 생성된 리소스의 위치를 `Location` 헤더에 담기도 합니다).
없는 유저 조회: **404 Not Found**. "요청한 리소스가 존재하지 않는다"는 의미입니다. 200을 주고 본문에 null을 담으면 캐싱·모니터링이 이를 정상으로 오인합니다.
</details>

**문제 3.** 아래 코드에서 잘못된 점 세 가지를 찾고, 어떻게 고쳐야 하는지 설명하세요.
```java
@RestController
@RequestMapping("/users")
public class UserController {
    private final UserRepository userRepository;

    @PostMapping("/createUser")
    public User create(@RequestBody User user) {
        if (user.getBalance() < 0) throw new RuntimeException("잔액 오류");
        return userRepository.save(user);
    }
}
```
<details><summary>힌트</summary>URL 설계, 계층 분리, Entity 노출, 검증 관점에서 각각 살펴보세요.</details>

<details><summary>해설</summary>
1. **URL에 동사(`/createUser`)** → `@PostMapping`만 두어 `POST /users`가 되게 한다.
2. **컨트롤러에 비즈니스 로직(잔액 검사)** → Service로 옮기고, 트랜잭션이 필요하면 `@Transactional`을 Service에 둔다. Repository도 Controller가 직접 의존하지 말고 Service를 거친다.
3. **Entity를 직접 요청/응답으로 사용** → 요청은 `CreateUserRequest` DTO로 받고 `@Valid`로 검증하며, 응답은 `UserDetailResponse` DTO로 변환해 민감정보 노출을 막는다.
</details>

**문제 4.** "우리 서비스는 프론트엔드에서 모든 입력을 철저히 검증하므로 서버 검증은 생략해도 안전하다"는 주장에 반박해 보세요. 구체적인 공격/우회 시나리오를 하나 들어 설명하세요.
<details><summary>힌트</summary>브라우저를 거치지 않고 API를 호출하는 방법을 떠올려 보세요.</details>

<details><summary>해설</summary>
클라이언트 검증은 브라우저 자바스크립트에서만 동작하며, 공격자는 이를 완전히 우회할 수 있습니다. 예: `curl -X POST /users -d '{"email":"not-an-email","password":"1"}'`처럼 API를 직접 호출하면 프론트 검증을 전혀 거치지 않습니다. 서버 검증이 없으면 잘못된 형식의 이메일, 8자 미만 비밀번호, 음수 잔액 등이 그대로 DB에 저장되어 데이터 무결성이 깨지고 보안 취약점이 됩니다. 서버 검증은 "최종 관문(창구 직원)"으로 반드시 존재해야 하며, 클라이언트 검증은 UX를 위한 보조 수단일 뿐입니다.
</details>

**문제 5.** 하나의 Service 메서드 안에서 "재고 차감 → 주문 저장" 두 개의 DB 작업이 순서대로 일어납니다. 재고 차감은 성공했지만 주문 저장 직전에 서버 오류가 발생했다면, `@Transactional`이 있을 때와 없을 때 데이터는 각각 어떻게 되나요?
<details><summary>힌트</summary>원자성(Atomicity)과 롤백을 떠올리세요.</details>

<details><summary>해설</summary>
- **`@Transactional`이 있을 때**: 두 작업이 하나의 트랜잭션으로 묶입니다. 주문 저장 직전 예외가 나면 재고 차감까지 **롤백**되어, 재고와 주문 모두 원래 상태로 돌아갑니다. 데이터 정합성이 유지됩니다("모두 성공 또는 모두 실패").
- **`@Transactional`이 없을 때**: 재고 차감은 이미 커밋되어 반영되고, 주문 저장은 실패합니다. 결과적으로 **재고는 줄었는데 주문은 없는** 데이터 불일치가 발생합니다. 이런 이유로 여러 DB 작업이 하나의 업무 단위라면 반드시 트랜잭션으로 묶어야 합니다.
</details>

## 요약

- **HTTP는 웹의 도로교통법이다.** 요청/응답이라는 단순한 편지 구조 위에 모든 웹이 세워져 있으며, 무상태(Stateless)라는 성질 덕분에 확장이 가능하다.
- **HTTP 메서드는 "의도"를 선언한다.** GET(조회, 안전), POST(생성), PUT(전체 수정, 멱등), PATCH(부분 수정), DELETE(삭제, 멱등). 안전성과 멱등성은 재시도·캐싱·크롤링의 안전을 보장하는 실무적 개념이다.
- **상태 코드는 기계와의 약속이다.** 2xx(성공)/4xx(클라이언트 잘못)/5xx(서버 잘못)를 정직하게 써야 모니터링·캐싱·로드밸런싱이 제대로 동작한다.
- **REST는 "리소스=명사, 행위=메서드"로 예측 가능성을 만든다.** URL에서 동사를 없애면 처음 보는 API도 짐작할 수 있어 인지 부하가 줄어든다.
- **Spring MVC는 DispatcherServlet(정문)을 중심으로 요청을 처리한다.** HandlerMapping이 담당 컨트롤러를 찾고, Controller→Service→Repository로 흐른다. 프론트 컨트롤러는 공통 관심사를 한곳에 모은다.
- **레이어드 아키텍처는 관심사의 분리다.** Controller(교통 정리)/Service(비즈니스+트랜잭션)/Repository(DB)로 역할을 나눠, 변경 영향 범위를 좁히고 테스트를 쉽게 한다. 비즈니스 로직은 Service에, 트랜잭션 경계도 Service에.
- **DTO는 Entity 직접 노출의 세 가지 문제(보안·결합·유연성)를 해결하는 방화벽이다.** 명함처럼 상황에 맞게 필요한 정보만 담아 전달한다.
- **검증은 반드시 서버에 있어야 한다.** 클라이언트 검증(번호표 기계)은 UX를 위한 편의일 뿐이고, 서버 검증(창구 직원)이 curl 우회를 막는 필수 방어선이다. `@Valid`+Bean Validation으로 선언적으로 처리하고, `@RestControllerAdvice`로 예외를 한곳에서 일관되게 변환한다.
- **필터는 DispatcherServlet 밖(서블릿 컨테이너 레벨), 인터셉터는 안(Spring MVC 레벨)이다.** 요청은 Filter → DispatcherServlet → preHandle → Controller → postHandle → afterCompletion → Filter 순서로 흐른다. 전면 방어(인증·인코딩)는 Filter에, 핸들러 문맥이 필요한 처리(인가·측정)는 Interceptor에 둔다. Spring Security가 Filter 체인인 이유도 "가장 바깥에서 차단"하기 위해서다. (실습: chapter05-spring-web)
