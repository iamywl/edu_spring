# Chapter 7: Docker - 왜 컨테이너인가?

> "내 컴퓨터에서는 되는데요?" — 이 한 문장 안에, Docker가 태어난 모든 이유가 들어 있다.

> **🐳 실습 환경 — 이 장의 실습 재료는 이 저장소의 compose 파일들 자체다**
> ```bash
> cd spring/docker && docker compose -f docker-compose-infra.yml up -d   # 공통 인프라: spring-postgres, spring-redis, spring-adminer(8081)
> cd spring/chapter06-spring-data-jpa && docker compose up --build       # 챕터별 예: spring-ch06-jpa + spring-ch06-postgres
> ```
> 멀티 컨테이너 예제는 chapter09(`spring-ch09-board` + `spring-ch09-postgres`/`-redis`/`-adminer`)가 가장 완전하다. 컨테이너 상태 확인: `docker ps`

세상의 거의 모든 개발자가 한 번쯤 해 본 말이 있다. **"내 컴퓨터에서는 되는데요(It works on my machine)."** 내 노트북에서는 완벽하게 돌아가던 코드가, 동료의 맥에서는 실행조차 안 되고, 테스트 서버에 올리면 알 수 없는 에러를 뱉으며, 정작 운영 서버에서는 새벽 3시에 죽어버린다. 코드는 한 글자도 바뀌지 않았는데 말이다.

이 문장은 농담처럼 쓰이지만, 그 뒤에는 소프트웨어 개발의 가장 뿌리 깊은 고통이 숨어 있다. 바로 **"환경(environment)"** 이라는 문제다. 프로그램은 진공 속에서 돌지 않는다. 운영체제, 언어 런타임, 라이브러리 버전, 환경 변수, 시스템 설정 — 이 수많은 전제 조건이 완벽하게 맞아떨어져야만 비로소 "실행"된다. 그런데 이 전제 조건들은 사람마다, 서버마다 조금씩 다르다. 그 미세한 차이가 쌓여 "내 컴퓨터에서는 되는데"라는 재앙을 만든다.

이 장에서는 Docker의 명령어를 외우는 것이 목적이 아니다. **왜 우리가 프로그램을 "컨테이너"라는 상자에 담기 시작했는가**, 그 근본적인 동기를 이해하는 것이 목표다. 그 동기를 이해하고 나면, Dockerfile의 각 줄이, docker-compose.yml의 각 항목이, 그리고 멀티스테이지 빌드라는 기법이 왜 그런 모양을 하고 있는지 자연스럽게 보이기 시작한다. 이 장이 끝날 때쯤이면, 여러분은 "내 컴퓨터에서는 되는데"라는 말을 두 번 다시 하지 않게 될 것이다.

## 7.1 Docker 이전의 문제 — "내 컴퓨터에서는 되는데"

### 개념: 코드는 같은데 환경이 다르다

소프트웨어를 배포한다는 것은, 내가 만든 프로그램을 **다른 컴퓨터로 옮겨 그곳에서도 똑같이 동작하게 만드는 일**이다. 그런데 이 "다른 컴퓨터"는 내 컴퓨터와 결코 같지 않다. 운영체제가 다르고, 설치된 프로그램의 버전이 다르며, 심지어 파일 경로 구분자(`/` vs `\`)조차 다르다.

문제는 프로그램이 이 모든 환경적 전제에 은근히 의존하고 있다는 점이다. 개발자는 자기 컴퓨터에 이미 필요한 모든 것이 깔려 있기 때문에 그 의존성을 눈치채지 못한다. 그러다 코드를 다른 곳으로 옮기는 순간, 숨어 있던 전제들이 하나씩 무너지며 정체불명의 에러가 쏟아진다.

### 원리: 환경 차이는 어디서 오는가

구체적으로 어떤 것들이 다를 수 있는지 살펴보자. 흔한 예로 세 명의 개발자와 두 개의 서버가 하나의 Spring Boot 프로젝트를 다룬다고 하자.

```
   같은 코드, 다른 환경 → 다른 결과
   ┌──────────────┬───────────┬──────────┬──────────────┐
   │              │    OS     │  Java    │ PostgreSQL   │
   ├──────────────┼───────────┼──────────┼──────────────┤
   │ 개발자 A     │  macOS    │  Java 21 │    16.2      │  ✅ 잘 됨
   │ 개발자 B     │  Ubuntu   │  Java 17 │    14.5      │  ❌ 컴파일 실패
   │ 개발자 C     │  Windows  │  Java 21 │    (없음)    │  ❌ DB 연결 실패
   │ 테스트 서버  │  Ubuntu   │  Java 17 │    16.0      │  ⚠️ 가끔 죽음
   │ 운영 서버    │  CentOS   │  Java 21 │    14.9      │  ❌ 문법 에러
   └──────────────┴───────────┴──────────┴──────────────┘
              ↑ 이 미세한 차이들이 "내 컴퓨터에서는 되는데"를 만든다
```

- **운영체제 차이**: 개발자 A는 macOS, B는 Ubuntu, C는 Windows를 쓴다. 파일 경로, 줄바꿈 문자, 시스템 명령어가 전부 다르다.
- **런타임 버전 차이**: A와 C는 Java 21인데 B와 테스트 서버는 Java 17이다. Java 21에서 쓴 새 문법(예: 최신 `record`, 패턴 매칭)이 Java 17에서는 컴파일조차 안 된다.
- **데이터베이스 버전 차이**: PostgreSQL 16에서 쓴 SQL 함수가 14에서는 존재하지 않아 런타임에 문법 에러가 터진다. C의 컴퓨터엔 아예 DB가 깔려 있지도 않다.

이 문제를 해결하려는 전통적인 방법은 **"설치 문서(README)"** 였다. "Java 21을 설치하세요, PostgreSQL 16을 설치하세요, 이 환경 변수를 세팅하세요…" 하지만 문서는 낡고, 사람은 실수하며, 무엇보다 문서에 적히지 않은 "숨은 전제"는 절대 재현되지 않는다.

### 비유: 이사 짐을 하나하나 옮기기 vs 컨테이너째 옮기기

이 상황을 이사에 비유해 보자.

전통적인 배포는 **이삿짐을 하나하나 옮기는 것**과 같다. 새 집(서버)에 도착해서, 침대를 조립하고, 냉장고 콘센트를 찾아 꽂고, 그릇을 찬장에 다시 넣는다. 그런데 새 집은 콘센트 규격이 다르고(OS 차이), 찬장 크기가 안 맞고(버전 차이), 어떤 방은 아예 없다(의존성 누락). 짐을 풀 때마다 "어? 여기선 안 맞네"를 반복한다.

Docker는 **집 전체를 컨테이너째 통째로 옮기는 것**이다. 가구, 콘센트, 배관, 심지어 벽까지 하나의 규격화된 상자(컨테이너)에 담아 그대로 배에 싣는다. 새 항구에 도착하면 상자를 내려놓기만 하면 된다. 상자 **안의 세계는 어디로 가든 완벽하게 동일**하기 때문에, "여기선 안 맞네"가 일어날 수 없다.

```
   전통 배포 (짐을 하나하나)         Docker (컨테이너째)
   ┌───────────────────┐            ┌───────────────────┐
   │ 서버마다 다시 설치 │            │  ┌─────────────┐  │
   │  - Java 설치       │            │  │  앱 + Java  │  │
   │  - DB 설치         │   ───▶     │  │  + 라이브러리│  │  통째로
   │  - 환경변수 세팅   │            │  │  + 설정      │  │  이동
   │  - "어? 버전이..." │            │  └─────────────┘  │
   └───────────────────┘            └───────────────────┘
   매번 조립 → 매번 다른 결과       어디서 열든 → 똑같은 결과
```

여기서 실제 해상 운송의 역사가 놀랍도록 좋은 비유가 된다. 20세기 중반까지 화물은 크기와 모양이 제각각이라 배에 싣고 내리는 데 엄청난 시간과 비용이 들었다. 그러다 **표준 규격의 컨테이너**가 등장하자, 화물의 내용물이 무엇이든 상관없이 크레인 하나로 척척 옮길 수 있게 되었고, 물류 비용은 극적으로 떨어졌다. Docker의 이름과 로고(고래 위의 컨테이너)가 여기서 왔다. **"내용물이 무엇이든 상관없이 똑같은 방식으로 다룰 수 있는 표준 상자"** — 이것이 컨테이너의 본질이다.

## 7.2 Docker 핵심 개념 — Dockerfile, Image, Container

### 개념: 세 단어만 알면 절반은 이해한 것

Docker의 세계는 복잡해 보이지만, 실은 **세 개의 핵심 명사와 두 개의 핵심 동사**로 이루어져 있다. 이 다섯 단어의 관계만 이해하면 Docker의 절반은 이해한 셈이다.

- **Dockerfile**: 이미지를 어떻게 만들지 적어 둔 **설명서(레시피)**. 텍스트 파일이다.
- **Image(이미지)**: Dockerfile로 만들어 낸 **읽기 전용 설계도**. 앱과 그 실행에 필요한 모든 것이 담긴 얼어붙은 스냅샷이다.
- **Container(컨테이너)**: 이미지를 실행해서 생겨난 **살아 움직이는 실체**. 실제로 프로세스가 돌아가는 격리된 공간이다.

그리고 이들을 잇는 두 동사가 **build(빌드)** 와 **run(실행)** 이다.

### 원리: build와 run — 정지된 설계도와 살아있는 실체

이 관계를 흐름으로 그리면 이렇다.

```
   Dockerfile  ──build──▶  Image  ──run──▶  Container
   (레시피)                (설계도)          (실체)
   텍스트 한 장            읽기 전용          실행 중인 프로세스

   ┌──────────┐           ┌──────────┐      ┌──────────┐
   │ FROM ...  │           │ ▓▓▓▓▓▓▓▓ │      │ ● 실행중 │
   │ COPY ...  │  build   │ ▓ 앱+환경│ run  │ ● 실행중 │
   │ RUN  ...  │ ───────▶ │ ▓ 얼어붙음│────▶│ ● 실행중 │  (여러 개
   │ ENTRYPOINT│           │ ▓▓▓▓▓▓▓▓ │      │ ● 실행중 │   가능)
   └──────────┘           └──────────┘      └──────────┘
    "어떻게 만들지"        "다 만들어진 것"   "실제로 돌아가는 것"
```

여기서 중요한 것은 **이미지는 읽기 전용(immutable)이고, 컨테이너는 그 이미지로부터 몇 개든 찍어낼 수 있다**는 점이다. 하나의 이미지로부터 컨테이너 100개를 띄워도, 모두 완전히 동일한 초기 상태에서 출발한다. 이 "동일함의 보장"이야말로 Docker가 "내 컴퓨터에서는 되는데" 문제를 뿌리 뽑는 핵심이다.

### 비유: 붕어빵 — 레시피, 틀, 붕어빵

이 세 단어의 관계는 **붕어빵**으로 완벽하게 설명된다.

```
   레시피        ──▶   붕어빵 틀      ──▶    붕어빵들
   (Dockerfile)        (Image)              (Container)

   "반죽 넣고           한 번 만들어         찍어낼 때마다
    팥 넣고             두면 계속            똑같은 붕어빵이
    2분 굽기"           재사용              무한정 나옴
```

- **레시피(Dockerfile)**: "이렇게 만들어라"라고 적힌 글이다. 그 자체로는 먹을 수 없다.
- **붕어빵 틀(Image)**: 레시피대로 한 번 만들어 놓은 틀. 단단하게 고정되어 있고(읽기 전용), 이 틀로 붕어빵을 몇 개든 찍어낼 수 있다.
- **붕어빵(Container)**: 틀로 실제로 찍어낸 결과물. 하나를 먹어치워도(컨테이너를 지워도) 틀만 있으면 언제든 똑같은 붕어빵을 또 찍어낼 수 있다.

이 비유가 알려주는 통찰이 하나 있다. **컨테이너는 언제든 버려도 되는(disposable) 존재**라는 것이다. 붕어빵 하나가 망가졌다고 걱정하지 않는다. 틀이 있으니까. 마찬가지로 컨테이너가 죽으면 그냥 이미지로부터 새로 띄우면 된다. 이 "가축처럼 다루기(cattle, not pets)"의 사고방식이 현대 인프라의 근간이다.

### 코드: Dockerfile 한 장 읽어보기

이제 실제 Spring Boot 애플리케이션을 위한 Dockerfile을 한 줄씩 읽어보자. 아래는 **멀티스테이지 빌드**(7.4에서 자세히)를 사용한 예시다.

```dockerfile
# ── 1단계: 빌드 스테이지 ──────────────────────────
# JDK가 포함된 이미지를 기반(base)으로 삼는다. 'builder'라는 별명을 붙인다.
FROM eclipse-temurin:21-jdk AS builder

# 컨테이너 안의 작업 디렉터리를 정한다. 이후 명령은 여기서 실행된다.
WORKDIR /app

# 소스 코드와 빌드 스크립트를 컨테이너 안으로 복사한다.
COPY . .

# Gradle로 실행 가능한 JAR를 빌드한다. (여기서 JDK가 필요하다)
RUN ./gradlew bootJar --no-daemon

# ── 2단계: 실행 스테이지 ──────────────────────────
# 이번엔 JDK가 아니라 훨씬 가벼운 JRE 이미지를 기반으로 삼는다.
FROM eclipse-temurin:21-jre

WORKDIR /app

# 1단계(builder)에서 만든 JAR '결과물만' 가져온다. 소스코드는 안 가져온다.
COPY --from=builder /app/build/libs/*.jar app.jar

# 컨테이너가 시작될 때 실행할 명령.
ENTRYPOINT ["java", "-jar", "app.jar"]
```

각 줄의 의미를 정리하면 이렇다.

- **`FROM`**: 어떤 이미지를 토대로 시작할지 정한다. 맨바닥에서 OS부터 만들 필요 없이, 이미 Java가 깔린 이미지 위에서 출발한다.
- **`WORKDIR`**: 컨테이너 내부의 "현재 작업 폴더"를 지정한다. 이후의 명령들이 이 폴더 기준으로 동작한다.
- **`COPY`**: 호스트(내 컴퓨터)의 파일을 컨테이너 안으로 복사한다. `--from=builder`를 붙이면 다른 스테이지에서 복사한다.
- **`RUN`**: 이미지를 **만드는 중에** 실행할 명령이다. 여기서는 JAR를 빌드한다.
- **`ENTRYPOINT`**: 이미지로부터 컨테이너를 **실행할 때** 돌아갈 명령이다. `RUN`(빌드 시점)과 `ENTRYPOINT`(실행 시점)의 시점 차이를 반드시 구분해야 한다.

`RUN`과 `ENTRYPOINT`의 차이가 헷갈리기 쉽다. `RUN`은 **붕어빵 틀을 만드는 동안** 하는 일(반죽 배합)이고, `ENTRYPOINT`는 **붕어빵을 찍어낼 때** 하는 일(굽기 시작)이라고 생각하면 된다.

## 7.3 Docker Compose — 여러 컨테이너를 지휘하다

### 개념: 앱 하나로는 세상이 안 돌아간다

지금까지는 컨테이너 하나만 다뤘다. 하지만 현실의 애플리케이션은 홀로 존재하지 않는다. 웹 애플리케이션 하나가 살아 움직이려면 보통 이런 동료들이 필요하다.

```
   현실의 서비스 구성 (게시판 예시)
   ┌────────────────────────────────────────────────┐
   │                                                  │
   │   [Nginx]  ──▶  [Spring Boot]  ──▶  [PostgreSQL] │
   │   리버스        애플리케이션         데이터베이스 │
   │   프록시            │                             │
   │                     ▼                             │
   │                  [Redis]                          │
   │                  캐시/세션                        │
   │                                                  │
   └────────────────────────────────────────────────┘
     4개의 컨테이너가 서로 협력해야 서비스가 돌아간다
```

Nginx는 요청을 받아 넘기고, Spring Boot는 로직을 처리하고, PostgreSQL은 데이터를 저장하고, Redis는 캐시와 세션을 담당한다. 이 네 개를 각각 `docker run` 명령으로 따로따로 띄우고, 네트워크로 연결하고, 실행 순서를 맞추는 일을 손으로 한다면? 명령어 네 줄이 각각 수십 개의 옵션으로 길어지고, 한 번이라도 순서나 옵션을 틀리면 전체가 무너진다.

### 원리: 왜 Compose가 필요한가 — 오케스트라의 지휘자

**Docker Compose**는 이 여러 컨테이너를 **하나의 설정 파일(`docker-compose.yml`)로 한꺼번에 정의하고 지휘**하는 도구다. 오케스트라에 비유하면, 각 컨테이너는 연주자이고 Compose는 지휘자다. 지휘자는 누가 언제 들어오고(실행 순서), 누가 누구와 화음을 맞추는지(네트워크)를 총괄한다.

`docker-compose.yml`의 구조를 보자.

```yaml
services:
  # ── 애플리케이션 서비스 ──
  app:
    build: .                     # 현재 폴더의 Dockerfile로 이미지를 빌드
    ports:
      - "8080:8080"              # 호스트 8080 ─▶ 컨테이너 8080 연결
    environment:                 # 컨테이너에 전달할 환경변수
      SPRING_DATASOURCE_URL: jdbc:postgresql://db:5432/board
      SPRING_REDIS_HOST: redis
    depends_on:                  # 이 서비스들이 '건강해진 뒤에' 시작
      db:
        condition: service_healthy
      redis:
        condition: service_started

  # ── 데이터베이스 서비스 ──
  db:
    image: postgres:16           # 버전을 명확히 고정한다 (latest 금지)
    environment:
      POSTGRES_DB: board
      POSTGRES_PASSWORD: secret
    volumes:
      - db-data:/var/lib/postgresql/data   # 데이터를 볼륨에 영속 저장
    healthcheck:                 # DB가 '진짜로' 준비됐는지 확인
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5

  # ── 캐시 서비스 ──
  redis:
    image: redis:7

# 이름 붙인 볼륨 선언 (컨테이너가 죽어도 데이터는 여기 남는다)
volumes:
  db-data:
```

이 파일 하나만 있으면, 단 한 줄의 명령으로 네 개의 컨테이너를 올바른 순서로 띄울 수 있다.

```bash
# 모든 서비스를 정의된 순서와 관계에 맞춰 한 번에 실행
docker compose up

# 백그라운드로 실행하려면 -d
docker compose up -d

# 전부 정리
docker compose down
```

### 원리: 서비스 간 통신 — 서비스 이름이 곧 호스트명이다

`docker-compose.yml`에서 가장 마법처럼 보이는 부분이 이것이다. app 서비스의 환경변수를 보면 데이터베이스 주소가 `jdbc:postgresql://db:5432/...`로 되어 있다. 여기서 `db`는 IP 주소가 아니라 **서비스 이름**이다.

```
   Compose 내부 네트워크에서의 통신
   ┌─────────────────────────────────────────────┐
   │   app 컨테이너                                │
   │     │  "db 서버 어디 있어?"                   │
   │     ▼                                         │
   │   [Compose 내장 DNS] ─── "db" → 172.x.x.x     │
   │     │                                         │
   │     ▼                                         │
   │   db 컨테이너 (5432 포트)                      │
   └─────────────────────────────────────────────┘
     서비스 이름 = 호스트명. IP를 몰라도 이름으로 부른다.
```

Compose는 같은 파일 안의 서비스들을 하나의 **가상 네트워크**로 묶고, **내장 DNS**를 제공한다. 그래서 컨테이너들은 서로의 IP 주소를 몰라도 **서비스 이름만으로** 통신할 수 있다. app 컨테이너가 `db`라는 이름을 찾으면, Compose의 DNS가 알아서 db 컨테이너의 실제 IP로 연결해 준다. IP는 컨테이너가 재시작될 때마다 바뀔 수 있지만 이름은 그대로이므로, 이름 기반 통신은 훨씬 안정적이다.

### 원리: depends_on과 healthcheck — 순서만으로는 부족하다

여러 컨테이너를 띄울 때 가장 흔히 겪는 함정이 **실행 순서 문제**다. Spring Boot 앱이 시작되면서 데이터베이스에 연결하려 하는데, 아직 DB가 완전히 준비되지 않았다면? 앱은 "연결 거부(Connection refused)" 에러를 내며 죽는다.

`depends_on`은 "db를 먼저 띄우고 나서 app을 띄워라"라는 순서를 지정한다. 하지만 여기에 미묘한 함정이 있다. **컨테이너가 "시작"된 것과, 그 안의 프로그램이 "요청을 받을 준비가 된" 것은 다르다.** PostgreSQL 컨테이너가 켜졌다고 해서 DB 엔진이 곧바로 접속을 받을 수 있는 상태는 아니다. 초기화에 몇 초가 걸릴 수 있다.

```
   ❌ depends_on만 (컨테이너 '시작'만 보장)
   db 컨테이너 켜짐 ──▶ app 시작 ──▶ DB에 연결 시도
                                        ↑ DB가 아직 초기화 중! → 연결 실패

   ✅ depends_on + healthcheck (진짜 '준비'를 보장)
   db 컨테이너 켜짐 ──▶ pg_isready 반복 확인 ──▶ 건강함 확인 ──▶ app 시작
                                                                    ↑ 안전하게 연결
```

그래서 `healthcheck`가 필요하다. 위 예시의 `pg_isready -U postgres`는 PostgreSQL이 **실제로 접속을 받을 수 있는지** 주기적으로 확인하는 명령이다. 그리고 app의 `depends_on`에서 `condition: service_healthy`를 지정하면, "db가 그냥 켜진 것"이 아니라 "db가 건강하다고 확인된 다음"에 app을 시작한다. 이 조합이 있어야 비로소 "연결 거부" 재앙을 막을 수 있다.

정리하면, `depends_on`은 순서를, `healthcheck`는 준비 상태를 책임진다. **둘은 함께 쓰여야 의미가 있다.**

## 7.4 멀티스테이지 빌드 — 왜 빌드와 실행을 나누는가

### 개념: 요리하는 주방과 손님상은 다르다

7.2에서 본 Dockerfile에는 `FROM`이 두 번 등장했다. 이것이 **멀티스테이지 빌드(multi-stage build)** 다. 하나의 Dockerfile 안에 여러 단계를 두고, 앞 단계의 결과물만 뒤 단계로 넘기는 기법이다. 왜 이렇게 번거롭게 나누는가? 답은 간단하다. **앱을 "만드는 데 필요한 것"과 "실행하는 데 필요한 것"이 다르기 때문이다.**

### 원리: JDK는 빌드할 때만, JRE는 실행할 때만

Java 애플리케이션을 예로 들자.

- **빌드(컴파일)** 하려면 **JDK(Java Development Kit)** 가 필요하다. 컴파일러, 빌드 도구, 각종 개발 도구가 들어 있어 덩치가 크다.
- **실행** 하려면 **JRE(Java Runtime Environment)** 만 있으면 된다. 이미 컴파일된 JAR를 돌리기만 하면 되므로 훨씬 가볍다.

단일 스테이지로 만들면, 최종 이미지에 빌드에만 쓰였던 JDK와 소스 코드, 빌드 캐시가 전부 남는다. 정작 실행에는 쓰이지 않는 짐이다. 멀티스테이지는 이 짐을 버리고 **꼭 필요한 결과물(JAR)과 JRE만** 최종 이미지에 담는다.

```
   단일 스테이지 이미지 (~800MB)      멀티스테이지 이미지 (~300MB)
   ┌────────────────────────┐        ┌────────────────────────┐
   │ ▓▓▓ JDK (컴파일러 등)   │        │ ░░░ JRE (실행만)        │
   │ ▓▓▓ 소스 코드          │        │ ░░░ app.jar            │
   │ ▓▓▓ 빌드 캐시/의존성   │        │                        │
   │ ▓▓▓ Gradle             │        │                        │
   │ ▓▓▓ app.jar            │        │                        │
   └────────────────────────┘        └────────────────────────┘
     실행에 불필요한 짐이 가득          실행에 필요한 것만 남김
              ~800MB                          ~300MB
```

멀티스테이지에서는 1단계(`AS builder`)에서 JDK로 JAR를 만들고, 2단계에서 가벼운 JRE 이미지를 새로 시작한 뒤 `COPY --from=builder`로 **JAR 결과물만** 가져온다. 1단계에 있던 JDK, 소스코드, 빌드 캐시는 최종 이미지에 **일절 포함되지 않는다.**

### 비유: 빵집에서 오븐까지 배달할 것인가, 빵만 배달할 것인가

이 원리를 빵집에 비유하면 이렇다.

단일 스테이지는 마치 **빵을 굽고 나서 오븐, 반죽기, 밀가루 포대까지 전부 트럭에 실어 손님에게 배달하는 것**과 같다. 손님이 원한 건 빵 한 덩이인데, 주방 설비 전체가 딸려 온다. 트럭은 무겁고, 배달은 느리고, 손님 집은 발 디딜 틈이 없다.

멀티스테이지는 **주방에서 빵만 꺼내 예쁘게 포장해서 배달하는 것**이다. 오븐과 반죽기는 빵집에 남겨 둔다. 배달 트럭은 가볍고, 배송은 빠르며, 손님은 빵만 받는다.

```
   단일 스테이지                     멀티스테이지
   ┌──────────────────┐            ┌──────────────────┐
   │  🚚 트럭         │            │  🚚 트럭         │
   │   ├ 빵           │            │   └ 빵 (포장)    │
   │   ├ 오븐         │  ───▶      │                  │
   │   ├ 반죽기       │            │                  │
   │   └ 밀가루 포대  │            │                  │
   └──────────────────┘            └──────────────────┘
    무겁고 느린 배달                가볍고 빠른 배달
```

### 원리: 작아진 이미지가 주는 세 가지 이득

이미지를 작게 만드는 것은 단순히 디스크를 아끼는 문제가 아니다. 실무에서 세 가지 큰 이득이 있다.

- **배포가 빠르다**: 이미지가 작을수록 네트워크로 주고받는 시간이 줄어든다. 수백 대 서버에 배포한다면 이 차이는 어마어마하다.
- **공격 표면(attack surface)이 줄어든다**: 이미지 안에 든 프로그램이 많을수록 보안 취약점이 숨어 있을 곳도 많아진다. 컴파일러, 빌드 도구, 셸 유틸리티 같은 것들은 실행에 불필요하면서 잠재적 위험이 된다. 실행에 필요한 최소한만 남기면 공격당할 지점 자체가 줄어든다.
- **관리가 명확해진다**: 최종 이미지에 무엇이 들어 있는지 명확하므로, 무엇이 왜 거기 있는지 추적하기 쉽다.

```
   이미지가 작으면...
   ┌─────────────────────────────────────────────┐
   │ ✅ 배포 속도 ↑   (전송할 데이터가 적음)       │
   │ ✅ 보안 ↑        (불필요한 프로그램 = 취약점)  │
   │ ✅ 관리 ↑        (뭐가 왜 들었는지 명확)       │
   └─────────────────────────────────────────────┘
```

## ⚠️ 흔한 오해와 함정

**오해 1: "컨테이너는 가벼운 가상머신(VM)이다"**
겉보기엔 비슷하지만 근본이 다르다. VM은 **하드웨어를 가상화**해서 그 위에 게스트 OS 전체(커널 포함)를 통째로 올린다. 그래서 무겁고 부팅이 느리다. 반면 컨테이너는 **호스트의 OS 커널을 공유**하고 프로세스만 격리한다. 그래서 훨씬 가볍고 몇 초 만에 뜬다. "컨테이너 = 미니 VM"이라고 이해하면 왜 컨테이너가 그렇게 빠른지, 왜 리눅스 컨테이너가 리눅스 커널 위에서만 돈다는 제약이 있는지를 영영 이해하지 못한다.

**오해 2: "`latest` 태그를 쓰면 항상 최신이라 편하다"**
`postgres:latest`처럼 `latest` 태그를 쓰면, 이미지를 받는 시점에 따라 **버전이 제멋대로 바뀐다.** 오늘은 PostgreSQL 16이던 것이 다음 달엔 17이 될 수 있고, 그러면 "내 컴퓨터에서는 되는데" 문제가 Docker 안에서 다시 재현된다. Docker의 존재 이유(환경 고정)를 스스로 무너뜨리는 셈이다. **항상 명시적 버전(`postgres:16.2`)을 고정하라.**

**함정 1: "볼륨 없이 데이터를 저장한다"**
컨테이너는 언제든 버려질 수 있는(disposable) 존재다. 데이터베이스 컨테이너를 볼륨(volume) 설정 없이 띄우면, 컨테이너가 삭제되는 순간 그 안의 모든 데이터가 함께 증발한다. **영속적으로 보존해야 할 데이터는 반드시 볼륨에 담아**, 컨테이너의 생명주기와 데이터의 생명주기를 분리해야 한다. (붕어빵은 버려도 되지만, 붕어빵을 먹은 손님의 주문 기록은 남아야 한다.)

**함정 2: "시크릿을 이미지에 하드코딩한다"**
DB 비밀번호, API 키 같은 비밀 값을 Dockerfile이나 이미지 안에 직접 박아 넣으면 안 된다. 이미지는 여러 곳에 공유되고 레지스트리에 올라가며, 이미지의 각 레이어는 뜯어볼 수 있어서 하드코딩된 시크릿이 그대로 노출된다. 시크릿은 **환경변수, Docker secrets, 외부 비밀 관리 도구** 등으로 실행 시점에 주입해야 한다.

**함정 3: "빌드가 느려도 어쩔 수 없다"**
Dockerfile의 레이어 캐시를 이해하지 못하면 매번 전체를 다시 빌드하게 된다. 자주 바뀌지 않는 것(의존성 다운로드)을 위에, 자주 바뀌는 것(소스 코드)을 아래에 배치하면, 소스만 바꿨을 때 의존성 단계는 캐시를 재사용해 빌드가 훨씬 빨라진다. 캐시는 "위에서 아래로, 바뀌기 전까지 재사용"된다는 원리를 기억하자.

<details>
<summary>해설</summary>

Dockerfile은 레시피(어떻게 만들지 적은 글), Image는 붕어빵 틀(레시피로 만든 읽기 전용 설계도), Container는 붕어빵(틀로 찍어낸 실행 중인 실체)이다. `build`는 레시피(Dockerfile)로 틀(Image)을 만드는 과정이고, `run`은 틀(Image)로 붕어빵(Container)을 찍어내는 과정이다. 하나의 이미지로부터 컨테이너를 몇 개든 찍어낼 수 있고, 모두 동일한 초기 상태에서 출발한다. 컨테이너는 버려도 되는 존재이며 이미지만 있으면 언제든 다시 만들 수 있다.
</details>

**문제 3.** Docker Compose에서 `depends_on`만으로는 부족하고 `healthcheck`가 함께 필요한 이유를 설명하라. `pg_isready`가 여기서 하는 일은 무엇인가?

<details>
<summary>해설</summary>

`depends_on`은 컨테이너의 **시작 순서**만 보장한다. 그러나 컨테이너가 "시작"된 것과 그 안의 프로그램이 "요청을 받을 준비가 된" 것은 다르다. PostgreSQL 컨테이너가 켜져도 DB 엔진 초기화에 시간이 걸려, 그 사이 앱이 연결을 시도하면 "연결 거부"로 죽는다. `healthcheck`의 `pg_isready`는 DB가 **실제로 접속을 받을 수 있는 상태인지** 주기적으로 확인하는 명령이며, app의 `depends_on`에 `condition: service_healthy`를 지정하면 "db가 건강하다고 확인된 뒤"에 app을 시작하므로 이 경쟁 상태를 막을 수 있다.
</details>

**문제 4.** 멀티스테이지 빌드가 이미지 크기를 ~800MB에서 ~300MB로 줄일 수 있는 이유를 "JDK와 JRE의 차이" 관점에서 설명하라. 그리고 이미지가 작아질 때의 이득 세 가지를 들라.

<details>
<summary>해설</summary>

빌드(컴파일)에는 JDK(컴파일러·빌드도구 포함, 무거움)가 필요하지만, 실행에는 JRE(런타임만, 가벼움)만 있으면 된다. 멀티스테이지는 1단계에서 JDK로 JAR를 빌드하고, 2단계에서 가벼운 JRE 이미지를 시작해 `COPY --from=builder`로 JAR 결과물만 가져온다. 그 결과 JDK·소스코드·빌드 캐시가 최종 이미지에서 빠져 크기가 크게 줄어든다. 이득: ① 배포 속도 향상(전송 데이터 감소), ② 보안 강화(불필요한 프로그램 = 잠재 취약점이 줄어 공격 표면 감소), ③ 관리 명확성(뭐가 왜 들었는지 추적 용이).
</details>

**문제 5.** 다음은 잘못된 습관들이다. 각각 무엇이 문제이며 어떻게 고쳐야 하는지 설명하라. (a) `image: postgres:latest` 사용, (b) DB 컨테이너를 볼륨 설정 없이 실행, (c) Dockerfile에 `ENV DB_PASSWORD=1234` 하드코딩.

<details>
<summary>해설</summary>

(a) `latest`는 받는 시점에 따라 버전이 바뀌어 "환경 고정"이라는 Docker의 목적을 스스로 깨뜨린다 → 명시적 버전(`postgres:16.2`)으로 고정한다. (b) 볼륨이 없으면 컨테이너 삭제 시 데이터가 함께 사라진다 → 영속 데이터는 named volume에 저장해 컨테이너 생명주기와 데이터 생명주기를 분리한다. (c) 이미지는 공유·배포되고 레이어를 뜯어볼 수 있어 하드코딩된 시크릿이 노출된다 → 시크릿은 환경변수·Docker secrets·외부 비밀 관리 도구로 실행 시점에 주입한다.
</details>

## 요약

- **Docker가 태어난 이유는 "내 컴퓨터에서는 되는데" 문제**다. 코드는 같아도 OS·런타임·DB 버전 등 환경 차이가 결과를 다르게 만든다. Docker는 앱과 실행 환경을 통째로 이미지에 담아, 어디서 실행하든 내부 세계가 동일함을 보장한다. (이사 짐을 하나하나 vs 컨테이너째)
- **핵심 3단어**: Dockerfile(레시피) → `build` → Image(읽기 전용 설계도) → `run` → Container(살아있는 실체). 붕어빵 비유: 레시피 / 틀 / 붕어빵. 이미지는 불변이고 컨테이너는 몇 개든 찍어낼 수 있으며 버려도 된다.
- **Dockerfile의 각 줄**: `FROM`(기반 이미지), `WORKDIR`(작업 폴더), `COPY`(파일 복사), `RUN`(빌드 시점 실행), `ENTRYPOINT`(실행 시점 명령). `RUN`과 `ENTRYPOINT`의 시점 차이가 핵심이다.
- **Docker Compose**는 여러 컨테이너(Spring+PostgreSQL+Redis+Nginx)를 하나의 `docker-compose.yml`로 정의하고 지휘한다. 서비스 이름이 곧 호스트명이 되어(내장 DNS) IP 없이 통신하며, `depends_on`(순서) + `healthcheck`(준비 상태, `pg_isready`)를 함께 써야 "연결 거부" 재앙을 막는다.
- **멀티스테이지 빌드**는 빌드(JDK 필요, 무거움)와 실행(JRE만 필요, 가벼움)을 분리한다. 1단계에서 빌드하고 2단계에서 결과물(JAR)만 가져와 이미지를 ~800MB → ~300MB로 줄인다. 배포 속도·보안·관리 모두 개선된다. (오븐까지 배달 vs 빵만 배달)
- **핵심 태도**: 컨테이너 ≠ VM(커널 공유), `latest` 금지·버전 고정, 데이터는 볼륨에, 시크릿은 절대 하드코딩 금지. 그리고 "왜 이미지를 작게, 왜 환경을 고정하는가"를 항상 물어라.
